## 二、WebEnv-OS 项目深度拷问

### 2.1 窗口管理系统

#### 拷问 1：窗口层级管理

> **面试官**："你说做了窗口管理系统，那我来问你几个问题。"
>
> "窗口的 z-index 你是怎么管理的？如果同时打开 20 个窗口，z-index 会不会乱？"

**考察点**：状态设计、层级算法、边界情况

**参考答案**：

在 WebEnv-OS 中，我们采用**最大 z-index 递增策略**：

```typescript
// 窗口激活时，设置为当前最大 z-index + 1
const activateWindow = useCallback((id: string) => {
  setActiveWindowId(id);
  setWindows((prev) => {
    const maxZIndex = Math.max(...prev.map((w) => w.zIndex), 100);
    return prev.map((w) =>
      w.id === id ? { ...w, zIndex: maxZIndex + 1 } : w
    );
  });
}, []);
```

**关键设计点**：
1. **基础层级基准**：所有窗口从 z-index: 100 开始，给系统组件留空间
2. **点击递增**：每次窗口获得焦点，z-index 设为最大值 + 1
3. **状态过滤**：只渲染非最小化的可见窗口

**可能的继续追问**：
- "z-index 会不会溢出？JavaScript Number 有上限吗？"
- "多标签页情况下，窗口层级怎么同步？"
- "拖拽动画期间点击其他窗口，怎么处理？"

**追问参考回答**：

1. **z-index 溢出问题**
  - JavaScript Number 的安全整数上限是 $2^{53}-1$，理论上很难点到溢出，但工程上不能“赌不会发生”。
  - 实际做法：当 `maxZIndex` 超过阈值（如 100000）时做一次**层级压缩重排**，按当前可见窗口顺序重新分配为 100~N，避免长期增长。

2. **多标签页同步**
  - 用 `BroadcastChannel` 或 `storage` 事件广播窗口激活事件（`windowId + timestamp`）。
  - 采用“最后写入优先（LWW）”并加标签页 `instanceId`，防止循环更新。
  - 只同步必要状态（激活窗口、布局版本号），不强制同步拖拽中间态，减少抖动。

3. **拖拽期间点击处理**
  - 拖拽开始后设置 `isDragging`，在 `pointerup` 前屏蔽其他窗口激活事件，避免焦点乱跳。
  - 若产品要求“可抢焦点”，则将点击事件入队，拖拽结束后再执行一次 `activateWindow`。
  - 统一使用 `pointer capture`，保证拖拽状态与点击状态一致收敛。

#### 拷问 2：窗口拖拽边界处理

> **面试官**："窗口拖拽功能，如果用户把窗口拖到屏幕外去了，怎么办？"

**参考答案**：

需要做**边界检测和吸附**：

```typescript
const handleDragMove = useCallback((e: MouseEvent) => {
  const deltaX = e.clientX - dragStartPos.current.x;
  const deltaY = e.clientY - dragStartPos.current.y;

  let newX = windowStartPos.current.x + deltaX;
  let newY = windowStartPos.current.y + deltaY;

  // 边界检测：限制在屏幕内
  const minX = 0;
  const maxX = window.innerWidth - 100; // 保留最小可见宽度
  const minY = 0;
  const maxY = window.innerHeight - 50; // 保留标题栏

  newX = Math.max(minX, Math.min(maxX, newX));
  newY = Math.max(minY, Math.max(maxY, newY));

  moveWindow(windowId, { x: newX, y: newY });
}, []);
```

### 2.2 文件系统

#### 拷问 3：虚拟文件系统设计

> **面试官**："你说用了虚拟文件系统，那你说说浏览器里怎么做文件存储？为什么不用 localStorage？"

**考察点**：浏览器存储方案、数据结构设计

**参考答案**：

**localStorage 的问题**：
- 容量只有 5-10MB
- 只能存字符串，需要 JSON 序列化
- 同步 API，阻塞主线程
- 不支持复杂数据结构

**IndexedDB 的优势**：
- 容量大（硬盘剩余空间）
- 支持复杂数据类型（Blob、File）
- 异步 API，不阻塞
- 支持索引和事务

**虚拟文件系统设计**：

```typescript
// ZenFS - 基于 IndexedDB 的虚拟文件系统
interface VirtualFileSystem {
  // 文件结构
  files: Map<string, VFile>;
  directories: Map<string, VDirectory>;

  // 核心操作
  readFile(path: string): Promise<VFile | null>;
  writeFile(path: string, content: string): Promise<void>;
  createDirectory(path: string): Promise<void>;
  delete(path: string): Promise<void>;
  listDirectory(path: string): Promise<VEntry[]>;
}

// 文件元数据
interface VFile {
  name: string;
  path: string;
  content: string | Blob;
  size: number;
  createdAt: Date;
  updatedAt: Date;
  isDirectory: false;
}
```

**追问**：
- "文件内容很大（比如视频）怎么办？"
- "怎么实现文件的模糊搜索？"
- "目录嵌套很深（比如 10 层），怎么优化遍历？"

**追问参考回答**：

1. **大文件存储（视频）**
  - 元数据（路径、大小、MIME、hash）放 IndexedDB；内容用 `Blob` 分片存储（chunk）。
  - 读取走流式（`ReadableStream`）或分段读取，避免一次性加载进内存。
  - 上传/同步使用断点续传（chunk + 校验），失败只重传缺失分片。

2. **模糊搜索**
  - 建立文件名倒排索引（token -> path 列表），支持前缀匹配。
  - 再做二次打分：编辑距离（Levenshtein）+ 最近访问时间 + 路径深度。
  - 搜索过程放到 Web Worker，主线程只接收前 N 条结果。

3. **深层目录遍历优化**
  - 目录树结构缓存为 `path -> childrenIds`，避免重复读取。
  - 遍历采用迭代（栈/队列）替代递归，防止极端深度栈溢出。
  - 懒展开：只遍历可见分支，后台做低优先级预取。

#### 拷问 4：文件树性能优化

> **面试官**："你说文件树做了性能优化，如果有 1 万个文件夹同时展示，会卡吗？"

**参考答案**：

需要**虚拟化渲染**：

```typescript
// 只渲染可见区域的文件
const VirtualFileTree: React.FC<{ entries: VEntry[] }> = ({ entries }) => {
  const [scrollTop, setScrollTop] = useState(0);
  const ITEM_HEIGHT = 32;
  const VISIBLE_COUNT = 15;

  // 计算可见范围
  const startIndex = Math.floor(scrollTop / ITEM_HEIGHT);
  const endIndex = Math.min(
    entries.length,
    startIndex + VISIBLE_COUNT + 2 // 额外渲染上下各 2 个
  );

  // 只渲染可见项
  const visibleEntries = entries.slice(startIndex, endIndex);

  return (
    <div onScroll={handleScroll}>
      <div style={{ height: entries.length * ITEM_HEIGHT, position: 'relative' }}>
        {visibleEntries.map((entry, index) => (
          <div
            style={{
              position: 'absolute',
              top: (startIndex + index) * ITEM_HEIGHT,
              height: ITEM_HEIGHT,
            }}
          >
            <FileTreeItem entry={entry} />
          </div>
        ))}
      </div>
    </div>
  );
};
```

**追问**：
- "展开/折叠状态怎么管理？"
- "虚拟列表和展开状态怎么结合？"

**追问参考回答**：

1. **展开/折叠状态管理**
  - 用 `Set<string>` 存展开节点 ID（如路径），更新是 $O(1)$。
  - 展开状态独立于数据源，目录刷新时按 ID 复用，避免 UI 闪烁。
  - 可选持久化到 localStorage/IndexedDB，恢复用户上次浏览位置。

2. **虚拟列表结合展开状态**
  - 先把树按当前展开态拍平成 `visibleNodes`（仅可见节点）。
  - 虚拟列表只消费 `visibleNodes.length` 与索引映射，不直接操作原树。
  - 展开/折叠时增量更新拍平结果，并配合 `requestAnimationFrame` 批量提交，减少重排。

### 2.3 终端模拟器

#### 拷问 5：xterm.js 与后端通信

> **面试官**："终端模拟器怎么和后端交互？命令执行结果怎么实时返回？"

**参考答案**：

使用**WebSocket 流式传输**：

```typescript
// 前端 - 订阅命令输出
useEffect(() => {
  const socket = io('/terminal');

  socket.on('output', (data: string) => {
    // 实时追加到终端
    term.write(data);
  });

  return () => socket.disconnect();
}, []);

// 后端 - 执行命令并流式输出
@SubscribeMessage('execute')
async handleExecute(client: Socket, command: string) {
  const process = spawn(command, [], { shell: true });

  // stdout 流式输出
  process.stdout.on('data', (data) => {
    client.emit('output', data.toString());
  });

  // stderr 也要输出
  process.stderr.on('data', (data) => {
    client.emit('output', `\x1b[31m${data}\x1b[0m`); // 红色
  });

  process.on('close', (code) => {
    client.emit('output', `\r\nprocess exited with code ${code}\r\n`);
  });
}
```

**追问**：
- "如果命令执行很慢（比如 `sleep 10`），怎么办？"
- "怎么支持命令历史？上下键切换？"

**追问参考回答**：

1. **慢命令处理**
  - 命令执行与会话分离：每条命令有 `commandId`，前端可取消（发送 kill 信号）。
  - 展示运行态（spinner + elapsed time），并设置超时策略（软超时提醒、硬超时终止）。
  - 长任务输出做背压/节流（例如每 50ms flush 一批），防止终端刷屏卡顿。

2. **命令历史与上下键**
  - 前端维护 ring buffer（如最多 500 条），持久化到 IndexedDB。
  - `ArrowUp/ArrowDown` 仅在光标位于输入末尾时接管，避免覆盖正常编辑行为。
  - 支持历史搜索（`Ctrl+R`）和去重策略（连续相同命令合并）。

### 2.4 WebContainer

#### 拷问 6：为什么需要 COOP/COEP 头？

> **面试官**："你说用了 WebContainer，那你知道为什么需要 COOP 和 COEP 头吗？"

**考察点**：浏览器安全策略、跨域隔离

**参考答案**：

**COOP** (Cross-Origin-Opener-Policy)
- 控制页面如何打开其他页面
- `same-origin`：只允许同源页面打开
- 防止恶意页面通过 window.open 控制你的页面

**COEP** (Cross-Origin-Embedder-Policy)
- 控制页面能否嵌入跨域资源
- `require-corp`：要求所有嵌入资源明确允许
- 只有启用 COEP，SharedArrayBuffer 才能使用

**WebContainer 需要**：
- SharedArrayBuffer 用于创建隔离的 JavaScript 运行环境
- 需要 COOP: same-origin + COEP: require-corp

```typescript
// next.config.js 配置
module.exports = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          { key: 'Cross-Origin-Opener-Policy', value: 'same-origin' },
          { key: 'Cross-Origin-Embedder-Policy', value: 'require-corp' },
        ],
      },
    ];
  },
};
```

**追问**：
- "如果服务器不能配置这个头怎么办？"
- "COOP 和 COEP 有什么副作用？"

**追问参考回答**：

1. **服务器不能配头的兜底方案**
  - 最优：在可控反向代理层（Nginx/Edge/Cloudflare）注入 COOP/COEP。
  - 若完全不可控，只能降级：不启用 WebContainer，改为远端容器执行（云端沙箱 + WebSocket）。
  - 明确提示能力降级：例如“当前环境不支持本地 Node 运行，仅支持只读预览”。

2. **COOP/COEP 副作用**
  - 一些第三方脚本/iframe 若未设置 CORP/CORS 会被拦截。
  - 跨窗口引用关系受限（`window.opener` 行为变化），部分旧登录流程会受影响。
  - 解决方式：资源白名单、改造第三方加载方式、将不兼容页面隔离到子域。

### 2.5 性能优化

#### 拷问 7：多窗口性能问题

> **面试官**："如果用户同时打开 10 个窗口（包括 Monaco Editor、3D 场景、多个文件编辑器），怎么保证流畅？"

**策略 3：降级处理**

```typescript
// 检测设备性能，降级渲染
const usePerformanceTier = () => {
  const [tier, setTier] = useState<'high' | 'medium' | 'low'>('high');

  useEffect(() => {
    // 根据 CPU 核心数和内存判断
    const cores = navigator.hardwareConcurrency || 4;
    const memory = (navigator as any).deviceMemory || 4;

    if (cores >= 8 && memory >= 8) {
      setTier('high');
    } else if (cores >= 4 && memory >= 4) {
      setTier('medium');
    } else {
      setTier('low');
    }
  }, []);

  return tier;
};
```

### 3.1 富文本编辑器

#### 拷问 8：为什么选择 TipTap？

> **面试官**："你说用了 TipTap 做编辑器，那为什么不用 Slate.js 或者 Quill？"

**考察点**：技术选型、框架对比

**参考答案**：

**TipTap vs Slate.js**：

| 维度 | TipTap | Slate.js |
|------|--------|----------|
| 底层 | ProseMirror | 自定义 |
| 生态 | 丰富插件 | 需要自己实现 |
| 协作 | 原生 Yjs 支持 | 需要集成 |
| 文档 | 完善 | 一般 |
| 学习曲线 | 较平缓 | 陡峭 |

**TipTap vs Quill**：

| 维度 | TipTap | Quill |
|------|--------|-------|
| 扩展性 | 非常强 | 一般 |
| 自定义 | 灵活 | 困难 |
| 协作 | 支持 | 需要自己集成 |

**为什么选 TipTap**：
1. 基于 ProseMirror（生产级验证）
2. Yjs 原生支持协作
3. 插件生态丰富
4. React 集成好

**追问**：
- "ProseMirror 和 Slate 的核心区别是什么？"
- "如果不用 TipTap，能自己基于 ProseMirror 封装吗？"

**追问参考回答**：

1. **ProseMirror vs Slate 核心区别**
  - ProseMirror 是“文档模型 + 事务 + 插件系统”的完整框架，约束强、可预测性高。
  - Slate 更偏 React-first 的可编程编辑器内核，自由度高，但很多能力需自行实现。
  - 协作与复杂 schema 场景下，ProseMirror 一般更稳；高度定制交互时 Slate 更灵活。

2. **是否可自行封装 ProseMirror**
  - 可以，TipTap 本质就是对 ProseMirror 的工程化封装。
  - 但需要补齐：命令系统、扩展规范、粘贴清洗、快捷键、历史与协作桥接、测试基线。
  - 结论：业务早期优先 TipTap 提效；当有强定制且团队有维护能力时再自研封装。

#### 拷问 9：编辑器数据存储

> **面试官**："编辑器的内容，你是怎么存储的？JSON 还是 HTML？各有什么优缺点？"

**参考答案**：

**JSON 格式（推荐）**：

```json
{
  "type": "doc",
  "content": [
    {
      "type": "paragraph",
      "content": [
        { "type": "text", "text": "Hello " },
        { "type": "text", "marks": [{ "type": "bold" }], "text": "World" }
      ]
    }
  ]
}
```

**优点**：
- 结构化，便于程序处理
- 便于协同编辑（Yjs 原生支持）
- 便于版本控制
- 便于转换格式

**缺点**：
- 体积较大
- 需要解析才能显示

**HTML 格式**：

```html
<p>Hello <strong>World</strong></p>
```

**优点**：
- 直接可显示
- 体积较小

**缺点**：
- 结构不清晰
- 协作困难
- 样式耦合

**FastDocument 的选择**：JSON + HTML 双存储

```typescript
// 保存时
const saveDocument = async (editor: Editor) => {
  // JSON 用于协作和程序处理
  const jsonContent = editor.getJSON();

  // HTML 用于展示
  const htmlContent = editor.getHTML();

  await api.save({
    json: jsonContent,
    html: htmlContent,
    updatedAt: new Date(),
  });
};
```

### 3.2 实时协作

#### 拷问 10：Yjs 冲突解决

> **面试官**："你说用了 Yjs 做协作，那两个人同时修改同一个地方，怎么解决冲突？"

**考察点**：CRDT、冲突解决

**参考答案**：

**Yjs 使用 CRDT（无冲突复制数据类型）**：

```typescript
// 1. 文本使用 RGA（Replicated Growable Array）
// 每个字符有唯一 ID：{ clientId, clock }
// 插入操作按 ID 排序，结果一致

// 示例：两个用户同时在 "ABC" 的 B 后插入
// 用户1：插入 "X" -> ID: {1, 1}
// 用户2：插入 "Y" -> ID: {2, 1}
// 结果：按 ID 排序 -> AXBYC 或 AYBXC（最终一致）

// 2. Y.Text 的操作
const ytext = ydoc.getText('content');

// 插入
ytext.insert(0, 'Hello', [{ bold: true }]); // 带格式

// 删除（标记删除，不影响其他插入）
ytext.delete(0, 5);

// 3. 观察变化
ytext.observe((event) => {
  event.delta.forEach((op) => {

      console.log('插入:', op.insert);
    } else if (op.delete) {
      console.log('删除:', op.delete, '个字符');
```

**追问**：
- "CRDT 和 OT（Operational Transformation）有什么区别？"
- "如果网络断开很长时间再重连，会发生什么？"
- "怎么防止恶意用户发送垃圾数据？"

**追问参考回答**：

1. **CRDT vs OT**
  - OT：通过转换并发操作来保持一致，依赖中心顺序与转换函数正确性。
  - CRDT：数据结构天然可交换/可合并，最终一致，不强依赖中心协调。
  - 实践上：OT 控制精细、实现复杂；CRDT 离线友好、实现心智更统一（Yjs 选 CRDT）。

2. **长时间离线后重连**
  - 客户端会把离线期间增量 update 与服务端状态合并。
  - 合并后文档达到最终一致，光标/在线状态等临时态会重新协商。
  - 需要做快照压缩（snapshot + gc），避免一次重连传输过大导致卡顿。

3. **防恶意垃圾数据**
  - 鉴权：WebSocket 握手 JWT + 文档级 ACL。
  - 限流：按用户/文档 QPS、包大小、频率阈值做 rate limit。
  - 校验：服务端验证 update 大小与 schema 合法性，异常客户端熔断与封禁。

**追问**：
- "STUN 和 TURN 服务器的区别是什么？"
- "如果有 100 个人同时开会，怎么办？"

**追问参考回答**：

1. **STUN vs TURN**
  - STUN：帮助发现公网地址并打洞，媒体仍走点对点，成本低。
  - TURN：当直连失败时走中继转发，成功率高但带宽成本高、时延更大。
  - 实际部署：优先 STUN，失败自动回落 TURN，监控 TURN 占比评估网络质量。

2. **100 人会议架构**
  - 不能全员 P2P，需 SFU（如 LiveKit/mediasoup）集中转发。
  - 订阅分层：主讲人高清，其余低清或仅音频；非活跃视频暂停订阅。
  - 前端做分页/舞台模式，限制同时解码路数（如 9~16 路）保证端上性能。

#### 追问 A：ICE 为什么会失败？怎么排查？

**参考答案**：

常见失败原因：
1. 双方都在对称 NAT，UDP 打洞失败
2. 企业网络屏蔽 UDP / 限制端口
3. TURN 配置错误（凭证过期、TLS 证书问题）

排查方法：
1. 打开 `chrome://webrtc-internals` 看 candidate pair 是否成功切换到 `succeeded`
2. 检查是否拿到了 `relay` candidate（没有通常是 TURN 不可用）
3. 服务端记录 ICE fail reason 和地区网络画像，针对性扩容 TURN 节点

---

#### 追问 B：为什么需要 signaling（信令）服务器？

**参考答案**：

WebRTC 不负责 offer/answer/candidate 的“传输通道”，只定义协议；
因此必须有信令层（WebSocket/HTTP）来交换：
- SDP offer/answer
- ICE candidates
- 房间与用户状态（谁进房、谁离房、谁上麦）

结论：**媒体是点对点/经 SFU，控制面一定要有中心信令系统**。

---

#### 追问 C：SDP 里到底有什么？你关心哪些字段？

**参考答案**：

SDP 主要描述媒体协商参数：
- 媒体类型与端口：`m=audio` / `m=video`
- 编解码能力：`rtpmap`（如 Opus、VP8、H264）
- 方向属性：`sendrecv` / `recvonly`
- 网络候选：`candidate`

实战最关注：
1. 编解码优先级是否符合预期（移动端常优先 H264）
2. 是否协商到 `rtcp-mux`、`bundle`（减少端口与开销）
3. 是否出现不兼容 profile 导致黑屏/无声

---

#### 追问 D：弱网下如何保通话质量？

**参考答案**：

核心是“自适应 + 降级”：
1. 开启带宽估计（GCC/Transport-CC），动态调码率
2. 开启 simulcast/SVC，多层码流按订阅端网络下发
3. 音频优先：网络差时先降视频分辨率和帧率，再考虑停视频
4. 启用 NACK/PLI/FEC，丢包时优先恢复关键帧

经验阈值：当丢包率持续 > 10% 且 RTT > 300ms，通常要进入“语音优先模式”。

#### 追问 E：怎么做屏幕共享？和摄像头流有什么差异？

**参考答案**：

屏幕共享使用 `getDisplayMedia`，摄像头使用 `getUserMedia`。

关键差异：
1. 屏幕内容细节多、文字多，需要更高分辨率/更稳定码率
2. 通常帧率可低于摄像头（例如 10~15fps 也可接受）
3. 要处理“用户停止共享”事件并自动回退到摄像头轨道

#### 追问 G：WebRTC 安全性怎么保证？

**参考答案**：

#### 拷问 14：React Three Fiber 原理

> **面试官**："你说用了 React Three Fiber，那它和原生 Three.js 有什么区别？为什么选它？"

**参考答案**：

**R3F vs 原生 Three.js**：

| 维度 | React Three Fiber | 原生 Three.js |
|------|-------------------|----------------|
| 开发范式 | React 组件化 | 命令式 API |
| 状态管理 | React 状态 | 手动管理 |
| 生命周期 | React 生命周期 | 手动管理 |
| 性能 | 需要注意 | 完全控制 |
| 学习曲线 | 熟悉 React 即可 | 需要学 Three.js |

**R3F 本质**：把 Three.js 的对象映射为 React 组件

```typescript

  <mesh>

    <meshStandardMaterial color="red" />
  </mesh>
</Canvas>

// 等价于原生 Three.js
const scene = new THREE.Scene();
scene.add(new THREE.AmbientLight(0.5));

const mesh = new THREE.Mesh(
  new THREE.BoxGeometry(),
  new THREE.MeshStandardMaterial({ color: 'red' })
);
scene.add(mesh);

const renderer = new THREE.WebGLRenderer();
renderer.render(scene, camera);
```

**追问**：
- "R3F 的性能会比原生 Three.js 差吗？"
- "如果不用 R3F，能直接用 Three.js 吗？"

**追问参考回答**：

1. **R3F 性能是否更差**
  - 理论上有少量 React 协调开销，但渲染瓶颈通常在 GPU/DrawCall，不在框架层。
  - 关键是避免把高频动画放进 React state，改用 `useFrame + ref` 直改对象。
  - 做到这一点后，R3F 在大多数业务场景可接近原生 Three.js 性能。

2. **不用 R3F 可否直接 Three.js**
  - 完全可以，且在极限性能与底层控制上更直接。
  - 代价是要自己管理生命周期、资源释放、事件系统和与 React 状态同步。
  - 选型原则：团队 React 经验强、迭代快选 R3F；极致性能/底层定制选原生。

**优化策略 2：对象池**

```typescript
// 预创建卡牌对象，避免运行时创建
class CardPool {
  private pool: THREE.Mesh[] = [];
  private activeCards: Set<string> = new Set();

  initialize(count: number) {
    for (let i = 0; i < count; i++) {
      const card = new THREE.Mesh(
        new THREE.BoxGeometry(0.6, 0.9, 0.01),
        new THREE.MeshStandardMaterial()
      );
      card.visible = false;
      this.pool.push(card);
      scene.add(card);
    }
  }

  acquire(id: string, position: THREE.Vector3) {
    const card = this.pool.find(c => !c.visible);
    if (card) {
      card.position.copy(position);
      card.visible = true;
      this.activeCards.add(id);
    }
    return card;
  }

  release(id: string) {
    const card = this.pool.find(c => c.visible && this.activeCards.has(id));
    if (card) {
      card.visible = false;
      this.activeCards.delete(id);
    }
  }
}
```

```typescript
interface AIStrategy {
  name: string;
  // 选择出哪张牌
  chooseCard(hand: Card[], topCard: Card, gameState: GameState): Card | null;
  // 是否喊 UNO
  shouldShoutUno(hand: Card[]): boolean;
  // 是否质疑

// 简单难度：随机出牌
const EasyAI: AIStrategy = {
  name: 'easy',
  chooseCard(hand, topCard) {
    // 随机选一张能出的牌
    const playable = hand.filter(card => canPlay(card, topCard));
    return playable[Math.floor(Math.random() * playable.length)] || null;
  },
  shouldShoutUno() {
    return Math.random() > 0.3; // 30% 概率喊
  },
  shouldChallenge() {
    return false; // 从不质疑
  },
};

// 困难难度：最优选择
const HardAI: AIStrategy = {
  name: 'hard',
  chooseCard(hand, topCard, gameState) {
    // 1. 如果能出 +4 或变色牌，优先出（摆脱难题）
    const wildCards = hand.filter(c => c.isWild);
    if (wildCards.length > 0) {
      return wildCards[0];
    }

    // 2. 统计手牌颜色，选择最多数量的颜色
    const colorCount = countColors(hand);
    const dominantColor = Object.entries(colorCount)
      .sort((a, b) => b[1] - a[1])[0][0];

    // 3. 找出能出的牌，优先出匹配颜色的
    const playable = hand.filter(c => canPlay(c, topCard));
    return playable.find(c => c.color === dominantColor) || playable[0] || null;
  },
  shouldShoutUno() {
    return true; // 手上剩 1 张必喊
  },
  shouldChallenge(target) {
    // 如果上家只剩 1 张且出了 +4，有 80% 概率质疑
    return target.handCount === 1 && target.lastCard?.isPlusFour && Math.random() > 0.2;
```

```typescript
// 2. 断线重连
socket.on('disconnect', (reason) => {
  if (reason === 'io server disconnect') {
    // 服务器主动断开，需要手动重连
    socket.connect();
  }
});

// 3. 消息确认
const sendWithAck = (event, data) => {
  return new Promise((resolve, reject) => {
    socket.emit(event, data, (ack) => {
      if (ack.success) {
        resolve(ack.data);
      } else {
        reject(ack.error);
      }
    });
  });
};

// 4. 游戏状态同步
socket.on('gameStateUpdate', (state) => {
  // 如果和本地状态不一致，以服务器为准
  // 但保留本地未发送的操作，重新应用
  const localPendingOps = getLocalPendingOps();
  applyServerState(state);

  // 重新应用本地操作
  localPendingOps.forEach(op => sendWithAck('reapply', op));
});
```

### 5.2 压力追问

> **面试官**："你说的方案有问题吗？能改进吗？"

**常见问题**：
1. "如果房间人数是 1000 人呢？"
2. "如果需要历史消息呢？"
3. "如果有人发垃圾消息怎么办？"
4. "如果服务器挂了怎么办？"

**追问参考回答**：

1. **房间 1000 人**
  - 拆为“逻辑房间 + 广播分组”（如每组 100~200），服务端聚合后分发。
  - 前端默认只订阅最近消息与活跃用户增量，成员列表分页/按需拉取。
  - 高频事件（输入中、在线状态）降采样，避免广播风暴。

2. **历史消息需求**
  - 热数据放 Redis（最近 N 条），冷数据落库（MySQL/ES/ClickHouse 按场景选）。
  - 游标分页（`messageId`/`timestamp`）向前翻页，避免深分页性能问题。
  - 做消息幂等 ID，防重放、防重复插入。

3. **垃圾消息治理**
  - 前置风控：敏感词、频率限制、验证码/人机校验。
  - 账号体系：禁言、踢出、黑名单、设备指纹联合惩罚。
  - 内容审核：异步审核 + 回撤机制，严重内容实时拦截。

4. **服务器故障**
  - 网关多副本 + 负载均衡 + 健康检查，连接自动迁移。
  - 消息链路做持久化与 ACK 重试，客户端断线重连后按游标补拉。
  - 关键指标（连接数、延迟、丢包、积压）告警，支持快速熔断与降级。

### 7.3 压力应对

```
面试官施压时的应对策略：

1. 承认不确定 → "这个我不太确定，但我猜测..."

3. 展示思考过程 → "让我分析一下...首先...其次..."
4. 诚实承认 → "这个问题很好，我之前确实没深入想过..."

记住：面试官不是在为难你，是在探测你的真实水平。
诚实 + 思考过程 > 硬撑。
```
