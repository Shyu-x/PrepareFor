## 一、网络协议基础

### 1.1 TCP 三次握手与四次挥手

**参考答案：**

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCP 三次握手                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│    客户端                              服务端                     │
│      │                                  │                        │
│      │  ──────  SYN = 1, seq = x  ────▶ │                       │
│      │                                  │                        │
│      │  ◀─────  SYN = 1, ACK = 1    ────│                       │
│      │          seq = y, ack = x + 1    │                        │
│      │                                  │                        │
│      │  ──────  ACK = 1, seq = x + 1  ──▶│                       │
│      │          ack = y + 1             │                        │
│      │                                  │                        │
│      │         [连接建立]                │                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**三次握手过程**：

| 步骤 | 客户端 | 服务端 | 说明 |
| :--- | :--- | :--- | :--- |
| 1 | SYN=1, seq=x | | 客户端发送连接请求 |
| 2 | | SYN=1, ACK=1, seq=y, ack=x+1 | 服务端确认并发送 SYN |
| 3 | ACK=1, seq=x+1, ack=y+1 | | 客户端确认，连接建立 |

**四次挥手过程**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCP 四次挥手                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│    客户端                              服务端                     │
│      │                                  │                        │
│      │  ──────  FIN = 1, seq = u   ────▶ │   客户端发送断开请求   │
│      │                                  │                        │
│      │  ◀─────  ACK = 1, ack = u + 1 ───│   服务端确认           │
│      │                                  │                        │
│      │                                  │   (服务端可能还有数据)  │
│      │                                  │                        │
│      │  ◀─────  FIN = 1, seq = v   ────│   服务端发送断开请求   │
│      │                                  │                        │
│      │  ──────  ACK = 1, ack = v + 1 ──▶│   客户端确认           │
│      │                                  │                        │
│      │    [等待 2MSL 后关闭]            │                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

| 步骤 | 客户端 | 服务端 | 说明 |
| :--- | :--- | :--- | :--- |
| 1 | FIN=1, seq=u | | 客户端发送断开请求 |
| 2 | | ACK=1, ack=u+1 | 服务端确认 |
| 3 | | FIN=1, seq=v | 服务端发送断开请求 |
| 4 | ACK=1, ack=v+1 | | 客户端确认，等待 2MSL 后关闭 |

**为什么需要三次握手**：
- 防止已失效的连接请求报文被服务器接收，导致错误建立连接
- 确保双方都能收发数据

**为什么需要四次挥手**：
- TCP 是全双工通信，需要双方分别发送 FIN
- 确保数据全部传输完毕

### 1.2 TCP 与 UDP 的区别

**参考答案：**

| 特性 | TCP | UDP |
| :--- | :--- | :--- |
| **连接性** | 面向连接 | 无连接 |
| **可靠性** | 可靠（确认、重传、排序） | 不可靠 |
| **传输方式** | 字节流 | 报文 |
| **效率** | 较低 | 较高 |
| **头部大小** | 20-60 字节 | 8 字节 |
| **拥塞控制** | 有 | 无 |
| **流量控制** | 有 | 无 |
| **使用场景** | HTTP、HTTPS、FTP、SMTP | DNS、VoIP、视频流 |

### 1.3 HTTP/1.0 / 1.1 / 2 / 3 区别

**参考答案：**

```
┌─────────────────────────────────────────────────────────────────┐
│                      HTTP 协议演进                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  HTTP/1.0                          HTTP/1.1                    │
│  ├─ 短连接（每次请求建连）         ├─ 持久连接（Keep-Alive）    │
│  ├─ 不支持断点续传                 ├─ 管线化（ pipelining）    │
│  └─ 无虚拟主机支持                 ├─ 支持断点续传             │
│                                       ├─ 虚拟主机支持           │
│                                       └─ 缓存更新               │
│                                                                  │
│  HTTP/2                              HTTP/3                     │
│  ├─ 二进制分帧                     ├─ QUIC 协议（UDP）         │
│  ├─ 多路复用（单一连接）           ├─ 0-RTT 连接               │
│  ├─ 头部压缩（HPACK）             ├─ 无队头阻塞               │
│  ├─ 服务端推送                     ├─ 连接迁移                 │
│  └─ 优先级设置                     └─ 更好的拥塞控制           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**HTTP/1.1 特性**：
- 持久连接（Keep-Alive）：复用 TCP 连接
- 管道化（Pipelining）：并行发送请求（但响应需按序）
- 断点续传（Range 请求）
- 虚拟主机（Host 头）

**HTTP/2 特性**：
- 二进制分帧：所有传输分割为更小的帧
- 多路复用：一个 TCP 连接可并行多个请求/响应
- HPACK 头部压缩：静态表 + 动态表
- 服务端推送：主动推送资源
- 优先级：可设置请求优先级

**HTTP/3 特性**：
- 基于 QUIC（UDP）：更快的连接建立
- 0-RTT：缓存密钥，快速恢复连接
- 无队头阻塞：流之间相互独立
- 连接迁移：切换网络时保持连接

| `max-age=3600` | 缓存有效时间 |
| `must-revalidate` | 过期后需验证 |

### 1.8 Cookie 与 Session

**参考答案：**

**Cookie**：

```http
Set-Cookie: name=value; expires=Thu, 01 Jan 2025 00:00:00 GMT; path=/; domain=.example.com; secure; HttpOnly; SameSite=Strict
```

| 属性 | 说明 |
| :--- | :--- |
| `name=value` | 键值对 |
| `expires` | 过期时间 |
| `path` | 生效路径 |
| `domain` | 生效域名 |
| `secure` | 仅 HTTPS 传输 |
| `HttpOnly` | 无法通过 JS 访问 |
| `SameSite` | CSRF 防护（Strict/Lax/None） |

**Session**：

```javascript
// 服务端 session 存储
session: {
  'session_id_123': {
    userId: 1,
    username: 'tom',
    loginTime: '2025-01-01'
  }
}

// Session + Cookie 流程
// 1. 登录成功后，服务端创建 session
// 2. 将 session_id 写入 cookie
// 3. 后续请求携带 cookie
// 4. 服务端根据 session_id 查找用户信息
```

**对比**：

| 特性 | Cookie | Session |
| :--- | :--- | :--- |
| 存储位置 | 浏览器 | 服务器 |
| 安全性 | 较低（可加密） | 较高 |
| 容量 | 4KB | 无限制 |
| 性能 | 每次请求携带 | 服务端查询 |

```
│                                                                  │
```

### 2.2 CSRF 攻击与防御

**参考答案：**

**CSRF（Cross-Site Request Forgery）**：跨站请求伪造。

```
┌─────────────────────────────────────────────────────────────────┐
│                      CSRF 攻击原理                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│    用户登录正常网站 A                                            │
│           │                                                     │
│           ▼                                                     │
│    用户被诱导访问恶意网站 B                                       │
│           │                                                     │
│           ▼                                                     │
│    网站 B 自动发送请求到 A                                       │
│    GET: /transfer?to=攻击者&amount=10000                        │
│           │                                                     │
│           ▼                                                     │
│    浏览器携带 A 的 Cookie 发请求                                 │
│           │                                                     │
│           ▼                                                     │
│    服务器认为是用户操作，执行转账                                  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**防御措施**：

| 措施 | 说明 |
| :--- | :--- |
| **SameSite Cookie** | `SameSite=Strict/Lax` |
| **CSRF Token** | 服务端生成随机 token，表单中携带 |
| **Referer 检查** | 检查请求来源 |
| **双重提交 Cookie** | Cookie 中设置 token，表单中也提交 |
| **验证码** | 短信/邮箱验证 |

## 六、HTTP 协议深度解析

### 6.1 HTTP 请求方法详解

**参考答案：**

HTTP 协议定义了多种请求方法，每种方法都有其特定的用途和行为。正确选择 HTTP 方法是设计 RESTful API 的基础。

#### HTTP 方法分类

```
┌─────────────────────────────────────────────────────────────────┐
│                      HTTP 请求方法                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  安全方法（只读操作，不改变服务器状态）                             │
│  ├─ GET    ─── 获取资源                                         │
│  ├─ HEAD   ─── 获取资源的元信息（无响应体）                      │
│  └─ OPTIONS ─── 获取服务器支持的请求方法                         │
│                                                                  │
│  幂等方法（多次执行结果相同）                                      │
│  ├─ PUT    ─── 更新/替换资源                                    │
│  ├─ DELETE ─── 删除资源                                         │
│  └─ DELETE ─── 删除资源（幂等）                                  │
│                                                                  │
│  非幂等方法（每次执行结果可能不同）                                 │
│  ├─ POST   ─── 创建资源                                         │
│  ├─ PATCH  ─── 部分更新资源                                     │
│  └─ CONNECT ── 建立隧道连接                                      │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

#### GET 方法详解

GET 是最常用的 HTTP 方法，用于请求服务器发送指定的资源。

```http
GET /api/users?page=1&size=10&sort=name HTTP/1.1
Host: api.example.com
Accept: application/json
Accept-Language: zh-CN,zh;q=0.9,en;q=0.8
Cache-Control: max-age=3600
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36
```

**GET 请求的特点**：
- 请求参数附加在 URL 后面，使用 `?` 分隔
- 多个参数使用 `&` 连接
- URL 长度受浏览器和服务器限制（通常 2KB 左右）
- 请求可以被缓存
- 请求会保存在浏览器历史记录中
- 请求不应该改变服务器状态（幂等性）

**GET 请求的 URL 编码**：

```javascript
// URL 编码示例
const params = {
  name: '张三',
  city: '北京',
  hobby: '编程 & 游戏'
};

// 手动编码
const url = '/api/search?name=' + encodeURIComponent('张三') +
            '&city=' + encodeURIComponent('北京');

// 使用 URLSearchParams（推荐）
const searchParams = new URLSearchParams(params);
console.log(searchParams.toString());
// 输出: name=%E5%BC%A0%E4%B8%89&city=%E5%8C%97%E4%BA%AC&hobby=%E7%BC%96%E7%A8%8B%20%26%20%E6%B8%B8%E6%88%8F
```

```javascript
// fetch API 发送 GET 请求
async function fetchUsers() {
  try {
    const response = await fetch('/api/users?page=1&size=10', {
      method: 'GET',
      headers: {
        'Accept': 'application/json',
        'Authorization': 'Bearer token123'
      },
      // GET 请求也可以设置缓存策略
      cache: 'no-cache' // no-cache, no-store, reload, force-cache
    });

    if (!response.ok) {
      throw new Error(`HTTP error! status: ${response.status}`);
    }

    const data = await response.json();
    console.log('Users:', data);
  } catch (error) {
    console.error('Fetch error:', error);
  }
}

// 使用 axios 发送 GET 请求
import axios from 'axios';

async function getUser() {
  try {
    const response = await axios.get('/api/user/123', {
      params: {
        include: 'profile,posts'
      },
      headers: {
        'X-Request-ID': 'uuid-123'
      },
      timeout: 5000
    });
    return response.data;
  } catch (error) {
    if (error.response) {
      // 服务器返回错误状态码
      console.error('Server error:', error.response.status);
    } else if (error.request) {
      // 请求发出但没有收到响应
      console.error('Network error:', error.message);
    }
    throw error;
  }
}
```


**POST 请求的 Content-Type**：

```javascript
// 1. application/json（最常用）
const jsonData = {
  title: '文章标题',
  content: '文章内容',
  author: '张三'
};
fetch('/api/posts', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify(jsonData)
});

// 2. application/x-www-form-urlencoded（表单提交）
const formData = new URLSearchParams();
formData.append('username', 'zhangsan');
formData.append('email', 'zhangsan@example.com');

  method: 'POST',
  headers: {
    'Content-Type': 'application/x-www-form-urlencoded'
  },
  body: formData.toString()
});

// 3. multipart/form-data（文件上传）
const formData = new FormData();
formData.append('username', 'zhangsan');
formData.append('avatar', fileInput.files[0]); // 文件对象
formData.append('bio', '这是一个简介');

  method: 'POST',
  body: formData
  // 注意：不需要设置 Content-Type，浏览器会自动设置
});
```

**POST 与 PUT 的区别**：

```javascript
// POST: 创建资源（服务器生成 ID）
// 请求: POST /api/users
// 响应: 201 Created
{
  "id": "12345",
  "username": "zhangsan",
  "createdAt": "2025-01-01T00:00:00Z"
}

// PUT: 更新/替换资源（客户端指定 ID）
// 请求: PUT /api/users/12345
// 响应: 200 OK
{
  "id": "12345",
  "username": "zhangsan",
  "updatedAt": "2025-01-01T00:00:00Z"
}

// PATCH: 部分更新资源
// 请求: PATCH /api/users/12345
// 响应: 200 OK
{
  "id": "12345",
  "email": "newemail@example.com"  // 只更新 email 字段
}
```

#### PUT vs PATCH 对比

```javascript
// PUT: 完整更新（替换整个资源）
// 假设原资源: { id: 1, name: "张三", age: 25, city: "北京" }
fetch('/api/users/1', {
  method: 'PUT',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: '李四', age: 30 })
});
// 结果: { id: 1, name: "李四", age: 30 }
// 注意: city 字段被删除（未提供）

// PATCH: 部分更新（只更新提供的字段）
fetch('/api/users/1', {
  method: 'PATCH',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ age: 30 })
});
// 结果: { id: 1, name: "张三", age: 30, city: "北京" }
// 注意: 其他字段保持不变
```

```javascript
// 删除单个资源
async function deleteUser(userId) {
  const response = await fetch(`/api/users/${userId}`, {

    headers: {
```

#### HEAD 方法详解

HEAD 方法与 GET 类似，但只返回响应头，不返回响应体。

```http
HEAD /api/large-file.zip HTTP/1.1
Host: download.example.com
```

```http
HTTP/1.1 200 OK
Content-Type: application/zip
Content-Length: 104857600  // 100MB
Last-Modified: Wed, 01 Jan 2025 00:00:00 GMT
ETag: "abc123def456"
Accept-Ranges: bytes
```

**HEAD 的应用场景**：

```javascript
// 1. 检查资源是否存在
async function checkResourceExists(url) {
  const response = await fetch(url, { method: 'HEAD' });
  return response.ok;
}

// 2. 获取资源的元信息（不下载大文件）
async function getFileInfo(url) {
  const response = await fetch(url, { method: 'HEAD' });
  return {
    contentType: response.headers.get('Content-Type'),
    contentLength: response.headers.get('Content-Length'),
    lastModified: response.headers.get('Last-Modified'),
    etag: response.headers.get('ETag')
  };
}

// 3. 检查资源是否更新（用于缓存）
async function isResourceUpdated(url, etag) {
  const response = await fetch(url, {
    method: 'HEAD',
    headers: {
      'If-None-Match': etag
    }
  });
  return response.status === 200; // 200 表示已更新，304 表示未更新
}
```

### 6.2 HTTP 状态码深度解析

**参考答案：**

HTTP 状态码是服务器向客户端返回的三位数字响应码，表示请求的处理结果。

#### 1xx 信息响应码

```
┌─────────────────────────────────────────────────────────────────┐
│                      1xx 信息响应码                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  100 Continue（继续）                                            │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  客户端发送了请求头，服务器愿意接受请求body              │  │
│  │  客户端应该继续发送请求体                                  │  │
│  │                                                            │  │
│  │  场景: 大文件上传时的分块传输                              │  │
│  │  客户端 ──▶ POST /upload ──▶ 100 Continue ──▶ 上传数据   │  │
│  └──────────────────────────────────────────────────────────┘  │
```

```javascript
  const limit = 100; // 每分钟 100 次
  const window = 60000; // 1 分钟

  if (!rateLimit.has(ip)) {

    const record = rateLimit.get(ip);

    if (now > record.resetTime) {
```

### 6.3 HTTPS 加密原理深度解析

HTTPS（HTTP Secure）是 HTTP 协议的安全版本，通过 TLS/SSL 加密传输数据，确保通信的机密性、完整性和真实性。

```javascript
// 对称加密算法示例

// AES-GCM (推荐)
const crypto = require('crypto');

function encryptAES(key, plaintext) {
  // 生成 12 字节的初始化向量 (IV)
  const iv = crypto.randomBytes(12);

  // 创建加密器
  const cipher = crypto.createCipheriv('aes-256-gcm', key, iv);

  // 加密
  let encrypted = cipher.update(plaintext, 'utf8', 'hex');
  encrypted += cipher.final('hex');

  // 获取认证标签
  const authTag = cipher.getAuthTag();

  return {
    iv: iv.toString('hex'),
    encryptedData: encrypted,
    authTag: authTag.toString('hex')
  };
}

function decryptAES(key, encryptedData, iv, authTag) {
  const decipher = crypto.createDecipheriv(
    'aes-256-gcm',
    key,
    Buffer.from(iv, 'hex')
  );

  decipher.setAuthTag(Buffer.from(authTag, 'hex'));

  let decrypted = decipher.update(encryptedData, 'hex', 'utf8');
  decrypted += decipher.final('utf8');

  return decrypted;
}

// 使用示例
const key = crypto.randomBytes(32); // 256 位密钥
const plaintext = 'Hello, HTTPS!';

const encrypted = encryptAES(key, plaintext);
console.log('加密:', encrypted);

const decrypted = decryptAES(
  key,
  encrypted.encryptedData,
  encrypted.iv,
  encrypted.authTag
);
console.log('解密:', decrypted);

// ChaCha20-Poly1305 (移动端推荐)
function encryptChaCha20(key, plaintext) {
  const nonce = crypto.randomBytes(12);
  const cipher = crypto.createCipheriv('chacha20-poly1305', key, nonce);

  const encrypted = Buffer.concat([
    cipher.update(plaintext),
    cipher.final()
  ]);

  const authTag = cipher.getAuthTag();

  return { nonce: nonce.toString('hex'), encrypted, authTag };
}

const { generateKeyPairSync, publicEncrypt, privateDecrypt } = require('crypto');

// 生成 RSA 密钥对

  modulusLength: 2048,
  publicKeyEncoding: { type: 'spki', format: 'pem' },
  privateKeyEncoding: { type: 'pkcs8', format: 'pem' }

// 使用公钥加密
function encryptWithRSA(publicKey, data) {
  const encrypted = publicEncrypt(
    {
      key: publicKey,
      padding: crypto.constants.RSA_PKCS1_OAEP_PADDING,
      oaepHash: 'sha256'
    },
    Buffer.from(data)
  );
  return encrypted.toString('base64');
}

// 使用私钥解密
function decryptWithRSA(privateKey, encryptedData) {
  const decrypted = privateDecrypt(
    {
      key: privateKey,
      padding: crypto.constants.RSA_PKCS1_OAEP_PADDING,
      oaepHash: 'sha256'
    },
    Buffer.from(encryptedData, 'base64')
  );
  return decrypted.toString('utf8');
}

// 使用示例
const data = 'Sensitive data';
const encrypted = encryptWithRSA(publicKey, data);
console.log('加密:', encrypted);

const decrypted = decryptWithRSA(privateKey, encrypted);
console.log('解密:', decrypted);

// ECDH 密钥交换示例
function generateECDHSecret(clientPrivateKey, serverPublicKey) {
  const ecdh = crypto.createECDH('secp256k1');
  ecdh.setPrivateKey(clientPrivateKey);

  const sharedSecret = ecdh.computeSecret(serverPublicKey);

  // 派生会话密钥
  return crypto.createHash('sha256').update(sharedSecret).digest();
}
```

### 7.1 TCP 三次握手详解

**参考答案：**

TCP（Transmission Control Protocol）是一种面向连接的、可靠的传输层协议。三次握手是 TCP 建立连接的过程。

#### 三次握手原理

```
┌─────────────────────────────────────────────────────────────────┐
│                      TCP 三次握手详解                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  第一次握手：SYN (Synchronize)                                  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                                                          │  │
│  │    客户端                              服务端             │  │
│  │      │                                  │                 │  │
│  │      │  ─── SYN=1, seq=x ──────────▶  │                 │  │
│  │      │                                  │                 │  │
│  │      │   [客户端发送 SYN 包            │                 │  │
│  │      │    同步序列号]                   │                 │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  第二次握手：SYN-ACK                                            │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                                                          │  │
│  │    客户端                              服务端             │  │
│  │      │                                  │                 │  │
│  │      │  ◀── SYN=1, ACK=1, seq=y ─────  │                 │  │
│  │      │        ack=x+1                  │                 │  │
│  │      │                                  │                 │  │
│  │      │   [服务端收到 SYN              │                 │  │
│  │      │    发送 SYN + ACK]             │                 │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  第三次握手：ACK                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                                                          │  │
│  │    客户端                              服务端             │  │
│  │      │                                  │                 │  │
│  │      │  ─── ACK=1, seq=x+1 ─────────▶ │                 │  │
│  │      │        ack=y+1                  │                 │  │
│  │      │                                  │                 │
│  │      │   [客户端确认                    │                 │  │
│  │      │    连接建立]                     │                 │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  为什么需要三次握手？                                     │  │
│  │                                                          │  │
│  │  1. 同步双方序列号                                        │  │
│  │     - 客户端发送 seq=x                                    │  │
│  │     - 服务端确认 ack=x+1                                  │  │
│  │     - 服务端发送 seq=y                                    │  │
│  │     - 客户端确认 ack=y+1                                  │  │
│  │                                                          │  │
│  │  2. 防止已失效的连接请求                                   │  │

│  │     - 三次握手确保双方都 ready                            │  │
│  │                                                          │  │
│  │  3. 验证双方通信能力                                      │  │
│  │     - 客户端能发能收                                      │  │
│  │     - 服务端能收能发                                      │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 TCP 四次挥手详解

**参考答案：**

四次挥手是 TCP 关闭连接的过程，由于 TCP 是全双工通信，需要双方分别发送 FIN 信号。

#### 四次挥手原理

```
┌─────────────────────────────────────────────────────────────────┐
│                      TCP 四次挥手详解                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  第一次挥手：FIN (客户端 -> 服务端)                              │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │    客户端                              服务端             │  │
│  │      │                                  │                 │  │
│  │      │  ─── FIN=1, seq=u ───────────▶  │                 │  │
│  │      │                                  │                 │  │
│  │      │   [客户端发送 FIN                │                 │  │
│  │      │    关闭写端]                      │                 │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  第二次挥手：ACK (服务端 -> 客户端)                             │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │    客户端                              服务端             │  │
│  │      │                                  │                 │  │
│  │      │  ◀── ACK=1, ack=u+1 ──────────  │                 │  │
│  │      │                                  │                 │  │
│  │      │   [服务端确认                    │                 │  │
│  │      │    但可能还有数据要发送]         │                 │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  第三次挥手：FIN (服务端 -> 客户端)                             │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │    客户端                              服务端             │  │
│  │      │                                  │                 │  │
│  │      │  ◀── FIN=1, seq=v ────────────  │                 │  │
│  │      │                                  │                 │  │
│  │      │   [服务端也关闭                  │                 │  │
│  │      │    写端]                         │                 │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  第四次挥手：ACK (客户端 -> 服务端)                             │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │    客户端                              服务端             │  │
│  │      │                                  │                 │  │
│  │      │  ─── ACK=1, ack=v+1 ─────────▶ │                 │  │
│  │      │                                  │                 │  │
│  │      │   [客户端确认                    │                 │  │
│  │      │    等待 2MSL 后关闭]             │                 │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  为什么需要四次挥手？                                     │  │
│  │                                                          │  │
│  │  1. TCP 全双工通信                                        │  │
│  │     - 客户端发送 FIN 表示不再发送数据                     │  │
│  │     - 但仍可以接收数据                                    │  │
│  │     - 服务端也需要单独关闭                                │  │
│  │                                                          │  │
│  │  2. 确保数据完整传输                                      │  │
│  │     - 服务端可能还有数据要发送                           │  │
│  │     - 需要等待数据传输完成                                │  │
│  │                                                          │  │
│  │  3. 2MSL 的作用                                           │  │
│  │     - Maximum Segment Lifetime                           │  │
│  │     - 等待服务端收到最后的 ACK                           │  │
│  │     - 防止延迟的 FIN 被新连接接收                        │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

#### TIME_WAIT 状态详解

```
┌─────────────────────────────────────────────────────────────────┐
│                      TIME_WAIT 状态详解                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  为什么需要 2MSL？                                               │
│                                                                  │
│  1. 确保最后的 ACK 到达                                          │
│     ┌─────────────────────────────────────────────────────┐    │
│  │  Client ── FIN ──▶ Server                            │    │
│  │  Client ◀── ACK ─── Server                           │    │
│  │  Client ── FIN ──▶ Server (可能丢失)                 │    │
│  │  Client ◀── ACK ─── Server (重传)                    │    │
│  │  Client 等待 2MSL 确认 Server 收到                   │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                                  │
│  2. 防止旧连接的数据被新连接接收                                  │
│     ┌─────────────────────────────────────────────────────┐    │
│  │  旧连接: A ── seq=1000, len=500 ──▶ B                │    │
│  │  新连接: A ── seq=1000 ──▶ B (可能被误认为重传)      │    │
│  │  等待 2MSL 确保旧数据在网络中消失                    │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                                  │
│  MSL (Maximum Segment Lifetime)                                  │
│  • Linux: 60秒                                                  │
│  • Windows: 120秒                                               │
│  • 可通过 tcp_time_wait_interval 调整                            │
│                                                                  │
│  TIME_WAIT 过多的问题:                                           │
│  • 原因: 高并发短连接                                           │
│  • 解决: 调整内核参数、使用连接池、HTTP Keep-Alive              │
│                                                                  │
│  /etc/sysctl.conf 配置:                                         │
│  net.ipv4.tcp_tw_reuse = 1      # 允许重用 TIME_WAIT 套接字   │
│  net.ipv4.tcp_tw_recycle = 1    # 快速回收 TIME_WAIT         │
│  net.ipv4.tcp_fin_timeout = 30  # FIN 超时时间                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**代码示例**：

```javascript

  console.log('Client connected:', socket.remoteAddress);

  // 设置 keep-alive 检测断开的客户端

    socket.destroy();
  });

  // 主动关闭连接
  // 方法1: end() 发送 FIN
  // socket.end('Goodbye');

  // 方法2: 立即销毁
  // socket.destroy();
});

  console.log('Server closed');
});
```

### 7.4 TCP 拥塞控制详解

**参考答案：**

TCP 拥塞控制是保证网络稳定性的重要机制，通过控制发送方的发送速率来避免网络过载。

#### 拥塞控制算法

```
┌─────────────────────────────────────────────────────────────────┐
│                      TCP 拥塞控制算法                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  拥塞窗口 (cwnd)                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  发送方维护的窗口大小                                     │  │
│  │  表示在未收到确认的情况下可以发送的数据量                  │  │
│  │  发送窗口 = min(cwnd, rwnd)                              │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  1. 慢启动 (Slow Start)                                   │  │
│  │                                                            │  │
│  │  初始 cwnd = 1 MSS (最大报文段)                           │  │
│  │  每个 RTT 翻倍增长                                        │  │
│  │                                                            │  │
│  │  cwnd: 1 → 2 → 4 → 8 → 16 → 32 → 64 ...                 │  │
│  │            ↑                     ↑                        │  │
│  │            1 MSS                 1 RTT                    │  │
│  │                                                            │  │
│  │  到达 ssthresh (慢启动门限) 后进入拥塞避免               │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  2. 拥塞避免 (Congestion Avoidance)                       │  │
│  │                                                            │  │
│  │  每个 RTT 增加 1 个 MSS                                   │  │
│  │                                                            │  │
│  │  cwnd: 64 → 65 → 66 → 67 ... (线性增长)                  │  │
│  │                                                            │  │
│  │  目的: 找到网络容量平衡点                                 │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  3. 快重传 (Fast Retransmit)                             │  │
│  │                                                            │  │
│  │  收到 3 个重复 ACK 时立即重传丢失的包                      │  │
│  │                                                            │  │
│  │  示例:                                                    │  │
│  │  发送: 1, 2, 3, 4, 5                                      │  │
│  │  收到: 1, 2, 4, 5, 3 (3 丢失)                             │  │
│  │  ACK: 2, 2, 2 (3 个重复 ACK)                             │  │
│  │  重传: 3 (快重传，不等待超时)                             │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  4. 快恢复 (Fast Recovery)                                │  │
│  │                                                            │  │
│  │  快重传后执行快恢复                                       │  │
│  │                                                            │  │
│  │  cwnd = cwnd / 2 + 3 (乘法减小)                          │  │
│  │  ssthresh = cwnd                                          │  │
│  │  然后进入拥塞避免                                         │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  5. 拥塞发生 (Loss Detection)                             │  │
│  │                                                            │  │
│  │  超时:                                                    │  │
│  │  • ssthresh = cwnd / 2                                    │  │
│  │  • cwnd = 1 MSS                                           │  │
│  │  • 重新慢启动                                            │  │
│  │                                                            │  │
│  │  重复 ACK:                                               │  │
│  │  • 执行快重传 + 快恢复                                   │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

#### BBR 拥塞控制算法

```
┌─────────────────────────────────────────────────────────────────┐
│                      BBR 拥塞控制算法                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  BBR (Bottleneck Bandwidth and RTT)                             │
│  • 区别于传统的丢包检测                                          │
│  • 基于带宽和 RTT 的实时测量                                     │
│                                                                  │
│  传统算法 vs BBR:                                                │
│  ┌──────────────────┬─────────────────┬─────────────────────┐  │
│  │      指标        │  传统 (Reno)    │       BBR           │  │
│  ├──────────────────┼─────────────────┼─────────────────────┤  │
│  │ 检测方式         │ 丢包             │ 带宽/RTT            │  │
│  │ 公平性           │ 较好             │ 好                  │  │
│  │ 延迟             │ 较高             │ 低                  │  │
│  │ 丢包率           │ 较高             │ 低                  │  │
│  │ 适用场景         │ 普通网络         │ 高带宽高延迟        │  │
│  └──────────────────┴─────────────────┴─────────────────────┘  │
│                                                                  │
│  BBR 工作原理:                                                   │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  1. 测量最大带宽 (BtlBw)                                 │  │
│  │     • 持续发送数据，观察吞吐量                          │  │
│  │                                                            │  │
│  │  2. 测量最小 RTT (RTtprop)                              │  │
│  │     • 记录所有 RTT 中的最小值                           │  │
│  │                                                            │  │
│  │  3. 计算发送速率                                         │  │
│  │     • Rate = BtlBw / RTtmin                            │  │
│  │                                                            │  │
│  │  4. 调整窗口                                             │  │
│  │     • pacing_rate = Rate * 1.25                         │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
│  Linux 启用 BBR:                                                 │
│  # modprobe tcp_bbr                                             │
│  # sysctl -w net.ipv4.tcp_congestion_control=bbr                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**代码示例**：

```javascript
// Node.js DNS 查询
const dns = require('dns');
const dnsPromises = dns.promises;

// 查询 A 记录
async function queryA(domain) {
  try {
    const addresses = await dnsPromises.resolve4(domain);
    console.log('A records:', addresses);
    return addresses;
  } catch (err) {
    console.error('DNS error:', err.message);
    throw err;
  }
}

// 查询 AAAA 记录 (IPv6)
async function queryAAAA(domain) {
  try {
    const addresses = await dnsPromises.resolve6(domain);
    console.log('AAAA records:', addresses);
    return addresses;
  } catch (err) {
    console.error('DNS error:', err.message);
    throw err;
  }
}

// 查询 MX 记录
async function queryMX(domain) {
  try {
    const records = await dnsPromises.resolveMx(domain);
    console.log('MX records:', records);
    // [{ priority: 10, exchange: 'mail.example.com' }, ...]
    return records;
  } catch (err) {
    console.error('DNS error:', err.message);
    throw err;
  }
}

// 查询 CNAME 记录
async function queryCNAME(domain) {
  try {
    const cnames = await dnsPromises.resolveCname(domain);
    console.log('CNAME records:', cnames);
    return cnames;
  } catch (err) {
    console.error('DNS error:', err.message);
    throw err;
  }
}

// 查询 TXT 记录
async function queryTXT(domain) {
  try {
    const records = await dnsPromises.resolveTxt(domain);
    console.log('TXT records:', records);
    return records;
  } catch (err) {
    console.error('DNS error:', err.message);
    throw err;
  }
}

// 查询 NS 记录
async function queryNS(domain) {
  try {
    const nameservers = await dnsPromises.resolveNs(domain);
    console.log('NS records:', nameservers);
    return nameservers;
  } catch (err) {
    console.error('DNS error:', err.message);
    throw err;
  }
}

// 查询所有记录
async function queryAll(domain) {
  try {
    const [a, aaaa, cname, mx, txt, ns] = await Promise.all([
      dnsPromises.resolve4(domain).catch(() => []),
      dnsPromises.resolve6(domain).catch(() => []),
      dnsPromises.resolveCname(domain).catch(() => []),
      dnsPromises.resolveMx(domain).catch(() => []),
      dnsPromises.resolveTxt(domain).catch(() => []),
      dnsPromises.resolveNs(domain).catch(() => [])
    ]);

    return { a, aaaa, cname, mx, txt, ns };
  } catch (err) {
    console.error('DNS error:', err.message);
    throw err;
  }
}

// 反向 DNS 查询
async function reverseDns(ip) {
  try {
    const hostnames = await dnsPromises.reverse(ip);
    console.log('Reverse DNS:', hostnames);
    return hostnames;
  } catch (err) {
    console.error('Reverse DNS error:', err.message);
    throw err;
  }
}

// 使用示例
(async () => {
  await queryA('example.com');
  await queryMX('example.com');
  await queryTXT('example.com');

  const allRecords = await queryAll('example.com');
  console.log('All records:', allRecords);
})();

// 底层 DNS 查询（可自定义 DNS 服务器）
dns.resolve4('example.com', { dns: '8.8.8.8' }, (err, addresses) => {
  if (err) {
    console.error('Error:', err);
    return;
  }
  console.log('Addresses:', addresses);
});

// DNS 缓存示例（简单实现）
const dnsCache = new Map();

async function queryWithCache(domain, ttl = 300000) { // 默认 5 分钟
  const cached = dnsCache.get(domain);

  if (cached && Date.now() < cached.expire) {
    console.log('Cache hit:', domain);
    return cached.addresses;
  }

  console.log('Cache miss:', domain);
  const addresses = await dnsPromises.resolve4(domain);

  dnsCache.set(domain, {
    addresses,
    expire: Date.now() + ttl
  });

  return addresses;
}

// 清理过期缓存
setInterval(() => {
  const now = Date.now();
  for (const [domain, cache] of dnsCache.entries()) {
    if (now >= cache.expire) {
      dnsCache.delete(domain);
    }
  }
}, 60000);
```

### 8.2 CDN 原理详解

**参考答案：**

CDN（Content Delivery Network，内容分发网络）通过将内容缓存到离用户最近的节点来加速访问。

#### CDN 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                      CDN 工作原理                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  无 CDN:                                                        │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                                                          │  │
│  │    用户 ──▶ ──▶ ──▶ 源服务器 (北京)                    │  │
│  │        (深圳)   跨省传输   延迟高                        │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                  │
```

```javascript
// 生成自签名证书（开发环境）
function generateSelfSignedCert() {
  const { publicKey, privateKey } = crypto.generateKeyPairSync('rsa', {
    modulusLength: 2048,
    publicKeyEncoding: { type: 'spki', format: 'pem' },
    privateKeyEncoding: { type: 'pkcs8', format: 'pem' }
  });

  const cert = crypto.createCertificate();
  cert.setSubject([
    { shortName: 'CN', value: 'localhost' },
    { shortName: 'O', value: 'Development' }
  ]);
  cert.setIssuer([
    { shortName: 'CN', value: 'localhost' },
    { shortName: 'O', value: 'Development' }
  ]);
  cert.setExtensions([
    { name: 'basicConstraints', cA: false },
    { name: 'keyUsage', digitalSignature: true, keyEncipherment: true },
    {
      name: 'subjectAltName',
      altNames: [
        { type: 2, value: 'localhost' },
        { type: 7, ip: '127.0.0.1' }
      ]
    }
  ]);
  cert.sign(privateKey, crypto.createHash('sha256'));

  return { key: privateKey, cert: cert.toString() };
}
```

```javascript
  async request(url, options = {}) {
    return new Promise((resolve, reject) => {
      const parsedUrl = new URL(url);
      const protocol = parsedUrl.protocol === 'https:' ? https : http;

      const requestOptions = {
        hostname: parsedUrl.hostname,
        port: parsedUrl.port || (parsedUrl.protocol === 'https:' ? 443 : 80),
        path: parsedUrl.pathname + parsedUrl.search,
        method: options.method || 'GET',
        headers: options.headers || {},
        timeout: options.timeout || 30000,
        // HTTPS 选项
        ...(parsedUrl.protocol === 'https:' && {
          rejectUnauthorized: options.verifySSL !== false
        })
      };

      const req = protocol.request(requestOptions, (res) => {
        let data = '';

        res.on('data', chunk => {
          data += chunk;
        });

        res.on('end', () => {
          try {
            const result = {
              status: res.statusCode,
              headers: res.headers,
              data: JSON.parse(data)
            };
            resolve(result);
          } catch (err) {
            resolve({
              status: res.statusCode,
              headers: res.headers,
              data: data
            });
          }
        });
      });

      req.on('error', reject);
      req.on('timeout', () => {
        req.destroy();
        reject(new Error('Request timeout'));
      });

      if (options.body) {
        req.write(options.body);
      }

      req.end();
    });
  }
```

```bash
# 持续 ping (Windows)
ping -t example.com

# 指定数据包大小
ping -s 1000 example.com

# -------------------- Traceroute --------------------
# 路由追踪 (Linux/Mac)
traceroute example.com

# 路由追踪 (Windows)
tracert example.com

# 使用 UDP 追踪
traceroute -U example.com

# -------------------- DNS 查询 --------------------
# A 记录查询
nslookup example.com

# 指定 DNS 服务器查询
nslookup example.com 8.8.8.8

# MX 记录查询
nslookup -type=mx example.com

# TXT 记录查询
nslookup -type=txt example.com

# 交互式查询
dig example.com

# 简洁输出
dig +short example.com

# 指定 DNS 服务器
dig @8.8.8.8 example.com

# 查询 CNAME
dig example.com CNAME

# -------------------- 网络状态 --------------------
# 查看端口监听
netstat -tulpn

# 查看所有连接
netstat -an

# 查看特定端口
netstat -an | grep 8080

# Windows 查看 TCP 连接
netstat -an | findstr ESTABLISHED

# 查看路由表
route -n (Linux)
netstat -r (Windows)

# -------------------- curl 命令 --------------------
# GET 请求
curl https://example.com

# 显示响应头
curl -I https://example.com

# 显示详细过程
curl -v https://example.com

# 下载并保存
curl -O https://example.com/file.zip

# POST 请求
curl -X POST -d "name=test" https://example.com/api

# JSON POST
curl -X POST -H "Content-Type: application/json" \
  -d '{"name":"test"}' https://example.com/api

# 带 Cookie
curl -c cookies.txt https://example.com
curl -b cookies.txt https://example.com

# 跟随重定向
curl -L https://example.com

# 限速
curl --limit-rate 100K https://example.com/file.zip

# 跳过 SSL 验证（测试用）
curl -k https://example.com

# -------------------- wget 命令 --------------------
# 下载文件
wget https://example.com/file.zip

# 后台下载
wget -b https://example.com/file.zip

# 断点续传
wget -c https://example.com/file.zip

# 递归下载整个网站
wget -r -np https://example.com

# -------------------- SSH 命令 --------------------
# 连接远程服务器
ssh user@example.com

# 指定端口
ssh -p 2222 user@example.com

# 密钥登录
ssh -i ~/.ssh/id_rsa user@example.com

# 端口转发
ssh -L 8080:localhost:80 user@example.com

# -------------------- 性能测试 --------------------
# 带宽测试
curl -o /dev/null -s -w "Time: %{time_total}s\nSize: %{size_download}bytes\nSpeed: %{speed_download}bytes/s\n" https://example.com

# 查看 HTTP2 支持
curl --http2 -I https://example.com

# 查看 TLS 版本
curl -vI https://example.com 2>&1 | grep "SSL connection"

# -------------------- 网络工具 --------------------
# IP 信息查询
curl ipinfo.io
curl ifconfig.me

# WHOIS 查询
whois example.com

# HTTP 状态码查询
curl -s -o /dev/null -w "%{http_code}" https://example.com

# SSL 证书信息
curl -vI https://example.com 2>&1 | grep -i "certificate"
```
