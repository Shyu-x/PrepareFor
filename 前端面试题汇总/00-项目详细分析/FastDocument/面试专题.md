## 1. 富文本编辑器面试问题

### 问题 1：TipTap/ProseMirror 的架构是怎么样的？为什么选择 TipTap？

**考察点：** 编辑器架构、ProseMirror、选型理由

**参考答案：**

#### ProseMirror 核心架构

ProseMirror 是 TipTap 的底层核心，它采用了一种独特的架构设计，分为四个核心概念：

```
┌─────────────────────────────────────────────────────────┐
│                    ProseMirror 架构                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─────────────┐    ┌─────────────┐    ┌────────────┐ │
│  │   Schema    │    │   State     │    │    View    │ │
│  │  (数据模型)  │───▶│  (编辑器状态) │───▶│  (渲染视图) │ │
│  └─────────────┘    └──────┬──────┘    └────────────┘ │
│                            │                           │
│                   ┌────────▼────────┐                  │
│                   │   Transaction  │                  │
│                   │     (事务)       │                  │
│                   └────────┬────────┘                  │
│                            │                           │
│                   ┌────────▼────────┐                  │
│                   │   Plugins       │                  │
│                   │   (插件系统)     │                  │
│                   └─────────────────┘                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

1. **Schema（模式）**：定义文档的数据结构
   - 定义允许的节点类型（node types）
   - 定义节点之间的关系
   - 定义属性和限制

2. **State（状态）**：文档的当前状态
   - `doc`：文档树结构
   - `selection`：光标/选区位置
   - `storedMarks`：当前样式

3. **View（视图）**：编辑器渲染和交互层
   - 将状态渲染为 DOM
   - 处理用户输入事件
   - 管理输入焦点

4. **Transaction（事务）**：状态变更的载体
   - 所有修改都通过事务进行
   - 事务可以包含多个步骤
   - 支持撤销/重做

#### 为什么选择 TipTap？

**TipTap 是基于 ProseMirror 的 React 封装**，它保留了 ProseMirror 的强大功能，同时提供了更好的开发体验：

```typescript
// TipTap 的优势
// 1. 基于 ProseMirror（生产级验证）
// ProseMirror 被 Notion、Atlassian、New York Times 等大厂使用

// 2. 完善的 React 集成
import { Editor, EditorContent } from '@tiptap/react';

function MyEditor() {
  const [editor, setEditor] = useState(null);

  return (
    <EditorContent editor={editor} />
  );
}

// 3. 强大的扩展系统
const extensions = [
  StarterKit,           // 基础功能
  Collaboration,        // 协作编辑
  CollaborationCursor,  // 协作光标
  Highlight,            // 高亮
  Typography,           // 排版优化
  // ... 更多扩展
];

// 4. 原生支持 Yjs 协作
import Collaboration from '@tiptap/extension-collaboration';

const editor = new Editor({
  extensions: [
    Collaboration.configure({
      document: ydoc,
    }),
  ],
});
```

**与其他编辑器的对比：**

| 特性 | TipTap | Slate | Quill |
|------|--------|-------|-------|
| 底层 | ProseMirror | 自研 | 自研 |
| 协作 | Yjs 原生支持 | 需自研 | 有限 |
| 定制性 | 高 | 非常高 | 中 |
| 文档质量 | 优秀 | 中 | 优秀 |
| 社区活跃度 | 高 | 高 | 中 |

**FastDocument 选型理由：**
- 项目需要多人实时协作，Yjs + TipTap 是业界最佳方案
- 团队对 ProseMirror 架构有一定了解
- 需要支持块级编辑（Block-based editing），TipTap 原生支持
- 需要支持 12+ 种块类型（text, h1-h3, todo, callout, divider, code, image, table, mindmap, flowchart, math）

**可能的追问：**

1. **Q: ProseMirror 和 Slate 的区别是什么？**
   - ProseMirror 是生产级框架，有完整的架构设计；Slate 更灵活但需要更多自定义代码
   - ProseMirror 的事务系统更强大，协作支持更好

2. **Q: TipTap 有什么局限性？**
   - 学习曲线较陡
   - 包体积较大（约 500KB+）
   - SSR 需要特殊处理

### 问题 2：编辑器的内容怎么存储？JSON 还是 HTML？各有什么优缺点？

**考察点：** 数据结构设计、数据格式选型

**参考答案：**

#### 两种存储方案对比

FastDocument 项目采用了**块级 JSON 结构**存储，这是经过技术调研后的选择。

#### 方案一：HTML 格式存储

```html
<!-- HTML 格式示例 -->
<div class="editor">
  <h1>标题</h1>
  <p>这是<strong>加粗</strong>文本</p>
  <ul>
    <li>列表项 1</li>
    <li>列表项 2</li>
  </ul>
</div>
```

**优点：**
- 前后端兼容性好，所有语言都能解析
- 展示简单，直接渲染即可
- 与富文本编辑习惯一致

**缺点：**
- 语义化差，解析成本高
- 难以进行局部更新（修改一个字需要重渲染整个段落）
- 协作冲突粒度粗（以整个节点为单位）
- 难以提取结构化数据

#### 方案二：JSON 格式存储（FastDocument 采用）

```json
// JSON 格式示例（块级结构）
{
  "id": "doc-123",
  "title": "我的文档",
  "blocks": [
    {
      "id": "block-1",
      "type": "h1",
      "content": "标题"
    },
    {
      "id": "block-2",
      "type": "text",
      "content": "这是普通文本",
      "properties": {
        "bold": true
      }
    },
    {
      "id": "block-3",
      "type": "todo",
      "content": "待办事项",
      "properties": {
        "checked": false
      }
    },
    {
      "id": "block-4",
      "type": "code",
      "content": "console.log('hello')",
      "properties": {
        "language": "javascript"
      }
    }
  ]
}
```

**优点：**
- 语义化清晰，结构明确
- 局部更新高效（修改单个块不影响其他块）
- 协作冲突粒度细（以块为单位）
- 易于进行数据分析、搜索、导出
- 类型安全，便于校验

**缺点：**
- 渲染需要额外处理
- 某些场景需要转换为 HTML
- 存储空间可能更大

#### FastDocument 的块级设计

```typescript
// 块接口定义
interface Block {
  id: string;           // 唯一标识
  type: BlockType;      // 块类型
  content: string;      // 文本内容
  properties?: {
    checked?: boolean;       // todo 项
    style?: string;           // 样式
    language?: string;        // 代码语言
    url?: string;            // 图片/链接 URL
    [key: string]: any;      // 扩展属性
  };
  order?: number;       // 排序
}

// 支持的块类型
type BlockType =
  | 'text'      // 普通文本
  | 'h1'        // 一级标题
  | 'h2'        // 二级标题
  | 'h3'        // 三级标题
  | 'todo'      // 待办事项
  | 'callout'   // 提示框
  | 'divider'   // 分隔线
  | 'code'      // 代码块
  | 'image'     // 图片
  | 'table'     // 表格
  | 'mindmap'   // 思维导图
  | 'flowchart' // 流程图
  | 'math';     // 数学公式
```

**存储格式选择建议：**

| 场景 | 推荐格式 | 理由 |
|------|----------|------|
| 协作编辑器 | JSON | 冲突粒度细，易于局部更新 |
| 简单富文本 | HTML | 兼容性好的 |
| 结构化文档 | JSON | 便于提取和处理数据 |
| 博客/CMS | HTML | SEO 友好 |

**可能的追问：**

1. **Q: JSON 怎么转成 HTML 渲染？**
   - 遍历 JSON 块数组
   - 根据 type 渲染对应的 React 组件
   - 使用 tippy.js 或自定义渲染器

2. **Q: 块级结构如何保证顺序？**
   - 使用 `order` 字段维护顺序
   - 支持拖拽排序
   - Yjs 的 Y.Array 可以自动处理顺序

### 问题 3：怎么做编辑器的撤销/重做？跟普通的 undo/redo 有什么区别？

**考察点：** 历史管理、编辑器原理

**参考答案：**

#### 普通 Undo/Redo 的问题

普通的 `window.history` 或简单的状态栈存在以下问题：

```typescript
// 普通方案的问题
const history = [];
let historyIndex = -1;

// 问题 1：内存占用大（存储完整状态副本）
function pushState(newState) {
  // 每次保存完整的文档副本
  history.push(JSON.parse(JSON.stringify(newState)));
  historyIndex++;
}

// 问题 2：无法区分协作操作
// 他人修改也会被算作一个历史记录点

// 问题 3：与协作冲突
// 撤销可能会撤销他人的合法操作
```

#### TipTap/Yjs 的撤销/重做方案

FastDocument 使用 **Yjs 的 UndoManager** 实现撤销/重做，这是专为协作场景设计的：

```typescript
import * as Y from 'yjs';
import { YUndoPlugin, YRedoPlugin } from 'y-prosemirror';

// 1. 创建 UndoManager
const undoManager = new YUndoManager(ytext, {
  trackedOrigins: new Set([null]),  // 跟踪所有变化
  captureTimeout: 5000,              // 5秒内的操作合并为一个撤销单元
});

// 2. 撤销/重做
undoManager.undo();  // 撤销
undoManager.redo();  // 重做

// 3. 监听状态
undoManager.on('stack-item-added', (event) => {
  console.log('可撤销:', event.stackItem.meta.get('undo'));
  console.log('可重做:', event.stackItem.meta.get('redo'));
});
```

#### Yjs UndoManager 的核心原理

```
┌─────────────────────────────────────────────────────────────┐
│              Yjs UndoManager 工作原理                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户输入 "Hello"  ──▶  Yjs 记录操作 (insert)              │
│                           │                                 │
│                           ▼                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              Operation Log (操作日志)                 │   │
│  │  [0] insert "H"  (clientID: 1, clock: 0)           │   │
│  │  [1] insert "e"  (clientID: 1, clock: 1)           │   │
│  │  [2] insert "l"  (clientID: 1, clock: 2)           │   │
│  │  [3] insert "l"  (clientID: 1, clock: 3)           │   │
│  │  [4] insert "o"  (clientID: 1, clock: 4)           │   │
│  └─────────────────────────────────────────────────────┘   │
│                           │                                 │
│                           ▼                                 │
│  5秒内操作合并为一个撤销单元 ──▶ 生成反向操作 (delete)        │
│                                                             │
│  撤销单元: delete "Hello"  (5个字符)                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 与普通 undo/redo 的区别

| 特性 | 普通方案 | Yjs UndoManager |
|------|----------|------------------|
| 存储内容 | 完整状态快照 | 操作日志 + 反向操作 |
| 内存占用 | O(n) 状态大小 | O(1) 固定大小 |
| 协作感知 | 无 | 可以排除他人操作 |
| 撤销粒度 | 固定 | 可配置（时间窗口） |
| 远程同步 | 困难 | 天然支持 |

#### 协作场景下的特殊处理

```typescript
// 排除他人操作的撤销/重做
const undoManager = new YUndoManager(ytext, {
  // 只跟踪本地客户端的操作
  trackedOrigins: new Set([clientID]),
});

// 或者跟踪特定用户
const undoManager = new YUndoManager(ytext, {
  trackedOrigins: new Set(['user-1', 'user-2']),
});

// 监听并过滤
ytext.observe((event) => {
  event.transaction.local = true;  // 标记为本地操作
  event.transaction.origin = clientID;  // 记录来源
});
```

#### FastDocument 的实现

```typescript
// documentStore.ts 中的历史管理
interface DocumentState {
  history: { blocks: Block[]; title: string }[];
  historyIndex: number;
  canUndo: boolean;
  canRedo: boolean;
}

// 撤销操作
function undo() {
  if (historyIndex > 0) {
    historyIndex--;
    const prevState = history[historyIndex];
    applyState(prevState);
  }
}

// 重做操作
function redo() {
  if (historyIndex < history.length - 1) {
    historyIndex++;
    const nextState = history[historyIndex];
    applyState(nextState);
  }
}
```

**可能的追问：**

1. **Q: 协作编辑时，撤销如何处理冲突？**
   - UndoManager 会记录操作的用户 ID
   - 可以选择只撤销自己的操作
   - 避免撤销他人正在进行的操作

2. **Q: 如何实现"撤销我的所有操作"功能？**
   - 遍历 UndoManager 的栈
   - 过滤出特定用户的操作
   - 批量执行反向操作

### 问题 4：多人同时编辑一个文档，怎么解决冲突？Yjs 的原理是什么？

**考察点：** CRDT、Yjs、冲突解决

**参考答案：**

#### 传统的冲突解决方式：OT (Operational Transformation)

```
┌─────────────────────────────────────────────────────────────┐
│                    OT 工作原理                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户 A: "Hello"  ──▶ 删除 [0,3] ──▶ "o"                   │
│                     │                                        │
│                     ▼                                        │
│              ┌──────────────┐                               │
│              │   服务端      │                               │
│              │  转换操作     │                               │
│              │ A: del[0,3]  │                               │
│              │ B: ins[5,"X"]│                               │
│              │ ─────────────│                               │
│              │ 转换后:      │                               │
│              │ A: del[0,3]  │                               │
│              │ B: ins[2,"X"]│  (位置偏移)                   │
│              └──────────────┘                               │
│                     │                                        │
│                     ▼                                        │
│  用户 B: "World" ──▶ 插入 [5,"X"] ──▶ "WoXrld"             │
│                                                             │
│  问题: 需要中心服务器，复杂度高                              │
└─────────────────────────────────────────────────────────────┘
```

#### Yjs 的解决方案：CRDT (无冲突复制数据类型)

**CRDT 的核心思想：不需要中心服务器转换，每个客户端独立操作，最终自动合并。**

```
┌─────────────────────────────────────────────────────────────┐
│                    CRDT 工作原理                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户 A                              用户 B                  │
│  "Hello"                            "World"                 │
│     │                                  │                    │
│     ▼                                  ▼                    │
│  删除 [0,3]                         插入 [5,"X"]            │
│     │                                  │                    │
│     │       网络同步 (无中心服务器)       │                    │
│     │◀─────────────────────────────────▶│                    │
│     │                                  │                    │
│     ▼                                  ▼                    │
│  各自独立应用操作                    各自独立应用操作         │
│                                     │                       │
│     ┌────────────────┐              │                       │
│     │ CRDT 保证:     │              │                       │
│     │ 1. 每个操作    │              │                       │
│     │    有唯一ID    │              │                       │
│     │ 2. 并发操作    │              │                       │
│     │    可任意顺序  │              │                       │
│     │ 3. 结果一致    │              │                       │
│     └────────────────┘              │                       │
│            │                        │                       │
│            ▼                        ▼                       │
│  结果: "oXrld"              结果: "oXrld"                  │
│  (一致!)                    (一致!)                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### Yjs 核心技术

```typescript
// 1. 每个操作都有唯一标识 (ClientID + Clock)
import * as Y from 'yjs';

const ydoc = new Y.Doc();
// ClientID: 客户端唯一标识（通常是随机生成的）
// Clock: 逻辑时钟，每个操作递增

// 2. 共享数据类型
const ytext = ydoc.getText('content');  // 文本
const yarray = ydoc.getArray('items');   // 数组
const ymap = ydoc.getMap('config');      // Map

// 3. 操作示例
ytext.insert(0, 'Hello');  // 插入操作
ytext.delete(0, 5);        // 删除操作

// 4. 编码为二进制
const update = Y.encodeStateAsUpdate(ydoc);
// update 是一个 Uint8Array，可以在网络传输
```

#### Yjs 的冲突解决策略

```typescript
// 1. 文本类型：RGA (Replicated Growable Array)
//    保留所有并发插入，按 ID 排序，结果一致

// 示例：两个用户同时在不同位置插入
// 用户 A: 在位置 0 插入 "X"
// 用户 B: 在位置 0 插入 "Y"
// 结果：都变成 "XY" 或 "YX"，取决于 ID 比较

// 2. Map 类型：LWW (Last-Writer-Wins)
//    最后的写入胜出

const ymap = ydoc.getMap('config');
ymap.set('theme', 'dark');   // 用户 A
ymap.set('theme', 'light');  // 用户 B（后到）

// 结果：theme = 'light'

// 3. Array 类型：需要小心使用
//    建议用 ID 替代索引
const yarray = ydoc.getArray('items');
// 不推荐: yarray.insert(0, ['item'])
// 推荐: 使用 Y.Map 或自定义 ID 索引
```

**可能的追问：**

1. **Q: Yjs 和 OT 哪个更好？**
   - Yjs: 去中心化，实现简单，适合协作场景
   - OT: 需要中心服务器，但更适合实时通信（如 Google Docs）
   - 现代协作编辑器多采用 Yjs（如 Figma, Notion）

2. **Q: Yjs 有什么性能问题？**
   - 大文档可能导致更新过大
   - 需要增量同步（Yjs 支持）
   - 内存占用随操作数量增长（需要 GC）

3. **Q: 如何处理"删除冲突"？**
   - Yjs 使用"墓碑"机制
   - 删除不立即清除，而是标记
   - 后续操作根据墓碑判断

### 问题 5：Yjs 的数据类型有哪些？Text、Array、Map 区别是什么？

**考察点：** Yjs 数据类型

**参考答案：**

#### Yjs 核心数据类型

```typescript
import * as Y from 'yjs';

const ydoc = new Y.Doc();

// Yjs 提供 5 种基础数据类型
const ytext = ydoc.getText('text');        // 文本
const yarray = ydoc.getArray('array');      // 数组
const ymap = ydoc.getMap('map');            // Map
const yxml = ydoc.getXmlFragment('xml');    // XML 片段
const yxmlElement = ydoc.getXmlElement('xmlElement'); // XML 元素
```

```typescript
// 操作队列
interface OperationQueue {
  pending: Y.Update[];     // 待同步操作
  synced: Y.Update[];      // 已确认同步
  failed: Y.Update[];     // 同步失败
}

// 1. 本地操作立即应用
function localUpdate(editor: Editor, change: any) {
  // 立即更新本地 Yjs
  ytext.applyDelta(change);

  // 加入待同步队列
  const update = Y.encodeStateAsUpdate(ydoc);
  operationQueue.pending.push(update);

  // 保存到 IndexedDB
  saveToIndexedDB(operationQueue);
}

// 2. 网络恢复后增量同步
async function syncPendingChanges() {
  for (const update of operationQueue.pending) {
    try {
      await socket.emitWithAck('syncUpdate', {
        documentId,
        update: Buffer.from(update).toString('base64'),
      });
      operationQueue.synced.push(update);
    } catch (e) {
      operationQueue.failed.push(update);
    }
  }

  // 清理已同步操作
  operationQueue.pending = operationQueue.failed;
  operationQueue.failed = [];
  operationQueue.synced = [];
}
```

```typescript
function useNetworkStatus() {
  const [isOnline, setIsOnline] = useState(navigator.onLine);

    const handleOnline = () => {
      setIsOnline(true);
      // 触发同步

    };
  }, []);

  return isOnline;
}
```

```typescript
  overscan: number;        // overscan 数量
  containerHeight: number; // 容器高度

// 计算可见范围
function calculateVisibleBlocks(
  blocks: Block[],
  scrollTop: number,
  containerHeight: number,
  itemHeight: number = 60,
  overscan: number = 5
) {
  const startIndex = Math.max(
    0,
    Math.floor(scrollTop / itemHeight) - overscan
  );

  const endIndex = Math.min(
    blocks.length - 1,
    Math.ceil((scrollTop + containerHeight) / itemHeight) + overscan
  );

  const visibleBlocks = blocks.slice(startIndex, endIndex + 1);

  return {
    visibleBlocks,
    startIndex,
    endIndex,
    totalHeight: blocks.length * itemHeight,
    offsetY: startIndex * itemHeight,
  };
}

// 虚拟列表组件
function VirtualBlockList({ blocks }: { blocks: Block[] }) {
  const [scrollTop, setScrollTop] = useState(0);
  const containerRef = useRef<HTMLDivElement>(null);

  const { visibleBlocks, totalHeight, offsetY } = useMemo(
    () => calculateVisibleBlocks(blocks, scrollTop, containerHeight),
    [blocks, scrollTop]
  );

  return (
    <div
      ref={containerRef}
      onScroll={(e) => setScrollTop(e.target.scrollTop)}
      style={{ height: '100%', overflow: 'auto' }}
    >
```

```typescript
// 1. 基础防抖实现
function debounce<T extends (...args: any[]) => any>(
  func: T,
  wait: number
): (...args: Parameters<T>) => void {
  let timeout: NodeJS.Timeout | null = null;

  return function (this: any, ...args: Parameters<T>) {
    if (timeout) clearTimeout(timeout);

    timeout = setTimeout(() => {
      func.apply(this, args);
    }, wait);
  };
}

// 2. 编辑器防抖保存
const saveDocument = debounce(async (blocks: Block[]) => {
```

#### 请求合并 (Batching)

```typescript
// 批量更新队列
class UpdateBatcher {
  private pending: BlockUpdate[] = [];
  private flushTimeout: NodeJS.Timeout | null = null;
  private readonly BATCH_DELAY = 100;  // 100ms 内合并

  add(update: BlockUpdate) {
    this.pending.push(update);

    // 如果已经有 flush 定时器，不要取消
    // 让多个更新一起发送
    if (!this.flushTimeout) {
      this.flushTimeout = setTimeout(() => {
        this.flush();
      }, this.BATCH_DELAY);
    }
  }

  private async flush() {
    if (this.pending.length === 0) return;

    // 合并更新
    const batchedUpdates = this.mergeUpdates([...this.pending]);
    this.pending = [];
    this.flushTimeout = null;

    // 发送请求
    await api.updateBlocksBatch(batchedUpdates);
  }

  private mergeUpdates(updates: BlockUpdate[]): BlockUpdate[] {
    // 按 blockId 分组
    const grouped = new Map<string, BlockUpdate>();

    for (const update of updates) {
      const existing = grouped.get(update.blockId);
      if (!existing) {
        grouped.set(update.blockId, update);
      } else {
        // 合并：后面的覆盖前面的
        grouped.set(update.blockId, {
          ...existing,
          content: update.content,
```

```typescript
// 保存策略配置
interface SaveStrategy {
  // 输入防抖
  inputDebounceMs: number;

  // 空闲保存（用户停止输入后）
  idleSaveMs: number;

  // 定期自动保存
  autoSaveIntervalMs: number;

  // 离开页面保存
  saveOnLeave: boolean;
}

const defaultStrategy: SaveStrategy = {
  inputDebounceMs: 300,
  idleSaveMs: 2000,
  autoSaveIntervalMs: 30000,
  saveOnLeave: true,
};

// 完整实现
class DocumentSaver {
  private strategy: SaveStrategy;
  private lastSaveTime: number = 0;
  private hasUnsavedChanges: boolean = false;
  private idleTimer: NodeJS.Timeout | null = null;

  constructor(strategy: SaveStrategy = defaultStrategy) {
    this.strategy = strategy;
    this.startAutoSave();
  }

  // 输入时防抖保存
  onInput(blocks: Block[]) {
    this.hasUnsavedChanges = true;

    debounce(() => {
      this.save(blocks);
    }, this.strategy.inputDebounceMs)();
  }

  // 空闲保存
  resetIdleTimer(blocks: Block[]) {
    if (this.idleTimer) clearTimeout(this.idleTimer);

    this.idleTimer = setTimeout(() => {
      if (this.hasUnsavedChanges) {
        this.save(blocks);
      }
    }, this.strategy.idleSaveMs);
  }

  // 定期自动保存
  private startAutoSave() {
    setInterval(() => {
      if (this.hasUnsavedChanges) {
        this.save();
      }
    }, this.strategy.autoSaveIntervalMs);
  }

  // 离开页面保存
  saveBeforeUnload(blocks: Block[]) {
    if (this.hasUnsavedChanges) {
      // 同步保存到本地
      localStorage.setItem(`draft-${documentId}`, JSON.stringify(blocks));
    }
  }
}
```

```typescript
// 权限装饰器
function requirePermission(permission: Permission) {
  return function (target: any, propertyKey: string, descriptor: PropertyDescriptor) {
    const originalMethod = descriptor.value;

    descriptor.value = async function (...args: any[]) {
      const user = getCurrentUser();
      const resourceId = args[0];

      // 检查权限
      const hasPermission = await checkPermission(user.id, resourceId, permission);
      if (!hasPermission) {
        throw new ForbiddenException('没有权限执行此操作');
      }

      return originalMethod.apply(this, args);
    };

    return descriptor;
  };
}
```
