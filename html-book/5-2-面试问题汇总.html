第2章 FastDocument

FastDocument 是一款面向团队协作的现代化文档编辑与管理平台，支持实时多人协作编辑、视频会议、版本历史管理等企业级功能。

1.1 问题1：编辑器架构是怎么样的？为什么选择块级编辑器？

考察点： 编辑器架构、ProseMirror、选型理由

参考答案：

块级编辑器核心架构

FastDocument 采用自定义块编辑器架构，核心特点：

  1. 原子化块设计：每个内容块独立管理（text, h1, h2, todo, code, image 等）
  2. 块级操作：支持拖拽排序、分块加载、块类型转换
  3. 独立渲染：每个块独立渲染，便于实现虚拟滚动
  4. 协作友好：以块为最小变更单元，冲突粒度更细
interface Block {
  id: string;           // 唯一标识
  type: BlockType;      // 块类型
  content: string;      // 文本内容
  properties?: {
    checked?: boolean;  // todo 项
    language?: string; // 代码语言
    url?: string;      // 图片 URL
  };
  order?: number;       // 排序
}

与富文本编辑器的对比

特性 块级编辑器 富文本编辑器
数据结构 JSON 块数组 HTML/XML
冲突粒度 块级 文档级
局部更新 高效 困难
虚拟滚动 原生支持 需深度定制
协作 更友好 复杂

问题 2：编辑器的内容怎么存储？JSON 还是 HTML？

考察点： 数据结构设计、数据格式选型

参考答案：

FastDocument 采用 JSON 块级结构

{
  "id": "doc-123",
  "title": "我的文档",
  "blocks": [
    { "id": "block-1", "type": "h1", "content": "标题" },
    { "id": "block-2", "type": "text", "content": "这是普通文本" },
    { "id": "block-3", "type": "todo", "content": "待办事项", "properties": { "checked": false } }
  ]
}

JSON 存储的优点：

  • 语义化清晰，结构明确
  • 局部更新高效（修改单个块不影响其他块）
  • 协作冲突粒度细（以块为单位）
  • 易于进行数据分析、搜索、导出
  • 类型安全，便于校验

问题 3：怎么做编辑器的撤销/重做？

考察点： 历史管理、编辑器原理

参考答案：

方案：操作日志 + 状态快照

interface DocumentState {
  history: { blocks: Block[]; title: string }[];
  historyIndex: number;
  canUndo: boolean;
  canRedo: boolean;
}

// 撤销操作
function undo() {
  if (historyIndex > 0) {
    historyIndex--;
    const prevState = history[historyIndex];
    applyState(prevState);
  }
}

// 重做操作
function redo() {
  if (historyIndex < history.length - 1) {
    historyIndex++;
    const nextState = history[historyIndex];
    applyState(nextState);
  }
}

2.1 问题4：多人同时编辑一个文档，怎么解决冲突？

考察点： CRDT、Yjs、冲突解决

参考答案：

两种主流方案对比

方案一：OT (Operational Transformation)

  • 中心服务器转换操作
  • Google Docs 早期使用
  • 需要中心服务器，复杂度高

方案二：CRDT (无冲突复制数据类型)

  • 每个客户端独立操作，最终自动合并
  • Yjs、Automerge 等库采用
  • 去中心化，更适合分布式

FastDocument 采用的方案

项目使用 Socket.io 事件流 + 乐观更新：

  1. 本地先更新：用户操作立即反映到 UI
  2. 发送到服务端：通过 WebSocket 发送变更
  3. 服务端广播：广播给房间内其他用户
  4. 冲突处理：块级变更，冲突概率更低
// 乐观更新示例
function handleUpdateBlock(blockId: string, content: string) {
  // 1. 本地立即更新
  updateLocalBlock(blockId, content);

  // 2. 发送到服务端
  socket.emit('updateBlock', { docId, blockId, content });
}

问题 5：如何保证协作的实时性？

考察点： WebSocket、实时通信

参考答案：

  1. WebSocket 全双工通信
  2. 心跳检测：定时 ping/pong 检测连接状态
  3. 断线重连：自动重连并恢复状态
  4. 增量同步：只同步变更部分，减少带宽
// 心跳检测
setInterval(() => {
  if (socket.connected) {
    socket.emit('ping');
  }
}, 30000);

// 断线重连
socket.on('disconnect', () => {
  reconnect();
});

6.1 Q: 遇到的最大技术挑战是什么？

回答框架：

  1. 描述挑战场景
  2. 尝试的解决方案
  3. 最终方案
  4. 学到的经验

示例：

在实现大文档编辑器时，遇到滚动卡顿的问题。最初尝试使用普通渲染，但 500+ 块的文档导致严重掉帧。

后来采用虚拟滚动方案，只渲染可见区域内的块，结合分块加载从服务端获取数据。

最终成功支持了 1000+ 块的大型文档，滚动帧率保持在 55FPS 以上。

8.1 30 秒电梯陈述

我在 FastDocument 里负责"协作编辑链路"的核心交付，覆盖 Editor、VirtualEditor、实时同步与版本历史。项目目标是把团队文档协作从"单人编辑 + 低效沟通"升级到"多人实时协同 + 可追溯版本"。我主导了块级编辑器与增量同步方案，解决了大文档卡顿与多人冲突问题，核心收益是：编辑流畅度显著提升、协作延迟可控、线上故障定位时间缩短，满足企业场景下稳定性与审计要求。

8.2 3 分钟深挖版本

  1. 业务目标拆解

    • 协作效率：减少"等待他人释放文档"的阻塞
    • 稳定与可信：支持版本回滚、评论追踪、分享权限
    • 可扩展：后续接入知识库、会议、任务看板的统一入口

  2. 关键技术决策

    • 文档数据模型采用块级结构而非整文字符串，降低冲突面，便于局部更新
    • 前端状态采用 Zustand，按域拆分 store，避免巨型全局状态
    • 大文档场景启用 VirtualEditor，将渲染复杂度从"与总块数线性增长"改为"与可视窗口相关"
    • 实时链路采用 Socket 事件 + 乐观更新，兼顾交互响应与最终一致性

  3. 难点与解法

    • 难点 A：多人同时编辑同一区域导致内容抖动
      • 解法：块级最小变更单元 + 服务端顺序广播 + 客户端幂等处理
    • 难点 B：大文档滚动和输入耦合导致掉帧
      • 解法：虚拟滚动 + 分层 memo + 高频事件节流
    • 难点 C：断网/弱网下用户感知差
      • 解法：本地操作队列 + 在线恢复后增量回放
