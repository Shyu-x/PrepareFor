1.1 模块化规范

1.1.1 模块化的发展历程

模块化是前端工程化的基础，它经历了从无到有、从简单到复杂的发展过程。理解模块化的发展历程有助于我们理解为什么需要模块化，以及不同模块化规范存在的意义。

概念语法详细解释

模块化的核心思想是将一个大型的程序拆分成相互依赖的小文件，这些小文件（模块）可以独立开发、测试和维护，最后通过特定的导入导出机制组合在一起。模块化解决了以下核心问题：

  • 命名空间污染：避免全局变量冲突
  • 依赖管理：明确模块之间的依赖关系
  • 代码复用：提高代码的可重用性
  • 可维护性：便于代码的维护和更新

早期的全局变量模式 在早期，JavaScript没有模块化机制，开发者通常使用全局变量来共享代码：

// module1.js
var name = 'Module1';

function sayHello() {
  console.log('Hello from ' + name);
}

// 使用全局变量
window.Module1 = {
  name: name,
  sayHello: sayHello
};

// main.js
Module1.sayHello(); // "Hello from Module1"

特别说明（常见误区、性能、兼容性、面试考点）

  • 全局变量模式是早期最简单的模块化尝试，但存在严重的命名冲突问题
  • 当项目规模增大时，全局变量模式会导致"变量命名战争"，不同团队开发的代码可能相互覆盖
  • 面试常考点：为什么全局变量模式不适合大型项目？如何解决命名冲突？

帮助理解（类比、场景分析、优缺点对比）

想象一个公司没有部门划分，所有人都在同一个大办公室里工作，每个人的文件都放在公共桌上。这就会造成：有人拿错了别人的文件、有人不小心删除了别人的数据、很难追踪是谁修改了什么。模块化就像是为每个部门建立独立的办公室，每个部门有自己的文件柜，只有通过特定的方式（门禁系统）才能访问其他部门的资源。

这种方式的缺点：

  • 命名冲突：不同模块可能使用相同的变量名
  • 依赖混乱：无法明确模块之间的依赖关系
  • 难以维护：模块之间的关系不清晰

IIFE（立即调用函数表达式） IIFE是一种早期的模块化解决方案，通过创建函数作用域来避免全局污染：

// module.js
var Module = (function() {
  var _private = 'I am private';

  function _privateMethod() {
    console.log(_private);
  }

  return {
    publicMethod: function() {
      _privateMethod();
    }
  };
})();

Module.publicMethod(); // "I am private"
console.log(Module._private); // undefined

概念语法详细解释

IIFE（Immediately Invoked Function Expression，立即调用函数表达式）是一种创建私有作用域的技术。它的核心原理是：

  • 使用函数作用域包裹代码，创建私有变量
  • 通过返回值（通常是一个对象）暴露有限的公共API
  • 函数在定义后立即执行，返回一个包含方法的对象

IIFE 实现了基本的封装和信息隐藏，是现代模块化规范的雏形。

// 模块依赖
var ModuleA = (function() {
  return {
    name: 'Module A'
  };
})();

var ModuleB = (function(ModuleA) {
  var name = 'Module B';

  function getFullName() {
    return name + ' depends on ' + ModuleA.name;
  }

  return {
    getFullName: getFullName
  };
})(ModuleA);

console.log(ModuleB.getFullName()); // "Module B depends on Module A"

具体应用实例

在实际项目中，IIFE 常用于：

  • 创建独立的第三方库作用域（如 jQuery）
  • 封装不需要暴露给外部的工具函数
  • 模拟私有方法和属性

帮助理解

IIFE 就像是一个"保险箱"。你把贵重物品（私有变量）放在保险箱里，只有通过特定的钥匙（返回的对象方法）才能访问。这样即使有人进入了你的房间（全局作用域），也无法直接拿走你的贵重物品。

1.1.2 CommonJS规范

CommonJS是Node.js使用的模块化规范，它使用require来引入模块，module.exports来导出模块。

概念语法详细解释

CommonJS 是 Node.js 的官方模块系统，其核心特点包括：

  • 同步加载：适合服务器端环境，因为文件都在本地磁盘
  • 运行时加载：模块在运行时才确定依赖关系
  • 值拷贝：导出的是值的拷贝，导出的原始值变化不会影响导入方
  • 缓存机制：模块只会加载一次，后续 require 直接从缓存获取

核心API：

  • require(id)：加载模块
  • module.exports：导出模块
  • exports：module.exports 的简写形式
  • __filename：当前文件的绝对路径
  • __dirname：当前目录的绝对路径

基本用法

// math.js - 导出模块
function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}

function multiply(a, b) {
  return a * b;
}

function divide(a, b) {
  if (b === 0) {
    throw new Error('Cannot divide by zero');
  }
  return a / b;
}

// 导出方式1：导出多个
module.exports = {
  add,
  subtract,
  multiply,
  divide
};

// index.js - 引入模块
const math = require('./math');

console.log(math.add(1, 2)); // 3
console.log(math.subtract(5, 3)); // 2
console.log(math.multiply(3, 4)); // 12
console.log(math.divide(10, 2)); // 5

特别说明

  • CommonJS 是同步的，在浏览器环境中会导致页面阻塞
  • webpack 等构建工具会将 CommonJS 转换为浏览器可用的格式
  • 注意区分 module.exports 和 exports，后者只是前者的引用

模块导出函数或类

// logger.js
class Logger {
  constructor(prefix) {
    this.prefix = prefix;
  }

  log(message) {
    console.log(`[${this.prefix}] ${message}`);
  }

  error(message) {
    console.error(`[${this.prefix}] ERROR: ${message}`);
  }

  warn(message) {
    console.warn(`[${this.prefix}] WARN: ${message}`);
  }
}

module.exports = Logger;

面试考点

  • CommonJS 的加载过程是怎样的？（从模块查找到缓存机制）
  • module.exports 和 exports 的区别
  • 为什么 CommonJS 不适合浏览器环境？

模块导出单例

// store.js
let count = 0;

function increment() {
  count++;
  return count;
}

function decrement() {
  count--;
  return count;
}

function getCount() {
  return count;
}

// 导出函数而非对象，每次require返回的都是同一个模块实例
module.exports = {
  increment,
  decrement,
  getCount
};

帮助理解

CommonJS 的单例模式就像一个"饮水机"。当多个地方的人都需要喝水时，他们不是各自买一台饮水机，而是共享同一台。第一个 require 的人"安装"了饮水机（执行模块代码），后面的人直接使用这台饮水机（从缓存获取）。

1.1.3 AMD规范

AMD（Asynchronous Module Definition）是异步模块定义规范，主要用于浏览器环境，解决了CommonJS在浏览器中同步加载的问题。

概念语法详细解释

AMD 的核心设计理念是"异步加载"：

  • 依赖前置：在定义模块时就声明所有依赖
  • 异步加载：所有依赖并行加载，不阻塞页面渲染
  • 回调机制：通过回调函数获取模块

核心API：

  • define(id?, dependencies?, factory)：定义模块
  • require(dependencies, callback)：加载模块

AMD 的全称是 Asynchronous Module Definition（异步模块定义），这个"异步"指的就是依赖的加载方式是异步的。

基本用法

// 定义模块
define('myModule', ['dependency1', 'dependency2'], function(dep1, dep2) {
  // 模块逻辑
  function doSomething() {
    return dep1.getValue() + dep2.getValue();
  }

  // 导出
  return {
    doSomething: doSomething
  };
});

// 引入模块
require(['myModule', 'simpleModule'], function(myModule, simpleModule) {
  console.log(myModule.doSomething());
  console.log(simpleModule.name);
});

具体应用实例

AMD 最著名的实现是 RequireJS，它解决了以下问题：

  • 浏览器端的模块加载
  • 多种文件类型的支持（JS、CSS等）
  • 模块之间的依赖管理
  • 动态加载和懒加载

特别说明

  • AMD 需要引入额外的库（如 RequireJS）才能使用
  • 依赖前置意味着即使某个依赖暂时用不到，也会提前加载
  • 语法相对冗长，需要维护大量的数组和回调

面试考点

  • AMD 和 CommonJS 的区别
  • 依赖前置 vs 依赖就近的优劣

1.1.4 CMD规范

CMD（Common Module Definition）是SeaJS推广的模块定义规范，与AMD类似但有所不同。CMD推崇依赖就近，延迟执行。

概念语法详细解释

CMD 与 AMD 的核心区别在于"何时加载依赖"：

  • 依赖就近：在需要使用依赖时才 require
  • 延迟执行：模块工厂函数在需要时才执行
  • 就近书写：依赖写在函数内部，代码更直观

CMD 的代表库是 SeaJS（阿里巴巴前端团队开发），虽然现在已经停止维护，但它对前端模块化的发展产生了重要影响。

基本用法

// SeaJS示例
// math.js
define(function(require, exports, module) {
  var add = function(a, b) {
    return a + b;
  };

  var subtract = function(a, b) {
    return a - b;
  };

  exports.add = add;
  exports.subtract = subtract;
});

AMD与CMD的区别

// AMD - 依赖前置
define(['dep1', 'dep2', 'dep3'], function(dep1, dep2, dep3) {
  // 所有依赖在定义时就已经加载
  dep1.doSomething();
  dep2.doSomething();
});

// CMD - 依赖就近
define(function(require, exports, module) {
  var dep1 = require('dep1'); // 使用时再加载
  dep1.doSomething();

  if (condition) {
    var dep2 = require('dep2'); // 条件加载
    dep2.doSomething();
  }
});

帮助理解

  • AMD 就像提前买好所有食材放进冰箱，做饭时直接从冰箱拿
  • CMD 就像去超市买菜，做饭时需要什么再去买什么
  • 两种方式各有优劣：AMD 预加载更快，CMD 按需加载更省资源

1.1.5 ES Module规范

ES Module（ESM）是ES6引入的官方模块化规范，使用import和export关键字。

概念语法详细解释

ES Module 是 ECMAScript 官方推出的模块系统，具有以下特点：

  • 静态语法：import/export 必须在模块顶层，不能在条件语句中
  • 编译时解析：在编译阶段就能确定依赖关系
  • 实时绑定：导入的是值的引用，导出的原始值变化会反映到导入方
  • 自动严格模式：ESM 自动使用严格模式
  • 独立的作用域：每个 ESM 模块有自己的顶层作用域

核心语法：

  • import { xxx } from 'module'：命名导入
  • import defaultExport from 'module'：默认导入
  • import * as namespace from 'module'：命名空间导入
  • export const/function/class：命名导出
  • export default：默认导出

基本导出方式

命名导出

// named-export.js
// 方式1：声明时导出
export const name = 'Module';
export const version = '1.0.0';

export function add(a, b) {
  return a + b;
}

export class Calculator {
  constructor() {
    this.result = 0;
  }

  add(n) {
    this.result += n;
    return this;
  }

  subtract(n) {
    this.result -= n;
    return this;
  }

  getResult() {
    return this.result;
  }
}

默认导出

// default-export.js
export default function() {
  return 'Default export';
}

特别说明

  • 一个模块可以同时有默认导出和命名导出
  • 命名导出适合导出多个相关功能，默认导出适合导出单一功能
  • 面试常考点：ES Module 和 CommonJS 的区别

基本导入方式

命名导入

// import-named.js
import { name, version, add, Calculator } from './named-export.js';

console.log(name); // 'Module'
console.log(version); // '1.0.0'
console.log(add(1, 2)); // 3

默认导入

// import-default.js
import defaultExport from './default-export.js';

console.log(defaultExport()); // 'Default export'

命名空间导入

// import-namespace.js
import * as module from './named-export.js';

console.log(module.name); // 'Module'
console.log(module.add(1, 2)); // 3

动态导入

// dynamic-import.js
// 返回Promise
import('./module.js')
  .then(module => {
    console.log(module.default);
    console.log(module.namedExport);
  });

// 使用async/await
async function loadModule() {
  const module = await import('./module.js');
  return module;
}

帮助理解

ES Module 就像是一个"智能文件柜"。当你需要某个文件时，你可以：

  • 只拿你需要的那一页（命名导入）
  • 拿整本手册（命名空间导入）
  • 拿默认版本（默认导入）
  • 等等需要的时候再拿（动态导入）

而且，这个柜子和文件是"实时同步"的——如果你修改了源文件，所有引用这个文件的地方都会看到更新。

ES Module的高级特性

静态导入与导出 ES Module使用静态语法，导入导出必须在模块顶层，这使得：

  • 静态分析成为可能
  • Tree Shaking可以移除未使用的代码
  • 打包时可以确定依赖关系

ES Module在浏览器中的使用

<!-- index.html -->
<script type="module">
  import { createApp } from './app.js';

  createApp();
</script>

面试考点

  • 为什么 ES Module 是静态的？这带来了哪些好处？
  • ES Module 和 CommonJS 的主要区别
  • Tree Shaking 的原理

1.1.6 UMD规范

UMD（Universal Module Definition）是一种兼容多种模块化规范的模式，可以在CommonJS、AMD和全局变量环境中使用。

概念语法详细解释

UMD 的设计目标是"一次编写，处处运行"。它通过检测当前环境来选择合适的模块加载方式：

  • 检测 CommonJS 环境（module.exports 存在）
  • 检测 AMD 环境（define 存在）
  • 都不是则使用全局变量

UMD 本质上是一个"适配器"模式，它让同一个文件可以在不同环境中正常工作。

UMD模式

// umd-module.js
(function(root, factory) {
  // 检测环境
  if (typeof module === 'object' && module.exports) {
    // CommonJS
    module.exports = factory();
  } else if (typeof define === 'function' && define.amd) {
    // AMD
    define([], factory);
  } else {
    // 全局变量
    root.MyModule = factory();
  }
}(typeof self !== 'undefined' ? self : this, function() {
  // 模块实现
  return {
    name: 'My Module',
    version: '1.0.0',
    init: function() {
      console.log('Module initialized');
    }
  };
}));

具体应用实例

很多流行的库（如 jQuery、moment.js）使用 UMD 来兼容不同的模块系统，这样无论用户使用哪种方式引入，都能正常工作。

特别说明

  • UMD 代码通常比单一模块规范更冗长
  • 现代项目推荐直接使用 ESM 或 CommonJS，UMD 主要是为了向后兼容
  • 如果要支持 IE11 等老旧浏览器，UMD 仍然是有用的

1.2.2 模块与组件的设计原则

SOLID原则在前端中的应用

单一职责原则（SRP） 每个模块、组件只负责一项功能。

// 正确：分离职责
// validators/user.js
export function validateUser(user) {
  if (!user.name || !user.email) {
    throw new Error('Invalid user data');
  }
  return true;
}

// formatters/user.js
export function formatUser(user) {
  return {
    name: user.name.trim(),
    email: user.email.toLowerCase()
  };
}

// services/user.js
export async function saveUser(user) {
  const response = await fetch('/api/users', {
    method: 'POST',
    body: JSON.stringify(user)
  });
  return response.json();
}

// 使用
import { validateUser } from './validators/user';
import { formatUser } from './formatters/user';
import { saveUser } from './services/user';

async function createUser(userData) {
  validateUser(userData);
  const formattedUser = formatUser(userData);
  return saveUser(formattedUser);
}

开闭原则（OCP） 对扩展开放，对修改封闭。

// 正确：使用策略模式
// 支付策略接口
class PaymentStrategy {
  pay(amount) {
    throw new Error('Method not implemented');
  }
}

// 具体支付策略
class AlipayStrategy extends PaymentStrategy {
  pay(amount) {
    console.log(`Alipay: ${amount}`);
  }
}

class WechatPayStrategy extends PaymentStrategy {
  pay(amount) {
    console.log(`Wechat Pay: ${amount}`);
  }
}

// 支付处理器
class PaymentProcessor {
  constructor(strategy) {
    this.strategy = strategy;
  }

  setStrategy(strategy) {
    this.strategy = strategy;
  }

  processPayment(amount) {
    return this.strategy.pay(amount);
  }
}

1.3.2 版本管理

语义化版本（SemVer）

版本格式：主版本.次版本.修订号

# ^ 允许次版本和修订号升级
npm install lodash@^4.17.21
# 会安装 4.x.x 中的最新版本

# ~ 允许修订号升级
npm install lodash@~4.17.21
# 会安装 4.17.x 中的最新版本

# 精确版本
npm install lodash@4.17.21
