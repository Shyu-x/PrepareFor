第1章 前端工程化

1.1 Git工作流

概念语法详细解释

Git工作流是指团队在项目中如何使用Git进行代码管理和协作的流程规范。一个良好的Git工作流可以：

  • 提高团队协作效率
  • 减少代码冲突
  • 保证代码质量
  • 支持并行开发
  • 便于问题追溯

常见的Git工作流有：Git Flow、GitHub Flow、GitLab Flow等，每种都有其适用场景。

1.1.1 Git基础

初始化和配置

# 初始化仓库
git init

# 配置用户信息
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

# 配置别名
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.cm commit

具体应用实例

配置Git别名可以显著提升开发效率，以下是一些推荐的别名配置：

# 高级别名配置
git config --global alias.lg "log --color --graph --pretty=format:'%Cred%h%Creset -%C(yellow)%d%Creset %s %Cgreen(%cr) %C(bold blue)<%an>%Creset' --abbrev-commit"
git config --global alias.last "log -1 HEAD --stat"
git config --global alias.unstage "reset HEAD --"
git config --global alias.graph "log --graph --oneline --all"

基本操作

# 克隆仓库
git clone https://github.com/user/repo.git

# 添加文件
git add filename.txt
git add .  # 添加所有文件

# 提交
git commit -m "Commit message"

# 查看状态
git status

# 查看差异
git diff

# 查看历史
git log
git log --oneline

# 撤销操作
git checkout -- filename  # 放弃工作区修改
git restore filename  # 同上（新版）
git reset HEAD~1  # 撤销最后一次提交

特别说明（常见误区）

  • git checkout -- file 和 git restore file 的区别：两者在新版本Git中作用相同，后者是更推荐的语法
  • git reset 的三种模式：soft（保留暂存区和工作区）、mixed（保留工作区，默认）、hard（丢弃所有修改）
  • 面试常考点：git reset、git revert、git checkout 的区别

帮助理解（类比）

  • git add 就像是"把文件放进收件箱"
  • git commit 就像是"把收件箱里的东西装进已发送文件夹"
  • git reset HEAD~1 就像是"从已发送文件夹里拿出来，放回收件箱"

1.1.2 分支操作

# 查看分支
git branch

# 创建分支
git branch feature-login
git checkout -b feature-login  # 创建并切换

# 切换分支
git checkout feature-login

# 合并分支
git checkout main
git merge feature-login

# 变基
git checkout feature-login
git rebase main

# 删除分支
git branch -d feature-login
git push origin --delete feature-login  # 删除远程分支

概念语法详细解释

Git分支是Git最强大的功能之一，它允许你在不影响主线的情况下进行开发：

  • 分支指针：分支本质上是一个指向特定提交的指针
  • HEAD：一个特殊的指针，指向当前检出的提交或分支
  • 快速前进（Fast-forward）：当没有分叉时，Git直接移动指针
  • 三方合并：当有分叉时，Git创建一个新的合并提交

面试考点

  • merge 和 rebase 的区别
  • 何时使用 rebase，何时使用 merge
  • 分支管理策略

帮助理解

想象你在一本书上做笔记：

  • merge 就像是在书的最后一页加上新的注释章节
  • rebase 就像是把你的注释直接插入到原始内容中
  • rebase 保持历史线性，merge 保留完整历史

1.1.3 远程仓库操作

# 添加远程仓库
git remote add origin https://github.com/user/repo.git

# 拉取
git fetch origin
git pull origin main

# 推送
git push origin main
git push -u origin feature-login

特别说明

  • git fetch 和 git pull 的区别：fetch只下载不合并，pull是fetch+merge
  • 建议经常使用 git fetch 查看远程更新，再决定是否合并

1.1.4 标签管理

# 创建标签
git tag v1.0.0
git tag -a v1.0.0 -m "Release v1.0.0"

# 查看标签
git tag
git tag -l "v1.*"

# 推送标签
git push origin v1.0.0
git push origin --tags

具体应用实例

标签常用于版本发布：

# 创建签名标签（需要GPG密钥）
git tag -s v1.0.0 -m "Signed release v1.0.0"

# 删除本地标签
git tag -d v1.0.0

1.1.5 工作流模式

Git Flow

main:     ━━━━●━━━━━━━━━━●━━━━━━━━━━●━━━━━━━━━━
               ↑          ↑          ↑
            release    release    release
                      /    \
feature:  ●━━━●━━━●━━━●      ↓
                       develop
develop:  ━━━━●━━━●━━━●━━━●━━━●━━━●━━━●━━━●

概念语法详细解释

Git Flow 是一个成熟的分支管理策略，适合有固定发布周期的项目：

  • main/master：生产环境代码，只接受合并
  • develop：开发主分支，集成所有功能分支
  • feature：功能分支，从develop创建，开发完成后合并回develop
  • release：发布分支，从develop创建，用于发布准备
  • hotfix：热修复分支，从main创建，用于紧急修复

适用场景

  • 适合有明确版本发布周期的项目
  • 适合多人在同一项目上协作
  • 不适合持续发布的项目

GitHub Flow

# 从main创建分支
git checkout -b feature/add-login

# 开发并提交
git add .
git commit -m "Add login feature"

# 推送并创建PR
git push -u origin feature/add-login
# 在GitHub创建Pull Request

帮助理解

GitHub Flow 就像是一个"随时可以发布"的模式：

  • 每次开发都在新分支进行
  • 完成后通过PR合并到main
  • 合并后立即可以部署
  • 适合持续部署的互联网产品

1.2 分支管理策略

1.2.1 常用分支类型

main (或 master)    - 生产环境代码
develop             - 开发主分支
feature/*           - 功能分支
release/*           - 发布分支
hotfix/*            - 热修复分支
bugfix/*            - Bug修复分支

特别说明

分支命名规范可以提高代码仓库的可读性：

  • feature/：新功能开发
  • bugfix/：Bug修复
  • hotfix/：紧急修复
  • release/：发布准备

1.2.2 分支命名规范

# 功能分支
feature/user-login
feature/add-payment

# Bug修复
bugfix/fix-login-error
bugfix/memory-leak

# 热修复
hotfix/security-patch

# 发布
release/1.0.0

1.3 代码审查

1.3.1 代码审查流程

概念语法详细解释

代码审查（Code Review）是保证代码质量的重要环节，通过同行评审来：

  • 发现潜在bug
  • 保证代码风格一致
  • 分享知识
  • 提高整体代码质量

一个有效的代码审查流程应该包括：审查清单、反馈规范、自动化工具集成。

Pull Request创建规范

## 描述
简要描述这个PR解决的问题或添加的功能

## 变更类型
- [ ] Bug修复
- [ ] 新功能
- [ ] 破坏性变更
- [ ] 文档更新

## 测试
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 手动测试

代码审查检查点

  1. 代码质量

    • 代码是否清晰易读
    • 命名是否有意义
    • 函数是否保持单一职责
    • 是否有重复代码

  2. 功能正确性

    • 代码是否实现了预期功能
    • 边界情况是否处理
    • 错误处理是否完善

  3. 性能

    • 是否有性能问题
    • 是否有内存泄漏风险

  4. 安全性

    • 是否有安全漏洞
    • 敏感数据是否正确处理

  5. 测试

    • 是否有足够的测试覆盖
    • 测试用例是否正确

帮助理解

代码审查就像是一个"集体Code Review"：

  • 每个人都可以提出问题
  • 每个人都可以学习新技术
  • 团队整体水平会不断提升

1.4 ESLint与Prettier配置

1.4.1 ESLint高级配置

概念语法详细解释

ESLint 是一个可配置的JavaScript和JSX检查工具，它：

  • 通过规则检测代码问题
  • 支持自定义规则
  • 可以自动修复部分问题
  • 与主流编辑器集成良好

ESLint 的配置核心包括：环境（env）、规则（rules）、插件（plugins）、继承配置（extends）。

配置文件详解

// .eslintrc.js
module.exports = {
  root: true,
  env: {
    browser: true,
    es2021: true,
    node: true
  },
  extends: [
    'eslint:recommended',
    'plugin:vue/vue3-essential',
    'plugin:@typescript-eslint/recommended',
    'prettier'
  ],
  parser: 'vue-eslint-parser',
  parserOptions: {
    ecmaVersion: 'latest',
    sourceType: 'module'
  },
  plugins: ['vue', '@typescript-eslint'],
  rules: {
    'no-console': process.env.NODE_ENV === 'production' ? 'warn' : 'off',
    'no-debugger': process.env.NODE_ENV === 'production' ? 'warn' : 'off',
    'prefer-const': 'error',
    'no-var': 'error'
  }
};

特别说明

  • extends 中的 ‘prettier’ 必须放在最后，用于关闭与 Prettier 冲突的 ESLint 规则
  • 生产环境应该更严格，开发环境可以适当放宽
  • 使用 eslint --fix 可以自动修复部分问题

1.4.2 Prettier高级配置

// .prettierrc.json
{
  "semi": true,
  "singleQuote": true,
  "tabWidth": 2,
  "useTabs": false,
  "trailingComma": "none",
  "printWidth": 100,
  "bracketSpacing": true,
  "arrowParens": "always"
}

概念语法详细解释

Prettier 是一个"固执己见"的代码格式化工具，它的核心理念是：

  • 有且只有一种"正确"的格式
  • 不需要配置太多选项
  • 保证团队代码风格完全一致

Prettier 主要关注：代码格式（缩进、分号、引号等），不检查代码逻辑错误。

1.4.3 Git Hooks配置

概念语法详细解释

Git Hooks 是在Git执行特定操作时自动运行的脚本：

  • pre-commit：提交前检查
  • commit-msg：提交信息检查
  • pre-push：推送前检查
  • post-checkout：检出后操作

Husky 是一个简化Git Hooks配置的工具。

使用Husky

# 安装
npm install husky --save-dev

# 初始化
npx husky install

# 添加prepare脚本
npm pkg set scripts.prepare="husky install"

# 创建pre-commit hook
npx husky add .husky/pre-commit "npm run lint"

使用lint-staged

// package.json
{
  "lint-staged": {
    "*.{js,ts,vue,jsx,tsx}": [
      "eslint --fix",
      "prettier --write"
    ]
  }
}

具体应用实例

一个完整的Git Hooks配置示例：

# commit-msg hook - 检查提交信息格式
npx husky add .husky/commit-msg 'npx --no -- commitlint --edit "$1"'

# pre-push hook - 运行测试
npx husky add .husky/pre-push "npm test"

1.5 代码规范

1.5.1 命名规范

概念语法详细解释

良好的命名是代码可读性的基础：

  • 变量名应该是名词，表示数据
  • 函数名应该是动词，表示行为
  • 布尔值应该用 is/has/can/should 开头
  • 类名应该用 PascalCase
  • 常量应该用 UPPER_SNAKE_CASE

变量和函数命名

// 变量命名
const userName = 'John';  // 名词，表示数据
const isActive = true;    // 布尔值，以is/has/can/should开头
const userList = [];      // 复数形式表示数组

// 函数命名 - 使用动词或动词短语
function getUserList() {}
function fetchUserInfo() {}
function validateForm() {}
function handleSubmit() {}

// 类命名 - 使用PascalCase
class UserService {}
class ApiClient {}

// 常量命名 - 使用大写和下划线
const MAX_RETRY_COUNT = 3;
const API_BASE_URL = 'https://api.example.com';

面试考点

  • 命名的基本原则
  • 如何平衡命名长度和可读性
  • 常见命名反模式

1.5.2 代码组织规范

文件组织

// 1. 导入语句放在文件顶部
// 标准库导入
import fs from 'fs';
import path from 'path';

// 第三方库导入
import axios from 'axios';

// 内部模块导入
import { UserService } from '@/services/user';

// 2. 类型定义（TypeScript）
interface User {
  id: number;
  name: string;
}

// 3. 常量定义
const DEFAULT_TIMEOUT = 5000;

// 4. 函数和类定义
function formatUser(user) {
  return {
    ...user,
    name: user.name.trim()
  };
}

// 5. 导出
export default UserRepository;
export { formatUser };

帮助理解

代码组织就像整理房间：

  • 衣服放衣柜（导入放顶部）
  • 书籍放书架（类型定义）
  • 工具放工具箱（常量和函数）
  • 东西要归类（按类型分组）

1.6 自动化测试

1.6.1 测试策略

概念语法详细解释

自动化测试是保证代码质量的重要手段，测试金字塔理论指导我们：

  • 单元测试：最多，针对最小代码单元
  • 集成测试：中等，测试模块间的协作
  • E2E测试：最少，模拟用户行为

正确的测试策略可以：快速发现问题、防止回归、提高重构信心。

测试金字塔

        /\
       /  \
      / E2E \        <- 少量：端到端测试
     /--------\
    / 集成测试 \    <- 中等：集成测试
   /------------\
  /  单元测试    \ <- 大量：单元测试
 /----------------\

测试覆盖率目标

  • 语句覆盖率：80%+
  • 分支覆盖率：75%+
  • 函数覆盖率：80%+
  • 行覆盖率：80%+

特别说明

测试覆盖率只是一个指标，不是目标。过度追求覆盖率可能会导致：

  • 测试质量下降
  • 开发时间浪费
  • 维护成本增加

1.6.2 单元测试示例

Jest配置详解

// jest.config.js
module.exports = {
  testEnvironment: 'jsdom',
  testMatch: [
    '**/__tests__/**/*.[jt]s?(x)',
    '**/?(*.)+(spec|test).[jt]s?(x)'
  ],
  transform: {
    '^.+\\.(ts|tsx|js|jsx|mjs){{CONTENT}}#x27;: 'babel-jest',
    '^.+\\.vue{{CONTENT}}#x27;: '@vue/vue3-jest'
  },
  moduleNameMapper: {
    '^@/(.*){{CONTENT}}#x27;: '<rootDir>/src/$1'
  },
  setupFilesAfterEnv: ['<rootDir>/tests/setupTests.js']
};

测试文件示例

// tests/unit/utils/format.spec.js
import { formatDate, formatCurrency } from '@/utils/format';

describe('formatDate', () => {
  it('should format date correctly', () => {
    const date = new Date('2023-01-15');
    expect(formatDate(date)).toBe('2023-01-15');
  });
});

describe('formatCurrency', () => {
  it('should format currency with default options', () => {
    expect(formatCurrency(1000)).toBe('$1,000.00');
  });
});

具体应用实例

一个好的单元测试应该遵循AAA模式：

  • Arrange：准备测试数据
  • Act：执行被测操作
  • Assert：验证结果

1.6.3 集成测试

// tests/integration/api/user.spec.js
import request from 'supertest';
import { app } from '../src/app.js';

describe('User API', () => {
  describe('GET /api/users', () => {

      const response = await request(app)

        .expect(200);

      expect(Array.isArray(response.body)).toBe(true);

  });
});

1.6.4 端到端测试

// cypress/integration/login.spec.js
describe('Login Flow', () => {
  beforeEach(() => {
    cy.visit('/login');
  });

  it('should display login form', () => {
    cy.get('[data-testid="login-form"]').should('be.visible');
  });

  it('should login successfully with valid credentials', () => {

    cy.get('[data-testid="password-input"]').type('password123');

帮助理解

三种测试就像三种"检查"：

  • 单元测试：检查零件是否合格
  • 集成测试：检查零件组装后是否正常工作
  • E2E测试：检查整个产品是否满足用户需求

1.7 CI/CD

1.7.1 CI/CD基础概念

概念语法详细解释

CI/CD 是现代软件开发的核心实践：

  • CI（持续集成）：频繁地将代码集成到主干，每次集成都通过自动化构建和测试来验证
  • CD（持续交付）：在CI的基础上，将代码自动部署到测试环境
  • CD（持续部署）：在持续交付的基础上，将代码自动部署到生产环境

CI/CD 的核心价值：快速反馈、降低风险、提高质量。

持续集成（CI）

  • 频繁地将代码集成到主分支
  • 每次提交触发自动化构建和测试
  • 快速发现集成错误

持续交付（CD）

  • 自动将代码部署到测试环境
  • 准备就绪后可以手动部署到生产
  • 确保代码始终可部署

持续部署

  • 自动将代码部署到生产环境
  • 完全自动化部署流程

1.7.2 GitHub Actions配置

基础工作流配置

# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  lint:
    name: Lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '18'

      - name: Install dependencies
        run: npm ci

      - name: Run ESLint
        run: npm run lint

  test:
    name: Test
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '18'

      - name: Run tests
        run: npm test

特别说明

GitHub Actions 的核心概念：

  • workflow：工作流，一个完整的自动化流程
  • job：作业，一组要执行的步骤
  • step：步骤，具体的操作
  • action：动作，可重用的步骤单元

1.7.3 持续部署

# .github/workflows/vercel.yml
name: Deploy to Vercel

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Vercel CLI
        run: npm install -g vercel@latest

      - name: Pull Vercel Environment Info
        run: vercel pull --yes --environment=production

      - name: Build Project
        run: vercel build --prod

      - name: Deploy to Vercel
        run: vercel deploy --prebuilt --prod

帮助理解

CI/CD 就像是一个"自动化工厂"：

  • 代码就像原材料
  • CI 就是加工生产线（组装、测试）
  • CD 就是物流系统（自动发货）
  • 最终产品快速、稳定地送到用户手中

        env:

具体应用实例

自动发布流程：

  1. 开发者push标签 git tag v1.0.0

  3. 自动运行测试、构建
  4. 自动发布到 npm
  5. 自动创建 GitHub Release
