# 自动化测试体系完全指南

## 前言：为什么你的代码需要"自动化测试"？

想象一下，你是一位餐厅厨师，每天要做100道菜。如果每道菜端给客人前，你都需要亲自尝一口才能确定是否好吃，那你会累死。更糟糕的是，你可能会因为太累而漏检，导致客人吃到难吃的菜。

软件开发也是同理。如果每改一行代码，你都需要手动测试所有功能，那你不仅会累死，还可能漏掉bug。

自动化测试就像给餐厅安装了一套"自动质检系统"：每道菜做好后，机器自动检测温度、味道、摆盘，不合格的自动打回重做。这样厨师就可以专注于做菜本身，而不是质量检查。

## 一、测试金字塔：理解测试的分层策略

### 1.1 测试金字塔是什么？

测试金字塔是一种测试策略的比喻，它告诉我们应该把测试资源按照什么比例分配：

```
                    ┌─────────────┐
                    │    UI层     │  ← E2E 测试（少量）
                    │   (E2E)     │    模拟真实用户操作
                   ├─────────────┤
                  │   集成测试    │  ← 集成测试（适量）
                  │ (Integration) │    测试多个组件配合
                 ├───────────────┤
                │    单元测试     │  ← 单元测试（大量）
                │   (Unit)       │    测试最小功能单元
               └─────────────────┘

    成本 ←─────────────────────────────→ 成本
   快速/廉价                         缓慢/昂贵
```

### 1.2 为什么是金字塔形？

| 层级 | 数量 | 成本 | 速度 | 隔离性 |
|------|------|------|------|--------|
| **单元测试** | 很多（上百个） | 低（毫秒级） | 快（每秒上千个） | 高（完全隔离） |
| **集成测试** | 适量（几十个） | 中（秒级） | 中（每秒几十个） | 中（部分依赖） |
| **E2E测试** | 少量（十几个） | 高（分钟级） | 慢（每分钟几个） | 低（完全真实） |

### 1.3 每层测试的目的

```
┌─────────────────────────────────────────────────────────────┐
│                    测试各层的职责                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  单元测试 (Unit Tests)                               │   │
│  │                                                       │   │
│  │  目的：验证每个"零件"是否正常工作                    │   │
│  │                                                       │   │
│  │  示例：                                              │   │
│  │  - sum(1, 2) 返回 3                                 │   │
│  │  - formatDate(new Date()) 返回 "2024-01-01"         │   │
│  │  - isEmailValid("test@example.com") 返回 true        │   │
│  │                                                       │   │
│  │  特点：                                              │   │
│  │  - 运行极快（毫秒级）                               │   │
│  │  - 可以精确指出哪个函数出问题                        │   │
│  │  - 不需要启动浏览器、数据库等外部依赖                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  集成测试 (Integration Tests)                        │   │
│  │                                                       │   │
│  │  目的：验证多个"零件"能否正常配合工作                │   │
│  │                                                       │   │
│  │  示例：                                              │   │
│  │  - 用户提交表单后，数据是否正确保存到数据库          │   │
│  │  - 组件A点击后，组件B是否正确更新                   │   │
│  │  - API调用后，返回的数据是否正确渲染到UI            │   │
│  │                                                       │   │
│  │  特点：                                              │   │
│  │  - 需要启动部分外部依赖（数据库、API）               │   │
│  │  - 测试组件之间的交互是否正常                        │   │
│  │  - 比单元测试慢，但比E2E快                           │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  端到端测试 (E2E Tests)                               │   │
│  │                                                       │   │
│  │  目的：模拟真实用户操作，验证整个系统是否正常        │   │
│  │                                                       │   │
│  │  示例：                                              │   │
│  │  - 用户登录 → 浏览商品 → 加入购物车 → 支付 → 成功   │   │
│  │  - 用户注册 → 邮箱验证 → 完善资料 → 登录           │   │
│  │                                                       │   │
│  │  特点：                                              │   │
│  │  - 使用真实浏览器，模拟真实用户操作                 │   │
│  │  │   需要启动完整的开发服务器                       │   │
│  │  - 最接近生产环境，但也是最慢和最脆弱的              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、Jest：JavaScript最流行的测试框架

### 2.1 Jest是什么？

Jest是Facebook开发的JavaScript测试框架，它的特点是：
- **零配置**：开箱即用，不需要复杂配置
- **运行速度快**：支持并行执行和增量测试
- **内置覆盖率**：不需要额外安装覆盖率工具
- **快照测试**：轻松测试UI组件
- **Mock支持**：强大的函数模拟能力

### 2.2 Jest安装和配置


### 2.3 Jest基本语法

#### describe和test/it

```javascript
// describe 用于分组相关的测试用例
describe("Calculator 计算器", () => {
  // test 或 it 用来编写具体的测试用例
  // 两者完全等价，只是风格不同

  test("sum函数：1 + 2 应该等于 3", () => {
    // expect 用来断言
    // toBe 是匹配器（matcher），表示相等
    expect(sum(1, 2)).toBe(3);
  });

  test("sum函数：负数相加", () => {
    expect(sum(-1, -2)).toBe(-3);
  });

  test("sum函数：大数相加", () => {
    expect(sum(999999999, 1)).toBe(1000000000);
  });

  // it 也可以使用箭头函数
  it("should multiply two numbers correctly", () => {
    expect(multiply(3, 4)).toBe(12);
  });
});
```

#### 常用匹配器

```javascript
describe("常用匹配器", () => {
  // 1. 相等匹配
  test("toBe - 严格相等（===）", () => {
    expect(1 + 1).toBe(2);
    expect("hello").toBe("hello");
    expect({ name: "张三" }).not.toBe({ name: "张三" }); // 对象比较地址
  });

  test("toEqual - 深度相等", () => {
    // 比较对象的值，而不是引用
    expect({ name: "张三", age: 18 }).toEqual({ name: "张三", age: 18 });
    expect([1, 2, 3]).toEqual([1, 2, 3]);
  });

  // 2. 真假匹配
  test("toBeTruthy / toBeFalsy", () => {
    expect(true).toBeTruthy();
    expect(false).toBeFalsy();
    expect(1).toBeTruthy(); // 1 也是 truthy
    expect(0).toBeFalsy();   // 0 是 falsy
    expect("").toBeFalsy();  // 空字符串是 falsy
    expect(null).toBeFalsy();
    expect(undefined).toBeFalsy();
  });

  test("toBeNull / toBeUndefined / toBeDefined", () => {
    expect(null).toBeNull();
    expect(undefined).toBeUndefined();
    expect(1).toBeDefined();
  });

  // 3. 数字匹配
  test("数字比较", () => {
    expect(10).toBe(10);
    expect(10).toEqual(10);
    expect(10).toBeGreaterThan(5);    // 大于
    expect(10).toBeLessThan(20);     // 小于
    expect(10).toBeGreaterThanOrEqual(10); // 大于等于
    expect(10).toBeLessThanOrEqual(10);   // 小于等于
    expect(0.1 + 0.2).toBeCloseTo(0.3); // 浮点数比较
  });

  // 4. 字符串匹配
  test("toMatch - 正则匹配", () => {
    expect("Hello World").toMatch(/Hello/);
    expect("Hello World").toMatch(/^Hello/);     // 以Hello开头
    expect("Hello World").toMatch(/World$/);     // 以World结尾
    expect("2024-01-01").toMatch(/^\d{4}-\d{2}-\d{2}$/); // 日期格式
  });

  test("toContain - 包含", () => {
    expect("Hello World").toContain("World");
    expect([1, 2, 3]).toContain(2);
    expect(["苹果", "香蕉", "橙子"]).toContain("香蕉");
  });

  // 5. 数组匹配
  test("toHaveLength - 检查长度", () => {
    expect([1, 2, 3]).toHaveLength(3);
    expect("Hello").toHaveLength(5);
  });

  test("toContainEqual - 数组包含特定元素", () => {
    const objects = [{ id: 1 }, { id: 2 }, { id: 3 }];
    expect(objects).toContainEqual({ id: 2 });
  });

  // 6. 对象匹配
  test("toHaveProperty - 检查属性", () => {
    const user = {
      name: "张三",
      age: 18,
      address: {
        city: "北京",
      },
    };

    expect(user).toHaveProperty("name");
    expect(user).toHaveProperty("age", 18);
    expect(user).toHaveProperty("address.city", "北京");
  });

  test("toMatchObject - 对象匹配（只检查部分属性）", () => {
    const user = { name: "张三", age: 18, gender: "男" };

    // 只检查存在的属性
    expect(user).toMatchObject({ name: "张三", age: 18 });
  });

  // 7. 异常匹配
  test("toThrow - 函数抛出异常", () => {
    // 方式1：只检查是否抛出异常
    expect(() => {
      throw new Error("出错了！");
    }).toThrow();

    // 方式2：检查异常消息
    expect(() => {
      throw new Error("出错了！");
    }).toThrow("出错了！");

    // 方式3：使用正则匹配
    expect(() => {
      throw new Error("Error: something went wrong");
    }).toThrow(/something went wrong/);
  });
});
```

### 2.4 Mock函数

Mock函数用于"模拟"函数行为，常用于：
- 模拟API调用
- 模拟第三方模块
- 隔离依赖

```javascript
describe("Mock函数", () => {
  // 1. 基本Mock
  test("模拟一个函数", () => {
    // 创建mock函数
    const mockCallback = jest.fn();

    // 设置mock返回值
    mockCallback.mockReturnValue(42);

    // 调用mock函数
    const result = mockCallback();

    expect(result).toBe(42);
    expect(mockCallback).toHaveBeenCalled(); // 被调用过
    expect(mockCallback).toHaveBeenCalledTimes(1); // 被调用1次
  });

  // 2. 模拟异步函数
  test("模拟API调用", async () => {
    const mockFetch = jest.fn();

    // 模拟返回Promise
    mockFetch.mockResolvedValue({ data: "success" });

    // 使用mock
    const result = await mockFetch("/api/user");

    expect(result).toEqual({ data: "success" });
    expect(mockFetch).toHaveBeenCalledWith("/api/user"); // 检查参数
  });

  // 3. 模拟模块
  test("模拟axios模块", () => {
    // jest.mock() 会自动提升到文件顶部
    jest.mock("axios");

    const axios = require("axios");

    // 设置模拟返回值
    axios.get.mockResolvedValue({ data: { name: "张三" } });

    // 调用函数
    const user = await getUser(1);

    expect(user.name).toBe("张三");
  });

  // 4. 模拟定时器
  test("模拟setTimeout", () => {
    jest.useFakeTimers();

    const callback = jest.fn();

    // 设置定时器
    setTimeout(callback, 1000);

    // 快进1秒
    jest.advanceTimersByTime(1000);

    // 验证被调用
    expect(callback).toHaveBeenCalled();

    // 恢复真实定时器
    jest.useRealTimers();
  });

  // 5. 监视函数调用
  test("监视函数调用参数", () => {
    const mockFn = jest.fn();

    mockFn(1);
    mockFn(2);
    mockFn(3);

    // 获取所有调用参数
    expect(mockFn.mock.calls).toEqual([[1], [2], [3]]);

    // 获取第一次调用的参数
    expect(mockFn.mock.calls[0]).toEqual([1]);

    // 获取返回值
    mockFn.mockReturnValue(42);
    mockFn(1);
    expect(mockFn.mock.results[1].value).toBe(42);
  });
});
```

## 三、React组件测试

### 3.1 Testing Library：更接近真实用户的测试

Testing Library的核心理念是：**"测试不应该像测试实现细节，而应该像用户使用一样"**。

```javascript
// 不推荐：测试实现细节
test("点击按钮时，count状态加1", () => {
  const wrapper = shallow(<Counter />);
  wrapper.find("button").simulate("click");
  expect(wrapper.state("count")).toBe(1);
});

// 推荐：测试用户能看到的行为
test("点击按钮后，显示的数字加1", async () => {
  render(<Counter />);

  // 获取按钮并点击
  const button = screen.getByRole("button", { name: /增加/i });
  await userEvent.click(button);

  // 验证结果显示
  expect(screen.getByText("1")).toBeInTheDocument();
});
```

### 3.3 组件交互测试

```javascript
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { LoginForm } from "./LoginForm";

describe("LoginForm 登录表单", () => {
  test("输入用户名和密码后点击登录", async () => {
    // 创建一个userEvent实例（推荐方式）
    const user = userEvent.setup();

    render(<LoginForm onSubmit={mockSubmit} />);

    // 1. 输入用户名
    const usernameInput = screen.getByLabelText("用户名");
    await user.type(usernameInput, "admin");

    // 2. 输入密码
    const passwordInput = screen.getByLabelText("密码");
    await user.type(passwordInput, "123456");

    // 3. 点击登录按钮
    const submitButton = screen.getByRole("button", { name: "登录" });
    await user.click(submitButton);

    // 4. 验证表单提交
    expect(mockSubmit).toHaveBeenCalledWith({
      username: "admin",
      password: "123456",
    });
  });

  test("输入为空时禁用登录按钮", () => {
    render(<LoginForm onSubmit={mockSubmit} />);

    const submitButton = screen.getByRole("button", { name: "登录" });

    // 初始状态，按钮应该禁用
    expect(submitButton).toBeDisabled();

    // 只输入用户名
    const usernameInput = screen.getByLabelText("用户名");
    userEvent.type(usernameInput, "admin");

    // 仍然禁用
    expect(submitButton).toBeDisabled();
  });

  test("显示错误消息", async () => {
    // 模拟API返回错误
    mockSubmit.mockResolvedValueOnce({
      success: false,
      error: "用户名或密码错误",
    });

    render(<LoginForm onSubmit={mockSubmit} />);

    // 触发提交
    const submitButton = screen.getByRole("button", { name: "登录" });
    await userEvent.click(submitButton);

    // 等待错误消息出现
    expect(await screen.findByText("用户名或密码错误")).toBeInTheDocument();
  });
});
```

## 四、集成测试

### 4.1 什么是集成测试？

集成测试是介于单元测试和E2E测试之间的测试，它测试多个组件或模块一起工作的效果：

```
┌─────────────────────────────────────────────────────────────┐
│                    集成测试 vs 单元测试                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  单元测试：                                                  │
│  ┌─────────┐                                                │
│  │ 组件A   │  ← 只测试组件A，完全隔离外部依赖                │
│  └─────────┘                                                │
│                                                             │
│  集成测试：                                                  │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐                      │
│  │ 组件A   │→│ 组件B   │→│ 组件C   │                      │
│  └─────────┘ └─────────┘ └─────────┘                      │
│       ↑            ↑            ↑                            │
│       └────────────┴────────────┘                          │
│                    ↓                                        │
│              测试它们的协作                                  │
│                                                             │
│  E2E测试：                                                   │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐         │
│  │ 前端    │→│ API      │→│ 后端    │→│ 数据库   │         │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘         │
│   真实浏览器    真实服务器      真实数据库                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 API集成测试

```javascript

import { render, screen, waitFor } from "@testing-library/react";
import { rest } from "msw"; // Mock Service Worker
import { setupServer } from "msw/node";
import { UserList } from "./UserList";

// 模拟API服务器
const server = setupServer(
  // 模拟 GET /api/users
  rest.get("/api/users", (req, res, ctx) => {
    return res(
      ctx.status(200),
      ctx.json([
        { id: 1, name: "张三", email: "zhangsan@example.com" },
        { id: 2, name: "李四", email: "lisi@example.com" },
      ])
    );
  }),

  // 模拟 GET /api/users/:id
  rest.get("/api/users/:id", (req, res, ctx) => {
    const { id } = req.params;
    return res(
      ctx.status(200),
      ctx.json({ id, name: "张三", email: "zhangsan@example.com" })
    );
  }),

  // 模拟错误情况
  rest.get("/api/users-error", (req, res, ctx) => {
    return res(ctx.status(500), ctx.json({ message: "服务器错误" }));
  })
);

// 测试开始前启动服务器
beforeAll(() => server.listen());

// 每个测试后重置处理函数
afterEach(() => server.resetHandlers());

// 测试结束后关闭服务器
afterAll(() => server.close());

describe("UserList 组件 - API集成测试", () => {
  test("成功加载用户列表", async () => {
    render(<UserList />);

    // 等待用户列表加载
    await waitFor(() => {
      expect(screen.getByText("张三")).toBeInTheDocument();
    });

    // 验证两个用户都显示了
    expect(screen.getByText("李四")).toBeInTheDocument();
  });

  test("加载失败时显示错误消息", async () => {
    // 使用error handler覆盖默认的handler
    server.use(
      rest.get("/api/users", (req, res, ctx) => {
        return res.once(ctx.status(500), ctx.json({ message: "加载失败" }));
      })
    );

    render(<UserList />);

    await waitFor(() => {
      expect(screen.getByText("加载失败")).toBeInTheDocument();
    });
  });
});
```

#### 页面对象模型（Page Object Model）

```typescript
// e2e/pages/LoginPage.ts
// 页面对象模型：封装页面元素和操作

import { Page, Locator, expect } from "@playwright/test";

export class LoginPage {
  // 页面元素
  readonly page: Page;
  readonly usernameInput: Locator;
  readonly passwordInput: Locator;
  readonly submitButton: Locator;
  readonly errorMessage: Locator;

  constructor(page: Page) {
    this.page = page;
    this.usernameInput = page.getByLabel("用户名");
    this.passwordInput = page.getByLabel("密码");
    this.submitButton = page.getByRole("button", { name: "登录" });
    this.errorMessage = page.getByRole("alert");
  }

  // 操作方法
  async goto() {
    await this.page.goto("/login");
  }

  async login(username: string, password: string) {
    await this.usernameInput.fill(username);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
  }

  async expectError(message: string) {
    await expect(this.errorMessage).toContainText(message);
  }

  async expectSuccess() {
    await this.page.waitForURL("/");
    await expect(this.page.getByText("欢迎回来")).toBeVisible();
  }
}

// e2e/pages/DashboardPage.ts
export class DashboardPage {
  readonly page: Page;
  readonly userName: Locator;
  readonly logoutButton: Locator;

  constructor(page: Page) {
    this.page = page;
    this.userName = page.getByTestId("user-name");
    this.logoutButton = page.getByRole("button", { name: "退出登录" });
  }

  async expectUserName(name: string) {
    await expect(this.userName).toHaveText(name);
  }

  async logout() {
    await this.logoutButton.click();
    await this.page.waitForURL("/login");
  }
}

// e2e/example.spec.ts
import { test } from "@playwright/test";
import { LoginPage } from "../pages/LoginPage";

  // 执行登录

  // 验证结果

});
```

```
│  4. 测试应该是确定性的                                      │
│  ─────────────────────────────────────────────────────────  │
│  不依赖随机数、网络状态、时间等不确定因素                     │
│                                                             │
│  5. 好的测试命名                                            │
│  ─────────────────────────────────────────────────────────  │
│  测试名称应该描述"什么情况"和"预期什么"                     │
│  "sum: 当输入负数时，应该返回正确的负数结果"                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.5 TDD开发流程

测试驱动开发（TDD）是一种开发方法：

```
┌─────────────────────────────────────────────────────────────┐
│                    TDD 开发流程                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────┐                                              │
│   │ 写一个  │  ← 先写一个会失败的测试                       │
│   │  失败的测试 │                                           │
│   └────┬────┘                                              │
│        │                                                   │
│        ▼                                                   │
│   ┌─────────┐                                              │
│   │ 运行测试 │  ← 验证测试确实失败                         │
│   │ 确认失败 │                                              │
│   └────┬────┘                                              │
│        │                                                   │
│        ▼                                                   │
│   ┌─────────┐                                              │
│   │ 写代码  │  ← 只写让测试通过的最小代码                  │
│   │ 通过测试 │                                              │
│   └────┬────┘                                              │
│        │                                                   │
│        ▼                                                   │
│   ┌─────────┐                                              │
│   │ 重构    │  ← 优化代码，保持测试通过                    │
│   │ 改进设计 │                                              │
│   └────┬────┘                                              │
│        │                                                   │
│        └──────────────→ 循环                               │
│                                                             │
│   红 → 绿 → 重构                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.1 GitHub Actions测试配置

```yaml
# .github/workflows/test.yml

name: Test

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  # 单元测试和集成测试
  test:
    name: Unit & Integration Tests
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Run lint
        run: npm run lint

      - name: Run unit tests
        run: npm run test:unit

      - name: Run integration tests
        run: npm run test:integration

      - name: Upload coverage
        uses: codecov/codecov-action@v3
        with:
          files: ./coverage/lcov.info
          fail_ci_if_error: true

  # E2E测试
  e2e:
    name: E2E Tests
    runs-on: ubuntu-latest
    needs: test  # 等待单元测试完成

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Install Playwright Browsers
        run: npx playwright install --with-deps

      - name: Run E2E tests
        run: npm run test:e2e

      - name: Upload test results
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 14
```

## 八、总结：构建完整的测试体系

### 8.1 测试矩阵

| 测试类型 | 工具 | 速度 | 隔离性 | 数量建议 |
|----------|------|------|--------|----------|
| **单元测试** | Jest | 极快 | 高 | 100+ |
| **集成测试** | Jest + msw | 快 | 中 | 20-50 |
| **组件测试** | Testing Library | 快 | 中 | 30+ |
| **E2E测试** | Playwright | 慢 | 低 | 10-20 |

### 8.2 推荐的测试组合

```
小型项目（< 1000行代码）：
├── 单元测试：70%
├── 集成测试：20%
└── E2E测试：10%

中型项目（1000-10000行代码）：
├── 单元测试：60%
├── 集成测试：25%
└── E2E测试：15%

大型项目（> 10000行代码）：
├── 单元测试：50%
├── 集成测试：30%
└── E2E测试：20%
```

### 8.3 常见测试问题

**Q: 测试太慢怎么办？**
A: 1) 使用 Jest 的 --onlyChanged 增量测试；2) 使用 SWC 或 esbuild 加速转换；3) 将 E2E 测试移到 CI 的非阻塞任务中。

**Q: 覆盖率要求太高达不到？**
A: 1) 逐步提高阈值（从30%开始）；2) 优先测试关键路径；3) 使用覆盖率的增量检查（新代码必须有测试）。

**Q: E2E测试不稳定怎么办？**
A: 1) 使用 Playwright 的自动等待；2) 添加适当的重试机制；3) 避免依赖测试执行顺序；4) 使用稳定的定位器。

**Q: 什么时候该Mock，什么时候不该？**
A: 原则：1) 真实环境快速且确定时，不用Mock；2) 调用外部API时，应该Mock；3) 数据库操作可以用内存数据库Mock；4) 第三方库整体Mock。
