# 前端工程化完全指南

## 概述

本文档是前端工程化的系统性教学指南，涵盖从模块化开发到CI/CD部署的全流程工程化知识体系。通过学习本文档，你将掌握现代前端工程化的核心概念、工具链和最佳实践，能够构建高效、可维护、可扩展的前端项目。

**前置知识**：JavaScript基础、命令行操作、版本控制概念

**学习目标**：建立完整的工程化思维，掌握主流工具链使用，理解工程化背后的设计哲学

## 一、前端工程化概述

### 1.1 什么是工程化

前端工程化是指将软件工程的方法和理念应用到前端开发过程中，通过规范化、模块化、自动化等手段，提升前端开发效率、代码质量和可维护性的一系列最佳实践和工具链。

工程化不是某个具体的工具或框架，而是一种**思维方式**和**方法论**。它关注的是如何让前端开发更加高效、可控、可持续。

从历史发展的角度看，前端工程化经历了以下几个阶段：

**第一阶段：刀耕火种时代**
早期的前端开发非常简单，HTML、CSS、JavaScript文件直接用script标签串起来，没有模块化、没有构建工具、没有自动化测试。这种方式在小项目中可以工作，但随着项目规模增长，代码维护变得极其困难。不同开发者的代码风格不一致，依赖管理混乱，全局变量命名冲突频发，重复代码难以复用。

**第二阶段：萌芽期**
随着jQuery等库的出现，前端开始有了代码组织的概念。模块化加载器如RequireJS、SeaJS解决了JavaScript模块加载的问题。但这个阶段的模块化方案大多是运行时加载，性能较差，且各方案互不兼容。

**第三阶段：构建时代**
Node.js的出现让JavaScript进入了服务端领域，也为前端带来了npm生态系统。Webpack、Gulp、Grunt等构建工具相继诞生，前端开始有了完整的构建流程。代码可以被压缩、合并、转换，资源可以被优化处理。

**第四阶段：现代化时代**
ES Modules成为标准，Vite、esbuild、Rolldown等新一代构建工具大幅提升了开发体验。TypeScript成为大型项目的首选，React、Vue、Angular等框架推动了组件化开发。Monorepo、CI/CD、微前端等架构方案日趋成熟。

**第五阶段：智能化时代**
AI辅助编程工具如Copilot、Cursor改变了代码编写方式。自动化测试覆盖率要求越来越高，SonarQube等工具实现了代码质量的自动化监控。前端工程化进入了智能化的新阶段。

### 1.2 工程化的价值

工程化对于前端团队和项目来说具有不可替代的价值。理解这些价值有助于我们在实际工作中更好地推动工程化建设。

**效率提升的价值**是最直观的。良好的工程化体系可以让开发效率提升数倍。以一个中型前端项目为例，没有工程化时，开发者需要手动处理文件依赖、手动压缩代码、手动刷新浏览器。建立了工程化体系后，这些重复性工作全部自动化，开发者的精力可以完全集中在业务逻辑上。热模块替换让代码修改即时生效，智能的类型提示减少了排查错误的时间，自动化测试让重构不再恐惧。

**质量保障的价值**体现在多个层面。ESLint和Prettier保证了代码风格一致，新成员加入后可以快速融入团队节奏。自动化测试在每次提交前发现潜在问题，避免了将bug带到生产环境。SonarQube等工具持续监控代码质量，技术债务得以可视化。Code Review流程让知识在团队中流动，代码审查者的经验成为团队共同的经验。

**协作优化的价值**在多人项目中尤为明显。没有工程化时，不同开发者可能使用不同的编辑器、不同的工具版本、不同的代码风格。工程化通过统一配置解决了这些问题。Git Hooks确保提交信息规范，分支策略清晰。Monorepo架构让多个项目共享代码，版本管理更加统一。CI/CD流水线让部署标准化，人为错误大大减少。

**可维护性的价值**决定了项目的长期健康。工程化建立了清晰的代码结构和规范，新增功能时开发者知道应该在哪里添加代码。组件库和工具库的复用减少了重复代码，维护工作量大为降低。文档自动化降低了新成员的学习成本。构建产物的优化让用户获得更好的体验。

### 1.3 工程化成熟度

前端工程化不是一蹴而就的，而是一个循序渐进的过程。不同规模的团队和项目处于不同的成熟度阶段。

**Level 1：基础阶段**
这个阶段的团队刚刚开始工程化的尝试。他们可能使用了npm scripts作为构建脚本，勉强实现了代码压缩和合并。没有自动化测试，代码质量靠人工检查。部署大多是手动操作，直接上传文件到服务器。这个阶段的痛点很明显：开发效率低、bug多、部署风险大。

**Level 2：规范化阶段**
团队开始重视代码规范，引入了ESLint和Prettier。Git Flow或类似的工作流开始执行，提交信息有了基本格式。基本的单元测试开始编写，虽然覆盖率可能不高。开始使用一些自动化工具，如自动刷新、热更新。这个阶段开始享受工程化的好处，团队协作更加顺畅。

**Level 3：系统化阶段**
完整的CI/CD流水线建立起来，代码提交后自动运行测试和构建。开始引入TypeScript，类型安全成为团队追求的目标。组件库开始建设，跨项目复用成为可能。性能优化被系统性地执行，Core Web Vitals成为关注指标。Monorepo架构可能被采用，多项目协调更加高效。

**Level 4：平台化阶段**
工程化能力成为平台输出，内部工具和脚手架开始建设。数据驱动的决策成为常态，代码质量、构建速度、性能指标都有数据支撑。微前端架构可能开始实施，大型应用被拆分为可独立开发的子应用。安全被系统性地考虑，依赖审计成为常规流程。

**Level 5：智能化阶段**
AI辅助编程深度集成到开发流程中。代码质量检测智能化，能自动发现潜在的漏洞和性能问题。自动化测试生成开始尝试。工程化指标成为团队绩效的重要参考。团队开始向业界输出工程化最佳实践。

### 1.4 我的思考：工程化是效率基础

在多年的前端开发实践中，我深刻体会到工程化对于团队效率的决定性影响。一个工程化成熟的项目，开发体验流畅如丝；一个工程化薄弱的项目，开发过程痛苦不堪。

工程化建设的投入产出比极高。前期花一周时间配置好的ESLint规则，可以在接下来的一年里每天节省开发者的时间，并避免无数次因为格式不一致导致的代码冲突。搭建一个组件库需要几个月，但之后每个新功能开发都能复用这些组件，效率提升是持续的。

工程化也有其代价。学习曲线是存在的，特别是对于新手来说，理解为什么需要这样做需要时间。配置工作有时很繁琐，特别是当需要支持特殊需求时。过度工程化也是陷阱，追求完美的配置而忽视了实际业务需求是本末倒置的。

我认为工程化的最佳状态是**恰到好处**。对于小型个人项目，不需要Monorepo，不需要复杂的CI/CD。对于大型团队项目，需要完整的测试覆盖、严格的质量门禁、自动化的流水线。根据项目规模和团队情况，选择合适的工程化程度，才是真正的工程化思维。

## 二、模块化开发

### 2.1 ES Modules

ES Modules（以下简称ESM）是ECMAScript官方推出的模块系统，从ES6开始成为JavaScript的标准。ESM采用静态导入导出的方式，这使得编译器可以在构建时进行优化，如Tree Shaking。

ESM的核心特性包括：静态语法允许编译器分析依赖关系；导入导出必须位于模块顶层，不能在条件语句中；导入的绑定是只读的引用；模块代码在严格模式下执行；每个模块只执行一次。

**基本导出**有两种形式：命名导出和默认导出。命名导出使用export关键字，可以导出常量、函数、类。默认导出每个模块只能有一个，使用export default。

```javascript
// 命名导出：导出多个
export const PI = 3.14159;
export function add(a, b) {
    return a + b;
}
export class Calculator {
    multiply(a, b) {
        return a * b;
    }
}

// 默认导出：一个模块一个
export default function main() {
    return 'Hello, ES Modules';
}
```

**基本导入**同样有两种形式。与命名导出对应的是命名导入，可以使用花括号指定要导入的内容。与默认导出对应的是默认导入，直接导入默认值。

```javascript
// 命名导入
import { PI, add, Calculator } from './math.js';
console.log(PI);
console.log(add(1, 2));

// 默认导入
import main from './main.js';
console.log(main());

// 导入所有内容到命名空间
import * as math from './math.js';
console.log(math.PI);

// 导入时重命名
import { add as addNumbers } from './math.js';
```

**动态导入**是ESM的重要补充，使用import()函数返回一个Promise。这允许在运行时根据条件加载模块，对于代码分割和懒加载非常有用。

```javascript
// 动态导入示例
async function loadMathModule() {
    try {
        const math = await import('./math.js');
        console.log(math.add(1, 2));
    } catch (error) {
        console.error('模块加载失败:', error);
    }
}

// 结合路由的懒加载示例
const routes = [
    {
        path: '/dashboard',
        component: () => import('./pages/Dashboard.jsx')
    },
    {
        path: '/settings',
        component: () => import('./pages/Settings.jsx')
    }
];
```

**ESM在浏览器中的使用**需要script标签的type属性设置为module。

```html
<script type="module" src="./app.js"></script>
<!-- 内联模块脚本 -->
<script type="module">
    import { add } from './utils.js';
    console.log(add(1, 2));
</script>
```

ESM模块无论在浏览器还是Node.js中，都有以下特点：模块是单例的，多次导入获取的是同一个实例；导入绑定是只读的，对导入门绑定的修改会反映到导出一侧；循环依赖是支持的，但需要小心处理。

### 2.2 CommonJS

CommonJS（以下简称CJS）是Node.js传统的模块系统，使用require()同步加载模块。了解CJS对于处理Node.js代码和某些遗留模块是必要的。

**模块导出**使用module.exports或exports。module.exports是导出的主体，exports是其引用。当直接赋值给module.exports时，exports就不再指向它了。

```javascript
// 方式一：直接赋值给module.exports
module.exports = {
    add: function(a, b) {
        return a + b;
    },
    multiply: function(a, b) {
        return a * b;
    }
};

// 方式二：给exports添加属性
exports.add = function(a, b) {
    return a + b;
};
exports.multiply = function(a, b) {
    return a * b;
};

// 方式三：导出单个函数（常见）
module.exports = function(a, b) {
    return a + b;
};
```

**模块导入**使用require()函数。导入的是module.exports导出的对象或值。

```javascript
// 导入整个模块
const math = require('./math');
console.log(math.add(1, 2));

// 解构导入
const { add, multiply } = require('./math');

// 导入时重命名
const { add: addNumbers } = require('./math');
```

**模块解析**遵循一定的规则。Node.js会检查模块路径是否以./、../或/开头来确定是相对模块还是绝对模块。内置模块如fs、path会直接返回。第三方模块会从node_modules目录中查找。

```javascript
// 相对路径导入
const utils = require('./utils');
const helper = require('../shared/helper');

// 绝对路径导入
const config = require('/etc/app/config');

// 内置模块导入
const fs = require('fs');
const path = require('path');

// 第三方模块导入
const express = require('express');
const axios = require('axios');
```

**CJS与ESM的主要区别**需要特别注意。语法上，CJS使用require()和module.exports，ESM使用import和export。加载方式上，CJS是同步加载，ESM支持异步动态导入。解析时间上，CJS在运行时解析，ESM在编译时静态解析。this指向不同，CJS中this指向module.exports，ESM中this是undefined（严格模式）。变量提升方面，CJS允许在require之后定义变量，ESM必须先导入再使用。

**混合使用**在项目中很常见。Node.js支持通过.mjs和.cjs扩展名或package.json中的type字段来指定模块类型。ESM可以导入CJS模块，但只能导入其default export。CJS不能直接导入ESM模块，需要使用动态import()。

### 2.3 模块规范

模块规范定义了如何组织、导出、导入和使用代码。理解各种规范有助于在项目中做出正确的选择。

**AMD（Asynchronous Module Definition）**是早期的异步模块规范，主要用于浏览器环境。RequireJS是最著名的AMD实现。

```javascript
// AMD规范示例（RequireJS）
define(['jquery', './utils'], function($, utils) {
    // 依赖被注入到参数中
    function doSomething() {
        utils.log('doing something');
    }

    return {
        doSomething: doSomething
    };
});
```

**CMD（Common Module Definition）**是国内发展出的模块规范，与AMD类似但更接近CommonJS的写法。SeaJS是CMD的代表实现。

```javascript
// CMD规范示例（SeaJS）
define(function(require, exports, module) {
    var $ = require('jquery');
    var utils = require('./utils');

    exports.doSomething = function() {
        utils.log('doing something');
    };
});
```

**UMD（Universal Module Definition）**是一种兼容多种模块规范的模式，使得模块可以在AMD、CJS和全局变量环境下运行。

```javascript
// UMD规范示例
(function(root, factory) {
    // 检测环境
    if (typeof define === 'function' && define.amd) {
        // AMD环境
        define(['jquery', 'underscore'], factory);
    } else if (typeof module === 'object' && module.exports) {
        // Node.js环境
        module.exports = factory(require('jquery'), require('underscore'));
    } else {
        // 浏览器全局变量环境
        root.MyModule = factory(root.jQuery, root._);
    }
}(typeof self !== 'undefined' ? self : this, function($, _) {
    // 模块工厂函数
    function myModule() {
        // 模块实现
    }

    return myModule;
}));
```

**模块规范的选择**需要根据项目情况决定。现代前端项目建议使用ESM，它是官方标准且得到浏览器的原生支持。对于Node.js项目，如果只需要支持Node.js环境可以使用CJS，如果有跨环境需求可以提供ESM和CJS双格式或使用打包工具转换。

### 2.4 Tree Shaking

Tree Shaking是指在构建过程中移除未使用的代码（dead code）的优化技术。这个名称来源于摇晃树木让枯叶掉落，形象地描述了移除无用代码的过程。

Tree Shaking有效的前提是模块系统必须是静态的。ESM的import/export是静态语法，编译器可以在不执行代码的情况下分析导入导出关系。这与CommonJS的动态require()形成对比，后者只有在运行时才能确定加载的模块，无法进行静态分析。

**Tree Shaking的原理**可以分为几个步骤。首先是标记阶段，构建工具分析模块依赖图，标记所有导出和导入的使用情况。然后是压缩阶段，使用压缩器（如Terser）移除未被使用的导出。最后是优化阶段，简化代码结构，进一步减少体积。

```javascript
// math.js
export function add(a, b) {
    return a + b;
}

export function multiply(a, b) {
    return a * b;
}

export function divide(a, b) {
    return a / b;
}

// app.js
import { add } from './math.js';
console.log(add(1, 2));

// Tree Shaking后，multiply和divide会被移除
```

**sideEffects标记**是Tree Shaking的重要配置项。在package.json中声明sideEffects可以告诉构建工具哪些文件有副作用。有副作用的文件不能被移除，即使它们导出的内容没有被使用。

```json
{
    "name": "my-library",
    "sideEffects": [
        "./src/polyfills.js",
        "*.css"
    ]
}
```

如果所有模块都没有副作用，可以将sideEffects设置为false，这样构建工具可以安全地移除任何未被使用的代码。

```json
{
    "name": "my-library",
    "sideEffects": false
}
```

**在Vite中配置Tree Shaking**是现代项目的主流做法。Vite基于Rolldown实现Tree Shaking，默认情况下会进行优化。可以在vite.config.js中进行配置。

```javascript
// vite.config.js
export default defineConfig({
    build: {
        // 启用或禁用Tree Shaking
        treeShaking: true,
        // 设置为true时，会将CSS中的未使用选择器也移除
        cssCodeSplit: true,
        // 控制rollup选项
        rollupOptions: {
            output: {
                // 手动启用或禁用Tree Shaking
                treeshake: {
                    // 是否在注释中保留删除的代码
                    comments: false,
                    // 尝试安全地删除其他内容
                    pureFuncNames: true
                }
            }
        }
    }
});
```

**在Webpack中配置Tree Shaking**使用optimization配置项。

```javascript
// webpack.config.js
module.exports = {
    mode: 'production', // 生产模式自动启用Tree Shaking
    optimization: {
        usedExports: true, // 标记使用的导出
        sideEffects: true, // 启用副作用处理
        concatenateModules: true, // 作用域合并，提升Tree Shaking效果
        minimizer: [
            new TerserPlugin({
                terserOptions: {
                    compress: {
                        dead_code: true, // 删除死代码
                        drop_console: true, // 删除console语句
                        drop_debugger: true // 删除debugger语句
                    }
                }
            })
        ]
    }
};
```

**Tree Shaking的局限性**需要注意。动态导入的模块不会被Tree Shaking，因为编译器无法确定哪些动态导入会被使用。跨模块边界的函数调用，如果函数是通过this或其他间接方式调用的，可能无法被正确识别。某些打包工具的实现可能存在bug，导致本应被移除的代码被保留。

### 2.5 实战：模块化实践

通过一个完整的示例来实践模块化的各种概念。

**项目结构设计**采用标准的分层结构。

```
src/
├── modules/              # 功能模块
│   ├── user/             # 用户模块
│   │   ├── index.js      # 模块入口
│   │   ├── userService.js    # 用户服务
│   │   ├── userModel.js      # 用户数据模型
│   │   └── userStore.js      # 用户状态管理
│   └── order/            # 订单模块
│       ├── index.js
│       ├── orderService.js
│       └── orderModel.js
├── utils/                # 工具函数
│   ├── index.js
│   ├── format.js         # 格式化工具
│   ├── validation.js     # 验证工具
│   └── request.js        # 网络请求
├── components/           # 组件
│   └── ...
├── styles/               # 样式
│   └── ...
└── main.js               # 入口文件
```

**模块入口文件**使用统一的导出模式。

```javascript
// modules/user/index.js
// 用户模块的入口文件，统一导出模块内所有内容

// 导入各个子模块
import { UserService } from './userService.js';
import { UserModel, UserRole } from './userModel.js';
import { createUserStore } from './userStore.js';

// 创建单例store实例
const userStore = createUserStore();
```

```javascript
// utils/format.js
// 格式化工具函数模块

/**
 * 格式化日期
 * @param {Date|string|number} date - 日期
 * @param {string} format - 格式字符串，默认'YYYY-MM-DD HH:mm:ss'
 * @returns {string} 格式化后的日期字符串
 */
export function formatDate(date, format = 'YYYY-MM-DD HH:mm:ss') {
    const d = new Date(date);

    const year = d.getFullYear();
    const month = String(d.getMonth() + 1).padStart(2, '0');
    const day = String(d.getDate()).padStart(2, '0');
    const hours = String(d.getHours()).padStart(2, '0');
    const minutes = String(d.getMinutes()).padStart(2, '0');
    const seconds = String(d.getSeconds()).padStart(2, '0');

    return format
        .replace('YYYY', year)
        .replace('MM', month)
        .replace('DD', day)
        .replace('HH', hours)
        .replace('mm', minutes)
        .replace('ss', seconds);
}

/**
 * 格式化文件大小
 * @param {number} bytes - 字节数
 * @returns {string} 格式化后的大小字符串
 */
export function formatFileSize(bytes) {
    if (bytes === 0) return '0 B';

    const units = ['B', 'KB', 'MB', 'GB', 'TB'];
    const k = 1024;
    const i = Math.floor(Math.log(bytes) / Math.log(k));

    return parseFloat((bytes / Math.pow(k, i)).toFixed(2)) + ' ' + units[i];
}

/**
 * 格式化手机号码
 * @param {string} phone - 手机号码
 * @returns {string} 格式化后的手机号
 */
export function formatPhone(phone) {
    if (!phone) return '';
    const cleaned = phone.replace(/\D/g, '');
    if (cleaned.length === 11) {
        return `${cleaned.slice(0, 3)}-${cleaned.slice(3, 7)}-${cleaned.slice(7)}`;
    }
    return phone;
}

/**
 * 数字千分位格式化
 * @param {number} num - 数字
 * @returns {string} 格式化后的字符串
 */
export function formatNumber(num) {
    if (num === null || num === undefined) return '';
    return num.toString().replace(/\B(?=(\d{3})+(?!\d))/g, ',');
}

/**
 * 字符串截断
 * @param {string} str - 原始字符串
 * @param {number} maxLength - 最大长度
 * @param {string} suffix - 截断后缀，默认'...'
 * @returns {string} 截断后的字符串
 */
export function truncate(str, maxLength, suffix = '...') {
    if (!str || str.length <= maxLength) return str;
    return str.slice(0, maxLength - suffix.length) + suffix;
}
```

### 3.1 原子设计

原子设计（Atomic Design）是由Brad Frost提出的一种组件设计方法论，将界面拆分为从最小单元到最大结构的五个层级。这种方法帮助我们系统性地构建可复用、可维护的组件体系。

**原子（Atoms）**是界面的最小单元，不可再分。原子包括HTML基础元素如按钮、输入框、标签，也包括更抽象的概念如图标、配色方案。

```jsx
// 按钮原子组件
// components/atoms/Button/Button.jsx

 * 按钮组件 - 原子级别

    type = 'button',

    const variantClass = `btn--${variant}`;
    const sizeClass = `btn--${size}`;
```

**分子（Molecules）**是由多个原子组合形成的简单UI单元。分子具有明确的功能性，是界面中的功能组件。

```jsx
// 搜索框分子组件
// components/molecules/SearchBox/SearchBox.jsx
import React, { useState } from 'react';
import { Button } from '../../atoms/Button/Button.jsx';
import { Input } from '../../atoms/Input/Input.jsx';
import { SearchIcon, ClearIcon } from '../../atoms/Icons/Icons.jsx';
import './SearchBox.css';

/**
 * 搜索框分子组件
 * 由输入框、搜索按钮、清除按钮组成
 */
export function SearchBox({
    placeholder = '搜索...',
    onSearch,
    onChange,
    value: externalValue,
    defaultValue = '',
    loading = false,
    size = 'medium',
    className = '',
    ...props
}) {
    // 支持受控和非受控模式
    const [internalValue, setInternalValue] = useState(defaultValue);
    const isControlled = externalValue !== undefined;
    const value = isControlled ? externalValue : internalValue;

    const handleChange = (e) => {
        if (!isControlled) {
            setInternalValue(e.target.value);
        }
        onChange?.(e.target.value);
    };

    const handleClear = () => {
        if (!isControlled) {
            setInternalValue('');
        }
        onChange?.('');
    };

    const handleSearch = () => {
        onSearch?.(value);
    };

    const handleKeyDown = (e) => {
        if (e.key === 'Enter') {
            handleSearch();
        }
    };

    return (
        <div className={`search-box search-box--${size} ${className}`} {...props}>
            <Input
                value={value}
                onChange={handleChange}
                onKeyDown={handleKeyDown}
                placeholder={placeholder}
                size={size}
                className="search-box__input"
            />
            {value && !loading && (
                <button
                    type="button"
                    className="search-box__clear"
                    onClick={handleClear}
                    aria-label="清除搜索"
                >
                    <ClearIcon />
                </button>
            )}
            <Button
                variant="primary"
                size={size}
                onClick={handleSearch}
                loading={loading}
                icon={<SearchIcon />}
            >
                搜索
            </Button>
        </div>
    );
}
```

### 3.2 组件拆分

组件拆分是组件化开发的核心技能。合理的拆分可以让组件更易复用、更易测试、更易维护。

**拆分原则**需要考虑多个维度。单一职责原则要求每个组件只做一件事。DRY原则（Don't Repeat Yourself）提醒我们当代码重复出现两次以上时应该考虑抽象。可复用性原则要求组件不依赖特定业务数据。组合优于继承原则建议使用组合而非继承来构建复杂组件。

**拆分粒度**是实践中常见的问题。太粗的粒度导致组件臃肿难以维护，太细的粒度导致组件过多难以管理。合适的粒度是让一个组件能在3-5行代码内描述清楚它的作用。

```jsx

// components/ButtonLoader/ButtonLoader.jsx
// 实际使用时需要组合多个小组件，很繁琐

// components/UserDashboard/UserDashboard.jsx - 一个组件包含整个用户仪表盘
```

**事件总线**适用于没有直接父子关系的组件间的通信。

```jsx
// 事件总线实现
// utils/eventBus.js
class EventBus {
    constructor() {
        // 存储事件监听器
        this.listeners = new Map();
    }

    /**
     * 订阅事件
     * @param {string} event - 事件名
     * @param {Function} callback - 回调函数
     * @returns {Function} 取消订阅的函数
     */
    on(event, callback) {
        if (!this.listeners.has(event)) {
            this.listeners.set(event, new Set());
        }
        this.listeners.get(event).add(callback);

        // 返回取消订阅函数
        return () => this.off(event, callback);
    }

    /**
     * 发送事件
     * @param {string} event - 事件名
     * @param {*} data - 事件数据
     */
    emit(event, data) {
        const callbacks = this.listeners.get(event);
        if (callbacks) {
            callbacks.forEach(callback => callback(data));
        }
    }

    /**
     * 取消订阅
     * @param {string} event - 事件名
     * @param {Function} callback - 回调函数
     */
    off(event, callback) {
        const callbacks = this.listeners.get(event);
        if (callbacks) {
            callbacks.delete(callback);
        }
    }

    /**
     * 只订阅一次
     * @param {string} event - 事件名
     * @param {Function} callback - 回调函数
     */
    once(event, callback) {
        const wrappedCallback = (data) => {
            callback(data);
            this.off(event, wrappedCallback);
        };
        this.on(event, wrappedCallback);
    }
}

// 创建单例
export const eventBus = new EventBus();

// 使用示例
// ComponentA - 发布事件
function ComponentA() {
    const handleClick = () => {
        eventBus.emit('user:login', { userId: 123, name: '张三' });
    };

    return <button onClick={handleClick}>登录</button>;
}

// ComponentB - 订阅事件
function ComponentB() {
    useEffect(() => {
        const unsubscribe = eventBus.on('user:login', (data) => {
            console.log('用户登录:', data);
        });

        return unsubscribe;
    }, []);

    return <div>Component B</div>;
}
```

```jsx
function usePagination({ total, pageSize = 10, initialPage = 1 }) {
    const [currentPage, setCurrentPage] = useState(initialPage);

    const totalPages = Math.ceil(total / pageSize);

    const goToPage = (page) => {
        const validPage = Math.max(1, Math.min(page, totalPages));
        setCurrentPage(validPage);
    };

    const goNext = () => goToPage(currentPage + 1);
    const goPrev = () => goToPage(currentPage - 1);
    const goFirst = () => goToPage(1);
    const goLast = () => goToPage(totalPages);

    const canGoPrev = currentPage > 1;
    const canGoNext = currentPage < totalPages;

    const offset = (currentPage - 1) * pageSize;

    const pageNumbers = [];
    for (let i = 1; i <= totalPages; i++) {
        pageNumbers.push(i);
    }

    return {
        currentPage,
        totalPages,
        pageSize,
        offset,
        canGoPrev,
        canGoNext,
        goToPage,
        goNext,
        goPrev,
        goFirst,
        goLast,
        pageNumbers,
    };

// hooks/useLocalStorage.js
function useLocalStorage(key, initialValue) {
    const [storedValue, setStoredValue] = useState(() => {
        try {
            const item = window.localStorage.getItem(key);
            return item ? JSON.parse(item) : initialValue;
        } catch (error) {
            return initialValue;
        }
    });

    const setValue = (value) => {
        try {
            const valueToStore = value instanceof Function
                ? value(storedValue)
                : value;
            setStoredValue(valueToStore);
            window.localStorage.setItem(key, JSON.stringify(valueToStore));
        } catch (error) {
            console.error('Error saving to localStorage:', error);
        }
    };

    return [storedValue, setValue];
}

// hooks/useMediaQuery.js
function useMediaQuery(query) {
    const [matches, setMatches] = useState(() => {
        if (typeof window !== 'undefined') {
            return window.matchMedia(query).matches;
        }
        return false;
    });

    useEffect(() => {
        const mediaQuery = window.matchMedia(query);
        const handler = (event) => setMatches(event.matches);

        mediaQuery.addEventListener('change', handler);
        return () => mediaQuery.removeEventListener('change', handler);
    }, [query]);

    return matches;
}
```

```jsx
// organisms/DataTable/DataTable.jsx
/**
 * 数据表格组件
 * 支持排序、筛选、分页、选择等功能的综合性表格
 */
import React, { useState, useMemo, useCallback } from 'react';
import { TableHead } from '../../molecules/TableHead/TableHead.jsx';
import { TableRow } from '../../molecules/TableRow/TableRow.jsx';
import { Pagination } from '../../molecules/Pagination/Pagination.jsx';
import { TableToolbar } from '../TableToolbar/TableToolbar.jsx';
import { EmptyState } from '../../atoms/EmptyState/EmptyState.jsx';
import { LoadingOverlay } from '../../atoms/LoadingOverlay/LoadingOverlay.jsx';
import './DataTable.css';

/**
 * 列定义
 * @typedef {Object} Column
 * @property {string} key - 列标识
 * @property {string} title - 列标题
 * @property {Function} render - 自定义渲染函数
 * @property {boolean} sortable - 是否可排序
 * @property {boolean} filterable - 是否可筛选
 * @property {string} width - 列宽度
 * @property {string} align - 对齐方式
 */

/**
 * DataTable组件属性
 * @param {Object[]} data - 表格数据
 * @param {Column[]} columns - 列定义
 * @param {boolean} loading - 加载状态
 * @param {boolean} selectable - 是否可选择
 * @param {Function} onSelectionChange - 选择变化回调
 * @param {Function} onSort - 排序变化回调
 * @param {Function} onFilter - 筛选变化回调
 * @param {Function} onPageChange - 页码变化回调
 * @param {Object} pagination - 分页配置
 * @param {number} pagination.page - 当前页码
 * @param {number} pagination.pageSize - 每页条数
 * @param {number} pagination.total - 总条数
 */
export function DataTable({
    data = [],
    columns = [],
    loading = false,
    selectable = false,
    selectedKeys = [],
    onSelectionChange,
    onSort,
    onFilter,
    onPageChange,
    pagination,
    emptyText = '暂无数据',
    className = '',
    ...props
}) {
    // 内部状态
    const [sortConfig, setSortConfig] = useState({ key: null, direction: 'asc' });
    const [filters, setFilters] = useState({});
    const [internalSelectedKeys, setInternalSelectedKeys] = useState([]);

    // 是否受控选择
    const isSelectionControlled = selectedKeys !== undefined;
    const currentSelectedKeys = isSelectionControlled ? selectedKeys : internalSelectedKeys;

    // 排序逻辑
    const sortedData = useMemo(() => {
        if (!sortConfig.key) return data;

        return [...data].sort((a, b) => {
            const aVal = a[sortConfig.key];
            const bVal = b[sortConfig.key];

            if (aVal === bVal) return 0;

            const comparison = aVal > bVal ? 1 : -1;
            return sortConfig.direction === 'asc' ? comparison : -comparison;
        });
    }, [data, sortConfig]);

    // 筛选逻辑
    const filteredData = useMemo(() => {
        return sortedData.filter(item => {
            return Object.entries(filters).every(([key, value]) => {
                if (!value) return true;
                const itemValue = item[key];
                if (Array.isArray(itemValue)) {
                    return itemValue.includes(value);
                }
                return String(itemValue).includes(String(value));
            });
        });
    }, [sortedData, filters]);

    // 处理排序
    const handleSort = useCallback((key) => {
        setSortConfig(prev => {
            const newDirection = prev.key === key && prev.direction === 'asc'
                ? 'desc'
                : 'asc';
            const newConfig = { key, direction: newDirection };

            onSort?.(newConfig);
            return newConfig;
        });
    }, [onSort]);

    // 处理筛选
    const handleFilter = useCallback((key, value) => {
        setFilters(prev => {
            const newFilters = { ...prev, [key]: value };
            onFilter?.(newFilters);
            return newFilters;
        });
    }, [onFilter]);

    // 处理选择
    const handleSelect = useCallback((key) => {
        let newSelectedKeys;
        if (currentSelectedKeys.includes(key)) {
            newSelectedKeys = currentSelectedKeys.filter(k => k !== key);
        } else {
            newSelectedKeys = [...currentSelectedKeys, key];
        }

        if (!isSelectionControlled) {
            setInternalSelectedKeys(newSelectedKeys);
        }
        onSelectionChange?.(newSelectedKeys);
    }, [currentSelectedKeys, isSelectionControlled, onSelectionChange]);

    // 处理全选
    const handleSelectAll = useCallback(() => {
        const allKeys = filteredData.map(item => item.id);
        const allSelected = allKeys.every(key => currentSelectedKeys.includes(key));

        let newSelectedKeys;
        if (allSelected) {
            newSelectedKeys = currentSelectedKeys.filter(key => !allKeys.includes(key));
        } else {
            newSelectedKeys = [...new Set([...currentSelectedKeys, ...allKeys])];
        }

        if (!isSelectionControlled) {
            setInternalSelectedKeys(newSelectedKeys);
        }
        onSelectionChange?.(newSelectedKeys);
    }, [filteredData, currentSelectedKeys, isSelectionControlled, onSelectionChange]);

    // 处理页码变化
    const handlePageChange = useCallback((page) => {
        onPageChange?.(page);
    }, [onPageChange]);

    // 渲染单元格
    const renderCell = useCallback((column, item) => {
        if (column.render) {
            return column.render(item[column.key], item);
        }
        return item[column.key];
    }, []);

    // 判断是否全选
    const isAllSelected = filteredData.length > 0 &&
        filteredData.every(item => currentSelectedKeys.includes(item.id));
    const isIndeterminate = !isAllSelected &&
        filteredData.some(item => currentSelectedKeys.includes(item.id));

    return (
        <div className={`data-table ${className}`} {...props}>
            <TableToolbar
                columns={columns}
                filters={filters}
                onFilter={handleFilter}
                selectedCount={currentSelectedKeys.length}
            />

            <div className="data-table__container">
                {loading && <LoadingOverlay />}

                <table className="data-table__table">
                    <thead>
                        <TableHead
                            columns={columns}
                            sortConfig={sortConfig}
                            onSort={handleSort}
                            selectable={selectable}
                            isAllSelected={isAllSelected}
                            isIndeterminate={isIndeterminate}
                            onSelectAll={handleSelectAll}
                        />
                    </thead>
                    <tbody>
                        {filteredData.length === 0 ? (
                            <tr>
                                <td colSpan={columns.length + (selectable ? 1 : 0)}>
                                    <EmptyState text={emptyText} />
                                </td>
                            </tr>
                        ) : (
                            filteredData.map(item => (
                                <TableRow
                                    key={item.id}
                                    item={item}
                                    columns={columns}
                                    selectable={selectable}
                                    isSelected={currentSelectedKeys.includes(item.id)}
                                    onSelect={() => handleSelect(item.id)}
                                    renderCell={renderCell}
                                />
                            ))
                        )}
                    </tbody>
                </table>
            </div>

            {pagination && (
                <Pagination
                    current={pagination.page}
                    pageSize={pagination.pageSize}
```

### 5.2 Testing Library

Testing Library是一套专注于测试实际用户体验的测试工具。它的核心理念是：测试越接近用户使用方式，测试就越有价值。

**查询方法优先级**。Testing Library提供了多种查询方法，从推荐到不推荐的顺序是：getByRole（最推荐）> getByLabelText > getByPlaceholderText > getByText > getByDisplayValue > getByTestId（最后使用）。

```typescript
import { render, screen } from '@testing-library/react';

// getByRole - 最推荐的方式
// 这是最接近用户认知的方式
render(<Button>提交</Button>);

// 按钮
screen.getByRole('button', { name: '提交' });

// 输入框
screen.getByRole('textbox', { name: '用户名' });

// 链接
screen.getByRole('link', { name: '了解更多' });

// 标题
screen.getByRole('heading', { name: '欢迎' });

// getByLabelText - 表单元素
render(
    <form>
        <label htmlFor="username">用户名</label>
        <input id="username" type="text" />
    </form>
);
screen.getByLabelText('用户名');

// getByText - 非交互式文本
render(<div>Hello World</div>);
screen.getByText('Hello World');

// getByTestId - 最后手段
render(<button data-testid="submit-btn">提交</button>);
screen.getByTestId('submit-btn');
```

```typescript
        this.passwordInput = page.getByLabel('密码');

        this.submitButton = page.getByRole('button', { name: '登录' });
```

**覆盖率指标说明**：

- **语句覆盖率（Statements）**：执行到的语句占总语句的比例
- **分支覆盖率（Branches）**：每个条件的每个分支是否都被执行
- **函数覆盖率（Functions）**：被调用的函数占总函数的比例
- **行覆盖率（Lines）**：执行到的代码行占总行数的比例

### 6.1 GitHub Actions

GitHub Actions是GitHub提供的CI/CD功能，允许自动化构建、测试、部署流程。

**基础概念**：

- **Workflow（工作流）**：自动化流程的定义
- **Job（任务）**：工作流中的执行单元
- **Step（步骤）**：任务中的具体操作
- **Action（动作）**：可复用的步骤单元
- **Runner**：执行工作流的服务器

**工作流配置文件**。

```yaml
# .github/workflows/ci.yml
name: CI/CD Pipeline

# 触发条件
on:
  # 在push和pull request时触发
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]
  # 定时触发
  schedule:
    - cron: '0 2 * * *'  # 每天凌晨2点
  # 手动触发
  workflow_dispatch:
    inputs:
      environment:
        description: '部署环境'
        required: true
        default: 'staging'

# 环境变量
env:
  NODE_VERSION: '20.x'

# 任务定义
jobs:
    # 检查任务
    lint:
        name: 代码检查
        runs-on: ubuntu-latest

        steps:
            - name: 检出代码
              uses: actions/checkout@v4

            - name: 设置Node.js
              uses: actions/setup-node@v4
              with:
                  node-version: ${{ env.NODE_VERSION }}
                  cache: 'npm'

            - name: 安装依赖
              run: npm ci

            - name: 运行ESLint
              run: npm run lint

            - name: 运行Prettier检查
              run: npm run format:check

    # 测试任务
    test:
        name: 单元测试
        needs: lint  # 依赖lint任务
        runs-on: ubuntu-latest

        services:
            postgres:
                image: postgres:15
                env:
                    POSTGRES_USER: test
                    POSTGRES_PASSWORD: test
                    POSTGRES_DB: test
                ports:
                    - 5432:5432
                options: >-
                    --health-cmd pg_isready
                    --health-interval 10s
                    --health-timeout 5s
                    --health-retries 5

            redis:
                image: redis:7
                ports:
                    - 6379:6379

        steps:
            - name: 检出代码
              uses: actions/checkout@v4

            - name: 设置Node.js
              uses: actions/setup-node@v4
              with:
                  node-version: ${{ env.NODE_VERSION }}
                  cache: 'npm'

            - name: 安装依赖
              run: npm ci

            - name: 运行测试
              run: npm run test:coverage
              env:
                  DATABASE_URL: postgresql://test:test@localhost:5432/test
                  REDIS_URL: redis://localhost:6379

            - name: 上传覆盖率报告
              uses: codecov/codecov-action@v4
              with:
                  files: ./coverage/lcov.info
                  fail_ci_if_error: true

    # E2E测试任务
    e2e:
        name: E2E测试
        needs: test
        runs-on: ubuntu-latest

        steps:
            - name: 检出代码
              uses: actions/checkout@v4

            - name: 设置Node.js
              uses: actions/setup-node@v4
              with:
                  node-version: ${{ env.NODE_VERSION }}
                  cache: 'npm'

            - name: 安装依赖
              run: npm ci

            - name: 安装Playwright浏览器
              run: npx playwright install --with-deps chromium

            - name: 运行E2E测试
              run: npm run test:e2e

            - name: 上传测试结果
              uses: actions/upload-artifact@v4
              if: always()
              with:
                  name: playwright-report
                  path: playwright-report/
                  retention-days: 30

    # 构建任务
    build:
        name: 构建
        needs: e2e
        runs-on: ubuntu-latest

        steps:
            - name: 检出代码
              uses: actions/checkout@v4

            - name: 设置Node.js
              uses: actions/setup-node@v4
              with:
                  node-version: ${{ env.NODE_VERSION }}
                  cache: 'npm'

            - name: 安装依赖
              run: npm ci

            - name: 构建项目
              run: npm run build
              env:
                  NEXT_PUBLIC_API_URL: ${{ secrets.NEXT_PUBLIC_API_URL }}

            - name: 上传构建产物
              uses: actions/upload-artifact@v4
              with:
                  name: dist
                  path: dist/
                  retention-days: 7

    # 部署任务
    deploy:
        name: 部署
        needs: build
        runs-on: ubuntu-latest
        if: github.ref == 'refs/heads/main'

        environment:
            name: production
            url: https://example.com

        steps:
            - name: 检出代码
              uses: actions/checkout@v4

            - name: 下载构建产物
              uses: actions/download-artifact@v4
              with:
                  name: dist
                  path: dist/

            - name: 部署到服务器
              uses: appleboy/ssh-action@v1
              with:
                  host: ${{ secrets.DEPLOY_HOST }}
                  username: ${{ secrets.DEPLOY_USER }}
                  key: ${{ secrets.DEPLOY_KEY }}
                  script: |
                      cd /var/www/app
                      npm ci --production
                      cp -r dist/* ./
                      pm2 restart app
```

```typescript
// 性能监控SDK
// utils/performance.ts
export class PerformanceMonitor {
    private marks: Map<string, number> = new Map();
    private measures: Array<{ name: string; duration: number }> = [];

    // 创建标记
    mark(label: string) {
        this.marks.set(label, performance.now());
    }

    // 创建测量
    measure(name: string, startMark: string, endMark?: string) {
        const startTime = this.marks.get(startMark);
        const endTime = endMark ? this.marks.get(endMark) : performance.now();

        if (startTime && endTime) {
            const duration = endTime - startTime;
            this.measures.push({ name, duration });
            console.log(`[Performance] ${name}: ${duration.toFixed(2)}ms`);
            return duration;
        }

        console.warn(`[Performance] Mark not found: ${startMark} or ${endMark}`);
        return null;
    }

    // 获取所有测量结果
    getMeasures() {
        return [...this.measures];
    }

    // 上报到服务器
    async report(url: string) {
        try {
            await fetch(url, {
                method: 'POST',
                headers: { 'Content-Type': 'application/json' },
                body: JSON.stringify({
                    measures: this.measures,
                    timestamp: Date.now(),
                    url: window.location.href,
                    userAgent: navigator.userAgent,
                }),
            });
        } catch (error) {
            console.error('[Performance] Report failed:', error);
        }
    }
}

export const perfMonitor = new PerformanceMonitor();
```

```javascript
/**
 * 计算两点之间的距离
 * @param {Object} point1 - 第一个点
 * @param {number} point1.x - 第一个点的X坐标
 * @param {number} point1.y - 第一个点的Y坐标
 * @param {Object} point2 - 第二个点
 * @param {number} point2.x - 第二个点的X坐标
 * @param {number} point2.y - 第二个点的Y坐标
 * @returns {number} 两点之间的距离
 * @throws {TypeError} 如果坐标不是数字
 * @example
 * // 返回 5
 * distance({ x: 0, y: 0 }, { x: 3, y: 4 });
 */
export function distance(point1, point2) {
    if (typeof point1.x !== 'number' || typeof point1.y !== 'number') {
        throw new TypeError('point1坐标必须是数字');
    }
    if (typeof point2.x !== 'number' || typeof point2.y !== 'number') {
        throw new TypeError('point2坐标必须是数字');
    }

    const dx = point2.x - point1.x;
    const dy = point2.y - point1.y;

    return Math.sqrt(dx * dx + dy * dy);
}

/**
 * 用户服务类
 * @class
 * @param {string} baseURL - API基础URL
 */
export class UserService {
    /**
     * @private
     */
    baseURL;

    /**
     * @readonly
     */
    static VERSION = '1.0.0';

    /**
     * 创建用户服务实例
     * @param {string} baseURL - API基础URL
     */
    constructor(baseURL) {
        this.baseURL = baseURL;
    }

    /**
     * 获取用户列表
     * @param {Object} options - 查询选项
     * @param {number} [options.page=1] - 页码
     * @param {number} [options.pageSize=20] - 每页数量
     * @param {string} [options.search] - 搜索关键词
     * @returns {Promise<{users: User[], total: number}>} 用户列表和总数
     */
    async getUsers({ page = 1, pageSize = 20, search } = {}) {
        const params = new URLSearchParams({
            page: String(page),
            pageSize: String(pageSize),
            ...(search && { search }),
        });

        const response = await fetch(`${this.baseURL}/users?${params}`);
        return response.json();
    }
}
```

### 11.1 XSS防护

```typescript
function escapeHtml(unsafe: string): string {
    return unsafe
        .replace(/&/g, '&amp;')
        .replace(/</g, '&lt;')
        .replace(/>/g, '&gt;')
        .replace(/"/g, '&quot;')

function sanitizeHtml(dirty: string): string {
```

### 11.2 CSRF防护

```typescript
res.cookie('session', token, {
    httpOnly: true,

// 3. 自定义请求头验证
const csrfHeaderName = 'x-csrf-token';

async function fetchWithCSRF(url, options = {}) {
    const token = getCSRFToken();

    return fetch(url, {
        ...options,
        headers: {
            ...options.headers,
            [csrfHeaderName]: token,
        },
        credentials: 'include',
    });
}

// 4. 双重提交Cookie
function DoubleSubmitCookie() {
    const verifyCSRF = async (req, res, next) => {
        const cookieToken = req.cookies['csrf-token'];
        const headerToken = req.headers['x-csrf-token'];

        if (!cookieToken || !headerToken) {
            return res.status(403).json({ error: 'CSRF验证失败' });
        }

        if (cookieToken !== headerToken) {
            return res.status(403).json({ error: 'CSRF token不匹配' });
        }

        next();
    };
}
```

```typescript
    return Object.entries(data).reduce((acc, [key, value]) => {
        if (sensitiveFields.some(field => key.toLowerCase().includes(field))) {
            acc[key] = '[REDACTED]';
        } else if (typeof value === 'object' && value !== null) {
            acc[key] = sanitizeForLog(value);
        } else {
            acc[key] = value;
        }
        return acc;
    }, {});
}

// 4. 密钥轮换

const keyVersions = {
    v1: 'old-secret-key',
    v2: 'current-secret-key',
};

function getSecretKey(version = 'v2') {
    return keyVersions[version];
```

```typescript
// lib/utils/debounce.ts
/**
 * 防抖函数
 * @param fn 要执行的函数
 * @param delay 延迟时间（毫秒）
 * @returns 防抖处理后的函数
 */
export function debounce<T extends (...args: any[]) => any>(
    fn: T,
    delay: number
): (...args: Parameters<T>) => void {
    let timeoutId: ReturnType<typeof setTimeout> | undefined;

    return function (this: any, ...args: Parameters<T>) {
        clearTimeout(timeoutId);

        timeoutId = setTimeout(() => {
            fn.apply(this, args);
        }, delay);
    };
}

/**
 * 节流函数
 * @param fn 要执行的函数
 * @param limit 限制时间（毫秒）
 * @returns 节流处理后的函数
 */
export function throttle<T extends (...args: any[]) => any>(
    fn: T,
    limit: number
): (...args: Parameters<T>) => void {
    let inThrottle: boolean;

    return function (this: any, ...args: Parameters<T>) {
        if (!inThrottle) {
            fn.apply(this, args);
            inThrottle = true;
            setTimeout(() => {
                inThrottle = false;
            }, limit);
        }
    };
}
```

### 12.5 我的思考：架构决定效率

前端架构设计是整个项目成功的关键基础。一个好的架构能够支撑业务的快速发展，一个糟糕的架构会成为团队效率的瓶颈。

**架构设计的重要性**体现在多个维度。首先是开发效率，良好的架构让开发者能够快速定位问题、理解代码、添加新功能。模块化的设计让团队可以并行开发不同模块，而不会相互干扰。组件化让UI开发可以高度复用，减少重复劳动。

其次是代码质量。清晰的架构本身就是一种约束，引导开发者写出结构良好的代码。分层架构让各层职责明确，便于编写针对性的测试。模块边界清晰让代码审查更容易进行。

第三是可维护性。好的架构让系统能够长期维护和演进。当需求变化时，良好的架构能够将变化局部化，不需要大规模修改代码。模块间的低耦合让修改一个模块不会影响其他模块。

**架构设计需要考虑的因素**包括：团队规模，大团队需要更严格的结构和规范；项目复杂度，复杂项目需要更完善的分层和模块化；业务特点，不同业务有不同的领域模型；技术选型，架构需要与选用的技术栈匹配；团队能力，架构需要考虑团队的学习曲线。

**架构不是一成不变的**。好的架构应该能够适应变化。随着项目的发展和业务的变化，架构也需要相应调整。过度设计和不足设计都需要避免。架构的目标是服务于业务，而不是追求完美。
