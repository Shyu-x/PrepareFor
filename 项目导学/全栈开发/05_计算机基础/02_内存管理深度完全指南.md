# 内存管理深度完全指南

## 前言：内存管理的艺术

想象一下，你经营一个巨大的仓库。这个仓库有一个特点：它被分成无数个小格子，每个格子都有固定的编号（地址）。你需要管理这个仓库，让不同的"租户"（程序）能够存放和取出他们的物品（数据），同时要确保：

1. 不同租户的物品不会混淆（隔离性）
2. 物品可以存放在任何空闲的格子里（灵活性）
3. 仓库的管理开销不能太大（效率）
4. 即使仓库满了，也要能想办法腾出空间（扩展性）

这就是操作系统内存管理的核心挑战。

---

## 第一章：物理内存与地址空间

### 1.1 什么是物理内存？

**物理内存（Physical Memory）**是计算机中真实的硬件内存条。你可以把物理内存想象成仓库本身——它是由一系列固定的存储单元组成的，每个单元都能存储一个字节（8位）的数据。

在 x86-64 架构的现代计算机中，通常会有：
- **DDR4/DDR5 内存条**：常见的容量有 8GB、16GB、32GB、64GB 等
- **物理地址总线**：通常是 48-52 位，能寻址 256TB-4PB 的地址空间
- **按字节寻址**：每个地址对应一个字节

### 1.2 地址空间的概念

**地址空间（Address Space）**是一个抽象概念。对于 CPU 来说，它只看到虚拟地址，而操作系统负责将虚拟地址转换为物理地址。

```
物理地址空间（硬件视角）
┌─────────────────────────────────────────────────────────────┐
│ 0x00000000                    │                    0xFFFFFFFF │
├──────────────────────────────┼───────────────────────────────┤
│        物理内存条             │         I/O设备等              │
└──────────────────────────────┴───────────────────────────────┘

虚拟地址空间（程序视角）
┌─────────────────────────────────────────────────────────────┐
│ 0x00000000                                                   │
├─────────────────────────────────────────────────────────────┤
│ 应用程序看到的是一个连续的、从0开始的地址空间                  │
│ 每个应用程序都以为自己拥有整个内存                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 为什么要用虚拟地址？

如果没有虚拟地址，会有以下问题：

| 问题 | 说明 | 解决方案 |
|------|------|----------|
| **地址冲突** | 两个程序可能都想用同样的地址 | 虚拟地址，每个程序有独立的地址空间 |
| **安全风险** | 程序可以直接访问物理内存 | 虚拟地址隔离，无法直接访问物理地址 |
| **内存碎片** | 物理内存可能到处都是小洞 | 虚拟地址连续，操作系统负责映射 |
| **.swap耦合** | 程序需要知道物理内存位置 | 虚拟地址抽象，swap对程序透明 |

## 第二章：虚拟内存

### 2.1 虚拟内存的工作原理

**虚拟内存（Virtual Memory）**是操作系统为每个进程提供的一种"假象"：让每个进程都以为自己拥有连续的、独立的地址空间。实际物理内存可能碎片化，但虚拟内存让程序看到的是连续的。

```
CPU 生成虚拟地址 (VA)
         │
         ▼
┌────────────────────┐
│   MMU (内存管理单元) │
└────────┬───────────┘
         │ 硬件转换
         ▼
物理地址 (PA) ──► 物理内存
```

### 2.2 页表（Page Table）

页表是虚拟地址到物理地址的映射表。CPU 的内存管理单元（MMU）会硬件加速这个查找过程。

```c
// 简化版的页表项结构
struct page_table_entry {
    // 页帧号：物理内存中页的编号
    unsigned long pfn : 55;

    // 有效位：该条目是否有效
    unsigned long valid : 1;

    // 访问权限：读/写/执行
    unsigned long read : 1;
    unsigned long write : 1;
    unsigned long execute : 1;

    // 脏位：页是否被修改过
    unsigned long dirty : 1;

    // 访问位：页是否被访问过（用于页面置换）
    unsigned long accessed : 1;

    // 其他标志
    unsigned long flags : 4;
};

// 地址转换过程（虚拟地址到物理地址）
// 假设：4KB页大小，48位虚拟地址，4级页表

/*
 * 虚拟地址结构 (48位):
 * [ PML4 (9位) | PDPT (9位) | PD (9位) | PT (9位) | Offset (12位) ]
 *
 * 转换过程:
 * 1. 从 CR3 寄存器获取 PML4 表的物理地址
 * 2. 用虚拟地址的 PML4 索引在 PML4 表中查找
 * 3. 获取 PDPT 表的物理地址
 * 4. 用虚拟地址的 PDPT 索引在 PDPT 表中查找
 * 5. 获取 PD 表的物理地址
 * 6. 用虚拟地址的 PD 索引在 PD 表中查找
 * 7. 获取 PT 表的物理地址
 * 8. 用虚拟地址的 PT 索引在 PT 表中查找
 * 9. 获取页帧号 (PFN)
 * 10. 用页帧号 × 页大小 + Offset = 物理地址
 */

virt_addr_t translate(virt_addr_t va) {
    // 1. 获取 PML4 表地址 (从 CR3 寄存器)
    paddr_t pml4_base = read_cr3();

    // 2. 提取各层索引
    int pml4_idx = (va >> 39) & 0x1FF;   // 位 39-47
    int pdpt_idx = (va >> 30) & 0x1FF;   // 位 30-38
    int pd_idx = (va >> 21) & 0x1FF;     // 位 21-29
    int pt_idx = (va >> 12) & 0x1FF;     // 位 12-20
    int offset = va & 0xFFF;              // 位 0-11

    // 3. 逐级查找页表
    pte_t* pml4 = phys_to_virt(pml4_base);
    pte_t pml4_entry = pml4[pml4_idx];

    if (!(pml4_entry.valid)) {
        // 页表不存在，触发缺页异常
        handle_page_fault(va);
    }

    pte_t* pdpt = phys_to_virt(pml4_entry.pfn << 12);
    pte_t pdpt_entry = pdpt[pdpt_idx];

    if (!(pdpt_entry.valid)) {
        handle_page_fault(va);
    }

    pte_t* pd = phys_to_virt(pdpt_entry.pfn << 12);
    pte_t pd_entry = pd[pd_idx];

    if (!(pd_entry.valid)) {
        handle_page_fault(va);
    }

    pte_t* pt = phys_to_virt(pd_entry.pfn << 12);
    pte_t pt_entry = pt[pt_idx];

    if (!(pt_entry.valid)) {
        handle_page_fault(va);
    }

    // 4. 组成物理地址
    paddr_t physical_addr = (pt_entry.pfn << 12) | offset;

    return physical_addr;
}
```

### 2.3 多级页表 vs 单级页表

为什么需要多级页表？想象一下 4GB 内存，4KB 页大小：

```
单级页表问题：
- 4GB / 4KB = 1,048,576 个页表项
- 每个页表项 8 字节 = 8MB 的页表
- 而且这 8MB 需要连续内存！

多级页表优势：
- 不需要的地址范围不需要顶级页表条目
- 每个子页表可以独立存在
- 大大节省内存
```

### 2.4 TLB（Translation Lookaside Buffer）

每次地址转换都要查页表，效率太低。TLB 是 CPU 内置的缓存，存放最近使用的虚拟地址到物理地址的映射。

```
TLB 命中流程 (Hit):
    VA → [TLB 查找] → PA → 物理内存
          ↑ 命中

TLB 未命中流程 (Miss):
    VA → [TLB 查找] → [页表查找] → PA → 物理内存
                   ↑ 未命中，需查页表
```

## 第三章：分页与分段

### 3.1 分页（Paging）

**分页**是将虚拟地址空间和物理地址空间都划分为固定大小的块。

| 概念 | 说明 | 常见大小 |
|------|------|----------|
| **页（Page）** | 虚拟地址空间的固定块 | 4KB、2MB、1GB |
| **页帧（Page Frame）** | 物理内存的固定块 | 与页大小相同 |
| **页表项（PTE）** | 映射关系记录 | 通常 8 字节 |

```
分页示意图

虚拟地址空间                    物理内存
┌──────────────┐               ┌──────────────┐
│ Page 0       │ ───────────► │ Page Frame 3  │
│ 0-4KB        │               │              │
├──────────────┤               ├──────────────┤
│ Page 1       │ ───X          │ Page Frame 7  │
│ 4KB-8KB      │  (未分配)      │              │
├──────────────┤               ├──────────────┤
│ Page 2       │ ───────────► │ Page Frame 1  │
│ 8KB-12KB     │               │              │
├──────────────┤               ├──────────────┤
│ Page 3       │ ───────────► │ Page Frame 5  │
│ 12KB-16KB    │               │              │
└──────────────┘               └──────────────┘
```

### 3.2 分段（Segmentation）

**分段**是将虚拟地址空间划分为大小不等的段：

```
代码段 ──► 存放程序代码
数据段 ──► 存放全局变量
堆段   ──► 动态分配的内存
栈段   ──► 函数调用、局部变量
```

### 3.3 段页式存储

现代系统通常结合分段和分页：

```
逻辑地址 → [段表] → 线性地址 → [页表] → 物理地址
```

```c
// 段表项结构
struct segment_descriptor {
    unsigned long base : 32;     // 段基址
    unsigned long flags : 8;     // 段标志
    unsigned long limit : 20;     // 段界限
};

// 段选择子结构
struct segment_selector {
    unsigned short index : 13;   // 段描述符索引
    unsigned short ti : 1;       // 0=GDT, 1=LDT
    unsigned short rpl : 2;      // 请求特权级
};

// 地址转换（段页式）
linear_addr_t segment_to_linear(struct segment_selector sel, offset_t offset) {
    // 1. 根据选择子获取段描述符
    struct segment_descriptor* desc;
    if (sel.ti == 0) {
        desc = &gdt[sel.index];  // 全局描述符表
    } else {
        desc = &ldt[sel.index];  // 本地描述符表
    }

    // 2. 检查界限
    if (offset > desc->limit) {
        raise_exception(GENERAL_PROTECTION_FAULT);
    }

    // 3. 计算线性地址
    return desc->base + offset;
}
```

## 第四章：内存映射（mmap）

### 4.1 mmap 的原理

**mmap** 是 Linux 中非常重要的系统调用，它将文件或设备映射到内存中，让程序可以像访问内存一样访问文件。

```c
// mmap 系统调用原型
#include <sys/mman.h>

void* mmap(void* addr,           // 建议的映射地址（通常为NULL）
           size_t length,         // 映射长度
           int prot,              // 保护标志 (PROT_READ/PROT_WRITE/PROT_EXEC)
           int flags,             // 映射类型 (MAP_SHARED/MAP_PRIVATE)
           int fd,                // 文件描述符
           off_t offset);         // 文件偏移量

// munmap 解除映射
int munmap(void* addr, size_t length);
```

### 4.2 mmap 的使用场景

#### 4.2.1 文件映射

```c
// 使用 mmap 读取文件（高效）
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/mman.h>
#include <sys/stat.h>
#include <fcntl.h>

int main() {
    // 打开文件
    int fd = open("test.txt", O_RDONLY);
    if (fd < 0) {
        perror("open");
        return 1;
    }

    // 获取文件大小
    struct stat st;
    if (fstat(fd, &st) < 0) {
        perror("fstat");
        return 1;
    }

    // 将文件映射到内存
    // NULL 让内核选择地址
    // st.st_size 映射整个文件
    // PROT_READ 文件只读
    // MAP_PRIVATE 私有映射（修改不会写回文件）
    char* content = mmap(NULL, st.st_size, PROT_READ, MAP_PRIVATE, fd, 0);

    if (content == MAP_FAILED) {
        perror("mmap");
        return 1;
    }

    // 现在可以像读内存一样读文件
    printf("文件内容: %.*s", (int)st.st_size, content);

    // 解除映射
    munmap(content, st.st_size);

    // 关闭文件
    close(fd);

    return 0;
}
```

#### 4.2.2 匿名映射（分配内存）

```c
// 使用 mmap 分配内存（代替 brk/sbrk）
#include <sys/mman.h>

void* allocate_anon_memory(size_t size) {
    // MAP_ANONYMOUS: 匿名映射，不关联任何文件
    // MAP_PRIVATE: 私有映射
    void* addr = mmap(NULL,
                      size,
                      PROT_READ | PROT_WRITE,
                      MAP_PRIVATE | MAP_ANONYMOUS,
                      -1,     // 无文件描述符
                      0);     // 无偏移

    if (addr == MAP_FAILED) {
        perror("mmap");
        return NULL;
    }

    return addr;
}

// 使用 munmap 释放内存
void free_anon_memory(void* addr, size_t size) {
    if (munmap(addr, size) < 0) {
        perror("munmap");
    }
}
```

### 4.4 零拷贝（Zero-Copy）

传统的文件传输需要多次数据拷贝：

```
传统方式：4次拷贝，2次上下文切换
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│  磁盘   │ ──► │ 内核缓冲区│ ──► │ 用户缓冲区│ ──► │ Socket缓冲区│ ──► 网络
└─────────┘     └─────────┘     └─────────┘     └─────────┘
      磁盘到内核     内核到用户      用户到内核     内核到网络

零拷贝：2次拷贝，0次上下文切换（使用 mmap + sendfile）
┌─────────┐     ┌─────────┐     ┌─────────┐
│  磁盘   │ ──► │ 内核缓冲区│ ──► │ Socket缓冲区│ ──► 网络
└─────────┘     └─────────┘     └─────────┘
      磁盘到内核     内核内部传输
```

```c
// 使用 sendfile 实现零拷贝
#include <sys/sendfile.h>

int zero_copy_sendfile(int out_fd, int in_fd, off_t* offset, size_t count) {
    // sendfile 在内核中完成数据传输
    // 文件描述符 in_fd 必须是套接字
    // out_fd 是目标文件描述符（通常是 socket）

    ssize_t bytes_sent = sendfile(out_fd, in_fd, offset, count);

    if (bytes_sent < 0) {
        perror("sendfile");
        return -1;
    }

    return bytes_sent;
}
```

## 第五章：页面置换算法

### 5.1 为什么需要页面置换？

当物理内存不够用时，操作系统需要把某些页"换出"到磁盘，腾出空间给需要的程序。这个过程就像仓库管理员把一些货物暂时搬到外面的仓库，腾出空间给新的货物。

### 5.2 页面置换算法

#### 5.2.1 LRU（最近最少使用）

**LRU** 是最常用的页面置换算法，选择最长时间没有被访问的页换出。

```c
// LRU 页置换的简化实现
struct lru_list {
    struct list_head head;      // 链表头（最久未使用的在前面）
    int num_pages;              // 页面数量
};

struct page {
    unsigned long vaddr;        // 虚拟地址
    unsigned long paddr;        // 物理地址
    int ref_bit;                // 参考位（硬件设置）
    struct list_head list;      // 链表节点
};

// 访问页面的处理
void access_page(struct page* p) {
    // 硬件会设置参考位
    p->ref_bit = 1;

    // 将页面移到链表末尾（最近使用的）
    list_del(&p->list);
    list_add_tail(&p->list, &lru_list.head);
}

// 页面置换
struct page* select_victim(void) {
    // 选择链表头部的页（最久未使用的）
    struct page* victim = list_entry(lru_list.head.next,
                                      struct page,
                                      list);

    // 如果是被修改过的页，需要写回磁盘
    if (victim->dirty) {
        write_page_to_disk(victim);
    }

    // 从内存中移除
    list_del(&victim->list);

    return victim;
}
```

#### 5.2.2 Clock 算法（第二次机会）

LRU 需要维护有序链表，开销大。Clock 算法是 LRU 的近似实现：

```
Clock 算法示意

指针 ──► ┌───┬───┬───┬───┬───┬───┬───┬───┐
        │ 1 │ 0 │ 1 │ 0 │ 1 │ 0 │ 1 │ 0 │
        └───┴───┴───┴───┴───┴───┴───┴───┘
              参考位

1. 指针指向页面A
2. 如果A的参考位是1，清零，继续
3. 如果A的参考位是0，换出A
4. 重复直到找到要换出的页
```

```c
// Clock 算法的实现
struct clock_data {
    unsigned char refs[NUM_PAGES];  // 参考位数组
    int hand;                       // 时钟指针位置
};

void clock_init(void) {
    for (int i = 0; i < NUM_PAGES; i++) {
        refs[i] = 0;  // 所有页的参考位清零
    }
    hand = 0;  // 指针从头开始
}

int clock_get_victim(void) {
    while (1) {
        // 检查当前指针位置的参考位
        if (refs[hand] == 1) {
            // 参考位是1，给它第二次机会，清零后继续
            refs[hand] = 0;
            hand = (hand + 1) % NUM_PAGES;
        } else {
            // 参考位是0，这个页就是被换出的候选
            int victim = hand;
            hand = (hand + 1) % NUM_PAGES;
            return victim;
        }
    }
}
```

#### 5.2.3 工作集（Working Set）算法

工作集是指进程当前正在使用的页面集合：

```c
// 工作集模型
struct working_set {
    unsigned long vm_area;       // 虚拟内存区域
    unsigned long referenced_pages[WS_SIZE];  // 最近引用的页
    unsigned long timestamps[WS_SIZE];         // 引用时间戳
    int size;                     // 当前工作集大小
};

// 定期扫描，更新工作集
void update_working_set(struct working_set* ws, unsigned long now) {
    unsigned long age_threshold = 100;  // 超过100个时间单位的页不在工作集

    for (int i = 0; i < ws->size; i++) {
        unsigned long age = now - ws->timestamps[i];

        if (age > age_threshold) {
            // 页太老了，从工作集移除
            remove_page_from_working_set(ws, i);
            i--;  // 调整索引
        }
    }
}

// 换出不在工作集的页
struct page* working_set_select_victim(void) {
    // 遍历所有页，选择不在工作集的页
    for (int i = 0; i < total_pages; i++) {
        if (!in_working_set(&pages[i])) {
            return &pages[i];
        }
    }
    // 如果所有页都在工作集，选择最老的
    return select_oldest_page();
}
```

### 5.3 页面置换的实际应用

```python
# Python 模拟 LRU 页面置换算法
from collections import OrderedDict

class LRUCache:
    """
    LRU 缓存实现
    这个原理和操作系统的页面置换是一样的
    """

    def __init__(self, capacity: int):
        # OrderedDict 会维护键值对的插入顺序
        self.cache = OrderedDict()
        self.capacity = capacity

    def access(self, page_num: int) -> bool:
        """
        访问一个页，如果不在缓存中返回 False（需要置换）
        """
        if page_num in self.cache:
            # 页已经在缓存中，移动到末尾（最近使用）
            self.cache.move_to_end(page_num)
            return True
        else:
            return False

    def insert(self, page_num: int) -> int:
        """
        插入一个新页，返回被置换出的页号（如果有）
        """
        victim = -1

        if page_num in self.cache:
            # 已经存在，更新位置
            self.cache.move_to_end(page_num)
        else:
            # 新页，检查是否需要置换
            if len(self.cache) >= self.capacity:
                # 缓存满了，移除最久未使用的（第一个）
                victim = next(iter(self.cache))
                del self.cache[victim]

            # 插入新页
            self.cache[page_num] = True

        return victim

    def display(self):
        """显示当前缓存状态"""
        print("缓存内容（从旧到新）:", list(self.cache.keys()))

def simulate_page_replacement():
    """
    模拟页面置换过程
    假设物理内存有3个页帧，访问序列：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2
    """
    frames = 3
    cache = LRUCache(frames)

    # 访问序列
    accesses = [7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2]

    hits = 0
    faults = 0

    print("页面置换模拟（LRU算法）")
    print(f"物理内存: {frames} 个页帧")
    print(f"访问序列: {accesses}")
    print("-" * 50)

    for access in accesses:
        if cache.access(access):
            # 命中（页已在内存）
            hits += 1
            status = "命中"
        else:
            # 未命中（需要置换）
            faults += 1
            victim = cache.insert(access)
            if victim >= 0:
                status = f"换入{access}，换出{victim}"
            else:
                status = f"换入{access}"

        print(f"访问页 {access:2d}: {status:20s}", end="")
        cache.display()

    print("-" * 50)
    print(f"命中次数: {hits}")
    print(f"缺页次数: {faults}")
    print(f"命中率: {hits/len(accesses)*100:.1f}%")

if __name__ == "__main__":
    simulate_page_replacement()
```

## 第六章：垃圾回收算法

### 6.1 什么是垃圾回收？

**垃圾回收（Garbage Collection，GC）**是自动管理内存的机制。程序员的"对象创建"由系统自动回收，不再需要手动释放内存。这避免了大部分内存泄漏和野指针问题。

就像一个自动清洁工，你用完东西（对象）后，清洁工会自动清理，不需要你自己扔垃圾。

### 6.2 引用计数（Reference Counting）

每个对象有个计数器，记录有多少引用指向它。计数器为0时，对象就是"垃圾"。

```c
// 引用计数的简单实现
typedef struct object {
    int ref_count;      // 引用计数
    void (*finalize)(struct object*);  // 析构函数
    // 对象数据...
} Object;

// 增加引用
void retain(Object* obj) {
    if (obj) {
        obj->ref_count++;
    }
}

// 释放引用
void release(Object* obj) {
    if (obj) {
        obj->ref_count--;
        if (obj->ref_count <= 0) {
            // 没有引用了，回收内存
            if (obj->finalize) {
                obj->finalize(obj);
            }
            free(obj);
        }
    }
}

// 创建新对象
Object* create_object(void) {
    Object* obj = malloc(sizeof(Object));
    obj->ref_count = 1;  // 创建时引用数为1
    obj->finalize = NULL;
    return obj;
}
```

**优点**：实时回收，没有暂停
**缺点**：无法处理循环引用

```
循环引用问题：

    A ──► B
    │     │
    └─────┘

A和B互相引用，引用计数都是1，但外部已经没有任何引用指向它们
引用计数算法无法回收它们
```

### 6.3 标记-清除（Mark and Sweep）

分两阶段：先标记所有可达对象，然后清除所有未标记对象。

```c
// 标记-清除垃圾回收器
typedef struct object_header {
    int marked;           // 标记位
    struct object_header* next;
    // 对象数据...
} ObjectHeader;

#define MAX_OBJECTS 10000
ObjectHeader* all_objects[MAX_OBJECTS];
int num_objects = 0;

// GC 根对象集合（全局变量、栈变量等）
ObjectHeader** gc_roots[MAX_ROOTS];
int num_roots = 0;

// 标记阶段：从根开始，递归标记所有可达对象
void mark_phase(void) {
    // 首先，取消所有对象的标记
    for (int i = 0; i < num_objects; i++) {
        all_objects[i]->marked = 0;
    }

    // 从根开始深度优先搜索
    for (int i = 0; i < num_roots; i++) {
        mark_object(gc_roots[i]);
    }
}

// 递归标记一个对象及其引用的对象
void mark_object(ObjectHeader* obj) {
    if (!obj || obj->marked) {
        return;  // 空指针或已标记
    }

    obj->marked = 1;  // 标记为可达

    // 递归标记这个对象引用的其他对象
    // 假设对象有一个引用数组
    ObjectHeader** refs = get_object_refs(obj);
    int num_refs = get_num_refs(obj);

    for (int i = 0; i < num_refs; i++) {
        mark_object(refs[i]);
    }
}

// 清除阶段：回收所有未标记的对象
void sweep_phase(void) {
    ObjectHeader** surviving = malloc(sizeof(ObjectHeader*) * MAX_OBJECTS);
    int surviving_count = 0;

    for (int i = 0; i < num_objects; i++) {
        ObjectHeader* obj = all_objects[i];

        if (!obj->marked) {
            // 未标记的对象是垃圾，回收
            if (obj->finalize) {
                obj->finalize(obj);
            }
            free(obj);
        } else {
            // 存活的对象保留
            surviving[surviving_count++] = obj;
        }
    }

    // 更新所有对象列表
    memcpy(all_objects, surviving, sizeof(ObjectHeader*) * surviving_count);
    num_objects = surviving_count;
    free(surviving);
}

// 运行一次 GC
void gc(void) {
    printf("开始垃圾回收... ");
    int before = num_objects;

    mark_phase();
    sweep_phase();

    printf("回收了 %d 个对象，还有 %d 个\n",
           before - num_objects, num_objects);
}
```

### 6.4 分代回收（Generational GC）

**核心观察**：大多数对象是"朝生夕死"的——创建后很快就不用了。

```python
# Python 的分代垃圾回收器模拟

class Generation:
    """
    分代回收将内存分为多个区域
    - 年轻代：新创建的对象
    - 老年代：存活时间长的对象

    对象的"年龄"由它经历的 GC 次数决定
    """

    def __init__(self, generation_id, max_size):
        self.generation_id = generation_id
        self.max_size = max_size
        self.objects = {}  # 对象ID -> 对象
        self.promotion_threshold = 2  # 晋升阈值

    def add_object(self, obj_id, obj):
        self.objects[obj_id] = {
            'data': obj,
            'age': 0  # 对象年龄
        }

    def get_live_objects(self):
        """返回所有存活对象"""
        return [info['data'] for info in self.objects.values()]

    def is_full(self):
        return len(self.objects) >= self.max_size

    def promote_objects(self, older_generation):
        """将存活时间长的对象晋升到老年代"""
        promoted = []
        for obj_id, info in list(self.objects.items()):
            if info['age'] >= self.promotion_threshold:
                promoted.append((obj_id, info['data']))
                del self.objects[obj_id]

        for obj_id, data in promoted:
            older_generation.add_object(obj_id, data)

        return len(promoted)

    def clear(self):
        """清除所有对象"""
        self.objects.clear()

class GenerationalGC:
    """
    分代垃圾回收器
    包含3个代：年轻代、中生代、老年代
    """

    def __init__(self):
        # 创建3个代
        self.generations = [
            Generation(0, 100),   # 年轻代，小而频繁回收
            Generation(1, 500),    # 中生代
            Generation(2, 2000),   # 老年代
        ]
        self.gc_count = 0

    def allocate(self, obj_id, obj):
        """分配新对象"""
        # 总是分配到年轻代（0代）
        young = self.generations[0]

        if young.is_full():
            self.collect_generation(0)

        young.add_object(obj_id, obj)

    def collect_generation(self, gen_id):
        """收集指定代"""
        self.gc_count += 1
        gen = self.generations[gen_id]

        print(f"\nGC #{self.gc_count}: 收集第 {gen_id} 代")
        print(f"  回收前: {len(gen.objects)} 个对象")

        # 对年轻代：活着的对象年龄+1或晋升
        if gen_id == 0:
            survivors = []
            for obj_id, info in list(gen.objects.items()):
                info['age'] += 1
                survivors.append((obj_id, info['data']))
            gen.objects.clear()

            # 将存活对象晋升到下一代
            if gen_id + 1 < len(self.generations):
                promoted = gen.promote_objects(self.generations[gen_id + 1])
                print(f"  晋升到第 {gen_id + 1} 代: {promoted} 个")

        # 对中生代和老年代：简单清除
        else:
            survivors = []
            for obj_id, info in list(gen.objects.items()):
                survivors.append((obj_id, info['data']))
            gen.objects.clear()

        # 重新加入存活对象
        for obj_id, data in survivors:
            gen.add_object(obj_id, data)

        print(f"  回收后: {len(gen.objects)} 个对象")

    def print_stats(self):
        """打印各代统计"""
        print("\n各代状态:")
        for i, gen in enumerate(self.generations):
            print(f"  第 {i} 代: {len(gen.objects)} 个对象")

def demo_generational_gc():
    """演示分代垃圾回收"""
    gc = GenerationalGC()

    # 模拟对象生命周期
    # 大多数对象很快变成垃圾，少数会存活很久

    object_id = 0

    print("=== 分代垃圾回收演示 ===\n")

    # 阶段1：大量创建短命对象
    print("阶段1: 创建大量短命对象")
    for i in range(50):
        gc.allocate(object_id, {'type': 'short_lived', 'data': i})
        object_id += 1

    gc.print_stats()

    # 阶段2：创建一些长命对象
    print("\n阶段2: 创建一些长命对象")
    long_lived_ids = []
    for i in range(10):
        gc.allocate(object_id, {'type': 'long_lived', 'data': i})
        long_lived_ids.append(object_id)
        object_id += 1

    gc.print_stats()

    # 阶段3：触发多次年轻代回收
    print("\n阶段3: 触发多次年轻代回收（短命对象应该被回收）")
    for i in range(30):
        gc.allocate(object_id, {'type': 'temp', 'data': i})
        object_id += 1

    gc.print_stats()

    # 检查长命对象是否还在
    print("\n长命对象状态:")
    young = gc.generations[0]
    middle = gc.generations[1]
    for lid in long_lived_ids:
        if lid in young.objects:
            print(f"  对象 {lid} 在年轻代")
        elif lid in middle.objects:
            print(f"  对象 {lid} 在中生代（已晋升）")

if __name__ == "__main__":
    demo_generational_gc()
```

```python
def demonstrate_cycle_reference():
    """演示循环引用"""
    print("\n=== 循环引用演示 ===")

    # 创建循环引用
    class Node:
        def __init__(self, value):
            self.value = value
            self.next = None
            print(f"  Node {value} 创建")

        def __del__(self):
            print(f"  Node {self.value} 被销毁")

    # 创建循环引用
    node1 = Node(1)
    node2 = Node(2)
    node1.next = node2
    node2.next = node1

    print("  形成循环引用: node1 -> node2 -> node1")

    # 删除外部引用
    del node1
    del node2
    print("  已删除外部引用，但对象未被销毁（循环引用）")

    # Python 的 GC 可以处理循环引用
    print("  手动触发垃圾回收...")
    gc.collect()
    print("  GC 完成后，对象被销毁")


def demonstrate_weakref():
    """演示弱引用（不增加引用计数）"""

def demonstrate_memory_leak():
    """演示内存泄漏"""
    print("\n=== 内存泄漏演示 ===")

    # 典型的内存泄漏：全局字典持续增长
    cache = {}

    def leaky_function(key, value):
        """
        这个函数有内存泄漏
        每次调用都往 cache 添加数据，但从不清理
        """
        # 正确做法：限制 cache 大小或使用 LRU cache
        cache[key] = value
        return value

    # 模拟不断调用
    for i in range(10000):
        leaky_function(f"key_{i}", f"value_{i}" * 100)

    print(f"  cache 大小: {len(cache)}")
    print(f"  字典占用的内存: {sys.getsizeof(cache)} 字节")
    print("  （正确做法：使用 functools.lru_cache 或定期清理）")
```

## 总结：内存管理的核心要点

1. **虚拟内存抽象**
   - 每个进程有独立的虚拟地址空间
   - MMU 硬件加速地址转换
   - TLB 缓存最近使用的映射

2. **分页与分段**
   - 分页：固定大小的内存块，简化管理
   - 分段：按逻辑功能划分，更符合程序视角
   - 现代系统：段页式结合

3. **页面置换算法**
   - LRU：选择最久未使用的页
   - Clock：LRU 的近似实现，更高效
   - 工作集：基于实际使用模式

4. **垃圾回收**
   - 引用计数：实时但无法处理循环引用
   - 标记-清除：完整但需要 STW
   - 分代回收：针对对象生命周期的优化
   - 三色标记：并发回收，减少停顿

5. **实战注意**
   - Node.js 使用 V8 引擎，分代 GC
   - Python 也有 GC，但要注意循环引用
   - 理解内存管理写出更高效的程序
