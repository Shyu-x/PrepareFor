# 计算机网络协议完全指南

## 概述

计算机网络协议是现代互联网通信的基石，无论是开发Web应用、设计分布式系统，还是保障网络安全，都离不开对网络协议的深入理解。本指南将从网络基础知识出发，全面讲解OSI七层模型、TCP/IP协议族、HTTP/HTTPS协议、TCP与UDP的机制，以及常见的网络安全问题和实战技巧。

本指南的目标是帮助读者建立完整的网络协议知识体系，不仅理解协议的工作原理，更要掌握其在实际开发中的应用场景。通过本指南的学习，你将能够熟练使用抓包工具分析网络通信，理解浏览器与服务器之间的交互细节，并编写出更加安全、高效的网络应用程序。

## 一、网络基础知识

### 1.1 OSI七层模型与TCP/IP四层模型

#### 1.1.1 OSI七层模型详解

OSI（Open Systems Interconnection Model，开放系统互连模型）是国际标准化组织（ISO）于1984年提出的网络通信标准模型。这个模型将网络通信划分为七个层次，每一层都有特定的功能和职责，层与层之间通过接口进行通信。OSI模型虽然是一个理论模型，但它为理解网络通信提供了清晰的框架，是学习网络协议的重要基础。

**物理层（Physical Layer）** 是OSI模型的最底层，负责在物理媒介上传输原始比特流。物理层关注的不是数据本身，而是如何将二进制数据转换为物理信号（如电信号、光信号、无线电波）并在物理媒介上传输。这一层涉及的因素包括电压、电流、光脉冲、射频等物理特性，以及网线、光纤、集线器等物理设备。物理层的主要任务包括比特同步、比特编码、传输速率控制、物理拓扑选择等。在实际开发中，物理层的问题通常表现为网线损坏、光纤弯曲过度、信号衰减等物理连接问题。

**数据链路层（Data Link Layer）** 建立在物理层之上，负责在相邻节点之间可靠地传输数据帧。数据链路层将网络层传来的IP数据报封装成帧（Frame），并添加必要的控制信息（如帧起始标志、源MAC地址、目标MAC地址、校验序列等）。这一层使用MAC地址（Media Access Control Address）来标识网络设备，每个网络接口卡（NIC）都有唯一的MAC地址。数据链路层的主要功能包括帧封装、帧同步、物理地址寻址、差错检测、流量控制等。常见的数据链路层协议有以太网（Ethernet）、PPP（Point-to-Point Protocol）、HDLC（High-Level Data Link Control）等。在交换机和网桥等设备中，数据链路层负责实现MAC地址学习与转发。

**网络层（Network Layer）** 是OSI模型中最重要的层次之一，负责在不同网络之间路由和转发数据包。网络层的核心功能是实现端到端的数据传输，包括IP寻址、路由选择、分段与重组、拥塞控制等。网络层使用IP地址来标识网络中的设备，IP地址分为网络部分和主机部分，通过子网掩码来确定网络边界。常见的网络层协议包括IP（IPv4/IPv6）、ICMP（Internet Control Message Protocol）、ARP（Address Resolution Protocol）、RARP（Reverse Address Resolution Protocol）等。路由器是工作在网络层的设备，它根据路由表和路由算法选择最佳路径将数据包从源主机转发到目标主机。网络层还负责处理不同网络之间的地址转换和协议转换问题。

**传输层（Transport Layer）** 提供了端到端的通信服务，是进程间通信的基础。传输层使用端口号来区分同一台主机上的不同应用进程，端口号是一个16位的数字，范围从0到65535，其中0到1023是知名端口（如HTTP使用80端口，HTTPS使用443端口）。传输层的主要协议有TCP（Transmission Control Protocol）和UDP（User Datagram Protocol）。TCP是面向连接的、可靠的传输协议，提供字节流服务、拥塞控制、流量控制等特性；UDP则是无连接的、不可靠的传输协议，提供简单的数据报服务。传输层还负责错误检测、顺序控制、连接管理等功能。在操作系统中，传输层的功能由传输控制块（TCB）来管理，每个TCP连接都有一个对应的TCB。

**会话层（Session Layer）** 负责建立、管理和终止应用程序之间的会话。会话是指两个应用进程之间的一次完整的对话过程，会话层提供了对话控制、同步、检查点管理等功能。在实际的网络协议中，会话层的功能通常被嵌入到应用层协议中，例如HTTP/1.1中的Keep-Alive机制、WebSocket的持久连接等。会话层的主要协议有NetBIOS、RPC（Remote Procedure Call）等。在现代网络应用中，会话管理更多是通过应用层协议（如HTTP Cookie、Session）来实现的。

**表示层（Presentation Layer）** 负责处理数据的格式化和转换，确保一个系统发送的数据能够被另一个系统正确理解。表示层主要处理数据格式的差异，例如字符编码转换、数据压缩、数据加密解密等。常见的表示层协议和技术包括ASCII、EBCDIC、JPEG、MPEG、GZIP、SSL/TLS等。表示层的工作确保了不同系统之间的数据互操作性，使得异构系统能够进行有效通信。在Web开发中，表示层的问题常常体现在字符编码不一致导致的乱码问题上。

**应用层（Application Layer）** 是OSI模型的最高层，也是直接为用户提供服务的层次。应用层包含了各种网络应用协议，这些协议定义了应用程序之间如何交换数据。常见的应用层协议包括HTTP（超文本传输协议）、FTP（文件传输协议）、SMTP（简单邮件传输协议）、DNS（域名系统）、SSH（安全外壳协议）、Telnet等。应用层协议是用户与网络之间的接口，它直接为用户的网络应用提供服务。在开发Web应用时，我们主要工作在应用层，使用HTTP、HTTPS、WebSocket等协议与客户端进行通信。

#### 1.1.2 TCP/IP四层模型

TCP/IP模型是事实上的网络通信标准，它将网络通信划分为四个层次，与OSI七层模型有明确的对应关系。TCP/IP模型更简洁实用，反映了实际网络协议的实现方式，因此在实际工作中更常被使用。TCP/IP模型的四层分别是网络接口层（或称链路层）、网络层（或称互联网层）、传输层和应用层。

**网络接口层（Network Interface Layer）** 对应OSI模型的物理层和数据链路层，负责在物理网络上发送和接收数据。这一层包含了以太网、PPP、帧中继等协议，处理MAC地址寻址、帧封装、帧校验等功能。在网络接口层，数据被封装成数据帧（Frame），通过物理网络进行传输。

**网络层（Internet Layer）** 对应OSI模型的网络层，是TCP/IP模型的核心层。这一层主要使用IP协议，负责将数据报从源主机路由到目标主机。IP协议是一种不可靠的、无连接的协议，它不保证数据报能够成功到达目的地，也不保证数据报的顺序。IP协议的主要功能包括寻址（使用IP地址）、路由选择、分段与重组。与OSI网络层一样，TCP/IP网络层也使用路由器来实现不同网络之间的数据转发。

**传输层（Transport Layer）** 与OSI传输层完全对应，主要使用TCP和UDP协议。TCP提供面向连接的可靠传输服务，通过三次握手建立连接、四次挥手关闭连接，采用序号、确认、重传、流量控制、拥塞控制等机制保证数据传输的可靠性。UDP提供无连接的不可靠传输服务，不保证数据到达和顺序，适用于对实时性要求高、对可靠性要求相对较低的应用场景。

**应用层（Application Layer）** 对应OSI模型的应用层、表示层和会话层。TCP/IP模型的应用层包含了所有高层协议，如HTTP、HTTPS、FTP、SMTP、POP3、DNS、SSH等。在Web开发中，我们主要使用HTTP和HTTPS协议与客户端进行通信。应用层协议直接为用户的应用程序提供服务，是用户与网络之间的接口。

#### 1.1.3 两种模型的对比

OSI七层模型和TCP/IP四层模型的主要区别在于层次划分的方式和适用范围。OSI模型是一种理论模型，层次划分更加细致，每一层都有明确的功能定义，但层次较多，理解和使用起来相对复杂。TCP/IP模型是实际使用的模型，层次较少，更加简洁实用，但某些层次的边界不够清晰。在实际学习和工作中，我们通常结合两种模型的优点，使用OSI模型的概念来理解网络通信的原理，使用TCP/IP模型的实践来分析和解决问题。

理解这两种模型对于学习网络协议至关重要。当我们讨论一个具体协议时，需要清楚它属于哪一层，以及它与上下层之间的关系。例如，HTTP是应用层协议，它建立在TCP之上，而TCP又建立在IP之上；我们在浏览器中输入一个URL时，数据会经过应用层（HTTP）、传输层（TCP）、网络层（IP）、数据链路层（以太网）和物理层，最终通过物理网络传输到服务器。

### 1.2 IP地址、端口、域名、DNS解析

#### 1.2.1 IP地址详解

IP地址（Internet Protocol Address）是互联网上标识每台设备地址的唯一标识符。IPv4地址是一个32位的二进制数，通常用点分十进制表示法来表示，即四个0到255之间的十进制数用点分隔，如192.168.1.1。IPv4地址理论上可以提供约43亿个地址，但随着互联网的快速发展，IPv4地址已经枯竭，因此IPv6应运而生。IPv6地址是128位的地址，通常用八组十六进制数表示，如2001:0db8:85a3:0000:0000:8a2e:0370:7334。

IPv4地址分为公网地址和私有地址两类。公网地址是全球唯一的，由互联网号码分配局（IANA）统一分配；私有地址是组织内部使用的，不能在互联网上直接路由。私有地址的范围包括10.0.0.0到10.255.255.255（10.0.0.0/8）、172.16.0.0到172.31.255.255（172.16.0.0/12）和192.168.0.0到192.168.255.255（192.168.0.0/16）。在家庭和办公网络中，我们通常使用私有地址，通过NAT（网络地址转换）与公网进行通信。

IPv4地址还可以分为主机地址、网络地址和广播地址。在一个网络中，主机地址用于标识网络中的具体设备，网络地址用于标识整个网络，广播地址用于向网络中的所有设备发送数据。例如，在192.168.1.0/24这个网络中，192.168.1.0是网络地址，192.168.1.1到192.168.1.254是主机地址，192.168.1.255是广播地址。子网掩码用于确定IP地址中哪些位是网络部分，哪些位是主机部分。常见的子网掩码包括255.255.255.0（/24）、255.255.0.0（/16）、255.0.0.0（/8）等。

IPv6地址相比IPv4有更大的地址空间、更好的安全性、更简化的路由配置等优势。IPv6地址分为多种类型，包括单播地址（用于单个接口）、多播地址（用于一组接口）、任播地址（用于任意一个接口）。IPv6还有链路本地地址、唯一本地地址、全球单播地址等分类。IPv6使用无状态地址自动配置（SLAAC）或DHCPv6来自动获取地址，大大简化了网络配置。虽然IPv6的普及还需要时间，但它是互联网发展的必然趋势。

#### 1.2.2 端口详解

端口（Port）是传输层协议用于区分同一台主机上不同应用进程的标识符。端口号是一个16位的无符号整数，范围从0到65535。其中，0到1023是知名端口（Well-Known Ports），由IANA统一分配和注册，如HTTP（80）、HTTPS（443）、FTP（21）、SSH（22）、SMTP（25）、DNS（53）等；1024到49151是注册端口（Registered Ports），通常由用户进程或应用程序动态使用；49152到65535是动态端口或私有端口，可供任何应用程序使用。

当一个应用程序需要使用网络服务时，它会绑定到一个特定的端口上进行监听，等待客户端的连接请求。例如，一个Web服务器会监听80端口（HTTP）或443端口（HTTPS），当客户端发送请求时，服务器会接受连接并处理请求。在客户端，操作系统会为每个连接分配一个临时端口，通常是从动态端口范围中选取。

在Node.js中，我们可以使用net模块创建TCP服务器并指定端口。以下是一个简单的TCP服务器示例，它监听3000端口并处理客户端连接：

```javascript
// 引入Node.js的net模块，用于创建TCP服务器
const net = require('net');

// 创建TCP服务器实例
const server = net.createServer((socket) => {
  // 当客户端连接时触发
  console.log('客户端已连接:', socket.remoteAddress, socket.remotePort);

  // 监听客户端发送的数据
  socket.on('data', (data) => {
    console.log('收到客户端数据:', data.toString());
    // 向客户端发送响应
    socket.write('服务器收到数据: ' + data);
  });

  // 监听连接关闭事件
  socket.on('end', () => {
    console.log('客户端连接关闭');
  });

  // 监听错误事件
  socket.on('error', (err) => {
    console.error('连接错误:', err.message);
  });
});

// 服务器监听3000端口
server.listen(3000, () => {
  console.log('TCP服务器已启动，监听端口: 3000');
});
```

#### 1.2.3 域名系统详解

域名（Domain Name）是IP地址的易记表示形式，由于IP地址难以记忆和识别，域名系统（DNS）应运而生。域名采用层次结构，从右到左分别是顶级域（TLD）、二级域（SLD）和子域（Subdomain）。例如，在www.example.com中，com是顶级域，example是二级域，www是子域。顶级域分为通用顶级域（gTLD，如.com、.org、.net、.edu等）和国家代码顶级域（ccTLD，如.cn、.jp、.uk等）。

域名的层次结构对应着DNS服务器的层次结构。全球共有13组根域名服务器（A到M），它们管理着顶级域的权威信息。根域名服务器知道所有顶级域服务器的地址，顶级域服务器知道二级域的权威域名服务器地址，以此类推。当我们需要解析一个域名时，会从根服务器开始，依次查询各级DNS服务器，直到获得最终的解析结果。

DNS记录有多种类型，常见的有A记录（将域名指向IPv4地址）、AAAA记录（将域名指向IPv6地址）、CNAME记录（将域名指向另一个域名）、MX记录（指定邮件服务器）、TXT记录（用于验证和SPF策略）、NS记录（指定域名服务器）等。通过配置不同的DNS记录，我们可以实现域名解析、邮件服务负载均衡、安全验证等多种功能。

#### 1.2.4 DNS解析过程详解

DNS解析是将域名转换为IP地址的过程，是互联网通信的关键环节。当用户在浏览器中输入一个域名时，系统会按照以下顺序进行DNS解析：首先检查浏览器DNS缓存，然后检查操作系统DNS缓存，接着检查本地DNS服务器（如ISP提供的DNS服务器或企业自建的DNS服务器），最后递归查询根域名服务器、顶级域名服务器和权威域名服务器，直到获得解析结果。

DNS解析过程使用了递归查询和迭代查询两种方式。当客户端向本地DNS服务器发起查询请求时，通常使用递归查询，即本地DNS服务器负责向其他DNS服务器查询并返回最终结果。而本地DNS服务器向根域名服务器、顶级域名服务器查询时，使用迭代查询，即每一步都返回下一个应该查询的DNS服务器地址，让本地DNS服务器继续查询。这种设计减轻了根服务器的负担，提高了DNS系统的可扩展性。

DNS解析结果通常会被缓存一段时间，以减少重复查询，提高解析效率。缓存时间由DNS记录中的TTL（Time To Live）值决定。合理的TTL设置需要在解析效率和更新速度之间取得平衡。对于频繁变动的DNS记录，应设置较短的TTL；对于相对稳定的记录，可以设置较长的TTL以减轻DNS服务器负担。

在实际开发中，我们可以使用nslookup、dig或ping等命令来查询域名的DNS信息。以下是使用nslookup查询域名的示例：

```bash
# 查询域名的A记录
nslookup www.example.com

# 查询域名的MX记录
nslookup -type=MX example.com

# 使用Google公共DNS查询
nslookup www.example.com 8.8.8.8
```

### 1.3 ARP协议与ARP欺骗

#### 1.3.1 ARP协议详解

ARP（Address Resolution Protocol，地址解析协议）是网络层与数据链路层之间的桥梁，负责将IP地址解析为MAC地址。在局域网中，数据帧是通过MAC地址进行寻址的，而不是IP地址。当主机需要向同一网络中的另一台主机发送数据时，它需要知道目标主机的MAC地址，这就是ARP协议存在的意义。ARP协议的工作过程可以分为请求和响应两个阶段。

当主机A需要向主机B发送数据时，如果主机A不知道主机B的MAC地址，它会首先检查自己的ARP缓存表（ARP Cache），看看是否已经有目标IP地址对应的MAC地址。如果缓存中没有，主机A会在局域网上广播一个ARP请求包，请求包含主机A的IP地址和MAC地址，以及目标主机B的IP地址。所有接收到这个请求的主机都会检查目标IP地址是否与自己匹配，如果不匹配则忽略这个请求。

当目标主机B接收到ARP请求并确认目标IP地址是自己时，它会向主机A发送一个ARP响应包，单播回复自己的MAC地址。主机A收到响应后，会将主机B的IP地址和MAC地址添加到自己的ARP缓存表中，并开始发送数据。ARP缓存表中的条目通常有一个老化时间（通常是20分钟），过期后需要重新查询。

ARP缓存表是网络通信性能的重要优化手段。通过缓存已解析的MAC地址，主机可以避免每次通信都进行ARP请求，减少网络流量和响应延迟。我们可以使用arp -a命令查看本机的ARP缓存表：

```bash
# Windows查看ARP缓存
arp -a

# Linux/Mac查看ARP缓存
arp -n

# 查看特定IP地址的MAC地址
arp -a 192.168.1.1
```

#### 1.3.2 ARP欺骗原理与防护

ARP欺骗（ARP Spoofing）是一种常见的网络攻击手段，攻击者通过发送伪造的ARP响应包来欺骗其他主机的ARP缓存，从而截获或篡改网络流量。ARP协议的设计存在一个严重的安全缺陷：它没有认证机制，任何主机都可以发送ARP响应，而接收主机不会验证响应的真实性。这个缺陷使得ARP欺骗攻击成为可能。

ARP欺骗的攻击原理是：攻击者主机C向目标主机A发送ARP响应，声称网关的MAC地址是攻击者的MAC地址；同时向网关发送ARP响应，声称主机A的MAC地址是攻击者的MAC地址。这样，攻击者就处于主机A和网关之间的中间人位置，可以窃听、篡改或阻断两者之间的通信。攻击者通常还会开启IP转发功能，将截获的数据包转发给真正的目标，使通信双方感觉不到攻击的存在。

ARP欺骗会造成多种危害：窃听敏感信息（如登录密码、银行账户）；篡改传输数据（如注入恶意代码）；阻断网络连接（导致网络中断）；进行中间人攻击（-session劫持）等。在企业内部网络和公共WiFi环境中，ARP欺骗攻击尤为常见。

防护ARP欺骗的措施包括：使用静态ARP表将重要的IP地址和MAC地址绑定；启用交换机的动态ARP检查（DAI）功能；使用VLAN分隔网络减少攻击面；部署ARP防火墙软件检测异常ARP活动；在安全性要求高的环境中使用IPsec等加密协议保护通信。以下是在Linux系统中绑定静态ARP条目的方法：

```bash
# 添加静态ARP条目（临时生效）
arp -s 192.168.1.1 00:11:22:33:44:55

# 使用iproute2命令添加永久ARP条目
ip neigh add 192.168.1.1 lladdr 00:11:22:33:44:55 dev eth0 nud permanent
```

### 1.4 DHCP动态主机配置协议

DHCP（Dynamic Host Configuration Protocol，动态主机配置协议）是用于自动分配IP地址和其他网络配置参数的应用层协议。在一个网络中，如果每台主机都需要手动配置IP地址、子网掩码、网关、DNS服务器等参数，工作量巨大且容易出错。DHCP协议通过自动化的方式简化了网络配置过程，使得设备能够即插即用，大大提高了网络管理的效率。

DHCP的工作过程采用客户端-服务器模式，包含四个主要步骤，称为DORA流程：Discover（发现）、Offer（提供）、Request（请求）和Acknowledgment（确认）。当一台新设备连接到网络时，它的DHCP客户端会首先广播一个DHCP Discover消息，寻找网络中的DHCP服务器。DHCP服务器收到请求后会从地址池中选择一个可用的IP地址，并广播DHCP Offer消息提供配置信息。客户端收到Offer后，会选择其中一个服务器并广播DHCP Request消息正式请求配置。服务器确认请求后，会发送DHCP Acknowledgment消息，包含分配的IP地址和其他配置参数。

DHCP服务器分配的参数不仅包括IP地址，还包括子网掩码、网关地址、DNS服务器地址、租约时间等。DHCP使用UDP协议的67端口（服务器端）和68端口（客户端）进行通信。分配的IP地址有一个租约期限，通常是几个小时到几天不等，客户端需要在租约到期前续租以继续使用该地址。

在企业网络和家庭网络中，路由器通常集成DHCP服务器功能。在Windows系统中，我们可以使用ipconfig命令查看和操作DHCP配置：

```bash
# 查看当前IP地址配置
ipconfig /all

# 释放当前DHCP获取的IP地址
ipconfig /release

# 重新向DHCP服务器请求IP地址
ipconfig /renew

# 查看DHCP租约信息
ipconfig /all | findstr "DHCP"
```

## 二、HTTP/HTTPS协议

### 2.1 HTTP请求与响应结构

#### 2.1.1 HTTP请求结构详解

HTTP（HyperText Transfer Protocol，超文本传输协议）是Web应用的基础协议，定义了客户端与服务器之间通信的格式和规则。HTTP是一种无状态的请求-响应协议，客户端发送请求，服务器返回响应。每次请求-响应完成后，连接通常会关闭，服务器不会记住客户端的状态。HTTP/1.1引入了Keep-Alive机制，可以在一次TCP连接中处理多个请求，提高了通信效率。

一个完整的HTTP请求由请求行（Request Line）、请求头（Request Headers）、空行和请求体（Request Body）四部分组成。请求行包含HTTP方法、请求URI和HTTP版本三个部分，例如：GET /index.html HTTP/1.1。请求头以键值对的形式提供客户端信息、请求内容、认证信息等。空行用于分隔请求头和请求体。请求体包含POST、PUT等方法要提交的数据，对于GET方法，请求体通常为空。

以下是一个完整的HTTP POST请求示例，包含请求头和请求体：

```
POST /api/users HTTP/1.1
Host: api.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36
Content-Type: application/json
Content-Length: 58
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Accept: application/json
Accept-Language: zh-CN,zh;q=0.9,en;q=0.8
Cache-Control: no-cache

{"username":"张三","email":"zhangsan@example.com","age":28}
```

在请求头中，Host字段是必需的，它指定了目标服务器的主机名和端口号。User-Agent字段标识客户端类型，服务器可以根据这个字段返回适合该客户端的内容。Content-Type字段指定请求体的数据格式，服务器根据这个字段来解析请求体。Authorization字段用于传递认证信息，常见的认证方式包括Basic认证和Bearer Token认证。

#### 2.1.2 HTTP响应结构详解

HTTP响应与请求结构类似，由状态行（Status Line）、响应头（Response Headers）、空行和响应体（Response Body）四部分组成。状态行包含HTTP版本、状态码和状态文本三部分，例如：HTTP/1.1 200 OK。响应头以键值对的形式提供服务器信息、响应内容、缓存策略等。响应体包含服务器返回的实际数据，可能是HTML文档、JSON数据、图片或其他内容。

以下是一个完整的HTTP响应示例：

```
HTTP/1.1 200 OK
Server: nginx/1.21.6
Date: Wed, 15 Mar 2026 10:30:00 GMT
Content-Type: application/json; charset=utf-8
Content-Length: 256
Connection: keep-alive
Cache-Control: max-age=3600
ETag: "64bytehash"
X-Request-ID: abc123

{"code":200,"message":"success","data":{"id":1,"name":"张三","email":"zhangsan@example.com"}}
```

在响应头中，Server字段标识服务器软件信息。Date字段表示响应生成的时间。Content-Type字段告诉客户端响应体的数据格式，客户端需要根据这个字段来解析响应体。Content-Length字段表示响应体的字节长度。Connection字段控制连接是否保持打开状态，keep-alive表示保持连接，close表示关闭连接。Cache-Control字段指导客户端如何缓存这个响应。ETag字段提供资源的版本标识，用于条件请求和缓存验证。

### 2.2 HTTP方法详解

HTTP定义了一组请求方法（也称为动词），用于指定对资源的操作类型。每种方法都有特定的语义和用途，正确选择HTTP方法体现了RESTful API设计的最佳实践。

**GET方法**是最常用的HTTP方法，用于请求指定的资源。GET请求应该是安全的（不会改变服务器状态）和幂等的（多次执行结果相同）。GET请求的参数通过URL查询字符串传递，长度受URL长度限制（通常不超过2048个字符）。GET请求可以被缓存，可以被浏览器历史记录保存。在实际应用中，GET用于获取数据列表、搜索查询等只读操作。

**POST方法**用于向服务器提交数据进行处理，常用于创建新资源。POST请求的参数放在请求体中，没有大小限制。POST请求不是幂等的，多次提交可能产生不同的结果（如创建多个资源）。POST请求一般不会被缓存，不会保存在浏览器历史记录中。POST用于用户注册、数据提交、表单处理等场景。

**PUT方法**用于更新指定资源的所有内容，或者创建指定URI的资源。PUT请求是幂等的，多次执行相同的PUT请求结果相同。PUT传递完整的资源数据，如果只更新部分字段，通常应该使用PATCH方法。

**DELETE方法**用于删除指定的资源。DELETE请求是幂等的，多次删除同一个资源，结果都是资源被删除（已不存在）。删除操作应该谨慎执行，因为它会永久性地移除数据。

**PATCH方法**用于部分更新指定资源的字段。与PUT不同，PATCH只传递要修改的字段，而不是完整的资源数据。PATCH不是完全幂等的，取决于具体的实现。

**HEAD方法**与GET类似，但服务器只返回响应头，不返回响应体。HEAD用于检查资源是否存在、获取资源的元信息、验证缓存有效性等场景，不需要传输完整的响应体，可以节省带宽。

**OPTIONS方法**用于获取目标资源支持的HTTP方法列表。浏览器在发送跨域请求前会先发送OPTIONS预检请求（Preflight Request），询问服务器是否允许该跨域请求。OPTIONS请求检查Access-Control-Allow-Origin、Access-Control-Allow-Methods等响应头，确定是否可以继续发送实际请求。

### 2.3 HTTP状态码详解

HTTP状态码是服务器对请求处理结果的数字描述，分为五类，用首位数字区分。正确理解和运用状态码对于API设计和错误处理至关重要。

**1xx informational（信息性状态码）** 表示请求已被服务器接收，继续处理。这类状态码是临时的，出现在实际响应之前。常见的1xx状态码包括100 Continue（客户端应继续发送请求）、101 Switching Protocols（服务器正在切换协议，如升级到WebSocket）。在HTTP/2中，1xx状态码用于传输元数据，不再用于表示Continue。

**2xx Success（成功状态码）** 表示请求已成功被服务器接收、理解并处理。200 OK是最常见的状态码，表示请求成功；201 Created表示资源已创建，常用于POST请求成功；204 No Content表示请求成功但响应体为空，常用于DELETE请求成功。2xx状态码表示客户端的操作已经成功完成。

**3xx Redirection（重定向状态码）** 表示需要进一步操作才能完成请求。301 Moved Permanently表示资源已永久移动到新位置，搜索引擎会更新索引；302 Found表示临时重定向，客户端应继续使用原始URI；304 Not Modified表示资源未修改，客户端可以使用缓存，这是浏览器缓存机制的重要组成部分。3xx状态码告诉客户端需要采取进一步行动，通常是跳转到一个新的URI。

**4xx Client Error（客户端错误状态码）** 表示请求包含语法错误或无法被服务器理解和处理。400 Bad Request表示请求格式错误或参数无效；401 Unauthorized表示需要认证，客户端应该提供认证信息；403 Forbidden表示服务器拒绝访问，即使提供认证信息也无法访问；404 Not Found表示请求的资源不存在；405 Method Not Allowed表示请求方法不被支持；422 Unprocessable Entity表示请求格式正确但语义错误；429 Too Many Requests表示请求频率超过限制。4xx状态码的责任在客户端，客户端需要修改请求才能成功。

**5xx Server Error（服务器错误状态码）** 表示服务器在处理请求时发生内部错误。500 Internal Server Error是通用的服务器错误，表示意外情况；502 Bad Gateway表示作为网关的服务器收到了无效响应；503 Service Unavailable表示服务器暂时不可用（可能过载或维护）；504 Gateway Timeout表示网关超时。5xx状态码的责任在服务器，客户端无法直接解决这个问题，只能等待服务器恢复或联系管理员。

### 2.4 HTTP头部详解

#### 2.4.1 通用头部

通用头部（General Headers）同时适用于请求和响应，不涉及具体的数据内容。Connection头部控制当前连接的行为，keep-alive表示保持连接不关闭，close表示关闭连接。Date头部表示消息产生的日期和时间。Cache-Control头部指导缓存机制的行为，no-cache表示必须先与服务器验证才能使用缓存，no-store表示不允许缓存，max-age表示缓存有效期。Transfer-Encoding指定消息体的传输编码方式，如chunked（分块传输）。

#### 2.4.2 请求头

请求头（Request Headers）由客户端发送到服务器，提供关于请求本身和客户端的信息。Accept头部告诉服务器客户端能够处理的内容类型，如text/html、application/json等。Accept-Language头部指定客户端偏好的语言。Accept-Encoding头部指定客户端支持的内容编码，如gzip、deflate、br等。User-Agent头部标识客户端应用程序类型和版本，服务器可以根据这个信息返回适合的内容。

Authorization头部携带认证凭证，常见的方式包括Basic认证（Base64编码的用户名:密码）和Bearer Token认证（JWT令牌）。Referer头部表示请求的来源URL，常用于统计分析、防盗链和安全检查。Origin头部用于跨域请求，指明请求的来源域名。Cookie头部携带服务器之前设置的Cookie值。If-Modified-Since、If-None-Match等头部用于条件请求，实现缓存验证。

#### 2.4.3 响应头

响应头（Response Headers）由服务器发送到客户端，提供关于响应的元数据。Content-Type头部指定响应体的MIME类型和字符编码，如application/json; charset=utf-8。Content-Length头部指定响应体的字节长度，服务器在发送响应前必须知道这个值才能正确处理分块传输。Content-Encoding头部指定响应体的编码方式，如gzip压缩。Last-Modified头部表示资源最后修改的时间。ETag头部提供资源的特定版本标识符。

Set-Cookie头部用于在客户端设置Cookie，包含名称、值、过期时间、路径、域、安全标志等属性。Access-Control-Allow-Origin等CORS头部控制跨域访问权限。X-Frame-Options头部防止点击劫持攻击。X-Content-Type-Options头部防止MIME类型嗅探。Strict-Transport-Security头部强制使用HTTPS连接。

#### 2.4.4 实体头

实体头（Entity Headers）用于描述消息体的内容，出现在请求和响应的消息体部分。Content-Type描述实体媒体的类型。Content-Length表示实体的大小。Content-Language表示实体使用的语言。Content-Encoding表示实体数据的编码方式。Content-Location给出实体相对应的资源位置。Content-MD5提供实体的MD5校验值（已废弃）。

### 2.5 Cookie与Session

#### 2.5.1 Cookie机制详解

Cookie是服务器发送到用户浏览器并保存在本地的小型数据片段，在后续请求中会被发送回同一服务器。Cookie的主要用途包括会话管理（用户登录状态）、个性化设置（主题、语言等）、行为追踪（访问历史、广告投放）等。Cookie以键值对的形式存储，由服务器通过Set-Cookie响应头设置，浏览器会自动在后续请求中通过Cookie请求头发送Cookie。

Cookie有多种类型：会话Cookie在浏览器关闭时自动删除，不设置过期时间；持久性Cookie保存到过期时间才删除；HttpOnly Cookie无法通过JavaScript的document.cookie访问，只能通过HTTP传输，有助于防止XSS攻击；Secure Cookie只能在HTTPS连接中传输；SameSite Cookie可以防止CSRF攻击。

Cookie的属性包括：Name（名称）、Value（值）、Domain（所属域）、Path（路径）、Expires/Max-Age（过期时间）、Size（大小）、HttpOnly（是否仅HTTP）、Secure（是否安全）、SameSite（跨站策略）。Domain属性决定了Cookie会被发送到哪些服务器，如果不设置，则默认是创建Cookie的服务器；设置Domain时，只能设置为当前域名或其父域名。Path属性进一步限制Cookie的发送范围，如设置为"/api"则只有/api开头的请求才会发送该Cookie。

以下是一个Set-Cookie响应头的示例：

```
Set-Cookie: session_id=abc123; expires=Wed, 15 Mar 2026 12:00:00 GMT; path=/; domain=.example.com; HttpOnly; Secure; SameSite=Strict
```

在Node.js中，我们可以使用Express框架的cookie-parser中间件来处理Cookie：

```javascript
// 引入Express框架
const express = require('express');
const cookieParser = require('cookie-parser');

const app = express();

// 使用cookie-parser中间件
app.use(cookieParser());

// 设置Cookie
app.get('/login', (req, res) => {
  // 设置一个会话Cookie
  res.cookie('session_id', 'abc123', {
    httpOnly: true,      // 防止XSS访问
    secure: false,       // HTTPS才传输（生产环境应为true）
    maxAge: 24 * 60 * 60 * 1000, // 24小时有效期
    sameSite: 'strict'   // 防止CSRF
  });
  res.send('登录成功');
});

// 读取Cookie
app.get('/profile', (req, res) => {
  console.log('收到的Cookie:', req.cookies);
  const sessionId = req.cookies.session_id;
  if (sessionId) {
    res.send(`欢迎回来，Session ID: ${sessionId}`);
  } else {
    res.status(401).send('请先登录');
  }
});

// 删除Cookie
app.get('/logout', (req, res) => {
  res.clearCookie('session_id');
  res.send('已退出登录');
});
```

#### 2.5.2 Session机制详解

Session（会话）是服务器端存储用户状态信息的机制。由于HTTP是无状态协议，服务器需要一种方式来区分不同的用户和会话。Session的工作原理是：当用户首次访问服务器时，服务器创建一个唯一的Session ID，并将这个ID存储在Cookie中返回给客户端；后续请求中，客户端会发送这个Session ID，服务器根据ID查找对应的会话数据，从而识别用户状态。

与Cookie相比，Session的优势在于：敏感数据存储在服务器端，不会在网络中传输，安全性更高；可以存储任意类型和大小的数据，不受Cookie大小限制（通常4KB）；更适合存储大量用户的状态信息。Session的缺点是需要服务器存储会话数据，增加服务器内存压力；分布式部署时需要考虑Session共享（如使用Redis存储）。

Session数据通常存储在服务器的内存、数据库或分布式缓存中。在生产环境中，建议使用Redis等高性能缓存存储Session，以提高访问速度和可扩展性。Session也有过期时间，通常与Cookie的过期时间一致或略长，确保用户长时间不操作后能够正常重新登录。

以下是使用Express和express-session中间件管理Session的示例：

```javascript
const express = require('express');
const session = require('express-session');
const RedisStore = require('connect-redis').default;

const app = express();

// 配置Redis存储Session
const redisClient = require('./redisClient'); // Redis客户端连接

app.use(session({
  store: new RedisStore({ client: redisClient }), // 使用Redis存储Session
  secret: 'your-secret-key-change-in-production', // 用于签名Session ID的密钥
  name: 'session_id',        // Cookie名称
  resave: false,             // 不强制保存未修改的Session
  saveUninitialized: false,  // 不保存未初始化的Session
  cookie: {
    httpOnly: true,          // 防止XSS攻击
    secure: false,           // HTTPS传输（生产环境应为true）
    maxAge: 24 * 60 * 60 * 1000, // 24小时
    sameSite: 'strict'      // 防止CSRF攻击
  }
}));

// 登录并创建Session
app.post('/login', (req, res) => {
  const { username, password } = req.body;

  // 验证用户身份（实际应用中需要查询数据库）
  if (username === 'admin' && password === 'password') {
    req.session.userId = 'user_123';
    req.session.username = username;
    res.json({ success: true, message: '登录成功' });
  } else {
    res.status(401).json({ success: false, message: '用户名或密码错误' });
  }
});

// 获取Session信息
app.get('/api/user', (req, res) => {
  if (req.session.userId) {
    res.json({
      userId: req.session.userId,
      username: req.session.username
    });
  } else {
    res.status(401).json({ message: '未登录' });
  }
});

// 销毁Session（登出）
app.post('/logout', (req, res) => {
  req.session.destroy((err) => {
    if (err) {
      res.status(500).json({ message: '退出失败' });
    } else {
      res.clearCookie('session_id');
      res.json({ success: true, message: '已退出' });
    }
  });
});
```

#### 2.5.3 Cookie与Session的选择

在实际应用中，Cookie和Session各有适用场景，我们需要根据具体需求选择合适的方案。对于简单的键值对数据、需要在客户端存储的标识符、不敏感的信息（如用户偏好设置），使用Cookie更加轻量。对于敏感数据、需要存储大量信息、需要跨子域共享的数据，使用Session更加安全。对于需要长期保存的信息（如登录状态），可以使用Session结合持久化Cookie；对于短期信息（如购物车），可以使用Session或Cookie。

现代Web应用通常采用混合方案：使用Cookie存储Session ID，后端使用Session存储用户状态，同时使用Token（如JWT）作为补充认证机制。这种方案结合了Cookie的便利性和Session的安全性，是目前最常用的认证方案。

### 2.6 缓存机制详解

#### 2.6.1 缓存基本概念

Web缓存是提升应用性能的关键技术，通过存储之前获取的资源（HTML页面、图片、CSS文件、JavaScript文件等），减少对服务器的请求次数，加快页面加载速度，降低服务器负载。缓存可以分为浏览器缓存、代理服务器缓存（如CDN）、网关缓存等不同层次。合理使用缓存可以显著提升用户体验，减少带宽消耗。

HTTP缓存主要通过请求头和响应头来控制。服务端通过设置响应头告诉浏览器如何缓存资源，浏览器根据响应头决定是否缓存以及缓存多久。下次请求时，浏览器会检查缓存是否有效，如果有效则直接使用缓存，如果无效则重新向服务器请求。

缓存策略分为两大类：强缓存和协商缓存。强缓存直接从本地缓存获取资源，不向服务器发送请求；协商缓存会向服务器发送验证请求，服务器根据条件判断是否可以使用缓存。

#### 2.6.2 强缓存

强缓存通过Cache-Control和Expires两个响应头来控制。Cache-Control是HTTP/1.1的标准，Expires是HTTP/1.0的标准，现代浏览器通常优先使用Cache-Control。

Cache-Control的主要指令包括：max-age=seconds设置缓存的最大有效期，单位为秒；no-cache强制每次使用缓存前都向服务器验证；no-store不允许缓存任何内容；private表示只能由浏览器缓存，不能被代理服务器缓存；public表示可以被任何中间节点缓存。

Expires头部指定缓存过期的绝对时间，格式为GMT时间字符串。如果当前时间在Expires指定的时间之前，浏览器就使用缓存；如果已过期，则重新向服务器请求。Expires的缺点是使用本地时间，如果客户端时间不准确可能导致缓存失效或提前过期。

```bash
# 强缓存示例：缓存有效期为1小时
Cache-Control: max-age=3600
# 或者使用Expires（不推荐，应优先使用Cache-Control）
Expires: Wed, 15 Mar 2026 13:00:00 GMT
```

#### 2.6.3 协商缓存

当强缓存失效或浏览器设置no-cache时，浏览器会发送协商缓存请求。服务器通过比较浏览器发送的验证条件与资源的当前状态，决定是否返回304 Not Modified（使用缓存）或200 OK（返回新资源）。

Last-Modified/If-Modified-Since是最早的协商缓存机制。服务器在响应头中设置Last-Modified，表示资源的最后修改时间；浏览器下次请求时在If-Modified-Since头中带上这个时间；服务器比较时间，如果资源未修改则返回304，如果有变化则返回新资源并更新Last-Modified。

ETag/If-None-Match是更精确的协商缓存机制。由于Last-Modified只能精确到秒，对于秒内的多次修改无法区分，ETag采用内容哈希来解决这个问题。服务器为每个资源生成唯一的ETag值，通常是文件内容的哈希值；浏览器在If-None-Match头中发送之前的ETag；服务器比较ETag，如果相同说明内容未变，返回304。

```bash
# 服务器响应（首次请求）
HTTP/1.1 200 OK
Last-Modified: Wed, 15 Mar 2026 10:00:00 GMT
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"
Content-Length: 1024

# 浏览器缓存失效后，发送协商请求
GET /static/app.js HTTP/1.1
If-Modified-Since: Wed, 15 Mar 2026 10:00:00 GMT
If-None-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"

# 服务器检查后返回304（内容未变）
HTTP/1.1 304 Not Modified
Last-Modified: Wed, 15 Mar 2026 10:00:00 GMT
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"
```

#### 2.6.4 实际应用中的缓存策略

在实际项目中，我们需要根据资源的特性制定合理的缓存策略。对于不经常变化的静态资源（如CSS、JavaScript、图片等），可以设置较长的强缓存时间（如一年），并使用基于内容的版本号或哈希值作为文件名的一部分，这样在资源更新时URL会变化，浏览器会重新请求新资源。对于频繁变化的资源（如HTML页面、API数据），应设置较短的缓存时间或使用no-cache策略，确保用户能够获取最新内容。

以下是一个使用Webpack打包资源时添加内容哈希的示例：

```javascript
// webpack.config.js
module.exports = {
  output: {
    // 打包后的文件名包含内容哈希
    filename: '[name].[contenthash].js',
    chunkFilename: '[name].[contenthash].chunk.js'
  },
  // ... 其他配置
};
```

```html
<!-- HTML中引用带哈希的资源 -->
<script src="/static/app.33a64df5.js"></script>
```

通过这种方式，当JavaScript文件内容变化时，哈希值会变化，HTML中引用的URL也会变化，从而确保用户获取到最新版本。同时，对于内容没有变化的资源，浏览器可以使用缓存，避免重复下载。

### 2.7 HTTPS加密原理

#### 2.7.1 SSL/TLS协议基础

HTTPS（HyperText Transfer Protocol Secure）是HTTP的安全版本，通过SSL/TLS协议对通信进行加密保护。HTTPS使用443端口（HTTP使用80端口），在HTTP和TCP之间增加了一层SSL/TLS层。所有通过HTTPS传输的数据都会被加密，防止被窃听、篡改或伪造。

SSL（Secure Sockets Layer，安全套接字层）是由Netscape公司开发的安全协议，经历了SSL 2.0、SSL 3.0等多个版本。由于SSL 3.0存在已知的安全漏洞，目前已被废弃。TLS（Transport Layer Security，传输层安全）是SSL的标准化版本，由IETF接管，目前主要使用TLS 1.2和TLS 1.3。TLS 1.3相比1.2有更快的握手速度、简化了密码套件、移除了不安全的加密算法等改进。

SSL/TLS协议主要解决三个安全问题：加密（Encryption）确保通信内容不被窃听，只有发送方和接收方能理解内容；完整性（Integrity）确保通信内容不被篡改，任何修改都能被检测到；认证（Authentication）确保通信对方的身份真实，防止被假冒。

#### 2.7.2 对称加密与非对称加密

SSL/TLS使用两种加密方式的组合来实现安全通信：对称加密和非对称加密。

对称加密使用相同的密钥进行加密和解密。优点是计算速度快，适合加密大量数据；缺点是密钥传输困难，如何将密钥安全地发送给接收方是个问题。常见的对称加密算法包括AES（Advanced Encryption Standard）、DES（Data Encryption Standard）、3DES、RC4等。AES是目前的行业标准，支持128位、192位和256位密钥长度。

非对称加密使用一对密钥：公钥和私钥。公钥可以公开分发，用于加密数据或验证签名；私钥必须严格保密，用于解密数据或生成签名。公钥和私钥是数学关联的，用公钥加密的数据只能用对应的私钥解密，反之亦然。优点是密钥传输更安全，缺点是计算速度慢，消耗更多资源。常见的非对称加密算法包括RSA（基于大数分解）、DSA（数字签名算法）、ECDSA（椭圆曲线数字签名算法）、DH（Diffie-Hellman密钥交换）等。

RSA是目前最广泛使用的非对称加密算法，其安全性基于大数分解的困难性。典型的RSA密钥长度为2048位或4096位。2048位的RSA密钥被认为可以提供足够的安全性到2030年左右。

#### 2.7.3 HTTPS加密通信过程

HTTPS的加密通信过程分为两个阶段：握手阶段和实际通信阶段。

握手阶段的目的是在客户端和服务器之间建立安全的加密通道，主要步骤包括：TCP三次握手建立连接（这与HTTP相同）；客户端发送ClientHello消息，包含支持的TLS版本、密码套件列表、客户端随机数、扩展名等；服务器发送ServerHello消息，选择双方都支持的TLS版本和密码套件，发送服务器随机数；服务器发送证书链（包含服务器证书和中间CA证书）；服务器发送ServerKeyExchange消息（对于某些密钥交换算法）；服务器发送ServerHelloDone消息表示服务器握手消息发送完成。

客户端收到证书后会验证证书的合法性：检查证书是否在有效期内；检查证书是否被吊销（通过CRL或OCSP）；检查证书的域名是否与访问的域名匹配；验证证书链的每个证书是否由可信的根CA签发。客户端使用服务器证书中的公钥验证服务器身份，确保服务器是真实的而不是被假冒的。

密钥交换是握手中的关键步骤，TLS 1.2和TLS 1.3的密钥交换方式有所不同。TLS 1.2常用的是RSA密钥交换（客户端生成预主密钥，用服务器公钥加密后发送）或DH/ECDH密钥交换（双方协商生成预主密钥）。TLS 1.3只支持前向安全的ECDH密钥交换。密钥交换后，双方使用预主密钥和随机数通过密钥派生函数生成会话密钥。

握手完成后，双方使用会话密钥进行对称加密通信。会话密钥通常包括用于加密数据的对称密钥和用于验证数据完整性的MAC密钥。整个握手过程完成后，才开始传输实际的HTTP请求和响应数据。

#### 2.7.4 CA证书体系

证书颁发机构（Certificate Authority，CA）是负责签发、管理和吊销数字证书的可信第三方。CA证书体系是HTTPS信任链的基础，确保了服务器身份的真实性和通信的保密性。

数字证书包含以下核心信息：证书持有人的身份信息（域名、组织名称等）；证书持有人的公钥信息；证书的有效期；证书的序列号（唯一标识）；CA的签名信息；证书的扩展信息（如SAN、密钥用途等）。证书使用CA的私钥进行数字签名，任何人都可以用CA的公钥验证证书的真伪。

证书信任链（Chain of Trust）建立了多层次的验证机制。根证书（Root Certificate）由受信任的CA自签名颁发，预装在操作系统和浏览器的可信根证书列表中。中级证书（Intermediate Certificate）由根CA签发，用于签发终端实体证书，可以有多级中间CA。终端实体证书（End-entity Certificate）由中间CA签发，部署在实际的Web服务器上。

Let's Encrypt是一个免费的自动化CA，提供免费的SSL/TLS证书，大大推动了HTTPS的普及。Let's Encrypt使用ACME协议自动化证书的申请、验证和续期过程，适合个人网站和小型应用使用。对于企业用户，可以使用DigiCert、GlobalSign、GeoTrust等商业CA获取支持更长密钥和更高级别担保的证书。

### 2.8 HTTP/2新特性

HTTP/2是HTTP协议的重大升级，于2015年正式发布，相比HTTP/1.1有多项重要改进。HTTP/2的主要目标是减少页面加载延迟，提高传输效率，同时保持对HTTP/1.1的向后兼容性。

**二进制分帧层（Binary Framing）** 是HTTP/2的核心改进。在HTTP/1.1中，请求和响应是文本格式的，容易出现粘包和半包问题。HTTP/2将所有传输的数据分割成更小的帧（Frame），并采用二进制格式编码。不同类型的帧包括HEADERS帧（存放元数据）、DATA帧（存放实际数据）、SETTINGS帧、PRIORITY帧等。二进制分帧提高了解析效率，消除了文本格式的歧义性。

**多路复用（Multiplexing）** 解决了HTTP/1.1中的队头阻塞问题。在HTTP/1.1中，虽然可以使用多个TCP连接来并发请求，但每个连接同时只能处理一个请求-响应。HTTP/2可以在单个TCP连接上并行传输多个请求和响应，每个请求-响应被称为一个流（Stream），每个流有唯一的ID。不同流的帧可以交错发送，接收端根据流ID重新组装，解决了TCP连接层面的队头阻塞问题。

**服务器推送（Server Push）** 允许服务器主动向客户端推送资源，而不需要客户端显式请求。当浏览器请求一个HTML页面时，服务器可以主动推送页面需要的CSS、JavaScript、图片等资源，减少往返时间（RTT）。服务器推送通过PUSH_PROMISE帧实现，客户端可以选择接受或拒绝推送的资源。

**头部压缩（Header Compression）** 使用HPACK算法压缩HTTP头部，减少传输的数据量。HTTP请求中，头部信息往往占据很大比例，尤其是Cookie、User-Agent等重复出现的头部。HPACK使用静态表、动态表和哈夫曼编码来压缩头部，首次请求后，后续请求只需要传输变化的增量部分，压缩率通常可达50%-90%。

**流控制（Stream Control）** 和**流量控制（Flow Control）** 机制确保发送方不会压垮接收方。HTTP/2为每个流提供了独立的流量控制窗口，接收方可以告知发送方可以发送多少数据。这对于移动网络和低带宽环境尤为重要。

#### 2.8.1 HTTP/2与HTTP/1.1性能对比

HTTP/2相比HTTP/1.1在性能上有显著提升，特别是在以下场景：资源数量多、需要多个往返的应用；高延迟网络环境（如移动网络）；高带宽但高延迟的连接。HTTP/2的多路复用消除了建立多个TCP连接的必要，减少了连接建立的开销和服务器负载。头部压缩减少了传输的数据量，降低了带宽消耗。

但是，HTTP/2并非在所有场景下都优于HTTP/1.1。对于资源数量少、单连接已足够快的场景，HTTP/1.1的简单性可能更好。HTTP/2使用单个TCP连接，在丢包率高的网络环境下可能反而不如多个TCP连接（因为TCP的拥塞控制会影响所有流）。此外，HTTP/2的实现复杂度更高，服务器需要更好的资源管理能力。

#### 2.8.2 实际应用中的HTTP/2配置

在Nginx中启用HTTP/2需要同时启用HTTPS，因为主流浏览器只支持基于TLS的HTTP/2。以下是Nginx配置HTTP/2的示例：

```nginx
server {
    listen 443 ssl http2;
    server_name www.example.com;

    ssl_certificate /path/to/certificate.pem;
    ssl_certificate_key /path/to/private.key;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256:TLS_AES_128_GCM_SHA256;

    root /var/www/html;
    index index.html;

    # HTTP/2服务器推送示例
    location / {
        http2_push /static/css/style.css;
        http2_push /static/js/app.js;
        http2_push /static/images/logo.png;
    }
}
```

### 2.9 HTTP/3（QUIC）新特性

HTTP/3是HTTP协议的第三个主要版本，于2022年正式发布。HTTP/3使用QUIC协议作为传输层，而不是TCP。QUIC是由Google开发的基于UDP的传输协议，旨在解决TCP的一些固有缺陷，提供更低的延迟和更好的拥塞控制。

#### 2.9.1 QUIC协议核心特性

**零RTT握手（0-RTT Handshake）** 是QUIC最重要的特性之一。对于曾经连接过的服务器，客户端可以在第一次发送数据时就包含加密数据，无需等待握手完成。这大大减少了连接建立的延迟，特别适合需要频繁建立连接的移动应用场景。0-RTT的安全性存在一定风险（可能受到重放攻击），但对于大多数应用来说，这个风险是可以接受的。

**连接迁移（Connection Migration）** 允许连接在改变IP地址或网络类型时保持活跃。传统的TCP连接由四元组（源IP、源端口、目标IP、目标端口）标识，网络变化时连接会断开。QUIC使用连接ID标识连接，当网络变化时，只要有一方使用相同的连接ID，连接就可以保持。这对于移动设备在WiFi和移动网络之间切换时特别有用。

**多路复用（Multiplexing without Head-of-Line Blocking）** 在QUIC中，每个流是独立管理的，一个流的丢包只会影响该流，不会阻塞其他流。而在HTTP/2 over TCP中，TCP的丢包会阻塞所有流，直到丢失的包被重传。

**改进的拥塞控制（Advanced Congestion Control）**。QUIC的拥塞控制是在应用层实现的，不依赖UDP或内核，可以更灵活地部署新的拥塞控制算法。QUIC还提供了更精确的RTT测量和拥塞状态跟踪。

#### 2.9.2 HTTP/3的应用场景

HTTP/3主要适用于以下场景：对延迟敏感的应用（如实时通信、在线游戏）；移动应用（网络切换频繁）；高丢包率网络环境（如无线网络）；大型Web应用（资源多，需要大量连接）。

在Nginx中启用HTTP/3（实验性支持）的配置示例：

```nginx
server {
    listen 443 ssl;
    listen 443 http3 reuseport;
    server_name www.example.com;

    ssl_certificate /path/to/certificate.pem;
    ssl_certificate_key /path/to/private.key;
    ssl_protocols TLSv1.3;

    # HTTP/3相关配置
    add_header Alt-Svc 'h3=":443"; ma=86400';
}
```

目前主流浏览器和主流网站都在逐步支持HTTP/3。根据Google的统计数据，Chrome中已有约25%的请求使用HTTP/3。未来的Web性能优化需要考虑HTTP/3的特性，特别是在移动网络环境下的应用。

## 三、TCP与UDP协议

### 3.1 TCP协议详解

#### 3.1.1 TCP协议特点

TCP（Transmission Control Protocol，传输控制协议）是互联网最重要的协议之一，提供了面向连接的、可靠的、基于字节流的传输服务。TCP的设计目标是解决IP层不可靠传输的问题，通过各种机制确保数据能够正确、有序、不重复地从一端传送到另一端。

TCP的主要特点包括：面向连接，通信前必须建立连接，通信结束后关闭连接；可靠传输，通过序号、确认、重传等机制保证数据正确到达；有序传输，数据按发送顺序到达；全双工通信，数据可以同时双向传输；流量控制，通过滑动窗口机制防止发送方发送过快；拥塞控制，防止网络过载导致性能下降。

TCP协议的工作过程可以分为三个阶段：建立连接（通过三次握手）、数据传输（双向交换数据）、关闭连接（通过四次挥手）。每个TCP连接都有唯一的一对端点，每个端点由IP地址和端口号组成，称为套接字（Socket）。

#### 3.1.2 TCP三次握手详解

三次握手（Three-way Handshake）是TCP建立连接的过程，通过三次报文交换确认双方的发送和接收能力，建立可靠的传输通道。之所以需要三次握手，是因为TCP是全双工协议，需要确认双方都能正常收发数据。

第一次握手：客户端发送SYN包（SYN=1, seq=x）给服务器，请求建立连接。SYN包不携带数据，但消耗一个序号。此时客户端进入SYN_SENT状态，等待服务器确认。

第二次握手：服务器收到SYN包后，如果同意建立连接，发送SYN+ACK包（SYN=1, ACK=1, seq=y, ack=x+1）作为响应。服务器进入SYN_RCVD状态。ack=x+1表示服务器确认收到了客户端的SYN。

第三次握手：客户端收到服务器的SYN+ACK包后，发送ACK包（ACK=1, seq=x+1, ack=y+1）给服务器。客户端进入ESTABLISHED状态。服务器收到ACK后也进入ESTABLISHED状态。

握手完成后，双方都确认了对方的发送能力和接收能力，可以开始传输数据。三次握手的目的是同步连接双方的序列号，为可靠传输和流量控制奠定基础。

#### 3.1.3 TCP四次挥手详解

四次挥手（Four-way Handshake）是TCP关闭连接的过程，由于TCP是全双工的，每个方向都需要单独关闭。四次挥手确保双方都将数据发送完毕后再断开连接。

第一次挥手：主动关闭方（假设是客户端）发送FIN包（FIN=1, seq=u）给服务器，请求关闭客户端到服务器方向的连接。客户端进入FIN_WAIT_1状态。

第二次挥手：服务器收到FIN包后，发送ACK包（ACK=1, ack=u+1）确认。服务器进入CLOSE_WAIT状态，客户端收到ACK后进入FIN_WAIT_2状态。此时客户端到服务器方向的连接已关闭，但服务器到客户端方向的连接仍然开放，服务器可以继续发送数据。

第三次挥手：当服务器也没有数据要发送时，服务器发送FIN包（FIN=1, seq=v）给客户端，请求关闭服务器到客户端方向的连接。服务器进入LAST_ACK状态。

第四次挥手：客户端收到FIN包后，发送ACK包（ACK=1, ack=v+1）确认。客户端进入TIME_WAIT状态，等待2MSL（Maximum Segment Lifetime，通常为2分钟）后进入CLOSED状态。服务器收到ACK后立即进入CLOSED状态。

TIME_WAIT状态的存在有重要原因：确保迟到的数据包被正确处理；确保最后的ACK能够到达服务器（如果ACK丢失，服务器会重发FIN）。在服务器端，需要通过合理配置来应对大量短连接导致的TIME_WAIT问题，如启用地址复用（SO_REUSEADDR）、调整超时时间等。

### 3.2 TCP可靠传输机制

#### 3.2.1 序号与确认机制

TCP的可靠传输建立在序号（Sequence Number）和确认（Acknowledgment）机制之上。每个字节在传输时都被分配一个序号，序号是32位的无符号整数，初始值在连接建立时随机选择。序号不是从0或1开始，而是随机选择，这是为了安全考虑，防止攻击者伪造TCP报文。

发送方在发送数据时，每个TCP报文段都包含该报文第一个字节的序号。接收方收到数据后，发送确认报文，ack字段表示期望收到的下一个字节的序号。例如，发送方发送了序号1001到2000的1000字节数据，接收方确认时应设置ack=2001，表示字节2001之前的数据都已收到。

TCP使用累计确认（ Cumulative Acknowledgment），即确认号表示该字节之前的所有数据都已收到，而不是确认收到了哪一段数据。这种方式简单高效，但无法确认非连续的数据块。当接收方收到乱序的数据时，它仍然发送对已收到连续数据的确认。

#### 3.2.2 超时重传机制

当发送方发送数据后，它会启动一个计时器（Retransmission Timer）。如果在计时器超时前收到确认，计时器取消；如果计时器超时，说明数据可能丢失，发送方会重传该数据。这个机制是TCP可靠传输的核心。

超时时间的设置非常关键。太短会导致不必要的重传，增加网络负载；太长会导致丢失数据恢复慢，影响性能。TCP使用自适应算法（Adaptive Retransmission）来动态计算超时时间。算法考虑往返时间（RTT）的变化，RTT是发送一个数据包到收到确认的时间。通过对RTT进行指数加权移动平均（EWMA），TCP估算当前的RTT，然后设置超时时间为估算RTT加上一定余量。

 Karn/Partridge算法和Jacobson/Karels算法是两种常用的RTT估算算法。Jacobson/Karels算法引入了平均偏差的概念，能够更快地适应RTT的变化，是目前广泛使用的算法。

#### 3.2.3 流量控制机制

TCP使用滑动窗口（Sliding Window）机制实现流量控制（Flow Control），防止发送方发送数据过快导致接收方无法处理。滑动窗口是接收方分配的一块缓冲区，用于存放已接收但尚未被上层应用读取的数据。

接收方在TCP头部中通告窗口大小（Window Size），表示接收方还有多少缓冲区空间。发送方根据窗口大小控制发送数据量，确保不发送超过接收方处理能力的数据。当接收方缓冲区接近满时，窗口大小会减小，甚至为0，表示暂停发送。

零窗口（Zero Window）是一种重要的流量控制手段。当接收方应用处理速度慢，导致缓冲区满时，它会通告窗口为0，发送方收到后停止发送数据。此时，发送方会启动探测定时器（Probe Timer），定期发送零窗口探测（Zero Window Probe）报文，询问接收方的窗口是否恢复。接收方收到探测后会回复当前的窗口大小。

#### 3.2.4 拥塞控制机制

拥塞控制（Congestion Control）是TCP最重要的特性之一，防止过多的数据注入网络，导致网络性能下降甚至拥塞崩溃。拥塞控制与流量控制不同，流量控制是端到端的，而拥塞控制涉及整个网络路径。

TCP的拥塞控制算法主要包括四个部分：慢启动（Slow Start）、拥塞避免（Congestion Avoidance）、快速重传（Fast Retransmit）和快速恢复（Fast Recovery）。

**慢启动**是TCP连接建立后或重传超时后的初始阶段。在这个阶段，拥塞窗口（cwnd）从较小的值开始，每收到一个确认就增加一个MSS（Maximum Segment Size，最大报文段长度）。慢启动使TCP的发送速率快速增长，直到达到慢启动阈值（ssthresh）或发生丢包。

**拥塞避免**发生在cwnd达到ssthresh后。在这个阶段，cwnd以线性速度增长，每经过一个RTT才增加一个MSS，比慢启动的增长速度慢很多。这是为了平稳地探索网络的可用带宽。

**快速重传**发生在接收方收到失序报文时。如果接收方收到一个失序的报文，它立即发送重复确认（Duplicate ACK），告诉发送方期望收到哪个序号。当发送方收到3个相同的重复确认时，它推断对应的报文可能丢失，立即重传该报文，而不必等待重传计时器超时。

**快速恢复**是快速重传后的恢复过程。当发送方执行快速重传后，它进入快速恢复状态，cwnd设置为ssthresh加上3个MSS（因为收到3个重复确认表明有3个报文段已离开网络），然后继续发送新数据（如果窗口允许），收到新的确认后退出快速恢复进入拥塞避免。

TCP还有两种不同的丢包检测机制，对应不同的拥塞控制行为：通过快速重传检测到丢包（三次重复确认），通常认为是非严重拥塞；通过重传计时器超时检测到丢包，通常认为是严重拥塞。超时导致的丢包会触发更严厉的惩罚：ssthresh设置为cwnd的一半，cwnd重置为1个MSS，重新进入慢启动。

### 3.3 UDP协议详解

#### 3.3.1 UDP协议特点

UDP（User Datagram Protocol，用户数据报协议）是TCP之外的另一个主要传输层协议。与TCP不同，UDP提供无连接的、不可靠的、基于数据报的传输服务。UDP的设计理念是简洁高效，尽量减少协议开销，适合对实时性要求高、能够容忍一定数据丢失的应用。

UDP的主要特点包括：无连接，通信前不需要建立连接，直接发送数据；不可靠，不保证数据能够到达目的地，不保证数据顺序，不进行重传；面向数据报，以报文为单位进行传输，每个UDP报文都有边界；开销小，UDP头部只有8个字节，而TCP头部至少20个字节；支持多播和广播，可以同时向多个接收方发送数据。

UDP头部只有4个字段，共8字节：源端口（16位）、目标端口（16位）、长度（16位，表示UDP头+数据的总长度）、校验和（16位，可选）。相比TCP的20字节固定头部加可选选项，UDP轻量得多。

#### 3.3.2 UDP的应用场景

UDP适用于对实时性要求高、能容忍少量数据丢失的应用场景。典型的应用包括：

**实时音视频通信**是UDP最典型的应用。视频会议、VoIP电话、视频直播等应用对延迟非常敏感，宁可偶尔丢帧也不愿等待重传。使用UDP可以快速发送音视频数据，即使丢失少量数据包，用户体验通常也比高延迟好。

**在线游戏**同样对延迟敏感。游戏需要实时更新玩家位置、动作等状态信息，偶尔的数据包丢失影响不大，但延迟过高会导致游戏卡顿不流畅。

**DNS查询**使用UDP53端口。DNS查询通常是简短的一问一答，TCP连接开销太大。DNS响应如果被截断，会回退到TCP。DNS服务器为了快速响应大量查询，UDP是更好的选择。

**物联网（IoT）设备**通常资源有限，使用UDP可以减少功耗和计算负担。许多物联网协议（如MQTT-SN、CoAP）都基于UDP。

**TFTP文件传输**使用UDP，适用于局域网内的简单文件传输，开销比FTP小。

#### 3.3.3 UDP与TCP的选择

选择UDP还是TCP需要根据应用的具体需求来判断。如果应用需要：可靠的数据传输、按序到达、流量控制、拥塞控制，那么TCP是更好的选择。如果应用对延迟敏感、能够容忍少量数据丢失、需要多播或广播支持，那么UDP可能更合适。

在实际开发中，很多应用会基于UDP实现自己的可靠传输机制，根据应用需求选择性地实现重传、排序、拥塞控制等功能。例如，QUIC协议基于UDP实现了可靠传输和拥塞控制；实时流媒体协议RTSP/RTP通常基于UDP，但包含自己的丢包补偿机制。

以下是使用Node.js的dgram模块创建UDP服务器和客户端的示例：

```javascript
// 引入dgram模块，用于创建UDP服务器和客户端
const dgram = require('dgram');

// 创建UDP服务器
const server = dgram.createSocket('udp4');

// 服务器监听消息事件
server.on('message', (msg, rinfo) => {
  console.log(`收到客户端消息: ${msg.toString()}`);
  console.log(`客户端地址: ${rinfo.address}:${rinfo.port}`);

  // 向客户端发送响应
  const response = Buffer.from(`服务器收到: ${msg}`);
  server.send(response, rinfo.port, rinfo.address, (err) => {
    if (err) {
      console.error('发送响应失败:', err);
    }
  });
});

// 服务器监听错误事件
server.on('error', (err) => {
  console.error('服务器错误:', err);
  server.close();
});

// 服务器监听监听事件
server.on('listening', () => {
  const address = server.address();
  console.log(`UDP服务器已启动，监听端口: ${address.port}`);
});

// 服务器绑定端口
server.bind(41234);

// 创建UDP客户端
const client = dgram.createSocket('udp4');

client.on('message', (msg, rinfo) => {
  console.log(`收到服务器响应: ${msg.toString()}`);
});

client.on('error', (err) => {
  console.error('客户端错误:', err);
});

// 发送消息给服务器
const message = Buffer.from('你好，服务器！');
client.send(message, 41234, 'localhost', (err) => {
  if (err) {
    console.error('发送消息失败:', err);
  } else {
    console.log('消息已发送');
  }
});

// 客户端绑定端口（系统自动分配）
client.bind(41235);
```

### 3.4 WebSocket与Socket.io

#### 3.4.1 WebSocket协议详解

WebSocket是一种在单个TCP连接上提供全双工通信的协议，解决了HTTP请求-响应模型的局限性。传统的HTTP通信是客户端主动请求、服务器被动响应的模式，服务器无法主动向客户端推送数据。WebSocket通过HTTP升级握手建立连接后，双方可以随时互相发送数据，非常适合实时性要求高的应用。

WebSocket的设计目标是：减少不必要的网络开销和延迟；支持双向通信；穿越防火墙和代理服务器；提供Web应用的实时交互能力。

WebSocket的建立过程始于HTTP请求。客户端发送一个特殊的HTTP请求，包含Upgrade头部，请求将连接升级为WebSocket协议：

```
GET /ws HTTP/1.1
Host: api.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
```

服务器如果支持WebSocket，响应101状态码表示协议切换：

```
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

握手完成后，连接从HTTP协议切换为WebSocket协议，双方可以自由地发送数据帧。WebSocket数据帧比HTTP头部紧凑得多，开销很小。

WebSocket数据帧有两种类型：文本帧和二进制帧。文本帧包含UTF-8编码的文本数据；二进制帧可以发送任意二进制数据。WebSocket还支持控制帧（Ping、Pong、Close），用于心跳检测和连接管理。

WebSocket的一个限制是同源策略的限制。浏览器中的WebSocket只能连接到同源服务器，除非服务器明确允许跨域连接。服务器通过检查Origin头部并设置Access-Control-Allow-Origin响应头来控制哪些来源可以连接。

#### 3.4.2 Socket.io库详解

Socket.io是一个流行的WebSocket封装库，提供了更方便的API、自动重连、房间管理、广播等高级功能。Socket.io在WebSocket不可用的环境中会回退到轮询（Polling）等其他技术，确保连接的可靠性。

Socket.io的主要特性包括：自动回退，在WebSocket不可用时回退到轮询；自动重连，连接断开后自动尝试重连；房间（Rooms）支持，可以在逻辑上对连接进行分组；广播（Broadcast），向房间内的所有连接或除发送者外的所有连接发送消息；中间件（Middleware），在连接和处理消息时可以插入中间件进行认证、日志等处理；跨浏览器和跨设备支持。

Socket.io使用自定义的协议，不兼容标准的WebSocket协议。如果客户端使用Socket.io，服务器端也必须使用Socket.io，反之亦然。对于需要与标准WebSocket互通的场景，应该直接使用WebSocket API。

Socket.io的命名空间（Namespace）功能允许在同一个连接上创建多个通信通道，每个通道可以独立订阅和广播。例如，可以创建/system和/user两个命名空间，分别处理系统消息和用户消息。

以下是在Node.js中使用Socket.io实现实时聊天的示例：

```javascript
// 引入Express框架和Socket.io
const express = require('express');
const { createServer } = require('http');
const { Server } = require('socket.io');

const app = express();
const httpServer = createServer(app);
const io = new Server(httpServer, {
  cors: {
    origin: '*',  // 允许所有来源（生产环境应设置具体域名）
    methods: ['GET', 'POST']
  }
});

// 静态文件服务
app.use(express.static('public'));

// Socket.io连接处理
io.on('connection', (socket) => {
  console.log('用户连接:', socket.id);

  // 加入房间
  socket.on('join-room', (roomId) => {
    socket.join(roomId);
    console.log(`用户 ${socket.id} 加入房间 ${roomId}`);
    // 通知房间内其他人
    socket.to(roomId).emit('user-joined', {
      userId: socket.id,
      message: '新用户加入了房间'
    });
  });

  // 离开房间
  socket.on('leave-room', (roomId) => {
    socket.leave(roomId);
    console.log(`用户 ${socket.id} 离开房间 ${roomId}`);
    socket.to(roomId).emit('user-left', {
      userId: socket.id,
      message: '用户离开了房间'
    });
  });

  // 发送聊天消息
  socket.on('chat-message', (data) => {
    console.log('收到消息:', data);
    // 广播消息给房间内所有人，包括发送者
    io.to(data.roomId).emit('new-message', {
      userId: socket.id,
      message: data.message,
      timestamp: new Date().toISOString()
    });
  });

  // 发送私信
  socket.on('private-message', (data) => {
    // data包含 { to: targetSocketId, message: text }
    io.to(data.to).emit('private-message', {
      from: socket.id,
      message: data.message,
      timestamp: new Date().toISOString()
    });
  });

  // 广播事件（所有连接的客户端）
  socket.on('broadcast', (data) => {
    io.emit('broadcast-message', {
      userId: socket.id,
      message: data.message
    });
  });

  // 断开连接
  socket.on('disconnect', () => {
    console.log('用户断开连接:', socket.id);
    // 通知所有房间的用户该用户已离开
    socket.rooms.forEach((room) => {
      if (room !== socket.id) {
        socket.to(room).emit('user-left', {
          userId: socket.id,
          message: '用户断开了连接'
        });
      }
    });
  });
});

// 中间件示例：认证
io.use((socket, next) => {
  const token = socket.handshake.auth.token;
  if (token === 'valid-token') {
    next(); // 认证通过
  } else {
    next(new Error('认证失败')); // 认证失败
  }
});

httpServer.listen(3000, () => {
  console.log('Socket.io服务器已启动，监听端口: 3000');
});
```

#### 3.4.3 客户端JavaScript代码

```javascript
// 客户端Socket.io使用示例
const socket = io('http://localhost:3000', {
  auth: {
    token: 'valid-token'  // 认证令牌
  }
});

// 连接成功
socket.on('connect', () => {
  console.log('已连接到服务器，Socket ID:', socket.id);
});

// 连接错误
socket.on('connect_error', (error) => {

});

// 断开连接
socket.on('disconnect', (reason) => {
  console.log('断开连接:', reason);
});

// 加入房间
function joinRoom(roomId) {
  socket.emit('join-room', roomId);
}

// 发送消息
function sendMessage(roomId, message) {
  socket.emit('chat-message', { roomId, message });
}

// 发送私信
function sendPrivateMessage(targetId, message) {
  socket.emit('private-message', { to: targetId, message });
}

// 发送广播
function sendBroadcast(message) {
  socket.emit('broadcast', { message });
}

// 监听新消息
socket.on('new-message', (data) => {
  console.log('收到群组消息:', data);
  addMessageToUI(data.userId, data.message, data.timestamp);
});

// 监听私信
socket.on('private-message', (data) => {
  console.log('收到私信:', data);
  addPrivateMessageToUI(data.from, data.message, data.timestamp);
});

// 监听广播
socket.on('broadcast-message', (data) => {
  console.log('收到广播:', data);
});
```

## 四、网络安全

### 4.1 XSS跨站脚本攻击

#### 4.1.1 XSS攻击原理

XSS（Cross-Site Scripting，跨站脚本攻击）是Web应用中最常见的安全漏洞之一。XSS攻击允许攻击者在受害者的浏览器中执行恶意JavaScript代码，从而窃取会话令牌、篡改网页内容、进行钓鱼攻击等。XSS之所以称为"跨站"脚本攻击，是因为攻击代码可以来自不同的网站，通过注入到目标网站的页面中执行。

XSS攻击的本质是Web应用对用户输入的数据没有进行充分的过滤和转义。当用户提交的数据被浏览器解析执行时，如果其中包含恶意代码，浏览器会像执行正常脚本一样执行这些恶意代码。浏览器的同源策略（SOP）虽然限制了不同源的脚本访问彼此的数据，但XSS攻击通过在目标网站执行脚本，实际上已经绕过了同源策略，因为恶意脚本是在目标网站的上下文中执行的。

XSS攻击可以造成多种危害：窃取用户Cookie和会话令牌，冒充用户身份进行操作；读取用户敏感信息（如键盘记录、表单数据）；篡改网页内容，插入虚假表单进行钓鱼；利用浏览器漏洞进行更严重的攻击（如安装恶意软件）。

#### 4.1.2 XSS攻击类型

**存储型XSS（Stored XSS）** 是最严重的XSS攻击类型。攻击者将恶意代码提交到服务器（如数据库），服务器将其存储。当其他用户访问包含恶意代码的页面时，恶意代码被从服务器取出并执行。典型的场景是用户评论、论坛帖子、社交网络帖子等。存储型XSS的危害最大，因为恶意代码会被永久存储，所有访问该内容的用户都会受到攻击。

**反射型XSS（Reflected XSS）** 是指恶意代码作为用户请求的一部分被服务器接收，然后立即反射回响应中，响应中包含这段恶意代码，浏览器执行后攻击生效。典型的场景是URL参数、搜索框输入等。攻击者通常诱骗用户点击一个包含恶意代码的链接，当用户点击后，恶意代码作为搜索结果或错误信息被浏览器执行。反射型XSS需要用户主动触发，危害相对较小。

**DOM型XSS（DOM-based XSS）** 是一种特殊的XSS攻击，恶意代码的注入和执行完全发生在客户端，通过操作DOM对象实现。服务器返回的页面本身不包含恶意代码，但页面中的JavaScript代码在处理用户输入时，动态地将恶意代码插入到DOM中执行。这种攻击完全客户端化，服务器端日志中可能完全看不到攻击痕迹，更难检测。

#### 4.1.3 XSS防护措施

防护XSS攻击需要从多个层面进行：输入过滤、输出转义、Content Security Policy、HttpOnly Cookie等。

**输入过滤**是在数据进入服务器时进行验证和清洗，拒绝或移除可疑的输入。但输入过滤很难做到完全准确，容易被绕过，因此不应作为主要的防护手段。

**输出转义**是在数据输出到HTML时进行转义，将特殊字符转换为HTML实体。例如，将<转换为&lt;，将>转换为&gt;，将"转换为&quot;，将'转换为&#x27;，将&转换为&amp;。这是防护XSS最有效的方法，确保浏览器将用户输入的数据作为文本内容而不是可执行代码。大多数现代Web框架都提供了自动的输出转义功能。

```javascript
// Node.js中使用he库进行HTML转义
const he = require('he');

// 手动转义函数
function escapeHtml(text) {
  return he.encode(text, {
    useNamedReferences: true,
    escapeEverything: false
  });
}

// 示例
const userInput = '<script>alert("XSS")</script>';
const safeOutput = escapeHtml(userInput);
console.log(safeOutput);
// 输出: &lt;script&gt;alert(&quot;XSS&quot;)&lt;/script&gt;
```

**Content Security Policy（CSP）** 是一种HTTP响应头，允许网站管理员控制哪些外部资源可以被加载和执行。通过设置CSP，网站可以禁止内联脚本执行（inline-script），禁止外域脚本加载，从而阻止XSS攻击即使恶意代码被注入也无法执行。

```http
# 严格的CSP策略示例
Content-Security-Policy:
  default-src 'self';                    # 默认只允许同源资源
  script-src 'self' 'nonce-{random}';     # 只允许同源脚本，并使用nonce
  style-src 'self' 'unsafe-inline';       # 允许内联样式（某些CSS-in-JS需要）
  img-src 'self' data: https:;           # 允许同源和HTTPS图片
  connect-src 'self' https://api.example.com;  # 限制API请求目标
  frame-ancestors 'none';                # 禁止被iframe嵌入
```

**HttpOnly Cookie** 是防止XSS窃取会话的最终防线。通过设置HttpOnly属性，Cookie无法通过JavaScript的document.cookie访问，即使页面被XSS注入，攻击者也无法读取Cookie。

```javascript
// Express中设置HttpOnly Cookie
res.cookie('session_id', 'abc123', {
  httpOnly: true,  // 禁止JavaScript访问
  secure: true,    // 仅HTTPS传输
  sameSite: 'strict'  // 防CSRF
});
```

### 4.2 CSRF跨站请求伪造

#### 4.2.1 CSRF攻击原理

CSRF（Cross-Site Request Forgery，跨站请求伪造）是一种利用用户已认证的身份，以用户的身份向Web应用发送非预期的请求的攻击方式。与XSS不同，CSRF不直接在用户浏览器中执行代码，而是利用浏览器的Cookie机制和Web应用对请求来源验证的不完善。

CSRF攻击的原理是：当用户登录网站A并获取有效的会话Cookie后，攻击者诱骗用户访问一个恶意网页，该网页中包含自动发送的请求到网站A。由于浏览器会自动发送Cookie，网站A会认为这个请求是用户主动发送的，从而以用户的身份执行操作。攻击者无法获取Cookie的内容，但可以利用Cookie的身份认证功能。

CSRF攻击的前提条件包括：用户已经登录目标网站并拥有有效的会话；目标网站接受来自用户浏览器的请求；攻击者能够诱导用户访问恶意网页。CSRF攻击的成功率取决于目标网站对请求来源的验证程度。

CSRF攻击可以造成的危害包括：以用户名义发送邮件、转账、修改密码、删除数据等。攻击的严重程度取决于被攻击网站的功能。例如，如果银行网站存在CSRF漏洞，攻击者可以诱骗用户转账。

#### 4.2.2 CSRF防护措施

**CSRF Token** 是最常用的CSRF防护机制。服务器在渲染表单时生成一个随机的CSRF Token，并将其包含在表单中作为一个隐藏字段，同时存储在用户的Session中。当表单提交时，服务器检查请求中的Token与Session中的Token是否匹配。由于攻击者无法获取Token的值（需要通过目标网站的页面获取），伪造的请求将缺少正确的Token。

```javascript
// Express中使用csurf中间件防护CSRF
const csrf = require('csurf');
const cookieParser = require('cookie-parser');

app.use(cookieParser());

// CSRF保护中间件
const csrfProtection = csrf({
  cookie: {
    httpOnly: true,
    secure: true
  }
});

// 应用CSRF保护到需要防护的路由
app.post('/transfer', csrfProtection, (req, res) => {
  // 处理转账请求
  const { toAccount, amount } = req.body;
  // ...转账逻辑
  res.json({ success: true, message: '转账成功' });
});

// 渲染表单页面时传递CSRF Token
app.get('/transfer', (req, res) => {
  // 生成CSRF Token并传递到模板
  res.render('transfer', {
    csrfToken: req.csrfToken(),
    user: req.user
  });
});
```

```html
<!-- 表单中包含CSRF Token -->
<form action="/transfer" method="POST">
  <input type="hidden" name="_csrf" value="{{csrfToken}}">
  <input type="text" name="toAccount" placeholder="目标账户">
  <input type="number" name="amount" placeholder="金额">
  <button type="submit">转账</button>
</form>
```

**SameSite Cookie** 是现代浏览器支持的Cookie属性，可以有效防止CSRF攻击。SameSite有三个可选值：Strict（仅允许同站请求发送Cookie）、Lax（允许部分安全的高导航请求发送Cookie）、None（允许所有请求发送Cookie，但必须配合Secure属性）。将Cookie的SameSite属性设置为Strict或Lax可以阻止大多数CSRF攻击。

```javascript
// 设置SameSite Cookie
res.cookie('session_id', 'abc123', {
  httpOnly: true,
  secure: true,
  sameSite: 'strict'  // 防止CSRF
});
```

**验证请求来源（Origin/Referer）** 是另一个常用的防护方法。服务器检查请求头中的Origin或Referer字段，确认请求是否来自可信的来源。但这种方法不够可靠，因为这些头部可以被伪造或被某些浏览器省略。

**双重提交Cookie** 是一种无状态的CSRF防护方法。将CSRF Token同时在Cookie和请求头/表单字段中发送，服务器验证两者是否匹配。由于浏览器会自动发送Cookie，但不会自动发送自定义请求头，这种方法可以有效防护CSRF。

### 4.3 SQL注入

#### 4.3.1 SQL注入原理

SQL注入（SQL Injection）是一种将恶意SQL代码插入到应用程序的SQL查询中，从而操纵数据库的攻击方式。当应用程序将用户输入直接拼接到SQL语句中而没有进行适当过滤或参数化查询时，攻击者可以利用这个漏洞执行任意SQL语句，包括查询、修改、删除数据库中的数据。

SQL注入的原理是：应用程序使用字符串拼接或字符串格式化构建SQL语句，将用户输入直接插入SQL语句中。例如，以下Node.js代码存在SQL注入漏洞：

```javascript
// 不安全的SQL查询 - 存在SQL注入风险
app.get('/search', (req, res) => {
  const searchTerm = req.query.term;
  // 直接将用户输入拼接到SQL语句
  const query = `SELECT * FROM products WHERE name LIKE '%${searchTerm}%'`;

  db.query(query, (err, results) => {
    if (err) {
      return res.status(500).send('查询失败');
    }
    res.json(results);
  });
});
```

如果攻击者输入term为`%' OR '1'='1`，SQL语句会变成：

```sql
SELECT * FROM products WHERE name LIKE '%%' OR '1'='1'
```

这会返回products表中的所有记录，而不仅仅是匹配搜索条件的产品。更危险的注入可能输入`'; DROP TABLE products; --`，直接删除整个表。

SQL注入可以造成的危害包括：数据泄露（查询数据库中的敏感信息）；数据篡改（修改或删除数据）；数据破坏（DROP TABLE等）；绕过认证（永远返回真条件）；远程代码执行（在某些数据库上）。

#### 4.3.2 SQL注入防护措施

**使用参数化查询（Parameterized Query）** 是防止SQL注入的最有效方法。参数化查询使用占位符代替用户输入，数据库驱动会先将查询结构发送给数据库编译，然后将用户输入作为安全的参数值发送，不会被当作SQL代码执行。

```javascript
// 使用参数化查询防止SQL注入
app.get('/search', (req, res) => {
  const searchTerm = req.query.term;
  // 使用?占位符，参数单独传递
  const query = 'SELECT * FROM products WHERE name LIKE ?';
  const params = [`%${searchTerm}%`];

  db.query(query, params, (err, results) => {
    if (err) {
      return res.status(500).send('查询失败');
    }
    res.json(results);
  });
});
```

```javascript
// 使用ORM（如Sequelize）进一步降低SQL注入风险
const { Op } = require('sequelize');

app.get('/search', async (req, res) => {
  try {
    const searchTerm = req.query.term;
    const products = await Product.findAll({
      where: {
        name: {
          [Op.like]: `%${searchTerm}%`
        }
      }
    });
    res.json(products);
  } catch (err) {
    res.status(500).send('查询失败');
  }
});
```

**输入验证（Input Validation）** 是第二道防线。对用户输入进行严格的验证，拒绝不符合预期格式的输入。例如，如果搜索词应该是字母数字，可以验证输入只包含字母和数字。

```javascript
// 输入验证示例
function validateSearchTerm(term) {
  // 只允许字母、数字、空格和基本标点
  const validPattern = /^[a-zA-Z0-9\s.,'-]+$/;
  return validPattern.test(term);
}

app.get('/search', (req, res) => {
  const searchTerm = req.query.term;

  if (!validateSearchTerm(searchTerm)) {
    return res.status(400).send('无效的搜索词');
  }

  // 执行查询...
});
```

**最小权限原则（Principle of Least Privilege）** 是数据库安全的重要原则。应用程序使用的数据库账户应该只拥有必要的权限，不要使用管理员账户。如果应用程序只需要读取数据，就不应该给予写权限。

**使用存储过程（Stored Procedure）** 可以在一定程度上防止SQL注入，因为存储过程中的SQL语句预先编译，但需要注意在存储过程中不要使用动态SQL。

### 4.4 中间人攻击

#### 4.4.1 中间人攻击原理

中间人攻击（Man-in-the-Middle Attack，MITM）是一种攻击者位于通信双方之间的攻击方式，攻击者可以在两方不知情的情况下拦截、篡改或伪造通信内容。中间人攻击可以发生在任何网络通信中，包括有线网络、无线网络、局域网等。

中间人攻击的基本流程是：攻击者首先处于通信双方的通信路径上，可能通过ARP欺骗、DNS劫持、WiFi嗅探等方式实现；然后攻击者分别与通信双方建立独立的连接，将自己伪装成双方各自的对端；在数据传输过程中，攻击者可以拦截、查看、修改数据，然后转发给真正的接收方。

中间人攻击可以造成的危害包括：窃听敏感信息（如密码、银行账户）；篡改传输数据（如注入恶意代码）；会话劫持（窃取认证会话）；SSL剥离（将HTTPS降级为HTTP）。

在公共WiFi环境中，中间人攻击尤其常见。攻击者可以设置一个与合法WiFi同名的恶意热点，用户连接后所有通信都会经过攻击者。攻击者可以查看和修改所有未加密的流量。

#### 4.4.2 中间人攻击防护措施

**使用HTTPS** 是防止中间人攻击的最基本措施。HTTPS使用TLS加密传输内容，即使攻击者能拦截通信，也无法查看或篡改加密的内容。确保网站使用有效的SSL/TLS证书，并启用HSTS（HTTP Strict Transport Security）。

**验证证书** 是HTTPS安全的关键。客户端应该验证服务器的证书是否由可信的CA签发、是否在有效期内、证书的域名是否匹配。浏览器的证书验证机制已经内置，但开发自制应用时需要正确实现证书验证。

```javascript
// Node.js中验证SSL证书
const https = require('https');
const fs = require('fs');

// 加载受信任的CA证书
const caCert = fs.readFileSync('/path/to/ca-cert.pem');

const options = {
  hostname: 'api.example.com',
  port: 443,
  path: '/data',
  method: 'GET',
  ca: caCert,  // 指定可信CA
  rejectUnauthorized: true  // 拒绝未授权的证书
};

const req = https.request(options, (res) => {
  console.log('状态码:', res.statusCode);
  res.on('data', (d) => {
    process.stdout.write(d);
  });
});

req.on('error', (e) => {
  console.error('请求错误:', e);
});

req.end();
```

**公钥钉扎（Public Key Pinning）** 是更高级的防护措施。网站可以声明自己只信任特定的公钥或CA，防止攻击者使用伪造证书进行中间人攻击。但公钥钉扎配置复杂，一旦证书更换不当会导致站点不可访问，目前大多数网站已不再使用。

**避免在公共WiFi上进行敏感操作** 是用户应该养成的好习惯。如果必须在公共网络进行敏感操作，使用VPN建立加密隧道是最安全的做法。

**检测ARP欺骗** 是局域网内的防护措施。可以使用arp -a命令检查ARP表中的MAC地址是否异常，或使用专门的ARP防火墙软件检测和阻止ARP欺骗攻击。

### 4.5 HTTPS安全最佳实践

#### 4.5.1 配置安全的HTTPS

正确配置HTTPS不仅需要安装证书，还需要配置合适的TLS版本、密码套件和其他安全选项。使用TLS 1.3是最安全的选择，如果必须兼容旧客户端，可以使用TLS 1.2，但应禁用TLS 1.0和TLS 1.1。

以下是Nginx中安全的HTTPS配置示例：

```nginx
server {
    listen 443 ssl http2;
    server_name www.example.com;

    # 证书配置
    ssl_certificate /path/to/fullchain.pem;
    ssl_certificate_key /path/to/privkey.pem;

    # TLS版本配置 - 只启用TLS 1.2和1.3
    ssl_protocols TLSv1.3 TLSv1.2;

    # 安全密码套件配置
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers on;

    # 启用OCSP Stapling
    ssl_stapling on;
    ssl_stapling_verify on;
    resolver 8.8.8.8 8.8.4.4 valid=300s;
    resolver_timeout 5s;

    # 安全响应头
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # HSTS - 强制使用HTTPS
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;

    # CSP策略
    add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' https://api.example.com; frame-ancestors 'none'" always;
}
```

#### 4.5.2 常见的HTTPS安全问题

**混合内容（Mixed Content）** 是指HTTPS页面中加载了HTTP资源。攻击者可能通过中间人攻击替换HTTP资源为恶意代码。应将所有资源链接改为HTTPS或使用协议相对URL（//example.com/resource.js）。

**过期证书** 是常见问题，导致用户看到安全警告。应使用自动化工具（如Let's Encrypt + Certbot）管理证书的续期。

**自签名证书** 在测试环境中有用，但在生产环境中会导致浏览器不信任。应使用受信任CA签发的证书。

**不完整的证书链** 会导致部分客户端无法验证证书。服务器应配置完整的证书链，包括中间证书。

## 五、实战技巧

### 5.1 Wireshark抓包分析

#### 5.1.1 Wireshark基础使用

Wireshark是功能最强大的网络协议分析工具，可以捕获和分析网络中的数据包，帮助理解网络通信的工作原理和排查网络问题。Wireshark支持数百种协议的分析，是网络工程师和安全专家的必备工具。

Wireshark的基本使用流程包括：选择网络接口进行捕获、设置捕获过滤器、开始捕获、分析捕获的数据包、应用显示过滤器进一步分析。

**选择捕获接口**：启动Wireshark后，会显示可用的网络接口列表。选择要捕获流量的接口，如以太网或WiFi。如果在Windows上使用，可能需要安装WinPcap或Npcap驱动。

**捕获过滤器（Capture Filter）** 在捕获时过滤数据包，减少捕获量。捕获过滤器使用BPF（Berkeley Packet Filter）语法。常用的捕获过滤器包括：host 192.168.1.1（只捕获与指定IP相关的数据包）、port 80（只捕获80端口的数据包）、tcp（只捕获TCP数据包）等。

```bash
# 捕获过滤器语法示例
# 只捕获与192.168.1.100的通信
host 192.168.1.100

# 只捕获HTTP和HTTPS流量
port 80 or port 443

# 只捕获TCP流量
tcp

# 排除某IP的流量
not host 192.168.1.1

# 组合条件
tcp and (port 80 or port 443) and host 192.168.1.100
```

**显示过滤器（Display Filter）** 在已捕获的数据包中过滤，使用更丰富的语法。显示过滤器是Wireshark最强大的功能之一，支持协议字段的精确匹配。

```bash
# 显示过滤器语法示例
# 按IP地址过滤
ip.addr == 192.168.1.1

# 按端口过滤
tcp.port == 443

# 按协议过滤
http

# 组合条件
http.request.method == "GET" and ip.addr == 192.168.1.1

# 过滤HTTP响应
http.response.code == 200

# 显示TCP重传
tcp.analysis.retransmission

# 显示握手和挥手
tcp.flags.syn == 1 or tcp.flags.fin == 1
```

#### 5.1.2 分析TCP三次握手和四次挥手

使用Wireshark分析TCP连接建立和关闭的过程，是理解TCP协议的重要方法。

**分析三次握手**：在Wireshark中输入显示过滤器`tcp.flags.syn == 1 or tcp.flags.synack == 1 or tcp.flags.ack == 1`，可以看到握手过程。三次握手的特征是：第一个包（SYN）从客户端到服务器，标志位SYN=1；第二个包（SYN+ACK）从服务器到客户端，标志位SYN=1, ACK=1；第三个包（ACK）从客户端到服务器，标志位ACK=1。每个TCP包都有Sequence Number和Acknowledgment Number，可以追踪数据的传输。

**分析四次挥手**：输入显示过滤器`tcp.flags.fin == 1 or (tcp.flags.ack == 1 and tcp.len == 0)`，可以看到挥手过程。四次挥手通常是：主动关闭方发送FIN，进入FIN_WAIT_1；接收方发送ACK，进入CLOSE_WAIT；接收方发送FIN，进入LAST_ACK；主动关闭方发送ACK，进入TIME_WAIT。

#### 5.1.3 分析HTTP/HTTPS流量

**HTTP明文流量分析**：如果服务器和客户端之间未加密，可以在Wireshark中直接看到HTTP请求和响应的内容。输入显示过滤器`http`，可以看到所有HTTP流量。展开HTTP协议的详细信息，可以查看请求方法、URI、响应状态码、头部信息等。右键点击HTTP包，选择"Follow" -> "TCP Stream"，可以看到完整的HTTP会话内容。

**HTTPS流量分析**：HTTPS流量是加密的，直接看是乱码。但通过导出TLS密钥并导入Wireshark，可以解密HTTPS流量查看内容。在Firefox或Chrome中设置环境变量SSLKEYLOGFILE指向一个文件，浏览器会将TLS会话密钥写入该文件。在Wireshark中通过Edit -> Preferences -> Protocols -> TLS，设置"(Pre)-Master-Secret log filename"为该文件，之后就能解密HTTPS流量。

### 5.2 curl命令调试HTTP

#### 5.2.1 curl基础用法

curl是一个强大的命令行HTTP客户端，支持各种HTTP协议操作，是调试Web服务和API的必备工具。curl的基本语法是`curl [options] [URL]`，可以发送各种类型的HTTP请求并显示服务器响应。

**发送GET请求**是最基本的用法：

```bash
# 基本GET请求
curl https://api.example.com/users

# 显示响应头
curl -i https://api.example.com/users

# 只显示响应头
curl -I https://api.example.com/users

# 下载文件并保存
curl -o filename.zip https://api.example.com/file.zip

# 显示完整请求和响应（包括请求头）
curl -v https://api.example.com/users
```

**发送POST请求**需要指定方法和数据：

```bash
# 发送表单数据
curl -X POST -d "username=admin&password=123456" https://api.example.com/login

# 发送JSON数据
curl -X POST -H "Content-Type: application/json" \
     -d '{"username":"admin","password":"123456"}' \
     https://api.example.com/api/login

# 发送文件数据
curl -X POST -F "file=@/path/to/file.txt" https://api.example.com/upload
```

#### 5.2.2 curl高级用法

**自定义请求头**：

```bash
# 添加自定义头部
curl -H "Authorization: Bearer token123" \
     -H "X-Custom-Header: value" \
     https://api.example.com/protected

# 模拟不同浏览器
curl -A "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36" \
     https://api.example.com
```

**处理Cookie**：

```bash
# 保存Cookie到文件
curl -c cookies.txt https://api.example.com/login

# 使用保存的Cookie
curl -b cookies.txt https://api.example.com/profile

# 发送Cookie
curl -b "session_id=abc123" https://api.example.com/profile
```

**处理重定向**：

```bash
# 自动跟随重定向（默认不跟随）
curl -L https://api.example.com/redirect

# 显示重定向过程
curl -L -v https://api.example.com/redirect
```

**处理SSL证书**：

```bash
# 忽略证书验证（用于测试）
curl -k https://api.example.com

# 指定CA证书
curl --cacert /path/to/ca.pem https://api.example.com

# 显示证书信息
curl -Iv https://api.example.com
```

**限速和断点续传**：

```bash
# 限制下载速度
curl --limit-rate 100K -o file.zip https://api.example.com/file.zip

# 断点续传
curl -C - -o file.zip https://api.example.com/file.zip
```

### 5.3 Node.js实现简单HTTP服务器

#### 5.3.1 原生Node.js HTTP服务器

Node.js内置了http模块，可以轻松创建HTTP服务器。以下是一个完整的HTTP服务器实现，支持静态文件服务和简单的API：

```javascript
// 引入Node.js内置模块
const http = require('http');
const fs = require('fs');
const path = require('path');
const url = require('url');

// 定义MIME类型映射
const MIME_TYPES = {
  '.html': 'text/html',
  '.css': 'text/css',
  '.js': 'application/javascript',
  '.json': 'application/json',
  '.png': 'image/png',
  '.jpg': 'image/jpeg',
  '.gif': 'image/gif',
  '.svg': 'image/svg+xml',
  '.ico': 'image/x-icon',
  '.pdf': 'application/pdf',
  '.txt': 'text/plain'
};

// 创建HTTP服务器
const server = http.createServer((req, res) => {
  // 解析请求URL
  const parsedUrl = url.parse(req.url, true);
  const pathname = parsedUrl.pathname;
  const query = parsedUrl.query;

  // 获取请求方法
  const method = req.method;

  // 日志记录
  console.log(`${new Date().toISOString()} ${method} ${pathname}`);

  // CORS头设置（开发环境）
  res.setHeader('Access-Control-Allow-Origin', '*');
  res.setHeader('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE, OPTIONS');
  res.setHeader('Access-Control-Allow-Headers', 'Content-Type, Authorization');

  // 处理预检请求
  if (method === 'OPTIONS') {
    res.writeHead(204);
    res.end();
    return;
  }

  // API路由处理
  if (pathname.startsWith('/api/')) {
    handleApiRequest(req, res, pathname, method, query);
    return;
  }

  // 静态文件服务
  serveStaticFile(req, res, pathname);
});

// API请求处理函数
function handleApiRequest(req, res, pathname, method, query) {
  // 设置JSON响应头
  res.setHeader('Content-Type', 'application/json');

  // 用户列表API
  if (pathname === '/api/users' && method === 'GET') {
    const users = [
      { id: 1, name: '张三', email: 'zhangsan@example.com' },
      { id: 2, name: '李四', email: 'lisi@example.com' },
      { id: 3, name: '王五', email: 'wangwu@example.com' }
    ];
    res.writeHead(200);
    res.end(JSON.stringify({ success: true, data: users }));
    return;
  }

  // 用户详情API
  const userMatch = pathname.match(/^\/api\/users\/(\d+)$/);
  if (userMatch && method === 'GET') {
    const userId = parseInt(userMatch[1]);
    const user = { id: userId, name: '用户' + userId, email: `user${userId}@example.com` };
    res.writeHead(200);
    res.end(JSON.stringify({ success: true, data: user }));
    return;
  }

  // 创建用户API
  if (pathname === '/api/users' && method === 'POST') {
    let body = '';
    req.on('data', chunk => {
      body += chunk.toString();
    });
    req.on('end', () => {
      try {
        const data = JSON.parse(body);
        const newUser = {
          id: Date.now(),
          name: data.name || '未命名',
          email: data.email || ''
        };
        res.writeHead(201);
        res.end(JSON.stringify({ success: true, data: newUser }));
      } catch (err) {
        res.writeHead(400);
        res.end(JSON.stringify({ success: false, message: '无效的JSON数据' }));
      }
    });
    return;
  }

  // 404 Not Found
  res.writeHead(404);
  res.end(JSON.stringify({ success: false, message: 'API不存在' }));
}

// 静态文件服务函数
function serveStaticFile(req, res, pathname) {
  // 默认文件
  if (pathname === '/') {
    pathname = '/index.html';
  }

  // 文件路径
  const filePath = path.join(__dirname, 'public', pathname);

  // 获取文件扩展名和MIME类型
  const ext = path.extname(filePath);
  const contentType = MIME_TYPES[ext] || 'application/octet-stream';

  // 读取并发送文件
  fs.readFile(filePath, (err, content) => {
    if (err) {
      if (err.code === 'ENOENT') {
        // 文件不存在
        res.writeHead(404);
        res.end('<h1>404 Not Found</h1>');
      } else {
        // 服务器错误
        res.writeHead(500);
        res.end('<h1>500 Internal Server Error</h1>');
      }
      return;
    }

    // 发送文件
    res.writeHead(200, { 'Content-Type': contentType });
    res.end(content);
  });
}

// 启动服务器
const PORT = process.env.PORT || 3000;
server.listen(PORT, () => {
  console.log(`HTTP服务器已启动，监听端口: ${PORT}`);
  console.log(`访问地址: http://localhost:${PORT}`);
});
```

#### 5.3.2 使用Express框架构建REST API

对于生产级应用，推荐使用Express框架，它提供了更简洁的API和丰富的中间件生态系统：

```javascript
// 引入Express框架
const express = require('express');
const cors = require('cors');
const helmet = require('helmet');
const morgan = require('morgan');
const rateLimit = require('express-rate-limit');

const app = express();
const PORT = process.env.PORT || 3000;

// 安全中间件
app.use(helmet());  // 设置安全HTTP头
app.use(cors());    // 启用CORS

// 日志中间件
app.use(morgan('combined'));

// 解析JSON请求体
app.use(express.json());

// 速率限制（防止DDoS和暴力破解）
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15分钟
  max: 100, // 每个IP最多100个请求
  message: { success: false, message: '请求过于频繁，请稍后再试' }
});
app.use('/api', limiter);

// 模拟用户数据
let users = [
  { id: 1, name: '张三', email: 'zhangsan@example.com', role: 'admin' },
  { id: 2, name: '李四', email: 'lisi@example.com', role: 'user' },
  { id: 3, name: '王五', email: 'wangwu@example.com', role: 'user' }
];

// 获取所有用户
app.get('/api/users', (req, res) => {
  res.json({ success: true, data: users });
});

// 获取单个用户
app.get('/api/users/:id', (req, res) => {
  const userId = parseInt(req.params.id);
  const user = users.find(u => u.id === userId);

  if (!user) {
    return res.status(404).json({ success: false, message: '用户不存在' });
  }

  res.json({ success: true, data: user });
});

// 创建用户
app.post('/api/users', (req, res) => {
  const { name, email, role = 'user' } = req.body;

  // 验证输入
  if (!name || !email) {
    return res.status(400).json({ success: false, message: '姓名和邮箱是必填项' });
  }

  // 检查邮箱唯一性
  if (users.some(u => u.email === email)) {
    return res.status(400).json({ success: false, message: '邮箱已被使用' });
  }

  const newUser = {
    id: users.length > 0 ? Math.max(...users.map(u => u.id)) + 1 : 1,
    name,
    email,
    role
  };

  users.push(newUser);
  res.status(201).json({ success: true, data: newUser });
});

// 更新用户
app.put('/api/users/:id', (req, res) => {
  const userId = parseInt(req.params.id);
  const userIndex = users.findIndex(u => u.id === userId);

  if (userIndex === -1) {
    return res.status(404).json({ success: false, message: '用户不存在' });
  }

  const { name, email, role } = req.body;

  // 更新用户数据
  users[userIndex] = {
    ...users[userIndex],
    name: name || users[userIndex].name,
    email: email || users[userIndex].email,
    role: role || users[userIndex].role
  };

  res.json({ success: true, data: users[userIndex] });
});

// 删除用户
app.delete('/api/users/:id', (req, res) => {
  const userId = parseInt(req.params.id);
  const userIndex = users.findIndex(u => u.id === userId);

  if (userIndex === -1) {
    return res.status(404).json({ success: false, message: '用户不存在' });
  }

  const deletedUser = users.splice(userIndex, 1)[0];
  res.json({ success: true, message: '用户已删除', data: deletedUser });
});

// 错误处理中间件
app.use((err, req, res, next) => {
  console.error('服务器错误:', err);
  res.status(500).json({
    success: false,
    message: '服务器内部错误'
  });
});

// 启动服务器
app.listen(PORT, () => {
  console.log(`Express服务器已启动，监听端口: ${PORT}`);
  console.log(`API地址: http://localhost:${PORT}/api/users`);
});
```
