# 进程与线程深度完全指南

## 前言：理解进程和线程的重要性

想象一下，你经营一家大型工厂。工厂里有多个车间（CPU核心），每个车间里有很多工人（线程）。工厂需要协调这些工人，确保他们不会互相干扰，同时又能高效地完成生产任务。这就是操作系统内核最核心的工作之一。

作为一个全栈开发者，你写的Node.js、Python或Go程序，最终都要运行在这些"工厂"之上。理解进程和线程的运作机制，能帮助你写出更高性能、更稳定的程序，也能让你在排查问题时有更深的洞察力。

## 第一章：进程——工厂里的一个个车间

### 1.1 什么是进程？

**进程（Process）**是操作系统进行资源分配和调度的基本单位。你可以把它想象成一座独立的工厂厂房。每个工厂有自己的土地（内存地址空间）、设备（文件描述符）、工人（线程）。

为什么需要进程？因为操作系统需要**隔离**。如果一个程序崩溃了，它不应该影响到其他程序。就像如果一个车间发生火灾，不应该把整个工厂都烧掉一样。

### 1.2 进程的诞生

当你双击一个程序图标，或者在终端输入`node server.js`时，操作系统会创建一个新的进程。这个过程大致如下：

```c
// 操作系统创建进程的核心步骤（伪代码，有中文注释）
pid_t fork() {
    // 1. 创建一个新的进程描述符（PCB）
    struct task_struct* new_task = allocate_task_struct();

    // 2. 复制父进程的地址空间
    copy_address_space(new_task, parent_task);

    // 3. 分配新的PID（进程ID）
    new_task->pid = next_available_pid();

    // 4. 设置父子关系
    new_task->parent = current_task;

    // 5. 返回新进程的PID
    return new_task->pid;
}
```

在Linux中，进程的创建是通过`fork()`系统调用完成的。`fork()`会创建一个子进程，子进程是父进程的"拷贝"——但这个拷贝是"写时复制"的，意味着只有当子进程尝试修改数据时，才会真正复制一份。

### 1.3 进程的五种状态

进程不是一成不变的，它会在不同的状态之间切换：

```
                    ┌─────────┐
                    │  创建   │ → 新进程正在被创建
                    └────┬────┘
                         │
                         ▼
┌─────────┐    ┌───────────────┐    ┌─────────┐
│  就绪   │◄──►│    运行中      │──►│  阻塞   │
└─────────┘    └───────┬───────┘    └─────────┘
     ▲                 │                │
     │                 │                │
     └─────────────────┴────────────────┘
                    (时间片用完等)

```

| 状态 | 含义 | 触发条件 |
|------|------|----------|
| **创建（New）** | 进程正在被创建 | 程序开始运行 |
| **就绪（Ready）** | 进程已准备好，等待CPU | 时间片用完、等待I/O完成 |
| **运行（Running）** | 进程正在CPU上执行 | 被调度器选中 |
| **阻塞（Blocked）** | 进程等待某个事件 | 等待I/O、等待信号 |
| **终止（Terminated）** | 进程结束 | main函数返回、调用exit() |

### 1.4 进程控制块（PCB）

每个进程在操作系统内核中都有一个对应的**进程控制块（Process Control Block，PCB）**。你可以把它想象成工厂车间的"档案袋"，里面记录了这个车间的所有重要信息。

```c
// Linux内核中的进程控制块简化结构
struct task_struct {
    // 进程标识
    pid_t pid;                    // 进程ID，唯一标识
    pid_t ppid;                    // 父进程ID
    char name[TASK_COMM_LEN];      // 进程名称

    // 进程状态
    volatile long state;           // 状态：运行/就绪/阻塞等
    unsigned int flags;           // 进程标志位

    // 调度信息
    int priority;                 // 调度优先级
    unsigned long runtime;        // 已经运行的CPU时间
    struct sched_entity se;       // 调度实体

    // 内存管理
    struct mm_struct* mm;         // 进程的地址空间
    unsigned long start_code;     // 代码段起始地址
    unsigned long end_code;       // 代码段结束地址
    unsigned long start_data;     // 数据段起始地址
    unsigned long brk;            // 堆的当前末尾

    // 文件描述符表
    struct files_struct* files;   // 打开的文件列表

    // 信号处理
    struct signal_struct* signal; // 信号处理函数
    struct sighand_struct* sighand;

    // 线程信息
    struct thread_struct thread;   // CPU特定的寄存器值

    // 亲子关系
    struct list_head children;    // 子进程链表
    struct list_head sibling;     // 兄弟进程链表
};
```

### 1.5 进程的内存布局

每个进程都有自己独立的**虚拟地址空间**。这就像每个工厂车间都有自己的"地盘"，不会互相重叠。

```
┌─────────────────────────────────────┐ 0xFFFFFFFF (4GB)
│         内核空间 (1GB)              │ ← 只有内核能访问
├─────────────────────────────────────┤ 0xC0000000
│                                   │
│          栈 (Stack)                │ ← 向下增长，存放局部变量
│         ↘                         │
│                                   │
│                                   │
│                                   │
│         ↗                         │
│          堆 (Heap)                 │ ← 向上增长，动态分配
│                                   │
├─────────────────────────────────────┤
│         未初始化数据 (BSS)          │ ← 未初始化的全局变量
├─────────────────────────────────────┤
│         已初始化数据 (Data)         │ ← 已初始化的全局变量
├─────────────────────────────────────┤
│         代码段 (Text)              │ ← 程序的可执行代码
└─────────────────────────────────────┘ 0x00000000
```

## 第二章：线程——车间里的工人

### 2.1 什么是线程？

**线程（Thread）**是进程内的执行单元。你可以把它想象成工厂车间里的工人。一个车间（进程）可以有一个或多个工人（线程），他们共享车间的资源，但各自做不同的事情。

为什么需要线程？假设你要开发一个Web服务器：
- 一个线程处理客户端A的请求
- 另一个线程处理客户端B的请求
- 第三个线程在后台记录日志

如果没有线程，你可能需要启动多个进程，每个进程处理一个任务。但进程创建和切换的代价很大，线程就轻量多了。

### 2.2 线程 vs 进程

| 特性 | 进程 | 线程 |
|------|------|------|
| **资源拥有** | 独立资源（内存、文件） | 共享进程资源 |
| **创建代价** | 大（需要复制整个地址空间） | 小（只需分配线程栈） |
| **切换代价** | 大（需要切换页表、缓存） | 小（只需切换寄存器和栈） |
| **通信** | 复杂（需要IPC机制） | 简单（直接读写共享内存） |
| **隔离性** | 高（进程间完全隔离） | 低（线程共享地址空间） |
| **崩溃影响** | 只影响自身 | 会导致整个进程崩溃 |

### 2.3 线程的创建

在不同的编程语言中，创建线程的方式各有特色：

```javascript
// JavaScript/Node.js 中创建线程
// Node.js 的 worker_threads 模块

const { Worker, isMainThread, parentPort, workerData } = require('worker_threads');

// 主线程
if (isMainThread) {
    // 创建新的 Worker 线程
    const worker = new Worker(__filename, {
        workerData: { start: 1, end: 1000000 }
    });

    // 监听线程消息
    worker.on('message', (result) => {
        console.log('计算结果是:', result);
    });

    // 监听线程错误
    worker.on('error', (err) => {
        console.error('线程错误:', err);
    });

    // 监听线程退出
    worker.on('exit', (code) => {
        if (code !== 0) {
            console.error('线程异常退出，代码:', code);
        }
    });
} else {
    // Worker 线程执行的代码
    // 接收主线程传来的数据
    const { start, end } = workerData;

    // 执行计算
    let sum = 0;
    for (let i = start; i <= end; i++) {
        sum += i;
    }

    // 向主线程发送结果
    parentPort.postMessage(sum);
}
```

```python
# Python 中创建线程
# 使用 threading 模块

import threading
import time

def calculate_prime(start, end, results, index):
    """计算素数的函数"""
    primes = []
    for num in range(start, end + 1):
        if num > 1:
            for i in range(2, int(num ** 0.5) + 1):
                if num % i == 0:
                    break
            else:
                primes.append(num)

    # 将结果存入共享的结果列表
    results[index] = primes
    print(f"线程 {index} 完成，找到 {len(primes)} 个素数")

def main():
    # 定义任务参数：把1-100000分成5个区间
    ranges = [(1, 20000), (20001, 40000), (40001, 60000),
              (60001, 80000), (80001, 100000)]

    # 存储结果的列表
    results = [None] * len(ranges)
    threads = []

    # 创建并启动所有线程
    start_time = time.time()
    for i, (start, end) in enumerate(ranges):
        # 创建线程，传入目标函数和参数
        t = threading.Thread(
            target=calculate_prime,
            args=(start, end, results, i)
        )
        threads.append(t)
        t.start()  # 启动线程

    # 等待所有线程完成
    for t in threads:
        t.join()

    end_time = time.time()

    # 合并所有结果
    all_primes = []
    for r in results:
        if r:
            all_primes.extend(r)

    print(f"总共找到 {len(all_primes)} 个素数")
    print(f"耗时: {end_time - start_time:.2f}秒")

if __name__ == "__main__":
    main()
```

```go
// Go 语言中的Goroutine（轻量级线程）
// Go的runtime会自动调度Goroutine到操作系统线程上

package main

import (
    "fmt"
    "sync"
    "time"
)

// 使用WaitGroup等待所有goroutine完成
func calculatePrimes(start, end int, results chan<- []int, id int) {
    // 存放素数的切片
    primes := []int{}

    // 判断素数的函数
    isPrime := func(n int) bool {
        if n < 2 {
            return false
        }
        for i := 2; i*i <= n; i++ {
            if n%i == 0 {
                return false
            }
        }
        return true
    }

    // 在指定范围内找素数
    for num := start; num <= end; num++ {
        if isPrime(num) {
            primes = append(primes, num)
        }
    }

    // 发送结果到通道
    results <- primes
    fmt.Printf("Goroutine %d 完成，找到 %d 个素数\n", id, len(primes))
}

func main() {
    // 定义计算范围，分成5个区间
    ranges := []struct {
        start, end int
        id         int
    }{
        {1, 20000, 0},
        {20001, 40000, 1},
        {40001, 60000, 2},
        {60001, 80000, 3},
        {80001, 100000, 4},
    }

    // 创建通道用于接收结果
    results := make(chan []int, len(ranges))

    startTime := time.Now()

    // 启动所有goroutine
    for _, r := range ranges {
        // 使用go关键字启动goroutine
        go calculatePrimes(r.start, r.end, results, r.id)
    }

    // 创建一个WaitGroup用于同步
    var wg sync.WaitGroup
    wg.Add(len(ranges))

    // 收集结果
    go func() {
        for range ranges {
            <-results  // 接收结果
            wg.Done()
        }
    }()

    // 等待所有goroutine完成
    wg.Wait()

    endTime := time.Now()
    fmt.Printf("总耗时: %v\n", endTime.Sub(startTime))
}
```

### 2.4 线程的栈

线程和进程最大的区别之一是：线程共享进程的地址空间，但每个线程有自己的**栈**。栈是用来保存函数调用历史、局部变量的地方。

```c
// 线程栈的简化示意
// 每个线程都有自己独立的栈空间

// 线程1的栈
┌─────────────────┐ 高地址
│   main() 返回地址 │
├─────────────────┤
│   main() 参数    │
├─────────────────┤
│   局部变量 i     │
├─────────────────┤
│   局部变量 sum   │
├─────────────────┤
│   嵌套函数调用   │
│   inner()       │
├─────────────────┤
│   ...           │
└─────────────────┘ 低地址

// 线程2的栈（独立于线程1）
┌─────────────────┐ 高地址
│   work() 返回地址 │
├─────────────────┤
│   work() 参数    │
├─────────────────┤
│   局部变量 data  │
├─────────────────┤
│   ...           │
└─────────────────┘ 低地址
```

在Linux中，默认的线程栈大小是8MB（在x86-64架构下）。这个大小对于大多数应用来说是足够的，但如果你的程序有深度递归或者大数组的局部变量，可能需要调整栈大小。

## 第三章：进程调度算法

### 3.1 调度是什么？

操作系统里有多个进程/线程，但CPU核心数是有限的（通常几个到几十个）。**调度（Scheduling）**就是决定哪个进程/线程该在哪个CPU核心上运行的过程。

调度就像交通警察指挥路口的车辆：
- 哪个车先走？（选择哪个进程）
- 走多久？（时间片长度）
- 走完之后的安排？（调度策略）

### 3.2 调度算法分类

#### 3.2.1 先来先服务（FCFS）

最简单的调度算法，就像排队买票——先来的人先服务。

**优点**：简单，实现容易
**缺点**：平均等待时间长，短作业可能等很久

```
假设有三个进程到达顺序和执行时间：
P1: 到达时间0，执行时间24
P2: 到达时间0，执行时间3
P3: 到达时间0，执行时间3

FCFS调度（按P1→P2→P3顺序）:
       ├────────┤       ├────┤  ├────┤
P1:    │        │       │    │  │    │
       └────────┘       └────┘  └────┘
       0              24     27  30

P2的等待时间 = 24
P3的等待时间 = 27
平均等待时间 = (0 + 24 + 27) / 3 = 17
```

#### 3.2.2 短作业优先（SJF）

顾名思义，让执行时间最短的先执行。

**优点**：平均等待时间最优
**缺点**：需要预估进程执行时间，可能导致长作业饿死

```
同样的三个进程，SJF调度（先P2和P3都是3，按顺序来）:
       ├────┤  ├────┤       ├────────┤
P1:    │    │  │    │       │        │
       └────┘  └────┘       └────────┘
       0     3  6          30

P2的等待时间 = 0
P3的等待时间 = 3
平均等待时间 = (0 + 3 + 24) / 3 = 9
```

#### 3.2.3 时间片轮转（RR）

每个进程执行一个时间片，然后被放回就绪队列末尾。

```c
// 时间片轮转调度器的伪代码实现
struct rq {
    struct list_head head;  // 就绪队列
    int time_slice;          // 时间片大小（通常10-100ms）
};

void scheduler_tick(void) {
    // 每次时钟中断时调用
    struct task_struct* current = get_current_task();

    // 减少当前进程的时间片
    current->time_slice--;

    if (current->time_slice == 0) {
        // 时间片用完了，重新设置并触发调度
        current->time_slice = DEFAULT_TIMESLICE;

        // 将进程放到队列末尾
        list_move_tail(&current->run_list, &rq.head);

        // 设置需要调度的标志
        set_tsk_need_resched(current);
    }
}

void schedule(void) {
    // 真正的调度函数
    struct task_struct* next;

    // 关闭中断
    preempt_disable();

    // 从队列头部取出下一个要运行的进程
    next = list_entry(rq.head.next, struct task_struct, run_list);

    // 切换到新进程
    switch_to(prev, next);

    // 恢复中断
    preempt_enable();
}
```

#### 3.2.4 优先级调度

给每个进程分配优先级，高优先级的先运行。Linux使用**CFS（Completely Fair Scheduler，完全公平调度器）**。

```c
// CFS调度器的核心思想：为每个进程分配一个"虚拟运行时间"
// 调度器总是选择虚拟运行时间最少的进程

struct cfs_rq {
    struct rb_root tasks_timeline;    // 红黑树，按虚拟运行时间排序
    struct rb_node* leftmost;         // 最左侧节点（虚拟时间最少）
    unsigned long min_vruntime;       // 最小虚拟运行时间
};

struct sched_entity {
    // 虚拟运行时间（值越小，优先级越高）
    u64 virtual_runtime;

    // 实际运行时间
    u64 sum_exec_runtime;

    // 在红黑树中的节点
    struct rb_node run_node;

    // 权重（优先级越高，权重越大）
    unsigned long weight;
};

// CFS选择下一个进程：总是选虚拟运行时间最少的
struct task_struct* cfs_schedule(struct cfs_rq* cfs_rq) {
    // 从红黑树中获取最左侧节点
    struct rb_node* left = rb_first(&cfs_rq->tasks_timeline);

    if (!left)
        return NULL;  // 没有可运行的进程

    // 获取对应的sched_entity
    struct sched_entity* entity = rb_entry(left, struct sched_entity, run_node);

    // 获取包含该entity的task
    return container_of(entity, struct task_struct, se);
}
```

### 3.3 多级反馈队列调度

这是Linux和许多操作系统实际使用的调度算法，它结合了多种策略的优点：

```
┌─────────────────────────────────────────────────────────┐
│                    多级反馈队列                          │
├─────────────────────────────────────────────────────────┤
│  队列0 (最高优先级)  │ 时间片: 8ms  │ 响应时间要求高     │
│  ┌──────┐ ┌──────┐  │              │                    │
│  │  P1  │ │  P2  │  │              │                    │
│  └──────┘ └──────┘  │              │                    │
├─────────────────────────────────────────────────────────┤
│  队列1 (次高优先级)  │ 时间片: 16ms │ 交互进程           │
│  ┌──────┐           │              │                    │
│  │  P3  │           │              │                    │
│  └──────┘           │              │                    │
├─────────────────────────────────────────────────────────┤
│  队列2 (普通优先级)  │ 时间片: 32ms │ 批处理进程         │
│  ┌──────┐           │              │                    │
│  │  P4  │           │              │                    │
│  └──────┘           │              │                    │
├─────────────────────────────────────────────────────────┤
│  队列3 (低优先级)    │ 时间片: 64ms │ 后台任务           │
│                     │              │                    │
└─────────────────────────────────────────────────────────┘
```

**调度规则**：
1. 新进程从最高优先级队列开始
2. 如果用完时间片还没完成，降一级
3. 如果没到时间片就阻塞了（比如等待I/O），保持原优先级
4. 调度器先处理高优先级队列，只有高优先级队列空了才处理低优先级

### 3.4 调度与Node.js的关系

Node.js是单线程的，但这不意味着它只能用一个CPU核心。通过libuv库，Node.js可以处理I/O操作时使用操作系统的异步机制，在多核系统上可以启动多个worker进程来处理CPU密集型任务。

```javascript
// Node.js 中使用 cluster 模块利用多核
const cluster = require('cluster');
const os = require('os');

const numCPUs = os.cpus().length;  // 获取CPU核心数

if (cluster.isMaster) {
    console.log(`主进程 PID: ${process.pid}`);
    console.log(`系统有 ${numCPUs} 个CPU核心`);

    // 启动多个工作进程
    for (let i = 0; i < numCPUs; i++) {
        cluster.fork();  // 创建worker进程
    }

    // 监听worker退出事件
    cluster.on('exit', (worker, code, signal) => {
        console.log(`worker ${worker.process.pid} 退出了`);
        // 如果worker意外退出，重新启动
        cluster.fork();
    });
} else {
    // worker进程执行的代码
    console.log(`Worker进程 PID: ${process.pid} 启动`);

    // 模拟一个HTTP服务器
    const http = require('http');
    http.createServer((req, res) => {
        res.writeHead(200);
        res.end(`处理此请求的进程: ${process.pid}\n`);
```

## 第四章：上下文切换

### 4.1 什么是上下文切换？

**上下文切换（Context Switch）**是CPU在多个进程/线程之间切换的技术。当操作系统决定要停止运行当前进程，转而运行另一个进程时，需要保存当前进程的状态（上下文），然后恢复另一个进程之前保存的状态。

就像一个管理人员在多个会议之间切换：需要把当前会议的笔记记下来（保存上下文），然后去看另一个会议的记录（恢复上下文），再继续那个会议。

### 4.2 上下文切换的过程

```
时间轴:
─────────────────────────────────────────────────────────►

进程A运行中                    上下文切换              进程B运行中
┌─────────────────┐           ┌─────────┐           ┌─────────────────┐
│ CPU寄存器: A的值│ ───────► │ 保存A的  │           │ CPU寄存器: B的值 │
│ 程序计数器: A的PC│          │ 上下文   │           │ 程序计数器: B的PC│
│ 栈指针: A的栈   │          │         │           │ 栈指针: B的栈   │
└─────────────────┘           └─────────┘           └─────────────────┘
                                  │
                                  ▼
                          ┌─────────────────┐
                          │ 1. 保存当前进程状态 │
                          │ 2. 更新进程控制块  │
                          │ 3. 切换内存地址空间 │
                          │ 4. 恢复新进程状态  │
                          │ 5. 跳转到新进程PC  │
                          └─────────────────┘
```

### 4.3 上下文切换的成本

上下文切换可不是免费的午餐，它涉及大量的操作：

```c
// Linux内核中上下文切换的核心代码
__switch_to(struct task_struct* prev, struct task_struct* next) {
    // 1. 保存当前进程的FP/SP寄存器（帧指针/栈指针）
    //    这是函数调用链的基础
    asm volatile(
        "pushl %%ebp\n\t"      // 保存帧指针
        "movl %%esp, %0\n\t"   // 保存栈指针到prev->thread.sp
        : "=m" (prev->thread.sp)
    );

    // 2. 保存程序计数器（告诉CPU下一条指令在哪里）
    //    这样下次切换回来时知道从哪里继续
    asm volatile(
        "movl %%eip, %0\n\t"   // 保存指令指针到prev->thread.ip
        : "=m" (prev->thread.ip)
    );

    // 3. 切换内核态栈
    //    每个进程在内核中都有独立的栈
    per_cpu(current_stack_pointer, smp_processor_id()) = next->thread.sp;

    // 4. 切换TLS（线程本地存储）
    //    让每个线程访问自己的线程本地变量
    load_TLS(next, next->task.thread.tls_array);

    // 5. 恢复下一个进程的寄存器
    //    这些操作让CPU恢复到next之前执行的状态
    asm volatile(
        "movl %0, %%esp\n\t"   // 恢复栈指针
        "popl %%ebp\n\t"       // 恢复帧指针
        : : "m" (next->thread.sp)
    );

    // 6. 跳转到之前保存的指令位置
    asm volatile(
        "jmp *%0\n\t"          // 跳转到next->thread.ip
        : : "m" (next->thread.ip)
    );
}
```

### 4.4 上下文切换的开销分析

| 操作 | 开销 | 说明 |
|------|------|------|
| **保存/恢复寄存器** | ~100-500ns | 取决于CPU架构和寄存器数量 |
| **切换内存地址空间** | ~1-10μs | 需要刷新TLB（页表缓存） |
| **刷新CPU缓存** | ~1-5μs | L1/L2缓存可能失效 |
| **切换内核栈** | ~100-200ns | 栈指针的切换 |
| **调度器运行** | ~1-5μs | 选择下一个运行的进程 |

在高性能服务器上，每秒可能发生数万甚至数十万次上下文切换！这就是为什么过度创建线程（每个线程都有自己的栈和内核调度开销）会严重影响性能。

Node.js使用**事件循环 + 非阻塞I/O**的模型，大大减少了上下文切换的开销：

## 第五章：线程同步

### 5.1 为什么需要同步？

想象一个场景：两个工人同时在一个仓库里工作。一个工人要拿走10个零件，另一个工人也要拿走10个零件，但仓库里只有12个零件。如果他们同时动手，可能会出问题：

```
时间顺序:
T1: 工人A读取库存：12个
T2: 工人B读取库存：12个  ← 两个工人同时读取，看到的都是12
T3: 工人A拿走10个，写回：2个
T4: 工人B拿走10个，写回：-8个  ← 错误！库存不能是负数
```

这就是**竞态条件（Race Condition）**——多个线程对共享资源的访问顺序会影响结果。

### 5.2 互斥锁（Mutex）

**互斥锁**是最基本的同步原语。就像一个厕所，一次只能一个人用。

```c
// 互斥锁的简化实现（基于原子操作）
// 核心思想：用一个原子变量表示锁的状态

struct mutex {
    atomic_t count;        // 锁的状态：1=可用，0=被占用
    struct list_head wait_list;  // 等待队列
};

// 加锁操作
void mutex_lock(struct mutex* lock) {
    // 尝试原子地将count从1改为0
    // 如果成功，说明拿到了锁
    // 如果失败，说明锁被占用，需要等待

    if (atomic_dec_and_test(&lock->count)) {
        // 获取到锁了
        return;
    }

    // 没拿到锁，进入等待队列
    add_wait_queue(&lock->wait_list, current);

    // 循环等待，直到锁被释放
    while (1) {
        // 标记自己正在等待
        set_current_state(TASK_UNINTERRUPTIBLE);

        // 再次尝试获取锁
        if (atomic_dec_and_test(&lock->count)) {
            remove_wait_queue(&lock->wait_list, current);
            return;
        }

        // 让出CPU，让其他线程运行
        schedule();
    }
}

// 解锁操作
void mutex_unlock(struct mutex* lock) {
    // 原子地将count从0改回1
    atomic_inc(&lock->count);

    // 唤醒等待队列中的一个线程
    if (!list_empty(&lock->wait_list)) {
        wake_up_process(list_entry(lock->wait_list.next,
                                    struct task_struct,
                                    wake_list));
    }
}
```

### 5.3 互斥锁在代码中的应用

```javascript

        const oldValue = sharedCounter;
        // 模拟一些处理时间
        await new Promise(resolve => setTimeout(resolve, 10));
        sharedCounter = oldValue + amount;
        return sharedCounter;
    } finally {
        // 一定要释放锁，即使发生异常

    }
}

// 使用示例

    }

    console.log('最终计数器值:', results[results.length - 1]);
    // 如果没有锁，可能是小于100的某个值
    // 有锁的情况下，应该是100
}

main();
```

### 5.4 读写锁（Read-Write Lock）

有时候，我们读共享资源的次数远多于写。如果每次读都要加锁，会严重影响性能。**读写锁**允许多个读者同时访问，但写者是独占的。

```
场景分析：
- 读者A、B、C同时读取数据 → 都可以通过（并发读）
- 写者D想要写入 → 必须等A、B、C全部释放锁
- 读者E想要读取 → 必须等D写入完成
```

```c
// 读写锁的简化实现
struct rw_lock {
    atomic_t readers;        // 当前读者数量
    atomic_t writers;        // 是否有写者等待或已持有
    struct list_head wait_queue;  // 等待的写者队列
};

// 读锁获取
void read_lock(struct rw_lock* lock) {
    // 增加读者计数
    atomic_inc(&lock->readers);

    // 如果有写者在等待，读者需要等待
    // 这样可以防止写者饿死
    if (atomic_read(&lock->writers) > 0) {
        // 有写者，读者需要等待
        atomic_dec(&lock->readers);

        // 加入等待队列
        add_to_wait_queue(current);

        // 睡等，直到没有写者
        while (atomic_read(&lock->writers) > 0) {
            schedule();
        }

        atomic_inc(&lock->readers);
    }
}

// 写锁获取
void write_lock(struct rw_lock* lock) {
    // 标记有写者在等待
    atomic_inc(&lock->writers);

    // 等待所有读者释放
    while (atomic_read(&lock->readers) > 0) {
        schedule();
    }
    // 此时没有读者了，写者可以进入
}

// 读锁释放
void read_unlock(struct rw_lock* lock) {
    atomic_dec(&lock->readers);
}

// 写锁释放
void write_unlock(struct rw_lock* lock) {
    atomic_dec(&lock->writers);
}
```

### 5.5 条件变量（Condition Variable）

**条件变量**用于线程间的等待和通知机制。比如：等原材料到了再生产，等数据准备好了再发送。

```c
// 条件变量的核心操作
struct condition_variable {
    struct list_head wait_queue;  // 等待的线程队列
};

// 等待条件
void cv_wait(struct condition_variable* cv, struct mutex* lock) {
    // 将当前线程加入等待队列
    add_wait_queue(&cv->wait_queue, current);

    // 解锁，让其他线程可以访问
    mutex_unlock(lock);

    // 标记为睡眠状态，让出CPU
    set_current_state(TASK_INTERRUPTIBLE);

    // 调度其他线程
    schedule();

    // 被唤醒后，重新获取锁
    mutex_lock(lock);
}

// 通知一个等待者
void cv_signal(struct condition_variable* cv) {
    if (!list_empty(&cv->wait_queue)) {
        // 唤醒队列中的第一个线程
        wake_up_process(list_entry(cv->wait_queue.next,
                                   struct task_struct,
                                   wake_list));
    }
}

// 通知所有等待者
void cv_broadcast(struct condition_variable* cv) {
    // 唤醒所有等待的线程
    while (!list_empty(&cv->wait_queue)) {
        wake_up_process(list_entry(cv->wait_queue.next,
                                   struct task_struct,
                                   wake_list));
    }
}
```

### 5.6 生产者-消费者问题

这是线程同步的经典问题：

```
        生产者                          消费者
    ┌──────────┐                   ┌──────────┐
    │  生产    │ ──────────────►  │  消费    │
    │  数据    │     缓冲区         │  数据    │
    └──────────┘                   └──────────┘
```

```python
# Python 实现生产者-消费者问题
import threading
import queue
import time
import random

# 创建有界队列（最多10个元素）
buffer = queue.Queue(maxsize=10)

# 控制生产的数量
ITEMS_TO_PRODUCE = 20

def producer(producer_id):
    """生产者函数：生产数据放入缓冲区"""
    produced = 0
    while produced < ITEMS_TO_PRODUCE // 2:  # 两个生产者
        item = random.randint(1, 100)
        buffer.put(item)  # 如果队列满会自动阻塞
        print(f"生产者 {producer_id} 生产了: {item}")
        produced += 1
        # 模拟生产时间
        time.sleep(random.uniform(0.1, 0.5))
    print(f"生产者 {producer_id} 完成生产")

def consumer(consumer_id):
    """消费者函数：从缓冲区取数据"""
    consumed = 0
    while consumed < ITEMS_TO_PRODUCE // 2:  # 两个消费者
        item = buffer.get()  # 如果队列空会自动阻塞
        print(f"消费者 {consumer_id} 消费了: {item}")
        consumed += 1
        # 模拟消费时间
        time.sleep(random.uniform(0.2, 0.6))
    print(f"消费者 {consumer_id} 完成消费")

# 创建和启动线程
producers = [
    threading.Thread(target=producer, args=(1,)),
    threading.Thread(target=producer, args=(2,))
]

consumers = [
    threading.Thread(target=consumer, args=(1,)),
    threading.Thread(target=consumer, args=(2,))
]

start_time = time.time()

# 启动所有线程
for p in producers:
    p.start()
for c in consumers:
    c.start()

# 等待完成
for p in producers:
    p.join()
for c in consumers:
    c.join()

end_time = time.time()

print(f"\n总耗时: {end_time - start_time:.2f}秒")
print("所有生产和消费完成！")
```

### 5.7 信号量（Semaphore）

**信号量**是一个更通用的同步原语，本质上是一个计数器：

- `wait()`（或`P()`）：计数器减1，如果小于0则等待
- `signal()`（或`V()`）：计数器加1，如果有等待者则唤醒

```c
// 信号量的实现
struct semaphore {
    atomic_t count;           // 计数器
    struct list_head wait_list;  // 等待队列
};

// P操作（等待）
void down(struct semaphore* sem) {
    // 原子地减少计数
    if (atomic_dec_and_test(&sem->count)) {
        // 计数还够，不用等待
        return;
    }

    // 计数不够，需要等待
    add_wait_queue(&sem->wait_list, current);

    // 循环等待
    while (1) {
        set_current_state(TASK_INTERRUPTIBLE);

        if (atomic_dec_and_test(&sem->count)) {
            remove_wait_queue(&sem->wait_list, current);
            return;
        }

        schedule();
    }
}

// V操作（释放）
void up(struct semaphore* sem) {
    // 增加计数
    atomic_inc(&sem->count);

    // 唤醒一个等待者（如果有）
    if (!list_empty(&sem->wait_list)) {
        wake_up_process(list_entry(sem->wait_list.next,
                                    struct task_struct,
                                    wake_list));
    }
}
```

## 第六章：进程间通信（IPC）

### 6.1 为什么进程需要通信？

进程是独立的，但有时候需要协作：

- 浏览器：多个标签页可能共享同一个渲染进程
- IDE：编辑器进程和编译器进程需要通信
- 服务器：主进程和工作进程需要传递任务

### 6.2 管道（Pipe）

**管道**是最基本的IPC机制，类似于生活中的水管——一端写入，一端读取。

```c
// 创建管道
int pipe(int fd[2]) {
    // fd[0]: 读端
    // fd[1]: 写端

    // 创建两个文件描述符
    if (pipe_create(fd) < 0)
        return -1;

    // fd[0] 用于读取
    // fd[1] 用于写入
    return 0;
}

// 使用示例：父子进程通过管道通信
int main() {
    int fd[2];

    // 创建管道
    if (pipe(fd) < 0) {
        perror("pipe error");
        return 1;
    }

    pid_t pid = fork();

    if (pid < 0) {
        perror("fork error");
        return 1;
    }

    if (pid == 0) {
        // 子进程：向管道写入数据
        close(fd[0]);  // 关闭读端

        const char* msg = "Hello from child!";
        write(fd[1], msg, strlen(msg) + 1);

        close(fd[1]);  // 写完后关闭写端
        exit(0);
    } else {
        // 父进程：从管道读取数据
        close(fd[1]);  // 关闭写端

        char buf[256];
        ssize_t n = read(fd[0], buf, sizeof(buf));

        printf("Parent received: %s\n", buf);

        close(fd[0]);  // 读完后关闭读端
        wait(NULL);
    }

    return 0;
}
```

### 6.3 消息队列（Message Queue）

**消息队列**允许进程发送格式化的消息，而不是字节流。

```python
# Python 中使用 POSIX 消息队列
# 注意：需要 python-mq 库

# 在Linux中，也可以通过 subprocess + mkfifo 实现类似效果

import subprocess
import os
import time

def main():
    # 创建两个命名管道用于双向通信
    pipe_parent_to_child = '/tmp/pipe_p2c'
    pipe_child_to_parent = '/tmp/pipe_c2p'

    # 删除已存在的管道
    for pipe in [pipe_parent_to_child, pipe_child_to_parent]:
        try:
            os.unlink(pipe)
        except FileNotFoundError:
            pass
        # 创建命名管道
        os.mkfifo(pipe)

    # 创建子进程
    pid = os.fork()

    if pid == 0:
        # 子进程
        # 打开父到子的管道（只读）
        p2c = os.open(pipe_parent_to_child, os.O_RDONLY)
        # 打开子到父的管道（只写）
        c2p = os.open(pipe_child_to_parent, os.O_WRONLY)

        while True:
            # 读取父进程的消息
            msg = os.read(p2c, 1024).decode()
            if msg == 'quit':
                break
            print(f"Child received: {msg}")
            # 发送响应
            response = f"Child received: {msg}"
            os.write(c2p, response.encode())

        os.close(p2c)
        os.close(c2p)

    else:
        # 父进程
        # 打开父到子的管道（只写）
        p2c = os.open(pipe_parent_to_child, os.O_WRONLY)
        # 打开子到父的管道（只读）
        c2p = os.open(pipe_child_to_parent, os.O_RDONLY)

        messages = ['Hello', 'World', 'Test', 'quit']
        for msg in messages:
            # 发送消息
            os.write(p2c, msg.encode())
            # 读取响应
            response = os.read(c2p, 1024).decode()
            print(f"Parent received: {response}")

        os.wait()  # 等待子进程
        os.close(p2c)
        os.close(c2p)

if __name__ == '__main__':
    main()
```

### 6.4 共享内存（Shared Memory）

**共享内存**是最快的IPC方式——直接让多个进程访问同一块物理内存。

```c
// 共享内存的创建和使用
#include <sys/shm.h>
#include <sys/ipc.h>

int main() {
    // 1. 创建共享内存段
    // key: 标识共享内存的键
    // size: 共享内存大小
    // shmflg: 权限标志
    int shmid = shmget(IPC_PRIVATE, 4096, IPC_CREAT | 0666);

    if (shmid < 0) {
        perror("shmget error");
        return 1;
    }

    // 2. 挂载共享内存到进程地址空间
    // shmaddr: 通常为NULL，让系统选择地址
    // shmflg: 读写权限
    void* addr = shmat(shmid, NULL, 0);

    if (addr == (void*) -1) {
        perror("shmat error");
        return 1;
    }

    // 3. fork创建子进程
    pid_t pid = fork();

    if (pid == 0) {
        // 子进程：写入共享内存
        char* shared = (char*) addr;
        sprintf(shared, "Data from child, PID=%d", getpid());

        // 卸载共享内存
        shmdt(addr);
        exit(0);
    } else {
        // 父进程：读取共享内存
        wait(NULL);  // 等待子进程完成

        char* shared = (char*) addr;
        printf("Parent read from shared memory: %s\n", shared);

        // 卸载共享内存
        shmdt(addr);

        // 删除共享内存段
        shmctl(shmid, IPC_RMID, NULL);
    }

    return 0;
}
```

### 6.5 套接字（Socket）

**套接字**是最通用的IPC方式，特别适合不同主机间的通信。

```javascript
// Node.js 中使用 Unix Domain Socket 进行本地进程通信
// Unix Socket 比 TCP Loopback 更快，因为不需要走网络协议栈

const net = require('net');
const fs = require('fs');

// Unix Socket 路径
const SOCKET_PATH = '/tmp/echo.sock';

// 如果 socket 已存在，先删除
if (fs.existsSync(SOCKET_PATH)) {
    fs.unlinkSync(SOCKET_PATH);
}

// 创建服务器
const server = net.createServer((socket) => {
    console.log('客户端连接成功');

    // 监听数据
    socket.on('data', (data) => {
        console.log('服务器收到:', data.toString());
        // 发送回显
        socket.write('服务器收到: ' + data.toString());
    });

    // 客户端关闭
    socket.on('close', () => {
        console.log('客户端断开连接');
    });
});

// 监听错误
server.on('error', (err) => {
    console.error('服务器错误:', err);
});

// 启动服务器
server.listen(SOCKET_PATH, () => {
    console.log(`服务器监听在 ${SOCKET_PATH}`);

    // 创建客户端连接
    const client = net.createConnection(SOCKET_PATH, () => {
        console.log('已连接到服务器');

        // 发送消息
        client.write('Hello from client!');
    });

    // 接收响应
    client.on('data', (data) => {
        console.log('客户端收到:', data.toString());
        // 关闭连接
        client.end();
        server.close();
    });
});
```

### 7.2 Go 的 Goroutine 调度

Go 使用**GMP模型**（Goroutine - Machine - Processor）进行调度：

```go
// Go 的 GMP 调度模型
// G（Goroutine）：待执行的代码
// M（Machine）：实际的操作系统线程
// P（Processor）：调度的上下文

package main

import (
    "fmt"
    "runtime"
    "sync"
    "time"
)

func main() {
    // 设置最大操作系统线程数
    runtime.GOMAXPROCS(4)

    var wg sync.WaitGroup

    // 启动100个goroutine
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            // 模拟工作
            time.Sleep(10 * time.Millisecond)
            fmt.Printf("Goroutine %d 完成\n", id)
        }(i)
    }

    wg.Wait()
    fmt.Println("所有goroutine完成")
}
```

### 7.3 Python 的 GIL 与多进程

Python 有 GIL（全局解释器锁），这意味着多线程不能真正并行执行 CPU 密集型任务：

```python
# Python 中的多进程 vs 多线程
# CPU密集型任务应该用多进程
# I/O密集型任务可以用多线程

import multiprocessing
import threading
import time
import math

def cpu_intensive_task(n, results, index):
    """CPU密集型任务：计算素数"""
    count = 0
    for i in range(1, n + 1):
        if is_prime(i):
            count += 1
    results[index] = count
    print(f"进程 {index} 完成，找到 {count} 个素数")

def is_prime(n):
    """判断素数"""
    if n < 2:
        return False
    for i in range(2, int(n ** 0.5) + 1):
        if n % i == 0:
            return False
    return True

def io_intensive_task(n, results, index):
    """I/O密集型任务：模拟网络请求"""
    import random
    count = 0
    for _ in range(n):
        # 模拟网络延迟
        time.sleep(random.uniform(0.001, 0.005))
        count += 1
    results[index] = count
    print(f"线程 {index} 完成，处理了 {count} 个请求")

def main():
    n = 100000  # 计算到10万的素数
    n_workers = 4

    # 测试 CPU 密集型：多进程
    print("=== CPU 密集型：多进程 ===")
    manager = multiprocessing.Manager()
    results = manager.list([0] * n_workers)

    start = time.time()
    processes = []
    chunk_size = n // n_workers
    for i in range(n_workers):
        start_num = i * chunk_size + 1
        end_num = (i + 1) * chunk_size if i < n_workers - 1 else n
        p = multiprocessing.Process(
            target=cpu_intensive_task,
            args=(end_num, results, i)
        )
        processes.append(p)
        p.start()

    for p in processes:
        p.join()

    print(f"多进程耗时: {time.time() - start:.2f}秒")
    print(f"找到的素数总数: {sum(results)}")

    # 测试 I/O 密集型：多线程
    print("\n=== I/O 密集型：多线程 ===")
    results = [0] * n_workers

    start = time.time()
    threads = []
    for i in range(n_workers):
        t = threading.Thread(
            target=io_intensive_task,
            args=(250, results, i)  # 每个线程处理250个请求
        )
        threads.append(t)
        t.start()

    for t in threads:
        t.join()

    print(f"多线程耗时: {time.time() - start:.2f}秒")

if __name__ == '__main__':
    main()
```

## 总结：进程与线程的核心要点

1. **进程是资源分配的单位，线程是CPU调度的单位**
   - 进程间隔离，线程共享资源

2. **调度算法决定了如何分配CPU时间**
   - Linux使用CFS，追求"完全公平"
   - 调度单位是线程，不是进程

3. **上下文切换有开销**
   - 保存/恢复寄存器、切换地址空间
   - 过多线程会影响性能

4. **同步机制保护共享资源**
   - 互斥锁、读写锁、信号量、条件变量
   - 选择合适的同步原语很重要

5. **IPC方式的选择**
   - 管道、消息队列、共享内存、套接字
   - 根据场景选择最合适的方案

理解这些概念，能帮助你写出更高效、更可靠的多线程/多进程程序！
