# 文件系统与存储原理

## 前言：从书架到硬盘

想象你有一个巨大的图书馆，里面有几百万本书。如果这些书随便堆在一起，那找一本书就像大海捞针。但如果有一个完善的分类系统——按作者、按年代、按主题——再找书就容易多了。

文件系统就是计算机世界的"图书管理系统"，它负责管理存储设备上的数据，让你能方便地存储、查找、删除文件。

## 第一章：文件系统的基本概念

### 1.1 什么是文件系统？

**文件系统（File System）**是操作系统用于管理存储设备上数据的系统。它提供：

1. **数据存储**：把数据保存到存储设备
2. **数据组织**：用目录和文件名组织数据
3. **数据访问**：提供读写数据的接口
4. **空间管理**：管理存储设备的可用空间
5. **权限控制**：控制谁可以访问什么数据

### 1.2 文件系统的层次结构

```
应用程序
    │
    ▼
┌─────────────────────────────────────┐
│         VFS (虚拟文件系统)           │
│  统一接口，屏蔽不同文件系统的差异     │
├─────────────────────────────────────┤
│     ext4    │   xfs   │  ntfs  │ btrfs │  ← 具体文件系统
├─────────────────────────────────────┤
│         块设备层 (Block Layer)       │
│     磁盘抽象，统一管理块设备          │
├─────────────────────────────────────┤
│         SCSI / NVMe 驱动            │
├─────────────────────────────────────┤
│           物理磁盘 / SSD             │
└─────────────────────────────────────┘
```

### 1.3 常见文件系统对比

| 文件系统 | 开发方 | 主要用途 | 特点 |
|----------|--------|----------|------|
| **ext4** | Linux | 通用 | 成熟稳定，日志功能 |
| **xfs** | Silicon Graphics | 大文件 | 高性能，大型企业存储 |
| **btrfs** | Oracle | 下一代 | 写时复制，快照，校验 |
| **ntfs** | Microsoft | Windows | Windows 默认 |
| **apfs** | Apple | macOS | 加密，快照 |
| **zfs** | Sun/Oracle | 存储服务器 | 128位，快照，RAID-Z |

## 第二章：inode 与文件存储

### 2.1 inode 的概念

**inode（Index Node）**是文件系统中用于描述文件元数据的数据结构。每个文件都有一个对应的 inode，inode 包含：

```
inode 结构（简化版）：

┌─────────────────────────────────────────────┐
│ inode 结构                                   │
├─────────────────────────────────────────────┤
│ inode 编号: 12345                           │
├─────────────────────────────────────────────┤
│ 文件类型: 普通文件 / 目录 / 符号链接 / ...   │
│ 文件权限: rwxr-xr-x                         │
│ 所有者: uid=1000                            │
│ 所属组: gid=1000                            │
│ 文件大小: 4096 字节                         │
│ 创建时间: 2024-01-15 10:30:00              │
│ 修改时间: 2024-01-15 11:45:00              │
│ 访问时间: 2024-01-15 12:00:00              │
├─────────────────────────────────────────────┤
│ 直接块指针: [块1][块2][块3][块4]             │ ← 指向数据块
│ 间接块指针: [间接块地址]                     │ ← 单间接
│ 双间接块指针: [双间接块地址]                 │ ← 双间接
│ 三间接块指针: [三间接块地址]                 │ ← 三间接
├─────────────────────────────────────────────┤
│ 链接计数: 3（硬链接数量）                    │
│ 块数量: 8                                   │
│ 块大小: 4096 字节                           │
└─────────────────────────────────────────────┘
```

### 2.2 inode 的代码实现

```c

    __le32  i_mtime;       // 修改时间
    __le32  i_dtime;       // 删除时间

    __le32  i_flags;       // 文件标志
    __le32  osd1;          // 操作系统特定
    __le32  i_block[15];   // 块指针数组（关键！）

```

### 2.3 块指针与文件数据

文件的数据存储在数据块中，而 inode 中的 `i_block[15]` 数组指向这些数据块：

```
ext4 块寻址机制：

直接块（12个）：
┌─────────────────────────────────────┐
│ i_block[0]  ──► 数据块 1            │
│ i_block[1]  ──► 数据块 2            │
│ i_block[2]  ──► 数据块 3            │
│ ...                                 │
│ i_block[11] ──► 数据块 12           │
└─────────────────────────────────────┘
最大直接寻址：12 × 4KB = 48KB

单间接块（1个）：
┌─────────────────────────────────────┐
│ i_block[12] ──► 间接块              │
│                 ┌─────────────────┤
│                 │ 数据块指针       │
│                 │ [块13] [块14]...│
│                 └─────────────────┘
└─────────────────────────────────────┘
间接块可寻址：1024 × 4KB = 4MB

双间接块（1个）：
┌─────────────────────────────────────┐
│ i_block[13] ──► 双间接块            │
│                 ┌─────────────────┤
│                 │ 单间接块指针    │
│                 │ ┌─────────────┤ │
│                 │ │ 数据块指针  │ │
│                 │ └─────────────┘ │
│                 └─────────────────┘
└─────────────────────────────────────┘
双间接可寻址：1024 × 1024 × 4KB = 4GB

三间接块（1个）：
└──► i_block[14] ──► 三间接块 ──► 双间接块 ──► 单间接块 ──► 数据块
```

## 第三章：ext4 与 xfs 对比

### 3.1 ext4 文件系统

**ext4（Fourth Extended File System）**是 Linux 最常用的文件系统：

```
ext4 特性：

1. 日志功能（Journaling）
   - 将元数据操作记录到日志
   - 崩溃后快速恢复
   - 避免文件系统损坏

2. 块组（Block Groups）
   - 将磁盘分成块组
   - 每个块组有自己的 inode 表和数据块
   - 减少碎片，提高局部性

3. 延迟分配（Extent）
   - 一次性分配多个块
   - 减少碎片
   - 提高大文件性能

4. 日志校验（Journal Checksumming）
   - 检测日志损坏
   - 提高可靠性
```

### 3.2 xfs 文件系统

**xfs** 是高性能日志文件系统，特别适合大文件和高吞吐量：

```
xfs 特性：

1. B+ 树索引
   - 使用 B+ 树管理空闲空间和目录
   - 大型目录依然高效

2. 实时子卷（Real-time Subvolume）
   - 数据和日志分离
   - 可选的实时日志

3. 分配组（Allocation Groups）
   - 磁盘分成多个分配组
   - 并行操作，提高并发性能
   - 适合多线程服务器

4. 日志限流
   - 日志操作不会阻塞数据操作
   - 更好的性能一致性
```

### 3.3 性能对比

```python
# 模拟 ext4 和 xfs 的性能特性

import time
import random

class FakeFileSystem:
    """模拟文件系统的基本操作"""

    def __init__(self, name):
        self.name = name
        self.blocks = {}  # block_id -> data
        self.inodes = {}  # inode_id -> metadata
        self.free_blocks = set(range(10000))

    def allocate_blocks(self, count, method='contiguous'):
        """分配数据块"""
        blocks = []

        if method == 'contiguous' or self.name == 'xfs':
            # xfs 优先连续分配
            if len(self.free_blocks) >= count:
                # 尝试找连续的块
                sorted_blocks = sorted(self.free_blocks)
                for i in range(len(sorted_blocks) - count + 1):
                    candidate = sorted_blocks[i:i+count]
                    if candidate[-1] - candidate[0] == count - 1:
                        blocks = candidate
                        break

        if not blocks:
            # ext4 或找不到连续块，随机分配
            for _ in range(count):
                blocks.append(self.free_blocks.pop())
        else:
            # 从空闲集合中移除
            for b in blocks:
                self.free_blocks.discard(b)

        return blocks

def benchmark_ext4_vs_xfs():
    """对比 ext4 和 xfs 的性能特点"""

    ext4 = FakeFileSystem("ext4")
    xfs = FakeFileSystem("xfs")

    print("=== 文件系统性能对比模拟 ===\n")

    # 测试1：创建大量小文件
    print("测试1: 创建 1000 个小文件（4KB 每个）")

    start = time.time()
    for i in range(1000):
        blocks = ext4.allocate_blocks(1)
        ext4.blocks[blocks[0]] = f"data_{i}"
    ext4_time = time.time() - start

    start = time.time()
    for i in range(1000):
        blocks = xfs.allocate_blocks(1)
        xfs.blocks[blocks[0]] = f"data_{i}"
    xfs_time = time.time() - start

    print(f"  ext4: {ext4_time*1000:.2f}ms")
    print(f"  xfs:  {xfs_time*1000:.2f}ms")

    # 测试2：创建大文件（顺序写入）
    print("\n测试2: 创建大文件（10MB，顺序写入）")

    ext4_blocks = ext4.allocate_blocks(2560, method='ext4')  # 4KB × 2560 = 10MB
    xfs_blocks = xfs.allocate_blocks(2560, method='xfs')

    # 计算顺序度
    ext4_seq = sum(1 for i in range(len(ext4_blocks)-1)
                   if ext4_blocks[i+1] - ext4_blocks[i] == 1) / len(ext4_blocks)
    xfs_seq = sum(1 for i in range(len(xfs_blocks)-1)
                  if xfs_blocks[i+1] - xfs_blocks[i] == 1) / len(xfs_blocks)

    print(f"  ext4 顺序度: {ext4_seq*100:.1f}% (块分布更随机)")
    print(f"  xfs 顺序度:  {xfs_seq*100:.1f}% (块连续分配)")
    print(f"  → xfs 在大文件顺序写入时性能更好")

    # 测试3：随机访问性能
    print("\n测试3: 随机访问大文件")

    random.seed(42)

    # ext4 随机访问（可能跨块组）
    start = time.time()
    for _ in range(10000):
        idx = random.randint(0, len(ext4_blocks)-1)
        block = ext4_blocks[idx]
        _ = ext4.blocks.get(block, None)
    ext4_random_time = time.time() - start

    # xfs 随机访问（分配组局部性）
    start = time.time()
    for _ in range(10000):
        idx = random.randint(0, len(xfs_blocks)-1)
        block = xfs_blocks[idx]
        _ = xfs.blocks.get(block, None)
    xfs_random_time = time.time() - start

    print(f"  ext4: {ext4_random_time*1000:.2f}ms")
    print(f"  xfs:  {xfs_random_time*1000:.2f}ms")
    print(f"  → xfs 分配组局部性更好，随机访问更稳定")

if __name__ == "__main__":
    benchmark_ext4_vs_xfs()
```

### 3.4 何时选择哪种文件系统

| 场景 | 推荐文件系统 | 原因 |
|------|-------------|------|
| **桌面 Linux** | ext4 | 成熟稳定，通用 |
| **服务器** | xfs | 大文件，高并发 |
| **需要快照** | btrfs / xfs | 支持写时复制快照 |
| **需要校验** | btrfs | 端到端校验 |
| **SSD 优化** | ext4 / xfs + discard | 都要开启 TRIM |
| **数据库** | xfs / ext4 | 根据数据库推荐 |

## 第四章：SSD 存储原理

### 4.1 SSD vs HDD

```
机械硬盘（HDD）工作原理：
┌─────────────────────────────────────────────────────┐
│                    磁盘盘片                          │
│              ┌─────────────────┐                    │
│              │                 │                    │
│              │    磁头臂       │ ◄── 读写磁头        │
│              │      ↑         │                    │
│              │      │         │                    │
│              └───────┼─────────┘                    │
│                      │                              │
│                    马达 ──► 旋转盘片                  │
└─────────────────────────────────────────────────────┘

问题：
- 需要旋转（7200 RPM = ~120 RPS）
- 需要移动磁头（寻道时间 5-10ms）
- 顺序访问还好，随机访问很慢

SSD 固态硬盘工作原理：
┌─────────────────────────────────────────────────────┐
│  NAND Flash 芯片                                     │
│  ┌────────┬────────┬────────┬────────┐              │
│  │ Block  │ Block  │ Block  │ Block  │              │
│  │ (256KB)│ (256KB)│ (256KB)│ (256KB)│              │
│  ├────────┼────────┼────────┼────────┤              │
│  │ Page   │ Page   │ Page   │ Page   │  ◄── 4KB     │
│  │ (4KB)  │ (4KB)  │ (4KB)  │ (4KB)  │              │
│  └────────┴────────┴────────┴────────┘              │
│                                                      │
│  无机械部件！直接电子访问                              │
└─────────────────────────────────────────────────────┘

优势：
- 无寻道时间
- 无旋转延迟
- 随机访问和顺序访问一样快（理论上）
- 功耗低，抗震
```

### 4.2 NAND Flash 的特性

```c
// NAND Flash 的物理特性

/*
 * NAND Flash 的层次结构：
 *
 * Block（块）：最小的擦除单位，通常 256KB - 4MB
 * Page（页）：最小的读写单位，通常 4KB - 16KB
 *
 * 重要特性：
 * 1. 读写按页，擦除按块
 * 2. 擦除次数有限（每个块 1000-100000 次）
 * 3. 写之前必须先擦除
 * 4. 擦除后全为 1，写操作只能把 1 改成 0
 */

// SSD 控制器的工作

struct ssd_controller {
    // 闪存通道（并行度）
    int num_channels;

    // 每个通道的芯片数
    int chips_per_channel;

    // FTL（Flash Translation Layer）表
    // 逻辑块地址 -> 物理块地址
    struct ftl_entry {
        uint32_t lba;      // 逻辑块地址
        uint32_t pba;      // 物理块地址
        uint32_t block;    // 块号
        uint32_t page;     // 页号
        int valid;         // 是否有效
    } *ftl_table;
};

// FTL 的作用：
// 1. 逻辑地址到物理地址的映射
// 2. 垃圾回收（Garbage Collection）
// 3. 磨损均衡（Wear Leveling）
// 4. 坏块管理
```

### 4.3 磨损均衡（Wear Leveling）

SSD 的每个块都有擦写次数限制，磨损均衡算法确保擦写均匀分布：

```
磨损均衡原理：

假设 SSD 有 1000 个块

初始状态：
Block 0: 擦写 100 次  [███████░░░░░░░░░]  10%
Block 1: 擦写 100 次  [███████░░░░░░░░░]  10%
Block 2: 擦写 100 次  [███████░░░░░░░░░]  10%
...
Block 999: 擦写 100 次 [███████░░░░░░░░░]  10%

写入新数据时：
- 优先选择擦写次数最少的块
- 定期迁移冷数据到磨损更多的块
- 保持各块擦写次数均衡

动态磨损均衡：
- 新数据写入擦写次数最少的块

静态磨损均衡：
- 定期检查冷数据
- 将长期未修改的数据移到高擦写次数的块
```

### 4.4 垃圾回收（Garbage Collection）

SSD 需要垃圾回收来回收已删除数据占用的空间：

```
垃圾回收过程：

1. 原始状态：
┌────────────────────────────────────────┐
│ Block A: [有效页][有效页][无效页][有效页]│
│ Block B: [空闲页][空闲页][空闲页][空闲页]│
└────────────────────────────────────────┘

2. 删除了某些数据后：
┌────────────────────────────────────────┐
│ Block A: [有效页][已删除][已删除][有效页]│ ← 有 50% 是无效的
│ Block B: [空闲页][空闲页][空闲页][空闲页]│
└────────────────────────────────────────┘

3. 触发垃圾回收：
┌────────────────────────────────────────┐
│ Block A: [有效页][有效页] ◄── 复制到这里 │
│         [新块1 ][新块2 ]                │
│ Block C: [复制后的有效页][复制后的有效页] │ ← 新的已擦除块
│         [空闲页][空闲页]                │
└────────────────────────────────────────┘

4. 回收完成后：
┌────────────────────────────────────────┐
│ Block A: 标记为擦除状态                  │
│ Block C: 变成空闲块                      │
└────────────────────────────────────────┘
```

## 第五章：RAID 存储技术

### 5.1 RAID 的概念

**RAID（Redundant Array of Independent Disks）**是将多个磁盘组合成逻辑单元的技术，提供：

- **性能提升**：并行读写
- **可靠性**：冗余数据
- **容量**：组合多盘

### 5.2 RAID 级别

```
RAID 0（条带化）：
┌─────────┬─────────┬─────────┬─────────┐
│  数据1  │  数据2  │  数据3  │  数据4  │  ← 分散到所有磁盘
│  磁盘0  │  磁盘1  │  磁盘2  │  磁盘3  │
└─────────┴─────────┴─────────┴─────────┘

优点：读写性能翻倍
缺点：无冗余，坏一块全丢
容量：N × 单盘容量

RAID 1（镜像）：
┌─────────┬─────────┬─────────┬─────────┐
│  数据1  │  数据1  │  数据2  │  数据2  │  ← 每份数据两份
│  磁盘0  │  磁盘1  │  磁盘2  │  磁盘3  │
└─────────┴─────────┴─────────┴─────────┘

优点：容错，一块盘坏数据不丢
缺点：容量利用率 50%
容量：N / 2 × 单盘容量

RAID 5（奇偶校验）：
┌─────────┬─────────┬─────────┬─────────┐
│  数据1  │  数据2  │  数据3  │  P(1,2,3)│ ← 分布式奇偶校验
│  磁盘0  │  磁盘1  │  磁盘2  │  磁盘3  │
└─────────┴─────────┴─────────┴─────────┘

优点：容错 + 容量利用率高
缺点：写性能略低，重建慢
容量：(N-1) × 单盘容量

RAID 6（双重奇偶）：
┌─────────┬─────────┬─────────┬─────────┐
│  数据1  │  数据2  │  数据3  │  P(1,2,3)│
│  磁盘0  │  磁盘1  │  磁盘2  │  磁盘3  │
└─────────┴─────────┴─────────┴─────────┘
                         ↓
              ┌──────────────────┐
              │  Q(1,2,3) 另一个  │
              │  独立校验数据    │
              └──────────────────┘

优点：可同时坏两块盘
缺点：容量利用率更低，写更慢
容量：(N-2) × 单盘容量
```

### 5.3 RAID 的软件实现

```bash

      --level=5 \
      --raid-devices=3 \
      --spare-devices=1 \
      /dev/sdb /dev/sdc /dev/sdd /dev/sde

# 移除故障盘
mdadm --manage /dev/md0 --remove /dev/sdc

# 添加新盘
mdadm --manage /dev/md0 --add /dev/sdf

# 停止 RAID
mdadm --stop /dev/md0

# 重新激活
mdadm --assemble /dev/md0
```

### 7.3 日志文件的处理

```javascript
// Node.js 高效日志处理

const fs = require('fs');
const stream = require('stream');

// 使用流式写入日志
class LogWriter {
    constructor(filename) {
        // 创建一个可写流（追加模式）
        this.stream = fs.createWriteStream(filename, {
            flags: 'a',  // 追加模式
            encoding: 'utf8'
        });
    }

    log(level, message) {
        const timestamp = new Date().toISOString();
        const entry = `[${timestamp}] [${level}] ${message}\n`;
        this.stream.write(entry);
    }

    info(message) { this.log('INFO', message); }
    warn(message) { this.log('WARN', message); }
    error(message) { this.log('ERROR', message); }

    close() {
        this.stream.end();
    }
}

// 使用管道进行日志处理
async function logProcessingDemo() {
    const { Transform } = stream;

    // 创建转换流：添加时间戳
    const timestampTransform = new Transform({
        transform(chunk, encoding, callback) {
            const data = chunk.toString();
            const lines = data.split('\n').filter(line => line.trim());

            for (const line of lines) {
                const timestamped = `[${new Date().toISOString()}] ${line}`;
                this.push(timestamped + '\n');
            }

            callback();
        }
    });

    // 读取源文件，转换后写入目标文件
    const readStream = fs.createReadStream('/tmp/source.log');
    const writeStream = fs.createWriteStream('/tmp/dest.log');

    // 使用管道连接流
    readStream
        .pipe(timestampTransform)
        .pipe(writeStream);

    await new Promise((resolve, reject) => {
        writeStream.on('finish', resolve);
        writeStream.on('error', reject);
    });

    console.log('日志处理完成');
}

// 文件锁示例（防止并发写入）
const FileLock = {
    locks: new Map(),

    async acquire(filepath) {
        while (this.locks.has(filepath)) {
            await new Promise(r => setTimeout(r, 10));
        }
        this.locks.set(filepath, true);
    },

    release(filepath) {
        this.locks.delete(filepath);
    },

    async withLock(filepath, fn) {
        await this.acquire(filepath);
        try {
            return await fn();
        } finally {
            this.release(filepath);
        }
    }
};

// 使用示例
async function main() {
    const logger = new LogWriter('/tmp/app.log');

    for (let i = 0; i < 10; i++) {
        await FileLock.withLock('/tmp/shared.log', async () => {
            // 多个进程/线程同时写入同一个文件时保证顺序
            const entry = `[PID ${process.pid}] Entry ${i}\n`;
            fs.appendFileSync('/tmp/shared.log', entry);
        });
    }

    logger.info('日志演示完成');
    logger.close();
}

main().catch(console.error);
```

## 总结：文件系统核心要点

1. **inode 是文件系统的核心**
   - 每个文件一个 inode
   - 存储元数据，通过块指针访问数据

2. **ext4 vs xfs**
   - ext4：通用，成熟，适合桌面和小文件
   - xfs：大文件，高并发，适合服务器

3. **SSD 特性**
   - 按页读写，按块擦除
   - 需要磨损均衡和垃圾回收
   - TRIM 命令让 SSD 知道哪些块可以回收

4. **RAID 提供冗余和性能**
   - RAID 0：性能，无冗余
   - RAID 1：镜像，冗余
   - RAID 5/6：奇偶校验，兼顾性能和冗余

5. **层次结构**
   - 文件系统 → LVM → RAID → 物理磁盘
   - 每层抽象，职责分明
