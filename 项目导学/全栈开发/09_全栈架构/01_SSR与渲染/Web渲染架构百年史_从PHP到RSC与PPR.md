# Web 渲染架构百年史：从 PHP、SPA 到 RSC 与 PPR (2026版)

## 1. 概述

Web 开发的本质是将服务器上的数据，渲染成用户屏幕上的像素。围绕着**“这段渲染代码到底该在服务器运行，还是在浏览器运行？”**这个核心矛盾，前端架构经历了长达 20 年的钟摆式演进。

理解这段演进史，是理解为什么 2026 年的业界会走向 React Server Components (RSC) 和局部预渲染 (PPR) 的唯一途径。本指南将带你拨开云雾，见证 Web 渲染引擎的发展脉络。

---

## 2. 远古时代：传统 SSR (Server-Side Rendering) 

**代表技术**：PHP, JSP, ASP, Ruby on Rails。

### 2.1 机制与繁荣
在 Web 的蛮荒时代，一切都在服务器发生。用户在浏览器输入 URL，请求到达 PHP 服务器。服务器连接 MySQL，查询数据，然后将数据与 HTML 模板（如 Blade 或 Smarty）拼接在一起，最后吐出一个**完整的、包含所有数据的 HTML 字符串**。

### 2.2 为什么被抛弃？
- **糟糕的 UX (用户体验)**：每次点击链接，整个页面都会闪烁、白屏并重新加载。
- **前后端高强度耦合**：前端写页面的不得不学一点 PHP，后端写接口的还得懂点 CSS，职责极其混乱。

---

## 3. 革命时代：SPA (Single Page Application) 与 CSR (Client-Side Rendering)

**代表技术**：AngularJS, React (早期的 create-react-app), Vue。

### 3.1 机制与繁荣
随着智能手机的普及和 V8 引擎的强大，工程师们决定：**把渲染的任务交给浏览器！**
- 服务器只负责吐出一个“空壳” HTML（`<div id="root"></div>`）和一坨巨大的 JavaScript 文件 (Bundle)。
- 浏览器下载完 JS 后，自己执行 React 代码，自己发起 AJAX 请求获取 JSON，自己拼接出 DOM 树。

### 3.2 致命缺陷 (SPA 的终结)
- **SEO 灾难**：爬虫爬不到数据，因为 HTML 源码是空的。
- **巨大的 FCP/LCP (首屏白屏)**：用户的网速决定了看白屏的时间。必须等那几兆的 JS 下载完并解析完，页面才能画出第一笔。也就是所谓的“菊花图 (Loading Spinner) 时代”。

---

## 4. 复辟时代：现代同构 SSR 与 Hydration (水合)

**代表技术**：Next.js (Pages Router), Nuxt.js。

为了拯救 SPA 的白屏和 SEO，业界搞出了一个缝合怪：**同构渲染 (Isomorphic Rendering)**。

### 4.1 Hydration 的魔法与诅咒
- **第一步 (服务端)**：Node.js 服务器假装自己是浏览器，执行一次 React 组件的渲染，生成带数据的完整 HTML 吐给用户（秒开，SEO 完美）。
- **第二步 (客户端)**：虽然看到了页面，但点击按钮没反应（因为没绑定事件）。浏览器必须**再次下载完整的 React JS Bundle，并在后台重新偷偷把组件再执行一遍**，把事件监听器挂载到已有的 HTML 上。这个叫醒死 HTML 的过程，就叫 **Hydration (水合)**。

### 4.2 为什么说同构 SSR 是个沉重的包袱？
- **Double Data (数据双传)**：为了保证服务端和客户端渲染出的结果绝对一致（否则会报 Hydration Mismatch），服务器不仅要传 HTML，还要把查询到的 JSON 数据塞进 HTML 末尾的 `<script>` 标签里传给客户端。
- **JS 依然庞大**：你引入了一个巨重的 `marked.js` 用来把 Markdown 转成 HTML，哪怕这事在服务端已经做完了，这个沉重的库还是会被打包发给浏览器参与 Hydration。

---

## 5. 破局时代：React Server Components (RSC)

**代表技术**：Next.js (App Router), React 19。

RSC 的出现，终结了上述所有的矛盾。它提出了一个天才的设想：**为什么组件不能被明确地划分界限呢？**

### 5.1 Server Component 的绝对纯洁
被标记为 Server Component 的组件，**只在服务器运行，它的代码永远不会被打包进客户端 JS！**
- 如果你在 RSC 里用了一个 2MB 的重度依赖库计算数据，客户端 JS 的体积增加量是 **0 KB**。
- 它不返回 HTML，而是返回一种特殊的二进制流 **RSC Payload**。

### 5.2 Client Component 的配合
带有 `'use client'` 的组件才会被打包发送到浏览器。RSC 和 Client Component 可以像三明治一样嵌套。
**架构哲学**：让服务器做沉重的数据获取和结构组装，让浏览器只负责轻巧的交互点击。这是 Web 渲染历史上的**最佳平衡点**。

---

## 6. 未来时代：PPR (Partial Prerendering - 局部预渲染)

RSC 解决了 JS 体积问题，但如果你的页面有很多动态数据（如依赖 Cookie 的用户头像），即使有 RSC，服务器每次也必须查询完数据库才能返回，失去了 CDN 边缘缓存的速度。

**PPR 是 2026 年的终极黑科技**：
它打破了“要么全静态 (SSG)，要么全动态 (SSR)”的二极管思维。

### 6.1 PPR 运行引擎机制
1. **构建时挖洞**：Next.js 在执行 `next build` 时，会将页面里用 `<Suspense>` 包裹的动态组件视为“洞 (Holes)”。页面其余的静态部分会被编译成一个极其轻薄的 HTML Shell，直接推送到全球 CDN。
2. **秒回 Shell**：用户访问时，CDN **0ms 延迟** 返回这个包含侧边栏、导航栏的静态 HTML。用户瞬间看到页面结构（体验超越任何框架）。
3. **流式填孔 (Streaming)**：与此同时，服务器开始执行那些被 `<Suspense>` 包裹的动态 RSC（如查询购物车），查询完毕后，通过同一个 HTTP 连接，将动态片段以流的形式注入到刚刚那个 Shell 里面。

## 7. 面试高频问题：渲染架构的底层思考

**Q1：Next.js 的 SSG (静态生成) 和 PPR (局部预渲染) 有什么本质区别？**
**答：** SSG 是**页面级**的。如果一个页面里有一小块数据需要实时获取，你就不能用 SSG，整个页面只能退化为 SSR，导致丧失 CDN 提速。而 PPR 是**组件级**的静态化，它允许页面 90% 的外壳走 CDN 的极致静态缓存，而剩下 10% 的动态数据走流式 SSR 注入，它融合了两者的所有优点。

**Q2：既然 RSC 这么好，那还有必要保留 Client Component 吗？**
**答：** 绝对必要。RSC 是一次性的，它像一个打印机，吐出结果后任务就结束了。它没有 `useState`，没有生命周期，也无法绑定 `onClick`。Web 应用的灵魂在于交互。未来的架构一定是 **RSC 主导页面骨架与数据链路**，**Client Component 散落在叶子节点负责微小交互**的混合树结构。
