# 大型项目组件管理与工程化质量验证 (2026版)

## 一、 前言：规模化带来的复杂性

在 2026 年，一个典型的中大型 React 项目可能包含 500+ 个组件、30+ 个业务模块以及跨团队协作的 UI 规范。此时，简单的“按照文件夹分类”已无法支撑项目的迭代。

我们面临的核心问题是：**如何确保组件在高度复用的同时，保持逻辑清晰、性能卓越且质量可控？**

---

## 二、 2026 组件管理架构：特性驱动 (Feature-based)

虽然**原子设计 (Atomic Design)** 在 UI 库中表现良好，但在业务开发中，**特性驱动架构**已成为大厂首选。

### 2.1 目录组织原则：垂直切片
将代码按业务功能（Feature）进行垂直拆分，每个 Feature 都是一个自包含的“微应用”。

```text
src/
├── components/          # 全局通用 UI 组件 (Atoms/Molecules)
│   └── ui/              # 纯 UI 原始组件 (shadcn/ui 风格)
├── features/            # 业务特性模块 (Vertical Slices)
│   ├── cart/            # 购物车模块
│   │   ├── components/  # 仅限该模块使用的组件
│   │   ├── hooks/       # 仅限该模块使用的逻辑
│   │   ├── types/       # 类型定义
│   │   ├── actions.ts   # React 19 Server Actions
│   │   └── index.ts     # 公共 API 导出 (Strict Barrel)
├── lib/                 # 第三方库配置
└── app/                 # Next.js 路由层 (粘合层)
```

### 2.2 导出约束：禁止“跨模块偷渡”
每个 Feature 必须通过 `index.ts` 明确导出。禁止其他 Feature 直接引用内部文件。
- **验证手段**：使用 ESLint 插件 `eslint-plugin-import` 的 `no-restricted-paths` 规则强制约束。

## 三、 高级组件设计模式：Compound + Headless

为了应对复杂 UI（如选择器、模态框），2026 年的标准做法是结合 **Headless** 的逻辑与 **Compound** 的声明式 API。

### 3.1 案例：极客版 Tabs 组件

```tsx
/**
 * 2026 推荐模式：利用 React 19 的 Context 与 Ref 简化封装
 */
export function Tabs({ children, defaultValue }) {
  const [activeTab, setActiveTab] = useState(defaultValue);
  
  return (
    <TabsContext.Provider value={{ activeTab, setActiveTab }}>
      <div className="tabs-container">{children}</div>
    </TabsContext.Provider>
  );
}

// 子组件直接挂载在父组件上，提高发现性
Tabs.Trigger = function Trigger({ value, children }) {
  const { activeTab, setActiveTab } = use(TabsContext); // React 19 简写
  return (
    <button 
      onClick={() => setActiveTab(value)}
      className={activeTab === value ? 'active' : ''}
    >
      {children}
    </button>
  );
};
```

## 四、 工程化质量验证：四维一体

在 2026 年，组件的质量不再由人工 Review 保证，而是由一套自动化体系支撑。

### 4.1 逻辑验证：Vitest Browser Mode
传统的 JSDOM 无法模拟真实的 CSS 样式和浏览器行为。2026 年我们强制要求使用 **Vitest Browser Mode** 配合 Playwright。

```typescript
// features/cart/components/CartItem.test.tsx
import { render, screen, userEvent } from '@testing-library/react';

test('增加商品数量应触发价格重算', async () => {
  render(<CartItem initialQty={1} price={100} />);
  
  const plusBtn = screen.getByRole('button', { name: '+' });
  await userEvent.click(plusBtn); // 模拟真实用户行为
  
  expect(screen.getByText('$200')).toBeInTheDocument();
});
```

### 4.2 视觉验证：Chromatic + Storybook 8
对于 UI 组件，逻辑通过不代表“看起来正确”。
- **方案**：每提交一个 PR，Chromatic 会自动将 Storybook 中的组件进行像素级比对。
- **验证点**：黑暗模式、不同语种（中文 vs 阿拉伯语）下的排版。

### 4.3 可访问性验证 (A11y)：vitest-axe
在 2026 年，不符合无障碍标准的组件被视为“Bug”。

```typescript
import { axe } from 'vitest-axe';

test('购物车组件应无 A11y 违规', async () => {
  const { container } = render(<Cart />);
  const results = await axe(container);
  expect(results).toHaveNoViolations();
});
```

### 4.4 性能验证：React DevTools Profiler 导出
在 CI 中运行脚本，导出组件的重绘次数和耗时。如果某个组件的渲染耗时由于逻辑修改而增加了 20% 以上，PR 将被自动阻塞。
