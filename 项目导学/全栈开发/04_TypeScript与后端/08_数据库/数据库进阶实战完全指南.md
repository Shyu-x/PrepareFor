## 一、事务深入

### 1.1 ACID 原则详解

事务是数据库管理系统执行过程中的逻辑单位，一个事务包含多个数据库操作。ACID是事务的四大特性，它们共同保证了数据库事务的可靠性和一致性。

**原子性（Atomicity）** 是事务最基本的特性。原子性要求事务是一个不可分割的最小单位，事务中的所有操作要么全部成功，要么全部失败回滚。就像硬币的两面，你不可能只看到一面。在实际开发中，如果事务执行到一半系统崩溃，数据库会自动回滚到事务开始前的状态，保证数据不会处于半完成状态。举例来说，用户转账场景中，从A账户扣除100元同时向B账户增加100元，这两个操作必须在同一个事务中完成，否则就会出现A扣了钱但B没收到的情况。

**一致性（Consistency）** 表示事务执行前后，数据库必须保持一致状态。一致性是由应用逻辑决定的，数据库本身无法完全保证。比如在转账场景中，转账前两个账户的总金额是2000元，转账后两个账户的总金额仍然是2000元，这就是一致性。数据库通过约束、触发器等机制来辅助保证一致性，但最终的一致性还需要应用层来维护。

**隔离性（Isolation）** 是指多个并发事务执行时，彼此之间应该相互隔离，不能互相干扰。隔离性决定了事务并发执行时的行为，如果隔离性不好，可能会出现脏读、不可重复读、幻读等问题。数据库通过锁机制和多版本并发控制来保证隔离性，不同的隔离级别会影响并发性能和数据一致性。

**持久性（Durability）** 是指事务一旦提交，对数据库的修改就是永久性的，即使系统崩溃也不会丢失。持久性通常通过日志机制来实现，事务提交前先将修改写入预写式日志（WAL），即使系统崩溃也可以通过日志恢复数据。

### 1.2 隔离级别深度剖析

SQL标准定义了四种隔离级别，每种级别对并发问题的容忍程度不同，级别越高并发性能越差，但数据一致性越好。

**读未提交（Read Uncommitted）** 是最低的隔离级别，事务可以读取其他事务未提交的数据。这个级别会导致脏读问题，即读取到了其他事务正在修改但尚未提交的数据。在实际生产环境中很少使用，因为数据一致性完全没有保障。想象一下，事务A正在更新用户余额，事务B此时读取到了更新后的余额，但事务A最后回滚了，导致事务B读取到的数据是错误的。

**读已提交（Read Committed）** 是大多数数据库的默认隔离级别（Oracle、SQL Server默认使用此级别）。该级别保证只能读取到已经提交的数据，避免了脏读问题。但这个级别仍然存在不可重复读问题，即在同一事务中两次读取同一行数据可能得到不同的结果，因为其他事务可能在这两次读取之间提交了修改。举例来说，事务A在时刻T1读取用户余额为1000元，事务B在时刻T2修改该用户余额为2000元并提交，事务A在时刻T3再次读取该用户余额时得到2000元，这就是不可重复读。

**可重复读（Repeatable Read）** 是MySQL默认的隔离级别（使用MVCC实现）。该级别保证在同一个事务中多次读取同一行数据时，结果始终一致，解决了不可重复读问题。但这个级别仍然存在幻读问题，即在事务执行过程中，其他事务插入了新的数据，导致查询结果不一致。幻读特别容易在范围查询时发生，比如事务A查询年龄在20到30岁之间的所有用户，在事务A执行过程中，事务B插入了一条年龄为25岁的新用户，当事务A再次查询时会发现多了一条记录。

**串行化（Serializable）** 是最高的隔离级别，通过强制事务串行执行来完全避免并发问题。这个级别相当于在读取的每一行数据上都加锁，导致并发性能极差，实际应用中很少使用。只有在确实需要严格保证数据一致性的场景才会考虑使用。

### 1.3 MVCC 多版本并发控制

MVCC（Multi-Version Concurrency Control）是现代数据库实现高并发的重要技术，它的核心思想是为每个事务创建一个数据快照，事务可以看到自己开始时的数据库状态，而不受其他并发事务的影响。

**版本链**是MVCC实现的基础。在InnoDB存储引擎中，每行数据都有两个隐藏列：DB_TRX_ID（最近修改的事务ID）和DB_ROLL_PTR（指向undo log的指针）。当数据被修改时，旧版本的数据会被写入undo log，形成一条版本链。每次修改都会创建一个新版本，通过undo log串起所有历史版本。

**ReadView**是MVCC中用于判断数据可见性的关键结构。ReadView包含四个重要属性：m_ids表示活跃事务ID列表，min_trx_id表示最小活跃事务ID，max_trx_id表示创建ReadView时最大事务ID，creator_trx_id表示当前事务ID。当读取数据时，数据库会根据ReadView和数据的版本链来判断哪个版本的数据对当前事务可见。

在读已提交级别下，每次读取都会重新创建ReadView，保证只能看到已提交的数据。在可重复读级别下，只在事务开始时创建一次ReadView，整个事务期间都使用同一个ReadView，保证了整个事务期间数据的一致性读取。

**undo log**记录了数据被修改前的值，是实现MVCC的关键。当事务需要读取旧版本数据时，数据库会根据版本链从最新版本开始向前遍历，直到找到对当前事务可见的版本。undo log分为两种：insert undo log用于插入操作，事务提交后可以立即删除；update undo log用于更新和删除操作，在没有事务需要读取时才能删除。

### 1.4 实战：事务实践

在实际项目开发中，正确使用事务是保证数据一致性的关键。以下是几种常见的事务实践场景和最佳实践。

**NestJS中事务的使用**：

```typescript
// 使用TypeORM进行事务处理
import { Injectable } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository, DataSource } from 'typeorm';
import { Account } from './account.entity';
import { Transfer } from './transfer.entity';

@Injectable()
export class TransferService {
  constructor(
    @InjectRepository(Account)
    private accountRepository: Repository<Account>,
    @InjectRepository(Transfer)
    private transferRepository: Repository<Transfer>,
    private dataSource: DataSource, // TypeORM数据源，包含事务管理
  ) {}

  /**
   * 转账业务的事务实现
   * 这个例子展示了如何保证跨多个表操作的数据一致性
   */
  async transfer(fromAccountId: string, toAccountId: string, amount: number) {
    // 创建查询构建器，用于执行事务
    const queryRunner = this.dataSource.createQueryRunner();

    // 连接到数据库
    await queryRunner.connect();
    // 开始事务
    await queryRunner.startTransaction();

    try {
      // 锁定源账户并检查余额（SELECT FOR UPDATE防止并发修改）
      const fromAccount = await queryRunner.manager.findOne(Account, {
        where: { id: fromAccountId },
        lock: { mode: 'pessimistic_write' }, // 行级锁
      });

      // 检查余额是否充足
      if (fromAccount.balance < amount) {
        throw new Error('余额不足');
      }

      // 扣除源账户金额
      fromAccount.balance -= amount;
      await queryRunner.manager.save(fromAccount);

      // 增加目标账户金额
      const toAccount = await queryRunner.manager.findOne(Account, {
        where: { id: toAccountId },
        lock: { mode: 'pessimistic_write' },
      });
      toAccount.balance += amount;
      await queryRunner.manager.save(toAccount);

      // 记录转账流水
      const transfer = new Transfer();
      transfer.fromAccountId = fromAccountId;
      transfer.toAccountId = toAccountId;
      transfer.amount = amount;
      transfer.status = 'completed';
      transfer.createdAt = new Date();
      await queryRunner.manager.save(transfer);

      // 提交事务
      await queryRunner.commitTransaction();

      return { success: true, transferId: transfer.id };
    } catch (error) {
      // 回滚事务
      await queryRunner.rollbackTransaction();
      throw error;
    } finally {
      // 释放连接
      await queryRunner.release();
    }
  }
}
```

```typescript
import { Observable, throwError } from 'rxjs';

import { DataSource } from 'typeorm';

/**
 * 事务拦截器
 * 自动管理事务的开启、提交和回滚
 * 使用方法：在Controller或方法上使用 @UseInterceptors(TransactionInterceptor)
 */
@Injectable()
export class TransactionInterceptor implements NestInterceptor {
  constructor(private dataSource: DataSource) {}

  async intercept(context: ExecutionContext, next: CallHandler): Promise<Observable<any>> {
    // 创建查询运行器
    const queryRunner = this.dataSource.createQueryRunner();

    // 连接到数据库
    await queryRunner.connect();
    // 开始事务
    await queryRunner.startTransaction();

    // 将queryRunner存入请求上下文，供服务层使用
    const request = context.switchToHttp().getRequest();
    request.queryRunner = queryRunner;

    return next.handle().pipe(
      tap(async () => {
        // 请求成功，提交事务
        await queryRunner.commitTransaction();
      }),
      catchError(async (error) => {
        // 请求失败，回滚事务
        await queryRunner.rollbackTransaction();
        return throwError(() => error);
      }),
      tap(async () => {
        // 释放连接
        await queryRunner.release();
      }),
    );
  }
}
```

```typescript
// 使用保存点实现部分回滚
async function complexOperationWithSavepoint(dataSource: DataSource) {
  const queryRunner = dataSource.createQueryRunner();
  await queryRunner.connect();
  await queryRunner.startTransaction();

  try {
    // 步骤1：创建订单（无保存点）
    const order = new Order();
    order.status = 'pending';
    await queryRunner.manager.save(order);

    // 创建保存点
    await queryRunner.query('SAVEPOINT order_created');

    try {
      // 步骤2：扣减库存
      await queryRunner.manager.query(
        'UPDATE inventory SET quantity = quantity - ? WHERE product_id = ?',
        [1, data.productId]
      );

      // 步骤3：扣除用户积分（可能失败）
      await queryRunner.manager.query(
        'UPDATE user_points SET points = points - ? WHERE user_id = ?',
        [data.points, data.userId]
      );

      // 步骤4：发送通知（可能失败）
      await sendNotification(data.userId);

      // 所有步骤成功，提交事务
      await queryRunner.commitTransaction();
    } catch (error) {
      // 扣除积分或发送通知失败，回滚到保存点
      // 只回滚步骤2、3、4，不回滚步骤1
      await queryRunner.query('ROLLBACK TO order_created');

      // 继续执行后续逻辑（如发送人工处理通知）
      await handleManualProcess(order.id);
    }
  } catch (error) {
    // 创建订单失败，全部回滚
    await queryRunner.rollbackTransaction();
    throw error;
  } finally {
    await queryRunner.release();
  }
}
```

## 二、索引优化

### 2.1 B+Tree 索引结构

B+Tree是数据库索引的核心数据结构，理解B+Tree的工作原理对于索引优化至关重要。与常见的二叉树不同，B+Tree是一种自平衡的多路搜索树，非常适合磁盘存储和范围查询。

**B+Tree的结构特点**使其成为数据库索引的首选。在B+Tree中，所有数据都存储在叶子节点，非叶子节点只存储索引键和指针。每个叶子节点包含数据行的实际位置（行指针）或完整的数据行。叶子节点之间通过双向链表连接，这使得范围查询（如BETWEEN、LIKE）非常高效，只需要找到起始节点然后顺序遍历即可。

**B+Tree的查询过程**从根节点开始，根据索引键值与节点中存储的键值比较，确定下一级子节点的指针，重复这个过程直到达到叶子节点。在叶子节点中找到对应的数据或数据位置。这种从根到叶子的路径长度称为树的高度，B+Tree的设计保证了树的高度通常在3到4层之间，这意味着最多只需要3到4次磁盘IO就能找到任何数据。

**InnoDB的B+Tree实现**中，主键索引（聚集索引）的叶子节点存储完整的行数据，而辅助索引的叶子节点存储主键值。这意味着通过辅助索引查询时，需要先查到主键，再通过主键索引查到完整数据，这个过程称为回表。如果查询的字段都在辅助索引中包含，可以避免回表，这就是覆盖索引的概念。

**B+Tree的维护代价**是插入和删除操作可能需要分裂或合并节点。InnoDB通过页分裂机制来处理插入，当一个页满了之后，会分裂成两个页。删除操作可能会触发页合并，以保持页的利用率。频繁的页分裂和合并会影响索引性能，这也是为什么建议使用单调递增的主键（如自增ID）来避免随机插入导致的页分裂。

### 2.2 索引设计原则

索引设计是数据库性能优化的关键，一个好的索引设计可以大幅提升查询性能，而一个糟糕的索引设计不仅浪费空间，还可能拖累写性能。

**选择合适的索引列**需要考虑查询条件和数据分布。索引应该创建在WHERE子句、JOIN条件和ORDER BY子句中使用的列上。对于高选择性的列（唯一值多）更适合创建索引，而低选择性的列（如性别、状态标志）即使创建了索引，查询优化器也可能选择全表扫描。

**复合索引的设计**需要遵循最左前缀原则。复合索引是多个列组合创建的索引，索引的列顺序非常重要。假设有一个索引（col1, col2, col3），则这个索引可以加速col1、col1+col2、col1+col2+col3的查询，但不能单独加速col2或col3的查询。因此，应该把选择性高的列放在前面，区分度大的列放在前面。

**覆盖索引的使用**可以避免回表，提升查询性能。如果一个查询的所有字段都包含在某个索引中，查询只需要扫描索引而不需要访问表数据。例如，如果经常需要查询用户的ID和姓名，而有一个索引（ID, name），则查询ID和name时直接可以从索引中获取，无需回表。

**索引长度与性能**需要权衡。索引列的前缀索引可以减少索引占用的空间，但会降低索引的选择性。对于很长的字符串列（如URL、文件名），可以考虑创建前缀索引。例如，对于一个存储URL的列，可以只索引前20个字符。但要注意，前缀索引不支持ORDER BY和GROUP BY，也不支持索引覆盖扫描。

### 2.3 最左前缀原则

最左前缀原则是理解复合索引的关键，也是正确使用复合索引的基础。

**最左前缀原则的定义**是：复合索引遵循最左边的列开始的顺序使用。例如，对于索引（a, b, c），查询条件可以只使用列a，也可以使用a和b的组合，但不能只使用b或c。这是因为索引在构建时按照从左到右的顺序排列，只有前面的列值确定了，后面的列值才有意义。

**为什么需要最左前缀**这与B+Tree的结构有关。B+Tree按照索引列从左到右的顺序构建树结构。第一列是树的第一级索引，只有第一列值确定了，才能确定第二列的值在树中的位置。打个比方，就像按姓名排序的字典，你先按姓排序，姓相同的情况下才按名排序。如果只查名，就无法快速定位。

**索引失效的情况**包括：查询条件不包含索引的最左列；查询条件使用了函数或表达式；查询条件使用LIKE以通配符开头（如LIKE '%abc'）；查询条件的数据类型不匹配（如字符串列用数字查询）；使用OR连接但OR两边的列不在同一索引中。

**最左前缀的实战应用**：

```sql
-- 创建复合索引
CREATE INDEX idx_user_status_created ON users(status, created_at, email);

-- 以下查询可以使用索引（遵循最左前缀）
SELECT * FROM users WHERE status = 'active';
SELECT * FROM users WHERE status = 'active' AND created_at > '2024-01-01';
SELECT * FROM users WHERE status = 'active' AND created_at > '2024-01-01' AND email LIKE 'a%';

-- 以下查询无法使用完整索引（不遵循最左前缀）
SELECT * FROM users WHERE created_at > '2024-01-01';  -- 跳过最左列status
SELECT * FROM users WHERE email LIKE 'a%';              -- 跳过最左两列
SELECT * FROM users WHERE created_at > '2024-01-01' AND email LIKE 'a%';
```

**索引列顺序的选择**需要考虑以下因素：将区分度高的列放在前面可以快速过滤数据；查询条件中经常使用的列优先；考虑查询是否覆盖索引；避免重复创建索引。

### 2.4 实战：索引优化

索引优化是数据库性能调优的核心工作，需要结合实际查询场景进行分析和优化。

**识别需要优化的查询**：

**索引优化实战案例**：

```sql
-- 原始查询（性能差）
SELECT * FROM orders
WHERE YEAR(created_at) = 2024
AND MONTH(created_at) = 3
AND status = 'completed'
ORDER BY created_at DESC
LIMIT 100;

-- 问题分析：
-- 1. YEAR()和MONTH()函数导致索引失效
-- 2. 使用SELECT * 可能导致回表
-- 3. 排序字段没有利用索引

-- 优化方案1：使用日期范围（推荐）
SELECT order_id, user_id, total_amount, created_at, status
FROM orders
WHERE created_at >= '2024-03-01'
AND created_at < '2024-04-01'
AND status = 'completed'
ORDER BY created_at DESC
LIMIT 100;

-- 创建复合索引
CREATE INDEX idx_orders_date_status ON orders(created_at, status);

-- 优化方案2：覆盖索引避免回表
-- 如果需要查询的字段都在索引中，可以创建覆盖索引
CREATE INDEX idx_orders_cover ON orders(created_at, status, order_id, user_id, total_amount);

-- 优化后的查询（利用覆盖索引）
SELECT order_id, user_id, total_amount, created_at, status
FROM orders
WHERE created_at >= '2024-03-01'
AND created_at < '2024-04-01'
AND status = 'completed'
ORDER BY created_at DESC;
```

**NestJS + TypeORM 索引优化实践**：

```typescript
import { Entity, Column, Index, PrimaryGeneratedColumn } from 'typeorm';

/**
 * 订单实体 - 展示索引设计
 * 索引设计的核心原则：
 * 1. 根据查询模式设计索引
 * 2. 复合索引遵循最左前缀原则
 * 3. 覆盖索引减少回表
 */
@Entity('orders')
@Index('idx_orders_user_status', ['userId', 'status'])  // 用户+状态组合查询
@Index('idx_orders_created_status', ['createdAt', 'status'])  // 日期+状态组合查询
@Index('idx_orders_cover', ['createdAt', 'status', 'orderId', 'userId'])  // 覆盖索引
export class Order {

   * 按日期范围查询订单（优化后）
   * 利用复合索引 idx_orders_created_status 和覆盖索引 idx_orders_cover

  async findOrdersByDateRange(
    startDate: Date,
    endDate: Date,
    status?: string,

      .createQueryBuilder('order')
      .select(['order.orderId', 'order.userId', 'order.totalAmount', 'order.createdAt', 'order.status'])
      .where('order.createdAt >= :startDate', { startDate })
      .andWhere('order.createdAt < :endDate', { endDate });

    if (status) {
      queryBuilder.andWhere('order.status = :status', { status });
    }

    // 利用覆盖索引，直接从索引获取数据，无需回表
    queryBuilder.orderBy('order.createdAt', 'DESC');

    return queryBuilder.getMany();
  }

  /**
   * 统计每日订单数据（优化后）
   * 利用索引避免排序和全表扫描
   */
  async getDailyOrderStats(startDate: Date, endDate: Date): Promise<any[]> {
    return this.orderRepository
      .createQueryBuilder('order')
      .select('DATE(order.createdAt)', 'date')
      .addSelect('COUNT(*)', 'totalOrders')
      .addSelect('SUM(order.totalAmount)', 'totalAmount')
      .where('order.createdAt >= :startDate', { startDate })
      .andWhere('order.createdAt < :endDate', { endDate })
      .andWhere('order.status = :status', { status: 'completed' })
      .groupBy('DATE(order.createdAt)')
      .orderBy('date', 'ASC')
      // 利用created_at索引，查询结果已排序，无需filesort
      .getRawMany();
  }
```

## 三、慢查询

### 3.1 EXPLAIN 分析

EXPLAIN是MySQL中用于分析查询执行计划的核心命令，掌握EXPLAIN的输出是进行SQL优化的基础。

**EXPLAIN的基本用法**是在查询语句前添加EXPLAIN关键字：

```sql
EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';
EXPLAIN FORMAT=JSON SELECT * FROM users WHERE email = 'test@example.com';  -- JSON格式详情
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'test@example.com';  -- MySQL 8.0+，包含实际执行信息
```

**EXPLAIN输出字段详解**：

id列表示查询中SELECT的序号，嵌套查询中id越大越先执行。select_type列表示查询类型，SIMPLE表示简单SELECT，PRIMARY表示外层查询，SUBQUERY表示子查询，DERIVED表示FROM子句的子查询，UNION表示UNION查询。

table列表示涉及的表名。type列表示关联类型，从好到差依次是：system（表只有一行）、const（最多匹配一行）、eq_ref（唯一索引扫描）、ref（非唯一索引扫描）、range（索引范围扫描）、index（全索引扫描）、ALL（全表扫描）。实际优化中至少要达到ref级别，避免ALL。

possible_keys列表示可能使用的索引，key列表示实际使用的索引。key_len列表示索引使用的字节数，可以判断复合索引使用了多少列。ref列表示索引比较的列或常量。rows列表示预计扫描的行数，越少越好。filtered列表示返回结果的行数占扫描行数的百分比。Extra列包含额外信息，如Using index（使用覆盖索引）、Using where（使用WHERE过滤）、Using temporary（使用临时表）、Using filesort（需要文件排序）等。

### 3.2 索引失效场景

索引失效是导致慢查询的常见原因，理解索引失效的场景可以避免很多性能问题。

**函数和表达式导致索引失效**是最常见的问题之一：

```sql
-- 索引失效：WHERE YEAR(created_at) = 2024
SELECT * FROM orders WHERE YEAR(created_at) = 2024;

-- 索引生效：使用范围查询
SELECT * FROM orders WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';

-- 索引失效：WHERE LOWER(email) = 'test'
SELECT * FROM users WHERE LOWER(email) = 'test';

-- 索引失效：WHERE id + 1 = 10
SELECT * FROM users WHERE id + 1 = 10;
```

**类型转换导致索引失效**也需要注意：

```sql
-- 假设phone是varchar类型
-- 索引失效：使用数字查询
SELECT * FROM users WHERE phone = 13800138000;

-- 索引生效：使用字符串查询
SELECT * FROM users WHERE phone = '13800138000';
```

**LIKE模式导致索引失效**的情况：

```sql
-- phone字段有索引
-- 索引失效：LIKE以通配符开头
SELECT * FROM users WHERE phone LIKE '%138%';

-- 索引生效：LIKE以固定字符开头
SELECT * FROM users WHERE phone LIKE '138%';
```

**OR条件导致索引失效**的情况：

```sql
-- 索引失效：OR两边有不同的条件
SELECT * FROM users WHERE name = '张三' OR email = 'zhangsan@example.com';

-- 索引生效：使用UNION
SELECT * FROM users WHERE name = '张三'
UNION
SELECT * FROM users WHERE email = 'zhangsan@example.com';

-- 索引失效：OR两边都没有索引
SELECT * FROM users WHERE name = '张三' OR age = 25;
```

**复合索引最左前缀原则**导致的索引失效：

```sql
-- 复合索引 (status, created_at, user_id)
-- 索引失效：跳过第一列
SELECT * FROM orders WHERE created_at > '2024-01-01';

-- 索引失效：中间跳过某一列
SELECT * FROM orders WHERE status = 'completed' AND user_id = '123';
```

### 3.3 查询优化技巧

掌握查询优化技巧可以显著提升数据库性能。

**避免SELECT *** 是优化查询的基本原则。SELECT * 会返回所有列，增加网络传输和内存消耗，更重要的是可能无法利用覆盖索引：

```sql
-- 低效：返回所有列
SELECT * FROM orders WHERE order_id = '12345';

-- 高效：只返回需要的列
SELECT order_id, user_id, total_amount, status
FROM orders WHERE order_id = '12345';
```

**分页查询优化**对于大表分页至关重要：

```sql
-- 低效的深度分页：偏移量越大越慢
SELECT * FROM orders ORDER BY created_at DESC LIMIT 1000000, 20;

-- 高效的游标分页：利用索引快速定位
SELECT * FROM orders
WHERE created_at < '2024-01-01'
ORDER BY created_at DESC
LIMIT 20;

-- 优化方案：记录上次的最大ID
SELECT * FROM orders
WHERE id < #{lastId}
ORDER BY id DESC
LIMIT 20;
```

**JOIN优化**需要注意的事项：

```sql
-- 小表驱动大表：MySQL优化器会自动选择，但明确指定更好
SELECT * FROM orders o
INNER JOIN users u ON o.user_id = u.id
WHERE u.status = 'vip';

-- 确保JOIN字段有索引
-- orders.user_id 和 users.id 都应该有索引

-- 避免SELECT * ，只选择需要的字段
SELECT o.order_id, o.total_amount, u.name, u.email
FROM orders o
INNER JOIN users u ON o.user_id = u.id;
```

**聚合查询优化**：

```sql
-- 在应用层处理：如果只需要估算值，可以使用SQL_CALC_FOUND_ROWS
SELECT SQL_CALC_FOUND_ROWS * FROM orders LIMIT 10;
SELECT FOUND_ROWS();  -- 获取总行数（不推荐，效率低）

-- 优化方案：使用EXPLAIN预估，或维护计数表
SELECT COUNT(*) FROM orders WHERE status = 'completed';

-- 实时计数需求：使用触发器或程序维护计数表
-- orders_count: { status: 'completed', count: 12345 }
```

### 3.4 实战：慢查询优化

以下是一个完整的慢查询优化实战案例。

**场景描述**：订单查询系统，随着数据量增长，查询变得越来越慢。

**原始查询分析**：

```sql
-- 慢查询：平均执行时间 3.5秒
SELECT o.*, u.name, u.email, p.product_name
FROM orders o
LEFT JOIN users u ON o.user_id = u.id
LEFT JOIN products p ON o.product_id = p.id
WHERE o.status IN ('pending', 'processing', 'completed')
AND o.created_at BETWEEN '2024-01-01' AND '2024-12-31'
ORDER BY o.created_at DESC
LIMIT 100;
```

**使用EXPLAIN分析**：

```sql
EXPLAIN SELECT o.*, u.name, u.email, p.product_name
FROM orders o
LEFT JOIN users u ON o.user_id = u.id
LEFT JOIN products p ON o.product_id = p.id
WHERE o.status IN ('pending', 'processing', 'completed')
AND o.created_at BETWEEN '2024-01-01' AND '2024-12-31'
ORDER BY o.created_at DESC
LIMIT 100;

-- 分析结果：
-- type: ALL（全表扫描）-- 索引失效
-- rows: 5000000（扫描500万行）-- 数据量大
-- Extra: Using filesort（文件排序）-- 没有利用索引排序
-- Extra: Using where（WHERE过滤）
```

**优化步骤**：

```sql
-- 步骤1：优化WHERE条件
-- 索引设计：created_at + status 复合索引
CREATE INDEX idx_orders_date_status ON orders(created_at, status);

-- 步骤2：避免IN，改用OR连接（IN有时会导致索引失效）
-- MySQL优化器对IN的处理有时不佳

-- 步骤3：只查询需要的字段，使用覆盖索引
CREATE INDEX idx_orders_cover ON orders(
  created_at,
  status,
  id,
  user_id,
  product_id,
  total_amount
);

-- 步骤4：确保JOIN字段有索引
-- users.id 应该有主键索引
CREATE INDEX idx_users_id ON users(id);  -- 通常已有
-- products.id 应该有主键索引
CREATE INDEX idx_products_id ON products(id);  -- 通常已有

-- 步骤5：添加JOIN字段索引（如果user_id和product_id不是外键）
CREATE INDEX idx_orders_user_id ON orders(user_id);
CREATE INDEX idx_orders_product_id ON orders(product_id);
```

**优化后的查询**：

```sql
-- 优化后查询：执行时间 50毫秒
SELECT
  o.id AS order_id,
  o.user_id,
  o.product_id,
  o.total_amount,
  o.status,
  o.created_at,
  u.name AS user_name,
  u.email AS user_email,
  p.product_name
FROM orders o
INNER JOIN users u ON o.user_id = u.id
INNER JOIN products p ON o.product_id = p.id
WHERE o.created_at >= '2024-01-01'
AND o.created_at < '2025-01-01'
AND o.status = 'completed'
ORDER BY o.created_at DESC
LIMIT 100;

-- 使用覆盖索引，无需回表
CREATE INDEX idx_orders_final ON orders(
  created_at,
  status,
  id,
  user_id,
  product_id,
  total_amount
);
```

## 四、分库分表

### 4.1 垂直分表

垂直分表是将一个大表按列拆分到多个表中，是数据库优化的常用手段。

**垂直分表的原理**是基于列的相关性将表拆分为多个子表。一般将访问频率高的热点字段放在一个表，访问频率低的冷数据放在另一个表。对于订单表，可以将订单基础信息（订单ID、用户ID、订单状态、总金额）放在主表，订单详情、商品快照等信息放在详情表。

**垂直分表的优势**包括：减少单表字段数量，降低表结构的复杂度；提高缓存效率，热点数据更集中；提升查询性能，减少磁盘IO；便于独立扩展不同表的容量。

**垂直分表的实施步骤**：首先分析表中字段的访问频率和数据量；然后将字段按访问模式分为热点字段和冷门字段；接着创建新的子表结构；最后通过迁移程序将数据迁移到新表，并建立表之间的关联关系。

**实战案例**：

```sql
-- 原始订单表：orders（约50个字段）
CREATE TABLE orders (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  order_no VARCHAR(32) NOT NULL,

  buyer_remark TEXT,
  seller_remark TEXT,
  created_at DATETIME,
  updated_at DATETIME,
  -- ... 还有40多个字段
);

-- 垂直拆分后：

-- 订单主表：orders（核心字段，访问频率高）
```

### 4.2 水平分表

水平分表是将一个大表按行拆分到多个表中，每个表结构相同但数据不同。

**水平分表的原理**是将大表的数据按某种规则（如按ID范围、按时间、按哈希）分散到多个表中。例如，按月份分表可以将每个月的数据放在单独的表中。

**水平分表的方式**：按范围分表适用于时间序列数据，如日志表、订单表，可以按月份或按年分表，新数据写入新表，历史数据保持不变。按哈希分表将数据均匀分布到多个表中，适合数据分布均匀的场景，但对范围查询不友好。

**分片键的选择**是水平分表的核心。分片键必须满足高选择性和数据分布均匀两个条件。常见的选择包括用户ID（适合用户相关的查询）、时间字段（适合时间序列查询）、订单ID（哈希后分布均匀）。

**实战案例**：

```sql
-- 按月份分表的订单表
-- 分表命名：orders_202401, orders_202402, ...

-- 创建分表的存储过程
DELIMITER //
CREATE PROCEDURE create_orders_monthly_table(IN table_suffix VARCHAR(6))
BEGIN
  SET @table_name = CONCAT('orders_', table_suffix);
  SET @sql = CONCAT('CREATE TABLE IF NOT EXISTS ', @table_name, ' (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no VARCHAR(32) NOT NULL,
    user_id BIGINT NOT NULL,
    status VARCHAR(20) NOT NULL,
    total_amount DECIMAL(10,2) NOT NULL,
    created_at DATETIME,
    updated_at DATETIME,
    INDEX idx_user_id (user_id),
    INDEX idx_status (status),
    INDEX idx_created_at (created_at)
  ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4');

  PREPARE stmt FROM @sql;
  EXECUTE stmt;
  DEALLOCATE PREPARE stmt;
END //
DELIMITER ;

-- 调用存储过程创建2024年1月的订单表
CALL create_orders_monthly_table('202401');
```

### 4.3 分片键设计

分片键是水平分表的关键，选择合适的分片键直接影响查询性能和扩展性。

**分片键的选择原则**：首先，分片键应该是查询中最常用的条件字段，这样查询可以直接定位到某个分片，避免跨分片查询。其次，分片键的值应该分布均匀，避免产生数据倾斜导致某些分片数据量过大。第三，分片键应该是不可变的，避免数据迁移。

**常见的分片键策略**：

按用户ID分片是最常见的策略，适合用户相关的业务场景，用户的所有数据都在同一分片，查询效率高。按时间分片适合日志、订单等时间序列数据，可以方便地进行历史数据归档。按地区分片适合O2O等地域相关的业务。

**跨分片查询的处理**是分库分表中最复杂的问题：对于能够直接定位分片的查询，直接查询目标分片；对于无法定位分片的查询，需要广播到所有分片然后聚合结果；对于分页查询，需要从多个分片获取数据然后排序分页。

**实战案例**：

```typescript
/**
 * 分库分表路由服务
 * 展示如何根据分片键定位数据存储位置
 */
@Injectable()
export class ShardingRouterService {
  /**
   * 用户库分片路由
   * 用户ID取模，分配到不同的数据库
   */
  getUserDatabase(userId: string): string {
    const hash = this.hashCode(userId);
    const dbIndex = hash % 4;  // 分配到4个库
    return `user_db_${dbIndex}`;
  }

  /**
   * 订单表分片路由
   * 按用户ID哈希，分配到256个表
   */
  getOrderTableName(userId: string, orderId?: string): string {
    const hash = this.hashCode(userId);
    const tableIndex = hash % 256;  // 256个表
    return `orders_${tableIndex.toString(16).padStart(2, '0')}`;
  }

  /**
   * 时间分片路由
   * 按月份分表
   */
  getMonthlyTableName(date: Date): string {
    const year = date.getFullYear();
    const month = String(date.getMonth() + 1).padStart(2, '0');
    return `logs_${year}${month}`;
  }

  /**
   * 复合分片键路由
   * 适用于需要按多个字段查询的场景
   * 使用一致性哈希环进行路由
   */
  getShardByCompositeKey(keys: string[]): string {
    const combinedKey = keys.sort().join(':');
    const hash = this.hashCode(combinedKey);
    const totalShards = 64;
    const shardIndex = Math.abs(hash % totalShards);
    return `shard_${shardIndex.toString(16).padStart(2, '0')}`;
  }

  /**
   * 跨分片查询
   * 从多个分片并行获取数据
   */
  async queryAcrossShards(
    userId: string,
    startDate: Date,
    endDate: Date,
  ): Promise<Order[]> {
    // 计算需要查询的分片范围
    const startMonth = this.getMonthKey(startDate);
    const endMonth = this.getMonthKey(endDate);

    // 获取所有需要查询的表
    const tables = this.getMonthlyTableRange(startMonth, endMonth);

    // 并行查询所有分片
    const promises = tables.map(table => {
      return this.dataSource.query(
        `SELECT * FROM ${table}
         WHERE user_id = ? AND created_at >= ? AND created_at < ?`,
        [userId, startDate, endDate]
      );
    });

    // 聚合结果
    const results = await Promise.all(promises);
    return results.flat().sort((a, b) => b.created_at - a.created_at);
  }

  private hashCode(str: string): number {
    let hash = 0;
    for (let i = 0; i < str.length; i++) {
      const char = str.charCodeAt(i);
      hash = ((hash << 5) - hash) + char;
      hash = hash & hash;  // 转换为32位整数
    }
    return hash;
  }

  private getMonthKey(date: Date): string {
    const year = date.getFullYear();
    const month = String(date.getMonth() + 1).padStart(2, '0');
    return `${year}${month}`;
  }

  private getMonthlyTableRange(start: string, end: string): string[] {
    const tables: string[] = [];
    let current = start;
    while (current <= end) {
      tables.push(`logs_${current}`);
      // 移动到下一个月
      const year = parseInt(current.substring(0, 4));
      const month = parseInt(current.substring(4, 6));
      const nextMonth = month === 12 ? 1 : month + 1;
      const nextYear = month === 12 ? year + 1 : year;
      current = `${nextYear}${String(nextMonth).padStart(2, '0')}`;
    }
    return tables;
  }
}
```

### 4.4 实战：分表设计

以下是一个完整的分库分表设计实战案例。

**场景描述**：订单系统需要支持日订单量100万，年订单量3.6亿，需要设计可扩展的分表方案。

**设计方案**：采用按用户ID哈希分表，256张订单表分布在8个数据库中，每个库32张表。

```typescript
/**
 * 订单分库分表配置服务
 */
@Injectable()
export class OrderShardingConfig {
  // 总共8个数据库
  private readonly DATABASE_COUNT = 8;
  // 每个库32张表
  private readonly TABLE_PER_DB = 32;
  // 总表数
  readonly TOTAL_TABLES = this.DATABASE_COUNT * this.TABLE_PER_DB;

  /**
   * 根据用户ID计算分片信息
   */
  calculateShard(userId: string): { database: string; table: string } {
    const hash = this.hashCode(userId);
    const dbIndex = hash % this.DATABASE_COUNT;
    const tableIndex = Math.floor(hash / this.DATABASE_COUNT) % this.TABLE_PER_DB;

    return {
      database: `order_db_${dbIndex}`,
      table: `orders_${tableIndex.toString(16).padStart(2, '0')}`,
    };
  }

  /**
   * 根据订单ID计算分片信息
   * 订单ID格式：时间戳 + 随机数 + 用户ID哈希
   * 前8位是时间戳，可以用于范围查询
   */
  calculateShardByOrderId(orderId: string): { database: string; table: string } {
    // 从订单ID中提取用户ID部分
    const userIdPart = orderId.substring(16, 32);
    return this.calculateShard(userIdPart);
  }

  /**
   * 获取所有分片的连接信息
   */
  getAllShardConnections(): { database: string; tables: string[] }[] {
    const shards = [];
    for (let db = 0; db < this.DATABASE_COUNT; db++) {
      const tables = [];
      for (let t = 0; t < this.TABLE_PER_DB; t++) {
        tables.push(`orders_${t.toString(16).padStart(2, '0')}`);
      }
      shards.push({
        database: `order_db_${db}`,
        tables,
      });
    }
    return shards;
  }

  /**
   * 生成建表SQL
   */
  generateCreateTableSQL(): string[] {
    const sqls = [];
    for (let db = 0; db < this.DATABASE_COUNT; db++) {
      for (let t = 0; t < this.TABLE_PER_DB; t++) {
        const tableName = `orders_${t.toString(16).padStart(2, '0')}`;
        const sql = `
CREATE TABLE ${tableName} (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  order_id VARCHAR(64) NOT NULL COMMENT '订单ID',
  user_id BIGINT NOT NULL COMMENT '用户ID',
  status VARCHAR(20) NOT NULL COMMENT '订单状态',
  total_amount DECIMAL(10,2) NOT NULL COMMENT '订单金额',

  payment_time DATETIME COMMENT '支付时间',

  INDEX idx_user_status (user_id, status),
  INDEX idx_created_at (created_at),
  INDEX idx_status_created (status, created_at)

        `.trim();
        sqls.push(sql);
      }
    }
    return sqls;
  }

  private hashCode(str: string): number {
    let hash = 0;
    for (let i = 0; i < str.length; i++) {
      const char = str.charCodeAt(i);
      hash = ((hash << 5) - hash) + char;
      hash = hash & hash;
    }
    return hash;

@Injectable()
export class OrderShardService {
  constructor(
    private orderShardingConfig: OrderShardingConfig,
    @Inject('DATA_SOURCE_MAP')
    private dataSourceMap: Map<string, DataSource>,
  ) {}

  /**
   * 插入订单
   * 根据用户ID路由到正确的分片
   */
  async insertOrder(order: Order): Promise<string> {
    const { database, table } = this.orderShardingConfig.calculateShard(
      order.userId.toString()
    );
    const dataSource = this.dataSourceMap.get(database);

    const sql = `
      INSERT INTO ${table}
      (order_id, user_id, status, total_amount, payment_method, created_at, updated_at)
      VALUES (?, ?, ?, ?, ?, ?, ?)
    `;

    await dataSource.query(sql, [
      order.orderId,
      order.userId,
      order.status,
      order.totalAmount,
      order.paymentMethod,
      order.createdAt,
      order.updatedAt,
    ]);

    return order.orderId;
  }

  /**
   * 按用户ID查询订单
   * 单分片查询，性能最佳
   */
  async findOrdersByUserId(
    userId: string,
    options?: { status?: string; limit?: number },
  ): Promise<Order[]> {
    const { database, table } = this.orderShardingConfig.calculateShard(userId);
    const dataSource = this.dataSourceMap.get(database);

    let sql = `SELECT * FROM ${table} WHERE user_id = ?`;
    const params: any[] = [userId];

    if (options?.status) {
      sql += ` AND status = ?`;
      params.push(options.status);
    }

    sql += ` ORDER BY created_at DESC`;

    if (options?.limit) {
      sql += ` LIMIT ?`;
      params.push(options.limit);
    }

    return dataSource.query(sql, params);
  }

  /**
   * 按时间范围查询订单
   * 需要扫描多个分片
   */
  async findOrdersByDateRange(
    startDate: Date,
    endDate: Date,
    limit: number = 100,
  ): Promise<Order[]> {
    // 按时间范围分片需要扫描所有分片
    const allShards = this.orderShardingConfig.getAllShardConnections();
    const promises: Promise<Order[]>[] = [];

    // 并行查询所有分片
    for (const shard of allShards) {
      for (const table of shard.tables) {
        const dataSource = this.dataSourceMap.get(shard.database);
        promises.push(
          dataSource.query(
            `SELECT * FROM ${table}
             WHERE created_at >= ? AND created_at < ?
             ORDER BY created_at DESC
             LIMIT ?`,
            [startDate, endDate, limit]
          )
        );
      }
    }

    // 等待所有查询完成
    const results = await Promise.all(promises);
    const orders = results.flat();

    // 合并后排序
    orders.sort((a, b) => b.created_at - a.created_at);

    return orders.slice(0, limit);
  }

  /**
   * 统计订单数量
   * 多分片聚合查询
   */
  async countOrders(status?: string): Promise<number> {
    const allShards = this.orderShardingConfig.getAllShardConnections();
    const promises: Promise<number>[] = [];

    for (const shard of allShards) {
      for (const table of shard.tables) {
        const dataSource = this.dataSourceMap.get(shard.database);
        let sql = `SELECT COUNT(*) as total FROM ${table}`;
        const params: string[] = [];

        if (status) {
          sql += ` WHERE status = ?`;
          params.push(status);
        }

        promises.push(
          dataSource.query(sql, params).then((result: any) => result[0].total)
        );
      }
    }

    const counts = await Promise.all(promises);
    return counts.reduce((sum, count) => sum + count, 0);
  }
}
```

## 五、主从复制

### 5.1 复制原理

MySQL主从复制是构建高可用数据库系统的基础，它允许将主库的数据变更同步到从库，实现数据冗余和读写分离。

**主从复制的核心原理**基于二进制日志（Binary Log）。当主库执行数据变更（如INSERT、UPDATE、DELETE）时，这些变更会以事件的形式记录到二进制日志中。从库通过IO线程连接到主库，请求主库发送二进制日志内容，主库通过Dump线程发送日志。从库的IO线程接收日志后写入本地中继日志（Relay Log），然后由SQL线程重放中继日志中的事件，实现数据同步。

**主从复制的数据流**：主库数据变更 -> 写入二进制日志 -> 从库IO线程读取日志 -> 写入从库中继日志 -> 从库SQL线程重放日志 -> 数据同步完成。这个过程是异步的，主库不需要等待从库确认就可以继续处理后续请求。

**GTID（全局事务ID）**是MySQL 5.6引入的特性，它为每个事务分配一个唯一的标识符，大大简化了主从复制的管理和故障恢复。使用GTID时，从库不需要指定二进制日志文件名和位置，只需要知道主库的GTID即可。GTID格式为server_uuid:transaction_id，例如a1b2c3d4:1-100。

**复制的配置参数**：

```bash
# 主库配置 (my.cnf)
[mysqld]
server-id = 1  # 唯一标识符
log-bin = /var/log/mysql/mysql-bin  # 二进制日志文件前缀
binlog_format = ROW  # 推荐使用ROW格式，兼容性更好
expire_logs_days = 7  # 二进制日志保留天数
sync_binlog = 1  # 每次事务提交同步二进制日志到磁盘

# 从库配置 (my.cnf)
[mysqld]
server-id = 2  # 唯一标识符，与主库不同
relay-log = /var/log/mysql/mysql-relay  # 中继日志文件前缀
read_only = ON  # 从库只读（但root用户仍然可以写）
super_read_only = ON  # 彻底只读，连super权限也无法写入
log_slave_updates = ON  # 从库写入的变更也记录到自己的二进制日志
```

### 5.2 读写分离

读写分离是主从复制最常见的应用场景，它将读操作分配到从库，写操作发送到主库，从而提升数据库的整体处理能力。

**读写分离的实现方式**：应用层实现是最灵活的方式，通过在代码中区分读和写操作，分别路由到不同的数据库实例。中间件实现使用如MySQL Proxy、Atlas、ShardingSphere等中间件自动实现读写分离。数据库驱动层实现使用支持读写分离的数据库驱动。

**读写分离的挑战**：数据一致性问题。由于复制是异步的，写入主库后立即读取从库可能读到旧数据。解决方案包括：读操作也路由到主库（强一致性场景）；使用半同步复制减少延迟；应用层处理短暂的数据不一致。复制延迟问题。长时间运行的查询会阻塞复制，导致从库数据滞后。解决方案包括：监控复制延迟；选择合适的场景使用从库；处理延迟告警。

```typescript
 * 多数据源管理器
 * 管理主库和多个从库的连接

@Injectable()
export class MultiDatabaseManager {
  private masterDataSource: DataSource;
  private replicaDataSources: DataSource[] = [];
  private currentReplicaIndex = 0;

  constructor(
    @Inject('MASTER_DATASOURCE') masterDataSource: DataSource,
    @Inject('REPLICA_DATASOURCES') replicaDataSources: DataSource[],
  ) {
    this.masterDataSource = masterDataSource;
    this.replicaDataSources = replicaDataSources;
  }

  /**
   * 获取主库数据源
   */
  getMaster(): DataSource {
    return this.masterDataSource;
  }

  /**
   * 获取从库数据源（轮询）
   */
  getReplica(): DataSource {
    if (this.replicaDataSources.length === 0) {
      return this.masterDataSource;
    }
    const replica = this.replicaDataSources[this.currentReplicaIndex];
    this.currentReplicaIndex =
      (this.currentReplicaIndex + 1) % this.replicaDataSources.length;
    return replica;
  }

  /**
   * 根据操作类型获取数据源
   */
  getDataSource(isReadonly: boolean): DataSource {
    return isReadonly ? this.getReplica() : this.getMaster();
  }
}

   */
  @ReadOnly()

    return db.manager.findOne(Order, { where: { orderId } });
  }

  /**
   * 查询用户订单列表（读操作，使用从库）
   */
  @ReadOnly()
  async findUserOrders(userId: string): Promise<Order[]> {
    const db = this.dbManager.getReplica();
    return db.manager.find(Order, {
      where: { userId },
      order: { createdAt: 'DESC' },
    });
  }

  /**
   * 创建订单（写操作，使用主库）
   */
  async createOrder(order: CreateOrderDto): Promise<Order> {
    const db = this.dbManager.getMaster();

    // 在事务中执行
    const queryRunner = db.createQueryRunner();
    await queryRunner.connect();
    await queryRunner.startTransaction();

    try {
      const newOrder = queryRunner.manager.create(Order, order);
      const savedOrder = await queryRunner.manager.save(newOrder);
      await queryRunner.commitTransaction();
      return savedOrder;
    } catch (error) {
      await queryRunner.rollbackTransaction();
      throw error;
    } finally {
      await queryRunner.release();
    }
  }

  /**
   * 统计订单（读操作，但要求强一致性，使用主库）
   * 虽然是读操作，但需要最新数据，必须走主库
   */
  async countRecentOrders(): Promise<number> {
    // 关键数据读取，不使用从库
    const db = this.dbManager.getMaster();
    return db.manager.count(Order, {
      where: {
        createdAt: MoreThan(new Date(Date.now() - 24 * 60 * 60 * 1000)),
      },
    });
  }
```

### 5.3 复制延迟问题

复制延迟是主从复制中最常见的问题，理解延迟的原因和解决方法对系统稳定性至关重要。

**延迟产生的原因**：从库处理能力不足。当主库写入压力大时，从库的SQL线程可能跟不上主库的Dump线程，导致中继日志堆积。网络延迟。主从之间的网络延迟或带宽不足会导致日志传输慢。从库负载过高。从库承担了大量查询负载，影响了复制日志的处理。大事务。主库执行了一个长时间运行的事务，产生大量日志，从库需要较长时间重放。

**监控复制延迟**：

```sql
-- 方法1：通过Show Slave Status查看
SHOW SLAVE STATUS\G;
-- Seconds_Behind_Master: 从库SQL线程相对于主库Dump线程的延迟秒数
-- Read_Master_Log_Pos: 已读取的主库日志位置
-- Relay_Log_Space: 中继日志总大小

-- 方法2：使用Performance Schema监控
SELECT * FROM performance_schema.replication_connection_status;
SELECT * FROM performance_schema.replication_applier_status;

-- 方法3：监控GTID模式下的延迟
SHOW SLAVE STATUS NONBLOCKING;
-- 或使用SHOW REPLICA STATUS (MySQL 8.0)
SHOW REPLICA STATUS\G;
```

**解决延迟的方法**：

```typescript
/**
 * 延迟监控服务
 */
@Injectable()
export class ReplicationLagMonitor {
  @Interval(10000)  // 每10秒检查一次
  async checkReplicationLag() {
    const lag = await this.getReplicaLag();

    if (lag > 30) {  // 超过30秒告警
      await this.sendAlert('Replication lag exceeds 30 seconds', lag);
    }

    if (lag > 60) {  // 超过60秒，触发自动切换
      await this.handleSevereLag(lag);
    }
  }

  private async getReplicaLag(): Promise<number> {
    // 通过Show Slave Status获取延迟
    const result = await this.masterDataSource.query('SHOW SLAVE STATUS');
    return result[0]?.Seconds_Behind_Master || 0;
  }

  private async handleSevereLag(lag: number) {
    // 1. 记录详细日志
    this.logger.error(`Severe replication lag detected: ${lag}s`);

    // 2. 可以考虑自动将从库切换为临时主库
    // 或暂停部分从库的查询，将资源用于追赶复制

    // 3. 发送告警通知DBA
    await this.notificationService.sendAlert({
      level: 'critical',
      message: `Replication lag: ${lag}s`,
      timestamp: new Date(),
    });
  }
}

/**
 * 读操作延迟路由策略
 * 根据从库延迟程度决定是否路由到从库
 */
@Injectable()
export class SmartReadRouter {
  private replicaLags: Map<string, number> = new Map();

  /**
   * 更新从库延迟
   */
  updateReplicaLag(replicaId: string, lag: number) {
    this.replicaLags.set(replicaId, lag);
  }

  /**
   * 获取可用的从库
   * 如果所有从库延迟都超过阈值，返回主库
   */
  getAvailableReplica(): string {
    const maxAllowedLag = 5;  // 允许的最大延迟（秒）

    for (const [replicaId, lag] of this.replicaLags) {
      if (lag < maxAllowedLag) {
        return replicaId;
      }
    }

    // 所有从库延迟都超过阈值，返回主库
    return 'master';
  }

  /**
   * 判断查询是否可以使用延迟的读库
   * 允许最终一致性，对实时性要求不高的场景
   */
  canUseLaggingReplica(queryContext: QueryContext): boolean {
    // 用户个人数据查询不允许从延迟的从库读
    if (queryContext.requiresStrongConsistency) {
      return false;
    }

    // 统计数据、报表等允许短暂延迟
    if (queryContext.allowStaleData) {
      return true;
    }

    // 有明确数据新鲜度要求的查询
    if (queryContext.maxStaleness && queryContext.maxStaleness > 0) {
      const minLag = Math.min(...Array.from(this.replicaLags.values()));
      return minLag < queryContext.maxStaleness;
    }

    return false;
  }
}

interface QueryContext {
  requiresStrongConsistency: boolean;  // 是否需要强一致性
  allowStaleData: boolean;              // 是否允许读取旧数据
  maxStaleness?: number;                // 允许的最大延迟（秒）
}
```

## 六、连接池

### 6.1 连接管理

数据库连接池是管理数据库连接的基础设施，正确理解和配置连接池对应用性能至关重要。

**为什么需要连接池**：数据库连接建立需要经过TCP三次握手、认证等过程，耗时较长，通常需要几十毫秒。如果每次请求都创建新连接，会严重影响性能。数据库能同时处理的连接数是有限的，应用服务器通常有多个，如果每个请求都占用一个连接，会很快耗尽数据库连接资源。连接池通过复用连接，避免了频繁创建和销毁连接的开销。

**连接池的工作原理**：连接池初始化时会创建一定数量的连接，这些连接处于空闲状态。当应用需要访问数据库时，从连接池获取一个空闲连接，使用完毕后归还连接池而不是关闭连接。如果所有连接都被占用，新的请求需要等待空闲连接。连接池会定期检测连接的可用性，清理无效连接，补充新连接。

**连接池的核心参数**：最小连接数（minIdle）表示连接池保持的最小空闲连接数。最大连接数（maxPoolSize）表示连接池允许的最大连接数。连接超时（connectionTimeout）表示获取连接的最大等待时间。空闲超时（idleTimeout）表示连接空闲多长时间后被关闭。最大生命周期（maxLifetime）表示连接的最长使用时间。

### 6.2 配置优化

连接池的合理配置对应用性能有重要影响。

**配置原则**：连接池大小不是越大越好，通常建议设置为CPU核心数的2-4倍。最大连接数应该小于数据库的max_connections配置。要考虑应用的并发量，合理设置最小和最大连接数。连接超时设置不宜过短，避免在高负载时频繁超时。

### 6.3 泄漏检测

连接泄漏是连接池使用中的严重问题，会导致连接池耗尽。

**连接泄漏的原因**：使用连接后没有调用close()或release()方法。在异常处理中没有正确释放连接。长事务没有正确提交或回滚。异步操作中连接没有正确传递和释放。查询超时设置过长导致连接长时间被占用。

**检测连接泄漏的方法**：

```typescript
import { Injectable, Logger } from '@nestjs/common';
import { DataSource } from 'typeorm';

/**
 * 连接池监控服务
 * 检测连接泄漏和连接池健康状况
 */
@Injectable()
export class ConnectionPoolMonitor {
  private readonly logger = new Logger(ConnectionPoolMonitor.name);

  constructor(@Inject('DATA_SOURCE') private dataSource: DataSource) {
    // 启动定时监控
    this.startMonitoring();
  }

  /**
   * 定时检查连接池状态
   */
  private startMonitoring() {
    setInterval(async () => {
      await this.checkPoolHealth();
    }, 60000);  // 每分钟检查一次
  }

  /**
   * 检查连接池健康状态
   */
  async checkPoolHealth() {
    try {
      // 获取连接池统计信息（以MySQL为例）
      const query = `
        SELECT
          @@max_connections AS maxConnections,
          @@thread_count AS currentConnections,
          (SELECT COUNT(*) FROM information_schema.processlist) AS activeProcesses
      `;

      const result = await this.dataSource.query(query);
      const { maxConnections, currentConnections, activeProcesses } = result[0];

      const usageRate = (currentConnections / maxConnections) * 100;

      // 记录连接池使用情况
      this.logger.log(
        `Connection Pool Status: ` +
        `Active: ${activeProcesses}, ` +
        `Current: ${currentConnections}/${maxConnections}, ` +
        `Usage: ${usageRate.toFixed(2)}%`
      );

      // 告警条件
      if (usageRate > 80) {
        this.logger.warn(`Connection pool usage rate exceeded 80%: ${usageRate.toFixed(2)}%`);
        await this.sendAlert(`High connection pool usage: ${usageRate.toFixed(2)}%`);
      }

      if (usageRate > 90) {
        this.logger.error(`Connection pool usage rate exceeded 90%: ${usageRate.toFixed(2)}%`);
      }
    } catch (error) {
      this.logger.error('Failed to check connection pool health', error);
    }
  }

  /**
   * 检测慢查询占用连接
   */
  async checkSlowQueries() {
    const slowQuerySql = `
      SELECT
        id,
        user,
        host,
        db,
        command,
        time,
        state,
        LEFT(info, 100) AS info
      FROM information_schema.processlist
      WHERE command != 'Sleep'
      AND time > 10  -- 运行超过10秒的查询
      ORDER BY time DESC
    `;

    const slowQueries = await this.dataSource.query(slowQuerySql);

    if (slowQueries.length > 0) {
      this.logger.warn(
        `Found ${slowQueries.length} slow queries:`,
        JSON.stringify(slowQueries, null, 2)
      );
    }
  }

  /**
   * 获取连接池详细信息
   */
  async getPoolDetails() {
    // 通过查询获取当前连接状态
    const activeConnections = await this.dataSource.query(
      'SELECT COUNT(*) AS count FROM information_schema.processlist WHERE user = ?',
      [process.env.DB_USER]
    );

    return {
      activeConnections: activeConnections[0].count,
      poolSize: 20,  // 需要从连接池获取
      waitingRequests: 0,  // 需要从连接池获取
      totalConnections: 20,  // 配置值
    };
  }
}

/**
 * 连接使用追踪
 * 记录连接的借用和释放，便于追踪泄漏
 */
@Injectable()
export class ConnectionTracker {
  private activeConnections: Map<string, {
    borrowedAt: Date;
    query: string;
    stack: string;
  }> = new Map();

  /**
   * 记录连接借用
   */
  trackBorrow(connectionId: string, query: string) {
    const stack = new Error().stack || '';
    this.activeConnections.set(connectionId, {
      borrowedAt: new Date(),
      query,
      stack: stack,
    });
  }

  /**
   * 记录连接释放
   */
  trackRelease(connectionId: string) {
    const info = this.activeConnections.get(connectionId);
    if (info) {
      const duration = Date.now() - info.borrowedAt.getTime();
      if (duration > 30000) {  // 超过30秒记录警告
        this.logger.warn(
          `Connection ${connectionId} held for ${duration}ms`,
          `Query: ${info.query}`,
          `Stack: ${info.stack}`
        );
      }
      this.activeConnections.delete(connectionId);
    }
  }

  /**
   * 获取未释放的连接
   */
  getActiveConnections() {
    return Array.from(this.activeConnections.entries()).map(([id, info]) => ({
      connectionId: id,
      borrowedAt: info.borrowedAt,
      duration: Date.now() - info.borrowedAt.getTime(),
      query: info.query,
    }));
  }
}
```

```typescript
import { Module, Global, Injectable, OnModuleDestroy } from '@nestjs/common';
import { DataSource, EntityManager } from 'typeorm';

/**
 * 自定义连接池管理器
 * 提供更灵活的连接池管理和监控
 */
@Global()
@Injectable()
export class PoolManager implements OnModuleDestroy {
  private mainPool: DataSource;
  private readonlyPool: DataSource[] = [];
  private readonlyRoundRobin = 0;

  constructor() {
    this.initializePools();
  }

  /**
   * 初始化连接池
   */
  private async initializePools() {
    // 主库连接池
    this.mainPool = new DataSource({
      name: 'main',
      type: 'mysql',
      host: process.env.DB_MASTER_HOST,
      port: parseInt(process.env.DB_MASTER_PORT || '3306'),
      username: process.env.DB_USER,
      password: process.env.DB_PASSWORD,
      database: process.env.DB_NAME,
      extra: {
        connectionLimit: 20,
        queueLimit: 0,
        connectTimeout: 10000,
        idleTimeout: 60000,
        maxLifetime: 1800000,
      },
      synchronize: false,
      logging: ['error', 'warn'],
    });

    // 从库连接池
    const replicaHosts = [
      { host: process.env.DB_REPLICA1_HOST, port: process.env.DB_REPLICA1_PORT },
      { host: process.env.DB_REPLICA2_HOST, port: process.env.DB_REPLICA2_PORT },
    ];

    for (const replica of replicaHosts) {
      const ds = new DataSource({
        name: `replica_${replica.host}`,
        type: 'mysql',
        host: replica.host,
        port: parseInt(replica.port || '3306'),
        username: process.env.DB_USER,
        password: process.env.DB_PASSWORD,
        database: process.env.DB_NAME,
        extra: {
          connectionLimit: 20,
          queueLimit: 0,
          connectTimeout: 10000,
          idleTimeout: 60000,
          maxLifetime: 1800000,
        },
        synchronize: false,
        logging: ['error'],
      });
      this.readonlyPool.push(ds);
    }

    // 初始化连接
    await this.mainPool.initialize();
    await Promise.all(this.readonlyPool.map(ds => ds.initialize()));
  }

  /**
   * 获取主库连接（用于写操作）
   */
  getMainPool(): DataSource {
    return this.mainPool;
  }

  /**
   * 获取从库连接（用于读操作，轮询）
   */
  getReadonlyPool(): DataSource {
    if (this.readonlyPool.length === 0) {
      return this.mainPool;
    }
    const pool = this.readonlyPool[this.readonlyRoundRobin];
    this.readonlyRoundRobin =
      (this.readonlyRoundRobin + 1) % this.readonlyPool.length;
    return pool;
  }

  /**
   * 获取连接池统计信息
   */
  async getPoolStats() {
    const stats = {
      main: await this.getPoolStat(this.mainPool),
      readonly: [] as any[],
    };

    for (const pool of this.readonlyPool) {
      stats.readonly.push(await this.getPoolStat(pool));
    }

    return stats;
  }

  private async getPoolStat(pool: DataSource): Promise<any> {
    try {
      // 获取连接池信息（MySQL特有）
      const [result] = await pool.query(`
        SELECT
          @@max_connections as maxConnections,
          (SELECT COUNT(*) FROM information_schema.processlist WHERE user = ?) as activeConnections
      `, [process.env.DB_USER]);

      return {
        name: pool.name,
        maxConnections: result.maxConnections,
        activeConnections: result.activeConnections,
        usagePercent: (result.activeConnections / result.maxConnections) * 100,
      };
    } catch (error) {
      return { name: pool.name, error: error.message };
    }
  }

  async onModuleDestroy() {
    // 关闭所有连接池
    await this.mainPool.destroy();
    await Promise.all(this.readonlyPool.map(ds => ds.destroy()));
  }
}

/**
 * 数据库操作服务基类
 * 提供读写分离的数据库操作方法
 */
@Injectable()
export class BaseRepository {
  constructor(private poolManager: PoolManager) {}

  /**
   * 获取读操作的EntityManager（自动路由到从库）
   */
  getReadManager(): EntityManager {
    return this.poolManager.getReadonlyPool().manager;
  }

  /**
   * 获取写操作的EntityManager（路由到主库）
   */
  getWriteManager(): EntityManager {
    return this.poolManager.getMainPool().manager;
  }

  /**
   * 执行读查询
   */
  async find<T>(entity: new () => T, options?: any): Promise<T[]> {
    return this.getReadManager().find(entity, options);
  }

  /**
   * 执行写操作
   */
  async save<T>(entity: new () => T, data: Partial<T>): Promise<T> {
    return this.getWriteManager().save(entity, data);
  }

  /**
   * 执行事务（始终使用主库）
   */
  async transaction<T>(callback: (manager: EntityManager) => Promise<T>): Promise<T> {
    return this.getWriteManager().transaction(callback);
  }
}
```

## 七、ORM优化

### 7.1 N+1问题

N+1问题是ORM使用中最常见的性能问题，理解它产生的原因和解决方法至关重要。

**N+1问题的产生**：假设有100个用户，每个用户有10篇文章。如果不使用预加载，查询所有用户的文章会这样执行：首先执行1次查询获取100个用户；然后循环每个用户，执行100次查询获取每个用户的文章；总共执行了1 + 100 = 101次查询，这就是N+1问题。

**N+1问题的危害**：每次查询都有网络开销和数据库执行开销，101次查询可能需要几秒钟。数据库连接资源被长时间占用。数据库负载急剧增加。

**N+1问题的解决**：使用JOIN查询一次性获取所有数据。使用预加载（Include/Eager Loading）一次性获取关联数据。使用批量加载。

### 7.2 预加载策略

预加载是解决N+1问题的核心策略。

**TypeORM预加载**：

```typescript
@Injectable()
export class ArticleService {
  constructor(
    @InjectRepository(Article)
    private articleRepository: Repository<Article>,
  ) {}

  /**
   * 预加载文章及其作者
   * 解决 N+1 问题
   */
  async findAllWithAuthor(): Promise<Article[]> {
    // 使用 leftJoinAndSelect 预加载关联数据
    // 这只会执行2次查询：1次查文章，1次查作者
    return this.articleRepository.find({
      relations: ['author'],  // 预加载author关系
    });
  }

  /**
   * 使用QueryBuilder进行更复杂的预加载
   */
  async findAllWithAuthorAndComments(): Promise<Article[]> {
    return this.articleRepository
      .createQueryBuilder('article')
      .leftJoinAndSelect('article.author', 'author')
      .leftJoinAndSelect('article.comments', 'comments')
      .leftJoinAndSelect('comments.author', 'commentAuthor')
      .where('article.published = :published', { published: true })
      .orderBy('article.createdAt', 'DESC')
      .getMany();
  }

  /**
   * 嵌套预加载
   * 文章 -> 作者 -> 作者的个人资料
   */
  async findWithNestedRelations(): Promise<Article[]> {
    return this.articleRepository.find({
      relations: {
        author: {
          profile: true,  // 嵌套预加载
        },
        tags: true,
      },
      where: {
        published: true,
      },
    });
  }
}
```

```typescript
 * 优化代码：使用预加载解决N+1

      title: article.title,
      content: article.content,
      authorName: article.author?.name,
      authorEmail: article.author?.email,
    }));
    // 总共执行：1（文章）+ 1（作者通过JOIN获取）= 2次查询
  }
```

### 7.3 批量操作

批量操作可以大幅减少数据库交互次数，提升性能。

**TypeORM批量操作**：

```typescript
@Injectable()
export class BatchOperationService {
  /**
   * 批量插入
   * 插入1000条记录，用批量操作只需几次SQL执行
   */
  async batchInsert(articles: CreateArticleDto[]): Promise<number> {
    // 使用 insert 批量插入（单次SQL）
    // 性能比循环单条插入快10-100倍
    const result = await this.articleRepository.insert(
      articles.map(dto => ({
        ...dto,
        createdAt: new Date(),
        updatedAt: new Date(),
      }))
    );
    return result.identifiers.length;
  }

  /**
   * 批量更新
   */
  async batchUpdate(ids: string[], status: string): Promise<number> {
    // 使用 update 批量更新（单次SQL）
    const result = await this.articleRepository
      .createQueryBuilder()
      .update(Article)
      .set({ status, updatedAt: new Date() })
      .whereInIds(ids)
      .execute();
    return result.affected;
  }

  /**
   * 批量删除
   */
  async batchDelete(ids: string[]): Promise<number> {
    const result = await this.articleRepository.delete(ids);
    return result.affected;
  }

  /**
   * 超大批量插入（分批执行）
   * 适用于插入数万条甚至更多数据
   */
  async hugeBatchInsert(articles: CreateArticleDto[], batchSize = 500): Promise<void> {
    const totalBatches = Math.ceil(articles.length / batchSize);

    for (let i = 0; i < totalBatches; i++) {
      const batch = articles.slice(i * batchSize, (i + 1) * batchSize);

      await this.articleRepository.insert(
        batch.map(dto => ({
          ...dto,
          createdAt: new Date(),
          updatedAt: new Date(),
        }))
      );

      // 每批插入后休息一下，避免数据库负载过高
      if (i < totalBatches - 1) {
        await this.sleep(100);
      }
    }
  }

  /**
   * Upsert操作（存在则更新，不存在则插入）
   */
  async upsert(article: Article): Promise<void> {
    // MySQL的ON DUPLICATE KEY UPDATE
    await this.articleRepository
      .createQueryBuilder()
      .insert()
      .into(Article)
      .values(article)
      .orUpdate(['title', 'content', 'updated_at'], ['article_id'])  // 更新这些字段
      .execute();
  }

  /**
   * 使用事务进行批量操作
   * 保证数据一致性
   */
  async batchInsertWithTransaction(articles: CreateArticleDto[]): Promise<number> {
    const queryRunner = this.dataSource.createQueryRunner();
    await queryRunner.connect();
    await queryRunner.startTransaction();

    try {
      let insertedCount = 0;
      const batchSize = 500;

      for (let i = 0; i < articles.length; i += batchSize) {
        const batch = articles.slice(i, i + batchSize);
        await queryRunner.manager.insert(Article, batch);
        insertedCount += batch.length;
      }

      await queryRunner.commitTransaction();
      return insertedCount;
    } catch (error) {
      await queryRunner.rollbackTransaction();
      throw error;
    } finally {
      await queryRunner.release();
    }
  }

  private sleep(ms: number): Promise<void> {
    return new Promise(resolve => setTimeout(resolve, ms));
  }
}
```

```typescript
      .leftJoinAndSelect('article.category', 'category')

      qb.andWhere('article.createdAt < :cursor', { cursor: new Date(cursor) });
    }

    const articles = await qb.getMany();

    const hasMore = articles.length > limit;
    const data = hasMore ? articles.slice(0, -1) : articles;
    const nextCursor = hasMore
      ? data[data.length - 1].createdAt.toISOString()
      : null;

    return { data, nextCursor };
  }
```

## 八、MongoDB

### 8.1 聚合管道

MongoDB的聚合管道是处理复杂数据分析和转换的强大工具。

**聚合管道基础**：聚合管道由多个阶段（Stage）组成，每个阶段对文档进行处理后传递给下一个阶段。每个阶段可以使用的操作符包括：$match（过滤）、$group（分组）、$sort（排序）、$project（投影）、$limit（限制）、$skip（跳过）、$lookup（关联查询）、$unwind（展开数组）。

**聚合管道的优势**：声明式语法，代码简洁易读。管道内文档流式处理，内存效率高。支持多种操作符，功能强大。支持索引优化，提高性能。

```javascript
/**
 * MongoDB聚合管道示例
 * 分析订单数据，生成销售报表
 */
async function generateSalesReport(startDate, endDate) {
  const pipeline = [
    // 阶段1：过滤日期范围内的已支付订单
    {
      $match: {
        status: 'completed',
        paidAt: {
          $gte: new Date(startDate),
          $lt: new Date(endDate),
        },
      },
    },

    // 阶段2：展开订单商品数组
    {
      $unwind: '$items',
    },

    // 阶段3：按商品分类分组统计
    {
      $group: {
        _id: '$items.categoryId',
        categoryName: { $first: '$items.categoryName' },
        totalQuantity: { $sum: '$items.quantity' },
        totalAmount: { $sum: '$items.subtotal' },
        orderCount: { $addToSet: '$_id' },  // 去重订单数
        avgOrderAmount: { $avg: '$totalAmount' },
      },
    },

    // 阶段4：计算订单数
    {
      $project: {
        _id: 1,
        categoryName: 1,
        totalQuantity: 1,
        totalAmount: 1,
        orderCount: { $size: '$orderCount' },
        avgOrderAmount: { $round: ['$avgOrderAmount', 2] },
      },
    },

    // 阶段5：按销售额降序排序
    {
      $sort: { totalAmount: -1 },
    },

    // 阶段6：添加排名
    {
      $facet: {
        rankedData: [
          {
            $addFields: {
              rank: { $sum: 1 },  // 需要在分组后计算
            },
          },
        ],
        summary: [
          {
            $group: {
              _id: null,
              totalCategories: { $sum: 1 },
              totalRevenue: { $sum: '$totalAmount' },
              totalOrders: { $sum: '$orderCount' },
            },
          },
        ],
      },
    },
  ];

  return await ordersCollection.aggregate(pipeline).toArray();
}
```

### 8.2 MongoDB索引

MongoDB支持多种类型的索引，合理使用索引对性能至关重要。

**索引类型**：单字段索引在单个字段上创建，适用于简单查询。复合索引在多个字段上创建，遵循最左前缀原则。多键索引在数组字段上创建，MongoDB会自动展开数组。文本索引支持文本搜索，但有性能开销。地理位置索引支持2dsphere和2d坐标查询。哈希索引按字段哈希值建立索引，支持分片集群。

**索引创建策略**：根据查询模式创建索引，索引字段应与查询条件匹配。考虑索引字段的选择性，高选择性的字段优先。复合索引要考虑字段顺序，将等值查询字段放在前面。避免创建太多索引，每个索引都会影响写性能。

```javascript
/**
 * MongoDB索引设计与优化
 */

// 为常用查询创建复合索引
db.orders.createIndex({
  userId: 1,           // 等值查询，放前面
  status: 1,            // 等值查询
  createdAt: -1,        // 排序字段
});

// 创建覆盖索引，索引包含所有查询需要的字段
db.orders.createIndex(
  { userId: 1, status: 1, createdAt: 1 },
  {
    name: 'coverIndex',
    projection: { userId: 1, status: 1, totalAmount: 1, createdAt: 1 },
  }
);

// 为数组字段创建多键索引
db.products.createIndex({ tags: 1 });

// 创建文本索引支持搜索
db.articles.createIndex(
  { title: 'text', content: 'text' },
  { weights: { title: 10, content: 1 } }  // 标题权重更高
);

// 查询分析：使用explain查看索引使用情况
db.orders.find({
  userId: 'user123',
  status: 'completed'
}).explain('executionStats');

// 查看索引使用情况
db.orders.aggregate([
  { $indexStats: {} }
]);
```

### 8.3 MongoDB分片

当单机MongoDB无法满足数据量需求时，需要使用分片集群。

**分片架构**：分片集群由多个分片节点组成，每个分片存储部分数据。查询路由器（mongos）作为客户端入口，转发请求到正确的分片。配置服务器（config servers）存储集群元数据，包括分片信息和数据分布。

**分片键选择**：分片键决定了数据如何分布到各个分片。好的分片键应具备：高基数（ cardinality），有足够多的不同值。数据分布均匀，避免单个分片数据量过大。查询局部性，相关数据在同一分片。

**分片策略**：范围分片按照分片键的值范围分布数据，适合按时间或有序字段查询。哈希分片按照分片键的哈希值分布数据，适合随机访问场景。

```javascript
/**
 * MongoDB分片集群配置与使用
 */

// 初始化分片集群
sh.initiate();
sh.addShard('shard1-replica-set/localhost:27017');
sh.addShard('shard2-replica-set/localhost:27018');
sh.addShard('shard3-replica-set/localhost:27019');

// 启用数据库分片
sh.enableSharding('mydb');

// 在集合上启用分片
sh.shardCollection('mydb.orders', { userId: 'hashed' });  // 哈希分片
sh.shardCollection('mydb.articles', { createdAt: 1 });    // 范围分片

// 查看集群状态
sh.status();

// 查看分片分布
db.adminCommand({ balancerStatus: 1 });
db.chunks.find({ ns: 'mydb.orders' }).count();

// 平衡器配置
```

```typescript
      return true;
    } catch (error) {

      this.logger.error('Credit transfer failed', error);

    }
```

## 九、Redis

### 9.1 数据结构

Redis提供了多种数据结构，每种数据结构都有其特定的应用场景。

**字符串（String）**是Redis最基本的数据结构，可以存储字符串、整数或浮点数。常用命令包括GET、SET、SETEX（带过期时间）、INCR（原子递增）、DECR、DEL。应用场景：缓存简单数据、计数器、分布式锁、session存储。

**哈希（Hash）**是键值对集合，适合存储对象。常用命令包括HGET、HSET、HGETALL、HINCRBY、HDEL。应用场景：存储对象数据、购物车数据、表单数据。

**列表（List）**是字符串链表，按插入顺序存储。常用命令包括LPUSH、RPUSH、LPOP、RPOP、LRANGE、LREM。应用场景：消息队列、最新消息列表、操作日志。

**集合（Set）**是无序且不重复的字符串集合。常用命令包括SADD、SREM、SMEMBERS、SISMEMBER、SCARD、SDIFF。应用场景：标签系统、好友关系、去重。

**有序集合（Sorted Set）**是带分数的集合，按分数排序。常用命令包括ZADD、ZRANGE、ZREVRANGE、ZSCORE、ZRANK。应用场景：排行榜、延时队列、优先级队列。

```typescript
 * Redis数据类型实战

import { Injectable } from '@nestjs/common';
import { RedisService } from '@nestjs-modules/ioredis';

@Injectable()
export class RedisDataStructureService {
  constructor(private redisService: RedisService) {}

  /**
   * String类型：分布式锁
   */
  async acquireLock(lockKey: string, expireMs: number = 30000): Promise<boolean> {
    const lockValue = `${process.pid}-${Date.now()}`;
    // SET key value NX PX milliseconds
    // NX：仅当键不存在时设置
    // PX：过期时间（毫秒）
    const result = await this.redisService.set(
      lockKey,
      lockValue,
      'PX',
      expireMs,
      'NX',
    );
    return result === 'OK';
  }

  /**
   * String类型：计数器
   */
  async incrementCounter(key: string, delta: number = 1): Promise<number> {
    return this.redisService.incrby(key, delta);
  }

  /**
   * Hash类型：存储用户信息
   */
  async setUserInfo(userId: string, userData: Record<string, string>) {
    const key = `user:${userId}`;
    await this.redisService.hmset(key, userData);
    // 设置24小时过期
    await this.redisService.expire(key, 86400);
  }

  async getUserInfo(userId: string): Promise<Record<string, string> | null> {
    const key = `user:${userId}`;
    return this.redisService.hgetall(key);
  }

  /**
   * List类型：消息队列
   */
  async pushMessage(queueName: string, message: string) {
    // 右插入，消费者的LPOP从左边取
    await this.redisService.rpush(`queue:${queueName}`, message);
  }

  async popMessage(queueName: string, timeout: number = 0): Promise<string | null> {
    // BLPOP阻塞等待，timeout为0表示无限等待
    if (timeout > 0) {
      return this.redisService.blpop(`queue:${queueName}`, timeout);
    }
    return this.redisService.lpop(`queue:${queueName}`);
  }

  /**
   * Set类型：标签系统
   */
  async addTags(itemId: string, tags: string[]) {
    const key = `item:${itemId}:tags`;
    await this.redisService.sadd(key, ...tags);
  }

  async getItemTags(itemId: string): Promise<string[]> {
    return this.redisService.smembers(`item:${itemId}:tags`);
  }

  async findItemsByTags(requiredTags: string[]): Promise<string[]> {
    const keys = requiredTags.map(tag => `tag:${tag}:items`);
    // 多个集合取交集
    return this.redisService.sinter(...keys);
  }

  /**
   * Sorted Set类型：排行榜
   */
  async updateScore(leaderboardKey: string, userId: string, score: number) {
    // ZADD key score member
    await this.redisService.zadd(leaderboardKey, score, userId);
  }

  async getRank(leaderboardKey: string, userId: string): Promise<number | null> {
    // ZREVRANK返回排名（0为最高分），ZSCORE返回分数
    const rank = await this.redisService.zrevrank(leaderboardKey, userId);
    const score = await this.redisService.zscore(leaderboardKey, userId);
    return rank !== null ? rank + 1 : null;  // 转为1-based排名
  }

  async getTopPlayers(leaderboardKey: string, count: number = 10): Promise<string[]> {
    // ZREVRANGE返回指定范围内的成员（按分数从高到低）
    return this.redisService.zrevrange(leaderboardKey, 0, count - 1, 'WITHSCORES');
  }

  /**
   * Sorted Set类型：延时队列
   */
  async addDelayTask(queueName: string, taskId: string, delayMs: number) {
    const score = Date.now() + delayMs;  // 执行时间戳作为分数
    await this.redisService.zadd(`delay:${queueName}`, score, taskId);
  }

  async pollDelayTask(queueName: string): Promise<string | null> {
    const now = Date.now();
    // 获取所有等待执行的任务（分数小于当前时间戳）
    const tasks = await this.redisService.zrangebyscore(
      `delay:${queueName}`,
      0,
      now,
      'LIMIT',
      0,
      1,
    );

    if (tasks.length === 0) {
      return null;
    }

    const taskId = tasks[0];
    // 原子性地移除任务
    const removed = await this.redisService.zrem(`delay:${queueName}`, taskId);

    return removed === 1 ? taskId : null;
  }
}
```

### 9.2 持久化

Redis提供了两种持久化方式：RDB和AOF，各有优缺点。

**RDB持久化**通过在指定时间点生成数据集快照来保存数据。优点：生成的备份文件紧凑，适合备份和灾难恢复。恢复大数据集时比AOF快。适合做冷备份。缺点：可能丢失最后一次快照后的数据。不适合实时持久化需求。

**AOF持久化**通过记录每个写操作命令来保存数据。优点：提供更好的数据安全性（可配置同步策略）。是追加写入，不会有后台fork的内存问题。缺点：AOF文件通常比RDB文件大。性能可能比RDB稍低。

**AOF同步策略**：always每个写命令都同步，最安全但最慢。everysec每秒同步，推荐选择，性能和数据安全的平衡。no由操作系统决定何时同步，性能最好但可能丢失较多数据。

### 9.3 集群

Redis集群提供数据分片和高可用能力。

**Redis Sentinel**是Redis的高可用方案，负责监控主从实例、自动故障转移、通知客户端。Sentinel通过PING命令监控主从实例的存活状态。当主实例不可用时，自动选举从实例晋升为新的主实例，并将其他从实例重新配置为复制新主实例。客户端通过连接Sentinel获取当前的主实例地址。

```typescript
import { Injectable, Logger } from '@nestjs/common';
import { RedisService } from '@nestjs-modules/ioredis';

/**
 * Redis实战：缓存服务
 * 展示Redis在项目中的实际应用
 */
@Injectable()
export class CacheService {
  private readonly logger = new Logger(CacheService.name);

  // 默认缓存过期时间
  private readonly DEFAULT_TTL = 3600;  // 1小时

  constructor(private redisService: RedisService) {}

  /**
   * 通用缓存方法
   */
  async cacheGet<T>(key: string): Promise<T | null> {
    const data = await this.redisService.get(key);
    if (!data) {
      return null;
    }
    try {
      return JSON.parse(data) as T;
    } catch {
      return data as any;
    }
  }

  async cacheSet(key: string, value: any, ttl: number = this.DEFAULT_TTL): Promise<void> {
    const data = typeof value === 'string' ? value : JSON.stringify(value);
    await this.redisService.setex(key, ttl, data);
  }

  async cacheDel(key: string): Promise<void> {
    await this.redisService.del(key);
  }

  /**
   * 缓存击穿：使用互斥锁
   */
  async cacheWithLock<T>(
    key: string,
    factory: () => Promise<T>,
    ttl: number = 300,
  ): Promise<T> {
    // 尝试获取缓存
    const cached = await this.cacheGet<T>(key);
    if (cached !== null) {
      return cached;
    }

    // 尝试获取锁
    const lockKey = `lock:${key}`;
    const lockAcquired = await this.acquireLock(lockKey, 10000);

    if (lockAcquired) {
      try {
        // 获取锁成功后，再次检查缓存（防止其他请求已经写入）
        const cached2 = await this.cacheGet<T>(key);
        if (cached2 !== null) {
          return cached2;
        }

        // 执行实际查询
        const data = await factory();
        await this.cacheSet(key, data, ttl);
        return data;
      } finally {
        await this.releaseLock(lockKey);
      }
    } else {
      // 未获取到锁，等待后重试
      await this.sleep(100);
      return this.cacheWithLock(key, factory, ttl);
    }
  }

  /**
   * 缓存雪崩：使用随机过期时间
   */
  async cacheSetWithJitter(
    key: string,
    value: any,
    baseTtl: number = 3600,
    jitterPercent: number = 0.1,
  ): Promise<void> {
    // 添加随机抖动，避免同时过期
    const jitter = Math.floor(baseTtl * jitterPercent * Math.random());
    const actualTtl = baseTtl + jitter;
    await this.cacheSet(key, value, actualTtl);
  }

  /**
   * 缓存穿透：使用布隆过滤器
   */
  async isProbablyExists(key: string): Promise<boolean> {
    // 使用BITCOUNT检查布隆过滤器
    const keyHash = this.hashKey(key);
    const bitIndex = keyHash % (1024 * 1024);  // 使用1M位的布隆过滤器
    const byteIndex = Math.floor(bitIndex / 8);

    const bitmap = await this.redisService.getbit('bloom:filter', byteIndex);
    const bitValue = (bitmap >> (bitIndex % 8)) & 1;

    return bitValue === 1;
  }

  async markAsExists(key: string): Promise<void> {
    const keyHash = this.hashKey(key);
    const bitIndex = keyHash % (1024 * 1024);

    await this.redisService.setbit('bloom:filter', bitIndex, 1);
  }

  private hashKey(key: string): number {
    let hash = 0;
    for (let i = 0; i < key.length; i++) {
      const char = key.charCodeAt(i);
      hash = ((hash << 5) - hash) + char;
      hash = hash & hash;
    }
    return Math.abs(hash);
  }

  /**
   * 分布式锁
   */
  async acquireLock(lockKey: string, expireMs: number = 30000): Promise<boolean> {
    const lockValue = `${process.pid}-${Date.now()}-${Math.random()}`;
    const result = await this.redisService.set(lockKey, lockValue, 'PX', expireMs, 'NX');
    return result === 'OK';
  }

  async releaseLock(lockKey: string): Promise<void> {
    // 使用Lua脚本确保只删除自己的锁
    const script = `
      if redis.call("get", KEYS[1]) == ARGV[1] then
        return redis.call("del", KEYS[1])
      else
        return 0
      end
    `;
    await this.redisService.eval(script, 1, lockKey, `${process.pid}`);
  }

  /**
   * 限流
   */
  async rateLimit(
    key: string,
    maxRequests: number,
    windowSec: number,
  ): Promise<{ allowed: boolean; remaining: number; resetAt: number }> {
    const now = Date.now();
    const windowKey = `ratelimit:${key}:${Math.floor(now / (windowSec * 1000))}`;

    const current = await this.redisService.incr(windowKey);

    if (current === 1) {
      // 第一个请求，设置过期时间
      await this.redisService.expire(windowKey, windowSec);
    }

    const allowed = current <= maxRequests;
    const remaining = Math.max(0, maxRequests - current);
    const resetAt = Math.ceil(now / (windowSec * 1000)) * (windowSec * 1000);

    return { allowed, remaining, resetAt };
  }

  /**
   * 分布式计数器
   */
  async incrementWithExpire(key: string, expireSec: number): Promise<number> {
    const exists = await this.redisService.exists(key);

    const pipeline = this.redisService.pipeline();
    pipeline.incr(key);

    if (!exists) {
      pipeline.expire(key, expireSec);
    }

    const results = await pipeline.exec();
    return results[0][1] as number;
  }

  /**
   * 延迟队列
   */
  async enqueueDelayTask(taskId: string, delayMs: number, payload: any): Promise<void> {
    const executeAt = Date.now() + delayMs;
    await this.redisService.zadd('delay:tasks', executeAt, `${taskId}:${JSON.stringify(payload)}`);
  }

  async dequeueDelayTask<T>(): Promise<{ taskId: string; payload: T } | null> {
    const now = Date.now();
    const results = await this.redisService.zrangebyscore('delay:tasks', 0, now, 'LIMIT', 0, 1);

    if (results.length === 0) {
      return null;
    }

    const [taskIdWithPayload] = results;
    const [taskId, payloadStr] = taskIdWithPayload.split(':', 1);
    const payload = JSON.parse(taskIdWithPayload.substring(taskId.length + 1));

    // 原子性删除
    await this.redisService.zrem('delay:tasks', taskIdWithPayload);

    return { taskId, payload };
  }

  private sleep(ms: number): Promise<void> {
    return new Promise(resolve => setTimeout(resolve, ms));
  }
}
```

## 十、数据库设计

### 10.1 范式

数据库规范化是设计关系型数据库的核心原则，通过范式可以消除数据冗余，保证数据完整性。

**第一范式（1NF）**要求数据库表的每一列都是不可分割的原子数据项。即每个字段只能存储单一值，不能有重复组或多值字段。例如，用户联系方式字段如果存储"电话:123,邮箱:abc"就不符合1NF，应该拆分为独立的电话和邮箱字段。1NF是数据库设计的基础，几乎所有现代数据库都天然满足1NF。

**第二范式（2NF）**在满足1NF的基础上，要求表中的非主键列完全依赖于主键，而不能只依赖于主键的一部分（针对复合主键）。例如，有一个订单明细表使用(orderId, productId)作为复合主键，如果表中还有productName字段，它只依赖于productId而不依赖于orderId，就违反了2NF，应该将productName拆分到单独的商品表中。2NF主要解决部分依赖问题。

**第三范式（3NF）**在满足2NF的基础上，要求非主键列之间不存在传递依赖。即非主键列只能直接依赖于主键，不能通过其他非主键列间接依赖于主键。例如，有一个用户表包含userId（主键）、userName、deptId、deptName，其中deptName依赖于deptId，deptId依赖于userId，这就是传递依赖，deptName应该拆分到部门表中。3NF主要解决传递依赖问题。

**BC范式（BCNF）**是第三范式的增强版，要求对于每一个非平凡的函数依赖X->Y，X必须包含码。通俗地说，BCNF要求每个决定因素都是超码。例如，有一个仓库管理表包含(仓库ID, 物品ID, 管理员ID, 物品数量)，假设每个仓库有多个管理员，每个管理员只能管理一个仓库，每个物品只能存放在一个仓库，那么仓库ID->管理员ID和物品ID->仓库ID都是函数依赖，但仓库ID不是超码，不符合BCNF，需要进一步拆分。

### 10.2 反范式

反范式是通过增加数据冗余来提升查询性能的优化策略。

**为什么需要反范式**：严格遵循范式会导致查询时需要频繁JOIN多个表，影响性能。在某些场景下，适当的冗余可以大大简化查询逻辑。对于读多写少的场景，反范式可以减少IO次数。反范式可以避免或减少聚合计算。

**常见的反范式场景**：在订单表中冗余用户名称，避免每次查询订单都JOIN用户表。在商品表中冗余分类名称，避免JOIN分类表。在订单详情表中冗余商品快照，防止商品信息修改后订单历史数据丢失。在论坛帖子表中冗余回复数量和最后回复时间。

```sql
-- 严格范式设计
-- 用户表
CREATE TABLE users (
  id BIGINT PRIMARY KEY,

  INDEX idx_department (department_id)

-- 部门表
CREATE TABLE departments (
  id BIGINT PRIMARY KEY,
```

### 10.3 表关联

表关联是数据库设计的核心，正确处理表之间的关系对系统性能和可维护性至关重要。

**一对一关联**表示两个表之间的一一对应关系。可以使用外键实现，也可以使用相同的主键值来实现。一对一关联适用于将一个大表拆分为多个小表、使用继承结构、将敏感信息分离到单独的表。例如，用户基本信息和用户详情可以是一对一关系。

**一对多关联**是最常见的关联类型，一个表的一条记录对应另一个表的多条记录。使用外键来实现，外键放在多的一方。例如，一个用户可以有多个订单，订单表中的userId就是指向用户表的外键。

**多对多关联**需要通过中间表来实现，两个表的多条记录可以对应对方的多条记录。例如，一个学生可以选多门课程，一门课程可以有多个学生，需要通过选课中间表来实现。

**自关联**是表与自身的关联，用于表示树形结构。例如，部门表中的parentId指向父部门，员工表中的managerId指向上级经理。

```sql
-- 一对一关联示例：用户和用户配置
```

## 十一、容灾备份

### 11.1 备份策略

数据是企业最重要的资产，完善的备份策略是保护数据安全的关键。备份策略需要根据业务需求、数据重要性和恢复时间目标（RTO）来制定。

**备份类型详解**：完全备份（Full Backup）会备份数据库的所有数据，是最完整的备份形式，但占用空间大，备份时间长。增量备份（Incremental Backup）只备份自上次备份以来发生变化的数据，节省存储空间但恢复时需要按顺序应用所有增量备份。差异备份（Differential Backup）备份自上次完全备份以来发生变化的数据，是完全备份和增量备份的折中方案。

**备份频率设计**：核心业务数据建议每小时增量备份，每天完全备份。对于变化不频繁的数据可以降低备份频率。日志数据可以根据业务需求选择备份策略。

**备份存储策略**：备份数据应该存储在与生产环境不同的物理位置，本地一份、异地一份。重要数据应该有多份副本。备份数据的保留时间应该满足合规要求和业务需要。

**备份验证**：定期验证备份文件的完整性，确保备份可以正常恢复。测试恢复流程，确保团队熟悉恢复操作。

```typescript
/**
 * 数据库备份服务
 * 展示自动备份策略的实现
 */
import { Injectable, Logger } from '@nestjs/common';
import { InjectDataSource } from '@nestjs/typeorm';
import { DataSource } from 'typeorm';
import * as fs from 'fs';
import * as path from 'path';
import * as crypto from 'crypto';

@Injectable()
export class BackupService {
  private readonly logger = new Logger(BackupService.name);
  private readonly backupDir = '/data/backups/mysql';
  private readonly retentionDays = 30;  // 保留30天

  constructor(
    @InjectDataSource() private dataSource: DataSource,
  ) {
    this.ensureBackupDir();
  }

  /**
   * 执行完全备份
   */
  async fullBackup(): Promise<BackupResult> {
    const timestamp = new Date().toISOString().replace(/[:.]/g, '-');
    const filename = `full-backup-${timestamp}.sql.gz`;
    const filepath = path.join(this.backupDir, filename);

    try {
      // 使用mysqldump进行完全备份
      const command = `mysqldump -h${process.env.DB_HOST} -u${process.env.DB_USER} -p${process.env.DB_PASSWORD} --all-databases --single-transaction --quick --routines --triggers | gzip > ${filepath}`;

      await this.execCommand(command);

      // 计算校验和
      const checksum = await this.calculateChecksum(filepath);

      // 记录备份元数据
      const metadata = {
        type: 'full',
        filename,
        filepath,
        checksum,
        size: fs.statSync(filepath).size,
        timestamp: new Date().toISOString(),
      };

      await this.saveMetadata(metadata);

      this.logger.log(`完全备份已完成: ${filename}`);

      return { success: true, filename, size: metadata.size };
    } catch (error) {
      this.logger.error(`完全备份失败: ${error.message}`);
      return { success: false, error: error.message };
    }
  }

  /**
   * 执行增量备份（基于二进制日志）
   */
  async incrementalBackup(): Promise<BackupResult> {
    const timestamp = new Date().toISOString().replace(/[:.]/g, '-');
    const filename = `increment-backup-${timestamp}`;
    const dirpath = path.join(this.backupDir, filename);

    try {
      // 创建增量备份目录
      fs.mkdirSync(dirpath, { recursive: true });

      // 刷新日志确保所有事务写入
      await this.dataSource.query('FLUSH LOGS');

      // 获取当前的二进制日志位置
      const [result] = await this.dataSource.query('SHOW MASTER STATUS');
      const currentLog = result.File;
      const currentPosition = result.Position;

      // 记录增量备份点
      const metadata = {
        type: 'incremental',
        baseBackupId: await this.getLatestFullBackupId(),
        startLogFile: currentLog,
        startLogPosition: currentPosition,
        timestamp: new Date().toISOString(),
      };

      await this.saveMetadata(metadata);

      this.logger.log(`增量备份已完成: ${filename}`);

      return { success: true, filename, logFile: currentLog, position: currentPosition };
    } catch (error) {
      this.logger.error(`增量备份失败: ${error.message}`);
      return { success: false, error: error.message };
    }
  }

  /**
   * 备份到远程存储
   */
  async backupToRemote(remotePath: string): Promise<boolean> {
    try {
      // 获取最新的备份文件
      const latestBackup = await this.getLatestBackup();

      if (!latestBackup) {
        throw new Error('没有找到可上传的备份');
      }

      // 使用scp或sftp上传到远程服务器
      const command = `scp -i ${process.env.SSH_KEY} ${latestBackup.filepath} ${process.env.REMOTE_USER}@${process.env.REMOTE_HOST}:${remotePath}`;

      await this.execCommand(command);

      // 验证上传的文件
      const verifyCommand = `ssh -i ${process.env.SSH_KEY} ${process.env.REMOTE_USER}@${process.env.REMOTE_HOST} "sha256sum ${remotePath}/${latestBackup.filename}"`;
      const verifyResult = await this.execCommand(verifyCommand);

      if (!verifyResult.includes(latestBackup.checksum)) {
        throw new Error('远程备份验证失败');
      }

      this.logger.log(`备份已上传到远程: ${remotePath}/${latestBackup.filename}`);
      return true;
    } catch (error) {
      this.logger.error(`远程备份失败: ${error.message}`);
      return false;
    }
  }

  /**
   * 清理过期备份
   */
  async cleanupOldBackups(): Promise<number> {
    const cutoffDate = new Date();
    cutoffDate.setDate(cutoffDate.getDate() - this.retentionDays);

    const backups = await this.getAllBackups();
    let deletedCount = 0;

    for (const backup of backups) {
      const backupDate = new Date(backup.timestamp);
      if (backupDate < cutoffDate) {
        // 删除本地文件
        if (fs.existsSync(backup.filepath)) {
          fs.unlinkSync(backup.filepath);
        }
        // 删除元数据记录
        await this.deleteMetadata(backup.id);
        deletedCount++;
      }
    }

    this.logger.log(`已清理 ${deletedCount} 个过期备份`);
    return deletedCount;
  }

  private async execCommand(command: string): Promise<string> {
    return new Promise((resolve, reject) => {
      require('child_process').exec(command, (error, stdout, stderr) => {
        if (error) {
          reject(error);
        } else {
          resolve(stdout || stderr);
        }
      });
    });
  }

  private async calculateChecksum(filepath: string): Promise<string> {
    const hash = crypto.createHash('sha256');
    const stream = fs.createReadStream(filepath);

    return new Promise((resolve, reject) => {
      stream.on('data', (data) => hash.update(data));
      stream.on('end', () => resolve(hash.digest('hex')));
      stream.on('error', reject);
    });
  }

  private ensureBackupDir() {
    if (!fs.existsSync(this.backupDir)) {
      fs.mkdirSync(this.backupDir, { recursive: true });
    }
  }

  private async saveMetadata(metadata: any): Promise<void> {
    const metadataFile = path.join(this.backupDir, 'backup-metadata.json');
    const existing = fs.existsSync(metadataFile)
      ? JSON.parse(fs.readFileSync(metadataFile, 'utf8'))
      : [];
    existing.push(metadata);
    fs.writeFileSync(metadataFile, JSON.stringify(existing, null, 2));
  }

  private async getAllBackups(): Promise<any[]> {
    const metadataFile = path.join(this.backupDir, 'backup-metadata.json');
    if (!fs.existsSync(metadataFile)) {
      return [];
    }
    return JSON.parse(fs.readFileSync(metadataFile, 'utf8'));
  }

  private async getLatestBackup(): Promise<any | null> {
    const backups = await this.getAllBackups();
    if (backups.length === 0) return null;
    return backups.sort((a, b) =>
      new Date(b.timestamp).getTime() - new Date(a.timestamp).getTime()
    )[0];
  }

  private async getLatestFullBackupId(): Promise<string> {
    const backups = await this.getAllBackups();
    const fullBackups = backups.filter(b => b.type === 'full');
    return fullBackups.length > 0 ? fullBackups[fullBackups.length - 1].id : '';
  }

  private async deleteMetadata(id: string): Promise<void> {
    // 实现删除元数据的逻辑
  }
}

interface BackupResult {
  success: boolean;
  filename?: string;
  size?: number;
  logFile?: string;
  position?: number;
  error?: string;
}
```

### 11.2 恢复演练

备份的价值在于能够成功恢复数据，定期进行恢复演练是备份策略的重要组成部分。通过恢复演练可以验证备份数据的有效性、测试恢复流程、测量恢复时间，以及发现备份策略中的问题。

**恢复演练的重要性**：验证备份数据的有效性，确保备份文件没有损坏。测试恢复流程，确保团队熟悉恢复步骤。测量恢复时间，评估是否能满足RTO（恢复时间目标）要求。发现备份策略中的问题，及时调整改进。

**恢复演练计划**：建议每季度至少进行一次完整的恢复演练。演练应该在隔离环境中进行，不影响生产环境。记录恢复过程中的每个步骤和耗时，形成文档。

**演练场景设计**：全量恢复演练，模拟数据库完全崩溃的场景。时间点恢复演练（PITR），验证基于二进制日志的恢复能力。单个表恢复演练，验证表级别恢复的可行性。跨数据库恢复演练，测试恢复到不同环境的可能性。

### 11.3 主从切换

主从切换是高可用数据库架构中的关键操作，需要精心规划和自动化处理。常见的主从切换场景包括主库硬件故障、主库维护升级、数据库负载过高等。

**为什么要主从切换**：主库硬件故障时需要将从库提升为新的主库以保证服务连续性。主库维护升级时需要切换流量到从库以实现零停机维护。数据库负载过高时可以将部分流量分担到从库以实现读写分离。

**手动切换流程**：停止主库写入，确认所有从库都已同步。选择一个从库作为新的主库，通常选择数据最新的从库。停止其他从库的复制，重新配置它们指向新的主库。更新应用层的数据库连接配置。验证新主库可正常写入。

**自动切换**：使用Redis Sentinel、Consul等工具实现自动故障检测和切换。配置虚拟IP漂移，实现连接透明切换。切换后自动更新DNS或负载均衡器配置。建立完善的切换审批和通知机制，确保关键切换操作有人知晓。

### 11.4 我的思考：数据是生命线

数据安全是数据库工作中永恒的主题，任何时候都不能掉以轻心。备份不是为了恢复，而是为了在灾难发生时能够快速恢复服务，每一个DBA都应该定期问自己：如果现在数据库崩溃了，我能在多长时间内恢复服务？这个问题的答案决定了备份策略是否足够完善。

**备份三二一原则**是业界公认的备份策略最佳实践：至少三份备份，存储在两种不同介质上，至少有一份存放在异地。这不是空话，而是经过无数实际案例验证的有效原则。

**恢复演练不是可选项**，而是必选项。纸上谈兵的备份毫无意义，只有经过实际恢复验证的备份才是可靠的。定期的恢复演练不仅是验证备份有效性，更是检验团队应急响应能力的过程。

**高可用不是终点，数据不丢失才是底线**。在设计数据库架构时，要从一开始就考虑容灾能力，而不是等到出问题才后悔莫及。数据一旦丢失，损失往往是无法估量的。

### 12.1 慢日志

慢查询日志是优化数据库性能的重要工具，记录执行时间超过指定阈值的查询。通过分析慢日志可以找出性能瓶颈，优化查询语句。

**慢日志配置**：slow_query_log开启慢查询日志功能。slow_query_log_file指定日志文件路径。long_query_time设置慢查询阈值（秒），默认是2秒。min_examined_row_limit设置只记录扫描行数超过指定值的查询。log_queries_not_using_indexes记录未使用索引的查询。

**慢日志分析**：定期分析慢日志找出性能瓶颈。关注执行时间长、扫描行数多、未使用索引的查询。对于频繁出现的慢查询，创建合适的索引。对于复杂查询，考虑SQL重构或应用层处理。

```typescript
   * 生成优化建议
   */
  private async generateRecommendations(stats: QueryStat[]): Promise<Recommendation[]> {
    const recommendations: Recommendation[] = [];

    for (const stat of stats) {
      // 检查是否缺少索引
      if (stat.noIndexCount > 0) {
        recommendations.push({
          type: 'missing_index',
          queryPattern: stat.pattern,
          impact: stat.noIndexCount,
          suggestion: `为以下字段创建索引: ${stat.missingIndexes.join(', ')}`,
        });
      }

      // 检查是否需要分页优化
      if (stat.avgExecutionTime > 1000 && stat.hasLimit === false) {
        recommendations.push({
          type: 'pagination',
          queryPattern: stat.pattern,
          impact: stat.avgExecutionTime,
          suggestion: '考虑添加LIMIT或使用游标分页',
        });
      }

      // 检查是否使用SELECT *
      if (stat.usesSelectStar) {
        recommendations.push({
          type: 'select_star',
          queryPattern: stat.pattern,
          impact: stat.avgRowsExamined - stat.avgRowsReturned,
          suggestion: '只选择需要的字段，使用覆盖索引',
        });
      }
    }

    return recommendations.sort((a, b) => b.impact - a.impact);
  }

  private async parseSlowQueryLog(): Promise<SlowQuery[]> {
    // 解析MySQL慢查询日志

  }

  private groupByQueryPattern(queries: SlowQuery[]): QueryStat[] {
    const groups = new Map<string, SlowQuery[]>();

    for (const query of queries) {
      // 归一化查询（替换具体值为占位符）
      const pattern = this.normalizeQuery(query.sql);
      const group = groups.get(pattern) || [];
      group.push(query);
      groups.set(pattern, group);
    }

    return Array.from(groups.entries()).map(([pattern, queries]) => ({
      pattern,
      count: queries.length,
      avgExecutionTime: queries.reduce((sum, q) => sum + q.queryTime, 0) / queries.length,
      maxExecutionTime: Math.max(...queries.map(q => q.queryTime)),
      avgRowsExamined: queries.reduce((sum, q) => sum + q.rowsExamined, 0) / queries.length,
      avgRowsReturned: queries.reduce((sum, q) => sum + q.rowsReturned, 0) / queries.length,
      noIndexCount: queries.filter(q => !q.usedIndex).length,
      hasLimit: queries.some(q => q.sql.includes('LIMIT')),
      usesSelectStar: queries.some(q => q.sql.includes('SELECT *')),
      missingIndexes: this.findMissingIndexes(queries),
    }));
  }

  private normalizeQuery(sql: string): string {
    // 归一化查询，将具体值替换为占位符

      .replace(/'[^']*'/g, '?')
      .replace(/\d+/g, '?');
```

### 12.2 性能指标

数据库性能指标是监控数据库健康状态的重要数据。通过监控这些指标可以及时发现性能问题，避免影响业务。

**核心指标**：QPS（Queries Per Second）每秒查询数，反映数据库吞吐量。TPS（Transactions Per Second）每秒事务数，反映数据库处理能力。连接数当前使用的连接数和最大连接数。缓存命中率InnoDB Buffer Pool命中率，Key Buffer命中率。磁盘IO读写IO次数和吞吐量。锁等待当前等待锁的会话数量。

**系统指标**：CPU使用率、内存使用率、磁盘空间、网络流量等系统资源使用情况。

```typescript
/**
 * 数据库性能指标服务
 */
@Injectable()
export class DatabaseMetricsService {
  private readonly logger = new Logger(DatabaseMetricsService.name);

  /**
   * 获取数据库核心指标
   */
  async getCoreMetrics(): Promise<CoreMetrics> {
    // 并行查询多个指标
    const [qps, tps, connections, bufferPool, diskIO] = await Promise.all([
      this.getQPS(),
      this.getTPS(),
      this.getConnectionStats(),
      this.getBufferPoolStats(),
      this.getDiskIOStats(),

    return {
      qps,
      tps,
      connections,
      bufferPool,
      diskIO,
      timestamp: new Date(),
    };
  }

  /**
   * 获取详细性能指标
   */

      this.getGlobalStatus(),
      this.getInnoDBStats(),
      this.getTableStats(),
      this.getIndexStats(),
    ]);

    return {
      globalStatus,
      innodbStats,
      tableStats,
      indexStats,
      timestamp: new Date(),
    };

  /**
   * 获取QPS

  private async getQPS(): Promise<number> {
    const result = await this.dataSource.query(`
      SHOW GLOBAL STATUS LIKE 'Questions'
    `);
    const questions = parseInt(result[0].Value);

    // 需要对比两次的值计算QPS
    // 实际应用中应该缓存上次查询的值
    return questions;
  }

  /**
   * 获取TPS
   */
  private async getTPS(): Promise<{ committed: number; rolledback: number }> {
    const [commits, rollbacks] = await Promise.all([
      this.dataSource.query(`SHOW GLOBAL STATUS LIKE 'Com_commit'`),
      this.dataSource.query(`SHOW GLOBAL STATUS LIKE 'Com_rollback'`),
    ]);

    return {
      committed: parseInt(commits[0].Value),
      rolledback: parseInt(rollbacks[0].Value),
    };
  }

   * 获取连接统计
   */
  private async getConnectionStats(): Promise<ConnectionStats> {

      this.dataSource.query(`SHOW VARIABLES LIKE 'max_connections'`),
      this.dataSource.query(`SHOW STATUS LIKE 'Threads_connected'`),
      this.dataSource.query(`SHOW STATUS LIKE 'Threads_cached'`),
    ]);

    const current = parseInt(currentConnections[0].Value);
    const max = parseInt(maxConnections[0].Value);

    return {
      current: current,
      max: max,
      usagePercent: (current / max) * 100,
      cached: parseInt(threadCache[0].Value),
    };
  }

      poolSize,

      pagesWritten,
    ] = await Promise.all([
      this.dataSource.query(`SHOW ENGINE INNODB STATUS`),
      // 从InnoDB Status中解析Buffer Pool信息
    ]);

    // 实际应该解析InnoDB Status输出
    return {
      totalSize: 0,
      freePages: 0,
      dataPages: 0,
      dirtyPages: 0,
      hitRate: 0,
      pagesRead: 0,
      pagesWritten: 0,
    };
  }

  /**
   * 获取磁盘IO统计
   */
  private async getDiskIOStats(): Promise<DiskIOStats> {
    // Linux系统可以从/proc/diskstats读取
    // 这里使用MySQL状态
    const [reads, writes] = await Promise.all([
      this.dataSource.query(`SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_read_requests'`),
      this.dataSource.query(`SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_write_requests'`),

      writes: parseInt(writes[0].Value),

  private async getTableStats(): Promise<any[]> {

        TABLE_SCHEMA,

      ORDER BY CARDINALITY DESC
      LIMIT 100
```

### 12.3 告警

数据库告警是及时发现和处理问题的关键，需要设置合理的告警规则和通知机制。一个好的告警系统应该在问题影响用户之前发出通知。

**告警级别**：Critical（严重）数据库不可用或严重性能问题，需要立即处理。Warning（警告）性能下降或资源使用率较高，需要关注。Info（信息）配置变更或非关键事件，供事后分析。

**常见告警项**：数据库连接数超过80%阈值。数据库响应时间超过阈值。复制延迟超过阈值。磁盘空间使用率超过80%。主从切换事件。慢查询数量突增。Buffer Pool命中率低于阈值。

```typescript
/**
 * 数据库告警服务
 */
@Injectable()
export class DatabaseAlertService {
  private readonly logger = new Logger(DatabaseAlertService.name);

  /**
   * 告警配置
   */
  private readonly alertRules: AlertRule[] = [
    // 连接数告警
    {
      name: 'connection_usage',
      metric: 'connections.usagePercent',

      comparison: 'gte',

      metric: 'connections.usagePercent',
      threshold: 95,
      comparison: 'gte',
      level: 'critical',
      duration: 60,

    {
      name: 'replication_lag',
      metric: 'replication.lag',
      threshold: 30,
      comparison: 'gt',
      level: 'warning',
      duration: 120,
    },
    {
      name: 'replication_lag_critical',
      metric: 'replication.lag',
      threshold: 60,
      comparison: 'gt',
      level: 'critical',
      duration: 30,
    },

    // 磁盘空间告警
    {
      name: 'disk_usage',
      metric: 'disk.usagePercent',
      threshold: 80,
      comparison: 'gte',
      level: 'warning',
      duration: 300,
    },
    {
      name: 'disk_usage_critical',
      metric: 'disk.usagePercent',
      threshold: 90,
      comparison: 'gte',
      level: 'critical',
      duration: 60,
    },

    // 慢查询告警
    {
      name: 'slow_query_count',
      metric: 'slowQueries.count',
      threshold: 100,
      comparison: 'gt',
      level: 'warning',
      duration: 300,
    },

    // Buffer Pool命中率告警
    {
      name: 'buffer_pool_hit_rate',
      metric: 'bufferPool.hitRate',
      threshold: 90,
      comparison: 'lt',
      level: 'warning',
      duration: 300,
    },
  ];

  /**
   * 启动告警监控
   */
  startMonitoring() {
    // 每分钟检查一次所有告警规则
    setInterval(async () => {
      await this.checkAllRules();
    }, 60000);
  }

  /**
   * 检查所有告警规则
   */
  private async checkAllRules() {
    try {
      const metrics = await this.metricsService.getCoreMetrics();

      for (const rule of this.alertRules) {
        await this.checkRule(rule, metrics);
      }
    } catch (error) {
      this.logger.error('检查告警规则失败', error);
    }
  }

  /**
   * 检查单条告警规则
   */
  private async checkRule(rule: AlertRule, metrics: CoreMetrics) {
    const value = this.getMetricValue(metrics, rule.metric);
    if (value === null) return;

    const isTriggered = this.compareValue(value, rule.threshold, rule.comparison);

    // 获取该规则的触发状态
    const alertState = this.getAlertState(rule.name);

    if (isTriggered) {
      if (!alertState.isTriggered) {
        // 首次触发
        alertState.triggeredAt = Date.now();
        alertState.triggeredValue = value;
      } else {
        // 持续触发，检查是否满足持续时间
        const duration = (Date.now() - alertState.triggeredAt) / 1000;
        if (duration >= rule.duration) {
          // 发送告警通知
          await this.sendAlert({
            rule: rule.name,
            level: rule.level,
            metric: rule.metric,
            value,
            threshold: rule.threshold,
            duration: duration,
            timestamp: new Date(),
          });
        }
      }
    } else {
      // 重置触发状态
      this.resetAlertState(rule.name);
    }
  }

  /**
   * 发送告警通知
   */
  private async sendAlert(alert: AlertNotification): Promise<void> {
    this.logger.warn(`告警 [${alert.level.toUpperCase()}]: ${alert.rule}`, alert);

    // 根据告警级别选择通知方式
    switch (alert.level) {
      case 'critical':
        await this.notifyCritical(alert);
        break;
      case 'warning':
        await this.notifyWarning(alert);
        break;
      default:
        await this.notifyInfo(alert);
    }

    // 记录告警历史
    await this.saveAlertHistory(alert);
  }

  /**
   * 严重告警通知

  private async notifyCritical(alert: AlertNotification): Promise<void> {
    // 严重告警：电话 + 短信 + 邮件 + 钉钉

      this.sendSMS(alert),

  }

  /**

    // 信息告警：邮件 + 钉钉

      this.sendEmail(alert),

    ]);
  }

  private async sendPhoneCall(alert: AlertNotification): Promise<void> {
    // 调用电话通知服务
    this.logger.log(`[电话] 呼叫值班人员: ${JSON.stringify(alert)}`);
  }

  private async sendSMS(alert: AlertNotification): Promise<void> {
    // 发送短信通知
    this.logger.log(`[短信] 发送告警: ${alert.rule}`);
  }

  private async sendEmail(alert: AlertNotification): Promise<void> {
    // 发送邮件通知
    this.logger.log(`[邮件] 发送告警: ${alert.rule}`);
  }

  private async sendDingTalk(alert: AlertNotification): Promise<void> {
    // 发送钉钉通知
    this.logger.log(`[钉钉] 发送告警: ${alert.rule}`);
  }

  private getMetricValue(metrics: CoreMetrics, path: string): number | null {
    const parts = path.split('.');
    let value: any = metrics;

    for (const part of parts) {
      if (value === undefined || value === null) return null;
      value = value[part];
    }

    return typeof value === 'number' ? value : null;

  private compareValue(value: number, threshold: number, comparison: string): boolean {
```

### 12.4 我的思考：监控是保障

监控不是为了看数据，而是为了及时发现问题。优秀的监控系统应该在用户感知到问题之前就发出告警，给运维人员留出足够的应急响应时间。

**监控指标不是越多越好**，关键是找到真正反映系统健康的指标，避免被海量数据淹没。常见的核心指标包括响应时间、吞吐量、错误率、资源使用率等。

**告警策略需要持续优化**，告警过多会导致"狼来了"效应，让人忽视真正的告警；告警过少可能遗漏重要问题。建议定期回顾告警记录，优化告警规则。

**自动化处理是监控的未来方向**，能够自动识别问题、自动分析原因、自动执行修复的智能监控系统将成为主流。但自动化不是万能的，对于复杂问题仍然需要人工介入。
