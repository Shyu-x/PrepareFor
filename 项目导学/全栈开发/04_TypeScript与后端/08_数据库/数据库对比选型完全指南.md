## 引言：数据库选型——全栈工程师的第一道分水岭

作为一个全栈工程师，你可能已经发现了：在实际项目中，数据库选型往往比代码实现更让人头疼。一个好的数据库选择，可以让后续的开发事半功倍；而一个错误的选择，则可能让你在项目后期付出十倍的代价来弥补。

我记得有一次，一个创业团队的朋友兴奋地告诉我，他们要用 MongoDB 重构整个电商后端。理由是"MongoDB 更灵活"、"不需要预先定义 schema"、"扩展性强"。三个月后，他们遇到了严重的问题：复杂的商品关联查询性能极差，事务一致性无法保证，最终不得不花大力气迁移回 PostgreSQL。

这个故事告诉我们：**没有银弹，只有合适**。本文将带你深入理解各种数据库的特点，学会在不同场景下做出正确的选择。

## 第一章：为什么数据库选择这么难

### 1.1 SQL vs NoSQL：这不是二选一

很多人初学数据库时，都会遇到一个经典的"选择题"：学 SQL 还是学 NoSQL？市面上的文章也经常把两者对立起来，仿佛它们是水火不容的两种技术。

但实际情况恰恰相反。**SQL 和 NoSQL 不是竞争关系，而是互补关系**。

**SQL（关系型数据库）的本质是：**

- 使用 SQL 语言进行操作
- 数据存储在预先定义好的表中
- 表与表之间可以通过外键建立关联
- 支持复杂的 JOIN 查询和事务

**NoSQL（泛指非关系型数据库）的本质是：**

- 不使用传统 SQL 语言（或使用类 SQL 语言）
- 数据存储方式灵活多变（文档、键值、列族、图等）
- 通常不支持 JOIN，但性能更高
- 弱化事务一致性，追求最终一致性

举一个生活中的例子：SQL 就像是一套固定的家具，你需要按照说明书来组装；NoSQL 就像是一堆积木，你可以自由组合成任何形状。它们各有各的用途，你不会用积木来盖房子，也不会用家具来搭乐高。

**实际项目中的选择：**

现代全栈应用很少只使用单一数据库。Twitter 早期只用 MySQL，但后来引入了 Redis 做缓存、Gizzard 做分布式存储；Netflix 同时使用 Cassandra、DynamoDB、Elasticsearch 和 MySQL。

我的经验法则是：**用 SQL 数据库作为主数据库，用 NoSQL 作为补充**。除非有明确的理由，否则不要把 NoSQL 当作主力数据库。

### 1.2 关系型 vs 文档型：各自擅长什么

关系型数据库（如 PostgreSQL、MySQL）和文档型数据库（如 MongoDB）是两个最常被拿来比较的类型。让我用一张表来清晰地展示它们的差异：

| 维度 | 关系型数据库 | 文档型数据库 |
|------|-------------|-------------|
| **数据结构** | 固定的表结构 | 灵活的 JSON 文档 |
| **关联查询** | 强大的 JOIN | 需要手动处理或嵌套 |
| **事务支持** | 原生 ACID 事务 | 有限的事务支持 |
| **扩展方式** | 垂直扩展为主 | 水平扩展为主 |
| **适用场景** | 强一致性业务 | 快速迭代、原型开发 |
| **查询语言** | SQL | 类 SQL 或 API |
| **社区生态** | 成熟稳定 | 快速迭代 |

**关系型数据库擅长的场景：**

1. **金融系统**：需要严格的 ACID 事务，比如转账操作必须同时成功或同时失败
2. **电商订单系统**：订单、用户、商品之间有复杂的关联关系
3. **ERP 系统**：组织结构、权限管理需要复杂的层级关系
4. **数据分析报表**：需要多表关联的复杂统计查询

**文档型数据库擅长的场景：**

1. **内容管理系统**：文章、评论、标签之间的嵌套关系
2. **用户画像系统**：灵活的用户属性，随时可以添加新字段
3. **日志收集系统**：海量的半结构化日志数据
4. **物联网数据**：设备传感器数据，格式不统一
5. **快速原型开发**：不确定的数据模型，需要快速迭代

### 1.3 我的思考：没有银弹，只有合适

我见过太多团队在数据库选型上犯错了。他们的共同点是：**被某个数据库的某个特性吸引，然后决定把它用在整个项目上**。

比如，我见过用 MongoDB 存储所有数据的团队，最后被复杂的关联查询折磨得死去活来；我也见过用 MySQL 存储日志的团队，数据库膨胀到几百 GB，每次查询都超时。

**我的选型原则是：**

1. **先想清楚业务场景**：是读多写少还是写多写少？需要强一致性吗？数据量预估多大？
2. **从最小化方案开始**：能用一个数据库解决的问题，绝不用两个
3. **让数据选择数据库**：而不是让团队偏好选择数据库
4. **考虑团队能力**：如果团队熟悉 PostgreSQL，不要为了"先进"而选择 CockroachDB
5. **为变化留有余地**：选择那些迁移成本低的方案

## 第二章：PostgreSQL vs MySQL

### 2.1 两大开源数据库的对决

在开源关系型数据库领域，PostgreSQL 和 MySQL 是当之无愧的两大巨头。它们都有着数十年的历史，都非常成熟稳定，都是开源免费的。那它们的区别在哪里呢？

**MySQL 的历史：**

- 1995 年诞生，由瑞典 MySQL AB 公司开发
- 2008 年被 Sun Microsystems 收购
- 2009 年 Oracle 收购 Sun，MySQL 成为 Oracle 的产品
- 2009 年，MySQL 原团队创建了 MariaDB 作为 MySQL 的开源分支

**PostgreSQL 的历史：**

- 1986 年由加州大学伯克利分校开发
- 1996 年发布第一个版本 6.0，正式命名为 PostgreSQL
- 一直由开源社区维护，没有被商业公司收购
- 被称为"世界上最先进的开源关系型数据库"

**架构上的核心差异：**

MySQL 的设计哲学是"简单高效"。它最初是为 Web 应用设计的，读多写少的场景优化得很好。MySQL 使用插件式存储引擎，InnoDB 是默认也是最重要的存储引擎。

PostgreSQL 的设计哲学是"功能完备"。它最初是为复杂查询设计的，支持几乎所有 SQL 标准特性。PostgreSQL 是单一存储引擎，但支持多种数据类型和索引方式。

### 2.2 JSON支持：PG后来居上

很长一段时间里，MySQL 的 JSON 支持都比 PostgreSQL 弱。但随着版本更新，两者在这个领域的差距已经越来越小。

**MySQL 的 JSON 支持（5.7+）：**

```sql
-- 创建包含 JSON 字段的表
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    profile JSON
);

-- 插入 JSON 数据
INSERT INTO users (id, name, profile) VALUES (
    1,
    '张三',
    '{"age": 28, "city": "北京", "interests": ["读书", "旅行"]}'
);

-- 查询 JSON 字段中的值
SELECT name, profile->>'$.city' AS city FROM users;
-- 结果：name=张三, city=北京

-- 使用 JSON_EXTRACT 函数
SELECT JSON_EXTRACT(profile, '$.age') AS age FROM users;
-- 结果：28
```

**PostgreSQL 的 JSON 支持（9.2+）：**

```sql
-- 创建包含 JSONB 字段的表（JSONB 是二进制格式，查询更快）
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    profile JSONB
);

-- 插入 JSON 数据
INSERT INTO users (name, profile) VALUES (
    '张三',
    '{"age": 28, "city": "北京", "interests": ["读书", "旅行"]}'::jsonb
);

-- 使用 JSONB 操作符查询
SELECT name, profile->>'city' AS city FROM users;
-- 结果：name=张三, city=北京

-- 创建 GIN 索引加速 JSON 查询
CREATE INDEX idx_users_profile ON users USING GIN (profile);

-- 使用表达式索引查询特定路径
SELECT * FROM users WHERE profile @> '{"city": "北京"}';
```

**两者对比：**

| 特性 | MySQL | PostgreSQL |
|------|-------|-----------|
| JSON 数据类型 | JSON | JSON / JSONB |
| JSON 查询函数 | JSON_EXTRACT | jsonb_each, jsonb_array_elements |
| JSON 索引 | 虚拟列索引 | GIN 索引、表达式索引 |
| JSON 写入性能 | 稍好 | 稍差（二进制存储） |
| JSON 查询性能 | 稍差 | 更好（索引支持） |
| JSON 修改函数 | JSON_SET, JSON_INSERT | jsonb_set, jsonb_insert |

**我的使用经验：**

如果你的业务需要频繁存储和查询 JSON 数据，PostgreSQL 是更好的选择。JSONB 类型可以建立索引，这在大量数据查询时性能差异非常明显。

### 2.3 性能对比：OLTP vs OLAP

性能对比是一个敏感话题，因为性能很大程度上取决于使用场景。我会从 OLTP（在线事务处理）和 OLAP（在线分析处理）两个维度来分析。

**OLTP 场景（典型 Web 应用）：**

```sql
-- 典型的电商订单查询
SELECT o.id, o.created_at, u.name, p.title
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id
WHERE o.status = 'paid' AND o.created_at > '2024-01-01'
ORDER BY o.created_at DESC
LIMIT 20;
```

在这个场景下，两者性能差异不大。MySQL 凭借简单的架构，在简单查询上略有优势；PostgreSQL 在复杂 JOIN 上优化更好。

**OLAP 场景（数据分析报表）：**

```sql
-- 典型的销售统计分析
SELECT
    DATE_TRUNC('month', o.created_at) AS month,
    c.category,
    COUNT(*) AS order_count,
    SUM(o.amount) AS total_amount,
    AVG(o.amount) AS avg_amount
FROM orders o
JOIN products p ON o.product_id = p.id
JOIN categories c ON p.category_id = c.id
WHERE o.created_at >= '2023-01-01'
GROUP BY DATE_TRUNC('month', o.created_at), c.category
ORDER BY month DESC, total_amount DESC;
```

在 OLAP 场景下，PostgreSQL 有明显优势：

1. **窗口函数**：PostgreSQL 原生支持窗口函数，MySQL 8.0 才开始支持
2. **CTE（公用表表达式）**：PostgreSQL 支持递归 CTE，MySQL 8.0 才支持
3. **物化视图**：PostgreSQL 支持物化视图加速复杂查询
4. **并行查询**：PostgreSQL 可以利用多核进行并行查询

**基准测试参考：**

根据 PostgreSQL 官方和第三方测试机构的测试结果：

- **简单查询**：MySQL 略快（5-10%）
- **复杂关联查询**：PostgreSQL 快 20-50%
- **大批量插入**：MySQL 快 10-20%（得益于 InnoDB 的插入缓冲）
- **高并发写入**：两者相当
- **地理空间查询**：PostgreSQL + PostGIS 远胜 MySQL

### 2.4 我的思考：为什么 MySQL 仍然是王者

尽管 PostgreSQL 功能更强大，但 MySQL 仍然是世界上使用最广泛的数据库。根据 DB-Engines 的统计，MySQL 排名第一，PostgreSQL 排名第四（仅次于 Oracle 和 Microsoft SQL Server）。

**MySQL 的优势：**

1. **生态成熟**：WordPress、Facebook、Twitter、阿里巴巴等巨头都在用，有大量的最佳实践
2. **工具丰富**：phpMyAdmin、MySQL WorkbenchNavicat 等工具非常成熟
3. **云支持好**：AWS RDS、阿里云 RDS、腾讯云数据库都优先支持 MySQL
4. **性能稳定**：在 Web 应用这个细分领域，MySQL 经过了大量生产环境验证
5. **学习资源多**：遇到问题更容易找到解决方案

**PostgreSQL 的优势：**

1. **功能完备**：如果需要 GIS、时序数据、全文检索，PostgreSQL 是更好的选择
2. **扩展性强**：可以安装扩展，比如 PostGIS、TimescaleDB、pgvector
3. **标准兼容**：SQL 标准支持更完整
4. **数据类型丰富**：数组、范围类型、JSONB、UUID、IP 地址等
5. **社区活跃**：近年来 PostgreSQL 社区发展迅速

**我的选择建议：**

- **个人项目、小型 Web 应用**：选 MySQL，生态成熟，遇到问题容易解决
- **企业级应用、需要 GIS/时序数据**：选 PostgreSQL
- **不确定**：先选 MySQL，等业务发展到需要 PostgreSQL 特性时再迁移
- **两个都要**：用 MySQL 做主数据库，PostgreSQL 做特定场景补充

## 第三章：MongoDB vs PostgreSQL

### 3.1 文档数据库的适用场景

MongoDB 是最流行的文档型数据库，它的核心特点是用 BSON 格式存储数据，类似于 JSON 但更高效。

**MongoDB 的数据结构：**

```javascript
// users 集合中的一个文档
{
    "_id": ObjectId("..."),
    "name": "张三",
    "email": "zhangsan@example.com",
    "profile": {
        "age": 28,
        "city": "北京",
        "education": {
            "school": "北京大学",
            "degree": "硕士"
        }
    },
    "interests": ["读书", "旅行", "摄影"],
    "created_at": ISODate("2024-01-01T00:00:00Z")
}
```

对比关系型数据库的表结构：

| users 表 | 关系 |
|---------|------|
| id | 主键 |
| name | 普通字段 |
| email | 普通字段 |
| profile | 需要 JSON 字段或一对一关联 |
| interests | 需要多对多关联表 |
| education | 需要外键关联 |

**MongoDB 擅长的场景：**

1. **内容管理系统（CMS）**

   ```javascript
   // 文章文档，包含评论和标签
   {
       "_id": ObjectId("..."),
       "title": "深入理解 React Hooks",
       "content": "...",
       "author": {
           "id": ObjectId("..."),
           "name": "李四",
           "avatar": "..."
       },
       "tags": ["React", "Hooks", "前端"],
       "comments": [
           {
               "user": "王五",
               "content": "写得真好！",
               "created_at": ISODate("...")
           }
       ],
       "published_at": ISODate("...")
   }
   ```

   在这个场景下，MongoDB 可以用一个文档存储完整的信息，而关系型数据库需要多个表和多次 JOIN。

2. **用户画像系统**

   ```javascript
   // 用户画像，灵活存储各种属性
   {
       "user_id": ObjectId("..."),
       "behavior": {
           "last_login": ISODate("..."),
           "login_count": 1523,
           "total_spent": 25800.50
       },
       "preferences": {
           "theme": "dark",
           "language": "zh-CN",
           "notifications": {
               "email": true,
               "push": false
           }
       },
       "dynamic_attrs": {
           "favorite_category": "电子产品",
           "avg_session_time": 1800,
           "purchase_power": "high"
       }
   }
   ```

   MongoDB 允许随时添加新字段，不需要修改表结构。

3. **物联网（IoT）数据**

   ```javascript
   // 传感器数据
   {
       "device_id": "sensor_001",
       "timestamp": ISODate("2024-01-15T10:30:00Z"),
       "readings": {
           "temperature": 23.5,
           "humidity": 65,
           "pressure": 1013.25,
           "air_quality": {
               "pm25": 35,
               "pm10": 58
           }
       },
       "location": {
           "type": "Point",
           "coordinates": [116.4074, 39.9042]
       }
   }
   ```

### 3.2 关联查询：MongoDB 的痛

MongoDB 最大的痛点就是关联查询。虽然 MongoDB 提供了 `$lookup` 进行聚合查询，但相比 SQL 的 JOIN，差得太远了。

**MongoDB 的关联查询（聚合管道）：**

```javascript
// 查找用户的所有订单
db.orders.aggregate([
    {
        // 匹配用户
        $match: { user_id: ObjectId("...") }
    },
    {
        // 关联用户信息
        $lookup: {
            from: "users",
            localField: "user_id",
            foreignField: "_id",
            as: "user"
        }
    },
    {
        // 展开用户数组
        $unwind: "$user"
    },
    {
        // 关联商品信息
        $lookup: {
            from: "products",
            localField: "product_id",
            foreignField: "_id",
            as: "product"
        }
    },
    {
        $unwind: "$product"
    },
    {
        // 选择需要的字段
        $project: {
            order_id: "$_id",
            user_name: "$user.name",
            product_title: "$product.title",
            amount: 1
        }
    }
])
```

对比 SQL：

```sql
SELECT o.id AS order_id, u.name AS user_name, p.title AS product_title, o.amount
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id
WHERE o.user_id = '...';
```

**问题在哪里？**

1. **语法复杂**：MongoDB 的聚合管道需要多步骤操作，SQL 一个 SELECT 就搞定了
2. **性能问题**：`$lookup` 是低效的关联，执行时间长
3. **维护困难**：聚合管道难以理解和调试
4. **功能有限**：不支持复杂的关联逻辑

### 3.3 事务支持：MongoDB 4.0 的转折

MongoDB 4.0 是转折点，它引入了副本集事务支持；MongoDB 4.2 引入了分片集群的事务支持。这意味着 MongoDB 终于可以处理跨文档事务了。

**MongoDB 事务示例：**

```javascript
const session = db.getMongo().startSession();

session.startTransaction({
    readConcern: { level: "snapshot" },
    writeConcern: { w: "majority" }
});

try {
    const ordersCollection = session.getDatabase("ecommerce").orders;
    const productsCollection = session.getDatabase("ecommerce").products;
    const usersCollection = session.getDatabase("ecommerce").users;

    // 创建订单
    ordersCollection.insertOne({
        user_id: ObjectId("..."),
        product_id: ObjectId("..."),
        quantity: 2,
        total_price: 199.98,
        status: "pending"
    }, { session });

    // 扣减库存
    productsCollection.updateOne(
        { _id: ObjectId("..."), stock: { $gte: 2 } },
        { $inc: { stock: -2 } },
        { session }
    );

    // 增加用户积分
    usersCollection.updateOne(
        { _id: ObjectId("...") },
        { $inc: { points: 200 } },
        { session }
    );

    // 提交事务
    session.commitTransaction();
} catch (error) {
    // 发生错误，回滚事务
    session.abortTransaction();
    throw error;
} finally {
    session.endSession();
}
```

**但是，事务是要付出代价的：**

1. **性能损耗**：事务需要额外的协调和锁管理
2. **复杂性增加**：应用代码需要显式管理事务
3. **不是万能药**：跨分片事务的性能仍然不理想

**我的建议**：如果你的业务需要频繁使用事务，优先考虑关系型数据库。MongoDB 的事务是"补救措施"，不是"核心能力"。

### 3.4 我的思考：什么时候该用 MongoDB

经过这么多年的经验，我总结了什么时候该用 MongoDB：

**适合用 MongoDB 的场景：**

1. **数据模型不确定，需要快速迭代**

   比如创业初期，业务模型还在探索阶段，数据结构可能频繁变化。MongoDB 的灵活 schema 允许你随时添加新字段，而不需要 `ALTER TABLE`。

2. **以文档为中心的数据模型**

   比如博客文章、用户资料、产品目录等，数据天然就是文档结构，使用 MongoDB 可以避免复杂的表关联。

3. **需要快速开发原型**

   MongoDB 的 CRUD 操作非常直观，可以用更少的代码完成更多的功能。

4. **数据量预估在 TB 级别，需要水平扩展**

   MongoDB 的分片功能对大数据量的支持比较好，而且配置相对简单。

**不适合用 MongoDB 的场景：**

1. **需要复杂关联查询**

   比如电商的订单统计、需要 JOIN 多张表的报表分析。

2. **需要强一致性事务**

   比如金融转账、库存扣减（虽然 MongoDB 支持事务，但性能和体验都不如关系型）。

3. **数据结构固定，不会频繁变化**

   如果数据结构很稳定，关系型数据库的性能和功能都更好。

4. **需要高级 SQL 特性的分析查询**

   比如窗口函数、物化视图、复杂的聚合查询。

### 4.2 持久化选项：RDB vs AOF

Redis 是内存数据库，但数据是可以持久化到磁盘的。Redis 提供了两种持久化方式：**RDB** 和 **AOF**。

**RDB（Redis Database）：**

RDB 是定时快照，会在指定的时间间隔内生成数据集的完整副本。

```bash
# 配置文件 redis.conf
save 900 1      # 900秒内至少1个key变化则保存
save 300 10     # 300秒内至少10个key变化则保存
save 60 10000    # 60秒内至少10000个key变化则保存

# 手动触发快照
BGSAVE
```

```javascript
// Node.js 中使用 ioredis 执行快照
await redis.bgsave();
```

**AOF（Append Only File）：**

AOF 是追加日志，记录所有写操作，可以在服务重启时重放。

```bash
# 配置文件 redis.conf
appendonly yes
appendfilename "appendonly.aof"

# 同步策略
appendfsync always      # 每次写都同步（最安全，最慢）
appendfsync everysec    # 每秒同步（默认，平衡方案）
appendfsync no          # 由操作系统决定何时同步（最快，可能丢失数据）
```

**两者的对比：**

| 特性 | RDB | AOF |
|------|-----|-----|
| **文件大小** | 小（完整快照） | 大（累积操作日志） |
| **恢复速度** | 快（直接加载） | 慢（重放操作） |
| **数据完整性** | 取决于快照间隔 | 取决于同步策略 |
| **写入性能** | 优秀（后台定期保存） | `everysec` 模式下良好 |
| **系统资源消耗** | 高（fork 进程） | 低（顺序写入） |

**最佳实践：同时开启两种持久化：**

```bash
# redis.conf
save 900 1
save 300 10
save 60 10000

appendonly yes
appendfsync everysec
```

在这种情况下，Redis 重启时会优先使用 AOF 恢复数据，因为 AOF 的数据更完整。只有当 AOF 文件不存在时，才会使用 RDB。

**AOF 重写：**

AOF 文件会随着时间不断增大，Redis 提供了重写机制来压缩 AOF 文件：

```bash
# 手动触发重写
BGREWRITEAOF
```

或者配置自动重写：

```bash
# redis.conf
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

### 4.3 集群方案：主从、哨兵、Cluster

Redis 有三种集群方案，分别是主从复制、哨兵模式和 Cluster 模式。

**主从复制（Replication）：**

主从复制是最简单的方案，一个主节点（master）可以有一个或多个从节点（slave）。主节点负责写入，从节点负责读取。

```
┌─────────────┐
│   Master    │  写：SET key value
│  (写入)     │
└──────┬──────┘
       │ 同步
       ▼
┌─────────────┐
│   Slave 1   │  读：GET key
│  (读取)     │
└─────────────┘
       │
┌─────────────┐
│   Slave 2   │  读：GET key
│  (读取)     │
└─────────────┘
```

```bash
# 从节点配置文件
replicaof 192.168.1.100 6379
replica-read-only yes
```

```javascript
// Node.js ioredis 配置主从
const redis = new Redis({
    host: '192.168.1.100',
    port: 6379,
    maxRetriesPerRequest: 3
});

// 读取操作会自动分发到从节点
```

**哨兵模式（Sentinel）：**

哨兵模式在主从复制的基础上，增加了自动故障转移功能。哨兵是一个独立进程，负责监控主从节点的健康状态，当主节点故障时，自动选举从节点为新的主节点。

```
         ┌─────────────┐
         │  Sentinel   │
         │  (监控)     │
         └──────┬──────┘
                │
       ┌────────┴────────┐
       ▼                 ▼
┌─────────────┐    ┌─────────────┐
│   Master    │    │   Slave 1   │
│  (主节点)   │    │  (从节点)   │
└─────────────┘    └─────────────┘
       │                 │
       └────────┬────────┘
                ▼
         ┌─────────────┐
         │   Slave 2   │
         │  (从节点)   │
         └─────────────┘
```

```bash
# sentinel.conf
sentinel monitor mymaster 192.168.1.100 6379 2
sentinel down-after-milliseconds mymaster 5000
sentinel failover-timeout mymaster 60000
```

```javascript
// Node.js 使用哨兵模式

    sentinels: [
        { host: '192.168.1.101', port: 26379 },
        { host: '192.168.1.102', port: 26379 },
        { host: '192.168.1.103', port: 26379 }
    ],
    name: 'mymaster',
    password: 'your-password'
});
```

**Cluster 模式（集群）：**

Cluster 模式是 Redis 3.0 引入的分布式方案，数据会自动分片到多个节点，每个节点负责一部分数据。Cluster 提供了自动分片和故障转移能力。

```
        ┌───────────────────┐
        │   Client (任意)   │
        └─────────┬─────────┘
                  │
        ┌─────────┴─────────┐
        ▼                   ▼
┌───────────────┐     ┌───────────────┐
│  Cluster Node │     │  Cluster Node │
│    (Node 1)   │─────│    (Node 2)   │
│  Slot 0-5460  │     │  Slot 5461-   │
└───────────────┘     │    10922      │
                      └───────────────┘
                              │
                      ┌───────────────┐
                      │  Cluster Node │
                      │    (Node 3)   │
                      │  Slot 10923-  │
                      │    16383      │
                      └───────────────┘
```

```bash

    192.168.1.100:6379 \
    192.168.1.101:6379 \
    192.168.1.102:6379 \
    192.168.1.103:6379 \
    192.168.1.104:6379 \
    192.168.1.105:6379 \
    --cluster-replicas 1
```

## 第五章：关系型 vs 文档型

### 5.1 什么时候用关系型

关系型数据库是大多数应用的主力选择，特别是在以下场景中：

**强一致性要求的场景：**

```sql
-- 转账场景：必须原子性完成
BEGIN TRANSACTION;

-- 检查余额
SELECT balance FROM accounts WHERE user_id = 123 FOR UPDATE;

-- 扣款
UPDATE accounts SET balance = balance - 1000 WHERE user_id = 123;

-- 打款
UPDATE accounts SET balance = balance + 1000 WHERE user_id = 456;

COMMIT;
```

如果使用 MongoDB，需要用事务，但性能和体验都不如关系型数据库。

**复杂关联查询的场景：**

```sql
-- 电商统计报表：涉及多表关联
SELECT
    DATE(o.created_at) AS date,
    c.category_name,
    p.product_name,
    COUNT(DISTINCT o.user_id) AS buyer_count,
    SUM(o.quantity) AS total_quantity,
    SUM(o.amount) AS total_amount,
    AVG(o.amount) AS avg_order_amount
FROM orders o
JOIN products p ON o.product_id = p.id
JOIN categories c ON p.category_id = c.id
JOIN users u ON o.user_id = u.id
WHERE o.status = 'completed'
    AND o.created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
    AND u.level IN ('gold', 'platinum')
GROUP BY DATE(o.created_at), c.category_name, p.product_name
HAVING SUM(o.amount) > 10000
ORDER BY date DESC, total_amount DESC;
```

关系型数据库的 JOIN 优化器可以高效执行这类查询，而 MongoDB 的 `$lookup` 在复杂场景下性能堪忧。

**数据完整性要求高的场景：**

```sql
-- 使用外键约束保证数据一致性
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    quantity INT NOT NULL CHECK (quantity > 0),
    amount DECIMAL(10, 2) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE RESTRICT,
    FOREIGN KEY (product_id) REFERENCES products(id) ON DELETE RESTRICT
);

-- 使用唯一索引防止重复
CREATE UNIQUE INDEX idx_user_product ON orders(user_id, product_id);
```

关系型数据库的约束机制（外键、唯一索引、检查约束）可以保证数据质量，MongoDB 需要在应用层实现这些逻辑。

```javascript
    // 缓存未命中，查 MongoDB
    product = await mongo.products.findOne({ _id: ObjectId(productId) });

    // 写入缓存，过期时间 1 小时
    await redis.setex(cacheKey, 3600, JSON.stringify(product));

    return product;
}

async function placeOrder(userId, productId, quantity) {
    const lockKey = `lock:product:${productId}`;

    // 获取分布式锁
    const lock = await redis.set(lockKey, '1', 'NX', 'EX', 10);
    if (!lock) {
        throw new Error('库存正在被处理，请稍后重试');
    }

    try {
        // 开启 PostgreSQL 事务
        await pg.transaction(async (trx) => {
            // 检查并扣减库存
            const result = await trx.query(
                `UPDATE products
                 SET stock = stock - $1
                 WHERE id = $2 AND stock >= $1
                 RETURNING *`,
                [quantity, productId]
            );

            if (result.rows.length === 0) {
                throw new Error('库存不足');
            }

            // 创建订单
            await trx.query(
                `INSERT INTO orders (user_id, product_id, quantity, total_amount)
                 VALUES ($1, $2, $3, $4)`,
                [userId, productId, quantity, result.rows[0].price * quantity]
            );
        });
    } finally {
        // 释放锁
        await redis.del(lockKey);
    }
}
```

### 5.4 我的思考：混用才是常态

我在实际项目中几乎没有见过只使用单一数据库的系统。**混用数据库不是过度设计，而是务实的选择**。

**我的数据库混用原则：**

1. **PostgreSQL 作为主数据库**

   所有需要强一致性、复杂关联查询的数据，都放在 PostgreSQL 中。用户账户、订单、支付、财务记录，这些是核心资产，不能妥协。

2. **MongoDB 作为文档数据库**

   所有数据结构灵活、不需要复杂关联的数据，放在 MongoDB 中。文章内容、用户生成内容、日志数据，这些用文档模型更自然。

3. **Redis 作为缓存和专用存储**

   所有对性能要求极高、不需要持久化的数据，放在 Redis 中。会话数据、热点缓存、排行榜、实时计数器，Redis 是最佳选择。

4. **Elasticsearch 作为搜索引擎**

   所有需要全文搜索、复杂查询的数据，放在 Elasticsearch 中。商品搜索、日志分析、数据可视化，Elasticsearch 是专业选择。

**数据库选型决策树：**

```
需要存储数据？
├── 数据结构是否灵活多变？
│   ├── 是 → MongoDB
│   └── 否 → 需要复杂关联查询吗？
│           ├── 是 → PostgreSQL
│           └── 否 → 数据量预估多大？
│                   ├── 是 → 考虑 Elasticsearch 或 ClickHouse
│                   └── 否 → MySQL（简单场景够用）
│
├── 需要强一致性事务吗？
│   ├── 是 → PostgreSQL（首选）或 MySQL
│   └── 否 → 可以考虑其他方案
│
├── 需要毫秒级响应吗？
│   ├── 是 → Redis（作为缓存层）
│   └── 否 → 其他方案
│
└── 需要全文搜索或复杂分析吗？
    ├── 是 → Elasticsearch 或 PostgreSQL 全文搜索
    └── 否 → 继续用当前选择
```

## 第六章：ORM vs 原生 SQL

### 6.1 ORM 的价值：快速开发

ORM（Object-Relational Mapping，对象关系映射）是一类将关系型数据库表映射为编程语言对象的工具。它的核心价值是：**让开发者可以用面向对象的方式操作数据库，而不需要写 SQL**。

**不用 ORM 的代码（原生 SQL）：**

```javascript
// Node.js 使用 node-postgres
const { Pool } = require('pg');
const pool = new Pool({
    connectionString: process.env.DATABASE_URL
});

async function getUserWithOrders(userId) {
    const client = await pool.connect();

    try {
        await client.query('BEGIN');

        // 查询用户
        const userResult = await client.query(
            'SELECT * FROM users WHERE id = $1',
            [userId]
        );

        if (userResult.rows.length === 0) {
            throw new Error('用户不存在');
        }

        const user = userResult.rows[0];

        // 查询订单
        const ordersResult = await client.query(
            `SELECT o.*, p.title as product_title, p.price as product_price
             FROM orders o
             JOIN products p ON o.product_id = p.id
             WHERE o.user_id = $1
             ORDER BY o.created_at DESC`,
            [userId]
        );

        await client.query('COMMIT');

        return {
            user,
            orders: ordersResult.rows
        };
    } catch (error) {
        await client.query('ROLLBACK');
        throw error;
    } finally {
        client.release();
    }
}
```

**使用 ORM 的代码（Prisma）：**

```javascript
// Prisma ORM
const { PrismaClient } = require('@prisma/client');
const prisma = new PrismaClient();

async function getUserWithOrders(userId) {
    const user = await prisma.user.findUnique({
        where: { id: userId },
        include: {
            orders: {
                include: {
                    product: true
                },
                orderBy: {
                    createdAt: 'desc'
                }
            }
        }
    });

    if (!user) {
        throw new Error('用户不存在');
    }

    return user;
}
```

**ORM 的核心优势：**

1. **类型安全**：TypeScript 项目中，ORM 可以提供完整的类型推断

   ```typescript
   // Prisma 生成的类型
   type User = {
       id: number;
       email: string;
       name: string | null;
       createdAt: Date;
       orders: Order[];
   };

   // 编译时检查，防止拼写错误
   const user = await prisma.user.findUnique({
       where: { id: 1 },
       select: { namee: true } // TypeScript 会报错：namee 不存在
   });
   ```

2. **迁移管理**：数据库版本控制

   ```bash
   # 创建迁移
   npx prisma migrate dev --name add_user_age

   # 生产环境应用迁移
   npx prisma migrate deploy
   ```

3. **数据库无关**：同一套代码可以切换数据库

   ```prisma
   // schema.prisma
   datasource db {
       provider = "postgresql"  // 改成 "mysql" 即可切换数据库
       url      = env("DATABASE_URL")
   }
   ```

4. **防止 SQL 注入**：参数化查询是内置的

   ```typescript
   // Prisma 自动使用参数化查询
   await prisma.user.findMany({
       where: {
           email: userInput // 自动转义，无需担心注入
       }
   });
   ```

### 6.2 ORM 的代价：性能损耗

ORM 不是银弹，它也有很多问题。

**性能损耗的来源：**

1. **N+1 查询问题**

   ```typescript
   // 看似只查了一次，实际上查了 1 + N 次
   const users = await prisma.user.findMany(); // 1次查询

   for (const user of users) {
       console.log(user.orders.length); // N次查询
   }

   // 解决方案：使用 include 或 join
   const users = await prisma.user.findMany({
       include: { orders: true } // 只查2次
   });
   ```

2. **生成的 SQL 不够优化**

   ```typescript
   // Prisma 生成的 SQL
   const users = await prisma.user.findMany({
       where: {
           OR: [
               { name: { contains: '张' } },
               { email: { contains: 'zhang' } }
           ]
       }
   });

   // 可能生成：
   // SELECT * FROM users
   // WHERE name ILIKE '%张%' OR email ILIKE '%zhang%'

   // 实际需要：
   // SELECT * FROM users
   // WHERE name ILIKE '%张%'
   // UNION
   // SELECT * FROM users
   // WHERE email ILIKE '%zhang%'
   // （使用 UNION 可能更高效）
   ```

3. **无法表达复杂查询**

   ```sql
   -- 某些复杂查询用 ORM 很难表达
   SELECT
       DATE_TRUNC('month', created_at) AS month,
       COUNT(*) AS order_count,
       PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY amount) AS median_amount
   FROM orders
   WHERE status = 'completed'
   GROUP BY DATE_TRUNC('month', created_at)
   HAVING COUNT(*) > 100
   ORDER BY month DESC;
   ```

**性能对比测试：**

```typescript
// 测试：查询1000个用户及其订单
// Prisma with include: ~150ms
const users = await prisma.user.findMany({
    include: { orders: true },
    take: 1000
});

// 原生 SQL: ~50ms
const result = await pool.query(`
    SELECT u.*, json_agg(o.*) as orders
    FROM users u
    LEFT JOIN orders o ON u.id = o.user_id
    WHERE u.id IN (SELECT id FROM users LIMIT 1000)
    GROUP BY u.id
`);
```

### 6.3 Prisma vs TypeORM vs Sequelize

Node.js 生态中最流行的三个 ORM 是 Prisma、TypeORM 和 Sequelize。让我对比一下它们的特点：

| 特性 | Prisma | TypeORM | Sequelize |
|------|--------|---------|-----------|
| **设计理念** | 新型，类型安全优先 | 传统，功能完备 | 传统，入门简单 |
| **TypeScript 支持** | 原生，类型推断完美 | 一般，需要手动类型 | 一般 |
| **迁移系统** | 内置，声明式 | 内置，命令式 | 需要额外工具 |
| **查询方式** | 链式 API | Repository / Active Record | 链式 API |
| **性能** | 优秀 | 中等 | 中等 |
| **学习曲线** | 平缓 | 陡峭 | 平缓 |
| **当前维护状态** | 活跃 | 活跃 | 维护但不活跃 |
| **星标数** | 35k+ | 35k+ | 29k+ |

**Prisma 示例：**

### 6.4 我的思考：为什么我选 Prisma

在实际项目中，我更倾向于使用 Prisma，原因是：

1. **类型安全是刚需**

   在 TypeScript 项目中，Prisma 的类型推断可以让我在写代码时就发现错误，而不是等到运行时。

2. **声明式迁移更可靠**

   Prisma 的迁移是基于 schema.prisma 文件的声明式迁移，状态清晰，不容易出错。

3. **性能足够好**

   虽然原生 SQL 更快，但 Prisma 的性能在大多数场景下都够用。只有在极端性能要求的场景下，才需要手写 SQL。

4. **团队协作更简单**

   有了 schema.prisma 文件，新成员可以快速了解数据库结构，而不需要看一堆迁移文件。

5. **现代化设计**

   Prisma 的设计理念更现代化，它的 Client 非常轻量，适合 Serverless 场景。

## 第七章：实战选型决策

### 7.1 社交应用：为什么选 PG + Redis

社交应用是典型的读多写多、数据关联复杂的场景。我曾参与过一个社交平台的开发，我们的选择是 PostgreSQL + Redis。

**数据特点分析：**

1. **用户数据**：需要强一致性，用户信息必须准确
2. **社交关系**：复杂的关联查询，朋友的朋友、关注、粉丝
3. **Feed 流**：海量数据，需要高效分页
4. **实时通知**：低延迟，必须快

**PostgreSQL 存储：**

```sql
-- 用户表
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    bio TEXT,
    avatar_url VARCHAR(500),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 关注关系（邻接表 + 路径存储）
CREATE TABLE follows (
    follower_id BIGINT REFERENCES users(id) ON DELETE CASCADE,
    following_id BIGINT REFERENCES users(id) ON DELETE CASCADE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (follower_id, following_id)
);

-- 创建索引加速查询
CREATE INDEX idx_follows_following ON follows(following_id);
CREATE INDEX idx_follows_follower ON follows(follower_id);

-- 帖子表
CREATE TABLE posts (
    id BIGSERIAL PRIMARY KEY,
    author_id BIGINT REFERENCES users(id) ON DELETE CASCADE,
    content TEXT NOT NULL,
    image_urls TEXT[],
    likes_count INT DEFAULT 0,
    comments_count INT DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_posts_author ON posts(author_id);
CREATE INDEX idx_posts_created ON posts(created_at DESC);
```

**Redis 存储：**

```typescript
// 用户会话
async function createSession(userId: number) {
    const sessionId = crypto.randomUUID();
    await redis.setex(
        `session:${sessionId}`,
        7 * 24 * 60 * 60, // 7天过期
        JSON.stringify({ userId, createdAt: Date.now() })
    );
    return sessionId;
}

// 关注列表缓存（ZSet，按时间排序）
async function followUser(followerId: number, followingId: number) {
    // 写入 Redis 关注列表
    await redis.zadd(
        `following:${followerId}`,
        Date.now(),
        followingId.toString()
    );

    // 增加粉丝计数
    await redis.hincrby(`user:${followingId}`, 'followers_count', 1);
    await redis.hincrby(`user:${followerId}`, 'following_count', 1);

    // 同步写入 PostgreSQL
    await prisma.follows.create({
        data: { followerId, followingId }
    });
}

// Feed 缓存（分页）
async function getFeed(userId: number, page: number, pageSize: number) {
    const cacheKey = `feed:${userId}:${page}`;

    // 先查缓存
    const cached = await redis.get(cacheKey);
    if (cached) {
        return JSON.parse(cached);
    }

    // 缓存未命中，查数据库
    const start = (page - 1) * pageSize;
    const feed = await prisma.posts.findMany({
        where: {
            authorId: {
                in: await getFollowingIds(userId) // 获取关注列表
            }
        },
        orderBy: { createdAt: 'desc' },
        skip: start,
        take: pageSize
    });

    // 写入缓存
    await redis.setex(cacheKey, 300, JSON.stringify(feed));

    return feed;
}
```

**为什么不用 MongoDB：**

社交关系的数据结构其实很规整——用户、关注、帖子，它们之间的关联是固定的。MongoDB 的灵活性在这种情况下反而是劣势，因为：

1. 关联查询在 MongoDB 中很痛苦
2. PostgreSQL 的 JOIN 性能远胜 MongoDB 的 `$lookup`
3. 社交图谱查询（朋友的朋友）用 PostgreSQL 的递归 CTE 更简单

### 7.2 电商系统：为什么选 MySQL + MongoDB

电商系统是一个混合场景，核心交易需要强一致性，但商品信息灵活多变。我们的选择是 MySQL + MongoDB。

**MySQL 存储（核心交易）：**

### 7.3 日志系统：为什么选 Elasticsearch

### 8.1 为什么要用数据库即服务

数据库即服务（DBaaS，Database as a Service）是指由云服务商托管的数据库服务，开发者不需要关心底层的服务器、网络、存储等基础设施。

| **安全补丁** | 需要手动更新 | 自动更新 |

### 8.2 备份策略：容灾不是可选的

数据是企业最重要的资产，备份是最后一道防线。

**备份策略三二一原则：**

```
备份策略三二一原则：
├── 3份数据副本
│   ├── 主数据
│   ├── 本地备份
│   └── 异地备份
├── 2种不同备份介质
│   ├── 磁盘备份（快速恢复）
│   └── 对象存储/磁带备份（长期保存）
└── 1份异地副本
    └── 至少在另一个数据中心或区域
```

**PostgreSQL 备份方案：**

```bash
# 1. 物理备份（pg_basebackup）
pg_basebackup -h localhost -U repl -D /backup/base -P -X stream -c fast

# 2. 逻辑备份（pg_dump）
pg_dump -Fc mydb > mydb.dump

# 3. 定时备份脚本
#!/bin/bash
BACKUP_DIR="/backup/postgresql"
DATE=$(date +%Y%m%d_%H%M%S)
DATABASE="mydb"

# 创建备份
pg_dump -Fc $DATABASE > $BACKUP_DIR/$DATABASE\_$DATE.dump

# 压缩备份
gzip $BACKUP_DIR/$DATABASE\_$DATE.dump

# 上传到对象存储
aws s3 cp $BACKUP_DIR/$DATABASE\_$DATE.dump.gz s3://my-bucket/backups/

# 删除7天前的本地备份
find $BACKUP_DIR -name "*.dump.gz" -mtime +7 -delete
```

**MySQL 备份方案：**

```bash
# 1. 逻辑备份（mysqldump）

# 2. 物理备份（xtrabackup）

# 3. 定时备份脚本
```

### 8.4 我的思考：数据库是应用的核心

作为一个全栈工程师，我深刻理解数据库的重要性。**数据库是应用的核心，所有业务逻辑最终都会落到数据上**。

**数据库运维的核心原则：**

1. **数据优先**

   任何时候，都要确保数据的安全。备份、容灾、多副本，这些投入永远不要省。

2. **监控先行**

   上线前就要部署监控，而不是等出问题再补救。慢查询、性能指标、连接数，这些都要实时监控。

3. **文档为王**

   数据库设计文档、ER 图、变更记录，这些文档是团队协作的基础。代码可以重写，数据很难挽回。

4. **变更谨慎**

   数据库变更的风险远高于代码变更。任何 DDL 操作都要经过评审，重要的变更要有回滚方案。

5. **定期检视**

   每季度检视一次数据库设计，检查是否有优化空间。索引是否合理、查询是否需要优化、容量是否足够。

## 第九章：新数据库的诱惑

### 9.1 为什么要用 NewSQL

NewSQL 是一类新兴的分布式关系型数据库，它们既有 NoSQL 的扩展性，又有传统 SQL 数据库的事务支持。

**NewSQL 的背景：**

2000 年代，互联网公司发现关系型数据库无法满足海量数据的扩展需求，于是 NoSQL 运动兴起。但 NoSQL 放弃了 SQL 和事务，导致很多业务场景无法支持。NewSQL 就是在这样的背景下诞生的。

**NewSQL 的代表产品：**

- **TiDB**：PingCAP 开源的分布式数据库，兼容 MySQL 协议
- **CockroachDB**：Google Spanner 的开源实现
- **YugaByteDB**：支持 PostgreSQL 和 Cassandra API
- **ClustrixDB**：分布式 MySQL 兼容数据库

**TiDB vs 传统 MySQL：**

```sql
-- TiDB 分布式特性：对用户透明
-- 自动分片
SELECT * FROM orders WHERE user_id = 123;
-- TiDB 自动将查询路由到正确的分片

-- 水平扩展：在线添加节点
-- 无需修改代码
ALTER TABLE orders SHARD_ROW_ID_BITS = 4;

-- 跨分片 JOIN
-- TiDB 优化器自动处理
SELECT u.name, COUNT(o.id) as order_count
FROM users u
JOIN orders o ON u.id = o.user_id
GROUP BY u.id;
```

### 9.2 TiDB、CockroachDB 适合什么

**TiDB 适用场景：**

1. **大数据量 OLTP**

   数据量在 TB 级别，需要水平扩展的传统 OLTP 场景。

2. **HTAP 混合负载**

   同时有 OLTP 和 OLAP 需求，不想维护两套系统。

3. **MySQL 迁移**

   现有 MySQL 数据量太大，想迁移到分布式数据库。

4. **业务高速增长**

   业务增长快，需要数据库能够快速水平扩展。

**CockroachDB 适用场景：**

1. **多地域部署**

   需要在全球多个地域保持强一致性，比如跨境电商、游戏服。

2. **高可用优先**

   对可用性要求极高，不能容忍任何停机时间。

3. **强一致性优先**

   需要分布式事务支持，不能接受数据冲突。

**典型案例：**

```yaml
# 某电商平台的选型历程

# 初期（数据量 < 100GB）
MySQL 单机 -> 足够了

# 成长期（100GB < 数据量 < 1TB）
MySQL 主从 + Redis 缓存 -> 加缓存、加从库

# 规模期（1TB < 数据量 < 10TB）
分库分表 -> 开始遇到瓶颈
痛点：
- 分库分表规则固定，扩容困难
- 跨库查询需要应用层处理
- 维护复杂度指数增长

# 转型期（数据量 > 10TB）
TiDB 迁移：
- 对应用透明，不需要改代码
- 自动分片、水平扩展
- OLTP 和 OLAP 统一平台
```

### 9.3 时序数据库、图数据库

**时序数据库（Time Series Database）：**

时序数据库专门用于存储和时间相关的数据，比如监控指标、传感器数据、金融行情。

```yaml
代表产品：
- InfluxDB：最流行的时序数据库
- TimescaleDB：PostgreSQL 扩展，时序数据
- Prometheus：监控系统，自带时序存储
- TDengine：国产时序数据库

时序数据特点：
- 按时间顺序写入
- 很少更新或删除
- 通常需要聚合查询
- 数据量大，压缩率高
```

```sql
-- TimescaleDB 示例
CREATE TABLE sensor_data (
    time TIMESTAMP NOT NULL,
    sensor_id INTEGER,
    temperature DOUBLEPRECISION,
    humidity DOUBLEPRECISION
);

SELECT create_hypertable('sensor_data', 'time');

-- 时间序列聚合查询
SELECT time_bucket('1 hour', time) AS hour,
       sensor_id,
       AVG(temperature) AS avg_temp,
       MAX(temperature) AS max_temp,
       MIN(temperature) AS min_temp
FROM sensor_data
WHERE time >= NOW() - INTERVAL '24 hours'
GROUP BY hour, sensor_id
ORDER BY hour DESC;
```

**图数据库（Graph Database）：**

图数据库专门存储实体之间的关系，比如社交网络、推荐系统、知识图谱。

```yaml
代表产品：
- Neo4j：最流行的图数据库
- Apache AGE：PostgreSQL 扩展
- Amazon Neptune：AWS 图数据库服务
- NebulaGraph：国产图数据库

图数据特点：
- 节点和边
- 高效的关系查询
- 路径发现
- 社区检测
```

```cypher
// Neo4j 示例（Cypher 查询语言）
// 创建社交网络
CREATE (alice:User {name: 'Alice', age: 28})
CREATE (bob:User {name: 'Bob', age: 30})
CREATE (carol:User {name: 'Carol', age: 25})

MATCH path = shortestPath((alice:User {name: 'Alice'})-[*]-(carol:User {name: 'Carol'}))
```

### 9.4 我的思考：不要为了新而新

NewSQL、时序数据库、图数据库，这些新技术都很好，但**不要为了新而新**。

**我的原则：**

1. **先用传统方案解决问题**

   MySQL/PostgreSQL 能解决 90% 的问题。在尝试新技术之前，先确认传统方案真的不行。

2. **新技术的学习成本很高**

   不仅是数据库本身，还有运维工具、性能调优、故障排查，这些都需要时间和经验。

3. **新技术往往更复杂**

   分布式系统比单机系统复杂一个数量级。CAP 理论、分布式事务、一致性协议，这些都不是小事。

4. **生态系统不够成熟**

   新技术的文档、工具、社区支持都不如成熟产品。遇到问题可能找不到解决方案。

5. **人才稀缺**

   招聘熟悉新技术的人才比熟悉传统技术更难、更贵。

**新技术适用的标志：**

```
新技术适用评估：

□ 业务确实遇到了瓶颈（不是想象出来的）
□ 现有方案确实无法解决（经过验证）
□ 团队有能力驾驭新技术（有学习意愿和时间）
□ 可以承受试错成本（有回滚方案）
□ 长期收益大于短期投入（不是一时兴起）
```

数据库是全栈工程师的核心能力之一。一个好的数据库设计，可以让系统稳定运行多年；一个坏的数据库设计，可能让团队疲于奔命。
