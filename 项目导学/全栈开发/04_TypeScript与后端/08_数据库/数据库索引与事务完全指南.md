## 一、数据库索引深入解析

数据库索引是提升查询性能的核心技术，理解索引的底层数据结构和工作机制是进行数据库优化的基础。本章节将深入分析各种索引类型的原理、适用场景和使用技巧。

### 1.1 索引数据结构详解

索引的本质是一种特殊的数据结构，其核心作用是加速数据检索。不同的数据结构决定了不同的查询效率和应用场景，我们需要根据实际的业务需求选择合适的索引类型。

#### 1.1.1 B-Tree索引原理

B-Tree（Balanced Tree，平衡树）是数据库中最常用的索引数据结构，其设计目标是在大规模数据场景下提供高效的查找、插入和删除操作。B-Tree通过保持树的平衡性，确保从根节点到任意叶子节点的路径长度相同，从而保证查询时间复杂度稳定在O(log n)。

B-Tree的每个节点可以包含多个键值和指针，这种设计使得每个节点可以存储更多的数据，减少了树的深度。在数据库中，B-Tree的节点通常与磁盘页大小对齐（如4KB或8KB），这样每次磁盘I/O可以读取整个节点，减少磁盘访问次数。

```
B-Tree结构示意图：

        [50, 100]
       /    |    \
   [20]   [70]   [120]
   /  \   /  \   /   \
 [10] [30] [60] [80] [110] [150]
```

B-Tree的查找过程从根节点开始，首先在节点内部通过二分查找定位键值所在的区间，然后根据指针进入子节点继续查找，直到到达叶子节点。叶子节点包含实际的数据行指针或数据本身。

B-Tree的插入操作需要找到合适的叶子节点，如果节点未满则直接插入，如果节点已满则需要分裂节点并向上传递，可能导致树的高度增加。为了保持树的平衡性，B-Tree采用自底向上的方式调整，而不是像AVL树那样需要旋转操作。

B-Tree的删除操作相对复杂，如果删除后节点键值数量过少，需要进行节点合并或借用相邻节点的键值来维持平衡。这些维护操作会增加写入的开销，这也是为什么大量写入操作可能导致索引性能下降的原因。

**B-Tree的关键特性：**

- 键值有序存储，支持范围查询和排序
- 所有叶子节点深度相同，查询时间稳定
- 节点分裂和合并代价较高，但频率可控
- 适合等值查询和范围查询，是大多数场景的最佳选择

#### 1.1.2 B+Tree索引原理

B+Tree是B-Tree的变体，在数据库存储引擎中得到广泛应用，尤其是InnoDB和PostgreSQL默认采用B+Tree作为索引结构。B+Tree与B-Tree的核心区别在于：所有数据都存储在叶子节点，非叶子节点只存储索引键值和指针。

```
B+Tree结构示意图：

        [50, 100]
       /    |    \
   [50]   [100]  [150]  <- 非叶子节点只存索引
    |      |      |
   ...    ...    ...    <- 叶子节点包含所有数据
   [10] [30] [60] [80] [110] [150] [180]
    |    |    |    |    |    |    |
   行   行   行   行   行   行   行   <- 叶子节点包含完整数据
```

B+Tree的叶子节点通过双向链表连接（部分实现使用单向链表），这一特性使得范围查询非常高效。查询一个范围的数据时，首先定位起始位置，然后沿着链表顺序遍历即可，无需回溯到父节点。

B+Tree的非叶子节点不存储数据，只存储索引，这意味着非叶子节点可以容纳更多的索引键值。在相同数据量下，B+Tree的树高通常比B-Tree更小，这意味着查询需要访问更少的节点，磁盘I/O次数更少。

**B+Tree的核心优势：**

- 叶子节点链表连接，范围查询只需顺序扫描叶子节点
- 非叶子节点不存储数据，相同内存可容纳更多索引项
- 查询稳定性更强，所有查询都需要到达叶子节点
- 适合大规模数据检索，是主流数据库的首选

#### 1.1.3 Hash索引原理

Hash索引使用哈希表作为底层数据结构，通过哈希函数计算键值的哈希值，然后根据哈希值直接定位数据位置。这种设计使得Hash索引在等值查询时具有O(1)的极致性能，理论上是最快的索引类型。

Hash索引的插入过程：首先计算键值的哈希值，然后根据哈希值找到对应的桶（bucket），将键值添加到桶中。如果发生哈希冲突（多个键值映射到相同的哈希值），通常采用链地址法或开放地址法解决。

Hash索引的查找过程：给定查询键值，计算其哈希值，定位到对应的桶，然后在桶中顺序查找目标键值。理想情况下查找复杂度为O(1)，但如果哈希冲突严重，最坏情况可能退化为O(n)。

```
Hash索引结构示意图：

键值        哈希函数        桶数组
"Alice"  ->  hash() ->  3  ->  桶3: ["Alice" -> 行1]
"Bob"    ->  hash() ->  7  ->  桶7: ["Bob" -> 行2]
"Charlie"->  hash() ->  3  ->  桶3: ["Alice" -> 行1] -> ["Charlie" -> 行3]
```

Hash索引的主要限制在于其适用范围有限。由于哈希值不保持键值的顺序关系，Hash索引无法支持范围查询、排序操作和最左前缀匹配。这些限制使得Hash索引主要用于等值查询场景，如缓存键、唯一约束等。

另外，Hash索引在处理大量哈希冲突时性能会明显下降。当桶的链表过长时，查找可能需要遍历多个元素。一些高级实现（如PostgreSQL的Hash索引）使用了逻辑分桶和动态扩容来优化性能。

**Hash索引的适用场景：**

- 等值查询为主的场景，如用户ID查询、订单号查询
- 需要唯一性约束的列
- 内存缓存层的快速查找
- 不适合范围查询、排序和前缀匹配场景

### 1.2 索引类型分类

数据库索引根据其用途和约束可以分为多种类型，每种类型都有特定的适用场景和创建方法。理解这些索引类型的区别对于设计高效的数据库 schema 至关重要。

#### 1.2.1 主键索引

主键索引是最重要的索引类型，每个表只能有一个主键索引。主键索引要求所有键值唯一且不能为NULL，其索引列自动创建唯一性约束和NOT NULL约束。在InnoDB中，主键索引的叶子节点存储完整的数据行（称为聚簇索引），因此主键选择对表性能有重大影响。

选择主键应考虑以下原则：主键值应保持稳定且不可变，避免使用业务相关的列作为主键；主键应尽可能紧凑，减少索引存储空间；优先选择自增整数或UUID等有序值，避免随机插入导致的页分裂。

```sql
-- 创建主键索引的两种方式
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,  -- 自增主键，推荐使用
    name VARCHAR(100),
    email VARCHAR(255)
);

-- 复合主键示例
CREATE TABLE order_items (
    order_id BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    quantity INT,
    PRIMARY KEY (order_id, product_id)  -- 复合主键
);
```

在PostgreSQL中，主键默认创建唯一性B+Tree索引。对于复合主键，索引键按照列定义的顺序组织，查询时需要满足最左前缀原则才能有效使用索引。

#### 1.2.2 唯一索引

唯一索引要求索引列的所有值唯一，但允许存在NULL值（具体取决于数据库实现，PostgreSQL允许多个NULL值）。唯一索引不仅可以保证数据完整性，还能像普通索引一样加速唯一性查询。

```sql
-- 创建唯一索引
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- 在现有表上创建唯一索引
CREATE UNIQUE INDEX idx_orders_order_no ON orders(order_no);

-- 复合唯一索引
CREATE UNIQUE INDEX idx_user_role ON user_roles(user_id, role_id);
```

唯一索引的实现机制与主键索引类似，通常也是使用B+Tree结构，但叶子节点存储的是索引键值和指向数据行的指针，而非完整的数据行（除非表没有主键，此时使用唯一索引作为聚簇索引的依据）。

唯一索引与主键索引的区别：一张表可以有多个唯一索引；唯一索引允许NULL值（InnoDB允许多个NULL，SQL Server只允许一个NULL）；唯一索引的键值可以是可变的数据类型。

#### 1.2.3 普通索引

普通索引是最基本的索引类型，没有任何约束条件，仅用于加速查询。普通索引允许重复值和NULL值，创建和使用方式最为灵活。

```sql
-- 创建普通索引的多种方式
CREATE INDEX idx_users_name ON users(name);

-- 多列索引
CREATE INDEX idx_orders_customer_date ON orders(customer_id, created_at);

-- 表达式索引
CREATE INDEX idx_users_email_lower ON users(LOWER(email));

-- 部分索引（PostgreSQL特性）
CREATE INDEX idx_users_active ON users(name) WHERE is_active = true;
```

普通索引的创建应遵循以下原则：选择区分度高的列（不同值越多越好）；避免在频繁更新的列上创建索引；考虑查询条件的模式，确保索引能够被有效利用。

#### 1.2.4 联合索引与最左前缀原则

联合索引（Composite Index）是由多个列组成的索引，其数据结构按照索引列的顺序依次排列。联合索引的键值是多个列值的组合，存储时先按第一列排序，第一列相同时按第二列排序，依次类推。

```
联合索引 (a, b, c) 的数据结构：

索引键值              物理位置
(1, 1, 1)    ->    数据行1
(1, 1, 2)    ->    数据行2
(1, 2, 1)    ->    数据行3
(2, 1, 1)    ->    数据行4
(2, 3, 2)    ->    数据行5
```

最左前缀原则（Leftmost Prefix Principle）是理解联合索引的核心概念。查询条件必须从索引的最左列开始，并且是连续的情况下，才能有效使用联合索引。这是因为索引的排序是按照列的顺序依次进行的。

**最左前缀原则的具体规则：**

- 查询条件必须包含索引的最左列才能使用索引
- 查询条件必须使用索引的连续列，中间不能跳过
- 如果查询条件只使用索引的后面列，则无法使用索引

```sql
-- 创建联合索引
CREATE INDEX idx_orders_composite ON orders(customer_id, status, created_at);

-- 以下查询可以使用索引
SELECT * FROM orders WHERE customer_id = 100;                    -- 使用a列
SELECT * FROM orders WHERE customer_id = 100 AND status = 'paid'; -- 使用a、b列
SELECT * FROM orders WHERE customer_id = 100 AND status = 'paid' AND created_at > '2024-01-01'; -- 使用全部列

-- 以下查询无法使用索引
SELECT * FROM orders WHERE status = 'paid';                       -- 跳过a列
SELECT * FROM orders WHERE customer_id = 100 AND created_at > '2024-01-01'; -- 跳过b列（中断）
```

理解最左前缀原则对于设计高效的联合索引至关重要。设计索引时，应将区分度高的列或经常作为查询条件的列放在前面。同时，应分析实际查询模式，将经常一起查询的列放在索引中相邻的位置。

#### 1.2.5 覆盖索引与索引下推

覆盖索引（Covering Index）是一种特殊的索引使用方式，当查询所需的所有列都包含在索引中时，数据库无需回表查询数据行，直接从索引中获取所需数据，从而大幅提升查询性能。

```sql
-- 创建覆盖索引
CREATE INDEX idx_orders_covering ON orders(customer_id, status, created_at)
INCLUDE (total_amount, discount);  -- PostgreSQL语法

-- 对于MySQL，可以使用索引覆盖
CREATE INDEX idx_orders_covering ON orders(customer_id, status, created_at, total_amount, discount);

-- 查询示例
SELECT customer_id, status, created_at, total_amount
FROM orders
WHERE customer_id = 100 AND status = 'paid';
-- 此查询可以完全通过索引满足，无需回表
```

索引下推（Index Condition Pushdown，ICP）是MySQL使用索引的一种优化技术。在不支持ICP的情况下，使用索引定位到数据行后，数据库需要在服务器层过滤其他条件；而启用ICP后，数据库可以将过滤条件下推到存储引擎层，在索引遍历过程中就完成过滤，减少回表次数。

```sql
-- 索引下推示例
CREATE INDEX idx_users_name_age ON users(name, age);

-- 查询：找出所有名为"张三"且年龄大于25岁的用户
SELECT * FROM users WHERE name = '张三' AND age > 25;

-- 启用ICP时：
-- 1. 使用索引定位 name = '张三' 的所有记录
-- 2. 在索引遍历过程中同时检查 age > 25 条件
-- 3. 只将满足两个条件的记录回表获取完整数据

-- 未启用ICP时：
-- 1. 使用索引定位 name = '张三' 的所有记录
-- 2. 回表获取完整数据
-- 3. 在服务器层过滤 age > 25 条件
```

索引下推可以显著减少不必要的回表操作，特别是对于索引列上有多个过滤条件的查询。在MySQL 5.6及更高版本中，ICP默认启用，可以通过optimizer_switch选项控制。

### 1.3 索引失效分析与解决方案

索引虽然能加速查询，但在某些情况下数据库优化器可能选择不使用索引，导致查询性能下降。理解索引失效的原因是进行SQL优化的基础，也是面试中的高频考点。

#### 1.3.1 LIKE查询与索引失效

使用LIKE进行前缀匹配时，如果模式串以通配符开头，索引将无法使用。这是因为索引按照键值的自然顺序排列，而通配符开头无法确定前缀的起始位置。

```sql
-- 创建索引
CREATE INDEX idx_users_email ON users(email);

-- 索引可用的LIKE查询（前缀匹配）
SELECT * FROM users WHERE email LIKE 'john%';   -- 可用索引
SELECT * FROM users WHERE email LIKE 'john_doe%'; -- 可用索引

-- 索引无法使用的LIKE查询（后缀或中间匹配）
SELECT * FROM users WHERE email LIKE '%john%';  -- 索引失效
SELECT * FROM users WHERE email LIKE '%john';   -- 索引失效
```

**解决方案：**

1. **反向索引技术**：如果确实需要支持后缀匹配，可以创建一个额外的反向列并建立索引

```sql
-- 反向索引解决方案
ALTER TABLE users ADD COLUMN email_reversed VARCHAR(255);
CREATE INDEX idx_users_email_rev ON users(email_reversed);

-- 查询时使用反向列
SELECT * FROM users WHERE email_reversed = REVERSE('%john'); -- 注意反向后的模式串
-- 对于 '%john'，反向后变为 'nhoj%'，可以使用前前缀匹配
```

2. **全文索引**：如果需要频繁进行模糊匹配，考虑使用全文索引

```sql
-- PostgreSQL全文搜索
CREATE INDEX idx_users_email_fts ON users USING gin(to_tsvector('english', email));

-- 查询
SELECT * FROM users WHERE to_tsvector('english', email) @@ to_tsquery('english', 'john');
```

3. **搜索引擎**：对于复杂的文本搜索需求，使用Elasticsearch等专业搜索引擎

#### 1.3.2 OR条件与索引失效

OR条件使用不当是导致索引失效的常见原因。当OR连接多个条件时，只有当每个条件都能使用索引时，数据库才会选择使用索引。如果其中任何一个条件无法使用索引，整个查询可能退化为全表扫描。

```sql
-- 创建索引
CREATE INDEX idx_users_age ON users(age);
CREATE INDEX idx_users_email ON users(email);

-- 索引失效的OR查询
SELECT * FROM users WHERE age = 25 OR email = 'john@example.com';
-- age列有索引，email列有索引，但由于OR的特性，优化器可能认为全表扫描更快

-- 解决方案：使用UNION替代OR
SELECT * FROM users WHERE age = 25
UNION ALL
SELECT * FROM users WHERE email = 'john@example.com' AND age != 25;
```

**OR索引失效的根本原因：** 当优化器估算OR查询的成本时，如果满足各条件的记录数较多，读取索引再回表的成本可能超过直接全表扫描的成本。优化器基于统计信息做出决策，在数据分布不均匀时可能做出错误选择。

**UNION替代OR的原理：** UNION会对每个子查询分别使用索引，最后合并结果。对于高选择性的条件，UNION通常比OR更高效。UNION ALL在确定子查询不重叠时可避免去重开销。

#### 1.3.3 函数与表达式与索引失效

在索引列上使用函数或表达式会导致索引失效，因为索引按照列值的原始形式排序，而函数操作改变了值的形态，无法直接利用索引的有序性。

```sql
-- 创建索引
CREATE INDEX idx_users_created ON users(created_at);

-- 索引失效的函数查询
SELECT * FROM users WHERE YEAR(created_at) = 2024;           -- 函数导致索引失效
SELECT * FROM users WHERE MONTH(created_at) = 3;              -- 函数导致索引失效
SELECT * FROM users WHERE DATE(created_at) = '2024-03-15';    -- 函数导致索引失效
SELECT * FROM users WHERE created_at + INTERVAL 1 DAY = '2024-03-16'; -- 表达式导致索引失效
```

**解决方案：**

1. **使用范围查询替代函数**

```sql
-- 改写为范围查询
SELECT * FROM users WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';
```

2. **创建表达式索引**

```sql
-- PostgreSQL表达式索引
CREATE INDEX idx_users_created_year ON users(DATE_PART('year', created_at));
CREATE INDEX idx_users_created_month ON users(DATE_PART('month', created_at));

-- MySQL表达式索引（5.7+支持Generated Column）
ALTER TABLE users ADD COLUMN created_year INT GENERATED ALWAYS AS (YEAR(created_at)) STORED;
CREATE INDEX idx_users_created_year ON users(created_year);
```

3. **使用函数索引（针对MySQL）**

```sql
-- MySQL 8.0+ 支持函数索引
CREATE INDEX idx_users_created_year ON users((YEAR(created_at)));
```

#### 1.3.4 隐式类型转换与索引失效

当查询条件中列的类型与传入值的类型不匹配时，数据库会进行隐式类型转换。这种转换可能导致索引失效，因为转换后的值可能无法直接利用索引的有序性。

```sql
-- created_at 为 TIMESTAMP 类型
-- 索引: CREATE INDEX idx_orders_created ON orders(created_at);

-- 隐式转换示例
SELECT * FROM orders WHERE created_at = '2024-03-15';  -- 字符串转TIMESTAMP，索引可用
SELECT * FROM orders WHERE created_at = 1710500000;     -- 整数转TIMESTAMP，索引可能失效

-- user_id 为 BIGINT 类型
SELECT * FROM users WHERE user_id = '12345';  -- 字符串转整数，可能触发全表扫描
```

**隐式转换导致索引失效的原理：** 当将字符串转换为数字类型时，数据库需要对字符串的每个字符进行计算才能比较，这个过程无法利用索引的有序性。因此优化器可能选择全表扫描。

**最佳实践：** 始终确保查询条件的数据类型与列定义一致；使用显式类型转换使代码更清晰；在应用层进行类型验证，避免隐式转换。

#### 1.3.5 其他索引失效场景

1. **使用IS NULL或IS NOT NULL**

```sql
-- IS NULL通常无法使用索引（取决于数据库和索引设计）
SELECT * FROM users WHERE email IS NULL;    -- 可能无法使用索引
SELECT * FROM users WHERE email IS NOT NULL; -- 通常无法使用索引

-- 解决方案：创建包含NULL值的复合索引
CREATE INDEX idx_users_email_null ON users(email) WHERE email IS NOT NULL;
```

2. **使用NOT IN和NOT EXISTS**

```sql
-- NOT IN通常导致索引失效
SELECT * FROM users WHERE id NOT IN (SELECT user_id FROM orders);

-- 替代方案：使用LEFT JOIN + IS NULL
SELECT u.* FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.user_id IS NULL;
```

3. **数据量过小导致全表扫描更快**

当表的数据量非常小（如几十行）时，读取索引再回表的成本可能高于直接读取数据页。优化器会自动选择全表扫描，这是正常的优化行为，不需要特别处理。

### 1.4 EXPLAIN执行计划分析

EXPLAIN命令是分析SQL查询性能的核心工具，通过查看执行计划，我们可以了解数据库如何执行查询、是否使用索引、查询的成本估算等信息。熟练掌握EXPLAIN的分析方法是进行SQL优化的必备技能。

#### 1.4.1 EXPLAIN输出详解

```sql
-- PostgreSQL EXPLAIN
EXPLAIN ANALYZE SELECT * FROM orders
WHERE customer_id = 100 AND status = 'paid'
ORDER BY created_at DESC;

-- 输出示例
Limit  (cost=0.43..155.43 rows=100 width=143)
  ->  Index Scan Backward using idx_orders_customer_status_date on orders
        (cost=0.43..155.43 rows=100 width=143)
        (actual time=0.042..0.089 rows=50 loops=1)
        Index Cond: ((customer_id = 100) AND ((status)::text = 'paid'::text))
        Filter: (created_at IS NOT NULL)
        Rows Removed by Filter: 0
Planning Time: 0.234 ms
Execution Time: 0.112 ms
```

**关键指标解释：**

- **cost=开始..结束**：优化器估算的查询成本
  - 第一个值：启动成本（返回第一行之前的成本）
  - 第二个值：总成本（返回所有行的成本）
  - width：每行的平均字节数估算

- **actual time=开始..结束**：实际执行时间（毫秒）
  - 第一个值：获取第一行的实际时间
  - 第二个值：获取所有行的实际时间

- **rows=估算行数**：优化器估算的返回行数
- **loops=循环次数**：该节点被执行的次数

#### 1.4.2 常见执行计划类型

| 类型 | 说明 | 性能特征 |
|------|------|----------|
| **Seq Scan** | 全表顺序扫描 | 大表全扫性能差，应尽量避免 |
| **Index Scan** | 索引扫描 + 回表 | 正常的选择，索引覆盖时性能好 |
| **Index Only Scan** | 仅索引扫描，无需回表 | 最优，索引完全覆盖查询 |
| **Bitmap Index Scan** | 位图索引扫描 | 适合多条件组合，使用位图合并 |
| **Nested Loop** | 嵌套循环连接 | 小表驱动大表时效率高 |
| **Hash Join** | 哈希连接 | 大表连接常用，需内存构建哈希表 |
| **Merge Join** | 归并连接 | 要求输入已排序，适用有序数据 |

## 二、事务与并发控制深入理解

事务是数据库管理系统执行过程中的一个逻辑单元，由一系列操作组成。事务的ACID特性确保了数据的一致性和可靠性，而并发控制机制则保证了多个事务同时执行时不会相互干扰。本章节将深入分析事务的底层原理和并发控制技术。

### 2.1 ACID特性详解

ACID是事务的四个基本特性的缩写：原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）和持久性（Durability）。理解这四个特性的底层实现机制对于设计可靠的数据系统至关重要。

#### 2.1.1 原子性实现原理

原子性确保事务中的所有操作要么全部成功，要么全部失败回滚，不存在中间状态。这一特性是事务最核心的特性，其他三个特性都建立在原子性的基础之上。

**PostgreSQL原子性实现：**

PostgreSQL通过MVCC（多版本并发控制）和WAL（预写式日志）实现原子性。当事务开始时，数据库会分配一个唯一的事务ID（xmin），所有在事务中修改的数据都会记录这个事务ID。事务提交时，所有修改通过WAL写入磁盘；事务回滚时，只需标记事务状态为_aborted_，无需实际撤销已写入的修改。

```sql
-- 事务提交示例
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- 扣款
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- 收款
COMMIT;  -- 原子性保证：要么全部成功，要么全部回滚

-- 事务回滚示例
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- 发生错误
ROLLBACK;  -- 所有修改被撤销，balance恢复原值
```

**原子性的代价：** 为了实现原子性，数据库需要记录事务修改前的状态（回滚段）和修改后的状态（WAL）。这意味着事务执行需要更多的存储空间和I/O操作。长时间运行的事务会阻止垃圾回收，影响数据库性能。

#### 2.1.2 一致性维护机制

一致性确保事务执行前后数据库始终处于一致状态，这种一致性包括数据库定义的约束（如主键唯一、外键关联）和业务规则（如账户余额不能为负数）。

```sql
-- 约束维护一致性
CREATE TABLE accounts (
    id SERIAL PRIMARY KEY,
    balance DECIMAL(15,2) NOT NULL CHECK (balance >= 0),  -- 业务约束
    user_id INT REFERENCES users(id)                       -- 外键约束
);

-- 事务中的约束检查
BEGIN;
UPDATE accounts SET balance = balance - 500 WHERE id = 1;  -- 如果余额不足，CHECK约束失败
COMMIT;  -- 事务失败，数据库保持一致状态
```

**一致性与隔离性的关系：** 一致性是目标，隔离性是手段。在不同隔离级别下，事务执行过程中可能看到不一致的中间状态，但事务提交后必须保证数据库达到一致状态。隔离级别越高，一致性保证越强，但并发性能越差。

#### 2.1.3 隔离级别与并发问题

SQL标准定义了四个隔离级别，从低到高依次为：读未提交（Read Uncommitted）、读已提交（Read Committed）、可重复读（Repeatable Read）和串行化（Serializable）。每个隔离级别都解决了特定的并发问题，但同时也带来不同的性能开销。

**脏读（Dirty Read）：** 指一个事务读取了另一个事务未提交的数据。如果另一个事务最终回滚，读取的数据就是无效的。脏读发生在隔离级别最低的读未提交级别。

```sql
-- 脏读示例
-- 事务A
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- 此时还未COMMIT

-- 事务B（设置隔离级别为READ UNCOMMITTED）
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT balance FROM accounts WHERE id = 1;  -- 可能读到未提交的脏数据
-- 如果事务A最终回滚，事务B读取的就是无效数据
```

**不可重复读（Non-repeatable Read）：** 指在同一事务中，两次读取同一行数据可能得到不同的结果，因为另一个事务在两次读取之间修改并提交了这行数据。

```sql
-- 不可重复读示例
-- 事务A
BEGIN;
SELECT balance FROM accounts WHERE id = 1;  -- 第一次读取：balance = 1000

-- 事务B（在事务A执行过程中）
BEGIN;
UPDATE accounts SET balance = balance - 500 WHERE id = 1;
COMMIT;  -- 提交修改

-- 事务A
SELECT balance FROM accounts WHERE id = 1;  -- 第二次读取：balance = 500
-- 同一事务中，两次读取结果不一致
```

**幻读（Phantom Read）：** 指在同一事务中，两次执行相同条件的查询可能得到不同的行数，因为另一个事务插入了或删除了满足条件的记录。

```sql
-- 幻读示例
-- 事务A
BEGIN;
SELECT COUNT(*) FROM orders WHERE status = 'pending';  -- 第一次：100条

-- 事务B（在事务A执行过程中）
BEGIN;
INSERT INTO orders (status) VALUES ('pending');  -- 插入一条新记录
COMMIT;

-- 事务A
SELECT COUNT(*) FROM orders WHERE status = 'pending';  -- 第二次：101条
-- 同一事务中，相同条件的查询结果行数不同
```

**各隔离级别解决的问题对照表：**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|----------|------|------------|------|
| 读未提交 | 可能 | 可能 | 可能 |
| 读已提交 | 不可能 | 可能 | 可能 |
| 可重复读 | 不可能 | 不可能 | 可能（PostgreSQL不可能） |
| 串行化 | 不可能 | 不可能 | 不可能 |

#### 2.1.4 持久性保障机制

持久性确保一旦事务提交，其对数据库的所有修改都必须永久保存，即使系统发生故障（如断电、崩溃）也不能丢失。持久性是数据库最核心的承诺，也是最难实现的特性。

**WAL机制：** 预写式日志（Write-Ahead Logging）是实现持久性的核心技术。在将数据页修改写入磁盘之前，必须先将修改记录写入日志文件。如果系统崩溃，数据库可以通过重放日志恢复已提交的事务。

```sql
-- PostgreSQL的WAL配置参数
-- wal_level: minimal, replica, logical
-- fsync: 是否同步写入磁盘（开启后性能下降但保证持久性）
-- synchronous_commit: 开启后等待WAL写入磁盘才返回提交成功

-- 检查WAL状态
SHOW wal_level;
SHOW fsync;
SHOW synchronous_commit;

-- 查看WAL写入统计
SELECT * FROM pg_stat_database WHERE datname = 'mydb';
```

**持久性的代价：** 为了保证持久性，每次事务提交都需要等待日志写入磁盘完成。磁盘I/O是数据库最耗时的操作之一，这就是为什么高并发写入场景下，数据库性能会遇到瓶颈。可以通过以下方式优化：

- 使用电池备份的写入缓存（BBU）的RAID控制器
- 使用SSD替代机械硬盘
- 批量提交减少fsync频率
- 使用主从复制将数据异步复制到备库

### 2.2 MVCC多版本并发控制

MVCC（Multi-Version Concurrency Control）是现代数据库实现高并发访问的核心技术。通过维护数据的多个版本，MVCC允许读写操作互不阻塞，极大地提升了数据库的并发性能。

#### 2.2.1 MVCC基本原理

MVCC的核心思想是为每个读操作创建数据的一个快照（Snapshot），而不是读取最新版本的数据。这样，读操作不会阻塞写操作，写操作也不会阻塞读操作，实现真正的并发访问。

```sql
-- PostgreSQL的MVCC示例
-- 事务A（事务ID=100）
BEGIN;
SELECT balance FROM accounts WHERE id = 1;  -- balance = 1000

-- 事务B（事务ID=101）
BEGIN;
UPDATE accounts SET balance = 500 WHERE id = 1;
COMMIT;

-- 事务A（仍在事务ID=100的快照中）
SELECT balance FROM accounts WHERE id = 1;  -- balance = 1000（读到的仍是旧版本）
-- 事务A不知道自己之外发生了什么变化
```

**PostgreSQL的版本链：** 在PostgreSQL中，每行数据有两个隐藏列：xmin（创建这行版本的事务ID）和xmax（删除这行版本的事务ID，0表示未删除）。当行被更新时，旧版本通过指针形成版本链，新版本指向新行。读操作根据事务ID判断可见性。

```
版本链示意图：

事务101更新前的版本（xmax=101）  事务101更新后的版本（xmin=101）
         |                              |
         v                              v
    [行版本1]  ----------------->   [行版本2]
    xmin=100                         xmin=101
    xmax=101                         xmax=0
    balance=1000                     balance=500

读取时：
- 事务100：看到行版本1（xmin=100, xmax=null，未开始更新）
- 事务101：看到行版本2（xmin=101，更新后的版本）
```

#### 2.2.2 事务ID与可见性判断

PostgreSQL使用事务ID（Transaction ID，XID）来标识事务和判断版本可见性。事务ID是一个32位无符号整数，有限的事务ID通过"XID环绕"机制循环使用。

```sql
-- 查看当前事务ID
SELECT txid_current();

-- 查看数据库的事务ID统计
SELECT datname, age(datfrozenxid), txid_age(datfrozenxid)
FROM pg_database;

-- 设置freeze列表年龄阈值
ALTER TABLE accounts SET (autovacuum_freeze_table_age = 1000000);
```

**可见性判断规则：** 当读取一行数据时，数据库根据以下规则判断该版本是否对当前事务可见：

1. 如果版本的xmin状态为_aborted_，则不可见
2. 如果xmin在当前事务之后启动，则不可见
3. 如果xmin未提交，则不可见
4. 如果xmax不为0且已提交，则该版本已被删除，不可见
5. 满足以上条件后，版本可见

#### 2.2.3 VACUUM与垃圾回收

在MVCC机制下，更新操作不会原地修改数据，而是创建新版本。这意味着旧版本会逐渐积累，形成"垃圾"。VACUUM进程负责清理这些垃圾，释放磁盘空间。

```sql
-- 手动VACUUM
VACUUM accounts;

-- VACUUM并分析统计信息
VACUUM ANALYZE accounts;

-- 彻底清理（重写表，回收所有空间）
VACUUM FULL accounts;

-- 查看表的膨胀程度
SELECT relname, n_live_tup, n_dead_tup, last_vacuum, last_autovacuum
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY n_dead_tup DESC;
```

**autovacuum机制：** PostgreSQL配置了autovacuum进程，自动定期执行VACUUM。autovacuum的触发条件由以下参数控制：

```sql
-- autovacuum配置参数
ALTER SYSTEM SET autovacuum_naptime = '1min';        -- 检查间隔
ALTER SYSTEM SET autovacuum_vacuum_threshold = 50;  -- 死元组阈值
ALTER SYSTEM SET autovacuum_vacuum_scale_factor = 0.1; -- 死元组比例
```

### 2.3 锁机制深度解析

锁机制是实现并发控制的基础，通过对数据对象加锁，可以确保多个事务之间的操作互不干扰。PostgreSQL实现了多种类型的锁，每种锁都有不同的粒度和用途。

#### 2.3.1 锁的分类与层级

PostgreSQL的锁分为多个层级，从低到高依次为：行级锁、表级锁、页面锁、事务锁。行级锁粒度最细，冲突最少，但开销最大；表级锁粒度最粗，冲突可能最多，但开销最小。

```sql
-- 行级锁
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;  -- FOR UPDATE行锁
SELECT * FROM accounts WHERE id = 1 FOR SHARE;   -- FOR SHARE共享锁

-- 表级锁
LOCK TABLE accounts IN ACCESS SHARE MODE;   -- 访问共享锁（SELECT默认）
LOCK TABLE accounts IN ROW SHARE MODE;     -- 行共享锁（SELECT FOR UPDATE）
LOCK TABLE accounts IN ROW EXCLUSIVE MODE; -- 行排他锁（UPDATE/DELETE/INSERT默认）
LOCK TABLE accounts IN SHARE UPDATE MODE;  -- 共享更新锁（VACUUM/ALTER TABLE）
LOCK TABLE accounts IN SHARE MODE;         -- 共享锁（CREATE INDEX）
LOCK TABLE accounts IN SHARE ROW EXCLUSIVE MODE; -- 共享行排他锁
LOCK TABLE accounts IN EXCLUSIVE MODE;     -- 排他锁（ALTER TABLE/DROP TABLE）
LOCK TABLE accounts IN ACCESS EXCLUSIVE MODE;   -- 访问排他锁（最严格）
```

**锁的兼容矩阵：**

| 锁模式 | ACCESS SHARE | ROW SHARE | SHARE UPDATE | SHARE | SHARE ROW EXCLUSIVE | EXCLUSIVE | ACCESS EXCLUSIVE |
|--------|--------------|-----------|--------------|-------|--------------------|-----------|-----------------|
| ACCESS SHARE | ✓ | | | | | | |
| ROW SHARE | | ✓ | | | | | |
| SHARE UPDATE | | ✓ | ✓ | | | | |
| SHARE | | | | ✓ | | | |
| SHARE ROW EXCLUSIVE | | | | | ✓ | | |
| EXCLUSIVE | | | | | | ✓ | |
| ACCESS EXCLUSIVE | | | | | | | ✓ |

#### 2.3.2 临键锁与间隙锁

临键锁（Next-Key Lock）是PostgreSQL在可重复读隔离级别下防止幻读的关键机制。它结合了行锁和间隙锁，既锁定存在的行，也锁定行之间的间隙。

```sql
-- 临键锁示例
BEGIN;
SELECT * FROM orders WHERE id BETWEEN 100 AND 200 FOR UPDATE;
-- 锁定范围：[MIN, 100), [100, 101), [101, 102), ..., [200, MAX]
-- 任何在这个范围内插入新行的操作都会被阻塞
```

**间隙锁的作用：** 间隙锁确保在事务执行期间，其他事务无法在锁定范围内插入新行，从而防止幻读。间隙锁只锁定"间隙"，不锁定具体的行，因此不会阻塞其他事务对已有行的更新操作。

#### 2.3.3 死锁检测与处理

死锁是指两个或多个事务相互持有对方需要的锁，形成循环等待的情况。如果不加以处理，死锁会导致事务永远等待下去。数据库通过死锁检测机制自动解决死锁问题。

```sql
-- PostgreSQL死锁超时设置
SHOW deadlock_timeout;  -- 默认1秒
SET deadlock_timeout = '500ms';

-- 查看当前等待的锁
SELECT blocked_locks.pid AS blocked_pid,
       blocking_locks.pid AS blocking_pid,
       blocked_activity.query AS blocked_query,
       blocking_activity.query AS blocking_query
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_locks.pid = blocked_activity.pid
JOIN pg_catalog.pg_locks blocking_locks
    ON blocking_locks.lock = blocked_locks.block
    AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_locks.pid = blocking_activity.pid
WHERE NOT blocked_locks.granted;
```

**死锁解决策略：** 当检测到死锁时，数据库会选择一个"受害者"事务进行回滚，释放其持有的锁，让其他事务继续执行。被回滚的事务需要应用层重试。

**避免死锁的最佳实践：**

- 多个事务按照相同的顺序访问资源
- 减少事务持有锁的时间，尽快提交
- 使用较低隔离级别
- 在同一个事务中访问尽可能少的行

### 2.4 分布式事务概述

在微服务架构和分布式系统中，事务需要跨越多个数据库或服务。传统的两阶段提交（2PC）协议是实现分布式事务的经典方案，但存在性能开销大和单点故障风险。

#### 2.4.1 两阶段提交协议

两阶段提交（Two-Phase Commit，2PC）是一种分布式事务协议，分为准备阶段和提交阶段。在准备阶段，协调者向所有参与者发送准备请求，所有参与者确认可以提交后，进入提交阶段。

```
2PC协议流程：

准备阶段：
协调者 -> 参与者1：PREPARE
协调者 -> 参与者2：PREPARE
参与者1：锁定资源，准备提交，返回YES
参与者2：锁定资源，准备提交，返回YES

提交阶段：
协调者 -> 参与者1：COMMIT
协调者 -> 参与者2：COMMIT
参与者1：提交事务，释放锁
参与者2：提交事务，释放锁
```

**2PC的问题：**

- 协调者单点故障：如果协调者在提交阶段崩溃，参与者将永远锁定资源
- 同步阻塞：所有参与者在准备阶段锁定资源，直到提交完成
- 数据不一致风险：如果某些参与者收到COMMIT而其他没收到

#### 2.4.2 Saga模式

Saga模式是一种分布式事务的替代方案，将大事务拆分为多个本地事务，每个本地事务有对应的补偿事务。当某个步骤失败时，按照反向顺序执行补偿事务。

```typescript
// Saga模式示例（TypeScript）
class OrderSaga {
  async execute(orderId: string): Promise<void> {
    const steps = [
      { name: 'createOrder', compensate: 'cancelOrder' },
      { name: 'reserveInventory', compensate: 'releaseInventory' },
      { name: 'processPayment', compensate: 'refundPayment' },
      { name: 'sendNotification', compensate: 'revokeNotification' },
    ];

    const completedSteps: string[] = [];

    try {
      for (const step of steps) {
        await this.executeStep(step.name, orderId);
        completedSteps.push(step.name);
      }
    } catch (error) {
      // 回滚已完成的步骤
      for (const stepName of completedSteps.reverse()) {
        const step = steps.find(s => s.name === stepName);
        await this.executeCompensation(step.compensate, orderId);
      }
      throw error;
    }
  }
}
```

### 3.1 慢查询分析与优化流程

```sql
-- 启用pg_stat_statements
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
```

### 3.2 分页查询优化

分页查询是Web应用中最常见的场景之一，但深度分页（如查询第10000页）往往性能很差，因为数据库需要跳过大量的数据。

#### 3.2.1 深度分页问题分析

```sql
-- 典型的深度分页查询（低效）
SELECT * FROM orders
ORDER BY created_at DESC
LIMIT 20 OFFSET 10000;
-- 数据库需要扫描并丢弃前10000条数据才能返回结果
-- 随着OFFSET增大，性能急剧下降

-- 性能测试对比
-- OFFSET=0:   ~5ms
-- OFFSET=1000:  ~50ms
-- OFFSET=10000: ~500ms
-- OFFSET=50000: ~2500ms
```

**深度分页性能问题的原因：** 即使只需要20条数据，数据库也需要读取并排序前10020条数据，然后丢弃前10000条。在没有索引的情况下，这是不可避免的全表扫描和排序。

#### 3.2.2 基于游标的分页优化

游标分页（Keyset Pagination）使用上一页最后一条记录的主键或排序键作为起点，避免OFFSET操作。

```sql
-- 传统的OFFSET分页（低效）
SELECT * FROM orders
WHERE customer_id = 100
ORDER BY created_at DESC
LIMIT 20 OFFSET 1000;

-- 基于游标的分页（高效）
-- 上一页最后一条记录：created_at = '2024-03-15 10:30:00', id = 9999
SELECT * FROM orders
WHERE customer_id = 100
  AND (created_at, id) < ('2024-03-15 10:30:00', 9999)
ORDER BY created_at DESC, id DESC
LIMIT 20;
-- 数据库直接从游标位置开始扫描，无需跳过数据

-- 性能对比（数据量1000万）：
-- OFFSET=0:     ~10ms
-- OFFSET=1000:  ~12ms
-- OFFSET=10000: ~15ms
-- OFFSET=50000: ~18ms（稳定）
-- 游标分页：    ~8ms（恒定）
```

**游标分页的限制：** 无法跳转到任意页面，只能按顺序翻页。适用于"加载更多"场景，不适用于"跳转到第100页"场景。

#### 3.2.3 优化实现示例

```sql
-- 创建分页专用索引
CREATE INDEX idx_orders_customer_date ON orders(customer_id, created_at DESC, id DESC);

-- 应用层分页查询实现（TypeScript/Node.js）
async function getOrdersPage(
  customerId: number,
  cursor: { createdAt: Date; id: number } | null,
  pageSize: number = 20
): Promise<{ orders: Order[]; nextCursor: { createdAt: Date; id: number } | null }> {
  let query = db
    .select()
    .from(orders)
    .where(eq(orders.customerId, customerId));

  if (cursor) {
    query = query.where(
      or(
        lt(orders.createdAt, cursor.createdAt),
        and(
          eq(orders.createdAt, cursor.createdAt),
          lt(orders.id, cursor.id)
        )
      )
    );
  }

  const results = await query
    .orderBy(desc(orders.createdAt), desc(orders.id))
    .limit(pageSize + 1);

  const hasNextPage = results.length > pageSize;
  const orders = hasNextPage ? results.slice(0, -1) : results;
  const nextCursor = hasNextPage
    ? { createdAt: orders[orders.length - 1].createdAt, id: orders[orders.length - 1].id }
    : null;

  return { orders, nextCursor };
}
```

### 3.3 COUNT(*)优化技巧

COUNT(*)是数据库中最常用但也是最容易产生性能问题的操作之一，特别是在大表上。

#### 3.3.1 COUNT(*)性能分析

```sql
-- 全表COUNT（低效）
SELECT COUNT(*) FROM orders;
-- 需要扫描全表或全索引统计行数

-- 查看表的统计信息
SELECT reltuples, relpages FROM pg_class WHERE relname = 'orders';
-- reltuples: 估算的行数
-- relpages: 数据页数
```

#### 3.3.2 优化方案

**使用近似COUNT：** 如果业务允许，可以使用PostgreSQL的统计信息获取近似行数。

```sql
-- 使用统计表估算行数（误差约5%）
SELECT reltuples AS estimated_count
FROM pg_class
WHERE relname = 'orders';

-- 使用pg_class实时统计（更准确但仍有误差）
SELECT n_live_tup + n_dead_tup AS total_tuples
FROM pg_stat_user_tables
WHERE relname = 'orders';
```

**使用覆盖索引：** 如果必须精确COUNT，确保使用覆盖索引避免回表。

```sql
-- 创建覆盖索引加速COUNT
CREATE INDEX idx_orders_customer ON orders(customer_id);

-- COUNT查询使用索引
SELECT COUNT(*) FROM orders WHERE customer_id = 100;
-- 直接扫描索引，无需访问数据表
```

**缓存COUNT结果：** 对于不频繁更新的表，可以定期计算COUNT并缓存。

```sql
-- 创建统计表
CREATE TABLE statistics (
    key VARCHAR(100) PRIMARY KEY,
    value BIGINT,
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 定期更新统计
INSERT INTO statistics (key, value, updated_at)
VALUES ('orders_count', (SELECT COUNT(*) FROM orders), NOW())
ON CONFLICT (key) DO UPDATE
SET value = (SELECT COUNT(*) FROM orders),
    updated_at = NOW();
```

### 3.4 JOIN优化与MRR

多表JOIN是SQL中常见的操作，但不当的JOIN可能导致性能急剧下降。理解JOIN的执行机制和优化技巧对于编写高效SQL至关重要。

#### 3.4.1 JOIN类型与性能特征

```sql
-- Nested Loop Join（小表驱动大表，索引存在时高效）
SELECT o.*, u.name
FROM orders o
JOIN users u ON o.user_id = u.id;
-- 外层循环遍历小表，内层循环通过索引查找大表

-- Hash Join（大表连接，内存密集型）
SELECT o.*, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.created_at > '2024-01-01';
-- 构建哈希表，适合无索引的大表连接

-- Merge Join（输入已排序，顺序扫描）
SELECT o.*, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
ORDER BY u.id;
-- 先排序，再归并，适合有序输入
```

**JOIN优化原则：**

- 确保JOIN列上有索引
- 让小表作为驱动表（外层循环）
- 过滤early，减少JOIN的数据量
- 避免SELECT *，只选择需要的列

#### 3.4.2 Multi-Range Read (MRR)

MRR是MySQL的优化技术，通过将随机磁盘读取转换为顺序读取来提升JOIN性能。当JOIN需要通过索引访问数据行时，MRR将主键排序后顺序读取，减少磁盘随机访问。

```sql
-- MySQL MRR优化示例
SET optimizer_switch='mrr=on,mrr_cost_based=off';

-- 查看是否使用了MRR
EXPLAIN
SELECT o.*, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.created_at > '2024-01-01';
-- Extra列显示 "Using MRR" 表示启用了MRR优化
```

#### 3.4.3 复杂查询改写技巧

```sql
-- 改写1：将子查询转为JOIN
-- 低效写法
SELECT * FROM users
WHERE id IN (SELECT user_id FROM orders WHERE amount > 1000);

-- 高效写法（使用EXISTS或JOIN）
SELECT DISTINCT u.*
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.amount > 1000;

-- EXISTS改写（保留语义）
SELECT * FROM users u
WHERE EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id AND o.amount > 1000
);

-- 改写2：将OR转为UNION ALL
-- 低效写法
SELECT * FROM orders
WHERE customer_id = 100 OR status = 'cancelled';

-- 高效写法
SELECT * FROM orders WHERE customer_id = 100
UNION ALL
SELECT * FROM orders WHERE status = 'cancelled' AND customer_id != 100;

-- 改写3：拆解复杂的NOT IN
-- 低效写法
SELECT * FROM users
WHERE id NOT IN (SELECT user_id FROM orders WHERE status = 'completed');

-- 高效写法
SELECT u.*
FROM users u
LEFT JOIN orders o ON u.id = o.user_id AND o.status = 'completed'
WHERE o.user_id IS NULL;
```

```sql
-- 优化后的查询（分步执行，避免数据放大）
-- 步骤1：先查询订单主表（使用索引）
WITH user_orders AS (
    SELECT o.id, o.status, o.total_amount, o.created_at
    FROM orders o
    WHERE o.user_id = 12345
      AND o.created_at > NOW() - INTERVAL '3 months'
)
-- 步骤2：关联用户信息（数据量小，使用NLJ+索引）
SELECT u.name as customer_name
FROM users u
WHERE u.id = 12345;

-- 完整优化查询
WITH user_orders AS (
    SELECT o.id, o.status, o.total_amount, o.created_at
    FROM orders o
    WHERE o.user_id = 12345
      AND o.created_at > NOW() - INTERVAL '3 months'
),
order_items_agg AS (
    SELECT oi.order_id,
           COUNT(*) as item_count,
           SUM(oi.quantity) as total_quantity,
           SUM(oi.price * oi.quantity) as items_amount
    FROM order_items oi
    WHERE oi.order_id IN (SELECT id FROM user_orders)
    GROUP BY oi.order_id
)
SELECT o.*,
       u.name as customer_name,
       oia.item_count,
       oia.total_quantity,
       oia.items_amount
FROM user_orders o
JOIN users u ON u.id = 12345
JOIN order_items_agg oia ON o.id = oia.order_id
ORDER BY o.created_at DESC;
```

```sql
async function getUserOrdersWithDetails(userId: number) {
  // 1. 查询订单主表
  const orders = await db.query(
    'SELECT * FROM orders WHERE user_id = ? AND created_at > ? ORDER BY created_at DESC',
    [userId, threeMonthsAgo]
  );

  // 2. 批量查询订单项（使用IN）

  const orderItems = await db.query(
    'SELECT * FROM order_items WHERE order_id = ANY(?)',
    [orderIds]
  );

  // 3. 批量查询产品信息
  const productIds = [...new Set(orderItems.map(oi => oi.product_id))];
  const products = await db.query(

    [productIds]
  );
  const productMap = new Map(products.map(p => [p.id, p]));

  // 4. 应用层组装数据
  return orders.map(order => ({
    ...order,
    items: orderItems
      .filter(oi => oi.order_id === order.id)
      .map(oi => ({
        ...oi,
        product: productMap.get(oi.product_id)
      }))
  }));
}
```

## 四、PostgreSQL高级特性详解

PostgreSQL是功能最强大的开源关系型数据库，其高级特性使其不仅是一个数据存储系统，更是一个完整的数据处理平台。本章节深入讲解PostgreSQL的高级SQL特性及其实际应用。

### 4.1 窗口函数深度应用

窗口函数（Window Functions）是SQL:2003标准引入的强大特性，允许在保持行级粒度的同时进行跨行计算。与聚合函数不同，窗口函数不会改变结果集的行数，只是为每一行添加额外的计算结果。

#### 4.1.1 窗口函数基础

窗口函数的基本语法是在聚合函数后加上OVER子句，定义窗口的分组、排序和框架规则。

```sql
-- 基础窗口函数示例
SELECT
    order_id,
    customer_id,
    amount,
    -- 窗口聚合：计算每个客户的订单总额
    SUM(amount) OVER (PARTITION BY customer_id) AS customer_total,
    -- 窗口聚合：计算所有订单的平均金额
    AVG(amount) OVER () AS overall_avg,
    -- 窗口计数：计算每个客户的订单序号
    ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY created_at) AS order_seq,
    -- 窗口排名：计算每个客户的订单排名
    RANK() OVER (PARTITION BY customer_id ORDER BY amount DESC) AS amount_rank,
    -- 窗口百分比排名
    PERCENT_RANK() OVER (PARTITION BY customer_id ORDER BY amount) AS pct_rank
FROM orders;
```

**OVER子句的三个关键组件：**

- **PARTITION BY**：定义窗口的分组边界，类似于GROUP BY但不会折叠行
- **ORDER BY**：定义窗口内行的顺序（可选）
- **ROWS/RANGE**：定义窗口的物理框架（可选，默认是整个分区）

#### 4.1.2 排名相关窗口函数

```sql
-- ROW_NUMBER：连续排名，无并列
SELECT
    product_name,
    sales,
    ROW_NUMBER() OVER (ORDER BY sales DESC) AS row_num,
    RANK() OVER (ORDER BY sales DESC) AS rank,           -- 有并列，跳过排名
    DENSE_RANK() OVER (ORDER BY sales DESC) AS dense_rank -- 有并列，不跳过排名
FROM products;

-- 结果示例：
-- product_name | sales | row_num | rank | dense_rank
-- -------------+-------+---------+------+------------
-- Product A    |  1000 |       1 |    1 |          1
-- Product B    |  1000 |       2 |    1 |          1
-- Product C    |   800 |       3 |    3 |          2  -- 注意rank跳过了2
-- Product D    |   800 |       4 |    3 |          2
```

#### 4.1.3 导航相关窗口函数

```sql
-- LAG和LEAD：获取前一行/后一行的值
SELECT
    order_id,
    created_at,
    amount,
    LAG(amount) OVER (ORDER BY created_at) AS prev_amount,        -- 上一条订单金额
    LEAD(amount) OVER (ORDER BY created_at) AS next_amount,        -- 下一条订单金额
    amount - LAG(amount) OVER (ORDER BY created_at) AS amount_diff -- 与上条订单的差值
FROM orders
WHERE customer_id = 100;

-- FIRST_VALUE和LAST_VALUE：获取窗口的第一个/最后一个值
SELECT
    order_id,
    created_at,
    amount,
    FIRST_VALUE(amount) OVER (
        PARTITION BY customer_id
        ORDER BY created_at
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS first_amount,
    LAST_VALUE(amount) OVER (
        PARTITION BY customer_id
        ORDER BY created_at
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS last_amount
FROM orders;

-- NTH_VALUE：获取窗口内第N行的值
SELECT
    order_id,
    amount,
    NTH_VALUE(amount, 2) OVER (
        ORDER BY amount
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS second_highest_amount
FROM orders;
```

#### 4.1.4 窗口框架详解

窗口框架（Window Frame）定义了窗口函数计算所使用的行范围，可以是相对于当前行的固定数量行或基于值的范围。

```sql
-- 移动平均示例（当前行及前2行）
SELECT
    order_id,
    created_at,
    amount,
    AVG(amount) OVER (
        ORDER BY created_at
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS moving_avg_3,
    -- 累计求和（从分区开始到当前行）
    SUM(amount) OVER (
        PARTITION BY customer_id
        ORDER BY created_at
        ROWS UNBOUNDED PRECEDING
    ) AS cumulative_sum
FROM orders;

-- RANGE vs ROWS的区别
-- ROWS：基于物理行位置
-- RANGE：基于逻辑值（相同值被视为同一组）
SELECT
    amount,
    SUM(amount) OVER (ORDER BY amount ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING) AS rows_sum,
    SUM(amount) OVER (ORDER BY amount RANGE BETWEEN 1 PRECEDING AND 1 FOLLOWING) AS range_sum
FROM (VALUES (100), (100), (200), (200), (300)) AS t(amount);
-- 对于amount=100：
-- ROWS: (100 + 100 + 200) / 3 = 133  (当前行、前1行、后1行)
-- RANGE: (100 + 100) / 2 = 100        (所有值为100的行)
```

### 4.2 CTE公用表表达式

CTE（Common Table Expression，公用表表达式）是一种临时命名结果集，可以在SQL语句中引用。CTE分为普通CTE和递归CTE两种类型。

#### 4.2.1 普通CTE

```sql
-- 简化复杂查询
WITH
-- 第一个CTE：获取活跃客户
active_customers AS (
    SELECT id, name, email
    FROM users
    WHERE status = 'active'
      AND last_login_at > NOW() - INTERVAL '30 days'
),
-- 第二个CTE：获取这些客户的订单统计
customer_orders AS (
    SELECT
        o.customer_id,
        COUNT(*) as order_count,
        SUM(o.total_amount) as total_spent
    FROM orders o
    JOIN active_customers ac ON o.customer_id = ac.id
    WHERE o.status = 'completed'
    GROUP BY o.customer_id
),
-- 第三个CTE：计算客户价值等级
customer_value AS (
    SELECT
        co.*,
        CASE
            WHEN total_spent > 10000 THEN 'VIP'
            WHEN total_spent > 5000 THEN 'Premium'
            WHEN total_spent > 1000 THEN 'Regular'
            ELSE 'Basic'
        END as value_tier
    FROM customer_orders co
)
-- 最终查询：关联所有信息
SELECT
    ac.name,
    ac.email,
    cv.order_count,
    cv.total_spent,
    cv.value_tier
FROM active_customers ac
JOIN customer_values cv ON ac.id = cv.customer_id
ORDER BY cv.total_spent DESC;
```

#### 4.2.2 递归CTE

递归CTE允许引用自己，从而实现递归计算。这在处理层级结构、图结构数据时非常有用。

```sql
-- 递归CTE：查询组织架构
WITH RECURSIVE org_tree AS (
    -- 基础情况：根节点（CEO）
    SELECT
        id,
        name,
        title,
        manager_id,
        1 AS level,
        name::text AS path
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    -- 递归情况：子节点
    SELECT
        e.id,
        e.name,
        e.title,
        e.manager_id,
        ot.level + 1,
        ot.path || ' > ' || e.name
    FROM employees e
    JOIN org_tree ot ON e.manager_id = ot.id
)
SELECT * FROM org_tree ORDER BY path;

-- 递归CTE：生成分面数据
WITH RECURSIVE date_series AS (
    SELECT DATE '2024-01-01' AS date

    UNION ALL

    SELECT date + 1
    FROM date_series
    WHERE date < DATE '2024-12-31'
)
SELECT * FROM date_series;

-- 递归CTE：获取最短路径（Dijkstra简化版）
WITH RECURSIVE shortest_path AS (
    -- 起点
    SELECT
        start_node,
        end_node,
        ARRAY[start_node] AS path,
        cost AS total_cost
    FROM edges
    WHERE start_node = 'A'

    UNION ALL

    -- 扩展路径
    SELECT
        sp.end_node,
        e.end_node,
        sp.path || e.end_node,
        sp.total_cost + e.cost
    FROM shortest_path sp
    JOIN edges e ON sp.path[array_upper(sp.path, 1)] = e.start_node
    WHERE NOT e.end_node = ANY(sp.path)  -- 避免环路
      AND array_length(sp.path, 1) < 20  -- 最大深度限制
)
SELECT * FROM shortest_path WHERE end_node = 'Z' ORDER BY total_cost LIMIT 1;
```

#### 4.2.3 递归CTE实战：树形评论系统

```sql
-- 评论表结构
CREATE TABLE comments (
    id SERIAL PRIMARY KEY,
    post_id INT NOT NULL,
    parent_id INT REFERENCES comments(id),
    user_id INT NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 递归查询：获取评论树（包含完整层级路径）
WITH RECURSIVE comment_tree AS (
    -- 根评论
    SELECT
        id,
        post_id,
        parent_id,
        user_id,
        content,
        created_at,
        1 AS depth,
        ARRAY[id] AS path
    FROM comments
    WHERE parent_id IS NULL AND post_id = 123

    UNION ALL

    -- 子评论
    SELECT
        c.id,
        c.post_id,
        c.parent_id,
        c.user_id,
        c.content,
        c.created_at,
        ct.depth + 1,
        ct.path || c.id
    FROM comments c
    JOIN comment_tree ct ON c.parent_id = ct.id
)
SELECT
    id,
    depth,
    path,
    REPEAT('  ', depth - 1) || content AS indented_content,
    created_at
FROM comment_tree
ORDER BY path;
```

### 4.3 JSONB数据类型与操作

PostgreSQL的JSONB数据类型存储高效的二进制JSON格式，支持完整的JSON语法，同时提供传统的索引能力。这使得PostgreSQL可以同时处理结构化和半结构化数据。

#### 4.3.1 JSONB基础操作

```sql
-- 创建JSONB列
CREATE TABLE events (
    id SERIAL PRIMARY KEY,
    event_type VARCHAR(50),
    payload JSONB,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 插入JSONB数据
INSERT INTO events (event_type, payload) VALUES
('user_signup', '{"user_id": 1001, "email": "john@example.com", "source": "organic"}'),
('user_login', '{"user_id": 1001, "ip": "192.168.1.1", "device": "mobile"}'),
('purchase', '{"order_id": 5001, "user_id": 1001, "items": [{"sku": "A001", "qty": 2}], "total": 199.99}');

-- JSONB操作符示例
-- 获取对象字段（-> vs ->>）
SELECT payload->>'email' FROM events WHERE event_type = 'user_signup';  -- 返回文本
SELECT payload->'user_id' FROM events WHERE event_type = 'user_signup'; -- 返回JSON值

-- 数组访问
SELECT payload->'items'->0->>'sku' FROM events WHERE event_type = 'purchase';

-- 检查JSONB属性存在
SELECT * FROM events WHERE payload ? 'ip';  -- 检查顶层key存在
SELECT * FROM events WHERE payload @> '{"user_id": 1001}'; -- 检查包含指定JSON

-- JSONB路径查询（GJSON风格）
SELECT payload#>>'{user_id}' FROM events;  -- 获取嵌套字段
```

#### 4.3.2 JSONB函数与聚合

```sql
-- JSONB函数
-- 将行转为JSONB对象
SELECT jsonb_object_agg(key, value)
FROM (VALUES ('name', 'John'::jsonb), ('age', '30'::jsonb)) AS t(key, value);

-- 合并JSONB对象
SELECT jsonb_merge(payload, '{"extra": "data"}')
FROM events WHERE id = 1;

-- 删除JSONB属性
SELECT payload - 'ip' FROM events WHERE event_type = 'user_login';

-- JSONB聚合
-- 将多行聚合为一个JSONB数组
SELECT event_type, jsonb_agg(payload) AS events
FROM events
GROUP BY event_type;

-- 将多行聚合为JSONB对象（键为ID）
SELECT jsonb_object_agg(id::text, payload) AS events_map
FROM events;
```

#### 4.3.3 JSONB索引

```sql
-- GIN索引：支持containment和exists查询
CREATE INDEX idx_events_payload ON events USING gin (payload);

-- JSONB路径索引：加速特定路径查询
CREATE INDEX idx_events_user_id ON events ((payload->>'user_id'));

-- 表达式索引：基于路径的值创建索引
CREATE INDEX idx_events_user_id_int ON events ((CAST(payload->>'user_id' AS INT)));

-- 查询使用索引的验证
EXPLAIN SELECT * FROM events WHERE payload @> '{"user_id": 1001}';
EXPLAIN SELECT * FROM events WHERE (payload->>'user_id')::INT = 1001;
```

#### 4.3.4 JSONB实战：事件溯源存储

```sql
-- 事件溯源表设计
CREATE TABLE event_store (
    id BIGSERIAL PRIMARY KEY,
    aggregate_id UUID NOT NULL,
    aggregate_type VARCHAR(100) NOT NULL,
    event_type VARCHAR(100) NOT NULL,
    event_data JSONB NOT NULL,
    metadata JSONB DEFAULT '{}',
    version INT NOT NULL,  -- 聚合版本号，用于乐观并发
    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(aggregate_id, version)
);

-- 追加事件
INSERT INTO event_store (aggregate_id, aggregate_type, event_type, event_data, version)
VALUES (
    'a123e560-1234-5678-9abc-def012345678',
    'Order',
    'OrderCreated',
    '{"order_no": "ORD-001", "customer_id": 1001, "items": []}'::jsonb,
    '{"user_id": 999, "correlation_id": "c456"}'::jsonb,
    1
);

-- 查询聚合的所有事件
SELECT event_type, event_data
FROM event_store
WHERE aggregate_id = 'a123e560-1234-5678-9abc-def012345678'
ORDER BY version;

-- 物化视图：聚合的当前状态
CREATE MATERIALIZED VIEW order_snapshots AS
SELECT
    aggregate_id,
    aggregate_type,
    event_data->>'order_no' AS order_no,
    event_data->>'customer_id' AS customer_id,
    version,
    created_at
FROM event_store
WHERE aggregate_type = 'Order'
DISTINCT ON (aggregate_id);

-- 重建聚合状态
CREATE OR REPLACE FUNCTION rebuild_order_state(order_id UUID)
RETURNS JSONB AS $$
DECLARE
    events RECORD;
    state JSONB := '{}';
BEGIN
    FOR events IN
        SELECT event_data, event_type, version
        FROM event_store
        WHERE aggregate_id = order_id
        ORDER BY version
    LOOP
        -- 根据事件类型应用状态变更
        CASE events.event_type
            WHEN 'OrderCreated' THEN
                state := events.event_data;
            WHEN 'OrderUpdated' THEN
                state := state || events.event_data;
            WHEN 'OrderCancelled' THEN
                state := state || '{"status": "cancelled"}'::jsonb;
        END CASE;
    END LOOP;

    RETURN state;
END;
$$ LANGUAGE plpgsql;
```

### 4.5 分区表深度应用

分区表（Partitioned Table）是PostgreSQL 10引入的特性，允许将一个大表物理拆分为多个较小的"子表"，每个子表存储特定范围的数据。

#### 4.5.1 分区策略

```sql
-- 范围分区：按时间范围
CREATE TABLE orders (
    id SERIAL,
    order_no VARCHAR(50) NOT NULL,
    customer_id INT NOT NULL,
    total_amount DECIMAL(10,2),
    created_at TIMESTAMP NOT NULL
) PARTITION BY RANGE (created_at);

-- 创建按月分区的子表
CREATE TABLE orders_2024_01 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE orders_2024_02 PARTITION OF orders
    FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

-- 列表分区：按枚举值
CREATE TABLE logs (
    id SERIAL,
    level VARCHAR(10) NOT NULL,
    message TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
) PARTITION BY LIST (level);

CREATE TABLE logs_error PARTITION OF logs FOR VALUES IN ('ERROR', 'FATAL');
CREATE TABLE logs_warn PARTITION OF logs FOR VALUES IN ('WARNING');
CREATE TABLE logs_info PARTITION OF logs FOR VALUES IN ('INFO', 'DEBUG');

-- Hash分区：均匀分布
CREATE TABLE user_sessions (
    id SERIAL,
    user_id INT NOT NULL,
    token VARCHAR(255),
    expires_at TIMESTAMP
) PARTITION BY HASH (user_id);

CREATE TABLE user_sessions_0 PARTITION OF user_sessions FOR VALUES WITH (MODULUS 4, REMAINDER 0);
CREATE TABLE user_sessions_1 PARTITION OF user_sessions FOR VALUES WITH (MODULUS 4, REMAINDER 1);
CREATE TABLE user_sessions_2 PARTITION OF user_sessions FOR VALUES WITH (MODULUS 4, REMAINDER 2);
CREATE TABLE user_sessions_3 PARTITION OF user_sessions FOR VALUES WITH (MODULUS 4, REMAINDER 3);
```

#### 4.5.2 分区表维护

```sql
-- 自动创建分区（需要定期维护任务）
CREATE OR REPLACE FUNCTION create_monthly_partitions()
RETURNS void AS $$
DECLARE
    next_month DATE;
    partition_name TEXT;
    start_date DATE;
    end_date DATE;
BEGIN
    -- 创建下个月的分区
    next_month := DATE_TRUNC('month', CURRENT_DATE) + INTERVAL '1 month';
    partition_name := 'orders_' || TO_CHAR(next_month, 'YYYY_MM');
    start_date := next_month;
    end_date := next_month + INTERVAL '1 month';

    EXECUTE format(
        'CREATE TABLE IF NOT EXISTS %I PARTITION OF orders FOR VALUES FROM (%L) TO (%L)',
        partition_name, start_date, end_date
    );
END;
$$ LANGUAGE plpgsql;

-- 删除旧分区（数据归档）
DROP TABLE IF EXISTS orders_2023_01;

-- 分离分区为独立表
ALTER TABLE orders DETACH PARTITION orders_2023_01;
ALTER TABLE orders_2023_01 RENAME TO orders_archive_2023_01;

-- 查看分区信息
SELECT
    parent.relname AS parent_table,
    child.relname AS partition,
    pg_get_expr(child.relpartbound, child.oid) AS partition_range
FROM pg_inherits
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid
JOIN pg_class child ON pg_inherits.inhrelid = child.oid
WHERE parent.relname = 'orders';
```

### 4.6 物化视图实战

物化视图（Materialized View）将查询结果物理存储为表，可以显著加速复杂聚合查询，但数据可能不是最新的。

#### 4.6.1 物化视图基础

```sql
-- 创建物化视图
CREATE MATERIALIZED VIEW monthly_sales AS
SELECT
    DATE_TRUNC('month', created_at) AS month,
    customer_id,
    COUNT(*) AS order_count,
    SUM(total_amount) AS total_sales,
    AVG(total_amount) AS avg_order_value
FROM orders
WHERE status = 'completed'
GROUP BY DATE_TRUNC('month', created_at), customer_id
WITH DATA;

-- 创建索引
CREATE UNIQUE INDEX idx_monthly_sales ON monthly_sales (month, customer_id);

-- 查询物化视图（极速）
SELECT * FROM monthly_sales WHERE month = '2024-01-01';

-- 刷新物化视图
REFRESH MATERIALIZED VIEW monthly_sales;           -- 全量刷新（阻塞查询）
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_sales; -- 并发刷新（不阻塞）
```

#### 4.6.2 实战：报表统计系统

```sql
-- 设计报表物化视图
CREATE MATERIALIZED VIEW report_orders_daily AS
SELECT
    -- 时间维度
    DATE_TRUNC('day', o.created_at) AS order_date,
    EXTRACT(DOW FROM o.created_at) AS day_of_week,
    EXTRACT(HOUR FROM o.created_at) AS hour_of_day,

    -- 客户维度
    o.customer_id,
    c.segment AS customer_segment,
    c.tier AS customer_tier,
    c.region AS customer_region,

    -- 订单维度
    o.status,
    CASE WHEN o.total_amount > 1000 THEN 'high' WHEN o.total_amount > 100 THEN 'medium' ELSE 'low' END AS order_value_tier,

    -- 商品维度
    oi.category_id,
    p.brand,
    p.is_premium,

    -- 聚合指标
    COUNT(DISTINCT o.id) AS order_count,
    COUNT(DISTINCT o.customer_id) AS unique_customers,
    SUM(o.total_amount) AS total_revenue,
    SUM(o.discount_amount) AS total_discount,
    AVG(o.total_amount) AS avg_order_value,
    SUM(oi.quantity) AS total_items
FROM orders o
JOIN order_items oi ON o.id = oi.order_id
JOIN customers c ON o.customer_id = c.id
JOIN products p ON oi.product_id = p.id
WHERE o.created_at >= '2023-01-01'
GROUP BY
    DATE_TRUNC('day', o.created_at),
    EXTRACT(DOW FROM o.created_at),
    EXTRACT(HOUR FROM o.created_at),
    o.customer_id,
    c.segment,
    c.tier,
    c.region,
    o.status,
    CASE WHEN o.total_amount > 1000 THEN 'high' WHEN o.total_amount > 100 THEN 'medium' ELSE 'low' END,
    oi.category_id,
    p.brand,
    p.is_premium
WITH DATA;

-- 创建复合索引加速常见查询
CREATE INDEX idx_report_daily_composite ON report_orders_daily (order_date, customer_segment);

-- 定时刷新任务（使用pg_cron或外部调度器）
SELECT cron.schedule(
    'refresh-daily-report',
    '0 2 * * *',  -- 每天凌晨2点
    'REFRESH MATERIALIZED VIEW CONCURRENTLY report_orders_daily'
);
```

## 五、数据库设计规范化

良好的数据库设计是系统性能和可维护性的基础。规范化设计减少数据冗余，保证数据一致性；反规范化设计提升查询性能，适应特定业务场景。

### 5.1 范式理论与实践

数据库规范化（Normalization）是通过组织表结构来减少数据冗余和提高数据完整性的过程。范式是设计规范的标准，从低到高分为1NF、2NF、3NF、BCNF等。

#### 5.1.1 第一范式（1NF）

第一范式要求表的每个字段都是原子值，不可再分。

```
-- 违反1NF的示例
users表：
id | name | phone_numbers
1  | John | 13800138000, 13900139000  -- 电话号码不是原子值

-- 符合1NF的设计
users表：
id | name
1  | John

phones表：
user_id | phone_number
1       | 13800138000
1       | 13900139000
```

#### 5.1.2 第二范式（2NF）

第二范式要求在满足1NF的基础上，非主键字段完全依赖于主键，不能只依赖主键的一部分（针对复合主键）。

```
-- 违反2NF的示例（订单项表）
order_items表（复合主键：order_id + product_id）：
order_id | product_id | product_name | quantity | price
1        | 100         | iPhone       | 2        | 6999
-- product_name和price只依赖product_id，不依赖order_id

-- 符合2NF的设计
products表：
product_id | product_name | price
100        | iPhone       | 6999

order_items表（主键：order_id + product_id）：
order_id | product_id | quantity
1        | 100        | 2
```

#### 5.1.3 第三范式（3NF）

第三范式要求在满足2NF的基础上，非主键字段之间不存在传递依赖。

```
-- 违反3NF的示例
orders表：
order_id | customer_id | customer_name | customer_city
1        | 100         | John          | Beijing
-- customer_name和customer_city依赖于customer_id，而不是直接依赖于order_id

-- 符合3NF的设计
customers表：
customer_id | customer_name | city_id

cities表：
city_id | city_name

orders表：
order_id | customer_id
```

#### 5.1.4 BCNF范式

BCNF是3NF的改进，要求任何决定因素都包含候选键。

```
-- 违反BCNF的示例
course_teacher表：
course | teacher | classroom
Math   | Alice   | Room 101
Science| Bob     | Room 102
Math   | Charlie | Room 101
-- 依赖关系：classroom -> teacher, teacher -> course
-- teacher不是候选键，但决定了course

-- 符合BCNF的设计
courses表：
course_id | course | teacher_id

teachers表：
teacher_id | teacher | classroom

course_assignments表：
course_id | teacher_id
```

#### 5.2.4 反规范化优化

```sql
-- 用户统计表（反规范化，加速查询）
CREATE TABLE user_stats (
    user_id BIGINT PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    follower_count INT DEFAULT 0,
    following_count INT DEFAULT 0,
    post_count INT DEFAULT 0,
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 触发器自动更新统计
CREATE OR REPLACE FUNCTION update_user_stats()
RETURNS TRIGGER AS $$
BEGIN
    IF TG_OP = 'INSERT' THEN
        IF TG_TABLE_NAME = 'follows' THEN
            IF NEW.follower_id = NEW.following_id THEN
                RETURN NEW;
            END IF;
            UPDATE user_stats SET
                follower_count = follower_count + 1
            WHERE user_id = NEW.following_id;
            UPDATE user_stats SET
                following_count = following_count + 1
            WHERE user_id = NEW.follower_id;
        ELSIF TG_TABLE_NAME = 'posts' THEN
            UPDATE user_stats SET
                post_count = post_count + 1
            WHERE user_id = NEW.user_id;
        END IF;
    ELSIF TG_OP = 'DELETE' THEN
        IF TG_TABLE_NAME = 'follows' THEN
            UPDATE user_stats SET
                follower_count = follower_count - 1
            WHERE user_id = OLD.following_id;
            UPDATE user_stats SET
                following_count = following_count - 1
            WHERE user_id = OLD.follower_id;
        ELSIF TG_TABLE_NAME = 'posts' THEN
            UPDATE user_stats SET
                post_count = post_count - 1
            WHERE user_id = OLD.user_id;
        END IF;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_follow_stats
    AFTER INSERT OR DELETE ON follows
    FOR EACH ROW EXECUTE FUNCTION update_user_stats();

CREATE TRIGGER trg_post_stats
    AFTER INSERT OR DELETE ON posts
    FOR EACH ROW EXECUTE FUNCTION update_user_stats();
```

```sql
-- 查看当前慢查询
SELECT
    pid,
    now() - query_start AS duration,
    state,
    query
FROM pg_stat_activity
WHERE state != 'idle'
  AND query_start < now() - INTERVAL '10 seconds'
ORDER BY duration DESC;

-- pg_stat_statements查看统计
SELECT
    query,
    calls,
    total_exec_time / 1000 AS total_seconds,
    mean_exec_time AS avg_ms,
    stddev_exec_time,
    rows / calls AS avg_rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

### 6.5 性能监控视图

```sql
-- 实时会话监控
SELECT
    pid,
    usename,
    application_name,
    client_addr,
    state,
    now() - query_start AS duration,
    wait_event_type,
    wait_event,
    left(query, 100) AS query_preview
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY duration DESC;

-- 表膨胀监控
SELECT
    schemaname,
    relname AS table_name,
    n_live_tup,
    n_dead_tup,
    round(n_dead_tup * 100.0 / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_tuple_pct,
    last_vacuum,
    last_autovacuum,
    vacuum_count,
    autovacuum_count
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY n_dead_tup DESC;

-- 索引使用统计
SELECT
    schemaname,
    relname AS table_name,
    indexrelname AS index_name,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND indisunique = false
ORDER BY pg_relation_size(indexrelid) DESC;

-- 缓存命中率
SELECT
    sum(heap_blks_read) AS heap_read,
    sum(heap_blks_hit) AS heap_hit,
    sum(heap_blks_hit) * 100.0 / NULLIF(sum(heap_blks_hit) + sum(heap_blks_read), 0) AS hit_ratio
FROM pg_statio_user_tables;

-- 连接数使用
SELECT
    count(*) AS current_connections,
    sum(numbackends) AS total_backends,
    setting AS max_connections
FROM pg_stat_database, pg_settings
WHERE name = 'max_connections';
```
