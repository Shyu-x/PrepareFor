## 1. 为什么需要分布式锁

### 1.1 多进程问题

在传统的单机应用中，我们使用进程内的锁（如Java的synchronized、Python的threading.Lock）就可以解决并发问题。因为所有线程共享同一块内存空间，锁可以有效地阻止多个线程同时访问共享资源。

然而，在现代分布式系统中，业务往往部署在多台服务器上，每台服务器运行着独立的进程。这些进程可能运行在不同的容器、不同的虚拟机甚至不同的物理机器上。它们各自拥有独立的内存空间，彼此之间无法直接访问对方的内存。

考虑一个典型的电商秒杀场景：

```javascript
// 假设我们有3台服务器处理秒杀请求
// 服务器A收到用户请求，需要扣减库存
// 服务器B收到另一个用户请求，也需要扣减库存
// 服务器C同样收到用户请求

// 在单机环境下，这没问题，因为有全局锁保护
// 但在分布式环境下，每个进程的锁只能保护自己

// 问题场景：
// 商品库存：100件
// 服务器A：读取库存100，扣减1，库存变为99
// 服务器B：读取库存100，扣减1，库存变为99（错误的！应该读到99）
// 服务器C：读取库存100，扣减1，库存变为99（更错误的！）

// 结果：实际卖出3件，但库存只扣了1件
```

这就是经典的**缓存穿透问题**和**超卖问题**的根源。每个进程都是独立运行的，它们看不到彼此对共享资源的修改。

### 1.2 进程间同步的挑战

在单机多线程环境中，进程内的锁通过操作系统的底层机制（如mutex）实现，这些机制确保了锁的原子性和可见性。当一个线程获取锁后，其他线程会被阻塞，直到锁被释放。

但是在分布式环境中，我们需要一种**跨进程的锁机制**。这带来了额外的挑战：

| 挑战 | 单机多线程 | 分布式多进程 |
|------|-----------|-------------|
| 锁的实现 | 操作系统内核 | 外部协调服务 |
| 可见性 | 共享内存，天然可见 | 需要网络通信 |
| 原子性 | CPU指令保证 | 需要分布式协议 |
| 性能 | 纳秒级 | 毫秒级（网络延迟） |
| 故障恢复 | 进程崩溃，锁自动释放 | 需要TTL和租约机制 |

### 1.3 锁的重要性：解决核心业务问题

分布式锁在以下场景中必不可少：

**场景一：库存扣减（防超卖）**

```javascript
// 没有分布式锁的情况
async function deductStock(productId, quantity) {
  // 第一步：查询当前库存
  const stock = await redis.get(`stock:${productId}`);
  // stock = 100

  // 第二步：检查库存是否充足
  if (stock < quantity) {
    throw new Error('库存不足');
  }

  // 第三步：扣减库存
  await redis.decrby(`stock:${productId}`, quantity);
  // 库存变为 99

  // 问题：如果多个进程同时执行，都会通过第2步检查，
  // 因为它们在第一步都读到了100，最终导致超卖
}

// 有分布式锁的情况
async function deductStockWithLock(productId, quantity) {
  // 获取锁，确保同一时间只有一个进程能执行扣减逻辑
  const lock = await redis.set(`lock:stock:${productId}`, '1', 'NX', 'EX', 10);

  if (lock !== 'OK') {
    throw new Error('系统繁忙，请稍后重试');
  }

  try {
    // 现在是安全的，因为只有获取到锁的进程才能执行到这里
    const stock = await redis.get(`stock:${productId}`);
    if (stock < quantity) {
      throw new Error('库存不足');
    }
    await redis.decrby(`stock:${productId}`, quantity);
  } finally {
    // 释放锁
    await redis.del(`lock:stock:${productId}`);
  }
}
```

**场景二：定时任务重复执行**

```javascript
// 定时任务可能部署多个实例，需要确保只有一个实例执行
async function runScheduledTask() {
  const lockKey = 'lock:daily-settlement';

  // 使用分布式锁，确保只有一个实例执行
  const acquired = await redis.set(lockKey, hostname, 'NX', 'EX', 3600);

  if (acquired !== 'OK') {
    console.log('另一个实例正在执行，跳过');
    return;
  }

  try {
    // 执行定时任务逻辑
    await performDailySettlement();
  } finally {
    // 只允许持有锁的实例释放
    const currentHolder = await redis.get(lockKey);
    if (currentHolder === hostname) {
      await redis.del(lockKey);
    }
  }
}
```

**场景三：幂等性控制**

```javascript
// 支付回调等操作需要幂等处理
async function handlePaymentCallback(orderId, paidAmount) {
  const lockKey = `lock:payment:${orderId}`;

  // 获取锁，防止重复处理
  const lock = await redis.set(lockKey, '1', 'NX', 'EX', 30);

  if (lock !== 'OK') {
    return { success: false, message: '正在处理中' };
  }

  try {
    // 检查订单状态
    const order = await db.getOrder(orderId);
    if (order.status === 'PAID') {
      return { success: true, message: '已处理' };
    }

    // 更新订单状态
    await db.updateOrder(orderId, { status: 'PAID', paidAmount });
    await db.updateStock(order.productId, -order.quantity);

    return { success: true };
  } finally {
    await redis.del(lockKey);
  }
}
```

### 1.4 分布式锁的基本要求

一个合格的分布式锁应该满足以下条件：

| 要求 | 描述 |
|------|------|
| **互斥性** | 在任意时刻，只有一个客户端能持有锁 |
| **不会死锁** | 即使锁定资源的客户端崩溃，也能自动释放锁 |
| **独占性** | 只有锁的持有者才能释放锁 |
| **公平性** | 按照请求顺序获取锁（可选） |
| **高性能** | 高并发下锁的获取和释放延迟要低 |
| **容错性** | 部分节点故障时，锁服务仍然可用 |

## 2. Redis分布式锁基础

### 2.1 SETNX + EXPIRE 方式

早期实现Redis分布式锁的方法是分两步操作：

```javascript
// 第一步：尝试设置锁（SET if Not eXists）
// 如果锁不存在，则设置成功返回1；如果锁已存在，则设置失败返回0
const result = await redis.setnx('lock:product:001', '1');

// 第二步：设置过期时间，防止锁永久存在导致死锁
if (result === 1) {
  await redis.expire('lock:product:001', 10); // 10秒后自动释放
}

// 执行业务逻辑...

// 最后：释放锁
await redis.del('lock:product:001');
```

这种实现方式看起来简单，但存在**严重的安全问题**：

```javascript
// 问题场景：进程A获取锁后，突然崩溃或阻塞
// 导致锁无法被正常释放

// 进程A获取锁
const result = await redis.setnx('lock:product:001', '1');
if (result === 1) {
  await redis.expire('lock:product:001', 10);

  // 进程A在这里崩溃了！还没来得及执行del
  // 或者GC暂停导致超时

  // ...业务逻辑还没执行完...

  await redis.del('lock:product:001'); // 这行永远不会执行
}

// 10秒后锁自动释放，其他进程可以获取锁
// 但进程A的业务逻辑可能还在执行，导致数据不一致
```

更危险的是，如果这两步之间发生故障，锁会永久存在：

```javascript
// 最危险的情况：setnx成功，但expire失败
const result = await redis.setnx('lock:product:001', '1');
// result = 1，锁已设置

// 就在这时，Redis连接断开了
// expire操作永远不会执行

// 结果：锁永久存在，形成死锁
```

### 2.2 SET key value NX EX 原子方式

为了解决上述问题，Redis 2.6.12版本引入了`SET`命令的增强选项，可以原子性地同时设置值和过期时间：

```javascript
/**
 * 使用SET命令的NX和EX选项实现分布式锁
 *
 * SET key value NX EX seconds
 * - NX: Only set the key if it does not already exist（互斥性）
 * - EX: Set the specified expire time, in seconds（防止死锁）
 *
 * 返回值：
 * - "OK" 表示成功获取锁
 * - null 表示获取锁失败（锁已被其他进程持有）
 */
const result = await redis.set('lock:product:001', 'unique-token-123', 'NX', 'EX', 10);

if (result === 'OK') {
  try {
    // 成功获取锁，执行业务逻辑
    console.log('成功获取锁，开始处理业务...');

    // 模拟业务处理时间
    await processOrder();

  } finally {
    // 释放锁
    await redis.del('lock:product:001');
  }
} else {
  console.log('获取锁失败，锁已被其他进程持有');
}
```

### 2.3 原子性问题的深层理解

为什么`SETNX + EXPIRE`不是原子性的，而`SET ... NX EX`是原子性的？

**SETNX + EXPIRE（非原子）的问题：**

```javascript
// Redis命令执行流程
// 客户端 -> TCP连接 -> Redis服务端

// SETNX lock:xxx 1  ---->  Redis执行SETNX，返回1
// [网络延迟/进程暂停/GC]
// EXPIRE lock:xxx 10 ---->  Redis执行EXPIRE，返回1

// 如果在这两步之间发生以下情况：
// 1. 进程崩溃 -> expire不会执行，锁永生
// 2. Redis主从切换 -> SETNX成功，但expire可能丢失
// 3. 超时断连 -> expire不会执行

// SET ... NX EX（原子操作）
// 客户端发送一条命令: SET lock:xxx 1 NX EX 10
// Redis收到完整命令，执行它，然后返回结果
// 不存在中间状态，整条命令要么成功要么失败
```

### 2.4 基础分布式锁实现

让我们实现一个完整的Redis分布式锁类：

```javascript
/**
 * Redis分布式锁实现类
 *
 * 设计要点：
 * 1. 使用SET NX EX保证原子性获取锁
 * 2. 使用唯一的token保证锁的独占性释放
 * 3. 使用Lua脚本保证释放锁的原子性
 */
class RedisLock {
  constructor(redis, options = {}) {
    // Redis客户端实例
    this.redis = redis;
    // 锁前缀，用于区分不同业务的锁
    this.prefix = options.prefix || 'lock:';
    // 默认锁超时时间（毫秒）
    this.defaultTtl = options.ttl || 10000;
  }

  /**
   * 生成唯一的锁标识
   * 使用随机字符串+时间戳+进程ID，保证在分布式环境下唯一性
   */
  generateToken() {
    return `${process.pid}-${Date.now()}-${Math.random().toString(36).substring(2, 15)}`;
  }

  /**
   * 获取分布式锁
   *
   * @param {string} key - 锁的key
   * @param {number} ttl - 锁的过期时间（毫秒），默认10秒
   * @returns {Promise<string|null>} - 成功返回token，失败返回null
   */
  async acquire(key, ttl = this.defaultTtl) {
    const lockKey = `${this.prefix}${key}`;
    const token = this.generateToken();
    const ttlSeconds = Math.ceil(ttl / 1000);

    /**
     * 使用SET命令原子性获取锁
     * NX - 只有键不存在时才设置（保证互斥）
     * PX - 过期时间以毫秒为单位（精确控制）
     */
    const result = await this.redis.set(lockKey, token, 'NX', 'PX', ttl);

    if (result === 'OK') {
      console.log(`[RedisLock] 获取锁成功: ${lockKey}, token: ${token}, ttl: ${ttl}ms`);
      return token;
    }

    console.log(`[RedisLock] 获取锁失败: ${lockKey}`);
    return null;
  }

  /**
   * 释放分布式锁
   *
   * @param {string} key - 锁的key
   * @param {string} token - 获取锁时返回的token
   * @returns {Promise<boolean>} - 成功释放返回true
   *
   * 注意：释放锁时必须验证token，防止误删其他进程的锁
   * 例如：进程A获取了锁，业务逻辑执行超时，锁自动释放
   *       进程B获取了同一个锁
   *       进程A恢复正常，执行del操作，如果不验证token，就会删除进程B的锁
   */
  async release(key, token) {
    const lockKey = `${this.prefix}${key}`;

    /**
     * 使用Lua脚本保证原子性：
     * 1. 检查锁的值是否等于token
     * 2. 如果相等，则删除锁
     * 3. 返回执行结果（1=成功，0=失败）
     *
     * Lua脚本在Redis中是原子执行的，不会被中断
     */
    const luaScript = `
      if redis.call('GET', KEYS[1]) == ARGV[1] then
        return redis.call('DEL', KEYS[1])
      else
        return 0
      end
    `;

    const result = await this.redis.eval(luaScript, 1, lockKey, token);

    if (result === 1) {
      console.log(`[RedisLock] 释放锁成功: ${lockKey}`);
      return true;
    }

    console.log(`[RedisLock] 释放锁失败，可能是token不匹配: ${lockKey}`);
    return false;
  }

  /**
   * 尝试获取锁，带重试机制
   *
   * @param {string} key - 锁的key
   * @param {number} ttl - 锁的过期时间
   * @param {number} retryTimes - 重试次数
   * @param {number} retryDelay - 重试间隔（毫秒）
   * @returns {Promise<{token: string, acquired: boolean}>}
   */
  async acquireWithRetry(key, ttl = this.defaultTtl, retryTimes = 3, retryDelay = 200) {
    for (let i = 0; i < retryTimes; i++) {
      const token = await this.acquire(key, ttl);

      if (token) {
        return { token, acquired: true };
      }

      // 如果还有重试次数，等待后重试
      if (i < retryTimes - 1) {
        console.log(`[RedisLock] 第${i + 1}次获取失败，${retryDelay}ms后重试...`);
        await new Promise(resolve => setTimeout(resolve, retryDelay));
      }
    }

    return { token: null, acquired: false };
  }

  /**
   * 使用锁执行回调函数（自动获取和释放）
   *
   * @param {string} key - 锁的key
   * @param {Function} callback - 要执行的异步回调函数
   * @param {number} ttl - 锁的过期时间
   * @returns {Promise<any>} - 回调函数的返回值
   *
   * 使用示例：
   * const result = await lock.withLock('order:123', async () => {
   *   return await processPayment();
   * });
   */
  async withLock(key, callback, ttl = this.defaultTtl) {
    const { token, acquired } = await this.acquireWithRetry(key, ttl);

    if (!acquired) {
      throw new Error(`无法获取锁: ${key}，系统繁忙，请稍后重试`);
    }

    try {
      // 执行回调函数
      return await callback();
    } finally {
      // 无论成功还是失败，都要释放锁
      await this.release(key, token);
    }
  }
}

// 使用示例
const Redis = require('ioredis');
const redis = new Redis({ host: 'localhost', port: 6379 });

const lock = new RedisLock(redis);

// 基础用法
async function basicUsage() {
  const token = await lock.acquire('product:001');

  if (!token) {
    console.log('获取锁失败');
    return;
  }

  try {
    // 业务逻辑
    console.log('执行业务逻辑...');
  } finally {
    await lock.release('product:001', token);
  }
}

// 自动释放用法
async function autoReleaseUsage() {
  try {
    await lock.withLock('order:123', async () => {
      console.log('在锁保护下执行...');
      await saveOrder();
      await deductStock();
      return { success: true };
    });
  } catch (error) {
    console.error('执行失败:', error.message);
  }
}
```

## 3. 锁的过期时间与续期机制

### 3.1 设置多长的过期时间

锁的过期时间（TTL）是分布式锁设计中最重要的参数之一。设置过短或过长都会带来问题：

**TTL过短的问题：**

```javascript
// 假设TTL设置为1秒
const result = await redis.set('lock:order:123', token, 'NX', 'PX', 1000);

if (result === 'OK') {
  // 获取锁成功，开始执行业务逻辑

  // 业务逻辑执行时间：
  // 1. 查询订单详情      50ms
  // 2. 验证用户权限     100ms
  // 3. 调用支付接口     500ms
  // 4. 更新订单状态     50ms
  // 总计：700ms，还在TTL内

  // 但如果支付接口响应慢一些...
  // 1. 查询订单详情      50ms
  // 2. 验证用户权限     100ms
  // 3. 调用支付接口    2000ms  <-- 外部接口慢
  // 4. 更新订单状态     50ms
  // 总计：2200ms，超过了1000ms的TTL！

  // 此时，锁自动过期释放
  // 其他进程获取到同一个锁，开始执行

  // 进程A还在执行支付回调...
  // 进程B也开始执行支付，可能导致重复支付！

  await processPayment(); // 假设这个花了2秒
  await updateOrderStatus();
}

// 释放锁时发现锁已经不存在了
await lock.release('lock:order:123', token); // 返回0，锁早已自动释放
```

**TTL过长的问题：**

```javascript
// 假设TTL设置为1小时
const result = await redis.set('lock:order:123', token, 'NX', 'PX', 3600000);

if (result === 'OK') {
  // 获取锁成功

  // 业务逻辑执行过程中：
  // 1. 进程崩溃了，没有释放锁
  // 2. 服务器宕机了，没有释放锁
  // 3. 网络分区了，无法连接到Redis

  // 无论哪种情况，锁都要等1小时才能自动释放
  // 这期间，所有需要操作这个订单的请求都会失败

  await complexBusinessLogic(); // 可能要很久
}

// 用户等了1小时后才能继续操作
```

### 3.2 业务执行时间评估

正确设置TTL需要评估业务的执行时间：

```javascript
/**
 * 业务执行时间评估方法
 */

// 方法一：基于历史数据分析
const businessTimeStats = {
  // 操作类型: {平均耗时, 最大耗时, P99耗时}
  'query_order': { avg: 50, max: 200, p99: 150 },
  'create_order': { avg: 100, max: 500, p99: 300 },
  'process_payment': { avg: 300, max: 2000, p99: 1500 },
  'send_notification': { avg: 100, max: 800, p99: 500 }
};

// 方法二：计算合理的TTL
function calculateTtl(operationType, multiplier = 2) {
  const stats = businessTimeStats[operationType];

  if (!stats) {
    // 未知操作，使用保守值
    return 30000; // 30秒
  }

  // TTL = P99耗时 * 倍数（保证在绝大多数情况下业务能完成）
  return stats.p99 * multiplier;
}

// 方法三：动态计算（考虑网络延迟）
async function calculateDynamicTtl(operationType) {
  // 测量当前到Redis的往返延迟
  const start = Date.now();
  await redis.ping();
  const redisLatency = Date.now() - start;

  const baseTime = calculateTtl(operationType);

  // 考虑Redis延迟的2倍（往返）+ 业务处理时间
  return baseTime + (redisLatency * 2);
}

// 使用示例
const ttl = calculateTtl('process_payment', 1.5);
console.log(`推荐TTL: ${ttl}ms`); // 输出：推荐TTL: 2250ms
```

### 3.3 续期机制（Watchdog）

为了解决TTL设置的两难问题，我们需要**锁续期机制**（也称为Watchdog）。其核心思想是：

1. 获取锁时设置一个较短的TTL
2. 启动一个后台任务，在业务执行过程中定期续期
3. 如果业务执行完成，主动释放锁，同时停止续期任务
4. 如果进程崩溃，后台任务也会停止，锁最终会自动释放

```javascript
/**
 * 带Watchdog的分布式锁
 *
 * 工作原理：
 * 1. 获取锁时，设置初始TTL（如10秒）
 * 2. 启动Watchdog后台任务，每3秒检查一次
 * 3. 只要锁还被持有（业务还在执行），就续期到10秒
 * 4. 业务执行完成，释放锁，停止Watchdog
 * 5. 如果进程崩溃，Watchdog也停止，锁在10秒后自动释放
 */
class RedisLockWithWatchdog {
  constructor(redis, options = {}) {
    this.redis = redis;
    this.prefix = options.prefix || 'lock:';
    this.defaultTtl = options.ttl || 10000;    // 默认TTL 10秒
    this.watchdogInterval = options.watchdogInterval || 3000; // 每3秒续期
    this.extendRatio = options.extendRatio || 0.5; // 续期到TTL的50%时开始续期
  }

  generateToken() {
    return `${process.pid}-${Date.now()}-${Math.random().toString(36).substring(2, 15)}`;
  }

  /**
   * 获取带Watchdog的锁
   */
  async acquire(key, ttl = this.defaultTtl) {
    const lockKey = `${this.prefix}${key}`;
    const token = this.generateToken();

    // 尝试获取锁
    const result = await this.redis.set(lockKey, token, 'NX', 'PX', ttl);

    if (result !== 'OK') {
      return null;
    }

    // 获取锁成功，启动Watchdog
    const watchdogTimer = this.startWatchdog(key, token, ttl);

    // 返回一个包装对象，包含token和停止Watchdog的方法
    return {
      token,
      stopWatchdog: () => {
        clearInterval(watchdogTimer);
        console.log(`[Watchdog] 停止监控: ${key}`);
      }
    };
  }

  /**
   * 启动Watchdog续期任务
   */
  startWatchdog(key, token, ttl) {
    console.log(`[Watchdog] 启动监控: ${key}, TTL: ${ttl}ms`);

    // 使用setInterval定期续期
    const timer = setInterval(async () => {
      try {
        // 检查锁是否还被我们持有
        const currentToken = await this.redis.get(`${this.prefix}${key}`);

        if (currentToken === token) {
          // 锁还在，续期
          const newTtl = await this.redis.pttl(`${this.prefix}${key}`);

          // 如果剩余TTL小于TTL的50%，则续期
          if (newTtl < ttl * this.extendRatio && newTtl > 0) {
            await this.redis.pexpire(`${this.prefix}${key}`, ttl);
            console.log(`[Watchdog] 续期成功: ${key}, 新TTL: ${ttl}ms`);
          }
        } else {
          // 锁已经不在了，停止Watchdog
          console.log(`[Watchdog] 锁已释放或被其他进程获取: ${key}`);
          clearInterval(timer);
        }
      } catch (error) {
        console.error(`[Watchdog] 续期失败: ${error.message}`);
      }
    }, this.watchdogInterval);

    return timer;
  }

  /**
   * 释放锁
   */
  async release(key, token) {
    const lockKey = `${this.prefix}${key}`;

    const luaScript = `
      if redis.call('GET', KEYS[1]) == ARGV[1] then
        return redis.call('DEL', KEYS[1])
      else
        return 0
      end
    `;

    const result = await this.redis.eval(luaScript, 1, lockKey, token);
    return result === 1;
  }

  /**
   * 续期锁（手动调用）
   */
  async extend(key, token, ttl) {
    const lockKey = `${this.prefix}${key}`;

    const luaScript = `
      if redis.call('GET', KEYS[1]) == ARGV[1] then
        return redis.call('PEXPIRE', KEYS[1], ARGV[2])
      else
        return 0
      end
    `;

    const result = await this.redis.eval(luaScript, 1, lockKey, token, ttl);
    return result === 1;
  }

  /**
   * 使用带Watchdog的锁执行回调
   */
  async withLock(key, callback, ttl = this.defaultTtl) {
    const lock = await this.acquire(key, ttl);

    if (!lock) {
      throw new Error(`无法获取锁: ${key}`);
    }

    try {
      return await callback();
    } finally {
      // 释放锁前先停止Watchdog
      lock.stopWatchdog();
      await this.release(key, lock.token);
    }
  }
}

// 使用示例
const lockWithWatchdog = new RedisLockWithWatchdog(redis);

async function processOrderWithWatchdog() {
  // 使用Watchdog，即使业务执行时间超过TTL也不会有问题
  await lockWithWatchdog.withLock('order:123', async () => {
    console.log('开始处理订单（Watchdog保护）...');

    // 模拟耗时操作
    await new Promise(resolve => setTimeout(resolve, 20000)); // 20秒

    console.log('订单处理完成');
    return { success: true };
  }, 10000); // TTL设为10秒，Watchdog会自动续期
}

// 问题场景演示：
// 1. TTL = 10秒
// 2. 业务执行需要20秒
// 3. Watchdog每3秒检查一次，发现剩余TTL不足5秒时续期到10秒
// 4. 最终业务完成，释放锁
```

### 3.4 续期实现详解

续期机制的关键在于**原子性**。我们需要确保检查和续期是原子操作：

```javascript
/**
 * 续期的几种实现方式
 */

// 方式一：使用Lua脚本（推荐）
async function extendWithLua(redis, key, token, ttl) {
  // Lua脚本保证原子性
  const luaScript = `
    -- 检查锁是否被当前进程持有
    if redis.call('GET', KEYS[1]) == ARGV[1] then
      -- 续期到指定时间
      return redis.call('PEXPIRE', KEYS[1], ARGV[2])
    else
      -- 锁已被其他进程获取或已释放
      return 0
    end
  `;

  return await redis.eval(luaScript, 1, key, token, ttl);
}

// 方式二：使用Redis事务（pipeline + WATCH）
async function extendWithTransaction(redis, key, token, ttl) {
  const maxRetries = 3;

  for (let i = 0; i < maxRetries; i++) {
    // 监视key，如果key在watch期间被修改，事务会失败
    await redis.watch(key);

    const currentToken = await redis.get(key);

    if (currentToken !== token) {
      // 锁已被其他进程获取
      await redis.unwatch();
      return false;
    }

    // 执行事务
    const multi = redis.multi();
    multi.set(key, token, 'PX', ttl);
    const results = await multi.exec();

    // 如果exec返回null，说明watch的key被修改了，事务失败
    if (results === null) {
      console.log('事务执行失败，重试...');
      continue;
    }

    return true;
  }

  return false;
}

// 方式三：使用Redisson的风格（看门狗自动续期）
class RedissonStyleLock {
  constructor(redis, options = {}) {
    this.redis = redis;
    this.internalLockLeaseTime = options.internalLockLeaseTime || 30000; // 30秒
    this.watchdogTimeout = options.watchdogTimeout || 10000; // 10秒
  }

  async lock(key, ttl = this.internalLockLeaseTime) {
    const lockKey = `lock:${key}`;
    const token = this.generateToken();

    // 获取锁
    const acquired = await this.redis.set(lockKey, token, 'NX', 'PX', ttl);

    if (!acquired) {
      return null;
    }

    // 启动看门狗
    this.scheduleRefreshLock(key, token, ttl);

    return token;
  }

  async scheduleRefreshLock(key, token, ttl) {
    const refreshTime = ttl / 3; // 每过1/3 TTL时间续期一次

    const timer = setInterval(async () => {
      try {
        // 检查锁是否还在
        const currentToken = await this.redis.get(`lock:${key}`);

        if (currentToken === token) {
          // 续期
          await this.redis.pexpire(`lock:${key}`, ttl);
          console.log(`[看门狗] 续期成功: ${key}`);
        } else {
          // 锁已不在，停止续期
          clearInterval(timer);
        }
      } catch (error) {
        console.error(`[看门狗] 续期失败: ${error.message}`);
      }
    }, refreshTime);

    return timer;
  }

  generateToken() {
    return `${Date.now()}-${Math.random().toString(36).substring(2)}`;
  }
}
```

## 4. 可重入锁实现

### 4.1 什么是可重入锁

可重入锁（Reentrant Lock）是指同一个线程在外层函数获取锁之后，再进入内层函数会自动获取锁（前提是锁的持有者相同）。在单机环境下，这是很常见的特性。

**为什么需要可重入锁：**

```javascript
// 场景：一个方法获取锁后调用另一个也需要锁的方法

class OrderService {
  constructor(lock) {
    this.lock = lock;
  }

  // 方法1：创建订单
  async createOrder(orderId) {
    // 获取锁
    const token = await this.lock.acquire(`order:${orderId}`);

    try {
      console.log('创建订单开始...');
      await this.validateStock(orderId);  // <-- 这里也需要锁
      await this.saveOrder(orderId);
      await this.sendNotification(orderId);  // <-- 这里也需要锁
      console.log('创建订单完成');
    } finally {
      await this.lock.release(`order:${orderId}`, token);
    }
  }

  // 方法2：验证库存（也需要锁）
  async validateStock(orderId) {
    // 如果锁不可重入，这里会尝试重新获取锁，导致死锁！
    const token = await this.lock.acquire(`stock:${orderId}`);

    try {
      console.log('验证库存...');
      // 库存验证逻辑
    } finally {
      await this.lock.release(`stock:${orderId}`, token);
    }
  }

  // 方法3：发送通知（也需要锁）
  async sendNotification(orderId) {
    const token = await this.lock.acquire(`notification:${orderId}`);

    try {
      console.log('发送通知...');
    } finally {
      await this.lock.release(`notification:${orderId}`, token);
    }
  }
}

// 问题：如果锁不可重入
// 1. createOrder获取锁 lock:order:123
// 2. 调用validateStock，尝试获取锁 lock:stock:123
// 3. 如果validateStock也需要获取 lock:order:123，就会死锁！

// 可重入锁的解决方案：
// 同一个线程可以多次获取同一个锁，只需要计数匹配即可
```

### 4.2 计数器模式实现

可重入锁的核心是**计数器**：同一把锁被获取多少次，就要释放多少次才真正释放：

```javascript
/**
 * 可重入分布式锁实现
 *
 * 设计原理：
 * 1. 锁的值存储：token:count 格式
 * 2. token: 当前持有锁的进程标识
 * 3. count: 重入次数计数
 * 4. 获取锁时，如果token匹配，count+1；否则创建新token，count=1
 * 5. 释放锁时，count-1；只有count=0时才真正删除锁
 */

// Lua脚本：获取可重入锁
const ACQUIRE_LOCK_SCRIPT = `
  local key = KEYS[1]
  local token = ARGV[1]
  local ttl = tonumber(ARGV[2])

  -- 获取当前锁的值
  local value = redis.call('GET', key)

  if value then
    -- 锁已存在
    local parts = {}
    for part in string.gmatch(value, "[^:]+") do
      table.insert(parts, part)
    end

    local existingToken = parts[1]
    local count = tonumber(parts[2])

    if existingToken == token then
      -- 同一进程重入，计数器+1
      redis.call('SET', key, token .. ':' .. (count + 1), 'PX', ttl)
      return 1
    else
      -- 不同进程，锁被占用
      return 0
    end
  else
    -- 锁不存在，创建新锁
    redis.call('SET', key, token .. ':1', 'PX', ttl)
    return 1
  end
`;

// Lua脚本：释放可重入锁
const RELEASE_LOCK_SCRIPT = `
  local key = KEYS[1]
  local token = ARGV[1]

  local value = redis.call('GET', key)

  if value then
    local parts = {}
    for part in string.gmatch(value, "[^:]+") do
      table.insert(parts, part)
    end

    local existingToken = parts[1]
    local count = tonumber(parts[2])

    if existingToken == token then
      if count > 1 then
        -- 还有重入次数，计数器-1
        redis.call('SET', key, token .. ':' .. (count - 1), 'PX', redis.call('PTTL', key))
        return 1
      else
        -- 最后一个重入，直接删除锁
        redis.call('DEL', key)
        return 1
      end
    else
      -- token不匹配，无权释放
      return 0
    end
  else
    -- 锁不存在
    return 0
  end
`;

class ReentrantRedisLock {
  constructor(redis, options = {}) {
    this.redis = redis;
    this.prefix = options.prefix || 'rlock:';
    this.defaultTtl = options.ttl || 30000;
  }

  generateToken() {
    // 使用进程ID+线程ID+随机数保证全局唯一
    return `${process.pid}-${Date.now()}-${Math.random().toString(36).substring(2, 15)}`;
  }

  /**
   * 获取可重入锁
   *
   * @param {string} key - 锁的key
   * @param {number} ttl - 过期时间
   * @returns {Promise<string|null>} - 成功返回token，失败返回null
   */
  async acquire(key, ttl = this.defaultTtl) {
    const lockKey = `${this.prefix}${key}`;
    const token = this.generateToken();

    const result = await this.redis.eval(
      ACQUIRE_LOCK_SCRIPT,
      1,
      lockKey,
      token,
      ttl
    );

    if (result === 1) {
      console.log(`[ReentrantLock] 获取锁成功: ${lockKey}, token: ${token}`);
      return token;
    }

    console.log(`[ReentrantLock] 获取锁失败: ${lockKey}`);
    return null;
  }

  /**
   * 释放可重入锁
   */
  async release(key, token) {
    const lockKey = `${this.prefix}${key}`;

    const result = await this.redis.eval(
      RELEASE_LOCK_SCRIPT,
      1,
      lockKey,
      token
    );

    return result === 1;
  }

  /**
   * 尝试获取锁（非阻塞）
   */
  async tryAcquire(key, ttl = this.defaultTtl) {
    const lockKey = `${this.prefix}${key}`;
    const token = this.generateToken();

    // 先检查锁是否存在
    const exists = await this.redis.exists(lockKey);

    if (exists) {
      // 锁存在，检查是否可以重入
      return await this.acquire(key, ttl);
    } else {
      // 锁不存在，直接获取
      const result = await this.redis.set(lockKey, `${token}:1`, 'NX', 'PX', ttl);

      if (result === 'OK') {
        return token;
      }

      return null;
    }
  }

  /**
   * 使用可重入锁执行回调
   */
  async withLock(key, callback, ttl = this.defaultTtl) {
    const token = await this.acquire(key, ttl);

    if (!token) {
      throw new Error(`无法获取锁: ${key}`);
    }

    try {
      return await callback();
    } finally {
      await this.release(key, token);
    }
  }
}

// 使用示例
const rlock = new ReentrantRedisLock(redis);

// 测试可重入特性
async function testReentrant() {
  const key = 'test:reentrant';
  const token = await rlock.acquire(key);

  console.log('第一次获取锁:', token ? '成功' : '失败');

  // 模拟重入：同一个线程再次获取锁
  const token2 = await rlock.acquire(key);
  console.log('第二次获取锁（重入）:', token2 ? '成功' : '失败');

  // 两次获取的是同一个token
  console.log('两次token相同:', token === token2);

  // 第一次释放
  await rlock.release(key, token);
  console.log('第一次释放锁');

  // 锁还在（因为重入了）
  const stillHolding = await rlock.redis.get(`${rlock.prefix}${key}`);
  console.log('锁是否还在:', stillHolding);

  // 第二次释放
  await rlock.release(key, token);
  console.log('第二次释放锁');

  // 锁已删除
  const deleted = await rlock.redis.get(`${rlock.prefix}${key}`);
  console.log('锁是否已删除:', deleted === null);
}

testReentrant().catch(console.error);
```

### 4.3 ThreadId存储模式

在Java等语言中，可重入锁通常使用线程ID来识别。但在Node.js中，由于单线程模型的特殊性，我们需要自己管理线程/协程标识：

```javascript
/**
 * 基于ThreadId的可重入锁实现
 *
 * 适用场景：多线程环境（如Java）或需要精确追踪调用栈的环境
 */

// 使用Map存储本地线程/协程的重入计数
const threadLocal = new Map();

// Lua脚本：基于ThreadId的可重入获取
const ACQUIRE_WITH_THREAD_ID = `
  local key = KEYS[1]
  local ttl = tonumber(ARGV[1])

  -- 获取当前线程ID（这里简化为使用redis存储）
  -- 实际应用中应使用进程/线程唯一的标识
  local threadId = ARGV[2]

  local value = redis.call('GET', key)

  if value then
    local parts = {}
    for part in string.gmatch(value, "[^:]+") do
      table.insert(parts, part)
    end

    local existingThreadId = parts[1]
    local count = tonumber(parts[2])

    if existingThreadId == threadId then
      -- 同一线程重入
      redis.call('SET', key, threadId .. ':' .. (count + 1), 'PX', ttl)
      return 1
    else
      return 0
    end
  else
    redis.call('SET', key, threadId .. ':1', 'PX', ttl)
    return 1
  end
`;

/**
 * 封装异步上下文的可重入锁
 *
 * 适用于NestJS等框架，可以在请求上下文中存储重入计数
 */
class ContextualReentrantLock {
  constructor(redis) {
    this.redis = redis;
    this.prefix = 'ctx_rlock:';
  }

  /**
   * 获取当前请求的唯一标识
   */
  getContextId() {
    // 在NestJS中可以从请求上下文获取
    // 这里使用简单的实现
    return `request-${Date.now()}-${Math.random().toString(36).substring(2)}`;
  }

  /**
   * 获取锁，自动处理重入
   */
  async acquire(key, contextId, ttl = 30000) {
    const lockKey = `${this.prefix}${key}`;

    // 先本地检查重入次数
    const localCount = threadLocal.get(contextId) || 0;

    if (localCount > 0) {
      // 已经持有锁，增加本地计数
      threadLocal.set(contextId, localCount + 1);

      // 远程也需要更新计数
      await this.redis.eval(ACQUIRE_WITH_THREAD_ID, 1, lockKey, ttl, contextId);
      return true;
    }

    // 尝试获取锁
    const result = await this.redis.set(lockKey, `${contextId}:1`, 'NX', 'PX', ttl);

    if (result === 'OK') {
      threadLocal.set(contextId, 1);
      return true;
    }

    return false;
  }

  /**
   * 释放锁
   */
  async release(key, contextId) {
    const lockKey = `${this.prefix}${key}`;

    // 本地计数减1
    const localCount = threadLocal.get(contextId) || 1;

    if (localCount > 1) {
      threadLocal.set(contextId, localCount - 1);
    } else {
      threadLocal.delete(contextId);
    }

    // 远程释放
    const RELEASE_SCRIPT = `
      local key = KEYS[1]
      local threadId = ARGV[1]

      local value = redis.call('GET', key)
      if value then
        local parts = {}
        for part in string.gmatch(value, "[^:]+") do
          table.insert(parts, part)
        end

        if parts[1] == threadId then
          if tonumber(parts[2]) > 1 then
            redis.call('SET', key, threadId .. ':' .. (tonumber(parts[2]) - 1), 'PX', redis.call('PTTL', key))
          else
            redis.call('DEL', key)
          end
          return 1
        end
      end
      return 0
    `;

    const result = await this.redis.eval(RELEASE_SCRIPT, 1, lockKey, contextId);
    return result === 1;
  }
}
```

## 5. Redlock红锁算法

### 5.1 单点Redis锁的问题

前面我们实现的分布式锁都依赖于单一Redis实例。这存在明显的**单点故障**问题：

```javascript
// 单点Redis锁的问题
const lock = new RedisLock(redis);

// 1. 获取锁成功
const token = await lock.acquire('product:001');

// 2. 业务逻辑执行中...
await processOrder();

// 3. Redis主节点突然宕机！
//    - 如果没有持久化，锁数据丢失
//    - 如果有持久化但复制延迟，数据恢复后锁不存在

// 4. 锁丢失了，其他进程可以获取同一个锁
// 5. 两个进程同时操作，导致数据不一致

// 解决方案：Redlock算法 - 使用多个独立的Redis节点
```

### 5.2 Redlock算法原理

Redlock是Redis作者Antirez提出的分布式锁算法，核心思想是：**在多个独立的Redis节点上获取锁，只有超过半数节点成功获取锁，才认为成功**。

```
Redlock算法图示：

假设我们有5个Redis节点

进程A请求获取锁：
┌─────────────────────────────────────────────────────────┐
│  Node1: ✓ 获取成功                                      │
│  Node2: ✓ 获取成功                                      │
│  Node3: ✓ 获取成功  ──> 3/5 = 60% ≥ 50%，锁获取成功    │
│  Node4: ✗ 获取失败                                      │
│  Node5: ✗ 获取失败                                      │
└─────────────────────────────────────────────────────────┘

即使其中2个节点宕机，只要3个节点正常工作，锁就能正常工作
```

### 5.3 Redlock实现

```javascript
/**
 * Redlock分布式锁实现
 *
 * 算法步骤：
 * 1. 获取当前时间戳（毫秒）
 * 2. 依次向N个Redis节点获取锁，使用相同的key和token
 * 3. 计算获取锁的总时间，如果超过TTL的一半，获取失败
 * 4. 计算成功的节点数，如果超过N/2，获取成功
 * 5. 如果获取成功，锁的有效时间为 TTL - 获取总时间
 * 6. 如果获取失败，向所有节点释放锁
 */

class Redlock {
  constructor(redisClients, options = {}) {
    // Redis客户端数组（至少5个）
    if (!Array.isArray(redisClients) || redisClients.length < 3) {
      throw new Error('Redlock需要至少3个Redis节点');
    }
    this.clients = redisClients;
    this.quorum = Math.floor(redisClients.length / 2) + 1; // 多数节点
    this.retryCount = options.retryCount || 3;      // 重试次数
    this.retryDelay = options.retryDelay || 200;     // 重试延迟
    this.clockDriftFactor = 0.01;                    // 时钟漂移因子
    this.prefix = options.prefix || 'redlock:';
  }

  /**
   * 生成唯一的锁标识
   */
  generateToken() {
    return `${Date.now()}-${Math.random().toString(36).substring(2, 15)}`;
  }

  /**
   * 向单个节点获取锁
   */
  async acquireOnNode(client, key, token, ttl) {
    try {
      const result = await client.set(
        `${this.prefix}${key}`,
        token,
        'PX',
        ttl,
        'NX'
      );
      return result === 'OK';
    } catch (error) {
      console.error(`节点获取锁失败: ${error.message}`);
      return false;
    }
  }

  /**
   * 向单个节点释放锁
   */
  async releaseOnNode(client, key, token) {
    const luaScript = `
      if redis.call('GET', KEYS[1]) == ARGV[1] then
        return redis.call('DEL', KEYS[1])
      else
        return 0
      end
    `;

    try {
      const result = await client.eval(luaScript, 1, `${this.prefix}${key}`, token);
      return result === 1;
    } catch (error) {
      console.error(`节点释放锁失败: ${error.message}`);
      return false;
    }
  }

  /**
   * 获取TTL
   */
  async getTtlOnNode(client, key) {
    try {
      const ttl = await client.pttl(`${this.prefix}${key}`);
      return ttl > 0 ? ttl : 0;
    } catch (error) {
      return 0;
    }
  }

  /**
   * 获取分布式锁
   *
   * @param {string} key - 锁的key
   * @param {number} ttl - 锁的过期时间（毫秒）
   * @returns {Promise<{token: string, validityTime: number} | null>}
   */
  async lock(key, ttl = 10000) {
    const token = this.generateToken();
    const startTime = Date.now();

    console.log(`[Redlock] 开始获取锁: ${key}, TTL: ${ttl}ms`);

    // 尝试在所有节点上获取锁
    const results = await Promise.all(
      this.clients.map(client =>
        this.acquireOnNode(client, key, token, ttl)
      )
    );

    // 统计成功的节点数
    const successCount = results.filter(r => r === true).length;
    const totalTime = Date.now() - startTime;

    console.log(`[Redlock] 获取结果: 成功 ${successCount}/${this.clients.length} 节点, 耗时: ${totalTime}ms`);

    // 没有获取到多数节点的锁，获取失败
    if (successCount < this.quorum) {
      console.log(`[Redlock] 获取锁失败：未达到多数节点要求 (${successCount} < ${this.quorum})`);

      // 释放所有已获取的锁
      await this.unlockOnAllNodes(key, token);

      return null;
    }

    // 计算锁的有效时间
    // 需要减去获取锁消耗的时间和时钟漂移
    const driftTime = Math.ceil(ttl * this.clockDriftFactor);
    const validityTime = ttl - totalTime - driftTime;

    console.log(`[Redlock] 获取锁成功, 有效性时间: ${validityTime}ms`);

    return {
      token,
      validityTime,
      successNodes: successCount
    };
  }

  /**
   * 释放锁
   */
  async unlock(key, lockInfo) {
    if (!lockInfo || !lockInfo.token) {
      return false;
    }

    return await this.unlockOnAllNodes(key, lockInfo.token);
  }

  /**
   * 在所有节点上释放锁
   */
  async unlockOnAllNodes(key, token) {
    const results = await Promise.all(
      this.clients.map(client =>
        this.releaseOnNode(client, key, token)
      )
    );

    const successCount = results.filter(r => r === true).length;
    console.log(`[Redlock] 释放锁: 成功 ${successCount}/${this.clients.length} 节点`);

    return successCount >= this.quorum;
  }

  /**
   * 延长锁的过期时间
   */
  async extend(key, lockInfo, ttl) {
    if (!lockInfo || !lockInfo.token) {
      return false;
    }

    const token = lockInfo.token;

    // 在所有节点上尝试续期
    const results = await Promise.all(
      this.clients.map(client => {
        const luaScript = `
          if redis.call('GET', KEYS[1]) == ARGV[1] then
            return redis.call('PEXPIRE', KEYS[1], ARGV[2])
          else
            return 0
          end
        `;
        return client.eval(luaScript, 1, `${this.prefix}${key}`, token, ttl);
      })
    );

    const successCount = results.filter(r => r === 1).length;
    return successCount >= this.quorum;
  }

  /**
   * 使用Redlock执行回调
   */
  async withLock(key, callback, ttl = 10000) {
    const lockInfo = await this.lock(key, ttl);

    if (!lockInfo) {
      throw new Error(`无法获取锁: ${key}`);
    }

    try {
      return await callback();
    } finally {
      await this.unlock(key, lockInfo);
    }
  }
}

// 使用示例
async function main() {
  // 创建5个独立的Redis客户端（模拟5台Redis服务器）
  const redisClients = [
    new (require('ioredis'))('redis://localhost:6379'),  // Node1
    new (require('ioredis'))('redis://localhost:6380'),  // Node2
    new (require('ioredis'))('redis://localhost:6381'),  // Node3
    new (require('ioredis'))('redis://localhost:6382'),  // Node4
    new (require('ioredis'))('redis://localhost:6383'),  // Node5
  ];

  // 创建Redlock实例
  const redlock = new Redlock(redisClients, {
    retryCount: 3,
    retryDelay: 200
  });

  try {
    // 使用分布式锁
    await redlock.withLock('seckill:product:001', async () => {
      console.log('开始秒杀处理...');

      // 模拟业务处理
      await new Promise(resolve => setTimeout(resolve, 1000));

      console.log('秒杀处理完成');
    }, 30000); // 30秒超时

  } catch (error) {
    console.error('执行失败:', error.message);
  } finally {
    // 关闭所有连接
    await Promise.all(redisClients.map(client => client.quit()));
  }
}
```

### 5.4 时钟漂移问题

Redlock算法中一个重要的概念是**时钟漂移（Clock Drift）**。由于不同机器上的时钟可能不同步，导致问题：

```javascript
/**
 * 时钟漂移问题分析
 */

// 问题场景：
// 1. 进程A在Node1上获取锁，设置TTL=10000ms
// 2. Node1的时钟比实际时间快500ms
// 3. Node2和Node3的时钟比实际时间慢500ms

// 进程A认为：锁在 10000-500 = 9500ms 后过期
// 但实际上：Node1在 10500ms 后才过期

// 如果Node1在9500ms时宕机，锁实际上还剩1000ms才过期
// 但Node2和Node3可能认为锁已经过期了

// Redlock的解决方案：引入时钟漂移因子
class RedlockWithClockDrift {
  constructor(clients, options = {}) {
    this.clients = clients;
    this.quorum = Math.floor(clients.length / 2) + 1;
    this.clockDriftFactor = 0.01; // 1%的时钟漂移补偿
    // 如果TTL=10000ms，时钟漂移补偿=100ms
  }

  calculateValidityTime(ttl, totalTime) {
    // 减去获取锁消耗的时间和时钟漂移
    const drift = Math.ceil(ttl * this.clockDriftFactor);
    return ttl - totalTime - drift;
    // 这样计算出的validityTime更保守，确保锁真正有效
  }
}

/**
 * 实际生产环境建议：
 *
 * 1. Redlock适用于对锁安全性要求极高的场景
 * 2. 大多数场景使用单节点Redis锁+主从复制已经足够
 * 3. 如果需要更高的可靠性，考虑使用Zookeeper
 * 4. 评估是否真的需要Redlock，因为它的延迟更高
 */
```

## 6. Zookeeper分布式锁

### 6.1 Zookeeper简介

Zookeeper是一个分布式协调服务，提供了分布式锁的另一种实现方式。与Redis相比，Zookeeper的锁更加可靠，但性能较低。

**Zookeeper的核心特性：**

| 特性 | 说明 |
|------|------|
| **临时顺序节点** | 客户端断开连接后自动删除，节点按创建顺序编号 |
| **Watch机制** | 节点变化时通知客户端 |
| **原子性** | 所有操作保证原子性 |
| **一致性** | ZAB协议保证强一致性 |
| **高可用** | 集群模式下无单点故障 |

### 6.2 Zookeeper锁的原理

```javascript
/**
 * Zookeeper分布式锁原理
 *
 * 利用临时顺序节点和Watch机制实现：
 *
 * 1. 创建一个锁目录 /locks
 * 2. 客户端获取锁时，在目录下创建临时顺序节点
 * 3. 客户端获取所有子节点，按顺序排序
 * 4. 如果创建的节点是序号最小的，就获取锁成功
 * 5. 否则，Watcher监听前一个节点的变化
 * 6. 当前一个节点删除时，收到通知，再次检查自己是否最小
 * 7. 重复步骤4-6，直到获取锁
 *
 * 图示：
 * /locks
 * ├── lock-000000001  (客户端A创建)
 * ├── lock-000000002  (客户端B创建)
 * └── lock-000000003  (客户端C创建)
 *
 * 初始状态：
 * - 客户端A创建的节点最小，A获取锁
 *
 * A释放锁后（节点删除）：
 * /locks
 * ├── lock-000000002  (客户端B创建)
 * └── lock-000000003  (客户端C创建)
 *
 * - 客户端B创建的节点最小，B获取锁
 */

// 使用zookeeper官方客户端
const zookeeper = require('zookeeper');

class ZookeeperLock {
  constructor(client, options = {}) {
    this.client = client;
    this.rootPath = options.rootPath || '/locks';
    this.lockPrefix = options.lockPrefix || 'lock-';
    this.sessionTimeout = options.sessionTimeout || 30000;
  }

  /**
   * 创建锁（使用临时顺序节点）
   */
  async createLockNode(myZookeeper, lockName) {
    return new Promise((resolve, reject) => {
      // 创建临时顺序节点
      myZookeeper.create(
        `${this.rootPath}/${lockName}-`,
        null,  // 数据为空
        zookeeper.const.ACL.OPEN_ACL_UNSAFE,  // ACL权限
        zookeeper.const.EPHEMERAL | zookeeper.const.SEQUENCE,  // 临时+顺序
        (error, path) => {
          if (error) {
            reject(error);
          } else {
            resolve(path);
          }
        }
      );
    });
  }

  /**
   * 获取锁
   *
   * @param {string} lockName - 锁名称
   * @returns {Promise<string>} - 返回锁的路径
   */
  async acquire(lockName) {
    // 创建临时顺序节点
    const myPath = await this.createLockNode(this.client, lockName);
    console.log(`[ZookeeperLock] 创建节点: ${myPath}`);

    // 获取所有子节点
    const children = await this.getChildren();

    // 按序号排序
    const sortedChildren = children.sort();

    // 获取当前节点的序号
    const myNodeName = myPath.split('/').pop();
    const myIndex = sortedChildren.indexOf(myNodeName);

    // 如果是最小节点，获取锁成功
    if (myIndex === 0) {
      console.log(`[ZookeeperLock] 获取锁成功: ${myPath}`);
      return myPath;
    }

    // 否则，监听前一个节点
    const prevNode = sortedChildren[myIndex - 1];
    const prevPath = `${this.rootPath}/${prevNode}`;

    console.log(`[ZookeeperLock] 等待锁，监听: ${prevPath}`);

    // 等待前一个节点删除
    await this.watchNode(prevPath);

    // 收到通知后，重新检查
    return await this.acquire(lockName);
  }

  /**
   * 获取子节点列表
   */
  async getChildren() {
    return new Promise((resolve, reject) => {
      this.client.getChildren(
        this.rootPath,
        false,  // 不使用watch
        (error, children) => {
          if (error && error.code !== zookeeper.exception.NO_NODE) {
            reject(error);
          } else {
            resolve(children || []);
          }
        }
      );
    });
  }

  /**
   * 监听节点变化
   */
  watchNode(path) {
    return new Promise((resolve, reject) => {
      // 使用exists监听节点
      this.client.exists(
        path,
        (type, state, pathWatch) => {
          // 节点被删除时会触发
          console.log(`[ZookeeperLock] 节点变化: ${path}`);
          resolve();
        },
        (error) => {
          if (error && error.code !== zookeeper.exception.NO_NODE) {
            reject(error);
          }
          // 如果节点不存在，直接继续
          resolve();
        }
      );
    });
  }

  /**
   * 释放锁（删除临时节点）
   */
  async release(lockPath) {
    return new Promise((resolve, reject) => {
      this.client.remove(
        lockPath,
        -1,  // 版本号，-1表示不检查版本
        (error) => {
          if (error) {
            console.error(`[ZookeeperLock] 释放锁失败: ${error.message}`);
            reject(error);
          } else {
            console.log(`[ZookeeperLock] 释放锁成功: ${lockPath}`);
            resolve();
          }
        }
      );
    });
  }
}

// 使用示例
async function zookeeperLockExample() {
  // 创建Zookeeper客户端
  const zk = new zookeeper({
    connect: 'localhost:2181',
    timeout: 30000,
    debug_level: zookeeper.const.LOG_LEVEL.WARN,
    host_provider_discovery: null
  });

  // 等待连接
  await new Promise((resolve) => {
    zk.on('connect', resolve);
  });

  const lock = new ZookeeperLock(zk, {
    rootPath: '/myapp/locks'
  });

  try {
    // 获取锁
    const lockPath = await lock.acquire('order-processing');

    // 执行业务逻辑
    console.log('开始处理订单...');
    await processOrder();

    // 释放锁
    await lock.release(lockPath);

  } catch (error) {
    console.error('处理失败:', error);
  } finally {
    zk.close();
  }
}
```

## 7. 数据库分布式锁

### 7.1 唯一索引锁

最简单的方式是利用数据库的唯一索引来实现分布式锁：

```javascript
/**
 * 基于数据库唯一索引的分布式锁
 *
 * 原理：
 * 1. 创建一张锁表，包含 lock_name 和 owner 字段
 * 2. lock_name 是唯一索引
 * 3. 获取锁时，插入一条记录
 * 4. 释放锁时，删除记录
 *
 * 注意：需要捕获唯一键冲突异常来判断获取是否成功
 */

// 锁表结构
const createLockTableSQL = `
  CREATE TABLE IF NOT EXISTS distributed_lock (
    lock_name VARCHAR(255) PRIMARY KEY,
    owner VARCHAR(255) NOT NULL,
    expire_at DATETIME NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_expire (expire_at)
  ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
`;

class DatabaseLock {
  constructor(db, options = {}) {
    this.db = db;
    this.tableName = options.tableName || 'distributed_lock';
    this.defaultTtl = options.ttl || 30000; // 默认30秒
  }

  generateOwner() {
    return `${process.pid}-${Date.now()}-${Math.random().toString(36).substring(2)}`;
  }

  /**
   * 获取数据库分布式锁
   *
   * @param {string} lockName - 锁名称
   * @param {number} ttl - 过期时间（毫秒）
   * @returns {Promise<string|null>} - 成功返回owner，失败返回null
   */
  async acquire(lockName, ttl = this.defaultTtl) {
    const owner = this.generateOwner();
    const expireAt = new Date(Date.now() + ttl);

    const sql = `
      INSERT INTO ${this.tableName} (lock_name, owner, expire_at)
      VALUES (?, ?, ?)
      ON DUPLICATE KEY UPDATE
        owner = IF(expire_at < NOW(), VALUES(owner), owner),
        expire_at = IF(expire_at < NOW(), VALUES(expire_at), expire_at)
    `;

    try {
      // 先尝试获取过期的锁（如果存在）
      const result = await this.db.query(sql, [lockName, owner, expireAt]);

      // 检查是否获取成功
      const current = await this.db.query(
        `SELECT owner FROM ${this.tableName} WHERE lock_name = ?`,
        [lockName]
      );

      if (current[0] && current[0].owner === owner) {
        console.log(`[DatabaseLock] 获取锁成功: ${lockName}, owner: ${owner}`);

        // 启动过期清理任务
        this.startExpireCleanup();

        return owner;
      }

      console.log(`[DatabaseLock] 获取锁失败: ${lockName}`);
      return null;

    } catch (error) {
      console.error(`[DatabaseLock] 获取锁异常: ${error.message}`);
      return null;
    }
  }

  /**
   * 释放锁
   */
  async release(lockName, owner) {
    const sql = `
      DELETE FROM ${this.tableName}
      WHERE lock_name = ? AND owner = ?
    `;

    try {
      const result = await this.db.query(sql, [lockName, owner]);

      if (result.affectedRows > 0) {
        console.log(`[DatabaseLock] 释放锁成功: ${lockName}`);
        return true;
      }

      console.log(`[DatabaseLock] 释放锁失败，可能不是锁的持有者: ${lockName}`);
      return false;

    } catch (error) {
      console.error(`[DatabaseLock] 释放锁异常: ${error.message}`);
      return false;
    }
  }

  /**
   * 延长锁的过期时间
   */
  async extend(lockName, owner, ttl) {
    const expireAt = new Date(Date.now() + ttl);

    const sql = `
      UPDATE ${this.tableName}
      SET expire_at = ?
      WHERE lock_name = ? AND owner = ?
    `;

    try {
      const result = await this.db.query(sql, [expireAt, lockName, owner]);

      return result.affectedRows > 0;

    } catch (error) {
      console.error(`[DatabaseLock] 续期异常: ${error.message}`);
      return false;
    }
  }

  /**
   * 清理过期的锁
   */
  async cleanupExpired() {
    const sql = `DELETE FROM ${this.tableName} WHERE expire_at < NOW()`;

    try {
      const result = await this.db.query(sql);
      if (result.affectedRows > 0) {
        console.log(`[DatabaseLock] 清理过期锁: ${result.affectedRows}个`);
      }
    } catch (error) {
      console.error(`[DatabaseLock] 清理过期锁失败: ${error.message}`);
    }
  }

  /**
   * 启动过期清理任务
   */
  startExpireCleanup(interval = 60000) {
    // 使用单例模式，避免多个定时任务
    if (this.cleanupTimer) return;

    this.cleanupTimer = setInterval(() => {
      this.cleanupExpired();
    }, interval);
  }

  /**
   * 使用锁执行回调
   */
  async withLock(lockName, callback, ttl = this.defaultTtl) {
    const owner = await this.acquire(lockName, ttl);

    if (!owner) {
      throw new Error(`无法获取锁: ${lockName}`);
    }

    try {
      return await callback();
    } finally {
      await this.release(lockName, owner);
    }
  }
}
```

### 7.2 悲观锁

使用数据库的悲观锁（如SELECT ... FOR UPDATE）也可以实现分布式锁：

```javascript
/**
 * 基于悲观锁的分布式锁
 *
 * 原理：
 * 1. 开启事务
 * 2. SELECT ... FOR UPDATE 锁定查询的行
 * 3. 执行业务逻辑
 * 4. 提交事务，锁自动释放
 *
 * 注意：需要使用索引列作为查询条件，否则可能锁表
 */

class PessimisticDatabaseLock {
  constructor(db) {
    this.db = db;
  }

  /**
   * 使用悲观锁执行操作
   *
   * @param {string} lockKey - 锁的key
   * @param {Function} callback - 业务逻辑回调
   * @returns {Promise<any>}
   */
  async withPessimisticLock(lockKey, callback) {
    const connection = await this.db.getConnection();

    try {
      // 开启事务
      await connection.beginTransaction();

      // 获取锁（SELECT ... FOR UPDATE）
      // 这里用了一个技巧：查询一个总是存在的记录，加上FOR UPDATE
      // 实际应用中应该根据业务逻辑设计
      await connection.execute(
        'SELECT 1 AS lock_col FROM DUAL WHERE ? = ? FOR UPDATE',
        [lockKey, lockKey]
      );

      // 执行业务逻辑
      const result = await callback();

      // 提交事务
      await connection.commit();

      return result;

    } catch (error) {
      // 回滚事务
      await connection.rollback();
      throw error;

    } finally {
      // 释放连接
      connection.release();
    }
  }

  /**
   * 更好的实现：为每种资源创建专门的锁记录
   */
  async withNamedPessimisticLock(tableName, recordId, callback) {
    const connection = await this.db.getConnection();

    try {
      await connection.beginTransaction();

      // 锁定特定的记录
      await connection.execute(
        `SELECT * FROM ${tableName} WHERE id = ? FOR UPDATE`,
        [recordId]
      );

      const result = await callback();

      await connection.commit();
      return result;

    } catch (error) {
      await connection.rollback();
      throw error;
    } finally {
      connection.release();
    }
  }
}
```

### 7.3 乐观锁（版本号）

乐观锁通过版本号机制实现，适用于读多写少且冲突较少的场景：

```javascript
/**
 * 基于乐观锁（版本号）的分布式锁
 *
 * 原理：
 * 1. 读取数据时获取版本号
 * 2. 更新数据时检查版本号是否变化
 * 3. 如果版本号变化，说明有其他事务修改了数据，更新失败
 * 4. 失败后重试，直到成功或达到最大重试次数
 *
 * 适用场景：
 * - 库存扣减
 * - 账户余额扣款
 * - 乐观并发更新
 */

// 库存表结构
const inventoryTableSchema = `
  CREATE TABLE IF NOT EXISTS inventory (
    product_id VARCHAR(64) PRIMARY KEY,
    stock INT NOT NULL DEFAULT 0,
    version INT NOT NULL DEFAULT 0,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
  ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
`;

class OptimisticDatabaseLock {
  constructor(db, options = {}) {
    this.db = db;
    this.maxRetries = options.maxRetries || 3;
    this.retryDelay = options.retryDelay || 100;
  }

  /**
   * 乐观锁扣减库存
   *
   * @param {string} productId - 商品ID
   * @param {number} quantity - 扣减数量
   * @returns {Promise<boolean>} - 成功返回true
   */
  async deductStock(productId, quantity) {
    for (let retry = 0; retry < this.maxRetries; retry++) {
      try {
        // 1. 查询当前库存和版本号
        const [rows] = await this.db.execute(
          'SELECT stock, version FROM inventory WHERE product_id = ?',
          [productId]
        );

        if (rows.length === 0) {
          throw new Error(`商品不存在: ${productId}`);
        }

        const currentStock = rows[0].stock;
        const currentVersion = rows[0].version;

        // 2. 检查库存是否充足
        if (currentStock < quantity) {
          throw new Error('库存不足');
        }

        // 3. 尝试更新库存（条件是版本号未变化）
        const [result] = await this.db.execute(
          'UPDATE inventory SET stock = stock - ?, version = version + 1 WHERE product_id = ? AND version = ?',
          [quantity, productId, currentVersion]
        );

        // 4. 检查更新是否成功（affectedRows = 1表示成功，0表示版本号已变化）
        if (result.affectedRows === 1) {
          console.log(`[OptimisticLock] 库存扣减成功: ${productId}, 扣减: ${quantity}`);
          return true;
        }

        console.log(`[OptimisticLock] 版本号冲突，重试: ${retry + 1}/${this.maxRetries}`);

        // 5. 版本号冲突，等待后重试
        await new Promise(resolve => setTimeout(resolve, this.retryDelay));

      } catch (error) {
        if (error.message.includes('库存不足') || error.message.includes('不存在')) {
          throw error;
        }
        console.error(`[OptimisticLock] 执行异常: ${error.message}`);
      }
    }

    throw new Error('库存扣减失败：并发冲突过多');
  }

  /**
   * 乐观锁扣减余额（通用实现）
   *
   * @param {string} tableName - 表名
   * @param {string} idColumn - ID列名
   * @param {string} idValue - ID值
   * @param {string} amountColumn - 金额列名
   * @param {number} amount - 扣减金额
   * @returns {Promise<boolean>}
   */
  async deductBalance(tableName, idColumn, idValue, amountColumn, amount) {
    for (let retry = 0; retry < this.maxRetries; retry++) {
      const [rows] = await this.db.execute(
        `SELECT ${amountColumn}, version FROM ${tableName} WHERE ${idColumn} = ?`,
        [idValue]
      );

      if (rows.length === 0) {
        throw new Error(`${tableName}记录不存在`);
      }

      const currentAmount = rows[0][amountColumn];
      const currentVersion = rows[0].version;

      if (currentAmount < amount) {
        throw new Error('余额不足');
      }

      const [result] = await this.db.execute(
        `UPDATE ${tableName} SET ${amountColumn} = ${amountColumn} - ?, version = version + 1 WHERE ${idColumn} = ? AND version = ?`,
        [amount, idValue, currentVersion]
      );

      if (result.affectedRows === 1) {
        return true;
      }

      await new Promise(resolve => setTimeout(resolve, this.retryDelay));
    }

    throw new Error('余额扣减失败：并发冲突过多');
  }
}
```

## 8. 分布式锁常见问题与解决方案

### 8.1 超时问题

```javascript
/**
 * 问题描述：
 * 业务逻辑执行时间超过锁的TTL，导致锁自动释放，
 * 其他进程获取锁后，两个进程同时操作共享资源
 */

class TimeoutProblem {
  constructor(lock) {
    this.lock = lock;
  }

  /**
   * 问题场景
   */
  async problematicScenario() {
    const token = await this.lock.acquire('resource:001', 5000); // 5秒TTL

    if (!token) {
      throw new Error('获取锁失败');
    }

    try {
      // 业务逻辑需要8秒
      console.log('开始执行耗时任务...');
      await this.longRunningTask(); // 8秒

    } finally {
      await this.lock.release('resource:001', token);
    }
  }

  /**
   * 解决方案1：设置合理的TTL
   *
   * 需要准确评估业务执行时间，并留有余量
   */
  async solution1_ProperTTL() {
    // 评估：业务逻辑最多需要10秒
    // TTL设置为30秒，留2倍余量
    const estimatedMaxDuration = 10000; // 10秒
    const safetyMargin = 3;
    const ttl = estimatedMaxDuration * safetyMargin; // 30秒

    const token = await this.lock.acquire('resource:001', ttl);

    try {
      await this.longRunningTask(); // 10秒内完成
    } finally {
      await this.lock.release('resource:001', token);
    }
  }

  /**
   * 解决方案2：Watchdog自动续期
   *
   * 业务执行期间自动续期，不受TTL限制
   */
  async solution2_Watchdog() {
    // 使用带Watchdog的锁
    const lock = new RedisLockWithWatchdog(redis);
    const ttl = 10000; // 基础TTL 10秒

    await lock.withLock('resource:001', async () => {
      // Watchdog会自动续期，直到业务完成
      await this.longRunningTask(); // 可以超过10秒
    }, ttl);
  }

  /**
   * 解决方案3：分段锁 + 状态记录
   *
   * 将长任务拆分为多个短任务，每个任务独立获取锁
   */
  async solution3_SegmentedLock() {
    const taskId = 'task-001';
    const totalSteps = 10;
    const stepTTL = 5000; // 每步5秒

    for (let step = 1; step <= totalSteps; step++) {
      const token = await this.lock.acquire(`${taskId}:step:${step}`, stepTTL);

      if (!token) {
        throw new Error(`步骤${step}获取锁失败`);
      }

      try {
        console.log(`执行步骤 ${step}/${totalSteps}`);
        await this.executeStep(step);

        // 记录进度
        await redis.set(`progress:${taskId}`, step.toString());

      } finally {
        await this.lock.release(`${taskId}:step:${step}`, token);
      }
    }
  }

  async longRunningTask() {
    await new Promise(resolve => setTimeout(resolve, 8000));
  }

  async executeStep(step) {
    await new Promise(resolve => setTimeout(resolve, 1000));
  }
}
```

### 8.2 脑裂问题

```javascript
/**
 * 问题描述：
 * 分布式系统中，由于网络分区等原因，
 * 可能出现多个节点都认为自己是锁的持有者的情况
 */

class SplitBrainProblem {
  /**
   * 问题场景：Redis主从切换导致脑裂
   *
   * 1. 进程A在主节点获取锁
   * 2. 主节点宕机，从节点升级为主节点
   * 3. 从节点没有锁数据（复制延迟）
   * 4. 进程B在新主节点获取同一个锁
   * 5. 进程A和进程B都认为持有锁
   */

  /**
   * 解决方案1：Redlock多节点
   *
   * 使用多个独立的Redis节点，只有超过半数成功才认为获取成功
   */
  async solution1_Redlock() {
    const redlock = new Redlock(redisClients);

    // 需要5个节点中至少3个成功才算获取成功
    // 即使2个节点宕机，也不会导致脑裂
    const lockInfo = await redlock.lock('resource:001', 10000);

    if (lockInfo) {
      // 执行业务逻辑
      await this.doWork();
      await redlock.unlock('resource:001', lockInfo);
    }
  }

  /**
   * 解决方案2：Fencing Token（栅栏令牌）
   *
   * 每次获取锁时分配一个递增的令牌
   * 业务操作时需要携带令牌，资源管理器拒绝旧令牌的请求
   */
  async solution2_FencingToken() {
    class FencingTokenLock {
      constructor(redis) {
        this.redis = redis;
      }

      async acquire(key) {
        // 使用INCR生成递增令牌
        const token = await this.redis.incr(`token:${key}`);
        return token.toString();
      }

      async doWithFencing(key, callback) {
        const token = await this.acquire(key);

        // 将令牌传递给业务操作
        return await callback(token);
      }
    }

    const lock = new FencingTokenLock(redis);
    const resourceManager = new ResourceManager();

    // 正确的使用方式
    await lock.doWithFencing('resource:001', async (token) => {
      // 执行业务操作前，先检查令牌是否有效
      const granted = await resourceManager.access(token, async () => {
        // 实际的操作逻辑
        await this.doWork();
      });

      if (!granted) {
        throw new Error('操作被拒绝：检测到锁冲突');
      }
    });
  }
}

/**
 * 资源管理器：使用Fencing Token防止脑裂
 */
class ResourceManager {
  constructor() {
    // 记录已处理的最新令牌
    this.lastAcceptedToken = 0;
  }

  /**
   * 尝试访问资源
   *
   * @param {number} token - 栅栏令牌
   * @param {Function} operation - 要执行的操作
   * @returns {Promise<boolean>} - 令牌有效返回true
   */
  async access(token, operation) {
    // 将token转为整数比较
    const tokenNum = parseInt(token, 10);

    // 拒绝旧令牌的请求（防止脑裂）
    if (tokenNum <= this.lastAcceptedToken) {
      console.warn(`[ResourceManager] 拒绝旧令牌: ${tokenNum} <= ${this.lastAcceptedToken}`);
      return false;
    }

    // 更新最新令牌
    this.lastAcceptedToken = tokenNum;

    // 执行操作
    await operation();

    return true;
  }
}
```

### 8.3 锁失效问题

```javascript
/**
 * 问题描述：
 * 锁失效可能由多种原因导致：
 * 1. 进程崩溃，未能释放锁
 * 2. Redis故障
 * 3. 网络抖动
 * 4. 锁自动过期
 */

class LockFailureScenarios {
  /**
   * 问题场景1：进程崩溃
   */
  async scenario1_ProcessCrash() {
    const token = await redis.set('lock:test', '1', 'NX', 'EX', 10);

    if (token === 'OK') {
      // 获取锁成功

      // 进程在这里突然崩溃
      // 没有执行任何清理代码
      // 锁会在10秒后自动释放

      // 如果崩溃前已经开始写数据，会导致数据不一致
      await this.writePartialData();
      process.exit(1); // 崩溃
    }
  }

  /**
   * 解决方案1：确保数据一致性
   */
  async solution1_DataConsistency() {
    // 使用Redis事务或Lua脚本保证原子性
    const luaScript = `
      local key = KEYS[1]
      local value = ARGV[1]
      local ttl = ARGV[2]

      -- 原子性地设置锁和初始状态
      redis.call('SET', key, value, 'PX', ttl)
      redis.call('HSET', key .. ':state', 'status', 'processing')

      return 'OK'
    `;

    await redis.eval(luaScript, 1, 'lock:test', '1', 10000);

    try {
      await this.doWork();
    } catch (error) {
      // 标记为失败
      await redis.hset('lock:test:state', 'status', 'failed');
      throw error;
    }

    // 完成后清理
    await redis.del('lock:test');
    await redis.del('lock:test:state');
  }

  /**
   * 问题场景2：Redis单点故障
   */
  async scenario2_RedisFailure() {
    // 主节点获取锁
    const token = await masterRedis.set('lock:test', '1', 'NX', 'EX', 10);

    if (token === 'OK') {
      // 写操作
      await this.writeData();

      // 主节点宕机！
      // 从节点可能没有这条数据
      // 锁丢失，新主节点可以获取同一个锁

      await this.moreWrites(); // 这里可能与新获取锁的进程冲突
    }
  }

  /**
   * 解决方案2：Redlock + 数据校验
   */
  async solution2_RedlockWithValidation() {
    const redlock = new Redlock(redisClients);
    const lockInfo = await redlock.lock('test:operation', 10000);

    if (lockInfo) {
      try {
        // 执行业务逻辑
        await this.doWork();

        // 使用乐观锁验证数据一致性
        const isValid = await this.validateDataConsistency();

        if (!isValid) {
          throw new Error('数据一致性校验失败');
        }

      } finally {
        await redlock.unlock('test:operation', lockInfo);
      }
    }
  }

  /**
   * 问题场景3：锁过期，业务仍在执行
   */
  async scenario3_LockExpiryDuringExecution() {
    // TTL设置为5秒
    const token = await redis.set('lock:order:123', '1', 'NX', 'PX', 5000);

    if (token === 'OK') {
      // 5秒后锁自动过期

      // 但业务逻辑需要15秒
      // 第6秒时，其他进程获取了锁

      // 10秒时，原进程还在执行，已经释放锁
      // 15秒时，原进程完成

      // 结果：两个进程都认为自己成功执行了
      await this.executeLongTask(); // 15秒
```

### 8.4 Race Condition（竞态条件）

```javascript
/**
 * 问题描述：
 * 竞态条件是指多个进程对共享资源的访问顺序依赖于程序的执行时序
 */

class RaceConditionProblems {

  /**
   * 问题场景：检查-执行-写入（Check-Then-Act）
   *
   * 最经典的竞态条件
   */
  async problem_CheckThenAct() {
    // 步骤1：检查库存（读取）
    const stock = await redis.get('stock:product:001');
    // stock = 10

    // 注意：这里可能发生上下文切换
    // 另一个进程也在检查库存

    // 步骤2：检查是否充足
    if (parseInt(stock) < 1) {
      throw new Error('库存不足');
    }

    // 步骤3：扣减库存（写入）
    // 此时另一个进程也执行到了这里
    // 两个进程都认为库存充足，都扣减了库存
    await redis.decr('stock:product:001');
  }

  /**
   * 解决方案1：使用锁保护整个检查-执行过程
   */
  async solution1_UseLock() {
    const lock = new RedisLock(redis);

    await lock.withLock('stock:product:001', async () => {
      // 检查和扣减在锁的保护下执行
      const stock = await redis.get('stock:product:001');

      if (parseInt(stock) < 1) {
        throw new Error('库存不足');
      }

      await redis.decr('stock:product:001');
    });
  }

  /**
   * 解决方案2：使用Lua脚本原子操作
   */
  async solution2_LuaScript() {
    const luaScript = `
      local stock = redis.call('GET', KEYS[1])
      if not stock then
        return -1  -- 商品不存在
      end

      stock = tonumber(stock)
      if stock < 1 then
        return 0  -- 库存不足
      end

      redis.call('DECR', KEYS[1])
      return 1  -- 扣减成功
    `;

    const result = await redis.eval(luaScript, 1, 'stock:product:001');

    if (result === 0) {
      throw new Error('库存不足');
    } else if (result === -1) {
      throw new Error('商品不存在');
    }
  }

  /**
   * 问题场景2：先到先得（Last Write Wins）
   *
   * 多个进程同时更新同一资源，最终结果取决于谁最后写入
   */
  async problem_LastWriteWins() {
    // 进程A读取用户信息
    const userA = await db.query('SELECT * FROM users WHERE id = ?', [1]);
    // userA.balance = 100

    // 进程B读取用户信息
    const userB = await db.query('SELECT * FROM users WHERE id = ?', [1]);
    // userB.balance = 100

    // 进程A：加50元
    await db.query('UPDATE users SET balance = 150 WHERE id = 1');

    // 进程B：加30元（覆盖了进程A的操作）
    await db.query('UPDATE users SET balance = 130 WHERE id = 1');

    // 最终余额是130，丢失了进程A的50元
  }

  /**
   * 解决方案3：乐观锁 + 重试
   */
  async solution3_OptimisticLock() {
    const maxRetries = 3;

    for (let retry = 0; retry < maxRetries; retry++) {
      // 读取当前余额和版本号
      const [user] = await db.query('SELECT balance, version FROM users WHERE id = ?', [1]);
      const { balance, version } = user;

      // 计算新余额
      const newBalance = balance + 30;

      // 尝试更新（检查版本号）
      const [result] = await db.query(
        'UPDATE users SET balance = ?, version = version + 1 WHERE id = ? AND version = ?',
        [newBalance, 1, version]
      );

      if (result.affectedRows === 1) {
        console.log('更新成功');
        return;
      }

      console.log(`版本冲突，重试: ${retry + 1}/${maxRetries}`);
      await new Promise(resolve => setTimeout(resolve, 100));
    }

    throw new Error('更新失败：并发冲突过多');
  }

  /**
   * 问题场景3：时序依赖（Order Dependency）
   *
   * 操作的成功与否依赖于其他操作的执行顺序
   */
  async problem_OrderDependency() {
    // 场景：先创建订单，再扣库存

    // 错误顺序：
    // 1. 扣库存成功
    // 2. 创建订单失败
    // 结果：库存扣了，但订单没创建

    // 正确顺序：
    // 1. 创建订单
    // 2. 扣库存
    // 3. 如果扣库存失败，回滚订单
  }

  /**
   * 解决方案4：分布式事务或补偿机制
   */
  async solution4_DistributedTransaction() {
    const saga = new Saga();

    // 定义每个步骤及其补偿操作
    saga.addStep({
      name: 'createOrder',
      execute: async () => {
        await db.query('INSERT INTO orders (user_id, product_id) VALUES (?, ?)', [1, 'SKU001']);
      },
      compensate: async () => {
        await db.query('DELETE FROM orders WHERE user_id = ? AND product_id = ?', [1, 'SKU001']);
      }
    });

    saga.addStep({
      name: 'deductStock',
      execute: async () => {
        await redis.decr('stock:SKU001');
      },
      compensate: async () => {
        await redis.incr('stock:SKU001');
      }
    });

    saga.addStep({
      name: 'sendNotification',
      execute: async () => {
        await sendEmail();
      },
      compensate: async () => {
        // 邮件无法撤回，记录日志人工处理
        await log('COMPENSATION_REQUIRED: sendNotification');
      }
    });

    // 执行saga
    await saga.execute();
  }
}

/**
 * 简单的Saga实现
 */
class Saga {
  constructor() {
    this.steps = [];
    this.executedSteps = [];
  }

  addStep(step) {
    this.steps.push(step);
  }

  async execute() {
    for (const step of this.steps) {
      try {
        console.log(`执行步骤: ${step.name}`);
        await step.execute();
        this.executedSteps.push(step);
      } catch (error) {
        console.error(`步骤失败: ${step.name}, 错误: ${error.message}`);
        await this.compensate();
        throw error;
      }
    }
  }

  async compensate() {
    // 逆序执行补偿操作
    for (const step of this.executedSteps.reverse()) {
      try {
        console.log(`执行补偿: ${step.name}`);
        await step.compensate();
      } catch (error) {
        console.error(`补偿失败: ${step.name}, 错误: ${error.message}`);
        // 记录日志，后续人工处理
      }
    }
  }
}
```

### 9.2 Redis分布式锁实现秒杀

```javascript
/**
 * 基于Redis分布式锁的秒杀系统
 */

const Redis = require('ioredis');

class SeckillSystem {
  constructor() {
    this.redis = new Redis({ host: 'localhost', port: 6379 });
    this.lockPrefix = 'seckill:lock:';
    this.stockPrefix = 'seckill:stock:';
    this.orderPrefix = 'seckill:order:';
    this.lockTtl = 5000; // 锁的TTL 5秒
  }

  /**
   * 初始化秒杀商品库存
   *
   * @param {string} productId - 商品ID
   * @param {number} stock - 初始库存
   */
  async initStock(productId, stock) {
    await this.redis.set(`${this.stockPrefix}${productId}`, stock);
    console.log(`[秒杀] 初始化商品 ${productId} 库存: ${stock}`);
  }

  /**
   * 获取分布式锁
   */
  async acquireLock(productId) {
    const lockKey = `${this.lockPrefix}${productId}`;
    const token = `${Date.now()}-${Math.random().toString(36).substring(2)}`;

    // SET NX PX 原子获取锁
    const result = await this.redis.set(lockKey, token, 'NX', 'PX', this.lockTtl);

    if (result === 'OK') {
      return token;
    }

    return null;
  }

  /**
   * 释放分布式锁
   */
  async releaseLock(productId, token) {
    const lockKey = `${this.lockPrefix}${productId}`;

    // Lua脚本保证原子性：只有持有锁的进程才能释放
    const luaScript = `
      if redis.call('GET', KEYS[1]) == ARGV[1] then
        return redis.call('DEL', KEYS[1])
      else
        return 0
      end
    `;

    const result = await this.redis.eval(luaScript, 1, lockKey, token);
    return result === 1;
  }

  /**
   * 秒杀请求处理
   *
   * @param {string} userId - 用户ID
   * @param {string} productId - 商品ID
   * @param {number} quantity - 购买数量
   * @returns {Promise<{success: boolean, message: string}>}
   */
  async seckill(userId, productId, quantity = 1) {
    const lockKey = `${this.lockPrefix}${productId}`;
    const stockKey = `${this.stockPrefix}${productId}`;
    const orderKey = `${this.orderPrefix}${productId}:${userId}`;

    // 1. 检查用户是否已经购买过（幂等性检查）
    const existingOrder = await this.redis.get(orderKey);
    if (existingOrder) {
      return { success: false, message: '您已经参与过本次秒杀' };
    }

    // 2. 获取分布式锁
    const token = await this.acquireLock(productId);

    if (!token) {
      return { success: false, message: '系统繁忙，请稍后重试' };
    }

    try {
      // 3. 检查库存
      const stock = await this.redis.get(stockKey);
      const currentStock = parseInt(stock, 10);

      if (currentStock < quantity) {
        return { success: false, message: '库存不足' };
      }

      // 4. 扣减库存
      await this.redis.decrby(stockKey, quantity);

      // 5. 创建订单（标记用户已购买）
      await this.redis.setex(orderKey, 86400, JSON.stringify({
        userId,
        productId,
        quantity,
        createTime: Date.now()
      }));

      console.log(`[秒杀] 用户 ${userId} 秒杀成功，商品 ${productId}，剩余库存: ${currentStock - quantity}`);

      return {
        success: true,
        message: '秒杀成功',
        data: {
          orderId: `${productId}-${userId}-${Date.now()}`
        }
      };

    } finally {
      // 6. 释放锁
      await this.releaseLock(productId, token);
    }
  }

  /**
   * 使用Lua脚本优化（减少网络往返）
   *
   * 将检查库存、扣减库存、下单合并为一个原子操作
   */
  async seckillOptimized(userId, productId, quantity = 1) {
    const stockKey = `${this.stockPrefix}${productId}`;
    const orderKey = `${this.orderPrefix}${productId}:${userId}`;

    // Lua脚本：原子性完成检查库存、扣减、下单
    const luaScript = `
      local stockKey = KEYS[1]
      local orderKey = KEYS[2]
      local quantity = tonumber(ARGV[1])
      local userId = ARGV[2]
      local orderExpire = tonumber(ARGV[3])

      -- 检查是否已购买
      local existing = redis.call('GET', orderKey)
      if existing then
        return {-1, 'ALREADY_PURCHASED'}
      end

      -- 检查库存
      local stock = redis.call('GET', stockKey)
      if not stock then
        return {-1, 'PRODUCT_NOT_FOUND'}
      end

      stock = tonumber(stock)
      if stock < quantity then
        return {0, 'STOCK_NOT_ENOUGH'}
      end

      -- 扣减库存
      redis.call('DECRBY', stockKey, quantity)

      -- 创建订单
      local orderData = cjson.encode({
        userId = userId,
        quantity = quantity,
        createTime = redis.call('TIME')[1]
      })
      redis.call('SETEX', orderKey, orderExpire, orderData)

      -- 返回新的库存数量
      return {1, stock - quantity}
    `;

    const result = await this.redis.eval(
      luaScript,
      2,
      stockKey,
      orderKey,
      quantity,
      userId,
      86400 // 订单24小时过期
    );

    const [success, data] = result;

    if (success === 1) {
      return {
        success: true,
        message: '秒杀成功',
        remainingStock: data
      };
    } else if (data === 'ALREADY_PURCHASED') {
      return { success: false, message: '您已经参与过本次秒杀' };
    } else if (data === 'STOCK_NOT_ENOUGH') {
      return { success: false, message: '库存不足' };
    } else {
      return { success: false, message: '秒杀失败' };
    }
  }

  /**
   * 获取秒杀结果
   */
  async getSeckillResult(userId, productId) {
    const orderKey = `${this.orderPrefix}${productId}:${userId}`;
    const order = await this.redis.get(orderKey);

    if (order) {
      return {
        success: true,
        message: '秒杀成功',
        order: JSON.parse(order)
      };
    }

    // 检查库存
    const stock = await this.redis.get(`${this.stockPrefix}${productId}`);

    if (parseInt(stock, 10) <= 0) {
      return {
        success: false,
        message: '秒杀已结束，库存已售罄'
      };
    }

    return {
      success: false,
      message: '正在排队，请稍后查询结果'
    };
  }

  /**
   * 防止黄牛（限流）
   */
  async rateLimit(userId, maxRequests = 5, windowSeconds = 60) {
    const key = `ratelimit:seckill:${userId}`;

    // 使用Redis计数器的滑动窗口
    const current = await this.redis.incr(key);

    if (current === 1) {
      // 第一次请求，设置过期时间
      await this.redis.expire(key, windowSeconds);
    }

    if (current > maxRequests) {
      return false; // 超过限制
    }

    return true; // 未超过限制
  }
}
```

```javascript
    const [product] = await this.db.query(
      `SELECT * FROM seckill_products
       WHERE id = ? AND seckill_start_time <= NOW() AND seckill_end_time >= NOW()`,
      [productId]
    );

    if (!product) {
      throw new Error('秒杀活动不存在或已结束');
    }

    // 2. 检查用户是否已购买
    const [existingOrder] = await this.db.query(
      'SELECT id FROM seckill_orders WHERE user_id = ? AND product_id = ?',
      [userId, productId]
    );

    if (existingOrder) {
      throw new Error('您已经参与过本次秒杀');
    }

    // 3. 乐观锁更新库存（带重试）
    for (let retry = 0; retry < this.maxRetries; retry++) {
      try {
        // 查询当前库存
        const [currentProduct] = await this.db.query(
          'SELECT stock, version FROM seckill_products WHERE id = ?',
          [productId]
        );

        const currentStock = currentProduct.stock;
        const currentVersion = currentProduct.version;

        // 检查库存
        if (currentStock < quantity) {
          throw new Error('库存不足');
        }

        // 乐观锁更新：WHERE条件包含version
        const [result] = await this.db.query(
          `UPDATE seckill_products
           SET stock = stock - ?, version = version + 1
           WHERE id = ? AND version = ? AND stock >= ?`,
          [quantity, productId, currentVersion, quantity]
        );

        if (result.affectedRows === 1) {
          // 更新成功
          console.log(`[乐观锁] 用户 ${userId} 扣减库存成功，剩余: ${currentStock - quantity}`);

          // 4. 创建订单
          await this.createOrder(productId, userId, quantity);

          return {
            success: true,
            message: '秒杀成功',
            remainingStock: currentStock - quantity
          };
        }

        // 版本号不匹配，说明有并发更新，重试
        console.log(`[乐观锁] 版本冲突，重试: ${retry + 1}/${this.maxRetries}`);

        // 短暂等待后重试
        await new Promise(resolve => setTimeout(resolve, this.retryDelay));

      } catch (error) {
        if (error.message === '库存不足') {
          throw error;
        }
        console.error(`[乐观锁] 执行异常: ${error.message}`);
      }
    }

    throw new Error('秒杀失败：并发竞争激烈，请重试');
  }
```

### 10.1 什么是幂等性

```javascript
/**
 * 幂等性（Idempotency）
 *
 * 定义：一个操作执行一次和执行多次的结果是一样的，不会产生副作用
 *
 * 例子：
 * - GET请求是幂等的（不会改变数据）
 * - PUT请求是幂等的（覆盖同一资源）
 * - DELETE请求是幂等的（删除一次和删除多次结果一样）
 * - POST请求不是幂等的（每次创建新资源）
 *
 * 为什么需要幂等性：
 * 1. 网络抖动导致请求超时
 * 2. 客户端重复点击
 * 3. 服务端处理超时，客户端重试
 * 4. 消息队列重复消费
 */

const idempotencyExamples = {
  // 幂等操作示例
  idempotent: {
    'GET /users/1': '获取用户信息，多次获取结果相同',
    'PUT /users/1': '更新用户信息，覆盖为相同数据，结果相同',
    'DELETE /users/1': '删除用户，删除一次和多次结果都是用户不存在',
    'POST /users/1/balance -30': '扣款30元，多次扣款结果不同（非幂等）'
  },

  // 非幂等操作示例
  nonIdempotent: {
    'POST /orders': '创建订单，每次创建新订单',
    'POST /payments': '支付，每次创建新的支付记录',
    'POST /users/1/balance +30': '存款，多次存款结果不同'
  }
};
```

### 10.2 Token幂等

```javascript
/**
 * 基于Token的幂等性实现
 *
 * 原理：
 * 1. 客户端在请求前向服务端获取一个唯一的token
 * 2. 客户端在后续请求中携带这个token
 * 3. 服务端使用Redis记录已处理的token
 * 4. 如果token已存在，说明请求已处理，直接返回结果
 * 5. 如果token不存在，处理请求并记录token
 */

class TokenIdempotency {
  constructor(redis) {
    this.redis = redis;
    this.tokenTtl = 3600; // Token有效期1小时
  }

  /**
   * 生成幂等Token
   *
   * @param {string} userId - 用户ID
   * @param {string} action - 操作类型
   * @returns {Promise<string>} - 返回Token
   */
  async generateToken(userId, action) {
    const token = `${userId}:${action}:${Date.now()}:${Math.random().toString(36).substring(2)}`;

    // 存储token，标记为未使用
    const key = `idempotent:pending:${token}`;
    await this.redis.setex(key, this.tokenTtl, 'pending');

    return token;
  }

  /**
   * 检查并标记Token为已使用
   *
   * @param {string} token - 幂等Token
   * @returns {Promise<{valid: boolean, cached: boolean, result: any}>}
   */
  async checkAndUseToken(token) {
    const key = `idempotent:pending:${token}`;
    const cachedKey = `idempotent:result:${token}`;

    // 1. 检查token是否存在（使用SET NX保证原子性）
    // 如果key不存在，设置值为used
    const isNew = await this.redis.set(key, 'used', 'NX');

    if (!isNew) {
      // token已被使用，检查是否有缓存结果
      const cached = await this.redis.get(cachedKey);

      if (cached) {
        return {
          valid: false,
          cached: true,
          result: JSON.parse(cached)
        };
      }

      // token存在但没有结果，说明正在处理中
      return {
        valid: false,
        cached: false,
        result: null
      };
    }

    // token是新分配的，可以继续处理
    return { valid: true, cached: false, result: null };
  }

  /**
   * 缓存处理结果
   *
   * @param {string} token - 幂等Token
   * @param {any} result - 处理结果
   */
  async cacheResult(token, result) {
    const cachedKey = `idempotent:result:${token}`;
    await this.redis.setex(cachedKey, this.tokenTtl, JSON.stringify(result));
  }

  /**
   * 执行幂等操作
   *
   * @param {string} token - 幂等Token
   * @param {Function} operation - 实际业务操作
   * @returns {Promise<any>}
   */
  async withIdempotency(token, operation) {
    // 1. 检查token
    const check = await this.checkAndUseToken(token);

    if (!check.valid) {
      if (check.cached) {
        console.log(`[幂等] 使用缓存结果: ${token}`);
        return check.result;
      }

      throw new Error('请求正在处理中，请勿重复提交');
    }

    try {
      // 2. 执行实际业务操作
      const result = await operation();

      // 3. 缓存结果
      await this.cacheResult(token, result);

      return result;

    } catch (error) {
      // 业务失败时，可以选择是否删除token让客户端重试
      // 这里选择保留token，防止重复提交
      throw error;
    }
  }
}

// 使用示例
async function tokenIdempotencyExample() {
  const redis = new Redis();
  const idempotency = new TokenIdempotency(redis);

  // 场景：用户点击多次提交订单
  const userId = 'USER-001';
  const action = 'CREATE_ORDER';

  // 1. 获取幂等token
  const token = await idempotency.generateToken(userId, action);
  console.log(`获取幂等Token: ${token}`);

  // 2. 提交订单（携带token）
  try {
    const result = await idempotency.withIdempotency(token, async () => {
      console.log('创建订单...');
      await new Promise(resolve => setTimeout(resolve, 100));
      return { orderId: 'ORD-123', status: 'SUCCESS' };
    });

    console.log('订单创建成功:', result);

    // 3. 模拟重复提交（使用相同的token）
    const duplicateResult = await idempotency.withIdempotency(token, async () => {
      console.log('这行不会执行');
      return { orderId: 'ORD-456' };
    });

    console.log('重复提交结果:', duplicateResult); // 应该返回第一次的结果

  } catch (error) {
    console.error('操作失败:', error.message);
  }
}
```

### 10.3 唯一键幂等

```javascript
/**
 * 基于唯一键的幂等性实现
 *
 * 原理：
 * 1. 为操作定义一个唯一键（如：userId + action + businessId）
 * 2. 使用数据库唯一索引或Redis SET NX保证唯一性
 * 3. 操作前尝试插入唯一键，操作后更新业务状态
 */

// 基于Redis的唯一键幂等
class UniqueKeyIdempotency {
  constructor(redis) {
    this.redis = redis;
    this.defaultTtl = 86400; // 24小时
  }

  /**
   * 生成唯一键
   */
  generateUniqueKey(userId, action, businessId = '') {
    return `idempotent:${action}:${userId}:${businessId}`;
  }

  /**
   * 尝试获取幂等锁
   *
   * @returns {Promise<{acquired: boolean, isDuplicate: boolean}>}
   */
  async tryAcquire(userId, action, businessId = '') {
    const key = this.generateUniqueKey(userId, action, businessId);

    // SET NX保证原子性
    // 值包含时间戳，用于判断是否为重复提交
    const result = await this.redis.set(
      key,
      Date.now().toString(),
      'NX',
      'EX',
      this.defaultTtl
    );

    if (result === 'OK') {
      // 成功获取，说明是第一次请求
      return { acquired: true, isDuplicate: false };
    }

    // 获取失败，检查是否重复提交
    const existingValue = await this.redis.get(key);
    const existingTime = parseInt(existingValue, 10);
    const timeDiff = Date.now() - existingTime;

    if (timeDiff < 1000) {
      // 1秒内的重复请求认为是重复提交
      return { acquired: false, isDuplicate: true };
    }

    // 老请求已过期（可能被删除了）
    // 允许重新尝试
    return { acquired: false, isDuplicate: false };
  }

  /**
   * 释放幂等锁（通常不需要主动调用）
   */
  async release(userId, action, businessId = '') {
    const key = this.generateUniqueKey(userId, action, businessId);
    await this.redis.del(key);
  }

  /**
   * 执行基于唯一键的幂等操作
   */
  async withUniqueKeyIdempotency(userId, action, businessId, operation) {
    const { acquired, isDuplicate } = await this.tryAcquire(userId, action, businessId);

    if (isDuplicate) {
      throw new Error('请勿重复提交请求');
    }

    if (!acquired) {
      throw new Error('系统繁忙，请稍后重试');
    }

    try {
      return await operation();
    } catch (error) {
      // 操作失败时可以选择释放锁
      // 但通常建议保留，防止重复提交
      throw error;
    }
  }
}

// 使用示例
async function uniqueKeyIdempotencyExample() {
  const redis = new Redis();
  const idempotency = new UniqueKeyIdempotency(redis);

  const userId = 'USER-001';
  const action = 'UPDATE_PROFILE';
  const profileId = 'PROFILE-123';

  try {
    // 第一次更新
    await idempotency.withUniqueKeyIdempotency(
      userId,
      action,
      profileId,
      async () => {
        console.log('执行更新操作...');
        return { success: true, message: '更新成功' };
      }
    );

    // 第二次更新（相同参数）
    await idempotency.withUniqueKeyIdempotency(
      userId,
      action,
      profileId,
      async () => {
        console.log('这行不会执行');
        return { success: false };
      }
    );

  } catch (error) {
    console.log('捕获错误:', error.message); // 会显示"请勿重复提交请求"
  }
}
```

### 10.4 去重表幂等

```javascript
/**
 * 基于数据库去重表的幂等性实现
 *
 * 原理：
 * 1. 创建一个去重表，记录已处理的请求
 * 2. 请求处理前先插入去重表（使用唯一索引防止重复）
 * 3. 处理成功后更新状态
 * 4. 查询时检查状态判断是否已处理
 */

const deduplicationTableSchema = `
  -- 去重表
  CREATE TABLE IF NOT EXISTS idempotent_dedup (
    idempotent_key VARCHAR(255) PRIMARY KEY,
    status ENUM('PENDING', 'PROCESSING', 'COMPLETED', 'FAILED') DEFAULT 'PENDING',
    result JSON,
    error_message TEXT,
    retry_count INT DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_status (status),
    INDEX idx_created (created_at)
  ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

  -- 业务表（以订单为例）
  CREATE TABLE IF NOT EXISTS orders (
    id VARCHAR(64) PRIMARY KEY,
    idempotent_key VARCHAR(255) UNIQUE,
    user_id VARCHAR(64) NOT NULL,
    product_id VARCHAR(64) NOT NULL,
    quantity INT NOT NULL,
    status VARCHAR(32) DEFAULT 'CREATED',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
  ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
`;

class DeduplicationTableIdempotency {
  constructor(db) {
    this.db = db;
  }

  /**
   * 生成幂等key
   */
  generateKey(userId, action, businessId = '') {
    return `${action}:${userId}:${businessId}`;
  }

  /**
   * 尝试获取幂等执行权
   *
   * @returns {Promise<{canExecute: boolean, status: string, cachedResult: any}>}
   */
  async tryAcquire(key) {
    // 1. 先查询是否已存在
    const [existing] = await this.db.query(
      'SELECT status, result FROM idempotent_dedup WHERE idempotent_key = ?',
      [key]
    );

    if (existing) {
      // 已存在，根据状态判断
      if (existing.status === 'COMPLETED') {
        return {
          canExecute: false,
          status: 'COMPLETED',
          cachedResult: JSON.parse(existing.result || '{}')
        };
      }

      if (existing.status === 'PROCESSING') {
        return {
          canExecute: false,
          status: 'PROCESSING',
          cachedResult: null
        };
      }

      if (existing.status === 'FAILED') {
        // 失败后可以重试，更新状态
        await this.db.query(
          'UPDATE idempotent_dedup SET status = ?, retry_count = retry_count + 1 WHERE idempotent_key = ?',
          ['PROCESSING', key]
        );

        return {
          canExecute: true,
          status: 'RETRY',
          cachedResult: null
        };
      }
    }

    // 2. 插入新记录（使用INSERT IGNORE或ON DUPLICATE KEY）
    try {
      await this.db.query(
        'INSERT INTO idempotent_dedup (idempotent_key, status) VALUES (?, ?) ON DUPLICATE KEY UPDATE status = ?',
        [key, 'PROCESSING', 'PROCESSING']
      );

      return { canExecute: true, status: 'NEW', cachedResult: null };

    } catch (error) {
      // 插入失败，说明并发请求已经插入了
      return { canExecute: false, status: 'PROCESSING', cachedResult: null };
    }
  }

  /**
   * 标记为完成
   */
  async markCompleted(key, result) {
    await this.db.query(
      'UPDATE idempotent_dedup SET status = ?, result = ? WHERE idempotent_key = ?',
      ['COMPLETED', JSON.stringify(result), key]
    );
  }

  /**
   * 标记为失败
   */
  async markFailed(key, errorMessage) {
    await this.db.query(
      'UPDATE idempotent_dedup SET status = ?, error_message = ? WHERE idempotent_key = ?',
      ['FAILED', errorMessage, key]
    );
  }

  /**
   * 执行幂等操作
   */
  async withIdempotency(key, operation) {
    const { canExecute, status, cachedResult } = await this.tryAcquire(key);

    if (!canExecute) {
      if (status === 'COMPLETED') {
        console.log(`[幂等] 使用缓存结果: ${key}`);
        return cachedResult;
      }

      throw new Error('请求正在处理中，请稍后重试');
    }

    try {
      // 执行实际业务操作
      const result = await operation();

      // 标记完成
      await this.markCompleted(key, result);

      return result;

    } catch (error) {
      // 标记失败
      await this.markFailed(key, error.message);
      throw error;
    }
  }

  /**
   * 清理过期记录（定时任务）
   */
  async cleanupExpired(days = 7) {
    const sql = `
      DELETE FROM idempotent_dedup
      WHERE status IN ('COMPLETED', 'FAILED')
        AND updated_at < DATE_SUB(NOW(), INTERVAL ? DAY)
    `;

    const result = await this.db.query(sql, [days]);
    console.log(`[清理] 删除 ${result.affectedRows} 条过期记录`);
  }
}
```

### 11.1 锁的粒度设计

```javascript
/**
 * 锁粒度设计原则
 *
 * 锁的粒度越粗，并发性能越差，但实现越简单
 * 锁的粒度越细，并发性能越好，但实现越复杂

// 反面示例：锁粒度过粗
class BadLockGranularity {
  async processAllOrders() {

class GoodLockGranularity {
```

### 11.2 锁的获取顺序

```javascript
/**
 * 锁获取顺序原则
 *
 * 多个锁的情况下，要保证所有进程以相同的顺序获取锁
 * 否则可能导致死锁
 */

// 死锁示例：不同的获取顺序
async function deadlockExample() {
  // 进程A：先锁X，再锁Y
  async function processA() {
    await redis.set('lock:X', '1', 'NX', 'EX', 10);
    await redis.set('lock:Y', '1', 'NX', 'EX', 10);
  }

  // 进程B：先锁Y，再锁X
  async function processB() {
    await redis.set('lock:Y', '1', 'NX', 'EX', 10);
    await redis.set('lock:X', '1', 'NX', 'EX', 10);
  }

  // 如果A获取了X，B获取了Y，就会死锁
}

// 解决方案：统一获取顺序
async function fixedOrderExample() {
  // 所有进程都按字典顺序获取锁
  async function getLocks(keys) {
    // 先排序
    const sortedKeys = keys.sort();

    const acquired = [];
    try {
      for (const key of sortedKeys) {
        const lock = await redis.set(`lock:${key}`, '1', 'NX', 'EX', 10);
        if (lock === 'OK') {
          acquired.push(key);
        } else {
          throw new Error(`获取锁失败: ${key}`);
        }
      }
    } finally {
      // 逆序释放
      for (const key of acquired.reverse()) {
        await redis.del(`lock:${key}`);
      }
    }
  }

  // 使用
  await getLocks(['X', 'Y']); // 进程A
  await getLocks(['X', 'Y']); // 进程B - 不会死锁
}
```

```javascript
      } else {
```

```javascript
/**
 * Redis分布式锁完整实现
 * 支持：自动续期、可重入、Redlock
 */

const Redis = require('ioredis');

class CompleteRedisLock {
  constructor(redis, options = {}) {
    this.redis = redis;
    this.prefix = options.prefix || 'lock:';
    this.defaultTtl = options.ttl || 10000;
    this.watchdogInterval = options.watchdogInterval || 3000;
  }

  // 生成唯一token
  generateToken() {
    return `${process.pid}-${Date.now()}-${Math.random().toString(36).substring(2, 15)}`;
  }

  // 获取锁
  async acquire(key, ttl = this.defaultTtl) {
    const lockKey = `${this.prefix}${key}`;
    const token = this.generateToken();

    const result = await this.redis.set(lockKey, token, 'NX', 'PX', ttl);

    if (result === 'OK') {
      return { key, token };
    }

    return null;
  }

  // 释放锁
  async release(lockInfo) {
    if (!lockInfo) return false;

    const { key, token } = lockInfo;
    const lockKey = `${this.prefix}${key}`;

    const luaScript = `
      if redis.call('GET', KEYS[1]) == ARGV[1] then
        return redis.call('DEL', KEYS[1])
      else
        return 0
      end
    `;

    const result = await this.redis.eval(luaScript, 1, lockKey, token);
    return result === 1;
  }

  // 带锁执行
  async withLock(key, callback, ttl = this.defaultTtl) {
    const lockInfo = await this.acquire(key, ttl);

    if (!lockInfo) {
      throw new Error(`获取锁失败: ${key}`);
    }

    try {
      return await callback();
    } finally {
      await this.release(lockInfo);
    }
  }
}

module.exports = CompleteRedisLock;
```
