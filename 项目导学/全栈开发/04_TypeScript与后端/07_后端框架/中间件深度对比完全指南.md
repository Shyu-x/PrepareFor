## 1. 中间件概述

### 1.1 什么是中间件

**中间件**是一种软件设计模式，它位于操作系统、数据库、网络协议与应用程序之间，在请求处理流程中起到承上启下的作用。在 Web 开发语境下，中间件本质上一个函数，它接收请求对象（Request）、响应对象（Response），以及一个指向下一个中间件的函数指针（通常称为 `next`）。

```
┌─────────────────────────────────────────────────────────────────┐
│                        请求处理管道                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   客户端请求                                                      │
│       │                                                          │
│       ▼                                                          │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                    中间件 1                              │   │
│   │   - 请求日志记录                                          │   │
│   │   - 修改请求对象                                           │   │
│   │   - 调用 next()                                           │   │
│   └────────────────────────┬────────────────────────────────┘   │
│                            │                                     │
│                            ▼                                     │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                    中间件 2                              │   │
│   │   - 身份认证验证                                          │   │
│   │   - 权限检查                                              │   │
│   │   - 调用 next()                                           │   │
│   └────────────────────────┬────────────────────────────────┘   │
│                            │                                     │
│                            ▼                                     │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                    中间件 3                              │   │
│   │   - 参数验证                                             │   │
│   │   - 数据转换                                             │   │
│   │   - 调用 next()                                           │   │
│   └────────────────────────┬────────────────────────────────┘   │
│                            │                                     │
│                            ▼                                     │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                    路由处理器                             │   │
│   │   - 执行业务逻辑                                          │   │
│   │   - 返回响应                                              │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 中间件模式：洋葱模型

**洋葱模型（Onion Model）** 是中间件最具代表性的执行模型，尤其在 Koa 框架中得到了完美体现。这个模型之所以被称为"洋葱"，是因为中间件层层嵌套，请求从外层向内层穿透，响应从内层向外层返回，恰好就像剥洋葱一样一层层进去、一层层出来。

```
┌─────────────────────────────────────────────────────────────────┐
│                         洋葱模型                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│                         请求入口                                  │
│                            │                                     │
│                            ▼                                     │
│              ┌───────────────────────────────┐                  │
│              │         中间件 A（外层）        │                  │
│              │  ┌─────────────────────────┐  │                  │
│              │  │     中间件 B             │  │                  │
│              │  │  ┌───────────────────┐  │  │                  │
│              │  │  │   中间件 C（内层）  │  │  │                  │
│              │  │  │  ┌─────────────┐  │  │  │                  │
│              │  │  │  │  业务逻辑   │  │  │  │                  │
│              │  │  │  └─────────────┘  │  │  │                  │
│              │  │  │   中间件 C 后置   │  │  │                  │
│              │  │  └───────────────────┘  │  │                  │
│              │  │     中间件 B 后置       │  │                  │
│              │  └─────────────────────────┘  │                  │
│              │         中间件 A 后置         │                  │
│              └───────────────────────────────┘                  │
│                            │                                     │
│                            ▼                                     │
│                         响应出口                                  │
│                                                                  │
│   执行顺序：                                                      │
│   1. A 前置逻辑 → B 前置逻辑 → C 前置逻辑 → 业务逻辑 →           │
│   2. C 后置逻辑 → B 后置逻辑 → A 后置逻辑 → 响应                 │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**洋葱模型的核心优势在于：**

1. **清晰的切面分离**：每个中间件可以独立关注前置处理或后置处理
2. **灵活的流程控制**：通过 `await next()` 可以控制何时向下传递
3. **易于组合和复用**：中间件可以单独测试、单独复用
4. **统一的错误传播**：异常可以自动向外层传播

### 1.3 常见中间件类型

在典型的 Web 应用中，中间件承担着多种职责：

| 中间件类型 | 主要职责 | 代表实现 |
|-----------|---------|---------|
| **日志中间件** | 记录请求信息、响应时间、错误日志 | morgan、winston、pino |
| **认证中间件** | 验证用户身份、JWT解析、Session检查 | passport、jwt-simple |
| **限流中间件** | 控制请求频率、防爬虫、DDoS防护 | express-rate-limit、koa-ratelimit |
| **缓存中间件** | 页面缓存、数据缓存、CDN缓存 | node-cache、redis |
| **验证中间件** | 参数校验、类型检查、Schema验证 | joi、zod、yup |
| **压缩中间件** | Gzip压缩、Brotli压缩 | compression |
| **安全中间件** | CSRF防护、XSS过滤、HTTPS强制 | helmet、csurf |
| **跨域中间件** | CORS配置、JSONP处理 | cors |
| **静态文件中间件** | 提供静态文件服务 | express.static、koa-static |
| **监控中间件** | 性能指标、健康检查 | prom-client、express-etrics |

### 1.4 中间件的执行顺序

中间件的执行顺序是理解中间件机制的关键。在 Express 中，中间件按照注册顺序依次执行，除非中间件内部终止了请求流程（如返回响应或抛出错误）。而在 Koa 中，中间件通过 `await next()` 来控制流程，下一个中间件必须等待当前中间件调用 `next()` 才会执行。

```javascript
// Express 风格的中间件执行流程
const app = express();

// 中间件按照注册顺序执行
app.use(middleware1);  // 第1个执行
app.use(middleware2);  // 第2个执行
app.use(middleware3);  // 第3个执行

// 路由处理器
app.get('/path', handler);  // 最后执行

// 请求流程: middleware1 → middleware2 → middleware3 → handler
```

```javascript
// Koa 风格的中间件执行流程（洋葱模型）
const app = new Koa();

// 中间件采用栈结构，后注册的先执行（类似栈的push）
app.use(middleware3);  // 第3个注册，但最先执行（最外层）
app.use(middleware2);  // 第2个注册
app.use(middleware1);  // 第1个注册，最后执行（最外层）

// 请求流程: middleware3 → middleware2 → middleware1 → handler
// 响应流程: middleware1 → middleware2 → middleware3 → 响应
```

### 1.5 我的思考：中间件是框架的灵魂

经过多年的后端开发实践，我深刻体会到：**中间件不仅是请求处理管道，更是框架设计理念的集中体现**。

首先，**中间件的设计直接决定了框架的扩展性**。一个好的中间件系统应该允许开发者在任何层级注入自定义逻辑，而不是被迫修改框架核心代码。Express 的线性中间件链和 Koa 的洋葱模型各有千秋，但都提供了良好的扩展能力。相比之下，NestJS 通过 Guards、Interceptors、Pipes、Filters 等多层次的拦截器机制，提供了更细粒度的请求控制能力。

其次，**中间件是关注点分离（Separation of Concerns）原则的最佳实践**。通过将日志、认证、验证、监控等横切关注点抽取为独立的中间件，业务逻辑层可以保持简洁清晰。这种设计模式与面向切面编程（AOP）的思想不谋而合。

最后，**中间件系统的设计往往反映了框架对异步处理的理解**。Express 的回调模式虽然简单，但在处理复杂异步流程时容易陷入回调地狱。Koa 基于 async/await 的设计让异步流程更加直观。NestJS 则通过 RxJS 的 Observable 提供了更强大的响应式编程能力。

## 2. Express vs Koa vs Fastify

### 2.1 Express：中间件缺陷与回调地狱

**Express** 是 Node.js 历史上最成功的 Web 框架，诞生于 2010 年左右，以其简洁的 API 和强大的中间件生态系统迅速走红。然而，随着 JavaScript 语言的发展和应用程序复杂度的提升，Express 的一些设计缺陷逐渐暴露出来。

#### 2.1.1 Express 的中间件模型

Express 采用的是**线性中间件链**模型，每个中间件函数接收 `req`、`res`、`next` 三个参数。中间件通过调用 `next()` 将控制权传递给下一个中间件。这种模型简单直观，但在处理复杂场景时存在明显不足。

```javascript
const express = require('express');
const app = express();

// 基础中间件结构
function loggerMiddleware(req, res, next) {
  // 记录请求开始时间
  const start = Date.now();

  // 为 req 对象添加自定义属性
  req.requestId = Math.random().toString(36).substring(7);

  // 监听响应完成事件，记录耗时
  res.on('finish', () => {
    const duration = Date.now() - start;
    console.log(`[${req.requestId}] ${req.method} ${req.url} - ${duration}ms`);
  });

  next(); // 必须调用 next() 才能继续
}

// 认证中间件
function authMiddleware(req, res, next) {
  const token = req.headers.authorization?.replace('Bearer ', '');

  if (!token) {
    return res.status(401).json({ error: '未授权访问' });
  }

  try {
    // 验证 JWT
    const decoded = verifyToken(token);
    req.user = decoded; // 将用户信息挂载到 req
    next();
  } catch (err) {
    return res.status(401).json({ error: '无效的令牌' });
  }
}

// 错误处理中间件（必须有4个参数）
function errorHandler(err, req, res, next) {
  console.error('错误:', err);

  const statusCode = err.statusCode || 500;
  const message = err.message || '服务器内部错误';

  res.status(statusCode).json({
    success: false,
    error: message,
    ...(process.env.NODE_ENV === 'development' && { stack: err.stack })
  });
}

// 应用中间件
app.use(express.json()); // 内置 JSON 解析中间件
app.use(loggerMiddleware); // 自定义日志中间件
app.use('/api', authMiddleware); // 仅在 /api 路径下应用认证

// 路由
app.get('/api/users', (req, res) => {
  res.json({ users: [], user: req.user });
});

// 错误处理（必须在所有路由之后）
app.use(errorHandler);
```

#### 2.1.2 Express 中间件的核心缺陷

**缺陷一：错误处理的不一致性**

Express 的错误处理依赖于回调函数和 `next(err)` 模式。在同步代码中，抛出异常会被 Express 自动捕获并传递给错误处理中间件。但在异步代码中，如果忘记调用 `next(err)`，错误就会被吞掉，导致请求挂起。

```javascript
// 同步代码 - 错误自动被捕获
app.get('/sync-error', (req, res, next) => {
  throw new Error('同步错误'); // Express 自动捕获
});

// 异步代码 - 错误可能被吞掉
app.get('/async-error-bad', (req, res, next) => {
  // 错误处理函数没有调用 next(err)，这个 Promise  rejection 会被忽略
  someAsyncOperation().catch(err => {
    console.error('错误被吞掉了'); // 永远不会执行
  });
  res.json({ success: true }); // 直接返回响应
});

// 正确的异步错误处理
app.get('/async-error-good', async (req, res, next) => {
  try {
    const result = await someAsyncOperation();
    res.json(result);
  } catch (err) {
    next(err); // 必须手动传递错误
  }
});
```

**缺陷二：回调地狱**

在复杂的业务逻辑中，Express 的回调模式容易导致回调嵌套过深，代码难以阅读和维护。

```javascript
// 回调地狱示例
app.post('/checkout', (req, res, next) => {
  // 验证库存
  checkInventory(req.body.productId, (err, inventory) => {
    if (err) return next(err);

    // 验证支付
    validatePayment(req.body.paymentToken, (err, payment) => {
      if (err) return next(err);

      // 创建订单
      createOrder(req.body, (err, order) => {
        if (err) return next(err);

        // 发送邮件
        sendConfirmationEmail(order, (err) => {
          if (err) return next(err);

          // 更新库存
          updateInventory(req.body.productId, inventory.quantity - 1, (err) => {
            if (err) return next(err);
            res.json({ success: true, order });
          });
        });
      });
    });
  });
});

// 使用 async/await 改造后
app.post('/checkout', async (req, res, next) => {
  try {
    const inventory = await checkInventory(req.body.productId);
    const payment = await validatePayment(req.body.paymentToken);
    const order = await createOrder(req.body);
    await sendConfirmationEmail(order);
    await updateInventory(req.body.productId, inventory.quantity - 1);
    res.json({ success: true, order });
  } catch (err) {
    next(err);
  }
});
```

**缺陷三：req/res 对象的不一致性**

Express 将请求和响应对象分离，虽然清晰但也导致了在中间件之间传递数据时需要手动挂载到 req 对象上，缺乏统一的数据流管理。

```javascript
// Express 中间件之间共享数据的常见方式
app.use((req, res, next) => {
  req.sharedData = { startTime: Date.now() }; // 手动挂载
  next();
});

app.use((req, res, next) => {
  console.log(req.sharedData.startTime); // 手动获取
  next();
});
```

#### 2.1.3 Express 的适用场景

尽管存在上述缺陷，Express 在以下场景仍然是首选：

- **快速原型开发**：Express 的灵活性使其非常适合快速搭建 MVP
- **学习 Node.js**：Express 的文档最完善，社区资源最丰富
- **中小型项目**：对于请求量不大、逻辑不复杂的应用，Express 足够使用
- **需要大量第三方中间件**：Express 生态的中间件数量远超其他框架

### 2.2 Koa：async/await 与洋葱模型

**Koa** 是 Express 原班人马打造的新一代框架，核心理念是"更小、更富表现力、更健壮"。Koa 团队认为 Express 过于笨重且存在设计缺陷，于是从头开始，重新设计了整个框架。

#### 2.2.1 Koa 的核心设计

Koa 相比 Express 有几个核心区别：

1. **极小体积**：Koa 核心只有约 550 行代码，所有功能都通过中间件实现
2. **async/await 原生支持**：Koa 的中间件都是 async 函数
3. **Context 对象**：Koa 将 req 和 res 封装为统一的 ctx 对象
4. **洋葱模型**：中间件的执行采用标准的洋葱模型

```javascript
const Koa = require('koa');
const Router = require('@koa/router');
const bodyParser = require('koa-bodyparser');

const app = new Koa();
const router = new Router();

// ============================================
// Koa Context 对象 (ctx)
// ============================================

// ctx.request - Koa 的请求对象（封装自 Node 原生 req）
// ctx.response - Koa 的响应对象（封装自 Node 原生 res）
// ctx.state - 推荐用于中间件之间共享数据的统一位置
// ctx.app - 应用实例引用
// ctx.cookies - Cookie 操作
// ctx.throw() - 抛出错误

// 示例：日志中间件
app.use(async (ctx, next) => {
  const start = Date.now();

  // 记录开始时间到 ctx.state
  ctx.state.startTime = start;

  await next(); // 等待下游中间件执行

  const duration = Date.now() - start;
  console.log(`${ctx.method} ${ctx.url} - ${duration}ms`);
});

// 示例：认证中间件
app.use(async (ctx, next) => {
  const token = ctx.headers.authorization?.replace('Bearer ', '');

  if (!token) {
    ctx.throw(401, '未提供认证令牌');
  }

  try {
    const decoded = verifyToken(token);
    ctx.state.user = decoded; // 使用 ctx.state 共享用户信息
    await next();
  } catch (err) {
    ctx.throw(401, '无效的令牌');
  }
});

// 错误处理中间件（必须放在所有中间件之后）
app.use(async (ctx, next) => {
  try {
    await next();
  } catch (err) {
    ctx.status = err.status || 500;
    ctx.body = {
      success: false,
      message: err.message,
      ...(process.env.NODE_ENV === 'development' && { stack: err.stack })
    };
  }
});

// 路由
router.get('/api/users', (ctx) => {
  ctx.body = { users: [], user: ctx.state.user };
});

app
  .use(router.routes())
  .use(router.allowedMethods());

app.listen(3000);
```

#### 2.2.2 Koa 洋葱模型的深度解析

Koa 的洋葱模型是其最具代表性的特性。理解这个模型的实现原理对于深入掌握 Koa 至关重要。

```javascript
// Koa 中间件的组合原理
// 源码简化版

// 中间件数组
const middlewares = [];

// use 方法：将中间件添加到数组
app.use = function(fn) {
  middlewares.push(fn);
};

// listen 方法：组合并执行中间件
app.listen = function() {
  // 从第一个中间件开始组合
  const composed = compose(middlewares);
  return composed(ctx, null);
};

// compose 函数：实现洋葱模型
function compose(middleware) {
  return function(ctx, next) {
    // 从第 0 个中间件开始执行
    let index = -1;

    function dispatch(i) {
      if (i <= index) {
        throw new Error('next() 被调用多次');
      }
      index = i;

      // 如果没有更多中间件，执行一个空的 next
      if (i === middleware.length) {
        return Promise.resolve();
      }

      const fn = middleware[i];

      try {
        // 执行当前中间件，传入下一个 dispatch
        // 注意：await next() 等价于 await dispatch(i + 1)
        return Promise.resolve(
          fn(ctx, function next() {
            return dispatch(i + 1);
          })
        );
      } catch (err) {
        return Promise.reject(err);
      }
    }

    return dispatch(0);
  };
}
```

**洋葱模型的执行流程详解：**

```javascript
// 完整示例：展示洋葱模型的每一个执行步骤
const app = new Koa();

app.use(async (ctx, next) => {
  console.log('【1】中间件 A - 前置逻辑');
  console.log('【2】调用 next()，进入中间件 B');
  await next();
  console.log('【9】中间件 A - 后置逻辑（响应前）');
});

app.use(async (ctx, next) => {
  console.log('【3】中间件 B - 前置逻辑');
  console.log('【4】调用 next()，进入中间件 C');
  await next();
  console.log('【8】中间件 B - 后置逻辑');
});

app.use(async (ctx, next) => {
  console.log('【5】中间件 C - 前置逻辑');
  console.log('【6】调用 next()，进入业务逻辑');
  await next();
  console.log('【7】中间件 C - 后置逻辑');
});

// 业务逻辑
app.use(async (ctx) => {
  console.log('【业务逻辑执行】');
  ctx.body = 'Hello Koa';
});

// 输出顺序:
// 【1】中间件 A - 前置逻辑
// 【2】调用 next()，进入中间件 B
// 【3】中间件 B - 前置逻辑
// 【4】调用 next()，进入中间件 C
// 【5】中间件 C - 前置逻辑
// 【6】调用 next()，进入业务逻辑
// 【业务逻辑执行】
// 【7】中间件 C - 后置逻辑
// 【8】中间件 B - 后置逻辑
// 【9】中间件 A - 后置逻辑（响应前）
```

#### 2.2.3 Koa 与 Express 的关键对比

| 特性 | Express | Koa |
|------|---------|-----|
| **核心代码量** | ~500KB | ~550 行代码 |
| **中间件模型** | 线性链式 | 洋葱模型 |
| **异步处理** | 回调模式 | async/await |
| **错误处理** | 回调+next(err) | try-catch |

### 2.3 Fastify：性能优先与 Schema 验证

**Fastify** 是目前最快的 Node.js Web 框架之一，以其卓越的性能和现代化的 API 设计著称。Fastify 诞生于 2016 年，由 Matteo Collina 和其他 Node.js 核心贡献者开发，旨在解决 Express 和 Koa 在高性能场景下的不足。

#### 2.3.1 Fastify 的核心特性

```javascript
const fastify = require('fastify')({ logger: true });

// Fastify 的性能优势来源：
// 1. 基于 Radix Tree 的极速路由匹配
// 2. fast-json-stringify 预编译 JSON 序列化
// 3. 最小化对象创建和内存分配

// ============================================
// Schema 验证（Fastify 独有特性）
// ============================================

// 定义请求/响应的 Schema
const userSchema = {
  body: {
    type: 'object',
    required: ['username', 'email', 'password'],
    properties: {
      username: { type: 'string', minLength: 2, maxLength: 50 },
      email: { type: 'string', format: 'email' },
      password: { type: 'string', minLength: 6 }
    }
  },
  response: {
    200: {
      type: 'object',
      properties: {
        success: { type: 'boolean' },
        user: {
          type: 'object',
          properties: {
            id: { type: 'string' },
            username: { type: 'string' },
            email: { type: 'string' }
          }
        }
      }
    }
  }
};

// Fastify 自动根据 Schema 生成验证逻辑和序列化逻辑
// 这比运行时验证快得多
fastify.post('/users', { schema: userSchema }, async (request, reply) => {
  const { username, email, password } = request.body;

  // 验证在进入处理器之前已经完成
  // 响应也会根据 Schema 自动序列化

  return {
    success: true,
    user: { id: '123', username, email }
  };
});

// ============================================
// 中间件装饰器
// ============================================

// 添加自定义属性到 request/reply
fastify.decorateRequest('user', null);
fastify.decorateReply('sendFormatted', function(data) {
  return this.send({ success: true, data });
});

// ============================================
// 插件系统（类似中间件但有独立作用域）
// ============================================

// 插件：封装独立功能
fastify.register(async function authPlugin(fastify, options) {
  // 在插件内部添加的装饰器只在该插件作用域内有效
  fastify.decorateRequest('token', null);

  fastify.addHook('preHandler', async (request, reply) => {
    const token = request.headers.authorization?.replace('Bearer ', '');
    if (!token) {
      reply.code(401).send({ error: '未授权' });
    }
  });
});

// ============================================
// 启动服务器
// ============================================

const start = async () => {
  try {
    await fastify.listen({ port: 3000 });
    console.log('Fastify 服务器运行中');
  } catch (err) {
    fastify.log.error(err);
    process.exit(1);
  }
};

start();
```

```

├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   Fastify     ████████████████████████████████████████████  极高  │
│   (Radix Tree路由 + 预编译JSON)                                   │
│                                                                  │
```

**Fastify 性能领先的原因：**

1. **Radix Tree 路由**：路由匹配时间复杂度为 O(k)，k 为路径长度，而非遍历数组
2. **JSON Schema 预编译**：使用 `fast-json-stringify` 将 Schema 编译为高效的序列化函数
3. **最小化中间件开销**：Fastify 的中间件（插件）系统经过优化，overhead 极低
4. **避免不必要的对象创建**：减少 GC 压力

### 2.5 我的思考：没有最好的框架

在多年的技术选型实践中，我深刻体会到：**框架的选择从来不是非此即彼的，而是需要根据具体场景权衡取舍**。

**Express 的价值**在于其极低的入门门槛和完善的生态系统。如果你需要快速交付一个中小型项目，或者你的团队对 Node.js 还不熟悉，Express 永远是最安全的选择。其庞大的中间件库意味着你不需要重复造轮子。

**Koa 的价值**在于其优雅的设计和对现代 JavaScript 特性的充分利用。洋葱模型让复杂业务流程的编写变得直观，async/await 让异步代码的可读性大幅提升。如果你的团队具备一定的 Node.js 经验，且项目需要处理复杂的中间件逻辑，Koa 是更好的选择。

**Fastify 的价值**在于其卓越的性能和对 TypeScript/Schema 验证的原生支持。在微服务架构中，每个服务的性能都会被放大，Fastify 的性能优势可以显著降低基础设施成本。同时，其 Schema 验证机制可以减少大量运行时错误，提升代码质量。

**我的建议是：**

- **个人项目/小团队**：Express + 1（足够用，不要过度工程）
- **中型项目/需要现代特性**：Koa + TypeScript
- **微服务/高性能需求**：Fastify
- **企业级应用/需要强类型**：NestJS（基于 Express/Fastify）

#### 3.1.2 Egg.js 中间件实现

```javascript
// app/middleware/auth.js
// 自定义中间件

module.exports = (options) => {
  // options 来自配置，用于参数化中间件
  return async function authMiddleware(ctx, next) {
    const token = ctx.headers.authorization?.replace('Bearer ', '');

    if (!token) {
      ctx.status = 401;
      ctx.body = { success: false, message: '未提供认证令牌' };
      return;
    }

    try {
      // 验证 token
      const decoded = ctx.service.user.verifyToken(token);
      ctx.state.user = decoded;
      await next();
    } catch (err) {
      ctx.status = 401;
      ctx.body = { success: false, message: '无效的认证令牌' };
    }
  };
};

// config/config.default.js
// 配置中间件

module.exports = {
  middleware: ['auth'],  // 启用 auth 中间件

  // 中间件配置
  auth: {
    ignore: ['/api/public', '/api/login'],  // 忽略不需要认证的路径
  }
};

// app/controller/user.js
// 控制器中使用

const Controller = require('egg').Controller;

class UserController extends Controller {
  async getUser() {
    // ctx.state.user 由 auth 中间件设置
    const { id } = ctx.params;
    const user = await ctx.service.user.findById(id);
    ctx.body = { success: true, data: user };
  }
}

module.exports = UserController;
```

### 3.2 NestJS：依赖注入与装饰器

**NestJS** 是目前最成熟的企业级 Node.js 框架，它将 Angular 的架构理念引入到后端开发中，提供了完整的依赖注入系统、模块化结构和面向切面编程能力。NestJS 构建在 Express 或 Fastify 之上，保留了底层框架的兼容性。

#### 3.2.1 NestJS 的模块系统与依赖注入

NestJS 的核心是**依赖注入容器**和**模块系统**。每个应用由多个模块组成，每个模块定义了自己的 Providers、Controllers 和导入的依赖模块。

```typescript
// user.module.ts
// 用户模块定义

import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { User } from './user.entity';
import { UserController } from './user.controller';
import { UserService } from './user.service';
import { AuthModule } from '../auth/auth.module';

@Module({
  // 导入其他模块
  imports: [
    // 导入 TypeORM 模块，使其可以在当前模块使用
    TypeOrmModule.forFeature([User]),
    // 导入 Auth 模块，可以直接使用 AuthModule 中 exports 的 Provider
    AuthModule,
  ],

  // 控制器 - 处理 HTTP 请求
  controllers: [UserController],

  // 服务提供者 - 业务逻辑和数据访问
  providers: [UserService],

  // 导出模块，供其他模块使用
  exports: [UserService],
})
export class UserModule {}

// user.service.ts
// 服务类 - 依赖注入的典型应用

import { Injectable, NotFoundException, ConflictException } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { User } from './user.entity';
import { CreateUserDto, UpdateUserDto } from './dto';

// @Injectable() 装饰器表示该类可以被 NestJS 容器管理
@Injectable()
export class UserService {
  // 依赖注入：通过构造函数自动注入 UserRepository
  constructor(
    @InjectRepository(User)  // 注入特定的 Repository
    private readonly userRepository: Repository<User>,
  ) {}

  // 查询所有用户
  async findAll(): Promise<User[]> {
    return this.userRepository.find();
  }

  // 根据 ID 查询
  async findById(id: string): Promise<User> {
    const user = await this.userRepository.findOne({ where: { id } });
    if (!user) {
      throw new NotFoundException(`用户 ID ${id} 不存在`);
    }
    return user;
  }

  // 创建用户
  async create(dto: CreateUserDto): Promise<User> {
    // 检查邮箱唯一性
    const existing = await this.userRepository.findOne({
      where: { email: dto.email }
    });

    if (existing) {
      throw new ConflictException('邮箱已被注册');
    }

    const user = this.userRepository.create(dto);
    return this.userRepository.save(user);
  }

  // 更新用户
  async update(id: string, dto: UpdateUserDto): Promise<User> {
    const user = await this.findById(id);
    Object.assign(user, dto);
    return this.userRepository.save(user);
  }

  // 删除用户
  async remove(id: string): Promise<void> {
    const user = await this.findById(id);
    await this.userRepository.remove(user);
  }
}

// user.controller.ts
// 控制器 - 接收 HTTP 请求，调用服务

import {
  Controller,
  Get,
  Post,
  Put,
  Delete,
  Body,
  Param,
  Query,
  UseGuards,
  HttpCode,
  HttpStatus,
} from '@nestjs/common';
import { UserService } from './user.service';
import { CreateUserDto, UpdateUserDto } from './dto';
import { JwtAuthGuard } from '../auth/jwt-auth.guard';
import { RolesGuard } from '../auth/roles.guard';
import { Roles } from '../auth/roles.decorator';

@Controller('users')
export class UserController {
  // 依赖注入服务
  constructor(private readonly userService: UserService) {}

  @Get()
  async findAll(@Query('page') page = 1, @Query('limit') limit = 10) {
    return this.userService.findAll();
  }

  @Get(':id')
  async findOne(@Param('id') id: string) {
    return this.userService.findById(id);
  }

  @Post()
  @HttpCode(HttpStatus.CREATED)
  async create(@Body() dto: CreateUserDto) {
    return this.userService.create(dto);
  }

  @Put(':id')
  async update(@Param('id') id: string, @Body() dto: UpdateUserDto) {
    return this.userService.update(id, dto);
  }

  @Delete(':id')
  @HttpCode(HttpStatus.NO_CONTENT)
  async remove(@Param('id') id: string) {
    return this.userService.remove(id);
  }
}
```

#### 3.2.2 NestJS 的请求生命周期

NestJS 提供了一套完整的请求处理生命周期，这是理解 NestJS 中间件系统的关键。

```

├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   1. Middleware（中间件）                                        │
│      │                                                          │

│      ├─ 路由中间件：MiddlewareConsumer                          │
│      │                                                          │
│      ▼                                                          │
```

#### 3.2.3 NestJS 守卫、拦截器、管道、过滤器详解

**守卫（Guards）** - 权限控制

```typescript
// jwt-auth.guard.ts
// JWT 认证守卫

import {
  Injectable,
  CanActivate,
  ExecutionContext,
  UnauthorizedException,
} from '@nestjs/common';
import { JwtService } from '@nestjs/jwt';
import { Observable } from 'rxjs';

@Injectable()
export class JwtAuthGuard implements CanActivate {
  constructor(private readonly jwtService: JwtService) {}

  canActivate(
    context: ExecutionContext,
  ): boolean | Promise<boolean> | Observable<boolean> {
    const request = context.switchToHttp().getRequest();

    // 从请求头获取 token
    const token = request.headers.authorization?.replace('Bearer ', '');

    if (!token) {
      throw new UnauthorizedException('未提供认证令牌');
    }

    try {
      // 验证 token
      const decoded = this.jwtService.verify(token);
      // 将用户信息挂载到请求对象
      request.user = decoded;
      return true;
    } catch (err) {
      throw new UnauthorizedException('无效的认证令牌');
    }
  }
}

// roles.guard.ts
// 角色守卫

@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private readonly reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    // 从处理器或类上获取 @Roles() 装饰器定义的角色
    const requiredRoles = this.reflector.get<string[]>(
      'roles',
      context.getHandler(),
    );

    // 如果没有定义角色要求，直接放行
    if (!requiredRoles) {
      return true;
    }

    // 获取请求对象中的用户信息（由 JwtAuthGuard 设置）
    const request = context.switchToHttp().getRequest();
    const user = request.user;

    // 检查用户角色是否匹配
    return requiredRoles.some(role => user.roles?.includes(role));
  }
}

// roles.decorator.ts
// 自定义角色装饰器

import { SetMetadata } from '@nestjs/common';

export const Roles = (...roles: string[]) => SetMetadata('roles', roles);

// 使用
@Controller('users')
@UseGuards(JwtAuthGuard, RolesGuard)
export class UserController {
  @Post()
  @Roles('admin')  // 只有 admin 角色可以访问
  async create(@Body() dto: CreateUserDto) {
    return this.userService.create(dto);
  }
}
```

**拦截器（Interceptors）** - 请求/响应处理

```typescript
// logging.interceptor.ts
// 日志拦截器

import {
  Injectable,
  NestInterceptor,
  ExecutionContext,
  CallHandler,
  Logger,
} from '@nestjs/common';
import { Observable } from 'rxjs';
import { tap } from 'rxjs/operators';

@Injectable()
export class LoggingInterceptor implements NestInterceptor {
  private readonly logger = new Logger(LoggingInterceptor.name);

  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    const request = context.switchToHttp().getRequest();
    const { method, url, body } = request;
    const now = Date.now();

    this.logger.log(`请求开始: ${method} ${url}`);
    if (body && Object.keys(body).length) {
      this.logger.debug(`请求体: ${JSON.stringify(body)}`);
    }

    return next.handle().pipe(
      tap({
        next: (data) => {
          const duration = Date.now() - now;
          this.logger.log(`请求完成: ${method} ${url} - ${duration}ms`);
        },
        error: (error) => {
          const duration = Date.now() - now;
          this.logger.error(
            `请求失败: ${method} ${url} - ${duration}ms - ${error.message}`,
          );
        },
      }),
    );
  }
}

// transform.interceptor.ts
// 响应转换拦截器

@Injectable()
export class TransformInterceptor<T>
  implements NestInterceptor<T, any>
{
  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    return next.handle().pipe(
      map((data) => ({
        success: true,
        data,
        timestamp: new Date().toISOString(),
      })),
    );
  }
}

// cache.interceptor.ts
// 缓存拦截器

@Injectable()
export class CacheInterceptor implements NestInterceptor {
  constructor(private readonly cacheService: CacheService) {}

  async intercept(context: ExecutionContext, next: CallHandler): Promise<Observable<any>> {
    const request = context.switchToHttp().getRequest();
    const key = `cache:${request.url}`;

    // 尝试从缓存获取
    const cached = await this.cacheService.get(key);
    if (cached) {
      return of(cached);
    }

    // 缓存未命中，执行处理方法并缓存结果
    return next.handle().pipe(
      tap(async (data) => {
        await this.cacheService.set(key, data, { ttl: 60 }); // 缓存 60 秒
      }),
    );
  }
}
```

**管道（Pipes）** - 数据转换与验证

```typescript
// validation.pipe.ts
// 全局验证管道

import {
  PipeTransform,
  Injectable,
  ArgumentMetadata,
  BadRequestException,
} from '@nestjs/common';
import { validate, ValidationError } from 'class-validator';
import { plainToInstance } from 'class-transformer';

@Injectable()
export class ValidationPipe implements PipeTransform<any> {
  async transform(value: any, { metatype, data }: ArgumentMetadata) {
    // 如果没有类型信息，直接返回
    if (!metatype || !this.toValidate(metatype)) {
      return value;
    }

    // 将 plain object 转换为 class 实例
    const object = plainToInstance(metatype, value);

    // 验证
    const errors = await validate(object as any);

    if (errors.length > 0) {
      const formattedErrors = this.formatErrors(errors);
      throw new BadRequestException({
        message: '参数验证失败',
        errors: formattedErrors,
      });
    }

    return object;
  }

  private toValidate(metatype: Function): boolean {
    const types: Function[] = [String, Boolean, Number, Array, Object];
    return !types.includes(metatype);
  }

  private formatErrors(errors: ValidationError[]): Record<string, string[]> {
    const result: Record<string, string[]> = {};

    for (const error of errors) {
      const property = error.property;
      const constraints = error.constraints
        ? Object.values(error.constraints)
        : [];

      if (error.children && error.children.length > 0) {
        // 递归处理嵌套对象
        const nested = this.formatErrors(error.children);
        for (const [key, messages] of Object.entries(nested)) {
          result[`${property}.${key}`] = messages;
        }
      } else {
        result[property] = constraints;
      }
    }

    return result;
  }
}

// parse-int.pipe.ts
// 自定义 ParseInt 管道

@Injectable()
export class ParseIntPipe implements PipeTransform<string, number> {
  transform(value: string, metadata: ArgumentMetadata): number {
    const val = parseInt(value, 10);

    if (isNaN(val)) {
      throw new BadRequestException(
        `参数 ${metadata.data} 必须是有效的整数`,
      );
    }

    return val;
  }
}

// 使用管道
@Controller('users')
export class UserController {
  @Get(':id')
  async findOne(
    @Param('id', ParseIntPipe) id: number,  // 自动转换为数字
    @Query('page', new DefaultValuePipe(1), ParseIntPipe) page: number,
  ) {
    return this.userService.findById(id);
  }
}
```

**异常过滤器（Exception Filters）** - 统一错误处理

```typescript
// http-exception.filter.ts
// HTTP 异常过滤器

import {
  ExceptionFilter,
  Catch,
  ArgumentsHost,
  HttpException,
  HttpStatus,
  Logger,
} from '@nestjs/common';
import { Request, Response } from 'express';

@Catch()
export class AllExceptionsFilter implements ExceptionFilter {
  private readonly logger = new Logger(AllExceptionsFilter.name);

  catch(exception: unknown, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const request = ctx.getRequest<Request>();

    // 确定 HTTP 状态码
    let status: number;
    let message: string | object;

    if (exception instanceof HttpException) {
      status = exception.getStatus();
      const exceptionResponse = exception.getResponse();
      message =
        typeof exceptionResponse === 'string'
          ? exceptionResponse
          : exceptionResponse;
    } else {
      // 非 HTTP 异常（系统错误）
      status = HttpStatus.INTERNAL_SERVER_ERROR;
      message = '服务器内部错误';

      // 记录错误日志
      this.logger.error(
        `系统错误: ${exception instanceof Error ? exception.message : 'Unknown'}`,
        exception instanceof Error ? exception.stack : undefined,
      );
    }

    // 格式化错误响应
    const errorResponse = {
      success: false,
      statusCode: status,
      message: message,
      timestamp: new Date().toISOString(),
      path: request.url,
      ...(process.env.NODE_ENV === 'development' &&
        exception instanceof Error && { stack: exception.stack }),
    };

    response.status(status).json(errorResponse);
  }
}

// 使用全局过滤器
async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  // 全局前缀
  app.setGlobalPrefix('api/v1');

  // 全局验证管道
  app.useGlobalPipes(
    new ValidationPipe({
      whitelist: true,           // 剥离非白名单属性
      forbidNonWhitelisted: true, // 拒绝非白名单属性
      transform: true,           // 自动类型转换
      transformOptions: {
        enableImplicitConversion: true,
      },
    }),
  );

  // 全局异常过滤器
  app.useGlobalFilters(new AllExceptionsFilter());

  // 全局拦截器
  app.useGlobalInterceptors(
    new LoggingInterceptor(),
    new TransformInterceptor(),
  );

  await app.listen(3000);
}
```

### 3.4 我的思考：企业级框架的选择

在企业级应用开发中，框架的选择需要考虑更多因素：

**技术团队背景**

- 如果团队主要来自 Java/Spring 背景，**NestJS** 的设计理念会更加熟悉
- 如果团队主要是前端背景，可能更容易接受 **Egg.js** 的目录约定
- 如果追求极限性能和简洁，**Fastify** + 手写架构是更好的选择

**项目需求复杂度**

- 简单 CRUD 应用：Express + 简单封装足以应对
- 中等复杂度：Egg.js 的分层架构可以保持代码组织
- 复杂企业级系统：NestJS 的模块化和依赖注入是最佳选择

**长期维护考虑**

- 团队更替：Egg.js 的约定俗成可能让新成员快速上手
- 架构演进：NestJS 的模块系统更利于大型项目的拆分和重构
- 类型安全：NestJS 的 TypeScript 原生支持在大型团队中价值显著

## 4. 日志中间件

### 4.1 Winston vs Pino vs Log4js 核心对比

日志系统是后端应用不可或缺的基础设施。在 Node.js 生态中，**Winston**、**Pino** 和 **Log4js** 是最主流的三个日志库。

#### 4.1.1 Winston - 最流行的日志库

Winston 是 Node.js 领域功能最全面的日志库，以其灵活的传输（Transports）机制著称。

```javascript
const winston = require('winston');

// ============================================
// Winston 核心配置
// ============================================

// 定义日志格式
const myFormat = winston.format.combine(
  winston.format.timestamp({ format: 'YYYY-MM-DD HH:mm:ss' }),
  winston.format.errors({ stack: true }),  // 支持错误堆栈
  winston.format.splat(),  // 支持字符串格式化
  winston.format.json()    // JSON 格式输出
);

// 定义控制台格式（开发环境友好）
const consoleFormat = winston.format.combine(
  winston.format.colorize(),  // 颜色高亮
  winston.format.timestamp({ format: 'HH:mm:ss' }),
  winston.format.printf(({ timestamp, level, message, ...meta }) => {
    const metaStr = Object.keys(meta).length ? JSON.stringify(meta) : '';
    return `${timestamp} [${level}]: ${message} ${metaStr}`;
  })
);

// 创建日志记录器
const logger = winston.createLogger({
  // 日志级别
  level: process.env.LOG_LEVEL || 'info',

  // 默认格式
  format: myFormat,

  // 传输目标
  transports: [
    // 控制台输出（开发环境）
    new winston.transports.Console({
      format: consoleFormat,
      level: 'debug',
    }),

    // 文件输出 - 错误日志（始终记录）
    new winston.transports.File({
      filename: 'logs/error.log',
      level: 'error',
      maxsize: 5242880,  // 5MB
      maxFiles: 5,        // 保留 5 个文件
      tailable: true,    // 追加模式
    }),

    // 文件输出 - 组合日志
    new winston.transports.File({
      filename: 'logs/combined.log',
      maxsize: 5242880,  // 5MB
      maxFiles: 5,
    }),

    // 文件输出 - JSON 格式（供日志收集系统使用）
    new winston.transports.File({
      filename: 'logs/json.log',
      format: winston.format.json(),
    }),
  ],

  // 异常处理
  exceptionHandlers: [
    new winston.transports.File({ filename: 'logs/exceptions.log' }),
  ],

  // 未捕获 Promise 拒绝处理
  rejectionHandlers: [
    new winston.transports.File({ filename: 'logs/rejections.log' }),
  ],
});

// ============================================
// Express 中间件集成
// ============================================

const express = require('express');
const app = express();

// 请求日志中间件
app.use((req, res, next) => {
  const start = Date.now();

  // 监听响应结束
  res.on('finish', () => {
    const duration = Date.now() - start;
    const { method, url, ip } = req;
    const { statusCode } = res;

    // 根据状态码选择日志级别
    const level = statusCode >= 500 ? 'error' :
                  statusCode >= 400 ? 'warn' : 'info';

    logger.log({
      level,
      message: 'HTTP 请求',
      method,
      url,
      ip,
      statusCode,
      duration,
      userAgent: req.get('user-agent'),
    });
  });

  next();
});

// 错误日志中间件
app.use((err, req, res, next) => {
  logger.error('未处理的错误', {
    message: err.message,
    stack: err.stack,
    url: req.url,
    method: req.method,
    body: req.body,
  });

  next(err);
});

// ============================================
// childlogger - 创建子日志记录器
// ============================================

const childLogger = logger.child({ service: 'user-service' });

childLogger.info('用户登录', { userId: '123', action: 'login' });
// 输出: { service: 'user-service', level: 'info', message: '用户登录', userId: '123', action: 'login' }

// ============================================
// 渐进式日志写入
// ============================================

// DailyRotateFile - 按日期分割日志
const DailyRotateFile = require('winston-daily-rotate-file');

const dailyRotate = new DailyRotateFile({
  filename: 'logs/app-%DATE%.log',
  datePattern: 'YYYY-MM-DD',
  zippedArchive: true,
  maxSize: '20m',
  maxFiles: '14d',
});

logger.add(dailyRotate);
```

#### 4.1.2 Pino - 最快的日志库

Pino 以其极致的性能著称，是 Winston 的 10-100 倍快。Pino 的设计哲学是"日志应该是廉价的"，通过减少不必要的对象创建和格式化操作来实现高性能。

```javascript
const pino = require('pino');

// ============================================
// Pino 基础配置
// ============================================

// 创建 logger
const logger = pino({
  // 基本配置
  level: process.env.LOG_LEVEL || 'info',
  base: { service: 'my-app' },  // 基础字段
  timestamp: pino.stdTimeFunctions.isoTime,  // ISO 时间格式

  // 序列化器
  serializers: {
    req: (req) => ({
      method: req.method,
      url: req.url,
      headers: {
        'user-agent': req.headers['user-agent'],
      },
    }),
    res: (res) => ({
      statusCode: res.statusCode,
    }),
    err: pino.stdSerializers.err,  // 标准错误序列化
  },
});

// ============================================
// Pino HTTP 中间件
// ============================================

const express = require('express');
const app = express();

// pino-http 提供 Express 集成
const pinoHttp = require('pino-http');

app.use(pinoHttp({
  logger: logger,
  autoLogging: {
    ignore: (req) => req.url === '/health',  // 忽略健康检查
  },
  customLogLevel: (req, res, err) => {
    if (res.statusCode >= 500 || err) return 'error';
    if (res.statusCode >= 400) return 'warn';
    return 'info';
  },
  customSuccessMessage: (req, res) => {
    return `${req.method} ${req.url} completed`;
  },
  customErrorMessage: (req, res, err) => {
    return `${req.method} ${req.url} failed`;
  },
  customAttributeKeys: {
    req: 'request',
    res: 'response',
    err: 'error',
    responseTime: 'duration',
  },
}));

// ============================================
// Pino 性能优势原理
// ============================================

/*
Pino 的高性能来自以下几个方面：

1. 同步写入：Pino 默认使用同步 I/O，避免了异步开销
2. 最小化对象创建：使用字符串拼接而非对象创建
3. 预编译格式化：日志格式在初始化时编译好
4. 递归序列化优化：避免不必要的函数调用

基准测试（简单字符串日志）：
- pino: ~800,000 ops/sec
- winston: ~50,000 ops/sec
- log4js: ~80,000 ops/sec

基准测试（JSON 对象日志）：
- pino: ~400,000 ops/sec
- winston: ~15,000 ops/sec
- log4js: ~20,000 ops/sec
*/

// ============================================
// Pino 与日志收集系统集成
// ============================================

// Pino 支持通过 pino-pretty 美化开发环境输出
// 通过 pino/file 写入文件
// 通过 pino-http 自动记录 HTTP 请求

// 写入文件（同步）
const fs = require('fs');
const transport = pino.transport({
  targets: [
    // 标准输出（供 Docker/日志收集）
    { level: 'info', target: 'pino/file', options: { destination: 1 } },
    // 错误输出到文件
    {
      level: 'error',
      target: 'pino/file',
      options: { destination: fs.openSync('logs/error.log', 'a') },
    },
  ],
});

const fastLogger = pino(transport);

// 异步写入（性能略低但不会阻塞事件循环）
const asyncTransport = pino.transport({
  targets: [
    {
      level: 'info',
      target: 'pino-pretty',
      options: { destination: 1, colorize: true },
    },
    {
      level: 'info',
      dest: fs.openSync('logs/app.log', 'a'),
    },
  ],
  dedupe: true,  // 减少重复写入
});
```

#### 4.1.3 Log4js - Java 风格的企业选择

Log4js 受 Java Log4j 启发，提供了类别（Category）系统和灵活的配置机制。

```javascript
const log4js = require('log4js');

// ============================================
// Log4js 配置
// ============================================

log4js.configure({
  // 输出器
  appenders: {
    // 控制台输出

      type: 'console',

      type: 'file',
      filename: 'logs/app.log',

        type: 'json',
      },

      type: 'dateFile',
      filename: 'logs/app',
      pattern: '-yyyy-MM-dd.log',
      alwaysIncludePattern: true,
    },

    // 错误专用
    errorFile: {
      type: 'file',
      filename: 'logs/error.log',
      layout: {
        type: 'pattern',
        pattern: '%d{yyyy-MM-dd hh:mm:ss} %p %c - %m%n stack: %ex',

    // 网络输出（供 ELK Stack）
    tcp: {
      type: 'tcp',
      host: 'localhost',
      port: 5000,
```

## 5. 认证中间件

### 5.1 JWT vs Session 认证

认证是 Web 应用安全的基础。**JWT（JSON Web Token）** 和 **Session** 是两种最主流的认证机制。

#### 5.1.1 Session 认证原理

Session 认证是一种服务端存储的认证机制。当用户登录成功后，服务器在内存或数据库中创建一个 Session 对象，并将 Session ID 返回给客户端存储在 Cookie 中。

```javascript
// ============================================
// Session 认证实现
// ============================================

const session = require('express-session');
const RedisStore = require('connect-redis')(session);
const redis = require('redis');

// 创建 Redis 客户端
const redisClient = redis.createClient({
  url: process.env.REDIS_URL || 'redis://localhost:6379',
});

// Session 中间件配置
app.use(session({
  store: new RedisStore({ client: redisClient }),  // 存储在 Redis
  secret: process.env.SESSION_SECRET || 'your-secret-key',
  name: 'session_id',  // Cookie 名称
  resave: false,        // 不强制保存未修改的 session
  saveUninitialized: false,  // 不保存未初始化的 session
  cookie: {
    secure: process.env.NODE_ENV === 'production',  // HTTPS only
    httpOnly: true,  // 禁止 JavaScript 访问
    maxAge: 24 * 60 * 60 * 1000,  // 24 小时
    sameSite: 'lax',  // CSRF 防护
  },
}));

// 登录处理
app.post('/api/login', async (req, res) => {
  const { username, password } = req.body;

  // 验证用户凭证
  const user = await validateCredentials(username, password);
  if (!user) {
    return res.status(401).json({ error: '用户名或密码错误' });
  }

  // 创建 Session
  req.session.userId = user.id;
  req.session.username = user.username;
  req.session.role = user.role;

  // 保存 Session（显式）
  req.session.save((err) => {
    if (err) {
      console.error('Session 保存失败:', err);
      return res.status(500).json({ error: '登录失败' });
    }

    res.json({
      success: true,
      message: '登录成功',
      user: { id: user.id, username: user.username },
    });
  });
});

// 登出处理
app.post('/api/logout', (req, res) => {
  req.session.destroy((err) => {
    if (err) {
      console.error('Session 销毁失败:', err);
      return res.status(500).json({ error: '登出失败' });
    }

    res.clearCookie('session_id');
    res.json({ success: true, message: '登出成功' });
  });
});

// Session 认证中间件
const requireAuth = (req, res, next) => {
  if (!req.session || !req.session.userId) {

};

// 受保护的路由

  res.json({
    user: {
      id: req.session.userId,
      username: req.session.username,
      role: req.session.role,
    },
  });
});
```

#### 5.1.2 JWT 认证原理

JWT 是一种无状态的认证机制。服务器不需要存储 Token，所有用户信息都编码在 Token 本身中。

```javascript
// ============================================
// JWT 认证实现
// ============================================

const jwt = require('jsonwebtoken');
const express = require('express');
const app = express();

// JWT 密钥（生产环境应使用复杂密钥）
const JWT_SECRET = process.env.JWT_SECRET || 'your-secret-key';
const JWT_EXPIRES_IN = '7d';

// 登录并生成 Token
app.post('/api/login', async (req, res) => {
  const { username, password } = req.body;

  // 验证用户凭证
  const user = await validateCredentials(username, password);
  if (!user) {
    return res.status(401).json({ error: '用户名或密码错误' });
  }

  // 生成 JWT
  const token = jwt.sign(
    {
      sub: user.id,           // 用户 ID（建议用 sub）
      username: user.username,
      role: user.role,
      iat: Math.floor(Date.now() / 1000),  // 签发时间
    },
    JWT_SECRET,
    { expiresIn: JWT_EXPIRES_IN }
  );

  // 返回 Token
  res.json({
    success: true,
    token,
    expiresIn: JWT_EXPIRES_IN,
  });
});

// JWT 验证中间件
const authenticateToken = (req, res, next) => {
  // 从 Authorization header 获取 Token
  const authHeader = req.headers['authorization'];
  const token = authHeader && authHeader.split(' ')[1];  // Bearer TOKEN

  if (!token) {
    return res.status(401).json({ error: '未提供认证令牌' });
  }

  // 验证 Token
  jwt.verify(token, JWT_SECRET, (err, decoded) => {
    if (err) {
      if (err.name === 'TokenExpiredError') {
        return res.status(401).json({ error: '令牌已过期' });
      }
      if (err.name === 'JsonWebTokenError') {
        return res.status(401).json({ error: '无效的令牌' });
      }
      return res.status(401).json({ error: '认证失败' });
    }

    // 将解码后的用户信息挂载到 req
    req.user = decoded;
    next();
  });
};

// Token 刷新（可选）
app.post('/api/refresh', authenticateToken, (req, res) => {
  // 验证旧 Token
  const user = req.user;

  // 生成新 Token
  const newToken = jwt.sign(
    {
      sub: user.sub,
      username: user.username,
      role: user.role,
    },
    JWT_SECRET,
    { expiresIn: JWT_EXPIRES_IN }
  );

  res.json({ success: true, token: newToken });
});

// 受保护的路由
app.get('/api/profile', authenticateToken, (req, res) => {
  res.json({
    user: {
      id: req.user.sub,
      username: req.user.username,
      role: req.user.role,
    },
  });
});

// ============================================
// JWT 安全最佳实践
// ============================================

/*
1. Token 存储安全
   - 避免 localStorage（XSS 攻击）
   - 推荐使用 HttpOnly Cookie

2. Token 传输安全
   - 必须使用 HTTPS
   - Authorization: Bearer <token>

3. Token 刷新策略
   - Access Token：短期（15分钟）
   - Refresh Token：长期（7天）
   - 使用 Refresh Token 获取新的 Access Token

4. Token 撤销机制
   - 使用黑名单（Redis）
   - 或者使用短过期时间 + 主动刷新

5. 防止 CSRF
   - 使用 SameSite Cookie
   - 或者验证 Origin/Referer 头
*/
```

#### 5.1.3 JWT 与 Session 对比

| 维度 | JWT | Session |
|------|-----|---------|
| **存储位置** | 客户端 | 服务端（内存/Redis/DB） |
| **扩展性** | 无状态，易于分布式 | 需要共享 Session 存储 |
| **安全性** | 防篡改，但可能被窃取 | 依赖 Cookie 安全 |
| **性能** | 无需查询，高性能 | 每次请求需查询 Session |
| **失效控制** | 困难（只能等过期） | 可立即失效 |
| **数据容量** | 小（受 Cookie 限制） | 无限制 |
| **适用场景** | 前后端分离、微服务 | 传统 Web 应用 |

```javascript
// JWT 认证路由
```

单点登录（Single Sign-On）允许用户使用一组凭证访问多个相关系统。

```javascript
    return next();

    // 没有 ticket，重定向到 CAS 登录

    if (err || !user) {
      return res.status(401).json({ error: '无效的 Ticket' });

    // 创建 session

    next();
  });
};
```

```javascript
// ============================================

class AuthService {
  constructor(redisClient) {
    this.redis = redisClient;
  }

  // 生成 Access Token
  generateAccessToken(user) {
    return jwt.sign(
      {
        sub: user.id,
        username: user.username,
        role: user.role,
        type: 'access',
      },
      AUTH_CONFIG.jwt.secret,
      {
        expiresIn: AUTH_CONFIG.jwt.expiresIn,
        issuer: AUTH_CONFIG.jwt.issuer,
        audience: AUTH_CONFIG.jwt.audience,
      }
    );
  }

  // 生成 Refresh Token
  generateRefreshToken(user) {
    return jwt.sign(
      {
        sub: user.id,
        type: 'refresh',
        jti: crypto.randomBytes(16).toString('hex'),  // JWT ID，用于撤销
      },
      AUTH_CONFIG.jwt.secret,
      {
        expiresIn: AUTH_CONFIG.jwt.refreshExpiresIn,
        issuer: AUTH_CONFIG.jwt.issuer,
      }
    );
  }

  // 验证 Token
  async verifyToken(token) {
    try {
      const decoded = jwt.verify(token, AUTH_CONFIG.jwt.secret, {
        issuer: AUTH_CONFIG.jwt.issuer,
        audience: AUTH_CONFIG.jwt.audience,
      });

      // 检查 Token 是否在黑名单
      if (decoded.type === 'access') {
        const isBlacklisted = await this.redis.get(`blacklist:${decoded.jti || decoded.sub}`);
        if (isBlacklisted) {
          throw new Error('Token 已被撤销');
        }
      }

      return decoded;
    } catch (err) {
      if (err.name === 'TokenExpiredError') {
        throw new AuthError('令牌已过期', 'TOKEN_EXPIRED');
      }
      if (err.name === 'JsonWebTokenError') {
        throw new AuthError('无效的令牌', 'INVALID_TOKEN');
      }
      throw err;
    }
  }

  // 撤销 Token（登出）
  async revokeToken(token) {
    const decoded = jwt.decode(token);

    if (decoded && decoded.exp) {
      const ttl = decoded.exp - Math.floor(Date.now() / 1000);
      if (ttl > 0) {
        const key = `blacklist:${decoded.jti || decoded.sub}`;
        await this.redis.setex(key, ttl, 'revoked');
      }
    }
  }

  // 刷新 Token
  async refreshTokens(refreshToken) {
    const decoded = await this.verifyToken(refreshToken);

    if (decoded.type !== 'refresh') {
      throw new AuthError('无效的 Refresh Token', 'INVALID_REFRESH_TOKEN');
    }

    // 获取用户信息
    const user = await User.findById(decoded.sub);
    if (!user) {
      throw new AuthError('用户不存在', 'USER_NOT_FOUND');
    }

    // 生成新的 Token 对
    return {
      accessToken: this.generateAccessToken(user),
      refreshToken: this.generateRefreshToken(user),
    };
  }
}
```

## 6. 限流中间件

### 6.1 限流算法：令牌桶、滑动窗口

限流是保护系统免受过载的重要手段。常见的限流算法包括**令牌桶算法**、**滑动窗口算法**、**漏桶算法**等。

#### 6.1.1 令牌桶算法

令牌桶算法的核心思想是：以固定速率向桶中添加令牌，每次请求从桶中取出一个令牌，桶满时新令牌被丢弃。

```javascript
// ============================================
// 令牌桶限流实现
// ============================================

class TokenBucket {
  constructor(capacity, refillRate) {
    this.capacity = capacity;      // 桶的容量
    this.tokens = capacity;        // 当前令牌数
    this.refillRate = refillRate;  // 每秒补充的令牌数
    this.lastRefill = Date.now();  // 上次补充时间
    this.refilling = false;        // 是否正在补充
  }

  // 补充令牌
  refill() {
    const now = Date.now();
    const elapsed = (now - this.lastRefill) / 1000;  // 秒
    const tokensToAdd = elapsed * this.refillRate;

    this.tokens = Math.min(this.capacity, this.tokens + tokensToAdd);
    this.lastRefill = now;
  }

  // 尝试获取令牌
  tryConsume(tokens = 1) {
    this.refill();

    if (this.tokens >= tokens) {
      this.tokens -= tokens;
      return true;
    }

    return false;
  }

  // 获取当前状态
  getStatus() {
    this.refill();
    return {
      tokens: this.tokens,
      capacity: this.capacity,
      refillRate: this.refillRate,
    };
  }
}

// 分布式令牌桶（基于 Redis）
class DistributedTokenBucket {
  constructor(redis, key, capacity, refillRate) {
    this.redis = redis;
    this.key = key;
    this.capacity = capacity;
    this.refillRate = refillRate;
  }

  // Lua 脚本保证原子性
  async tryConsume(tokens = 1) {
    const luaScript = `
      local key = KEYS[1]
      local capacity = tonumber(ARGV[1])
      local refillRate = tonumber(ARGV[2])
      local now = tonumber(ARGV[3])
      local requested = tonumber(ARGV[4])

      -- 获取当前状态
      local data = redis.call('HMGET', key, 'tokens', 'lastRefill')
      local currentTokens = tonumber(data[1]) or capacity
      local lastRefill = tonumber(data[2]) or now

      -- 计算应补充的令牌
      local elapsed = (now - lastRefill) / 1000
      local tokensToAdd = elapsed * refillRate
      currentTokens = math.min(capacity, currentTokens + tokensToAdd)

      -- 检查是否可以获取令牌
      if currentTokens >= requested then
        currentTokens = currentTokens - requested
        redis.call('HMSET', key, 'tokens', currentTokens, 'lastRefill', now)
        redis.call('EXPIRE', key, 3600)
        return 1
      else
        redis.call('HMSET', key, 'tokens', currentTokens, 'lastRefill', now)
        redis.call('EXPIRE', key, 3600)
        return 0
      end
    `;

    const result = await this.redis.eval(
      luaScript,
      1,
      this.key,
      this.capacity,
      this.refillRate,
      Date.now(),
      tokens
    );

    return result === 1;
  }
}

// 使用
const bucket = new TokenBucket(100, 10);  // 容量100，每秒补充10个

app.get('/api/data', async (req, res) => {
  if (bucket.tryConsume()) {
    res.json({ data: 'success' });
  } else {
    res.status(429).json({ error: '请求过于频繁' });
  }
});
```

#### 6.1.2 滑动窗口算法

滑动窗口算法将时间划分为更小的窗口，统计当前窗口内的请求数，实现更平滑的限流。

```javascript
// ============================================
// 滑动窗口限流实现
// ============================================

class SlidingWindow {
  constructor(windowSizeMs, maxRequests) {
    this.windowSizeMs = windowSizeMs;  // 窗口大小（毫秒）
    this.maxRequests = maxRequests;    // 窗口内最大请求数
    this.requests = [];                 // 请求时间戳数组
  }

  // 清理过期的请求记录
  cleanOldRequests() {
    const now = Date.now();
    const cutoff = now - this.windowSizeMs;
    this.requests = this.requests.filter(time => time > cutoff);
  }

  // 尝试处理请求
  tryConsume() {
    this.cleanOldRequests();

    if (this.requests.length < this.maxRequests) {
      this.requests.push(Date.now());
      return true;
    }

    return false;
  }

  // 获取当前窗口请求数
  getCount() {
    this.cleanOldRequests();
    return this.requests.length;
  }

  // 获取剩余配额
  getRemaining() {
    this.cleanOldRequests();
    return Math.max(0, this.maxRequests - this.requests.length);
  }
}

// 分布式滑动窗口（使用 Redis Sorted Set）
class DistributedSlidingWindow {
  constructor(redis, key, windowSizeMs, maxRequests) {
    this.redis = redis;
    this.key = key;
    this.windowSizeMs = windowSizeMs;
    this.maxRequests = maxRequests;
  }

  async tryConsume() {
    const now = Date.now();
    const windowStart = now - this.windowSizeMs;

    // 事务保证原子性
    const multi = this.redis.multi();

    // 移除窗口外的记录
    multi.zremrangebyscore(this.key, 0, windowStart);

    // 获取当前窗口请求数
    multi.zcard(this.key);

    // 添加当前请求
    multi.zadd(this.key, now, `${now}-${Math.random()}`);

    // 设置过期时间
    multi.expire(this.key, Math.ceil(this.windowSizeMs / 1000) + 1);

    const results = await multi.exec();

    // results[1] 是当前窗口请求数
    const currentCount = results[1];

    if (currentCount < this.maxRequests) {
      return true;
    }

    // 如果超限，删除刚添加的请求
    await this.redis.zremrangebyscore(this.key, now, now);
    return false;
  }

  async getStatus() {
    const now = Date.now();
    const windowStart = now - this.windowSizeMs;

    await this.redis.zremrangebyscore(this.key, 0, windowStart);
    const count = await this.redis.zcard(this.key);

    return {
      remaining: Math.max(0, this.maxRequests - count),
      limit: this.maxRequests,
      resetMs: this.windowSizeMs,
    };
  }
}
```

#### 6.1.3 漏桶算法

漏桶算法以固定速率从桶中流出（处理请求），桶满时新请求被丢弃。

```javascript
// ============================================
// 漏桶限流实现
// ============================================

class LeakyBucket {
  constructor(bucketSize, leakRate) {
    this.bucketSize = bucketSize;  // 桶的容量
    this.leakRate = leakRate;      // 漏出速率（每秒处理请求数）
    this.water = 0;                // 当前水量（积压请求数）
    this.lastLeak = Date.now();    // 上次漏水时间
  }

  // 漏水
  leak() {
    const now = Date.now();
    const elapsed = (now - this.lastLeak) / 1000;
    const leaked = elapsed * this.leakRate;

    this.water = Math.max(0, this.water - leaked);
    this.lastLeak = now;
  }

  // 尝试处理请求
  tryConsume() {
    this.leak();

    if (this.water < this.bucketSize) {
      this.water += 1;
      return true;
    }

    return false;
  }
}
```

```javascript
// ============================================
// LRU 缓存实现
// ============================================

class LRUCache {
  constructor(maxSize, ttl) {
    this.maxSize = maxSize;      // 最大缓存数量
    this.ttl = ttl || 0;         // TTL（毫秒），0 表示永不过期
    this.cache = new Map();       // 使用 Map 保持插入顺序
  }

  get(key) {
    if (!this.cache.has(key)) {
      return undefined;
    }

    const entry = this.cache.get(key);

    // 检查过期
    if (this.ttl && Date.now() - entry.timestamp > this.ttl) {
      this.cache.delete(key);
      return undefined;
    }

    // 移动到末尾（最近使用）
    this.cache.delete(key);
    this.cache.set(key, entry);

    return entry.value;
  }

  set(key, value) {
    // 如果已存在，先删除
    if (this.cache.has(key)) {
      this.cache.delete(key);
    }

    // 如果已满，删除最旧的
    if (this.cache.size >= this.maxSize) {
      const oldestKey = this.cache.keys().next().value;
      this.cache.delete(oldestKey);
    }

    // 添加新条目
    this.cache.set(key, {
      value,
      timestamp: Date.now(),
    });
  }

  has(key) {
    const value = this.get(key);
    return value !== undefined;
  }

  delete(key) {
    return this.cache.delete(key);
  }

  clear() {
    this.cache.clear();
  }

  size() {
    return this.cache.size;
  }
}
```

### 7.2 Redis 缓存：ioredis

Redis 是生产环境最常用的缓存方案。

```javascript
// ============================================
// ioredis 缓存封装
// ============================================

const Redis = require('ioredis');

class RedisCache {
  constructor(options = {}) {
    this.redis = new Redis({
      host: options.host || 'localhost',
      port: options.port || 6379,
      password: options.password,
      db: options.db || 0,
      retryStrategy: (times) => Math.min(times * 50, 2000),
      maxRetriesPerRequest: 3,
    });

    this.prefix = options.prefix || 'cache:';
    this.defaultTTL = options.ttl || 3600;  // 默认 1 小时

    this.redis.on('error', (err) => {
      console.error('Redis 连接错误:', err);
    });
  }

  // 生成带前缀的键
  _key(key) {
    return `${this.prefix}${key}`;
  }

  // 设置缓存
  async set(key, value, ttl = this.defaultTTL) {
    const serialized = typeof value === 'string' ? value : JSON.stringify(value);
    const fullKey = this._key(key);

    if (ttl > 0) {
      await this.redis.setex(fullKey, ttl, serialized);
    } else {
      await this.redis.set(fullKey, serialized);
    }
  }

  // 获取缓存
  async get(key) {
    const fullKey = this._key(key);
    const value = await this.redis.get(fullKey);

    if (!value) {
      return undefined;
    }

    try {
      return JSON.parse(value);
    } catch {
      return value;
    }
  }

  // 获取并删除
  async take(key) {
    const fullKey = this._key(key);
    const value = await this.redis.getdel(fullKey);

    if (!value) {
      return undefined;
    }

    try {
      return JSON.parse(value);
    } catch {
      return value;
    }
  }

  // 删除缓存
  async del(key) {
    return await this.redis.del(this._key(key));
  }

  // 批量删除（支持通配符）
  async delPattern(pattern) {
    const fullPattern = this._key(pattern);
    const keys = await this.redis.keys(fullPattern);

    if (keys.length > 0) {
      return await this.redis.del(...keys);
    }

    return 0;
  }

  // 设置 Hash 缓存
  async hSet(key, field, value) {
    return await this.redis.hset(this._key(key), field, JSON.stringify(value));
  }

  async hGet(key, field) {
    const value = await this.redis.hget(this._key(key), field);
    return value ? JSON.parse(value) : undefined;
  }

  async hGetAll(key) {
    const data = await this.redis.hgetall(this._key(key));
    const result = {};

    for (const [field, value] of Object.entries(data)) {
      try {
        result[field] = JSON.parse(value);
      } catch {
        result[field] = value;
      }
    }

    return result;
  }

  // 增加/减少
  async incr(key) {
    return await this.redis.incr(this._key(key));
  }

  async decr(key) {
    return await this.redis.decr(this._key(key));
  }

  // 设置过期时间
  async expire(key, ttl) {
    return await this.redis.expire(this._key(key), ttl);
  }

  // 获取 TTL
  async ttl(key) {
    return await this.redis.ttl(this._key(key));
  }

  // 批量获取
  async mGet(keys) {
    if (keys.length === 0) {
      return [];
    }

    const fullKeys = keys.map(k => this._key(k));
    const values = await this.redis.mget(...fullKeys);

    return values.map(v => {
      if (!v) return undefined;
      try {
        return JSON.parse(v);
      } catch {
        return v;
      }
    });
  }

  // 批量设置
  async mSet(pairs, ttl = this.defaultTTL) {
    const args = [];

    for (const [key, value] of Object.entries(pairs)) {
      args.push(this._key(key));
      args.push(typeof value === 'string' ? value : JSON.stringify(value));
    }

    if (ttl > 0) {
      // 使用 pipeline 设置带 TTL 的多个键
      const pipeline = this.redis.pipeline();
      for (let i = 0; i < args.length; i += 2) {
        pipeline.setex(args[i], ttl, args[i + 1]);
      }
      await pipeline.exec();
    } else {
      await this.redis.mset(...args);
    }
  }
}

// ============================================
// Cache-Aside 模式
// ============================================

/*
Cache-Aside（旁路缓存）是最常用的缓存策略：

1. 读操作：
   - 先查缓存
   - 缓存命中 → 直接返回
   - 缓存未命中 → 查数据库 → 写入缓存 → 返回数据

2. 写操作：
   - 先更新数据库
   - 再删除缓存（注意是删除而非更新）
*/

class UserService {
  constructor(cache) {
    this.cache = cache;
    this.cacheTTL = 3600;  // 1 小时
  }

  // 获取用户（Cache-Aside 读）
  async getUser(userId) {
    const cacheKey = `user:${userId}`;

    // 1. 先查缓存
    let user = await this.cache.get(cacheKey);

    if (user) {
      console.log('缓存命中:', userId);
      return user;
    }

    // 2. 缓存未命中，查数据库
    console.log('缓存未命中，查询数据库:', userId);
    user = await db.query('SELECT * FROM users WHERE id = ?', [userId]);

    if (user) {
      // 3. 写入缓存
      await this.cache.set(cacheKey, user, this.cacheTTL);
    }

    return user;
  }

  // 更新用户（Cache-Aside 写）
  async updateUser(userId, data) {
    // 1. 先更新数据库
    await db.query('UPDATE users SET ? WHERE id = ?', [data, userId]);

    // 2. 删除缓存（不是更新！）
    // 为什么是删除而不是更新？
    // - 更新可能导致数据不一致（并发情况）
    // - 删除更简单，下次访问自动重新缓存
    await this.cache.del(`user:${userId}`);

    return true;
  }

  // 删除用户
  async deleteUser(userId) {
    // 1. 先删除数据库
    await db.query('DELETE FROM users WHERE id = ?', [userId]);

    // 2. 删除缓存
    await this.cache.del(`user:${userId}`);

    return true;
  }
}
```

### 7.3 多级缓存实现

```javascript
// ============================================
// 多级缓存实现
// ============================================

class MultiLevelCache {
  constructor(options = {}) {
    // L1: 内存缓存
    this.l1 = new NodeCache({
      stdTTL: options.l1Ttl || 60,      // 1 分钟
      checkperiod: options.checkperiod || 30,
    });

    // L2: Redis 缓存
    this.l2 = new RedisCache({
      host: options.redisHost,
      port: options.redisPort,
      prefix: options.prefix || 'mlc:',
      ttl: options.l2Ttl || 3600,       // 1 小时
    });
  }

  // 获取缓存（先 L1，再 L2）
  async get(key) {
    // L1 查找
    let value = this.l1.get(key);
    if (value !== undefined) {
      console.log('L1 命中:', key);
      return value;
    }

    // L2 查找
    value = await this.l2.get(key);
    if (value !== undefined) {
      console.log('L2 命中:', key);
      // 回填 L1
      this.l1.set(key, value, 60);
      return value;
    }

    console.log('L1/L2 都未命中:', key);
    return undefined;
  }

  // 设置缓存（同时写入 L1 和 L2）
  async set(key, value, options = {}) {
    const l1Ttl = options.l1Ttl || 60;
    const l2Ttl = options.l2Ttl || 3600;

    // 同时写入两级缓存
    this.l1.set(key, value, l1Ttl);
    await this.l2.set(key, value, l2Ttl);
  }

  // 删除缓存（同时删除 L1 和 L2）
  async del(key) {
    this.l1.del(key);
    await this.l2.del(key);
  }

  // 失效（只删除 L1，用于缓存预热场景）
  async invalidate(key) {
    this.l1.del(key);
  }

  // 清空所有缓存
  async flushAll() {
    this.l1.flushAll();
    await this.l2.delPattern('*');
  }
}

// ============================================
// 使用多级缓存
// ============================================

const multiCache = new MultiLevelCache({
  redisHost: 'localhost',
  redisPort: 6379,
  l1Ttl: 60,
  l2Ttl: 3600,
});

// 获取数据
async function getArticle(articleId) {
  const cacheKey = `article:${articleId}`;

  // 尝试从缓存获取
  let article = await multiCache.get(cacheKey);

  if (!article) {
    // 缓存未命中，从数据库获取
    article = await db.query('SELECT * FROM articles WHERE id = ?', [articleId]);

    if (article) {
      // 写入缓存
      await multiCache.set(cacheKey, article);
    }
  }

  return article;
}
```

```javascript
// ============================================
// 洋葱模型实现原理
// ============================================

// compose 函数实现
const compose = (middlewares) => {
  return (context, next) => {
    let index = -1;

    const dispatch = (i) => {
      if (i <= index) {
        throw new Error('next() 被调用多次');
      }
      index = i;

      if (i === middlewares.length) {
        // 所有中间件都执行完毕，调用最后的 next
        return next ? next(context) : Promise.resolve();
      }

      const middleware = middlewares[i];

      try {
        // 执行中间件，传入下一个 dispatch
        return Promise.resolve(
          middleware(context, () => dispatch(i + 1))
        );
      } catch (err) {
        return Promise.reject(err);
      }
    };

    return dispatch(0);
  };
};

// ============================================
// Koa 风格中间件示例
// ============================================
```

```javascript
const express = require('express');

// 1. 请求 ID 中间件

const requestIdMiddleware = (req, res, next) => {
  // 生成或使用传入的请求 ID
  req.id = req.headers['x-request-id'] || generateUUID();
  res.setHeader('X-Request-Id', req.id);

  next();
};

// ============================================
// 2. 请求日志中间件
// ============================================

const requestLoggingMiddleware = (req, res, next) => {
  const startTime = Date.now();

  // 请求开始日志
  req.log('info', '请求开始', {
    method: req.method,
    url: req.url,
    ip: req.ip,
    userAgent: req.get('user-agent'),
  });

  // 监听响应完成
  res.on('finish', () => {
    const duration = Date.now() - startTime;

    // 根据状态码选择日志级别
    const level = res.statusCode >= 500 ? 'error' :
                  res.statusCode >= 400 ? 'warn' : 'info';

    req.log(level, '请求完成', {
      method: req.method,
      url: req.url,
      statusCode: res.statusCode,
      duration,
    });
  });

  next();
};

// ============================================
// 3. 请求超时中间件
// ============================================

const timeoutMiddleware = (timeout = 30000) => {
  return (req, res, next) => {
    const timer = setTimeout(() => {
      if (!res.headersSent) {
        req.log('warn', '请求超时', { timeout });
        res.status(408).json({
          success: false,
          error: {
            code: 'REQUEST_TIMEOUT',
            message: `请求超时（${timeout}ms）`,
          },
        });
      }
    }, timeout);

    // 响应完成时清除超时
    res.on('finish', () => clearTimeout(timer));

    next();
  };
};

// ============================================
// 4. CORS 中间件
// ============================================

const corsMiddleware = (options = {}) => {
  const defaults = {
    origin: '*',
    methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'OPTIONS'],
    allowedHeaders: ['Content-Type', 'Authorization'],
    exposedHeaders: ['X-Request-Id'],
    credentials: false,
    maxAge: 86400,  // 24 小时
  };

  const config = { ...defaults, ...options };

  return (req, res, next) => {
    const origin = req.headers.origin;

    // 检查 origin 是否被允许
    if (typeof config.origin === 'function') {
      config.origin(req, (err, allow) => {
        if (err) return next(err);
        if (allow) {
          res.setHeader('Access-Control-Allow-Origin', origin || '*');
        }
      });
    } else if (config.origin === '*') {
      res.setHeader('Access-Control-Allow-Origin', '*');
    } else if (Array.isArray(config.origin) && config.origin.includes(origin)) {
      res.setHeader('Access-Control-Allow-Origin', origin);
    }

    if (config.credentials) {
      res.setHeader('Access-Control-Allow-Credentials', 'true');
    }

    if (config.exposedHeaders) {
      res.setHeader('Access-Control-Expose-Headers', config.exposedHeaders.join(','));
    }

    if (config.methods) {
      res.setHeader('Access-Control-Allow-Methods', config.methods.join(','));
    }

    if (config.allowedHeaders) {
      res.setHeader('Access-Control-Allow-Headers', config.allowedHeaders.join(','));
    }

    if (config.maxAge) {
      res.setHeader('Access-Control-Max-Age', String(config.maxAge));
    }

    // 处理预检请求
    if (req.method === 'OPTIONS') {
      return res.status(204).end();
    }

    next();
  };
};

    const accept = req.headers['accept-encoding'] || '';
    if (accept.includes('br')) {
      res.setHeader('Content-Encoding', 'br');
      return 'br';
    }
    if (accept.includes('gzip')) {
      res.setHeader('Content-Encoding', 'gzip');
      return 'gzip';
    }
    if (accept.includes('deflate')) {
      res.setHeader('Content-Encoding', 'deflate');
      return 'deflate';
    }

    return false;
  },
  threshold: 1024,  // 小于 1KB 的响应不压缩
});

// ============================================
// 应用所有中间件
// ============================================

app.use(requestIdMiddleware);
app.use(corsMiddleware({ origin: process.env.ALLOWED_ORIGINS?.split(',') }));
app.use(compressionMiddleware);
app.use(timeoutMiddleware(30000));
app.use(requestLoggingMiddleware);
app.use(express.json());
```

### 12.4 我的思考：选型要有依据

在多年的技术选型实践中，我总结了以下原则：

**第一，不要为了技术而技术**

引入新的中间件或框架意味着增加维护成本。在选型之前，先问自己：现有的方案是否真的无法满足需求？很多时候，简单的手写实现比引入一个重型库更合适。

**第二，考虑团队的长期维护能力**

再好的技术，如果团队无法驾驭，就是风险。选择团队有经验的技术栈，或者选择学习曲线平缓的替代方案。

**第三，预留扩展能力**

选型时要考虑未来的变化。例如，选择日志库时要考虑是否支持多种输出格式、是否可以接入日志收集系统。

**第四，性能与可维护性的平衡**

极致性能意味着更少的抽象和更多的手动配置。如果业务复杂度高、需求变化快，适当放弃性能换取可维护性是值得的。

**第五，生态与社区**

选择有活跃社区和完善文档的技术。万一遇到问题，需要能够快速找到解决方案。

```
推荐顺序：
```

### 附录 B：常见中间件性能问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 中间件执行慢 | 中间件中有阻塞操作 | 使用 async/await，避免同步操作 |
| 内存泄漏 | 中间件中创建的定时器/事件监听器未清理 | 在响应完成后清理资源 |
| 高 CPU | 复杂的验证规则 | 预编译 Schema，缓存结果 |
| 网络延迟 | 中间件中有外部 API 调用 | 添加超时，优雅降级 |
