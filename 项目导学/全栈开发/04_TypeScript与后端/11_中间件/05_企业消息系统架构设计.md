# 企业消息系统架构设计

## 前言：为什么企业需要消息系统架构？

想象一下你是一个大型企业的 IT 架构师。公司的业务系统越来越多：

- 订单系统需要通知库存系统
- 库存系统需要通知物流系统
- 物流系统需要通知用户
- 财务系统需要结算
- 监控系统需要收集所有系统的状态

如果每个系统都直接调用其他系统，会形成一张密密麻麻的网，这就是"烟囱式架构"的问题：

- 一个系统挂了，会影响很多其他系统
- 新增一个需求，需要改很多系统的代码
- 很难知道消息是否被正确处理
- 无法处理高并发流量

**企业消息系统架构就是来解决这些问题的**，它通过消息队列作为"中介"，把系统之间的调用关系变成"发布-订阅"关系，让整个系统变得松耦合、高可用、可扩展。

## 第一章：多环境隔离架构设计

### 1.1 为什么要多环境隔离？

想象一下你家的电路设计：
- 厨房有独立的空开
- 卫生间有独立的空开
- 卧室有独立的空开

这样设计的好处是：
- 厨房短路了，不会影响卧室的灯
- 可以单独给某个房间断电维修
- 出问题时容易定位是哪个房间

**多环境隔离就是这个原理**。企业消息系统需要隔离不同环境：
- 开发环境：开发人员测试
- 测试环境：QA 团队测试
- 预发布环境：上线前最后的验证
- 生产环境：真实用户使用

### 1.2 RabbitMQ 多环境隔离方案

```python
# RabbitMQ 多环境隔离架构
# ================================================

"""
方案一：虚拟主机（Virtual Host）隔离

每个环境一个独立的虚拟主机，完美隔离

开发环境 ──▶ vhost: /dev ──▶ Exchange/Queue
测试环境 ──▶ vhost: /test ──▶ Exchange/Queue
预发布环境 ──▶ vhost: /staging ──▶ Exchange/Queue
生产环境 ──▶ vhost: /prod ──▶ Exchange/Queue
"""

import pika

class RabbitMQMultiEnvConfig:
    """
    RabbitMQ 多环境配置管理器
    """

    # 环境配置
    ENVIRONMENTS = {
        'dev': {
            'host': 'rabbitmq-dev.internal',
            'port': 5672,
            'username': 'dev_user',
            'password': 'dev_pass',
            'vhost': '/dev'
        },
        'test': {
            'host': 'rabbitmq-test.internal',
            'port': 5672,
            'username': 'test_user',
            'password': 'test_pass',
            'vhost': '/test'
        },
        'staging': {
            'host': 'rabbitmq-staging.internal',
            'port': 5672,
            'username': 'staging_user',
            'password': 'staging_pass',
            'vhost': '/staging'
        },
        'prod': {
            'host': 'rabbitmq-prod.internal',
            'port': 5672,
            'username': 'prod_user',
            'password': 'prod_pass',
            'vhost': '/prod'
        }
    }

    @staticmethod
    def get_connection(env='dev'):
        """获取指定环境的连接"""
        config = RabbitMQMultiEnvConfig.ENVIRONMENTS[env]

        credentials = pika.PlainCredentials(
            config['username'],
            config['password']
        )

        connection_params = pika.ConnectionParameters(
            host=config['host'],
            port=config['port'],
            virtual_host=config['vhost'],
            credentials=credentials,
            # 连接池配置
            connection_attempts=3,
            retry_delay=5,
            socket_timeout=5
        )

        return pika.BlockingConnection(connection_params)

    @staticmethod
    def declare_infrastructure(env='dev'):
        """
        在指定环境声明基础设施

        包含：
        - 交换机
        - 队列
        - 绑定关系
        """
        connection = RabbitMQMultiEnvConfig.get_connection(env)
        channel = connection.channel()

        # 1. 声明交换机
        # -----------------------------------------------
        # 订单交换机（Direct 类型）
        channel.exchange_declare(
            exchange='order_exchange',
            exchange_type='direct',
            durable=True
        )

        # 通知交换机（Fanout 类型）
        channel.exchange_declare(
            exchange='notification_exchange',
            exchange_type='fanout',
            durable=True
        )

        # 事件交换机（Topic 类型）
        channel.exchange_declare(
            exchange='event_exchange',
            exchange_type='topic',
            durable=True
        )

        # 2. 声明队列
        # -----------------------------------------------
        # 订单处理队列
        channel.queue_declare(
            queue='order_processing_queue',
            durable=True,
            arguments={
                'x-message-ttl': 86400000,  # 24小时 TTL
                'x-dead-letter-exchange': 'dlx_exchange',
                'x-dead-letter-routing-key': 'dead_order'
            }
        )

        # 死信队列
        channel.queue_declare(
            queue='dead_letter_queue',
            durable=True
        )

        # 通知队列
        channel.queue_declare(
            queue='notification_queue',
            durable=True
        )

        # 3. 建立绑定
        # -----------------------------------------------
        # 订单交换机绑定
        channel.queue_bind(
            exchange='order_exchange',
            queue='order_processing_queue',
            routing_key='order.created'
        )

        # 死信队列绑定
        channel.queue_bind(
            exchange='dlx_exchange',
            queue='dead_letter_queue',
            routing_key='dead_order'
        )

        # 通知交换机绑定（广播模式）
        channel.queue_bind(
            exchange='notification_exchange',
            queue='notification_queue'
        )

        connection.close()
        print(f"环境 {env} 的基础设施已声明")


# 使用示例
if __name__ == '__main__':
    # 在各环境声明基础设施
    for env in ['dev', 'test', 'staging', 'prod']:
        RabbitMQMultiEnvConfig.declare_infrastructure(env)
```

```python
# Kafka 多环境隔离架构
# ================================================

"""
方案：多集群 + Topic 命名隔离

开发环境 ──▶ Kafka Cluster A ──▶ Topic: dev_orders, dev_payments
测试环境 ──▶ Kafka Cluster B ──▶ Topic: test_orders, test_payments
生产环境 ──▶ Kafka Cluster C ──▶ Topic: orders, payments
"""

from kafka import KafkaProducer, KafkaConsumer
from kafka.admin import KafkaAdminClient, NewTopic

class KafkaMultiEnvConfig:
    """
    Kafka 多环境配置管理器
    """

    ENVIRONMENTS = {
        'dev': {
            'bootstrap_servers': ['kafka-dev-1:9092', 'kafka-dev-2:9092'],
            'security_protocol': 'PLAINTEXT',
        },
        'test': {
            'bootstrap_servers': ['kafka-test-1:9092', 'kafka-test-2:9092'],
            'security_protocol': 'PLAINTEXT',
        },
        'staging': {
            'bootstrap_servers': ['kafka-staging-1:9092', 'kafka-staging-2:9092'],
            'security_protocol': 'SASL_PLAINTEXT',
        },
        'prod': {
            'bootstrap_servers': ['kafka-prod-1:9092', 'kafka-prod-2:9092', 'kafka-prod-3:9092'],
            'security_protocol': 'SASL_SSL',
        }
    }

    # Topic 命名规范
    TOPIC_PREFIX = {
        'dev': 'dev_',
        'test': 'test_',
        'staging': 'staging_',
        'prod': ''  # 生产环境无前缀
    }

    @staticmethod
    def get_producer(env='dev'):
        """获取指定环境的生产者"""
        config = KafkaMultiEnvConfig.ENVIRONMENTS[env]

        producer = KafkaProducer(
            bootstrap_servers=config['bootstrap_servers'],
            security_protocol=config['security_protocol'],
            # 可靠性配置
            acks='all',
            retries=3,
            # 性能配置
            batch_size=16384,
            linger_ms=10,
            compression_type='snappy'
        )

        return producer

    @staticmethod
    def get_consumer(env='dev', group_id='default_group'):
        """获取指定环境的消费者"""
        config = KafkaMultiEnvConfig.ENVIRONMENTS[env]
        prefix = KafkaMultiEnvConfig.TOPIC_PREFIX[env]

        consumer = KafkaConsumer(
            bootstrap_servers=config['bootstrap_servers'],
            group_id=f'{prefix}{group_id}',  # 消费者组加前缀
            security_protocol=config['security_protocol'],
            # 消费配置
            auto_offset_reset='earliest',
            enable_auto_commit=False,
            max_poll_records=500
        )

        return consumer

    @staticmethod
    def create_topic(env, topic_name, num_partitions=6, replication_factor=3):
        """在指定环境创建 Topic"""
        config = KafkaMultiEnvConfig.ENVIRONMENTS[env]
        prefix = KafkaMultiEnvConfig.TOPIC_PREFIX[env]

        admin_client = KafkaAdminClient(
            bootstrap_servers=config['bootstrap_servers'],
            security_protocol=config['security_protocol']
        )

        topic = NewTopic(
            name=f'{prefix}{topic_name}',
            num_partitions=num_partitions,
            replication_factor=replication_factor
        )

        admin_client.create_topics([topic])
        print(f"环境 {env} 的 Topic {prefix}{topic_name} 已创建")


# Topic 命名规范示例
"""
# 生产环境
orders
payments
inventory
notifications

# 开发环境
dev_orders
dev_payments
dev_inventory
dev_notifications

这样命名的好处：
1. 不同环境的 Topic 不会混淆
2. 可以用相同的 Topic 名开发，只需切换前缀
3. 便于监控和治理
"""
```

### 1.4 统一配置中心设计

```python
# 消息队列统一配置中心
# ================================================

"""
配置中心架构：

┌─────────────────────────────────────────────────────────────────┐
│                      配置中心                                    │
│  ┌─────────────────────────────────────────────────────────┐  │
│  │  QueueConfig {                                          │  │
│  │    "dev": { "host": "...", "vhost": "/dev" },          │  │
│  │    "prod": { "host": "...", "vhost": "/prod" }          │  │
│  │  }                                                      │  │
│  │  QueueTopology {                                        │  │
│  │    "order_queue": { "env": ["dev", "prod"] },          │  │
│  │    "payment_queue": { "env": ["prod"] }                │  │
│  │  }                                                      │  │
│  └─────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
"""

import json
from typing import Dict, List, Any

class QueueConfigCenter:
    """
    消息队列统一配置中心

    功能：
    1. 统一管理所有环境的配置
    2. 定义 Topic/Queue 拓扑
    3. 动态下发配置
    4. 配置版本管理
    """

    def __init__(self):
        # 基础配置
        self.base_config = {
            'rabbitmq': RabbitMQMultiEnvConfig.ENVIRONMENTS,
            'kafka': KafkaMultiEnvConfig.ENVIRONMENTS
        }

        # Topic/Queue 拓扑定义
        self.topology = self._load_topology()

        # 当前环境
        self.current_env = 'dev'

    def _load_topology(self) -> Dict[str, Any]:
        """加载队列拓扑配置"""
        return {
            # 订单相关
            'order_exchange': {
                'type': 'rabbitmq',
                'exchange_type': 'direct',
                'queues': ['order_processing_queue', 'order_notification_queue'],
                'envs': ['dev', 'test', 'staging', 'prod']
            },
            'order_topic': {
                'type': 'kafka',
                'partitions': 6,
                'replication_factor': 3,
                'envs': ['dev', 'test', 'staging', 'prod']
            },

            # 支付相关（仅生产）
            'payment_topic': {
                'type': 'kafka',
                'partitions': 10,
                'replication_factor': 3,
                'envs': ['prod']
            },

            # 通知相关
            'notification_exchange': {
                'type': 'rabbitmq',
                'exchange_type': 'fanout',
                'queues': ['email_queue', 'sms_queue', 'push_queue'],
                'envs': ['dev', 'test', 'staging', 'prod']
            }
        }

    def get_queue_config(self, queue_name: str, env: str = None) -> Dict[str, Any]:
        """获取队列配置"""
        if env is None:
            env = self.current_env

        topology = self.topology.get(queue_name, {})

        if not topology:
            raise ValueError(f"Queue {queue_name} not found in topology")

        if env not in topology.get('envs', []):
            raise ValueError(f"Queue {queue_name} not available in {env} env")

        # 返回环境特定的配置
        return {
            'name': queue_name,
            'type': topology['type'],
            'env': env,
            'config': self.base_config[topology['type']][env],
            'topology': topology
        }

    def get_producer(self, queue_name: str, env: str = None):
        """获取生产者"""
        config = self.get_queue_config(queue_name, env)

        if config['type'] == 'rabbitmq':
            return RabbitMQMultiEnvConfig.get_connection(env)
        elif config['type'] == 'kafka':
            return KafkaMultiEnvConfig.get_producer(env)

    def get_consumer(self, queue_name: str, group_id: str, env: str = None):
        """获取消费者"""
        config = self.get_queue_config(queue_name, env)

        if config['type'] == 'rabbitmq':
            return RabbitMQMultiEnvConfig.get_connection(env)
        elif config['type'] == 'kafka':
            return KafkaMultiEnvConfig.get_consumer(env, group_id)

    def deploy_topology(self, env: str):
        """部署拓扑到指定环境"""
        print(f"部署拓扑到环境: {env}")

        for queue_name, topology in self.topology.items():
            if env not in topology.get('envs', []):
                continue

            if topology['type'] == 'rabbitmq':
                RabbitMQMultiEnvConfig.declare_infrastructure(env)

            elif topology['type'] == 'kafka':
                KafkaMultiEnvConfig.create_topic(
                    env,
                    queue_name,
                    num_partitions=topology.get('partitions', 6),
                    replication_factor=topology.get('replication_factor', 3)
                )

        print(f"环境 {env} 拓扑部署完成")
```

## 第二章：优先级队列设计

### 2.1 优先级队列的使用场景

想象一下医院的挂号系统：
- 急诊病人：立即就诊（最高优先级）
- 预约病人：按预约时间就诊（中等优先级）
- 普通病人：排队等候（最低优先级）

**优先级队列就是为这种场景设计的**，让重要消息优先被处理。

**典型使用场景**：
- 电商订单：VIP 订单优先处理
- 金融交易：大额交易优先确认
- 工单系统：紧急工单优先处理
- 监控系统：严重告警优先通知

### 2.2 RabbitMQ 优先级队列实现

```python
# RabbitMQ 优先级队列实现
# ================================================

"""
RabbitMQ 优先级队列原理：

1. 声明队列时设置 x-max-priority 参数
2. 发送消息时设置 priority 属性
3. RabbitMQ 按优先级排序，优先级高的先消费

限制：
- 优先级需要消费者支持
- 队列长度满时，低优先级消息可能被丢弃
"""

import pika
import json

class PriorityOrderSystem:
    """
    优先级订单系统

    优先级定义：
    - 0: 普通订单
    - 1: 加急订单
    - 2: VIP 订单
    - 3: 紧急订单（需要立即处理）
    """

    PRIORITY_MAP = {
        'normal': 0,
        'urgent': 1,
        'vip': 2,
        'emergency': 3
    }

    def __init__(self, env='dev'):
        from RabbitMQMultiEnvConfig import RabbitMQMultiEnvConfig
        self.connection = RabbitMQMultiEnvConfig.get_connection(env)
        self.channel = self.connection.channel()

    def declare_priority_queue(self):
        """声明优先级队列"""
        # x-max-priority: 最大优先级（0-255）
        # x-queue-mode: 队列模式，lazy 适合大量消息
        self.channel.queue_declare(
            queue='priority_orders',
            durable=True,
            arguments={
                'x-max-priority': 10,  # 支持 0-10 优先级
                'x-message-ttl': 86400000  # 24小时 TTL
            }
        )

    def publish_order(self, order_id, order_type, priority_level):
        """
        发布优先级订单

        参数：
        - order_id: 订单ID
        - order_type: 订单类型
        - priority_level: 优先级（0-10）
        """
        order_data = {
            'order_id': order_id,
            'order_type': order_type,
            'content': f'处理订单 {order_id}'
        }

        # 发送消息，设置优先级
        self.channel.basic_publish(
            exchange='',
            routing_key='priority_orders',
            body=json.dumps(order_data),
            properties=pika.BasicProperties(
                delivery_mode=2,  # 持久化
                priority=priority_level,  # 消息优先级
                content_type='application/json'
            )
        )

        print(f"订单 {order_id} 已发送，优先级: {priority_level}")

    def consume_orders(self):
        """消费订单（按优先级顺序）"""
        self.channel.basic_qos(prefetch_count=1)  # 一次处理一个

        def callback(ch, method, properties, body):
            order_data = json.loads(body)
            priority = properties.priority

            print(f"处理订单: {order_data['order_id']}, "
                  f"优先级: {priority}, "
                  f"类型: {order_data['order_type']}")

            # 模拟处理
            import time
            time.sleep(1)

            # 确认消息
            ch.basic_ack(delivery_tag=method.delivery_tag)

        self.channel.basic_consume(
            queue='priority_orders',
            on_message_callback=callback,
            auto_ack=False
        )

        print("开始消费优先级订单...")
        self.channel.start_consuming()


# 批量发送测试
def send_priority_orders():
    """发送不同优先级的订单"""
    system = PriorityOrderSystem()

    # 测试数据：不同优先级的订单
    orders = [
        # 普通订单（优先级 0）
        ('ORDER_001', 'normal', 0),
        ('ORDER_002', 'normal', 0),
        ('ORDER_003', 'normal', 0),

        # 加急订单（优先级 5）
        ('ORDER_004', 'urgent', 5),
        ('ORDER_005', 'urgent', 5),

        # VIP 订单（优先级 8）
        ('ORDER_006', 'vip', 8),

        # 紧急订单（优先级 10）
        ('ORDER_007', 'emergency', 10),
    ]

    # 按随机顺序发送
    import random
    random.shuffle(orders)

    for order_id, order_type, priority in orders:
        system.publish_order(order_id, order_type, priority)

    # 消费时会按优先级顺序处理：10 -> 8 -> 5 -> 0 -> 0 -> 0 -> 0
    system.consume_orders()
```

### 2.3 Kafka 优先级实现方案

```python
# Kafka 优先级实现方案
# ================================================

"""
Kafka 原生不支持优先级队列，但可以通过以下方案实现：

方案一：多 Topic 方案（推荐）
- 为每个优先级创建独立的 Topic
- 高优先级 Topic 消费能力更强

方案二：分区方案
- 按优先级分组到不同分区
- 不同消费者组配置不同的消费能力

方案三：自定义优先级队列
- 使用优先级排序 + 延迟重试
"""

class KafkaPrioritySystem:
    """
    Kafka 优先级队列系统（多 Topic 方案）
    """

    TOPIC_MAP = {
        10: 'priority_emergency',  # 紧急
        8: 'priority_high',        # 高优先级
        5: 'priority_medium',      # 中优先级
        0: 'priority_low'           # 普通
    }

    def __init__(self):
        from KafkaMultiEnvConfig import KafkaMultiEnvConfig
        self.producer = KafkaMultiEnvConfig.get_producer('prod')
        self.consumer_configs = {}

        # 为每个优先级创建消费者
        for priority, topic in self.TOPIC_MAP.items():
            consumer = KafkaMultiEnvConfig.get_consumer('prod', f'priority_consumer_{priority}')
            consumer.subscribe([topic])
            self.consumer_configs[priority] = consumer

    def send_message(self, message, priority=0):
        """
        发送消息（根据优先级路由到不同 Topic）
        """
        # 确定 Topic
        topic = self._get_topic_for_priority(priority)

        # 发送消息
        self.producer.send(
            topic,
            key=message.get('key'),
            value=json.dumps(message).encode('utf-8')
        )

        print(f"消息已发送到 {topic}，优先级: {priority}")

    def _get_topic_for_priority(self, priority):
        """根据优先级确定 Topic"""
        if priority >= 10:
            return self.TOPIC_MAP[10]
        elif priority >= 8:
            return self.TOPIC_MAP[8]
        elif priority >= 5:
            return self.TOPIC_MAP[5]
        else:
            return self.TOPIC_MAP[0]

    def consume_all(self):
        """
        消费所有优先级的消息

        注意：需要多线程或协程分别消费各优先级队列
        """
        import threading

        def consume_priority(priority, consumer):
            print(f"启动优先级 {priority} 的消费者")
            for message in consumer:
                print(f"处理消息: {message.value}, 优先级: {priority}")

        # 为每个优先级启动一个消费者线程
        threads = []
        for priority, consumer in self.consumer_configs.items():
            t = threading.Thread(target=consume_priority, args=(priority, consumer))
            t.start()
            threads.append(t)

        # 等待所有线程
        for t in threads:
            t.join()
```

## 第三章：消息追踪与链路监控

### 3.1 为什么要消息追踪？

想象一下快递追踪系统：
- 你买了一个快递，从广州发到北京
- 快递经过：广州仓库 → 中转站1 → 中转站2 → 北京仓库 → 派送员
- 你可以实时查看快递在哪一个环节

**消息追踪就是消息的"快递追踪系统"**，让你知道消息从产生到消费经过了哪些系统、花了多少时间、是否被正确处理。

**解决的问题**：
1. 消息丢失了？不知道在哪丢的
2. 消息延迟了？不知道在哪慢的
3. 消息重复了？不知道在哪重复的

### 3.2 消息追踪实现方案

```python
# 消息追踪系统实现
# ================================================

"""
消息追踪架构：

┌─────────────────────────────────────────────────────────────────┐
│                      消息追踪系统                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Producer ──▶ TraceId ──▶ Message ──▶ Broker ──▶ Consumer      │
│       │                                                           │
│       │ 记录发送时间                                               │
│       │                                                           │
│  Broker ──▶ 记录接收时间                                          │
│       │                                                           │
│  Consumer ──▶ 记录消费时间                                         │
│       │                                                           │
│       ▼                                                           │
│  Trace Store (Elasticsearch/MySQL)                               │
│       │                                                           │
│       ▼                                                           │
│  Trace Query API ──▶ Dashboard                                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
"""

import json
import time
import uuid
from datetime import datetime
from typing import Dict, List, Optional
from dataclasses import dataclass, asdict
from functools import wraps

@dataclass
class MessageTrace:
    """
    消息追踪实体

    记录消息的完整生命周期
    """
    trace_id: str           # 全局追踪ID
    message_id: str         # 消息ID
    producer_id: str        # 生产者ID
    topic: str              # Topic/Queue 名称
    send_time: float        # 发送时间戳
    send_status: str        # 发送状态：success/fail
    broker_recv_time: Optional[float]  # Broker 接收时间
    broker_store_time: Optional[float]  # Broker 存储时间
    consumer_recv_time: Optional[float]  # 消费者接收时间
    consumer_ack_time: Optional[float]  # 消费者确认时间
    consumer_status: Optional[str]       # 消费状态
    error_message: Optional[str]         # 错误信息
    tags: Dict[str, str]    # 自定义标签


class MessageTracer:
    """
    消息追踪器

    功能：
    1. 生成 TraceId
    2. 记录消息追踪信息
    3. 异步上报追踪数据
    """

    def __init__(self, store_backend='elasticsearch'):
        self.traces = []  # 实际应使用异步队列
        self.store_backend = store_backend

    def generate_trace_id(self) -> str:
        """生成全局追踪ID"""
        return f"trace_{uuid.uuid4().hex[:16]}"

    def generate_message_id(self) -> str:
        """生成消息ID"""
        return f"msg_{uuid.uuid4().hex[:16]}"

    def record_send(self, trace_id: str, message_id: str,
                   producer_id: str, topic: str, status: str = 'success',
                   error: str = None):
        """记录发送事件"""
        trace = MessageTrace(
            trace_id=trace_id,
            message_id=message_id,
            producer_id=producer_id,
            topic=topic,
            send_time=time.time(),
            send_status=status,
            broker_recv_time=None,
            broker_store_time=None,
            consumer_recv_time=None,
            consumer_ack_time=None,
            consumer_status=None,
            error_message=error,
            tags={}
        )

        self._save_trace(trace)
        return trace

    def record_broker_recv(self, trace_id: str, message_id: str):
        """记录 Broker 接收"""
        self._update_trace(trace_id, message_id, 'broker_recv_time', time.time())

    def record_broker_store(self, trace_id: str, message_id: str):
        """记录 Broker 存储"""
        self._update_trace(trace_id, message_id, 'broker_store_time', time.time())

    def record_consumer_recv(self, trace_id: str, message_id: str):
        """记录消费者接收"""
        self._update_trace(trace_id, message_id, 'consumer_recv_time', time.time())

    def record_consumer_ack(self, trace_id: str, message_id: str, status: str = 'success'):
        """记录消费者确认"""
        self._update_trace(trace_id, message_id, 'consumer_ack_time', time.time())
        self._update_trace(trace_id, message_id, 'consumer_status', status)

    def record_error(self, trace_id: str, message_id: str, error: str):
        """记录错误"""
        self._update_trace(trace_id, message_id, 'error_message', error)

    def add_tag(self, trace_id: str, message_id: str, key: str, value: str):
        """添加标签"""
        trace = self._get_trace(trace_id, message_id)
        if trace:
            trace.tags[key] = value

    def _save_trace(self, trace: MessageTrace):
        """保存追踪记录"""
        # 实际应异步发送到存储
        self.traces.append(trace)
        print(f"[追踪] trace_id={trace.trace_id}, msg_id={trace.message_id}, "
              f"topic={trace.topic}, send_time={trace.send_time}")

    def _update_trace(self, trace_id: str, message_id: str, field: str, value):
        """更新追踪记录"""
        for trace in self.traces:
            if trace.trace_id == trace_id and trace.message_id == message_id:
                setattr(trace, field, value)
                print(f"[追踪] trace_id={trace_id}, msg_id={message_id}, "
                      f"{field}={value}")
                break

    def _get_trace(self, trace_id: str, message_id: str) -> Optional[MessageTrace]:
        """获取追踪记录"""
        for trace in self.traces:
            if trace.trace_id == trace_id and trace.message_id == message_id:
                return trace
        return None

    def get_trace_summary(self, trace_id: str) -> Dict:
        """获取追踪摘要"""
        traces = [t for t in self.traces if t.trace_id == trace_id]

        if not traces:
            return {}

        trace = traces[0]
        summary = {
            'trace_id': trace.trace_id,
            'message_id': trace.message_id,
            'topic': trace.topic,
            'total_time': 0,
            'stages': []
        }

        # 计算各阶段耗时
        if trace.send_time and trace.broker_recv_time:
            broker_delay = trace.broker_recv_time - trace.send_time
            summary['stages'].append(f"发送→Broker: {broker_delay*1000:.2f}ms")

        if trace.broker_recv_time and trace.broker_store_time:
            store_delay = trace.broker_store_time - trace.broker_recv_time
            summary['stages'].append(f"Broker处理: {store_delay*1000:.2f}ms")

        if trace.consumer_recv_time and trace.send_time:
            total_time = trace.consumer_ack_time - trace.send_time
            summary['total_time'] = f"{total_time*1000:.2f}ms"

        return summary


# 追踪装饰器
def traced_producer(topic_name):
    """生产者追踪装饰器"""
    tracer = MessageTracer()

    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            trace_id = tracer.generate_trace_id()
            message_id = tracer.generate_message_id()

            try:
                # 发送前记录
                tracer.record_send(trace_id, message_id, 'producer_1', topic_name)

                # 执行发送
                result = func(*args, **kwargs)

                # 返回追踪信息
                return {
                    'trace_id': trace_id,
                    'message_id': message_id,
                    'result': result
                }

            except Exception as e:
                tracer.record_error(trace_id, message_id, str(e))
                raise

        return wrapper
    return decorator


# 使用示例
if __name__ == '__main__':
    tracer = MessageTracer()

    # 模拟发送消息
    trace_id = tracer.generate_trace_id()
    message_id = tracer.generate_message_id()

    time.sleep(0.1)  # 模拟延迟

    tracer.record_broker_recv(trace_id, message_id)
    time.sleep(0.05)

    tracer.record_broker_store(trace_id, message_id)
    time.sleep(0.2)

    tracer.record_consumer_recv(trace_id, message_id)
    time.sleep(0.3)

    tracer.record_consumer_ack(trace_id, message_id, 'success')

    # 添加标签
    tracer.add_tag(trace_id, message_id, 'order_type', 'vip')
    tracer.add_tag(trace_id, message_id, 'amount', '10000')

    # 获取追踪摘要
    summary = tracer.get_trace_summary(trace_id)
    print("\n追踪摘要:")
    print(json.dumps(summary, indent=2, default=str))
```

```python
# 消息链路监控
# ================================================

"""
监控大盘指标：

1. 发送指标
   - 发送速率（条/秒）
   - 发送成功率
   - 发送延迟 P99

2. 消费指标
   - 消费速率（条/秒）
   - 消费成功率
   - 消费延迟 P99
   - 消费堆积数

3. 队列指标
   - 队列深度
   - 消息 TTL
   - 死信数量

4. 系统指标
   - CPU 使用率
   - 内存使用率
   - 磁盘 IO
"""

class MessageQueueMonitor:
    """
    消息队列监控器
    """

    def __init__(self):
        self.metrics = {
            'send': [],
            'consume': [],
            'queue': []
        }

    def record_send_metric(self, topic, count, latency, success):
        """记录发送指标"""
        self.metrics['send'].append({
            'topic': topic,
            'count': count,
            'latency': latency,
            'success': success,
            'timestamp': time.time()
        })

    def record_consume_metric(self, topic, count, latency, success, lag):
        """记录消费指标"""
        self.metrics['consume'].append({
            'topic': topic,
            'count': count,
            'latency': latency,
            'success': success,
            'lag': lag,
            'timestamp': time.time()
        })

    def record_queue_metric(self, topic, depth, dlr_count):
        """记录队列指标"""
        self.metrics['queue'].append({
            'topic': topic,
            'depth': depth,
            'dead_letter_count': dlr_count,
            'timestamp': time.time()
        })

    def get_dashboard_data(self):
        """获取监控大盘数据"""
        now = time.time()
        window = 300  # 5分钟窗口

        # 计算发送速率
        recent_sends = [m for m in self.metrics['send']
                       if now - m['timestamp'] < window]

        send_rate = sum(m['count'] for m in recent_sends) / window if recent_sends else 0
        send_success_rate = (sum(1 for m in recent_sends if m['success']) /
                           len(recent_sends) * 100) if recent_sends else 0

        # 计算消费延迟 P99
        recent_consumes = [m for m in self.metrics['consume']
                          if now - m['timestamp'] < window]

        latencies = sorted([m['latency'] for m in recent_consumes])
        p99_latency = latencies[int(len(latencies) * 0.99)] if latencies else 0

        # 消费堆积
        total_lag = sum(m['lag'] for m in recent_consumes)

        return {
            'send_rate': send_rate,
            'send_success_rate': send_success_rate,
            'consume_latency_p99': p99_latency,
            'consume_lag': total_lag,
            'timestamp': now
        }

    def check_alerts(self):
        """检查告警"""
        alerts = []

        data = self.get_dashboard_data()

        # 发送成功率告警
        if data['send_success_rate'] < 99:
            alerts.append({
                'level': 'critical',
                'message': f"发送成功率过低: {data['send_success_rate']:.2f}%"
            })

        # 消费延迟告警
        if data['consume_latency_p99'] > 1000:  # 1秒
            alerts.append({
                'level': 'warning',
                'message': f"消费延迟过高: {data['consume_latency_p99']:.2f}ms"
            })

        # 消费堆积告警
        if data['consume_lag'] > 10000:
            alerts.append({
                'level': 'critical',
                'message': f"消息堆积严重: {data['consume_lag']} 条"
            })

        return alerts
```

## 第四章：幂等设计

### 4.1 幂等性的重要性

**幂等**是指一个操作执行一次和执行多次的效果相同。

**为什么重要？**

想象银行转账场景：
- 用户点击"转账100元"，网络超时
- 系统重试，又转了100元
- 结果：用户被扣了200元！

**这就是没有幂等设计的后果！**

消息队列中的幂等场景：
1. **生产者重试**：网络超时导致消息重复发送
2. **消费者重试**：消息处理失败后重新消费
3. **消息堆积**：消费者重启后重新消费积压消息

### 4.2 幂等实现方案

#### 4.2.1 唯一消息 ID 方案

```python
# 幂等实现方案一：唯一消息 ID
# ================================================

"""
原理：
1. 每条消息分配唯一 ID
2. 消费者处理前检查 ID 是否已处理
3. 已处理则跳过，未处理则处理并记录

适用场景：
- 消息量不大
- 需要精确去重
"""

import redis
import json

class IdempotentConsumer:
    """
    基于唯一 ID 的幂等消费者
    """

    def __init__(self, redis_client):
        self.redis = redis_client
        self.processed_key = 'idempotent:processed'
        self.ttl = 86400 * 7  # 7 天过期

    def is_processed(self, message_id: str) -> bool:
        """检查消息是否已处理"""
        return self.redis.exists(f'{self.processed_key}:{message_id}')

    def mark_processed(self, message_id: str):
        """标记消息已处理"""
        self.redis.setex(
            f'{self.processed_key}:{message_id}',
            self.ttl,
            '1'
        )

    def process_message(self, message_id: str, message_data: dict):
        """
        处理消息（幂等）

        流程：
        1. 检查是否已处理
        2. 已处理则跳过
        3. 未处理则处理并标记
        """
        # 1. 检查
        if self.is_processed(message_id):
            print(f"消息 {message_id} 已处理，跳过")
            return True

        # 2. 处理
        print(f"处理消息: {message_id}")

        # 模拟业务处理
        result = self._do_process(message_data)

        if result:
            # 3. 标记
            self.mark_processed(message_id)
            print(f"消息 {message_id} 处理完成，已标记")

        return result

    def _do_process(self, message_data: dict) -> bool:
        """实际业务处理"""
        # 业务逻辑
        return True


# 使用示例
```

```python
# 幂等实现方案二：数据库唯一约束
# ================================================

"""
原理：
1. 业务表添加唯一约束
2. 插入/更新时使用唯一约束字段
3. 违反唯一约束则跳过

适用场景：
- 订单处理
- 支付确认
- 状态更新
"""

class IdempotentOrderProcessor:
    """
    基于数据库唯一约束的幂等订单处理器
    """

    def __init__(self, db_connection):
        self.db = db_connection

    def process_order(self, order_id: str, amount: float, status: str):
        """
        处理订单（幂等）

        使用 INSERT ... ON DUPLICATE KEY UPDATE
        保证幂等性
        """
        sql = """
        INSERT INTO orders (order_id, amount, status, create_time, update_time)
        VALUES (%s, %s, %s, NOW(), NOW())
        ON DUPLICATE KEY UPDATE
            amount = VALUES(amount),
            status = VALUES(status),
            update_time = NOW()
        """

        cursor = self.db.cursor()
        try:
            cursor.execute(sql, (order_id, amount, status))
            self.db.commit()

            # 判断是插入还是更新
            if cursor.rowcount == 1:
                print(f"订单 {order_id} 新建成功")
            else:
                print(f"订单 {order_id} 已存在，更新成功（幂等）")

            return True

        except Exception as e:
            self.db.rollback()
            print(f"订单处理失败: {e}")
            return False

    def process_payment(self, payment_id: str, order_id: str, amount: float):
        """
        处理支付（幂等）

        使用唯一键保证幂等
        """
        sql = """
        INSERT INTO payments (payment_id, order_id, amount, status, create_time)
        VALUES (%s, %s, %s, 'success', NOW())
        ON DUPLICATE KEY UPDATE
            status = CASE
                WHEN status = 'success' THEN status
                ELSE 'success'
            END,
            update_time = NOW()
        """

        cursor = self.db.cursor()
        try:
            cursor.execute(sql, (payment_id, order_id, amount))
            self.db.commit()
            return True

        except Exception as e:
            self.db.rollback()
            print(f"支付处理失败: {e}")
            return False
```

#### 4.2.3 Redis 分布式锁方案

```python
# 幂等实现方案三：Redis 分布式锁
# ================================================

"""
原理：
1. 处理消息前获取分布式锁
2. 锁的 key 为消息 ID
3. 处理完成后释放锁

适用场景：
- 需要精确控制并发
- 消息处理时间不确定
"""

import redis
import time
import uuid

class DistributedLockIdempotent:
    """
    基于分布式锁的幂等处理器
    """

    def __init__(self, redis_client):
        self.redis = redis_client
        self.lock_prefix = 'lock:'
        self.lock_timeout = 30  # 锁超时时间（秒）

    def acquire_lock(self, message_id: str) -> str:
        """
        获取锁

        返回锁的 value（用于释放锁）
        """
        lock_key = f'{self.lock_prefix}{message_id}'
        lock_value = str(uuid.uuid4())  # 唯一值，用于安全释放

        # SET key value NX EX seconds
        # NX: 不存在时设置
        # EX: 设置过期时间
        acquired = self.redis.set(
            lock_key,
            lock_value,
            nx=True,
            ex=self.lock_timeout
        )

        return lock_value if acquired else None

    def release_lock(self, message_id: str, lock_value: str) -> bool:
        """
        释放锁

        使用 Lua 脚本保证原子性
        只有锁的值匹配时才释放
        """
        lock_key = f'{self.lock_prefix}{message_id}'

        # Lua 脚本：只有值匹配才删除
        lua_script = """
        if redis.call("get", KEYS[1]) == ARGV[1] then
            return redis.call("del", KEYS[1])
        else
            return 0
        end
        """

        result = self.redis.eval(lua_script, 1, lock_key, lock_value)
        return result == 1

    def process_with_lock(self, message_id: str, process_func):
        """
        带锁的消息处理
        """
        # 1. 获取锁
        lock_value = self.acquire_lock(message_id)

        if not lock_value:
            print(f"消息 {message_id} 正在被处理或处理超时，跳过")
            return False

        try:
            # 2. 执行处理
            result = process_func(message_id)
            return result

        finally:
            # 3. 释放锁
            self.release_lock(message_id, lock_value)

    def process_message(self, message_id: str, message_data: dict):
        """
        处理消息（带锁的幂等）
        """

        def do_process(msg_id):
            print(f"处理消息: {msg_id}")
            # 业务逻辑
            return True

        return self.process_with_lock(message_id, do_process)


# 使用示例
r = redis.Redis(host='localhost', port=6379)
processor = DistributedLockIdempotent(r)

# 处理消息
processor.process_message('MSG_001', {'data': 'test'})
```

### 4.3 幂等设计最佳实践

```python
# 幂等设计最佳实践总结
# ================================================

"""
幂等设计检查清单：

1. 消息层面
   □ 每条消息都有唯一 ID
   □ 消息 ID 包含时间戳和序号
   □ 消息 ID 足够长，避免冲突

2. 消费者层面
   □ 处理前检查是否已处理
   □ 使用数据库唯一约束
   □ 处理完成后立即标记
   □ 设置合理的标记过期时间

3. 业务层面
   □ 所有写操作都是幂等的
   □ 使用状态机控制状态流转
   □ 避免并发处理同一业务 ID

4. 监控层面
   □ 记录重复消息数量
   □ 记录幂等跳过数量
   □ 异常情况及时告警
"""

class IdempotencyBestPractices:
    """
    幂等最佳实践实现
    """

    @staticmethod
    def generate_message_id(business_type: str, business_id: str) -> str:
        """
        生成业务消息 ID

        格式：{业务类型}_{业务ID}_{时间戳}_{序号}
        示例：ORDER_ORDER123_1699999999_001
        """
        import time
        timestamp = int(time.time() * 1000)
        sequence = redis_client.incr(f'idempotent:seq:{business_type}')

        return f"{business_type}_{business_id}_{timestamp}_{sequence:04d}"

    @staticmethod
    def should_process(redis_client, message_id: str, ttl: int = 86400 * 7) -> bool:
        """
        判断是否应该处理

        使用 SETNX 保证原子性
        """
        key = f'idempotent:processed:{message_id}'

        # SETNX: 不存在时设置，返回 True
        # 存在时不设置，返回 False
        return redis_client.set(key, '1', nx=True, ex=ttl)

    @staticmethod
    def process_with_idempotency(redis_client, message_id: str, process_func,
                                 ttl: int = 86400 * 7):
        """
        带幂等性的消息处理
        """
        # 1. 检查是否应该处理
        if not IdempotencyBestPractices.should_process(redis_client, message_id, ttl):
            print(f"消息 {message_id} 已处理，跳过")
            return None

        # 2. 执行处理
        try:
            result = process_func()
            return result
        except Exception as e:
            # 处理失败，删除幂等标记，允许重试
            redis_client.delete(f'idempotent:processed:{message_id}')
            raise e
```

## 第五章：容错与高可用设计

### 5.1 消息系统容错原则

```
容错设计原则：

1. 消息不丢失
   ├── 生产者确认（acks=all）
   ├── 消息持久化（durable=true）
   ├── 副本机制（replication factor > 1）
   └── 消费确认（manual ack）

2. 消费者容错
   ├── 处理失败 → 重新入队（有限重试）
   ├── 最大重试 → 死信队列
   └── 幂等设计 → 防止重复处理

3. 系统容错
   ├── 集群部署 → 避免单点
   ├── 健康检查 → 故障转移
   └── 定时巡检 → 及时发现
```

```python
def consumer_with_retry(channel, queue_name):
    """
    带重试的消费者
    """
    def callback(ch, method, properties, body):
        max_retries = 3
        retry_count = 0

        while retry_count < max_retries:
            try:
                # 处理消息
                process_message(body)

                # 成功，确认消息
                ch.basic_ack(method.delivery_tag)
                return

            except Exception as e:
                retry_count += 1
                print(f"处理失败，第 {retry_count} 次重试")

                if retry_count >= max_retries:
                    # 超过最大重试，进入死信队列
                    print(f"超过最大重试次数，进入死信队列")
                    ch.basic_nack(method.delivery_tag, requeue=False)
                else:
                    # 重试前等待
                    time.sleep(2 ** retry_count)  # 指数退避
                    ch.basic_nack(method.delivery_tag, requeue=True)

    channel.basic_qos(prefetch_count=10)
    channel.basic_consume(
        queue=queue_name,
        on_message_callback=callback,
        auto_ack=False
    )
```

## 总结：企业消息系统架构设计要点

**核心设计原则**：

1. **隔离原则**
   - 环境隔离：dev/test/staging/prod 独立配置
   - 租户隔离：多租户场景下数据隔离
   - 业务隔离：不同业务使用不同的队列

2. **可靠性原则**
   - 消息不丢失：持久化 + 副本 + 确认
   - 幂等设计：防止重复消息
   - 容错处理：重试 + 死信队列

3. **可观测原则**
   - 消息追踪：trace_id 贯穿全链路
   - 监控指标：发送、消费、堆积
   - 告警机制：及时发现问题

4. **可扩展原则**
   - 接口抽象：方便切换底层实现
   - 流量规划：预留扩容空间
   - 架构演进：支持架构升级

**架构师 Checklist**：
- [ ] 环境配置是否正确隔离？
- [ ] 消息是否有幂等设计？
- [ ] 消费者是否有容错处理？
- [ ] 是否有消息追踪机制？
- [ ] 是否有监控告警？
- [ ] 是否有死信处理？
- [ ] 是否有容量规划？

记住：**好的架构是设计出来的，不是调试出来的**。在设计阶段多花一分精力，上线后就少十分救火。
