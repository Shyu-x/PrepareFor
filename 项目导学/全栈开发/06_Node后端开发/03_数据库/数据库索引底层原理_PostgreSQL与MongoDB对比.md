# 数据库索引底层原理：PostgreSQL 与 MongoDB 深度对比 (2026版)

## 1. 概述

在全栈开发中，数据库往往是系统性能的最终瓶颈。无论是关系型数据库（RDBMS）还是 NoSQL，其极速查询的秘密都隐藏在底层的索引数据结构中。

本指南将深入数据库的底层引擎，剖析 PostgreSQL 的 B-Tree 索引与 MongoDB 的 WiredTiger 引擎存储机制，从根本上理解为什么有些查询快如闪电，而有些则会导致全表扫描（Seq Scan）。

---

## 2. 关系型霸主：PostgreSQL 与 B-Tree 引擎

PostgreSQL 是目前最先进的开源关系型数据库。它的默认索引结构是 B-Tree（平衡多路查找树）。

### 2.1 为什么是 B-Tree 而不是红黑树/哈希表？
- **哈希表**：时间复杂度是 O(1)，极快。但它完全**不支持范围查询**（如 `age > 18`），并且在发生哈希冲突时性能会下降。
- **红黑树/二叉搜索树**：虽然支持范围查询，但每个节点只有两个子节点。当数据量达到千万级时，树的高度会非常高。因为数据库数据存储在磁盘上，树的每一次层级下降都意味着一次**磁盘 I/O（极其缓慢）**。
- **B-Tree 的降维打击**：它是一个“矮胖”的树。一个节点可以包含几百个键值，树的高度通常只有 3 到 4 层。这意味着即使在十亿级的数据表中查找一条记录，最多也只需要 3-4 次磁盘 I/O。

### 2.2 B-Tree 索引的内部结构
在 PostgreSQL 中，B-Tree 索引页面分为：
1. **根节点 (Root Node)** 和 **内部节点 (Internal Nodes)**：只存储键值（Key）和指向子节点的指针，不存储实际行数据。
2. **叶子节点 (Leaf Nodes)**：不仅存储键值，还存储一个**TID (Tuple Identifier，元组标识符)**。
   - TID 包含了：`数据块号 (Block Number)` + `块内偏移量 (Offset)`。
   - 通过 TID，PostgreSQL 可以直接去磁盘上精准读取那一行的完整数据，这个过程称为 **回表 (Table Access By Index ROWID)**。

### 2.3 联合索引与最左前缀法则
如果你创建了联合索引 `CREATE INDEX idx_a_b_c ON users (a, b, c);`
B-Tree 会先按照 `a` 排序；当 `a` 相同时，再按照 `b` 排序，依此类推。
- **命中索引**：`WHERE a = 1 AND b = 2`
- **索引失效**：`WHERE b = 2 AND c = 3`（因为跳过了 `a`，B-Tree 无法利用树结构进行二分查找，只能退化为全表扫描）。

## 3. NoSQL 之王：MongoDB 与 WiredTiger 引擎

MongoDB 在文档存储领域占据统治地位。自 3.2 版本起，它默认使用了底层的 **WiredTiger** 存储引擎。

### 3.1 B-Tree 的变形与文档存储
MongoDB 的索引同样基于 B-Tree 结构，但其叶子节点存储的不是行偏移量，而是指向 BSON 文档的指针。
与 PostgreSQL 的行式存储不同，MongoDB 存储的是自包含的 JSON/BSON 文档。

### 3.2 WiredTiger 的内存与缓存机制
WiredTiger 的性能神话来自于它激进的内存管理：
- **内部缓存 (Internal Cache)**：默认占用机器物理内存的 **50% - 1GB**。它不仅缓存磁盘上的原始数据块，还会缓存在内存中解压后的 BSON 对象。
- **无锁并发 (Document-Level Concurrency)**：MongoDB 早期的版本是数据库级别/集合级别的锁。而 WiredTiger 实现了**文档级锁**（类似于 InnoDB 的行级锁），使得并发写操作性能呈指数级提升。

### 3.3 复合索引与 ESR 原则
在 MongoDB 中建立复合索引，官方强烈推荐 **ESR 原则**：
1. **E (Equality, 等值匹配)**：把精确匹配的字段放在索引最前面（如 `user_id = 123`）。
2. **S (Sort, 排序)**：把用于 `sort()` 的字段放在中间，这样 MongoDB 可以直接按照索引顺序扫描，避免在内存中进行极其昂贵的排序操作（In-memory Sort）。
3. **R (Range, 范围)**：把范围查询（`$gt`, `$lt`）的字段放在最后。因为范围扫描一旦开始，索引的边界就被打破了，后续字段无法再用于精确定位。

## 4. 核心对比与技术选型

| 维度 | PostgreSQL | MongoDB |
| :--- | :--- | :--- |
| **数据模型** | 关系型 (Schema-rigid) | 文档型 (Schema-less, BSON) |
| **存储架构** | 行式存储 (Row-based) | WiredTiger 压缩块存储 |
| **事务支持** | 极致强事务 (ACID), MVCC 完美实现 | 支持多文档 ACID，但性能损耗较大 |
| **JOIN 性能** | 极强，支持 Hash Join, Merge Join | 较弱，使用 `$lookup` 性能开销大 |
| **适用场景** | 金融、电商订单、复杂报表统计 | 物联网数据、日志系统、高度动态的 CMS |

### 4.1 面试高频问题

**Q1：什么是索引覆盖 (Covering Index / Index-Only Scan)？**
**答：** 无论是在 PG 还是 MongoDB 中，如果查询的字段（`SELECT` 或 `find` 的投影）**完全包含**在你建立的索引中，数据库引擎就不需要拿着 TID 去磁盘上“回表”读取完整行数据了，而是直接从索引树的叶子节点返回数据。这能减少极大的磁盘 I/O 开销，是 SQL 调优的终极目标。

**Q2：为什么不建议给所有列都加上索引？**
**答：** 
1. **空间浪费**：索引本身就是一棵 B-Tree 物理文件，过多索引会导致磁盘爆炸。
2. **写放大 (Write Amplification)**：每次执行 `INSERT`、`UPDATE`、`DELETE` 时，不仅要修改表数据，还需要同步更新所有的 B-Tree 索引。索引越多，写入性能越差。
