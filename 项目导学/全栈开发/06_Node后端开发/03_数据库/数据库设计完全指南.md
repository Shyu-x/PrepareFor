### 1.2 数据库设计原则

```
数据库设计范式（Normalization）

第一范式（1NF）
└── 字段不可再分

第二范式（2NF）
└── 消除部分函数依赖

第三范式（3NF）
└── 消除传递函数依赖

BCNF范式
└── 主属性不依赖于主属性

第四范式（4NF）
└── 消除多值依赖

第五范式（5NF）
└── 消除连接依赖

反范式化（Denormalization）
└── 适当冗余，提升查询性能
```

### 1.3 数据库范式深度解析

#### 1.3.1 第一范式（1NF）—— 原子性原则

**定义**：每个字段都是不可分割的最小数据单元，表中不能有重复组。

```sql
-- ❌ 违反1NF：字段可再分
CREATE TABLE bad_orders (
    id SERIAL PRIMARY KEY,
    customer_name VARCHAR(100),     -- 包含姓名和称谓
    phone_numbers VARCHAR(200),     -- 多个电话用逗号分隔
    address VARCHAR(500),           -- 完整地址未拆分
    product_info VARCHAR(1000)      -- 产品信息混杂
);

-- 违反1NF的示例数据：
-- id | customer_name | phone_numbers       | address                  | product_info
-- 1  | "张总 先生"    | "13800138000,13900139000" | "北京市朝阳区建国路88号"   | "产品A:2件,产品B:1件"

-- ✅ 符合1NF：字段不可再分
CREATE TABLE customers (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) NOT NULL,      -- 姓名（不含称谓）
    title VARCHAR(10),               -- 称谓（先生/女士/博士等）
    phone VARCHAR(20),              -- 电话（单个）
    province VARCHAR(50),           -- 省
    city VARCHAR(50),               -- 市
    district VARCHAR(50),          -- 区
    street VARCHAR(200),            -- 街道地址
    postal_code VARCHAR(10)        -- 邮政编码
);

-- ✅ 符合1NF：多值使用单独表
CREATE TABLE customer_phones (
    id SERIAL PRIMARY KEY,
    customer_id INTEGER REFERENCES customers(id),
    phone VARCHAR(20) NOT NULL,
    phone_type VARCHAR(20) DEFAULT 'mobile'  -- mobile/home/work
);

CREATE TABLE order_items (
    id SERIAL PRIMARY KEY,
    order_id INTEGER REFERENCES orders(id),
    product_id INTEGER REFERENCES products(id),
    quantity INTEGER NOT NULL CHECK (quantity > 0)
);
```

**为什么必须满足1NF？**

| 违反1NF的问题 | 影响 | 解决方案 |
|--------------|------|----------|
| 无法单独查询姓名字段 | 查询效率低 | 拆分为独立字段 |
| 无法对电话单独操作 | 更新/删除困难 | 使用单独表存储 |
| 无法按城市统计 | 数据分析受限 | 地址字段拆分 |
| 数据验证困难 | 数据质量差 | 规范化字段类型 |

#### 1.3.2 第二范式（2NF）—— 完全函数依赖

**定义**：在满足1NF的基础上，所有非主键字段完全依赖于主键，不能只依赖主键的一部分。

```sql
-- ❌ 违反2NF：存在部分函数依赖
-- 主键是 (order_id, product_id)
-- 但 product_name 只依赖于 product_id，不完全依赖于主键
CREATE TABLE order_details_bad (
    id SERIAL PRIMARY KEY,
    order_id INTEGER,
    product_id INTEGER,
    product_name VARCHAR(200),     -- 只依赖 product_id，违反2NF
    product_price DECIMAL(10,2),  -- 只依赖 product_id，违反2NF
    quantity INTEGER,
    subtotal DECIMAL(10,2)
);

-- ❌ 违反2NF的问题数据：
-- id | order_id | product_id | product_name | product_price | quantity | subtotal
-- 1  | 100      | 1          | "iPhone 15"  | 5999.00       | 2        | 11998.00
-- 2  | 101      | 1          | "iPhone 15"   | 5999.00       | 1        | 5999.00
-- product_name 和 product_price 重复存储了！

-- ✅ 符合2NF：拆分产品信息到独立表
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(200) NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    description TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE order_items (
    id SERIAL PRIMARY KEY,
    order_id INTEGER REFERENCES orders(id),
    product_id INTEGER REFERENCES products(id),  -- 完全依赖主键
    quantity INTEGER NOT NULL,
    unit_price DECIMAL(10,2) NOT NULL,         -- 订单时的价格（快照）
    subtotal DECIMAL(10,2) NOT NULL
);
-- order_items 的所有字段都完全依赖于 (order_id, product_id)
```

**复合主键 vs 代理主键：**

```sql
-- 使用复合主键（需要确保唯一性）
CREATE TABLE order_items (
    order_id INTEGER NOT NULL,
    product_id INTEGER NOT NULL,
    quantity INTEGER NOT NULL,
    PRIMARY KEY (order_id, product_id)
);

-- 或使用代理主键（更推荐）
CREATE TABLE order_items (
    id SERIAL PRIMARY KEY,
    order_id INTEGER NOT NULL,
    product_id INTEGER NOT NULL,
    quantity INTEGER NOT NULL,
    UNIQUE (order_id, product_id)  -- 确保订单产品组合唯一
);

-- 什么时候使用复合主键？
-- 1. 关系表本身有业务含义（如订单明细的序号）
-- 2. 需要确保两个外键组合的唯一性
-- 3. 业务逻辑上不允许单个外键重复

-- 什么时候使用代理主键？
-- 1. 主键没有业务含义
-- 2. 需要统一的ID格式（UUID/自增）
-- 3. 避免主键更新级联
```

#### 1.3.3 第三范式（3NF）—— 传递依赖消除

**定义**：在满足2NF的基础上，非主键字段之间不能存在传递依赖。

```sql
-- ❌ 违反3NF：存在传递依赖
-- order_id -> customer_id -> customer_city -> city_population
CREATE TABLE orders_bad (
    id SERIAL PRIMARY KEY,
    order_number VARCHAR(50) UNIQUE NOT NULL,
    customer_id INTEGER,
    customer_name VARCHAR(100),       -- 依赖于 customer_id
    customer_city VARCHAR(50),       -- 依赖于 customer_id（传递依赖）
    city_population INTEGER,         -- 依赖于 customer_city（传递依赖）
    total_amount DECIMAL(10,2),
    created_at TIMESTAMP
);

-- 问题分析：
-- customer_name 传递依赖于 customer_id
-- customer_city 传递依赖于 customer_id
-- city_population 传递依赖于 customer_city

-- ✅ 符合3NF：拆分到多个表
CREATE TABLE customers (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    city_id INTEGER REFERENCES cities(id),
    email VARCHAR(255),
    created_at TIMESTAMP
);

CREATE TABLE cities (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    province VARCHAR(50),
    population INTEGER,  -- 城市人口与城市ID直接关联
    area_sqkm DECIMAL(10,2)
);

CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    order_number VARCHAR(50) UNIQUE NOT NULL,
    customer_id INTEGER REFERENCES customers(id),
    total_amount DECIMAL(10,2),
    created_at TIMESTAMP
);
-- orders表中的所有字段都直接依赖于 order_id，不存在传递依赖
```

**传递依赖的识别方法：**

```
传递依赖判定流程：

1. 确定主键 A
2. 找出所有非主键字段 B, C, D...
3. 检查每个非主键字段的依赖关系：

   A -> B (直接依赖) ✓
   A -> C (直接依赖) ✓
   B -> C (非主键依赖非主键) ✗ 传递依赖！
   B -> D (非主键依赖非主键) ✗ 传递依赖！

示例分析：
order(id, customer_id, customer_name, customer_city)
- id 是主键
- customer_name 依赖于 customer_id（A->B）
- customer_city 依赖于 customer_id（A->B）
- customer_name 和 customer_city 之间没有直接关系 ✓

problematic_order(id, customer_id, customer_city, city_weather)
- customer_city 依赖于 customer_id（A->B）
- city_weather 依赖于 customer_city（B->C）
- 存在传递依赖：id -> customer_id -> customer_city -> city_weather
```

#### 1.3.4 BCNF范式—— 主属性约束

**定义**：在满足3NF的基础上，任何表的主属性都不能对主键产生部分依赖或传递依赖。

```sql
-- ❌ 违反BCNF
-- 主键是 (student_id, course_id)
-- 但 teacher_id 决定了 course_name
CREATE TABLE student_courses_bad (
    student_id INTEGER,
    course_id INTEGER,
    course_name VARCHAR(100),    -- 依赖于 course_id
    teacher_id INTEGER,          -- 决定了 course_name
    grade VARCHAR(2),
    PRIMARY KEY (student_id, course_id)
);

-- 问题分析：
-- course_id -> teacher_id (course_id 决定 teacher_id)
-- course_id -> course_name (course_id 决定 course_name)
-- teacher_id 是主属性，但不能完全依赖于主键

-- ✅ 符合BCNF：拆分课程表
CREATE TABLE courses (
    course_id SERIAL PRIMARY KEY,
    course_name VARCHAR(100) NOT NULL,
    teacher_id INTEGER REFERENCES teachers(id)
);

CREATE TABLE enrollments (
    student_id INTEGER REFERENCES students(id),
    course_id INTEGER REFERENCES courses(id),
    grade VARCHAR(2),
    PRIMARY KEY (student_id, course_id)
);

CREATE TABLE teachers (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL
);
```

#### 1.3.5 范式选择建议

```
范式选择决策树：

                    开始设计
                        │
                        ▼
            ┌───────────────────────┐
            │ 数据修改频繁吗？       │
            └───────────────────────┘
                  │           │
                 是          否
                  │           │
                  ▼           ▼
        ┌─────────────┐  ┌─────────────┐
        │ 高范式      │  │ 考虑性能   │
        │ (3NF/BCNF) │  │ 需求       │
        │ 减少更新异常│  └─────────────┘
        └─────────────┘        │
                              ▼
                  ┌───────────────────────┐
                  │ 查询多还是写入多？   │
                  └───────────────────────┘
                        │           │
                   查询多        写入多
                        │           │
                        ▼           ▼
              ┌─────────────┐  ┌─────────────┐
              │ 可适当冗余   │  │ 严格遵循   │
              │ 减少JOIN   │  │ 高范式     │
              └─────────────┘  └─────────────┘

实际项目建议：
- 核心业务表（订单、支付）：遵循3NF/BCNF，保证数据一致性
- 统计报表、历史数据：可适度反范式，提高查询性能
- 日志表、审计表：优先考虑写入性能
- 缓存表：完全反范式，以空间换时间
```

## 2. 关系型数据库设计

### 2.1 ER图设计

#### 2.1.1 ER图基本概念

```
ER图元素说明：

┌─────────────┐
│    用户      │  ← 实体（Entity）：用矩形表示，代表业务中的对象
│    User     │
└─────────────┘
      │
      │ id, name, email, phone
      │ 属性（Attribute）：用椭圆表示
      ▼
┌─────────────┐
│   (PK) id   │  ← 主键属性用下划线标注
└─────────────┘

关系类型：
┌─────────┐         ┌─────────┐         ┌─────────┐
│  用户   │───1:N───│  订单   │───N:1───│  产品   │
└─────────┘         └─────────┘         └─────────┘
   一个用户           一个订单           一个产品
   有多个订单         属于一个用户       可被多个订单购买
```

#### 2.1.3 ER图设计步骤

```
ER图设计流程：

步骤1：识别实体
├── 从业务需求中提取名词
├── 例如：用户、商品、订单、支付、物流
└── 每个实体对应一张表

步骤2：识别属性
├── 每个实体的特征
├── 主键（唯一标识）
├── 普通属性
└── 关系属性（外键）

步骤3：识别关系
├── 动宾短语识别关系
├── 例如：用户"下"订单、订单"包含"商品
├── 确定关系类型（1:1, 1:N, M:N）
└── 确定是否需要关联表

步骤4：绘制ER图
├── 使用工具绘制
├── 标注基数（ Cardinality）
├── 标注参与约束（Total/Partial）
└── 标注属性

步骤5：规范化检查
├── 检查是否满足3NF
├── 检查是否有传递依赖
└── 必要时拆分实体
```

#### 2.1.4 完整电商ER图示例

```
电商系统ER图

┌──────────────────┐         ┌──────────────────┐
│      用户         │         │      角色         │
│   (users)        │         │    (roles)       │
├──────────────────┤         ├──────────────────┤
│ (PK) id          │         │ (PK) id          │

        │                            │
        │ 1:N                       │ M:N
        │                            │
        ▼                            │
┌──────────────────┐                 │
│      订单         │                 │
│   (orders)       │                 │
├──────────────────┤                 │
│ (PK) id          │                 │
│ (UK) order_no    │                 │
│ (FK) user_id     │──────────────►user_roles◄────────┘
│     total_amount │                 │ (关联表)
│     status       │                 │
│     created_at   │                 │
└───────┬──────────┘
        │
        │ 1:N
        ▼
┌──────────────────┐         ┌──────────────────┐
│    订单明细       │         │      产品         │

├──────────────────┤         ├──────────────────┤
│ (PK) id          │    N:1  │ (PK) id          │
│ (FK) order_id    │────────►│ (UK) sku         │
│ (FK) product_id  │────────►│     name         │
│     product_name │(快照)   │     price        │
│     quantity     │         │     stock        │
│     unit_price   │         │ (FK) category_id │
└──────────────────┘         └───────┬──────────┘
                                      │
                                      │ 1:N
                                      ▼
                            ┌──────────────────┐
                            │      分类         │
                            │  (categories)    │
                            ├──────────────────┤
                            │ (PK) id          │
                            │ (UK) slug        │
                            │     name         │
                            │ (FK) parent_id   │──┐
                            │     level        │  │
                            └──────────────────┘  │
                                      ▲          │
                                      │ self-ref│
                                      └──────────┘

关系说明：
- users 1:N orders：一个用户可以有多个订单
- users M:N roles：通过 user_roles 关联表实现多对多
- orders 1:N order_items：一个订单包含多个明细
- products 1:N order_items：一个产品可以在多个订单明细中出现
- products N:1 categories：一个产品属于一个分类
- categories self-ref：自关联，parent_id 指向上级分类
```

#### 2.1.5 ER图代码表示

```sql
-- PostgreSQL 实现上述ER图的DDL

-- 用户表
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    phone VARCHAR(20),
    status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 角色表
CREATE TABLE roles (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) UNIQUE NOT NULL,
    description TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 用户角色关联表
CREATE TABLE user_roles (
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    role_id INTEGER NOT NULL REFERENCES roles(id) ON DELETE CASCADE,
    assigned_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    assigned_by INTEGER REFERENCES users(id),
    PRIMARY KEY (user_id, role_id)
);

-- 分类表（支持无限级分类）
CREATE TABLE categories (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    slug VARCHAR(100) UNIQUE NOT NULL,
    parent_id INTEGER REFERENCES categories(id) ON DELETE SET NULL,
    level INTEGER DEFAULT 1,
    path VARCHAR(255),  -- 路径格式: "1,2,3"
    sort_order INTEGER DEFAULT 0,
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 产品表
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    sku VARCHAR(100) UNIQUE NOT NULL,
    name VARCHAR(200) NOT NULL,
    description TEXT,
    price DECIMAL(10, 2) NOT NULL CHECK (price >= 0),
    stock INTEGER DEFAULT 0 CHECK (stock >= 0),
    category_id INTEGER REFERENCES categories(id) ON DELETE SET NULL,
    status VARCHAR(20) DEFAULT 'draft',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 订单表
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    order_no VARCHAR(50) UNIQUE NOT NULL,
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
    total_amount DECIMAL(10, 2) NOT NULL,
    status VARCHAR(20) DEFAULT 'pending',
    shipping_address JSONB NOT NULL,
    payment_method VARCHAR(20),
    payment_status VARCHAR(20) DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 订单明细表
CREATE TABLE order_items (
```

```sql
-- 2. 复杂查询优化
-- 使用CTE（Common Table Expression）
WITH active_users AS (
    SELECT id, username
    FROM users
    WHERE status = 'active'
),
recent_orders AS (
    SELECT user_id, COUNT(*) as order_count
    FROM orders
    WHERE created_at > NOW() - INTERVAL '30 days'
    GROUP BY user_id
)
SELECT
    au.username,
    COALESCE(ro.order_count, 0) as recent_orders
FROM active_users au
LEFT JOIN recent_orders ro ON au.id = ro.user_id
ORDER BY recent_orders DESC;

    id,
    user_id,
    total_amount,
    created_at,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) as order_rank,

-- 8. 存储过程
CREATE OR REPLACE FUNCTION create_order(
    p_user_id INTEGER,
    p_items JSONB,
    p_shipping_address JSONB
) RETURNS INTEGER AS $$
DECLARE
    v_order_id INTEGER;
    v_total_amount DECIMAL(10, 2) := 0;
    v_item JSONB;
BEGIN
    -- 计算总金额
    FOR v_item IN SELECT * FROM jsonb_array_elements(p_items)
    LOOP
        v_total_amount := v_total_amount + (v_item->>'price')::DECIMAL * (v_item->>'quantity')::INTEGER;
    END LOOP;

    -- 创建订单
    INSERT INTO orders (
        user_id,
        total_amount,
        shipping_address,
        status
    ) VALUES (
        p_user_id,
        v_total_amount,
        p_shipping_address,
        'pending'
    ) RETURNING id INTO v_order_id;

    -- 创建订单明细
    INSERT INTO order_items (
        order_id,
        product_id,
        product_name,
        unit_price,
        quantity,
        subtotal
    )
    SELECT
        v_order_id,
        (item->>'product_id')::INTEGER,
        (item->>'product_name'),
        (item->>'price')::DECIMAL,
        (item->>'quantity')::INTEGER,
        (item->>'price')::DECIMAL * (item->>'quantity')::INTEGER
    FROM jsonb_array_elements(p_items) AS item;

    RETURN v_order_id;
END;
$$ LANGUAGE plpgsql;

-- 使用存储过程

    '{"recipient": "张三", "city": "北京"}'::jsonb
);
```

```sql

-- ❌ 不好的做法
```

```sql
-- 2. 使用LIMIT
-- ✅ 限制结果集

-- ❌ 不好的做法（索引失效）

WHERE LOWER(email) = 'user@example.com';

-- ✅ 好的做法
SELECT * FROM users
WHERE email = 'user@example.com';

-- 4. 使用EXISTS代替IN
-- ❌ 不好的做法
SELECT * FROM orders
WHERE user_id IN (SELECT id FROM users WHERE status = 'active');

-- ✅ 好的做法
SELECT o.*
FROM orders o
WHERE EXISTS (
    SELECT 1 FROM users u
    WHERE u.id = o.user_id AND u.status = 'active'
);

-- 5. 避免OR条件
-- ❌ 不好的做法
SELECT * FROM orders
WHERE user_id = 123 OR status = 'paid';

-- ✅ 好的做法
SELECT * FROM orders WHERE user_id = 123
UNION ALL
SELECT * FROM orders WHERE status = 'paid' AND user_id != 123;

-- ❌ 不好的做法

    (SELECT COUNT(*) FROM order_items WHERE order_id = orders.id) as item_count

FROM orders o
```

## 5. 分布式数据库架构

### 5.1 主从复制

```
主从复制架构（Master-Slave Replication）

      主库（Master）
          │
          ├─写入─→ 应用服务器
          │
          └─复制─┬─→ 从库1（Slave 1）
                 ├─→ 从库2（Slave 2）
                 └─→ 从库3（Slave 3）

特点：
- 主库：处理写操作和读操作
- 从库：只处理读操作，从主库同步数据
- 异步复制：性能好，但可能数据延迟
- 同步复制：数据一致性好，但性能影响大
```

#### 6.3.3 反范式维护策略

```sql
-- 策略1：触发器自动维护
CREATE OR REPLACE FUNCTION update_order_summary()
RETURNS TRIGGER AS $$
BEGIN
    IF TG_OP = 'INSERT' THEN
        UPDATE user_order_summary
        SET
            order_count = order_count + 1,
            total_spent = total_spent + NEW.total_amount,
            avg_order_value = total_spent / order_count,
            last_order_date = NEW.created_at,
            last_order_no = NEW.order_no,
            updated_at = CURRENT_TIMESTAMP
        WHERE user_id = NEW.user_id;
        RETURN NEW;
    ELSIF TG_OP = 'UPDATE' THEN
        -- 处理金额变化
        UPDATE user_order_summary
        SET total_spent = total_spent - OLD.total_amount + NEW.total_amount,
            updated_at = CURRENT_TIMESTAMP
        WHERE user_id = NEW.user_id;
        RETURN NEW;
    ELSIF TG_OP = 'DELETE' THEN
        UPDATE user_order_summary
        SET
            order_count = order_count - 1,
            total_spent = total_spent - OLD.total_amount,
            updated_at = CURRENT_TIMESTAMP
        WHERE user_id = OLD.user_id;
        RETURN OLD;
    END IF;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_order_summary
AFTER INSERT OR UPDATE OR DELETE ON orders
FOR EACH ROW EXECUTE FUNCTION update_order_summary();

-- 策略2：定时任务批量同步
CREATE OR REPLACE FUNCTION sync_user_order_summary()
RETURNS void AS $$
BEGIN
    -- 使用UPSERT批量更新
    INSERT INTO user_order_summary (user_id, user_name, user_email, order_count, total_spent, ...)
    SELECT
        u.id,
        u.name,
        u.email,
        COUNT(o.id),
        COALESCE(SUM(o.total_amount), 0),
        ...
    FROM users u
    LEFT JOIN orders o ON u.id = o.user_id
    GROUP BY u.id, u.name, u.email
    ON CONFLICT (user_id) DO UPDATE SET
        user_name = EXCLUDED.user_name,
        user_email = EXCLUDED.user_email,
        order_count = EXCLUDED.order_count,
        total_spent = EXCLUDED.total_spent,
        avg_order_value = EXCLUDED.total_spent / NULLIF(EXCLUDED.order_count, 0),
        updated_at = CURRENT_TIMESTAMP;
END;
$$ LANGUAGE plpgsql;

-- 每天凌晨2点执行
```

```markdown
- [ ] 字段名使用小写字母和下划线
- [ ] 每个字段都有适当的数据类型
- [ ] NOT NULL字段有默认值或确认为必填
- [ ] 金额字段使用DECIMAL而非FLOAT
- [ ] 大文本字段使用TEXT而非VARCHAR(大数值)

## 索引设计检查
- [ ] 外键字段有索引
- [ ] WHERE条件中频繁使用的字段有索引
- [ ] ORDER BY字段有索引
- [ ] 复合索引遵循最左前缀原则
- [ ] 定期检查并删除未使用的索引

## 关系设计检查
- [ ] 外键关系明确（ON DELETE/UPDATE）
- [ ] 多对多关系使用关联表
- [ ] 没有循环外键引用
- [ ] 一对一关系考虑合并

## 规范化检查
- [ ] 满足第一范式（字段不可再分）
- [ ] 满足第二范式（非主键字段完全依赖主键）
- [ ] 满足第三范式（非主键字段无传递依赖）
- [ ] 或明确记录了反范式设计的原因

## 性能考虑
- [ ] 大表考虑分区
- [ ] 频繁查询考虑冗余字段
```
