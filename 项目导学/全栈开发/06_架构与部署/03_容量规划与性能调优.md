# 容量规划与性能调优

## 前言：容量规划的重要性

想象你要举办一场演唱会。作为场馆经理，你需要回答几个问题：
- 需要准备多少座位？
- 需要多少安检通道才能不让观众排太久的队？
- 后台需要多少工作人员？
- 餐饮服务需要准备多少食材？

如果座位准备太少，很多买了票的粉丝进不去，粉丝会闹翻天；如果座位准备太多，空着也是浪费。容量规划就是在"不够"和"浪费"之间找到平衡点。

软件系统的容量规划也是一样：你需要提前知道系统能扛多少流量、什么时候该扩容、怎么用最少的资源撑起最大的业务。

很多公司等到系统扛不住了才想起来扩容，这时候往往已经晚了——用户已经在排队、投诉纷至沓来。好的容量规划能让你提前几个月就知道系统什么时候会达到瓶颈，给你足够的准备时间。

这一章，我们将从容量评估模型讲起，涵盖性能测试方法、扩容策略，以及如何优化成本。内容会比较硬核，但我尽量用通俗的语言来解释。

### 1.2 容量评估模型

```typescript
/**
 * 容量评估模型
 *
 * 如何计算系统容量
 */

// 基准测试法
const benchmarkModel = {
  principle: `
  先测量单个资源单元的处理能力，
  然后根据资源数量计算总容量
  `,

  example: `
  假设：
  - 一台4核CPU的服务器每秒能处理1000个请求
  - 我们有10台这样的服务器
  - 负载均衡均匀分配流量

  那么系统的总容量是：
  1000 * 10 = 10000 QPS
  `,

  calculation: `
  总容量 = 单机容量 × 服务器数量 × 负载均衡效率

  注意考虑：
  - 负载均衡是否均匀
  - 是否有单点瓶颈
  - 资源是否独立
  `,
};

// 瀑布模型
const waterfallModel = {
  principle: `
  系统容量取决于整个链路中最薄弱的那个环节
  `,

  example: `
  假设系统链路是：
  客户端 -> CDN -> 负载均衡 -> API服务器 -> 数据库

  每个环节的容量：
  - CDN: 100000 QPS
  - 负载均衡: 50000 QPS
  - API服务器: 10000 QPS
  - 数据库: 5000 QPS

  整个系统的容量 = min(100000, 50000, 10000, 5000) = 5000 QPS

  数据库成了瓶颈！
  `,
};

/**
 * 容量计算公式
 */
class CapacityCalculator {
  /**
   * 根据业务量计算所需容量
   *
   * @param {object} params - 业务参数
   * @returns {object} 容量需求
   */
  static calculateFromBusiness(params) {
    const {
      dailyRequests,      // 每日请求量
      peakQPS,            // 峰值QPS（通常是平均的3-5倍）
      avgResponseTime,    // 平均响应时间（毫秒）
      targetAvailability,  // 目标可用性
    } = params;

    // 估算所需服务器数量
    const requestsPerSecond = dailyRequests / (24 * 3600);
    const estimatedPeakQPS = peakQPS || requestsPerSecond * 4;  // 默认峰值是平均的4倍

    // 单机QPS能力（假设）
    const singleServerQPS = 1000;

    // 考虑可用性的容量冗余
    const availabilityFactor = 1 / targetAvailability;  // 99.9%需要1.1倍容量
    const peakWithBuffer = estimatedPeakQPS * availabilityFactor;

    // 计算服务器数量（向上取整）
    const serverCount = Math.ceil(peakWithBuffer / singleServerQPS);

    return {
      estimatedDailyRequests: dailyRequests,
      estimatedPeakQPS: Math.round(estimatedPeakQPS),
      requiredServerCount: serverCount,
      singleServerQPS: singleServerQPS,
      utilizationAtPeak: (estimatedPeakQPS / (serverCount * singleServerQPS) * 100).toFixed(1) + '%',
    };
  }

  /**
   * 根据资源使用率推算容量上限
   *
   * @param {object} currentLoad - 当前负载
   * @param {number} targetUtilization - 目标资源使用率
   * @returns {object} 容量上限分析
   */
  static analyzeCapacityCeiling(currentLoad, targetUtilization = 70) {
    const {
      currentQPS,
      cpuUsage,
      memoryUsage,
      diskUsage,
      networkUsage,
    } = currentLoad;

    // 找到最紧张的资源
    const resources = [
      { name: 'CPU', usage: cpuUsage },
      { name: '内存', usage: memoryUsage },
      { name: '磁盘', usage: diskUsage },
      { name: '网络', usage: networkUsage },
    ];

    const bottleneck = resources.reduce((max, r) =>
      r.usage > max.usage ? r : max
    );

    // 计算各资源的扩容上限
    const capacityAnalysis = resources.map(r => ({
      name: r.name,
      current: r.usage + '%',
      target: targetUtilization + '%',
      headroom: targetUtilization - r.usage,  // 剩余空间
      // 当前使用率到目标使用率，还能承受多少额外负载
      additionalLoadPercent: ((targetUtilization - r.usage) / r.usage * 100).toFixed(1) + '%',
    }));

    return {
      currentQPS,
      bottleneck: bottleneck.name,
      bottleneckUsage: bottleneck.usage + '%',
      capacityAnalysis,
      // 整体扩容上限（受限于瓶颈资源）
      maxScalableQPS: Math.round(currentQPS * (targetUtilization / bottleneck.usage)),
      recommendation: bottleneck.usage > 80
        ? `警告：${bottleneck.name}使用率过高，建议扩容`
        : `${bottleneck.name}是主要瓶颈，可以继续扩容`,
    };
  }
}

// 使用示例
const businessParams = {
  dailyRequests: 10_000_000,    // 每天1000万请求
  peakQPS: null,                 // 使用默认值估算
  avgResponseTime: 100,         // 100ms
  targetAvailability: 0.999,    // 99.9%
};

const capacity = CapacityCalculator.calculateFromBusiness(businessParams);
console.log('容量需求分析:', capacity);

// 当前负载分析
const currentLoad = {
  currentQPS: 5000,
  cpuUsage: 65,
  memoryUsage: 70,
  diskUsage: 40,
  networkUsage: 30,
};

const ceiling = CapacityCalculator.analyzeCapacityCeiling(currentLoad, 70);
console.log('容量上限分析:', ceiling);
```

// Node.js性能优化
const nodeJsOptimization = {
  // 避免阻塞事件循环
  eventLoopBlocking: `
  // 错误示例：在事件循环中做重计算
  function badExample() {
    let result = 0;
    for (let i = 0; i < 1000000000; i++) {
      result += i;  // 这个循环会阻塞事件循环
    }
    return result;
  }

  // 正确示例：使用setImmediate分片
  function goodExample() {
    return new Promise((resolve) => {
      function compute(start, end, acc) {
        if (start >= end) {
          resolve(acc);
          return;
        }

        // 每10000次后让出事件循环
        setImmediate(() => {
          let result = acc;
          for (let i = start; i < Math.min(start + 10000000, end); i++) {

## 附录：常见问题解答

### Q1: 如何判断系统容量是否足够？

A：几个关键指标：
1. 资源使用率（CPU/内存/带宽）是否长期>70%
2. 响应时间是否在SLO范围内
3. 错误率是否在可接受范围
4. 是否能应对预期的峰值流量
如果任何一项不满足，就需要考虑扩容。

### Q2: 垂直扩展和水平扩展该怎么选？

A：一般建议：
- 早期业务，架构未定时，先垂直扩展（简单）
- 单机有性能瓶颈时，考虑水平扩展（更彻底）
- 无状态服务优先水平扩展
- 有状态服务（如数据库）谨慎水平扩展

### Q3: 如何避免过度扩容浪费成本？

A：
1. 监控资源使用率，长期<30%说明配置过高
2. 使用弹性扩容，按需付费
3. 非核心服务使用更低的配置
4. 定期review并清理闲置资源

### Q4: 性能调优的优先级是什么？

A：架构 > 系统 > 应用
1. 先看架构是否合理（缓存、读写分离）
2. 再看系统配置（Nginx、数据库参数）
3. 最后优化代码（SQL、算法）

### Q5: 如何说服老板投入容量规划？

A：用数据说话：
1. 展示一次故障的损失（用户流失、订单损失）
2. 计算投入产出比（扩容费用 vs 故障风险）
3. 对比竞争对手的SLA
4. 展示容量规划带来的成本节省
