# 性能压测与容量规划完全指南

## 前言：从健身理解性能压测

想象一下，你正在训练跑马拉松。你不能一开始就跑42公里，而是：
- **先测试**：用不同速度跑，观察心率、呼吸、肌肉状态
- **找到极限**：知道自己的最大摄氧量、最大心率
- **制定计划**：根据测试结果，制定科学的训练计划
- **持续调整**：随着训练深入，不断调整计划

软件性能测试也是如此：
- **压测**：用负载探测系统极限
- **容量规划**：根据测试结果规划资源
- **调优**：发现瓶颈后进行优化
- **监控**：持续观察系统状态

## 一、性能压测基础

### 1.1 什么是性能压测？

性能压测就是**给系统施加压力，观察它的表现**。就像：
- 汽车碰撞测试：用不同速度撞击，观察车身变形
- 桥梁承重测试：逐步增加重量，观察桥梁形变
- 服务器压测：逐步增加并发，观察系统响应

### 1.2 性能压测的类型

| 类型 | 目的 | 施加的压力 | 持续时间 |
|------|------|-----------|----------|
| **冒烟测试** | 验证系统能正常工作 | 最小正常负载 | 短 |
| **基准测试** | 建立性能基线 | 预期负载 | 中 |
| **负载测试** | 验证系统在预期负载下的表现 | 预期负载的100% | 长 |
| **压力测试** | 找出系统极限 | 超过预期负载 | 中 |
| **浸泡测试** | 验证长时间运行的稳定性 | 正常负载 | 长（数天） |
| **尖峰测试** | 验证突发流量的处理能力 | 快速增加/减少 | 短 |


### 1.3 核心性能指标

| 指标 | 含义 | 重要性 |
|------|------|--------|
| **QPS (Queries Per Second)** | 每秒查询数 | 核心指标 |
| **TPS (Transactions Per Second)** | 每秒事务数 | 核心指标 |
| **RT (Response Time)** | 响应时间 | 核心指标 |
| **并发用户数** | 同时请求的用户数 | 重要 |
| **错误率** | 失败请求的比例 | 关键 |
| **CPU使用率** | CPU忙碌程度 | 重要 |
| **内存使用率** | 内存使用情况 | 重要 |
| **吞吐量** | 单位时间处理的数据量 | 重要 |

```bash
# ========== 核心指标计算公式 ==========
#
# QPS = 并发用户数 × 平均请求率
# TPS = QPS × 平均事务占比
# RT = QPS × 平均响应时间
#
# 示例：
# 100个并发用户
# 每个用户每10秒发一个请求
# 平均响应时间 200ms
#
# QPS = 100 / 10 = 10 QPS
# 或者 = 100 × (1/10) = 10 QPS
#
# 平均响应时间 = 200ms
# 吞吐率 = 10 × 200ms = 2000ms = 2秒
```

## 二、JMeter详解

### 2.1 JMeter核心概念

JMeter是Apache开源的压测工具，功能强大，使用广泛。

| 组件 | 作用 |
|------|------|
| **Thread Group（线程组）** | 模拟用户 |
| **Sampler（采样器）** | 发送请求 |
| **Logic Controller（逻辑控制器）** | 控制请求流程 |
| **Listener（监听器）** | 查看结果 |
| **Config Element（配置元件）** | 公共配置 |
| **Assertion（断言）** | 验证响应 |

### 2.3 JMeter命令行执行

```bash
# ========== 命令行运行JMeter ==========

# 基本命令
jmeter -n -t test-plan.jmx -l results.jtl -e -o output

# 参数说明：
# -n: 非GUI模式
# -t: 指定测试计划文件
# -l: 指定结果文件
# -e: 生成HTML报告
# -o: 报告输出目录

# 完整示例
jmeter -n \
  -t /opt/jmeter/tests/api-test.jmx \
  -l /opt/jmeter/results/results.jtl \
  -e \
  -o /opt/jmeter/results/html \
  -j /opt/jmeter/results/jmeter.log

# 覆盖变量
jmeter -n \
  -t test.jmx \
  -JTHREAD_COUNT=200 \
  -JRAMP_TIME=120 \
  -JDURATION=600 \
  -l results.jtl

# 分布式测试（控制多台机器）
jmeter -n \
  -t test.jmx \
  -R 192.168.1.101,192.168.1.102,192.168.1.103 \
  -l results.jtl
```

### 3.2 K6完整脚本示例

```javascript
// api-load-test.js
// API负载测试脚本

import http from 'k6/http';
import { check, sleep, group } from 'k6';
import { Rate, Trend, Counter } from 'k6/metrics';

// ========== 自定义指标 ==========

// 错误率
const errorRate = new Rate('errors');

// 响应时间趋势
const loginDuration = new Trend('login_duration');
const getUserDuration = new Trend('get_user_duration');
const createOrderDuration = new Trend('create_order_duration');

// 特定错误计数器
const authErrors = new Counter('auth_errors');
const serverErrors = new Counter('server_errors');

// ========== 测试配置 ==========

export const options = {
  // 场景定义
  scenarios: {
    // 场景1：持续负载测试
    constant_load: {
      executor: 'constant-vus',
      vus: 100,              // 100个虚拟用户
      duration: '10m',       // 持续10分钟
    },

    // 场景2：阶梯负载测试（逐步增加）
    ramping_load: {
      executor: 'ramping-vus',
      startVUs: 0,
      stages: [
        { duration: '2m', target: 50 },   // 2分钟内增加到50用户
        { duration: '5m', target: 50 },   // 保持50用户5分钟
        { duration: '2m', target: 100 },  // 2分钟内增加到100用户
        { duration: '5m', target: 100 },  // 保持100用户5分钟
        { duration: '2m', target: 200 },  // 2分钟内增加到200用户
        { duration: '5m', target: 200 },  // 保持200用户5分钟
        { duration: '2m', target: 0 },    // 2分钟内降到0
      ],
    },

    // 场景3：尖峰测试
    spike_test: {
      executor: 'ramping-vus',
      startVUs: 10,
      stages: [
        { duration: '1m', target: 10 },   // 基础10用户
        { duration: '30s', target: 500 },  // 突增到500用户
        { duration: '1m', target: 500 },   // 保持30秒
        { duration: '30s', target: 10 },   // 恢复到10用户
        { duration: '2m', target: 0 },     // 逐步退出
      ],
    },
  },

  // 阈值配置
  thresholds: {
    // HTTP请求相关阈值
    'http_req_duration': ['p(95)<500', 'p(99)<1000'],  // 95%请求<500ms，99%<1s
    'http_req_failed': ['rate<0.01'],                    // 错误率<1%

    // 自定义指标阈值
    'login_duration': ['p(95)<1000'],                   // 登录P95<1s
    'get_user_duration': ['p(95)<300'],                 // 获取用户P95<300ms
    'create_order_duration': ['p(95)<2000'],            // 创建订单P95<2s

    // 全局错误率
    'errors': ['rate<0.05'],                            // 总错误率<5%
  },

  // 系统配置
  ext: {
    loadimpact: {
      // Load Impact云测试配置（可选）
      projectId: 'your-project-id',
      name: 'API Load Test',
    },
  },
};

// ========== 测试数据 ==========

// 从CSV文件读取测试用户
const testUsers = open('./test-data/users.csv');

// 解析CSV（简单实现）
function parseCSV(content) {
  const lines = content.split('\n');
  const headers = lines[0].split(',');
  return lines.slice(1).map(line => {
    const values = line.split(',');
    const obj = {};
    headers.forEach((h, i) => obj[h.trim()] = values[i]?.trim());
    return obj;
  });
}

// ========== 默认函数（每个VUs执行）==========

export default function () {
  // 解析用户数据
  const users = parseCSV(testUsers);

  // 轮流使用用户数据
  const user = users[Math.floor(Math.random() * users.length)];

  // ========== 用户登录 ==========
  group('用户登录', () => {
    const loginRes = http.post(
      'https://api.example.com/api/v1/auth/login',
      JSON.stringify({
        username: user.username,
        password: user.password,
      }),
      {
        headers: {
          'Content-Type': 'application/json',
        },
        tags: { name: 'Login' },
      }
    );

    // 记录响应时间
    loginDuration.add(loginRes.timings.duration);

    // 检查响应
    const loginSuccess = check(loginRes, {
      '登录-状态码是200': (r) => r.status === 200,
      '登录-返回token': (r) => r.json('data.token') !== undefined,
      '登录-响应时间<1s': (r) => r.timings.duration < 1000,
    });

    // 记录错误
    if (!loginSuccess) {
      errorRate.add(1);
      authErrors.add(1);
    } else {
      errorRate.add(0);
    }

    // 提取token
    const token = loginRes.json('data.token');
    const headers = {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${token}`,
    };

    // ========== 获取用户信息 ==========
    const userRes = http.get(
      'https://api.example.com/api/v1/users/me',
      { headers, tags: { name: 'GetUser' } }
    );

    getUserDuration.add(userRes.timings.duration);

    check(userRes, {
      '获取用户-状态码是200': (r) => r.status === 200,
      '获取用户-返回用户信息': (r) => r.json('data.id') !== undefined,
    });

    // ========== 创建订单（需要用户登录）==========
    const orderRes = http.post(
      'https://api.example.com/api/v1/orders',
      JSON.stringify({
        items: [
          { productId: 'P001', quantity: 2 },
          { productId: 'P002', quantity: 1 },
        ],
        totalAmount: 299.99,
      }),
      { headers, tags: { name: 'CreateOrder' } }
    );

    createOrderDuration.add(orderRes.timings.duration);

    const orderSuccess = check(orderRes, {
      '创建订单-状态码是201': (r) => r.status === 201,
      '创建订单-返回订单号': (r) => r.json('data.orderId') !== undefined,
    });

    if (!orderSuccess) {
      if (orderRes.status === 401) {
        authErrors.add(1);
      } else if (orderRes.status >= 500) {
        serverErrors.add(1);
      }
    }

    // 随机等待0.5-2秒，模拟用户思考时间
    sleep(Math.random() * 1.5 + 0.5);
  });

  // 模拟不同用户行为
  if (Math.random() < 0.3) {
    // 30%概率浏览商品
    http.get('https://api.example.com/api/v1/products');
    sleep(1);
  }
}

// ========== 生命周期钩子 ==========

// 测试开始前执行
export function setup() {
  console.log('测试开始，准备数据...');

  // 可以在这里做一些准备工作
  // 比如验证测试环境是否可用
  const healthCheck = http.get('https://api.example.com/health');

// 测试结束后执行
export function teardown(data) {
  console.log('测试结束，清理资源...');
  console.log(`测试总时长: ${(Date.now() - data.startTime) / 1000}秒`);
}

// 每次迭代结束后执行（可选）
export function handleSummary(data) {
  return {
    stdout: textSummary(data, { indent: ' ', enableColors: true }),
    'summary.json': JSON.stringify(data),
  };
}

// 文本格式的摘要
function textSummary(data, options) {
  const { metrics } = data;

  let summary = '\n========== 压测结果摘要 ==========\n\n';

  summary += `总迭代次数: ${metrics.iterations.values.count}\n`;
  summary += `总请求数: ${metrics.http_reqs.values.count}\n`;
  summary += `测试时长: ${(metrics.http_req_duration.values.p99 / 1000).toFixed(2)}s\n\n`;

  summary += '响应时间统计:\n';
  summary += `  - 平均: ${metrics.http_req_duration.values.avg.toFixed(2)}ms\n`;
  summary += `  - 中位数: ${metrics.http_req_duration.values.med.toFixed(2)}ms\n`;
  summary += `  - P95: ${metrics.http_req_duration.values['p(95)'].toFixed(2)}ms\n`;
```

## 四、QPS/TPS计算与容量预估

### 4.1 核心公式

```bash
# ========== 容量规划核心公式 ==========
#
# 1. 并发用户数 -> QPS
#    QPS = 并发用户数 × (1 / 平均请求间隔)
#
#    示例：
#    1000个并发用户
#    每个用户每10秒发一个请求
#    QPS = 1000 × (1/10) = 100 QPS
#
# 2. QPS -> 并发用户数
#    并发用户数 = QPS × 平均响应时间
#
#    示例：
#    QPS = 1000
#    平均响应时间 = 200ms = 0.2s
#    并发用户数 = 1000 × 0.2 = 200
#
# 3. 目标QPS -> 所需服务器数
#    服务器数 = 目标QPS / 单台服务器QPS
#
#    示例：
#    目标QPS = 10000
#    单台服务器能处理 QPS = 500
#    服务器数 = 10000 / 500 = 20 台
#
# 4. 峰值QPS计算
#    峰值QPS = 平均QPS × 峰值倍数
#
#    示例：
#    平均QPS = 1000
#    峰值倍数 = 3 ~ 5
#    峰值QPS = 1000 × 5 = 5000
#
# 5. 容量预留
#    规划QPS = 峰值QPS × 冗余系数
#
#    示例：
#    峰值QPS = 5000
#    冗余系数 = 1.5（留50%余量）
#    规划QPS = 5000 × 1.5 = 7500

# ========== 不同业务场景的QPS估算 ==========
#
# 场景1：普通Web应用
# - 平均QPS ≈ DAU × 0.1 / 86400
# - 峰值QPS ≈ 平均QPS × 5
#
# 示例：
# DAU = 100000
# 平均QPS = 100000 × 0.1 / 86400 ≈ 116 QPS
# 峰值QPS = 116 × 5 = 580 QPS
# 规划QPS = 580 × 1.5 = 870 QPS
#
# 场景2：电商秒杀系统
# - 预热期 QPS ≈ 平时峰值的 2-3 倍
# - 秒杀开始 QPS ≈ 平时峰值的 10-50 倍
# - 远超平时峰值
#
# 场景3：即时通讯
# - 长连接模式，QPS不高
# - 但并发连接数是核心指标
```

### 4.2 容量规划实战

```bash
# ========== 容量规划案例 ==========
#
# 背景：开发一个在线教育平台
#
# 预估：
# - 预计第一年DAU 10万
# - 每个用户平均每天看3个视频
# - 每个视频需要一次请求获取播放地址
# - 峰值是平均的5倍
#
# 计算：

# 1. DAU -> 平均日请求
DAU = 100000
请求/用户/天 = 3
平均日请求 = 100000 × 3 = 300000 次

# 2. 日请求 -> 平均QPS
平均QPS = 300000 / 86400 ≈ 3.5 QPS

# 3. 平均QPS -> 峰值QPS
峰值倍数 = 5
峰值QPS = 3.5 × 5 = 17.5 QPS

# 4. 峰值QPS -> 规划QPS（留50%余量）
冗余系数 = 1.5
规划QPS = 17.5 × 1.5 ≈ 26 QPS

# 结论：
# 单台服务器足够应对（单台能处理500-1000 QPS）

# ========== 扩容后的容量规划 ==========
#
# 假设：
# - 业务快速发展，第二年DAU增长到100万
# - 需要重新规划
#
# 重算：
DAU = 1000000
平均日请求 = 1000000 × 3 = 3000000 次
平均QPS = 3000000 / 86400 ≈ 35 QPS
峰值QPS = 35 × 5 = 175 QPS
规划QPS = 175 × 1.5 = 262 QPS

# 考虑：
# - 视频播放服务器单台QPS ≈ 500
# - 需要服务器数 = 262 / 500 = 0.5 → 至少2台（高可用）
# - 加上负载均衡器、缓存、数据库等
# - 总计需要：
#   - 视频服务器：2-4台
#   - API服务器：2-4台
#   - Redis集群：3台
#   - PostgreSQL主从：2台
```

### 8.3 容量规划速查表

```bash
# ========== 快速容量计算公式 ==========

# 应用服务器
需要的QPS = 预估DAU × 每日请求数 / 86400 × 峰值倍数 × 安全系数
服务器数 = 需要的QPS / 单台QPS

# 数据库服务器
连接数需求 = 应用服务器数 × 每台连接池大小
存储需求 = 数据量 × 增长系数 × 冗余系数

# 缓存服务器
内存需求 = 热数据大小 × 冗余系数
Redis内存 = (key数量 × 平均key大小) × 1.5（开销）

# 带宽估算
带宽 = (峰值QPS × 平均响应大小 × 8) / 压缩率
```
