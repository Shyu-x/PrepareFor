## 一、分布式系统基础

### 1.1 什么是分布式系统

分布式系统（Distributed System）是由多个独立的计算机节点组成的系统，这些节点通过网络通信协调工作，对用户呈现为单一的连贯系统。简单来说，分布式系统就是"多个节点协同完成一个任务"。

**分布式系统的核心特征：**

| 特征 | 说明 | 示例 |
|------|------|------|
| 多节点 | 系统由多个计算机节点组成 | 100台服务器组成的集群 |
| 网络通信 | 节点间通过消息传递通信 | HTTP、RPC、消息队列 |
| 透明性 | 对用户隐藏复杂性，看起来像单一系统 | 访问淘宝像访问单一网站 |
| 协作性 | 节点协同工作完成共同目标 | 分布式数据库查询 |
| 独立性 | 每个节点可以独立运行和故障 | 某台服务器宕机不影响其他 |

**常见的分布式系统类型：**

```java
// 1. 分布式数据库
// MongoDB分片集群、Cassandra集群、HBase分布式存储
// 特点：数据分散存储在多个节点，支持水平扩展

// 2. 分布式计算
// Hadoop MapReduce、Spark、Flink
// 特点：将大任务分解到多个节点并行处理

// 3. 分布式存储
// HDFS、GlusterFS、Ceph
// 特点：文件分散存储，提供高可靠性和高可用性

// 4. 分布式消息队列
// Kafka集群、RabbitMQ集群、RocketMQ
// 特点：消息分散存储，实现异步和解耦

// 5. 分布式微服务
// Spring Cloud微服务、Dubbo微服务
// 特点：服务分散部署，通过RPC通信
```

### 1.2 分布式 vs 单机系统

在深入分布式系统之前，我们需要理解为什么需要分布式，以及分布式与单机系统的本质区别。

**单机系统的局限性：**

```java

    // CPU瓶颈：单处理器核心数有限，无法处理高并发
    // 现代服务器通常最多64-128核心
    // 而高并发场景可能需要处理百万级请求

    // 内存瓶颈：单机器内存有限
    // 大型应用可能需要TB级内存（如缓存、搜索索引）
    // 单机无法满足

    // 磁盘IO瓶颈：磁盘读写速度有限

    // 可用性瓶颈：单机宕机意味着服务不可用
```

### 1.3 CAP理论

CAP理论是分布式系统领域最重要的理论之一，由Eric Brewer在2000年提出，2002年被证明为定理。

**CAP三要素：**

```
C (Consistency) - 一致性
    所有节点在同一时刻看到相同的数据
    写操作完成后，任何后续访问都能看到更新后的值
    强一致性代价高，需要同步确认

A (Availability) - 可用性
    每个请求都能在合理时间内得到响应
    系统始终可读可写，不会拒绝服务
    可用性意味着可能返回过期数据

P (Partition Tolerance) - 分区容错性
    系统在网络分区发生时仍能运行
    网络分区：节点间通信中断，但各自独立运行
    分布式系统必须容忍网络分区
```

**CAP三角关系：**

```
                    CAP
                     │
         ┌───────────┼───────────┐
         │           │           │
         ▼           ▼           ▼
    Consistency    Availab    Partition
         │           │           │
         └───────────┼───────────┘
                     │
              必须选择其中一个

    // CA (单机数据库)：不存在的假命题
    // 分布式系统中，P是必须容忍的
    // 所以实际上是 CP 或 AP 的选择
```

**CP vs AP 场景分析：**

```java
// CP系统示例：Zookeeper、HBase、MongoDB副本集

    /**
     * Zookeeper的CP设计
     *
     * Zookeeper保证一致性：当leader节点故障时
     * 系统会重新选举leader，在选举期间服务不可用
     *
     * 特点：
     * - 强一致性写入
     * - 选举期间短暂不可用
     * - 适合协调服务、配置管理
     */
    public void zookeeperCP() {
        // 写入需要多数节点确认
        // leader故障时选举期间拒绝服务
    }
}

// AP系统示例：Cassandra、DynamoDB、Eureka
public class APSystemExample {

    /**
     * Cassandra的AP设计
     *
     * Cassandra保证可用性：当节点故障时
     * 仍可响应读请求，可能返回过期数据
     *
     * 特点：
     * - 写入始终成功（可配置一致性级别）
     * - 故障节点不影响读写
     * - 适合对可用性要求高的场景
     */
    public void cassandraAP() {
        // 最终一致性
        // 可配置复制因子和一致性级别
        // 冲突解决：最后写入者胜出(LWW)
    }
}
```

## 二、一致性理论

### 2.1 CAP定理的深入理解

CAP定理是分布式系统的理论基础，理解CAP对于架构设计至关重要。

**CAP定理的形式化定义：**

```
CAP定理证明：
在一个存在信息分组不可恢复的异步分布式系统中，
不可能同时实现一致性、可用性和分区容错性。

数学表达：
CAP(C, A, P) → 最多同时满足两个

实际意义：
由于网络分区不可避免（P必须满足），
所以系统只能在C和A之间权衡。
```

**一致性级别详解：**

```java
/**
 * 一致性级别分类
 *
 * 1. 强一致性（Strong Consistency）
 *    - 写入后立即可见
 *    - 所有副本数据完全一致
 *    - 实现代价高，性能差
 *    - 示例：银行转账、库存扣减
 *
 * 2. 弱一致性（Weak Consistency）
 *    - 写入后不保证立即可见
 *    - 系统不承诺何时达到一致
 *    - 实现代价低，性能好
 *    - 示例：DNS更新、缓存失效
 *
 * 3. 最终一致性（Eventual Consistency）
 *    - 写入后保证最终达到一致
 *    - 存在不一致窗口期
 *    - 典型窗口：几毫秒到几秒
 *    - 示例：Cassandra、 DynamoDB
 */
public class ConsistencyLevels {

    // 强一致性：同步复制
    // 写入主节点，同时写入N个从节点
    // 所有写入完成后才算成功

    // 最终一致性：异步复制
    // 写入主节点后立即返回
    // 后台异步复制到从节点

    // 最终一致性的变种：
    // - 因果一致性：保证因果顺序
    // - 读己之所写：自己写入立即可见
    // - 会话一致性：同一会话内保证
    // - 单调读一致性：不会读到更旧数据
}
```

### 2.2 BASE理论

BASE理论是对CAP中一致性和可用性权衡的实践总结，由eBay提出。

**BASE三要素：**

```
BASE = Basically Available + Soft State + Eventually Consistent

基本可用（Basically Available）：
    系统保证可用性，但可能降级
    故障时采用降级策略：返回默认值、简化功能
    示例：促销时关闭退款功能

软状态（Soft State）：
    系统状态可能随时间变化
    不要求强一致，允许中间状态
    示例：订单"处理中"状态

最终一致（Eventually Consistent）：
    系统在未来的某个时刻达到一致
    不需要实时保证一致性
    示例：异步复制完成后的数据一致
```

**BASE vs ACID：**

```java
/**
 * ACID：传统关系型数据库的事务特性
 *
 * Atomicity（原子性）：
 *   事务要么全部成功，要么全部失败
 *
 * Consistency（一致性）：
 *   事务前后数据库状态一致
 *
 * Isolation（隔离性）：
 *   并发事务互不干扰
 *
 * Durability（持久性）：
 *   事务提交后永久保存
 *
 * BASE vs ACID对比：
 *
 * ACID追求强一致，牺牲可用性
 * BASE追求可用性，接受最终一致
 *
 * ACID适合：银行转账、订单处理（强一致场景）
 * BASE适合：社交Feed、评论系统（高可用场景）
 */
public class ACIDvsBASE {

    // ACID模型示例：银行转账
    public void bankTransfer() {
        // 转账必须原子完成
        // 转账前后余额必须一致
        // 不允许并发转账导致数据错误
        // 转账结果必须持久化
    }

    // BASE模型示例：社交网络点赞
    public void socialLike() {
        // 点赞可能短暂失败（可用性降级）
        // 点赞数可能暂时不准确（软状态）
        // 最终会达到一致（最终一致）
        // 用户体验优先于精确计数
    }
}
```

### 2.3 一致性模型详解

**不同一致性模型的选择：**

```java
/**
 * 一致性模型对比
 *
 * 线性一致性（Linearizability）：
 * - 最强的一致性模型
 * - 所有操作像在单一时刻发生
 * - 实现代价极高，性能最差
 * - 示例：Zookeeper、etcd
 *
 * 顺序一致性（Sequential Consistency）：
 * - 保证操作顺序一致
 * - 不要求实时性（后发的操作可能先完成）
 * - 比线性一致性弱，但实现成本较低
 * - 示例：多线程程序的寄存器
 *
 * 因果一致性（Causal Consistency）：
 * - 只保证因果相关的操作顺序
 * - 无因果关系的操作可以乱序
 * - 实现成本适中
 * - 示例：社交网络评论系统
 *
 * 最终一致性（Eventual Consistency）：
 * - 最弱的一致性保证
 * - 只保证在没有新更新的情况下最终一致
 * - 实现成本最低，性能最好
 * - 示例：Cassandra、DynamoDB
 */
public class ConsistencyModels {

    /**
     * 一致性模型选择指南
     *
     * 金融交易场景：线性一致性
     * - 股票交易、资金转账
     * - 必须保证强一致
     *
     * 业务数据场景：顺序一致性
     * - 订单处理、库存管理
     * - 需要保证操作顺序
     *
     * 用户内容场景：因果一致性
     * - 评论系统、社交网络
     * - 允许一定程度的乱序
     *
     * 日志分析场景：最终一致性
     * - 埋点数据、日志收集
     * - 允许短暂不一致
     */
    public void chooseConsistencyModel() {
        // 根据业务需求选择合适的一致性模型
        // 原则：够用就好，不要过度追求一致性
    }
}
```

### 2.4 我的思考：CAP之间的权衡

在实际项目中，我深刻体会到CAP权衡的复杂性。

```java
/**
 * CAP权衡的实践经验
 *
 * 1. 电商系统
 *    - 核心交易链路：CP（订单、支付）
 *    - 商品展示：AP（商品信息、库存快照）
 *    - 用户评价：AP（允许短暂不一致）
 *
 * 2. 社交系统
 *    - 消息系统：AP优先（用户体验）
 *    - 好友关系：CP（关系一致性）
 *    - 内容推荐：AP（实时性更重要）
 *
 * 3. 金融系统
 *    - 银行转账：CP（必须强一致）
 *    - 账户查询：可以弱一致（允许延迟）
 *    - 交易记录：CP（日志必须一致）
 */
public class CAPTradeoff {

    /**
     * 我的教训
     *
     * 教训1：过早追求强一致
     * - 以为强一致更安全
     * - 结果：系统可用性极差
```

### 3.2 2PC：两阶段提交

2PC（Two Phase Commit）是最基础的分布式事务协议。

**2PC执行流程：**

```
阶段1：准备阶段（Prepare Phase）
    协调者向所有参与者发送 Prepare 请求
    参与者锁定资源，执行事务，但不提交
    参与者投票：如果可以提交返回 Yes，否则返回 No

阶段2：提交阶段（Commit Phase）
    协调者收到所有参与者的 Yes 投票
    向所有参与者发送 Commit 请求
    参与者提交事务，释放锁
    协调者收到所有参与者的 Ack

失败处理：
    如果任何参与者返回 No 或超时
    协调者向所有参与者发送 Rollback
    参与者回滚事务，释放锁
```

**2PC的Java实现：**

```java
/**
 * 2PC 两阶段提交实现
 *
 * 角色：
 * - Coordinator（协调者）：事务管理器
 * - Participant（参与者）：资源管理器
 */
public class TwoPhaseCommit {

    // 协调者实现
    public static class Coordinator {
        private List<Participant> participants = new ArrayList<>();

        /**
         * 提交事务
         *
         * 阶段1：发送 Prepare 请求
         * 阶段2：根据投票结果决定 Commit 或 Rollback
         */
        public boolean commit() {
            // 阶段1：准备阶段
            // 向所有参与者发送 Prepare 请求
            for (Participant participant : participants) {
                boolean prepared = participant.prepare();
                if (!prepared) {
                    // 有人投票反对，回滚
                    rollback();
                    return false;
                }
            }

            // 阶段2：提交阶段
            // 所有参与者都准备好了，发送 Commit
            for (Participant participant : participants) {
                participant.commit();
            }

            return true;
        }

        private void rollback() {
            // 向所有参与者发送 Rollback
            for (Participant participant : participants) {
                participant.rollback();
            }
        }
    }

    // 参与者实现
    public static class Participant {
        private boolean committed = false;

        /**
         * 准备阶段
         *
         * 锁定资源，执行事务，但不提交
         * 返回 true 表示可以提交
         */
        public boolean prepare() {
            try {
                // 锁定相关资源
                // 执行SQL操作（不提交）
                // 记录Undo日志
                // 预提交检查

                // 检查是否满足提交条件
                // 例如：库存是否充足、余额是否足够

                return true; // 投票Yes
            } catch (Exception e) {
                return false; // 投票No
            }
        }

        /**
         * 提交阶段
         *
         * 真正提交事务
         */
        public void commit() {
            // 提交本地事务
            // 释放锁
            committed = true;
        }

        /**
         * 回滚阶段
         *
         * 回滚本地事务
         */
        public void rollback() {
            // 根据Undo日志回滚
            // 释放锁
            committed = false;
        }
    }
}

/**
 * 2PC 的问题
 *
 * 1. 同步阻塞
 *    - 整个过程是同步的
 *    - 参与者锁定资源时间长
 *    - 其他事务需要等待
 *
 * 2. 单点故障
 *    - 协调者故障会导致事务卡住
 *    - 参与者一直等待指令
 *
 * 3. 数据不一致
 *    - 阶段1完成后协调者故障
 *    - 部分参与者已提交，部分未提交
 *    - 数据不一致
 *
 * 4. 无法处理故障恢复
 *    - 参与者故障恢复后无法确定事务状态
 *    - 需要人工干预
 */
```

### 3.3 3PC：三阶段提交

3PC（Three Phase Commit）是2PC的改进版本，试图解决2PC的同步阻塞问题。

**3PC执行流程：**

```
阶段1：CanCommit（能否提交）
    协调者询问所有参与者是否可以提交
    参与者检查自身状态，返回 Yes 或 No
    如果全部返回 Yes，进入下一阶段

阶段2：PreCommit（预提交）
    协调者向所有参与者发送 PreCommit 请求
    参与者锁定资源，执行预提交
    参与者返回 Ack

阶段3：DoCommit（正式提交）
    协调者收到所有 Ack 后发送 DoCommit
    参与者正式提交事务
    协调者收到所有 Ack 完成
```

**3PC的改进点：**

```java
/**
 * 3PC 对 2PC 的改进
 *
 * 改进1：添加 CanCommit 阶段
 *    - 在真正锁定资源前检查可行性
 *    - 提前发现可能失败的情况
 *
 * 改进2：打破同步阻塞
 *    - PreCommit 阶段参与者会超时自动提交
 *    - 不会一直等待协调者指令
 *
 * 改进3：更好的故障处理
 *    - 参与者可以判断是否应该提交
 *    - 减少未知状态的事务
 */
public class ThreePhaseCommit {

    public static class Coordinator {

        public boolean commit() {
            // 阶段1：CanCommit
            // 询问所有参与者是否可以提交
            // 不锁定资源，只做检查

            // 阶段2：PreCommit
            // 所有参与者返回 Yes
            // 发送 PreCommit，锁定资源

            // 阶段3：DoCommit
            // 收到所有 Ack 后发送 DoCommit
            // 参与者正式提交
            return true;
        }
    }
}

/**
 * 3PC 的问题
 *
 * 1. 仍然可能数据不一致
 *    - 阶段2到阶段3之间协调者故障
 *    - 部分参与者会超时自动提交
 *    - 但其他参与者可能未提交
 *
 * 2. 网络分区问题
 *    - 网络分区时可能部分提交
 *    - 分区两侧行为不一致
 *
 * 3. 实现复杂度高
 *    - 需要处理多种超时状态
 *    - 逻辑复杂，难以正确实现
 *
 * 总结：3PC 在理论上优于 2PC
 * 但实际应用中，2PC 仍然更常用
 * 因为 2PC 的问题可以通过其他方式解决（如重试补偿）
 */
```

### 3.4 TCC：补偿事务

TCC（Try-Confirm-Cancel）是一种补偿式事务模式，与2PC/3PC不同，TCC是业务层面的实现。

**TCC三阶段设计：**

```
Try（预留阶段）：
    尝试预留资源
    检查业务可行性
    锁定必要资源
    例如：冻结库存、预扣余额

Confirm（确认阶段）：
    确认执行事务
    使用 Try 阶段预留的资源
    必须成功，不允许失败

Cancel（取消阶段）：
    取消执行业务
    释放 Try 阶段预留的资源
    处理 Try 阶段失败的情况
```

**TCC的Java实现：**

```java
/**
 * TCC 事务参与者实现
 *
 * TCC 是一种业务层面的分布式事务实现
 * 需要业务代码实现 Try、Confirm、Cancel 三个方法
 */
public class TCCTransaction {

    // TCC 服务接口
    public interface TCCService<T> {
        /**
         * Try 阶段
         *
         * 资源预留：
         * - 检查业务可行性
         * - 预留资源（如冻结库存）
         * - 返回事务ID
         */
        T tryCommit();

        /**
         * Confirm 阶段
         *
         * 确认执行：
         * - 使用预留的资源
         * - 完成业务操作
         * - 必须成功
         */
        void confirm(T transactionId);

        /**
         * Cancel 阶段
         *
         * 取消执行：
         * - 释放预留的资源
         * - 回滚业务操作
         * - 处理失败情况
         */
        void cancel(T transactionId);
    }

    // 库存服务 TCC 实现
    public static class InventoryTCCService implements TCCService<Long> {

        @Override
        public Long tryCommit() {
            // Try 阶段：冻结库存
            // 检查库存是否充足
            // 冻结（预留）要扣减的数量
            // 返回冻结记录ID
            Long freezeId = inventoryDAO.freeze(10); // 冻结10件
            return freezeId;
        }

        @Override
        public void confirm(Long freezeId) {
            // Confirm 阶段：确认扣减
            // 删除冻结记录
            // 减少实际库存
            inventoryDAO.confirmFreeze(freezeId);
        }

        @Override
        public void cancel(Long freezeId) {
            // Cancel 阶段：取消冻结
            // 恢复冻结的库存
            inventoryDAO.cancelFreeze(freezeId);
        }
    }

    // 订单服务 TCC 实现
    public static class OrderTCCService implements TCCService<Long> {

        @Override
        public Long tryCommit() {
            // Try 阶段：创建预订单
            // 创建订单，状态为"待确认"
            // 返回订单ID
            Long orderId = orderDAO.createPendingOrder();
            return orderId;
        }

        @Override
        public void confirm(Long orderId) {
            // Confirm 阶段：确认订单
            // 更新订单状态为"已确认"
            // 触发后续流程（支付、发货）
            orderDAO.confirmOrder(orderId);
        }

        @Override
        public void cancel(Long orderId) {
            // Cancel 阶段：取消订单
            // 删除或更新订单状态为"已取消"
            orderDAO.cancelOrder(orderId);
        }
    }

    // TCC 协调者
    public static class TCCCoordinator {
        private List<TCCService> services = new ArrayList<>();

        public boolean commit() {
            List<Long> transactionIds = new ArrayList<>();

            try {
                // Try 阶段：调用所有服务的 tryCommit
                for (TCCService service : services) {
                    Long txId = (Long) service.tryCommit();
                    transactionIds.add(txId);
                }

                // Try 全部成功，执行 Confirm
                for (int i = 0; i < services.size(); i++) {
                    services.get(i).confirm(transactionIds.get(i));
                }

                return true;
            } catch (Exception e) {
                // Try 失败，执行 Cancel
                for (int i = 0; i < services.size(); i++) {
                    if (transactionIds.get(i) != null) {
                        services.get(i).cancel(transactionIds.get(i));
                    }
                }
                return false;
            }
        }
    }
}

/**
 * TCC 的优势
 *
 * 1. 性能高
 *    - 执行阶段是并行的
 *    - 不需要锁定资源
 *
 * 2. 灵活
 *    - 业务自己定义补偿逻辑
 *    - 可以处理复杂的业务场景
 *
 * 3. 隔离性好
 *    - Try 阶段就完成了资源预留
 *    - 业务隔离度较高
 *
 * TCC 的问题
 *
 * 1. 实现复杂
 *    - 每个业务都需要实现 Try/Confirm/Cancel
 *    - 代码侵入性高
 *
 * 2. 状态管理
 *    - 需要记录事务状态
 *    - 协调者故障需要恢复机制
 *
 * 3. 空回滚
 *    - Try 未执行就收到 Cancel
 *    - 需要防重机制
 *
 * 4. 悬挂
 *    - Cancel 先于 Confirm 执行
 *    - 需要防悬挂机制
 */
```

### 3.5 Saga模式

Saga模式是处理长事务的一种方式，将大事务拆分为多个小事务，每个小事务都有对应的补偿事务。

**Saga的执行方式：**

```
正向补偿（Saga Forward）：
    T1 → T2 → T3 → T4 → ... → Tn
    每个事务执行后如果失败，逆向执行补偿
    T1失败：直接结束
    T2失败：补偿T1
    T3失败：补偿T1、T2
    ...

补偿模式（Saga Compensation）：
    T1 → T2 → T3 → T4 → ... → Tn
    每个事务对应一个补偿事务 C1、C2、C3...
    失败时逆向执行补偿：
    T3失败：执行C2、C1
    T4失败：执行C3、C2、C1
```

**Saga的Java实现：**

```java
/**
 * Saga 模式实现
 *
 * Saga 将长事务拆分为多个子事务
 * 每个子事务都有对应的补偿事务
 */
public class SagaPattern {

    // Saga 步骤定义
    public static class SagaStep {
        private String name;
        private Runnable execute;
        private Runnable compensate;

        public SagaStep(String name, Runnable execute, Runnable compensate) {
            this.name = name;
            this.execute = execute;
            this.compensate = compensate;
        }

        public void execute() {
            execute.run();
        }

        public void compensate() {
            compensate.run();
        }
    }

    // Saga 执行器
    public static class Saga {
        private List<SagaStep> steps = new ArrayList<>();
        private List<SagaStep> executedSteps = new ArrayList<>();

        public Saga addStep(SagaStep step) {
            steps.add(step);
            return this;
        }

        public boolean execute() {
            for (SagaStep step : steps) {
                try {
                    step.execute();
                    executedSteps.add(step);
                } catch (Exception e) {
                    // 失败，执行补偿
                    compensate();
                    return false;
                }
            }
            return true;
        }

        private void compensate() {
            // 逆向执行补偿
            Collections.reverse(executedSteps);
            for (SagaStep step : executedSteps) {
                try {
                    step.compensate();
                } catch (Exception e) {
                    // 补偿失败，记录日志，后续处理
                    log.error("Compensate failed: {}", step.name, e);
                }
            }
        }
    }

    // 订单处理 Saga 示例
    public static class OrderSaga {

        public void createOrder() {
            Saga saga = new Saga();

            // 步骤1：创建订单
            saga.addStep(new SagaStep(
                "createOrder",
                () -> orderService.create(),
                () -> orderService.cancel()
            ));

            // 步骤2：扣减库存
            saga.addStep(new SagaStep(
                "deductInventory",
                () -> inventoryService.deduct(),
                () -> inventoryService.restore()
            ));

            // 步骤3：扣减余额
            saga.addStep(new SagaStep(
                "deductBalance",
                () -> accountService.deduct(),
                () -> accountService.refund()
            ));

            // 步骤4：增加积分
            saga.addStep(new SagaStep(
                "addPoints",
                () -> pointsService.add(),
                () -> pointsService.deduct()
            ));

            boolean success = saga.execute();
            if (!success) {
                throw new BusinessException("订单创建失败");
            }
        }
    }
}

/**
 * Saga vs TCC 对比
 *
 * Saga：
 * - 没有 Try 阶段，直接执行
 * - 失败时执行补偿事务
 * - 适合长事务场景
 * - 补偿是逆向执行
 *
 * TCC：
 * - 有 Try 阶段预留资源
 * - Confirm/Cancel 是预定义的
 * - 适合需要资源预留的场景
 * - 补偿是释放预留资源
 *
 * 选择建议：
 * - 库存扣减、余额扣减：TCC（需要预留）
 * - 跨系统调用、长流程：Saga（简单补偿）
 */
```

### 3.6 实战：订单系统分布式事务实现

```java
/**
 * 订单系统分布式事务实战
 *
 * 业务场景：用户下单
 * - 扣减商品库存
 * - 创建订单记录
 * - 扣减用户余额
 * - 增加用户积分
 *
 * 技术选型：TCC + Saga 混合模式
 */
public class OrderDistributedTransaction {

    // 订单服务
    public static class OrderService {

        // 订单仓储
        private OrderRepository orderRepository;

        // 库存服务
        private InventoryService inventoryService;

        // 账户服务
        private AccountService accountService;

        // 积分服务
        private PointsService pointsService;

        /**
         * 创建订单 - 分布式事务处理
         */
        public void createOrder(Long userId, Long productId, Integer quantity) {
            // 1. TCC 预留库存
            Long freezeId = inventoryService.tryFreeze(productId, quantity);

            try {
                // 2. Saga 编排订单流程
                Saga saga = new Saga();

                // 步骤1：创建订单（本地事务）
                saga.addStep(new SagaStep(
                    "createOrder",
                    () -> {
                        Order order = new Order();
                        order.setUserId(userId);
                        order.setProductId(productId);
                        order.setQuantity(quantity);
                        order.setStatus(OrderStatus.PENDING);
                        orderRepository.save(order);
                    },
                    () -> {
                        // 补偿：删除订单
                        orderRepository.deleteByUserIdAndProductId(userId, productId);
                    }
                ));

                // 步骤2：扣减余额（TCC Confirm）
                saga.addStep(new SagaStep(
                    "deductBalance",
                    () -> accountService.confirmDeduct(userId, calculateAmount()),
                    () -> {
                        // 补偿：退还余额
                        accountService.refund(userId, calculateAmount());
                    }
                ));

                // 步骤3：增加积分（Saga）
                saga.addStep(new SagaStep(
                    "addPoints",
                    () -> pointsService.addPoints(userId, calculatePoints()),
                    () -> {
                        // 补偿：扣除积分
                        pointsService.deductPoints(userId, calculatePoints());
                    }
                ));

                // 执行 Saga
                boolean success = saga.execute();

                if (success) {
                    // 4. TCC Confirm 库存冻结
                    inventoryService.confirmFreeze(freezeId);

                    // 5. 更新订单状态
                    orderRepository.updateStatus(OrderStatus.CONFIRMED);
                } else {
                    // Saga 失败，补偿
                    inventoryService.cancelFreeze(freezeId);
                    throw new BusinessException("订单创建失败");
                }

            } catch (Exception e) {
                // 异常情况：Cancel 库存冻结
                inventoryService.cancelFreeze(freezeId);
                throw e;
            }
        }

        private BigDecimal calculateAmount() {
            // 计算订单金额
            return BigDecimal.valueOf(100);
        }

        private Integer calculatePoints() {
            // 计算积分
            return 100;
        }
    }

    // 库存服务 - TCC 实现
    public static class InventoryService {

        public Long tryFreeze(Long productId, Integer quantity) {
            // Try 阶段：检查库存并冻结
            Product product = productRepository.findById(productId);
            if (product.getStock() < quantity) {
                throw new BusinessException("库存不足");
            }

            // 冻结库存
            FreezeRecord freeze = new FreezeRecord();
            freeze.setProductId(productId);
            freeze.setQuantity(quantity);
            freeze.setStatus(FreezeStatus.FROZEN);
            freezeRepository.save(freeze);

            // 减少可用库存
            product.setAvailableStock(product.getAvailableStock() - quantity);
            productRepository.save(product);

            return freeze.getId();
        }

        public void confirmFreeze(Long freezeId) {

            FreezeRecord freeze = freezeRepository.findById(freezeId);
            freeze.setStatus(FreezeStatus.CONFIRMED);
```

### 4.2 Redis分布式锁

Redis是最常用的分布式锁实现方案，性能高，支持丰富的数据结构。

**Redis分布式锁实现原理：**

```java
/**
 * Redis 分布式锁实现
 *
 * 核心命令：SETNX + EXPIRE
 *
 * 获取锁：
 * SET lock_key unique_value NX EX 30
 *
 * - NX：只有key不存在时才设置
 * - EX 30：30秒过期时间
 *
 * 释放锁：
 * - 简单方案：DEL lock_key
 * - 安全方案：Lua脚本校验 + DEL
 */
public class RedisDistributedLock {

    private Jedis jedis;

    /**
     * 简单的加锁实现
     *
     * 问题：
     * - SETNX 和 EXPIRE 不是原子操作
     * - 如果 SETNX 成功后 EXPIRE 执行前进程崩溃
     * - 锁将永远不会过期（死锁）
     */
    public boolean lock_simple(String key, String value, int expireSeconds) {
        // 获取锁
        Long result = jedis.setnx(key, value);
        if (result == 1) {
            // 设置过期时间
            jedis.expire(key, expireSeconds);
            return true;
        }
        return false;
    }

    /**
     * 改进的加锁实现（原子操作）
     *
     * 使用 SET key value NX EX seconds
     * 将设置和过期时间合并为一个原子操作
     */
    public boolean lock_atomic(String key, String value, int expireSeconds) {
        // SET key value NX EX seconds 是原子操作
        String result = jedis.set(key, value, "NX", "EX", expireSeconds);
        return "OK".equals(result);
    }

    /**
     * 释放锁（安全释放）
     *
     * 不能直接 DEL，因为：
     * - 可能释放别人的锁（超时导致别人获取了锁）
     *
     * 正确做法：
     * - 校验 value 是否一致
     * - 只有持有者才能释放锁
     */
    public boolean unlock(String key, String value) {
        // Lua 脚本：只有 value 匹配才删除
        String script =
            "if redis.call('get', KEYS[1]) == ARGV[1] then " +
            "    return redis.call('del', KEYS[1]) " +
            "else " +
            "    return 0 " +
            "end";

        Object result = jedis.eval(script, 1, key, value);
        return Long.valueOf(1).equals(result);
    }

    /**
     * 完整的分布式锁实现
     */
    public class RedisLock {
        private static final String LOCK_PREFIX = "lock:";

        private String lockKey;
        private String lockValue;
        private int expireSeconds;

        public RedisLock(String lockKey, int expireSeconds) {
            this.lockKey = LOCK_PREFIX + lockKey;
            this.lockValue = UUID.randomUUID().toString();
            this.expireSeconds = expireSeconds;
        }

        /**
         * 尝试获取锁
         *
         * @param retryTimes 重试次数
         * @param retryInterval 重试间隔（毫秒）
         */
        public boolean tryLock(int retryTimes, long retryInterval) {
            for (int i = 0; i < retryTimes; i++) {
                if (lock_atomic(lockKey, lockValue, expireSeconds)) {
                    return true;
                }

                // 重试前等待
                if (retryInterval > 0) {
                    try {
                        Thread.sleep(retryInterval);
                    } catch (InterruptedException e) {
                        Thread.currentThread().interrupt();
                        return false;
                    }
                }
            }
            return false;
        }

        /**
         * 释放锁
         */
        public boolean unlock() {
            return unlock(lockKey, lockValue);
        }
    }
}
```

### 4.3 Redlock：红锁算法

Redlock是Redis作者提出的分布式锁算法，用于提高分布式锁的可靠性。

**Redlock算法原理：**

```java
/**
 * Redlock 红锁算法
 *
 * 核心思想：
 * - 使用多个独立的Redis节点
 * - 获取锁时在多数节点上成功才算成功
 * - 即使部分节点故障，仍能保证锁的正确性
 *
 * 算法步骤：
 * 1. 获取当前时间戳（毫秒）
 * 2. 依次向N个Redis节点获取锁
 * 3. 计算获取锁的耗时
 * 4. 如果在多数节点上获取成功，且耗时小于锁的过期时间
 *    则认为锁获取成功
 * 5. 否则，需要在所有节点上释放锁
 *
 * 参数建议：
 * - N = 5（5个Redis节点）
 * - 锁过期时间 = 10秒
 * - 获取节点超时 = 5秒
 */
public class RedlockAlgorithm {

    private List<Jedis> jedisPool;
    private int N = 5;
    private int quorum = 3; // 多数节点

    /**
     * Redlock 获取锁
     */
    public String lock(String resourceName, long ttlMillis) {
        String lockValue = UUID.randomUUID().toString();

        List<Long> startTimes = new ArrayList<>();
        int successCount = 0;

        // 计算获取锁的截止时间
        long deadline = System.currentTimeMillis() + ttlMillis;

        for (Jedis jedis : jedisPool) {
            try {
                long start = System.currentTimeMillis();

                // 尝试获取锁
                String result = jedis.set(resourceName, lockValue, "NX", "PX", ttlMillis);

                if ("OK".equals(result)) {
                    successCount++;
                    startTimes.add(start);
                }
            } catch (Exception e) {
                // 节点故障，继续下一节点
            }
        }

        // 检查是否在多数节点上获取成功
        if (successCount >= quorum) {
            // 计算锁的存活时间
            // = 原始TTL - 获取锁耗时
            long elapsedTime = System.currentTimeMillis() -
                startTimes.stream().mapToLong(Long::longValue).max().getAsLong();
            long validTime = ttlMillis - elapsedTime;

            if (validTime > 0) {
                return lockValue;
            }
        }

        // 失败：释放所有已获取的锁
        unlock(lockValue, resourceName);
        return null;
    }

    /**
     * Redlock 释放锁
     */
    public void unlock(String lockValue, String resourceName) {
        for (Jedis jedis : jedisPool) {
            try {
                // Lua 脚本释放锁
                String script =
                    "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                    "    return redis.call('del', KEYS[1]) " +
                    "else " +
                    "    return 0 " +
                    "end";
                jedis.eval(script, 1, resourceName, lockValue);
            } catch (Exception e) {
                // 忽略单节点失败
            }
        }
    }
}

/**
 * Redlock 的争议
 *
 * 支持观点：
 * - 提高可靠性：单节点故障不影响
 * - 简单有效：不需要复杂协调
 *
 * 反对观点：
 * - 假设所有节点独立：实际上可能有关联故障
 * - 性能降低：需要访问多个节点
 * - 不适合CAP：仍然基于Redis，可能不满足CAP
 *
 * 我的观点：
 * - Redlock 适合对可靠性要求高的场景
 * - 需要评估故障模型是否满足需求
 * - 可以结合业务场景做简化
 */
```

### 4.4 Zookeeper分布式锁

Zookeeper是另一种分布式锁实现方案，基于ZAB协议，提供强一致性保证。

**Zookeeper分布式锁实现原理：**

```java
/**
 * Zookeeper 分布式锁实现
 *
 * 实现原理：临时顺序节点 + Watcher
 *
 * 获取锁流程：
 * 1. 创建临时顺序节点
 * 2. 获取所有临时顺序节点
 * 3. 判断是否是序号最小的节点
 *    - 是：获取锁成功
 *    - 否：监听前一个节点
 * 4. 前一个节点消失时，收到通知，重复步骤3
 *
 * 释放锁流程：
 * - 删除临时节点即可
 * - 连接断开时自动删除
 */
public class ZookeeperDistributedLock {

    private ZooKeeper zooKeeper;
    private String lockPath;

    /**
     * Zookeeper 锁节点
     */
    public class ZkLock implements AutoCloseable {
        private String nodePath;

        /**
         * 获取锁
         */
        public boolean acquire(int timeout) throws Exception {
            // 1. 创建临时顺序节点
            nodePath = zooKeeper.create(
                lockPath + "/lock_",
                new byte[0],
                ZooDefs.Ids.OPEN_ACL_UNSAFE,
                CreateMode.EPHEMERAL_SEQUENTIAL
            );

            // 2. 获取所有子节点
            List<String> children = zooKeeper.getChildren(lockPath, false);

            // 3. 按序号排序
            Collections.sort(children);

            // 4. 检查是否是序号最小的
            String nodeName = nodePath.substring(nodePath.lastIndexOf('/') + 1);
            if (children.get(0).equals(nodeName)) {
                return true; // 获取锁成功
            }

            // 5. 不是最小节点，监听前一个节点
            int myIndex = children.indexOf(nodeName);
            String prevNode = children.get(myIndex - 1);

            // 使用 CountDownLatch 等待
            CountDownLatch latch = new CountDownLatch(1);

            // 注册 Watcher
            Stat stat = zooKeeper.exists(lockPath + "/" + prevNode, event -> {
                if (event.getType() == Event.EventType.NodeDeleted) {
                    latch.countDown();
                }
            });

            // 检查节点是否还存在
            if (stat != null) {
                latch.await(timeout, TimeUnit.MILLISECONDS);
            }

            // 重新检查
            children = zooKeeper.getChildren(lockPath, false);
            Collections.sort(children);
            nodeName = nodePath.substring(nodePath.lastIndexOf('/') + 1);

            return children.get(0).equals(nodeName);
        }

        /**
         * 释放锁
         */
        @Override
        public void close() throws Exception {
            if (nodePath != null) {
                zooKeeper.delete(nodePath, -1);
            }
        }
    }
}

/**
 * Redis vs Zookeeper 分布式锁对比

 * Redis 锁：

 * - 单节点有单点故障风险（用Redlock解决）

 *
 * Zookeeper 锁：
 * - 可靠性高
 * - 实现简单（基于临时节点和Watcher）
 * - 性能相对较低（每次需要查询）
 * - CAP中选择CP
 *
 * 选型建议：
 * - 性能优先：Redis（Redlock）
```

### 4.6 实战：实现可靠的分布式锁

```java
/**
 * 可靠的分布式锁实战
 *
 * 使用 Redisson 实现生产级分布式锁
 */
public class ReliableDistributedLock {

    // Redisson 客户端
    private RedissonClient redissonClient;

    /**
     * 库存扣减分布式锁
     */
    public boolean deductStock(Long productId, Integer quantity) {
        String lockKey = "stock:lock:" + productId;
        RLock lock = redissonClient.getLock(lockKey);

        try {
            // 尝试获取锁，等待10秒，锁定30秒
            // 看门狗会自动续期
            boolean acquired = lock.tryLock(10, 30, TimeUnit.SECONDS);

            if (!acquired) {
                return false; // 获取锁失败
            }

            // 执行业务逻辑
            Product product = productRepository.findById(productId);
            if (product.getStock() < quantity) {
                return false; // 库存不足
            }

            product.setStock(product.getStock() - quantity);
            productRepository.save(product);

            return true;

        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return false;
        } finally {
            // 释放锁
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }

    /**
     * 分布式锁最佳实践
     *
     * 1. 锁粒度要细
     *    - 按资源ID锁定，而非按资源类型
     *    - 减少锁竞争
```

### 5.4 熔断降级

熔断降级是保护系统稳定性的重要机制。

**熔断器原理：**

```java
/**
 * 熔断器模式（Circuit Breaker）
 *
 * 状态机：
 *
 *        ┌────────────────────────────────────┐
 *        │                                    │
 *        ▼                                    │
 * ┌──────────┐    失败率超阈值    ┌───────────┐    熔断时间窗口到期   ┌──────────┐
 * │  CLOSED  │ ───────────────►  │   OPEN    │ ─────────────────►  │  HALF_   │
 * │  关闭    │                    │   打开    │                    │  OPEN    │
 * └──────────┘                    └───────────┘                    │  半开    │
 *      ▲                              │                              └──────────┘
 *      │                              │                                    │
 *      │     成功                      │ 失败                                │ 成功
 *      └───────────────────────────────┴────────────────────────────────────┘
 *                                                失败
 */

/**
 * 熔断器实现
 */
public class CircuitBreaker {

    // 熔断器状态
    public enum State {
        CLOSED,    // 关闭：正常调用
        OPEN,      // 打开：直接拒绝调用
        HALF_OPEN  // 半开：允许探测调用
    }

    // 配置
    private int failureThreshold = 5;      // 失败次数阈值
    private int successThreshold = 3;      // 成功次数阈值（半开转关闭）
    private long sleepWindow = 60000;      // 熔断窗口（毫秒）
    private double failureRate = 0.5;       // 失败率阈值（50%）

    // 状态
    private State currentState = State.CLOSED;
    private int failureCount = 0;
    private int successCount = 0;
    private long lastFailureTime = 0;

    /**
     * 调用服务
     */
    public Result call(Service service) {
        if (currentState == State.OPEN) {
            // 熔断打开，直接拒绝
            if (System.currentTimeMillis() - lastFailureTime > sleepWindow) {
                // 进入半开状态
                currentState = State.HALF_OPEN;
                successCount = 0;
            } else {
                return Result.fallback(); // 返回降级结果
            }
        }

        try {
            Result result = service.execute();

            // 调用成功
            onSuccess();
            return result;

        } catch (Exception e) {
            // 调用失败
            onFailure();
            return Result.fallback();
        }
    }

    /**
     * 调用成功处理
     */
    private void onSuccess() {
        if (currentState == State.HALF_OPEN) {
            successCount++;
            if (successCount >= successThreshold) {
                // 半开转关闭
                currentState = State.CLOSED;
                failureCount = 0;
            }
        } else {
            // 关闭状态下重置失败计数
            failureCount = 0;
        }
    }

    /**
     * 调用失败处理
     */
    private void onFailure() {
        failureCount++;
        lastFailureTime = System.currentTimeMillis();

        if (currentState == State.HALF_OPEN) {
            // 半开状态下失败，转打开
            currentState = State.OPEN;
        } else if (failureCount >= failureThreshold) {
            // 达到阈值，转打开
            currentState = State.OPEN;
        }
    }
}
```

### 5.5 限流算法

限流是保护系统的常用手段。

**限流算法对比：**

```java
/**
 * 限流算法详解
 *
 * 1. 计数器算法（Fixed Window Counter）
 *
 * 原理：
 * - 固定时间窗口内统计请求数
 * - 超过阈值则拒绝
 *
 * 优点：简单、易实现
 * 缺点：窗口边界可能出现突发流量（2倍阈值）
 *
 * 2. 滑动窗口算法（Sliding Window Counter）
 *
 * 原理：
 * - 将时间窗口划分为多个小窗口
 * - 统计当前时间和之前窗口的请求总数
 *
 * 优点：平滑流量、精确限流
 * 缺点：实现复杂、占用内存
 *
 * 3. 令牌桶算法（Token Bucket）
 *
 * 原理：
 * - 以固定速率向桶中添加令牌
 * - 请求需要获取令牌才能执行
 * - 桶满时不添加，满了不丢弃
 *
 * 优点：允许突发流量
 * 缺点：实现稍复杂
 *
 * 4. 漏桶算法（Leaky Bucket）
 *
 * 原理：
 * - 请求以任意速率进入漏桶
 * - 漏桶以固定速率漏出
 * - 桶满时拒绝请求
 *
 * 优点：平滑输出、稳定流量
 * 缺点：不支持突发流量
 */
public class RateLimitAlgorithms {

    /**
     * 计数器算法实现
     */
    public class CounterRateLimiter {

        private int maxRequests;        // 时间窗口内最大请求数
        private long windowSize;         // 时间窗口大小（毫秒）
        private AtomicLong counter;      // 当前计数
        private long windowStart;        // 窗口开始时间

        public CounterRateLimiter(int maxRequests, long windowSize) {
            this.maxRequests = maxRequests;
            this.windowSize = windowSize;
            this.counter = new AtomicLong(0);
            this.windowStart = System.currentTimeMillis();
        }

        public synchronized boolean tryAcquire() {
            long now = System.currentTimeMillis();

            // 窗口过期，重置
            if (now - windowStart >= windowSize) {
                windowStart = now;
                counter.set(0);
            }

            return counter.incrementAndGet() <= maxRequests;
        }
    }

    /**
     * 令牌桶算法实现
     */
    public class TokenBucketRateLimiter {

        private double rate;              // 令牌添加速率（个/秒）
        private double bucketCapacity;    // 桶容量
        private double currentTokens;     // 当前令牌数
        private long lastRefillTime;      // 最后填充时间

        public TokenBucketRateLimiter(double rate, double bucketCapacity) {
            this.rate = rate;
            this.bucketCapacity = bucketCapacity;
            this.currentTokens = bucketCapacity;
            this.lastRefillTime = System.currentTimeMillis();
        }

        public synchronized boolean tryAcquire(int permits) {
            // 先填充令牌
            refill();

            if (currentTokens >= permits) {
                currentTokens -= permits;
                return true;
            }
            return false;
        }

        private void refill() {
            long now = System.currentTimeMillis();
            double elapsed = (now - lastRefillTime) / 1000.0;

            // 添加令牌
            double tokensToAdd = elapsed * rate;
            currentTokens = Math.min(bucketCapacity, currentTokens + tokensToAdd);
            lastRefillTime = now;
        }
    }

    /**
     * 滑动窗口算法实现
     */
    public class SlidingWindowRateLimiter {

        private int maxRequests;           // 窗口内最大请求数
        private int windowSizeInSeconds;  // 窗口大小（秒）
        private ConcurrentLinkedQueue<Long> timestamps;

        public SlidingWindowRateLimiter(int maxRequests, int windowSizeInSeconds) {
            this.maxRequests = maxRequests;
            this.windowSizeInSeconds = windowSizeInSeconds;
            this.timestamps = new ConcurrentLinkedQueue<>();
        }

        public synchronized boolean tryAcquire() {
            long now = System.currentTimeMillis();
            long windowStart = now - windowSizeInSeconds * 1000L;

            // 移除窗口外的记录
            while (!timestamps.isEmpty() && timestamps.peek() < windowStart) {
                timestamps.poll();
            }

            if (timestamps.size() < maxRequests) {
                timestamps.offer(now);
                return true;
            }
            return false;
        }
    }
}
```

```java
 * 接口限流实战
 *
 * 基于 Redis + Lua 实现分布式限流
 */
public class DistributedRateLimiter {

    private Jedis jedis;

    /**
     * Redis + Lua 滑动窗口限流
     *
     * Lua 脚本保证原子性
     */
    public boolean isAllowed(String key, int maxRequests, int windowSeconds) {
        String luaScript =
            "local key = KEYS[1] " +
            "local max = tonumber(ARGV[1]) " +
            "local window = tonumber(ARGV[2]) " +
            "local now = tonumber(ARGV[3]) " +
            "local windowStart = now - window * 1000 " +

            // 删除窗口外的记录
            "redis.call('ZREMRANGEBYSCORE', key, 0, windowStart) " +

            // 获取当前请求数
            "local current = redis.call('ZCARD', key) " +

            "if current < max then " +
            // 添加当前请求
            "    redis.call('ZADD', key, now, now .. ':' .. math.random()) " +
            "    redis.call('PEXPIRE', key, window * 1000) " +
            "    return 1 " +
            "else " +
            "    return 0 " +
            "end";

        long now = System.currentTimeMillis();
        Object result = jedis.eval(
            luaScript,
            1,
            key,
            String.valueOf(maxRequests),
            String.valueOf(windowSeconds),
            String.valueOf(now)
        );

        return Long.valueOf(1).equals(result);
    }

     * 令牌桶限流（Redis 实现）
     */

            "local rate = tonumber(ARGV[1]) " +
            "local capacity = tonumber(ARGV[2]) " +
            "local now = tonumber(ARGV[3]) " +

            // 获取当前令牌数和时间戳
            "local data = redis.call('HMGET', key, 'tokens', 'lastTime') " +
            "local tokens = tonumber(data[1]) " +
            "local lastTime = tonumber(data[2]) " +

            // 初始化
            "if tokens == nil then " +
            "    tokens = capacity " +
            "    lastTime = now " +
            "end " +

            // 计算应补充的令牌数
            "local elapsed = (now - lastTime) / 1000 " +
            "local补给 = elapsed * rate " +
            "tokens = math.min(capacity, tokens + 补给) " +

            // 检查是否足够
            "if tokens >= 1 then " +
            "    tokens = tokens - 1 " +
            "    redis.call('HMSET', key, 'tokens', tokens, 'lastTime', now) " +
            "    redis.call('EXPIRE', key, 60) " +
            "    return 1 " +
            "else " +
            "    redis.call('HMSET', key, 'tokens', tokens, 'lastTime', now) " +
            "    redis.call('EXPIRE', key, 60) " +
            "    return 0 " +
            "end";

        long now = System.currentTimeMillis();
        Object result = jedis.eval(
            luaScript, 1, key,
            String.valueOf(rate),
            String.valueOf(capacity),
            String.valueOf(now)
        );

        return Long.valueOf(1).equals(result);
    }

    /**
     * Spring Cloud Gateway 限流配置
```

```java
/**
 * 简化版任务调度系统实现
 *
 * 核心组件：
 * - 调度器（Scheduler）：管理任务调度
 * - 执行器（Executor）：执行具体任务
 * - 注册中心（Registry）：管理执行器节点
 *
 * 简化实现，不考虑分布式、高可用等复杂场景
 */
public class SimpleTaskScheduler {

    /**
     * 任务定义
     */
    public static class Task {
        private String id;
        private String name;
        private String cron;
        private String handlerClass;
        private TaskStatus status;

        public enum TaskStatus {
            STOPPED,
            RUNNING,
            PAUSED
        }
    }

    /**
     * 任务执行记录
     */
    public static class TaskExecution {
        private String taskId;
        private String executorId;
        private long startTime;
        private long endTime;
        private ExecutionStatus status;
        private String errorMessage;

        public enum ExecutionStatus {
            SUCCESS,
            FAILED,
            RUNNING
        }
    }

    /**
     * 任务处理器接口
     */
    public interface TaskHandler {
        void execute(String params) throws Exception;
    }

    /**
     * 任务执行器
     */
    public class TaskExecutor implements Runnable {

        private Task task;
        private TaskHandler handler;
        private ExecutorService executorService;

        @Override
        public void run() {
            // 执行任务
            try {
                handler.execute(task.getId());
            } catch (Exception e) {
                // 记录错误
            }
        }
    }

    /**
     * 调度器核心
     */
    public class Scheduler {

        private Map<String, Task> tasks = new ConcurrentHashMap<>();
        private Map<String, List<String>> runningTasks = new ConcurrentHashMap<>();
        private ScheduledExecutorService scheduledExecutor;

        /**
         * 添加任务
         */
        public void addTask(Task task) {
            tasks.put(task.getId(), task);
        }

        /**
         * 启动调度
         */
        public void start() {
            scheduledExecutor = Executors.newScheduledThreadPool(4);

            // 每秒检查一次任务触发
            scheduledExecutor.scheduleAtFixedRate(() -> {
                for (Task task : tasks.values()) {
                    if (shouldRun(task)) {
                        executeTask(task);
                    }
                }
            }, 0, 1, TimeUnit.SECONDS);
        }

        /**
         * 判断任务是否应该执行
         */
        private boolean shouldRun(Task task) {
            if (task.getStatus() != Task.TaskStatus.RUNNING) {
                return false;
            }

            // 检查Cron表达式
            CronExpression cron = new CronExpression(task.getCron());
            Date nextRun = cron.getNextValidTimeAfter(new Date());

            // 如果下一执行时间在1秒内，返回true
            return nextRun != null &&
                   (nextRun.getTime() - System.currentTimeMillis()) < 1000;
        }

        /**
         * 执行任务
         */
        private void executeTask(Task task) {
            // 获取执行器
            ExecutorService executor = Executors.newSingleThreadExecutor();

            // 提交任务
            executor.submit(() -> {
                try {
                    // 执行业务逻辑
                    // ...
                } finally {
                    executor.shutdown();
                }
            });
        }

        /**
         * 停止任务
         */
        public void stopTask(String taskId) {
            Task task = tasks.get(taskId);
            if (task != null) {
                task.setStatus(Task.TaskStatus.STOPPED);
            }
        }
    }
}
```

### 9.3 库存扣减方案

```java
/**
 * 库存扣减方案
 *
 * 核心问题：如何保证库存不超卖
 *
 * 方案1：数据库乐观锁
 * UPDATE inventory SET stock = stock - 1
 * WHERE product_id = ? AND stock > 0
 *
 * 方案2：Redis原子操作
 * DECR stock_key
 *
 * 方案3：Redis + 数据库双写
 *
 * 推荐方案：Redis预扣 + 数据库确认
 */
public class InventoryDeduction {

    /**
     * Redis Lua脚本扣减库存
     *
     * 优点：
     * - 原子性操作，不会超卖
     * - 性能高，单机可达10万QPS
     *
     * 缺点：
     * - Redis故障时需要降级
     */
    public class RedisInventoryDeduction {

        private Jedis jedis;

        /**
         * 扣减库存Lua脚本
         */
        public boolean deductStock(String productId, int quantity) {
            String script =
                "local stock = redis.call('GET', KEYS[1]) " +
                "if stock == false then " +
                "    return -1 " +  // 库存不存在
                "end " +
                "stock = tonumber(stock) " +
                "if stock < tonumber(ARGV[1]) then " +
                "    return 0 " +    // 库存不足
                "end " +
                "stock = stock - tonumber(ARGV[1]) " +
                "redis.call('SET', KEYS[1], stock) " +
                "return stock";      // 返回剩余库存

            String key = "stock:" + productId;
            Object result = jedis.eval(script, 1, key, String.valueOf(quantity));

            Long remaining = Long.valueOf(result.toString());
            return remaining >= 0;
        }

        /**
         * 库存回滚Lua脚本
         *
         * 用于取消订单时回滚库存
         */
        public void rollbackStock(String productId, int quantity) {
            String script =
                "redis.call('INCRBY', KEYS[1], ARGV[1])";

            String key = "stock:" + productId;
            jedis.eval(script, 1, key, String.valueOf(quantity));
        }
    }

    /**
     * 数据库乐观锁扣减
     *
     * 优点：
     * - 数据一致性高
     * - 不依赖Redis
     *
     * 缺点：
     * - 性能较低
     * - 并发高时大量失败重试
     */
    public boolean deductStockByDB(Long productId, int quantity) {
        // 乐观锁：使用版本号或状态控制
        String sql =
            "UPDATE inventory " +
            "SET stock = stock - ?, " +
            "    version = version + 1 " +
            "WHERE product_id = ? " +
            "  AND stock >= ? " +
            "  AND version = ?";

        int affected = jdbcTemplate.update(sql,
            quantity, productId, quantity, currentVersion);

        return affected > 0;
    }
```

```java
 * - 而是请求进入队列
```

## 总结

### 分布式系统核心知识点

```
一、基础理论
├── CAP理论：一致性、可用性、分区容错性
├── BASE理论：基本可用、软状态、最终一致
└── 一致性模型：强一致、弱一致、最终一致

二、分布式事务
├── 2PC：两阶段提交，强一致但有阻塞
├── 3PC：三阶段提交，改进了阻塞问题
├── TCC：补偿事务，业务层面实现
└── Saga：长事务拆分，补偿式执行

三、分布式锁
├── Redis分布式锁：SETNX + Lua原子操作
├── Redlock：多节点提高可靠性
├── Zookeeper分布式锁：临时顺序节点
└── 坑：超时、续期、可重入、误删

四、服务治理
├── 注册中心：Eureka、Nacos、Consul
├── 负载均衡：客户端、服务端
├── 熔断降级：Hystrix、Sentinel
└── 限流算法：计数器、滑动窗口、令牌桶、漏桶

五、配置中心
├── Apollo：功能完善，适合大规模
├── Nacos：轻量级，国产推荐
└── 热更新：动态配置生效

六、网关设计
├── Kong：插件丰富，性能高
├── Nginx：负载均衡，反向代理
└── Spring Cloud Gateway：Spring生态

七、分布式任务
├── XXL-JOB：国产轻量级调度平台
├── Elastic Job：支持分片，特性丰富
└── 任务分片：并行处理大规模任务

八、实战案例
└── 秒杀系统：限流、缓存、锁、队列的综合应用
```
