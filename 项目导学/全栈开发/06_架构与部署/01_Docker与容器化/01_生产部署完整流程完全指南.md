# 生产部署完整流程完全指南

> 本文将带你从零开始理解生产部署的完整生命周期，包括开发流程、上线步骤、灰度发布策略和回滚机制。假设你是一个刚接手项目的开发新人，我会用通俗易懂的语言配合实际案例来讲解。

## 一、生产部署到底是什么？

### 1.1 像个餐厅的菜品上线流程

想象一下，你是一家餐厅的厨师长，要推出一道新菜品。这个过程和软件部署其实非常相似：

| 餐厅环节 | 软件部署环节 | 说明 |
|---------|-------------|------|
| 研发新菜品 | 开发新功能 | 在测试环境研发 |
| 内部试吃 | 内测/灰度测试 | 让小部分用户先体验 |
| 调整口味 | 修复Bug | 根据反馈优化 |
| 大厨做菜 | 构建生产包 | 打包可部署版本 |
| 传菜员上菜 | 部署到服务器 | 真正让用户使用 |
| 收集反馈 | 监控与日志 | 看用户是否满意 |
| 紧急撤菜 | 回滚 | 出问题时快速恢复 |

### 1.2 生产环境的特殊性

**开发环境**就像是你家的厨房，你想怎么折腾都行，锅碗瓢盆随便用，食材不够了随时买。但**生产环境**就像是一家米其林餐厅的后厨，每一步都必须标准化、规范化，因为出了问题影响的是成千上万的顾客。

```
开发环境（你家厨房）          生产环境（米其林餐厅后厨）
├── 可以随便试错              ├── 必须严格按照流程
├── 日志随便打                 ├── 日志要有策略地记录
├── 性能无所谓                 ├── 性能必须优化
├── 可以用本地数据库           ├── 必须用高可用数据库
├── 端口随便开                 ├── 安全端口才开放
└── 团队1-2人                  └── 可能有上千人同时访问
```

### 2.2 各阶段详细说明

#### 第一阶段：代码编写（开发者的日常）

这是你每天都在做的事情，但即使是这个阶段，也有一些为生产环境做准备的好习惯：

```typescript
// ❌ 开发环境写法：hardcode配置
const API_URL = 'http://localhost:3000';
const DB_HOST = 'localhost';

// ✅ 生产就绪写法：使用环境变量
const API_URL = process.env.NEXT_PUBLIC_API_URL;
const DB_HOST = process.env.DB_HOST;

/**
 * 为什么这样做？
 * 因为生产环境的地址和开发环境完全不同
 * 开发环境可能是 localhost:3000
 * 生产环境可能是 api.example.com
 * 如果hardcode，上线后还要改代码
 */
```

#### 第二阶段：本地测试

本地测试不仅要测功能，还要模拟生产环境的配置：

```bash
# 创建 .env.local 文件（本地开发用）
cat > .env.local << EOF
# 数据库配置
DATABASE_URL=postgresql://localhost:5432/myapp_dev

# API配置
NEXT_PUBLIC_API_URL=http://localhost:3000/api

# 第三方服务（测试key）
STRIPE_TEST_KEY=sk_test_xxx
SENDGRID_TEST_KEY=SG.xxx

# 日志级别
LOG_LEVEL=debug
EOF

# 创建 .env.production 文件（生产环境用）
cat > .env.production << EOF
# 数据库配置 - 生产数据库地址
DATABASE_URL=postgresql://prod-db.example.com:5432/myapp

# API配置 - 真实的生产域名
NEXT_PUBLIC_API_URL=https://api.example.com/api

# 第三方服务（真实的生产key）
STRIPE_LIVE_KEY=sk_live_xxx
SENDGRID_LIVE_KEY=SG.xxx

# 日志级别 - 生产环境用error级别，减少日志量
LOG_LEVEL=error
EOF
```

#### 第三阶段：Git提交与分支管理

生产部署的Git流程是有讲究的，不是想提交就提交：

```bash
# 常见的Git分支策略：GitFlow
#
# master (或 main)    ──── 生产环境永远从这个分支部署
#                              ▲
#                              │ 只有发布经理才能合并
#                              │
# develop                    ──┴── 开发集成分支
#      ▲
#      │ 开发完成后合并
#      │
# feature/xxx               ──── 功能开发分支
# hotfix/xxx                ──── 紧急修复分支（从master切出）

# 举例：新功能开发流程

# 1. 从develop创建功能分支
git checkout develop
git pull origin develop
git checkout -b feature/user-dashboard

# 2. 开发完成后，提交PR到develop
git add .
git commit -m "feat: 添加用户仪表盘功能"
git push origin feature/user-dashboard

# 3. 代码review通过后，合并到develop
git checkout develop
git merge feature/user-dashboard
git push origin develop

# 4. 测试通过后，从develop创建release分支
git checkout -b release/v1.2.0

# 5. 发布前最后的bug修复都在release分支进行
git commit -m "fix: 修复发布前发现的样式问题"

# 6. 发布完成后，合并回master和develop
git checkout master
git merge release/v1.2.0
git tag -a v1.2.0 -m "版本1.2.0发布"
git push origin master --tags

git checkout develop
git merge release/v1.2.0
git push origin develop

# 7. 删除已经完成的功能分支
git branch -d feature/user-dashboard
git push origin --delete feature/user-dashboard
```

### 2.3 CI/CD自动化流程

现在大多数公司都用自动化构建和部署，就像一条流水线：

```yaml
# .github/workflows/deploy.yml
name: 生产部署流水线

on:
  # 当master分支有新提交时触发
  push:
    branches: [master]
  # 也可以手动触发
  workflow_dispatch:

jobs:
  # ==================== 第一步：构建阶段 ====================
  build:
    runs-on: ubuntu-latest

    steps:
      # 1. 拉取代码
      - name: 拉取源码
        uses: actions/checkout@v4

      # 2. 设置Node.js环境
      - name: 设置Node.js 20
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      # 3. 安装依赖
      - name: 安装项目依赖
        run: npm ci

      # 4. 运行类型检查
      - name: TypeScript类型检查
        run: npm run type-check

      # 5. 运行代码检查
      - name: ESLint代码检查
        run: npm run lint

      # 6. 运行单元测试
      - name: 执行单元测试
        run: npm run test:unit

      # 7. 构建生产版本
      - name: 构建生产环境
        run: npm run build
        env:
          # 注入生产环境变量
          NEXT_PUBLIC_API_URL: ${{ secrets.PROD_API_URL }}
          DATABASE_URL: ${{ secrets.PROD_DATABASE_URL }}

      # 8. 上传构建产物（供后续阶段使用）
      - name: 上传构建产物
        uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: .next
          retention-days: 7

  # ==================== 第二步：测试阶段 ====================
  test:
    runs-on: ubuntu-latest
    needs: build

    steps:
      - name: 拉取源码
        uses: actions/checkout@v4

      - name: 设置Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: 安装依赖
        run: npm ci

      - name: 下载构建产物
        uses: actions/download-artifact@v4
        with:
          name: build-output
          path: .next

      - name: 启动预览服务器
        run: npm run start &
        sleep 5  # 等待服务器启动

      - name: 运行E2E测试
        run: npm run test:e2e
        env:
          E2E_BASE_URL: http://localhost:3000

  # ==================== 第三步：部署阶段 ====================
  deploy:
    runs-on: ubuntu-latest
    needs: test
    # 生产部署必须确认代码没问题
    if: github.ref == 'refs/heads/master'

    steps:
      - name: 拉取源码
        uses: actions/checkout@v4

      - name: 下载构建产物
        uses: actions/download-artifact@v4
        with:
          name: build-output
          path: .next

      - name: 部署到服务器
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.PROD_HOST }}
          username: ${{ secrets.PROD_USER }}
          key: ${{ secrets.PROD_SSH_KEY }}
          script: |
            # 进入部署目录
            cd /var/www/myapp

            # 备份当前版本（出问题可以快速回滚）
            cp -r current previous_version_backup

            # 停止当前服务
            pm2 stop myapp || true

            # 部署新版本
            cp -r .next current
            cp .env.production current/.env

            # 安装依赖（生产环境精简）
            npm ci --production

            # 启动服务
            pm2 start ecosystem.config.js

            # 验证部署是否成功
            curl -f https://example.com/api/health || exit 1

            # 清理旧的备份（保留最近3个）
            ls -dt previous_version_backup* | tail -n +4 | xargs rm -rf

  # ==================== 第四步：通知阶段 ====================
  notify:
    runs-on: ubuntu-latest
    needs: deploy

    steps:
      - name: 发送部署成功通知
        if: success()
        uses: actions/github-script@v7
        with:
          script: |
            // 发送钉钉/飞书/Slack通知
            fetch('${{ secrets.DEPLOY_WEBHOOK }}', {
              method: 'POST',
              body: JSON.stringify({
                msgtype: 'text',
                text: {
                  content: '✅ 生产环境部署成功\n版本：${{ github.sha }}\n负责人：${{ github.actor }}'
                }
              })
            });

      - name: 发送部署失败通知
        if: failure()
        uses: actions/github-script@v7
        with:
          script: |
            // 发送告警通知
            fetch('${{ secrets.ALERT_WEBHOOK }}', {
              method: 'POST',
              body: JSON.stringify({
                msgtype: 'text',
                text: {
                  content: '🚨 生产环境部署失败！\n版本：${{ github.sha }}\n请立即检查！'
                }
              })
            });
```

## 三、上线步骤详解

### 3.1 上线前的检查清单

上线就像飞机起飞前的检查，不能漏掉任何一项：

```bash
# ============================================
# 上线前必做检查清单（建议做成shell脚本）
# ============================================

#!/bin/bash

echo "========================================"
echo "       生产部署前检查清单"
echo "========================================"

# 1. 检查代码是否已经review
echo "[1/15] 检查PR是否已Review..."
if [ -z "$PR_APPROVED" ]; then
  echo "❌ 错误：PR还未被Review！"
  exit 1
fi
echo "✅ PR已Review"

# 2. 检查测试覆盖率
echo "[2/15] 检查测试覆盖率..."
COVERAGE=$(cat coverage/coverage-summary.json | jq '.total.lines.pct')
if (( $(echo "$COVERAGE < 80" | bc -l) )); then
  echo "❌ 错误：测试覆盖率仅${COVERAGE}%，低于80%标准！"
  exit 1
fi
echo "✅ 测试覆盖率：${COVERAGE}%"

# 3. 检查是否有敏感信息泄露
echo "[3/15] 检查敏感信息..."
if grep -r "password\s*=\s*['\"][^'\"]*['\"]" --include="*.ts" --include="*.js" .; then
  echo "❌ 错误：发现硬编码密码！"
  exit 1
fi
echo "✅ 无硬编码密码"

# 4. 检查环境变量配置
echo "[4/15] 检查生产环境变量..."
REQUIRED_VARS=(
  "DATABASE_URL"
  "NEXT_PUBLIC_API_URL"
  "REDIS_URL"
  "JWT_SECRET"
)

for var in "${REQUIRED_VARS[@]}"; do
  if [ -z "${!var}" ]; then
    echo "❌ 错误：缺少环境变量 $var"
    exit 1
  fi
done
echo "✅ 环境变量配置完整"

# 5. 检查依赖安全漏洞
echo "[5/15] 检查依赖安全漏洞..."
npm audit --audit-level=high
if [ $? -ne 0 ]; then
  echo "❌ 错误：发现高危安全漏洞！"
  exit 1
fi
echo "✅ 无高危安全漏洞"

# 6. 检查数据库迁移脚本
echo "[6/15] 检查数据库迁移..."
if [ -d "prisma/migrations" ]; then
  echo "检测到Prisma迁移文件:"
  ls -la prisma/migrations/
  read -p "确认已备份生产数据库？ (y/n) " -n 1 -r
  echo
  if [[ ! $REPLY =~ ^[Yy]$ ]]; then
    echo "❌ 取消部署"
    exit 1
  fi
fi
echo "✅ 数据库迁移检查完成"

# 7. 通知相关人员
echo "[7/15] 通知团队成员..."
# 发送上线通知到钉钉群
curl -X POST "$WEBHOOK_URL" \
  -H 'Content-Type: application/json' \
  -d '{"msgtype": "text", "text": {"content": "🔔 即将进行生产部署\n版本：'"$VERSION"'\n负责人：'"$USER"'\n预计时间：5分钟"}}'
echo "✅ 已发送通知"

# ... 其他检查项 ...

echo "========================================"
echo "       所有检查通过！准备部署"
echo "========================================"
```

### 3.2 手动部署操作步骤

虽然现在大多用自动化部署，但了解手动部署的步骤很重要，万一自动化出问题了呢：

```bash
#!/bin/bash
# ============================================
# 生产环境手动部署脚本
# ============================================

set -e  # 任何命令失败就停止

# 定义颜色输出
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
NC='\033[0m' # No Color

log_info() { echo -e "${GREEN}[INFO]${NC} $1"; }
log_warn() { echo -e "${YELLOW}[WARN]${NC} $1"; }
log_error() { echo -e "${RED}[ERROR]${NC} $1"; }

# 配置变量
APP_NAME="myapp"
APP_DIR="/var/www/$APP_NAME"
BACKUP_DIR="/var/www/backups"
CURRENT_LINK="$APP_DIR/current"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)

# ============================================
# 步骤1：准备工作
# ============================================
log_info "========== 步骤1：准备工作 =========="

# 确认是生产环境
if [ "$ENVIRONMENT" != "production" ]; then
  log_error "这不是生产环境！当前环境：$ENVIRONMENT"
  exit 1
fi

# 检查磁盘空间（至少保留20%可用）
DISK_USAGE=$(df -h $APP_DIR | tail -1 | awk '{print $5}' | sed 's/%//')
if [ "$DISK_USAGE" -gt 80 ]; then
  log_error "磁盘空间不足！当前使用率：${DISK_USAGE}%"
  exit 1
fi

# 检查服务器负载
LOAD=$(uptime | awk -F'load average:' '{print $2}')
log_info "当前服务器负载：$LOAD"

# 创建备份目录
mkdir -p $BACKUP_DIR

log_info "准备工作完成"

# ============================================
# 步骤2：备份当前版本
# ============================================
log_info "========== 步骤2：备份当前版本 =========="

if [ -d "$CURRENT_LINK" ]; then
  BACKUP_NAME="${BACKUP_DIR}/${APP_NAME}_${TIMESTAMP}"

  log_info "正在备份当前版本到：$BACKUP_NAME"

  # 备份整个应用目录
  cp -r $CURRENT_LINK $BACKUP_NAME

  # 备份数据库（如果使用文件数据库）
  if [ -f "$CURRENT_LINK/data.db" ]; then
    cp $CURRENT_LINK/data.db "${BACKUP_NAME}/data.db"
  fi

  # 记录备份信息
  echo "Backup created at: $TIMESTAMP" > "${BACKUP_NAME}/backup_info.txt"
  echo "App version: $(cat ${CURRENT_LINK}/version.txt 2>/dev/null || echo 'unknown')" >> "${BACKUP_NAME}/backup_info.txt"

  log_info "备份完成！备份目录：$BACKUP_NAME"
else
  log_warn "没有找到当前版本，跳过备份"
fi

# ============================================
# 步骤3：停止服务
# ============================================
log_info "========== 步骤3：停止服务 =========="

# 使用PM2停止应用
if command -v pm2 &> /dev/null; then
  log_info "使用PM2停止应用..."
  pm2 stop $APP_NAME || true
  pm2 delete $APP_NAME || true

  # 等待进程完全停止
  sleep 3

  # 确认进程已停止
  if pm2 list | grep -q $APP_NAME; then
    log_error "应用停止失败，仍在运行"
    exit 1
  fi
fi

# 如果使用Nginx反向代理，先停止Nginx
# systemctl stop nginx

log_info "服务已停止"

# ============================================
# 步骤4：部署新版本
# ============================================
log_info "========== 步骤4：部署新版本 =========="

# 创建新的发布目录
RELEASE_DIR="${APP_DIR}/releases/${TIMESTAMP}"
mkdir -p $RELEASE_DIR

log_info "新版本目录：$RELEASE_DIR"

# 从构建服务器拉取最新代码（或者直接复制本地构建产物）
log_info "复制构建产物..."

# 如果是远程服务器，使用scp拉取：
# scp -r user@build-server:/path/to/build/* $RELEASE_DIR/

# 如果是本地，直接复制
cp -r .next/* $RELEASE_DIR/
cp .env.production $RELEASE_DIR/.env

# 设置正确的文件权限
chown -R www-data:www-data $RELEASE_DIR
find $RELEASE_DIR -type d -exec chmod 755 {} \;
find $RELEASE_DIR -type f -exec chmod 644 {} \;

# 记录版本信息
git rev-parse HEAD > $RELEASE_DIR/version.txt
echo "Deployed at: $(date)" >> $RELEASE_DIR/version.txt

# 符号链接切换（实现无 downtime 部署）
log_info "切换到新版本..."
ln -sfn $RELEASE_DIR $CURRENT_LINK

# ============================================
# 步骤5：启动服务
# ============================================
log_info "========== 步骤5：启动服务 =========="

# 安装生产依赖
log_info "安装生产依赖..."
cd $CURRENT_LINK
npm ci --production --prefer-offline

# 使用PM2启动应用
log_info "启动应用..."
NODE_ENV=production pm2 start ecosystem.config.js --name $APP_NAME

# 等待应用启动
sleep 5

# 检查应用是否正常启动
if ! pm2 list | grep -q "$APP_NAME.*online"; then
  log_error "应用启动失败！"
  pm2 logs $APP_NAME --lines 50
  exit 1
fi

# 保存PM2进程列表（以便重启后自动恢复）
pm2 save

log_info "应用启动成功！"

# ============================================
# 步骤6：验证部署
# ============================================
log_info "========== 步骤6：验证部署 =========="

# 检查健康检查接口
log_info "检查健康检查接口..."
for i in {1..5}; do
  HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/api/health)

  if [ "$HTTP_CODE" = "200" ]; then
    log_info "✅ 健康检查通过！HTTP状态码：$HTTP_CODE"
    break
  else
    log_warn "健康检查失败，重试第${i}次..."
    sleep 5
  fi

  if [ $i -eq 5 ]; then
    log_error "❌ 健康检查全部失败！开始回滚..."
    # 自动回滚
    $0 rollback
    exit 1
  fi
done

# 检查关键页面是否可以访问
log_info "检查关键页面..."
curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/ | grep -q "200" && \
  log_info "✅ 首页访问正常" || log_warn "⚠️ 首页访问异常"

# ============================================
# 步骤7：清理旧版本
# ============================================
log_info "========== 步骤7：清理旧版本 =========="

# 保留最近5个版本
log_info "清理旧版本备份..."
cd $BACKUP_DIR
ls -dt ${APP_NAME}_* | tail -n +6 | xargs rm -rf 2>/dev/null || true
```

### 3.3 一键回滚脚本

回滚是生产部署的救命稻草，必须熟练掌握：

```bash
#!/bin/bash
# ============================================
# 生产环境一键回滚脚本
# ============================================

set -e

RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
NC='\033[0m'

log_info() { echo -e "${GREEN}[INFO]${NC} $1"; }
log_warn() { echo -e "${YELLOW}[WARN]${NC} $1"; }
log_error() { echo -e "${RED}[ERROR]${NC} $1"; }

APP_NAME="myapp"
APP_DIR="/var/www/$APP_NAME"
BACKUP_DIR="/var/www/backups"
CURRENT_LINK="$APP_DIR/current"

# ============================================
# 回滚函数
# ============================================
do_rollback() {
  log_warn "========== 开始回滚操作 =========="

  # 发送告警通知
  curl -X POST "$WEBHOOK_URL" \
    -H 'Content-Type: application/json' \
    -d '{
      "msgtype": "text",
      "text": {
        "content": "🚨 开始生产环境回滚！\n原因：'"$REASON"'\n操作人：'"$USER"'"
      }
    }' || true

  # 检查是否有可用的备份
  if [ ! -d "$BACKUP_DIR" ]; then
    log_error "没有找到备份目录！"
    exit 1
  fi

  # 获取最新的备份
  LATEST_BACKUP=$(ls -dt ${BACKUP_DIR}/${APP_NAME}_* | head -1)

  if [ -z "$LATEST_BACKUP" ]; then
    log_error "没有找到可用的备份！"
    exit 1
  fi

  log_info "找到最新备份：$LATEST_BACKUP"

  # 停止当前服务
  log_info "停止当前服务..."
  pm2 stop $APP_NAME || true
  pm2 delete $APP_NAME || true

  # 备份当前出问题的版本（以防万一）
  if [ -d "$CURRENT_LINK" ]; then
    PROBLEM_BACKUP="${BACKUP_DIR}/${APP_NAME}_problem_$(date +%Y%m%d_%H%M%S)"
    log_info "备份当前问题版本到：$PROBLEM_BACKUP"
    cp -r $CURRENT_LINK $PROBLEM_BACKUP
  fi

  # 切换到上一个正常版本
  log_info "切换到上一个正常版本..."
  ln -sfn $LATEST_BACKUP $CURRENT_LINK

  # 安装依赖
  log_info "安装依赖..."
  cd $CURRENT_LINK
  npm ci --production --prefer-offline

  # 启动服务
  log_info "启动服务..."
  NODE_ENV=production pm2 start ecosystem.config.js --name $APP_NAME

  # 等待启动
  sleep 5

  # 验证启动成功
  if pm2 list | grep -q "$APP_NAME.*online"; then
    log_info "✅ 回滚成功！应用已正常启动"

    # 发送通知
    curl -X POST "$WEBHOOK_URL" \
      -H 'Content-Type: application/json' \
      -d '{
        "msgtype": "text",
        "text": {
          "content": "✅ 生产环境回滚成功\n回滚到版本：'"$(cat $LATEST_BACKUP/version.txt)"'\n如有问题请检查！"
        }
      }'
  else
    log_error "❌ 回滚后应用启动失败！需要人工介入！"
    exit 1
  fi

  log_warn "========== 回滚操作完成 =========="
}

# ============================================
# 主程序
# ============================================

# 显示用法
if [ "$1" = "-h" ] || [ "$1" = "--help" ]; then
  echo "用法: $0 [选项]"
  echo ""
  echo "选项:"
  echo "  rollback        执行回滚（使用上一个备份）"
  echo "  list            列出所有可用的备份"
  echo "  to <版本>       回滚到指定版本（备份目录名）"
  echo ""
  echo "示例:"
  echo "  $0 rollback              # 回滚到上一个版本"
  echo "  $0 list                   # 查看所有备份"
  echo "  $0 to myapp_20240315_1430 # 回滚到指定版本"
  exit 0
fi

# 列出所有备份
if [ "$1" = "list" ]; then
  echo "========================================"
  echo "       可用的备份版本"
  echo "========================================"
  ls -lht ${BACKUP_DIR}/${APP_NAME}_* | head -10
  echo ""
  echo "当前版本："
  if [ -L "$CURRENT_LINK" ]; then
    readlink -f $CURRENT_LINK
    cat $CURRENT_LINK/version.txt 2>/dev/null || echo "无法读取版本信息"
  else
    echo "没有当前版本"
  fi
  exit 0
fi

# 回滚到指定版本
if [ "$1" = "to" ] && [ -n "$2" ]; then
  SPECIFIC_BACKUP="$2"
  if [ ! -d "$SPECIFIC_BACKUP" ]; then
    # 尝试在备份目录中查找
    SPECIFIC_BACKUP="${BACKUP_DIR}/${APP_NAME}_${2}"
  fi

  if [ ! -d "$SPECIFIC_BACKUP" ]; then
    log_error "指定的备份不存在：$2"
    exit 1
  fi

  REASON="回滚到指定版本：$2"
  do_rollback
  exit 0
fi

# 默认执行回滚
if [ -z "$1" ] || [ "$1" = "rollback" ]; then
  REASON=${REASON:-"手动触发回滚"}
  do_rollback
  exit 0
fi

log_error "未知参数：$1"
echo "使用 $0 --help 查看帮助"
exit 1
```

## 四、灰度发布策略

### 4.1 灰度发布的意义

灰度发布就像是你推出新菜品时，先让一小部分顾客试吃，根据反馈再决定是否全面推出。这个策略的好处是：

```
┌─────────────────────────────────────────────────────────┐
│                    灰度发布的价值                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 降低风险                                              │
│     └── 新版本有问题，只影响小部分用户                      │
│                                                         │
│  2. 收集真实反馈                                          │
│     └── 让真实用户告诉你哪里不好用                         │
│                                                         │
│  3. 发现隐藏Bug                                          │
│     └── 大量用户使用时才能暴露的问题                        │
│                                                         │
│  4. 便于快速修复                                          │
│     └── 小范围出问题，修复影响也小                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 4.2 灰度发布的常见策略

#### 策略一：按用户比例灰度

```typescript
// 使用用户ID的哈希值来决定是否在灰度范围内
// 这样可以保证同一个用户每次访问都得到一致的体验

function isInGrayGroup(userId: string, grayPercent: number): boolean {
  // 使用一个简单的哈希函数
  let hash = 0;
  for (let i = 0; i < userId.length; i++) {
    const char = userId.charCodeAt(i);
    hash = ((hash << 5) - hash) + char;
    hash = hash & hash; // 转换为32位整数
  }

  // 取绝对值后取模，得到0-99的值
  const bucket = Math.abs(hash) % 100;

  // 如果bucket小于灰度百分比，则在灰度组内
  return bucket < grayPercent;
}

// 使用示例
function getFeatureFlags(userId: string) {
  return {
    // 开启新的编辑器，10%的用户可以使用
    newEditor: isInGrayGroup(userId, 10),

    // 开启深色模式，50%的用户可以使用
    darkMode: isInGrayGroup(userId, 50),

    // 新用户默认开启深色模式
    darkModeForNewUser: true,
  };
}
```

#### 策略二：按用户标签灰度

```typescript
// 按用户属性进行灰度分组

interface User {
  id: string;
  plan: 'free' | 'pro' | 'enterprise';  // 用户等级
  registrationDate: Date;                // 注册时间
  region: string;                        // 地区
  vipLevel: number;                      // VIP等级
}

/**
 * 判断用户是否应该开启某个功能
 *
 * @param user - 用户对象
 * @param featureName - 功能名称
 * @param config - 灰度配置
 */
function shouldEnableFeature(
  user: User,
  featureName: string,
  config: {
    // 允许哪些plan的用户使用
    allowedPlans?: ('free' | 'pro' | 'enterprise')[];
    // 注册超过多少天的用户才能使用（新用户可能不稳定）
    minRegisterDays?: number;
    // 允许哪些地区的用户使用
    allowedRegions?: string[];
    // 最低VIP等级
    minVipLevel?: number;
  }
): boolean {
  // 检查用户等级
  if (config.allowedPlans && !config.allowedPlans.includes(user.plan)) {
    return false;
  }

  // 检查注册时间
  if (config.minRegisterDays) {
    const daysSinceRegister = Math.floor(
      (Date.now() - user.registrationDate.getTime()) / (1000 * 60 * 60 * 24)
    );
    if (daysSinceRegister < config.minRegisterDays) {
      return false;
    }
  }

  // 检查地区
  if (config.allowedRegions && !config.allowedRegions.includes(user.region)) {
    return false;
  }

  // 检查VIP等级
  if (config.minVipLevel && user.vipLevel < config.minVipLevel) {
    return false;
  }

  return true;
}

// 使用示例
function getUserFeatures(user: User) {
  const features = {
    // 新编辑器只对Pro及以上用户开放
    newEditor: shouldEnableFeature(user, 'newEditor', {
      allowedPlans: ['pro', 'enterprise'],
      minRegisterDays: 30,  // 注册超过30天
      minVipLevel: 1,
    }),

    // 高级分析功能只对企业用户开放
    advancedAnalytics: shouldEnableFeature(user, 'advancedAnalytics', {
      allowedPlans: ['enterprise'],
    }),

    // A/B测试：给北京和上海的用户试用新功能
    experimentalFeature: shouldEnableFeature(user, 'experimentalFeature', {
      allowedRegions: ['beijing', 'shanghai'],
    }),
  };

  return features;
}
```

## 五、回滚机制详解

### 5.1 什么时候需要回滚？

```
┌─────────────────────────────────────────────────────────┐
│                   需要立即回滚的情况                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  🚨 严重错误（必须回滚）                                  │
│  ├── 服务完全不可用（500错误）                            │
│  ├── 数据库连接失败                                      │
│  ├── 核心功能完全失效                                    │
│  ├── 安全漏洞被发现                                      │
│  └── 用户数据丢失或损坏                                  │
│                                                         │
│  ⚠️ 警告问题（可以观察后决定）                            │
│  ├── 性能下降 20-30%                                    │
│  ├── 部分用户偶发错误                                    │
│  ├── 非核心功能异常                                      │
│  └── 样式/布局在小部分设备上异常                           │
│                                                         │
│  💡 优化问题（不需要回滚）                                │
│  ├── 新功能用户反馈不佳                                   │
│  ├── 想撤回新功能                                        │
│  └── A/B测试结果不符合预期                                │
│     └── 这些情况应该用功能开关，而不是回滚                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.2 回滚的决策流程

```typescript
// 回滚决策树

interface Incident {
  severity: 'critical' | 'high' | 'medium' | 'low';
  impact: {
    usersAffected: number;           // 受影响用户数
    errorRate: number;               // 错误率百分比
    revenueImpact: number;           // 预计收入损失（元）
    duration: number;                // 持续时间（分钟）
  };
  rootCause: 'known' | 'unknown';
  fixAvailable: boolean;             // 是否有现成的修复方案
}

/**
 * 根据事故等级和影响决定是否回滚
 *
 * 决策原则：
 * - 影响越大，越倾向回滚
 * - 有现成修复方案，可以考虑不立即回滚
 * - 原因不明时，先回滚再排查
 */
function shouldRollback(incident: Incident): {
  decision: 'immediate_rollback' | 'rollback_after_attempt' | 'monitor' | 'no_rollback';
  reason: string;
  priority: 'P0' | 'P1' | 'P2' | 'P3';
} {
  const { severity, impact, rootCause, fixAvailable } = incident;

  // P0：必须立即回滚
  if (severity === 'critical' || impact.errorRate > 10) {
    return {
      decision: 'immediate_rollback',
      reason: '严重级别事故或错误率超过10%，立即回滚',
      priority: 'P0'
    };
  }

  // P1：高优先级回滚
  if (severity === 'high' && impact.usersAffected > 1000) {
    return {
      decision: 'immediate_rollback',
      reason: '高严重级别且影响用户超过1000，立即回滚',
      priority: 'P1'
    };
  }

  // P2：尝试修复后再决定
  if (fixAvailable && impact.duration < 30) {
    return {
      decision: 'rollback_after_attempt',
      reason: '有修复方案，先尝试修复，30分钟无效则回滚',
      priority: 'P2'
    };
  }

  // P3：持续监控
  if (severity === 'medium' || impact.revenueImpact < 1000) {
    return {
      decision: 'monitor',
      reason: '中等级别，影响可控，持续监控',
      priority: 'P3'
    };
  }

  // 低优先级问题不需要回滚
  return {
    decision: 'no_rollback',
    reason: '问题影响小，使用功能开关禁用问题功能即可',
    priority: 'P4'
  };
}

// 使用示例
const incident: Incident = {
  severity: 'critical',
  impact: {
    usersAffected: 5000,
    errorRate: 25,
    revenueImpact: 50000,
    duration: 10
  },
  rootCause: 'unknown',
  fixAvailable: false
};

const decision = shouldRollback(incident);
console.log(decision);
// 输出：
// {
//   decision: 'immediate_rollback',
//   reason: '严重级别事故或错误率超过10%，立即回滚',
//   priority: 'P0'
// }
```

### 5.3 回滚的注意事项

```bash

# - 产品团队（可能需要通知用户）
# - 客服团队（准备好应对用户询问）
# - 管理层（重大事件需要知悉）

# 回滚只是止损，不是解决问题
# 回滚后必须：
# - 立即开始排查问题根因

# - 准备重新发布

# 回滚后的操作清单
# ============================================

curl -s http://localhost:3000/api/health

# 4. 检查错误日志
```

FastDocument是一个文档协作平台，下面是它从开发到上线的完整流程：

```bash
# ============================================
# FastDocument 生产部署实战

# 项目结构
# FastDocument/
# ├── frontend/          # Next.js前端
# ├── backend/           # NestJS后端

# 1. 前端部署
# ============================================

# 前端使用Next.js，需要先构建再部署

# 设置环境变量
export NEXT_PUBLIC_API_URL=https://api.fastdocument.com

# 构建生产版本

ls -la .next/

# ============================================

# 后端使用NestJS，同样需要构建

cd backend
npm ci
npm run build

# 构建产物在 dist/ 目录

# 4. Nginx反向代理配置
# ============================================

# /etc/nginx/conf.d/fastdocument.conf

upstream frontend {
    server 127.0.0.1:3001;  # Next.js默认端口
}

upstream backend {
    server 127.0.0.1:3000;  # NestJS后端端口

server {
    listen 443 ssl http2;
    server_name fastdocument.com;

    ssl_certificate /etc/ssl/certs/fastdocument.pem;
    ssl_certificate_key /etc/ssl/private/fastdocument.key;
    ssl_session_timeout 1d;
    ssl_session_cache shared:SSL:50m;

    # 安全头
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # 前端静态文件
    location / {
        proxy_pass http://frontend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;

        # 静态资源缓存
        location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2)$ {
            proxy_pass http://frontend;
            expires 30d;
            add_header Cache-Control "public, immutable";
        }
    }

    # API请求
    location /api/ {
        proxy_pass http://backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';

        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 86400;
```

## 七、总结

### 7.1 核心要点回顾

```
┌─────────────────────────────────────────────────────────┐
│                    生产部署核心要点                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 环境分离                                             │
│     ├── 开发环境：本地，随便折腾                          │
│     ├── 测试环境：模拟生产，完整测试                      │
│     └── 生产环境：真实用户，必须稳定                      │
│                                                         │
│  2. 部署流程                                            │
│     编写代码 → 本地测试 → Git提交 → CI构建 → 部署验证    │
│                                                         │
│  3. 灰度发布                                            │
│     ├── 小范围先试（10%）                               │
│     ├── 监控指标正常后扩大（50%）                       │
│     └── 全部用户（100%）                                │
│                                                         │
│  4. 回滚机制                                            │
│     ├── 出现问题立即回滚                                 │
│     ├── 回滚前通知相关人员                               │
│     └── 回滚后立即排查问题                               │
│                                                         │
│  5. 自动化是关键                                         │
│     └── 手动部署容易出错，自动化更可靠                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```
