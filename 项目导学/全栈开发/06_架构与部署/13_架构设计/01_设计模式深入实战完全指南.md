# 设计模式深入实战完全指南

## 前言：为什么要学习设计模式？

想象一下，你小时候玩乐高积木。设计师给你提供了各种形状的积木块——有的是带孔的，有的是带凸起的，有的是轮子，有的是门窗。当你按照说明书拼装时，发现这些积木块可以组合成汽车、房子、飞机等各种造型。设计模式就像是编程世界里的"积木块"，是前人总结出来的、可以被重复使用的"拼装方案"。

为什么要学习设计模式？因为软件开发本质上就是在"组合积木"。当你遇到一个具体问题时，如果你知道某种设计模式，就能快速找到一个经过验证的解决方案。设计模式不是死板的教条，而是灵活的思想。掌握了它们，就像掌握了一门艺术，让你的代码从"能跑就行"变成"优雅可维护"。

本文档将带你深入理解23种设计模式，通过大量的代码示例、UML图和实际应用场景，让你不仅"知道"这些模式，更能"运用"它们。

## 一、设计模式概述

### 1.1 设计模式的诞生

1994年，四位作者（Erich Gamma、Richard Helm、Ralph Johnson、John Vlissides）出版了一本名为《设计模式：可复用面向对象软件的基础》的书。这本书被大家亲切地称为"GoF book"（Gang of Four，四人组）。他们总结了23种在面向对象软件开发中反复出现的设计模式，从此改变了整个软件行业对代码结构的认知方式。

### 1.2 设计模式的分类

23种设计模式可以分为三大类：

| 类别 | 数量 | 描述 | 核心思想 |
|------|------|------|----------|
| **创建型模式** | 5种 | 专注于对象的创建机制 | 如何安全、高效地创建对象 |
| **结构型模式** | 7种 | 关注对象之间的组合方式 | 如何将对象组装成更大的结构 |
| **行为型模式** | 11种 | 关注对象之间的职责分配 | 如何让对象之间优雅地协作 |

### 1.3 设计模式的六大原则

在深入具体模式之前，我们需要理解这些模式背后共同遵循的设计原则。这些原则就像宪法，其他所有模式都是在这些原则指导下的具体"治国方略"。

**1. 单一职责原则（SRP）**

一个类应该只有一个引起它变化的原因。想象一个餐厅，服务员不应该同时负责做菜和收银。每个角色各司其职，餐厅才能运转顺畅。

```typescript
// ❌ 违反SRP：一个类承担了多个职责
class User {
  name: string;
  email: string;

  // 职责1：用户数据管理
  constructor(name: string, email: string) {
    this.name = name;
    this.email = email;
  }

  // 职责2：数据库操作（不应该属于User类）
  saveToDatabase() {
    // 保存用户到数据库
  }

  // 职责3：发送邮件（不应该属于User类）
  sendEmail() {
    // 发送邮件给用户
  }
}

// ✅ 符合SRP：每个类只负责一件事
class User {
  name: string;
  email: string;

  constructor(name: string, email: string) {
    this.name = name;
    this.email = email;
  }
}

// 用户仓储：负责数据库操作
class UserRepository {
  save(user: User) {
    // 保存用户到数据库
  }

  findById(id: string): User {
    // 从数据库查询用户
    return new User('张三', 'zhangsan@example.com');
  }
}

// 邮件服务：负责发送邮件
class EmailService {
  send(user: User, content: string) {
    // 发送邮件给用户
  }
}
```

**2. 开闭原则（OCP）**

软件实体应该对扩展开放，对修改关闭。想象一个万能充电器的发明：我们不需要为每个手机型号发明一个新的充电器，而是让充电器通过可拆卸的转换头来适应不同的手机。

```typescript
// ❌ 违反OCP：每次添加新功能都需要修改原有代码
class AreaCalculator {
  calculateRectangle(width: number, height: number): number {
    return width * height;
  }

  // 添加圆形计算需要修改这个类
  calculateCircle(radius: number): number {
    return Math.PI * radius * radius;
  }
}

// ✅ 符合OCP：通过扩展来添加新功能
interface Shape {
  // 所有形状都必须实现这个方法
  getArea(): number;
}

class Rectangle implements Shape {
  constructor(private width: number, private height: number) {}

  getArea(): number {
    return this.width * this.height;
  }
}

class Circle implements Shape {
  constructor(private radius: number) {}

  getArea(): number {
    return Math.PI * this.radius * this.radius;
  }
}

// 以后添加Triangle只需要创建新类，不需要修改AreaCalculator
class Triangle implements Shape {
  constructor(private base: number, private height: number) {}

  getArea(): number {
    return (this.base * this.height) / 2;
  }
}

class AreaCalculator {
  // 这个方法可以计算任何实现了Shape接口的图形
  calculate(shapes: Shape[]): number {
    return shapes.reduce((sum, shape) => sum + shape.getArea(), 0);
  }
}
```

**3. 里氏替换原则（LSP）**

子类型必须能够替换其基类型。想象一个乐高小人，无论你把它放在什么位置——汽车里、房子里、船上——它都应该能正常工作，因为它符合"小人"这个抽象的所有要求。

```typescript
// ❌ 违反LSP：子类改变了父类的行为契约
class Bird {
  fly(): string {
    return '鸟在飞翔';
  }
}

class Penguin extends Bird {
  // ❌ 企鹅不会飞！这个重写违反了LSP
  fly(): string {
    return '企鹅不会飞';
  }
}

function makeBirdFly(bird: Bird) {
  console.log(bird.fly()); // 对于企鹅，这句话会输出错误信息
}

// ✅ 符合LSP：子类严格遵守父类的行为契约
abstract class Bird {
  abstract move(): string;
}

class Sparrow extends Bird {
  move(): string {
    return '麻雀在飞翔';
  }
}

class Penguin extends Bird {
  move(): string {
    return '企鹅在游泳'; // 企鹅用游泳代替飞翔，但仍然是一种"移动"
  }
}

function letBirdMove(bird: Bird) {
  console.log(bird.move()); // 无论什么鸟，调用move()都能正常工作
}
```

**4. 依赖倒置原则（DIP）**

高层模块不应该依赖低层模块，两者都应该依赖抽象。想象一个灯泡和插座的关系：灯泡不应该"焊接死"在某个特定的插座上，而应该设计成"任何符合标准的插座都能用"。

```typescript
// ❌ 违反DIP：高层模块直接依赖低层模块
class MySQLDatabase {
  connect() {
    return '连接MySQL数据库';
  }

  query(sql: string) {
    return '执行SQL查询';
  }
}

class UserService {
  private database: MySQLDatabase;

  constructor() {
    // 直接依赖具体的数据库实现
    this.database = new MySQLDatabase();
  }

  getUser(id: string) {
    return this.database.query(`SELECT * FROM users WHERE id = ${id}`);
  }
}

// ✅ 符合DIP：依赖抽象（接口），而不是具体实现
interface Database {
  connect(): void;
  query(sql: string): any;
}

class MySQLDatabase implements Database {
  connect(): void {
    console.log('连接MySQL数据库');
  }

  query(sql: string): any {
    return 'MySQL查询结果';
  }
}

class PostgreSQLDatabase implements Database {
  connect(): void {
    console.log('连接PostgreSQL数据库');
  }

  query(sql: string): any {
    return 'PostgreSQL查询结果';
  }
}

class UserService {
  // 依赖抽象，而不是具体实现
  constructor(private database: Database) {}

  getUser(id: string) {
    return this.database.query(`SELECT * FROM users WHERE id = ${id}`);
  }
}

// 使用时可以注入任何实现了Database接口的实现
const mysqlDb = new MySQLDatabase();
const userService = new UserService(mysqlDb);
```

**5. 接口隔离原则（ISP）**

不应该强迫客户依赖它们不使用的接口。想象一个电视遥控器，上面有100个按钮，但对于只看有限频道的老人来说，他只需要10个常用的按钮就够了。接口应该设计得"小巧精干"，而不是"大而全"。

```typescript
// ❌ 违反ISP：强迫用户依赖不需要的方法
interface Worker {
  work(): void;
  eat(): void;
  sleep(): void;
  attendMeeting(): void;
  codeReview(): void;
  // ... 可能有几十个方法
}

class Developer implements Worker {
  work() { /* 写代码 */ }
  eat() { /* 吃饭 */ }
  sleep() { /* 睡觉 */ }
  attendMeeting() { /* 参加会议 */ }
  codeReview() { /* 代码审查 */ }
}

class Robot implements Worker {
  work() { /* 机器人工作 */ }
  eat() { /* 机器人不需要吃饭，但必须实现这个方法 */ }
  sleep() { /* 机器人不需要睡觉，但必须实现这个方法 */ }
  attendMeeting() { /* 机器人参加会议 */ }
  codeReview() { /* 机器人审查代码 */ }
}

// ✅ 符合ISP：将大接口拆分为小而专注的接口
interface Workable {
  work(): void;
}

interface Eatable {
  eat(): void;
}

interface Sleepable {
  sleep(): void;
}

interface Meetingable {
  attendMeeting(): void;
}

class Developer implements Workable, Eatable, Sleepable, Meetingable {
  work() { /* 写代码 */ }
  eat() { /* 吃饭 */ }
  sleep() { /* 睡觉 */ }
  attendMeeting() { /* 参加会议 */ }
}

class Robot implements Workable, Meetingable {
  work() { /* 机器人工作 */ }
  attendMeeting() { /* 机器人参加会议 */ }
  // Robot不需要实现eat和sleep
}
```

**6. 合成/聚合复用原则（CARP）**

优先使用组合（has-a）而不是继承（is-a）。想象一个电脑和它的组件：电脑"有"键盘、"有"鼠标、"有"显示器。通过组合的方式，我们可以随时更换鼠标或键盘，而如果是继承（比如"游戏电脑"继承"电脑"），那更换零件就困难多了。

```typescript
// ❌ 过度使用继承：类层次过深，难以维护
class Animal {
  eat() { }
}

class Mammal extends Animal {
  feedWithMilk() { }
}

class Dog extends Mammal {
  bark() { }
}

class Cat extends Mammal {
  meow() { }
}

// 问题：如果想创造一个"会飞的狗"，该怎么办？
// 需要创建FlyingDog继承自Dog，但会飞的动物不一定都是狗
// 类层次会变得非常复杂

// ✅ 使用组合：更灵活的设计
class Flyable {
  fly() {
    console.log('我可以飞翔');
  }
}

class Swimmable {
  swim() {
    console.log('我可以游泳');
  }
}

class Walkable {
  walk() {
    console.log('我可以走路');
  }
}

class Dog {
  private walkable = new Walkable();
  private swimmable = new Swimmable();

  walk() {
    this.walkable.walk();
  }

  swim() {
    this.swimmable.swim();
  }
}

class FlyingDog {
  private walkable = new Walkable();
  private flyable = new Flyable();
  private swimmable = new Swimmable();

  walk() {
    this.walkable.walk();
  }

  fly() {
    this.flyable.fly();
  }

  swim() {
    this.swimmable.swim();
  }
}
```

## 二、创建型模式（5种）

创建型模式关注的是"如何创建对象"。想象一个汽车工厂，工厂需要根据订单生产不同型号的汽车，但工厂内部有统一的生产流程。创建型模式就是软件世界里的"工厂"，它们定义了创建对象的最佳实践，让对象创建过程既灵活又可控。

### 2.1 单例模式（Singleton Pattern）

**模式定义**：确保一个类只有一个实例，并提供一个全局访问点。

**生活类比**：就像一个国家的总统，全世界只有一个总统实例。无论你从哪里查询，得到的都是同一个人。总统这个"类"在任何一个时间点都只有一个实例在运行。

**UML图**：

```
┌─────────────────────────┐
│     <<singleton>>       │
│       President         │
├─────────────────────────┤
│ - instance: President   │
├─────────────────────────┤
│ - constructor()         │
│ + getInstance(): Presi..│
│ + getName(): string     │
│ + getPolicy(): string   │
└─────────────────────────┘
```

**代码示例**：

```typescript
/**
 * 单例模式：确保类只有一个实例
 *
 * 应用场景：
 * - 数据库连接池（整个应用共享一个连接池）
 * - 全局配置管理器（所有模块读取同一份配置）
 * - 日志记录器（所有模块写入同一个日志文件）
 * - 缓存管理器（全局缓存，所有请求共享）
 */

// TypeScript实现单例模式
class President {
  // 静态属性：用于存储唯一的实例
  // 为什么要用静态属性？因为我们需要一个类级别的变量，而不是实例级别的
  private static instance: President;

  // 私有构造函数：防止外部使用new创建新实例
  // 这就像总统不是随便就能"生成"的，需要通过特定程序产生
  private constructor(
    private name: string,
    private policy: string
  ) {
    console.log(`新总统 ${this.name} 就职`);
  }

  /**
   * 静态工厂方法：获取单例实例
   * 这是单例模式的核心：通过这个方法确保只创建一个实例
   * 如果实例已存在，直接返回；如果不存在，创建新的
   */
  public static getInstance(): President {
    // 为什么要检查null？因为instance初始是undefined
    if (!President.instance) {
      // 首次调用时创建实例
      President.instance = new President('特朗普', '让美国再次伟大');
    }
    return President.instance;
  }

  // 获取总统姓名
  public getName(): string {
    return this.name;
  }

  // 发表政策演讲
  public makeSpeech(): string {
    return `${this.name} 说：${this.policy}`;
  }

  // 更换政策
  public setPolicy(newPolicy: string): void {
    this.policy = newPolicy;
    console.log(`政策已更新为：${this.policy}`);
  }
}

// 测试单例模式
function testSingleton() {
  console.log('--- 单例模式测试 ---');

  // 获取第一个实例
  const president1 = President.getInstance();
  console.log(`第一次获取：${president1.getName()}`);

  // 尝试创建新实例（这在实际中不会创建新实例）
  const president2 = President.getInstance();
  console.log(`第二次获取：${president2.getName()}`);

  // 验证两个变量引用的是同一个实例
  console.log(`是否是同一个实例：${president1 === president2}`);

  // 通过一个实例修改政策
  president1.setPolicy('新的政策方针');

  // 另一个实例也能看到这个变化
  console.log(president2.makeSpeech());
}

testSingleton();
```

**使用场景**：

1. **数据库连接池**：整个应用只需要一个连接池实例，避免频繁创建销毁连接
2. **全局配置管理**：应用的配置信息应该全局唯一
3. **日志记录器**：所有模块使用同一个日志实例，确保日志格式统一
4. **线程池管理**：控制线程数量，避免资源耗尽

**注意事项**：

- 单例模式在多线程环境下需要特别注意（Java/C++），但JavaScript是单线程的，所以没有这个问题
- 过度使用单例可能导致全局状态污染，难以测试
- 单例模式有时被称为"反模式"，因为它引入了全局状态

### 2.2 工厂方法模式（Factory Method Pattern）

**模式定义**：定义一个创建对象的接口，但让子类决定实例化哪个类。

**生活类比**：想象一个手机工厂。工厂有"生产手机"的能力，但具体生产什么型号的手机，由车间主任决定。每个车间可以生产不同型号的手机，但都遵循统一的"手机生产流程"。

**UML图**：

```
         ┌──────────────────┐
         │    <<abstract>>  │
         │  ProductFactory  │
         ├──────────────────┤
         │ +createProduct() │◄─────────── 抽象工厂方法
         └──────────────────┘
                   △
                   │
      ┌────────────┴────────────┐
      │                         │
┌─────┴─────┐            ┌──────┴──────┐
│AppleFactory│            │SamsungFactory│
├───────────┤            ├─────────────┤
│+createProduct()         │+createProduct()
│ : Product               │ : Product
└───────────┘            └─────────────┘
```

**代码示例**：

```typescript
/**
 * 工厂方法模式：定义创建对象的接口，让子类决定实例化哪个类
 *
 * 与简单工厂的区别：
 * - 简单工厂：所有创建逻辑集中在一个类中
 * - 工厂方法：创建逻辑分散到子类中，更符合开闭原则
 *
 * 应用场景：
 * - UI组件库（创建不同主题的按钮、输入框等）
 * - 数据库驱动（创建MySQL、PostgreSQL等连接）
 * - 文档解析器（创建PDF、Word、Excel解析器）
 */

/**
 * 产品接口：所有具体产品必须实现这个接口
 * 这就像所有手机都必须"能打电话"、"能发短信"一样
 */
interface Phone {
  // 获取手机品牌
  getBrand(): string;

  // 获取手机价格
  getPrice(): number;

  // 打电话
  call(number: string): void;

  // 发短信
  sendMessage(number: string, content: string): void;
}

/**
 * 具体产品：iPhone
 * 每个具体产品都实现了产品接口
 */
class iPhone implements Phone {
  constructor(private model: string, private price: number) {}

  getBrand(): string {
    return 'Apple';
  }

  getPrice(): number {
    return this.price;
  }

  call(number: string): void {
    console.log(`使用iPhone ${this.model} 拨打 ${number}`);
  }

  sendMessage(number: string, content: string): void {
    console.log(`iPhone ${this.model} 发送短信给 ${number}: ${content}`);
  }
}

/**
 * 具体产品：Samsung手机
 */
class SamsungPhone implements Phone {
  constructor(private model: string, private price: number) {}

  getBrand(): string {
    return 'Samsung';
  }

  getPrice(): number {
    return this.price;
  }

  call(number: string): void {
    console.log(`使用Samsung ${this.model} 拨打 ${number}`);
  }

  sendMessage(number: string, content: string): void {
    console.log(`Samsung ${this.model} 发送短信给 ${number}: ${content}`);
  }
}

/**
 * 具体产品：小米手机
 */
class XiaomiPhone implements Phone {
  constructor(private model: string, private price: number) {}

  getBrand(): string {
    return 'Xiaomi';
  }

  getPrice(): number {
    return this.price;
  }

  call(number: string): void {
    console.log(`使用小米 ${this.model} 拨打 ${number}`);
  }

  sendMessage(number: string, content: string): void {
    console.log(`小米 ${this.model} 发送短信给 ${number}: ${content}`);
  }
}

/**
 * 抽象工厂类：定义创建产品的接口
 * 为什么不直接创建产品？因为创建过程可能很复杂，需要封装
 */
abstract class PhoneFactory {
  /**
   * 工厂方法：创建手机
   * 为什么用抽象方法？因为具体创建什么手机，由子类决定
   * 为什么这个方法不是私有的？因为需要被子类重写
   */
  abstract createPhone(model: string, price: number): Phone;

  /**
   * 模板方法：定义创建手机的完整流程
   * 这个方法定义了创建手机的步骤顺序
   */
  public assemblePhone(model: string, price: number): Phone {
    console.log('--- 开始组装手机 ---');

    // 步骤1：创建手机
    const phone = this.createPhone(model, price);

    // 步骤2：进行质量检测
    console.log(`品牌：${phone.getBrand()}, 价格：${phone.getPrice()}`);

    // 步骤3：包装出厂
    console.log('--- 手机组装完成 ---');

    return phone;
  }
}

/**
 * Apple工厂：专门创建Apple手机
 */
class AppleFactory extends PhoneFactory {
  // 实现工厂方法：创建Apple手机
  createPhone(model: string, price: number): Phone {
    console.log(`Apple工厂正在创建 ${model}...`);
    return new iPhone(model, price);
  }
}

/**
```

### 2.3 抽象工厂模式（Abstract Factory Pattern）

**模式定义**：提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们具体的类。

**生活类比**：想象一个家具店，它提供"北欧风格家具套餐"和"中式风格家具套餐"。每个套餐包含沙发、茶几、椅子等，但风格统一。你选择哪个套餐，就会得到一套风格统一的所有家具。

**UML图**：

```
┌─────────────────────────────────────────────────────────┐
│                   <<abstract>>                          │
│                  AbstractFactory                        │
├─────────────────────────────────────────────────────────┤
│ +createButton(): Button                                 │
│ +createInput(): Input                                   │
│ +createSelect(): Select                                 │
└─────────────────────────────────────────────────────────┘
              △                    △
              │                    │
┌─────────────┴─────┐    ┌─────────┴────────┐
│  LightThemeFactory│    │ DarkThemeFactory │
├───────────────────┤    ├──────────────────┤
│+createButton()    │    │+createButton()    │
│+createInput()     │    │+createInput()     │
│+createSelect()    │    │+createSelect()    │
└───────────────────┘    └──────────────────┘
```

### 2.4 建造者模式（Builder Pattern）

**模式定义**：将一个复杂对象的构建与它的表示分离，使得同样的构建过程可以创建不同的表示。

**生活类比**：想象去麦当劳点餐。你说"我要一个巨无霸套餐"，服务员不需要知道巨无霸是怎么做的——它由面包、肉饼、蔬菜、酱料等组成，厨师按照固定流程组装。你也可以说"我要一个自定义套餐"，自己选择面包、肉饼、蔬菜等。建造者模式就像这个"点餐系统"，它让你能灵活地构建复杂对象。

**UML图**：

```
┌──────────────┐         ┌──────────────┐
│   Director   │────────▶│   <<i>>      │
├──────────────┤         │    Builder   │
│ +build()     │         ├──────────────┤
└──────────────┘         │+buildPartA() │
                        │+buildPartB() │
                        │+getResult()  │
                        └──────────────┘
                               △
                               │
              ┌────────────────┴────────────────┐
              │                                │
     ┌────────┴───────┐              ┌─────────┴────────┐
     │ ConcreteBuilder│              │ ConcreteBuilder2│
     ├────────────────┤              ├──────────────────┤
     │ +buildPartA() │              │ +buildPartA()    │
     │ +buildPartB() │              │ +buildPartB()    │
     │ +getResult()  │              │ +getResult()     │
     └────────────────┘              └──────────────────┘
```

**代码示例**：

```typescript

 * 建造者模式：将复杂对象的构建与表示分离
 *
 * 与工厂模式的区别：
 * - 工厂模式：一次性创建完整对象
 * - 建造者模式：分步骤构建复杂对象
 *
 * 应用场景：
 * - 构建SQL查询语句（SELECT、FROM、WHERE等逐步构建）
```

### 2.5 原型模式（Prototype Pattern）

**模式定义**：通过复制现有对象（原型）来创建新对象，而无需知道其具体类型。

**生活类比**：想象细胞分裂。一个细胞分裂成两个完全相同的细胞，这两个细胞有相同的DNA（属性），但之后可以独立进化（修改属性不会相互影响）。原型模式就是软件的"细胞分裂"，通过复制现有对象来创建新对象。

**UML图**：

```
┌──────────────────────────────────┐
│      <<interface>>               │
│        Prototype                │
├──────────────────────────────────┤
│ + clone(): Prototype            │
└──────────────────────────────────┘
              △
              │
    ┌─────────┴─────────┐
    │                   │
┌───┴────┐        ┌─────┴────┐
│Concrete│        │ Concrete │
│Proto A │        │ Proto B  │
├────────┤        ├──────────┤
│+clone()│        │+clone()  │
└────────┘        └──────────┘
```

**代码示例**：

```typescript
/**
 * 原型模式：通过复制现有对象创建新对象
 *
 * 与工厂模式的区别：
 * - 工厂模式：通过类创建对象
 * - 原型模式：通过对象复制创建对象
 *
 * 应用场景：
 * - 需要创建多个相似对象
 * - 对象创建成本较高（数据库查询、文件读取等）
 * - 需要避免子类爆炸
 */

/**
 * 原型接口：定义复制方法
 */
interface Prototype<T> {
  /**
   * 克隆方法：复制当前对象
   * 返回T类型，具体由实现类决定
   */
  clone(): T;
}

/**
 * 邮件类：原型具体实现
 * 场景：邮件系统需要快速创建大量相似邮件
 */
class Email implements Prototype<Email> {
  constructor(
    public to: string,
    public from: string,
    public subject: string,
    public body: string,
    public attachments: string[] = []
  ) {}

  // 克隆邮件
  clone(): Email {
    // 使用展开运算符复制所有属性
    // 这是一个浅拷贝，如果attachments有引用类型，需要深拷贝
    const cloned = new Email(
      this.to,
      this.from,
      this.subject,
      this.body,
      [...this.attachments] // 复制数组（浅拷贝）
    );
    return cloned;
  }

  // 显示邮件信息
  display(): void {
    console.log('=== 邮件信息 ===');
    console.log(`收件人: ${this.to}`);
    console.log(`发件人: ${this.from}`);
    console.log(`主题: ${this.subject}`);
    console.log(`正文: ${this.body}`);
    console.log(`附件: ${this.attachments.join(', ')}`);
    console.log('================');
  }
}

/**
 * 深拷贝版本邮件类
 * 当对象包含嵌套引用时，需要深拷贝
 */
class DeepEmail implements Prototype<DeepEmail> {
  constructor(
    public to: string,
    public from: string,
    public subject: string,
    public body: string,
    public attachments: Attachment[] = []
  ) {}

  // 深拷贝：递归复制所有嵌套对象
  clone(): DeepEmail {
    // 递归拷贝attachments数组中的每个对象
    const clonedAttachments = this.attachments.map(
      att => new Attachment(att.name, att.path, att.size)
    );

    return new DeepEmail(
      this.to,
      this.from,
      this.subject,
      this.body,
      clonedAttachments
    );
  }

  display(): void {
    console.log('=== 邮件信息（深拷贝版）===');
    console.log(`收件人: ${this.to}`);
    console.log(`发件人: ${this.from}`);
    console.log(`主题: ${this.subject}`);
    console.log(`正文: ${this.body}`);
    console.log('附件:');
    this.attachments.forEach(att => {
      console.log(`  - ${att.name} (${att.path}, ${att.size}MB)`);
    });
    console.log('================================');
  }
}

/**
 * 附件类

class Attachment {
  constructor(
    public name: string,
    public path: string,
    public size: number
  ) {}

/**
 * 原型管理器：管理常用的原型对象
 * 避免每次都创建新的原型实例
 */
class PrototypeRegistry {
  // 存储原型的Map
  private prototypes: Map<string, Prototype<any>> = new Map();

  /**
   * 注册原型
   * @param key 原型标识
   * @param prototype 原型对象
   */
  register(key: string, prototype: Prototype<any>): void {
    this.prototypes.set(key, prototype);
  }

  /**
   * 获取并克隆原型
   * @param key 原型标识
   * @returns 克隆后的新对象
   */
  get(key: string): Prototype<any> | undefined {
    const prototype = this.prototypes.get(key);
    if (!prototype) {
      console.log(`未找到原型: ${key}`);
      return undefined;
    }
    return prototype.clone();
  }
}
```

## 三、结构型模式（7种）

结构型模式关注的是"如何组合对象"。想象一个乐高城市：有的人用小积木拼小房子，有的人用大积木拼大厦。结构型模式就是告诉你"如何巧妙地把积木组合起来"的技巧。

### 3.1 适配器模式（Adapter Pattern）

**模式定义**：将一个类的接口转换成客户希望的另外一个接口，使得原本由于接口不兼容而不能一起工作的类可以一起工作。

**生活类比**：想象一个出国旅行的人。他带了一个中国标准的充电器，但目的地的插座是英国标准。这时需要一个"电源适配器"，它一边插在英国插座上，一边让中国充电器能正常充电。适配器模式就是这个"转换头"，它让不兼容的接口能够一起工作。

**UML图**：

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│   Client    │─────▶│   Target    │      │   Adaptee   │
│   (客户)     │      │   (目标接口) │      │   (被适配者) │
├─────────────┤      ├─────────────┤      ├─────────────┤
│             │      │+request()   │      │+specificReq()│
└─────────────┘      └─────────────┘      └─────────────┘
                           △                      △
                           │                      │
                           │                      │
                     ┌─────┴─────┐                │
                     │  Adapter  │◄───────────────┘
                     │  (适配器)  │
                     ├───────────┤
                     │+request() │
                     └───────────┘
```

**代码示例**：

```typescript
/**
 * 适配器模式：让不兼容的接口能够一起工作
 *
 * 应用场景：
 * - 集成第三方库：第三方库的接口与项目接口不一致
 * - 兼容旧系统：新系统需要调用旧系统接口
 * - 数据格式转换：XML转JSON、不同数据格式互转
 */

/**
 * 场景：中国支付系统接口
 * 我们的系统使用这个接口
 */
interface ChinaPayment {
  // 人民币支付
  payInRMB(amount: number, currency: string): boolean;
}

/**
 * 场景：美国支付系统接口
 * 第三方库提供的接口
 */
interface USAPayment {
  // 美元支付
  processPayment(amount: number, currency: string): string;
}

/**
 * 我们的中国支付实现
 * 这个类实现了我们的接口
 */
class ChinaPaymentImpl implements ChinaPayment {
  payInRMB(amount: number, currency: string): boolean {
    if (currency !== 'CNY') {
      console.log(`错误：只支持人民币支付，当前货币是 ${currency}`);
      return false;
    }
    console.log(`成功使用人民币支付 ${amount} 元`);
    return true;
  }
}

/**
 * 美国支付系统的实现（第三方库）
 * 这个类实现了美国接口，我们无法修改
 */
class USAPaymentImpl implements USAPayment {
  processPayment(amount: number, currency: string): string {
    if (currency !== 'USD') {
      console.log(`Error: Only USD supported, current is ${currency}`);
      return 'ERROR';
    }
    console.log(`Payment processed: $${amount} USD`);
    return 'SUCCESS';
  }
}

/**
 * 适配器：让美国支付系统适配我们的接口
 * 这个适配器"包裹"了美国支付系统，使其符合我们的接口
 */
class USAPaymentAdapter implements ChinaPayment {
  // 组合：持有美国支付系统的实例
  private usaPayment: USAPayment;

  constructor(usaPayment: USAPayment) {
    this.usaPayment = usaPayment;
  }

  /**
   * 将人民币转换为美元，然后调用美国支付系统
   * 1 CNY ≈ 0.14 USD
   */
  payInRMB(amount: number, currency: string): boolean {
    if (currency === 'CNY') {
      // 转换为美元（简化计算）
      const usdAmount = amount * 0.14;
      console.log(`汇率转换：${amount} CNY → ${usdAmount.toFixed(2)} USD`);

      // 调用美国支付系统
      const result = this.usaPayment.processPayment(usdAmount, 'USD');
      return result === 'SUCCESS';
    }

    // 如果已经是美元，直接调用
    if (currency === 'USD') {
      const result = this.usaPayment.processPayment(amount, 'USD');
      return result === 'SUCCESS';
    }

    console.log(`不支持的货币类型：${currency}`);
    return false;
  }
}

/**
 * 支付管理器：处理各种支付
 */
class PaymentManager {
  // 接受任何实现了ChinaPayment接口的支付方式
  processPayment(payment: ChinaPayment, amount: number, currency: string): void {
    console.log(`处理支付：${amount} ${currency}`);
    const success = payment.payInRMB(amount, currency);
    console.log(`支付结果：${success ? '成功' : '失败'}\n`);
  }
}

// 测试适配器模式
function testAdapter() {
  console.log('=== 适配器模式测试 ===\n');

  const paymentManager = new PaymentManager();

  // 1. 使用我们自己的中国支付系统
  console.log('--- 使用中国支付系统 ---');
  const chinaPayment = new ChinaPaymentImpl();
  paymentManager.processPayment(chinaPayment, 1000, 'CNY');

  // 2. 使用美国支付系统（通过适配器）
  console.log('--- 使用美国支付系统（通过适配器）---');
  const usaPayment = new USAPaymentImpl();
  const adapter = new USAPaymentAdapter(usaPayment);
  paymentManager.processPayment(adapter, 1000, 'CNY');

  // 3. 直接使用美元支付
  console.log('--- 使用美元支付 ---');
  paymentManager.processPayment(adapter, 100, 'USD');
}

testAdapter();
```

**使用场景**：

1. **集成旧系统**：新系统通过适配器调用旧系统API
2. **第三方库集成**：统一不同第三方库的接口
3. **数据格式转换**：XML解析器适配JSON接口
4. **插件系统**：为主程序提供统一接口，底层调用具体插件实现

**注意事项**：

- 适配器不是改变原有类的行为，而是转换接口
- 可以使用对象组合（推荐）或类继承实现适配器
- 如果能直接改造原有接口使其兼容，就不需要适配器

### 3.2 桥接模式（Bridge Pattern）

**模式定义**：将抽象部分与它的实现部分分离，使它们都可以独立地变化。

**生活类比**：想象遥控器和电视机的关系。遥控器是"抽象"，电视机是"实现"。有各种类型的遥控器（简单遥控、智能遥控），也有各种类型的电视机（LCD、LED、OLED）。桥接模式让遥控器和电视机可以独立变化——你可以用一个遥控器控制所有类型的电视机，反之亦然。

**UML图**：

```
        ┌─────────────┐
        │ Abstraction │
        │   (抽象)     │
        ├─────────────┤
        │ +operation()│
        └──────┬──────┘
               │
               │ imp
               ▼
        ┌─────────────┐
        │Implementor  │
        │ (实现者接口) │
        ├─────────────┤
        │+operationImpl()│
        └─────────────┘
               △
               │
    ┌──────────┴──────────┐
    │                     │
┌───┴─────┐          ┌────┴────┐
│ConcreteA│          │ConcreteB│
│Impl     │          │Impl     │
└─────────┘          └─────────┘
```

**代码示例**：

```typescript
/**
 * 桥接模式：将抽象部分与实现部分分离，各自独立变化
 *
 * 与适配器模式的区别：
 * - 适配器：改变已有接口，使其兼容
 * - 桥接：预防性地分离抽象和实现，避免类爆炸
 *
 * 应用场景：
 * - 不同平台的消息通知系统（短信、邮件、推送 × 紧急、普通、促销）
 * - 不同设备的图形绘制（Windows/Mac/Linux × 2D/3D/矢量）
 * - 数据库驱动（MySQL/PostgreSQL × 连接池/事务/日志）
 */

/**
 * 实现者接口：定义消息发送的具体实现
 * 这个接口是"发送通道"的抽象
 */
interface MessageSender {
  // 发送消息
  send(message: string): boolean;

  // 获取发送器名称
  getName(): string;
}

/**
 * 具体实现者：邮件发送器
 */
class EmailSender implements MessageSender {
  send(message: string): boolean {
    console.log(`[Email] 发送邮件: ${message}`);
    return true;
  }

  getName(): string {
    return '邮件';
  }
}

/**
 * 具体实现者：短信发送器
 */
class SMSSender implements MessageSender {
  send(message: string): boolean {
    console.log(`[SMS] 发送短信: ${message}`);
    return true;
  }

  getName(): string {
    return '短信';
  }
}

/**
 * 具体实现者：推送通知发送器
 */
class PushSender implements MessageSender {
  send(message: string): boolean {
    console.log(`[Push] 发送推送: ${message}`);
    return true;
  }

  getName(): string {
    return '推送通知';
  }
}

/**
 * 抽象类：消息通知器
 * 这个类定义了通知器的"抽象"，但具体发送逻辑委托给MessageSender
 * 使用protected是因为子类需要访问这个属性
 */
abstract class Notifier {
  // 组合实现者：使用protected让子类可以访问
  protected sender: MessageSender;

  constructor(sender: MessageSender) {
    this.sender = sender;
  }

  /**
   * 抽象方法：发送通知（具体逻辑由子类实现）
   */
  abstract notify(message: string): void;

  /**
   * 模板方法：定义发送通知的流程
   */
  public sendNotification(message: string): void {
    console.log(`=== 通过${this.sender.getName()}发送通知 ===`);
    this.notify(message);
    console.log('发送完成\n');
  }
}

/**
 * 具体抽象：普通通知器
 */
class NormalNotifier extends Notifier {
  constructor(sender: MessageSender) {
    super(sender);
  }

  notify(message: string): void {
    // 普通通知直接发送
    this.sender.send(message);
  }
}

/**
 * 具体抽象：紧急通知器
 */
class UrgentNotifier extends Notifier {
  constructor(sender: MessageSender) {
    super(sender);
  }

  notify(message: string): void {
    // 紧急通知添加前缀并重试
    const urgentMessage = `【紧急】${message}`;
    const success = this.sender.send(urgentMessage);

    if (!success) {
      console.log('发送失败，尝试重新发送...');
      this.sender.send(urgentMessage);
    }
  }
}

/**
 * 具体抽象：营销通知器
 */
class MarketingNotifier extends Notifier {
  constructor(sender: MessageSender) {
    super(sender);
  }

  notify(message: string): void {
    // 营销通知添加优惠信息
    const marketingMessage = `${message}\n限时优惠，点击领取！`;
    this.sender.send(marketingMessage);
  }
}

// 测试桥接模式
function testBridge() {

  const emailSender = new EmailSender();

  const pushSender = new PushSender();

  const normalSMS = new NormalNotifier(smsSender);
  normalSMS.sendNotification('您的订单已发货');

  // 2. 紧急通知 × 不同发送器
  console.log('--- 紧急通知 ---');
  const urgentEmail = new UrgentNotifier(emailSender);
  urgentEmail.sendNotification('您的账户存在安全风险，请立即处理');

  const urgentPush = new UrgentNotifier(pushSender);
  urgentPush.sendNotification('服务器异常，请立即检查');

  // 3. 营销通知 × 不同发送器
  console.log('--- 营销通知 ---');
```

1. **消息通知系统**：通知类型（普通/紧急/营销）× 发送渠道（邮件/短信/推送）
2. **跨平台图形系统**：图形类型（2D/3D）× 平台（Windows/Mac/Linux）
3. **数据库抽象层**：操作类型（CRUD）× 数据库类型（MySQL/PostgreSQL/MongoDB）
4. **设备驱动**：设备类型（打印机/扫描仪）× 操作系统（Windows/Mac/Linux）

**注意事项**：

- 桥接模式强调"独立变化"，适用于类数量可能爆炸的场景
- 抽象和实现之间是组合关系，而不是继承关系
- 可以通过抽象部分切换实现部分

### 3.3 组合模式（Composite Pattern）

**模式定义**：将对象组合成树形结构以表示"部分-整体"的层次结构，使得用户对单个对象和组合对象的使用具有一致性。

**生活类比**：想象一个文件系统。文件夹可以包含文件，也可以包含子文件夹。文件和文件夹有共同的操作：可以打开、可以删除、可以复制。但文件夹还有一些特殊操作：可以添加内容、可以列出内容。组合模式让我们能以相同的方式处理"单个对象（文件）"和"组合对象（文件夹）"。

**UML图**：

```
         ┌───────────────┐
         │   Component   │
         │   (组件接口)   │
         ├───────────────┤
         │+operation()   │
         │+add()         │
         │+remove()      │
         │+getChild()    │
         └───────┬───────┘
                 △
       ┌─────────┴─────────┐
       │                   │
┌──────┴─────┐      ┌──────┴──────┐
│   Leaf     │      │  Composite  │
│   (叶子)    │      │   (组合)     │
├────────────┤      ├─────────────┤
│+operation()│      │+operation() │
└────────────┘      │+add()       │
                    │+remove()    │
                    │+getChild()  │
                    └─────────────┘
```

**代码示例**：

```typescript
/**
 * 组合模式：以树形结构组织对象，一致性处理单个对象和组合对象
 *
 * 与命令模式的区别：
 * - 命令模式：封装请求为对象
 * - 组合模式：组织对象为树形结构
 *
 * 应用场景：
 * - 文件系统：文件和文件夹
 * - 组织架构：公司、部门、员工
 * - UI组件：容器组件和原子组件
 * - 菜单系统：菜单项和子菜单
 */

/**
 * 组件接口：定义文件系统和组织架构的共同操作
 */
interface FileSystemItem {
  // 获取名称
  getName(): string;

  // 获取大小（文件返回字节数，文件夹返回所有内容总大小）
  getSize(): number;

  // 打印结构（带缩进）
  print(indent: string): void;
}

/**
 * 文件类：叶子节点
 * 文件是最小的计算单元，不能包含其他元素
 */
class File implements FileSystemItem {
  constructor(
    private name: string,     // 文件名
    private size: number,      // 文件大小（字节）
    private extension: string  // 扩展名
  ) {}

  getName(): string {
    return `${this.name}.${this.extension}`;
  }

  getSize(): number {
    return this.size;
  }

  print(indent: string): void {
    // 打印文件信息
    console.log(`${indent}📄 ${this.getName()} (${this.formatSize()})`);
  }

  // 格式化文件大小显示
  private formatSize(): string {
    if (this.size < 1024) {
      return `${this.size} B`;
    } else if (this.size < 1024 * 1024) {
      return `${(this.size / 1024).toFixed(2)} KB`;
    } else {
      return `${(this.size / (1024 * 1024)).toFixed(2)} MB`;
    }
  }
}

/**
 * 文件夹类：组合节点
 * 文件夹可以包含文件和子文件夹
 */
class Folder implements FileSystemItem {
  // 存储文件夹中的所有内容（文件和子文件夹）
  private children: FileSystemItem[] = [];

  constructor(private name: string) {}

  getName(): string {
    return this.name;
  }

  // 计算文件夹总大小：所有子项大小之和
  getSize(): number {
    return this.children.reduce((total, child) => total + child.getSize(), 0);
  }

  /**
   * 添加文件或文件夹
   * @param item 要添加的项
   */
  add(item: FileSystemItem): void {
    this.children.push(item);
    console.log(`添加 ${item.getName()} 到文件夹 ${this.name}`);
  }

  /**
   * 移除文件或文件夹
   * @param item 要移除的项
   */
  remove(item: FileSystemItem): void {
    const index = this.children.indexOf(item);
    if (index > -1) {
      this.children.splice(index, 1);
      console.log(`从文件夹 ${this.name} 移除 ${item.getName()}`);
    }
  }

  /**
   * 获取子项
   */
  getChild(index: number): FileSystemItem | undefined {
    return this.children[index];
  }

  /**
   * 获取子项数量
   */
  getChildCount(): number {
    return this.children.length;
  }

  print(indent: string): void {
    // 打印文件夹头
    console.log(`${indent}📁 ${this.getName()} (${this.formatSize()})`);

    // 递归打印所有子项（增加缩进）
    const childIndent = indent + '    ';
    for (const child of this.children) {
      child.print(childIndent);
    }
  }

  // 格式化大小显示
  private formatSize(): string {
    const size = this.getSize();
    if (size < 1024) {
      return `${size} B`;
    } else if (size < 1024 * 1024) {
      return `${(size / 1024).toFixed(2)} KB`;
    } else if (size < 1024 * 1024 * 1024) {
      return `${(size / (1024 * 1024)).toFixed(2)} MB`;
    } else {
      return `${(size / (1024 * 1024 * 1024)).toFixed(2)} GB`;
    }
  }
}
```

### 3.4 装饰器模式（Decorator Pattern）

**模式定义**：动态地给对象添加一些额外的职责，就增加功能来说，装饰器模式比继承更加灵活。

**生活类比**：想象给手机贴膜和装手机壳。手机本身可以打电话、发短信。贴膜后增加了防刮功能，装壳后增加了防摔功能。这些装饰是可叠加的，你可以同时贴膜+装壳，也可以随时拆卸。装饰器模式就是这种"可以叠加的装饰"。

**UML图**：

```
┌─────────────┐         ┌─────────────┐
│  Component  │◄────────│  Decorator   │
│  (组件接口)   │         │  (装饰器基类) │
├─────────────┤         ├─────────────┤
│+operation()  │         │-wrapped:Comp │
├─────────────┤         │+operation()  │
└─────────────┘         └──────────────┘
       △                        △
       │                        │
       │              ┌─────────┴─────────┐
       │              │                   │
┌──────┴─────┐  ┌─────┴─────┐       ┌─────┴─────┐
│ConcreteComp│  │DecoratorA │       │DecoratorB │
├────────────┤  ├───────────┤       ├───────────┤
│+operation()│  │+operation()│       │+operation()│
└────────────┘  └───────────┘       └───────────┘
```

**代码示例**：

```typescript
/**
 * 装饰器模式：动态给对象添加功能，比继承更灵活
 *
 * 与继承的区别：
 * - 继承：在编译时静态添加功能
 * - 装饰器：在运行时动态添加功能
 *
 * 应用场景：
 * - IO流：FileInputStream + BufferedInputStream + DataInputStream
 * - Web中间件：Logger + Auth + CORS 层层包装
 * - UI组件：基础组件 + 边框 + 阴影 + 动画
 */

/**
 * 组件接口：定义被装饰对象的基本行为
 */
interface Coffee {
  // 获取咖啡描述
  getDescription(): string;

  // 获取咖啡价格
  getCost(): number;
}

/**
 * 具体组件：基础咖啡
 * 这是被装饰的原始对象
 */
class SimpleCoffee implements Coffee {
  getDescription(): string {
    return '浓缩咖啡';
  }

  getCost(): number {
    return 25; // 基础咖啡25元
  }
}

/**
 * 装饰器基类：所有装饰器的父类
 * 实现了Coffee接口，持有一个Coffee实例（被装饰的对象）
 */
abstract class CoffeeDecorator implements Coffee {
  // 组合：持有被装饰的咖啡对象
  protected coffee: Coffee;

  constructor(coffee: Coffee) {
    this.coffee = coffee;
  }

  // 默认实现：委托给被装饰的对象
  getDescription(): string {
    return this.coffee.getDescription();
  }

  getCost(): number {
    return this.coffee.getCost();
  }
}

/**
 * 具体装饰器：加牛奶
 */
class MilkDecorator extends CoffeeDecorator {
  getDescription(): string {
    // 在原有描述基础上添加牛奶
    return `${this.coffee.getDescription()} + 牛奶`;
  }

  getCost(): number {
    // 在原有价格基础上加牛奶价格
    return this.coffee.getCost() + 5;
  }
}

/**
 * 具体装饰器：加糖
 */
class SugarDecorator extends CoffeeDecorator {
  getDescription(): string {
    return `${this.coffee.getDescription()} + 糖`;
  }

  getCost(): number {
    return this.coffee.getCost() + 2;
  }
}

/**
 * 具体装饰器：加焦糖
 */
class CaramelDecorator extends CoffeeDecorator {
  getDescription(): string {
    return `${this.coffee.getDescription()} + 焦糖`;
  }

  getCost(): number {
    return this.coffee.getCost() + 8;
  }
}

/**
 * 具体装饰器：加奶油
 */
class WhipDecorator extends CoffeeDecorator {
  getDescription(): string {
    return `${this.coffee.getDescription()} + 鲜奶油`;
  }

  getCost(): number {
    return this.coffee.getCost() + 10;
  }
}

/**
 * 具体装饰器：加大杯
 */
class LargeSizeDecorator extends CoffeeDecorator {
  getDescription(): string {
    return `${this.coffee.getDescription()}（大杯）`;
  }

  getCost(): number {
    return this.coffee.getCost() + 8;
  }
}

  console.log(`描述: ${coffee.getDescription()}`);

  // 4. 创建一杯复杂的咖啡：大杯 + 焦糖 + 奶油 + 牛奶
  console.log('--- 豪华咖啡（大杯+焦糖+奶油+牛奶）---');
  let luxuryCoffee: Coffee = new SimpleCoffee();
  luxuryCoffee = new LargeSizeDecorator(luxuryCoffee);
```

**注意事项**：

### 3.5 外观模式（Facade Pattern）

**模式定义**：为子系统中的一组接口提供一个一致的界面，定义一个高层接口，使得这一子系统更加容易使用。

**生活类比**：想象汽车仪表盘。汽车内部有发动机、变速箱、刹车系统、转向系统等复杂组件，但司机只需要通过方向盘、刹车踏板、油门踏板这些简单的"界面"来操作。外观模式就是这个仪表盘，它隐藏了内部复杂性，提供了一个简单的操作界面。

**UML图**：

```
┌─────────────┐
│   Client    │
│   (客户)     │
└──────┬──────┘
       │ 使用外观
       ▼
┌─────────────┐         ┌──────────────────────────┐
│   Facade    │────────▶│      Subsystem           │
│   (外观)     │         ├──────────────────────────┤
├─────────────┤         │ - subsystemClassA        │
│ +operation()│         │ - subsystemClassB        │
└─────────────┘         │ - subsystemClassC        │
                        │ + subsystemOperationA() │
                        │ + subsystemOperationB() │
                        │ + subsystemOperationC() │
                        └──────────────────────────┘
```

**代码示例**：

```typescript
/**
 * 外观模式：为子系统提供统一的高层接口
 *
 * 与适配器模式的区别：
 * - 适配器：改变接口使其兼容
 * - 外观：简化接口，隐藏复杂性
 *
 * 应用场景：
 * - 电脑启动：BIOS/UEFI隐藏了所有硬件初始化复杂性
 * - 家庭影院：一键开启DVD、功放、电视
 * - 编译系统：一条命令完成词法分析、语法分析、代码生成
 * - 订单处理：下单 → 库存检查 → 支付 → 发货
 */

/**
 * 子系统类：订单系统
 */
class OrderSystem {
  // 创建订单
  createOrder(items: string[]): string {
    console.log('OrderSystem: 创建订单...');
    const orderId = 'ORD-' + Date.now();
    console.log(`OrderSystem: 订单 ${orderId} 已创建，包含 ${items.length} 个商品`);
    return orderId;
  }

  // 取消订单
  cancelOrder(orderId: string): void {
    console.log(`OrderSystem: 取消订单 ${orderId}`);
  }
}

/**
 * 子系统类：库存系统
 */
class InventorySystem {
  // 检查库存
  checkStock(itemId: string): boolean {
    console.log(`InventorySystem: 检查商品 ${itemId} 库存`);
    return true; // 假设有库存
  }

  // 锁定库存
  reserveStock(itemId: string, quantity: number): boolean {
    console.log(`InventorySystem: 锁定商品 ${itemId}，数量 ${quantity}`);
    return true;

  }
}

/**
 * 子系统类：支付系统
 */
class PaymentSystem {
  // 处理支付

  track(trackingId: string): string {

/**
 * 子系统类：通知系统
 */
class NotificationSystem {

    console.log(`NotificationSystem: 发送邮件给 ${to}: ${subject}`);

/**
 * 外观类：订单处理Facade
 * 简化整个订单处理流程，对外提供简单的接口
 */
class OrderFacade {
  // 组合所有子系统
  private orderSystem: OrderSystem;
  private inventorySystem: InventorySystem;
  private paymentSystem: PaymentSystem;
  private shippingSystem: ShippingSystem;
  private notificationSystem: NotificationSystem;

  constructor() {
    this.orderSystem = new OrderSystem();
    this.inventorySystem = new InventorySystem();
    this.paymentSystem = new PaymentSystem();
    this.shippingSystem = new ShippingSystem();
    this.notificationSystem = new NotificationSystem();
  }

  /**
   * 简化的一键下单方法
   * 客户只需要调用这一个方法，就能完成整个下单流程
   */
  placeOrder(
    customerId: string,
    items: string[],
    address: string,
    paymentMethod: string
  ): { success: boolean; orderId?: string; message: string } {
    console.log('\n========== 开始处理订单 ==========\n');

    // 步骤1：创建订单
    const orderId = this.orderSystem.createOrder(items);

    // 步骤2：检查并锁定库存
    let allInStock = true;
    for (const item of items) {
      if (!this.inventorySystem.checkStock(item)) {
        allInStock = false;
        break;
      }
      this.inventorySystem.reserveStock(item, 1);
    }

    if (!allInStock) {
      console.log('库存不足，订单取消');
      return { success: false, message: '库存不足' };
    }

    // 步骤3：处理支付
    const totalAmount = items.length * 100; // 简化计算
    const paymentId = this.paymentSystem.processPayment(orderId, totalAmount);

    // 步骤4：发货
    const trackingId = this.shippingSystem.ship(orderId, address);

    // 步骤5：发送通知
    this.notificationSystem.sendEmail(
      `${customerId}@example.com`,
      '订单确认',
      `您的订单 ${orderId} 已确认，正在配送中`
    );

    console.log('\n========== 订单处理完成 ==========');
    console.log(`订单号: ${orderId}`);
    console.log(`支付号: ${paymentId}`);
    console.log(`快递号: ${trackingId}\n`);

    return {
      success: true,
      orderId,
      message: '订单处理成功'
    };
  }

  /**
   * 简化的一键取消订单方法
   */
  cancelOrder(orderId: string): { success: boolean; message: string } {
    console.log('\n========== 开始取消订单 ==========\n');

    // 步骤1：取消订单
    this.orderSystem.cancelOrder(orderId);

    // 步骤2：释放库存
    this.inventorySystem.releaseStock('ITEM-001', 1);

    // 步骤3：退款
    this.paymentSystem.refund('PAY-123');

    // 步骤4：发送通知
    this.notificationSystem.sendSMS('13800138000', `您的订单 ${orderId} 已取消`);

    console.log('\n========== 订单取消完成 ==========\n');

    return { success: true, message: '订单已取消' };
  }
}
```

### 3.6 享元模式（Flyweight Pattern）

**模式定义**：运用共享技术有效地支持大量细粒度的对象。

**生活类比**：想象一个网游服务器。成千上万的玩家同时在线，但如果每个玩家都创建一个独立的角色模型实例，那内存很快就爆了。实际上，所有同职业的玩家可以共享同一个"角色模板"（模型、骨骼动画），每个玩家只需要存储自己的"独特数据"（位置、装备、等级）。享元模式就是这个"角色模板共享"的思想。

**UML图**：

```
┌─────────────────────────────────────────────────┐
│                 Client                          │
│         (使用享元对象的客户端)                     │
└─────────────────────┬───────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────┐
│               FlyweightFactory                  │
│                 (享元工厂)                       │
├─────────────────────────────────────────────────┤
│ - flyweights: Map                               │
├─────────────────────────────────────────────────┤
│ + getFlyweight(key): Flyweight                  │
│ + listAllFlyweights(): void                     │
└─────────────────────┬───────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────┐
│              <<interface>>                      │
│                Flyweight                        │
├─────────────────────────────────────────────────┤
│ + operation(extrinsicState): void              │
└─────────────────────┬───────────────────────────┘
                      │
                      ▼
         ┌────────────────────────┐
         │    ConcreteFlyweight   │
         │    (具体享元)           │
         ├────────────────────────┤
         │ - intrinsicState       │
         │ (内部状态，可共享)       │
         ├────────────────────────┤
         │ + operation(extrinsic) │
         └────────────────────────┘
```

**代码示例**：

```typescript
/**
 * 享元模式：通过共享大量细粒度对象来节省内存
 *
 * 与对象池的区别：
 * - 对象池：复用整个对象，减少创建销毁开销
 * - 享元模式：共享对象的内部状态，减少内存占用
 *
 * 应用场景：
 * - 文本编辑器：大量字符共享字体样式对象
 * - 游戏开发：大量子弹共享纹理，同职业角色共享模型
 * - 图形系统：大量图形共享样式
 * - 树形结构：DOM节点共享样式对象
 */

/**
 * 享元接口：定义共享对象的行为
 */
interface TreeType {
  // 绘制树
  draw(canvas: string): void;
}

/**
 * 具体享元：树的类型（内部状态）
 * 这些是可以共享的通用数据
 */
class TreeTypeImpl implements TreeType {
  constructor(
    private name: string,        // 树种名称
    private color: string,       // 颜色
    private texture: string       // 纹理
  ) {
    console.log(`创建树类型: ${name} (${color})`);
  }

  draw(canvas: string): void {
    console.log(`[${canvas}] 绘制 ${this.name} 树，颜色=${this.color}`);
  }

  getName(): string {
    return this.name;
  }
}

/**
 * 享元工厂：管理享元对象
 * 负责创建和缓存享元对象
 */
class TreeFactory {
  // 缓存所有树类型（享元池）
  private static treeTypes: Map<string, TreeTypeImpl> = new Map();

  /**
   * 获取树类型（如果不存在则创建）
   */
  static getTreeType(name: string, color: string): TreeTypeImpl {
    // 生成缓存键
    const key = `${name}-${color}`;

    // 检查缓存
    if (!this.treeTypes.has(key)) {
      // 创建新类型并存入缓存
      const texture = this.generateTexture(name);
      this.treeTypes.set(key, new TreeTypeImpl(name, color, texture));
    }

    return this.treeTypes.get(key)!;
  }

  /**
   * 生成纹理（实际应用中可能是加载图片等操作）
   */
  private static generateTexture(name: string): string {
    return `texture_${name.toLowerCase()}.png`;
  }

  /**
   * 获取已缓存的类型数量
   */
  static getCachedTypeCount(): number {
    return this.treeTypes.size;
  }

  /**
   * 列出所有缓存的树类型
   */
  static listCachedTypes(): void {
    console.log('--- 已缓存的树类型 ---');
    for (const [key, type] of this.treeTypes) {
      console.log(`  ${key}: ${type.getName()}`);
    }
  }
}

/**
 * 树（场景对象）：使用享元
 * 外部状态（位置）由每个树自己存储
 */
class Tree {
  constructor(
    private x: number,              // X坐标（外部状态）
    private y: number,                // Y坐标（外部状态）
    private treeType: TreeType        // 树类型（内部状态，共享）
  ) {}

  // 绘制这棵树
  draw(canvas: string): void {
    console.log(`绘制位置: (${this.x}, ${this.y})`);
    this.treeType.draw(canvas);
  }

  getPosition(): string {
    return `(${this.x}, ${this.y})`;
  }
}

/**
 * 森林：管理所有树
 */
class Forest {
  // 存储场景中的所有树
  private trees: Tree[] = [];

  /**
   * 种植一棵树
   */
  plantTree(x: number, y: number, name: string, color: string): void {
    // 从工厂获取树类型（内部状态，共享）
    const treeType = TreeFactory.getTreeType(name, color);

    // 创建树实例（外部状态，独立）
    const tree = new Tree(x, y, treeType);
    this.trees.push(tree);

    console.log(`种植了 ${name} 树在位置 (${x}, ${y})`);
  }

  /**
   * 在画布上绘制所有树
   */
  draw(canvas: string): void {
    console.log(`\n=== 在 ${canvas} 上绘制森林 ===`);
    for (const tree of this.trees) {
      tree.draw(canvas);
    }
  }

  /**
   * 获取树的总数
   */
  getTreeCount(): number {
    return this.trees.length;
  }
}

// 测试享元模式
function testFlyweight() {
  console.log('=== 享元模式测试：森林种植系统 ===\n');

  // 创建森林
  const forest = new Forest();

  console.log('--- 种植大量树木 ---\n');

  // 模拟种植10000棵树
  // 如果不使用享元模式，每棵树都存储完整的数据

  // 种植松树（绿色）
  for (let i = 0; i < 5000; i++) {
    forest.plantTree(

    );
  }

  // 种植橡树（绿色）
  for (let i = 0; i < 3000; i++) {
    forest.plantTree(

      '橡树',
      '翠绿色'
    );
  }

  // 种植枫树（红色）
  for (let i = 0; i < 2000; i++) {
    forest.plantTree(
      Math.floor(Math.random() * 1000),
      Math.floor(Math.random() * 1000),

  }

  console.log('\n--- 统计信息 ---');

  // 显示缓存的树类型

  console.log('\n--- 内存优化效果 ---');

```

**使用场景**：

1. **文本编辑器**：大量字符共享字符样式对象
2. **游戏开发**：大量同类型子弹/怪物共享纹理和模型
3. **图形系统**：大量图形共享绘制样式
4. **浏览器DOM**：多个同类元素共享事件处理器
5. **数据库连接池**：复用连接对象

**注意事项**：

- 享元模式适用于对象数量大、对象可共享内部状态
- 内部状态（可共享）与外部状态（独立）需要分离
- 线程安全问题：享元对象通常需要是线程安全的

### 3.7 代理模式（Proxy Pattern）

**模式定义**：为其他对象提供一种代理以控制对这个对象的访问。

**生活类比**：想象你去银行存钱。你不直接把现金交给柜员，而是把钱放进"信封"里，通过柜台小姐传递。柜台小姐就是"代理"，她控制着对"真正金库"的访问。你可以把钱给她，她帮你存；她可以检查钱的数量；她还可以拒绝处理可疑的存款请求。代理模式就是这种"中间人"模式。

**UML图**：

```
┌─────────────┐       ┌─────────────┐
│   Client    │       │   Subject   │
│   (客户)     │─────▶│   (主题)    │
├─────────────┤       ├─────────────┤
└─────────────┘       │+request()  │
                      └──────┬──────┘
                             △
              ┌──────────────┴──────────────┐
              │                             │
       ┌──────┴──────┐              ┌───────┴───────┐
       │   Proxy    │              │ RealSubject   │
       │   (代理)    │              │ (真实主题)    │
       ├─────────────┤              ├───────────────┤
       │-realSubject │─────────────▶│+request()    │
       │+request()   │              └───────────────┘
       └─────────────┘
```

**代码示例**：

```typescript
/**
 * 代理模式：为对象提供代理，控制对原对象的访问
 *
 * 与装饰器模式的区别：
 * - 装饰器：添加功能
 * - 代理：控制访问，不一定添加功能
 *
 * 应用场景：
 * - 虚拟代理：延迟加载大图片、大文件
 * - 保护代理：权限验证、访问控制
 * - 远程代理：访问远程服务（RPC）
 * - 智能引用：访问对象时执行额外操作（日志、缓存）
 */

/**
 * 主题接口：定义代理和真实对象共同实现的接口
 */
interface Image {
  // 显示图片
  display(): void;

  // 获取图片信息
  getInfo(): string;
}

/**
 * 真实主题：大图片（实际加载图片文件）
 */
class RealImage implements Image {
  private filename: string;
  private data: string | null = null;

  constructor(filename: string) {
    this.filename = filename;
    // 模拟：创建RealImage时并不立即加载数据
    console.log(`RealImage: 构造函数收到文件名 ${filename}`);
  }

  /**
   * 真正加载图片（可能很慢）
   */
  private loadFromDisk(): void {
    console.log(`RealImage: 从磁盘加载图片 ${this.filename}...`);
    // 模拟加载耗时
    this.data = `[图片数据: ${this.filename}]`;
    console.log(`RealImage: 加载完成，数据大小 ${this.data.length} 字节`);
  }

  display(): void {
    // 懒加载：首次显示时才真正加载
    if (!this.data) {
      this.loadFromDisk();
    }
    console.log(`RealImage: 显示图片 ${this.filename}`);
  }

  getInfo(): string {
    return `图片: ${this.filename}, 状态: ${this.data ? '已加载' : '未加载'}`;
  }
}

/**
 * 虚拟代理：图片代理（延迟加载）
 * 在真实图片加载之前，使用代理占位
 */
class ImageProxy implements Image {
  private filename: string;
  private realImage: RealImage | null = null;

  constructor(filename: string) {
    this.filename = filename;
    console.log(`ImageProxy: 创建代理 ${filename}`);
  }

  /**
   * 代理的display：延迟加载真实图片
   */
  display(): void {
    console.log(`ImageProxy: 调用display for ${this.filename}`);

    // 首次显示时才创建并加载真实图片
    if (!this.realImage) {
      console.log(`ImageProxy: 首次显示，创建真实图片对象`);
      this.realImage = new RealImage(this.filename);
    }

    // 委托给真实图片
    this.realImage.display();
  }

  getInfo(): string {
    if (this.realImage) {
      return this.realImage.getInfo();
    }
    return `图片: ${this.filename}, 状态: 代理中(未加载)`;
  }
}

/**
 * 保护代理：验证访问权限
 */
class SecureImageProxy implements Image {
  private filename: string;
  private realImage: RealImage | null = null;
  private userRole: string;

  constructor(filename: string, userRole: string) {
    this.filename = filename;
    this.userRole = userRole;
  }

  display(): void {
    console.log(`SecureImageProxy: 用户角色=${this.userRole} 尝试访问 ${this.filename}`);

    // 访问控制：只有管理员才能查看
    if (this.userRole !== 'admin') {
      console.log(`SecureImageProxy: 访问被拒绝！只有admin角色可以查看`);
      return;
    }

    // 验证通过，创建真实图片
    if (!this.realImage) {
      this.realImage = new RealImage(this.filename);
    }

    this.realImage.display();
  }

  getInfo(): string {
    if (this.userRole !== 'admin') {
      return `[权限不足，无法查看 ${this.filename} 的信息]`;
    }
    if (this.realImage) {
      return this.realImage.getInfo();
    }
    return `图片: ${this.filename}, 状态: 未加载`;
  }
}

/**
 * 日志代理：记录访问日志
 */
class LoggingImageProxy implements Image {
  private filename: string;
  private realImage: RealImage | null = null;

  constructor(filename: string) {
    this.filename = filename;
  }

  display(): void {
    console.log(`[LOG] 访问图片: ${this.filename}`);

    if (!this.realImage) {
      this.realImage = new RealImage(this.filename);
    }

    const startTime = Date.now();
    this.realImage.display();
    const endTime = Date.now();

    console.log(`[LOG] 显示耗时: ${endTime - startTime}ms`);
  }

  getInfo(): string {
    console.log(`[LOG] 获取图片信息: ${this.filename}`);

    if (!this.realImage) {
      return `图片: ${this.filename}, 状态: 未加载`;
    }

    return this.realImage.getInfo();
  }
}
```

**使用场景**：

1. **图片懒加载**：网页中的大图，先用占位图显示
2. **访问控制**：API接口的权限验证
3. **远程服务调用**：RPC、GraphQL的客户端代理

## 四、行为型模式（11种）

行为型模式关注的是"对象之间的协作"。想象一个交响乐团，有指挥家、小提琴手、大提琴手、鼓手等。行为型模式就是告诉你"这些乐手如何配合演奏"的艺术——谁先演奏、谁后演奏、谁听谁的。

### 4.1 策略模式（Strategy Pattern）

**模式定义**：定义一系列算法，把它们一个个封装起来，并且使它们可以相互替换。

**生活类比**：想象你去北京旅游。你可以选择不同的交通方式：坐地铁、打车、骑单车、步行。每种方式都能让你到达目的地，但各有优缺点——地铁快但要换乘，打车快但贵，单车便宜但慢。你可以根据当天的情况（天气、时间、预算）选择最合适的策略。策略模式就是让你能"换乘"不同算法。

**UML图**：

```
┌─────────────┐       ┌─────────────────┐
│   Context   │─────▶│   <<interface>> │
│   (上下文)   │      │     Strategy   │
├─────────────┤      ├─────────────────┤
│ - strategy  │      │ + execute()     │
├─────────────┤      └─────────────────┘
│ +execute()  │              △
└─────────────┘              │
                      ┌──────┴──────┐
                      │             │
              ┌───────┴─────┐ ┌────┴──────┐
              │  StrategyA  │ │ StrategyB │
              ├─────────────┤ ├───────────┤
              │ + execute() │ │ + execute()│
              └─────────────┘ └───────────┘
```

**代码示例**：

```typescript
/**
 * 策略模式：定义一系列算法，可相互替换
 *
 * 与状态模式的区别：
 * - 策略模式：算法可以相互替换，由客户端选择
 * - 状态模式：状态自动触发行为，状态改变时行为自动切换
 *
 * 应用场景：
 * - 支付方式：支付宝、微信、银行卡
 * - 排序算法：快排、归并、堆排（根据数据量选择）
 * - 出行路线：驾车、公交、步行、骑行
 * - 压缩算法：ZIP、RAR、7z
 */

/**
 * 策略接口：定义算法的公共接口
 */
interface SortStrategy<T> {
  /**
   * 排序方法
   * @param data 要排序的数据
   */
  sort(data: T[]): T[];

  // 获取策略名称
  getName(): string;
}

/**
 * 具体策略：快速排序
 */
class QuickSortStrategy<T> implements SortStrategy<T> {
  getName(): string {
    return '快速排序';
  }

  sort(data: T[]): T[] {
    console.log(`QuickSort: 排序 ${data.length} 个元素`);

    if (data.length <= 1) {
      return data;
    }

    const pivot = data[0];
    const left: T[] = [];
    const right: T[] = [];

    for (let i = 1; i < data.length; i++) {
      if (data[i] < pivot) {
        left.push(data[i]);
      } else {
        right.push(data[i]);
      }
    }

    return [...this.sort(left), pivot, ...this.sort(right)];
  }
}

/**
 * 具体策略：冒泡排序
 */
class BubbleSortStrategy<T> implements SortStrategy<T> {
  getName(): string {
    return '冒泡排序';
  }

  sort(data: T[]): T[] {
    console.log(`BubbleSort: 排序 ${data.length} 个元素`);

    const result = [...data];
    const n = result.length;

    for (let i = 0; i < n - 1; i++) {
      for (let j = 0; j < n - i - 1; j++) {
        if (result[j] > result[j + 1]) {
          [result[j], result[j + 1]] = [result[j + 1], result[j]];
        }
      }
    }

    return result;
  }
}

/**
 * 具体策略：归并排序
 */
class MergeSortStrategy<T> implements SortStrategy<T> {
  getName(): string {
    return '归并排序';
  }

  sort(data: T[]): T[] {
    console.log(`MergeSort: 排序 ${data.length} 个元素`);

    if (data.length <= 1) {
      return data;
    }

    const mid = Math.floor(data.length / 2);
    const left = this.sort(data.slice(0, mid));
    const right = this.sort(data.slice(mid));

    return this.merge(left, right);
  }

  private merge(left: T[], right: T[]): T[] {
    const result: T[] = [];
    let i = 0, j = 0;

    while (i < left.length && j < right.length) {
      if (left[i] <= right[j]) {
        result.push(left[i++]);
      } else {
        result.push(right[j++]);
      }
    }

    return [...result, ...left.slice(i), ...right.slice(j)];
  }
}

/**
 * 上下文：排序执行器
 * 持有策略引用，委托给策略执行排序
 */
class Sorter<T> {
  private strategy: SortStrategy<T>;

  constructor(strategy: SortStrategy<T>) {
    this.strategy = strategy;
  }

  /**
   * 切换策略
   */
  setStrategy(strategy: SortStrategy<T>): void {
    this.strategy = strategy;
  }

  /**
   * 执行排序
   */
  sort(data: T[]): T[] {
    console.log(`\n使用策略: ${this.strategy.getName()}`);
    const start = Date.now();
    const result = this.strategy.sort(data);
    const end = Date.now();
    console.log(`排序耗时: ${end - start}ms`);
    return result;
  }
}
```

### 4.2 观察者模式（Observer Pattern）

**模式定义**：定义对象间的一种一对多的依赖关系，当一个对象的状态发生改变时，所有依赖于它的对象都会收到通知并自动更新。

**生活类比**：想象你订阅了一个YouTube频道。你（观察者）关注了某个UP主（被观察者），当UP主发布新视频时，所有订阅者都会收到通知。观察者模式就是这个"订阅-发布"机制。

**UML图**：

```
┌─────────────┐       ┌─────────────────┐
│  Subject    │─────▶│    Observer     │
│  (被观察者)   │      │   (观察者)       │
├─────────────┤      ├─────────────────┤
│+attach()    │      │+update(message) │
│+detach()    │      └─────────────────┘
│+notify()    │              △
└──────┬──────┘              │
       │                    │
       ▼                    │
┌─────────────┐             │
│ConcreSubject│             │
├─────────────┤             │
│+state       │             │
│+notify()    │◀────────────┘
└─────────────┘
```

**代码示例**：

```typescript
/**
 * 观察者模式：一对多的依赖关系，状态变化自动通知
 *
 * 与发布-订阅模式的区别：
 * - 观察者模式：观察者和被观察者直接通信
 * - 发布-订阅模式：通过消息队列解耦
 *
 * 应用场景：
 * - GUI事件处理：按钮点击触发多个处理器
 * - MVC架构：Model变化自动更新View
 * - 数据绑定：Vue/Angular的双向绑定
 * - 消息推送：订阅主题，接收更新
 */

/**
 * 观察者接口：定义接收通知的方法
 */
interface Observer {
  // 接收更新通知
  update(message: string): void;

  // 获取观察者名称
  getName(): string;
}

/**
 * 具体观察者：订阅者A
 */
class SubscriberA implements Observer {
  constructor(private name: string) {}

  update(message: string): void {
    console.log(`[${this.name}] 收到通知: ${message}`);
  }

  getName(): string {
    return this.name;
  }
}

/**
 * 具体观察者：订阅者B
 */
class SubscriberB implements Observer {
  constructor(private name: string) {}

  update(message: string): void {
    console.log(`[${this.name}] 收到通知: ${message}`);
  }

  getName(): string {
    return this.name;
  }
}

/**
 * 被观察者接口：定义订阅/退订/通知方法
 */
interface Subject {
  // 添加观察者
  attach(observer: Observer): void;

  // 移除观察者
  detach(observer: Observer): void;

  // 通知所有观察者
  notify(): void;
}

/**
 * 具体被观察者：YouTube频道
 */
class YouTubeChannel implements Subject {
  // 存储所有观察者
  private observers: Observer[] = [];

  // 频道名称
  private channelName: string;

  // 最新视频
  private latestVideo: string = '';

  constructor(channelName: string) {
    this.channelName = channelName;
  }

  /**
   * 订阅频道
   */
  attach(observer: Observer): void {
    // 检查是否已存在（避免重复订阅）
    const exists = this.observers.some(o => o.getName() === observer.getName());
    if (exists) {
      console.log(`${observer.getName()} 已经订阅过了`);
      return;
    }
    this.observers.push(observer);
    console.log(`${observer.getName()} 订阅了 ${this.channelName} 频道`);
  }

  /**
   * 退订频道
   */
  detach(observer: Observer): void {
    const index = this.observers.findIndex(o => o.getName() === observer.getName());
    if (index > -1) {
      this.observers.splice(index, 1);
      console.log(`${observer.getName()} 退订了 ${this.channelName} 频道`);
    } else {
      console.log(`${observer.getName()} 没有订阅过 ${this.channelName} 频道`);
    }
  }

  /**
   * 发布新视频，通知所有订阅者
   */
  publishVideo(videoTitle: string): void {
    console.log(`\n${this.channelName} 发布了新视频: ${videoTitle}`);
    this.latestVideo = videoTitle;
    this.notify();
  }

  /**
   * 通知所有观察者
   */
  notify(): void {
    console.log(`通知 ${this.observers.length} 个订阅者...`);
    for (const observer of this.observers) {
      observer.update(`【${this.channelName}】新视频: ${this.latestVideo}`);
    }
  }

  // 获取订阅者数量
  getSubscriberCount(): number {
    return this.observers.length;
  }
}

// 测试观察者模式
function testObserver() {
  console.log('=== 观察者模式测试：YouTube订阅系统 ===\n');

  // 创建YouTube频道
  const channel = new YouTubeChannel('程序员老李');

  // 创建订阅者
  const alice = new SubscriberA('Alice');
  const bob = new SubscriberB('Bob');
  const charlie = new SubscriberA('Charlie');

  // 订阅频道
  console.log('--- 订阅频道 ---');
  channel.attach(alice);
  channel.attach(bob);
  channel.attach(charlie);

  console.log(`当前订阅者数量: ${channel.getSubscriberCount()}`);

  // 发布第一个视频
  channel.publishVideo('TypeScript入门教程');

  // Charlie取消订阅
  console.log('\n--- Charlie退订 ---');
  channel.detach(charlie);
  console.log(`当前订阅者数量: ${channel.getSubscriberCount()}`);

  // 发布第二个视频
  channel.publishVideo('React最佳实践');

  // 尝试重复订阅
  console.log('\n--- 重复订阅测试 ---');
  channel.attach(alice);

  // 尝试退订未订阅的人
  console.log('\n--- 退订未订阅者 ---');
  channel.detach(new SubscriberB('Dave'));
}

testObserver();
```

**使用场景**：

1. **GUI事件处理**：点击按钮触发多个事件处理器
2. **数据绑定**：Model更新自动刷新View
3. **消息推送系统**：用户订阅主题，主题更新推送给所有订阅者
4. **社交媒体**：关注/取消关注，发布动态推送通知
5. **股票行情**：价格变化通知所有关注者

**注意事项**：

- 观察者模式分为推模型（push）和拉模型（pull）
- 观察者过多可能导致性能问题
- 观察者和被观察者应该避免循环引用

### 4.3 命令模式（Command Pattern）

**模式定义**：将请求封装成对象，从而允许你参数化客户端、对请求排队或记录请求日志，以及支持可撤销的操作。

**生活类比**：想象你去餐厅点餐。你对服务员说"来一份宫保鸡丁"，服务员把订单交给厨房，厨房按订单做菜。在这个过程中，订单本身就是"命令对象"，它封装了"做什么"的信息。服务员不需要知道宫保鸡丁怎么做，他只需要把订单传递给厨房。命令模式就是这个"订单传递"机制。

**UML图**：

```
┌─────────────┐       ┌─────────────────┐
│   Client    │       │    Command      │
│   (客户)     │─────▶│    (命令接口)    │
├─────────────┤       ├─────────────────┤
└─────────────┘       │ +execute()      │
                      │ +undo()         │
                      └────────┬────────┘
                               △
                    ┌──────────┴──────────┐
                    │                     │
           ┌────────┴───────┐   ┌─────────┴────────┐
           │ ConcreteCommand│   │ ConcreteCommand2 │
           ├─────────────────┤   ├──────────────────┤
           │-receiver        │   │-receiver         │
           │+execute()       │   │+execute()        │
           │+undo()          │   │+undo()           │
           └─────────────────┘   └──────────────────┘
                    │
                    ▼
           ┌────────────────┐
           │    Receiver    │
           │    (接收者)     │
           ├────────────────┤
           │ +action()      │
           └────────────────┘
```

**代码示例**：

```typescript
class RemoteControl {
  // 存储已执行的命令（支持撤销）
  private history: Command[] = [];

  // 当前命令（用于显示）
  private currentCommand: Command | null = null;

  /**
   * 执行命令
   */
  executeCommand(command: Command): void {
    console.log(`执行: ${command.getDescription()}`);
    command.execute();
    this.history.push(command);
    this.currentCommand = command;
  }

  /**
   * 撤销上一次命令
   */
  undoCommand(): void {
    if (this.history.length === 0) {
      console.log('没有可撤销的命令');
      return;
    }

    const lastCommand = this.history.pop()!;
    console.log(`撤销: ${lastCommand.getDescription()}`);
    lastCommand.undo();
    this.currentCommand = this.history.length > 0
      ? this.history[this.history.length - 1]
      : null;
  }

  /**
   * 显示当前状态
   */
  showStatus(): void {
    console.log('--- 遥控器状态 ---');
    console.log(`已执行命令数: ${this.history.length}`);
    console.log(`当前命令: ${this.currentCommand?.getDescription() || '无'}`);
  }
}
```

### 4.4 模板方法模式（Template Method Pattern）

**模式定义**：定义一个算法的骨架，将某些步骤延迟到子类中实现。

**生活类比**：想象做菜谱。菜谱说"第一步：准备食材；第二步：切菜；第三步：烹饪；第四步：装盘"。不同菜品的具体做法不同，但流程骨架是固定的。模板方法模式就是这个"菜谱骨架"。

**UML图**：

```
┌────────────────────────────────┐
│       AbstractClass            │
│       (抽象类)                  │
├────────────────────────────────┤
│ +templateMethod()              │ ← 模板方法，定义算法骨架
│ #step1()                       │ ← 具体步骤（可选实现）
│ #step2()                       │ ← 抽象步骤（必须重写）
│ #step3()                       │ ← 抽象步骤（必须重写）
└────────────────────────────────┘
              △
              │
┌─────────────┴─────────────┐
│   ConcreteClass           │
│   (具体类)                │
├───────────────────────────┤
│ #step2() { ... }          │ ← 重写抽象步骤
│ #step3() { ... }          │ ← 重写抽象步骤
└───────────────────────────┘
```

**代码示例**：

```typescript
/**
 * 模板方法模式：定义算法骨架，具体实现延迟到子类
 *
 * 与策略模式的区别：
 * - 策略模式：算法可以整体替换
 * - 模板方法模式：算法骨架固定，部分步骤可替换
 *
 * 应用场景：
 * - 框架生命周期：初始化 → 运行 → 销毁
 * - 数据处理流程：读取 → 转换 → 写入
 * - 测试框架：setup → execute → teardown
 * - 咖啡/茶饮：烧水 → 冲泡 → 倒入杯中 → 加调料
 */

/**
 * 抽象类：泡饮品的基类
 * 定义了制作饮品的算法骨架
 */
abstract class BeverageMaker {
  /**
   * 模板方法：定义制作饮品的固定流程
   * 这个方法声明为final，防止子类重写
   */
  public final makeBeverage(): void {
    console.log('=== 开始制作饮品 ===');

    // 步骤1：烧水（通用步骤）
    this.boilWater();

    // 步骤2：选择原料（通用步骤）
    this.chooseIngredient();

    // 步骤3：冲泡（抽象步骤，子类实现）
    this.brew();

    // 步骤4：倒入杯中（通用步骤）
    this.pourInCup();

    // 步骤5：加调料（钩子方法，子类可选择重写）
    this.addCondiments();

    console.log('=== 饮品制作完成 ===\n');
  }

  /**
   * 烧水：通用步骤
   */
  protected boilWater(): void {
    console.log('烧水至100°C');
  }

  /**
   * 选择原料：通用步骤
   */
  protected chooseIngredient(): void {
    console.log('选择原料');
  }

  /**
   * 冲泡：抽象步骤，必须被子类重写
   */
```

### 4.5 迭代器模式（Iterator Pattern）

**模式定义**：提供一种方法顺序访问一个聚合对象中的各个元素，而不暴露其内部的表示。

**生活类比**：想象一个书架。你不需要知道书架是怎么搭建的（木头架子还是金属架子），你只需要知道如何"从第一本书开始，一本一本往下翻"。迭代器就是"翻书这个动作"的抽象，它让你不需要了解书架的内部结构，就能遍历所有书籍。

**UML图**：

```
┌─────────────┐       ┌─────────────────┐
│  <<interface>> │   │    Iterator     │
│  Aggregate    │───▶│   (迭代器)       │
│  (聚合对象)   │    ├─────────────────┤
├─────────────┤    │ +next()          │
│ +createIterator()│ │ +hasNext(): bool│
└─────────────┘    └─────────────────┘
       △
       │
       │ creates
       ▼
┌─────────────┐
│ConcreteAgg..│
│(具体聚合)   │
├─────────────┤
│ +createIterator()│
│ +getItem()   │
└─────────────┘
```

**代码示例**：

```typescript
/**
 * 迭代器模式：顺序访问集合元素，不暴露内部结构
 *
 * 与for...of的区别：
 * - for...of：JavaScript内置的迭代语法
 * - 迭代器模式：可以自定义遍历逻辑
 *
 * 应用场景：
 * - 遍历树形结构：DOM树、文件系统
 * - 遍历图结构：社交网络好友遍历
 * - 懒加载遍历：大量数据的惰性迭代
 * - 多种遍历方式：前序/中序/后序遍历
 */

/**
 * 迭代器接口：定义迭代器的基本操作
 */
interface Iterator<T> {
  // 是否还有下一个元素
  hasNext(): boolean;

  // 获取下一个元素
  next(): T | null;

  // 重置迭代器到开头
  reset(): void;

  // 获取当前位置
  getCurrentIndex(): number;
}

/**
 * 具体迭代器：数组迭代器
 */
class ArrayIterator<T> implements Iterator<T> {
  private index: number = 0;

  constructor(private collection: T[]) {}

  hasNext(): boolean {
    return this.index < this.collection.length;
  }

  next(): T | null {
    if (this.hasNext()) {
      return this.collection[this.index++];
    }
    return null;
  }

  reset(): void {
    this.index = 0;
  }

  getCurrentIndex(): number {
    return this.index;
  }
}

/**
 * 具体迭代器：反向迭代器
 */
class ReverseIterator<T> implements Iterator<T> {
  private index: number;

  constructor(private collection: T[]) {
    this.index = collection.length - 1;
  }

  hasNext(): boolean {
    return this.index >= 0;
  }

  next(): T | null {
    if (this.hasNext()) {
      return this.collection[this.index--];
    }
    return null;
  }

  reset(): void {
    this.index = this.collection.length - 1;
  }

  getCurrentIndex(): number {
    return this.index;
  }
}

/**
 * 具体迭代器：奇数索引迭代器（只遍历奇数位置）
 */
class OddIndexIterator<T> implements Iterator<T> {
  private index: number = 1; // 从1开始（第二个元素，索引为1）

  constructor(private collection: T[]) {}

  hasNext(): boolean {
    return this.index < this.collection.length;
  }

  next(): T | null {
    if (this.hasNext()) {
      const item = this.collection[this.index];
      this.index += 2; // 跳到下一个奇数位置
      return item;
    }
    return null;
  }

  reset(): void {
    this.index = 1;
  }

  getCurrentIndex(): number {
    return this.index;
  }
}

/**
 * 聚合接口：定义集合的基本操作
 */
interface Collection<T> {
  // 创建正向迭代器
  createIterator(): Iterator<T>;

  // 创建反向迭代器
  createReverseIterator(): Iterator<T>;

  // 获取集合大小
  getSize(): number;

  // 获取指定位置的元素
  get(index: number): T | undefined;
}

/**
 * 具体聚合：歌曲列表
 */
class SongList implements Collection<string> {
  private songs: string[] = [];

  add(song: string): void {
    this.songs.push(song);
  }

  createIterator(): Iterator<string> {
    return new ArrayIterator(this.songs);
  }

  createReverseIterator(): Iterator<string> {
    return new ReverseIterator(this.songs);
  }

  createOddIterator(): Iterator<string> {
    return new OddIndexIterator(this.songs);
  }

  getSize(): number {
    return this.songs.length;
  }

  get(index: number): string | undefined {
    return this.songs[index];
  }
}

// 测试迭代器模式
function testIterator() {
  console.log('=== 迭代器模式测试：歌曲播放列表 ===\n');

  // 创建歌曲列表
  const playlist = new SongList();
  playlist.add('晴天 - 周杰伦');
  playlist.add('稻香 - 周杰伦');
  playlist.add('七里香 - 周杰伦');
  playlist.add('夜曲 - 周杰伦');
  playlist.add('青花瓷 - 周杰伦');

  // 使用正向迭代器遍历
  console.log('--- 正向遍历 ---');
  const iterator = playlist.createIterator();
  while (iterator.hasNext()) {
    const song = iterator.next();
    if (song) {
      console.log(`播放: ${song}`);
    }
  }

  // 使用反向迭代器遍历
  console.log('\n--- 反向遍历 ---');
  const reverseIterator = playlist.createReverseIterator();
  while (reverseIterator.hasNext()) {
    const song = reverseIterator.next();
    if (song) {
      console.log(`播放: ${song}`);
    }
  }

  // 使用奇数索引迭代器
  console.log('\n--- 奇数索引遍历（跳过第一首）---');
  const oddIterator = playlist.createOddIterator();
  while (oddIterator.hasNext()) {
    const song = oddIterator.next();
    if (song) {
      console.log(`播放: ${song}`);
    }
  }

  // 演示迭代器独立性：同一个集合可以用不同迭代器
  console.log('\n--- 同时使用多个迭代器 ---');
  const it1 = playlist.createIterator();
  const it2 = playlist.createIterator();

  console.log('迭代器1 - 前两首:');
  console.log(`  ${it1.next()?.toString()}`);
  console.log(`  ${it1.next()?.toString()}`);

  console.log('迭代器2 - 独立遍历第一首:');
  console.log(`  ${it2.next()?.toString()}`);
}
```

1. **集合遍历**：统一遍历接口，隐藏内部实现
2. **树形结构遍历**：DOM树、文件系统、AST
3. **图遍历**：DFS、BFS
4. **数据库游标**：分页查询
5. **惰性加载**：大数据集的按需遍历

**注意事项**：

- 可以为同一个集合提供多种迭代器

### 4.6 责任链模式（Chain of Responsibility Pattern）

**模式定义**：使多个对象都有机会处理请求，从而避免请求的发送者和接收者之间的耦合关系。

**生活类比**：想象你去政府部门办事。你把申请材料交给窗口A，窗口A说"这个不归我管，找窗口B"，窗口B说"这个需要先盖章，找窗口C"，以此类推。每个窗口只知道自己的职责，不知道整个流程。这就是责任链模式。

**UML图**：

```
┌──────────────────┐       ┌──────────────────┐
│   Handler        │       │   Client         │
│   (处理者接口)    │       │   (客户端)        │
├──────────────────┤       └────────┬─────────┘
│ +setNext()       │                │
│ +handle()        │                ▼
└────────┬─────────┘       (发起请求)
         │                        │
         ▼                        ▼
┌──────────────────┐      ┌───────┴─────────┐
│ConcreteHandlerA  │─────▶│ ConcreteHandlerB│─────▶...
├──────────────────┤      ├─────────────────┤
│ +handle()        │      │ +handle()        │
│   (处理请求)      │      │   (处理请求)      │
└──────────────────┘      └─────────────────┘
```

**代码示例**：

```typescript
/**
 * 责任链模式：请求沿着处理者链传递，直到被某个处理者处理
 *
 * 与中介者模式的区别：
 * - 责任链模式：请求沿着链传递，多个对象都可能处理
 * - 中介者模式：对象通过中介者通信，所有通信集中一点
 *
 * 应用场景：
 * - Web中间件：Express/Koa的middleware链
 * - 审批流程：请假申请 → 主管 → 经理 → 总监
 * - 日志级别：DEBUG → INFO → WARN → ERROR
 * - 事件处理：JS事件冒泡
 */

/**
 * 处理者接口：定义处理请求的接口
 */
interface Handler {
  // 设置下一个处理者
  setNext(handler: Handler): Handler;

  // 处理请求（由具体类实现）
  handle(request: string): string | null;
}

/**
 * 基础处理者：实现公共逻辑
 */
abstract class BaseHandler implements Handler {
  // 下一个处理者
  private nextHandler: Handler | null = null;

  setNext(handler: Handler): Handler {
    this.nextHandler = handler;
    return handler; // 返回handler是为了链式调用
  }

  /**
   * 处理请求的模板方法
   */
  handle(request: string): string | null {
    // 如果当前处理者能处理，就处理
    if (this.canHandle(request)) {
      return this.process(request);
    }

    // 如果有下一个处理者，传递给下一个
    if (this.nextHandler) {
      console.log(`${this.constructor.name} 无法处理，传递给下一个`);
      return this.nextHandler.handle(request);
    }

    // 没有处理者能处理
    console.log(`${this.constructor.name} 无法处理，且没有下一个处理者`);
    return null;
  }

  /**
   * 子类实现：判断是否能处理这个请求
   */
  protected abstract canHandle(request: string): boolean;

  /**
   * 子类实现：实际处理请求
   */
  protected abstract process(request: string): string;
}

/**
 * 具体处理者：客服
 * 负责处理简单咨询
 */
class CustomerServiceHandler extends BaseHandler {
  protected canHandle(request: string): boolean {
    // 能处理包含"咨询"的请求
    return request.includes('咨询');
  }

  protected process(request: string): string {
    console.log('CustomerServiceHandler: 正在处理');
    return `【客服】已处理您的咨询: ${request}`;
  }
}

/**
 * 具体处理者：技术支持
 * 负责处理技术问题
 */
class TechSupportHandler extends BaseHandler {
  protected canHandle(request: string): boolean {
    // 能处理包含"bug"、"故障"、"报错"的请求
    return request.includes('bug') ||
           request.includes('故障') ||
           request.includes('报错');
  }

  protected process(request: string): string {
    console.log('TechSupportHandler: 正在处理');
    return `【技术支持】已处理您的问题: ${request}`;
  }
}

/**
 * 具体处理者：经理
 * 负责处理投诉和复杂问题
 */
class ManagerHandler extends BaseHandler {
  protected canHandle(request: string): boolean {
    // 能处理包含"投诉"、"升级"、"严重"的请求

           request.includes('升级') ||
           request.includes('严重');
  }

  protected process(request: string): string {
    console.log('ManagerHandler: 正在处理');
    return `【经理】已处理您的请求: ${request}`;
  }
}

/**
 * 具体处理者：紧急处理
 * 负责处理紧急问题（优先级最高）
 */
class EmergencyHandler extends BaseHandler {
  protected canHandle(request: string): boolean {
    // 能处理包含"紧急"、"立即"、"宕机"的请求
    return request.includes('紧急') ||
           request.includes('立即') ||
           request.includes('宕机');
  }

  protected process(request: string): string {
    console.log('EmergencyHandler: 正在处理');
    return `【紧急处理】立即处理: ${request}`;
  }
}

// 测试责任链模式
function testChainOfResponsibility() {
  console.log('=== 责任链模式测试：客户问题处理系统 ===\n');

  // 创建处理者
  const customerService = new CustomerServiceHandler();
  const techSupport = new TechSupportHandler();
  const manager = new ManagerHandler();
  const emergency = new EmergencyHandler();

  // 构建责任链：紧急处理 → 客服 → 技术支持 → 经理
  emergency.setNext(customerService).setNext(techSupport).setNext(manager);

  // 测试各种请求
  const requests = [
    '紧急：服务器宕机了！',
    '咨询：如何重置密码？',
    'bug：登录页面报错500',
    '投诉：服务态度太差',
    '故障：无法打开文档',
    '请问你们的营业时间是多少？',
  ];

  console.log('--- 处理请求 ---\n');

  for (const request of requests) {
    console.log(`\n收到请求: "${request}"`);
    const response = emergency.handle(request);
    if (response) {
      console.log(`处理结果: ${response}`);
```

### 4.7 状态模式（State Pattern）

**模式定义**：允许对象在其内部状态改变时改变它的行为，看起来好像修改了它的类。

**生活类比**：想象一个自动售货机。它有几种状态：待机、投币中、可选择商品、出货中、找零中。不同状态下，你按相同的按钮会得到不同的结果。比如"选择商品"按钮，在"待机"状态下不起作用，在"投币后"状态下才会出货。状态模式就是让对象"根据状态改变行为"。

**UML图**：

```
┌──────────────────┐
│     Context      │
│     (上下文)      │
├──────────────────┤
│ - state: State   │
├──────────────────┤
│ +request()       │
└────────┬─────────┘
         │
         │ uses
         ▼
┌──────────────────┐       ┌──────────────────┐
│ <<interface>>    │◀──────│ ConcreteStateA   │
│     State        │       │   (具体状态A)     │
├──────────────────┤       ├──────────────────┤
│ +handle()        │       │ +handle()        │
└──────────────────┘       └──────────────────┘
         △                         △
         │                         │
         │                ┌────────┴────────┐
         │                │                 │
         │        ┌───────┴──────┐ ┌───────┴───────┐
         └───────▶│ ConcreteStateB│ │ ConcreteStateC│
                  │   (具体状态B)  │ │   (具体状态C)  │
                  ├───────────────┤ ├───────────────┤
                  │ +handle()     │ │ +handle()     │
                  └───────────────┘ └───────────────┘
```

**代码示例**：

```typescript
/**
 * 状态模式：对象内部状态改变时，行为自动改变
 *
 * 与策略模式的区别：
 * - 策略模式：客户端主动选择算法
 * - 状态模式：状态自动触发行为转换
 *
 * 应用场景：
 * - 自动售货机状态机
 * - 订单状态流转（待支付 → 已支付 → 已发货 → 已完成）
 * - TCP连接状态机
 * - 游戏角色状态（站立、行走、跳跃、攻击）
 */

/**
 * 状态接口：定义状态的行为
 */
interface State {
  // 处理请求（插入货币）
  insertCoin(): void;

  // 处理请求（退回货币）
  ejectCoin(): void;

  // 处理请求（转动曲柄）
  turnCrank(): void;

  // 处理请求（分配商品）
  dispense(): void;

  // 获取状态名称
  getStateName(): string;
}

/**
 * 具体状态：无硬币状态（初始状态）
 */
class NoCoinState implements State {
```

### 4.8 备忘录模式（Memento Pattern）

**模式定义**：在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态，以便恢复。

**生活类比**：想象你在玩游戏时打了一个Boss，打之前你存了个存档（快照）。打完发现打不过，你可以读档回到之前的状态。备忘录模式就是这个"存档/读档"机制。

**UML图**：

```
┌─────────────┐       ┌─────────────┐       ┌─────────────┐
│ Originator  │──────▶│   Memento   │◀──────│   Caretaker │
│  (发起人)    │       │   (备忘录)   │       │   (管理者)   │
├─────────────┤       ├─────────────┤       ├─────────────┤
│+createMemento()│     │ -state      │       │ -mementos[] │
│+restore()    │       └─────────────┘       └─────────────┘
└─────────────┘
```

**代码示例**：

```typescript
/**
 * 备忘录模式：保存对象状态，支持撤销/恢复
 *
 * 与命令模式的区别：
 * - 命令模式：保存操作，可以重做
 * - 备忘录模式：保存状态，可以恢复
 *
 * 应用场景：
 * - 游戏存档
 * - 编辑器撤销/恢复
 * - 数据库事务回滚

/**
 * 备忘录：存储对象状态
 * 这个类的设计是只读的，不提供修改状态的方法
 */
class GameMemento {
  // 存档数据（私有，防止外部修改）
  private level: number;
  private health: number;
  private gold: number;
  private inventory: string[];
  private timestamp: Date;

  constructor(level: number, health: number, gold: number, inventory: string[]) {
    this.level = level;
    this.health = health;
    this.gold = gold;
    this.inventory = [...inventory]; // 复制数组，防止引用修改
    this.timestamp = new Date();
  }

  // 提供只读访问
  getLevel(): number {
    return this.level;
  }

  getHealth(): number {
    return this.health;
  }

  getGold(): number {
    return this.gold;
  }

  getInventory(): string[] {
    return [...this.inventory]; // 返回副本
  }

  getTimestamp(): Date {
    return this.timestamp;
  }
}

 * 发起人：游戏
 * 创建备忘录和从备忘录恢复

class Game {
  private level: number = 1;
  private health: number = 100;
  private gold: number = 0;
  private inventory: string[] = [];

  constructor() {
    console.log('新游戏开始');
    this.showStatus();
  }

  // 玩游戏：打怪升级
  fightBoss(): void {
    console.log('\n--- 挑战Boss ---');
    this.health -= 30;
    this.gold += 100;
    this.level += 1;
    this.inventory.push(`掉落的装备_${this.level}`);

    if (this.health <= 0) {
      console.log('你死了！');
      this.health = 0;
    }

    console.log('Boss战斗结束');
    this.showStatus();
  }

  // 获得宝藏
  findTreasure(): void {
    console.log('\n--- 发现宝藏 ---');
    this.gold += 500;
    this.inventory.push('黄金宝箱');
    console.log('获得500金币和黄金宝箱');
    this.showStatus();
  }

  // 失去生命
  takeDamage(): void {
    console.log('\n--- 受到攻击 ---');
    this.health -= 20;
    if (this.health <= 0) {
      console.log('你死了！');
      this.health = 0;
    }
    this.showStatus();
  }

  // 创建备忘录（存档）
  createMemento(): GameMemento {
    console.log('\n--- 创建存档 ---');
    return new GameMemento(
      this.level,
      this.health,
      this.gold,
      this.inventory
    );
  }

  // 从备忘录恢复（读档）
  restore(memento: GameMemento): void {
    console.log('\n--- 读取存档 ---');
    this.level = memento.getLevel();
    this.health = memento.getHealth();
    this.gold = memento.getGold();
    this.inventory = memento.getInventory();
    console.log(`存档时间: ${memento.getTimestamp().toLocaleString()}`);
    this.showStatus();
  }

  // 显示状态
  showStatus(): void {
    console.log(`等级: ${this.level}, 生命: ${this.health}, 金币: ${this.gold}`);
    console.log(`背包: ${this.inventory.join(', ') || '空'}`);
  }
}

/**

class SaveManager {

   * 读取存档

  load(key: string): GameMemento | undefined {
    return this.saves.get(key);

   * 删除存档
   */

   * 列出所有存档

    for (const [key, memento] of this.saves) {
      console.log(`${key}: Lv.${memento.getLevel()}, ${memento.getTimestamp().toLocaleString()}`);
    }
  }
}
```

### 4.9 解释器模式（Interpreter Pattern）

**模式定义**：定义一个语言的文法，并且建立一个解释器来解释该语言中的句子。

**生活类比**：想象学习英语。老师教你语法规则（主语+谓语+宾语），然后给你一些句子，让你按照语法规则理解。解释器模式就是这种"根据语法规则解释句子"的机制，不过这里的"语言"不一定是自然语言，也可以是编程语言、正则表达式、配置文件等。

**UML图**：

```
┌──────────────────┐
│    Context       │
│    (上下文)       │
├──────────────────┤
│ +lookup()        │
│ +assign()        │
└──────────────────┘

┌──────────────────┐       ┌──────────────────┐
│ <<interface>>     │       │     Client      │
│   Expression     │◀──────│                 │
│   (表达式接口)    │       └─────────────────┘
├──────────────────┤
│ +interpret()     │
└────────┬─────────┘
         △
    ┌────┴────┐
    │         │
┌───┴───┐ ┌───┴─────┐
│Termina │ │NonTerm..│
│Expr    │ │Expr     │
├────────┤ ├─────────┤
│+inter..│ │+inter.. │
└────────┘ └─────────┘
```

**代码示例**：

```typescript
/**
 * 解释器模式：定义语法规则，解释语言表达式
 *
 * 与其他模式的区别：
 * - 解释器模式：自定义语言或规则引擎
 * - 策略模式：可替换的算法
 * - 职责链模式：请求传递直到被处理
 *
 * 应用场景：
 * - SQL解析
 * - 正则表达式引擎
 * - 数学表达式计算器
 * - 配置文件解析
 * - 业务规则引擎
 */

/**
 * 上下文：存储变量和值
 */
class Context {
  private variables: Map<string, boolean> = new Map();

  // 设置变量值
  setVariable(name: string, value: boolean): void {
    this.variables.set(name, value);
  }

  // 获取变量值
  getVariable(name: string): boolean | undefined {
    return this.variables.get(name);
  }

  // 获取所有变量
  getAllVariables(): Map<string, boolean> {
    return this.variables;
  }
}

/**
 * 表达式接口：定义解释器的基本操作
 */
interface Expression {
  // 解释表达式
  interpret(context: Context): boolean;

  // 转为字符串表示
  toString(): string;
}

/**
 * 终结符表达式：变量
 * 例如：x, y, isActive
 */
class VariableExpression implements Expression {
  constructor(private name: string) {}

  interpret(context: Context): boolean {
    const value = context.getVariable(this.name);
    if (value === undefined) {
      throw new Error(`变量 ${this.name} 未定义`);
    }
    return value;
  }

  toString(): string {
    return this.name;
  }
}

/**
 * 终结符表达式：字面量（true/false）
 */
class LiteralExpression implements Expression {
  constructor(private value: boolean) {}

  interpret(): boolean {
    return this.value;
  }

  toString(): string {
    return this.value ? 'TRUE' : 'FALSE';
  }
}

/**
 * 非终结符表达式：And（且）
 * 例如：A AND B
 */
class AndExpression implements Expression {
  constructor(
    private left: Expression,
    private right: Expression
  ) {}

  interpret(context: Context): boolean {
    return this.left.interpret(context) && this.right.interpret(context);
  }

  toString(): string {
    return `(${this.left.toString()} AND ${this.right.toString()})`;
  }
}

/**
 * 非终结符表达式：Or（或）
 * 例如：A OR B
 */
class OrExpression implements Expression {
  constructor(
    private left: Expression,
    private right: Expression
  ) {}

  interpret(context: Context): boolean {
    return this.left.interpret(context) || this.right.interpret(context);
  }

  toString(): string {
    return `(${this.left.toString()} OR ${this.right.toString()})`;
  }
}

/**
 * 非终结符表达式：Not（非）
 * 例如：NOT A
 */
class NotExpression implements Expression {
  constructor(private expression: Expression) {}

  interpret(context: Context): boolean {
    return !this.expression.interpret(context);
  }

  toString(): string {
    return `NOT ${this.expression.toString()}`;
  }
}

/**
 * 业务规则引擎：构建和解释业务规则
 */
class RuleEngine {
  /**
   * 判断用户是否有权限
   * 规则：管理员 OR (已认证用户 AND 非禁用)
   */
  static createPermissionRule(): Expression {
    const isAdmin = new VariableExpression('isAdmin');
    const isAuthenticated = new VariableExpression('isAuthenticated');
    const isDisabled = new VariableExpression('isDisabled');

    // (isAdmin) OR (isAuthenticated AND NOT isDisabled)
    const authenticatedAndNotDisabled = new AndExpression(
      isAuthenticated,
      new NotExpression(isDisabled)
    );

    return new OrExpression(isAdmin, authenticatedAndNotDisabled);
  }

  /**
   * 判断是否发送通知
   * 规则：(已验证邮箱 AND 已验证手机) OR (是VIP AND 已验证邮箱)
   */
  static createNotificationRule(): Expression {
    const emailVerified = new VariableExpression('emailVerified');
    const phoneVerified = new VariableExpression('phoneVerified');
    const isVIP = new VariableExpression('isVIP');

    // (emailVerified AND phoneVerified) OR (isVIP AND emailVerified)
    const verifiedBoth = new AndExpression(emailVerified, phoneVerified);
    const vipWithEmail = new AndExpression(isVIP, emailVerified);

    return new OrExpression(verifiedBoth, vipWithEmail);
  }
}

// 测试解释器模式
function testInterpreter() {
  console.log('=== 解释器模式测试：业务规则引擎 ===\n');

  // 创建上下文
  const context = new Context();

  // 测试规则1：权限判断
  console.log('--- 测试权限规则 ---');
  console.log('规则: (isAdmin) OR (isAuthenticated AND NOT isDisabled)');

  const permissionRule = RuleEngine.createPermissionRule();
  console.log(`规则表达式: ${permissionRule.toString()}\n`);

  // 场景1：普通用户，正常状态
  console.log('场景1：普通用户，正常状态');
  context.setVariable('isAdmin', false);
  context.setVariable('isAuthenticated', true);
  context.setVariable('isDisabled', false);
  console.log(`结果: ${permissionRule.interpret(context) ? '允许访问' : '拒绝访问'}`);

  // 场景2：禁用用户
  console.log('\n场景2：禁用用户');
  context.setVariable('isAdmin', false);
  context.setVariable('isAuthenticated', true);
```

### 4.10 中介者模式（Mediator Pattern）

**模式定义**：用一个中介对象来封装一系列的对象交互，使各对象不需要显式地相互引用。

**生活类比**：想象一个机场的塔台。飞机之间不直接通信，而是通过塔台中转。飞机A说"我要降落"，塔台告诉它"等待"，然后塔台告诉飞机B"减速让行"。中介者模式就是这个"塔台"，它让所有通信都集中在一个地方，避免了对象之间的直接耦合。

**UML图**：

```
┌──────────────┐        ┌──────────────────┐        ┌──────────────┐
│   ColleagueA │◀──────▶│     Mediator     │◄─────▶│   ColleagueB │
│   (同事A)     │        │    (中介者)       │        │   (同事B)     │
├──────────────┤        ├──────────────────┤        ├──────────────┤
│ -mediator    │        │ +notify()        │        │ -mediator    │
└──────────────┘        └──────────────────┘        └──────────────┘
```

**代码示例**：

```typescript
/**
 * 中介者模式：通过中介者集中对象间的通信
 *
 * 与观察者模式的区别：
 * - 观察者模式：订阅-发布，一对多，发布者不知道订阅者
 * - 中介者模式：中心化协调，多对多，对象之间解耦
 *
 * 应用场景：
 * - GUI对话框：按钮、输入框、下拉框通过对话框协调
 * - 聊天室：用户之间不直接通信，通过聊天室中转
 * - 航空管制：飞机通过塔台通信
 * - 微服务：API网关作为中介
 */

/**
 * 中介者接口：定义同事之间的通信方法
 */
interface Mediator {
  // 通知中介者有消息
  notify(sender: Colleague, event: string): void;
}

/**
 * 同事基类：所有需要通信的对象的基类
 */
abstract class Colleague {
  // 持有中介者引用
  protected mediator: Mediator;

  constructor(mediator: Mediator) {
    this.mediator = mediator;
  }

  // 发送消息给中介者
  send(event: string): void {
    console.log(`${this.constructor.name} 发送: ${event}`);
    this.mediator.notify(this, event);
  }

  // 接收消息（由子类实现）
  abstract receive(event: string): void;
}

/**
 * 具体同事：按钮
 */
class Button extends Colleague {
  private label: string;

  constructor(mediator: Mediator, label: string) {
    super(mediator);
    this.label = label;
  }

  receive(event: string): void {
    console.log(`[Button:${this.label}] 收到事件: ${event}`);
  }

  // 点击按钮
  click(): void {
    console.log(`[Button:${this.label}] 被点击`);
    this.send('click');
  }

  // 设置是否可用
  setEnabled(enabled: boolean): void {
    console.log(`[Button:${this.label}] ${enabled ? '启用' : '禁用'}`);
  }
}

/**
 * 具体同事：文本框
 */
class TextBox extends Colleague {
  private content: string = '';

  constructor(mediator: Mediator, placeholder: string) {
    super(mediator);
    this.placeholder = placeholder;
  }

  receive(event: string): void {
    console.log(`[TextBox:${this.placeholder}] 收到事件: ${event}`);
  }

  // 输入文本
  input(text: string): void {
    this.content = text;
    console.log(`[TextBox:${this.placeholder}] 输入: ${text}`);
    this.send('input');
  }

  getContent(): string {
    return this.content;
  }

  setContent(content: string): void {
    this.content = content;
    console.log(`[TextBox:${this.placeholder}] 内容设置为: ${content}`);
  }

  private placeholder: string;
}

/**
 * 具体同事：下拉框
 */
class Dropdown extends Colleague {
  private options: string[] = [];
  private selectedIndex: number = -1;

  constructor(mediator: Mediator, options: string[] = []) {
    super(mediator);
    this.options = options;
  }

  receive(event: string): void {
    console.log(`[Dropdown] 收到事件: ${event}`);
  }

  select(index: number): void {
    if (index >= 0 && index < this.options.length) {
      this.selectedIndex = index;
      console.log(`[Dropdown] 选择: ${this.options[index]}`);
      this.send('select');
    }
  }

  getSelected(): string | null {
    return this.selectedIndex >= 0 ? this.options[this.selectedIndex] : null;
  }

  setOptions(options: string[]): void {
    this.options = options;
    console.log(`[Dropdown] 选项更新为: ${options.join(', ')}`);
  }
}

/**
 * 具体中介者：用户表单中介者
 * 协调表单组件之间的交互
 */
class UserFormMediator implements Mediator {
  private button: Button | null = null;
  private usernameInput: TextBox | null = null;
  private emailInput: TextBox | null = null;
  private roleDropdown: Dropdown | null = null;
  private submitButton: Button | null = null;

  // 注入所有同事
  setButton(button: Button): void {
    this.button = button;
  }

  setUsernameInput(input: TextBox): void {
    this.usernameInput = input;
  }

  setEmailInput(input: TextBox): void {
    this.emailInput = input;
  }

  setRoleDropdown(dropdown: Dropdown): void {
    this.roleDropdown = dropdown;
  }

  setSubmitButton(button: Button): void {
    this.submitButton = button;
  }

  /**
   * 中介者的核心方法：处理同事的消息
   */
  notify(sender: Colleague, event: string): void {
    console.log(`[Mediator] 收到来自 ${sender.constructor.name} 的事件: ${event}`);

    // 根据不同的事件进行协调
    if (event === 'input') {
      // 输入事件：检查表单是否有效，控制提交按钮状态
      this.checkFormValidity();
    } else if (event === 'select') {
      // 选择事件：管理员角色时，禁用用户名输入
      this.handleRoleChange();
    } else if (event === 'click') {
      // 点击事件：执行提交逻辑
      this.handleSubmit();
    }
  }

  // 检查表单是否有效
  private checkFormValidity(): void {
    if (!this.usernameInput || !this.emailInput || !this.submitButton) {
      return;
    }

    const username = this.usernameInput.getContent();
    const email = this.emailInput.getContent();
    const isValid = username.length >= 3 && email.includes('@');

    this.submitButton.setEnabled(isValid);
    console.log(`[Mediator] 表单验证: ${isValid ? '有效' : '无效'}`);
  }

  // 处理角色变更
  private handleRoleChange(): void {
    if (!this.roleDropdown || !this.usernameInput) {
      return;
    }

    const role = this.roleDropdown.getSelected();

    if (role === '管理员') {
      // 管理员角色：禁用用户名输入（自动生成）
      console.log('[Mediator] 管理员角色：禁用用户名输入');
    } else {
      // 其他角色：启用用户名输入
      console.log('[Mediator] 其他角色：启用用户名输入');
    }
  }

  // 处理提交
  private handleSubmit(): void {
    if (!this.usernameInput || !this.emailInput || !this.roleDropdown) {
      return;
    }

    const username = this.usernameInput.getContent();
    const email = this.emailInput.getContent();
    const role = this.roleDropdown.getSelected();

    console.log('\n========== 提交用户信息 ==========');
    console.log(`用户名: ${username}`);
    console.log(`邮箱: ${email}`);
    console.log(`角色: ${role}`);
    console.log('===================================\n');
  }
}

function testMediator() {
  console.log('=== 中介者模式测试：用户表单协调 ===\n');

  // 创建中介者
  const mediator = new UserFormMediator();

  // 创建组件（同事）
  const usernameInput = new TextBox(mediator, '用户名');
  const emailInput = new TextBox(mediator, '邮箱');
  const roleDropdown = new Dropdown(mediator, ['普通用户', '管理员', '超级管理员']);
  const submitButton = new Button(mediator, '提交');
  const cancelButton = new Button(mediator, '取消');

  // 注册组件到中介者
  mediator.setUsernameInput(usernameInput);
  mediator.setEmailInput(emailInput);
  mediator.setRoleDropdown(roleDropdown);

  mediator.setButton(submitButton);

  // 模拟用户交互

  console.log('\n--- 用户完善信息 ---');
  usernameInput.input('john_doe'); // 输入完整的用户名
  emailInput.input('john@example.com');

  console.log('\n--- 用户选择角色 ---');
```

1. **GUI表单组件**：输入框、按钮、下拉框通过表单协调
2. **聊天室**：用户通过聊天室中转消息
3. **航空管制**：飞机通过塔台通信
4. **微服务网关**：API网关作为服务间的中介
5. **监控系统**：监控组件通过监控中心协调

**注意事项**：

- 中介者模式将对象间的耦合集中到中介者
- 中介者可能变成"上帝对象"（知道太多、做太多）
- 适用于对象之间交互复杂、耦合严重的情况

### 4.11 访问者模式（Visitor Pattern）

**模式定义**：表示一个作用于某对象结构中的各元素的操作，使操作可以在不改变各元素的类的前提下定义作用于这些元素的新操作。

**生活类比**：想象一个博物馆。博物馆里有各种文物（雕塑、绘画、陶瓷），也有各种参观者（学生、教授、记者）。文物不知道自己被谁参观，但参观者知道如何"参观"自己。学生会数一数有多少文物，教授会分析文物年代，记者会拍照。访问者模式就是让"参观者"（操作）作用于"文物"（元素），而不改变文物本身。

**UML图**：

```
┌──────────────┐        ┌──────────────┐
│   ElementA   │◄───────│   Visitor    │
├──────────────┤        ├──────────────┤
│ +accept()    │        │ +visitA()     │
└──────────────┘        │ +visitB()     │
       △                 └──────────────┘
       │                         △
       │                         │
┌──────┴──────┐                  │
│  ObjectStr │                  │
│  (对象结构) │                  │
└─────────────┘                  │
                                 │
       ┌─────────────────────────┘
       │
       ▼
┌──────────────┐        ┌──────────────┐
│   ElementB   │◄───────│ConcreteVisit.│
├──────────────┤        ├──────────────┤
│ +accept()    │        │ +visitA()     │
└──────────────┘        │ +visitB()     │
                         └──────────────┘
```

**代码示例**：

```typescript
/**
 * 访问者模式：在不改变元素类的前提下，定义新操作
 *
 * 与策略模式的区别：
 * - 策略模式：算法可以替换
 * - 访问者模式：在元素上定义新操作，不改变元素
 *
 * 应用场景：
 * - 编译器AST遍历：不同访问者做不同分析
 * - 文件系统遍历：计算大小、统计文件数、杀毒扫描
 * - 报表生成：HTML报表、PDF报表、Excel报表
 * - 序列化：JSON序列化、XML序列化
 */

/**
 * 元素接口：定义接受访问者的方法
 */
interface DocumentElement {
  // 接受访问者访问
  accept(visitor: DocumentVisitor): void;

  // 获取元素内容
  getContent(): string;
}

/**
 * 具体元素：段落
 */
class Paragraph implements DocumentElement {
  constructor(private text: string) {}

  accept(visitor: DocumentVisitor): void {
    // 将控制权交给访问者
    visitor.visitParagraph(this);
  }

  getContent(): string {
    return this.text;
  }
}

/**
 * 具体元素：图片
 */
class Image implements DocumentElement {
  constructor(
    private url: string,
    private alt: string
  ) {}

  accept(visitor: DocumentVisitor): void {
    visitor.visitImage(this);
  }

  getContent(): string {
    return `[图片: ${this.alt}]`;
  }

  getUrl(): string {
    return this.url;
  }

  getAlt(): string {
    return this.alt;
  }
}

/**
 * 具体元素：表格
 */
class Table implements DocumentElement {
  constructor(
    private headers: string[],
    private rows: string[][]
  ) {}

  accept(visitor: DocumentVisitor): void {
    visitor.visitTable(this);
  }

  getContent(): string {
    return `[表格: ${this.headers.join(', ')}]`;
  }

  getHeaders(): string[] {
    return this.headers;
  }

  getRows(): string[][] {
    return this.rows;
  }
}

/**
 * 访问者接口：定义对每种元素的访问操作
 */
interface DocumentVisitor {
  // 访问段落
  visitParagraph(paragraph: Paragraph): void;

  // 访问图片
  visitImage(image: Image): void;

  // 访问表格
  visitTable(table: Table): void;
}

/**
 * 具体访问者：HTML导出器
 */
class HTMLExporter implements DocumentVisitor {
  private html: string = '';

  visitParagraph(paragraph: Paragraph): void {
    this.html += `<p>${paragraph.getContent()}</p>\n`;
  }

  visitImage(image: Image): void {
    this.html += `<img src="${image.getUrl()}" alt="${image.getAlt()}" />\n`;
  }

  visitTable(table: Table): void {
    this.html += '<table>\n';
    this.html += '<thead><tr>';
    for (const header of table.getHeaders()) {
      this.html += `<th>${header}</th>`;
    }
    this.html += '</tr></thead>\n';
    this.html += '<tbody>\n';
    for (const row of table.getRows()) {
      this.html += '<tr>';
      for (const cell of row) {
        this.html += `<td>${cell}</td>`;
      }
      this.html += '</tr>\n';
    }
    this.html += '</tbody>\n</table>\n';
  }

  // 获取生成的HTML
  getHTML(): string {
    return `<!DOCTYPE html>
<html>
<head><title>Document</title></head>
<body>
${this.html}
</body>
</html>`;
  }
}

/**
 * 具体访问者：纯文本导出器
 */
class PlainTextExporter implements DocumentVisitor {
  private text: string = '';

  visitParagraph(paragraph: Paragraph): void {
    this.text += `${paragraph.getContent()}\n\n`;
  }

  visitImage(image: Image): void {
    this.text += `[图片: ${image.getAlt()}]\n\n`;
  }

  visitTable(table: Table): void {
    // 简单格式化输出表格
    const headers = table.getHeaders();
    this.text += headers.join(' | ') + '\n';
    this.text += headers.map(() => '---').join(' | ') + '\n';
    for (const row of table.getRows()) {
      this.text += row.join(' | ') + '\n';
    }
    this.text += '\n';
  }

  getText(): string {
    return this.text;
  }
}

/**
 * 具体访问者：字数统计器
 */
class WordCounter implements DocumentVisitor {
  private wordCount: number = 0;
  private imageCount: number = 0;
  private tableCount: number = 0;
  private paragraphCount: number = 0;

  visitParagraph(paragraph: Paragraph): void {
    this.paragraphCount++;
    const words = paragraph.getContent().split(/\s+/).filter(w => w.length > 0);
    this.wordCount += words.length;
  }

  visitImage(image: Image): void {
    this.imageCount++;
  }

  visitTable(table: Table): void {
    this.tableCount++;
    // 统计表格中的文字
    for (const row of table.getRows()) {
      for (const cell of row) {
        const words = cell.split(/\s+/).filter(w => w.length > 0);
        this.wordCount += words.length;
      }
    }
  }

  // 获取统计报告
  getReport(): string {
    return `
========== 文档统计 ==========
段落数: ${this.paragraphCount}
图片数: ${this.imageCount}
表格数: ${this.tableCount}
总字数: ${this.wordCount}
============================`;
  }
}

/**
 * 对象结构：文档
 * 存储所有元素，提供遍历接口
 */
class Document {
  private elements: DocumentElement[] = [];

  add(element: DocumentElement): void {
    this.elements.push(element);
  }

  // 接受访问者访问
  accept(visitor: DocumentVisitor): void {
    for (const element of this.elements) {
      element.accept(visitor);
    }
  }
}
```

## 五、设计模式的组合应用

在实际项目中，很少只使用一种设计模式。多个设计模式通常会组合使用来解决复杂问题。

### 5.1 典型组合：工厂方法 + 单例

```typescript
/**
 * 工厂方法 + 单例组合

class DatabasePool {

      DatabasePool.instance = new DatabasePool();

    return DatabasePool.instance;
  }

  getConnection(): any {
    // 获取连接
    return {};
  }
}
```

### 5.2 典型组合：观察者 + 命令

```typescript
/**
 * 观察者 + 命令组合
 * 实现撤销/重做功能
 */
interface Command {
  execute(): void;
  undo(): void;
}

class UndoableSubject {
  private observers: ((cmd: Command) => void)[] = [];

  executeCommand(cmd: Command) {
    cmd.execute();
    this.notifyObservers(cmd);
  }

  private notifyObservers(cmd: Command) {
    this.observers.forEach(obs => obs(cmd));
  }
}
```

### 5.3 典型组合：装饰器 + 组合

```typescript
/**
 * 装饰器 + 组合组合
 * 文件系统操作
 */

// 文件节点（组合模式）
interface FileNode {
  getSize(): number;
}

// 装饰器添加功能
class EncryptedFile implements FileNode {
  constructor(private file: FileNode) {}

  getSize(): number {
    return this.file.getSize();
  }

  getEncryptedContent(): string {
    return '加密内容';
  }
}
```

## 六、设计模式的选型指南

### 6.1 根据场景选型

| 场景 | 推荐模式 |
|------|----------|
| 对象创建复杂，需要灵活创建 | 工厂方法、抽象工厂、建造者 |
| 需要全局唯一实例 | 单例模式 |
| 需要复制对象 | 原型模式 |
| 接口不兼容，需要转换 | 适配器模式 |
| 需要处理对象组合 | 组合模式、装饰器模式 |
| 需要延迟加载 | 代理模式、享元模式 |
| 算法可以替换 | 策略模式 |
| 状态自动切换 | 状态模式 |
| 需要撤销/重做 | 命令模式、备忘录模式 |
| 对象需要知道状态变化 | 观察者模式 |
| 需要遍历集合 | 迭代器模式 |
| 需要按顺序处理请求 | 责任链模式 |
| 需要解释语言/规则 | 解释器模式 |
| 对象交互复杂 | 中介者模式 |
| 需要在元素上定义操作 | 访问者模式 |

### 6.2 反模式警告

以下情况应该避免使用设计模式：

1. **过度设计**：简单的if-else能解决问题，不要引入策略模式
2. **为模式而模式**：不要为了展示自己懂设计模式而强行使用
3. **模式误用**：用适配器模式做转换，用装饰器模式做继承
4. **模式叠加**：不要一个功能用5个设计模式

## 七、总结

### 7.1 设计模式核心思想

1. **封装变化**：将可能变化的部分封装起来
2. **组合优于继承**：使用组合来扩展功能，而不是继承
3. **面向接口编程**：依赖抽象，而不是具体实现
4. **职责单一**：每个类只做一件事

### 7.2 学习建议

1. **理解原则**：先理解SOLID原则，再学习设计模式
2. **动手实践**：不要只看理论，要实际编码
3. **项目应用**：在真实项目中应用，不要纸上谈兵
4. **持续反思**：思考为什么这个模式适合/不适合当前场景

### 7.3 进阶路径

```
第一阶段：会用
├── 了解23种模式的基本概念
└── 能识别代码中使用的模式

第二阶段：会用
├── 能在项目中正确应用模式
└── 能根据场景选择合适的模式

第三阶段：精通
├── 能灵活组合多个模式
└── 能识别模式的变体和误用

第四阶段：超越
├── 能创造新的模式变体
└── 能识别模式背后的核心思想
```
