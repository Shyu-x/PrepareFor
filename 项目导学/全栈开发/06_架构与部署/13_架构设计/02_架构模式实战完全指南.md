# 架构模式实战完全指南

## 前言：什么是架构模式？

想象你要建造一座房子。你可以选择不同的建筑风格：中式四合院、欧式城堡、现代公寓。不同的风格有不同特点：四合院注重庭院布局，城堡强调防御功能，公寓追求空间效率。软件架构模式就像是建筑的"风格指南"，它告诉我们如何组织代码结构，让系统更稳定、更易维护、更高性能。

本文档将带你深入理解七种经典架构模式：分层架构、微内核架构、事件驱动架构、CQRS、EDA事件驱动架构、六边形架构和洋葱架构。每种架构都有其适用场景和核心思想，我会通过大量实际案例帮助你理解。

## 一、分层架构（Layered Architecture）

### 1.1 什么是分层架构？

分层架构是最常见的架构模式，就像一栋写字楼的楼层划分：每层有不同的功能，有电梯（接口）连接各层。经典的MVC（Model-View-Controller）就是分层架构的典型代表。

**核心思想**：将系统划分为多个层次，每个层次只依赖其下层，职责分明。

**生活类比**：想象一家餐厅，有服务员（展示层）、厨师（业务逻辑层）、采购员（数据访问层）。服务员不直接去菜市场买菜，而是告诉厨师要什么菜，厨师告诉采购员买什么。每一层只做好自己的事。

### 1.2 分层架构的层次划分

```
┌─────────────────────────────────────────────┐
│           Presentation Layer                │
│              (展示层/UI层)                   │
│   负责：用户界面、数据展示、用户交互           │
├─────────────────────────────────────────────┤
│           Application Layer                 │
│              (应用层/服务层)                  │
│   负责：业务流程编排、事务管理、权限控制       │
├─────────────────────────────────────────────┤
│             Domain Layer                    │
│              (领域层/业务层)                 │
│   负责：核心业务逻辑、领域模型、业务规则       │
├─────────────────────────────────────────────┤
│          Infrastructure Layer                │
│              (基础设施层)                   │
│   负责：数据库访问、文件存储、外部服务调用     │
└─────────────────────────────────────────────┘
```

### 1.3 代码示例：电商系统分层架构

```typescript
/**
 * 分层架构示例：电商订单系统
 *
 * 架构分层：
 * - Presentation Layer（展示层）：处理用户请求和响应
 * - Application Layer（应用层）：编排业务流程
 * - Domain Layer（领域层）：核心业务逻辑
 * - Infrastructure Layer（基础设施层）：数据持久化
 */

/**
 * ========== 领域层（Domain Layer）==========
 * 这一层包含核心业务逻辑，是系统最稳定的一层
 * 不依赖任何其他层，是整个系统的"心脏"
 */

/**
 * 订单实体：领域模型
 * 代表订单这个业务概念，包含订单的所有业务规则
 */
class Order {
  // 订单ID
  private id: string;
  // 订单状态
  private status: OrderStatus;
  // 订单项
  private items: OrderItem[];
  // 订单总价
  private totalAmount: number;
  // 创建时间
  private createdAt: Date;

  constructor(id: string) {
    this.id = id;
    this.status = OrderStatus.PENDING;
    this.items = [];
    this.totalAmount = 0;
    this.createdAt = new Date();
  }

  /**
   * 添加订单项
   * 这是订单的核心业务方法，会触发业务规则验证
   */
  addItem(item: OrderItem): void {
    // 业务规则：已支付的订单不能添加商品
    if (this.status === OrderStatus.PAID) {
      throw new Error('已支付的订单不能添加商品');
    }

    this.items.push(item);
    this.recalculateTotal();
  }

  /**
   * 重新计算总价
   */
  private recalculateTotal(): void {
    this.totalAmount = this.items.reduce(
      (sum, item) => sum + item.getPrice() * item.getQuantity(),
      0
    );
  }

  /**
   * 支付订单
   * 核心业务方法
   */
  pay(paymentMethod: string): void {
    // 业务规则：只有待支付状态才能支付
    if (this.status !== OrderStatus.PENDING) {
      throw new Error('订单状态不正确，无法支付');
    }

    // 业务规则：订单必须有商品
    if (this.items.length === 0) {
      throw new Error('订单不能为空');
    }

    this.status = OrderStatus.PAID;
  }

  /**
   * 取消订单
   * 核心业务方法
   */
  cancel(): void {
    // 业务规则：已支付的订单取消需要退款
    if (this.status === OrderStatus.PAID) {
      // 这里只是标记，实际退款由应用层处理
      this.status = OrderStatus.CANCELLED_REFUNDING;
    } else if (this.status === OrderStatus.PENDING) {
      this.status = OrderStatus.CANCELLED;
    } else {
      throw new Error('当前状态无法取消订单');
    }
  }

  // Getter方法
  getId(): string { return this.id; }
  getStatus(): OrderStatus { return this.status; }
  getTotalAmount(): number { return this.totalAmount; }
  getItems(): OrderItem[] { return [...this.items]; }
}

/**
 * 订单项
 */
class OrderItem {
  constructor(
    private productId: string,
    private productName: string,
    private price: number,
    private quantity: number
  ) {}

  getPrice(): number { return this.price; }
  getQuantity(): number { return this.quantity; }
  getProductId(): string { return this.productId; }
}

/**
 * 订单状态枚举
 */
enum OrderStatus {
  PENDING = 'pending',           // 待支付
  PAID = 'paid',               // 已支付
  CANCELLED = 'cancelled',     // 已取消（未支付）
  CANCELLED_REFUNDING = 'cancelled_refunding', // 取消中（退款）
  COMPLETED = 'completed'      // 已完成
}

/**
 * ========== 应用层（Application Layer）==========
 * 这一层编排业务流程，组合多个领域对象完成任务
 * 协调多个领域服务，处理事务边界
 */

/**
 * 订单应用服务
 * 负责订单相关业务流程的编排
 */
class OrderApplicationService {
  /**
   * 创建订单
   * 应用层方法：协调领域对象和基础设施
   */
  async createOrder(request: CreateOrderRequest): Promise<OrderResponse> {
    console.log('应用层：开始创建订单...');

    // 1. 调用基础设施层创建订单
    const orderRepository = new OrderRepository();
    const order = new Order(this.generateOrderId());

    // 2. 添加订单项（领域逻辑）
    for (const itemRequest of request.items) {
      const item = new OrderItem(
        itemRequest.productId,
        itemRequest.productName,
        itemRequest.price,
        itemRequest.quantity
      );
      order.addItem(item);
    }

    // 3. 持久化订单
    await orderRepository.save(order);

    console.log('应用层：订单创建完成');

    // 4. 返回响应
    return {
      orderId: order.getId(),
      status: order.getStatus(),
      totalAmount: order.getTotalAmount()
    };
  }

  /**
   * 支付订单
   * 演示事务管理和跨层协调
   */
  async payOrder(orderId: string, paymentMethod: string): Promise<PayOrderResponse> {
    console.log(`应用层：开始支付订单 ${orderId}...`);

    // 1. 获取订单
    const orderRepository = new OrderRepository();
    const order = await orderRepository.findById(orderId);

    if (!order) {
      throw new Error('订单不存在');
    }

    // 2. 执行业务操作（领域逻辑）
    order.pay(paymentMethod);

    // 3. 创建支付记录（基础设施）
    const paymentService = new PaymentService();
    const paymentResult = await paymentService.processPayment(
      order.getTotalAmount(),
      paymentMethod
    );

    // 4. 更新订单状态
    await orderRepository.save(order);

    console.log('应用层：订单支付完成');

    return {
      success: true,
      orderId: order.getId(),
      paymentId: paymentResult.paymentId
    };
  }

  /**
   * 取消订单
   * 演示复杂业务场景的处理
   */
  async cancelOrder(orderId: string): Promise<CancelOrderResponse> {
    console.log(`应用层：开始取消订单 ${orderId}...`);

    const orderRepository = new OrderRepository();
    const order = await orderRepository.findById(orderId);

    if (!order) {
      throw new Error('订单不存在');
    }

    // 执行业务操作
    order.cancel();

    // 如果是已支付订单，需要处理退款
    if (order.getStatus() === OrderStatus.CANCELLED_REFUNDING) {
      console.log('应用层：需要退款，触发退款流程...');
      // 退款逻辑由支付服务处理
    }

    // 保存订单
    await orderRepository.save(order);

    return {
      success: true,
      orderId: order.getId(),
      status: order.getStatus()
    };
  }

  private generateOrderId(): string {
    return 'ORD-' + Date.now();
  }
}

// 请求/响应DTO
interface CreateOrderRequest {
  items: {
    productId: string;
    productName: string;
    price: number;
    quantity: number;
  }[];
}

interface OrderResponse {
  orderId: string;
  status: OrderStatus;
  totalAmount: number;
}

interface PayOrderResponse {
  success: boolean;
  orderId: string;
  paymentId: string;
}

interface CancelOrderResponse {
  success: boolean;
  orderId: string;
  status: OrderStatus;
}

/**
 * ========== 基础设施层（Infrastructure Layer）==========
 * 这一层处理技术细节：数据库、缓存、文件、外部服务
 * 依赖领域层，但不包含业务逻辑
 */

/**
 * 订单仓储：处理订单的数据持久化
 */
class OrderRepository {
  // 模拟数据库存储
  private database: Map<string, Order> = new Map();

  async save(order: Order): Promise<void> {
    console.log(`基础设施层：保存订单 ${order.getId()} 到数据库`);
    this.database.set(order.getId(), order);
  }

  async findById(orderId: string): Promise<Order | null> {
    console.log(`基础设施层：从数据库查询订单 ${orderId}`);
    return this.database.get(orderId) || null;
  }

  async findByStatus(status: OrderStatus): Promise<Order[]> {
    console.log(`基础设施层：查询状态为 ${status} 的订单`);
    return Array.from(this.database.values())
      .filter(order => order.getStatus() === status);
  }
}

/**
 * 支付服务：处理支付相关的外部集成
 */
class PaymentService {
  async processPayment(amount: number, method: string): Promise<{ paymentId: string }> {
    console.log(`基础设施层：调用支付网关，金额 ${amount}，方式 ${method}`);

    // 模拟调用第三方支付API
    await this.simulateNetworkCall();

    return {
      paymentId: 'PAY-' + Date.now()
    };
  }

  private async simulateNetworkCall(): Promise<void> {
    return new Promise(resolve => setTimeout(resolve, 100));
  }
}

/**
 * ========== 展示层（Presentation Layer）==========
 * 这一层处理HTTP请求、响应格式化、参数验证
 * 直接面向用户/客户端
 */

/**
 * 订单控制器：处理HTTP请求
 */
class OrderController {
  // 依赖应用服务
  private orderService: OrderApplicationService;

  constructor() {
    this.orderService = new OrderApplicationService();
  }

  /**
   * 处理创建订单请求
   */
  async handleCreateOrder(request: CreateOrderRequest): Promise<OrderResponse> {
    console.log('\n展示层：收到创建订单请求');

    // 参数验证（展示层职责）
    if (!request.items || request.items.length === 0) {
      throw new Error('订单项不能为空');
    }

    // 调用应用层处理业务
    const response = await this.orderService.createOrder(request);

    console.log('展示层：返回订单响应');
    return response;
  }

  /**
   * 处理支付请求
   */
  async handlePayOrder(orderId: string, paymentMethod: string): Promise<PayOrderResponse> {
    console.log('\n展示层：收到支付请求');

    const response = await this.orderService.payOrder(orderId, paymentMethod);

    console.log('展示层：返回支付响应');
    return response;
  }

  /**
   * 处理取消订单请求
   */
  async handleCancelOrder(orderId: string): Promise<CancelOrderResponse> {
    console.log('\n展示层：收到取消订单请求');

    const response = await this.orderService.cancelOrder(orderId);

    console.log('展示层：返回取消响应');
    return response;
  }
}
```

### 1.4 分层架构的优缺点

**优点**：
- 职责清晰，每层有明确的职责
- 易于开发和维护
- 便于测试，每层可以独立测试
- 便于团队分工，不同层可以交给不同团队

**缺点**：
- 层级过多可能导致性能开销
- 不太适合需要高度灵活性的场景
- 领域逻辑可能分散在多层中

**适用场景**：
- 企业级应用（ERP、CRM）
- 传统Web应用
- 业务逻辑复杂的系统

## 二、微内核架构（Microkernel Architecture）

### 2.1 什么是微内核架构？

微内核架构就像一个操作系统的设计：内核提供最基本的功能（进程调度、内存管理等），其他功能（文件系统、网络协议）作为插件运行在内核之上。这种架构强调核心系统的稳定性和插件的独立性。

**核心思想**：将系统分为一个小型核心系统和一个插件系统，核心系统提供最小功能集，插件负责扩展功能。

**生活类比**：想象一台基础款汽车，厂商提供了发动机、底盘、轮胎等核心部件。但消费者可以根据需求加装GPS、倒车影像、真皮座椅等配件。这些配件就是"插件"，可以自由选择和组合。

### 2.2 微内核架构的组成

```
┌──────────────────────────────────────────────────────────┐
│                    Plugin System                          │
│                     (插件系统)                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐│
│  │ Plugin 1 │  │ Plugin 2 │  │ Plugin 3 │  │ Plugin 4 ││
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘│
├──────────────────────────────────────────────────────────┤
│                     Core System                           │
│                      (核心系统)                            │
│  ┌─────────────────────────────────────────────────────┐ │
│  │              Plugin Registry (插件注册表)            │ │
│  ├─────────────────────────────────────────────────────┤ │
│  │                  Event Bus (事件总线)                │ │
│  ├─────────────────────────────────────────────────────┤ │
│  │              Extension Points (扩展点)               │ │
│  └─────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────┘
```

### 2.3 代码示例：IDE插件系统

```typescript
/**
 * 微内核架构示例：IDE插件系统
 *
 * 架构说明：
 * - Core System（核心系统）：提供基础编辑功能、插件管理
 * - Plugin System（插件系统）：提供语言支持、代码检查等扩展
 */

/**
 * ========== 核心系统（Core System）==========
 * 核心系统保持最小化，只包含绝对必要的功能
 */

/**
 * 核心接口：所有核心功能通过接口暴露
 */
interface CoreExtensionPoint {
  // 扩展点名称
  getName(): string;

  // 扩展点描述
  getDescription(): string;
}

/**
 * 插件接口：所有插件必须实现这个接口
 */
interface Plugin {
  // 插件ID
  getId(): string;

  // 插件名称
  getName(): string;

  // 插件版本
  getVersion(): string;

  // 插件描述
  getDescription(): string;

  // 初始化插件
  initialize(context: PluginContext): void;

  // 销毁插件
  dispose(): void;
}

/**
 * 插件上下文：插件与核心系统交互的接口
 */
class PluginContext {
  // 插件注册表
  private pluginRegistry: PluginRegistry;

  // 事件总线
  private eventBus: EventBus;

  constructor(registry: PluginRegistry, eventBus: EventBus) {
    this.pluginRegistry = registry;
    this.eventBus = eventBus;
  }

  /**
   * 注册扩展点实现
   */
  registerExtension(extensionPointId: string, extension: any): void {
    console.log(`插件上下文：注册扩展点 ${extensionPointId}`);
    this.pluginRegistry.registerExtension(extensionPointId, extension);
  }

  /**
   * 发布事件
   */
  publishEvent(event: string, data: any): void {
    this.eventBus.publish(event, data);
  }

  /**
   * 订阅事件
   */
  subscribeEvent(event: string, handler: (data: any) => void): void {
    this.eventBus.subscribe(event, handler);
  }

  /**
   * 获取配置
   */
  getConfig(key: string): any {
    return ConfigManager.get(key);
  }
}

/**
 * 事件总线：核心系统内的通信机制
 */
class EventBus {
  private handlers: Map<string, Set<(data: any) => void>> = new Map();

  /**
   * 订阅事件
   */
  subscribe(event: string, handler: (data: any) => void): void {
    if (!this.handlers.has(event)) {
      this.handlers.set(event, new Set());
    }
    this.handlers.get(event)!.add(handler);
  }

  /**
   * 发布事件
   */
  publish(event: string, data: any): void {
    console.log(`事件总线：发布事件 ${event}`);
    const handlers = this.handlers.get(event);
    if (handlers) {
      handlers.forEach(handler => {
        try {
          handler(data);
        } catch (error) {
          console.error(`事件处理器错误: ${error}`);
        }
      });
    }
  }

  /**
   * 取消订阅
   */
  unsubscribe(event: string, handler: (data: any) => void): void {
    const handlers = this.handlers.get(event);
    if (handlers) {
      handlers.delete(handler);
    }
  }
}

/**
 * 插件注册表：管理所有插件
 */
class PluginRegistry {
  // 已注册的插件
  private plugins: Map<string, Plugin> = new Map();

  // 扩展点注册表
  private extensions: Map<string, Set<any>> = new Map();

  // 核心系统的扩展点定义
  private extensionPoints: Map<string, CoreExtensionPoint> = new Map();

  /**
   * 注册核心扩展点
   */
  registerExtensionPoint(extensionPoint: CoreExtensionPoint): void {
    console.log(`注册扩展点: ${extensionPoint.getName()}`);
    this.extensionPoints.set(extensionPoint.getName(), extensionPoint);
  }

  /**
   * 注册扩展
   */
  registerExtension(extensionPointId: string, extension: any): void {
    if (!this.extensions.has(extensionPointId)) {
      this.extensions.set(extensionPointId, new Set());
    }
    this.extensions.get(extensionPointId)!.add(extension);
    console.log(`注册扩展到 ${extensionPointId}`);
  }

  /**
   * 安装插件
   */
  installPlugin(plugin: Plugin): void {
    if (this.plugins.has(plugin.getId())) {
      throw new Error(`插件 ${plugin.getId()} 已安装`);
    }

    console.log(`安装插件: ${plugin.getName()} v${plugin.getVersion()}`);

    const context = new PluginContext(this, new EventBus());
    plugin.initialize(context);

    this.plugins.set(plugin.getId(), plugin);
  }

  /**
   * 卸载插件
   */
  uninstallPlugin(pluginId: string): void {
    const plugin = this.plugins.get(pluginId);
    if (!plugin) {
      throw new Error(`插件 ${pluginId} 未安装`);
    }

    plugin.dispose();
    this.plugins.delete(pluginId);
    console.log(`卸载插件: ${plugin.getName()}`);
  }

  /**
   * 获取插件
   */
  getPlugin(pluginId: string): Plugin | undefined {
    return this.plugins.get(pluginId);
  }

  /**
   * 获取所有插件
   */
  getAllPlugins(): Plugin[] {
    return Array.from(this.plugins.values());
  }

  /**
```

## 三、事件驱动架构（Event-Driven Architecture）

### 3.1 什么是事件驱动架构？

事件驱动架构就像你的手机通知系统。手机不会不停地问你"有新消息吗？"，而是当有新消息时，系统主动通知你。事件驱动架构让系统中的各个组件通过"事件"进行通信，而不是直接调用。

**核心思想**：组件之间通过发布和订阅事件进行通信，发送者不知道谁会接收，接收者也不知道谁发送。

**生活类比**：想象餐厅的厨房。当你点了一道菜，服务员把你的订单（事件）交给厨房。厨房不知道也不关心你是谁，只负责做菜，做好了就通知服务员上菜。这就是事件驱动的思想。

```typescript
/**
 * 事件总线：核心事件通信机制
 */
class EventBus {
  // 订阅者Map：事件名 -> 处理器列表
  private subscribers: Map<string, Array<(event: DomainEvent) => void>> = new Map();

  /**
   * 订阅事件
   */
  subscribe(eventName: string, handler: (event: DomainEvent) => void): void {
    if (!this.subscribers.has(eventName)) {
      this.subscribers.set(eventName, []);
    }
    this.subscribers.get(eventName)!.push(handler);
    console.log(`事件总线：订阅事件 ${eventName}`);
  }

  /**
   * 取消订阅
   */
  unsubscribe(eventName: string, handler: (event: DomainEvent) => void): void {
    const handlers = this.subscribers.get(eventName);
    if (handlers) {
      const index = handlers.indexOf(handler);
      if (index > -1) {
        handlers.splice(index, 1);
      }
    }
  }

  /**
   * 发布事件
   */
  publish(event: DomainEvent): void {
    console.log(`\n事件总线：发布事件 ${event.eventName}`);
    console.log(`  事件ID: ${event.eventId}`);
    console.log(`  发生时间: ${event.occurredAt.toISOString()}`);

    const handlers = this.subscribers.get(event.eventName);
    if (handlers) {
      console.log(`  找到 ${handlers.length} 个订阅者`);
      handlers.forEach((handler, index) => {
        try {
          console.log(`  处理中 [${index + 1}/${handlers.length}]...`);
          handler(event);
        } catch (error) {
          console.error(`  处理错误: ${error}`);
        }
      });
    } else {
      console.log(`  没有找到订阅者`);
    }
  }

  /**
   * 批量发布事件
   */
  publishAll(events: DomainEvent[]): void {
    events.forEach(event => this.publish(event));
  }
}
```

### 3.4 事件驱动架构的优缺点

**优点**：
- 松耦合：发布者和订阅者不需要知道彼此
- 可扩展性：可以轻松添加新的订阅者
- 异步处理：事件可以异步处理，提高系统响应性
- 审计友好：所有事件都有记录，可追溯

**缺点**：
- 事件处理可能乱序
- 调试困难（异步调用链路不清晰）
- 事件爆炸：事件太多难以管理
- 事务处理复杂：跨多个事件的一致性

**适用场景**：
- 微服务架构中的服务间通信
- 实时系统（聊天、通知）
- 数据管道（ETL、实时分析）
- 事件溯源（Event Sourcing）

## 四、CQRS模式（Command Query Responsibility Segregation）

### 4.1 什么是CQRS？

CQRS（命令查询职责分离）就像图书馆的管理系统。借书和还书是一种操作（命令），查找图书目录是另一种操作（查询）。在传统架构中，我们用同一个数据模型处理读写。但在CQRS中，我们把读操作（查询）和写操作（命令）完全分开，使用不同的模型。

**核心思想**：将读取数据（查询）和修改数据（命令）分开，使用不同的模型和处理逻辑。

**生活类比**：想象你去银行。你存钱（命令）和查询余额（查询）走的是不同的通道，由不同的柜员处理，效率更高。

### 4.2 CQRS的架构

```
┌──────────────────────────────────────────────────────────────┐
│                         Client                               │
└──────────────────────────┬───────────────────────────────────┘
                           │
            ┌──────────────┴──────────────┐
            │                             │
            ▼                             ▼
┌───────────────────────────┐  ┌───────────────────────────────┐
│         Commands          │  │          Queries              │
│         (命令端)           │  │         (查询端)              │
├───────────────────────────┤  ├───────────────────────────────┤
│ - CreateOrder             │  │ - GetOrderSummary             │
│ - UpdateOrder             │  │ - GetOrderDetails             │
│ - CancelOrder             │  │ - GetCustomerOrderHistory      │
│ - Payment                 │  │ - GetDashboardData            │
└───────────────────────────┘  └───────────────────────────────┘
            │                             │
            ▼                             ▼
┌───────────────────────────┐  ┌───────────────────────────────┐
│      Command Handler      │  │       Query Handler           │
│       (命令处理器)          │  │       (查询处理器)             │
└───────────────────────────┘  └───────────────────────────────┘
            │                             │
            ▼                             ▼
┌───────────────────────────┐  ┌───────────────────────────────┐
│    Write Database         │  │     Read Database             │
│    (写入数据库)            │  │     (读取数据库)              │
│    (事务性存储)            │  │     (专门优化的读取模型)       │
└───────────────────────────┘  └───────────────────────────────┘
```

```typescript
 * CQRS模式示例：订单系统
 *
 * 架构说明：
 * - Command Side：处理创建、更新、删除等命令
 * - Query Side：处理数据查询，返回专门优化的视图
 * - 两个模型完全独立，各司其职

/**
 * ========== 命令端（Command Side）==========
 * 命令端专注于业务逻辑和数据写入
 * 使用标准的领域模型

/**
 * 订单命令 - 创建订单
 */
interface CreateOrderCommand {
  orderId: string;
  customerId: string;
  items: { productId: string; quantity: number; price: number }[];
  shippingAddress: string;
}

/**
 * 订单命令 - 支付订单
 */
interface PayOrderCommand {
  orderId: string;
  paymentMethod: string;
  paymentAmount: number;
}

/**
 * 订单命令 - 取消订单
 */
interface CancelOrderCommand {
  orderId: string;
  reason: string;

/**
 * 订单聚合根 - 领域模型
 */
class OrderAggregate {
  private orderId: string;
  private customerId: string;
  private status: string;
  private items: any[];
  private totalAmount: number;
  private paymentId: string | null;

  constructor(orderId: string, customerId: string) {
    this.orderId = orderId;
    this.customerId = customerId;
    this.status = 'CREATED';
    this.items = [];
    this.totalAmount = 0;
    this.paymentId = null;
  }

  /**
   * 添加订单项
   */
  addItem(productId: string, quantity: number, price: number): void {
    if (this.status !== 'CREATED') {
      throw new Error('只能在创建状态添加商品');
    }

    this.items.push({ productId, quantity, price });
    this.recalculateTotal();
  }

  private recalculateTotal(): void {
    this.totalAmount = this.items.reduce(
      (sum, item) => sum + item.price * item.quantity,
      0
    );
  }

  /**
   * 支付订单
   */
  pay(paymentId: string, paymentAmount: number): void {
    if (this.status !== 'CREATED') {
      throw new Error('订单状态不正确');
    }

    if (paymentAmount < this.totalAmount) {
      throw new Error('支付金额不足');
    }

    this.paymentId = paymentId;
    this.status = 'PAID';
  }

  /**
   * 取消订单
   */
  cancel(reason: string): void {
    if (this.status === 'SHIPPED' || this.status === 'COMPLETED') {
      throw new Error('已发货或完成的订单不能取消');
    }

    this.status = reason.includes('退款') ? 'REFUNDING' : 'CANCELLED';
  }

  // Getter方法
  getOrderId(): string { return this.orderId; }
  getStatus(): string { return this.status; }
  getTotalAmount(): number { return this.totalAmount; }
  getItems(): any[] { return [...this.items]; }

  // 转换为存储格式
  toStorageFormat(): any {
    return {
      orderId: this.orderId,
      customerId: this.customerId,
      status: this.status,
      items: this.items,
      totalAmount: this.totalAmount,
      paymentId: this.paymentId,
      updatedAt: new Date()
    };
  }
}

class OrderCommandHandler {
  // 模拟写入数据库
  private writeDb: Map<string, any> = new Map();

  /**
   * 处理创建订单命令
   */
  handleCreateOrder(command: CreateOrderCommand): void {
    console.log('\n命令处理器：处理 CreateOrderCommand');
    console.log(`  订单ID: ${command.orderId}`);

    // 创建订单聚合
    const order = new OrderAggregate(command.orderId, command.customerId);

    // 添加订单项
    for (const item of command.items) {
      order.addItem(item.productId, item.quantity, item.price);
    }

    // 保存到写入数据库
    this.writeDb.set(command.orderId, order.toStorageFormat());

    console.log(`  订单创建成功，总金额: ${order.getTotalAmount()}`);
  }

  /**
   * 处理支付命令
   */
  handlePayOrder(command: PayOrderCommand): void {
    console.log('\n命令处理器：处理 PayOrderCommand');
    console.log(`  订单ID: ${command.orderId}`);

    const orderData = this.writeDb.get(command.orderId);
    if (!orderData) {
      throw new Error('订单不存在');
    }

    // 更新订单状态
    orderData.status = 'PAID';
    orderData.paymentId = 'PAY-' + Date.now();
    orderData.paidAt = new Date();

    console.log(`  订单已支付，支付ID: ${orderData.paymentId}`);
  }

  /**
   * 处理取消命令
   */
  handleCancelOrder(command: CancelOrderCommand): void {
    console.log('\n命令处理器：处理 CancelOrderCommand');
    console.log(`  订单ID: ${command.orderId}`);
    console.log(`  取消原因: ${command.reason}`);

    const orderData = this.writeDb.get(command.orderId);
    if (!orderData) {
      throw new Error('订单不存在');
    }

    // 更新状态
    if (orderData.status === 'PAID') {
      orderData.status = 'REFUNDING';
    } else {
      orderData.status = 'CANCELLED';
    }

    console.log(`  订单已取消`);
  }
}
```

### 4.4 CQRS的优缺点

**优点**：
- 读写分离优化：读模型和写模型可以独立优化
- 性能提升：查询端可以使用专门优化的读取数据库
- 扩展性：读写可以独立扩展
- 灵活性：可以为不同查询需求创建不同的读取模型

**缺点**：
- 复杂度增加：需要处理数据同步
- 一致性挑战：读写模型之间可能存在延迟
- 需要事件源或同步机制
- 学习成本较高

**适用场景**：
- 读写比例差异大的系统
- 需要高性能读取的系统
- 复杂业务逻辑的系统
- 微服务架构

## 五、EDA事件驱动架构（Event-Driven Architecture）

### 5.1 什么是EDA？

EDA（Event-Driven Architecture，事件驱动架构）是一种专注于事件生产、检测、消费和响应的架构模式。与传统的事件驱动不同，EDA强调事件的持久化和事件溯源。

**核心思想**：系统中的所有业务操作都产生事件，这些事件被持久化存储，形成事件日志。系统的当前状态可以通过重放事件来重建。

**生活类比**：想象银行的存折。每笔交易（存款、取款、转账）都被记录在存折上。存折就是事件日志，你的余额是通过从第一笔交易开始逐笔计算得出的。

### 5.2 EDA vs 传统事件驱动

| 特性 | 传统事件驱动 | EDA |
|------|-------------|-----|
| 事件持久化 | 通常不持久化 | 必须持久化 |
| 事件溯源 | 不支持 | 支持 |
| 状态重建 | 需要额外存储 | 通过重放事件 |
| 审计能力 | 有限 | 完整 |

### 5.3 代码示例：EDA订单系统

```typescript
/**
 * EDA事件驱动架构示例：订单事件溯源系统
 *
 * 架构说明：

 * - Aggregate：从事件流重建状态
 * - Event Handler：处理事件副作用
 */

/**

/**

interface Event {

class OrderPaidEvent implements Event {

/**
 * ========== 事件存储（Event Store）==========
 * 事件存储是EDA的核心，它持久化所有事件
 */

class EventStore {
  // 事件存储：aggregateId -> events[]
  private events: Map<string, Event[]> = new Map();

  /**
   * 追加事件
   */
  append(event: Event): void {
    console.log(`事件存储：追加事件 ${event.eventType} -> ${event.aggregateId}`);

    if (!this.events.has(event.aggregateId)) {
      this.events.set(event.aggregateId, []);
    }

    this.events.get(event.aggregateId)!.push(event);
  }

  /**
   * 获取聚合的所有事件
   */
  getEvents(aggregateId: string): Event[] {
    return this.events.get(aggregateId) || [];
  }

  /**
   * 获取所有事件（用于投影）
   */
  getAllEvents(): Event[] {
    const allEvents: Event[] = [];
    for (const events of this.events.values()) {
      allEvents.push(...events);
    }
    return allEvents.sort((a, b) =>
      new Date(a.occurredAt).getTime() - new Date(b.occurredAt).getTime()
    );
  }

  /**
   * 获取特定类型的所有事件
   */
  getEventsByType(eventType: string): Event[] {
    return this.getAllEvents().filter(e => e.eventType === eventType);
  }
}

// 全局事件存储
const eventStore = new EventStore();

/**
 * ========== 聚合根（Aggregate）==========
 * 聚合根是状态的来源，通过重放事件来重建状态
 */

/**
 * 订单聚合根
 */
class OrderAggregate {
  // 状态
  private orderId: string;
  private customerId: string | null = null;
  private status: 'NEW' | 'PAID' | 'CANCELLED' = 'NEW';
  private items: { productId: string; quantity: number; price: number }[] = [];
  private paymentId: string | null = null;

  constructor(orderId: string) {
    this.orderId = orderId;
  }

  /**
   * 从事件流重建状态
   * 这是EDA的核心：通过重放事件来重建聚合根状态
   */
  static rebuild(events: Event[]): OrderAggregate {
    console.log(`\n订单聚合：重建聚合根，共 ${events.length} 个事件`);

    // 从第一个事件获取orderId
    const firstEvent = events[0] as any;
    const aggregate = new OrderAggregate(firstEvent?.aggregateId || 'UNKNOWN');

    // 按顺序重放每个事件
    for (const event of events) {
      aggregate.replayEvent(event);
    }

    return aggregate;
  }

  /**
   * 重放单个事件
   */
  private replayEvent(event: Event): void {
    switch (event.eventType) {
      case 'OrderCreated':
        this.onOrderCreated(event as OrderCreatedEvent);
        break;
      case 'OrderItemAdded':
        this.onOrderItemAdded(event as OrderItemAddedEvent);
        break;
      case 'OrderPaid':
        this.onOrderPaid(event as OrderPaidEvent);
        break;
      case 'OrderCancelled':
        this.onOrderCancelled(event as OrderCancelledEvent);
        break;
    }
  }

  private onOrderCreated(event: OrderCreatedEvent): void {
    this.customerId = event.customerId;
    this.items = [...event.items];
    this.status = 'NEW';
  }

  private onOrderItemAdded(event: OrderItemAddedEvent): void {
    this.items.push({
      productId: event.productId,
      quantity: event.quantity,
      price: event.price
    });
  }

  private onOrderPaid(event: OrderPaidEvent): void {
    this.status = 'PAID';
    this.paymentId = event.paymentId;
  }

  private onOrderCancelled(event: OrderCancelledEvent): void {
    this.status = 'CANCELLED';
  }

  // 命令方法（创建新状态，但不直接修改）
  createOrder(customerId: string, items: { productId: string; quantity: number; price: number }[]): Event {
    if (this.status !== 'NEW') {
      throw new Error('订单已创建');
    }
    return new OrderCreatedEvent(this.orderId, customerId, items);
  }

  addItem(productId: string, quantity: number, price: number): Event {
    return new OrderItemAddedEvent(this.orderId, productId, quantity, price);
  }

  pay(paymentId: string, paymentMethod: string): Event {
    if (this.status !== 'NEW') {
      throw new Error('订单状态不正确');
    }
    return new OrderPaidEvent(this.orderId, paymentId, paymentMethod);
  }

  cancel(reason: string): Event {
    if (this.status === 'PAID') {
      throw new Error('已支付的订单需要退款');
    }
    return new OrderCancelledEvent(this.orderId, reason);
  }

  // 获取当前状态
  getState() {
    return {
      orderId: this.orderId,
      customerId: this.customerId,
      status: this.status,
      items: [...this.items],
      paymentId: this.paymentId,
      totalAmount: this.items.reduce((sum, item) => sum + item.price * item.quantity, 0)
    };
  }
}

class OrderCommandHandler {
  /**
   * 处理创建订单命令

  createOrder(command: { orderId: string; customerId: string; items: any[] }): void {

  /**

  payOrder(command: { orderId: string; paymentId: string; paymentMethod: string }): void {

    eventStore.append(event);
  }
}

/**
 * 投影构建器：从事件流构建查询视图
 */
class ProjectionBuilder {
  /**
   * 构建订单列表投影
   */
  buildOrderListProjection(): any[] {
    console.log('\n投影构建：从事件流构建订单列表');

    const ordersById = new Map<string, any>();

    for (const event of eventStore.getAllEvents()) {
      if (event.eventType === 'OrderCreated') {
        const e = event as OrderCreatedEvent;
        ordersById.set(event.aggregateId, {
          orderId: event.aggregateId,
          customerId: e.customerId,
          status: 'NEW',
          items: [...e.items],
          totalAmount: e.items.reduce((sum, i) => sum + i.price * i.quantity, 0)
        });
      } else if (event.eventType === 'OrderPaid') {
        const order = ordersById.get(event.aggregateId);
        if (order) {
          order.status = 'PAID';
          order.paymentId = (event as OrderPaidEvent).paymentId;
        }
      } else if (event.eventType === 'OrderCancelled') {
        const order = ordersById.get(event.aggregateId);
        if (order) {
          order.status = 'CANCELLED';
        }
      }
    }

    return Array.from(ordersById.values());
  }

  /**
   * 获取订单当前状态（通过重放事件）
   */
  getOrderState(orderId: string): any {
    const events = eventStore.getEvents(orderId);
    const aggregate = OrderAggregate.rebuild(events);
    return aggregate.getState();
  }
}
```

### 5.4 EDA的优缺点

**优点**：
- 完整审计日志：所有操作都有记录
- 事件溯源：可以重建任意时间点的状态
- 时间旅行：可以回溯到过去分析问题
- 解耦：事件生产者和消费者完全解耦
- 灵活查询：可以从事件流构建不同的视图

**缺点**：
- 事件膨胀：长时间运行的系统事件会很多
- 性能挑战：重放大量事件可能很慢
- 事件变更困难：历史事件不易修改
- 复杂度高：需要处理幂等性、顺序性

**适用场景**：
- 金融系统（需要完整审计）
- 订单系统（需要事件溯源）
- 协作平台（多用户操作记录）
- 分布式系统（事件总线）

## 六、六边形架构（Hexagonal Architecture）

### 6.1 什么是六边形架构？

六边形架构（也称为端口和适配器架构）将系统分为"内部"和"外部"两部分。内部是核心业务逻辑（领域模型），外部是与外部系统交互的适配器（数据库、API、UI等）。六边形形象地表示了系统可以根据需要连接多个外部系统。

**核心思想**：核心业务逻辑（领域）不应该依赖外部基础设施，而应该通过端口（接口）进行交互。

**生活类比**：想象一个USB接口。U盘、鼠标、键盘都通过同一个USB接口连接到电脑。电脑（核心）不知道也不关心连接的是什么设备，只需要知道USB协议（端口）。六边形架构就是这种"插拔式"设计。

### 6.3 代码示例：六边形架构订单系统

```typescript
/**
 * 六边形架构示例：订单系统
 *
 * 架构说明：
 * - Domain（核心）：包含所有业务逻辑，不依赖任何外部
 * - Ports（端口）：定义核心与外部交互的接口
 * - Adapters（适配器）：实现端口，连接具体外部系统
 */

/**
 * ========== Domain（核心领域）==========
 * 这一层完全不依赖任何外部，只包含纯业务逻辑
 */

/**
 * 订单实体 - 领域模型
 */
class Order {
  private id: string;
  private customerId: string;
  private status: OrderStatus;
  private items: OrderItem[];
  private totalAmount: number;

  constructor(id: string, customerId: string) {
    this.id = id;
    this.customerId = customerId;
    this.status = OrderStatus.CREATED;
    this.items = [];
    this.totalAmount = 0;
  }

  /**
   * 添加订单项
   */
  addItem(productId: string, quantity: number, price: number): void {
    if (this.status !== OrderStatus.CREATED) {
      throw new Error('订单已支付后不能再添加商品');
    }

    this.items.push({ productId, quantity, price });
    this.recalculateTotal();
  }

  private recalculateTotal(): void {
    this.totalAmount = this.items.reduce(
      (sum, item) => sum + item.price * item.quantity,
      0
    );
  }

  /**
   * 支付
   */
  pay(): void {
    if (this.status !== OrderStatus.CREATED) {
      throw new Error('订单状态不正确');
    }

    if (this.items.length === 0) {
      throw new Error('订单不能为空');
    }

    this.status = OrderStatus.PAID;
  }

  /**
   * 取消
   */
  cancel(): void {
    if (this.status === OrderStatus.SHIPPED || this.status === OrderStatus.COMPLETED) {
      throw new Error('已发货或完成的订单不能取消');
    }

    this.status = this.status === OrderStatus.PAID
      ? OrderStatus.REFUNDING
      : OrderStatus.CANCELLED;
  }

  // Getters
  getId(): string { return this.id; }
  getCustomerId(): string { return this.customerId; }
  getStatus(): OrderStatus { return this.status; }
  getTotalAmount(): number { return this.totalAmount; }
  getItems(): OrderItem[] { return [...this.items]; }
}

interface OrderItem {
  productId: string;
  quantity: number;
  price: number;
}

enum OrderStatus {
  CREATED = 'created',
  PAID = 'paid',
  SHIPPED = 'shipped',
  COMPLETED = 'completed',
  CANCELLED = 'cancelled',
  REFUNDING = 'refunding'
}

/**
 * ========== Ports（端口）==========
 * 端口是核心领域定义的接口，不包含实现
 * 入站端口：外部调用核心的接口
 * 出站端口：核心调用外部的接口
 */

/**
 * 入站端口：订单服务接口
 * 定义了外部可以调用的操作
 */
interface OrderServicePort {
  createOrder(customerId: string, items: { productId: string; quantity: number; price: number }[]): Promise<string>;
  payOrder(orderId: string): Promise<void>;
  cancelOrder(orderId: string): Promise<void>;
  getOrder(orderId: string): Promise<OrderDTO | null>;
}

/**
 * 出站端口：订单仓储接口
 * 核心需要持久化时调用的接口
 */
interface OrderRepositoryPort {
  save(order: Order): Promise<void>;
  findById(orderId: string): Promise<Order | null>;
}

/**
 * 出站端口：支付网关接口
 * 核心需要支付时调用的接口
 */
interface PaymentGatewayPort {
  processPayment(amount: number, method: string): Promise<PaymentResult>;
}

/**
 * 出站端口：消息通知接口
 * 核心需要发送通知时调用的接口
 */
interface NotificationPort {
  sendNotification(customerId: string, message: string): Promise<void>;
}

/**
 * ========== DTO（数据传输对象）==========
 */

/**
 * 订单DTO - 用于向外部展示数据
 */
interface OrderDTO {
  orderId: string;
  customerId: string;
  status: string;
  items: { productId: string; quantity: number; price: number }[];
  totalAmount: number;
}

/**
 * ========== Adapters（适配器）==========
 * 适配器实现端口，连接具体外部系统
 */

/**
 * 订单服务适配器 - 实现入站端口
 * 协调各组件完成业务流程
 */
class OrderServiceAdapter implements OrderServicePort {
  constructor(
    private orderRepository: OrderRepositoryPort,
    private paymentGateway: PaymentGatewayPort,
    private notificationPort: NotificationPort
  ) {}

  async createOrder(
    customerId: string,
    items: { productId: string; quantity: number; price: number }[]
  ): Promise<string> {
    console.log('\n[OrderServiceAdapter] 创建订单');

    const order = new Order('ORD-' + Date.now(), customerId);
    items.forEach(item => order.addItem(item.productId, item.quantity, item.price));

    await this.orderRepository.save(order);

    await this.notificationPort.sendNotification(
      customerId,
      `订单已创建，总金额 ¥${order.getTotalAmount()}`
    );

    return order.getId();
  }

  async payOrder(orderId: string): Promise<void> {
    console.log(`\n[OrderServiceAdapter] 支付订单 ${orderId}`);

    const order = await this.orderRepository.findById(orderId);
    if (!order) {
      throw new Error('订单不存在');
    }

    const paymentResult = await this.paymentGateway.processPayment(
      order.getTotalAmount(),
      'credit_card'
    );

    order.pay();
    await this.orderRepository.save(order);

    await this.notificationPort.sendNotification(
      order.getCustomerId(),
      `订单已支付，支付流水号 ${paymentResult.transactionId}`
    );
  }

  async cancelOrder(orderId: string): Promise<void> {
    console.log(`\n[OrderServiceAdapter] 取消订单 ${orderId}`);

    const order = await this.orderRepository.findById(orderId);
    if (!order) {
      throw new Error('订单不存在');
    }

    order.cancel();
    await this.orderRepository.save(order);

    await this.notificationPort.sendNotification(
      order.getCustomerId(),
      `订单已取消`
    );
  }

  async getOrder(orderId: string): Promise<OrderDTO | null> {
    console.log(`\n[OrderServiceAdapter] 查询订单 ${orderId}`);

    const order = await this.orderRepository.findById(orderId);
    if (!order) {
      return null;
    }

    return {
      orderId: order.getId(),
      customerId: order.getCustomerId(),
      status: order.getStatus(),
      items: order.getItems(),
      totalAmount: order.getTotalAmount()
    };
  }
}

/**
 * MySQL仓储适配器 - 实现出站端口
 */
class MySQLOrderRepository implements OrderRepositoryPort {
  private db: Map<string, Order> = new Map();

  async save(order: Order): Promise<void> {
    console.log(`[MySQL] 保存订单 ${order.getId()}`);
    this.db.set(order.getId(), order);
  }

  async findById(orderId: string): Promise<Order | null> {
    console.log(`[MySQL] 查询订单 ${orderId}`);
    return this.db.get(orderId) || null;
  }
}

/**
 * 支付宝网关适配器 - 实现出站端口
 */
class AlipayGateway implements PaymentGatewayPort {
  async processPayment(amount: number, method: string): Promise<PaymentResult> {
    console.log(`[Alipay] 处理支付 ¥${amount} via ${method}`);
    await new Promise(resolve => setTimeout(resolve, 100));
    return {
      success: true,
      transactionId: 'TXN-' + Date.now()
    };
  }
}

/**
 * 邮件通知适配器 - 实现出站端口
 */
class EmailNotification implements NotificationPort {
  async sendNotification(customerId: string, message: string): Promise<void> {
    console.log(`[Email] 发送通知给 ${customerId}: ${message}`);
  }
}

/**
 * ========== 依赖注入容器 ==========
 */

/**
 * 简单依赖注入容器
 */
class DIContainer {
  private services: Map<string, any> = new Map();

  register<T>(name: string, instance: T): void {
    this.services.set(name, instance);
  }

  resolve<T>(name: string): T {
    return this.services.get(name);
  }
}

// 创建并配置容器
const container = new DIContainer();

// 注册适配器
container.register('OrderRepository', new MySQLOrderRepository());
container.register('PaymentGateway', new AlipayGateway());
container.register('Notification', new EmailNotification());

// 创建服务
const orderService = new OrderServiceAdapter(
  container.resolve('OrderRepository'),
  container.resolve('PaymentGateway'),
  container.resolve('Notification')
);

// 测试六边形架构
function testHexagonalArchitecture() {
  console.log('=== 六边形架构测试：订单系统 ===\n');

  // 1. 创建订单
  console.log('>>> 创建订单 <<<');
  orderService.createOrder('CUST-001', [
    { productId: 'P001', quantity: 1, price: 6999 },
    { productId: 'P002', quantity: 2, price: 999 }
  ]);

  // 2. 支付订单
  console.log('\n>>> 支付订单 <<<');
  orderService.payOrder('ORD-' + Date.now());

  // 3. 查询订单
  console.log('\n>>> 查询订单 <<<');
  orderService.getOrder('ORD-' + Date.now());

  // 4. 取消订单
  console.log('\n>>> 取消订单 <<<');
```

### 6.4 六边形架构的优缺点

**优点**：
- 高度解耦：核心业务逻辑与外部系统完全解耦
- 可测试性：核心逻辑可以独立测试
- 可替换性：可以轻松替换外部系统（换适配器）
- 清晰架构：端口和适配器职责分明

**缺点**：
- 复杂度增加：需要处理依赖注入
- 学习成本：端口-适配器的概念需要理解
- 可能过度设计：对于简单系统可能太复杂

**适用场景**：
- 需要集成多个外部系统
- 核心逻辑需要高度稳定
- 需要高可测试性
- DDD项目

## 七、洋葱架构（Onion Architecture）

### 7.1 什么是洋葱架构？

洋葱架构是六边形架构的进化版本，强调从内到外的依赖方向。核心业务逻辑在最内层，外层依次是应用服务、端口、适配器。依赖只能从外向内，不能反向。

**核心思想**：越在内层越稳定，越在外层越容易变化。依赖方向始终指向圆心。

**生活类比**：想象洋葱，从外向内一层一层剥开。最中心是你真正想要的（核心），外面都是保护层。洋葱架构就是这样，最核心的是你的业务逻辑，外面层层包装着各种基础设施。

### 7.2 洋葱架构的层次

```
┌─────────────────────────────────────────────────────────────┐
│                    Infrastructure                           │
│                   (基础设施层/最外层)                        │
│   Web适配器、数据库适配器、消息队列适配器                      │
├─────────────────────────────────────────────────────────────┤
│                    Ports Layer                             │
│                      (端口层)                               │
│   入站端口（API接口）、出站端口（仓储接口）                    │
├─────────────────────────────────────────────────────────────┤
│                   Application Layer                         │
│                     (应用层)                                │
│   应用服务、用例、命令处理器                                   │
├─────────────────────────────────────────────────────────────┤
│                      Domain Layer                          │
│                      (领域层)                               │
│   领域实体、值对象、领域服务、领域事件                         │
├─────────────────────────────────────────────────────────────┤
│                    Core (核心)                              │
│                     (最内层)                                │
│   业务规则、业务不变式                                        │
└─────────────────────────────────────────────────────────────┘
```

### 7.3 代码示例：洋葱架构订单系统

```typescript
/**
 * 洋葱架构示例：订单系统
 *
 * 架构层次（从内到外）：
 * 1. Domain Core（核心）：最纯粹的领域规则
 * 2. Domain Layer（领域层）：领域实体、领域服务
 * 3. Application Layer（应用层）：用例、应用服务
 * 4. Ports Layer（端口层）：接口定义
 * 5. Infrastructure Layer（基础设施层）：适配器实现
 */

/**
 * ========== Domain Core（核心层）==========
 * 包含最纯粹的领域规则和业务不变式
 * 这一层不依赖任何其他层
 */

/**
 * 业务不变式：定义领域的基本约束
 */
class OrderBusinessRules {
  /**
   * 验证订单是否可以支付
   */
  static canPay(status: string): boolean {
    return status === 'CREATED';
  }

  /**
   * 验证订单是否可以取消
   */
  static canCancel(status: string): boolean {
    return ['CREATED', 'PAID'].includes(status);
  }

  /**
   * 验证是否需要退款
   */
  static needsRefund(status: string): boolean {
    return status === 'PAID';
  }

  /**
   * 计算订单总价
   */
  static calculateTotal(items: { price: number; quantity: number }[]): number {
    return items.reduce((sum, item) => sum + item.price * item.quantity, 0);
  }
}

/**
 * ========== Domain Layer（领域层）==========
 * 包含领域实体和领域服务
 */

/**
 * 订单项值对象
 */
class OrderItem {
  constructor(
    public readonly productId: string,
    public readonly productName: string,
    public readonly price: number,
    public readonly quantity: number
  ) {
    // 值对象的相等性基于属性值
  }

  /**
   * 计算小计
   */
  getSubtotal(): number {
    return this.price * this.quantity;
  }
}

/**
 * 订单状态值对象
 */
enum OrderStatus {
  CREATED = 'CREATED',
  PAID = 'PAID',
  CANCELLED = 'CANCELLED',
  REFUNDING = 'REFUNDING'
}

/**
 * 订单实体
 */
class OrderEntity {
  private _id: string;
  private _customerId: string;
  private _status: OrderStatus;
  private _items: OrderItem[];
  private _createdAt: Date;

  constructor(id: string, customerId: string) {
    this._id = id;
    this._customerId = customerId;
    this._status = OrderStatus.CREATED;
    this._items = [];
    this._createdAt = new Date();
  }

  // 聚合方法
  addItem(item: OrderItem): void {
    if (this._status !== OrderStatus.CREATED) {
      throw new Error('只能在创建状态添加商品');
    }
    this._items.push(item);
  }

  pay(): void {
    if (!OrderBusinessRules.canPay(this._status)) {
      throw new Error('订单状态不允许支付');
    }
    if (this._items.length === 0) {
      throw new Error('订单不能为空');
    }
    this._status = OrderStatus.PAID;
  }

  cancel(): void {
    if (!OrderBusinessRules.canCancel(this._status)) {
      throw new Error('订单状态不允许取消');
    }
    this._status = OrderBusinessRules.needsRefund(this._status)
      ? OrderStatus.REFUNDING
      : OrderStatus.CANCELLED;
  }

  // Getters
  get id(): string { return this._id; }
  get customerId(): string { return this._customerId; }
  get status(): OrderStatus { return this._status; }
  get items(): OrderItem[] { return [...this._items]; }
  get totalAmount(): number {
    return OrderBusinessRules.calculateTotal(this._items);
  }
  get createdAt(): Date { return this._createdAt; }
}

/**
 * ========== Application Layer（应用层）==========
 * 包含用例和应用服务
 */

/**
 * 创建订单命令
 */
interface CreateOrderCommand {
  customerId: string;
  items: { productId: string; productName: string; price: number; quantity: number }[];
}

/**
 * 订单DTO
 */
interface OrderDTO {
  id: string;
  customerId: string;
  status: string;
  items: { productId: string; productName: string; price: number; quantity: number; subtotal: number }[];
  totalAmount: number;
  createdAt: string;
}

/**
 * 应用服务
 */
class OrderApplicationService {
  constructor(
    private orderRepository: IOrderRepository,
    private paymentService: IPaymentService,
    private notificationService: INotificationService
  ) {}

  /**
   * 创建订单用例
   */
  async createOrder(command: CreateOrderCommand): Promise<OrderDTO> {
    console.log('\n[应用服务] 执行创建订单用例');

    // 创建订单实体
    const orderId = 'ORD-' + Date.now();
    const order = new OrderEntity(orderId, command.customerId);

    // 添加订单项
    command.items.forEach(item => {
      order.addItem(new OrderItem(
        item.productId,
        item.productName,
        item.price,
        item.quantity
      ));
    });

    // 持久化
    await this.orderRepository.save(order);

    // 发送通知
    await this.notificationService.notify(
      command.customerId,
      `订单已创建，总金额 ¥${order.totalAmount}`
    );

    return this.toDTO(order);
  }

  /**
   * 支付订单用例
   */
  async payOrder(orderId: string): Promise<OrderDTO> {
    console.log(`\n[应用服务] 执行支付订单用例 ${orderId}`);

    // 获取订单
    const order = await this.orderRepository.findById(orderId);
    if (!order) {
      throw new Error('订单不存在');
    }

    // 处理支付
    await this.paymentService.processPayment(order.totalAmount);

    // 更新状态
    order.pay();
    await this.orderRepository.save(order);

    // 发送通知
    await this.notificationService.notify(
      order.customerId,
      `订单已支付`
    );

    return this.toDTO(order);
  }

  /**
   * 取消订单用例
   */
  async cancelOrder(orderId: string): Promise<OrderDTO> {
    console.log(`\n[应用服务] 执行取消订单用例 ${orderId}`);

    const order = await this.orderRepository.findById(orderId);
    if (!order) {
      throw new Error('订单不存在');
    }

    order.cancel();
    await this.orderRepository.save(order);

    await this.notificationService.notify(
      order.customerId,
      `订单已取消`
    );

    return this.toDTO(order);
  }

  /**
   * 转换为DTO
   */
  private toDTO(order: OrderEntity): OrderDTO {
    return {
      id: order.id,
      customerId: order.customerId,
      status: order.status,
      items: order.items.map(item => ({
        productId: item.productId,
        productName: item.productName,
        price: item.price,
        quantity: item.quantity,
        subtotal: item.getSubtotal()
      })),
      totalAmount: order.totalAmount,
      createdAt: order.createdAt.toISOString()
    };
  }
}

/**
 * ========== Ports Layer（端口层）==========
 * 定义接口（依赖倒置）
 */

/**
 * 订单仓储端口
 */
interface IOrderRepository {
  save(order: OrderEntity): Promise<void>;
  findById(orderId: string): Promise<OrderEntity | null>;
}

/**
 * 支付服务端口
 */
interface IPaymentService {

}

/**
 * 通知服务端口
 */
interface INotificationService {
  notify(customerId: string, message: string): Promise<void>;
}

/**
 * ========== Infrastructure Layer（基础设施层）==========
 * 实现端口层定义的接口
 */

/**
 * 内存订单仓储
 */
class InMemoryOrderRepository implements IOrderRepository {
  private db: Map<string, OrderEntity> = new Map();

  async save(order: OrderEntity): Promise<void> {
    console.log(`[仓储] 保存订单 ${order.id}`);
    this.db.set(order.id, order);
  }

  async findById(orderId: string): Promise<OrderEntity | null> {
    console.log(`[仓储] 查询订单 ${orderId}`);
    return this.db.get(orderId) || null;
  }
}

/**

class MockPaymentService implements IPaymentService {

    await new Promise(resolve => setTimeout(resolve, 100));
    return { transactionId: 'TXN-' + Date.now() };
  }
}

  }
}
```

### 7.4 洋葱架构的优缺点

**优点**：
- 依赖方向清晰：始终指向内层
- 可测试性高：核心逻辑可以独立测试
- 稳定性高：核心层不依赖外层
- 可替换性：外层可以轻松替换

**缺点**：
- 复杂度较高
- 需要良好的依赖注入
- 对于小项目可能过度设计

**适用场景**：
- 大型企业应用
- DDD项目
- 需要长期维护的系统
- 测试驱动开发（TDD）

## 八、架构模式对比总结

### 8.1 各模式对比

| 架构模式 | 核心思想 | 适用场景 | 复杂度 |
|---------|---------|---------|-------|
| 分层架构 | 职责分层 | 企业应用、传统Web | 中 |
| 微内核架构 | 核心+插件 | 平台产品、可扩展系统 | 中 |
| 事件驱动 | 事件通信 | 实时系统、微服务 | 高 |
| CQRS | 读写分离 | 高性能查询系统 | 高 |
| EDA | 事件溯源 | 金融系统、需要审计 | 高 |
| 六边形架构 | 端口适配 | 集成多个外部系统 | 中 |
| 洋葱架构 | 依赖向内 | DDD项目、可测试系统 | 中高 |

### 8.2 模式选择指南

```
问题：我需要选择什么架构？
│
├── 我的系统需要集成多个外部系统吗？
│   ├── 是 → 六边形架构
│   └── 否 ↓
│
├── 我的系统需要高可扩展性吗？
│   ├── 是 → 微内核架构
│   └── 否 ↓
│
├── 我的系统读写比例差异大吗？
│   ├── 是 → CQRS
│   └── 否 ↓
│
├── 我的系统需要完整审计吗？
│   ├── 是 → EDA
│   └── 否 ↓
│
├── 我的系统需要实时响应吗？
│   ├── 是 → 事件驱动架构
│   └── 否 → 分层架构（保守选择）
│
└── 我的系统是DDD项目吗？
    ├── 是 → 洋葱架构
    └── 否 → 分层架构
```

### 8.3 模式组合建议

在实际项目中，架构模式通常会组合使用：

1. **分层 + 六边形**：分层组织代码，六边形处理外部集成
2. **CQRS + EDA**：CQRS处理读写分离，EDA处理事件同步
3. **微内核 + 事件驱动**：微内核提供核心功能，事件驱动实现插件通信
4. **洋葱 + 六边形**：洋葱组织依赖，六边形处理适配

## 九、架构模式实践建议

### 9.1 开始新项目

1. **从分层架构开始**：简单、清晰、够用
2. **预留扩展点**：为未来可能的变化留有余地
3. **不要过度设计**：YAGNI原则（You Aren't Gonna Need It）

### 9.2 改造现有项目

1. **识别核心业务**：找出系统中最重要的领域逻辑
2. **渐进式重构**：不要一次性重写
3. **保持向后兼容**：确保现有功能正常

### 9.3 架构决策

1. **考虑团队能力**：架构要匹配团队的技术水平
2. **考虑运维成本**：越复杂的架构运维成本越高
3. **考虑业务复杂度**：简单业务用简单架构
