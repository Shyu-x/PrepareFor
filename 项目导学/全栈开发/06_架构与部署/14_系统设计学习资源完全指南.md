```
                │         │        │
           ┌────▼──────────▼───────▼──┐
           │        Databases         │
           │   (主从复制/分片)         │
           └─────────────────────────┘
```

### 2. 负载均衡算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **Round Robin** | 轮询分配 | 简单 | 不考虑服务器负载 |
| **Weighted RR** | 权重轮询 | 可控分配比例 | 需手动配置权重 |
| **Least Connections** | 最少连接数 | 动态平衡 | 实现复杂 |
| **IP Hash** | IP地址哈希 | 会话粘性 | 负载不均 |
| **Random** | 随机分配 | 简单 | 不保证均匀 |

### 3. 缓存策略

```
┌────────────────────────────────────────────────────────────┐
│                      缓存多级分类                           │
├────────────────────────────────────────────────────────────┤
│  L1 Cache (本地)    │ CPU L1/L2/L3 │ 纳秒级 │ KB-MB       │
│  L2 Cache (分布式)  │ Redis/Memcached │ 微秒级 │ GB       │
│  L3 Cache (CDN)     │ 全球节点     │ 毫秒级 │ TB          │
└────────────────────────────────────────────────────────────┘
```

**缓存淘汰策略**：
- LRU（最近最少使用）
- LFU（最不经常使用）
- FIFO（先进先出）
- TTL（时间过期）

**缓存一致性问题**：
- Cache Aside（旁路缓存）
- Read Through（读穿透）
- Write Through（写穿透）
- Write Behind（异步写入）

### 1. 设计短链接服务（URL Shortener）

**典型问题**：设计类似TinyURL的短链接服务

**核心需求分析**：

| 功能 | 需求描述 | 预估QPS |

| 短链接生成 | 6-8位随机字符串 | 10k/s |

| 短链接详情 | 点击统计、有效期 | 1k/s |

**架构设计**：

```python
# 短链接生成算法
import hashlib
import base62

def generate_short_url(long_url, chars=6):
    """生成指定长度的短链接"""
    # 方法1：哈希+Base62编码
    hash_obj = hashlib.md5(long_url.encode())
    hash_int = int(hash_obj.hexdigest(), 16)
    return base62.encode(hash_int)[:chars]

def generate_short_url_unique(long_url, counter, chars=6):
    """唯一短链接生成（基于计数器）"""
    return base62.encode(counter)[:chars]
```

### 4. 设计秒杀系统

**核心挑战**：

```
┌────────────────────────────────────────────────────────────┐
│                      秒杀系统挑战                           │
├────────────────────────────────────────────────────────────┤
│  1. 高并发        │ 瞬时流量洪峰，远超平时                  │
│  2. 超卖问题      │ 库存有限，如何防止超卖                 │
│  3. 一致性        │ 库存扣减与订单创建的一致性              │
│  4. 系统可用性    │ 如何保证高可用                         │
│  5. 反作弊        │ 防止恶意抢购、机器脚本                 │
└────────────────────────────────────────────────────────────┘
```

**架构方案**：

```python
# 秒杀系统核心流程
class FlashSaleService:
    """秒杀服务"""

    def prepare(self, product_id, stock):
        """秒杀前的准备工作"""
        # 1. 预热库存到Redis
        redis.set(f"stock:{product_id}", stock)

        # 2. 限流令牌桶
        redis.execute('pipeline', [
            ['set', f'rate:{product_id}:tokens', 1000],
            ['expire', f'rate:{product_id}:tokens', 60]
        ])

        # 3. 标记活动开始
        redis.set(f'seckill:{product_id}:active', 1)

    def seckill(self, user_id, product_id):
        """执行秒杀"""
        # 1. 用户限流（同一人频率限制）
        if not self.check_user_rate_limit(user_id, product_id):
            raise SeckillException('请求过于频繁')

        # 2. 活动有效性检查
        if not redis.exists(f'seckill:{product_id}:active'):
            raise SeckillException('秒杀未开始')

        # 3. 库存扣减（原子操作）
        stock = redis.decr(f"stock:{product_id}")
        if stock < 0:
            redis.incr(f"stock:{product_id}")  # 回补
            raise SeckillException('库存不足')

        # 4. 创建订单（异步）
        self.create_order_async(user_id, product_id)

        return True

    def check_user_rate_limit(self, user_id, product_id):
        """用户限流：每秒最多1次请求"""
        key = f'rate:user:{user_id}:{product_id}'
        result = redis.set(key, 1, nx=True, ex=1)
        return result is not None
```

### 6. 设计搜索建议系统

**功能拆解**：

| 组件 | 功能 | 技术选型 |
|------|------|----------|
| 词库 | 热词、拼音、错别字 | Trie树、词典 |
| 索引 | 前缀匹配、相关性 | Elasticsearch |
| 排序 | 点击率、相关度 | 机器学习模型 |
| 缓存 | TopK热词 | Redis |

**Trie树实现**：

```python
class TrieNode:
    def __init__(self):
        self.children = {}
        self.is_end = False
        self.hot_score = 0  # 热度分数

class TrieTree:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word, score=1):
        """插入词语及其热度"""
        node = self.root
        for char in word:
            if char not in node.children:
                node.children[char] = TrieNode()
            node = node.children[char]
            node.hot_score += score
        node.is_end = True

    def search_prefix(self, prefix, limit=10):
        """搜索前缀匹配的topK词语"""
        node = self.root
        for char in prefix:
            if char not in node.children:
                return []
            node = node.children[char]

        # DFS收集所有匹配的词语
        results = []
        self._dfs_collect(node, prefix, results)

        # 按热度排序
        results.sort(key=lambda x: x[1], reverse=True)
        return results[:limit]

    def _dfs_collect(self, node, path, results):
        if node.is_end:
            results.append((path, node.hot_score))
        for char, child in node.children.items():
            self._dfs_collect(child, path + char, results)
```

**秒杀系统回答框架**：

```
1. 需求澄清（2分钟）
   - 明确QPS峰值、库存量、活动时长
   - 确定核心功能范围

2. 总体设计（5分钟）
   - 画出高层架构图
   - 列出关键组件

3. 核心设计（10分钟）
   - 库存预热方案
   - 多层限流策略
   - 异步订单处理
   - 缓存一致性

4. 优化与扩展（3分钟）
   - 如何防止刷单
   - 如何做压测
   - 如何监控
```

## 实践方法论

### 1. 四步分析法

```
┌────────────────────────────────────────────────────────────┐
│                  系统设计四步分析法                          │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Step 1: 需求澄清 (Requirements Clarification)            │
│  ├─ 功能性需求                                             │
│  │   ├─ 用户是谁？                                         │
│  │   ├─ 核心功能有哪些？                                    │
│  │   └─ 数据规模多大？                                      │
│  └─ 非功能性需求                                           │
│      ├─ 需要支持多少QPS？                                   │
│      ├─ 需要多少可用性？                                    │
│      └─ 延迟要求是什么？                                    │
│                                                            │
│  Step 2: 高层设计 (High-Level Design)                      │
│  ├─ 系统架构图                                              │
│  ├─ 数据模型                                                │
│  └─ API设计                                                │
│                                                            │
│  Step 3: 核心组件设计 (Detailed Design)                     │
│  ├─ 核心组件的详细设计                                       │
│  ├─ 数据存储方案                                            │
│  └─ 关键算法                                                │
│                                                            │
│  Step 4: 权衡与扩展 (Trade-offs & Scalability)              │
│  ├─ 方案的优缺点                                            │
│  ├─ 如何监控                                                │
│  └─ 如何优化                                                │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 2. 自我评估清单

完成设计后，对照检查：

```
□ 需求是否完整澄清？
□ 架构图是否清晰？
□ 扩展性是否考虑？
□ 可用性是否保障？
□ 一致性如何保证？
□ 性能指标是否明确？
□ 备选方案是否考虑？
□ 权衡取舍是否说明？
□ 监控告警是否覆盖？
□ 容灾备份是否考虑？
```

### 3. 练习方法

1. **每日一题**：每天设计一个系统，做笔记
2. **对比学习**：先自己想，再看答案，对比差异
3. **画图练习**：用draw.io或Excalidraw画架构图
4. **项目实践**：参与开源项目或自己实现
5. **模拟面试**：找伙伴互相练习

### 4. 推荐练习题目顺序

```
入门级（1-2周）：
  1. 设计短链接服务
  2. 设计URL访问计数器

  9. 设计分布式锁

大师级（持续）：

```

## 总结

系统设计是后端工程师进阶的必经之路，需要：

1. **扎实的基础知识**：网络、数据库、操作系统、分布式理论
2. **丰富的实践经验**：多读源码，多做项目，多总结
3. **系统的学习方法**：循序渐进，由浅入深，持续迭代
4. **良好的表达能趑**：清晰表达设计思路，善于沟通协作

**推荐学习路径**：

```
第1周 │ 系统设计Primer通读 + 基础概念学习
第2周 │ 10道入门级题目练习
第3周 │ 10道进阶级题目练习
第4周 │ 复杂系统深入研究
持续  │ 跟进新技术，保持学习
```

---

**最后提醒**：系统设计没有标准答案，重要的是过程而非结果。要学会权衡取舍，理解不同方案的优缺点，在面试中展现自己的思考过程。
