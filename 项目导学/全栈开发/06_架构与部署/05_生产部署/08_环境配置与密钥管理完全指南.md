# 环境配置与密钥管理完全指南

## 前言：为什么密钥管理如此重要？

想象一下，你家门上有一把锁，但钥匙却贴在门口的便利贴上写着"备用钥匙在这里"。这就是很多团队处理密钥的方式——把最敏感的东西放在最容易被发现的地方。

密钥管理不当的后果：
- **2019年**: 某知名云服务商因为GitHub上的密钥泄露，导致5亿用户数据泄露
- **2020年**: 某电商平台工程师把数据库密码提交到GitHub，被黑客利用，泄露千万用户信息
- **2021年**: 某社交媒体公司内部密钥泄露，导致系统被入侵

这些不是技术问题，而是**管理问题**。让我们学习如何正确管理密钥和环境配置。

## 一、环境配置管理

### 1.1 什么是环境配置？

环境配置就像**餐厅的菜谱**——告诉厨师（应用程序）应该用什么食材（数据库）、什么火候（端口）、什么调味品（API密钥）。不同的餐厅（开发环境、测试环境、生产环境）可能用不同的配方。

### 1.2 环境类型与配置

```bash
# ========== 环境类型详解 ==========
#
# 开发环境 (Development)
# - 用途：日常开发、调试
# - 特点：修改频繁、日志详细、允许不稳定

# - 特点：与生产环境完全一致的配置
# - 数据：使用脱敏后的生产数据副本
#
# 生产环境 (Production)
# - 用途：真实用户使用
# - 特点：最保守的配置、最严格的访问
```

### 1.4 多环境配置加载

```typescript
// lib/config-loader.ts
// 多环境配置加载器

import * as fs from 'fs';
import * as path from 'path';
import yaml from 'js-yaml';

/**
 * 配置加载器
 * 按照以下优先级加载配置（后面的覆盖前面的）：
 * 1. 默认配置 (defaults/)
 * 2. 环境特定配置 (development/, production/)
 * 3. 本地覆盖 (.env.local, .local.yaml)
 */

interface AppConfig {
  app: {
    name: string;
    version: string;
    port: number;
    host: string;
  };
  logging: {
    level: string;
    format: string;
    output: string;
  };
  features: Record<string, boolean | string | number>;
  api: {
    timeout: number;
    retryAttempts: number;
    rateLimit: {
      windowMs: number;
      maxRequests: number;
    };
  };
  security?: {
    corsOrigins: string[];
    hstsEnabled: boolean;
  };
}

interface ConfigLoaderOptions {
  /** 环境名称：development, staging, production */
  env: string;
  /** 配置文件目录 */
  configDir: string;
  /** 是否允许本地覆盖 */
  allowLocalOverride: boolean;
}

/**
 * 深度合并两个对象
 * 用途：将环境特定配置合并到默认配置上
 */
function deepMerge<T extends Record<string, any>>(
  base: T,
  override: Partial<T>
): T {
  const result = { ...base };

  for (const key in override) {
    const baseValue = base[key];
    const overrideValue = override[key];

    // 如果两个值都是对象，递归合并
    if (
      typeof baseValue === 'object' &&
      baseValue !== null &&
      typeof overrideValue === 'object' &&
      overrideValue !== null &&
      !Array.isArray(baseValue) &&
      !Array.isArray(overrideValue)
    ) {
      result[key] = deepMerge(baseValue, overrideValue);
    } else {
      // 否则用override的值覆盖
      result[key] = overrideValue as any;
    }
  }

  return result;
}

/**
 * 加载配置文件
 */
function loadConfig(options: ConfigLoaderOptions): AppConfig {
  const { env, configDir, allowLocalOverride } = options;

  // 1. 加载默认配置
  const defaultConfigPath = path.join(configDir, 'defaults', 'app.yaml');
  let config: AppConfig = yaml.load(
    fs.readFileSync(defaultConfigPath, 'utf8')
  ) as AppConfig;

  // 2. 加载环境特定配置
  const envConfigPath = path.join(configDir, env, 'app.yaml');
  if (fs.existsSync(envConfigPath)) {
    const envConfig = yaml.load(
      fs.readFileSync(envConfigPath, 'utf8')
    ) as Partial<AppConfig>;
    config = deepMerge(config, envConfig);
  }

  // 3. 加载本地覆盖（仅开发环境）
  if (allowLocalOverride) {
    const localConfigPath = path.join(process.cwd(), `.${env}.local.yaml`);
    if (fs.existsSync(localConfigPath)) {
      const localConfig = yaml.load(
        fs.readFileSync(localConfigPath, 'utf8')
      ) as Partial<AppConfig>;
      config = deepMerge(config, localConfig);
    }
  }

  return config;
}

/**
 * 获取当前配置
 * 使用示例：
 */
export function getConfig(): AppConfig {
  const env = process.env.NODE_ENV || 'development';
  const configDir = path.join(process.cwd(), 'config');

  return loadConfig({
    env,
    configDir,
    allowLocalOverride: env !== 'production'
  });
}

// 使用示例
const config = getConfig();
console.log(`应用: ${config.app.name}`);
console.log(`环境: ${process.env.NODE_ENV}`);
```

## 二、密钥管理基础

### 2.1 什么是密钥？

密钥是**访问敏感资源的凭证**，就像：
- 你的家门钥匙（访问服务器的SSH密钥）
- 你的银行卡密码（访问数据库的密码）
- 你的身份证（第三方服务的API密钥）

**常见密钥类型：**

| 类型 | 用途 | 风险 |
|------|------|------|
| 数据库密码 | 连接数据库 | 数据泄露 |
| API密钥 | 第三方服务认证 | 资源滥用、账单超支 |
| SSH密钥 | 服务器访问 | 服务器被入侵 |
| JWT密钥 | 生成访问令牌 | 伪造身份 |
| 加密密钥 | 数据加密 | 数据泄露 |

### 2.2 密钥管理原则

```
密钥管理八荣八耻（改编自八荣八耻）：

以使用密钥管理工具为荣，以写在代码里为耻
以定期轮换密钥为荣，以永久使用同一密钥为耻
以最小权限原则为荣，以过度授权为耻
以加密存储为荣，以明文存储为耻
以审计日志为荣，以无记录操作为耻
以隔离环境为荣，以混用密钥为耻
以多因素认证为荣，以单因素认证为耻
以安全销毁为荣，以随意丢弃为耻
```

### 2.3 常见密钥管理方案对比

| 方案 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| **环境变量** | 小型项目 | 简单 | 不够安全 |
| **HashiCorp Vault** | 中大型项目 | 功能完善、高安全 | 部署复杂 |
| **AWS Secrets Manager** | AWS项目 | 原生集成 | 绑定AWS |
| **Azure Key Vault** | Azure项目 | 原生集成 | 绑定Azure |
| **1Password/Secrets Hub** | 团队协作 | 易于分享 | 不是自动化的 |
| **GitHub Secrets** | GitHub Actions | CI/CD集成 | 只能在Actions用 |

## 三、Docker环境变量与密钥

### 3.1 Docker中的密钥处理

```dockerfile
# Dockerfile - 生产镜像构建
# 注意：不要在Dockerfile中硬编码密钥！

FROM node:20-alpine AS builder

# 设置工作目录
WORKDIR /app

# 复制依赖文件
COPY package*.json ./

# 安装依赖（利用缓存层）
RUN npm ci --only=production

# 复制源代码
COPY . .

# 构建应用（注意：构建时可能需要密钥）
# 但不应该把密钥写在Dockerfile里！
# RUN NEXT_PUBLIC_API_KEY=xxx npm run build  ❌ 错误做法

# 最终镜像
FROM node:20-alpine

WORKDIR /app

# 从builder复制构建产物
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./

# 以非root用户运行（安全最佳实践）
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodejs -u 1001
USER nodejs

EXPOSE 3000

CMD ["node", "dist/server.js"]
```

## 四、配置中心架构

### 4.1 什么是配置中心？

配置中心就像**餐厅的中央厨房管理系统**——所有分店的菜谱（配置）都存储在中央厨房，分店只需连接就能获取最新配方。如果要改配方，不需要去每个分店改，只需在中央改，所有分店自动同步。

```java
// 5. 监听配置变化（热更新）
package com.example.listener;

import com.ctrip.framework.apollo.model.ConfigChangeEvent;
import com.ctrip.framework.apollo.spring.annotation.ApolloConfigChangeListener;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.stereotype.Component;

/**
 * 配置变更监听器
 * 当配置中心中的配置发生变化时，自动触发回调
 * 用途：无需重启应用，动态更新配置
 */
@Component
public class ConfigChangeListener {

    private static final Logger logger = LoggerFactory.getLogger(ConfigChangeListener.class);

    @ApolloConfigChangeListener
    public void onConfigChange(ConfigChangeEvent changeEvent) {
        // 遍历所有变化的key
        for (String key : changeEvent.changedKeys()) {
            // 获取变化详情
            var change = changeEvent.getChange(key);

            logger.info("配置变更 - key: {}, 旧值: {}, 新值: {}, 变化类型: {}",
                key,
                change.getOldValue(),
                change.getNewValue(),
                change.getChangeType()
            );
        }
```

```typescript
 * 获取数据库连接凭据
 * 用途：从Vault获取动态生成的用户名和密码
 * 特点：每次获取都是新的凭据，有过期时间
 */
async function getDatabaseCredentials() {
  try {
    // 读取动态生成的数据库凭据
    const result = await vaultClient.read('database/creds/myapp-role');

      host: 'db.example.com',
      port: 5432,
      database: 'myapp',

      password: result.data.password,
```

### 6.2 密钥轮换策略

```bash
# ========== 密钥轮换的最佳实践 ==========
#
# 为什么需要轮换？
# 1. 降低密钥泄露后的风险（过期密钥无法使用）
# 2. 满足合规要求（PCI-DSS、SOC2等要求定期轮换）
# 3. 员工离职时必须更换相关密钥
#
# 轮换频率建议：
# 高敏感密钥（数据库密码）：每30-90天
# API密钥：每90-180天
# SSH密钥：每180-365天
# JWT密钥：每30-60天

# ========== 数据库密码轮换脚本 ==========
#!/bin/bash
# rotate-db-password.sh - 数据库密码轮换

set -e

# 配置
DB_HOST="db.example.com"
DB_PORT="5432"
DB_NAME="myapp"
OLD_USER="app_user"
VAULT_SECRET_PATH="secret/myapp/database"

# 从Vault获取旧密码（或者从环境变量）
OLD_PASSWORD=$(vault kv get -field=password $VAULT_SECRET_PATH)

# 生成新密码（使用随机字符串）
NEW_PASSWORD=$(openssl rand -base64 32)

echo "开始轮换数据库密码..."

# 1. 在数据库中更新密码
psql -h $DB_HOST -p $DB_PORT -U postgres -c \
  "ALTER USER $OLD_USER WITH PASSWORD '$NEW_PASSWORD';"

echo "✓ 数据库密码已更新"

# 2. 更新到Vault
vault kv put $VAULT_SECRET_PATH \
  host=$DB_HOST \
  port=$DB_PORT \
  username=$OLD_USER \
  password=$NEW_PASSWORD \
  last_rotated=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

echo "✓ Vault中的密钥已更新"

# 3. 通知应用重新加载配置
# 方式1：调用应用的刷新端点
curl -X POST https://myapp.example.com/admin/refresh-secrets \
  -H "X-Admin-Token: $ADMIN_TOKEN"

# 方式2：触发Kubernetes Secret滚动更新
kubectl rollout restart deployment/myapp -n production

echo "✓ 应用配置已刷新"

# 4. 验证新密码可以连接
PGPASSWORD=$NEW_PASSWORD psql -h $DB_HOST -p $DB_PORT -U $OLD_USER -d $DB_NAME -c "SELECT 1;" > /dev/null

if [ $? -eq 0 ]; then
    echo "✓ 新密码验证成功"
    echo "数据库密码轮换完成！"
else
    echo "❌ 新密码验证失败，紧急回滚..."
    # 回滚到旧密码
    psql -h $DB_HOST -p $DB_PORT -U postgres -c \
      "ALTER USER $OLD_USER WITH PASSWORD '$OLD_PASSWORD';"
    exit 1
fi

# 5. 记录轮换日志（审计用）
echo "$(date -u +"%Y-%m-%dT%H:%M:%SZ") - DB password rotated for $OLD_USER" >> /var/log/key-rotation.log
```

## 八、总结与检查清单

### 8.1 密钥管理检查清单

```
日常检查：

```

### 8.2 环境配置最佳实践

| 方面 | 最佳实践 |
|------|----------|
| **文件管理** | 使用版本控制配置，使用环境覆盖特定配置 |
| **密钥分离** | 密钥永远不要提交到代码仓库 |
| **最小权限** | 每个环境使用独立的凭据 |
| **审计追踪** | 记录所有配置和密钥的访问和变更 |
| **自动化** | 使用配置中心实现动态配置 |
| **文档化** | 记录所有环境变量的用途和格式 |

### 8.3 快速参考表

```
环境变量命名规范：
├── NODE_ENV              # 环境名称
├── PORT                  # 服务端口
├── <SERVICE>_HOST        # 服务地址
├── <SERVICE>_PORT        # 服务端口
├── <SERVICE>_URL         # 服务完整URL
├── <SERVICE>_USER        # 用户名
├── <SERVICE>_PASSWORD    # 密码
├── API_KEY               # API密钥
├── JWT_SECRET            # JWT签名密钥
├── AWS_ACCESS_KEY_ID     # AWS访问密钥
├── AWS_SECRET_ACCESS_KEY # AWS私钥
└── DATABASE_URL          # 数据库连接URL

常见错误：
❌ apikey=sk_live_xxxxx
❌ db_password=password123
❌ secret_key=mysecret

正确做法：
✅ API_KEY=sk_live_xxxxx (使用下划线，无空格)
✅ DB_PASSWORD=password123 (使用下划线，无空格)
✅ SECRET_KEY=mysecret (使用下划线，无空格)
```

记住：**密钥安全不仅是技术问题，更是意识和流程问题**。再好的加密算法也挡不住把密钥贴在公告栏上。
