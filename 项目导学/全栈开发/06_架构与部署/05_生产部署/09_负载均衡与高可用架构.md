# 负载均衡与高可用架构完全指南

## 前言：从快递站点理解负载均衡

想象一下，你开了一家快递公司。一开始只有一家门店，所有订单都从这家店收发。但随着业务增长：

- 一家门店忙不过来了（服务器扛不住）
- 一家门店出问题了，所有包裹都送不了（单点故障）

于是你想到：
- 在城市不同区域开设多家门店（负载均衡集群）
- 设立一个调度中心，根据每家门店的忙碌程度分配订单（负载均衡算法）
- 某家门店出了问题，调度中心自动把订单转到其他门店（故障转移）

这就是负载均衡与高可用的基本思想。

## 一、负载均衡基础

### 1.1 什么是负载均衡？

负载均衡（Load Balancing）就像**餐厅的前台接待员**：
- 把客人（请求）合理分配到不同的服务员（服务器）
- 不让任何一个人太忙或太闲
- 某个人请假时，其他人能接手他的工作

### 1.2 负载均衡的核心功能

```
负载均衡核心功能：

┌─────────────────────────────────────────────────────────┐
│                    用户请求                               │
│                    /    \    \                           │
│                   /      \      \                        │
│                  ▼        ▼       ▼                      │
│         ┌─────────────────────────────────┐            │
│         │         负载均衡器               │            │
│         │  ┌─────────────────────────────┐ │            │
│         │  │  1. 连接管理（请求分发）     │ │            │
│         │  │  2. 健康检查（剔除故障节点） │ │            │
│         │  │  3. 会话保持（可选）         │ │            │
│         │  │  4. SSL终结（可选）           │ │            │
│         │  │  5. 故障转移（高可用）        │ │            │
│         │  └─────────────────────────────┘ │            │
│         └─────────────────────────────────┘            │
│                   /      |      \                       │
│                  ▼       ▼       ▼                      │
│              ┌────┐  ┌────┐  ┌────┐                    │
│              │ S1 │  │ S2 │  │ S3 │                    │
│              └────┘  └────┘  └────┘                    │
└─────────────────────────────────────────────────────────┘
```

### 1.3 负载均衡 vs 反向代理 vs 正向代理

| 概念 | 形象比喻 | 作用 | 常见工具 |
|------|----------|------|----------|
| **正向代理** | 帮你跑腿的秘书 | 隐藏客户端身份，客户端知道服务器 | VPN、Shadowsocks |
| **反向代理** | 前台接待员 | 隐藏服务器身份，用户不知道真实服务器 | Nginx、HAProxy |
| **负载均衡** | 调度中心 | 分配请求到多台服务器，实现扩展性 | Nginx、LVS、ELB |

注意：**反向代理和负载均衡经常一起使用**，Nginx 两者都能做。

## 二、负载均衡算法详解

### 2.1 常见算法对比

| 算法 | 工作原理 | 优点 | 缺点 | 适用场景 |
|------|----------|------|------|----------|
| **Round Robin** | 轮询分配 | 简单、公平 | 不考虑服务器性能差异 | 服务器性能相近 |
| **Weighted RR** | 按权重轮询 | 可分配权重 | 需要手动配置 | 服务器性能不同 |
| **Least Connections** | 分配给连接最少的 | 动态适应 | 实现复杂 | 长连接场景 |
| **IP Hash** | 根据IP做哈希 | 会话保持 | 可能不均匀 | 需要会话保持 |
| **URL Hash** | 根据URL做哈希 | 缓存友好 | 可能不均匀 | 缓存命中优化 |

### 2.2 轮询算法（Round Robin）

这是最简单也是最常用的算法，就像**按顺序叫号**。

```nginx
# Nginx Round Robin 配置
# 这是默认方式，不需要特别配置

upstream backend {
    # 默认就是轮询算法
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;
}

server {
    listen 80;

    location / {
        proxy_pass http://backend;
    }
}

# 请求分发示例：
# 请求1 -> 192.168.1.101
# 请求2 -> 192.168.1.102
# 请求3 -> 192.168.1.103
# 请求4 -> 192.168.1.101 (循环)
```

### 2.3 加权轮询算法（Weighted Round Robin）

有些服务器性能强，想让它多干点活，就用加权轮询。

```nginx
# 加权轮询配置
# 性能好的服务器权重高

upstream backend {
    # weight参数表示权重，默认是1
    # 101服务器性能最强，分配更多请求
    server 192.168.1.101:8080 weight=5;   # 权重5
    server 192.168.1.102:8080 weight=3;   # 权重3
    server 192.168.1.103:8080 weight=2;   # 权重2
}

# 分发比例：
# 101服务器：每收到 5/(5+3+2) = 50% 的请求
# 102服务器：每收到 3/(5+3+2) = 30% 的请求
# 103服务器：每收到 2/(5+3+2) = 20% 的请求

# 实际分发：
# 请求1 -> 101
# 请求2 -> 101
# 请求3 -> 101
# 请求4 -> 101
# 请求5 -> 101
# 请求6 -> 102
# 请求7 -> 102
# 请求8 -> 102
# 请求9 -> 103
# 请求10 -> 103
```

### 2.4 最少连接算法（Least Connections）

轮询不考虑"忙碌程度"，最少连接算法会**把请求发给当前最空闲的服务器**。

```nginx
# 最少连接算法
# 适合长连接场景，如WebSocket、数据库连接池

upstream backend {
    least_conn;  # 启用最少连接算法

    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;
}

# 工作原理：
# 1. 服务器A当前有5个连接
# 2. 服务器B当前有3个连接
# 3. 服务器C当前有8个连接
# 4. 新请求来时 -> 分配给B（连接最少）
```

```yaml
# HAProxy 最少连接配置
# haproxy.cfg

backend api_servers
    # 最少连接算法
    balance leastconn

    # 服务器定义
    server api1 192.168.1.101:8080 check inter 2000 rise 2 fall 3
    server api2 192.168.1.102:8080 check inter 2000 rise 2 fall 3
    server api3 192.168.1.103:8080 check inter 2000 rise 2 fall 3

    # check: 启用健康检查
    # inter 2000: 每2秒检查一次
    # rise 2: 连续2次成功认为服务器上线
    # fall 3: 连续3次失败认为服务器下线
```

### 2.5 IP哈希算法（IP Hash）

有些场景需要**同一用户的请求都打到同一台服务器**（会话保持），比如用户登录后把会话存在本地服务器。

```nginx
# IP Hash算法
# 相同IP的请求会打到同一台服务器

upstream backend {
    ip_hash;  # 启用IP哈希

    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;

    # 注意：权重不能直接用weight参数，要用server指令的weight
    # 如果需要临时移除服务器，先用down标记
    # server 192.168.1.103:8080 down;
}

# 工作原理：
# 1. 计算客户端IP的哈希值
# 2. 用哈希值对服务器数量取模
# 3. 分配到对应服务器
#
# 假设hash(IP) = 100
# 100 % 3 = 1 -> 服务器2 (index=1)
```

### 2.6 一致性哈希算法（Consistent Hashing）

IP Hash的问题是**服务器数量变化时，所有哈希值都要重新分配**，可能导致缓存全部失效。

一致性哈希让这个过程**更平滑**。

```
一致性哈希原理：

                    ┌──────────┐
                    │  服务器A  │ (哈希环上的节点)
                    └──────────┘
                         │
        ┌────────────────┼────────────────┐
        ▼                ▼                ▼
   ┌─────────┐     ┌─────────┐     ┌─────────┐
   │ 键->A   │     │ 键->C   │     │ 键->B   │
   └─────────┘     └─────────┘     └─────────┘
        │                │                │
        ▼                ▼                ▼
   ┌─────────────────────────────────────────┐
   │            哈希环                        │
   │  0 ──────► 哈希值 ◄────── 2^32          │
   └─────────────────────────────────────────┘

# 特性：
# 1. 服务器映射到哈希环上的某个点
# 2. 键映射到环上，顺时针遇到的第一个服务器就是目标
# 3. 增加/删除服务器时，只影响相邻的键
```

## 三、Nginx负载均衡实战

### 3.1 Nginx核心配置

```nginx
# /etc/nginx/nginx.conf

# 运行用户
user nginx;

# 工作进程数（一般设置为CPU核心数）
worker_processes auto;

# 错误日志
error_log /var/log/nginx/error.log warn;

# PID文件
pid /var/run/nginx.pid;

events {
    # 每个工作进程的最大连接数
    worker_connections 1024;

    # 使用epoll（Linux高性能IO模型）
    use epoll;

    # 允许一次接受多个连接
    multi_accept on;
}

http {
    # 基础配置
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    # 日志格式
    log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                    '$status $body_bytes_sent "$http_referer" '
                    '"$http_user_agent" "$http_x_forwarded_for" '
                    'upstream: $upstream_addr '
                    'upstream_status: $upstream_status';

    access_log /var/log/nginx/access.log main;

    # 性能优化
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;

    # 超时配置
    keepalive_timeout 65;
    client_max_body_size 20M;

    # Gzip压缩
    gzip on;
    gzip_types text/plain text/css application/json application/javascript;

    # ========== 负载均衡配置 ==========
    upstream backend_servers {
        # 方式1：默认轮询
        server 192.168.1.101:8080;
        server 192.168.1.102:8080;

        # 方式2：加权轮询
        server 192.168.1.101:8080 weight=5;
        server 192.168.1.102:8080 weight=3;
        server 192.168.1.103:8080 weight=2;

        # 方式3：最少连接
        # least_conn;

        # 方式4：IP哈希（会话保持）
        # ip_hash;

        # 统一配置
        keepalive 32;  # 保持连接数
        max_fails = 3;  # 最大失败次数
        fail_timeout = 30s;  # 失败超时时间
    }

    server {
        listen 80;
        server_name example.com;

        # 请求日志
        access_log /var/log/nginx/example.access.log main;

        location / {
            # 代理到上游服务器
            proxy_pass http://backend_servers;

            # ========== 代理头部设置 ==========
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;

            # 超时设置
            proxy_connect_timeout 60s;
            proxy_send_timeout 60s;
            proxy_read_timeout 60s;

            # 缓冲设置
            proxy_buffering on;
            proxy_buffer_size 4k;
            proxy_buffers 8 4k;
        }

        # 健康检查接口
        location /health {
            access_log off;
            return 200 "healthy\n";
            add_header Content-Type text/plain;
        }
    }
}
```

        proxy_send_timeout 86400s;

        proxy_set_header Host $host;

        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

## 四、Keepalived实现高可用

### 4.1 什么是Keepalived？

Keepalived就像**电梯的备用控制系统**：
- 主系统正常时，电梯由主系统控制
- 主系统坏了，备用系统立即接管
- 切换过程对用户透明

在服务器场景中，Keepalived实现**VRRP协议**，提供**虚IP漂移**功能。

### 4.2 Keepalived工作原理

```
VRRP协议工作原理：

                    ┌─────────────────┐
                    │   虚拟IP (VIP)   │
                    │   192.168.1.100  │
                    └────────┬────────┘
                             │
              ┌──────────────┴──────────────┐
              │                             │
              ▼                             ▼
     ┌─────────────────┐           ┌─────────────────┐
     │    MASTER       │           │    BACKUP       │
     │   优先级: 100    │◄─────────►│   优先级: 80     │
     │   状态: ACTIVE   │  VRRP心跳  │   状态: BACKUP  │
     └────────┬────────┘           └────────┬────────┘
              │                             │
              │        物理网络              │
              └──────────────┬──────────────┘
                             │
                             ▼
                     ┌───────────────┐
                     │     客户端      │
                     └───────────────┘

# 选举过程：
# 1. Master定期发送VRRP广告（组播）
# 2. Backup收不到时，认为Master挂了
# 3. 优先级高的Backup成为新的Master
# 4. 虚拟IP漂移到新Master
# 5. 客户端无需改变配置，继续访问虚拟IP
```

### 4.3 Keepalived安装与配置

```bash
# ========== 安装Keepalived ==========
# Ubuntu/Debian
apt-get update
apt-get install keepalived

# CentOS/RHEL
yum install keepalived

# ========== Master节点配置 ==========
# /etc/keepalived/keepalived.conf

global_defs {
    # 邮件通知（可选）
    notification_email {
        admin@example.com
    }
    notification_email_from keepalived@example.com
    smtp_server 127.0.0.1
    smtp_connect_timeout 30

    # Keepalived进程ID
    router_id nginx_master

    # 防止脑裂（同一网段有多个Master）
    # 设置nopreempt时必须设置unicast
    vrrp_unicast_bind
    vrrp_unicast_peer 192.168.1.102
}

# 健康检查脚本
vrrp_script check_nginx {
    # 脚本路径（检查nginx是否存活）
    script "/etc/keepalived/check_nginx.sh"

    # 执行间隔（秒）
    interval 2

    # 权重变化（脚本返回0时增加）
    weight -20

    # 初始状态
    initial_backup  # 初始为backup状态
}

vrrp_instance VI_1 {
    # 初始状态：MASTER | BACKUP | FAULT
    state BACKUP

    # 网卡名
    interface eth0

    # VRRP实例ID（同一集群要相同）
    virtual_router_id 51

    # 优先级（MASTER要高于BACKUP）
    priority 100

    # VRRP广告间隔（秒）
    advert_int 1

    # 认证
    authentication {
        auth_type PASS
        auth_pass 1234
    }

    # 虚拟IP地址（可以多个）
    virtual_ipaddress {
        192.168.1.100/24 dev eth0
    }

    # 跟踪脚本（健康检查）
    track_script {
        check_nginx
    }

    # 切换脚本（可选）
    notify_master "/etc/keepalived/notify.sh master"
    notify_backup "/etc/keepalived/notify.sh backup"
    notify_fault "/etc/keepalived/notify.sh fault"

    # 使用单播模式（防止脑裂）
    unicast_peer {
        192.168.1.102
    }
}

# ========== Backup节点配置 ==========
# /etc/keepalived/keepalived.conf

global_defs {
    router_id nginx_backup
    vrrp_unicast_bind
    vrrp_unicast_peer 192.168.1.101  # 指向Master
}

vrrp_script check_nginx {
    script "/etc/keepalived/check_nginx.sh"
    interval 2
    weight -20
}

vrrp_instance VI_1 {
    state BACKUP
    interface eth0
    virtual_router_id 51

    # Backup优先级较低
    priority 80

    advert_int 1

    authentication {
        auth_type PASS
        auth_pass 1234
    }

    virtual_ipaddress {
        192.168.1.100/24 dev eth0
    }

    track_script {
        check_nginx
    }

    unicast_peer {
        192.168.1.101  # 指向Master
    }
}
```

### 4.4 健康检查脚本

```bash
#!/bin/bash
# /etc/keepalived/check_nginx.sh
# Nginx健康检查脚本

# 检查nginx进程是否存在
nginx_process=$(pgrep nginx | wc -l)

if [ $nginx_process -eq 0 ]; then
    # nginx没有运行，尝试启动
    systemctl start nginx

    # 等待一下再检查
    sleep 2

    # 再次检查
    nginx_process=$(pgrep nginx | wc -l)
    if [ $nginx_process -eq 0 ]; then
        # 启动失败，报告失败
        exit 1
    fi
fi

# 检查nginx是否能正常响应
nginx_response=$(curl -s -o /dev/null -w "%{http_code}" http://localhost/health)

if [ "$nginx_response" != "200" ]; then
    # nginx不能正常响应，报告失败
    exit 1
fi

# 一切正常
exit 0

# ========== 通知脚本 ==========
#!/bin/bash
# /etc/keepalived/notify.sh
# 状态切换通知脚本

case "$1" in
    "master")
        echo "切换为Master" | logger -t keepalived
        # 可以发送邮件、钉钉通知等
        ;;
    "backup")
        echo "切换为Backup" | logger -t keepalived
        ;;
    "fault")
        echo "进入Fault状态" | logger -t keepalived
        ;;
esac
```

    │   MASTER        │ │   BACKUP        │ │                 │
    │   优先级: 100    │ │   优先级: 80     │ │                 │
    └────────┬────────┘ └────────┬────────┘ │                 │
             │                     │          │                 │
             ▼                     ▼          │                 │
    ┌─────────────────┐ ┌─────────────────┐    │                 │
    │     Nginx       │ │     Nginx       │    │                 │
    │   192.168.1.101 │ │   192.168.1.102 │    │                 │
    └────────┬────────┘ └────────┬────────┘    │                 │
             │                     │             │                 │
             └──────────┬──────────┘             │
                        │                        │
                        ▼                        ▼
              ┌──────────────────────┐   ┌──────────────────────┐

# 故障场景：
# 1. Master Nginx挂了 -> Keepalived检测到，VIP漂移到Backup

### 5.2 被动健康检查

被动健康检查是**根据实际请求结果判断服务器是否健康**。

```nginx
# 被动健康检查配置
# 当连续失败达到阈值时，标记服务器为不可用

upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;

    # 连续失败次数阈值
    max_fails = 3;

    # 不可用时间窗口
    fail_timeout = 30s;

    # 补充：如果需要，可以标记某台为backup或down
    # server 192.168.1.103:8080 backup;
    # server 192.168.1.104:8080 down;
}

# 配置proxy_next_upstream处理故障转移
server {
    location / {
        proxy_pass http://backend;

        # 发生这些错误时，自动尝试下一个服务器
        proxy_next_upstream error timeout http_502 http_503 http_504;

        # 请求超时
        proxy_connect_timeout 5s;
        proxy_read_timeout 10s;
    }
}

# max_fails行为说明：
# 1. 如果30秒内有3次请求失败，服务器被认为不可用
# 2. 接下来的30秒内，不会向这台服务器分发请求
# 3. 30秒后，服务器重新尝试（如果期间有请求成功，立即可用）
```

## 八、实战踩坑经验

### 8.1 负载均衡常见问题

```nginx
# ========== 踩坑1：会话突然断开 ==========
# 问题：用户登录后，下次请求变成未登录状态

# 原因：没有会话保持，用户请求被分配到不同服务器
#      但Session存储在单机内存中

# 解决1：使用IP Hash（同一IP访问同一服务器）
upstream backend {
    ip_hash;
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
}

# 解决2：使用共享Session（Redis）
# 应用代码使用Redis存储Session
# set session:user123 '{"userId": 123, "name": "张三"}'

# ========== 踩坑2：后端服务器负载不均 ==========
# 问题：明明配置了轮询，但某台服务器CPU特别高

# 原因：某些请求特别耗时（如文件上传），被分配到同一台服务器

# 解决：使用最少连接算法
upstream backend {
    least_conn;  # 动态分配到连接数最少的服务器
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
}

# ========== 踩坑3：健康检查导致频繁切换 ==========
# 问题：服务器偶尔响应慢，被误判为不可用

# 解决：增加失败阈值和超时时间
upstream backend {
    server 192.168.1.101:8080 max_fails=5 fail_timeout=60s;
    # 连续5次失败才标记为不可用，且60秒后重新尝试
}

# ========== 踩坑4：长连接没有正确复用 ==========
# 问题：Nginx和后端的连接频繁断开

# 解决：配置keepalive
upstream backend {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;

    keepalive 32;  # 保持32个空闲连接
}

location / {
    proxy_pass http://backend;
    proxy_http_version 1.1;  # 启用HTTP/1.1
    proxy_set_header Connection "";  # 清除Connection头
}
```

### 8.2 Keepalived常见问题

```bash
# ========== 踩坑5：脑裂问题（两个节点都是Master） ==========
# 问题：VIP同时出现在两台服务器上

# 原因：
# 1. 心跳线断了，两台服务器互相认为对方挂了
# 2. 认证配置不一致

# 解决：
# 1. 使用双心跳线（eth0和eth1）
# 2. 设置nopreempt避免抢主
# 3. 添加脚本检查脑裂情况

# 检查脑裂的脚本
#!/bin/bash
# /etc/keepalived/check-split-brain.sh

VIP_COUNT=$(ip addr show | grep -c 192.168.1.100)

if [ $VIP_COUNT -gt 1 ]; then
    echo "检测到脑裂！VIP在多个节点上" | logger -t keepalived -p daemon.error
    # 可以发送告警
    # 或者强制关闭本机keepalived
    # systemctl stop keepalived
fi

# ========== 踩坑6：VIP不能访问 ==========
# 问题：VIP ping不通，无法连接

# 检查清单：
# 1. 防火墙是否放行了VRRP协议（组播224.0.0.18）
iptables -I INPUT -p vrrp -j ACCEPT

# 2. 网卡是否支持NETWORKING
#    检查 /proc/sys/net/ipv4/ip_nonlocal_bind
cat /proc/sys/net/ipv4/ip_nonlocal_bind
# 如果是0，设置为1
echo 1 > /proc/sys/net/ipv4/ip_nonlocal_bind

# 3. 永久设置
echo "net.ipv4.ip_nonlocal_bind = 1" >> /etc/sysctl.conf
sysctl -p
```
