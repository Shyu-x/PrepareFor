# 日志收集与可观测性实战

## 前言：从debug到可观测性

想象你是一个酒店的经理。以前酒店小的时候，客人投诉房间有问题，你可以直接去现场看看：空调不制冷？去看看空调。热水没了？去检查热水器。那时候问题简单，靠"直觉"就能定位。

但现在酒店变大了，有几十层楼、几百个房间。你不可能每一层都跑去检查。怎么办？

聪明的经理会建立一套"监控系统"：
- 每个房间有温度传感器、烟雾探测器
- 前台有监控大屏，实时显示各楼层状态
- 客人打电话投诉时，你能在系统里立刻看到对应房间的信息
- 系统还能提前预警：某个区域的温度开始上升了，可能有问题

软件系统也是一样。系统小的时候，你可以直接登录服务器看日志、debug。但系统大了、日活高了，你就需要一套完整的"监控体系"——这就是可观测性（Observability）。

这一章，我们将从最基础的日志讲起，涵盖结构化日志、ELK/Loki日志平台、OpenTelemetry标准，以及链路追踪。内容有点多，但每块都会讲透。

/**
 * 结构化日志工具
 *
 * 统一的日志格式和输出
 */

// 日志字段接口
interface LogFields {
  // 基础字段
  timestamp: string;
  level: 'DEBUG' | 'INFO' | 'WARN' | 'ERROR' | 'CRITICAL';
  service: string;
  host: string;
  pid: number;

  // 追踪字段
  traceId?: string;
  spanId?: string;
  requestId?: string;

  // 业务字段
  message: string;
  userId?: string;
  duration?: number;

  // 错误字段
  error?: {
    name: string;
    message: string;
    stack?: string;
  };

  // 其他
  [key: string]: any;
}

// 日志输出器接口
interface LogTransport {
  write(log: LogFields): void;
}

// Console输出（开发环境）
class ConsoleTransport implements LogTransport {
  write(log: LogFields) {
    const output = {
      // 时间使用ISO格式
      timestamp: new Date(log.timestamp).toISOString(),

      // 级别用大写
      level: log.level,

      // 服务名
      service: log.service,

      // 消息
      message: log.message,

      // 追踪ID
      traceId: log.traceId,

      // 业务相关字段
      ...pick(log, ['userId', 'duration', 'error']),
    };

    // 根据级别使用不同的console方法
    switch (log.level) {
      case 'DEBUG':
      case 'INFO':
        console.log(JSON.stringify(output));
        break;
      case 'WARN':
        console.warn(JSON.stringify(output));
        break;
      case 'ERROR':
      case 'CRITICAL':
        console.error(JSON.stringify(output));
        break;
    }
  }
}

// File输出（生产环境）
class FileTransport implements LogTransport {
  private stream: WriteStream;

  constructor(filename: string) {
    this.stream = createWriteStream(filename, { flags: 'a' });
  }

  write(log: LogFields) {
    this.stream.write(JSON.stringify(log) + '\\n');
  }

  close() {
    this.stream.end();
  }
}

// 日志器类
class Logger {
  private service: string;
  private host: string;
  private pid: number;
  private transports: LogTransport[];

  constructor(service: string, transports: LogTransport[]) {
    this.service = service;
    this.host = os.hostname();
    this.pid = process.pid;
    this.transports = transports;
  }

  // 创建基础字段
  private baseLog(level: LogFields['level'], message: string): LogFields {
    return {
      timestamp: new Date().toISOString(),
      level,
      service: this.service,
      host: this.host,
      pid: this.pid,
      message,
    };
  }

  // 添加追踪信息
  withTrace(traceId: string, spanId?: string) {
    // 返回新的logger实例，带有追踪信息
    return {
      debug: (message: string, fields?: object) =>
        this.debug(message, { traceId, spanId, ...fields }),
      info: (message: string, fields?: object) =>
        this.info(message, { traceId, spanId, ...fields }),
      warn: (message: string, fields?: object) =>
        this.warn(message, { traceId, spanId, ...fields }),
      error: (message: string, fields?: object) =>
        this.error(message, { traceId, spanId, ...fields }),
    };
  }

  // 记录日志
  private log(level: LogFields['level'], message: string, extra?: object) {
    const logEntry: LogFields = {
      ...this.baseLog(level, message),
      ...extra,
    };

    for (const transport of this.transports) {
      transport.write(logEntry);
    }
  }

  debug(message: string, fields?: object) {
    this.log('DEBUG', message, fields);
  }

  info(message: string, fields?: object) {
    this.log('INFO', message, fields);
  }

  warn(message: string, fields?: object) {
    this.log('WARN', message, fields);
  }

  error(message: string, fields?: object) {
    this.log('ERROR', message, fields);
  }
}

// 工具函数：选取对象的部分属性
function pick(obj: object, keys: string[]): object {
  const result = {};
  for (const key of keys) {
    if (key in obj) {
      result[key] = obj[key];
    }
  }
  return result;
}

// 使用示例
const logger = new Logger('user-service', [
  new ConsoleTransport(),
  new FileTransport('/var/log/user-service.log'),
]);

// 记录日志
logger.info('User logged in', {
  userId: '12345',
  ip: '192.168.1.100',
  duration: 45,
});

// 记录错误
try {
  // 业务逻辑
  throw new Error('Database connection failed');
} catch (err) {
  logger.error('Operation failed', {
    error: {
      name: err.name,
      message: err.message,
      stack: err.stack,
    },
  });
}
`;

Fluentd的工作流程：
Input → Filter → Buffer → Output

- Input：从哪里收集日志（文件、TCP、HTTP等）
- Filter：如何处理日志（解析、转换、过滤）
- Buffer：如何缓冲日志（防止数据丢失）
- Output：发送到哪个存储（Elasticsearch、S3等）

/**
 * Fluentd配置文件示例

const fluentdConfig = `
# fluent.conf

# ========================
# 1. 读取应用日志
# ========================
<source>
  @type tail
  @id input_tail_app
  path /var/log/myapp/*.log
  pos_file /var/log/fluentd/app.log.pos

  # 解析JSON格式日志
  @type json

  # 添加标签
  tag app.log

  # 解析时间戳
  <parse>
    @type json
    time_type string
    time_format %Y-%m-%dT%H:%M:%S.%L%z
  </parse>
</source>

# ========================
# 2. 读取系统日志
# ========================
<source>
  @type tail
  @id input_tail_syslog
  path /var/log/syslog
  pos_file /var/log/fluentd/syslog.pos

  tag syslog

  <parse>
    @type syslog
    time_format %b %d %H:%M:%S
  </parse>
</source>

# ========================
# 3. Filter: 添加额外字段
# ========================
<filter app.log>
  @type record_transformer

  # 添加静态字段
  <record>
    hostname "#{Socket.gethostname}"
    environment production
    app_name myapp
  </record>

  # 修改字段
  <record>
    # 把user_id改成userId
    userId \${record["user_id"]}
  </record>
</filter>

# ========================
# 4. Filter: 过滤日志
# ========================
<filter app.log>
  @type grep

  # 只保留ERROR级别的日志
  <exclude>
    key level
    pattern /^(DEBUG|INFO|WARN)$/
  </exclude>
</filter>

# ========================
# 5. Buffer: 缓冲配置
# ========================
<match app.log>
  @type elasticsearch

  # 缓冲配置
  <buffer>
    @type file
    path /var/log/fluentd/buffer/app
    flush_mode interval
    flush_interval 10s
    retry_type exponential_backoff
    retry_wait 10s
    retry_max_interval 30s
    retry_timeout 72h
    chunk_size 8m
    total_limit_size 1G
  </buffer>

  # Elasticsearch配置
  host elasticsearch.local
  port 9200
  logstash_format true
  logstash_prefix myapp-logs
  include_tag_key true
  tag_key @log_name

  # 认证
  user elastic
  password your_password

  # SSL
  ssl_verify false
  scheme https
</match>

# ========================
# 6. 输出到文件（备份）
# ========================
<match syslog>
  @type file

  path /var/log/fluentd/syslog-backup

  <buffer>
    @type file
    path /var/log/fluentd/buffer/syslog
    flush_mode interval
    flush_interval 60s
  </buffer>
</match>

# ========================
# 7. 错误日志单独处理
# ========================
<match app.error>
  @type elasticsearch

  # 单独的索引
  index_name myapp-error

  <buffer>
    @type file
    path /var/log/fluentd/buffer/error
  </buffer>

## 附录：常见问题解答

### Q1: ELK和Loki该怎么选？

A：
- 需要复杂全文搜索、聚合分析 → ELK
- 只是按时间+标签查日志，资源有限 → Loki
- 已经用Prometheus/Grafana → Loki（天然集成）

### Q2: 结构化日志和普通日志哪个好？

A：生产环境推荐结构化日志。
- 结构化日志方便程序解析和查询
- 普通日志方便人类阅读（调试时可以用）
- 建议：JSON格式存储（机器读），Pretty格式输出（开发调试）

### Q3: OpenTracing和OpenTelemetry什么关系？

A：OpenTracing是旧标准，已经不维护了。OpenTelemetry是继任者，统一了Traces、Metrics、Logs。新项目直接用OpenTelemetry。

### Q4: 日志收集影响性能吗？

A：有影响，但通常可接受。
- Fluentd/Promtail资源消耗很低（~50MB内存）
- 异步发送，不阻塞应用
- 建议：生产环境开启INFO级别，DEBUG按需开启

### Q5: 日志保留多久合适？

A：看需求。
- 错误日志：至少保留30天（便于排查）
- 业务日志：至少保留7天
- 合规要求：金融、医疗等可能要求1年+
- 成本考虑：超过30天的日志可以转存到冷存储
