# 状态同步与数据拉取场景完全指南

## 前言

在现代前端开发中，状态同步与数据拉取是每个开发者都必须面对的核心问题。当我们构建一个复杂的应用时，数据如何在组件之间流动、如何保持一致性、如何优雅地处理加载状态，这些问题直接影响着用户体验和代码质量。

本文将以一个**社交媒体Feed场景**为线索，从最基础的组件间通信开始，逐步深入到全局状态管理、数据获取策略、乐观更新、骨架屏加载以及缓存失效等核心知识点。通过大量的代码示例，帮助读者建立完整的状态管理知识体系。

---

## 第一部分：什么是"状态同步"？

### 1.1 通俗理解状态同步

让我们用一个生活中的例子来理解什么是"状态同步"。

想象你和你的室友共用一个冰箱。你们约定好：谁取了最后一样食材（比如鸡蛋），就要告诉对方"鸡蛋没了，记得买"。但是，如果有一天你取完鸡蛋后忘了告诉她，那么她打开冰箱时就会发现：冰箱显示还有鸡蛋（这是旧数据），但实际上已经没有鸡蛋了。这就是"状态不同步"——两个人对冰箱里的东西有了不同的认知。

在前端开发中，"状态"就是应用中的数据，"同步"就是让所有需要知道这个数据的组件都能得到一致的结果。当我们说"状态不同步"时，意思是：某些组件看到的数据和其他组件看到的数据不一致。

### 1.2 为什么会产生状态不同步？

产生状态不同步的原因主要有以下几种：

**第一种：多数据源问题**

现代应用通常从多个地方获取数据——有的从API来，有的从本地存储来，有的从WebSocket实时推送来。当这些数据源之间没有良好协调时，就容易产生不一致。

```typescript
// 假设我们从API获取用户信息
const userFromAPI = await fetch('/api/user/1').then(r => r.json());

// 同时也从本地存储读取（可能是旧的）
const userFromStorage = JSON.parse(localStorage.getItem('user') || '{}');

// 如果这两个数据不一致，界面就可能显示错误的信息
console.log(userFromAPI.name, userFromStorage.name); // "张三" vs "张"
```

**第二种：并发更新问题**

当多个操作同时发生时，如果没有适当的协调机制，就会产生竞态条件（Race Condition）。

```typescript
// 用户快速点击了两次"点赞"按钮
// 请求1 和 请求2 同时发出
const result1 = await fetch('/api/posts/1/like', { method: 'POST' });
const result2 = await fetch('/api/posts/1/like', { method: 'POST' });

// 如果服务器没有做好幂等性处理
// 结果可能是：只加了1个赞，而不是2个
console.log(result1.likes, result2.likes); // 101, 102（而不是103）
```

**第三种：缓存过期问题**

当使用缓存时，如果缓存的过期时间设置不当，或者缓存失效逻辑有bug，就会导致组件显示过期数据。

```typescript
// 缓存了5分钟的用户信息
const cache = {
  data: null,
  timestamp: 0,
  ttl: 5 * 60 * 1000 // 5分钟
};

function getUser() {
  const now = Date.now();
  // 如果缓存还有效，返回旧数据
  if (cache.data && (now - cache.timestamp) < cache.ttl) {
    return cache.data; // 这里返回的可能已经是5分钟前的旧数据了
  }
  // 否则重新获取
  return fetch('/api/user').then(r => r.json());
}
```

**第四种：组件树层级过深**

当状态需要从顶层组件逐层传递到深层组件时，中间任何一个环节出问题都会导致数据不一致。特别是在 React 中，props 的传递链路很长时，维护起来非常困难。

```typescript
// 顶层组件知道用户信息
<App>
  <Layout>
    <Sidebar>
      <UserPanel>  // 第4层：终于用到用户信息了
        <Avatar user={user} /> // 第5层
      </UserPanel>
    </Sidebar>
  </Layout>
</App>
```

### 1.3 状态同步的解决方案概述

面对这些问题，前端社区发展出了多种解决方案：

| 方案 | 解决问题 | 适用场景 | 代表库 |
|------|----------|----------|-------|
| Props Drilling | 深层组件传参 | 简单层级 | React 原生 |
| Context | 跨层级共享 | 主题、语言 | React 原生 |
| 事件总线 | 跨组件通信 | 解耦通信 | mitt、EventEmitter |
| 全局状态管理 | 应用级状态 | 复杂应用 | Zustand、Jotai、Recoil |
| 数据获取库 | 服务端状态 | API数据 | SWR、React Query |
| 乐观更新 | 即时反馈 | 用户交互 | 结合上述方案 |

本文将逐一讲解这些方案，并展示如何在一个完整的社交媒体Feed场景中综合运用它们。

## 第二部分：组件间通信

### 2.1 Props Drilling（属性穿透）

Props Drilling 是最原始也最直接的组件间通信方式。父组件通过 props 将数据传递给子组件，子组件再继续向下传递。

**优点：**

- 实现简单，不需要任何额外依赖
- 数据流向清晰，易于追踪
- React 原生支持，无需配置

**缺点：**

- 当层级过深时，代码会变得冗余
- 中间层的组件被迫接收它们不需要的 props
- 修改数据结构时，需要改动很多文件

### 2.2 Context（上下文）

Context 是 React 提供的内置方案，用于在组件树中传递数据，而无需显式地通过每一层级的 props 手动传递。

**核心概念：**

- **Provider（提供者）**：在组件树的上层定义，持有需要共享的数据
- **Consumer（消费者）**：在组件树的任意位置消费这些数据
- **useContext Hook**：函数组件中消费 Context 的现代方式

**优点：**

- 避免层层传递 props
- 可以跨层级共享数据
- React 原生支持

**缺点：**

- 过度使用会导致组件复用困难
- Context 的值变化会触发所有消费者重新渲染
- 不适合频繁变化的数据

```typescript
// 创建User Context
import { createContext, useContext, ReactNode, useState } from 'react';

// 定义Context的类型
interface UserContextType {
  // 当前登录用户
  currentUser: User | null;
  // 更新用户方法
  updateUser: (user: Partial<User>) => void;
  // 登出方法
  logout: () => void;
}

// 创建Context，初始值为undefined
const UserContext = createContext<UserContextType | undefined>(undefined);

// Provider组件：持有用户状态并提供操作方法
function UserProvider({ children }: { children: ReactNode }) {
  const [currentUser, setCurrentUser] = useState<User | null>(null);

  // 更新用户信息
  const updateUser = (updates: Partial<User>) => {
    if (currentUser) {
      setCurrentUser({ ...currentUser, ...updates });
    }
  };

  // 登出
  const logout = () => {
    setCurrentUser(null);
    localStorage.removeItem('auth_token');
  };

  const value: UserContextType = {
    currentUser,
    updateUser,
    logout
  };

  return (
    <UserContext.Provider value={value}>
      {children}
    </UserContext.Provider>
  );
}

// 自定义Hook：方便消费UserContext
function useUser() {
  const context = useContext(UserContext);
  if (context === undefined) {
    throw new Error('useUser必须在UserProvider内部使用');
  }
  return context;
}

// Theme Context：另一个常用场景
interface Theme {
  mode: 'light' | 'dark';
  primaryColor: string;
  backgroundColor: string;
}

interface ThemeContextType {
  theme: Theme;
  toggleTheme: () => void;
}

const ThemeContext = createContext<ThemeContextType | undefined>(undefined);

function ThemeProvider({ children }: { children: ReactNode }) {
  const [theme, setTheme] = useState<Theme>({
    mode: 'light',
    primaryColor: '#1890ff',
    backgroundColor: '#ffffff'
  });

  const toggleTheme = () => {
    setTheme(prev => ({
      ...prev,
      mode: prev.mode === 'light' ? 'dark' : 'light',
      backgroundColor: prev.mode === 'light' ? '#1a1a1a' : '#ffffff'
    }));
  };

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

// 使用示例：在任意深层组件中获取用户信息
function UserProfile() {
  const { currentUser, logout } = useUser();

  if (!currentUser) {
    return <div>请先登录</div>;
  }

  return (
    <div className="user-profile">
      <img src={currentUser.avatar} alt={currentUser.name} />
      <div>
        <h2>{currentUser.name}</h2>
        <button onClick={logout}>退出登录</button>
      </div>
    </div>
  );
}
```

### 2.3 事件总线（Event Bus）

事件总线是一种发布-订阅模式，用于解耦组件间的通信。当组件之间没有直接的父子关系，但需要通信时，事件总线是一个很好的选择。

**优点：**

- 完全解耦：发布者和订阅者不需要知道彼此的存在
- 可以跨任意组件通信
- 适合处理一次性事件或全局事件

**缺点：**

- 调试困难：事件流程不直观，难以追踪
- 容易滥用：过度使用会导致代码逻辑混乱
- 类型安全较差（需要额外处理）

```typescript
// 事件总线实现
type EventHandler<T = unknown> = (data: T) => void;

class EventBus {
  // 存储所有事件处理器
  private handlers: Map<string, Set<EventHandler>> = new Map();

  // 订阅事件
  on<T>(event: string, handler: EventHandler<T>): () => void {
    if (!this.handlers.has(event)) {
      this.handlers.set(event, new Set());
    }
    this.handlers.get(event)!.add(handler as EventHandler);

    // 返回取消订阅的函数
    return () => {
      this.handlers.get(event)?.delete(handler as EventHandler);
    };
  }

  // 发布事件
  emit<T>(event: string, data: T): void {
    const handlers = this.handlers.get(event);
    if (handlers) {
      handlers.forEach(handler => {
        try {
          handler(data);
        } catch (error) {
          console.error(`Error in event handler for "${event}":`, error);
        }
      });
    }
  }

  // 订阅一次性事件
  once<T>(event: string, handler: EventHandler<T>): void {
    const wrappedHandler = (data: T) => {
      handler(data);
      this.off(event, wrappedHandler);
    };
    this.on(event, wrappedHandler);
  }

  // 取消订阅
  off<T>(event: string, handler: EventHandler<T>): void {
    this.handlers.get(event)?.delete(handler);
  }

  // 清除所有处理器
  clear(): void {
    this.handlers.clear();
  }
}

// 创建全局事件总线实例
const eventBus = new EventBus();

// 定义事件类型
interface AppEvents {
  // 用户相关事件
  'user:login': { userId: string; token: string };
  'user:logout': void;
  'user:update': Partial<User>;

  // 帖子相关事件
  'post:created': Post;
  'post:deleted': { postId: string };
  'post:liked': { postId: string; likes: number };

  // 通知相关事件
  'notification:show': { type: 'success' | 'error' | 'info'; message: string };

  // 主题相关事件
  'theme:changed': 'light' | 'dark';
}

// 社交媒体Feed场景中的事件使用

// 1. 在Header组件中监听登录事件，显示用户信息
function Header() {
  const [isLoggedIn, setIsLoggedIn] = useState(false);

  useEffect(() => {
    // 订阅用户登录事件
    const unsubscribe = eventBus.on('user:login', (data) => {
      console.log('用户登录了:', data.userId);
      setIsLoggedIn(true);
    });

    return unsubscribe;
  }, []);

  return (
    <header className="app-header">
      <div className="logo">MySocial</div>
      <div className="user-area">
        {isLoggedIn ? <UserAvatar /> : <LoginButton />}
      </div>
    </header>
  );
}

// 2. 在Login组件中发布登录事件
function LoginForm() {
  const handleLogin = async (credentials: { email: string; password: string }) => {
    try {
      const response = await fetch('/api/auth/login', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(credentials)
      });

      if (response.ok) {
        const data = await response.json();
        // 登录成功后发布事件
        eventBus.emit('user:login', {
          userId: data.user.id,
          token: data.token
        });
        // 同时显示通知
        eventBus.emit('notification:show', {
          type: 'success',
          message: '登录成功！'
        });
      }
    } catch (error) {
      eventBus.emit('notification:show', {
        type: 'error',
        message: '登录失败，请重试'
      });
    }
  };

  return <form onSubmit={handleLogin}>...</form>;
}

// 3. 在Feed组件中监听新帖子事件，实时更新
function Feed() {
  const [posts, setPosts] = useState<Post[]>([]);

  useEffect(() => {
    // 监听新帖子创建事件
    const unsubscribe = eventBus.on('post:created', (newPost) => {
      // 将新帖子添加到列表开头
      setPosts(prev => [newPost, ...prev]);
    });

    return unsubscribe;
  }, []);

  return <div className="feed">{posts.map(post => <PostCard key={post.id} post={post} />)}</div>;
}

// 4. 在PostCard组件中监听点赞事件，更新本地状态
function PostCard({ post }: { post: Post }) {
  const [likes, setLikes] = useState(post.likes);
  const [hasLiked, setHasLiked] = useState(false);

  useEffect(() => {
    // 监听其他组件对同一个帖子的点赞
    const unsubscribe = eventBus.on('post:liked', (data) => {
      if (data.postId === post.id) {
        setLikes(data.likes);
      }
    });

    return unsubscribe;
  }, [post.id, post.likes]);

  const handleLike = () => {
    const newHasLiked = !hasLiked;
    const newLikes = newHasLiked ? likes + 1 : likes - 1;

    // 乐观更新：立即更新UI
    setHasLiked(newHasLiked);
    setLikes(newLikes);

    // 发布事件通知其他组件
    eventBus.emit('post:liked', { postId: post.id, likes: newLikes });

    // 发送请求到服务器
    fetch(`/api/posts/${post.id}/like`, { method: newHasLiked ? 'POST' : 'DELETE' })
      .catch(() => {
        // 失败时回滚
        setHasLiked(!newHasLiked);
        setLikes(newHasLiked ? newLikes - 1 : newLikes + 1);
        eventBus.emit('post:liked', {
          postId: post.id,
          likes: newHasLiked ? newLikes - 1 : newLikes + 1
        });
      });
  };

  return (
    <button onClick={handleLike} className={hasLiked ? 'liked' : ''}>
      &#x2764; {likes}
    </button>
  );
}
```

## 第三部分：数据获取策略

### 3.1 为什么需要专门的数据获取库？

在前端开发中，我们需要从服务器获取数据。传统的方式是使用`fetch`或`axios`，但这种方式存在一些问题：

1. **重复请求**：用户可能在不同页面需要同一份数据，但没有缓存机制
2. **加载状态**：需要手动管理 loading、error、data 等状态
3. **缓存同步**：当数据更新后，如何让所有使用这个数据的地方都更新？
4. **预加载**：如何在用户访问之前就提前准备好数据？
5. **乐观更新**：如何让用户感觉操作是即时完成的？

数据获取库（如 SWR 和 React Query / TanStack Query）正是为了解决这些问题而生的。

### 3.2 SWR：Stale-While-Revalidate

SWR 是一个由 Vercel 开发的 React Hooks 库，用于数据获取和缓存。它的名字来源于 HTTP 的 stale-while-revalidate 缓存策略。

**SWR 的核心特性：**

- **缓存优先**：首先返回缓存数据（可能是旧的），然后发送请求获取新数据
- **自动重新验证**：当窗口重新获得焦点时会自动重新请求
- **乐观更新**：支持乐观更新模式
- **预加载**：可以在用户访问之前就预加载数据
- **分页和无限滚动**：内置支持分页数据

```typescript
import useSWR from 'swr';

// 创建fetcher函数
const fetcher = (url: string) => fetch(url).then(res => {
  if (!res.ok) throw new Error('Network response was not ok');
  return res.json();
});

// 基础使用：获取帖子列表
function Feed() {
  // 使用SWR获取数据
  const { data, error, isLoading } = useSWR<Post[]>('/api/posts', fetcher, {
    // 数据freshness：1分钟内不需要重新请求
    revalidateOnFocus: true,      // 窗口聚焦时重新验证
    revalidateOnReconnect: true,  // 网络重新连接时重新验证
    dedupingInterval: 2000,       // 2秒内相同请求去重
    refreshInterval: 0,           // 不自动轮询
  });

  if (isLoading) return <Skeleton posts={3} />;
  if (error) return <ErrorMessage message="加载失败，请刷新重试" />;
  if (!data) return null;

  return (
    <div className="feed">
      {data.map(post => (
        <PostCard key={post.id} post={post} />
      ))}
    </div>
  );
}

// 获取单个帖子详情
function PostDetail({ postId }: { postId: string }) {
  const { data, error, isLoading, mutate } = useSWR<Post>(`/api/posts/${postId}`, fetcher);

  // 手动触发重新获取
  const refreshPost = () => mutate();

  if (isLoading) return <PostSkeleton />;
  if (error) return <ErrorMessage message="帖子不存在或已被删除" />;
  if (!data) return null;

  return (
    <article className="post-detail">
      <h1>{data.content}</h1>
      <button onClick={refreshPost}>刷新</button>
    </article>
```

### 3.3 TanStack Query（React Query）

TanStack Query（原名 React Query）是目前最强大的服务端状态管理库之一。与 SWR 相比，它提供了更多的功能和更细粒度的控制。

**TanStack Query vs SWR：**

| 特性 | TanStack Query | SWR |
|------|----------------|-----|
| 缓存管理 | 强大，支持精确缓存控制 | 基础缓存 |
| 乐观更新 | 原生支持 | 需要手动实现 |
| 无限滚动 | 原生支持 | 需额外配置 |
| DevTools | 官方提供 | 社区提供 |
| 包裹（wrappers） | 多框架支持 | 主要React |

```typescript
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';

// 创建API客户端
const apiClient = {
  get: <T>(url: string) => fetch(url).then(r => r.json() as Promise<T>),
  post: <T>(url: string, data: unknown) =>
    fetch(url, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(data)
    }).then(r => r.json() as Promise<T>),
  delete: (url: string) => fetch(url, { method: 'DELETE' }).then(r => r.json())
};

// 查询键管理（推荐的做法，便于缓存复用和失效）
const queryKeys = {
  posts: ['posts'] as const,
  post: (id: string) => ['posts', id] as const,
  userPosts: (userId: string) => ['users', userId, 'posts'] as const,
  user: (id: string) => ['users', id] as const,
};

// 获取帖子列表
function usePosts() {
  return useQuery({
    queryKey: queryKeys.posts,
    queryFn: () => apiClient.get<Post[]>('/api/posts'),
    // 基础配置
    staleTime: 1000 * 60 * 5,        // 数据5分钟内是新鲜的
    gcTime: 1000 * 60 * 10,         // 缓存10分钟后被垃圾回收
    refetchOnWindowFocus: true,     // 窗口聚焦时重新获取
    retry: 3,                        // 失败重试3次
    // 初始数据（SSR场景下有用）
    initialData: [],
  });
}

// 获取单个帖子
function usePost(postId: string) {
  return useQuery({
    queryKey: queryKeys.post(postId),
    queryFn: () => apiClient.get<Post>(`/api/posts/${postId}`),
    enabled: !!postId, // 只有当postId存在时才执行
  });
}

// 创建帖子的mutation
function useCreatePost() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (newPost: { content: string }) =>
      apiClient.post<Post>('/api/posts', newPost),

    // 成功后的回调
    onSuccess: (newPost) => {
      // 使帖子列表缓存失效
      queryClient.invalidateQueries({ queryKey: queryKeys.posts });

      // 或者使用setQueriesData进行乐观更新
      queryClient.setQueriesData<Post[]>({ queryKey: queryKeys.posts }, (oldData) => {
        return oldData ? [newPost, ...oldData] : [newPost];
      });
    },

    // 失败回调
    onError: (error, newPost, context) => {
      console.error('创建帖子失败:', error);
      // 可以回滚操作
    },
  });
}

// 点赞帖子的mutation（带乐观更新）
function useLikePost() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: ({ postId, action }: { postId: string; action: 'like' | 'unlike' }) =>
      apiClient.post<{ likes: number }>(
        `/api/posts/${postId}/${action}`,
        {}
      ),

    // 乐观更新：在请求发送前先更新UI
    onMutate: async ({ postId, action }) => {
      // 取消所有正在进行的对同一个帖子的请求（防止竞态）
      await queryClient.cancelQueries({ queryKey: queryKeys.post(postId) });

      // 保存当前数据，用于可能的回滚
      const previousPost = queryClient.getQueryData<Post>(queryKeys.post(postId));

      // 立即更新缓存
      queryClient.setQueryData<Post>(queryKeys.post(postId), (old) => {
        if (!old) return old;
        return {
          ...old,
          likes: action === 'like' ? old.likes + 1 : old.likes - 1,
          hasLiked: action === 'like'
        };
      });

      // 同时更新帖子列表缓存
      queryClient.setQueriesData<Post[]>({ queryKey: queryKeys.posts }, (oldPosts) => {
        if (!oldPosts) return oldPosts;
        return oldPosts.map(post =>
          post.id === postId
            ? { ...post, likes: action === 'like' ? post.likes + 1 : post.likes - 1 }
            : post
        );
      });

      // 返回上下文，用于在失败时回滚
      return { previousPost };
    },

    // 请求失败时的回滚逻辑
    onError: (err, variables, context) => {
      // 回滚帖子详情
      if (context?.previousPost) {
        queryClient.setQueryData(queryKeys.post(variables.postId), context.previousPost);
      }
      // 回滚帖子列表
      queryClient.invalidateQueries({ queryKey: queryKeys.posts });
    },

    // 请求完成后（无论成功失败）清道
    onSettled: (data, error, variables) => {
      // 重新获取最新数据以确保一致
      queryClient.invalidateQueries({ queryKey: queryKeys.post(variables.postId) });
    },
  });
}

// 删除帖子的mutation
function useDeletePost() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (postId: string) => apiClient.delete(`/api/posts/${postId}`),

    onSuccess: (_, postId) => {
      // 从缓存中移除被删除的帖子
      queryClient.setQueriesData<Post[]>({ queryKey: queryKeys.posts }, (oldPosts) => {
        return oldPosts?.filter(post => post.id !== postId) ?? [];
      });
    },
  });
}

// Feed组件：组合使用多个hook
function Feed() {
  // 获取帖子列表
  const { data: posts, isLoading, error, refetch } = usePosts();

  // 创建帖子的mutation
  const createPost = useCreatePost();

  // 点赞的mutation
  const likePost = useLikePost();

  // 处理新帖子提交
  const handleCreatePost = (content: string) => {
    createPost.mutate({ content });
  };

  // 处理点赞
  const handleLike = (postId: string, hasLiked: boolean) => {
    likePost.mutate({
      postId,
      action: hasLiked ? 'unlike' : 'like'
    });
  };

  if (isLoading) {
    return <FeedSkeleton count={5} />;
  }

  if (error) {
    return (
      <div className="error-container">
        <p>加载失败</p>
        <button onClick={() => refetch()}>重试</button>
      </div>
    );
  }

  return (
    <div className="feed">
      {/* 发布新帖子区域 */}
      <CreatePostForm onSubmit={handleCreatePost} isPending={createPost.isPending} />

      {/* 帖子列表 */}
      <div className="posts-list">
        {posts.map(post => (
          <PostCard
            key={post.id}
            post={post}
            onLike={() => handleLike(post.id, post.hasLiked)}
            isLiking={likePost.isPending}
          />
        ))}
      </div>
    </div>
  );
}
```

### 4.1 Zustand

Zustand 是一个轻量级、简洁的 React 状态管理库。它不需要 Provider 包裹，使用 hooks 的方式管理状态，非常直观。

**Zustand 的特点：**

- **极简API**：不需要 Provider
- **灵活更新**：可以直接修改状态，也可以通过函数更新
- **中间件支持**：可以添加 persist、redux 风格的 reducers 等
- **TypeScript 友好**：完整的类型推断

```typescript
import { create } from 'zustand';
import { persist, createJSONStorage } from 'zustand/middleware';

// 定义状态类型
interface UserState {
  // 状态
  currentUser: User | null;
  isAuthenticated: boolean;
  preferences: {
    theme: 'light' | 'dark';
    language: 'zh-CN' | 'en-US';
    notifications: boolean;
  };
  // 操作
  login: (user: User) => void;
  logout: () => void;
  updatePreferences: (prefs: Partial<UserState['preferences']>) => void;
}

// 创建带持久化的用户store
const useUserStore = create<UserState>()(
  persist(
    (set, get) => ({
      // 初始状态
      currentUser: null,
      isAuthenticated: false,
      preferences: {
        theme: 'light',
        language: 'zh-CN',
        notifications: true,
      },

      // 登录：设置用户并标记已认证
      login: (user) => set({
        currentUser: user,
        isAuthenticated: true
      }),

      // 登出：清除用户并重置认证状态
      logout: () => set({
        currentUser: null,
        isAuthenticated: false
      }),

      // 更新偏好设置（浅合并）
      updatePreferences: (prefs) => set((state) => ({
        preferences: { ...state.preferences, ...prefs }
      })),
    }),
    {
      // 持久化配置
      name: 'user-storage', // localStorage的key
      storage: createJSONStorage(() => localStorage),
      // 只持久化这些字段
      partialize: (state) => ({
        currentUser: state.currentUser,
        preferences: state.preferences,
      }),
    }
  )
);

// 帖子Store：包含更多操作
interface PostState {
  posts: Post[];
  currentPost: Post | null;
  isLoading: boolean;
  error: string | null;
  // 操作
  fetchPosts: () => Promise<void>;
  fetchPost: (id: string) => Promise<void>;
  addPost: (post: Post) => void;
  updatePost: (id: string, updates: Partial<Post>) => void;
  deletePost: (id: string) => void;
  likePost: (id: string) => void;
  unlikePost: (id: string) => void;
}

const usePostStore = create<PostState>((set, get) => ({
  // 初始状态
  posts: [],
  currentPost: null,
  isLoading: false,
  error: null,

  // 获取所有帖子
  fetchPosts: async () => {
    set({ isLoading: true, error: null });
    try {
      const response = await fetch('/api/posts');
      if (!response.ok) throw new Error('获取帖子失败');
      const posts = await response.json();
      set({ posts, isLoading: false });
    } catch (error) {
      set({ error: (error as Error).message, isLoading: false });
    }
  },

  // 获取单个帖子
  fetchPost: async (id) => {
    set({ isLoading: true, error: null });
    try {
      const response = await fetch(`/api/posts/${id}`);
      if (!response.ok) throw new Error('获取帖子详情失败');
      const post = await response.json();
      set({ currentPost: post, isLoading: false });
    } catch (error) {
      set({ error: (error as Error).message, isLoading: false });
    }
  },

  // 添加帖子（通常用于乐观更新后同步）
  addPost: (post) => set((state) => ({
    posts: [post, ...state.posts]
  })),

  // 更新帖子
  updatePost: (id, updates) => set((state) => ({
    posts: state.posts.map(post =>
      post.id === id ? { ...post, ...updates } : post
    ),
    currentPost: state.currentPost?.id === id
      ? { ...state.currentPost, ...updates }
      : state.currentPost
  })),

  // 删除帖子
  deletePost: (id) => set((state) => ({
    posts: state.posts.filter(post => post.id !== id),
    currentPost: state.currentPost?.id === id ? null : state.currentPost
  })),

  // 点赞
  likePost: (id) => set((state) => ({
    posts: state.posts.map(post =>
      post.id === id
        ? { ...post, likes: post.likes + 1, hasLiked: true }
        : post
    ),
    currentPost: state.currentPost?.id === id
      ? { ...state.currentPost, likes: state.currentPost.likes + 1, hasLiked: true }
      : state.currentPost
  })),

  // 取消点赞
  unlikePost: (id) => set((state) => ({
    posts: state.posts.map(post =>
      post.id === id
        ? { ...post, likes: Math.max(0, post.likes - 1), hasLiked: false }
        : post
    ),
    currentPost: state.currentPost?.id === id
      ? { ...state.currentPost, likes: Math.max(0, state.currentPost.likes - 1), hasLiked: false }
      : state.currentPost
  })),
}));

// 在组件中使用
```

### 4.2 Jotai

Jotai 是一个原子化（Atomic）状态管理库，灵感来自 Recoil，但更加轻量和简洁。

**Jotai 的特点：**

- **原子化设计**：状态由最小的原子组成，按需读取
- **极致性能**：只有使用的原子才会导致重渲染
- **TypeScript 原生支持**：完整类型推断
- **React Suspense 集成**：可以与 Suspense 无缝配合

```typescript
import { atom, useAtom, useAtomValue, useSetAtom } from 'jotai';
import { atomWithStorage } from 'jotai/utils';

// 创建原子：最小的状态单元
// 用户原子
const userAtom = atomWithStorage<User | null>('user', null);

// 帖子列表原子
const postsAtom = atom<Post[]>([]);

// 加载状态原子
const postsLoadingAtom = atom<boolean>(false);

// 错误状态原子
const postsErrorAtom = atom<string | null>(null);

// 派生原子：从其他原子计算得出的值
// 当前用户是否登录
const isAuthenticatedAtom = atom((get) => get(userAtom) !== null);

// 获取当前用户的帖子
const currentUserPostsAtom = atom((get) => {
  const user = get(userAtom);
  const posts = get(postsAtom);
  if (!user) return [];
  return posts.filter(post => post.author.id === user.id);
});

// 获取帖子的评论数（通过派生计算）
const postCommentsCountAtom = atom((get) => (postId: string) => {
  const posts = get(postsAtom);
  const post = posts.find(p => p.id === postId);
  return post?.comments ?? 0;
});

// 异步原子：用于数据获取
const fetchPostsAtom = atom(null, async (get, set) => {
  set(postsLoadingAtom, true);
  set(postsErrorAtom, null);

  try {
    const response = await fetch('/api/posts');
    if (!response.ok) throw new Error('获取失败');
    const posts = await response.json();
    set(postsAtom, posts);
  } catch (error) {
    set(postsErrorAtom, (error as Error).message);
  } finally {
    set(postsLoadingAtom, false);
  }
});

// 组合原子：管理更复杂的状态
const likePostAtom = atom(
  null,
  async (get, set, { postId, action }: { postId: string; action: 'like' | 'unlike' }) => {
    // 乐观更新
    const posts = get(postsAtom);
    const postIndex = posts.findIndex(p => p.id === postId);

    if (postIndex !== -1) {
      const updatedPosts = [...posts];
      const post = updatedPosts[postIndex];

      updatedPosts[postIndex] = {
        ...post,
        likes: action === 'like' ? post.likes + 1 : Math.max(0, post.likes - 1),
        hasLiked: action === 'like'
      };

      set(postsAtom, updatedPosts);
    }

    // 发送请求
    try {
      await fetch(`/api/posts/${postId}/${action}`, { method: 'POST' });
    } catch (error) {
      // 回滚
      set(postsAtom, posts);
      console.error('操作失败:', error);
    }
  }
);

// 在组件中使用Jotai
function Feed() {
  // 获取帖子列表
  const posts = useAtomValue(postsAtom);

  // 获取加载状态
  const isLoading = useAtomValue(postsLoadingAtom);

  // 获取错误
  const error = useAtomValue(postsErrorAtom);

  // 获取操作函数
  const fetchPosts = useSetAtom(fetchPostsAtom);

  useEffect(() => {
    fetchPosts();
  }, [fetchPosts]);

  if (isLoading) return <Skeleton />;
  if (error) return <ErrorMessage message={error} />;

  return (
    <div className="feed">
      {posts.map(post => (
        <PostCard key={post.id} post={post} />
      ))}
    </div>
  );
}

// 精细订阅：只获取需要的字段
function LikeButton({ postId }: { postId: string }) {
  // 只订阅这个帖子的hasLiked状态
  const hasLiked = useAtomValue(
    atom((get) => get(postsAtom).find(p => p.id === postId)?.hasLiked ?? false)
  );

  const likePost = useSetAtom(likePostAtom);

  return (
    <button onClick={() => likePost({ postId, action: hasLiked ? 'unlike' : 'like' })}>
      {hasLiked ? '已赞' : '赞'}
    </button>
  );
}

// 评论数显示：使用派生原子函数
function CommentsCount({ postId }: { postId: string }) {
  const getCommentsCount = useAtomValue(postCommentsCountAtom);

  return <span>{getCommentsCount(postId)} 条评论</span>;
}
```

### 4.3 Recoil

Recoil 是 Facebook（现Meta）开发的状态管理库，采用了原子图（Atom Graph）的概念。

**Recoil 的特点：**

- **原子图**：状态由原子组成，组件按需订阅
- **派生数据**：支持同步和异步的派生选择器
- **React Suspense 支持**：可以与 Suspense 集成
- **时间旅行调试**：支持 Redux DevTools 类似的调试功能

```typescript
import { atom, selector, useRecoilState, useRecoilValue, useSetRecoilState } from 'recoil';

// 定义原子：最小的状态单元
const userState = atom<User | null>({
  key: 'userState',
  default: null,
});

const postsState = atom<Post[]>({
  key: 'postsState',
  default: [],
});

const postsLoadingState = atom<boolean>({
  key: 'postsLoadingState',
  default: false,
});

const postsErrorState = atom<string | null>({
  key: 'postsErrorState',
  default: null,
});
```

## 第五部分：乐观更新详解

### 5.1 什么是乐观更新？

乐观更新（Optimistic Update）是一种提升用户体验的技术：当你执行一个操作（如点赞、发帖）时，不等待服务器响应，立即更新用户界面，让用户感觉操作已经成功了。只有当服务器返回错误时，才回滚到之前的状态。

**传统更新流程（悲观更新）：**
```
用户点击点赞 → 显示loading → 等待服务器响应 → 更新UI
总耗时：200-500ms（用户感受到延迟）
```

**乐观更新流程：**
```
用户点击点赞 → 立即更新UI → 后台发送请求 → 如果失败则回滚
总耗时：0ms（用户感觉是即时完成）
```

### 5.2 乐观更新的实现模式

```typescript

import { useState, useCallback } from 'react';

// 点赞的乐观更新Hook
function useOptimisticLike(post: Post) {
  // 本地状态
  const [likes, setLikes] = useState(post.likes);
  const [hasLiked, setHasLiked] = useState(post.hasLiked);
  const [isError, setIsError] = useState(false);

  // 点赞处理
  const handleLike = useCallback(async () => {
    // 1. 保存当前状态（用于可能的回滚）
    const previousLikes = likes;
    const previousHasLiked = hasLiked;

    // 2. 立即更新UI（乐观更新）
    setHasLiked(!previousHasLiked);
    setLikes(previousHasLiked ? previousLikes - 1 : previousLikes + 1);
    setIsError(false);

    try {
      // 3. 发送请求到服务器
      const response = await fetch(
        `/api/posts/${post.id}/${previousHasLiked ? 'unlike' : 'like'}`,
        { method: 'POST' }
      );

      if (!response.ok) throw new Error('操作失败');

      // 4. 可选：服务器返回最新状态，用服务器状态覆盖本地状态
      // const updatedPost = await response.json();
      // setLikes(updatedPost.likes);
      // setHasLiked(updatedPost.hasLiked);

    } catch (error) {
      // 5. 失败时回滚
      setLikes(previousLikes);
      setHasLiked(previousHasLiked);
      setIsError(true);

      console.error('点赞失败:', error);
    }
  }, [likes, hasLiked, post.id]);

  return { likes, hasLiked, isError, handleLike };
}

  const { likes, hasLiked, isError, handleLike } = useOptimisticLike(post);

  return (

        onClick={handleLike}
        className={hasLiked ? 'liked' : ''}
        disabled={isError}
      >
        {hasLiked ? '♥' : '♡'} {likes}
      </button>
      {isError && <span className="error-text">操作失败</span>}

// 结合React Query的乐观更新
function useOptimisticLikeWithQuery(postId: string) {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: ({ action }: { action: 'like' | 'unlike' }) =>
      apiClient.post(`/api/posts/${postId}/${action}`, {}),

    // 在mutation函数执行前触发
    onMutate: async ({ action }) => {
      // 取消任何正在进行的请求
      await queryClient.cancelQueries({ queryKey: ['post', postId] });

      // 保存当前数据
      const previousPost = queryClient.getQueryData(['post', postId]);

      // 乐观更新
      queryClient.setQueryData(['post', postId], (old: Post | undefined) => {
        if (!old) return old;
        return {
          ...old,
          likes: action === 'like' ? old.likes + 1 : Math.max(0, old.likes - 1),
          hasLiked: action === 'like'
        };
      });

      // 返回上下文
      return { previousPost };
    },

    // 失败时回滚
    onError: (err, variables, context) => {
      if (context?.previousPost) {
        queryClient.setQueryData(['post', postId], context.previousPost);
      }
    },

    // 完成后总是重新验证
    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ['post', postId] });
    },
  });
}
```

## 第六部分：骨架屏Loading

### 6.1 骨架屏的概念与优势

骨架屏（Skeleton Screen）是一种加载占位符，它展示了内容的预期布局，让用户在等待内容加载时有一个大致的预期。相比传统的 Spinner，骨架屏有如下优势：

- **感知加载更快**：用户看到内容结构，心理上感觉更快
- **减少焦虑**：明确的预期比空白屏幕或转圈更友好
- **渐进式体验**：内容逐步出现，而非突然全部显示

## 第七部分：数据缓存与失效策略

### 7.1 缓存策略基础

缓存策略决定了数据何时被获取、何时被视为过期、何时被清除。合理的缓存策略可以大幅提升应用性能和用户体验。

```typescript
// 缓存配置类型
interface CacheConfig {
  // 缓存key
  key: string;
  // 缓存数据
  data: unknown;
  // 缓存时间戳
  timestamp: number;
  // 过期时间（毫秒）
  ttl: number;
}

// 缓存管理器类
class CacheManager {
  private cache: Map<string, CacheConfig> = new Map();

  // 设置缓存
  set(key: string, data: unknown, ttl: number = 5 * 60 * 1000) {
    this.cache.set(key, {
      key,
      data,
      timestamp: Date.now(),
      ttl
    });
  }

  // 获取缓存
  get<T>(key: string): T | null {
    const item = this.cache.get(key);
    if (!item) return null;

    const isExpired = Date.now() - item.timestamp > item.ttl;
    if (isExpired) {
      this.cache.delete(key);
      return null;
    }

    return item.data as T;
  }

  // 判断缓存是否新鲜（未过期）
  isFresh(key: string): boolean {
    const item = this.cache.get(key);
    if (!item) return false;
    return Date.now() - item.timestamp <= item.ttl;
  }

  // 使缓存失效
  invalidate(key: string) {
    this.cache.delete(key);
  }

  // 清除所有缓存
  clear() {
    this.cache.clear();
  }

  // 清除过期缓存
  prune() {
    const now = Date.now();
    for (const [key, item] of this.cache.entries()) {
      if (now - item.timestamp > item.ttl) {
        this.cache.delete(key);
      }
    }
  }
}

// 使用示例
const cacheManager = new CacheManager();

// 获取帖子列表（带缓存）
async function fetchPosts(forceRefresh = false): Promise<Post[]> {
  const cacheKey = 'posts:list';

  // 如果不是强制刷新，先尝试从缓存获取
  if (!forceRefresh) {
    const cached = cacheManager.get<Post[]>(cacheKey);
    if (cached) {
      console.log('从缓存获取帖子列表');
      return cached;
    }
  }

  // 从API获取
  console.log('从API获取帖子列表');
  const response = await fetch('/api/posts');
  const posts = await response.json();

  // 存入缓存，5分钟过期
  cacheManager.set(cacheKey, posts, 5 * 60 * 1000);

  return posts;
}
```

### 7.2 缓存失效策略

缓存失效是状态同步中最复杂的部分。常见的失效策略有：

**1. 基于时间的失效（TTL）**
```typescript
// 设置带TTL的缓存
function setWithTTL<T>(key: string, value: T, ttlMs: number): void {
  localStorage.setItem(key, JSON.stringify({
    value,
    expiry: Date.now() + ttlMs
  }));
}

// 获取缓存，自动处理过期
function getWithTTL<T>(key: string): T | null {
  const item = localStorage.getItem(key);
  if (!item) return null;

  const { value, expiry } = JSON.parse(item);

  if (Date.now() > expiry) {
    localStorage.removeItem(key);
    return null;
  }

  return value as T;
}
```

**2. 基于版本号的失效**
```typescript
// 缓存版本管理器
const cacheVersion = 'v1';
```

```typescript
// 层级缓存管理
const tieredCache = {
  // L1: 内存缓存（最快，容量小）
  memory: new Map<string, CacheConfig>(),

  // L2: localStorage缓存（较慢，容量大）
  local: new CacheManager(),

  // 设置数据（同时写入多层）
  set<T>(key: string, value: T, ttlMs: number) {
    // L1: 存入内存
    this.memory.set(key, {
      key,
      data: value,
      timestamp: Date.now(),
      ttl: ttlMs
    });

    // L2: 存入localStorage
    this.local.set(key, value, ttlMs);
  },

  // 获取数据（从最快层级开始）
  get<T>(key: string): T | null {
    // L1检查
    const memItem = this.memory.get(key);
    if (memItem && Date.now() - memItem.timestamp <= memItem.ttl) {
      return memItem.data as T;
    }

    // L2检查
    const localItem = this.local.get<T>(key);
    if (localItem) {
      // 恢复L1
      this.memory.set(key, {
        key,
        data: localItem,
        timestamp: Date.now(),
        ttl: 60000 // 1分钟
      });
      return localItem;
    }

    return null;
  }
};
```

```typescript
    setOptimisticLikes(prevHasLiked ? prevLikes - 1 : prevLikes + 1);
    setIsError(false);

    try {
      // 发送请求
      if (prevHasLiked) {
        await api.unlikePost(post.id);
      } else {
        await api.likePost(post.id);
      }
      // 通知父组件刷新数据
      onMutate();
    } catch (error) {
      // 失败回滚
      setOptimisticLikes(prevLikes);
      setOptimisticHasLiked(prevHasLiked);
      setIsError(true);
      console.error('点赞失败:', error);
    }
  };

  return (
    <button
      className={`like-button ${optimisticHasLiked ? 'liked' : ''}`}
      onClick={handleLike}
      disabled={isError}
    >
      <span className="like-icon">{optimisticHasLiked ? '❤️' : '🤍'}</span>
      <span className="like-count">{optimisticLikes}</span>
    </button>
  );
}
```

## 第九部分：总结与最佳实践

### 9.1 状态同步策略选择指南

面对不同的场景，我们应该如何选择合适的状态同步方案？

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 简单的父子组件通信 | Props | 最简单直接 |
| 跨多层级的全局配置（主题、语言） | Context | 原生支持，无需依赖 |
| 一次性事件（登录成功、通知） | 事件总线 | 完全解耦 |
| 服务端数据获取与缓存 | SWR / TanStack Query | 专为数据获取设计 |
| 复杂客户端状态 | Zustand / Jotai | 轻量且灵活 |
| 需要乐观更新 | TanStack Query / Zustand + 自定义 | 内置支持 |
| 需要调试工具 | Recoil / TanStack Query | DevTools支持 |

### 9.2 性能优化建议

**1. 避免不必要的重渲染**

```typescript
// 不好：整个组件在任何状态变化时都会重渲染
function Feed() {
  const posts = useStore(state => state.posts);
  const user = useStore(state => state.user); // user变化也会导致重渲染
  // ...
}

// 好：使用选择器精细控制
function Feed() {
  const posts = useStore(useShallow(state => state.posts));
  // user变化不会导致Feed重渲染
}

// 好：使用Zustand的派生
const postsSelector = (state) => state.posts;
function Feed() {
  const posts = useStore(postsSelector);
  // ...
}
```

**2. 合理设置缓存策略**

```typescript
// SWR配置示例
const { data } = useSWR('/api/posts', fetcher, {
  // 5分钟内数据是新鲜的
  staleTime: 1000 * 60 * 5,

  // 窗口聚焦时重新验证（可选）
  revalidateOnFocus: true,

  // 失败重试2次
  retry: 2,

  // dedupe: 5秒内相同请求去重
  dedupingInterval: 5000,
});
```

**3. 批量更新减少渲染次数**

```typescript
// Zustand批量更新
const useStore = create((set) => ({
  // 批量更新：只触发一次重渲染
  updateMultiple: (data) => set((state) => ({
    ...state,
    ...data
  })),
}));
```

### 9.3 常见问题与解决方案

**问题1：Context导致整个组件树重渲染**

解决方案：拆分Context，只订阅需要部分

```typescript
// 不好：所有消费者都会在任何一个值变化时重渲染
const AppContext = createContext({ user, theme, settings });

// 好：拆分为多个独立的Context
const UserContext = createContext<User | null>(null);
const ThemeContext = createContext<Theme | null>(null);
const SettingsContext = createContext<Settings | null>(null);
```

**问题2：数据获取库与全局状态管理冲突**

解决方案：明确职责边界

- **TanStack Query / SWR**：负责服务端状态（来自API的数据）
- **Zustand / Jotai**：负责客户端状态（UI状态、用户偏好）

两者可以共存，分别管理不同类型的state。

**问题3：乐观更新失败后的回滚不一致**

解决方案：使用事务式更新

```typescript
// TanStack Query的onMutate + onError组合确保一致性
useMutation({
  mutationFn: likePost,

  onMutate: async () => {
    // 取消正在进行的请求
    await queryClient.cancelQueries(['posts']);

    // 快照之前的数据
    const previous = queryClient.getQueryData(['posts']);

    // 乐观更新
    queryClient.setQueryData(['posts'], (old) => /* 更新逻辑 */);

    return { previous };
  },

  onError: (err, variables, context) => {
    // 失败时回滚到快照
    queryClient.setQueryData(['posts'], context.previous);
  },

  onSettled: () => {
    // 无论成功失败，都重新获取最新数据
    queryClient.invalidateQueries(['posts']);
  },
});
```
