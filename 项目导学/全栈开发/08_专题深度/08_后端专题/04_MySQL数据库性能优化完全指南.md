## 二、索引原理深度剖析

### 2.1 索引是什么？

**索引**就像**餐厅的菜单簿**：没有菜单时，想找某道菜需要翻遍整本书；有菜单时，按拼音/分类快速定位。索引的本质是**减少遍历的数据结构**。

```
┌────────────────────────────────────────────────────────────────────┐
│                           索引加速原理                               │
├────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  无索引（全表扫描）：                                                 │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │  查询：WHERE name = '宫保鸡丁'                                │ │
│  │                                                              │ │
│  │  数据：                                                       │ │
│  │  [宫保鸡丁] ← 找到了！                                        │ │
│  │  [麻婆豆腐] ← 没找到，继续                                    │ │
│  │  [糖醋里脊] ← 没找到，继续                                    │ │
│  │  ... 翻遍所有10000条记录                                       │ │
│  └──────────────────────────────────────────────────────────────┘ │
│  时间复杂度：O(n)                                                   │
│                                                                      │
│  有索引（B+树）：                                                    │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │  查询：WHERE name = '宫保鸡丁'                                │ │
│  │                                                              │ │
│  │  B+树索引：                                                   │ │
│  │                        ┌─────┐                               │ │
│  │                       /       \                              │ │
│  │                  ┌─────┐     ┌─────┐                        │ │
│  │                 /       \   /       \                       │ │
│  │              [宫-辣]  [面-糖]  [西-粤]                       │ │
│  │                 ↓        ↓       ↓                           │ │
│  │            宫保鸡丁   麻婆豆腐  ...                          │ │
│  │  查找路径：根节点 → 中间节点 → 叶子节点 → 定位记录             │ │
│  └──────────────────────────────────────────────────────────────┘ │
│  时间复杂度：O(log n)                                              │
│                                                                      │
└────────────────────────────────────────────────────────────────────┘
```

### 2.2 B+树数据结构

```sql
-- MySQL InnoDB使用B+树作为索引结构
-- B+树特性：
-- 1. 所有数据都存储在叶子节点
-- 2. 叶子节点之间用链表连接
-- 3. 适合范围查询

-- 创建示例表
CREATE TABLE employee (
    id INT PRIMARY KEY,           -- 主键索引（聚簇索引）
    name VARCHAR(50),             -- 普通索引
    age INT,
    department VARCHAR(50),
    salary DECIMAL(10,2),
    INDEX idx_name (name),        -- 普通索引
    INDEX idx_dept_age (department, age),  -- 联合索引
    FULLTEXT INDEX ft_name (name) -- 全文索引
) ENGINE=InnoDB;
```

### 2.3 聚簇索引 vs 非聚簇索引

```sql
-- 聚簇索引（Clustered Index）
-- 表数据按主键顺序存储，索引的叶子节点存储完整数据

-- 主键 = 聚簇索引
CREATE TABLE orders (
    order_id INT PRIMARY KEY,  -- 聚簇索引，叶子节点存储整行数据
    customer_name VARCHAR(50),
    amount DECIMAL(10,2),
    order_date DATE
);

-- 主键结构示意：
--
-- 聚簇索引：
-- ┌────────────────────────────────────────┐
-- │  主键ID   │  customer_name  │  amount  │  order_date │
-- ├───────────┼─────────────────┼──────────┼──────────────┤
-- │    1     │   张三          │  100.00  │  2024-01-01 │
-- │    2     │   李四          │  200.00  │  2024-01-02 │
-- │    3     │   王五          │  150.00  │  2024-01-03 │
-- └───────────┴─────────────────┴──────────┴──────────────┘
--           ↑
--      叶子节点存储完整数据行
--

-- 非聚簇索引（Secondary Index）
-- 叶子节点存储主键值，需要回表查询

CREATE INDEX idx_customer ON orders(customer_name);

-- 非聚簇索引结构：
--
-- 非聚簇索引：
-- ┌────────────────────┐
-- │  customer_name     │  主键值
-- ├────────────────────┼────────────────────┐
-- │   张三             │         1          │
-- │   李四             │         2          │
-- │   王五             │         3          │
-- └────────────────────┴────────────────────┘
--           ↑
--      叶子节点存储主键，需要"回表"获取完整数据
--

-- 回表查询示例
SELECT * FROM orders WHERE customer_name = '张三';

-- 执行流程：
-- 1. 在idx_customer索引树查找'张三'，得到主键ID=1
-- 2. 用ID=1回到聚簇索引查找完整数据行
-- 3. 返回完整数据
--
-- 优化：覆盖索引（无需回表）
SELECT customer_name, amount FROM orders WHERE customer_name = '张三';
-- 因为要查询的列(name, amount)都在索引中，无需回表
```

```java
/**
 * 索引类型详解
 */

/*
 * 索引分类：
 *
 * 1. 主键索引 (Primary Key)
 *    - 唯一且非空
 *    - 每个表只能有一个
 *    - InnoDB会自动为主键创建聚簇索引
 *
 * 2. 唯一索引 (Unique)
 *    - 值唯一，可以有空值
 *    - 可创建多个
 *    - 会自动创建非聚簇索引
 *
 * 3. 普通索引 (Index)
 *    - 无唯一性限制
 *    - 用于加速查询
 *
 * 4. 联合索引 (Composite)
 *    - 多个列组成的索引
 *    - 遵循最左前缀原则
 *
 * 5. 全文索引 (Full-text)
 *    - 用于文本搜索
 *    - MySQL 5.6+ InnoDB支持
 */

/*
 * 联合索引示例：
 *
 * INDEX idx_dept_name (department, name)
 *
 * 数据：
 * department    name       age
 * ---------------------------------
 * 研发部        张三       25
 * 研发部        李四       30
 * 销售部        王五       28
 * 销售部        赵六       35
 *
 * B+树结构：
 *                        ┌─────────────────┐
 *                       /                   \
 *              ┌─────────┴───┐       ┌─────────┴───┐
 *           研发部            销售部              ← 先按部门排序
 *           /   \              /   \
 *       张三    李四        王五    赵六            ← 相同部门内按名字排序
 *
 * 最左前缀原则：
 * - 可以使用：department
 * - 可以使用：department, name
 * - 不能使用：name（跳过部门）
 * - 不能使用：age（跳过部门、名字）
 */

/*
 * 覆盖索引示例：
 *
 * 查询：SELECT name, age FROM employee WHERE department = '研发部'
 *
 * 使用覆盖索引 idx_dept_name(department, name, age)：
 *
 * - 索引包含所有要查询的列
 * - 无需回表，直接从索引返回结果
 * - 性能最优
 *
 * EXPLAIN分析：
 * Using index condition: 使用了索引
 * Using index: 覆盖索引，无需回表
 */
```

### 2.4 索引失效场景

```sql
-- 索引失效的典型场景

-- 1. 对索引列使用函数或运算
SELECT * FROM orders WHERE YEAR(order_date) = 2024;
-- 失效原因：对YEAR()结果比较，无法使用索引
-- 优化：WHERE order_date >= '2024-01-01' AND order_date < '2025-01-01'

-- 2. 字符串不加引号
SELECT * FROM employee WHERE name = 123;
-- 失效原因：MySQL类型转换，导致索引失效
-- 优化：WHERE name = '123'

-- 3. LIKE以%开头
SELECT * FROM employee WHERE name LIKE '%张%';
-- 失效原因：无法确定前缀，无法使用B+树
-- 优化：使用ElasticSearch处理全文搜索
-- 可用：WHERE name LIKE '张%'

-- 4. OR条件中包含非索引列
SELECT * FROM employee WHERE name = '张三' OR age = 25;
-- 失效原因：age列没有索引，整个查询可能全表扫描
-- 优化：为age列创建索引，或拆分为UNION

-- 5. 使用NOT IN / NOT EXISTS
SELECT * FROM employee WHERE id NOT IN (1, 2, 3);
-- 失效原因：通常全表扫描
-- 优化：使用LEFT JOIN或NOT EXISTS

-- 6. 数据量过小时（优化器选择全表扫描）
-- 当MySQL认为全表扫描更快时，即使有索引也可能不使用
-- 解决：强制使用索引（不推荐），或优化统计信息

-- 7. 联合索引不满足最左前缀
-- 假设索引是 (A, B, C)
-- 可用：WHERE A = x
-- 可用：WHERE A = x AND B = y
-- 可用：WHERE A = x AND B = y AND C = z
-- 不可用：WHERE B = y
-- 不可用：WHERE C = z

-- 8. 索引列对比
SELECT * FROM employee WHERE age > 20;
-- 可用：age有索引
-- 但范围查询后，索引可能中断

-- 9. IS NULL不使用索引（部分情况）
SELECT * FROM employee WHERE name IS NULL;
-- 优化：可以使用覆盖索引
```

## 三、事务隔离级别

### 3.1 事务是什么？

**事务**就像**餐厅的点餐-做菜-上菜流程**：要么全套完成（提交），要么全套取消（回滚）。

```
┌────────────────────────────────────────────────────────────────────┐
│                           事务特性 (ACID)                            │
├────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Atomic（原子性）                                                    │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │  点餐：                                                       │  │
│  │  BEGIN → 下单 → 扣款 → 通知厨房 → 完成                       │  │
│  │  如果扣款成功但通知厨房失败，整个回滚！                        │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                      │
│  Consistency（一致性）                                               │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │  约束：                                                       │  │
│  │  - 用户余额 + 餐厅余额 = 总资产（不变）                        │  │
│  │  - 订单状态必须合法                                           │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                      │

│  ┌──────────────────────────────────────────────────────────────┐  │

│                                                                      │
│  Durability（持久性）                                                │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │  订单完成：                                                   │  │
│  │  - 数据必须持久化到磁盘                                        │  │
```

### 3.2 并发问题详解

```sql
-- 并发事务导致的经典问题

-- 1. 脏读 (Dirty Read)
-- 事务A读取了事务B未提交的数据
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
START TRANSACTION;
-- T1: 事务A读取用户余额
SELECT balance FROM account WHERE user_id = 1;  -- 余额1000
-- T2: 事务B扣款（未提交）
UPDATE account SET balance = balance - 500 WHERE user_id = 1;
-- T3: 事务A再次读取
SELECT balance FROM account WHERE user_id = 1;  -- 余额500（脏读）
-- T4: 事务B回滚
ROLLBACK;  -- 余额恢复1000
-- 事务A读到了不存在的数据！

-- 2. 不可重复读 (Non-Repeatable Read)
-- 事务A两次读取同一行数据，结果不同
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
START TRANSACTION;
-- T1: 事务A读取订单
SELECT * FROM orders WHERE id = 1;  -- 订单状态=待支付
-- T2: 事务B修改订单状态（提交）
UPDATE orders SET status = '已支付' WHERE id = 1;
COMMIT;
-- T3: 事务A再次读取
SELECT * FROM orders WHERE id = 1;  -- 订单状态=已支付（不可重复读）
COMMIT;

-- 3. 幻读 (Phantom Read)
-- 事务A两次查询，结果集数量不同
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
START TRANSACTION;
-- T1: 事务A查询订单
SELECT COUNT(*) FROM orders WHERE status = '待支付';  -- 5条
-- T2: 事务B插入新订单（提交）
INSERT INTO orders (status) VALUES ('待支付');
COMMIT;
-- T3: 事务A再次查询
SELECT COUNT(*) FROM orders WHERE status = '待支付';  -- 6条（幻读）
COMMIT;

-- 4. 丢失更新 (Lost Update)
-- 两个事务同时更新，后者覆盖了前者的更新
START TRANSACTION;
-- T1: 事务A读取商品库存=10
SELECT stock FROM product WHERE id = 1;  -- stock=10
-- T2: 事务B读取商品库存=10
SELECT stock FROM product WHERE id = 1;  -- stock=10
-- T3: 事务A更新库存=10-3=7
UPDATE product SET stock = 10 - 3 WHERE id = 1;  -- stock=7
-- T4: 事务B更新库存=10-5=5（覆盖了事务A的更新！）
UPDATE product SET stock = 10 - 5 WHERE id = 1;  -- stock=5
COMMIT;
-- 正确应该是stock=2，但实际是5，丢失了事务A的更新！
```

### 3.3 隔离级别

```sql
-- MySQL 4种隔离级别

-- 查看当前隔离级别
SELECT @@tx_isolation;
SELECT @@transaction_isolation;

-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;  -- MySQL默认
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

/*
 * 隔离级别对比：
 *
 * ┌────────────────┬────────────┬────────────┬────────────┬────────────┐
 * │   隔离级别      │   脏读     │  不可重复读 │   幻读     │   并发性能  │
 * ├────────────────┼────────────┼────────────┼────────────┼────────────┤
 * │ READ UNCOMMITTED│    能     │     能    │     能    │    最高    │
 * ├────────────────┼────────────┼────────────┼────────────┼────────────┤
 * │ READ COMMITTED │   不能     │     能    │     能    │    中等    │
 * ├────────────────┼────────────┼────────────┼────────────┼────────────┤
 * │ REPEATABLE READ │   不能    │    不能    │   MySQL不能│    中等偏上│
 * ├────────────────┼────────────┼────────────┼────────────┼────────────┤
 * │ SERIALIZABLE   │   不能     │    不能    │    不能    │    最低    │
 * └────────────────┴────────────┴────────────┴────────────┴────────────┘
 *
 * 各种锁：
 *
 * READ UNCOMMITTED：无锁
 * READ COMMITTED：行级锁
 * REPEATABLE READ：间隙锁 + 行级锁（MySQL默认）
 * SERIALIZABLE：表级锁
 */
```

## 四、锁机制

### 4.1 锁的分类

```sql
-- 锁的分类

-- 1. 共享锁 (Shared Lock, S锁)
-- 读取数据时加锁，其他事务可以读但不能写
SELECT * FROM orders WHERE id = 1 LOCK IN SHARE MODE;

-- 2. 排他锁 (Exclusive Lock, X锁)
-- 修改数据时加锁，其他事务不能读也不能写
SELECT * FROM orders WHERE id = 1 FOR UPDATE;

-- 3. 记录锁 (Record Lock)
-- 锁定索引记录，而非数据本身
-- id=1的记录被锁定
SELECT * FROM orders WHERE id = 1 FOR UPDATE;

-- 4. 间隙锁 (Gap Lock)
-- 锁定索引记录之间的间隙
-- 防止插入
SELECT * FROM orders WHERE id BETWEEN 1 AND 10 FOR UPDATE;
-- 锁定 (1, 10) 这个区间，阻止插入id在2-9之间的记录

-- 5. Next-Key Lock
-- 记录锁 + 间隙锁的组合
-- InnoDB在REPEATABLE READ下的默认锁
SELECT * FROM orders WHERE id = 5 FOR UPDATE;
-- 锁定 id=5 这条记录，以及(上一个索引, 5] 和 [5, 下一个索引) 的间隙

-- 6. 意向锁 (Intention Lock)
-- 表级锁，表明事务有意向对某些行加锁
-- 意向共享锁(IS)：事务想要获得某些行的S锁
-- 意向排他锁(IX)：事务想要获得某些行的X锁
LOCK TABLE orders WRITE;  -- 会自动加IX锁
```

### 4.2 锁的兼容矩阵

```sql
/*
 * 锁兼容矩阵：
 *
 * ┌────────┬───────┬───────┐
 * │        │   S   │   X   │
 * ├────────┼───────┼───────┤
 * │   S    │  兼容  │ 不兼容 │
 * ├────────┼───────┼───────┤
 * │   X    │ 不兼容 │ 不兼容 │
 * └────────┴───────┴───────┘
 *
 * S锁和S锁兼容：两个事务可以同时读取同一行
 * X锁和任何锁都不兼容：写操作必须独占
 */

/*
 * 意向锁兼容矩阵：
 *
 * ┌────────┬───────┬───────┬───────┬───────┐
 * │        │   IS  │   IX  │   S   │   X   │
 * ├────────┼───────┼───────┼───────┼───────┤
 * │   IS  │  兼容  │  兼容  │  兼容  │ 不兼容 │
 * ├────────┼───────┼───────┼───────┼───────┤
 * │   IX  │  兼容  │  兼容  │ 不兼容 │ 不兼容 │
 * ├────────┼───────┼───────┼───────┼───────┤
 * │   S   │  兼容  │ 不兼容 │  兼容  │ 不兼容 │
 * ├────────┼───────┼───────┼───────┼───────┤
 * │   X   │ 不兼容 │ 不兼容 │ 不兼容 │ 不兼容 │
 * └────────┴───────┴───────┴───────┴───────┘
 *
 * 意向锁的作用：
 * - 快速判断表是否有行锁
 * - 避免逐行检查锁的冲突
 */
```

### 4.3 死锁及处理

```sql
-- 死锁示例

-- 事务A
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;  -- 锁定id=1
UPDATE account SET balance = balance + 100 WHERE id = 2;  -- 锁定id=2
COMMIT;

-- 事务B
START TRANSACTION;
UPDATE account SET balance = balance - 200 WHERE id = 2;  -- 锁定id=2
UPDATE account SET balance = balance + 200 WHERE id = 1;  -- 等待id=1...
-- 死锁！事务A持有id=1，等待id=2；事务B持有id=2，等待id=1

-- 查看死锁日志
SHOW ENGINE INNODB STATUS;

-- 死锁处理策略：
-- 1. InnoDB自动检测死锁，回滚小事务
-- 2. innodb_lock_wait_timeout 设置超时时间
-- 3. 业务层面合理设计锁顺序

-- 解决死锁的方法
-- 1. 统一锁顺序：始终先锁id=1，再锁id=2
-- 2. 减少锁范围：锁定后尽快释放
-- 3. 降低隔离级别
-- 4. 使用乐观锁代替悲观锁
```

## 五、慢查询优化

### 5.1 EXPLAIN分析

```sql
-- 使用EXPLAIN分析查询

EXPLAIN SELECT * FROM orders
WHERE customer_id = 100
ORDER BY order_date DESC
LIMIT 10;

-- 返回字段解析：
/*
 * id: 查询序号
 * select_type: 查询类型（SIMPLE/PRIMARY/SUBQUERY等）
 * table: 表名
 * type: 连接类型
 * possible_keys: 可用的索引
 * key: 实际使用的索引
 * key_len: 索引长度
 * ref: 与索引比较的列
 * rows: 预计扫描的行数
 * filtered: 过滤后百分比
 * Extra: 额外信息（Using index/Using filesort等）
 */

-- type字段详解：
-- system > const > eq_ref > ref > range > index > ALL
-- 最好到最差

-- const: 主键或唯一索引等值查询
EXPLAIN SELECT * FROM orders WHERE id = 1;

-- eq_ref: 关联查询，使用主键或唯一索引
EXPLAIN SELECT * FROM orders o JOIN order_items i ON o.id = i.order_id;

-- ref: 使用普通索引等值查询
EXPLAIN SELECT * FROM orders WHERE customer_id = 100;

-- range: 索引范围扫描
EXPLAIN SELECT * FROM orders WHERE id BETWEEN 1 AND 100;

-- index: 全索引扫描
EXPLAIN SELECT id FROM orders;

-- ALL: 全表扫描（需要优化！）
EXPLAIN SELECT * FROM orders WHERE customer_name = '张三';

-- Extra字段常见值：
-- Using index: 使用覆盖索引
-- Using where: 使用WHERE条件
-- Using temporary: 使用临时表
-- Using filesort: 文件排序（需要优化！）
-- Using index condition: 使用索引下推
```

### 5.2 优化实战案例

```sql
-- 慢查询优化案例

-- 原始慢查询
SELECT o.*, c.name, c.phone
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'pending'
AND o.created_at >= '2024-01-01'
ORDER BY o.created_at DESC
LIMIT 100;

-- 问题分析：
-- 1. 全表扫描 orders (status没有索引)
-- 2. 文件排序 (ORDER BY created_at)
-- 3. 需要回表查询

-- 优化1：添加合适的索引
CREATE INDEX idx_status_created ON orders(status, created_at);

-- 优化2：使用覆盖索引（避免回表）
CREATE INDEX idx_covering ON orders(status, created_at, customer_id);

-- 优化后查询
SELECT o.id, o.created_at, c.name, c.phone
FROM orders o
JOIN customers c USE INDEX(idx_customer_name) ON o.customer_id = c.id
WHERE o.status = 'pending'
AND o.created_at >= '2024-01-01'
ORDER BY o.created_at DESC
LIMIT 100;

-- 验证优化效果
EXPLAIN SELECT ... -- 检查是否使用索引

-- 分页优化（深度分页问题）
-- 原始：
SELECT * FROM orders ORDER BY id LIMIT 1000000, 10;
-- 问题：MySQL先扫描前1000010条，再返回最后10条

-- 优化1：利用主键索引
SELECT * FROM orders
WHERE id > 1000000
ORDER BY id
LIMIT 10;

-- 优化2：延迟关联
SELECT o.* FROM orders o
INNER JOIN (
    SELECT id FROM orders
    WHERE status = 'pending'
    ORDER BY created_at DESC
    LIMIT 1000000, 10
) AS t ON o.id = t.id;
```

### 6.2 分片策略

```sql
-- 分片键选择原则：
-- 1. 尽可能选择查询最频繁的字段
-- 2. 数据分布均匀
-- 3. 避免跨分片查询

-- 1. Hash分片
-- 根据分片键的Hash值分配到不同的表
-- SHARDING_KEY = hash(分片键) % 分片数

-- 示例：按user_id hash分片
CREATE TABLE orders (
    id BIGINT,
    user_id BIGINT,
    ...
) ENGINE=InnoDB
PARTITION BY HASH(user_id)
PARTITIONS 4;

-- 优点：数据分布均匀
-- 缺点：跨分片查询困难

-- 2. Range分片
-- 按分片键的范围分配
CREATE TABLE orders (
    id BIGINT,
    created_at DATE,
    ...
) ENGINE=InnoDB
PARTITION BY RANGE(YEAR(created_at)) (
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN MAXVALUE
);

-- 优点：范围查询快
-- 缺点：可能导致热点分片

-- 3. List分片
-- 按枚举值分配
CREATE TABLE orders (
    id BIGINT,
    region VARCHAR(20),
    ...
) ENGINE=InnoDB
PARTITION BY LIST_COLUMNS(region) (
    PARTITION p-east VALUES IN ('北京', '天津', '河北'),
    PARTITION p-west VALUES IN ('四川', '重庆', '云南')
);

-- 4. 复合分片
-- 使用多个分片键
-- SHARD_KEY = hash(field1 + field2) % 分片数
```

```
│  │                                                              │ │
│  │  全同步复制：                                                 │ │
│  │  - 主库等待所有从库确认收到日志                               │ │
│  │  - 数据最安全，但性能影响大                                   │ │
│  │                                                              │ │
│  └──────────────────────────────────────────────────────────────┘ │
│                                                                      │
└────────────────────────────────────────────────────────────────────┘
```

### 7.2 读写分离配置

```sql
-- 主库配置
[mysqld]
server-id = 1
log-bin = mysql-bin
binlog_format = ROW
sync-binlog = 1

-- 从库配置
[mysqld]
server-id = 2
relay-log = relay-bin
read_only = 1
super_read_only = 1

-- 从库设置只读
SET GLOBAL read_only = 1;
SET GLOBAL super_read_only = 1;

-- 主从复制命令
SHOW MASTER STATUS;  -- 查看主库状态（Binlog位置）
SHOW SLAVE STATUS\G;  -- 查看从库状态

-- 跳过复制错误
SET GLOBAL sql_slave_skip_counter = 1;
START SLAVE;

-- 配置读写分离（Java应用层面）
-- 使用ShardingSphere、MyCat等中间件
-- 或使用Spring的AbstractRoutingDataSource
```

## 八、实战：数据库优化检查清单

```sql
-- MySQL数据库优化检查清单

-- 1. 索引检查
-- 查看表的索引
SHOW INDEX FROM orders;

-- 查找未使用的索引
SELECT * FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE object_schema = 'your_database'
AND index_name IS NOT NULL
AND count_star = 0;

-- 2. 慢查询日志
-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';

-- 查看慢查询
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 3. 连接数检查
SHOW STATUS LIKE 'Threads_connected';
SHOW VARIABLES LIKE 'max_connections';

-- 调整最大连接数
SET GLOBAL max_connections = 1000;

-- 4. 缓冲池大小检查
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';
-- 推荐：设置为服务器内存的70-80%

-- 5. Key Buffer大小（MyISAM）
SHOW VARIABLES LIKE 'key_buffer_size';

-- 6. 表统计信息
-- 更新表统计信息
ANALYZE TABLE orders;

-- 查看表大小
SELECT
    table_schema,
    table_name,
    ROUND(data_length / 1024 / 1024, 2) AS '数据大小(MB)',
    ROUND(index_length / 1024 / 1024, 2) AS '索引大小(MB)'
FROM information_schema.tables
WHERE table_schema = 'your_database'
ORDER BY data_length DESC;

-- 7. 锁等待检查
SHOW ENGINE INNODB STATUS;
-- 查看死锁信息
SHOW VARIABLES LIKE 'innodb_print_all_deadlocks';

-- 8. 配置文件优化示例
[mysqld]
# 连接优化
max_connections = 1000
wait_timeout = 600
interactive_timeout = 600

# 缓冲池优化
innodb_buffer_pool_size = 4G
innodb_buffer_pool_instances = 4

# 日志优化
innodb_log_file_size = 1G
innodb_log_buffer_size = 64M
innodb_flush_log_at_trx_commit = 1

# 线程优化
innodb_thread_concurrency = 16
innodb_read_io_threads = 8
innodb_write_io_threads = 8

# 查询优化
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 1
```
