## 一、JVM原理深度剖析

### 1.1 JVM是什么？

想象一下，**JVM就像一家大型餐厅的厨房**，而Java程序就是一道道菜品。

- **厨师（CPU）** 只认识一种语言：执行指令
- **菜品原料（.java源码）** 经过 **洗切加工（编译器）** 变成 **半成品（字节码 .class）**
- **厨房（JVM）** 负责把半成品烹饪成美味佳肴

```
┌─────────────────────────────────────────────────────────────┐
│                      Java程序运行流程                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   源代码(.java)  ──编译──>  字节码(.class)  ──加载──>  JVM   │
│        │                              │                     │
│        ▼                              ▼                     │
│   程序员写的代码                   虚拟机执行                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**为什么Java能够"一次编写，到处运行"？**

因为不同的操作系统（Windows、Linux、Mac）都有各自的厨房（JVM实现），只要半成品（字节码）是统一的，就能在任何厨房里烹饪出同样的菜品。

### 1.3 各区域详解

#### 1.3.1 程序计数器 - "当前菜品的食谱页码"

程序计数器就像烹饪时记下的**当前步骤页码**。

- 每个线程独立拥有程序计数器
- 如果执行的是Java方法，记录**当前执行的字节码指令地址**
- 如果执行的是Native方法，计数器值为**空（Undefined）**
- 这是唯一一个不会发生OutOfMemoryError的区域

```java
// 程序计数器示例：记录执行到哪一行
public class PCRegisterDemo {
    public static void main(String[] args) {
        int a = 1;        // 字节码指令位置：0
        int b = 2;        // 字节码指令位置：1
        int c = a + b;    // 字节码指令位置：2
        System.out.println(c); // 字节码指令位置：3
    }
}

/*
 * 编译后的字节码：
 * 0: iconst_1      // 把常量1放入操作数栈
 * 1: istore_1     // 从栈顶弹出1，存入局部变量表位置1（变量a）
 * 2: iconst_2     // 把常量2放入操作数栈
 * 3: istore_2     // 从栈顶弹出2，存入局部变量表位置2（变量b）
 * 4: iload_1      // 从局部变量表位置1加载a到栈顶
 * 5: iload_2      // 从局部变量表位置2加载b到栈顶
 * 6: iadd         // 弹出两个int，相加，结果入栈
 * 7: istore_3     // 结果存入位置3（变量c）
 * 8: getstatic    // 获取System.out
 * 11: iload_3     // 加载c的值
 * 12: invokevirtual// 调用println方法
 *
 * 程序计数器会记录下一条要执行的指令位置
 */
```

#### 1.3.2 虚拟机栈 - "每个厨师的独立工作台"

想象每个厨师都有一个独立的工作台，**栈帧就是工作台上的一个工具箱**。

```java
public class StackDemo {
    /**
     * 递归调用会导致栈帧不断入栈
     * 如果递归没有终止条件，就会栈溢出（StackOverflowError）
     */
    public static int factorial(int n) {
        if (n <= 1) {
            return 1;
        }
        return n * factorial(n - 1); // 每次递归都会创建一个新的栈帧
    }

    public static void main(String[] args) {
        int result = factorial(5); // 5! = 120
        System.out.println("结果：" + result);
    }
}

/*
 * factorial(5) 的栈帧变化：
 *
 * 主线程栈内存：
 * ┌─────────────────────────┐
 * │  main() 栈帧            │  ← 最先入栈，最后出栈
 * │  -局部变量: args         │
 * │  -返回值地址             │
 * ├─────────────────────────┤
 * │  factorial(5) 栈帧     │
 * │  -局部变量: n=5         │
 * │  -返回地址              │
 * ├─────────────────────────┤
 * │  factorial(4) 栈帧     │
 * │  -局部变量: n=4         │
 * │  -返回地址              │
 * ├─────────────────────────┤
 * │  factorial(3) 栈帧     │
 * │  -局部变量: n=3         │
 * │  -返回地址              │
 * ├─────────────────────────┤
 * │  factorial(2) 栈帧     │
 * │  -局部变量: n=2         │
 * │  -返回地址              │
 * ├─────────────────────────┤
 * │  factorial(1) 栈帧     │  ← 最后入栈，最先出栈
 * │  -局部变量: n=1         │
 * │  -返回地址              │
 * └─────────────────────────┘
 */
```

#### 1.3.3 本地方法栈 - "外籍厨师的专业工作台"

本地方法栈就像为**不会说中文的外籍厨师**准备的工作台，他们有自己独特的工作方式（Native Method）。

```java
/**
 * 本地方法示例：Object类的几个本地方法
 */
public class NativeMethodDemo {
    /**
     * 返回对象哈希码，本地实现
     * 底层通常是C/C++实现，直接返回内存地址或经过计算的哈希值
     */
    public native int hashCode();

    /**
     * 创建对象副本，本地实现
     * 底层直接复制内存数据，比Java实现更高效
     */
    protected native Object clone();

    /**
     * 线程唤醒，本地实现
     * 直接操作底层操作系统线程调度
     */
    public final native void notify();

    /**
     * 线程等待，本地实现
     * 直接挂起线程，不消耗CPU资源
     */
    public final native void wait(long timeout);
}
```

#### 1.3.4 堆区 - "最大的食材仓库"

堆是JVM中**最大的一块内存区域**，所有对象都在这里"存放"，就像餐厅的**中央大仓库**。

```java
public class HeapDemo {
    public static void main(String[] args) {
        // 每次new都会在堆中分配内存
        String ref1 = new String("食材1"); // 堆中分配
        String ref2 = new String("食材2"); // 堆中分配
        String ref3 = ref1;                // 引用赋值，不创建新对象

        System.out.println(ref1 == ref2); // false：引用不同
        System.out.println(ref1 == ref3); // true：同一个对象

        // 数组也是对象，在堆中分配
        int[] array = new int[1000]; // 堆中分配数组对象
        System.out.println("数组长度：" + array.length);
    }
}

/*
 * 堆内存布局：
 *
 * ┌──────────────────────────────────────────────────────────────┐
 * │                        堆 Heap                                │
 * ├──────────────────────────────────────────────────────────────┤
 * │                                                              │
 * │   ┌──────────────────┐  ┌──────────────────┐                 │
 * │   │  String "食材1"  │  │  String "食材2"  │                 │
 * │   │  (对象1)         │  │  (对象2)         │                 │
 * │   └────────┬─────────┘  └────────┬─────────┘                 │
 * │            │                     │                            │
```

## 二、垃圾回收（GC）原理

### 2.1 为什么要垃圾回收？

**程序运行过程中会产生垃圾**，就像餐厅做饭会产生厨余垃圾一样。如果不清理：
- 食材仓库会爆满（新对象没地方放）
- 厨房环境会恶化（内存泄漏导致系统变慢）

**哪些是"垃圾"？**

```java
public class GarbageDemo {
    public static void main(String[] args) {
        // 局部变量，在方法结束时就会被回收
        int localVar = 1;

        // 对象创建在堆中
        GarbageDemo obj = new GarbageDemo();
        obj = null; // 现在obj引用不再指向这个对象，它就变成了"垃圾"

        // 另一个引用也指向这个对象
        GarbageDemo anotherRef = obj; // 这里的obj已经是null，所以anotherRef也是null
        // 此时，没有任何引用指向最早创建的那个GarbageDemo对象
        // 它就是垃圾了！

        // 方法结束，局部变量全部出栈，localVar的内存被回收
    }
}

/*
 * 垃圾判定示意图：
 *
 *                    堆内存
 *     ┌─────────────────────────────────┐
 *     │                                 │
 *     │    ┌─────────────────────┐     │
 *     │    │  GarbageDemo 对象    │     │
 *     │    │  (已经没有任何引用指向) │ ← 垃圾！
 *     │    └─────────────────────┘     │
 *     │                                 │
 *     └─────────────────────────────────┘
 *
 *     ┌─────────────────────────────────┐
 *     │  虚拟机栈                        │
 *     │  ┌─────────────────────────┐    │
 *     │  │ main方法的栈帧          │    │
 *     │  │ 局部变量表:              │    │
 *     │  │   - args: [...]        │    │
 *     │  │   - obj: null          │    │ ← 引用指向null
 *     │  │   - anotherRef: null   │    │
 *     │  └─────────────────────────┘    │
 *     └─────────────────────────────────┘
 */
```

### 2.2 垃圾判定算法

#### 2.2.1 引用计数法 - "数一数有几个人在用它"

引用计数法就像给每个食材贴上**使用计数标签**。

```java
/**
 * 引用计数示意
 */
public class ReferenceCountingDemo {
    public static void main(String[] args) {
        // 创建对象A，引用计数 = 1
        CounterObj A = new CounterObj("A");

        // 创建对象B
        CounterObj B = new CounterObj("B");

        // A被B引用，计数 = 2
        A.instance = B;

        // B被A引用，计数 = 2
        B.instance = A;

        // 现在让它们互相不引用
        A.instance = null;
        B.instance = null;

        // 此时A和B的引用计数都是1（只有main方法的局部变量引用）
        // 但如果还有其他地方引用...

        // 让局部变量也不引用
        A = null;
        B = null;

        // 现在A和B的引用计数都是0
        // 它们都是垃圾了！

        // 触发垃圾回收
        System.gc();
    }
}

/*
 * 引用计数法的缺陷：
 *
 *  ┌─────────────────────────────────────────────────────────┐
 *  │              循环引用问题                                │
 *  │                                                         │
 *  │      A.instance ─────────→ B                           │
 │  │          ↑                    │                        │
 │  │          │                    │                        │
 │  │          └────────────────────┘                        │
 │  │              B.instance                                │
 │  │                                                         │
 │  │  虽然A和B互相引用，但它们已经不可达了（从GC Roots出发    │
 │  │  找不到它们），实际上是垃圾。但引用计数都是1，无法识别。 │
 │  └─────────────────────────────────────────────────────────┘
 *
 *  优点：简单，判定效率高
 *  缺点：无法处理循环引用
 *
 *  Java没有使用引用计数法，而是使用了可达性分析算法
 */
```

#### 2.2.2 可达性分析算法 - "从根开始顺藤摸瓜"

**GC Roots** 就像餐厅的**总进货渠道**，从它开始能找到的所有对象就是"活着的"，找不到的就是垃圾。

```java
public class GCRootsDemo {
    // 静态变量，持有引用，是GC Roots之一
    public static GCNode staticNode = new GCNode("静态变量节点");

    // 静态变量
    public static GCNode anotherStatic = new GCNode("另一个静态节点");

    // 实例方法
    public void method() {
        // 方法中的局部变量，是GC Roots之一
        GCNode localNode = new GCNode("局部变量节点");

        System.out.println(localNode);
    }

    public static void main(String[] args) {
        // 创建对象
        GCNode obj1 = new GCNode("对象1");
        GCNode obj2 = new GCNode("对象2");
        GCNode obj3 = new GCNode("对象3");

        // 建立引用关系（形成引用链）
        obj1.next = obj2;
        obj2.next = obj3;

        // obj3没有后续引用
        obj3.last = obj1;

        // 调用方法
        GCRootsDemo demo = new GCRootsDemo();
        demo.method();

        // 切断引用链
        obj1 = null;
        obj2 = null;

        System.gc();
    }
}
```

#### 2.3.2 复制算法 - "一半一半，复制存活对象"

复制算法就像**把仓库分成两半，先用一半，用完了就把存活的物品复制到另一半**。

```java
public class CopyingDemo {
    /**
     * 模拟复制算法
     */
    public static void main(String[] args) {
        // 假设堆内存分为两块：From区和To区
        Object[] fromSpace = new Object[100];
        Object[] toSpace = new Object[100]; // 空闲区

        // 初始状态
        System.out.println("初始状态：From区使用中，To区空闲");

        // 假设From区满了，触发GC
        // GC会把存活的对象复制到To区

        // 模拟存活对象
        Object存活1 = new Object();
        Object存活2 = new Object();
        Object存活3 = new Object();

        // 复制到To区
        Object[] toSpaceUsed = new Object[3];
        toSpaceUsed[0] = 存活1;
        toSpaceUsed[1] = 存活2;
        toSpaceUsed[2] = 存活3;

        // From区和To区交换
        Object[] temp = fromSpace;
        fromSpace = toSpace;
        toSpace = temp;

        System.out.println("GC后：From区和To区交换角色");
    }
}
```

#### 2.3.3 标记-整理算法 - "标记后整理，紧凑排列"

标记-整理算法就像**整理仓库：先标记要保留的物品，然后把剩下的空位都填上**。

```java
public class MarkCompactDemo {
    public static void main(String[] args) {
        // 模拟标记-整理过程
        Object[] heap = new Object[10];

        // 初始分配
        heap[0] = new Object(); // 存活
        heap[1] = null; // 垃圾
        heap[2] = new Object(); // 存活
        heap[3] = null; // 垃圾
        heap[4] = new Object(); // 存活
        heap[5] = null; // 垃圾
        heap[6] = new Object(); // 存活
        heap[7] = null; // 垃圾
        heap[8] = new Object(); // 存活
        heap[9] = null; // 垃圾

        System.out.println("整理前：存活对象分散在堆中");
        for (int i = 0; i < heap.length; i++) {
            System.out.print(heap[i] == null ? "[空]" : "[存]");
        }
        System.out.println();

        // 模拟整理：把所有存活对象向一端移动
        int writeIndex = 0;
        for (int i = 0; i < heap.length; i++) {
            if (heap[i] != null) {
                heap[writeIndex] = heap[i];
                if (writeIndex != i) {
                    heap[i] = null; // 原位置清空
                }
                writeIndex++;
            }
        }

        System.out.println("整理后：存活对象紧凑排列");
        for (int i = 0; i < heap.length; i++) {
            System.out.print(heap[i] == null ? "[空]" : "[存]");
        }
    }
}
```

### 2.4 分代收集理论

#### 2.4.1 为什么需要分代？

**不同对象的"寿命"不同**，就像餐厅里：
- 有些食材（蔬菜）很快就用掉了（新对象）
- 有些调料可以用很久（长期存活的对象）

```java
public class GenerationalDemo {
    public static void main(String[] args) {
        // 大量短命对象的例子：循环中创建的对象
        for (int i = 0; i < 100000; i++) {
            // 这些对象大多是短命的
            String temp = "临时对象" + i;
            // 处理...
        }
        // 循环结束后，这些临时对象大部分都变成垃圾了

        // 长寿对象的例子：缓存、单例
        Map<String, Object> cache = new HashMap<>();
        for (int i = 0; i < 100; i++) {
            cache.put("key" + i, new byte[1024]); // 长期存活
        }
        // cache中的对象会存活很长时间
    }
}
```

#### 2.4.3 GC分类详解

```java
public class GCTypesDemo {
    /**
     * Minor GC (年轻代GC)
     * - 发生在新生代的Eden区
     * - Eden区满时触发
     * - 频率高，停顿时间短
     * - 采用复制算法
     */
    public void minorGC() {
        // 模拟Minor GC场景
        for (int i = 0; i < 1000; i++) {
            // 每个新对象都在Eden区分配
            byte[] data = new byte[1024]; // 1KB
        }
        // Eden区满时触发Minor GC
    }

    /**
     * Full GC (老年代GC / Major GC)
     * - 发生在老年代
     * - 老年代空间不足时触发
     * - 频率低，但停顿时间长
     * - 采用标记-整理或标记-清除算法
     */
    public void fullGC() {
        // 模拟Full GC场景
        // 大量长生命周期对象占据老年代
        List<byte[]> largeData = new ArrayList<>();
        for (int i = 0; i < 1000; i++) {
            largeData.add(new byte[1024 * 1024]); // 1MB对象
            // 老年代空间不足时触发Full GC
        }
    }

    public static void main(String[] args) {
        GCTypesDemo demo = new GCTypesDemo();
        demo.minorGC();
        demo.fullGC();
    }
}
```

### 2.5 经典GC算法

#### 2.5.1 Serial GC - "单线程清洁工"

Serial GC是最基础的GC，就像**只有一个清洁工的餐厅**。

```java
public class SerialGCDemo {
    /**
     * Serial GC特点：
     * - 单线程执行GC
     * - GC时需要Stop-The-World（STW），停止所有应用线程
     * - 简单高效，适合客户端模式
     */
    public static void main(String[] args) {
        // 启用Serial GC的配置：
        // -XX:+UseSerialGC

        // 模拟对象分配
        List<Object> holdObjects = new ArrayList<>();
        for (int i = 0; i < 100; i++) {
            holdObjects.add(new byte[1024]); // 保持对象不被回收
        }

        System.out.println("Serial GC示例运行完成");
    }
}

/*
 * Serial GC工作示意：
 *
 *  用户线程： ████████████████████████████  →  停止  →  ████████████████
 *             (正常执行)                        (STW)        (继续执行)
 *                                                       ↑
 *  GC线程：    [空闲]  →  [开始GC]  →  [GC中...]  →  [完成]
 *                                                       ↑
 *            ┌───────────────────────────────────────────┐
 *            │  STW期间，所有用户线程必须等待GC完成        │
 *            │  这就是为什么GC会造成系统"卡顿"            │
 *            └───────────────────────────────────────────┘
 */
```

#### 2.5.3 CMS GC - "边做饭边打扫"

CMS (Concurrent Mark Sweep) GC就像**勤快的清洁工，在餐厅营业时也能慢慢打扫**。

```java
public class CMSGCDemo {
    /**
     * CMS GC目标：
     * - 减少GC停顿时间
     * - GC线程与应用线程并发执行
     *
     * CMS过程：
     * 1. 初始标记（STW）：标记GC Roots直接引用的对象
     * 2. 并发标记：沿着引用链标记存活对象（与应用并发）
     * 3. 重新标记（STW）：修正并发标记期间产生的变化
     * 4. 并发清除：清除垃圾对象（与应用并发）
     */
    public static void main(String[] args) {
        // 启用CMS GC：
        // -XX:+UseConcMarkSweepGC

        // 模拟Web应用
        WebApplication app = new WebApplication();
        for (int i = 0; i < 10000; i++) {
            app.handleRequest(new Request(i));
        }
    }
}

class WebApplication {
    private List<Object> sessionCache = new ArrayList<>();

    public void handleRequest(Request req) {
        // 模拟处理请求
        sessionCache.add(new byte[1024]);

        // 模拟处理时间
        try {
            Thread.sleep(1);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }

        // 定期清理
        if (sessionCache.size() > 1000) {
            sessionCache.clear();
        }
    }
}

/*
 * CMS GC各阶段示意：
 *
 *  时间轴：
 *  ───────────────────────────────────────────────────────────────────→
 *
 *  阶段：  [初始标记]    [并发标记]       [重新标记]    [并发清除]
 *          ────        ─────────────        ────       ─────────────
 *  STW:     ████                      ████
 *
 *  用户线程： ████████████████████████████   （大部分时间并发执行）
 *
 *
 *  详细过程：
 *  1. 初始标记（Stop-The-World）
 *     - 暂停所有应用线程
 *     - 标记GC Roots直接引用的对象（如：静态变量、本地变量表中的对象引用）
 *     - 停顿时间很短（通常<100ms）
 *
 *  2. 并发标记（Concurrent）
 *     - GC线程和应用线程并发执行
 *     - 从初始标记的对象出发，遍历引用链，标记所有存活对象
 *     - 应用线程不受影响，但GC线程会占用一部分CPU
 *
 *  3. 重新标记（Stop-The-World）
 *     - 暂停所有应用线程
 *     - 修正并发标记期间因应用线程运行而产生的变化
 *     - 停顿时间比初始标记长，但比Full GC短
 *
 *  4. 并发清除（Concurrent）
 *     - GC线程和应用线程并发执行
 *     - 清除未被标记的垃圾对象
 *     - 不需要STW
 *
 *  CMS的缺点：
 *  - CPU敏感：并发阶段会占用CPU资源
 *  - 内存碎片：采用标记-清除，会产生碎片
 *  - 并发失败：如果并发阶段老年代内存不足，会退化为Serial GC
 */
```

#### 2.5.4 G1 GC - "智能分区清洁工"

G1 (Garbage First) GC是现代JVM默认的垃圾收集器，它把堆分成**多个小区块（Region），优先回收垃圾最多的区域**。

```java

     * - 兼具年轻代和老年代的特点（不再严格区分）
     *
     * 常用参数：
     * -XX:+UseG1GC                启用G1
     * -XX:MaxGCPauseMillis=200   最大停顿时间目标
     * -XX:G1HeapRegionSize=       Region大小（1MB-32MB）
     */
    public static void main(String[] args) {
        // G1将堆划分为多个Region
        simulateG1Regions();
    }

    private static void simulateG1Regions() {
        // G1将堆最多分成约2048个Region
        // 每个Region大小 = 堆大小 / 2048（取决于堆大小）

        // Region状态可以是：
        // - Eden（新生代Eden区）
        // - Survivor（幸存区）
        // - Old（老年代）
        // - Humongous（大对象区，超过Region 50%的对象）

        System.out.println("G1 GC Region模拟");
    }
}

/*
 * G1 GC堆结构：
 *
 *  传统分代：                        G1分代：
 *  ┌─────────────────────┐         ┌─────────────────────┐
 *  │      新生代           │         │  Region 1  [Eden]   │
 *  │  ┌───┬───┬───┬───┐   │         ├─────────────────────┤
 *  │  │Ede│S0 │S1 │   │   │         │  Region 2  [Eden]   │
 *  │  │ n │   │   │   │   │         ├─────────────────────┤
 *  │  └───┴───┴───┴───┘   │         │  Region 3  [Surv]   │
 *  ├─────────────────────┤         ├─────────────────────┤
 *  │                     │         │  Region 4  [Old]    │
 *  │       老年代         │         ├─────────────────────┤
 *  │                     │         │  Region 5  [Surv]   │
 *  │                     │         ├─────────────────────┤
 *  │                     │         │  Region 6  [Old]    │
 *  └─────────────────────┘         └─────────────────────┘
 *                                       ├─────────────────────┤
 *                                       │  Region N  [Humongous]│
 *                                       └─────────────────────┘
 *
 *  G1的Young GC：
 *  - 收集所有年轻代的Eden和Survivor区
 *  - 使用复制算法，将存活对象复制到Survivor区或老年代
 *  - 停顿时间可控
 *
 *  G1的Mixed GC：
 *  - 收集所有年轻代 + 部分老年代
 *  - 优先选择垃圾最多的老年代Region
 *  - 可以设置回收老年代的比例
 *
 *  G1的工作过程：
 *  1. Young GC：收集年轻代Region
 *  2. 并发标记：在应用运行时标记存活对象
 *  3. Mixed GC：收集年轻代 + 部分老年代Region
 *
 *  为什么G1能做到停顿时间可控？
 *  因为G1可以指定目标停顿时间，然后选择垃圾最多的Region进行回收，
 *  牺牲一些吞吐量来换取更短的停顿时间。
 */
```

### 3.3 双亲委派模型

**双亲委派模型**就像**餐厅的采购流程**：现场采购员不会自己采购，而是先问主管，主管再问经理，层层向上请示。

```java
/**
 * 双亲委派模型示意
 *
 * "双亲"指的是父加载器，但这个翻译容易误解

public class ParentDelegateDemo {

     * 1. 检查类是否已加载

     * 4. 如果父加载器加载失败，自己尝试加载

    public static void main(String[] args) {
        // 尝试加载一个类
        try {

 *  │  - 找不到                                           │
 *  │  - 返回null                                        │

 *  1. 安全性：防止核心类被篡改

 *     由于双亲委派，核心类库的String会优先被Bootstrap加载
 *     用户的String永远不会被加载
 *
 *  2. 类的唯一性：保证同一个类不会被重复加载
 *     父加载器加载的类与子加载器加载的类是不同的
 *     有了双亲委派，全局只有一份类
 *
 *  3. 类的层次关系：符合类的依赖关系
 *     java.lang.Object 一定由Bootstrap加载
 *     其他类都继承Object，所以也一定是可见的
 */
```

### 3.4 自定义类加载器

```java
/**
 * 自定义类加载器示例：从指定目录加载类
 */
public class CustomClassLoader extends ClassLoader {

    // 要加载的目录
    private String classPath;

    public CustomClassLoader(String classPath) {
        this.classPath = classPath;
    }

    /**
     * 重写findClass方法，实现自定义加载逻辑
     * @param name 类的全限定名
     * @return Class对象
     * @throws ClassNotFoundException
     */
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        // 1. 获取类的字节数组
        byte[] classData = loadClassData(name);
        if (classData == null) {
            throw new ClassNotFoundException("找不到类：" + name);
        }

        // 2. 调用父类的defineClass方法定义类
        // defineClass会将字节数组转换为Class对象
        return defineClass(name, classData, 0, classData.length);
    }

    /**
     * 从文件系统中读取类的字节码
     * @param className 类名
     * @return 字节数组
     */
    private byte[] loadClassData(String className) {
        // 将包名转换为文件路径
        // com.example.Test → com/example/Test.class
        String fileName = classPath + File.separatorChar
            + className.replace('.', File.separatorChar) + ".class";

        try (FileInputStream fis = new FileInputStream(fileName);
             ByteArrayOutputStream bos = new ByteArrayOutputStream()) {

            // 读取文件内容
            byte[] buffer = new byte[1024];
            int bytesRead;
            while ((bytesRead = fis.read(buffer)) != -1) {
                bos.write(buffer, 0, bytesRead);
            }

            return bos.toByteArray();
        } catch (IOException e) {
            return null; // 文件不存在或读取失败
        }
    }

    /**
     * 使用自定义类加载器加载类
     */
    public static void main(String[] args) throws Exception {
        // 创建自定义类加载器，指定加载目录
        CustomClassLoader loader = new CustomClassLoader("D:/myclasses");

        // 加载类
        Class<?> clazz = loader.loadClass("com.example.Test");

        // 创建实例
        Object instance = clazz.getDeclaredConstructor().newInstance();

        System.out.println("类加载器：" + clazz.getClassLoader());
        System.out.println("类名：" + clazz.getName());
    }
}

/*
 * 自定义类加载器的使用场景：
 *
 * 1. 热部署（Hot Deployment）
 *    - 应用不重启的情况下加载新版本的类
 *    - Web容器、插件系统常用
 *
 * 2. 模块化系统
 *    - 不同模块使用不同的类加载器
 *    - 实现模块隔离
 *
 * 3. 加密保护
 *    - 类文件加密后，自定义加载器解密
 *    - 防止反编译
 *
 * 4. 从非标准来源加载
 *    - 网络、数据库、压缩包等
 *    - 实现类的动态加载
 *
 * 5. 类加载器隔离
 *    - 同一个类名可以存在多个版本
 *    - 不同类加载器加载的类不可见
 *
 * Tomcat的类加载器结构：
 * ┌─────────────────────────────────────────────────────┐
 * │          WebappClassLoader (加载WEB-INF/lib, classes)│
 * │  - 每个Web应用有自己的类加载器                        │
 * │  - 优先加载应用自己的类                               │
 * │  - 支持热部署                                        │
 * ├─────────────────────────────────────────────────────┤
 * │          StandardLoader (加载Tomcat核心类)            │
 * ├─────────────────────────────────────────────────────┤
```

### 4.1 线程基础

#### 4.1.1 线程与进程的区别

**进程就像餐厅，线程就像餐厅里的厨师**。

- 一个餐厅（进程）可以雇用多个厨师（线程）
- 餐厅拥有独立的资源（厨房、设备）

- 每个厨师可以独立执行任务
- 一个厨师卡住了不会影响其他厨师

```java
/**
 * 线程创建的三种方式
 */
public class ThreadCreationDemo {

    // 方式一：继承Thread类
    static class MyThread extends Thread {
        @Override
        public void run() {
            // 线程执行的任务
            System.out.println("方式一：MyThread线程执行中，线程名：" + getName());
        }
    }

    // 方式二：实现Runnable接口
    static class MyRunnable implements Runnable {
        @Override
        public void run() {
            System.out.println("方式二：MyRunnable线程执行中，线程名：" +
                Thread.currentThread().getName());
        }
    }

    // 方式三：实现Callable接口（可以返回结果、抛出异常）
    static class MyCallable implements Callable<Integer> {
        @Override
        public Integer call() throws Exception {
            System.out.println("方式三：MyCallable线程执行中，线程名：" +
                Thread.currentThread().getName());
            // 可以有返回值
            return 42;
        }
    }

    public static void main(String[] args) throws ExecutionException, InterruptedException {
        // 方式一：继承Thread
        MyThread thread1 = new MyThread();
        thread1.start(); // 启动线程，不是run()！

        // 方式二：实现Runnable
        Thread thread2 = new Thread(new MyRunnable());
        thread2.start();

        // 方式三：Lambda简化（Runnable的现代写法）
        Thread thread3 = new Thread(() -> {
            System.out.println("方式二（Lambda）：线程执行中，线程名：" +
                Thread.currentThread().getName());
        });
        thread3.start();

        // 方式三：实现Callable，使用FutureTask
        FutureTask<Integer> futureTask = new FutureTask<>(new MyCallable());
        Thread thread4 = new Thread(futureTask);
        thread4.start();

        // 获取Callable的返回值
        Integer result = futureTask.get(); // 阻塞等待结果
        System.out.println("Callable返回结果：" + result);

        // 主线程继续执行
        System.out.println("主线程继续执行：" + Thread.currentThread().getName());

        // 等待所有子线程结束
        thread1.join();
        thread2.join();
        thread3.join();
        thread4.join();

        System.out.println("所有线程执行完毕！");
    }
}
```

### 4.2 线程状态与生命周期

```java
/**
 * 线程状态演示
 */
public class ThreadStateDemo {

    public static void main(String[] args) throws InterruptedException {
        // 创建一个线程
        Thread thread = new Thread(() -> {
            System.out.println("子线程状态：" + Thread.currentThread().getState());
            try {
                // 在TIMED_WAITING状态
                Thread.sleep(1000);
                System.out.println("子线程睡醒了");
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            System.out.println("子线程任务完成");
        });

        System.out.println("线程创建后：" + thread.getState()); // NEW

        // 启动线程
        thread.start();
        Thread.sleep(100); // 等待一小段时间
        System.out.println("线程启动后：" + thread.getState()); // RUNNABLE

        // 等待线程结束
        thread.join();
        System.out.println("线程结束后：" + thread.getState()); // TERMINATED
    }
}

/*
 * 线程状态转换图（Java Thread.State）：
 *
 *                    ┌─────────────────┐
 *                    │     NEW         │  创建了线程对象
 *                    │   (新建状态)     │  但还没start()
 *                    └────────┬────────┘
 *                             │ start()
 *                             ▼
 *                    ┌─────────────────┐
 *                    │    RUNNABLE     │ 可运行状态
 *                    │   (就绪/运行中) │ 可能在运行
 *                    └────────┬────────┘  可能正在等待CPU
 *                             │
 *              ┌──────────────┼──────────────┐
 *              │              │              │
 *              ▼              ▼              ▼
 *     ┌────────────┐  ┌────────────┐  ┌────────────┐
 *     │  BLOCKED   │  │  WAITING   │  │TIMED_WAITING│
 *     │  (阻塞状态) │  │  (等待状态) │  │ (计时等待)  │
 *     └────────────┘  └────────────┘  └────────────┘
 *          ↑              ↑              ↑
 *      synchronized    wait()        sleep()
 *       锁未获取        join()        wait(timeout)
 *                       notify()     sleep(timeout)
 *                       notifyAll()   join(timeout)
 *                       |              |
 *                       └──────┬───────┘
 *                              ▼
 *                    ┌─────────────────┐
 *                    │   TERMINATED   │
 *                    │   (终止状态)    │
 *                    │   线程执行完毕   │
 *                    └─────────────────┘
 *
 *
 * 各状态详解：
 *
 * NEW（新建）
 * - Thread对象创建，但还没调用start()
 * - 线程还没有开始运行
 *
 * RUNNABLE（可运行）
 * - 调用了start()，线程准备运行
 * - JVM调度器可能在运行它，也可能还在就绪队列等待CPU
 * - 包含OS层面的"Running"和"Ready"两个子状态
 *
 * BLOCKED（阻塞）
 * - 等待获取锁（synchronized）
 * - 进入synchronized块/方法，但锁被其他线程持有
 *
 * WAITING（无限等待）
 * - 调用Object.wait()、Thread.join()、LockSupport.park()
 * - 等待其他线程显式唤醒
 *
 * TIMED_WAITING（计时等待）
 * - 调用sleep()、wait(timeout)、join(timeout)
 * - 指定等待时间，时间到了自动唤醒
 *
 * TERMINATED（终止）
 * - run()方法执行完毕
 * - 线程已经完成执行
 */

/**
 * 线程阻塞与等待的区别
 */
class ThreadBlockVsWait {
    public void blockedExample() {
        // synchronized获取锁失败的场景
        synchronized (this) {
            // 持有锁的逻辑
        }
        // 其他线程想进入synchronized块，会进入BLOCKED状态
    }

    public void waitingExample() throws InterruptedException {
        // wait()方法的场景
        synchronized (this) {
            while (someCondition) {
                this.wait(); // 进入WAITING状态，释放锁
            }
        }
        // 其他线程调用notify()唤醒
    }
}
```

### 4.3 线程同步机制

#### 4.3.1 synchronized关键字

**synchronized就像餐厅的"单人间"**：一次只能一个人使用。

```java
/**
 * synchronized同步示例
 */
public class SynchronizedDemo {

    // 方式一：同步方法（锁定this对象）
    public synchronized void syncMethod() {
        System.out.println("同步方法开始，线程：" + Thread.currentThread().getName());
        try {
            Thread.sleep(1000);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        System.out.println("同步方法结束，线程：" + Thread.currentThread().getName());
    }

    // 方式二：同步代码块（指定锁对象）
    public void syncBlock() {
        System.out.println("同步代码块外面，线程：" + Thread.currentThread().getName());
        synchronized (this) {
            System.out.println("同步代码块开始，线程：" + Thread.currentThread().getName());
            try {
                Thread.sleep(1000);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            System.out.println("同步代码块结束，线程：" + Thread.currentThread().getName());
        }
    }

    // 方式三：静态同步方法（锁定Class对象）
    public static synchronized void staticSyncMethod() {
        System.out.println("静态同步方法开始，线程：" + Thread.currentThread().getName());
        try {
            Thread.sleep(1000);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        System.out.println("静态同步方法结束，线程：" + Thread.currentThread().getName());
    }

    public static void main(String[] args) {
        SynchronizedDemo demo = new SynchronizedDemo();

        // 创建多个线程
        Thread t1 = new Thread(() -> demo.syncMethod(), "线程1");
        Thread t2 = new Thread(() -> demo.syncMethod(), "线程2");
        Thread t3 = new Thread(() -> demo.syncBlock(), "线程3");

        // 注意：由于t1和t2调用的是同一个对象的方法，t2需要等t1
        // t3调用的是不同方法，但锁对象是this，所以t3也要等t1/t2
        t1.start();
        t2.start();
        t3.start();
    }
}

/*
 * synchronized锁的原理：
 *
 * 每个对象都有一个monitor（监视器锁）
 * ┌─────────────────────────────────────────────────────────┐
 * │                    对象头（Object Header）              │
 *  ─────────────────────────────────────────────────────────
 * │  Mark Word（标记字段）                                    │
 │ │  - 状态：无锁 | 偏向锁 | 轻量级锁 | 重量级锁               │
 * │  - 持有锁的线程ID                                        │
 │ │  - 偏向时间戳                                           │
 │ ├─────────────────────────────────────────────────────────┤
> │  Klass Pointer（类型指针）                                │
 │ │  - 指向方法区类信息的指针                                │
 │ └─────────────────────────────────────────────────────────┘
 │
> *  synchronized的锁升级过程（不可逆）：
> *
> *   无锁状态 → 偏向锁 → 轻量级锁 → 重量级锁
> *   (一次CAS)   (多线程竞争)  (自旋超过阈值)
> *
> *  1. 无锁状态（001）
> *     - 初始状态，没有线程获取锁
> *
> *  2. 偏向锁（101）
> *     - 第一次被线程获取时，使用CAS记录线程ID
> *     - 之后该线程进入同步块不需要CAS
> *     - 适用于只有一个线程访问同步块的场景
> *     - 减少了一次CAS操作的开销
> *
> *  3. 轻量级锁（00）
> *     - 多个线程交替进入同步块（没有实际竞争）
> *     - 使用CAS自旋，避免线程阻塞/唤醒的开销
> *     - 适用于线程交替执行同步块的场景
> *     - 自旋超过一定次数（默认10次），升级为重量级锁
> *
> *  4. 重量级锁（10）
> *     - 多个线程同时竞争锁
> *     - 未获取锁的线程会被阻塞（BLOCKED状态）
> *     - 涉及OS层面mutex操作，开销较大
> *     - 但不会再消耗CPU自旋
> *
> * synchronized锁的工作过程：
> *
> *  线程A进入同步方法：
> *  ┌─────────────────────────────────────────────────────────┐
> *  │ object对象头                                          │
> *  │ ┌─────────────────────────────────────────────────────┐│
> *  │ │ Mark Word: 重量级锁(10) + 指向线程A的Monitor的指针  ││
> *  │ └─────────────────────────────────────────────────────┘│
> *  │                    │                                   │
> *  │                    ▼                                   │
> *  │  ┌─────────────────────────────────────────────────────┐│
> *  │  │ Monitor（监视器锁）                                 ││
> *  │  │ - _owner: 线程A                                     ││
> *  │  │ - _WaitSet: 空（没有等待的线程）                    ││
> *  │  │ - _EntryList: 空（没有竞争锁的线程）                ││
> *  │  │ - _recursions: 1（重入次数）                        ││
> *  │  └─────────────────────────────────────────────────────┘│
> *  └─────────────────────────────────────────────────────────┘
> *
> *  线程B尝试获取锁（竞争）：
> *  ┌─────────────────────────────────────────────────────────┐
> *  │ object对象头                                          │
> *  │ ┌─────────────────────────────────────────────────────┐│
> *  │ │ Mark Word: 重量级锁(10) + 指向线程A的Monitor的指针  ││
> *  │ └─────────────────────────────────────────────────────┘│
> *  │                    │                                   │
> *  │                    ▼                                   │
> *  │  ┌─────────────────────────────────────────────────────┐│
> *  │  │ Monitor                                            ││
> *  │  │ - _owner: 线程A                                     ││
> *  │  │ - _WaitSet: 空                                     ││
> *  │  │ - _EntryList: [线程B] ← 线程B在此等待               ││
> *  │  │ - _recursions: 1                                    ││
> *  │  └─────────────────────────────────────────────────────┘│
> *  └─────────────────────────────────────────────────────────┘
> *                                   │
> *                                   │ 线程A释放锁
> *                                   ▼
> *  ┌─────────────────────────────────────────────────────────┐
> *  │ Monitor通知EntryList中的线程竞争                        │
> *  │ - 线程B获得锁（通过CAS设置Mark Word）                    │
> *  │ - 线程B成为Monitor的新owner                             │
> *  └─────────────────────────────────────────────────────────┘
 */
```

#### 4.3.2 volatile关键字

**volatile就像餐厅的"公示牌"**：每次更新都会立即公示，所有人都能看到最新状态。

```java
/**
 * volatile特性演示
 */
public class VolatileDemo {

    // 普通变量：线程有自己的副本，可能不同步
    // volatile变量：所有线程共享主内存副本，每次读取都从主内存读
    private volatile boolean running = true;
    private int counter = 0;

    public void run() {
        System.out.println("开始执行...");

        // 线程尝试读取running变量
        while (running) {
            // 如果running不是volatile，线程可能一直读不到更新
            // 因为编译器可能优化：把running缓存到寄存器
            counter++;
        }

        System.out.println("停止执行，counter=" + counter);
    }

    public void stop() {
        // 修改running值
        running = false; // 写入主内存，对所有线程可见
    }

    public static void main(String[] args) throws InterruptedException {
        VolatileDemo demo = new VolatileDemo();

        Thread t = new Thread(() -> demo.run(), "计算线程");
        t.start();

        // 主线程睡1秒
        Thread.sleep(1000);

        // 主线程修改running
        demo.stop();

        // 等待子线程结束
        t.join();
    }
}

/*
 * volatile的内存语义：
 *
 * ┌─────────────────────────────────────────────────────────────┐
 *                      volatile读写原理                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  volatile写操作：                                            │
│  1. 线程修改值                                               │
│  2. 强制将值写回主内存                                        │
│  3. 让其他CPU缓存中的副本失效                                │
 │                                                             │
 │  volatile读操作：                                           │
 │  1. CPU缓存中的副本失效                                      │
 │  2. 从主内存读取最新值                                       │
 │                                                             │
 ├─────────────────────────────────────────────────────────────┤
 │                                                             │
 │  普通变量读写：                                              │
 │  ┌─────────┐         ┌─────────┐         ┌─────────┐      │
 │  │ 线程A   │ ─写─>   │ CPU缓存 │ ─写─>   │ 主内存  │      │
 │  └─────────┘         └─────────┘         └─────────┘      │
 │      │                                           │         │
 │      │               ┌─────────┐                   │         │
 │      └─────读───────│ CPU缓存 │◄────────────────────┘         │
 │                      └─────────┘                   │         │
 │                                                   │         │
 │                     可能读到过期数据！              │         │
 │                                                             │
 ├─────────────────────────────────────────────────────────────┤
 │                                                             │
 │  volatile读写：                                             │
 │  ┌─────────┐                                   ┌─────────┐  │
 │  │ 线程A   │ ──强制写入──>                   ──>│ 主内存  │  │
 │  └─────────┘      主内存                      │         │  │
 │      │                            缓存失效      │         │  │
 │      │◄──────强制读取───────    ───────────────┘         │
 │      │                    主内存                              │
 │                                                             │
 └─────────────────────────────────────────────────────────────┘
 *
 * volatile vs synchronized：
 *
 * ┌────────────────┬────────────────────────────────────────────┐
 * │     特性        │                  说明                       │
 * ├────────────────┼────────────────────────────────────────────┤
> │  原子性          │  synchronized保证                        │
> │                 │  volatile不保证原子性                      │
> ├────────────────┼────────────────────────────────────────────┤
> │  可见性          │  两者都保证                               │

> └────────────────┴────────────────────────────────────────────┘
 */
```

### 4.4 JUC并发包

#### 4.4.1 ReentrantLock - "更灵活的门禁"

ReentrantLock就像**比synchronized更智能的门禁系统**：可以try-lock、可以超时、可以公平/非公平选择。

```java
import java.util.concurrent.locks.ReentrantLock;
import java.util.concurrent.TimeUnit;

/**
 * ReentrantLock使用示例
 */
public class ReentrantLockDemo {

    // 创建ReentrantLock（false=非公平锁，true=公平锁）
    private final ReentrantLock lock = new ReentrantLock(false);

    // 统计
    private int count = 0;

    /**
     * 基本lock/unlock使用
     */
    public void basicLock() {
        lock.lock(); // 获取锁
        try {
            // 临界区代码
            count++;
            System.out.println("基本锁定，count=" + count + ", 线程=" + Thread.currentThread().getName());
        } finally {
            lock.unlock(); // 必须释放锁
        }
    }

    /**
     * tryLock：尝试获取锁，不阻塞
     * @return 是否获取成功
     */
    public boolean tryLockExample() {
        // 尝试一次获取锁
        if (lock.tryLock()) {
            try {
                count++;
                System.out.println("tryLock成功，count=" + count);
                return true;
            } finally {
                lock.unlock();
            }
        } else {
            System.out.println("tryLock失败，锁被占用");
            return false;
        }
    }

    /**
     * tryLock带超时：尝试获取锁，超时则放弃
     */
    public void tryLockWithTimeout() {
        try {
            // 尝试获取锁，最多等待5秒
            if (lock.tryLock(5, TimeUnit.SECONDS)) {
                try {
                    count++;
                    System.out.println("带超时tryLock成功，count=" + count);
                } finally {
                    lock.unlock();
                }
            } else {
                System.out.println("带超时tryLock超时，放弃");
            }
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }

    /**
     * 可中断锁：等待锁的过程中可以被中断
     */
    public void lockInterruptibly() throws InterruptedException {
        lock.lockInterruptibly(); // 可中断的获取锁
        try {
            count++;
            System.out.println("可中断锁，count=" + count);
        } finally {
            lock.unlock();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        ReentrantLockDemo demo = new ReentrantLockDemo();

        // 测试基本锁定
        for (int i = 0; i < 5; i++) {
            new Thread(() -> demo.basicLock(), "线程" + i).start();
        }

        Thread.sleep(100);

        // 测试tryLock
        System.out.println("\n=== 测试tryLock ===");
        demo.tryLockExample();
        demo.tryLockExample();
    }
}

/*
 * ReentrantLock vs synchronized：
 *
 * ┌─────────────────┬────────────────────────────────────────────┐
 * │     功能         │                  说明                      │
 * ├─────────────────┼────────────────────────────────────────────┤
 * │  尝试获取锁      │  synchronized：只能阻塞等待                 │
> │                 │  ReentrantLock：tryLock()立即返回          │
> │                 │  ReentrantLock：tryLock(timeout)超时等待   │
> ├─────────────────┼────────────────────────────────────────────┤
> │  可中断等待      │  synchronized：不可中断                      │
> │                 │  ReentrantLock：lockInterruptibly()可中断  │
> ├─────────────────┼────────────────────────────────────────────┤
> │  公平锁          │  synchronized：非公平锁                     │
> │                 │  ReentrantLock：可选择公平/非公平            │
> ├─────────────────┼────────────────────────────────────────────┤
> │  多个条件变量    │  synchronized：只能有一个条件（wait/notify）│
> │                 │  ReentrantLock：多个Condition              │
> ├─────────────────┼────────────────────────────────────────────┤
> │  锁获取顺序      │  synchronized：非公平                      │
> │                 │  ReentrantLock公平锁：等待最久的先获取       │
> └─────────────────┴────────────────────────────────────────────┘
 *
 * ReentrantLock可重入原理：
 *
 * Monitor中记录了：
 * - _owner：当前持有锁的线程
 * - _recursions：重入次数
 *
 * 当线程再次获取锁时：
 * 1. 检查owner是否是当前线程
 * 2. 如果是，recursions++，允许进入
 * 3. 如果不是，阻塞等待
 *
 * 释放锁时：
 * 1. recursions--
 * 2. 当recursions=0时，真正释放锁
 *
 * ┌─────────────────────────────────────────────────────────────┐
> │  可重入示意：                                                 │
> │                                                              │
> │  synchronized method1() {                                   │
> │      synchronized method2() {                               │
> │          // 同一线程可以进入                                │
> │          // 因为锁是可重入的                                │
> │      }                                                      │
> │  }                                                          │
> │                                                              │
> │  // 递归调用也是可重入的                                     │
> │  void recursive(int n) {                                   │
> │      synchronized(this) {                                 │
> │          if (n > 0) {                                      │
> │              recursive(n - 1);                             │
> │          }                                                  │
> │      }                                                      │
> │  }                                                          │
> └─────────────────────────────────────────────────────────────┘
 */
```

#### 4.4.2 CountDownLatch - "倒数计数器"

CountDownLatch就像**发令枪前的倒计时**：等所有人都准备好了，才开始。

```java
import java.util.concurrent.CountDownLatch;

/**
 * CountDownLatch示例：模拟比赛开始
 */
public class CountDownLatchDemo {

    // 裁判使用的倒计时锁
    private static CountDownLatch startSignal = new CountDownLatch(1);

    // 运动员完成信号
    private static CountDownLatch endSignal = new CountDownLatch(5);

    static class Athlete implements Runnable {
        private String name;
        private CountDownLatch startSignal;
        private CountDownLatch endSignal;

        public Athlete(String name, CountDownLatch startSignal, CountDownLatch endSignal) {
            this.name = name;
            this.startSignal = startSignal;
            this.endSignal = endSignal;
        }

        @Override
        public void run() {
            try {
                // 运动员等待裁判发令
                System.out.println(name + "：准备完毕，等待发令...");
                startSignal.await(); // 等待startSignal计数为0

                // 收到发令，开始跑步
                System.out.println(name + "：发令枪响，开始跑！");

                // 模拟跑步耗时
                Thread.sleep((long) (Math.random() * 2000));

                // 到达终点
                System.out.println(name + "：到达终点！");
                endSignal.countDown(); // 完成，计数减1

            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
    }

    public static void main(String[] args) throws InterruptedException {
        System.out.println("===== 比赛开始 =====");

        // 创建5个运动员线程
        for (int i = 1; i <= 5; i++) {
            new Thread(new Athlete("运动员" + i, startSignal, endSignal)).start();
        }

        // 裁判做准备...
        Thread.sleep(1000);
        System.out.println("裁判：各就各位！");

        // 等待1秒
        Thread.sleep(1000);
        System.out.println("裁判：预备！");

        // 等待1秒
        Thread.sleep(1000);
        System.out.println("裁判：砰！发令枪响！");

        // 计数减1，所有等待的线程同时开始
        startSignal.countDown();

        // 等待所有运动员完成
        endSignal.await();

        System.out.println("裁判：所有运动员都到达终点，比赛结束！");
    }
}

/*
 * CountDownLatch工作原理：
 *
 * ┌─────────────────────────────────────────────────────────────┐
 *                      CountDownLatch工作流程                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
 │  初始化：CountDownLatch latch = new CountDownLatch(3);     │
 │                         │                                  │
 │                         ▼                                  │
 │              计数器初始值 = 3                               │
 │                                                             │
 │  ┌─────────────────────────────────────────────────────┐   │
 │  │                    计数器状态                         │   │
 │  │              ● ● ●  (计数为3)                        │   │
 │  │              │ │ │                                   │   │
 │  │              ▼ ▼ ▼                                   │   │
 │  │           countDown()                               │   │
 │  │              countDown()                             │   │
 │  │              countDown()                             │   │
 │  └─────────────────────────────────────────────────────┘   │
 │                         │                                  │
 │                         ▼                                  │
 │              计数器 = 0，所有await()的线程被唤醒            │
 │                                                             │
 │  典型用法：                                                  │
 │  1. 等待多个前置任务完成：latch = new CountDownLatch(N)     │
```

#### 4.4.3 CyclicBarrier - "集合点"

CyclicBarrier就像**旅行团的集合点**：所有人都到了才出发，人齐了还能再来一轮。

```java
import java.util.concurrent.CyclicBarrier;
import java.util.concurrent.BrokenBarrierException;

/**
 * CyclicBarrier示例：模拟旅行团出游
 */
public class CyclicBarrierDemo {

    // 旅行团有5个人，约好在景点门口集合
    private static CyclicBarrier barrier = new CyclicBarrier(5, () -> {
        // 当所有人都到达时的回调（可选）
        System.out.println("所有人到齐了，拍照合影！");
    });

    static class Tourist implements Runnable {
        private String name;

        public Tourist(String name) {
            this.name = name;
        }

        @Override
        public void run() {
            try {
                // 模拟从不同地方出发
                int travelTime = (int) (Math.random() * 3000);
                Thread.sleep(travelTime);

                System.out.println(name + "：到达景点门口，还差" +
                    (barrier.getNumberWaiting()) + "人");

                // 到达集合点，等待其他人
                barrier.await();

                // 所有人都到达后，继续游览
                System.out.println(name + "：开始游览！");

            } catch (InterruptedException e) {
                e.printStackTrace();
            } catch (BrokenBarrierException e) {
                // 如果屏障被重置，会抛出这个异常
                e.printStackTrace();
            }
        }
    }

    public static void main(String[] args) throws InterruptedException {
        System.out.println("===== 旅行团出游 =====");

        // 创建5个游客
        for (int i = 1; i <= 5; i++) {
            new Thread(new Tourist("游客" + i)).start();
        }

        Thread.sleep(10000);

        // CyclicBarrier可以重复使用
        System.out.println("\n===== 第一轮结束，第二轮开始 =====");
        for (int i = 1; i <= 5; i++) {
            new Thread(new Tourist("游客" + i)).start();
        }
    }
}
```

#### 4.4.4 Semaphore - "停车场限流"

Semaphore就像**停车场的闸机**：限制同时进入的数量。

```java
import java.util.concurrent.Semaphore;
import java.util.concurrent.TimeUnit;

/**
 * Semaphore示例：停车场管理
 */
public class SemaphoreDemo {

    // 停车场有3个车位
    private static Semaphore semaphore = new Semaphore(3);

    static class Car implements Runnable {
        private String carNumber;

        public Car(String carNumber) {
            this.carNumber = carNumber;
        }

        @Override
        public void run() {
            try {
                // 尝试获取车位（获取1个许可）
                System.out.println(carNumber + "：准备进入停车场...");

                // tryAcquire()：非阻塞获取
                // acquire()：阻塞获取
                boolean acquired = semaphore.tryAcquire(3, TimeUnit.SECONDS);

                if (acquired) {
                    // 获取车位成功
                    System.out.println(carNumber + "：驶入停车场，占用车位");
                    System.out.println("当前可用车位：" + semaphore.availablePermits());

                    // 模拟停车
                    Thread.sleep((long) (Math.random() * 5000));

                    // 离开停车场，释放车位
                    System.out.println(carNumber + "：离开停车场");
                    semaphore.release();

                } else {
                    // 超时获取失败
                    System.out.println(carNumber + "：等待超时，去别的停车场");
                }

            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
    }

    public static void main(String[] args) {
        System.out.println("===== 停车场系统启动 =====");
        System.out.println("车位总数：" + 3);

        // 模拟10辆车来停车
        for (int i = 1; i <= 10; i++) {
            new Thread(new Car("车牌" + i)).start();
            try {
                Thread.sleep(500); // 每0.5秒来一辆
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
    }
}

/*
 * Semaphore工作原理：
 *
 * ┌─────────────────────────────────────────────────────────────┐
> │                    Semaphore原理                             │
├─────────────────────────────────────────────────────────────┤
> │                                                             │
│  构造函数：Semaphore permits = new Semaphore(3);            │
 │                                                             │
 │  内部结构：                                                  │
 │  ┌─────────────────────────────────────────────────────┐   │
> │  │                   Semaphore                          │   │
 │  │  - permits: 3                                        │   │
> │  │  - _sync: 同步器                                    │   │
> │  └─────────────────────────────────────────────────────┘   │
> │                                                             │
 │  acquire()获取许可：                                        │
 │  1. 检查permits > 0                                        │
 │  2. 如果是，permits--，获取成功                            │
 │  3. 如果否，阻塞等待                                        │
 │                                                             │
 │  release()释放许可：                                        │
 │  1. permits++                                              │
 │  2. 唤醒一个等待的线程                                     │
 │                                                             │
 ├─────────────────────────────────────────────────────────────┤
 │                                                             │
 │  限流示意：                                                  │
 │                                                             │
 │  ┌─────────┐    ┌─────────┐    ┌─────────┐                 │
 │  │ 车位1   │    │ 车位2   │    │ 车位3   │   空车位：3    │
 │  └─────────┘    └─────────┘    └─────────┘                 │
 │                                                             │
 │  来了5辆车：                                                │
 │  车1 ──进入──> 车位1   空车位：2                           │
 │  车2 ──进入──> 车位2   空车位：1                           │
 │  车3 ──进入──> 车位3   空车位：0                           │
 │  车4 ──阻塞等待──>                                                     │
 │  车5 ──阻塞等待──>                                                     │
 │                                                             │
 │  车1离开：                                                  │
 │  空车位：1                                                 │
 │  车4 ──进入──> 车位1   空车位：0                           │
 │                                                             │
 └─────────────────────────────────────────────────────────────┘
 *
 * Semaphore的两种模式：
 *
 * 1. 公平模式（FIFO）
 *    - 等待最久的线程先获取许可
 *    - new Semaphore(3, true)
 *
 * 2. 非公平模式（默认）
 *    - 允许"插队"
 *    - 新线程可能比等待的线程先获取许可
 *    - 吞吐量更高
 */
```

#### 4.4.5 CompletableFuture - "异步任务编排"

CompletableFuture就像**餐厅的智能点餐系统**：可以异步下单、等待结果、组合多个任务。

```java
import java.util.concurrent.CompletableFuture;
import java.util.concurrent.ExecutionException;

/**
 * CompletableFuture示例：异步菜品制作
 */
public class CompletableFutureDemo {

    public static void main(String[] args) throws ExecutionException, InterruptedException {
        System.out.println("===== 智能点餐系统 =====");

        // 1. 异步任务：创建 CompletableFuture
        CompletableFuture<String> orderFuture = CompletableFuture.supplyAsync(() -> {
            // 模拟下单
            System.out.println("订单已提交，正在处理...");
            try {
                Thread.sleep(2000); // 模拟处理时间
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            return "订单12345";
        });

        // 2. thenApply：处理结果（同步）
        CompletableFuture<String> cookingFuture = orderFuture.thenApply(orderId -> {
            // 模拟开始烹饪
            System.out.println(orderId + "：开始烹饪");
            return orderId + " - 宫保鸡丁";
        });

        // 3. thenCompose：扁平化（用于异步任务链）
        CompletableFuture<String> servingFuture = cookingFuture.thenCompose(dish ->
            CompletableFuture.supplyAsync(() -> {
                System.out.println(dish + "：烹饪完成，正在装盘");
                try {
                    Thread.sleep(500);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
                return dish + " - 上桌";
            })
        );

        // 4. thenCombine：组合两个独立任务
        CompletableFuture<String> drinkFuture = CompletableFuture.supplyAsync(() -> {
            System.out.println("饮品准备中...");
            try {
                Thread.sleep(1000);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            return "可乐一杯";
        });

        // 等待两个任务都完成
        String finalResult = servingFuture.thenCombine(drinkFuture, (dish, drink) -> {
            return dish + " + " + drink + " = 套餐完成";
        }).get();

        System.out.println("\n最终结果：" + finalResult);

        // 5. 异常处理
        CompletableFuture<String> errorFuture = CompletableFuture
            .supplyAsync(() -> {
                throw new RuntimeException("网络错误");
            })
            .exceptionally(ex -> {
                System.out.println("捕获异常：" + ex.getMessage());
                return "默认值";
            })
            .thenApply(result -> result + " - 降级处理");

        System.out.println("异常处理结果：" + errorFuture.get());
    }
}

/*
 * CompletableFuture常用方法：
 *
 * ┌─────────────────────────────────────────────────────────────┐
> │                    CompletableFuture方法图                  │
├─────────────────────────────────────────────────────────────┤
> │                                                             │
> │  创建阶段：                                                  │
> │  - supplyAsync(() -> ...)  异步执行，有返回值               │
> │  - runAsync(() -> ...)     异步执行，无返回值               │
> │                                                             │
> │  转换阶段（接受上一步结果）：                                 │
> │  - thenApply(fn)           同步转换                        │
> │  - thenApplyAsync(fn)      异步转换                        │
> │                                                             │
> │  消费阶段（不接受上一步结果）：                               │
> │  - thenAccept(consumer)    同步消费                        │
> │  - thenAcceptAsync(consumer) 异步消费                      │
> │                                                             │
> │  组合阶段：                                                  │
> │  - thenCompose(fn)         扁平化，fn返回CompletableFuture │
> │  - thenCombine(future, fn) 组合两个独立任务                 │
> │                                                             │
> │  异常处理：                                                  │
> │  - exceptionally(fn)       异常时执行                       │
> │  - handle(fn)              不论正常/异常都执行             │
> │  - whenComplete(consumer)  完成时执行（不改变结果）          │
> │                                                             │

> │  - join()                  阻塞获取结果（不抛异常）          │
```

## 五、实战代码：生产者-消费者问题

```java
import java.util.LinkedList;
import java.util.Queue;
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.ReentrantLock;

/**
 * 生产者-消费者问题实现
 *
 * 场景：厨师（生产者）做菜，服务员（消费者）端菜
 * 厨师做了5道菜后停止生产，服务员消费完所有菜后停止
 */
public class ProducerConsumerDemo {

    // 缓冲区：最多存放5道菜
    private static final int BUFFER_SIZE = 5;

    // 共享缓冲区
    private static Queue<String> buffer = new LinkedList<>();

    // 锁
    private static ReentrantLock lock = new ReentrantLock();

    // 条件变量：缓冲区不满/不空
    private static Condition notFull = lock.newCondition();
    private static Condition notEmpty = lock.newCondition();

    // 统计
    private static int producedCount = 0;
    private static int consumedCount = 0;

    /**
     * 生产者（厨师）
     */
    static class Producer implements Runnable {
        private String name;

        public Producer(String name) {
            this.name = name;
        }

        @Override
        public void run() {
            while (producedCount < 20) { // 生产20道菜
                lock.lock();
                try {
                    // 如果缓冲区满，等待消费者消费
                    while (buffer.size() >= BUFFER_SIZE) {
                        System.out.println(name + "：缓冲区满了，等待消费...");
                        notFull.await();
                    }

                    // 生产一道菜
                    String dish = "菜品" + (++producedCount);
                    buffer.offer(dish);
                    System.out.println(name + "：生产了 " + dish +
                        "，缓冲区大小：" + buffer.size());

                    // 通知消费者
                    notEmpty.signal();

                } catch (InterruptedException e) {
                    e.printStackTrace();
                } finally {
                    lock.unlock();
                }

                // 模拟生产时间
                try {
                    Thread.sleep(200);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        }
    }

    /**
     * 消费者（服务员）
     */
    static class Consumer implements Runnable {
        private String name;

        public Consumer(String name) {
            this.name = name;
        }

        @Override
        public void run() {
            while (consumedCount < 20) { // 消费20道菜
                lock.lock();
                try {
                    // 如果缓冲区空，等待生产者生产
                    while (buffer.isEmpty()) {
                        // 如果生产已完成，退出
                        if (producedCount >= 20 && buffer.isEmpty()) {
                            System.out.println(name + "：生产已全部完成，退出");
                            return;
                        }
                        System.out.println(name + "：缓冲区空了，等待生产...");
                        notEmpty.await();
                    }

                    // 消费一道菜
                    String dish = buffer.poll();
                    consumedCount++;
                    System.out.println(name + "：消费了 " + dish +
                        "，缓冲区大小：" + buffer.size());

                    // 通知生产者
                    notFull.signal();

                } catch (InterruptedException e) {
                    e.printStackTrace();
                } finally {
                    lock.unlock();
                }

                // 模拟消费时间
                try {
                    Thread.sleep(500);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        }
    }

    public static void main(String[] args) {
        System.out.println("===== 餐厅开始营业 =====");
        System.out.println("缓冲区大小：" + BUFFER_SIZE);
        System.out.println("生产数量：" + 20);
        System.out.println("消费数量：" + 20);
        System.out.println();

        // 创建1个生产者，2个消费者
        Thread producerThread = new Thread(new Producer("厨师老张"));
        Thread consumer1Thread = new Thread(new Consumer("服务员小李"));
        Thread consumer2Thread = new Thread(new Consumer("服务员小王"));

        producerThread.start();
        consumer1Thread.start();
        consumer2Thread.start();

        try {
            producerThread.join();
            consumer1Thread.join();
            consumer2Thread.join();
        } catch (InterruptedException e) {
            e.printStackTrace();
        }

        System.out.println("\n===== 餐厅打烊 =====");
        System.out.println("总生产：" + producedCount);
        System.out.println("总消费：" + consumedCount);
    }
}

/*
 * 生产者-消费者问题核心要点：
 *
 * 1. 线程安全
 *    - 使用ReentrantLock保护共享缓冲区
 *    - 每次操作缓冲区都要获取锁
 *
 * 2. 同步机制
 *    - notFull.await()：缓冲区满时，生产者等待
 *    - notEmpty.await()：缓冲区空时，消费者等待
 *    - signal()：消费/生产后唤醒对方
 *
 * 3. 避免虚假唤醒
 *    - while循环判断条件，而不是if
 *    - 原因：多线程环境下，await可能被多个线程同时唤醒
 *
 * 4. 锁的释放
 *    - await()会释放锁，让其他线程可以进入
 *    - signal()只是唤醒，不立即释放锁
 *    - 被唤醒的线程需要重新竞争锁
 *
 * 关键问题分析：
 *
 * Q: 为什么要用while而不是if判断条件？
 * A: 假设缓冲区大小为1，消费者A被唤醒消费后，
 *    在它释放锁之前，消费者B也被唤醒了。
 *    此时缓冲区已经空了，但B仍可能继续执行。
 *    while会再次检查，if不会。
 *
 * Q: 为什么不使用notify()而使用signal()？
 * A: notify()随机唤醒，可能导致"饥饿"
 *    signal()唤醒等待时间最长的线程，更公平
 *    在Condition中推荐使用signal()
 *
 * 运行结果示例：
 * 厨师老张：生产了 菜品1，缓冲区大小：1
 * 服务员小李：消费了 菜品1，缓冲区大小：0
 * 厨师老张：生产了 菜品2，缓冲区大小：1
 * ...
 */
```
