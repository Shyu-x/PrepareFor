# 前端微服务架构完全指南

## 前言：什么是前端微服务？

### 打个比方来理解

想象一下，你小时候玩的**乐高积木**。每一块乐高都是一个独立的小部件：
- 你有一块红色的2x4基础砖
- 你有一块蓝色的1x2薄板
- 你有一块绿色的1x1圆柱

这些积木块都是**独立存在**的，你可以单独拿起任何一块，可以单独替换任何一块，可以单独给任何一块涂色。但它们又能**组合在一起**，拼成一个大城堡、飞船或者任何你能想象的东西。

**前端微服务**就是这样的"乐高积木"。你的整个应用不是一整块大积木，而是由很多很多小块积木组成的：
- 用户模块是一块积木
- 订单模块是一块积木
- 购物车模块是一块积木
- 评论区是一块积木
- 视频播放器是一块积木

每块积木可以：
- **独立开发**：不同的团队可以同时开发不同的积木
- **独立测试**：每块积木可以单独测试，不用等其他积木
- **独立部署**：任何一块积木坏了，不会影响其他积木
- **独立替换**：想换掉购物车模块？没问题，直接拔掉换新的
- **按需加载**：用户不点购物车，购物车积木就永远不用加载

这就是前端微服务！把一个巨大的前端应用拆成很多小而美的独立模块，让它们既能独自运作，又能组合成完整应用。

### 从后端微服务说起

你可能听说过后端微服务。简单来说，后端微服务就是：

```
传统后端架构（单体应用）：
┌─────────────────────────────────┐
│         后端服务器               │
│  ┌─────┐ ┌─────┐ ┌─────┐       │
│  │用户 │ │订单 │ │支付 │       │
│  │服务 │ │服务 │ │服务 │       │
│  └─────┘ └─────┘ └─────┘       │
│  所有功能都在一个应用中           │
└─────────────────────────────────┘

后端微服务架构：
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 用户服务  │  │ 订单服务  │  │ 支付服务  │
│ (独立部署)│  │ (独立部署)│  │ (独立部署)│
└──────────┘  └──────────┘  └──────────┘
      ↓             ↓             ↓
   数据库1       数据库2       数据库3
```

后端微服务的核心理念是：**把一个巨大的后端应用拆成多个独立的小服务，每个服务负责一个明确的业务功能，可以独立开发、独立部署、独立扩展**。

前端微服务借鉴了这个思想，但解决的问题不太一样。

## 一、为什么前端需要微服务？

### 1.1 大型前端项目的困境

想象一下，你负责开发一个巨型前端应用，比如阿里巴巴的淘宝或者字节跳动的抖音。这样的应用：

- **代码量巨大**：可能有上百万行代码
- **团队规模庞大**：可能有几百个前端开发同时协作
- **业务复杂度高**：无数的功能模块交织在一起
- **迭代速度快**：每天可能都要发布新版本

如果还用传统的方式开发，会遇到这些问题：

#### 问题一：代码冲突频繁

```
传统开发模式：
                    main分支
                       │
        ┌──────────────┼──────────────┐
        │              │              │
     小明开发        小红开发        小刚开发
      (用户模块)      (订单模块)      (支付模块)
        │              │              │
        └──────────────┼──────────────┘
                       │
                   合并到main
                       │
               ⚠️ 冲突！冲突！冲突！
```

每天光是解决代码冲突就能让人疯掉。你改一行代码，可能影响到了别人；别人改一行代码，可能把你的功能搞挂了。几百个人的团队，几百万行代码，全都揉在一个代码仓库里，想象一下那混乱程度。

#### 问题二：构建速度慢得像蜗牛

你以为代码冲突是最痛苦的吗？too young too simple！

当你修改了一行代码，按下保存键，然后...然后你等了**30分钟**构建完成。为了一行代码的改动，你等了30分钟。为什么？因为你这个应用太大了，每次构建都要把整个应用重新打包一遍。

```
传统构建流程：
┌─────────────────────────────────────────────┐
│  开始构建                                    │
│    ↓                                        │
│  编译 TypeScript → 10分钟                    │
│    ↓                                        │
│  打包所有模块 → 15分钟                        │
│    ↓                                        │
│  优化 bundle → 5分钟                         │
│    ↓                                        │
│  完成！总共30分钟                            │
└─────────────────────────────────────────────┘

你的改动：只需要改一行代码
但是：整个应用要重新构建
```

#### 问题三：技术栈升级困难

想象一下，你的项目是三年前创建的，用的是 Vue 2。现在 Vue 3 发布了，性能提升巨大，生态也更丰富了。但是...你能升级吗？

- 升级意味着重写很多代码
- 重写意味着引入大量bug风险
- 引入bug意味着项目可能崩溃
- 项目崩溃意味着...你可能要被祭天

所以很多人选择了"摆烂"——不升了，能跑就行。于是项目越来越旧，技术债务越堆越高，最后成了一个谁都不敢动的祖传代码库。

#### 问题四：发布风险高

```
传统发布流程：
┌─────────────────────────────────────────────┐
│  发布新版本                                  │
│    ↓                                        │
│  ┌─────────────────────────────────────┐   │
│  │ 一次性发布整个应用                     │   │
│  │ 任何一个小bug都可能导致整个应用崩溃     │   │
│  └─────────────────────────────────────┘   │
│    ↓                                        │
│  ⚠️ 风险：所有功能一起发布，无法精细控制      │
└─────────────────────────────────────────────┘
```

你想改一个小按钮的颜色？要发布。整个应用重新打包、重新测试、重新部署。万一这个发布把支付模块搞挂了怎么办？你承担得起这个风险吗？

### 1.2 前端微服务如何解决这些问题

前端微服务就像给混乱的乐高世界带来了秩序。来看看它是怎么解决上面那些问题的：

#### 解决方案一：独立代码仓库（减少冲突）

```
前端微服务架构：
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  用户模块    │  │  订单模块    │  │  支付模块    │
│  (独立仓库)  │  │  (独立仓库)  │  │  (独立仓库)  │
└─────────────┘  └─────────────┘  └─────────────┘
      ↓                ↓                ↓
   独立部署         独立部署         独立部署
      ↓                ↓                ↓
   独立测试         独立测试         独立测试
```

每个团队负责自己的小模块，每个模块有自己的代码仓库。小明只改用户模块的代码，他的代码仓库和订单模块、支付模块的仓库完全隔离，怎么可能冲突？做梦都冲突不了！

#### 解决方案二：独立构建和部署（构建速度飙升）

```
前端微服务架构下的构建流程：
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  用户模块    │  │  订单模块    │  │  支付模块    │
│  (独立构建)  │  │  (独立构建)  │  │  (独立构建)  │
│   3分钟     │  │   5分钟     │  │   4分钟     │
└─────────────┘  └─────────────┘  └─────────────┘
      ↓                ↓                ↓
   独立部署         独立部署         独立部署
      ↓                ↓                ↓
   只需发布         只需发布         只需发布
   用户模块         订单模块         支付模块
```

你想改用户模块？好的，只构建用户模块，3分钟完成。其他模块？不需要动。一个改动，从30分钟变成了3分钟，你说爽不爽？

#### 解决方案三：技术栈自由（按模块选择）

```
传统模式（被迫统一）：
┌─────────────────────────────────┐
│        整个应用统一技术栈         │
│  ┌─────────────────────────────┐ │
│  │ React 15 + jQuery + Angular │ │
│  │ (各种混搭，欲哭无泪)         │ │
│  └─────────────────────────────┘ │
└─────────────────────────────────┘

微服务模式（各取所需）：
┌─────────────┬─────────────┬─────────────┐
│  用户模块    │  订单模块    │  支付模块    │
│  (React 18) │  (Vue 3)   │  (Svelte)  │
└─────────────┴─────────────┴─────────────┘
```

用户模块想用React 18？没问题！订单模块觉得Vue 3更适合？也没问题！支付模块想试试Svelte这种新潮框架？还是没问题！每个模块可以选择自己最合适的技术栈，不用被迫统一。

#### 解决方案四：独立发布（风险可控）

```
前端微服务架构下的发布流程：
┌─────────────────────────────────────────────┐
│  发布新版本                                  │
│    ↓                                        │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐      │
│  │用户模块 │  │订单模块 │  │支付模块 │      │
│  │发布v2.1 │  │发布v1.5 │  │发布v3.0 │      │
│  └─────────┘  └─────────┘  └─────────┘      │
│    ↓                                        │
│  某个模块发布出问题？只影响那个模块             │
│  其他模块正常运行，用户几乎无感知              │
└─────────────────────────────────────────────┘
```

你想改购物车模块？好的，只发布购物车模块。其他模块照常运行，购物车模块出问题了，用户还是能正常浏览商品、下订单，只是不能把东西加入购物车而已。这风险，比传统模式低了不知道多少倍。

## 二、前端微服务的核心概念

### 2.1 什么是模块解耦？

想象一下你和朋友的相处模式：

**紧密耦合模式（不健康的关系）**：
- 你和室友共享所有的生活用品
- 你的牙刷=室友的牙刷
- 你的毛巾=室友的毛巾
- 你买了一件新衣服，所有人的衣柜都要重新整理
- 有一天室友把牙刷弄丢了，结果你也无法刷牙了

**独立解耦模式（理想的关系）**：
- 每个人都有自己独立的牙刷、毛巾、衣柜
- 你买新衣服，只需要整理自己的衣柜
- 室友把牙刷弄丢了？那是他的问题，不影响你刷牙

前端微服务就是让你的代码模块从"紧密耦合"变成"独立解耦"。

#### 什么是耦合？

耦合就是模块之间的依赖程度。耦合越高，模块之间的依赖越强；耦合越低，模块越独立。

```
紧密耦合的代码示例：
```typescript
// 用户模块直接依赖订单模块的实现细节
// 这种写法，订单模块改一点点，用户模块就可能出问题
class UserService {
  private orderService = new OrderService(); // 直接实例化！

  // 用户模块直接调用订单模块的内部方法
  getUserWithOrders() {
    // 如果OrderService改了个方法名，整个UserService都要改
    const orders = this.orderService.getAllOrders();
    const user = this.getUserInfo();
    return { user, orders };
  }
}
```

```
解耦后的代码示例：
```typescript
// 用户模块只通过接口/事件与其他模块通信
class UserService {
  // 通过事件总线通信，不知道也不关心谁在监听
  publishUserUpdated(userId: number) {
    eventBus.publish('user:updated', { userId });
  }

  // 通过接口订阅事件，不知道是谁发布的
  subscribeToUserChanges(callback: (data) => void) {
    eventBus.subscribe('user:updated', callback);
  }
}
```

#### 解耦的好处：维修站比喻

想象你的汽车：
- 紧密耦合：发动机坏了，你可能要把整个车拆了才能修
- 解耦模式：发动机是一个独立模块，拆下来换一个新的就行，其他零件不用动

前端微服务就是让每个模块成为这样的"可拆卸零件"。

### 2.2 什么是独立部署？

独立部署就像是每个乐高小人都有自己独立的能力：

```
传统部署模式：
┌─────────────────────────────────────────────┐
│              打包整个应用                    │
│    包含：用户模块 + 订单模块 + 支付模块 + ...   │
│           (不管用户是否需要)                 │
└─────────────────────────────────────────────┘
                       ↓
                   一次性部署
                       ↓
              任何改动都要重新打包整个应用

微服务部署模式：
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  用户模块    │  │  订单模块    │  │  支付模块    │
│  (独立部署)  │  │  (独立部署)  │  │  (独立部署)  │
│  100KB      │  │  200KB      │  │  150KB      │
└─────────────┘  └─────────────┘  └─────────────┘
      ↓                ↓                ↓
   今天部署       明天部署       后天部署
   用户模块       订单模块       支付模块
```

#### 独立部署的三大好处

**好处一：故障隔离**

```
场景：支付模块突然崩溃了

传统模式：
┌─────────────────────────────────────┐
│ ⚠️ 整个应用崩溃！                    │
│ 用户无法：                          │
│  - 浏览商品                         │
│  - 查看订单                         │
│  - 使用购物车                        │
│  - 支付（这是应该的）                 │
│  - 搜索商品                         │
│  损失：巨大                          │
└─────────────────────────────────────┘

微服务模式：
┌─────────────────────────────────────┐
│ ✅ 只有支付模块不可用                │
│ 用户仍然可以：                       │
│  - 浏览商品（正常）                   │
│  - 查看订单（正常）                   │
│  - 使用购物车（正常）                 │
│  - 支付（不行）                      │
│  损失：很小                          │
└─────────────────────────────────────┘
```

**好处二：按需扩展**

```
场景：促销活动来了，订单量暴增

传统模式：
┌─────────────────────────────────────┐
│ 需要扩展整个应用，因为无法只扩展订单模块 │
│ 成本：需要扩容整个服务器               │
│ 浪费：用户模块、支付模块其实不需要扩容   │
└─────────────────────────────────────┘

微服务模式：
┌─────────────────────────────────────┐
│ 只需要扩容订单模块的服务器             │
│ 成本：只需要为订单模块付费              │
│ 效率：其他模块不受影响                 │
│                                    │
│  ┌─────────────┐                   │
│  │ 订单模块 x3  │ ← 扩容了          │
│  │ (3台服务器)  │                   │
│  └─────────────┘                   │
│                                    │
│  ┌─────────────┐                   │
│  │ 用户模块 x1  │ ← 保持不变        │
│  └─────────────┘                   │
└─────────────────────────────────────┘
```

**好处三：独立发布**

```
场景：产品经理说"我想改一下用户头像的样式"

传统模式：
1. 修改代码
2. 构建整个应用（30分钟）
3. 测试整个应用（2小时）
4. 发布整个应用
5. 祈祷不要出问题

微服务模式：
1. 修改用户模块代码
2. 构建用户模块（3分钟）
3. 测试用户模块（15分钟）
4. 发布用户模块
5. 如果出问题，只影响用户模块
```

### 2.3 什么是异步加载？

异步加载就像是自助餐厅和传统餐厅的区别：

**传统餐厅（同步加载）**：
- 你坐下来
- 服务员端上一道菜
- 你吃完
- 服务员再端下一道
- 所有菜必须一次全上齐你才能离开（或者全上齐才开始吃）

**自助餐厅（异步加载）**：
- 你坐下来
- 你去拿你想吃的菜
- 你边吃边可以去拿下一道
- 你不需要等所有菜都上齐才开始吃

前端微服务就是"自助餐厅模式"：

```
传统加载（同步）：
┌─────────────────────────────────────────────┐
│ 1. 加载整个应用                              │
│    ↓                                        │
│ 2. 加载完成之前，用户只能看到空白页面          │
│    ↓                                        │
│ 3. 所有功能一次性就绪                        │
│    ↓                                        │
│ 4. 用户才能开始使用                          │
└─────────────────────────────────────────────┘

微服务加载（异步）：
┌─────────────────────────────────────────────┐
│ 1. 先加载核心框架和主应用                     │
│    ↓ (用户可以看到页面骨架)                  │
│ 2. 加载用户模块（按需）                       │
│    ↓ (用户登录后)                           │
│ 3. 加载订单模块（按需）                      │
│    ↓ (用户点击订单)                         │
│ 4. 加载支付模块（按需）                      │
│    ↓ (用户点击支付)                         │
│ 5. ...模块们陆续加载，互不阻塞                │
└─────────────────────────────────────────────┘
```

#### 异步加载的实现原理

```typescript
// 同步加载方式（传统）
// 这种方式会在页面加载时一次性加载所有模块
import { UserModule } from './modules/user';
import { OrderModule } from './modules/order';
import { PaymentModule } from './modules/payment';

// 应用启动时，所有模块一起加载
const app = new App({
  modules: [UserModule, OrderModule, PaymentModule]
});

// 异步加载方式（微服务）
// 这种方式只在需要时才加载对应模块
class MicroAppLoader {
  // 定义模块的加载路径和依赖关系
  private moduleRegistry: Map<string, ModuleConfig> = new Map();

  // 注册一个模块，告诉系统怎么加载它
  registerModule(name: string, config: ModuleConfig) {
    this.moduleRegistry.set(name, config);
  }

  // 异步加载指定模块
  async loadModule(moduleName: string) {
    const config = this.moduleRegistry.get(moduleName);
    if (!config) {
      throw new Error(`模块 ${moduleName} 不存在`);
    }

    // 动态导入模块代码
    // 这行代码会在需要时才开始加载，而不是应用启动时就加载
    const module = await import(config.entry);

    // 初始化模块
    return this.bootstrapModule(module.default, config);
  }

  // 启动模块的引导程序
  private async bootstrapModule(moduleClass, config) {
    // 创建模块实例
    const instance = new moduleClass();

    // 挂载到指定DOM节点
    instance.mount(config.mountPoint);

    return instance;
  }
}

// 使用示例
const loader = new MicroAppLoader();

// 注册用户模块
loader.registerModule('user', {
  name: '用户模块',
  entry: './modules/user/index.ts',  // 模块的入口文件
  mountPoint: '#user-root',           // 模块要挂载的DOM节点
  dependencies: []                    // 依赖的其他模块

// 当用户点击"我的"按钮时，才加载用户模块
document.getElementById('user-btn')?.addEventListener('click', async () => {
  // 在这里才去加载用户模块的代码
  await loader.loadModule('user');

### 2.4 什么是主应用和子应用？

在微前端架构中，应用分为两种角色：

**主应用（Container App / Shell App）**：
- 就像是乐高基地板，所有的积木都要放在这个板上
- 负责整体布局、导航、路由管理
- 不知道也不关心各个子模块的具体实现
- 只负责加载和调度子模块

**子应用（Micro App / Child App）**：
- 就像是放在基地板上的各种乐高积木
- 负责具体的业务功能
- 不知道自己被谁加载，在哪里被加载
- 只需要按照规定的方式暴露自己

```
┌─────────────────────────────────────────────────────────┐
│                      主应用（基地板）                    │
│  ┌─────────────────────────────────────────────────────┐ │
│  │  顶部导航栏  │  侧边栏  │                           │ │
│  │             │          │                            │ │
│  │  ┌─────────┐│          │                            │ │
│  │  │ 用户模块 ││          │                            │ │
│  │  │(子应用1) ││          │                            │ │
│  │  └─────────┘│          │                            │ │
│  │             │          │                            │ │
│  │  ┌─────────┐│          │                            │ │
│  │  │ 订单模块 ││          │                            │ │
│  │  │(子应用2) ││          │                            │ │
│  │  └─────────┘│          │                            │ │
│  │             │          │                            │ │
│  │  ┌─────────┐│          │                            │ │
│  │  │ 支付模块 ││          │                            │ │
│  │  │(子应用3) ││          │                            │ │
│  │  └─────────┘│          │                            │ │
│  └─────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

#### 主应用的核心职责

```typescript
// 主应用的简化实现
class MainApp {
  // 存储所有子应用的信息
  private microApps: Map<string, MicroAppConfig> = new Map();

  // 注册子应用
  registerMicroApp(name: string, config: MicroAppConfig) {
    this.microApps.set(name, config);
  }

  // 根据路由加载对应的子应用
  async handleRouteChange(path: string) {
    // 找到匹配的子应用
    const matchedApp = this.findMatchedApp(path);

    if (matchedApp) {
      // 加载并启动子应用
      await this.loadMicroApp(matchedApp);
    }
  }

  // 加载子应用
  private async loadMicroApp(config: MicroAppConfig) {
    // 1. 动态加载子应用的代码
    // 这里使用了动态import，会在浏览器中发起一个新的请求去加载子应用
    const microApp = await import(config.entry);

    // 2. 创建子应用实例
    const instance = new microApp.default();

    // 3. 挂载到指定的DOM节点
    // 注意：子应用不知道也不关心这个DOM节点在哪里
    instance.mount(config.container);
  }
}
```

#### 子应用的核心要求

```typescript
// 子应用必须导出一个标准化的接口
// 这样主应用才能统一管理和调度

// 方式一：使用对象导出
const microApp = {
  // 应用启动时调用
  bootstrap() {
    console.log('子应用启动');
    // 初始化操作：创建组件、设置状态、绑定事件
  },

  // 应用挂载时调用
  mount(container) {
    console.log('子应用被挂载到', container);
    // 真正的渲染操作：把组件渲染到container中
  },

  // 应用卸载时调用
  unmount() {
    console.log('子应用被卸载');
    // 清理操作：取消事件绑定、清理定时器、释放资源
  },

  // 应用每次被激活时调用（比如从后台切回来）
  activate() {
    console.log('子应用被激活');
  }
};

// 方式二：使用类导出
class OrderModule {
  private container: HTMLElement;

  // 启动钩子 - 主应用加载完子应用代码后首先调用
  bootstrap() {
    console.log('订单模块初始化中...');

  // 挂载钩子 - 子应用要被渲染到页面时调用

    this.render();
  }

  // 卸载钩子 - 子应用要从页面移除时调用
  unmount() {
    // 清理所有资源
    this.container.innerHTML = '';
    this.removeEventListeners();
  }

  private render() {
    // 渲染订单列表的逻辑
  }

  private removeEventListeners() {
    // 移除事件监听器等清理操作
  }
}

// 导出默认实例
export default new OrderModule();

## 三、前端微服务的实现方式

### 3.1 Module Federation（Webpack的解决方案）

Module Federation就像是**拼图游戏**。你有一块大拼图，被分成了很多小块：

```
拼图游戏：
┌─────────┬─────────┬─────────┐
│  模块A  │  模块B  │  模块C  │
│ (本地)  │ (共享)  │ (远程)  │
├─────────┼─────────┼─────────┤
│  模块D  │  模块E  │  模块F  │
│ (远程)  │ (共享)  │ (远程)  │
├─────────┼─────────┼─────────┤
│  模块G  │  模块H  │  模块I  │
│ (本地)  │ (远程)  │ (本地)  │
└─────────┴─────────┴─────────┘

有些拼图块在你手上（本地模块）
有些拼图块在朋友手上（远程模块）
你们可以互相借拼图块来完成任务
```

Module Federation让Webpack支持这种"拼图"式的加载方式，可以动态加载其他打包产物中的模块。

```javascript
// Webpack 5 的 Module Federation 配置示例
// webpack.config.js

// 主机应用（Host）的配置
const ModuleFederationPlugin = require('webpack/lib/container/ModuleFederationPlugin');
const deps = require('./package.json').dependencies;

module.exports = {
  // ... 其他webpack配置

  plugins: [
    new ModuleFederationPlugin({
      // 给这个应用起个名字
      name: 'host_app',

      // 声明要共享的模块（其他应用可以使用这些模块）
      // 就像是在说："这些模块我可以借给你们用"
      shared: {
        // 共享react及react-dom
        // version表示版本，singleton表示单例模式
        react: {
          singleton: true,           // 单例模式：整个应用只用一个react实例
          requiredVersion: deps.react  // 需要的版本
        },
        'react-dom': {
          singleton: true,
          requiredVersion: deps['react-dom']
        }
      }
    })
  ]
};
```

```javascript
// 子应用（Remote）的配置
// webpack.config.js

const ModuleFederationPlugin = require('webpack/lib/container/ModuleFederationPlugin');
const deps = require('./package.json').dependencies;

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      // 给子应用起个名字
      name: 'remote_app',

      // 声明要暴露给其他应用使用的模块
      // 就像是在说："这些模块我可以借给你们"
      exposes: {
        // 键是模块的路径，值是模块的实际文件
        './OrderModule': './src/components/OrderModule.tsx',
        './UserProfile': './src/components/UserProfile.tsx'
      },

      // 声明需要从主机应用共享的模块
      // 就像是在说："我需要从你们那里借这些模块"
      shared: {
        react: {
          singleton: true,
          requiredVersion: deps.react
        },
        'react-dom': {
          singleton: true,
          requiredVersion: deps['react-dom']
        }
      }
    })
  ]
};
```

### 3.3 iframe方案（最简单但最笨重）

iframe方案就像是在一个大房间里用玻璃隔开不同区域：

**优点**：
- 实现超级简单
- 天然隔离：每个子应用在独立的iframe中，样式、JS完全隔离
- 兼容性好：什么浏览器都能用

**缺点**：
- 通信困难：跨iframe通信需要postMessage，麻烦
- 体验差：iframe的边框、滚动条、加载状态都很难看
- SEO不友好：搜索引擎基本不抓取iframe内容
- 性能差：每个iframe都要独立运行一套React/Vue，内存占用大

```html
<!-- 最原始的iframe方案 -->
<!DOCTYPE html>
<html>
<head>
  <title>主应用</title>
</head>
<body>
  <nav>
    <a href="#" onclick="switchApp('user')">用户模块</a>
    <a href="#" onclick="switchApp('order')">订单模块</a>
  </nav>

  <div id="app-container">
    <!-- 子应用将在这里渲染 -->
    <!-- 每个子应用都是一个独立的iframe -->
    <iframe id="user-iframe" src="http://localhost:3001" style="display:none"></iframe>
    <iframe id="order-iframe" src="http://localhost:3002" style="display:none"></iframe>
  </div>

  <script>
    function switchApp(appName) {
      // 隐藏所有iframe
      document.querySelectorAll('iframe').forEach(iframe => {
        iframe.style.display = 'none';
      });

      // 显示目标iframe
      document.getElementById(`${appName}-iframe`).style.display = 'block';
    }
  </script>
</body>
</html>

### 4.2 挑战大盘点

#### 挑战一：公共依赖的管理

当多个子应用都需要React时，是每个子应用都打包一份，还是共享一份？

```
问题场景：
┌─────────────────────────────────────────┐
│  主机应用 + 5个子应用                     │
│  每个都用React                           │
│                                         │
│  方案A：每个子应用都打包React              │
│  → 总共6份React代码，浪费带宽和内存        │
│                                         │
│  方案B：共享React                         │
│  → 需要版本协调，复杂度上升               │
└─────────────────────────────────────────┘
```

解决方案：使用共享依赖（Shared Dependencies）

```javascript
// Webpack的Module Federation配置
// 告诉Webpack：这些依赖我希望大家共享
new ModuleFederationPlugin({
  shared: {
    // 我希望react被共享
    // singleton: true 表示只允许存在一个实例
    // requiredVersion: '^18.0.0' 表示需要18.x版本
    react: {
      singleton: true,
      requiredVersion: '^18.0.0',
      // 如果版本不匹配怎么办？strict: true表示报错
      strictVersion: true
    },
    'react-dom': {
      singleton: true,
      requiredVersion: '^18.0.0'
    }
  }
})
```

#### 挑战二：样式冲突

当多个子应用都在页面上时，CSS样式可能会互相污染：

```css
/* 子应用A定义的样式 */
.button {
  background: blue;
  color: white;
}

/* 子应用B也定义了同样的类名 */
.button {
  background: red;
  color: black;
}

/* 结果：谁后加载，谁的样式就覆盖前面的 */
```

解决方案一：CSS命名空间（给每个子应用的样式加前缀）

```css
/* 子应用A */
.app-a .button {
  background: blue;
  color: white;
}

/* 子应用B */
.app-b .button {
  background: red;
  color: black;
}
```

解决方案二：CSS Modules（把类名编译成唯一的哈希值）

```css
/* Button.module.css */
/* 编译后可能变成 .button_abc123 */
.button {
  background: blue;
}
```

解决方案三：Shadow DOM（最彻底的隔离）

```typescript
// 使用Shadow DOM来隔离子应用的样式
class MicroApp extends HTMLElement {
  constructor() {
    super();
    // 创建Shadow DOM，内部的样式不会影响外部
    this.attachShadow({ mode: 'open' });
  }

  connectedCallback() {
    // 渲染逻辑...
    this.shadowRoot.innerHTML = `
      <style>
        .button { background: blue; }
      </style>
      <button class="button">点击我</button>
    `;
  }
}
```

#### 挑战三：跨子应用通信

当子应用之间需要传递数据时，该怎么做？

```
问题场景：
┌─────────────────────────────────────────┐
│           用户模块（子应用A）              │
│              点击了"修改地址"              │
│                    ↓                     │
│              需要通知                      │
│                    ↓                     │
│           订单模块（子应用B）              │
│              刷新地址显示                 │
└─────────────────────────────────────────┘
```

解决方案一：基于CustomEvent的发布订阅

```typescript
// 事件总线工具类
class EventBus {
  // 发布事件
  static publish(eventName: string, data: any) {
    // 创建一个自定义事件
    const event = new CustomEvent(eventName, { detail: data });
    // 在window上触发事件
    window.dispatchEvent(event);
  }

  // 订阅事件
  static subscribe(eventName: string, callback: (data: any) => void) {
    // 添加事件监听器
    window.addEventListener(eventName, (event: CustomEvent) => {
      callback(event.detail);
    });
  }

  // 取消订阅
  static unsubscribe(eventName: string, callback: (data: any) => void) {
    window.removeEventListener(eventName, callback);
  }
}

// 子应用A - 用户模块
function UserModule() {
  // 用户修改了地址
  function onAddressChange(newAddress) {
    // 发布一个地址变更事件
    EventBus.publish('address:changed', {
      userId: '123',
      address: newAddress,
      timestamp: Date.now()
    });
  }
}

// 子应用B - 订单模块
function OrderModule() {
  // 订阅地址变更事件
  EventBus.subscribe('address:changed', (data) => {
    console.log('地址变更收到:', data);
    // 刷新订单中的地址显示
    refreshAddress(data.userId, data.address);
  });
}
```

解决方案二：基于Props的父子通信（通过主应用）

```typescript
// 主应用
function MainApp() {
  const [sharedState, setSharedState] = useState({});

  return (
    <>
      {/* 传递状态和更新方法给子应用 */}
      <UserApp
        sharedState={sharedState}
        onStateChange={setSharedState}
      />

      {/* 子应用通过props拿到状态和方法 */}
      <OrderApp
        sharedState={sharedState}
        onStateChange={setSharedState}
      />
    </>
  );
}
```

### 5.2 场景二：企业级中台系统

**背景**：
一个大型企业的内部中台系统，包含人力资源、财务、审批、报表等多个模块。

**问题**：
- 不同部门使用不同的技术栈，HR用Angular，财务用React，审批用Vue
- 各个模块由不同部门的开发团队维护，协调成本高
- 系统迭代缓慢，一个小改动要走很长的发布流程

**效果**：
- 允许不同部门使用自己擅长的技术栈
- 各模块独立部署，互不影响
- 可以让合适的团队维护合适的模块
- 打破了部门之间的技术壁垒

### 5.3 场景三：低代码平台

## 六、踩坑经验总结

### 坑一：子应用之间的DOM冲突

**问题描述**：
多个子应用同时运行时，可能出现DOM元素的ID或Class冲突，导致样式和行为错乱。

**踩坑经历**：
```
我第一次用qiankun时，兴冲冲地加载了两个子应用：
- 子应用A：有个id="modal"的div
- 子应用B：也有个id="modal"的div

结果：子应用B的modal把子应用A的modal覆盖了！
```

**解决方案**：

方案一：使用唯一前缀

```typescript
// 在子应用初始化时，给所有DOM元素添加唯一前缀
function initializeApp(appName: string) {
  // 给body添加一个唯一标识
  document.body.classList.add(`app-${appName}`);

  // 或者使用Shadow DOM完全隔离
  const shadowRoot = document.createElement('div').attachShadow({ mode: 'open' });
}
```

方案二：使用CSS Modules

```typescript
// Vue组件中使用 scoped CSS
// <style scoped>
// .modal { ... }
// </style>

// 编译后变成
// .modal[data-v-xxxxx] { ... }
```

### 坑二：子应用关闭后内存泄漏

**问题描述**：
子应用卸载后，JavaScript对象、定时器、事件监听器仍然存在，导致内存泄漏。

**踩坑经历**：
```
我加载了一个子应用，然后关闭它。
再次加载时，发现内存占用比之前高了50MB！
原来是定时器没有被清理...
```

**解决方案**：

```typescript
// 子应用必须正确实现卸载逻辑
class MyMicroApp {
  private timer: any;
  private eventListeners: any[] = [];

  // 启动时
  bootstrap() {
    // 设置定时器
    this.timer = setInterval(() => {
      this.doSomething();
    }, 1000);

    // 添加事件监听
    const handler = () => console.log('click');
    window.addEventListener('click', handler);
    this.eventListeners.push(handler);
  }

  // 卸载时 - 必须清理所有资源！
  unmount() {
    // 清理定时器
    if (this.timer) {
      clearInterval(this.timer);
      this.timer = null;
    }

    // 清理所有事件监听器
    this.eventListeners.forEach(handler => {
      window.removeEventListener('click', handler);
    });
    this.eventListeners = [];

    // 清理DOM
    this.container.innerHTML = '';

    // 清理任何其他资源，比如WebSocket连接、IndexedDB等
    if (this.ws) {
      this.ws.close();
    }
  }
}
```

### 坑三：共享依赖版本不一致

**问题描述**：
主机应用和子应用依赖不同版本的React，导致状态不同步或渲染异常。

**踩坑经历**：
```
主机应用用React 18
子应用A用React 17
子应用B用React 18

结果：点击事件触发后，子应用A完全没反应！
原因：事件系统在不同版本的React中实现不同
```

**解决方案**：

```javascript
// Webpack配置中严格管理共享依赖
new ModuleFederationPlugin({
  shared: {
    react: {
      singleton: true,           // 单例模式，只允许一个版本
      requiredVersion: '^18.0.0', // 严格的版本要求
      strictVersion: true        // 版本不匹配就报错
    },
    'react-dom': {
      singleton: true,
      requiredVersion: '^18.0.0'
    }
  }
})
```

```typescript
// 或者在qiankun中配置
loadMicroApp({
  name: 'order-app',
  entry: '//localhost:7001',
  container: '#container',
  // 显式声明共享的依赖
  props: {
    // 主应用传递给子应用的props
  }
}, {
  // 共享的依赖配置
  shareScope: 'default',  // 共享域
  modules: {
    // 强制使用这些版本
    react: '18.2.0',
    'react-dom': '18.2.0'
  }
})
```

### 坑四：子应用加载白屏

**问题描述**：
子应用加载时，页面出现短暂的白屏，用户体验很差。

**踩坑经历**：
```
用户点击"订单"标签
等待3秒...白屏
等待4秒...白屏
终于！订单页面出来了
用户投诉：你们页面怎么这么慢？
```

**解决方案**：

方案一：骨架屏 + loading状态

```typescript
// 主应用
function MainApp() {
  const [loading, setLoading] = useState(false);

  // 加载子应用前显示loading
  async function loadApp(appName) {
    setLoading(true);

    // 显示骨架屏
    showSkeleton(appName);

    await loadMicroApp(appName);

    setLoading(false);
    hideSkeleton();
  }
}
```

方案二：预加载子应用

```typescript
// 在用户还没点击之前，就预先加载子应用
// 比如用户在浏览商品时，就预先加载订单模块
class AppPreloader {
  // 预加载策略
  preloadApps() {
    // 预加载订单模块（猜测用户可能会点击）
    loadMicroApp({ name: 'order-app' }, { immediate: false });

    // 预加载用户模块
    loadMicroApp({ name: 'user-app' }, { immediate: false });
  }
}

// 用户鼠标悬停在某个导航项时预加载
navItems.forEach(item => {
  item.addEventListener('mouseenter', () => {
    // 鼠标悬停就预加载
    preloader.preloadApp(item.appName);
  });
});
```

### 什么时候应该用前端微服务？

**适合使用微服务**：
- ✅ 超大型前端项目（几十人以上团队）
- ✅ 需要技术栈多样性（不同模块想用不同技术）
- ✅ 需要独立部署（不同模块发布频率差异大）
- ✅ 需要渐进式升级（想逐步迁移旧项目）
- ✅ 需要故障隔离（某个模块挂了不影响其他）

**不适合使用微服务**：
- ❌ 小型项目（3-5人团队，代码量不大）
- ❌ 简单应用（功能单一，不需要拆分）
- ❌ 追求极致性能（微服务有额外开销）
- ❌ 团队经验不足（微服务需要更高的架构能力）

### 最佳实践

1. **从小开始**：不要一开始就想着把所有模块都拆成微服务，先从1-2个模块开始
2. **明确边界**：模块之间的边界要清晰，避免循环依赖
3. **接口先行**：先定义好模块之间的接口，再开始独立开发
4. **自动化测试**：每个模块都要有完善的测试用例，确保独立运行时不会出错
5. **监控到位**：做好监控和日志，及时发现和定位问题
6. **文档完善**：每个模块都要有清晰的使用文档，降低协作成本

前端微服务不是银弹，它解决了一些问题的同时，也引入了一些新的复杂性。在决定使用之前，请确保你真的需要它。
