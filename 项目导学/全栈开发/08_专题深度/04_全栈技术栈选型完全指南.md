# 全栈技术栈选型完全指南

## 一、引言

技术选型是全栈开发中最关键的决策之一，它直接影响项目的开发效率、运维成本和可扩展性。本指南将系统性地讲解全栈开发中各个层面的技术选型决策，帮助开发者和团队根据项目规模、业务需求和资源约束做出最佳选择。

技术选型的核心原则是：**没有最好的技术，只有最合适的技术**。每一项技术都有其适用场景和局限性，优秀的架构师能够根据实际情况权衡利弊，选择最匹配的技术方案。

本指南涵盖的技术选型维度包括：架构模式、后端框架、数据库、认证方案、实时通信、缓存策略和部署方案。我们将从原理出发，分析各方案的优缺点，并给出不同规模项目的推荐配置。

## 二、架构模式选型

### 2.1 BFF架构设计

**BFF（Backend For Frontend）** 即服务于前端的后端架构，是一种针对前端需求专门设计的后端服务层。BFF模式的核心思想是在后端与前端之间增加一个聚合层，统一处理来自多个微服务的响应数据，按前端需要的形式进行组织和裁剪。

**BFF架构的典型应用场景：**

```typescript
// 传统模式：前端需要调用多个接口
// 接口1：获取用户基本信息
const userInfo = await fetch('/api/user/info');
// 接口2：获取用户订单列表
const orders = await fetch('/api/user/orders');
// 接口3：获取用户收藏
const favorites = await fetch('/api/user/favorites');

// BFF模式：前端只需调用一个接口
const userDashboard = await fetch('/api/bff/user-dashboard');
// 返回聚合后的完整数据
```

**BFF架构的优势分析：**

| 优势 | 说明 |
|------|------|
| **减少网络请求** | 前端无需多次往返，聚合层一次性返回所有数据 |
| **保护后端服务** | 前端无法直接访问内部微服务，降低耦合度 |
| **接口定制化** | 根据不同前端（Web/iOS/Android）返回不同格式 |
| **逻辑复用** | 跨多个接口的业务逻辑在BFF层统一处理 |
| **独立演进** | 前端和后端可以独立迭代，通过BFF解耦 |

**BFF的实现方式：**

```typescript
// 使用Node.js/Express实现BFF层
// bff-server/index.ts
import express from 'express';
import { createProxyMiddleware } from 'http-proxy-middleware';

const app = express();

// 用户信息聚合接口
app.get('/api/bff/user-dashboard', async (req, res) => {
  try {
    // 并行调用多个后端服务
    const [userInfo, orders, favorites] = await Promise.all([
      fetch('http://user-service:3001/info'),
      fetch('http://order-service:3002/list'),
      fetch('http://favorite-service:3003/items')
    ]);

    // 数据聚合与转换
    const dashboardData = {
      user: {
        id: userInfo.id,
        name: userInfo.name,
        avatar: userInfo.avatarUrl
      },
      summary: {
        orderCount: orders.total,
        favoriteCount: favorites.items.length
      },
      recentOrders: orders.items.slice(0, 5),
      topFavorites: favorites.items.slice(0, 3)
    };

    res.json(dashboardData);
  } catch (error) {
    res.status(500).json({ error: 'BFF聚合失败' });
  }
});

// 移动端专用接口（返回更少数据，节省流量）
app.get('/api/bff/mobile/user-dashboard', async (req, res) => {
  // 移动端定制逻辑
});
```

**BFF架构的适用性判断：**

BFF架构并非适用于所有项目。建议在以下情况下考虑使用BFF：

1. **微服务架构**：当后端拆分为多个独立服务时，前端需要聚合数据
2. **多端适配**：同一业务需要支持Web、iOS、Android等多个前端
3. **性能敏感**：减少网络往返次数对用户体验有显著影响
4. **接口版本管理**：需要同时维护多个版本的API

对于小型项目或单体应用，BFF架构可能增加不必要的复杂度。

### 2.2 前后端分离vs一体化

前后端分离和前后端一体化是两种主流的架构风格，各有其适用场景。

**前后端一体化（Monolithic）架构特点：**

```java
// Spring Boot一体架构示例
// 同一个应用中同时包含前端资源和后端逻辑
// src/main/resources/static/index.html  // 前端页面
// src/main/java/com/example/controller/ // 后端API
// src/main/java/com/example/service/    // 业务逻辑

@RestController
@RequestMapping("/api/users")
public class UserController {
    @GetMapping("/{id}")
    public User getUser(@PathVariable Long id) {
        return userService.findById(id);
    }
}
```

| 一体化架构 | 优势 | 劣势 |
|------------|------|------|
| **开发体验** | 调试方便，全在一个项目内 | 前后端开发可能相互影响 |
| **部署简单** | 单一部署单元 | 扩展不灵活 |
| **团队协作** | 小团队沟通成本低 | 大团队职责边界模糊 |
| **性能** | 内网通信，速度快 | 扩展时需要整体扩容 |

**前后端分离架构特点：**

```
项目结构：
├── frontend/          # 独立前端项目
│   ├── src/
│   ├── package.json
│   └── next.config.ts
│
├── backend/           # 独立后端项目
│   ├── src/
│   ├── package.json
│   └── nest-cli.json
│
└── shared/            # 共享类型定义
    └── types/
```

| 分离架构 | 优势 | 劣势 |
|----------|------|------|
| **技术选型** | 前后端可独立选择技术栈 | 需要API契约约定 |
| **独立部署** | 前端可CDN部署，后端弹性扩展 | 部署流程更复杂 |
| **团队边界** | 前后端团队职责清晰 | 跨团队沟通成本 |
| **性能优化** | 各自独立优化 | 网络延迟增加 |

**决策流程图：**

```
项目启动 → 评估团队规模
    ↓
团队规模 < 5人?
    ├─ 是 → 一体化架构（Next.js/Nuxt.js全栈框架）
    │         ↓
    │      团队技术栈统一?
    │         ├─ 是 → 直接使用框架自带API
    │         └─ 否 → 前后端独立项目，但用同一语言
    │
    └─ 否 → 评估业务复杂度
              ↓
           业务复杂度低?
              ├─ 是 → 前后端分离，但可共用代码仓库
              └─ 否 → 完全分离 + BFF层
```

**推荐方案：**

| 项目规模 | 架构选择 | 技术组合 |
|----------|----------|----------|
| 个人项目 | 一体化 | Next.js / Nuxt.js |
| 小团队（2-5人） | 渐进分离 | Next.js + 独立API服务 |
| 中型团队（5-20人） | 前后端分离 | React/Vue + Node BFF |
| 大型团队（20+人） | 微服务+BFF | 多前端 + BFF + 微服务 |

## 三、数据库选型

### 3.1 关系型vs非关系型数据库

数据库选型是后端架构中最核心的决策之一。关系型数据库和非关系型数据库各有其适用场景，正确的选型能够大大简化开发工作并提升系统性能。

**关系型数据库（RDBMS）核心特征：**

| 特性 | 说明 |
|------|------|
| **ACID事务** | 原子性、一致性、隔离性、持久性保证 |
| **SQL查询** | 结构化查询语言，强大的查询能力 |
| **Schema约束** | 预定义表结构，数据完整性高 |
| **关联查询** | JOIN操作支持复杂的数据关系 |
| **成熟生态** |  decades of tooling and optimization |

**主流关系型数据库对比：**

| 数据库 | 适用场景 | 优势 | 劣势 |
|--------|----------|------|------|
| **PostgreSQL** | 通用场景、需要高级特性 | 功能丰富、扩展性强、性能优秀 | 相对较重 |
| **MySQL** | Web应用、高并发读 | 生态广泛、运维简单 | 功能相对有限 |
| **SQLite** | 嵌入式、轻量级 | 零配置、文件级存储 | 并发有限 |
| **MariaDB** | MySQL替代 | 兼容MySQL + 开源 | 生态稍弱 |

**PostgreSQL高级特性示例：**

```sql
-- 1. 数组类型支持
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    interests TEXT[]  -- PostgreSQL数组类型
);

INSERT INTO users (name, interests)
VALUES ('张三', ARRAY['编程', '阅读', '旅行']);

-- 2. JSONB类型（兼顾灵活性和查询能力）
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    data JSONB,  -- 灵活存储订单详情
    created_at TIMESTAMP DEFAULT NOW()
);

-- 支持JSON查询
SELECT * FROM orders
WHERE data->>'status' = 'completed';

-- 3. 全文搜索
CREATE INDEX idx_products_search ON products
USING GIN (to_tsvector('chinese', name || ' ' || description));

-- 4. 事务支持
BEGIN;
UPDATE accounts SET balance = balance - 1000 WHERE user_id = 1;
UPDATE accounts SET balance = balance + 1000 WHERE user_id = 2;
COMMIT;
```

**非关系型数据库（NoSQL）核心特征：**

| 特性 | 说明 |
|------|------|
| **Schema灵活** | 无固定表结构，存储灵活 |
| **水平扩展** | 天然分布式，易于扩容 |
| **高性能** | 优化后的读写性能 |
| **多样化模型** | 文档、键值、列族、图等多种模型 |

**非关系型数据库分类与代表产品：**

| 类型 | 代表产品 | 适用场景 |
|------|----------|----------|
| **文档数据库** | MongoDB, CouchDB | 内容管理、日志、灵活Schema |
| **键值数据库** | Redis, DynamoDB | 缓存、Session、会话存储 |
| **列族数据库** | Cassandra, HBase | 时序数据、大数据存储 |
| **图数据库** | Neo4j, ArangoDB | 社交关系、推荐系统 |

### 3.2 数据库选型决策树

```
业务需求分析
    ↓
是否需要强一致性事务?
    ├─ 是 → 选择关系型数据库（PostgreSQL/MySQL）
    │         ↓
    │      是否需要复杂关联查询?
    │         ├─ 是 → PostgreSQL（更强的事务和查询能力）
    │         └─ 否 → MySQL（更轻量）
    │
    └─ 否 → 评估数据结构
              ↓
           数据结构是否灵活多变?
              ├─ 是 → MongoDB（文档模型）
              │
              └─ 否 → 评估访问模式
                        ↓
                     访问模式特点?
                        ├─ 高频读写 → Redis（缓存层）+ PostgreSQL
                        ├─ 简单键值 → Redis/DynamoDB
                        ├─ 图关系强 → Neo4j（图数据库）
                        └─ 时序数据 → TimescaleDB/Cassandra
```

## 四、认证方案选型

### 4.1 JWT vs Session vs OAuth

认证是Web应用安全的基础，选择合适的认证方案对系统的安全性和用户体验都至关重要。

**Session认证机制：**

```typescript
// Session认证流程
// 1. 用户登录成功，服务端创建Session
// 2. Session ID存储在Cookie中
// 3. 后续请求自动携带Cookie
// 4. 服务端通过Session ID验证用户身份

// Express + express-session 示例
import session from 'express-session';
import connectRedis from 'connect-redis';

const redisStore = new RedisStore({ client: redis });

app.use(session({
  store: redisStore,  // Session存储在Redis
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
  cookie: {
    secure: process.env.NODE_ENV === 'production',
    httpOnly: true,      // 防止XSS攻击
    maxAge: 24 * 60 * 60 * 1000  // 24小时
  }
}));

app.post('/api/login', async (req, res) => {
  const { email, password } = req.body;

  // 验证用户凭证
  const user = await validateCredentials(email, password);
  if (!user) {
    return res.status(401).json({ error: '无效的凭证' });
  }

  // 创建Session
  req.session.userId = user.id;
  req.session.email = user.email;

  res.json({ message: '登录成功', user: { id: user.id, email: user.email } });
});

app.get('/api/profile', (req, res) => {
  // Session自动验证
  if (!req.session.userId) {
    return res.status(401).json({ error: '未登录' });
  }

  res.json({ userId: req.session.userId, email: req.session.email });
});
```

**Session认证优缺点分析：**

| 优势 | 说明 |
|------|------|
| **服务端可控** | Session存储在服务端，可以随时撤销 |
| **安全性高** | Session ID随机生成，难以预测 |
| **简单直接** | 认证逻辑清晰，易于理解和调试 |

| 劣势 | 说明 |
|------|------|
| **扩展困难** | 分布式环境下Session同步复杂 |
| **存储成本** | 服务端需要存储所有活跃Session |
| **延迟增加** | 每次请求都需要查询Session |

**JWT认证机制：**

```typescript
// JWT认证流程
// 1. 用户登录成功，服务端签发JWT
// 2. JWT存储在客户端（localStorage或Cookie）
// 3. 后续请求携带JWT
// 4. 服务端验证JWT签名（无需查询数据库）

// jsonwebtoken 示例
import jwt from 'jsonwebtoken';

const JWT_SECRET = process.env.JWT_SECRET;
const JWT_EXPIRES_IN = '7d';

// 登录签发Token
app.post('/api/login', async (req, res) => {
  const { email, password } = req.body;

  const user = await validateCredentials(email, password);
  if (!user) {
    return res.status(401).json({ error: '无效的凭证' });
  }

  // 签发JWT
  const token = jwt.sign(
    {
      sub: user.id,        // 用户ID
      email: user.email,    // 用户邮箱
      role: user.role       // 用户角色
    },
    JWT_SECRET,
    { expiresIn: JWT_EXPIRES_IN }
  );

  res.json({ token });
});

// JWT验证中间件
const authenticateJWT = (req, req, res, next) => {
  const authHeader = req.headers.authorization;

  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return res.status(401).json({ error: '缺少认证Token' });
  }

  const token = authHeader.split(' ')[1];

  try {
    // 验证JWT（无需数据库查询）
    const payload = jwt.verify(token, JWT_SECRET);
    req.user = payload;
    next();
  } catch (error) {
    if (error.name === 'TokenExpiredError') {
      return res.status(401).json({ error: 'Token已过期' });
    }
    return res.status(403).json({ error: '无效的Token' });
  }
};

// 受保护的路由
app.get('/api/profile', authenticateJWT, (req, res) => {
  res.json({
    userId: req.user.sub,
    email: req.user.email
  });
});
```

**JWT的刷新Token策略：**

```typescript
// Access Token + Refresh Token 双Token机制
// 解决JWT无法撤销的问题

interface TokenPair {
  accessToken: string;      // 短期Token（15分钟）
  refreshToken: string;     // 长期Token（7天）
  expiresIn: number;
}

// 签发双Token
app.post('/api/login', async (req, res) => {
  const user = await validateCredentials(req.body.email, req.body.password);

  const accessToken = jwt.sign(
    { sub: user.id, type: 'access' },
    JWT_SECRET,
    { expiresIn: '15m' }
  );

  const refreshToken = jwt.sign(
    { sub: user.id, type: 'refresh' },
    REFRESH_SECRET,  // 使用不同的密钥
    { expiresIn: '7d' }
  );

  // Refresh Token存储到数据库/Redis，用于撤销
  await storeRefreshToken(user.id, refreshToken);

  res.json({ accessToken, refreshToken });
});

// 刷新Access Token
app.post('/api/refresh', async (req, res) => {
  const { refreshToken } = req.body;

  try {
    const payload = jwt.verify(refreshToken, REFRESH_SECRET);

    // 验证Refresh Token未被撤销
    const isValid = await validateRefreshToken(payload.sub, refreshToken);
    if (!isValid) {
      return res.status(401).json({ error: 'Refresh Token已撤销' });
    }

    // 签发新的Access Token
    const newAccessToken = jwt.sign(
      { sub: payload.sub, type: 'access' },
      JWT_SECRET,
      { expiresIn: '15m' }
    );

    res.json({ accessToken: newAccessToken });
  } catch (error) {
    res.status(401).json({ error: '无效的Refresh Token' });
  }
});

// 登出时撤销Refresh Token
app.post('/api/logout', authenticateJWT, async (req, res) => {
  const { refreshToken } = req.body;
  await revokeRefreshToken(req.user.sub, refreshToken);
  res.json({ message: '已登出' });
});
```

**OAuth 2.0认证：**

```typescript
// OAuth 2.0 + OpenID Connect 实现第三方登录
// 以GitHub登录为例

import github from 'passport-github2';

// Passport配置
```

## 五、实时通信方案选型

### 5.1 WebSocket vs SSE vs Long Polling

实时通信是现代Web应用的重要需求，从简单的通知推送到大规模实时协作，都需要选择合适的实时通信技术。

**WebSocket通信：**

```typescript
// WebSocket服务端（Node.js + ws库）
import { WebSocketServer, WebSocket } from 'ws';

const wss = new WebSocketServer({ port: 8080 });

// 连接管理
const clients = new Map<string, WebSocket>();

wss.on('connection', (ws, req) => {
  const clientId = generateClientId();

  // 保存客户端连接
  clients.set(clientId, ws);

  console.log(`客户端连接: ${clientId}`);

  // 接收消息
  ws.on('message', (data) => {
    try {
      const message = JSON.parse(data.toString());
      handleMessage(clientId, message);
    } catch (error) {
      console.error('消息解析失败:', error);
    }
  });

  // 心跳检测
  ws.on('pong', () => {
    // 客户端存活
  });

  // 断开连接
  ws.on('close', () => {
    clients.delete(clientId);

  // 错误处理
  ws.on('error', (error) => {
    console.error('WebSocket错误:', error);

// 广播消息给所有客户端

  const data = JSON.stringify(message);
  clients.forEach((client) => {
    if (client.readyState === WebSocket.OPEN) {
      client.send(data);
    }
  });
}
```

```typescript
// Long Polling服务端
app.get('/api/poll', async (req, res) => {
  const { userId } = req.query;
  const timeout = 30000; // 30秒超时

  try {
    // 等待新消息或有新消息立即返回
    const messages = await pollMessages(userId, { timeout });

    res.json({
      success: true,
      messages,
      timestamp: Date.now()

    res.status(500).json({ error: '轮询失败' });

// 轮询中间件（带退避策略）
function createPollingHandler(pollFn) {
  let retryCount = 0;
  const maxRetries = 10;
  const baseDelay = 1000;

  return async function poll(req, res) {
    try {
      const result = await pollFn(req);
      retryCount = 0; // 重置重试计数
      res.json(result);
    } catch (error) {
      retryCount++;

      if (retryCount > maxRetries) {
        return res.status(503).json({
          error: '服务暂时不可用',
          retryAfter: 60
        });
      }

      // 指数退避
      const delay = baseDelay * Math.pow(2, retryCount);
      setTimeout(() => {
        poll(req, res);
      }, delay);
    }
  };
}
```

```typescript
// Long Polling客户端
class LongPollingClient {
  private userId: string;
  private isPolling = false;
  private retryDelay = 1000;

  constructor(userId: string) {
    this.userId = userId;
  }

  async start() {
    this.isPolling = true;
    this.poll();
  }

  stop() {
    this.isPolling = false;
  }

  private async poll() {
    if (!this.isPolling) return;

    try {
      const response = await fetch(`/api/poll?userId=${this.userId}`);
      const data = await response.json();

      if (data.messages && data.messages.length > 0) {
        this.handleMessages(data.messages);
      }

      // 成功接收后立即继续轮询
      this.retryDelay = 1000;
      this.poll();
    } catch (error) {
      console.error('轮询错误:', error);

      // 指数退避
      await new Promise(resolve => setTimeout(resolve, this.retryDelay));
      this.retryDelay = Math.min(this.retryDelay * 2, 30000);

      if (this.isPolling) {
        this.poll();
      }
    }
  }

  private handleMessages(messages: any[]) {
    messages.forEach(msg => {
      console.log('收到消息:', msg);
    });
  }
}
```

```typescript
// 1. 分布式锁
import Redis from 'ioredis';

class DistributedLock {
  constructor(private redis: Redis) {}

  async acquire(key: string, ttl: number = 30000): Promise<string | null> {
    const lockValue = generateUUID();

    // SET key value NX PX ttl
    const result = await this.redis.set(
      `lock:${key}`,
      lockValue,
      'PX',
      ttl,
      'NX'
    );

    return result === 'OK' ? lockValue : null;
  }

  async release(key: string, value: string): Promise<boolean> {
    // Lua脚本保证原子性
    const script = `
      if redis.call("get", KEYS[1]) == ARGV[1] then
        return redis.call("del", KEYS[1])
      else
        return 0
      end
    `;

    const result = await this.redis.eval(script, 1, `lock:${key}`, value);
    return result === 1;
  }
}

  try {
    // 处理订单逻辑
    await doProcessOrder(orderId);
  } finally {
    await lock.release(`order:${orderId}`, lockValue);

}
```

```typescript
// 2. 延迟队列（用于订单超时取消）
import { Redis } from 'ioredis';

class DelayQueue {
  constructor(private redis: Redis) {}

  // 添加延迟任务
  async add(taskId: string, delayMs: number, payload: any) {
    const executeAt = Date.now() + delayMs;
    const score = executeAt;

    // 添加到有序集合
    await this.redis.zadd('delay:queue', score, JSON.stringify({
      id: taskId,
      payload,
      createdAt: Date.now()
    }));
  }

  // 消费延迟任务
  async consume(callback: (task: any) => Promise<void>) {
    while (true) {
      // 获取已到期的任务
      const tasks = await this.redis.zrangebyscore(
        'delay:queue',
        0,
        Date.now(),
        'LIMIT',
        0,
        10
      );

      for (const taskStr of tasks) {
        const task = JSON.parse(taskStr);

        try {
          await callback(task.payload);
          // 执行成功，从队列中移除
          await this.redis.zrem('delay:queue', taskStr);
        } catch (error) {
          console.error('处理延迟任务失败:', error);
          // 可选：重试或移到死信队列
        }
      }

      // 等待一段时间再检查
      await new Promise(resolve => setTimeout(resolve, 1000));
    }
  }
}

// 使用延迟队列处理订单超时
```

```typescript
// 3. 缓存预热与自动刷新
class CacheManager {
  constructor(private redis: Redis) {}

  // 缓存数据（带过期时间）
  async cache(key: string, data: any, ttlSeconds: number) {
    await this.redis.setex(`cache:${key}`, ttlSeconds, JSON.stringify(data));
  }

  // 获取缓存（穿透保护）
  async getOrFetch<T>(
    key: string,
    fetcher: () => Promise<T>,
    ttlSeconds: number = 300
  ): Promise<T> {
    const cached = await this.redis.get(`cache:${key}`);

    if (cached) {
      return JSON.parse(cached);
    }

    // 缓存未命中，执行查询
    const data = await fetcher();

    // 异步更新缓存（不影响返回速度）
    this.cache(key, data, ttlSeconds);

    return data;
  }

  // 缓存预热
  async warmup(keys: string[], loader: (key: string) => Promise<any>) {
    const pipeline = this.redis.pipeline();

    for (const key of keys) {
      const data = await loader(key);
      pipeline.setex(`cache:${key}`, 3600, JSON.stringify(data));
    }

    await pipeline.exec();
    console.log(`缓存预热完成: ${keys.length}个键`);
  }

  // 缓存自动刷新（快过期时异步更新）
  async getOrFetchWithRefresh<T>(
    key: string,
    fetcher: () => Promise<T>,
    ttlSeconds: number = 300
  ): Promise<T> {
    const cached = await this.redis.get(`cache:${key}`);

    if (cached) {
      const data = JSON.parse(cached);

      // 检查是否快过期（TTL < 20%）
      const ttl = await this.redis.ttl(`cache:${key}`);
      if (ttl < ttlSeconds * 0.2) {
        // 异步刷新，不阻塞
        fetcher().then(freshData => {
          this.cache(key, freshData, ttlSeconds);
        }).catch(console.error);
      }

      return data;
    }

    // 首次加载
    const data = await fetcher();
    this.cache(key, data, ttlSeconds);
    return data;
  }
}
```

```typescript
  // 静态资源：长期缓存
  if (url.match(/\.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2)$/)) {
    res.setHeader('Cache-Control', 'public, max-age=31536000, immutable');
    res.setHeader('ETag', generateHash(fileContent));
  }

  // API响应：不缓存
  else if (url.startsWith('/api/')) {
    res.setHeader('Cache-Control', 'no-store, no-cache, must-revalidate');
    res.setHeader('Pragma', 'no-cache');
  }

  // 页面：短期缓存
  else if (!url.includes('.')) {
    res.setHeader('Cache-Control', 'private, max-age=300');
  }

  // 处理ETag验证
  if (req.fresh) {
    res.status(304).end();
    return;
  }

  next();
});
```

### 6.2 缓存方案选型清单

| 场景 | 推荐方案 | 说明 |
|------|----------|------|
| **简单KV缓存** | Memcached | 高性能、内存效率高 |
| **复杂数据结构** | Redis | 支持多种数据类型 |
| **Session存储** | Redis | 跨服务器共享、支持过期 |
| **实时排行** | Redis Sorted Set | 天然支持排行榜 |

| **静态资源** | CDN + 浏览器缓存 | 减轻源站压力 |

```typescript
import express, { Express } from 'express';
import { Server } from 'http';
import { Handler, Context, APIGatewayEvent } from 'aws-lambda';

const app: Express = express();

// 路由
app.get('/api/health', (req, res) => {
  res.json({ status: 'ok', timestamp: Date.now() });
});

app.get('/api/users/:id', async (req, res) => {
  const user = await getUserById(req.params.id);
  res.json(user);
});

// Serverless适配
let server: Server;

function setup(): Server {
  server = createServer(app);
  return server;
}

export const handler: Handler = async (
  event: APIGatewayEvent,
  context: Context
) => {
  if (!server) {
    setup();
  }

  return proxy(server, event, context, 'PROMISE').promise;
};
```

**水平扩展 vs 垂直扩展：**

```typescript
// 读写分离示例
// 配置主从数据库
const dbConfig = {
  master: {
    host: 'master.db.example.com',
    port: 5432
  },
  slaves: [
    { host: 'slave1.db.example.com', port: 5432 },
    { host: 'slave2.db.example.com', port: 5432 }
  ]
};

// 自动判断读写操作
class ReadWriteRouter {
  async query(sql: string, params?: any[]) {
    if (this.isReadQuery(sql)) {
      // 读操作：路由到从库
      const slave = this.selectSlave();
      return slave.query(sql, params);
    } else {
      // 写操作：路由到主库
      return this.master.query(sql, params);
    }
  }

  private isReadQuery(sql: string): boolean {
    const readKeywords = ['SELECT', 'SHOW', 'DESCRIBE'];
    return readKeywords.some(k => sql.toUpperCase().includes(k));
  }

  private selectSlave(): Database {
    // 负载均衡选择从库
    const index = Math.floor(Math.random() * this.slaves.length);
    return this.slaves[index];
  }
}
```

## 十一、总结

技术选型是一项需要综合权衡的决策，没有放之四海而皆准的最佳方案。本指南从架构模式、数据库、认证方案、实时通信、缓存策略和部署方案六个维度进行了详细分析，并针对不同规模的项目给出了具体的技术组合建议。

**核心选型原则回顾：**

1. **合适优先**：根据团队能力和项目需求选择，而非盲目追求新技术
2. **渐进演进**：从简单方案开始，逐步引入复杂方案
3. **成本意识**：综合考虑开发成本、运维成本和扩展成本
4. **风险可控**：避免单一技术依赖，保持架构灵活性
5. **文档先行**：选型决策要有据可查，便于后续维护
