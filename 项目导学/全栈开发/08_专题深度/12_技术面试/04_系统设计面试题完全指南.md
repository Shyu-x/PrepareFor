# 系统设计面试题完全指南

## 概述

系统设计面试是全栈工程师和后端开发岗位面试中最具挑战性的环节之一。它不仅考察候选人对计算机科学基础的理解，更考验其在真实场景下分析问题、权衡利弊、设计解决方案的综合能力。与算法题不同，系统设计题没有标准答案，面试官更关注候选人的思考过程、沟通能力和工程判断力。

本指南将系统性地讲解系统设计的核心方法论，并提供10+道经典系统设计题目的完整分析。每道题目都包含需求理解、数据模型设计、API设计、扩展性考虑、容量规划等维度，帮助读者在面试中游刃有余，在实际工作中也能设计出高质量的系统。

## 第一部分：系统设计方法论

### 1.1 系统设计的核心原则

系统设计本质上是在各种约束条件下寻找最优解的过程。优秀的系统设计需要兼顾功能需求、非功能需求（性能、可扩展性、可用性、安全性）、成本效益等多个维度。以下是几个核心原则：

**简约设计原则**：系统设计应避免过度工程化。在面试和实际工作中，首先要确保能够满足当前需求，而不是预设未来可能的需求。YAGNI（You Aren't Gonna Need It）原则提醒我们，只实现当下需要的功能，避免为可能永远不会到来的需求预先设计。过度设计不仅增加开发成本，还可能引入不必要的复杂性，导致系统难以维护。

**可扩展性优先**：优秀的系统应该在不改变整体架构的情况下能够水平扩展。这意味着需要将状态 Stateless 化、设计无状态的计算层、使用分布式存储等。在设计之初就要考虑数据量和请求量的增长趋势，确保系统能够通过增加节点来应对流量增长。

**一致性 vs 可用性权衡**：分布式系统中的 CAP 定理告诉我们，一致性和可用性往往不可兼得。根据业务场景的不同，需要在这两者之间做出合理的权衡。对于金融交易系统，一致性是首要考虑；对于社交媒体的 feed 流，可用性可能更重要。理解业务场景是做出正确权衡的前提。

### 1.2 如何理解需求

需求理解是系统设计的第一步，也是最关键的一步。很多面试者急于给出解决方案，却忽略了与面试官确认需求，这往往会导致设计偏离方向。需求理解包括以下几个层面：

**功能需求澄清**：首先需要明确系统需要支持哪些核心功能。以设计一个短链接系统为例，需要确认：短链接的最大长度是多少？是否需要支持自定义短链接？链接是否需要设置过期时间？是否需要统计点击量？这些细节问题直接影响数据模型和存储设计。与面试官沟通需求不是怯懦，而是展现工程素养的重要方式。

**非功能需求量化**：除了功能，还需要明确性能指标。系统的 QPS（每秒查询数）是多少？延迟要求是多少毫秒？可用性需要达到几个 9？数据规模预计是多少？这些量化指标将指导后续的容量规划和架构设计。如果面试官没有给出具体数字，可以根据常识进行合理假设，并说明这些假设。

**约束条件识别**：了解系统的技术约束和资源限制同样重要。团队的技术栈是什么？可以使用哪些现成的服务？预算限制是多少？是否需要考虑合规要求？这些约束条件将影响架构选择的可行性。

### 1.3 数据模型设计

数据模型是系统设计的基础，它决定了数据如何存储、查询和更新。良好的数据模型应该能够高效支持系统的核心操作，同时保持一定的灵活性以应对需求变化。

**实体识别与关系建模**：首先识别系统中的核心实体及其关系。以电商系统为例，核心实体包括用户（User）、商品（Product）、订单（Order）、评论（Review）等。实体之间存在各种关系：用户与订单是一对多关系，订单与商品是多对多关系（通过订单项连接），用户与评论是一对多关系。使用 ER 图来可视化这些关系可以帮助理清思路。

**表结构设计**：在关系型数据库中，需要设计合适的表结构。关键考虑包括：选择合适的主键策略（自增 ID、UUID、业务 ID）；定义适当的索引以加速查询；考虑反范式化以优化读取性能；设计合适的字段类型以节省存储。例如，对于需要频繁查询的字段，应该创建索引；对于文本内容，可以限制长度或使用 TEXT 类型并考虑是否分库分表。

**扩展性考量**：数据量增长可能导致单库单表无法应对。需要在设计之初就考虑分库分表策略：是按用户 ID 分片还是按时间分片？分片键的选择如何影响查询模式？是否需要引入分布式数据库如 Cassandra 或 MongoDB？这些决策需要根据数据规模和查询模式来综合判断。

### 1.4 API 设计

API 是系统对外提供服务的接口，其设计质量直接影响开发者的使用体验和系统的前后端集成。RESTful API 是目前最流行的 API 设计风格，但在某些场景下，GraphQL 或 gRPC 可能更合适。

**资源导向设计**：RESTful API 应该围绕资源进行设计。使用名词而非动词来标识资源，如 `/users` 表示用户集合，`/users/{id}` 表示特定用户。HTTP 方法表示对资源的操作：GET 用于查询，POST 用于创建，PUT 用于完整更新，PATCH 用于部分更新，DELETE 用于删除。

**版本管理策略**：API 需要考虑版本演进。常见的版本管理策略包括 URL 路径版本（如 `/api/v1/users`）、请求头版本（如 `Accept: application/vnd.api.v1+json`）和查询参数版本（如 `/api/users?version=1`）。URL 路径版本最直观，也是最常用的策略。

**错误处理规范**：统一的错误处理规范对于良好的开发者体验至关重要。应该定义清晰的错误码体系，返回有意义的错误信息，包含错误追踪 ID 以便排查。典型的错误响应格式包括错误码、错误消息、详细信息和请求 ID。

### 1.5 扩展性设计

扩展性是系统应对流量增长和数据增长的能力。良好的扩展性设计应该允许通过增加资源来提升系统容量，而不需要重新设计系统架构。

**水平扩展 vs 垂直扩展**：水平扩展通过增加节点来提升系统容量，适合无状态服务；垂直扩展通过升级单机硬件来提升容量，受到硬件限制。对于无状态的服务如 Web 服务器，水平扩展是首选；对于有状态的服务如数据库，需要更仔细的规划。

**无状态设计**：将状态外置到分布式存储中（如 Redis、Memcached），服务本身不存储状态，可以无限制地水平扩展。Session 状态、配置信息、缓存键等都应该考虑外置。无状态设计不仅便于扩展，还简化了故障恢复和负载均衡的实现。

**服务拆分策略**：单体应用在初期开发效率高，但随着业务增长，可能成为瓶颈。根据业务边界和技术异构性，可以将系统拆分为多个微服务。拆分粒度需要权衡：太粗则退化为小单体，太细则增加运维复杂度。领域驱动设计（DDD）可以帮助识别合适的业务边界。

### 1.6 容量规划

容量规划是确保系统能够承载预期负载的重要环节。它包括估算数据量、计算资源需求、规划存储容量等工作。

**数据量估算**：根据业务场景估算数据总量和增长趋势。例如，一个日活 1000 万用户的社交应用，假设每个用户平均关注 200 人，发布 0.5 条动态，那么每天需要存储 500 万条动态，需要存储的关注关系约为 20 亿条。这些估算将指导数据库选型和存储规划。

**QPS 估算**：根据用户规模和操作频率估算系统的 QPS。假设日活 1000 万，活跃用户每秒产生 100 次请求，QPS 约为 1000。考虑峰值是平均的 10 倍，峰值 QPS 约为 10000。留有一定的余量，设计的系统应该能够应对 15000 QPS。

**资源规划**：根据 QPS 和延迟要求，计算需要的服务器数量。假设单台服务器能够处理 1000 QPS，考虑到 CPU 利用率和故障冗余，需要部署至少 20 台服务器。数据库连接数、缓存容量、带宽等资源都需要综合规划。

## 第二部分：经典系统设计题目

### 题目一：设计一个短链接系统

#### 题目描述

设计一个短链接服务，实现长 URL 到短 URL 的转换，并能够根据短 URL 跳转到原始 URL。例如，将 `https://www.example.com/products/category/item?id=12345` 转换为类似 `https://short.link/abc123` 的短链接。

#### 考察点

这道题考察候选人对以下方面的理解：

**编码算法**：如何将长 URL 唯一映射到短字符串。常见的算法包括 Base62 编码（使用 0-9、a-z、A-Z 共 62 个字符）和 MD5 哈希。Base62 的优点是长度固定，缺点是需要在存储中查找；MD5 的优点是无需存储，缺点是存在哈希碰撞风险。

**存储设计**：如何存储短链接与原始 URL 的映射关系。需要考虑查询效率、存储成本、数据持久化等因素。MySQL 适合存储关旭型数据，Redis 适合做缓存层以提升读取性能。

**重定向策略**：302 临时重定向适合统计点击场景，301 永久重定向适合追求性能的 SEO 场景。浏览器对 301 有缓存，可能影响统计准确性。

#### 思路分析

**需求澄清**：首先需要与面试官确认以下问题：短链接的长度限制是多少？是否需要支持自定义短链接？链接是否需要设置过期时间？是否需要统计点击量？是否需要密码保护？这些细节将影响数据模型设计。

**核心功能拆解**：

1. 短链接生成：将长 URL 转换为短字符串
2. 短链接存储：保存短字符串与原始 URL 的映射
3. 访问解析：根据短字符串查询原始 URL 并重定向
4. 点击统计（可选）：记录访问次数、时间、地理位置等信息
5. 链接管理（可选）：过期时间设置、自定义短链接、密码保护

**数据模型设计**：核心表结构包括链接表（存储短码、原始 URL、创建时间、过期时间、点击数等信息）和可选的点击日志表（存储每次访问的详细信息）。短码字段应建立唯一索引以加速查询。

```
链接表 (urls)
├── id: BIGINT 主键，自增
├── short_code: VARCHAR(20) 短码，唯一索引
├── original_url: TEXT 原始 URL
├── created_at: DATETIME 创建时间
├── expires_at: DATETIME 过期时间，可为空表示永不过期
├── click_count: INT 点击次数
├── is_password_protected: BOOLEAN 是否密码保护
└── password_hash: VARCHAR(255) 密码哈希，可为空
```

#### 架构设计

**整体架构**：系统采用分层设计，包括接入层、应用层和存储层。接入层处理 HTTP 请求，可以使用 Nginx 或 CDN；应用层实现业务逻辑，使用无状态的 Web 服务器，支持水平扩展；存储层包括 MySQL（持久化存储）和 Redis（缓存层）。

**短码生成策略**：采用分布式 ID 生成器（如 Snowflake 算法）生成分布式唯一 ID，然后转换为 Base62 编码。例如，Snowflake 生成的 64 位整数 ID，转换为 6 位 Base62 字符串（62^6 ≈ 568 亿种组合）。这种方法的优势是无需存储即可保证唯一性。

**读写分离优化**：读多写少的场景下，可以采用读写分离策略。主库处理写请求，多个从库处理读请求。也可以引入 Redis 缓存层，将热点数据缓存在内存中，进一步提升读取性能。

**高可用设计**：使用多副本部署避免单点故障。MySQL 采用主从复制，Redis 采用哨兵或集群模式。应用层部署多个无状态实例，前端使用负载均衡器分发请求。

#### 加分项

**自定义短链接**：允许用户指定有意义的短码，如 `example.com/myprofile`。需要校验唯一性，已被占用的需要拒绝或提示用户选择替代方案。

**链接过期与自动清理**：支持设置过期时间，后台任务定期清理过期数据。可以使用定时任务（如 cron）或消息队列来实现。

**点击数据分析**：记录每次点击的时间、IP、User-Agent、地理位置等信息，为运营分析提供数据支持。

**密码保护**：支持为短链接设置密码，访问时需要输入密码验证。

#### 可能的追问

**追问1：如何防止恶意刷接口？**

可以采用限流策略，使用 Redis 实现滑动窗口限流。每个 IP 或用户有固定的请求配额，超过配额返回 429 Too Many Requests。也可以引入验证码或人机验证机制。

**追问2：如何实现分布式环境下的 ID 生成？**

可以使用 Snowflake 算法（Twitter 开源），生成分布式唯一 ID。Snowflake 将 64 位分为三部分：时间戳（41 位）、机器 ID（10 位）、序列号（12 位）。也可以使用数据库自增 ID + 批量预取，或使用 Zookeeper、Kubernetes 等协调服务。

**追问3：如何实现高并发写入？**

写入压力主要来自短链接创建。可以采用批量写入策略，将多个写请求合并批量处理。也可以使用异步写入，将创建请求放入消息队列，后台 worker 异步处理。需要权衡一致性和性能。

### 题目二：设计一个实时聊天系统

#### 题目描述

设计一个即时通讯（IM）系统，支持一对一聊天、群组聊天、消息推送、已读未读等功能。例如，设计一个类似微信或 Slack 的聊天系统。

#### 考察点

这道题全面考察了候选人对以下技术的理解：

**WebSocket 实时通信**：相比 HTTP 的请求-响应模式，WebSocket 提供了全双工通信能力，适合实时性要求高的场景。需要处理连接管理、心跳检测、断线重连等问题。

**消息推送机制**：如何将消息实时推送给目标用户。涉及 WebSocket 路由、在线状态管理、离线消息存储等。

**消息存储与同步**：如何存储聊天记录，如何同步未读消息，如何实现消息漫游（多设备同步）。

**高并发与高可用**：IM 系统通常用户量大、消息量大，需要考虑水平扩展、负载均衡、容灾备份等。

#### 思路分析

**需求澄清**：需要确认系统是一对一的私聊还是包括群聊？消息量级预计是多少？是否需要支持消息加密？是否需要支持消息撤回、编辑？是否需要支持 Typing Indicator（正在输入提示）？这些细节影响架构设计。

**核心功能拆解**：

1. 用户认证与授权：用户登录、Token 验证
2. 连接管理：WebSocket 连接建立、维护、心跳
3. 消息收发：一对一消息、群组消息
4. 消息存储：消息持久化、消息历史查询
5. 未读同步：未读计数、已读回执、消息同步
6. 推送通知：离线消息推送（如 APNs、FCM）
7. 群组管理：创建群组、成员管理

#### 架构设计

**整体架构**：系统采用微服务架构，包括多个独立的服务模块。API 网关处理所有外部请求，负责鉴权和路由；WebSocket 服务处理长连接；消息服务处理消息的发送和存储；推送服务处理离线消息推送；用户服务管理用户关系和在线状态。

**消息流转流程**：用户 A 发送消息给用户 B 的流程如下：

1. 客户端 A 通过 WebSocket 发送消息到服务器
2. WebSocket 服务接收消息，验证用户身份
3. 消息服务将消息存储到数据库
4. 如果用户 B 在线，通过 WebSocket 实时推送消息给 B
5. 如果用户 B 离线，将消息加入离线推送队列
6. 返回发送成功响应给用户 A

**在线状态管理**：使用 Redis 存储用户在线状态和连接的 WebSocket 服务器。Key 为 `user:{userId}:online`，Value 为服务器节点标识。当用户建立 WebSocket 连接时设置此 Key，断开时删除。可以使用 Redis 的过期机制实现心跳检测。

```
用户在线状态存储结构：
user:12345:online -> "ws-server-1"  (字符串，存储连接的服务节点)
user:12345:channels -> ["channel_1", "channel_2"]  (Set，用户加入的频道)
```

**消息存储设计**：消息表设计需要考虑查询模式。消息主要按会话查询，因此需要建立 `conversation_id` 和 `created_at` 的复合索引。

```
消息表 (messages)
├── id: BIGINT 主键
├── conversation_id: VARCHAR(64) 会话 ID（如 user1_user2）
├── sender_id: BIGINT 发送者用户 ID
├── receiver_id: BIGINT 接收者用户 ID（私聊时）
├── group_id: BIGINT 群组 ID（群聊时）
├── message_type: TINYINT 消息类型（文本、图片、文件等）
```

```javascript
// WebSocket 服务伪代码
class ChatServer {
    constructor() {
        // 存储用户连接映射
        // key: userId, value: WebSocket connection
        this.userConnections = new Map();

        if (!user) {

        await this.subscribeUserChannel(userId);

            userId: userId,

        }));

        // 6. 发送未读消息同步
        await this.syncUnreadMessages(userId, socket);
    }

    // 处理心跳检测
    async handlePing(socket, userId) {
        // 更新 Redis 中的心跳时间
        await this.redis.set(`user:${userId}:heartbeat`, Date.now());
        socket.send(JSON.stringify({ type: 'pong' }));
    }

    // 处理断连
    async handleDisconnect(socket, userId) {
        // 1. 清理连接映射
        this.userConnections.delete(userId);
        this.nodeUsers.delete(userId);

        // 2. 标记用户离线
        await this.redis.del(`user:${userId}:online`);

        // 3. 取消订阅消息频道
        await this.unsubscribeUserChannel(userId);
    }
```

**消息发送与路由**：

```javascript
// 消息发送服务伪代码
class MessageService {
    // 发送私聊消息
    async sendMessage(senderId, receiverId, content, messageType) {
        // 1. 验证发送者权限
        const sender = await this.userService.getUser(senderId);
        if (!sender) throw new Error('Sender not found');

        // 2. 验证接收者存在
        const receiver = await this.userService.getUser(receiverId);
        if (!receiver) throw new Error('Receiver not found');

        // 3. 生成会话 ID（确保相同的两用户对话有唯一的会话 ID）
        const conversationId = this.generateConversationId(senderId, receiverId);

        // 4. 存储消息
        const message = await this.messageRepository.create({
            conversationId,
            senderId,
            receiverId,
            content,
            messageType,
            status: 'sent'
        });

        // 5. 更新发送者最新消息缓存
        await this.updateLatestMessage(senderId, conversationId, message);

        // 6. 推送消息给接收者
        await this.pushMessage(receiverId, message);

        // 7. 如果接收者离线，存储离线消息
        if (!await this.isUserOnline(receiverId)) {
            await this.storeOfflineMessage(receiverId, message);
        }

        return message;
    }

    // 推送消息给在线用户
    async pushMessage(userId, message) {
        // 1. 获取用户当前连接的节点
        const nodeId = await this.redis.get(`user:${userId}:online`);
        if (!nodeId) return; // 用户离线

        // 2. 如果是本节点用户，直接发送
        if (nodeId === this.nodeId) {
            const socket = this.userConnections.get(userId);
            if (socket) {
                socket.send(JSON.stringify({
                    type: 'message',
                    data: message
                }));
            }
        } else {
            // 3. 跨节点，通过 Redis Pub/Sub 转发到对应节点
            await this.redis.publish(`node:${nodeId}:message`, JSON.stringify({
                userId,
                message
            }));
        }
    }

    // 生成会话 ID
    generateConversationId(userId1, userId2) {
        // 确保相同用户对的会话 ID 唯一且确定
        const sorted = [userId1, userId2].sort();
        return `${sorted[0]}_${sorted[1]}`;
    }
}
```

#### 可能的追问

**追问1：如何保证消息的可靠送达？**

```javascript
// 分片上传服务伪代码
class ChunkUploadService {
    // 1. 初始化分片上传
    async initUpload(userId, fileName, fileSize, fileHash) {
        // 检查是否已存在相同的文件（去重）
        const existing = await this.fileService.findByHash(fileHash);
        if (existing) {
            return {
                duplicated: true,
                fileId: existing.id,
                fileUrl: existing.url
            };
        }

        // 计算分片数量（每个分片 5MB）
        const chunkSize = 5 * 1024 * 1024;
        const totalChunks = Math.ceil(fileSize / chunkSize);

        // 创建上传任务
        const uploadTask = await this.uploadTaskRepository.create({
            userId,
            fileName,
            fileSize,
            fileHash,
            status: 'initializing',
            totalChunks,
            uploadedChunks: 0
        });

        return {
            duplicated: false,
            uploadId: uploadTask.uploadId,
            chunkSize,
            totalChunks
        };
    }

    // 2. 上传单个分片
    async uploadChunk(uploadId, chunkIndex, chunkData, chunkHash) {
        // 验证上传任务存在且未完成
        const task = await this.uploadTaskRepository.findOne({ uploadId });
        if (!task || task.status === 'completed') {
            throw new Error('Upload task not found or completed');
        }

        // 验证分片序号有效
        if (chunkIndex < 0 || chunkIndex >= task.totalChunks) {
            throw new Error('Invalid chunk index');
        }

        // 存储分片到临时目录
        const chunkPath = `temp/${uploadId}/${chunkIndex}`;
        await this.storage.write(chunkPath, chunkData);

        // 记录分片上传完成
        await this.chunkRepository.upsert({
            uploadId,
            chunkIndex,
            chunkSize: chunkData.length,
            chunkHash,
            storagePath: chunkPath,
            status: 'completed'
        });

        // 更新任务已上传分片数
        task.uploadedChunks = await this.chunkRepository.count({
            uploadId,
            status: 'completed'
        });
        await task.save();

        return {
            chunkIndex,
            uploadedChunks: task.uploadedChunks,
            totalChunks: task.totalChunks
        };
    }

    // 3. 完成分片上传
    async completeUpload(uploadId) {
        const task = await this.uploadTaskRepository.findOne({ uploadId });
        if (!task) throw new Error('Upload task not found');

        // 验证所有分片都已上传
        const completedChunks = await this.chunkRepository.count({
            uploadId,
            status: 'completed'
        });

        if (completedChunks !== task.totalChunks) {
            throw new Error(`Missing chunks: ${task.totalChunks - completedChunks} chunks not uploaded`);
        }

        // 合并分片
        const finalPath = `files/${task.userId}/${task.fileHash}/${task.fileName}`;
        await this.mergeChunks(task, finalPath);

        // 创建文件记录
        const file = await this.fileRepository.create({
            userId: task.userId,
            fileName: task.fileName,
            fileSize: task.fileSize,
            fileHash: task.fileHash,
            storagePath: finalPath,
            mimeType: this.getMimeType(task.fileName),
            status: 'active'
        });

        // 清理临时分片文件
        await this.cleanupChunks(uploadId);

        // 更新任务状态
        task.status = 'completed';
        task.storagePath = finalPath;
        await task.save();

        return {
            fileId: file.id,
            fileUrl: this.generateFileUrl(file)
        };
    }

    // 4. 查询上传进度（用于断点续传）
    async getUploadProgress(uploadId) {
        const task = await this.uploadTaskRepository.findOne({ uploadId });
        if (!task) throw new Error('Upload task not found');

        const completedChunks = await this.chunkRepository.findAll({
            uploadId,
            status: 'completed'
        });

        return {
            uploadId,
            status: task.status,
            uploadedChunks: completedChunks.map(c => c.chunkIndex),
            totalChunks: task.totalChunks,
            percentage: Math.round(completedChunks.length / task.totalChunks * 100)
        };
    }
}
```

**断点续传实现**：客户端记录已上传的分片列表，上传前先查询服务端的已上传分片，然后只上传缺失的分片。服务端通过 uploadId 识别上传任务，返回已完成的分片列表。

```javascript
// 客户端断点续传伪代码
class UploadClient {
    async uploadFile(file) {
        // 1. 计算文件哈希
        const fileHash = await this.calculateFileHash(file);

        // 2. 初始化上传，获取 uploadId 和已上传分片
        const initResult = await this.api.initUpload({
            fileName: file.name,
            fileSize: file.size,
            fileHash
        });

        // 如果是重复文件，直接返回
        if (initResult.duplicated) {
            return initResult.fileUrl;
        }

        const { uploadId, chunkSize, totalChunks } = initResult;
        const uploadedChunks = initResult.uploadedChunks || [];

        // 3. 查询服务端已上传的分片
        const progress = await this.api.getUploadProgress(uploadId);
        const completedChunks = new Set(progress.uploadedChunks);

        // 4. 上传缺失的分片
        for (let i = 0; i < totalChunks; i++) {
            if (completedChunks.has(i)) {
                console.log(`Chunk ${i} already uploaded, skipping`);
                continue;
            }

            // 读取分片数据
            const start = i * chunkSize;
            const end = Math.min(start + chunkSize, file.size);
            const chunk = file.slice(start, end);

            // 计算分片哈希
            const chunkHash = await this.calculateChunkHash(chunk);

            // 上传分片，带重试机制
            await this.uploadWithRetry(() =>
                this.api.uploadChunk(uploadId, i, chunk, chunkHash)
            );

            // 更新本地已上传记录
            this.saveProgress(uploadId, i);
        }

        // 5. 完成上传
        const result = await this.api.completeUpload(uploadId);
        return result.fileUrl;
    }

    // 带重试的上传
    async uploadWithRetry(fn, maxRetries = 3) {
        for (let i = 0; i < maxRetries; i++) {
            try {
                return await fn();
            } catch (error) {
                if (i === maxRetries - 1) throw error;
                // 指数退避等待
                await this.sleep(Math.pow(2, i) * 1000);
            }
        }
    }
}
```

#### 加分项

**文件去重**：通过文件哈希（如 MD5、SHA256）检测重复文件，已存在的文件直接返回链接，避免重复上传浪费带宽和存储。

**秒传功能**：用户上传前先计算文件哈希，查询服务端是否已存在。已存在则直接返回 URL，实现"秒传"。

**图片处理**：支持上传后自动生成缩略图、转换格式、水印添加等。可使用 ImageMagick 或云服务的图片处理功能。

**视频转码**：上传视频后异步进行转码，生成多种分辨率的版本，支持自适应码率流媒体播放。

**异步处理**：大文件的后处理（合并、转码、截图等）通过消息队列异步执行，不阻塞上传响应。

#### 可能的追问

**追问1：如何保证分片上传的完整性？**

每个分片上传时可以计算并传递分片哈希，服务端收到后验证哈希是否匹配。所有分片上传完成后，还可以对整个文件计算哈希进行最终校验，确保传输过程中没有损坏。

**追问2：如何处理并发上传同一个文件？**

可以通过文件哈希作为去重的依据。如果检测到相同的文件正在上传或已存在，可以直接返回已有文件的引用，避免重复存储。这需要在上传前就计算哈希并查询。

**追问3：分布式存储下如何保证数据可靠性？**

分布式存储系统（如 HDFS、Ceph、GlusterFS）通过数据冗余来保证可靠性。每个文件存储多个副本，分布在不同的节点和机架上。当某个节点故障时，可以从其他副本恢复数据。一般采用三副本策略，存储成本是实际大小的 3 倍。

```
商品索引表 (product_index)
```

```python
class SuggestionService:
    def __init__(self, redis_client):
        self.redis = redis_client

    def get_suggestions(self, prefix, limit=10):
        """获取搜索建议"""
        suggestions = []

        # 1. 前缀匹配建议
        prefix_suggestions = self.get_prefix_suggestions(prefix, limit)
        suggestions.extend(prefix_suggestions)

        # 2. 热门搜索补充
        if len(suggestions) < limit:
            hot_searches = self.get_hot_searches(limit - len(suggestions))
            suggestions.extend(hot_searches)

        # 3. 纠错建议（当用户输入有拼写错误时）
        corrections = self.get_corrections(prefix, limit)
        suggestions.extend(corrections)

        return suggestions[:limit]

    def get_prefix_suggestions(self, prefix, limit):
        """前缀匹配：使用 Trie 树或 Redis Sorted Set"""
        # 方案1：使用 Redis ZSet 存储前缀索引
        # prefix:keyword:prefixes 存储以该前缀开头的所有关键词
        # score 为搜索热度
        cache_key = f"suggest:prefix:{prefix.lower()}"
        cached = self.redis.get(cache_key)
        if cached:
            return json.loads(cached)

        # 从索引中查找前缀匹配
        keywords = self.suggestion_index.prefix_search(prefix.lower())
        # 返回 top N
        result = keywords[:limit]

        # 缓存结果
        self.redis.setex(cache_key, 3600, json.dumps(result))

        return result

    def record_search(self, keyword):
        """记录搜索词，用于生成热门搜索和建议"""
        # 更新搜索词计数
        key = "search:keyword:count"
        self.redis.zincrby(key, 1, keyword.lower())

        # 更新用户的搜索历史
        # 使用 Redis List 存储用户最近搜索
        user_history_key = "search:user:history:{userId}"
        self.redis.lpush(user_history_key, keyword)
        self.redis.ltrim(user_history_key, 0, 19)  # 只保留最近 20 条
        self.redis.expire(user_history_key, 30 * 24 * 3600)  # 30 天过期

    def get_hot_searches(self, limit):
        """获取热门搜索词"""
        key = "search:keyword:count"
        # 返回热度最高的前 N 个词
        hot_keywords = self.redis.zrevrange(key, 0, limit - 1)
        return hot_keywords
```

```python
# 通知服务伪代码
class NotificationService:
    def __init__(self, mq_client, template_repo, preference_service):
        self.mq = mq_client
        self.template_repo = template_repo
        self.preference_service = preference_service

    async def send_notification(self, user_id, template_code, channel, variables, payload=None):
        """发送通知的主入口"""
        # 1. 获取用户偏好
        preferences = await self.preference_service.get_user_preferences(user_id)

        # 2. 检查用户是否开启了该渠道
        if not preferences.is_channel_enabled(channel):
            return {"status": "skipped", "reason": "channel disabled by user"}

        # 3. 检查是否在免打扰时段
        if preferences.is_in_quiet_hours():
            # 延迟到免打扰结束再发送
            scheduled_at = preferences.quiet_hours_end
        else:
            scheduled_at = datetime.now()

        # 4. 渲染模板
        template = await self.template_repo.find_by_code(template_code)
        rendered = self.render_template(template, variables)

        # 5. 创建通知任务
        task = await self.notification_repo.create({
            user_id,
            channel,
            template_code,
            title=rendered['title'],
            content=rendered['content'],
            payload,
            status='pending',
            scheduled_at
        })

        # 6. 写入消息队列
        await self.mq.publish('notification:send', {
            'task_id': task.task_id,
            'user_id': user_id,
            'channel': channel,
            'title': rendered['title'],
            'content': rendered['content'],
            'payload': payload
        })

        return {"status": "queued", "task_id": task.task_id}

    def render_template(self, template, variables):
        """渲染通知模板"""
        title = template.title_template
        content = template.content_template

        # 简单的模板变量替换
        # 实际生产中应使用专业的模板引擎如 Jinja2
        for key, value in variables.items():
            placeholder = f"{{{key}}}"
            title = title.replace(placeholder, str(value))
            content = content.replace(placeholder, str(value))

        return {"title": title, "content": content}
```

**消息队列消费与重试**：

```python
# 通知 Worker 伪代码
class NotificationWorker:
    def __init__(self, mq_client, channel_providers):
        self.mq = mq_client
        self.providers = channel_providers  # 各渠道的发送provider
        self.max_retries = 3

    async def start(self):
        """启动 Worker"""
        await self.mq.consume('notification:send', self.handle_message)

    async def handle_message(self, message):
        """处理消息"""
        task_id = message['task_id']

        try:
            # 获取任务详情
            task = await self.notification_repo.find_by_task_id(task_id)
            if not task:
                return  # 任务不存在，丢弃

            # 调用对应渠道发送
            provider = self.providers[task.channel]
            result = await provider.send(
                user_id=task.user_id,
                title=task.title,
                content=task.content,
                payload=task.payload
            )

            if result.success:
                # 更新任务状态为已送达
                await self.notification_repo.update_status(task_id, 'delivered')
            else:
                # 处理发送失败
                await self.handle_failure(task, result.error)

        except Exception as e:
            # 捕获异常，记录错误
            await self.handle_failure(task, str(e))

    async def handle_failure(self, task, error):
        """处理发送失败"""
        task.retry_count += 1

        if task.retry_count < self.max_retries:
            # 指数退避，重新入队
            delay = 2 ** task.retry_count  # 2s, 4s, 8s
            await asyncio.sleep(delay)

            # 重新发布到队列
            await self.mq.publish_with_delay(
                'notification:send',
                message,
                delay_ms=delay * 1000
            )

            await self.notification_repo.update_status(
                task.task_id,
                'pending',
                retry_count=task.retry_count
            )
        else:
            # 超过最大重试次数，标记为失败
            await self.notification_repo.update_status(
                task.task_id,
                'failed',
                error_message=error
            )

            # 写入死亡队列，供人工处理
            await self.mq.publish('notification:dlq', {
                'task_id': task.task_id,
                'error': error,
                'retry_count': task.retry_count,
                'failed_at': datetime.now().isoformat()
            })
```

**核心功能拆解**：

2. 关注关系：用户关注、取消关注
3. Feed 分发：将新内容推送给关注者

6. 内容排序：按时间或相关性排序

#### 架构设计

**Feed 推拉模式对比**：

**Pull 模式（读扩散）**：用户获取 Feed 时，从所有关注者的内容库中拉取并合并。优点是存储成本低（新内容只存储一份），缺点是读取成本高（每次 Feed 访问都需要合并大量数据）。

**Push 模式（写扩散）**：用户发布内容时，主动推送到所有关注者的 Feed 缓存。优点是读取速度快（直接读取缓存），缺点是存储成本高（每条内容复制多份），且对于大 V 用户（粉丝数百万），推送会消耗大量资源。

**混合模式**：对于普通用户采用 Push 模式，对于大 V 用户采用 Pull 模式。这样可以在大多数场景下获得快速的读取速度，同时避免对大 V 的过度推送。

```python
        start = (page - 1) * page_size
        end = start + page_size - 1

        # 获取指定范围的 post_id
        post_ids = await self.feed_cache.zrevrange(
            cache_key, start, end
        )

        if not post_ids:
            # 缓存为空，可能是冷启动，执行 Pull 模式补全
            post_ids = await self.pull_feed_from_followings(user_id, page, page_size)
            # 写入缓存
            if post_ids:
                await self.cache_user_feed(user_id, post_ids)

        # 2. 根据 post_id 批量获取内容详情
        posts = await self.post_repo.get_by_ids(post_ids)

        # 3. 按 Feed 中的顺序返回（保持时间线顺序）
        posts_map = {p.id: p for p in posts}
        ordered_posts = [posts_map[pid] for pid in post_ids if pid in posts_map]

        return ordered_posts

    async def pull_feed_from_followings(self, user_id, page, page_size):
        """Pull 模式：从关注者的内容库拉取"""
        # 1. 获取用户关注列表
        following_ids = await self.follow_repo.get_following_ids(user_id)

        if not following_ids:
            return []

        # 2. 从每个关注者的内容中拉取最新内容
        all_posts = []
        for following_id in following_ids:
            posts = await self.post_repo.get_user_posts(
                following_id,
                limit=page_size * 2  # 多拉取一些用于过滤
            )
            all_posts.extend(posts)

        # 3. 按时间排序并分页
        all_posts.sort(key=lambda p: p.created_at, reverse=True)
        return [p.id for p in all_posts[:page_size]]
```

```python
# 写扩散服务伪代码
class FanoutService:
    # 大V阈值：粉丝数超过此值采用 Pull 模式
    BIG_FAN_THRESHOLD = 100000

    def __init__(self, follow_repo, feed_cache, redis_client):
        self.follow_repo = follow_repo
        self.feed_cache = feed_cache
        self.redis = redis_client

    async def publish_to_followers(self, post_id, user_id, created_at):
        """将新内容推送给关注者"""
        # 1. 获取粉丝列表
        follower_ids = await self.follow_repo.get_follower_ids(user_id)
        fan_count = len(follower_ids)

        # 2. 根据粉丝数量选择策略
        if fan_count > self.BIG_FAN_THRESHOLD:
            # 大V：只写入自己的 Feed，使用 Pull 模式
            await self.feed_cache.zadd(
                f"feed:{user_id}",
                {post_id: created_at.timestamp()}
            )
        else:
            # 普通用户：写扩散，推送给所有粉丝
            # 使用 Redis Pipeline 批量写入，提升性能
            pipe = self.redis.pipeline()

            for follower_id in follower_ids:
                cache_key = f"feed:{follower_id}"
                pipe.zadd(cache_key, {post_id: created_at.timestamp()})
                # 设置缓存过期时间（如 7 天）
                pipe.expire(cache_key, 7 * 24 * 3600)

            await pipe.execute()

    async def follow_user(self, follower_id, following_id):
        """处理关注操作：补全新粉丝的 Feed"""
        # 1. 记录关注关系
        await self.follow_repo.create(follower_id, following_id)

        # 2. 将被关注者最近的内容写入新粉丝的 Feed
        recent_posts = await self.post_repo.get_user_posts(
            following_id,
            limit=20  # 只补全最近 20 条
        )

        if recent_posts:
            for post in reversed(recent_posts):  # 保持时间顺序
                await self.feed_cache.zadd(
                    f"feed:{follower_id}",
                    {post.id: post.created_at.timestamp()}
                )

    async def unfollow_user(self, follower_id, following_id):
        """处理取消关注：从 Feed 中移除被关注者的内容"""
        # 1. 删除关注关系
        await self.follow_repo.delete(follower_id, following_id)

        # 2. 从 Feed 中移除被关注者的内容
        # 注意：这需要遍历删除，效率较低
        # 实际系统中，可以采用标记删除或异步清理策略
        post_ids = await self.post_repo.get_user_post_ids(following_id)

        if post_ids:
            await self.feed_cache.zrem(f"feed:{follower_id}", *post_ids)
```

### 题目七：设计一个分布式 ID 生成系统

#### 题目描述

设计一个分布式唯一 ID 生成器，能够在分布式环境下生成全局唯一、有序、可递增的唯一 ID，支持高并发调用。

#### 考察点

这道题考察候选人对以下技术的理解：

**分布式 ID 需求**：为什么数据库自增 ID 不够用？分布式 ID 需要满足哪些特性？

**Snowflake 算法**：时间戳 + 机器 ID + 序列号的组合原理和实现。

**时钟回拨处理**：分布式环境下时钟可能出现回拨，如何处理？

**高性能设计**：如何实现高并发下的低延迟 ID 生成。

#### 思路分析

**分布式 ID 的核心特性**：

**唯一性**：在分布式环境下全局唯一，不允许重复。这是基本要求。

**有序性**：ID 尽量保持递增或趋势递增。有序 ID 对数据库索引友好（减少叶子节点分裂），便于排序和分页。

**可预测性**：ID 的格式是否可解析（如包含时间戳、机器 ID 等），还是完全随机。不同场景有不同需求。

**高性能**：ID 生成服务需要支持高并发、低延迟。一个系统的核心链路（如订单创建）依赖 ID 生成服务，可用性要求很高。

**常见的分布式 ID 方案**：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| UUID | 128 位随机数 | 本地生成、无依赖 | 无序、存储空间大 |
| 数据库自增 | 数据库 AUTO_INCREMENT | 简单、连续 | 单点、无法水平扩展 |
| Snowflake | 时间戳+机器+序列号 | 有序、高性能 | 依赖时钟 |
| 雪花ID变种 | 基于 Snowflake 优化 | 可定制 | 复杂度增加 |

#### 架构设计

**Snowflake 算法原理**：Snowflake 将 64 位整数分为四个部分：

```
| 符号位 (1 bit) | 时间戳 (41 bits) | 机器 ID (10 bits) | 序列号 (12 bits) |
|      0         |   0000000000      |    0000000000      |    000000000000  |
```

- 符号位：固定为 0，表示正数
- 时间戳：使用 41 位表示毫秒级时间戳，可以支持约 69 年的使用（2^41 - 1 毫秒）
- 机器 ID：使用 10 位表示机器 ID，最多支持 1024 个节点（2^10）
- 序列号：使用 12 位表示每毫秒内的序列号，每个节点每毫秒最多生成 4096 个 ID（2^12）

**Snowflake 算法实现**：

```python
# Snowflake ID 生成器实现
import time
import threading
from dataclasses import dataclass

# 时间纪元：2020-01-01 00:00:00 UTC
TWEPOCH = 1577836800000

# 各部分占用的位数
TIMESTAMP_BITS = 41
WORKER_ID_BITS = 10
SEQUENCE_BITS = 12

# 各部分的取值范围
MAX_WORKER_ID = -1 ^ (-1 << WORKER_ID_BITS)  # 1023
MAX_SEQUENCE = -1 ^ (-1 << SEQUENCE_BITS)    # 4095

# 左移位数
WORKER_ID_SHIFT = SEQUENCE_BITS
TIMESTAMP_SHIFT = SEQUENCE_BITS + WORKER_ID_BITS

@dataclass
class Snowflake:
    """Snowflake 分布式 ID 生成器"""
    worker_id: int  # 机器 ID (0-1023)
    sequence: int = 0  # 当前序列号
    last_timestamp: int = -1  # 上次时间戳
    lock: threading.Lock = None

    def __post_init__(self):
        if self.worker_id < 0 or self.worker_id > MAX_WORKER_ID:
            raise ValueError(f"worker_id 必须在 0-{MAX_WORKER_ID} 之间")
        self.lock = threading.Lock()

    def next_id(self) -> int:
        """生成下一个 ID"""
        with self.lock:
            timestamp = self._current_millis()

            # 处理时钟回拨
            if timestamp < self.last_timestamp:
                # 等待时钟追上
                timestamp = self._wait_until_millis(self.last_timestamp)

            # 处理同一毫秒内的请求
            if timestamp == self.last_timestamp:
                self.sequence = (self.sequence + 1) & MAX_SEQUENCE
                if self.sequence == 0:
                    # 序列号用尽，等待下一毫秒
                    timestamp = self._wait_until_millis(timestamp + 1)
            else:
                self.sequence = 0

            self.last_timestamp = timestamp

            # 组装 ID
            return ((timestamp - TWEPOCH) << TIMESTAMP_SHIFT) | \
                   (self.worker_id << WORKER_ID_SHIFT) | \
                   self.sequence

    def _current_millis(self) -> int:
        """获取当前毫秒时间戳"""
        return int(time.time() * 1000)

    def _wait_until_millis(self, target_millis: int) -> int:
        """等待直到达到目标时间戳"""
        while True:
            current = self._current_millis()
            if current >= target_millis:
                return current
            # 让出 CPU，避免空转消耗资源
            time.sleep(0.001)

    @staticmethod
    def parse_id(id_value: int) -> dict:
        """解析 ID 的各部分"""
        timestamp = (id_value >> TIMESTAMP_SHIFT) + TWEPOCH
        worker_id = (id_value >> WORKER_ID_SHIFT) & MAX_WORKER_ID
        sequence = id_value & MAX_SEQUENCE
        return {
            'timestamp': timestamp,
            'worker_id': worker_id,
            'sequence': sequence
        }
```

**时钟回拨处理策略**：

```python
class SnowflakeWithClockFix(Snowflake):
    """带时钟回拨处理的 Snowflake 实现"""

    # 允许的最大时钟回拨时间（毫秒）
    MAX_CLOCK_BACKWARD_MS = 1000

    def __init__(self, worker_id, redis_client=None):
        super().__init__(worker_id)
        self.redis = redis_client

        # 如果有 Redis，记录上次生成的时间戳
        # 用于在服务重启时恢复
        if redis_client:
            self._restore_timestamp()

    def _restore_timestamp(self):
        """从持久化存储恢复时间戳"""
        key = f"snowflake:last_timestamp:{self.worker_id}"
        saved_timestamp = self.redis.get(key)
        if saved_timestamp:
            self.last_timestamp = int(saved_timestamp)

    def _save_timestamp(self):
        """持久化时间戳"""
        if self.redis:
            key = f"snowflake:last_timestamp:{self.worker_id}"
            self.redis.set(key, str(self.last_timestamp))

    def next_id(self) -> int:
        """生成下一个 ID（带时钟回拨处理）"""
        with self.lock:
            timestamp = self._current_millis()

            # 检测时钟回拨
            if timestamp < self.last_timestamp:
                # 计算回拨时间
                backoff = self.last_timestamp - timestamp

                if backoff > self.MAX_CLOCK_BACKWARD_MS:
                    # 回拨过大，可能是严重问题，记录日志并抛出异常
                    raise ClockBackwardsError(
                        f"时钟回拨超过阈值: {backoff}ms"
                    )
                else:
                    # 在允许范围内，等待回拨时间恢复
                    timestamp = self._wait_until_millis(self.last_timestamp)

            # 同一毫秒内序列号递增
            if timestamp == self.last_timestamp:
                self.sequence = (self.sequence + 1) & MAX_SEQUENCE
                if self.sequence == 0:
                    timestamp = self._wait_until_millis(timestamp + 1)
            else:
                self.sequence = 0

            self.last_timestamp = timestamp

            # 异步保存时间戳到 Redis
            if self.redis:
                self._save_timestamp()

            return ((timestamp - TWEPOCH) << TIMESTAMP_SHIFT) | \
                   (self.worker_id << WORKER_ID_SHIFT) | \
                   self.sequence
```

#### 加分项

**Worker ID 自动分配**：服务启动时从 ZooKeeper、etcd 或数据库自动获取可用的 Worker ID，故障时自动释放。

**批量预生成**：将 ID 预生成到本地队列，减少网络开销。使用 RingBuffer 实现无锁并发消费。

**多格式支持**：支持 Long（64 位）、String（Base62/Base36 编码）、UUID 等多种格式。

#### 可能的追问

**追问1：如何处理 Worker ID 分配问题？**

可以使用数据库表管理所有 Worker 节点，服务启动时向数据库插入一行获取自增 ID，或者使用 ZooKeeper/etcd 的临时有序节点来分配 ID。需要处理节点故障后的 ID 回收问题。

**追问2：Snowflake 算法如何保证 ID 有序？**

Snowflake 的 ID 包含毫秒级时间戳，且序列号在同一毫秒内递增，因此生成的 ID 是趋势递增的。但严格递增需要在集群内协调时钟，而 Snowflake 只保证单机递增，集群间可能存在交错。

**追问3：还有其他分布式 ID 方案吗？**

常见的替代方案包括：ULID（Universally Unique Lexicographically Sortable Identifier，比 UUID 有序且保持 128 位）、KSUID（类似 ULID 但包含时间戳）、Leaf（美团的解决方案，支持号段模式）等。

```python
# 库存服务伪代码
class InventoryService:
    def __init__(self, redis_client, db_client):
        self.redis = redis_client
        self.db = db_client

    async def check_and_reduce_stock(self, performance_id, ticket_type, quantity=1):
        """检查库存并扣减（使用 Redis + Lua 保证原子性）"""
        # Redis 库存 key
        stock_key = f"stock:{performance_id}:{ticket_type}"

        # Lua 脚本：原子检查和扣减库存
        lua_script = """
        local stock = redis.call('GET', KEYS[1])
        if stock == false then
            return -1  -- 库存未初始化
        end
        stock = tonumber(stock)
        local quantity = tonumber(ARGV[1])
        if stock < quantity then
            return 0  -- 库存不足
        end
        redis.call('DECRBY', KEYS[1], quantity)
        return stock - quantity  -- 返回剩余库存
        """

        result = await self.redis.eval(lua_script, 1, stock_key, quantity)

        if result == -1:
            # 库存未初始化，从数据库加载
            stock = await self._load_stock_from_db(performance_id, ticket_type)
            if stock <= 0:
                return False
            await self.redis.set(stock_key, stock)
            # 重新扣减
            result = await self.redis.eval(lua_script, 1, stock_key, quantity)

        return result >= 0

    async def _load_stock_from_db(self, performance_id, ticket_type):
        """从数据库加载库存到 Redis"""
        row = await self.db.query_one(
            "SELECT available FROM inventory WHERE performance_id = %s AND ticket_type = %s",
            (performance_id, ticket_type)
        )
        return row['available'] if row else 0

    async def revert_stock(self, performance_id, ticket_type, quantity):
        """回补库存（订单取消或超时）"""
        stock_key = f"stock:{performance_id}:{ticket_type}"
        await self.redis.incrby(stock_key, quantity)
        # 异步同步到数据库
        await self._sync_stock_to_db(performance_id, ticket_type, quantity)

    async def _sync_stock_to_db(self, performance_id, ticket_type, quantity):
        """异步同步库存到数据库"""
        # 使用数据库的乐观锁更新
        await self.db.execute(
            "UPDATE inventory SET available = available + %s WHERE performance_id = %s AND ticket_type = %s",
            (quantity, performance_id, ticket_type)
        )
```

```python
class RateLimitService:
    def __init__(self, redis_client):
        self.redis = redis_client

    async def check_rate_limit(self, user_id, action, limit, window_seconds):
        """
        滑动窗口限流
        返回 True 表示通过，False 表示被限流
        """
        key = f"rate_limit:{action}:{user_id}"
        now = time.time()
        window_start = now - window_seconds

        # 使用 Redis 事务保证原子性
        pipe = self.redis.pipeline()
        # 删除窗口外的记录
        pipe.zremrangebyscore(key, 0, window_start)
        # 统计窗口内请求数
        pipe.zcard(key)
        # 添加当前请求
        pipe.zadd(key, {str(now): now})
        # 设置过期
        pipe.expire(key, window_seconds)
        results = await pipe.execute()

        request_count = results[1]  # zcard 结果

        if request_count >= limit:
            return False  # 被限流

        return True

    async def check_seckill_limit(self, user_id, performance_id):
        """检查用户抢购限制（如每个用户限购 2 张）"""
        key = f"seckill:limit:{performance_id}:{user_id}"
        count = await self.redis.get(key)

        if count and int(count) >= 2:
            return False  # 已达购买上限

        return True

    async def increment_seckill_count(self, user_id, performance_id, expire_seconds=900):
        """增加用户抢购计数"""
        key = f"seckill:limit:{performance_id}:{user_id}"
        pipe = self.redis.pipeline()
        pipe.incr(key)
        pipe.expire(key, expire_seconds)  # 15分钟内有效
        await pipe.execute()
```

#### 题目描述

#### 思路分析

**延迟任务的类型**：

**定时任务**：在指定时间点执行，如每天早上 9 点发送报表。

**延迟任务**：从当前时间开始，等待一定时间后执行，如订单创建 30 分钟后检查是否付款。

**周期任务**：按照一定周期重复执行，如每分钟同步一次数据。

**延迟队列实现方案对比**：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|

| 数据库 + 轮询 | 将任务存储在数据库，轮询获取到期任务 | 简单、易于持久化 | 性能较低 |

```python
# 延迟任务服务伪代码
import asyncio
import json
import uuid
from typing import Callable, Dict, Any

class DelayTaskService:
    def __init__(self, redis_client, task_repo):
        self.redis = redis_client
        self.task_repo = task_repo
        # 任务队列的 Key
        self.queue_key = "delayed_tasks:queue"

    async def submit_task(self, task_type: str, task_data: Dict[str, Any],
                         delay_seconds: int, max_retries: int = 3) -> str:
        """提交延迟任务"""
        task_id = str(uuid.uuid4())
        execute_at = int(time.time() * 1000) + delay_seconds * 1000

        # 1. 存储任务信息到数据库（持久化）
        await self.task_repo.create({
            'task_id': task_id,
            'task_type': task_type,
            'task_data': json.dumps(task_data),
            'execute_at': execute_at,
            'status': 'pending',
            'max_retries': max_retries
        })

        # 2. 加入 Redis ZSet（用于快速获取到期任务）
        await self.redis.zadd(self.queue_key, {task_id: execute_at})

        return task_id

    async def submit_cron_task(self, task_type: str, task_data: Dict[str, Any],
                               cron_expression: str) -> str:
        """提交定时任务（使用 Cron 表达式）"""
        # 计算下次执行时间
        execute_at = self.cron_parser.parse(cron_expression)
        task_id = str(uuid.uuid4())

        await self.task_repo.create({
            'task_id': task_id,
            'task_type': task_type,
            'task_data': json.dumps(task_data),
            'execute_at': execute_at,
            'status': 'pending',
            'max_retries': 0,
            'is_cron': True,
            'cron_expression': cron_expression
        })

        await self.redis.zadd(self.queue_key, {task_id: execute_at})

        return task_id

    async def get_due_tasks(self, batch_size: int = 100) -> List[str]:
        """获取到期任务"""
        now = int(time.time() * 1000)

        # 获取到期任务（score <= now）
        # 使用 ZRANGEBYSCORE 获取，同时删除（ZPOPMIN）
        tasks = await self.redis.zpopmin(self.queue_key, batch_size)

        return [(task_id, score) for task_id, score in tasks]

    async def reschedule_task(self, task_id: str, delay_seconds: int):
        """重新调度任务（用于失败重试）"""
        execute_at = int(time.time() * 1000) + delay_seconds * 1000

        await self.task_repo.update(task_id, {
            'execute_at': execute_at,
            'status': 'pending'
        })

        await self.redis.zadd(self.queue_key, {task_id: execute_at})

    async def mark_completed(self, task_id: str, success: bool, error: str = None):
        """标记任务完成状态"""
        if success:
            await self.task_repo.update(task_id, {
                'status': 'completed',
                'completed_at': datetime.now()
            })
        else:
            # 失败，更新错误信息，由调度器决定是否重试
            await self.task_repo.update(task_id, {
                'status': 'failed',
                'error_message': error
            })
```

```python
# 调度器伪代码
class TaskScheduler:
    def __init__(self, task_service, worker_id, handlers: Dict[str, Callable]):
        self.task_service = task_service
        self.worker_id = worker_id
        self.handlers = handlers
        self.running = False

    async def start(self):
        """启动调度器"""
        self.running = True
        asyncio.create_task(self._poll_loop())

    async def stop(self):
        """停止调度器"""
        self.running = False

    async def _poll_loop(self):
        """轮询获取到期任务"""
        while self.running:
            try:
                # 获取到期任务
                tasks = await self.task_service.get_due_tasks(batch_size=10)

                for task_id, execute_at in tasks:
                    # 分发任务给执行器
                    asyncio.create_task(self._execute_task(task_id))

            except Exception as e:
                # 记录错误，继续运行
                print(f"轮询任务失败: {e}")

            # 轮询间隔，根据任务量动态调整
            await asyncio.sleep(0.1)

    async def _execute_task(self, task_id: str):
        """执行单个任务"""
        # 获取任务信息
        task = await self.task_service.get_task(task_id)
        if not task:
            return

        # 检查任务处理器是否存在
        handler = self.handlers.get(task['task_type'])
        if not handler:
            await self.task_service.mark_completed(
                task_id, False, f"Unknown task type: {task['task_type']}"
            )
            return

        # 更新任务状态为执行中
        await self.task_service.update_status(task_id, 'running', self.worker_id)

        start_time = time.time()
        success = False
        error_message = None

        try:
            # 执行任务
            task_data = json.loads(task['task_data'])
            await handler(task_data)
            success = True

        except Exception as e:
            error_message = str(e)

        finally:
            execution_time = int((time.time() - start_time) * 1000)

            if success:
                await self.task_service.mark_completed(task_id, True)
                # 记录执行日志
                await self.task_service.log_execution(
                    task_id, self.worker_id, 'success', execution_time
                )
            else:
                # 失败处理
                await self._handle_failure(task, error_message)

    async def _handle_failure(self, task: Dict, error_message: str):
        """处理任务失败"""
        task_id = task['task_id']
        retry_count = task.get('retry_count', 0) + 1
        max_retries = task['max_retries']

        await self.task_service.update_status(task_id, 'failed',
                                             retry_count=retry_count,
                                             error_message=error_message)

        if retry_count < max_retries:
            # 指数退避重试
            delay = 2 ** retry_count
            await self.task_service.reschedule_task(task_id, delay)

            await self.task_service.log_execution(
                task_id, self.worker_id, 'retry',
                error_message=f"{error_message}, retry #{retry_count}"
            )
        else:
            # 超过最大重试次数，进入死信队列
            await self.task_service.move_to_dlq(task_id, error_message)

            await self.task_service.log_execution(
                task_id, self.worker_id, 'failed',
                error_message=f"Max retries exceeded: {error_message}"
            )
```

### 题目十：设计一个分布式锁

#### 题目描述

设计一个分布式锁服务，支持互斥锁、锁超时、自动续期、可重入等功能。

#### 考察点

这道题考察候选人对以下技术的理解：

**分布式一致性问题**：为什么本地锁不够用，如何保证分布式环境下的互斥。

**Redis 分布式锁**：SETNX + EXPIRE 的问题及解决方案（SET NX EX）。

**Redlock 算法**：多节点分布式锁实现。

**锁的安全性**：死锁、锁重入、锁超时等边界情况处理。

#### 思路分析

**分布式锁的需求**：

**互斥性**：同一时刻只能有一个客户端持有锁。

**无死锁**：持有锁的客户端崩溃后，锁能够自动释放，避免死锁。

**可重入**：同一客户端可以多次获取同一把锁。

**公平性**：按照请求顺序获取锁（FIFO）。

**分布式锁实现方案对比**：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| Redis SETNX | 使用 SET key value NX EX 设置锁 | 性能高 | 无自动续期 |
| Redis Redlock | 多数节点过半机制 | 高可用 | 实现复杂 |
| ZooKeeper | 临时有序节点 + Watch | 有自动过期机制 | 性能较低 |
| etcd | 乐观锁 + Lease | 支持 MVCC | 复杂度较高 |

#### 架构设计

**Redlock 算法原理**：Redlock 通过在多个独立的 Redis 节点上获取锁来提高可靠性。客户端需要从 N 个节点中获取到 N/2+1 个节点的锁才算成功。

```
假设 N=5（5 个 Redis 节点）：
1. 客户端获取当前时间 T1
2. 依次向 5 个节点发送获取锁请求（SET key value NX EX timeout）
3. 客户端计算获取锁的耗时 T2 = T1 - 当前时间
4. 如果获取到 >= 3 个节点的锁，且耗时 < 锁过期时间，则认为获取锁成功
5. 锁的有效期 = 原始过期时间 - T2（扣除获取锁的耗时）
```

**Redis 分布式锁实现**：

```python
# 分布式锁服务伪代码
import uuid
import time
import threading
from contextlib import contextmanager

class DistributedLock:
    """Redis 分布式锁实现"""

    def __init__(self, redis_client, lock_name: str,
                 expire_seconds: int = 30,
                 retry_times: int = 3,
                 retry_interval: float = 0.1):
        self.redis = redis_client
        self.lock_name = f"distributed_lock:{lock_name}"
        self.expire_seconds = expire_seconds
        self.retry_times = retry_times
        self.retry_interval = retry_interval

        # 生成唯一标识，用于标识锁持有者
        self.client_id = f"{uuid.uuid4()}:{threading.current_thread().ident}"
        self.lock_value = f"{self.client_id}:0"  # 格式: client_id:count

    async def acquire(self) -> bool:
        """
        尝试获取锁
        返回 True 表示获取成功，False 表示获取失败
        """
        for _ in range(self.retry_times):
            # 使用 SET NX EX 原子操作获取锁
            acquired = await self.redis.set(
                self.lock_name,
                self.lock_value,
                nx=True,  # Only set if Not eXists
                ex=self.expire_seconds  # 过期时间
            )

            if acquired:
                # 获取成功，启动看门狗自动续期
                asyncio.create_task(self._watchdog())
                return True

            # 重试前等待
            await asyncio.sleep(self.retry_interval)

        return False

    async def release(self):
        """
        释放锁
        只有锁的持有者才能释放锁
        """
        # 使用 Lua 脚本保证检查和删除的原子性
        lua_script = """
        if redis.call('GET', KEYS[1]) == ARGV[1] then
            return redis.call('DEL', KEYS[1])
        else
            return 0
        end
        """
        await self.redis.eval(lua_script, 1, self.lock_name, self.lock_value)

    async def extend(self, extend_seconds: int = None):
        """
        延长锁的过期时间
        只有锁的持有者才能延长
        """
        if extend_seconds is None:
            extend_seconds = self.expire_seconds

        lua_script = """
        if redis.call('GET', KEYS[1]) == ARGV[1] then
            return redis.call('EXPIRE', KEYS[1], ARGV[2])
        else
            return 0
        end
        """
        result = await self.redis.eval(
            lua_script, 1,
            self.lock_name, self.lock_value, extend_seconds
        )
        return bool(result)

    async def _watchdog(self):
        """
        看门狗：自动续期
        在锁过期前一半时间时自动续期，直到锁被释放
        """
        while True:
            await asyncio.sleep(self.expire_seconds / 2)

            # 检查锁是否还被自己持有
            current_value = await self.redis.get(self.lock_name)
            if current_value == self.lock_value:
                await self.extend()
            else:
                # 锁已被释放或被他人获取
                break

    @contextmanager
    async def lock(self):
        """上下文管理器方式使用锁"""
        acquired = await self.acquire()
        if not acquired:
            raise LockAcquireError(f"Failed to acquire lock: {self.lock_name}")

        try:
            yield
        finally:
            await self.release()
```

**可重入锁实现**：

```python
class ReentrantDistributedLock(DistributedLock):
    """可重入分布式锁"""

    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        # 记录当前线程的重入次数
        self.local = threading.local()

    async def acquire(self) -> bool:
        # 获取当前线程的重入计数
        if not hasattr(self.local, 'count'):
            self.local.count = 0

        if self.local.count == 0:
            # 首次获取，使用父类方法
            acquired = await super().acquire()
            if acquired:
                self.local.count = 1
            return acquired
        else:
            # 已持有锁，直接重入
            self.local.count += 1
            return True

    async def release(self):
        """释放锁，减少重入计数"""
        if not hasattr(self.local, 'count'):
            return

        self.local.count -= 1

        if self.local.count == 0:
            # 重入计数归零，释放锁
            await super().release()
```

**Redlock 多节点实现**：

```python
class Redlock:
    """Redlock 多节点分布式锁"""

    def __init__(self, redis_clients: List, retry_times: int = 3):
        """
        redis_clients: Redis 客户端列表（如 5 个节点）
        """
        self.redis_clients = redis_clients
        self.retry_times = retry_times
        self.quorum = len(redis_clients) // 2 + 1  # 多数节点

    async def lock(self, lock_name: str, expire_ms: int = 30000) -> bool:
        """
        获取 Redlock
        返回 True 表示获取成功
        """
        lock_value = f"{uuid.uuid4()}:{threading.current_thread().ident}"

        # 向所有节点发送获取锁请求
        async def try_acquire(client, lock_name, expire_ms):
            try:
                return await client.set(
                    f"redlock:{lock_name}",
                    lock_value,
                    nx=True,
                    px=expire_ms
                )
            except Exception:
                return False

        # 并发获取所有节点的锁
        results = await asyncio.gather(*[
            try_acquire(client, lock_name, expire_ms)
            for client in self.redis_clients
        ])

        success_count = sum(1 for r in results if r)

        if success_count >= self.quorum:
            return True

        # 获取锁失败，释放所有已获取的锁
        await self._unlock_all(lock_name, lock_value)
        return False

    async def _unlock_all(self, lock_name: str, lock_value: str):
        """释放所有节点的锁"""
        await asyncio.gather(*[
            self._unlock_single(client, lock_name, lock_value)
            for client in self.redis_clients
        ])

    async def _unlock_single(self, client, lock_name: str, lock_value: str):
        """释放单个节点的锁"""
        try:
            lua_script = """
            if redis.call('GET', KEYS[1]) == ARGV[1] then
                return redis.call('DEL', KEYS[1])
            else
                return 0
            end
            """
            await client.eval(lua_script, 1, f"redlock:{lock_name}", lock_value)
        except Exception:
            pass
```

#### 加分项

**公平锁实现**：使用 Redis 的有序集合（Sorted Set）实现 FIFO 队列，保证锁获取的公平性。

**锁等待队列**：获取锁失败时，加入等待队列，锁释放时按顺序唤醒等待者。

**乐观锁优化**：对于读多写少的场景，使用版本号实现乐观锁，减少阻塞。

#### 可能的追问

**追问1：Redis 分布式锁有什么安全问题？**

Redis 分布式锁存在多个问题：单机模式下，如果 Redis 主节点故障，可能导致锁丢失；Redlock 算法虽然提高了可靠性，但实现复杂且仍有争议；锁的过期时间难以设置，太长会影响性能，太短可能导致锁提前释放。

**追问2：ZooKeeper 实现分布式锁的原理是什么？**

ZooKeeper 使用临时有序节点实现分布式锁。每个请求创建临时有序节点，根据节点序号判断是否获得锁；Watcher 机制实现等待队列，当前一个节点删除时自动通知后续节点。

**追问3：如何选择分布式锁的实现方案？**

根据场景选择：如果对性能要求高，Redis 是常见选择；如果对可靠性要求极高，使用 Redlock 或 ZooKeeper；如果已使用 etcd/Consul 作为配置中心，可以复用其分布式锁能力。

## 第三部分：面试技巧与注意事项

### 3.1 面试中的沟通技巧

系统设计面试是一个双向沟通过程，面试官不仅考察技术能力，也关注候选人的沟通方式和思维过程。以下是一些重要的沟通技巧：

**主动澄清需求**：在开始设计之前，不要急于给出答案。首先与面试官确认需求的细节，包括数据规模、性能要求、功能边界等。这展示了你谨慎思考的态度，也能避免设计偏离方向。可以说："在开始设计之前，我想先确认几个问题：系统的用户规模是多少？主要的功能优先级是什么？有什么技术约束吗？"

**明确假设条件**：如果某些条件没有给出，可以基于常识做出合理假设，并明确说明这些假设。例如："假设系统需要支持 100 万日活用户，平均每个用户每天产生 10 次请求，那么系统的日请求量约为 1000 万，峰值 QPS 约为 1000。"

**边说边画**：使用图表来辅助说明架构设计。一张清晰的架构图比长篇大论的文字描述更有效。可以使用方块表示服务、箭头表示数据流向、标注说明关键组件。

**展示决策过程**：不要只说"我选择 X 方案"，而是解释为什么选择 X 而不选择 Y。说明每个方案的优缺点，以及在当前场景下为什么 X 更合适。

**接受反馈**：如果面试官提出质疑或建议，不要防御性地反驳。认真听取并思考，给出合理的解释或调整方案。这展示了你接受批评和建议的能力。

### 3.2 时间管理

系统设计面试通常持续 30-45 分钟，需要在有限时间内完成需求分析、架构设计、详细设计和总结。需要合理分配时间：

**需求澄清（5-10 分钟）**：确认核心功能和约束条件，明确设计边界。

**高层设计（10-15 分钟）**：画出系统架构图，讲解核心组件和数据流。

**详细设计（15-20 分钟）**：深入讨论关键模块的实现，如数据模型、API 设计、算法选择。

**总结与扩展（5 分钟）**：总结设计要点，讨论可能的优化和扩展。

### 3.3 常见错误

**过度设计**：在面试中不要设计过于复杂的系统，特别是时间有限的情况下。先给出一个简单可行的方案，再讨论可能的优化和扩展。面试官更看重的是你的思考过程，而非一步到位的完美设计。

**忽略约束**：没有考虑数据规模、延迟要求、可用性等约束条件。设计一个能应对 100 QPS 的系统很简单，但能应对 100 万 QPS 的系统需要完全不同的架构。

**忽视 tradeoff**：只描述方案的优势，没有讨论潜在的缺点和权衡。优秀的工程师知道每个方案都有代价，需要在多个因素间权衡。

**缺乏实践**：只停留在理论层面，没有结合实际项目经验。如果有相关的实践经验，一定要结合具体案例来讲解。
