# 代码质量保障体系完全指南

## 前言：为什么你的代码需要"质量保障"？

想象一下，你正在建造一座房子。如果没有建筑师审核图纸、没有工程监理检查施工质量、没有验收标准，那么这座房子很可能存在各种隐患——墙壁可能歪斜、电线可能乱接、地下室可能漏水。代码项目也是同理，没有质量保障的代码，就像没有监理的建筑工程，随时可能"坍塌"。

代码质量保障体系，就是软件工程的"监理体系"。它包含了一系列工具、流程和规范，确保我们的代码：
- **能够正常工作**（功能正确性）
- **易于维护和理解**（代码可读性）
- **性能良好**（运行效率）
- **没有安全隐患**（安全性）
- **易于扩展和修改**（可维护性）

这一章，我们将详细介绍前端工程化中最常用的代码质量保障工具和方法。

## 一、ESLint：代码的"语法检查器"

### 1.1 ESLint是什么？

ESLint就像一位严格的语文老师，它会检查你写的"作文"（代码），找出各种语法错误、格式问题和不良编程习惯。不同的是，这位老师不仅能告诉你"哪里错了"，还能自动帮你"改正"。

### 1.2 为什么需要ESLint？

想象一下团队协作的场景：

```
小张喜欢这样写代码：
if(true){
    console.log("Hello")
}

小李喜欢这样写：
if (true) {
  console.log("Hello");
}

小王则喜欢这样：
if(true){console.log("Hello")}
```

如果没有统一规范，代码会变得"百花齐放"，阅读和维护都极其困难。ESLint就像一位严格的班主任，强制所有人使用统一的"作业格式"。

### 1.3 ESLint的核心概念

#### 规则（Rules）

ESLint通过规则来检查代码。每个规则都有三个选项：
- `"off"` 或 `0`：关闭该规则
- `"warn"` 或 `1`：警告（不影响运行）
- `"error"` 或 `2`：错误（会导致退出码非0）

```javascript
// .eslintrc.js 配置文件示例

module.exports = {
  // 继承官方推荐配置
  extends: ["eslint:recommended"],

  // 环境配置：告诉ESLint我们使用的是Node和现代浏览器
  env: {
    node: true,
    browser: true,
    es2021: true,
  },

  // 解析器：使用@babel/eslint-parser来解析TS/TSX
  parser: "@babel/eslint-parser",

  // 解析器选项
  parserOptions: {
    ecmaVersion: 2021,
    sourceType: "module",
    // 如果使用JSX，需要开启
    ecmaFeatures: {
      jsx: true,
    },
  },

  // 规则配置
  rules: {
    // 禁止使用console（生产环境）
    "no-console": "error",

    // 强制使用===而不是==
    eqeqeq: ["error", "always"],

    // 强制使用单引号
    quotes: ["error", "single"],

    // 强制使用const而不是let（未使用的变量报错）
    "no-unused-vars": "error",

    // 强制使用尾随逗号（ES5兼容）
    "comma-dangle": ["error", "always-multiline"],

    // 强制缩进为2个空格
    indent: ["error", 2],

    // 强制分号结尾
    semi: ["error", "always"],
  },

  // 针对特定文件覆盖规则
  overrides: [
    {
      // 只对测试文件开启jest环境
      files: ["**/*.test.js", "**/*.spec.js"],
      env: {
        jest: true,
      },
    },
  ],
};
```

### 1.4 ESLint的规则分类

ESLint的规则可以分为几大类：

#### 1. 潜在问题（Possible Problems）

这类规则帮助避免语法错误或逻辑bug：

```javascript
// 1. no-duplicate-imports：禁止重复导入
// 错误示例
import { foo } from "module";
import { bar } from "module"; // 重复导入

// 正确示例
import { foo, bar } from "module";

// 2. no-irregular-whitespace：禁止不规则的空白字符
// 这可以避免一些隐藏的bug，比如全角空格导致的语法错误

// 3. no-prototype-builtins：禁止直接调用Object.prototype的方法
// 错误示例
const has = obj.hasOwnProperty("foo");

// 正确示例
const has = Object.prototype.hasOwnProperty.call(obj, "foo");
// 或者使用 lodash 的 has 函数
import has from "lodash/has";
```

#### 2. 最佳实践（Best Practices）

这类规则帮助你写出更健壮的代码：

```javascript
// 1. no-eval()：禁止使用eval
// eval就像黑魔法，虽然强大但极其危险
// 错误示例
const result = eval("2 + 2"); // "2 + 2" 被当作代码执行

// 正确示例
const result = 2 + 2; // 直接计算

// 2. no-implicit-globals：禁止在全局作用域声明变量
// 在浏览器中，这会污染全局window对象

// 错误示例（在脚本顶层）
var globalVar = "我在污染全局作用域";

// 正确示例
(function() {
  // 使用IIFE包裹，或者使用ES6模块
  const localVar = "我只在局部作用域";
})();

// 3. consistent-return：要求函数返回值类型一致
// 错误示例
function process(data) {
  if (data.valid) {
    return true;
  }
  // 有时候返回，有时候不返回
}

// 正确示例
function process(data) {
  if (data.valid) {
    return true;
  }
  return false; // 始终返回值
}
```

#### 3. 变量声明（Variable Declarations）

```javascript
// 1. no-undef：禁止使用未声明的变量
// 错误示例
function foo() {
  console.log(x); // x未声明
}

// 正确示例
function foo() {
  const x = 10;
  console.log(x);
}

// 2. no-const-assign：禁止修改const声明的变量
// 错误示例
const x = 1;
x = 2; // 报错！const声明的变量不能重新赋值

// 3. prefer-const：如果变量从不重新赋值，使用const
// 警告示例
let x = 1;
x = 2; // 如果x之后不再修改，应该用const

// 正确示例
const x = 1; // x不会被重新赋值
```

#### 4. Node.js和CommonJS规则

```javascript
// 1. no-sync：如果有异步版本，就不要用同步方法
// 错误示例
const fs = require("fs");
const content = fs.readFileSync("/path/to/file"); // 同步阻塞

// 正确示例
const fs = require("fs").promises;
const content = await fs.readFile("/path/to/file"); // 异步非阻塞

// 2. no-process-exit：禁止直接调用process.exit()
// 因为这会强制终止进程，可能导致清理逻辑不执行
```

### 1.5 ESLint配置继承

ESLint允许我们继承现有的配置集，而不需要从头编写所有规则：

```javascript
// .eslintrc.js

module.exports = {
  // 继承多个配置（后面的会覆盖前面的）
  extends: [
    // 1. ESLint官方推荐配置（包含一些基本规则）
    "eslint:recommended",

    // 2. React项目推荐配置
    "plugin:react/recommended",

    // 3. React Hooks规则
    "plugin:react-hooks/recommended",

    // 4. TypeScript推荐配置
    "plugin:@typescript-eslint/recommended",

    // 5. Next.js推荐配置
    "next/core-web-vitals",
  ],

  // 配置插件
  plugins: [
    "react",           // React规则
    "react-hooks",     // React Hooks规则
    "@typescript-eslint", // TypeScript规则
  ],

  // 共享设置（会被每个规则共享访问）
  settings: {
    // 告诉ESLint我们的React版本
    react: {
      version: "detect", // 自动检测package.json中的版本
    },
  },

  // 针对特定文件应用不同配置
  overrides: [
    {
      // 对测试文件应用不同规则
      files: ["**/*.test.ts", "**/*.test.tsx"],
      extends: ["plugin:jest/recommended"],
      rules: {
        // 测试文件中允许使用describe和it
        "no-unused-vars": "off",
      },
    },
  ],
};
```

### 1.6 常用ESLint插件

#### @typescript-eslint/eslint-plugin

TypeScript的ESLint插件，提供了大量TypeScript特定的规则：

```javascript
// TypeScript特定规则示例
rules: {
  // 1. ts/no-unused-vars：TypeScript版本的未使用变量检查
  // 比原生规则更智能，能理解类型
  "@typescript-eslint/no-unused-vars": [
    "error",
    {
      argsIgnorePattern: "^_",      // 以下划线开头的参数被忽略
      varsIgnorePattern: "^_",      // 以下划线开头的变量被忽略
      caughtErrorsIgnorePattern: "^_",
    },
  ],

  // 2. ts/explicit-function-return-type：要求显式声明函数返回类型
  // 错误示例
  function add(a, b) {
    return a + b;
  }

  // 正确示例
  function add(a: number, b: number): number {
    return a + b;
  }

  // 3. ts/no-explicit-any：禁止使用any类型
  "@typescript-eslint/no-explicit-any": "error",

  // 4. ts/consistent-type-imports：强制一致的类型导入方式
  // 推荐使用 import type { Type } 而不是 import { Type }
  "@typescript-eslint/consistent-type-imports": [
    "error",
    {
      prefer: "type-imports", // 推荐使用 type 修饰符
    },
  ],
}
```

#### eslint-plugin-react

React特定规则：

```javascript
// React规则示例
rules: {
  // 1. react/prop-types：要求为props声明类型
  // 帮助你捕获prop类型错误
  "react/prop-types": "error",

  // 2. react/display-name：要求为组件声明displayName
  // 有助于调试时识别组件
  "react/display-name": "error",

  // 3. react/no-children-prop：禁止传递children作为prop
  // 错误示例
  <Component children={<div>内容</div>} />

  // 正确示例
  <Component>
    <div>内容</div>
  </Component>

  // 4. react-hooks/exhaustive-deps：检查useEffect依赖
  // 确保useEffect的依赖数组完整
  "react-hooks/exhaustive-deps": "warn",
}
```

#### eslint-plugin-import

导入排序和路径检查：

```javascript
// import规则示例
rules: {
  // 1. import/order：强制导入顺序
  // 推荐顺序：React -> 第三方 -> 相对导入 -> 样式
  "import/order": [
    "error",
    {
      groups: [
        "builtin",    // Node内置模块（如fs, path）
        "external",   // npm包
        "internal",   // 内部模块（通过alias配置）
        ["parent", "sibling"], // 相对导入
        "index",      // 导入索引文件
        "object",     // type导入
        "type",       // 类型导入
      ],
      "newlines-between": "always", // 组之间需要空行
      alphabetize: {
        order: "asc", // 按字母顺序排列
        caseInsensitive: true,
      },
    },
  ],

  // 2. import/no-unresolved：检查导入路径是否存在
  "import/no-unresolved": "error",

  // 3. import/named：检查命名导入是否存在
  "import/named": "error",

  // 4. import/default：检查默认导入
  "import/default": "error",
}
```

## 二、Prettier：代码的"自动排版师"

### 2.1 Prettier是什么？

如果说ESLint是一位严格的语文老师，那么Prettier就是一位强迫症级别的排版师。Prettier不关心你代码的"语法"对不对，它只关心你的代码"长得好看"。

Prettier的核心理念是：**"只有一个正确的格式"**。它会移除所有可以自定义的格式选项，强制团队使用统一格式。

### 2.2 Prettier vs ESLint

| 特性 | ESLint | Prettier |
|------|--------|----------|
| **主要职责** | 代码质量、潜在bug | 代码格式、排版 |
| **可配置性** | 高度可配置 | 几乎不可配置 |
| **修复能力** | 自动修复部分问题 | 自动格式化全部代码 |
| **检查范围** | 语法、逻辑、风格 | 缩进、引号、分号等 |
| **运行速度** | 较慢（需要解析代码） | 较快 |

**最佳实践**：两者配合使用，ESLint负责代码质量，Prettier负责代码格式。

### 2.3 Prettier配置

```javascript
// .prettierrc.js

module.exports = {
  // 1. 单引号还是双引号
  // 推荐：false（使用双引号），但React/JSX中通常用单引号
  singleQuote: false,

  // 2. 是否使用尾随逗号
  // "es5" - ES5兼容（对象、数组等）
  // "all" - 所有可能的地方（包括函数参数）
  // "none" - 不用尾随逗号
  trailingComma: "es5",

  // 3. 分号
  // 推荐：true（始终添加分号）
  semi: true,

  // 4. Tab宽度
  tabWidth: 2,

  // 5. 是否使用Tab
  // 推荐：false（用空格而不是Tab）
  useTabs: false,

  // 6. 行宽（每行最大字符数）
  printWidth: 100,

  // 7. JSX引号
  jsxSingleQuote: false,

  // 8. JSX属性引号
  // "preserve" - 保持原样
  quoteProps: "as-needed",

  // 9. 括号内部是否需要空格
  bracketSpacing: true,

  // 10. HTML空白字符处理
  htmlWhitespaceSensitivity: "css",

  // 11. 箭头函数的参数括号
  // "always" - 始终有括号：x => x
  // "avoid" - 尽可能没有括号：x => x
  // "force" - 单参数也加强制括号：(x) => x
  arrowParens: "always",

  // 12. 行结束符
  // "lf" - Unix风格（\n）
  // "crlf" - Windows风格（\r\n）
  // "auto" - 保持原样
  endOfLine: "lf",

  // 13. 是否格式化嵌入的内容（如HTML中的<script>）
  embeddedLanguageFormatting: "auto",
};
```

## 三、 Husky：Git钩子的"自动化门卫"

### 3.1 Git钩子是什么？

在讲解Husky之前，我们先理解Git钩子（Hooks）的概念。

Git钩子是Git操作生命周期中插入的"钩子"（回调函数）。你可以把这些钩子理解为一个"安检门"，每次你执行git commit、git push等操作时，都会触发相应的安检程序。

```
Git操作流程：
1. git commit 执行
2. 触发 pre-commit 钩子（提交前检查）
   ├── 运行 ESLint 检查代码
   ├── 运行测试
   └── 如果检查失败 → 阻止提交
3. 提交成功
4. 触发 post-commit 钩子（提交后通知）
```

### 3.2 常见的Git钩子

| 钩子名称 | 触发时机 | 常见用途 |
|----------|----------|----------|
| pre-commit | git commit之前 | 运行linters、格式化代码、运行测试 |
| commit-msg | git commit之前，编辑完提交信息后 | 检查提交信息格式 |
| pre-push | git push之前 | 运行测试、构建检查 |
| pre-rebase | git rebase之前 | 代码质量检查 |
| post-commit | git commit之后 | 通知、CI触发 |
| post-push | git push之后 | 部署、通知 |

### 3.3 Husky是什么？

原生Git钩子需要你在`.git/hooks/`目录下编写shell脚本。但这些脚本有几个问题：

1. **不易共享**：每个开发者都需要手动创建
2. **难以维护**：shell脚本不易编写和调试
3. **无法版本控制**：hooks目录不会被提交到仓库

Husky就是为了解决这些问题而生的。它允许你通过`package.json`或配置文件来定义Git钩子，并且这些定义会和项目一起提交到仓库。

### 3.4 Husky安装和配置

#### 安装

```bash
# 1. 安装 Husky（作为开发依赖）
npm install -D husky

# 2. 初始化Husky（在项目根目录创建.husky目录）
npx husky install

# 3. 配置Git prepare脚本（自动安装hooks）
# 在package.json中添加prepare脚本
npm pkg set prepare="husky install"
```

#### 配置package.json

```json
{
  "scripts": {
    "prepare": "husky install"  // npm install 时自动运行
  }
}
```

### 3.5 创建Git钩子

#### 创建pre-commit钩子

pre-commit钩子是最常用的钩子，用于在提交前检查代码质量。

```bash
# 创建pre-commit钩子
npx husky add .husky/pre-commit "npm run lint"
```

这会在`.husky/pre-commit`文件中创建：

```shell
#!/bin/sh
# .husky/pre-commit

# 运行npm run lint命令
npm run lint
```

#### 创建commit-msg钩子

commit-msg钩子用于检查提交信息格式：

```bash
npx husky add .husky/commit-msg 'npx --no -- commitlint --edit "$1"'
```

#### 创建pre-push钩子

pre-push钩子在push之前运行测试：

```bash
npx husky add .husky/pre-push "npm run test"
```

### 3.6 Husky配置文件

Husky 7+版本支持在`husky.config.js`中配置：

```javascript
// husky.config.js

module.exports = {
  // hook的目录
  hookDir: ".husky",

  // 需要安装的hooks
  hooks: {
    // pre-commit钩子
    "pre-commit": "npm run lint && npm run format:check",

    // commit-msg钩子
    "commit-msg": "npx --no -- commitlint --edit $1",

    // pre-push钩子
    "pre-push": "npm run test:unit && npm run build",
  },
};
```

### 3.7 完整的Husky工作流

一个典型的团队项目会配置这样的钩子：

```
┌─────────────────────────────────────────────────────────────┐
│                     Git Commit 流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 开发者执行 git commit                                   │
│           │                                                 │
│           ▼                                                 │
│  ┌─────────────────┐                                        │
```

## 四、 lint-staged：只检查"待提交"的代码

### 4.1 为什么需要lint-staged？

想象一下你的项目有10000行代码，你修改了其中的50行，然后执行git commit。按照传统的lint配置，ESLint会检查全部10000行代码，这非常耗时。

lint-staged就是为了解决这个问题。它只会检查**即将提交的文件**，大大提高了检查效率。

### 4.2 lint-staged配置

```javascript
// lint-staged.config.js

module.exports = {
  // 检查所有暂存的文件
  // $ git add file1.ts file2.ts
  // lint-staged只检查这两个文件

  // 对不同类型的文件执行不同的检查
  "*.{js,jsx,ts,tsx}": [
    // 1. 先用Prettier格式化
    "prettier --write",

    // 2. 再用ESLint检查（fix自动修复）
    "eslint --fix",

    // 3. 运行单元测试
    "npm run test:unit -- --bail",
  ],

  // 3. 对JSON文件只做格式化
  "*.json": ["prettier --write"],

  // 4. 对Markdown文件做格式化和拼写检查
  "*.md": [
    "prettier --write",
    // 可以添加markdownlint等工具
  ],

  // 5. 对CSS文件做格式化和lint
  "*.{css,scss,sass}": [
    "prettier --write",
    "stylelint --fix",
  ],
};
```

### 4.3 在Husky中集成lint-staged

修改pre-commit钩子：

```bash
# 创建使用lint-staged的pre-commit钩子
npx husky add .husky/pre-commit "npx lint-staged"
```


## 五、 SonarQube：代码质量的"全方位体检中心"

### 5.1 SonarQube是什么？

如果说ESLint是"语文老师"，Prettier是"排版师"，那么SonarQube就是一个"全方位体检中心"。

SonarQube（现在叫Sonar）是一个开源的代码质量管理平台，它能够：
- **静态分析代码**：不需要运行代码就能发现问题
- **检测Bug**：发现潜在的代码缺陷
- **安全漏洞**：发现SQL注入、XSS等安全问题
- **代码异味**：发现难以维护的代码
- **代码覆盖率**：统计测试覆盖了多少代码
- **技术债务**：评估修复这些问题需要多少时间

### 5.2 SonarQube的核心指标

#### 1. 可维护性（Maintainability）

代码的可维护性如何？是否难以理解和修改？

```
评分等级：
A - 优秀（0-0.1% 技术债务率）
B - 良好（0.1-0.2%）
C - 一般（0.2-0.5%）
D - 较差（0.5-1%）
E - 很差（>1%）
```

#### 2. 可靠性（Reliability）

代码是否存在潜在的Bug？

```
评分等级：
A - 无Bug
B - 1个轻微Bug
C - 至少1个主要Bug
D - 至少1个严重Bug
E - 至少1个阻断性Bug
```

#### 3. 安全（Security）

代码是否存在安全漏洞？

```
评分等级：
A - 无漏洞
B - 1个轻微漏洞
...
E - 多个严重漏洞
```

#### 4. 安全热点（Security Hotspots）

需要人工审核的安全敏感代码区域。

### 5.3 SonarQube规则分类

#### Bug（虫子）

可能导致生产环境问题的代码：

```typescript
// 1. no-insignificant-blank-lines：禁止使用无意义的空行
// 有时候开发者用多个空行来"分组"代码，这是不推荐的

// 2. no-unused-variables：禁止未使用的变量
const unused = "这个变量声明了但从未使用";

// 3. no-constant-conditions：禁止常量条件表达式
if (true) {  // 永远为true的条件
  // ...
}

// 4. no-empty-functions：禁止空函数
function doNothing() {
  // 空函数！可能是个bug
}
```

#### 漏洞（Vulnerabilities）

安全相关问题：

```typescript
// 1. sql-injection：SQL注入风险
// 危险示例
const query = `SELECT * FROM users WHERE name = '${userInput}'`;

// 安全示例 - 使用参数化查询
const query = `SELECT * FROM users WHERE name = ?`;
db.execute(query, [userInput]);

// 2. xss：跨站脚本攻击风险
// 危险示例
element.innerHTML = userInput; // 用户输入被当作HTML执行

// 安全示例
element.textContent = userInput; // 纯文本，不会被执行

// 3. hard-coded-credentials：硬编码凭证
// 危险示例
const apiKey = "sk-1234567890abcdef"; // API密钥被暴露

// 安全示例 - 使用环境变量
const apiKey = process.env.API_KEY;
```

#### 代码异味（Code Smells）

降低可维护性的问题：

```typescript
// 1. cognitively-complex-functions：认知复杂度过高的函数
// 过于复杂的函数难以理解和维护
function processOrder(order) {
  if (order.valid) {
    if (order.items.length > 0) {
      if (order.payment) {
        if (order.payment.status === "paid") {
          if (order.shipping) {
            // 嵌套太深了！重构吧
          }
        }
      }
    }
  }
}

// 2. duplicated-blocks：重复代码块
// 重复代码应该抽取为函数

// 3. long-functions：过长的函数
// 函数应该保持简短，每个函数只做一件事

// 4. magic-numbers：魔法数字
// 应该使用命名常量
const maxRetryCount = 3;
if (attemptCount > maxRetryCount) {
  // 不要用 if (attemptCount > 3)
}
```

## 六、 代码覆盖率：测试的"盲区探测器"

### 6.1 什么是代码覆盖率？

代码覆盖率衡量的是"测试覆盖了多少代码"。就像考试一样，覆盖率高意味着更多代码被测试到了。

```
代码覆盖率类型：

┌─────────────────────────────────────────────────────────┐
│  1. 行覆盖率（Line Coverage）                            │
│     测试执行了多少行代码                                  │
│                                                         │
│     覆盖率 = (执行的代码行数 / 总代码行数) × 100%        │
│                                                         │
├─────────────────────────────────────────────────────────┤
│  2. 分支覆盖率（Branch Coverage）                        │
│     测试覆盖了多少分支（if/else、switch等）              │
│                                                         │
│     覆盖率 = (执行的分支数 / 总分支数) × 100%            │
│                                                         │
├─────────────────────────────────────────────────────────┤
│  3. 函数覆盖率（Function Coverage）                      │
│     测试调用了多少函数                                    │
│                                                         │
│     覆盖率 = (调用的函数数 / 总函数数) × 100%            │
│                                                         │
├─────────────────────────────────────────────────────────┤
│  4. 语句覆盖率（Statement Coverage）                     │
│     测试执行了多少语句                                    │
│                                                         │
│     覆盖率 = (执行的语句数 / 总语句数) × 100%            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 6.2 覆盖率报告工具

#### Istanbul（NYC）

Istanbul是Node.js最常用的覆盖率工具：

```bash
# 安装
npm install -D nyc

# 在package.json中配置
{
  "scripts": {
    // 使用nyc运行测试并生成覆盖率报告
    "test:coverage": "nyc npm run test",

    // 或者直接使用 Jest（内置覆盖率支持）
    "test:coverage": "jest --coverage"
  }
}
```

### 6.3 Jest覆盖率配置

```javascript
// jest.config.js

module.exports = {
  // 覆盖率配置
  collectCoverage: true,  // 启用覆盖率收集

  // 覆盖率目录
  coverageDirectory: "coverage",

  // 覆盖率报告格式
  coverageReporters: [
    "text",        // 控制台文本报告
    "lcov",        // HTML报告（可用浏览器打开）
    "clover",      // Clover XML格式
    "html",        // HTML格式
  ],

  // 需要收集覆盖率的目录
  collectCoverageFrom: [
    "src/**/*.{js,jsx,ts,tsx}",  // 收集src下所有文件的覆盖率
    "!src/**/*.d.ts",            // 排除类型定义文件
    "!src/**/*.test.{js,ts}",    // 排除测试文件本身
    "!src/**/index.ts",         // 排除入口文件
  ],

  // 强制覆盖率阈值（达不到会失败CI）
  coverageThreshold: {
    // 全局阈值
    global: {
      branches: 50,      // 分支覆盖率 ≥ 50%
      functions: 50,     // 函数覆盖率 ≥ 50%
      lines: 50,         // 行覆盖率 ≥ 50%
      statements: 50,    // 语句覆盖率 ≥ 50%
    },

    // 针对特定文件设置不同阈值
    "./src/components/**/*": {
      branches: 80,
      functions: 80,
      lines: 80,
      statements: 80,
    },

    // 关键文件要求100%覆盖率
    "./src/utils/**/*": {
      branches: 100,
      functions: 100,
      lines: 100,
      statements: 100,
    },
  },
};
```

### 6.5 提高覆盖率的策略

#### 1. 从关键代码开始

```
优先级顺序：
1. 业务逻辑（utils、helpers）→ 核心功能，100%覆盖
2. 复杂Hooks → 逻辑复杂，容易出错
3. 复杂组件 → UI逻辑
4. 简单组件 → 优先级低
```

#### 2. 测试边界条件

```typescript
// 待测试函数
function divide(a: number, b: number): number {
  return a / b;
}

// 测试用例
describe("divide", () => {
  // 基本测试
  test("正常除法", () => {
    expect(divide(6, 2)).toBe(3);
  });

  // 边界测试
  test("0作为除数", () => {
    expect(() => divide(6, 0)).toThrow(); // 抛出错误
  });

  // 负数测试
  test("负数除法", () => {
    expect(divide(-6, 2)).toBe(-3);
  });

  // 小数测试
  test("小数结果", () => {
    expect(divide(7, 2)).toBe(3.5);
  });
});
```

#### 3. 测试真实场景

```typescript
// 测试用户交互（而非简单调用）
describe("登录流程", () => {
  test("输入正确凭证后显示成功消息", async () => {
    render(<LoginForm />);

    // 输入用户名
    const usernameInput = screen.getByLabelText("用户名");
    fireEvent.change(usernameInput, { target: { value: "admin" } });

    // 输入密码
    const passwordInput = screen.getByLabelText("密码");
    fireEvent.change(passwordInput, { target: { value: "123456" } });

    // 点击登录
    const submitButton = screen.getByRole("button", { name: "登录" });
    fireEvent.click(submitButton);

    // 验证结果
    await waitFor(() => {
      expect(screen.getByText("登录成功")).toBeInTheDocument();
    });
  });
});
```

```yaml
# .github/workflows/ci.yml

name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  # 1. 代码质量检查
  lint:
    name: ESLint & Prettier检查
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - run: npm ci

      - name: ESLint检查
        run: npm run lint

      - name: Prettier格式检查
        run: npm run format:check

  # 2. 单元测试
  test:
    name: 单元测试
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - run: npm ci

      - name: 运行测试
        run: npm run test:coverage

      - name: 上传覆盖率报告
        uses: codecov/codecov-action@v3
        with:
          files: ./coverage/lcov.info
          fail_ci_if_error: true

  # 3. 构建检查
  build:
    name: 构建检查
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - run: npm ci

      - name: 构建项目
        run: npm run build

      - name: 上传构建产物
        uses: actions/upload-artifact@v4
        with:
          name: build
          path: .next
          retention-days: 7

  # 4. SonarCloud分析
  sonar:
    name: SonarCloud代码分析
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - run: npm ci

      - name: 运行测试生成覆盖率
        run: npm run test:coverage

      - name: SonarCloud扫描
        uses: SonarSource/sonarcloud-github-action@master
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
        with:
          projectKey: my-org_my-project
```

### 8.2 团队实践建议

#### 1. 分阶段引入

```
阶段1：基础配置（1-2天）
├── 配置ESLint + Prettier
├── 配置IDE格式化保存
└── 统一代码风格

阶段2：自动化检查（2-3天）
├── 配置Husky
├── 配置lint-staged
└── 提交前自动检查

阶段3：持续集成（1周）
├── 配置GitHub Actions
├── 自动化测试
├── 覆盖率报告
└── SonarQube集成
```

#### 2. 阈值建议

| 指标 | 初始阈值 | 目标阈值 |
|------|----------|----------|
| ESLint错误 | 0 | 0 |
| 测试覆盖率 | 50% | 80% |
| 关键代码覆盖率 | 80% | 100% |
| 技术债务率 | <5% | <2% |

#### 3. 常见问题

**Q: ESLint和Prettier冲突怎么办？**
A: 安装`eslint-config-prettier`并在ESLint配置最后extends它。

**Q: 覆盖率要求太高导致CI失败？**
A: 可以先设置较低的阈值，逐步提高；或者使用`nyc --check-coverage`来动态调整。

**Q: Husky钩子不执行？**
A: 检查`.husky/`目录权限；确保`prepare`脚本在npm install时运行。

**Q: 团队成员不遵守规范？**
A: 1) 确保IDE配置自动格式化；2) 提交前检查应该足够快；3) 通过code review监督。

### 8.3 持续改进

代码质量保障不是一劳永逸的，需要持续改进：

```
每月Review流程：
1. 查看SonarQube报告，找出新增的问题
2. 分析覆盖率变化，调整测试策略
3. 更新ESLint规则，添加针对项目特性的规则
4. 优化lint-staged配置，加快检查速度
5. 分享最佳实践，提升团队意识
```

## 九、 附录：常用命令速查

```bash
# ESLint
npx eslint src/                    # 检查src目录
npx eslint src/ --fix             # 自动修复
npx eslint --init                  # 初始化ESLint配置

# Prettier
npx prettier --write src/          # 格式化src目录
npx prettier --check src/         # 检查格式
npx prettier --config .prettierrc  # 指定配置文件

# Husky
npx husky install                   # 安装husky
npx husky add .husky/pre-commit "..."  # 添加钩子
git config core.hooksPath .husky    # 设置hooks路径

# lint-staged
npx lint-staged                     # 运行lint-staged

# Jest覆盖率
npx jest --coverage                 # 运行测试并生成覆盖率
npx jest --coverage --coverageThreshold='{"global":{"branches":80}}'  # 设置阈值

# SonarScanner
npx sonar-scanner                   # 运行扫描
```
