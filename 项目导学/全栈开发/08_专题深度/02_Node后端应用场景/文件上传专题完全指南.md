# 文件上传专题完全指南

## 概述

文件上传是Web开发中最常见也是最核心的功能之一。从用户头像上传、文档分享到视频发布，几乎每一个互联网应用都离不开文件上传功能。本指南将从最基础的表单上传开始，逐步深入到分片上传、断点续传、大文件优化以及云存储集成等高级主题，帮助开发者全面掌握文件上传的核心技术。

在实际开发中，文件上传面临着诸多挑战：如何处理大文件、如何保证上传的安全性、如何实现断点续传、如何集成云存储服务等。本指南将通过大量的实战代码示例，帮助读者解决这些实际问题。无论是前端开发者还是后端开发者，都能在本指南中找到所需的技术方案。

本指南的所有代码示例都基于Node.js生态，使用Express或NestJS作为Web框架，multer作为文件上传中间件，并集成阿里云OSS、AWS S3等云存储服务。所有代码都经过实际验证，读者可以直接应用到项目中。

## 第一章 文件上传基础

### 1.1 表单编码：multipart/form-data

文件上传的核心在于HTTP协议的multipart/form-data编码方式。理解这种编码方式是掌握文件上传的第一步。在传统的URL编码（application/x-www-form-urlencoded）中，所有数据都被编码为键值对的形式，文字可以直接传输，但二进制文件（图片、视频、音频等）无法直接编码。multipart/form-data正是为了解决这一问题而设计的。

multipart/form-data编码的工作原理是将表单数据分割成多个部分，每个部分都有自己的Content-Disposition头部和Content-Type头部。文件内容作为独立的二进制部分包含在请求体中。这种编码方式支持传输任意类型的文件，也支持同时传输多个文件和普通表单字段。

当我们查看一个multipart/form-data请求时，会发现它的请求体与普通表单请求完全不同。普通表单请求的请求体可能是这样的：`username=张三&age=25`，简单直观。但multipart/form-data请求体则复杂得多，它看起来像是包含了一个完整的邮件格式，每部分之间用随机的边界字符串分隔。例如，一个简单的头像上传请求可能包含以下结构：首先是 `------WebKitFormBoundary7MA4YWxkTrZu0gW` 这样的边界标记，然后是Content-Disposition头部说明这是表单字段"avatar"，Content-Type说明文件类型，然后是空行，最后是文件的二进制内容。

这种编码方式的优势在于它的通用性和完整性。通用性体现在任何类型的文件都可以通过这种方式上传，无论是文本文件、图片、视频还是压缩包。完整性体现在它可以同时携带文件名、文件类型等元信息，以及文件的实际二进制内容。这为后续的文件处理提供了丰富的上下文信息。

### 1.2 文件二进制传输

文件上传的本质是将文件的二进制数据从客户端传输到服务器端。理解二进制传输的原理对于排查上传问题、优化上传性能至关重要。在Node.js中，文件数据以Buffer的形式存在，Buffer是Node.js专门为处理二进制数据而创建的数据类型。

在文件上传过程中，HTTP请求的Content-Type头部指定为multipart/form-data，并且包含一个特殊的boundary参数，用于分隔请求体中的各个部分。服务器端接收到请求后，需要按照这个boundary来解析请求体，将每个部分分离出来。对于文件部分，解析出来的就是原始的二进制数据。

文件二进制传输涉及几个关键的技术点。首先是请求体的解析方式，传统的解析方式是先将整个请求体读取到内存中，然后进行解析。但对于大文件来说，这种方式会导致内存占用过高。因此，实际应用中通常采用流式解析，边读取边处理。其次是传输编码，HTTP协议支持对请求体进行编码传输，如gzip压缩。但对于已经高度压缩的文件（如jpg、png、mp4等），再进行gzip压缩往往效果不佳，甚至可能增加文件大小。

在前端实现文件二进制传输时，需要使用FormData对象来构建请求体。FormData是Web API的一部分，它可以动态构建包含文件和表单字段的数据集合。当使用Fetch API或XMLHttpRequest发送FormData时，浏览器会自动设置正确的Content-Type头部，包括multipart/form-data和自动生成的boundary。这个过程是浏览器自动完成的，开发者通常不需要手动设置。

### 1.3 前端FormData构建

在前端JavaScript中，构建文件上传请求主要依靠FormData API。FormData对象可以包含多个键值对，每个键可以是一个字符串值或一个文件对象。通过append方法可以向FormData添加数据，通过delete方法可以删除数据，通过set方法可以设置（替换）指定键的值。

创建一个基本的FormData对象并添加文件非常简单。首先，通过document.getElementById或querySelector获取input元素，然后通过files属性获取用户选择的文件对象。files属性是一个FileList对象，类似数组，可以使用索引访问单个File对象。然后，将File对象append到FormData中，指定一个字段名。最后，将FormData作为请求体发送给服务器。

在实际应用中，FormData常常需要同时包含文件和其他表单字段。例如，用户上传头像时可能需要同时提交用户名、描述等信息。这些都可以通过append方法添加到FormData中。服务器端接收到数据后，可以根据字段名区分普通表单字段和文件字段。需要注意的是，FormData中的字段顺序可能影响服务器端的解析结果，因此应该将文件字段放在最后，或者确保服务器端能够正确处理任意顺序的字段。

现代前端开发中，通常使用Fetch API来发送包含FormData的请求。Fetch API提供了更现代化、更简洁的API设计，支持Promise和async/await语法。相比传统的XMLHttpRequest，Fetch API的代码更易读、更易维护。但需要注意，Fetch API在某些场景下的进度监控不如XMLHttpRequest完善，如果需要显示上传进度，可能需要使用XMLHttpRequest或新的Upload API。

### 1.4 实战：最简上传

让我们通过一个完整的示例来理解文件上传的最基本实现。这个示例包含前端和后端两部分，前端使用原生JavaScript构建FormData并发送请求，后端使用Express和multer接收并保存文件。

首先是后端代码。我们使用Express框架创建服务器，multer中间件处理文件上传。multer是Express生态中最流行的文件上传处理库，它简化了multipart/form-data的解析过程，提供了便捷的API来获取上传的文件信息。

```javascript
// server.js - 最简文件上传后端
const express = require('express');
const multer = require('multer');
const path = require('path');
const fs = require('fs');

const app = express();
const PORT = 3000;

// 确保上传目录存在
const uploadDir = path.join(__dirname, 'uploads');
if (!fs.existsSync(uploadDir)) {
  fs.mkdirSync(uploadDir, { recursive: true });
}

// 配置 multer 存储选项
const storage = multer.diskStorage({
  // 设置存储路径
  destination: function (req, file, cb) {
    cb(null, uploadDir);
  },
  // 设置文件名格式
  filename: function (req, file, cb) {
    // 使用时间戳 + 原始扩展名，避免文件名冲突
    const uniqueSuffix = Date.now() + '-' + Math.round(Math.random() * 1E9);
    const ext = path.extname(file.originalname);
    cb(null, file.fieldname + '-' + uniqueSuffix + ext);
  }
});

// 创建 multer 实例
const upload = multer({
  storage: storage,
  // 设置文件大小限制为 10MB
  limits: {
    fileSize: 10 * 1024 * 1024
  }
});

// 简单路由：提供上传表单页面
app.get('/', (req, res) => {
  res.send(`
    <!DOCTYPE html>
    <html lang="zh-CN">
    <head>
      <meta charset="UTF-8">
      <title>最简文件上传</title>
      <style>
        body { font-family: Arial, sans-serif; max-width: 600px; margin: 50px auto; padding: 20px; }
        .upload-form { border: 2px dashed #ccc; padding: 30px; text-align: center; border-radius: 10px; }
        input[type="file"] { margin: 15px 0; }
        button { background: #007bff; color: white; border: none; padding: 10px 30px; border-radius: 5px; cursor: pointer; }
        button:hover { background: #0056b3; }
        #result { margin-top: 20px; padding: 15px; border-radius: 5px; }
        .success { background: #d4edda; color: #155724; }
        .error { background: #f8d7da; color: #721c24; }
      </style>
    </head>
    <body>
      <h1>最简文件上传示例</h1>
      <form id="uploadForm" class="upload-form">
        <input type="file" name="file" id="fileInput" required />
        <br>
        <button type="submit">上传文件</button>
      </form>
      <div id="result"></div>
      <script>
        const form = document.getElementById('uploadForm');
        const resultDiv = document.getElementById('result');

        form.addEventListener('submit', async (e) => {
          e.preventDefault();

          const fileInput = document.getElementById('fileInput');
          if (!fileInput.files.length) {
            resultDiv.className = 'error';
            resultDiv.textContent = '请选择文件';
            return;
          }

          // 构建 FormData
          const formData = new FormData();
          formData.append('file', fileInput.files[0]);

          try {
            const response = await fetch('/upload', {
              method: 'POST',
              body: formData
            });

            const result = await response.json();

            if (response.ok) {
              resultDiv.className = 'success';
              resultDiv.innerHTML = '上传成功！<br>文件名：' + result.filename +
                '<br>文件大小：' + result.size + ' 字节';
            } else {
              resultDiv.className = 'error';
              resultDiv.textContent = '上传失败：' + result.message;
            }
          } catch (error) {
            resultDiv.className = 'error';
            resultDiv.textContent = '请求错误：' + error.message;
          }
        });
      </script>
    </body>
    </html>
  `);
});

// 处理文件上传
app.post('/upload', upload.single('file'), (req, res) => {
  // req.file 包含上传文件的信息
  if (!req.file) {
    return res.status(400).json({ message: '没有上传文件' });
  }

  // 返回上传成功信息
  res.json({
    message: '文件上传成功',
    filename: req.file.filename,
    originalName: req.file.originalname,
    size: req.file.size,
    mimetype: req.file.mimetype
  });
});

// 启动服务器
app.listen(PORT, () => {
  console.log('服务器已启动，访问 http://localhost:' + PORT);
});
```

这个示例展示了一个完整的文件上传功能。前端页面包含一个文件选择框和一个提交按钮，用户选择文件后点击提交，JavaScript代码会构建FormData并发送到服务器。后端使用multer处理上传请求，将文件保存到uploads目录，并返回上传结果。

需要安装的依赖：npm install express multer。这个最简示例虽然代码不多，但包含了文件上传的核心流程，是理解更复杂上传方案的基础。在后续章节中，我们将在这个基础上逐步增加文件验证、大文件处理、云存储集成等功能。

## 第二章 Node.js文件接收

### 2.1 multer中间件详解

multer是Express生态中处理文件上传的标准解决方案，它封装了multipart/form-data解析的复杂性，提供了简洁而强大的API。理解multer的工作原理和使用方式，是掌握Node.js文件上传的关键。

multer中间件的核心概念包括storage、limits、fileFilter三个配置项，以及single、array、fields、none、any五个文件处理方法。storage配置决定了上传文件的存储方式，可以选择磁盘存储或内存存储。limits配置用于限制上传文件的大小、请求体的大小等。fileFilter配置用于过滤上传文件的类型。

storage存储选项有两种主要模式：diskStorage和memoryStorage。diskStorage将文件保存到磁盘指定目录，优点是可以处理大文件（不受内存限制），缺点是磁盘I/O速度相对较慢。memoryStorage将文件保存在内存中，优点是处理速度快，缺点是受内存限制，不适合大文件。实际应用中，应根据文件大小和使用场景选择合适的存储方式。

multer的single方法是处理单文件上传最常用的方式。传入表单字段名，multer会解析该字段对应的文件，保存到配置好的存储中，然后在request对象上附加file属性（单文件）或files属性（多文件）。如果上传的是图片，想要立即在浏览器中显示，可以将文件路径返回给前端。

使用multer时还需要注意错误处理。multer会在解析过程中检测各种错误，如文件大小超出限制、内容类型不匹配等。这些错误通过multer的主体错误处理中间件捕获。常见的multer错误包括LIMIT_FILE_SIZE（文件大小超限）、LIMIT_UNEXPECTED_FILE（意外的字段）、LIMIT_PART_COUNT（请求体中的部分数量超限）等。良好的错误处理可以提升用户体验，给用户明确的错误提示。

### 2.2 内存 vs 磁盘存储

选择内存存储还是磁盘存储，是文件上传功能设计时需要首先考虑的问题。这两种存储方式各有优缺点，适用于不同的场景。

内存存储（memoryStorage）将上传的文件内容保存在Buffer对象中，通过req.file.buffer可以访问完整的文件内容。这种方式的优势在于处理速度快，没有磁盘I/O开销，适合处理小文件（几MB以内）。同时，内存中的数据更容易进行后续处理，如图片处理、数据加密等。但内存存储的劣势也很明显：受Node.js进程内存限制，不适合处理大文件；如果忘记清理，内存占用会持续增长。

磁盘存储（diskStorage）将上传的文件写入磁盘文件系统。multer支持高度可配置的存储行为，可以指定存储目录、文件名生成规则等。磁盘存储的优势在于可以处理任意大小的文件（受磁盘空间限制），文件持久化保存，不会因为进程重启而丢失。但磁盘存储也有缺点：磁盘I/O速度相对较慢，特别是机械硬盘；大量小文件会给文件系统带来压力；需要处理文件清理、磁盘空间监控等问题。

对于大多数Web应用来说，磁盘存储是更常见的选择。但这里的"磁盘存储"通常不是直接存储在应用服务器上，而是存储在专门的存储服务（如云存储）中。直接存储在应用服务器上只适合小规模、单机部署的场景。在生产环境中，建议将文件存储到云存储服务（如阿里云OSS、AWS S3等），这样可以解耦应用服务器和存储，实现更好的可扩展性和可靠性。

实际应用中，还可以采用混合策略：小文件使用内存存储以获得更好的性能，大文件直接流式传输到云存储（如AWS S3的 multipart upload），避免占用过多内存。这种策略需要更复杂的实现，但可以优化资源利用。

### 2.3 文件流处理

Node.js的最大优势之一就是强大的流处理能力。文件上传本质上就是数据流的处理过程，理解流的概念和操作方式，对于实现高效、可靠的文件上传功能至关重要。

流是一种抽象的数据处理模式，它代表数据从源到目的地的有序传输。在文件上传场景中，客户端的HTTP请求就是一个可读流，服务器写入的文件是可写流。通过管道（pipe）操作，可以将可读流直接导向可写流，实现边读边写，避免将整个文件加载到内存中。

使用multer时，diskStorage的文件写入也是通过流来完成的。multer会自动创建可写流，将HTTP请求中的文件数据写入磁盘。但有时候我们需要对上传的文件进行特殊处理，如添加水印、转换格式等，这时可以拦截文件流，在保存之前进行处理。

对于大文件处理，流式处理尤为重要。假设要上传一个1GB的视频文件，如果不使用流式处理，就需要将1GB数据全部加载到内存中，这显然是不可行的。而使用流式处理，数据会被分块读取和写入，每块可能只有几KB，这样内存占用就维持在一个稳定的低水平。

Node.js的pipeline API是处理流的高级方式，它会正确处理背压（backpressure）问题，即写入速度跟不上读取速度时自动暂停读取，避免内存溢出。pipeline还会在发生错误时自动清理已创建的资源，防止内存泄漏。在处理文件流时，应优先使用pipeline而不是pipe，因为前者更安全。

### 2.4 实战：multer实战

下面通过一个完整的实战示例，展示multer的多种用法和配置方式。这个示例涵盖单文件上传、多文件上传、条件过滤等常见场景。

```javascript
// comprehensive-upload/server.js - multer 综合实战
const express = require('express');
const multer = require('multer');
const path = require('path');
const fs = require('fs');
const crypto = require('crypto');

const app = express();
const PORT = 3000;

// 创建必要的目录
const dirs = ['uploads', 'uploads/images', 'uploads/documents', 'uploads/archives'];
dirs.forEach(dir => {
  const fullPath = path.join(__dirname, dir);
  if (!fs.existsSync(fullPath)) {
    fs.mkdirSync(fullPath, { recursive: true });
  }
});

// 配置 diskStorage - 详细版本
const storage = multer.diskStorage({
  // destination 可以是字符串或函数
  destination: function (req, file, cb) {
    // 根据文件类型选择存储目录
    let subDir = 'others';
    if (file.mimetype.startsWith('image/')) {
      subDir = 'images';
    } else if (file.mimetype.startsWith('application/pdf') ||
               file.mimetype.startsWith('application/msword') ||
               file.mimetype.startsWith('application/vnd')) {
      subDir = 'documents';
    } else if (file.mimetype.startsWith('application/zip') ||
               file.mimetype.startsWith('application/x-rar')) {
      subDir = 'archives';
    }

    const dest = path.join(__dirname, 'uploads', subDir);
    cb(null, dest);
  },

  // filename 自定义文件名
  filename: function (req, file, cb) {
    // 生成唯一文件名：时间戳-随机数-原始扩展名
    const uniqueSuffix = Date.now() + '-' + Math.round(Math.random() * 1E9);
    const ext = path.extname(file.originalname);
    const baseName = path.basename(file.originalname, ext);

    // 清理原始文件名中的特殊字符
    const cleanName = baseName.replace(/[^a-zA-Z0-9\u4e00-\u9fa5]/g, '_');

    // 组合新文件名
    cb(null, cleanName + '-' + uniqueSuffix + ext);
  }
});

// 文件过滤器 - 根据 MIME 类型过滤
const fileFilter = function (req, file, cb) {
  // 定义允许的 MIME 类型
  const allowedMimes = [
    'image/jpeg',
    'image/png',
    'image/gif',
    'image/webp',
    'image/svg+xml',
    'application/pdf',
    'application/msword',
    'application/vnd.openxmlformats-officedocument.wordprocessingml.document',
    'application/vnd.ms-excel',
    'application/vnd.openxmlformats-officedocument.spreadsheetml.sheet',
    'application/zip',
    'application/x-zip-compressed'
  ];

  if (allowedMimes.includes(file.mimetype)) {
    // 允许上传
    cb(null, true);
  } else {
    // 拒绝上传，创建错误对象
    const err = new Error('不支持的文件类型：' + file.mimetype);
    err.code = 'INVALID_FILE_TYPE';
    cb(err, false);
  }
};

// 创建 multer 实例
const upload = multer({
  storage: storage,
  limits: {
    // 文件大小限制：10MB
    fileSize: 10 * 1024 * 1024,
    // 文件数量限制
    files: 5,
    // 请求体大小限制（包含所有字段和文件）
    fieldSize: 50 * 1024 * 1024
  },
  fileFilter: fileFilter
});

// 错误处理中间件（用于捕获 multer 错误）
const handleMulterError = (err, req, res, next) => {
  if (err instanceof multer.MulterError) {
    // multer 错误
    if (err.code === 'LIMIT_FILE_SIZE') {
      return res.status(400).json({
        success: false,
        message: '文件大小超出限制，最大允许 10MB'
      });
    }
    if (err.code === 'LIMIT_FILE_COUNT') {
      return res.status(400).json({
        success: false,
        message: '文件数量超出限制，最多允许 5 个文件'
      });
    }
    if (err.code === 'INVALID_FILE_TYPE') {
      return res.status(400).json({
        success: false,
        message: err.message
      });
    }
    return res.status(400).json({
      success: false,
      message: '文件上传错误：' + err.message
    });
  }

  if (err) {
    // 其他错误
    return res.status(500).json({
      success: false,
      message: err.message
    });
  }

  next();
};

// ============ 路由定义 ============

// 单文件上传
app.post('/upload/single', upload.single('avatar'), handleMulterError, (req, res) => {
  if (!req.file) {
    return res.status(400).json({ success: false, message: '没有上传文件' });
  }

  res.json({
    success: true,
    message: '上传成功',
    file: {
      filename: req.file.filename,
      originalName: req.file.originalname,
      size: req.file.size,
      mimetype: req.file.mimetype,
      path: req.file.path,
      url: '/uploads/' + path.basename(req.file.path)
    }
  });
});

// 多文件上传（同一字段多个文件）
app.post('/upload/multiple', upload.array('photos', 5), handleMulterError, (req, res) => {
  if (!req.files || req.files.length === 0) {
    return res.status(400).json({ success: false, message: '没有上传文件' });
  }

  const files = req.files.map(file => ({
    filename: file.filename,
    originalName: file.originalname,
    size: file.size,
    mimetype: file.mimetype
  }));

  res.json({
    success: true,
    message: '上传成功，共 ' + files.length + ' 个文件',
    files: files
  });
});

// 多字段上传（不同字段的不同数量文件）
app.post('/upload/fields', upload.fields([
  { name: 'avatars', maxCount: 3 },
  { name: 'documents', maxCount: 5 }
]), handleMulterError, (req, res) => {
  const result = {};

  if (req.files.avatars) {
    result.avatars = req.files.avatars.map(file => ({
      filename: file.filename,
      size: file.size
    }));
  }

  if (req.files.documents) {
    result.documents = req.files.documents.map(file => ({
      filename: file.filename,
      size: file.size
    }));
  }

  res.json({
    success: true,
    message: '上传成功',
    files: result
  });
});

// 访问上传的文件
app.use('/uploads', express.static(path.join(__dirname, 'uploads')));

// 启动服务器
app.listen(PORT, () => {
  console.log('服务器已启动，访问 http://localhost:' + PORT);
});
```

这个综合示例展示了multer的多种配置和使用方式。storage配置部分展示了如何根据文件类型动态选择存储目录，如何生成安全的文件名。fileFilter部分展示了如何验证文件类型，拒绝不支持的文件格式。upload.single、upload.array、upload.fields分别对应不同的上传场景。

错误处理是生产环境中的重要环节。multer的错误通过特定的方式传递给Express，需要专门的错误处理中间件来捕获和格式化。本示例中的handleMulterError函数展示了如何区分和处理不同类型的multer错误，提供了友好的错误信息给客户端。

## 第三章 文件类型验证

### 3.1 扩展名验证：不安全

文件扩展名验证是最简单、最直接的文件类型判断方式，但同时也是最不安全的方式。恶意用户可以轻易伪造文件扩展名，将可执行文件伪装成图片或文档上传到服务器。

扩展名验证的基本原理是检查文件名的最后一部分是否属于允许的扩展名列表。例如，允许上传jpg、png、gif格式的图片，就检查文件扩展名是否为这些值之一。这种验证方式简单高效，可以快速过滤掉明显不合法的文件，减少后续处理的开销。

然而，扩展名验证存在严重的安全隐患。首先，文件扩展名只是文件名的一部分，可以随意修改。将恶意文件改名为example.jpg再上传，服务器仅通过扩展名验证是无法检测的。其次，某些系统漏洞可能允许绕过扩展名验证，如利用特殊的文件命名技巧（如在末尾添加多个扩展名、利用空字符截断等）。第三，扩展名验证不能确保文件的实际内容与扩展名匹配，一张改名为jpg的病毒文件仍然会被接受。

真正危险的场景是服务器会执行或提供用户上传的文件。如果上传的PHP文件、JavaScript文件或其他可执行文件被服务器以脚本方式执行，或者被其他用户直接访问，就可能导致严重的安全问题。即使是看似无害的图片文件，如果服务器存在解析漏洞（如某些图片处理库的缓冲区溢出漏洞），也可能被利用来执行恶意代码。

因此，扩展名验证应该被视为第一道粗筛，用于快速过滤明显不合法的请求，但绝对不能作为唯一的安全验证手段。实际的系统应该采用多层次的验证策略，包括后续将要介绍的MIME类型验证和魔数验证。

### 3.2 MIME类型验证

MIME类型（Multipurpose Internet Mail Extensions）是互联网标准中用于表示文件类型的标识符。当浏览器上传文件时，会在Content-Type头部中包含文件的MIME类型。服务器可以通过检查这个MIME类型来初步判断文件类型。

MIME类型验证比扩展名验证稍微可靠一些，因为MIME类型是由浏览器根据文件内容自动检测的，理论上应该与文件的实际类型一致。但MIME类型验证同样存在被伪造的可能。攻击者可以通过设置请求头或使用工具构造任意MIME类型的请求，绕过这一验证。

在Node.js中，multer会自动从请求中提取MIME类型信息。对于普通字段，req.body中包含的是文本值；对于文件字段，req.file中包含文件的元信息，其中mimetype字段就是MIME类型。通过检查这个字段，可以实现MIME类型验证。

MIME类型验证的实现方式有两种：白名单和黑名单。白名单方式只允许已知安全的MIME类型，如只允许图片类型image/jpeg、image/png等，拒绝其他所有类型。黑名单方式拒绝已知危险的MIME类型，如application/javascript、text/html等，允许其他类型。黑名单方式安全性较低，因为可能有新的危险类型不断出现。

在实际应用中，MIME类型验证应该结合其他验证手段一起使用。可以通过multer的fileFilter选项在文件处理之前进行MIME类型检查，拒绝不合法的文件。同时，应该设置适当的文件大小限制，防止大文件攻击。

### 3.3 魔数验证：文件头

魔数（Magic Number）是文件开头的一段特定字节序列，用于标识文件的真实格式。每种文件格式都有其独特的魔数，图片、音频、视频、文档等都有各自的标准。例如，PNG文件的魔数是89 50 4E 47 0D 0A 1A 0A（十六进制），JPEG文件以FF D8 FF开头，PDF文件以25 50 44 46开头。

魔数验证是目前最可靠的文件类型检测方法。通过读取文件开头的若干字节并与已知的魔数进行比对，可以准确判断文件的真实格式，而不管文件扩展名或MIME类型如何设置。这种方法即使攻击者修改了文件扩展名或请求头，也无法欺骗服务器。

魔数验证的实现需要使用文件流读取。Node.js中可以使用fs.readFile或创建可读流来读取文件头部。对于大文件，不需要读取整个文件，只需要读取前几个字节即可，这使得验证过程非常高效。

下面是一个完整的魔数验证实现示例：

```javascript
// magic-number-validator.js - 魔数验证实现
const fs = require('fs');
const path = require('path');

// 常见文件格式的魔数定义
const MAGIC_NUMBERS = {
  // 图片格式
  'image/jpeg': {
    bytes: [0xFF, 0xD8, 0xFF],
    offset: 0,
    description: 'JPEG图片'
  },
  'image/png': {
    bytes: [0x89, 0x50, 0x4E, 0x47, 0x0D, 0x0A, 0x1A, 0x0A],
    offset: 0,
    description: 'PNG图片'
  },
  'image/gif': {
    bytes: [0x47, 0x49, 0x46, 0x38],
    offset: 0,
    description: 'GIF图片'
  },
  'image/webp': {
    bytes: [0x52, 0x49, 0x46, 0x46], // RIFF header
    offset: 0,
    verify: (buffer) => {
      // WEBP 在 RIFF 之后是 WEBP，然后是文件大小（4字节），然后是 WEBP 标识
      return buffer.slice(8, 12).toString() === 'WEBP';
    },
    description: 'WEBP图片'
  },
  'image/bmp': {
    bytes: [0x42, 0x4D],
    offset: 0,
    description: 'BMP图片'
  },
  'image/tiff': {
    bytes: [0x49, 0x49, 0x2A, 0x00], // 小端序
    offset: 0,
    description: 'TIFF图片（小端）'
  },
  'image/tiff': {
    bytes: [0x4D, 0x4D, 0x00, 0x2A], // 大端序
    offset: 0,
    description: 'TIFF图片（大端）'
  },

  // 文档格式
  'application/pdf': {
    bytes: [0x25, 0x50, 0x44, 0x46],
    offset: 0,
    description: 'PDF文档'
  },
  'application/zip': {
    bytes: [0x50, 0x4B, 0x03, 0x04],
    offset: 0,
    description: 'ZIP压缩文件'
  },
  'application/x-rar-compressed': {
    bytes: [0x52, 0x61, 0x72, 0x21],
    offset: 0,
    description: 'RAR压缩文件'
  },

  // 视频格式
  'video/mp4': {
    bytes: [0x00, 0x00, 0x00, 0x18, 0x66, 0x74, 0x79, 0x70], // ftyp
    offset: 4,
    description: 'MP4视频'
  },
  'video/x-msvideo': {
    bytes: [0x52, 0x49, 0x46, 0x46], // RIFF
    offset: 0,
    verify: (buffer) => {
      return buffer.slice(8, 12).toString() === 'AVI ';
    },
    description: 'AVI视频'
  },

  // 音频格式
  'audio/wav': {
    bytes: [0x52, 0x49, 0x46, 0x46], // RIFF
    offset: 0,
    verify: (buffer) => {
      return buffer.slice(8, 12).toString() === 'WAVE';
    },
    description: 'WAV音频'
  },
  'audio/mpeg': {
    bytes: [0xFF, 0xFB],
    offset: 0,
    description: 'MP3音频'
  }
};

/**
 * 读取文件的魔数
 * @param {string} filePath - 文件路径
 * @param {number} bytesToRead - 读取的字节数，默认32字节
 * @returns {Promise<Buffer>} - 文件头数据
 */
async function readFileHeader(filePath, bytesToRead = 32) {
  return new Promise((resolve, reject) => {
    const buffer = Buffer.alloc(bytesToRead);
    fs.open(filePath, 'r', (err, fd) => {
      if (err) {
        reject(err);
        return;
      }
      fs.read(fd, buffer, 0, bytesToRead, 0, (err, bytesRead) => {
        fs.close(fd, () => {});
        if (err) {
          reject(err);
          return;
        }
        resolve(buffer.slice(0, bytesRead));
      });
    });
  });
}

/**
 * 验证文件魔数
 * @param {Buffer} header - 文件头数据
 * @param {object} magicInfo - 魔数定义
 * @returns {boolean} - 是否匹配
 */
function verifyMagicNumber(header, magicInfo) {
  const { bytes, offset = 0, verify } = magicInfo;

  // 检查基础魔数匹配
  for (let i = 0; i < bytes.length; i++) {
    if (header[offset + i] !== bytes[i]) {
      return false;
    }
  }

  // 如果有额外的验证函数，执行验证
  if (verify) {
    return verify(header);
  }

  return true;
}

/**
 * 检测文件真实类型
 * @param {string} filePath - 文件路径
 * @returns {Promise<object>} - 检测结果
 */
async function detectFileType(filePath) {
  const header = await readFileHeader(filePath);

  for (const [mimeType, magicInfo] of Object.entries(MAGIC_NUMBERS)) {
    if (verifyMagicNumber(header, magicInfo)) {
      return {
        detected: true,
        mimeType: mimeType,
        description: magicInfo.description
      };
    }
  }

  return {
    detected: false,
    mimeType: 'application/octet-stream',
    description: '未知类型'
  };
}

/**
 * 完整文件验证
 * @param {string} filePath - 文件路径
 * @param {string[]} allowedTypes - 允许的MIME类型数组
 * @returns {Promise<object>} - 验证结果
 */
async function validateFile(filePath, allowedTypes = []) {
  const typeInfo = await detectFileType(filePath);

  // 如果启用了允许列表，检查是否在允许范围内
  if (allowedTypes.length > 0) {
    if (!allowedTypes.includes(typeInfo.mimeType)) {
      return {
        valid: false,
        reason: '文件类型不在允许列表中',
        detectedType: typeInfo.mimeType,
        allowedTypes: allowedTypes
      };
    }
  }

  return {
    valid: true,
    mimeType: typeInfo.mimeType,
    description: typeInfo.description
  };
}

// 导出函数
module.exports = {
  MAGIC_NUMBERS,
  readFileHeader,
  verifyMagicNumber,
  detectFileType,
  validateFile
};
```

这个魔数验证模块定义了常见文件格式的魔数，并提供了检测和验证功能。魔数定义包含了基本的字节序列、偏移量（某些格式的魔数不在文件开头）和可选的验证函数（用于需要二次确认的格式如WEBP和AVI）。

在实际应用中，可以将这个模块集成到multer的fileFilter中，在文件保存之前先验证文件的真实类型。如果魔数验证不通过，则拒绝上传并返回错误信息。

### 3.4 实战：完整验证

将扩展名验证、MIME类型验证和魔数验证结合起来，构建一个完整的文件验证方案。这个实战示例展示了如何在实际项目中实现多层文件验证。

```javascript
// complete-validator.js - 完整文件验证方案
const fs = require('fs');
const path = require('path');
const { MAGIC_NUMBERS, detectFileType, validateFile } = require('./magic-number-validator');

/**
 * 完整文件验证器
 * 实现三层验证：扩展名 -> MIME类型 -> 魔数
 */
class FileValidator {
  constructor(options = {}) {
    // 允许的文件类型映射
    this.allowedTypes = new Map([
      ['image/jpeg', { ext: ['.jpg', '.jpeg'], description: 'JPEG图片' }],
      ['image/png', { ext: ['.png'], description: 'PNG图片' }],
      ['image/gif', { ext: ['.gif'], description: 'GIF图片' }],
      ['image/webp', { ext: ['.webp'], description: 'WEBP图片' }],
      ['application/pdf', { ext: ['.pdf'], description: 'PDF文档' }],
      ['application/zip', { ext: ['.zip'], description: 'ZIP压缩包' }]
    ]);

    // 文件大小限制
    this.maxFileSize = options.maxFileSize || 10 * 1024 * 1024; // 默认10MB

    // 是否严格模式（同时验证MIME和魔数）
    this.strictMode = options.strictMode !== false;
  }

  /**
   * 第一层验证：扩展名
   */
  validateExtension(filename) {
    const ext = path.extname(filename).toLowerCase();
    for (const [mimeType, info] of this.allowedTypes.entries()) {
      if (info.ext.includes(ext)) {
        return {
          valid: true,
          expectedMimeType: mimeType,
          extension: ext
        };
      }
    }
    return {
      valid: false,
      message: '不支持的文件扩展名：' + ext,
      allowedExtensions: this.getAllExtensions()
    };
  }

  /**
   * 第二层验证：MIME类型
   */
  validateMimeType(mimeType) {
    if (this.allowedTypes.has(mimeType)) {
      return {
        valid: true,
        mimeType: mimeType
      };
    }
    return {
      valid: false,
      message: '不支持的MIME类型：' + mimeType,
      allowedMimeTypes: Array.from(this.allowedTypes.keys())
    };
  }

  /**
   * 第三层验证：魔数（文件头）
   */
  async validateMagicNumber(filePath) {
    try {
      const result = await detectFileType(filePath);
      if (result.detected && this.allowedTypes.has(result.mimeType)) {
        return {
          valid: true,
          detectedType: result.mimeType,
          description: result.description
        };
      }
      return {
        valid: false,
        message: '文件内容与声明类型不匹配',
        detectedType: result.mimeType,
        description: result.description
      };
    } catch (error) {
      return {
        valid: false,
        message: '魔数验证失败：' + error.message
      };
    }
  }

  /**
   * 完整验证流程
   */
  async validate(filePath, originalName, declaredMimeType) {
    const errors = [];

    // 第一层：扩展名验证
    const extResult = this.validateExtension(originalName);
    if (!extResult.valid) {
      errors.push(extResult.message);
    }

    // 第二层：MIME类型验证
    const mimeResult = this.validateMimeType(declaredMimeType);
    if (!mimeResult.valid) {
      errors.push(mimeResult.message);
    }

    // 如果基础验证失败，直接返回
    if (errors.length > 0) {
      return {
        valid: false,
        errors: errors,
        passedLevels: 0
      };
    }

    // 第三层：魔数验证（严格模式）
    if (this.strictMode) {
      const magicResult = await this.validateMagicNumber(filePath);
      if (!magicResult.valid) {
        return {
          valid: false,
          errors: [magicResult.message],
          passedLevels: 2,
          detectedType: magicResult.detectedType
        };
      }
    }

    return {
      valid: true,
      passedLevels: this.strictMode ? 3 : 2,
      mimeType: extResult.expectedMimeType
    };
  }

  /**
   * 检查文件大小
   */
  validateFileSize(filePath) {
    try {
      const stats = fs.statSync(filePath);
      if (stats.size > this.maxFileSize) {
        return {
          valid: false,
          message: '文件大小超出限制（最大 ' + this.formatSize(this.maxFileSize) + '）',
          actualSize: stats.size,
          maxSize: this.maxFileSize
        };
      }
      return {
        valid: true,
        size: stats.size
      };
    } catch (error) {
      return {
        valid: false,
        message: '无法读取文件大小：' + error.message
      };
    }
  }

  /**
   * 获取所有允许的扩展名
   */
  getAllExtensions() {
    const extensions = new Set();
    for (const info of this.allowedTypes.values()) {
      info.ext.forEach(ext => extensions.add(ext));
    }
    return Array.from(extensions);
  }

  /**
   * 格式化文件大小
   */
  formatSize(bytes) {
    if (bytes < 1024) return bytes + ' B';
    if (bytes < 1024 * 1024) return (bytes / 1024).toFixed(2) + ' KB';
    if (bytes < 1024 * 1024 * 1024) return (bytes / (1024 * 1024)).toFixed(2) + ' MB';
    return (bytes / (1024 * 1024 * 1024)).toFixed(2) + ' GB';
  }
}

// 集成到 Express 路由
const express = require('express');
const multer = require('multer');
const { v4: uuidv4 } = require('uuid');
const path = require('path');

const app = express();
const PORT = 3000;

// 创建验证器实例
const validator = new FileValidator({
  maxFileSize: 10 * 1024 * 1024, // 10MB
  strictMode: true
});

// 配置 multer（暂时不设置 fileFilter，在路由中手动验证）
const storage = multer.diskStorage({
  destination: './uploads',
  filename: (req, file, cb) => {
    const uniqueName = uuidv4() + path.extname(file.originalname);
    cb(null, uniqueName);
  }
});

const upload = multer({
  storage: storage,
  limits: { fileSize: 10 * 1024 * 1024 }
});

// 安全文件上传路由
app.post('/upload/secure', upload.single('file'), async (req, res) => {
  if (!req.file) {
    return res.status(400).json({ success: false, message: '没有上传文件' });
  }

  const filePath = req.file.path;

  try {
    // 完整验证
    const result = await validator.validate(
      filePath,
      req.file.originalname,
      req.file.mimetype
    );

    if (!result.valid) {
      // 验证失败，删除已上传的文件
      fs.unlinkSync(filePath);
      return res.status(400).json({
        success: false,
        message: '文件验证失败',
        errors: result.errors,
        passedLevels: result.passedLevels + '/3'
      });
    }

    // 验证通过
    res.json({
      success: true,
      message: '文件上传并验证通过',
      file: {
        id: path.basename(filePath, path.extname(filePath)),
        originalName: req.file.originalname,
        mimeType: result.mimeType,
        size: req.file.size,
        storedAs: req.file.filename
      }
    });

  } catch (error) {
    // 发生错误，清理文件
    if (fs.existsSync(filePath)) {
      fs.unlinkSync(filePath);
    }
    res.status(500).json({
      success: false,
      message: '处理文件时发生错误：' + error.message
    });
  }
});

app.listen(PORT, () => {
  console.log('安全上传服务器已启动');
});

module.exports = { FileValidator };
```

这个完整验证方案实现了三层验证机制：第一层检查文件扩展名，快速过滤明显不合法的文件；第二层检查MIME类型，确保请求中声明的文件类型是允许的；第三层通过魔数验证文件的真实内容，即使攻击者伪造了扩展名和MIME类型，也无法绕过魔数验证。这种多层防御策略可以大大提高文件上传的安全性。

## 第四章 文件大小限制

### 4.1 请求体大小限制

在文件上传功能中，请求体大小限制是一个非常重要的安全配置。没有限制的请求体大小可能导致恶意用户上传超大文件，耗尽服务器磁盘空间或内存，使服务不可用。

Express框架使用body-parser中间件解析请求体，multer也提供了limits配置选项来限制上传文件的大小。正确配置这些限制可以防止多种类型的攻击：大文件攻击（上传超大文件耗尽资源）、多次小文件攻击（上传大量小文件累积占用空间）、请求体过大导致解析失败等。

请求体大小限制需要在多个层面设置。在nginx反向代理层面，需要设置client_max_body_size参数，否则大文件请求可能在到达Express之前就被nginx拒绝。在Express层面，可以通过express.json()和express.urlencoded()的limit选项设置解析器的限制。在multer层面，通过limits.fileSize设置单个文件的大小限制，通过limits.fieldSize设置整个请求体的大小限制。

合理的大小限制应该根据实际业务需求来设定。头像图片通常限制在2MB以内，文档类文件（如PDF、Word）通常限制在10-20MB，压缩包或视频可能需要更大的限制，但一般也不应超过100MB。如果业务确实需要处理超大文件（如视频编辑平台），则需要采用分片上传方案，而不是简单地增加限制。

### 4.2 文件大小限制

文件大小限制是保护服务器资源的重要手段。multer提供了直观易用的fileSize限制选项，可以限制单个文件的最大大小。这个配置与请求体大小限制相辅相成，共同构成完整的文件大小保护体系。

multer的fileSize限制以字节为单位，接受一个数字值。在前面的示例中，我们设置了10 * 1024 * 1024，即10MB。当上传的文件超过这个限制时，multer会触发LIMIT_FILE_SIZE错误，这个错误可以在错误处理中间件中被捕获和处理。

除了multer本身的限制，还可以在更底层的HTTP服务器层面设置请求大小限制。Node.js的http模块创建的服务器可以接收express实例作为请求处理函数，但底层的请求解析仍然受服务器配置影响。不过通常情况下，使用nginx等反向代理已经在网络层面做了限制，Express层面的配置更多是作为应用层的最终保护。

实现时应该同时设置多个层面的限制：nginx的反向代理配置作为第一道防线，Express的body-parser配置作为第二道防线，multer的fileSize限制作为最后一道防线。这种纵深防御策略可以应对不同层次的攻击，同时也能提供更友好的错误提示。

### 4.3 进度监控

对于大文件上传来说，进度监控是提升用户体验的关键功能。用户需要知道文件正在上传、已经完成了多少、预计还需要多长时间。没有进度反馈的上传功能会让用户感到焦虑，特别是上传较大的文件时。

在传统的前端方案中，XMLHttpRequest原生支持上传进度事件。通过监听xhr.upload的progress事件，可以获取已上传的字节数和总字节数，从而计算上传进度。Fetch API在进度监控方面支持较差，虽然可以通过ReadableStream实现进度监控，但目前浏览器支持还不完善。因此，如果需要精确的进度反馈，推荐使用XMLHttpRequest。

前端进度监控的实现需要创建XMLHttpRequest对象，手动设置请求头，然后监听upload.onprogress事件。事件回调函数接收一个ProgressEvent对象，其loaded属性表示已上传的字节数，total属性表示文件的总字节数。通过这两个值可以计算出进度百分比。

下面是一个完整的进度监控实现示例：

```javascript
// upload-with-progress.js - 带进度监控的文件上传

/**
 * 带进度监控的文件上传函数
 * @param {string} url - 上传地址
 * @param {File} file - 文件对象
 * @param {Function} onProgress - 进度回调函数
 * @returns {Promise<object>} - 上传结果
 */
function uploadWithProgress(url, file, onProgress) {
  return new Promise((resolve, reject) => {
    const xhr = new XMLHttpRequest();

    // 监听上传进度
    xhr.upload.addEventListener('progress', (event) => {
      if (event.lengthComputable) {
        const percentComplete = Math.round((event.loaded / event.total) * 100);
        const uploadedMB = (event.loaded / (1024 * 1024)).toFixed(2);
        const totalMB = (event.total / (1024 * 1024)).toFixed(2);

        onProgress({
          percent: percentComplete,
          loaded: event.loaded,
          total: event.total,
          uploadedMB: uploadedMB,
          totalMB: totalMB
        });
      }
    });

    // 监听请求完成
    xhr.addEventListener('load', () => {
      if (xhr.status >= 200 && xhr.status < 300) {
        try {
          resolve(JSON.parse(xhr.responseText));
        } catch {
          resolve(xhr.responseText);
        }
      } else {
        reject(new Error('上传失败：' + xhr.status + ' ' + xhr.statusText));
      }
    });

    // 监听错误
    xhr.addEventListener('error', () => {
      reject(new Error('网络错误'));
    });

    // 监听超时
    xhr.addEventListener('timeout', () => {
      reject(new Error('上传超时'));
    });

    // 设置超时时间（30秒）
    xhr.timeout = 30000;

    // 打开连接并发送请求
    xhr.open('POST', url);

    // 创建 FormData 并添加文件
    const formData = new FormData();
    formData.append('file', file);

    // 发送请求
    xhr.send(formData);
  });
}

// 使用示例
const uploadBtn = document.getElementById('uploadBtn');
const progressBar = document.getElementById('progressBar');
const progressText = document.getElementById('progressText');
const fileInput = document.getElementById('fileInput');

uploadBtn.addEventListener('click', async () => {
  const file = fileInput.files[0];
  if (!file) {
    alert('请选择文件');
    return;
  }

  try {
    const result = await uploadWithProgress(
      '/upload',
      file,
      (progress) => {
        // 更新进度条
        progressBar.style.width = progress.percent + '%';
        progressBar.textContent = progress.percent + '%';
        progressText.textContent =
          `已上传：${progress.uploadedMB} MB / ${progress.totalMB} MB`;
      }
    );

    console.log('上传成功:', result);
    alert('上传成功！');

  } catch (error) {
    console.error('上传失败:', error);
    alert('上传失败：' + error.message);
  }
});
```

服务端也需要支持断点续传和进度记录。后端应该能够处理Range请求头，返回已上传的分片信息，以及在上传过程中记录进度到数据库或Redis。这些内容将在后续的断点续传章节详细讲解。

### 4.4 实战：大文件处理

处理大文件上传需要前后端的紧密配合。前端需要实现分片上传逻辑，后端需要支持分片接收和合并。本节先介绍基本的大文件处理方案，分片上传的完整实现在后续章节详细讲解。

大文件处理的核心挑战是内存管理。如果将整个大文件加载到内存中再处理，可能会导致内存溢出。正确的做法是使用流式处理，边读边写，保持恒定的内存占用。

```javascript
// large-file-handler.js - 大文件处理示例

const express = require('express');
const multer = require('multer');
const fs = require('fs');
const path = require('path');
const cors = require('cors');

const app = express();
const PORT = 3000;

app.use(cors());
app.use(express.json());

// 大文件存储目录
const LARGE_FILES_DIR = path.join(__dirname, 'large-uploads');
if (!fs.existsSync(LARGE_FILES_DIR)) {
  fs.mkdirSync(LARGE_FILES_DIR, { recursive: true });
}

// 临时文件存储目录（用于分片上传）
const TEMP_DIR = path.join(__dirname, 'temp-uploads');
if (!fs.existsSync(TEMP_DIR)) {
  fs.mkdirSync(TEMP_DIR, { recursive: true });
}

// 配置 multer 处理大文件
// 注意：这里不限制文件大小，由分片逻辑控制
const storage = multer.diskStorage({
  destination: (req, file, cb) => {
    // 为每个上传任务创建临时目录
    const uploadId = req.body.uploadId || 'default';
    const tempPath = path.join(TEMP_DIR, uploadId);
    if (!fs.existsSync(tempPath)) {
      fs.mkdirSync(tempPath, { recursive: true });
    }
    cb(null, tempPath);
  },
  filename: (req, file, cb) => {
    // 使用分片索引作为文件名，便于排序和合并
    const chunkIndex = req.body.chunkIndex || 0;
    const ext = path.extname(file.originalname);
    cb(null, 'chunk-' + chunkIndex + ext);
  }
});

const upload = multer({ storage });

/**
 * 初始化大文件上传
 * 创建上传记录和临时目录
 */
app.post('/upload/init', (req, res) => {
  const { filename, fileSize, chunkSize = 5 * 1024 * 1024, mimeType } = req.body;

  if (!filename || !fileSize) {
    return res.status(400).json({
      success: false,
      message: '缺少必要参数：filename, fileSize'
    });
  }

  // 生成唯一上传ID
  const uploadId = Date.now() + '-' + Math.random().toString(36).substr(2, 9);

  // 计算分片数量
  const totalChunks = Math.ceil(fileSize / chunkSize);

  // 创建临时目录
  const tempPath = path.join(TEMP_DIR, uploadId);
  if (!fs.existsSync(tempPath)) {
    fs.mkdirSync(tempPath, { recursive: true });
  }

  // 保存上传元信息
  const uploadMeta = {
    uploadId,
    filename,
    fileSize,
    chunkSize,
    totalChunks,
    mimeType,
    uploadedChunks: [],
    createdAt: new Date().toISOString()
  };

  // 可以将元信息保存到Redis或数据库
  // 这里简化为保存到文件
  fs.writeFileSync(
    path.join(tempPath, 'meta.json'),
    JSON.stringify(uploadMeta, null, 2)
  );

  res.json({
    success: true,
    uploadId,
    totalChunks,
    chunkSize
  });
});

/**
 * 上传分片
 */
app.post('/upload/chunk',
  upload.array('chunk', 1),
  (req, res) => {
    if (!req.files || req.files.length === 0) {
      return res.status(400).json({
        success: false,
        message: '没有上传分片'
      });
    }

    const { uploadId, chunkIndex } = req.body;

    if (!uploadId || chunkIndex === undefined) {
      return res.status(400).json({
        success: false,
        message: '缺少必要参数：uploadId, chunkIndex'
      });
    }

    const tempPath = path.join(TEMP_DIR, uploadId);
    const metaPath = path.join(tempPath, 'meta.json');

    if (!fs.existsSync(metaPath)) {
      return res.status(400).json({
        success: false,
        message: '上传ID不存在或已过期'
      });
    }

    // 更新上传记录
    const meta = JSON.parse(fs.readFileSync(metaPath, 'utf8'));
    meta.uploadedChunks.push(parseInt(chunkIndex));

    fs.writeFileSync(metaPath, JSON.stringify(meta, null, 2));

    res.json({
      success: true,
      message: '分片上传成功',
      chunkIndex: parseInt(chunkIndex),
      uploadedChunks: meta.uploadedChunks.length,
      totalChunks: meta.totalChunks
    });
  }
);

/**
 * 合并分片
 */
app.post('/upload/merge', async (req, res) => {
  const { uploadId } = req.body;

  if (!uploadId) {
    return res.status(400).json({
      success: false,
      message: '缺少uploadId'
    });
  }

  const tempPath = path.join(TEMP_DIR, uploadId);
  const metaPath = path.join(tempPath, 'meta.json');

  if (!fs.existsSync(metaPath)) {
    return res.status(400).json({
      success: false,
      message: '上传ID不存在'
    });
  }

  const meta = JSON.parse(fs.readFileSync(metaPath, 'utf8'));

  // 检查是否所有分片都已上传
  if (meta.uploadedChunks.length !== meta.totalChunks) {
    const missingChunks = [];
    for (let i = 0; i < meta.totalChunks; i++) {
      if (!meta.uploadedChunks.includes(i)) {
        missingChunks.push(i);
      }
    }
    return res.status(400).json({
      success: false,
      message: '还有分片未上传',
      missingChunks,
      uploadedChunks: meta.uploadedChunks.length,
      totalChunks: meta.totalChunks
    });
  }

  // 合并文件
  const finalPath = path.join(LARGE_FILES_DIR, meta.filename);
  const writeStream = fs.createWriteStream(finalPath);

  // 按顺序读取并写入每个分片
  for (let i = 0; i < meta.totalChunks; i++) {
    const chunkPath = path.join(tempPath, 'chunk-' + i + path.extname(meta.filename));
    const chunkData = fs.readFileSync(chunkPath);
    writeStream.write(chunkData);

    // 删除分片文件
    fs.unlinkSync(chunkPath);
  }

  writeStream.end();

  // 等待写入完成
  await new Promise((resolve, reject) => {
    writeStream.on('finish', resolve);
    writeStream.on('error', reject);
  });

  // 删除临时目录和元信息
  fs.rmSync(tempPath, { recursive: true, force: true });

  res.json({
    success: true,
    message: '文件合并成功',
    file: {
      filename: meta.filename,
      size: meta.fileSize,
      path: finalPath
    }
  });
});

/**
 * 获取上传进度
 */
app.get('/upload/status/:uploadId', (req, res) => {
  const { uploadId } = req.params;
  const metaPath = path.join(TEMP_DIR, uploadId, 'meta.json');

  if (!fs.existsSync(metaPath)) {
    return res.status(404).json({
      success: false,
      message: '上传任务不存在'
    });
  }

  const meta = JSON.parse(fs.readFileSync(metaPath, 'utf8'));

  res.json({
    success: true,
    uploadId,
    filename: meta.filename,
    fileSize: meta.fileSize,
    uploadedChunks: meta.uploadedChunks.length,
    totalChunks: meta.totalChunks,
    percent: Math.round((meta.uploadedChunks.length / meta.totalChunks) * 100)
  });
});

app.listen(PORT, () => {
  console.log('大文件上传服务器已启动，端口：' + PORT);
});
```

这个示例展示了一个基本的大文件分片上传方案。核心思路是将大文件分割成多个小块（分片），分别上传每个分片，然后在所有分片上传完成后按顺序合并成完整文件。这种方案可以有效解决大文件上传的内存问题，并且为后续实现断点续传提供了基础。

## 第五章 存储方案

### 5.1 本地存储

本地存储是最简单直接的文件存储方式，适用于单机部署的小型应用或开发测试环境。将上传的文件直接保存到应用服务器的磁盘上，通过静态文件服务暴露访问路径。

本地存储的优势在于实现简单、部署快捷。不需要额外的云服务配置，不需要理解复杂的API，直接使用文件系统API即可完成存储。调试也非常方便，可以直接在服务器上查看、修改、删除文件。但本地存储的缺点也很明显：不适合分布式部署，在多服务器环境下需要共享存储；难以扩展，磁盘空间受限于单台服务器；难以管理，无法统一管理备份、CDN分发等。

实现本地存储时，目录结构的设计很重要。常见的做法是按日期或类型组织文件，如uploads/2024/03/15/目录下存放当天的上传文件。文件名应该使用唯一的标识符（如UUID或时间戳加随机数），避免文件名冲突。还需要注意处理文件名中的特殊字符，防止路径遍历攻击。

本地存储还需要考虑备份和清理策略。重要的上传文件应该定期备份，临时文件（如未完成的上传）应该设置过期时间自动清理。可以使用cron任务或node-cron库来实现定时清理任务。

### 5.2 阿里云OSS

阿里云对象存储（OSS）是国内最流行的云存储服务之一，提供海量、安全、高可靠的云端存储服务。OSS支持标准RESTful API，与各种应用无缝集成，是生产环境中存储用户上传文件的首选方案之一。

阿里云OSS的主要优势包括：海量存储空间，按需付费，无需容量规划；高可靠性，数据多副本冗余存储，承诺99.999999999%的数据可靠性；高可用性，自动容灾切换，保证服务可用性；全球加速，支持CDN加速访问；丰富的数据处理能力，图片缩略、水印、转码等。

OSS的基本概念包括Bucket（存储桶）和Object（对象）。Bucket是存储的容器，类似于顶级目录，每个Bucket有全局唯一的名称。Object是存储的具体文件，由Key（键）和Data（数据）组成，Key是文件的唯一标识。

集成阿里云OSS需要使用官方提供的SDK。OSS SDK提供了丰富的API，包括上传文件（简单上传、分片上传、表单上传）、下载文件、复制文件、删除文件等。对于Web前端，还可以生成预签名URL，让用户直接上传到OSS，而不需要经过自己的服务器中转。

### 5.3 AWS S3

AWS S3（Simple Storage Service）是亚马逊云服务提供的对象存储服务，是云计算领域最经典、最广泛使用的存储服务之一。S3的耐久性和可用性都非常高，标准存储承诺99.999999999%的对象耐久性和99.99%的可用性。

AWS S3同样使用Bucket和Object的概念，与阿里云OSS非常相似。S3的存储类别（Storage Classes）非常丰富，包括标准存储、不频繁访问存储 Glacier归档存储等，可以根据数据的访问频率优化成本。S3的生命周期策略可以自动将数据在不同存储类别之间转移，进一步优化成本。

AWS S3与各种AWS服务有深度集成，如CloudFront CDN、Lambda计算服务等。可以很方便地构建完整的文件处理流水线，如图片上传后自动触发Lambda进行缩略图生成。

AWS S3的SDK同样支持Node.js、Python、Java等多种语言，前端也可以通过预签名URL实现直传。S3的IAM策略提供了细粒度的权限控制，可以精确控制每个用户对每个Bucket的访问权限。

### 5.4 七牛云

七牛云是国内另一家主流的云存储服务商，以CDN加速能力强、功能丰富著称。七牛云的存储服务与CDN深度整合，对于需要加速访问的文件（如图片、音视频），七牛云是很好的选择。

七牛云提供了简单直观的API和丰富的SDK支持。其上传策略（PutPolicy）机制允许服务端生成上传凭证，前端凭此凭证直接上传到七牛云存储，无需经过自己的服务器。这种设计可以大幅减轻服务器负担，提升上传速度。

七牛云还提供了强大的音视频处理能力，如转码、截图、水印等。如果应用涉及大量音视频内容，七牛云的一体化解决方案可以简化架构。

### 5.5 实战：OSS集成

下面通过完整的代码示例，展示如何将阿里云OSS集成到Node.js应用中，实现文件上传、下载和管理功能。

```javascript
// oss-upload.js - 阿里云OSS集成示例

const OSS = require('ali-oss');
const path = require('path');
const fs = require('fs');

/**
 * 阿里云OSS客户端封装
 */
class OSSClient {
  constructor(options) {
    // 阿里云OSS配置
    this.client = new OSS({
      region: options.region,           // 如：oss-cn-beijing
      accessKeyId: options.accessKeyId,
      accessKeySecret: options.accessKeySecret,
      bucket: options.bucket,            // 存储桶名称
      // 可选：使用cname自定义域名
      cname: options.cname,
      // 可选：secure安全模式（https）
      secure: true
    });

    this.bucket = options.bucket;
  }

  /**
   * 上传文件到OSS
   * @param {string} localFile - 本地文件路径
   * @param {string} objectName - OSS上的对象名称
   */
  async uploadFile(localFile, objectName) {
    try {
      const result = await this.client.put(objectName, localFile);
      return {
        success: true,
        url: result.url,
        name: objectName,
        etag: result.etag
      };
    } catch (error) {
      console.error('OSS上传失败：', error);
      throw error;
    }
  }

  /**
   * 上传Buffer数据到OSS
   * @param {Buffer} buffer - 缓冲区数据
   * @param {string} objectName - OSS上的对象名称
   * @param {string} contentType - 内容类型
   */
  async uploadBuffer(buffer, objectName, contentType = 'application/octet-stream') {
    try {
      const result = await this.client.put(objectName, buffer, {
        headers: {
          'Content-Type': contentType
        }
      });
      return {
        success: true,
        url: result.url,
        name: objectName
      };
    } catch (error) {
      console.error('OSS Buffer上传失败：', error);
      throw error;
    }
  }

  /**
   * 生成预签名URL
   * 用于前端直接上传或下载
   * @param {string} objectName - OSS上的对象名称
   * @param {number} expires - URL有效期（秒）
   */
  async generateSignedUrl(objectName, expires = 3600) {
    try {
      const url = await this.client.signatureUrl(objectName, {
        expires: expires
      });
      return {
        success: true,
        url: url,
        expires: expires
      };
    } catch (error) {
      console.error('生成签名URL失败：', error);
      throw error;
    }
  }

  /**
   * 生成上传凭证
   * 适用于前端直传场景
   * @param {string} objectName - OSS上的对象名称
   * @param {number} expires - 凭证有效期（秒）
   */
  async generateUploadToken(objectName, expires = 3600) {
    const date = new Date();
    date.setSeconds(date.getSeconds() + expires);

    const policy = JSON.stringify({
      expiration: date.toISOString(),
      conditions: [
        ['content-length-range', 0, 100 * 1024 * 1024], // 最大100MB
        ['eq', '$key', objectName]
      ]
    });

    // Base64编码的policy
    const policyBase64 = Buffer.from(policy).toString('base64');

    // 签名
    const signature = this.client.calculateSignature(policyBase64);

    return {
      success: true,
      accessKeyId: this.client.options.accessKeyId,
      policy: policyBase64,
      signature: signature,
      host: `https://${this.bucket}.oss-cn-beijing.aliyuncs.com`,
      objectName: objectName
    };
  }

  /**
   * 删除OSS上的文件
   * @param {string} objectName - OSS上的对象名称
   */
  async deleteFile(objectName) {
    try {
      await this.client.delete(objectName);
      return {
        success: true,
        message: '删除成功'
      };
    } catch (error) {
      console.error('OSS删除失败：', error);
      throw error;
    }
  }

  /**
   * 检查文件是否存在
   * @param {string} objectName - OSS上的对象名称
   */
  async fileExists(objectName) {
    try {
      await this.client.head(objectName);
      return true;
    } catch (error) {
      if (error.name === 'NoSuchKeyError' || error.status === 404) {
        return false;
      }
      throw error;
    }
  }

  /**
   * 列出文件
   * @param {string} prefix - 前缀过滤
   * @param {number} limit - 返回数量限制
   */
  async listFiles(prefix = '', limit = 100) {
    try {
      const result = await this.client.list({
        prefix: prefix,
        'max-keys': limit
      });

      return {
        success: true,
        files: result.objects.map(obj => ({
          name: obj.name,
          url: obj.url,
          size: obj.size,
          lastModified: obj.lastModified
        })),
        isTruncated: result.isTruncated,
        nextMarker: result.nextMarker
      };
    } catch (error) {
      console.error('OSS列出文件失败：', error);
      throw error;
    }
  }
}
```

## 第六章 云存储SDK

### 6.1 阿里云OSS SDK

阿里云OSS SDK是官方提供的Node.js SDK，封装了OSS的所有API，提供了简洁易用的接口。SDK支持Promise和async/await语法，可以方便地集成到现代Node.js应用中。

SDK的核心是OSS.Client类，需要配置region、accessKeyId、accessKeySecret和bucket等参数创建客户端实例。实例方法包括put（上传）、get（下载）、head（获取文件信息）、delete（删除）、list（列出文件）等基本操作，以及签名URL生成、分片上传等高级功能。

分片上传（Multipart Upload）是OSS处理大文件的核心功能。当文件较大时，SDK会自动将文件分割成多个分片并行上传，提高上传速度和可靠性。如果上传中断，已上传的分片会保留在OSS上，可以继续上传剩余的分片。分片上传还支持上传进度回调，可以实现上传进度显示。

SDK还支持各种配置选项，如上传重试次数、超时时间、并发数等。通过配置这些选项，可以优化上传性能，适应不同的网络环境。

### 6.2 AWS S3 SDK

AWS S3 SDK（@aws-sdk/client-s3）是AWS官方提供的S3客户端SDK，支持S3的所有操作。SDK使用模块化设计，只加载需要的组件，减少包体积。

S3 SDK的使用方式与阿里云OSS SDK类似。创建S3Client客户端实例，配置region和credentials（访问密钥），然后调用各种命令（Command）执行操作。命令包括GetObjectCommand（下载）、PutObjectCommand（上传）、DeleteObjectCommand（删除）、ListObjectsV2Command（列出文件）等。

S3的预签名URL功能非常强大。可以生成用于上传的预签名PUT URL或用于下载的预签名GET URL。预签名URL包含了签名和过期时间，可以在不暴露AWS密钥的情况下授权他人访问指定的S3对象。

S3的存储类别（Storage Classes）提供了成本优化选项。CreateMultipartUpload时可以通过StorageClass参数指定存储类别。生命周期策略可以自动将对象在不同存储类别之间转移，如将30天未访问的对象转移到STANDARD_IA（不频繁访问），90天后转移到Glacier（归档存储）。

### 6.3 预签名URL

预签名URL（Pre-signed URL）是云存储服务提供的授权访问机制。用户在没有云存储访问权限的情况下，可以通过预签名URL临时访问存储对象。预签名URL包含签名、过期时间和操作权限信息。

预签名URL的工作原理是：服务端使用云存储的访问密钥对请求参数（对象名称、操作类型、过期时间等）进行签名，生成签名字符串和URL。客户端拿到预签名URL后，在有效期内可以直接访问。超过有效期或篡改参数，签名会失效，访问被拒绝。

对于文件上传场景，预签名URL允许用户直接上传文件到云存储，跳过应用服务器中转。这带来了几个好处：减轻服务器带宽压力，特别是大文件上传；提高上传速度，客户端直连云存储，走CDN最优路径；支持断点续传，因为上传不经过服务器。

预签名URL的安全性取决于签名密钥的保护和过期时间的合理设置。过期时间太短可能导致上传未完成就无法继续，太长则增加安全风险。一般建议上传凭证的过期时间设置为1小时左右，给用户充足的上传时间。

## 第七章 安全防护

### 7.1 文件名注入

文件名注入是文件上传功能中最常见的安全漏洞之一。恶意用户可能通过构造特殊的文件名来执行路径遍历攻击、跨站脚本攻击（XSS）或覆盖敏感文件。

路径遍历攻击利用程序对文件路径处理的不当验证，通过在文件名中包含"../"等特殊序列，试图跳出指定的上传目录，访问或覆盖服务器上的其他文件。例如，上传文件名为"../../../etc/passwd"，可能导致系统密码文件被覆盖或读取。

防范文件名注入的措施包括：对上传的文件名进行严格验证，只允许包含安全的字符（如字母、数字、下划线、连字符）；重命名所有上传的文件，使用随机生成的唯一名称，而不是用户提供的原始文件名；验证最终的完整路径确保在允许的目录范围内；限制文件扩展名，只允许上传安全的文件类型。

文件名处理还应该考虑Unicode相关的攻击。有些Unicode字符在视觉上与普通字符相同，但在规范化后会变成不同的字符，这可能导致绕过文件名验证。使用严格的白名单验证和文件名重命名策略可以有效防止这类攻击。

### 7.2 路径遍历

路径遍历（Path Traversal）是另一种常见的安全漏洞，攻击者利用不安全的文件路径处理来访问服务器上的敏感文件或目录。在文件上传场景中，攻击者可能通过构造特殊的文件路径来覆盖系统文件或读取敏感数据。

典型的路径遍历攻击使用"../"序列来向上遍历目录。如果服务器代码直接使用用户提供的文件名来构建完整路径，而没有进行充分的验证，就可能被攻击。例如，服务器代码可能这样写：

```javascript
const uploadDir = '/var/www/uploads/';
const filePath = uploadDir + userProvidedFilename;
```

如果用户提供的文件名是"../../../etc/passwd"，最终的文件路径就变成了"/var/www/uploads/../../../etc/passwd"，等价于"/etc/passwd"。如果文件写入成功，就可能导致系统文件被覆盖。

正确的做法是使用path.resolve或path.join来构建路径，这些方法会规范化路径并检测异常：

```javascript
const uploadDir = '/var/www/uploads/';
const filePath = path.join(uploadDir, userProvidedFilename);

// 验证最终路径在允许的目录内
if (!filePath.startsWith(path.resolve(uploadDir))) {
  throw new Error('非法文件路径');
}
```

还应该限制文件名长度和字符集，防止超长文件名或特殊字符导致的缓冲区溢出或解析错误。对于Windows系统，还需要处理驱动器相关的路径序列。

### 7.3 恶意文件

恶意文件上传是Web安全中最危险的问题之一。攻击者可能上传包含恶意代码的文件，如WebShell、病毒、木马等。如果服务器以不安全的方式处理或执行这些文件，可能导致服务器被完全控制。

WebShell是最常见的恶意文件形式。攻击者上传一个看似无害的图片或文档文件，但实际上包含可执行的服务器端代码（如PHP、ASP、JSP等）。如果服务器配置不当，直接访问这个文件就会执行其中的代码，攻击者就获得了服务器的控制权。

防范恶意文件的关键措施包括：严格验证文件类型（使用魔数验证而不仅仅是扩展名）；重命名上传文件，使用随机名称而不是原始名称；将上传目录设置为不可执行，防止通过Web访问执行文件；存储文件到Web根目录之外，或使用云存储而不是本地存储；对于图片等文件，尝试解析并重新生成，确保是可执行的有效格式。

文件内容扫描也是重要的防护手段。可以集成ClamAV等杀毒引擎对上传的文件进行扫描，检测已知病毒和恶意软件。但这会增加上传延迟，需要根据实际场景权衡。

### 7.4 图片安全处理

图片是文件上传中最常见的类型，也是安全风险较高的类型。除了前面提到的恶意代码嵌入，图片还可能包含敏感信息（如GPS位置、拍摄设备信息等）或被用于钓鱼攻击（如将恶意图片伪装成名人照片）。

图片安全处理的第一步是重新生成图片而不是直接保存用户上传的原始文件。通过使用sharp等图像处理库读取、上传的图片并重新生成，可以去除EXIF信息（包含拍摄设备、时间、地点等隐私信息）、确保图片格式正确、验证图片可解析等。

下面是一个完整的图片安全处理实现：

```javascript
// image-processor.js - 图片安全处理

const sharp = require('sharp');
const path = require('path');
const crypto = require('crypto');

/**
 * 图片安全处理器
 * 执行多种安全检查和处理
 */
class ImageSecurityProcessor {
  constructor(options = {}) {
    this.maxWidth = options.maxWidth || 4096;
    this.maxHeight = options.maxHeight || 4096;
    this.maxSize = options.maxSize || 10 * 1024 * 1024; // 10MB
    this.allowedFormats = options.allowedFormats || ['jpeg', 'png', 'webp', 'gif'];
    this.outputFormat = options.outputFormat || 'jpeg';
  }

  /**
   * 处理上传的图片
   * @param {string} inputPath - 输入文件路径
   * @param {string} outputDir - 输出目录
   * @returns {Promise<object>} - 处理结果
   */
  async process(inputPath, outputDir) {
    // 生成安全的随机文件名
    const uniqueId = crypto.randomBytes(16).toString('hex');
    const outputFilename = uniqueId + '.' + this.outputFormat;
    const outputPath = path.join(outputDir, outputFilename);

    try {
      // 读取并验证图片
      const image = sharp(inputPath, { failOn: 'none' });

      // 获取图片元信息
      const metadata = await image.metadata();

      // 安全检查1：验证格式
      if (!this.allowedFormats.includes(metadata.format)) {
        throw new Error('不支持的图片格式：' + metadata.format);
      }

      // 安全检查2：验证尺寸
      if (metadata.width > this.maxWidth || metadata.height > this.maxHeight) {
        throw new Error('图片尺寸超出限制');
      }

      // 安全检查3：去除EXIF信息（去除GPS、拍摄设备等隐私信息）
      // 安全检查4：重新编码图片，确保是可解析的有效图片

      // 构建处理流水线
      let pipeline = image
        .rotate() // 自动旋转（根据EXIF orientation）
        .jpeg({ quality: 85 }) // 转为JPEG格式
        .greyscale(false); // 保持色彩

      // 如果启用了去EXIF选项
      pipeline = pipeline.withMetadata({ orientation: undefined });

      // 如果启用了尺寸限制，应用resize
      if (metadata.width > this.maxWidth || metadata.height > this.maxHeight) {
        pipeline = pipeline.resize(this.maxWidth, this.maxHeight, {
          fit: 'inside',
          withoutEnlargement: true
        });
      }

      // 保存处理后的图片
      await pipeline.toFile(outputPath);

      // 获取处理后的文件信息
      const outputStats = require('fs').statSync(outputPath);

      return {
        success: true,
        outputPath: outputPath,
        filename: outputFilename,
        originalMetadata: {
          width: metadata.width,
          height: metadata.height,
          format: metadata.format,
          size: metadata.size
        },
        outputMetadata: {
          width: (await sharp(outputPath).metadata()).width,
          height: (await sharp(outputPath).metadata()).height,
          format: this.outputFormat,
          size: outputStats.size
        }
      };

    } catch (error) {
      // 处理失败，清理可能生成的文件
      if (require('fs').existsSync(outputPath)) {
        require('fs').unlinkSync(outputPath);
      }
      throw error;
    }
  }

  /**
   * 生成缩略图
   */
  async generateThumbnail(inputPath, outputDir, width = 200, height = 200) {
    const uniqueId = crypto.randomBytes(8).toString('hex');
    const outputFilename = 'thumb_' + uniqueId + '.jpg';
    const outputPath = path.join(outputDir, outputFilename);

    await sharp(inputPath)
      .resize(width, height, {
        fit: 'cover',
        position: 'center'
      })
      .jpeg({ quality: 80 })
      .toFile(outputPath);

    return {
      path: outputPath,
      filename: outputFilename
    };
  }

  /**
   * 添加水印
   */
  async addWatermark(inputPath, outputDir, watermarkText = 'Watermark') {
    const uniqueId = crypto.randomBytes(8).toString('hex');
    const outputFilename = 'wm_' + uniqueId + '.jpg';
    const outputPath = path.join(outputDir, outputFilename);

    // 获取图片尺寸
    const metadata = await sharp(inputPath).metadata();

    // 创建SVG水印
    const svgText = `
      <svg width="${metadata.width}" height="${metadata.height}">
        <style>
          .watermark {
            fill: rgba(255, 255, 255, 0.5);
            font-size: 24px;
            font-family: Arial, sans-serif;
          }
        </style>
        <text
          x="20"
          y="${metadata.height - 20}"
          class="watermark"
        >${watermarkText}</text>
      </svg>
    `;

    await sharp(inputPath)
      .composite([{
        input: Buffer.from(svgText),
        top: 0,
        left: 0
      }])
      .jpeg({ quality: 90 })
      .toFile(outputPath);

    return {
      path: outputPath,
      filename: outputFilename
    };
  }
}

/**
 * 检查图片是否为真正的图片（而非伪装的可执行文件）
 */
async function isValidImage(filePath) {
  try {
    const metadata = await sharp(filePath, { failOn: 'none' }).metadata();
    // sharp能读取元数据说明是有效的图片
    return !!metadata.width && !!metadata.height;
  } catch {
    return false;
  }
}

/**
 * 提取并清除EXIF信息
 */
async function stripExif(inputPath, outputPath) {
  await sharp(inputPath)
    .withMetadata({ orientation: undefined }) // 去除所有EXIF，保留orientation
    .toFile(outputPath);
}

module.exports = {
  ImageSecurityProcessor,
  isValidImage,
  stripExif
};
```

这个图片安全处理器实现了多层安全检查：格式验证确保上传的是允许的图片格式；尺寸验证防止超大图片；去除EXIF信息防止隐私泄露；重新编码确保图片是有效的可解析格式。生成的缩略图和水印功能也常用于实际业务场景。

### 7.5 实战：安全上传

综合前面的安全措施，构建一个完整的安全文件上传系统：

```javascript
// secure-upload.js - 安全文件上传完整实现

const express = require('express');
const multer = require('multer');
const path = require('path');
const fs = require('fs');
const crypto = require('crypto');
const { ImageSecurityProcessor, isValidImage } = require('./image-processor');

const app = express();
const PORT = 3000;

// ============ 安全配置 ============

// 1. 上传目录 - 放在Web根目录之外
const UPLOAD_DIR = path.join(__dirname, 'secure-uploads');
const TEMP_DIR = path.join(__dirname, 'temp-uploads');

// 确保目录存在
[UPLOAD_DIR, TEMP_DIR].forEach(dir => {
  if (!fs.existsSync(dir)) {
    fs.mkdirSync(dir, { recursive: true });
  }
});

// 2. 文件验证配置
const ALLOWED_EXTENSIONS = new Set([
  '.jpg', '.jpeg', '.png', '.gif', '.webp', '.pdf', '.doc', '.docx', '.xls', '.xlsx', '.zip'
]);

const ALLOWED_MIME_TYPES = new Set([
  'image/jpeg',
  'image/png',
  'image/gif',
  'image/webp',
  'application/pdf',
  'application/msword',
  'application/vnd.openxmlformats-officedocument.wordprocessingml.document',
  'application/vnd.ms-excel',
  'application/vnd.openxmlformats-officedocument.spreadsheetml.sheet',
  'application/zip',
  'application/x-zip-compressed'
]);

// 3. 大小限制
const MAX_FILE_SIZE = 50 * 1024 * 1024; // 50MB
const MAX_IMAGE_SIZE = 10 * 1024 * 1024; // 10MB

// 4. 图片处理器
const imageProcessor = new ImageSecurityProcessor({
  maxWidth: 4096,
  maxHeight: 4096,
  maxSize: 10 * 1024 * 1024,
  outputFormat: 'jpeg'
});

// ============ 安全工具函数 ============

/**
 * 生成安全的随机文件名
 */
function generateSecureFilename(originalName) {
  const ext = path.extname(originalName).toLowerCase();
  const uniqueId = crypto.randomBytes(16).toString('hex');
  const timestamp = Date.now();
  return `${timestamp}-${uniqueId}${ext}`;
}

/**
 * 验证文件扩展名
 */
function validateExtension(filename) {
  const ext = path.extname(filename).toLowerCase();
  if (!ALLOWED_EXTENSIONS.has(ext)) {
    return {
      valid: false,
      message: '不支持的文件扩展名：' + ext
    };
  }
  return { valid: true, extension: ext };
}

/**
 * 验证MIME类型
 */
function validateMimeType(mimeType) {
  if (!ALLOWED_MIME_TYPES.has(mimeType)) {
    return {
      valid: false,
      message: '不支持的文件类型：' + mimeType
    };
  }
  return { valid: true, mimeType };
}

/**
 * 规范化并验证文件路径
 */
function sanitizePath(basePath, userPath) {
  // 使用path.resolve解析绝对路径
  const fullPath = path.resolve(basePath, userPath);

  // 确保解析后的路径在基准路径内
  if (!fullPath.startsWith(basePath)) {
    return null; // 检测到路径遍历
  }

  return fullPath;
}

// ============ Multer配置 ============

const storage = multer.diskStorage({
  destination: (req, file, cb) => {
    cb(null, TEMP_DIR);
  },
  filename: (req, file, cb) => {
    // 生成安全的随机文件名
    const safeFilename = generateSecureFilename(file.originalname);
    cb(null, safeFilename);
  }
});

// 文件过滤器
const fileFilter = (req, file, cb) => {
  // 检查扩展名
  const extResult = validateExtension(file.originalname);
  if (!extResult.valid) {
    return cb(new Error(extResult.message), false);
  }

  // 检查MIME类型
  const mimeResult = validateMimeType(file.mimetype);
  if (!mimeResult.valid) {
    return cb(new Error(mimeResult.message), false);
  }

  // MIME类型与扩展名匹配检查
  const extToMime = {
    '.jpg': 'image/jpeg',
    '.jpeg': 'image/jpeg',
    '.png': 'image/png',
    '.gif': 'image/gif',
    '.webp': 'image/webp',
    '.pdf': 'application/pdf'
  };

  if (extToMime[extResult.extension] &&
      extToMime[extResult.extension] !== file.mimetype) {
    return cb(new Error('文件扩展名与内容类型不匹配'), false);
  }

  cb(null, true);
};

const upload = multer({
  storage: storage,
  fileFilter: fileFilter,
  limits: {
    fileSize: MAX_FILE_SIZE,
    files: 5
  }
});

// ============ 错误处理中间件 ============

const errorHandler = (err, req, res, next) => {
  console.error('上传错误：', err);

  if (err instanceof multer.MulterError) {
    switch (err.code) {
      case 'LIMIT_FILE_SIZE':
        return res.status(400).json({
          success: false,
          message: '文件大小超出限制'
        });
      case 'LIMIT_FILE_COUNT':
        return res.status(400).json({
          success: false,
          message: '文件数量超出限制'
        });
      default:
        return res.status(400).json({
          success: false,
          message: '上传错误：' + err.message
        });
    }
  }

  res.status(400).json({
    success: false,
    message: err.message
  });
};

// ============ 路由 ============

/**
 * 普通文件上传
 */
app.post('/upload/file', upload.single('file'), errorHandler, async (req, res) => {
  if (!req.file) {
    return res.status(400).json({
      success: false,
      message: '没有上传文件'
    });
  }

  const tempPath = req.file.path;
  const finalFilename = req.file.filename;
  const finalPath = path.join(UPLOAD_DIR, finalFilename);

  try {
    // 如果是图片，进行安全处理
    if (req.file.mimetype.startsWith('image/')) {
      const result = await imageProcessor.process(tempPath, UPLOAD_DIR);

      // 删除临时文件
      fs.unlinkSync(tempPath);

      // 返回图片URL（实际项目中应该返回云存储URL）
      res.json({
        success: true,
        message: '图片上传并处理成功',
        file: {
          filename: result.filename,
          url: '/secure-uploads/' + result.filename,
          width: result.outputMetadata.width,
          height: result.outputMetadata.height,
          size: result.outputMetadata.size
        }
      });
    } else {
      // 非图片文件，直接移动到最终目录
      fs.renameSync(tempPath, finalPath);

      res.json({
        success: true,
        message: '文件上传成功',
        file: {
          filename: finalFilename,
          url: '/secure-uploads/' + finalFilename,
          size: req.file.size
        }
      });
    }

  } catch (error) {
    // 清理临时文件
    if (fs.existsSync(tempPath)) {
      fs.unlinkSync(tempPath);
    }
    throw error;
  }
});

/**
 * 多文件上传
 */
app.post('/upload/multiple', upload.array('files', 5), errorHandler, async (req, res) => {
  if (!req.files || req.files.length === 0) {
    return res.status(400).json({
      success: false,
      message: '没有上传文件'
    });
  }

  const results = [];

  for (const file of req.files) {
    try {
      const result = await imageProcessor.process(file.path, UPLOAD_DIR);
      fs.unlinkSync(file.path);
      results.push({
        success: true,
        filename: result.filename,
        url: '/secure-uploads/' + result.filename
      });
    } catch (error) {
      results.push({
        success: false,
        originalName: file.originalname,
        error: error.message
      });
    }
  }

  res.json({
    success: true,
    message: '处理完成',
    results: results
  });
});

/**
 * 生成缩略图
 */
app.post('/upload/thumbnail', upload.single('image'), errorHandler, async (req, res) => {
  if (!req.file || !req.file.mimetype.startsWith('image/')) {
    return res.status(400).json({
      success: false,
      message: '请上传图片文件'
    });
  }

  try {
    const result = await imageProcessor.generateThumbnail(
      req.file.path,
      UPLOAD_DIR,
      200,
      200
    );

    // 删除原图
    fs.unlinkSync(req.file.path);

    res.json({
      success: true,
      thumbnail: {
        url: '/secure-uploads/' + result.filename
      }
    });

  } catch (error) {
    if (fs.existsSync(req.file.path)) {
      fs.unlinkSync(req.file.path);
    }
    throw error;
  }
});

app.listen(PORT, () => {
  console.log('安全上传服务器已启动，端口：' + PORT);
});

module.exports = app;
```

这个安全上传系统综合应用了前面介绍的所有安全措施：文件名随机化防止路径遍历和文件名注入；扩展名和MIME类型双重验证；扩展名与MIME类型匹配检查；图片内容重新生成去除恶意代码；文件存储在Web根目录之外；完善的错误处理机制。这些措施共同构成了纵深防御体系，可以有效防止各种文件上传相关的安全威胁。

## 第八章 图片处理

### 8.1 sharp图像处理

sharp是目前Node.js生态中最流行、性能最好的图像处理库。sharp基于libvips开发，处理速度比ImageMagick和GraphicsMagick快得多，内存占用也更少。sharp支持多种图片格式，包括JPEG、PNG、WebP、GIF、TIFF等，可以完成图片缩放、裁剪、旋转、格式转换等常见操作。

sharp的API设计非常优雅，采用链式调用风格。创建sharp实例时传入图片源（可以是文件路径、Buffer或流），然后通过链式方法进行各种变换，最后调用toFile、toBuffer或pipe方法输出结果。这种设计使得复杂的图片处理流程可以用简洁的代码表达。

sharp的一个核心优势是处理大图片时的低内存占用。libvips的按需加载机制使得处理超大图片时不需要将整个图片加载到内存中，只有当前处理的像素区域才会驻留内存。这使得sharp非常适合处理用户上传的高分辨率图片。

### 8.2 缩略图生成

缩略图生成是图片处理中最常见的需求。用户上传图片后，通常需要生成不同尺寸的缩略图用于列表展示、头像显示等场景。sharp提供了resize方法来实现高质量的缩略图生成。

resize方法支持多种缩放模式：fit模式保持宽高比，限制在指定尺寸内；cover模式裁剪到正好填满指定尺寸；contain模式在指定尺寸内完整显示图片（可能留白）；fill模式在指定尺寸内拉伸或压缩图片。每种模式适用于不同的场景。

生成缩略图时还需要注意质量控制。JPEG和WebP格式支持质量参数，可以在文件大小和图片质量之间取得平衡。对于缩略图，使用较低的质量（如70-80）通常不会有明显的视觉差异，但可以显著减少文件大小，加快页面加载。

```javascript
// thumbnail-generator.js - 缩略图生成示例

const sharp = require('sharp');
const path = require('path');
const fs = require('fs');

/**
 * 缩略图生成器
 */
class ThumbnailGenerator {
  constructor(options = {}) {
    this.sizes = options.sizes || [
      { name: 'small', width: 100, height: 100 },
      { name: 'medium', width: 300, height: 300 },
      { name: 'large', width: 800, height: 800 }
    ];
    this.outputDir = options.outputDir || './thumbnails';
    this.quality = options.quality || 80;

    if (!fs.existsSync(this.outputDir)) {
      fs.mkdirSync(this.outputDir, { recursive: true });
    }
  }

  /**
   * 生成多个尺寸的缩略图
   * @param {string} inputPath - 输入图片路径
   * @param {string} baseName - 基础文件名
   * @returns {Promise<object>} - 生成的缩略图信息
   */
  async generateAll(inputPath, baseName) {
    const results = {};
    const metadata = await sharp(inputPath).metadata();

    for (const size of this.sizes) {
      const outputFilename = `${baseName}-${size.name}.jpg`;
      const outputPath = path.join(this.outputDir, outputFilename);

      await sharp(inputPath)
        .resize(size.width, size.height, {
          fit: 'cover',
          position: 'centre', // 从中心裁剪
          kernel: 'lanczos3' // 高质量缩放算法
        })
        .jpeg({
          quality: this.quality,
          progressive: true // 渐进式JPEG
        })
        .toFile(outputPath);

      const outputMeta = await sharp(outputPath).metadata();

      results[size.name] = {
        filename: outputFilename,
        path: outputPath,
        width: outputMeta.width,
        height: outputMeta.height,
        size: fs.statSync(outputPath).size
      };
    }

    return results;
  }

  /**
   * 生成自适应缩略图
   * 根据原图尺寸和目标尺寸自动选择最佳裁剪位置
   */
  async generateSmart(inputPath, baseName, targetWidth = 300, targetHeight = 300) {
    const metadata = await sharp(inputPath).metadata();
    const outputFilename = `${baseName}-smart.jpg`;
    const outputPath = path.join(this.outputDir, outputFilename);

    // 计算裁剪区域
    const aspectRatio = metadata.width / metadata.height;
    const targetRatio = targetWidth / targetHeight;

    let width, height, left, top;

    if (aspectRatio > targetRatio) {
      // 原图更宽，以高度为基准裁剪宽度
      height = metadata.height;
      width = Math.round(height * targetRatio);
      left = Math.round((metadata.width - width) / 2);
      top = 0;
    } else {
      // 原图更高，以宽度为基准裁剪高度
      width = metadata.width;
      height = Math.round(width / targetRatio);
      left = 0;
      top = Math.round((metadata.height - height) / 2);
    }

    await sharp(inputPath)
      .extract({ left, top, width, height })
      .resize(targetWidth, targetHeight)
      .jpeg({ quality: this.quality })
      .toFile(outputPath);

    return {
      filename: outputFilename,
      width: targetWidth,
      height: targetHeight
    };
  }
}

module.exports = { ThumbnailGenerator };
```

### 8.3 水印添加

水印是保护图片版权的常用手段。sharp提供了composite方法，可以将水印图片或SVG叠加到原图上，实现文字水印或图片水印功能。

文字水印通常使用SVG来创建矢量图形，然后用composite方法叠加到原图上。SVG水印的优势是放大后不会失真，生成速度快。可以创建包含版权信息、公司Logo的水印。

水印的位置可以设置为固定位置（如右下角、左下角）或使用随机位置。还可以根据原图尺寸动态调整水印大小，确保水印在不同尺寸的图片上都清晰可见但不会遮挡主要内容。

```javascript
// watermark.js - 水印添加示例

const sharp = require('sharp');
const path = require('path');
const fs = require('fs');

/**
 * 水印添加器
 */
class WatermarkProcessor {
  constructor(options = {}) {
    this.text = options.text || 'Watermark';
    this.fontSize = options.fontSize || 24;
    this.color = options.color || 'rgba(255, 255, 255, 0.5)';
    this.position = options.position || 'southeast'; // north, south, east, west, center, southeast, etc.
    this.margin = options.margin || 20;
  }

  /**
   * 创建文字水印SVG
   */
  createTextSvg(width, height) {
    return `
      <svg width="${width}" height="${height}">
        <style>
          .watermark {
            fill: ${this.color};
            font-size: ${this.fontSize}px;
            font-family: Arial, sans-serif;
            font-weight: bold;
          }
        </style>
        <text x="10" y="${height - 10}" class="watermark">${this.text}</text>
      </svg>
    `;
  }

  /**
   * 计算水印位置
   */
  calculatePosition(imageWidth, imageHeight, watermarkWidth, watermarkHeight) {
    const positionMap = {
      northwest: { left: this.margin, top: this.margin },
      north: { left: Math.round((imageWidth - watermarkWidth) / 2), top: this.margin },
      northeast: { left: imageWidth - watermarkWidth - this.margin, top: this.margin },
      west: { left: this.margin, top: Math.round((imageHeight - watermarkHeight) / 2) },
      center: {
        left: Math.round((imageWidth - watermarkWidth) / 2),
        top: Math.round((imageHeight - watermarkHeight) / 2)
      },
      east: {
        left: imageWidth - watermarkWidth - this.margin,
        top: Math.round((imageHeight - watermarkHeight) / 2)
      },
      southwest: { left: this.margin, top: imageHeight - watermarkHeight - this.margin },
      south: {
        left: Math.round((imageWidth - watermarkWidth) / 2),
        top: imageHeight - watermarkHeight - this.margin
      },
      southeast: {
        left: imageWidth - watermarkWidth - this.margin,
        top: imageHeight - watermarkHeight - this.margin
      }
    };

    return positionMap[this.position] || positionMap.southeast;
  }

  /**
   * 添加文字水印
   */
  async addTextWatermark(inputPath, outputPath) {
    const image = sharp(inputPath);
    const metadata = await image.metadata();

    // 水印尺寸（可以根据图片尺寸按比例调整）
    const watermarkWidth = this.fontSize * this.text.length * 0.6;
    const watermarkHeight = this.fontSize * 1.5;

    // 计算位置
    const position = this.calculatePosition(
      metadata.width,
      metadata.height,
      watermarkWidth,
      watermarkHeight
    );

    // 创建水印SVG
    const svg = this.createTextSvg(metadata.width, metadata.height);

    // 叠加水印
    await image
      .composite([{
        input: Buffer.from(svg),
        top: 0,
        left: 0
      }])
      .toFile(outputPath);

    return {
      success: true,
      outputPath: outputPath
    };
  }

  /**
   * 添加图片水印（Logo）
   */
  async addImageWatermark(inputPath, logoPath, outputPath) {
    const image = sharp(inputPath);
    const imageMeta = await image.metadata();
    const logoMeta = await sharp(logoPath).metadata();

    // Logo尺寸（最大为图片的10%）
    const maxLogoWidth = Math.round(imageMeta.width * 0.1);
    const maxLogoHeight = Math.round(imageMeta.height * 0.1);

    let logoWidth = logoMeta.width;
    let logoHeight = logoMeta.height;

    // 如果Logo过大，按比例缩小
    if (logoWidth > maxLogoWidth || logoHeight > maxLogoHeight) {
      const ratio = Math.min(maxLogoWidth / logoWidth, maxLogoHeight / logoHeight);
      logoWidth = Math.round(logoWidth * ratio);
      logoHeight = Math.round(logoHeight * ratio);
    }

    // 调整Logo大小
    const resizedLogo = await sharp(logoPath)
      .resize(logoWidth, logoHeight)
      .toBuffer();

    // 计算位置
    const position = this.calculatePosition(
      imageMeta.width,
      imageMeta.height,
      logoWidth,
      logoHeight
    );

    // 叠加水印
    await image
      .composite([{
        input: resizedLogo,
        top: position.top,
        left: position.left
      }])
      .toFile(outputPath);

    return {
      success: true,
      outputPath: outputPath
    };
  }
}

module.exports = { WatermarkProcessor };
```

### 8.4 格式转换

sharp支持图片格式的相互转换，可以轻松实现图片格式转换功能。常见的格式转换包括：将PNG转JPEG（减少文件大小）、将图片转WebP（现代浏览器支持的高压缩比格式）、生成渐进式JPEG（适合网络传输）等。

格式转换时需要考虑不同格式的特性。JPEG是有损压缩格式，适合照片等复杂图片；PNG是无损压缩，支持透明度；WebP在保持相似质量的同时文件更小，但兼容性稍差（IE不支持）。实际应用中可以根据用户代理动态选择合适的格式。

sharp还支持图片的元数据操作，可以读取或去除EXIF、IPTC、XMP等元数据。这对于保护用户隐私（去除GPS位置等信息）和减少文件大小都很有用。

## 第九章 分片上传

### 9.1 分片大小选择

分片上传是将大文件分割成多个小块分别上传的技术。合理选择分片大小对于上传成功率和性能至关重要。

分片大小的选择需要考虑几个因素：网络环境、服务器配置、业务需求。一般建议单个分片大小在1MB到10MB之间。对于稳定的网络环境，较大的分片可以减少HTTP请求数量，提高效率。对于不稳定的网络环境，较小的分片可以减少单次失败的影响范围。

分片太小（如小于1MB）会导致HTTP请求过多，增加服务器处理开销和网络延迟。分片太大（如超过100MB）会导致单次上传时间过长，增加失败重试的成本。如果在分片上传过程中网络中断，已上传的大分片需要重新上传，造成浪费。

常见的配置是5MB作为默认分片大小，这个大小在大多数场景下都能取得较好的平衡。对于移动端或弱网络环境，可以适当减小到1-2MB。对于企业内部的高速网络，可以增大到10-20MB。

### 9.2 前端分片逻辑

前端分片上传的核心是将大文件分割成多个Blob块，然后按顺序上传每个块。File对象提供了slice方法可以方便地分割文件。这个方法接受起始和结束位置，返回一个新的Blob对象。

分片上传的流程是：首先发送初始化请求获取上传ID和分片信息；然后按顺序上传每个分片，可以并发上传多个分片提高速度；最后发送合并请求，服务端将所有分片合并成完整文件。

前端还需要实现进度追踪、错误重试、并发控制等逻辑。并发控制可以限制同时上传的分片数量，避免占用过多浏览器资源或压垮服务器。错误重试机制应该在分片上传失败后自动重试，通常设置3-5次重试机会。

```javascript
// chunk-upload.js - 前端分片上传实现

/**
 * 分片上传管理器
 */
class ChunkUploader {
  constructor(options = {}) {
    this.chunkSize = options.chunkSize || 5 * 1024 * 1024; // 默认5MB
    this.maxConcurrent = options.maxConcurrent || 3; // 最大并发数
    this.maxRetries = options.maxRetries || 3; // 最大重试次数
    this.onProgress = options.onProgress || (() => {});
    this.onComplete = options.onComplete || (() => {});
    this.onError = options.onError || (() => {});
  }

  /**
   * 计算分片信息
   */
  calculateChunks(file) {
    const chunks = [];
    let start = 0;
    let index = 0;

    while (start < file.size) {
      const end = Math.min(start + this.chunkSize, file.size);
      chunks.push({
        index: index,
        start: start,
        end: end,
        size: end - start,
        blob: file.slice(start, end)
      });

      start = end;
      index++;
    }

    return chunks;
  }

  /**
   * 上传单个分片
   */
  async uploadChunk(uploadId, chunk, retries = 0) {
    try {
      const formData = new FormData();
      formData.append('uploadId', uploadId);
      formData.append('chunkIndex', chunk.index);
      formData.append('totalChunks', chunk.totalChunks);
      formData.append('filename', chunk.filename);
      formData.append('chunk', chunk.blob);

      const response = await fetch('/upload/chunk', {
        method: 'POST',
        body: formData
      });

      if (!response.ok) {
        throw new Error('上传失败：' + response.status);
      }

      return await response.json();

    } catch (error) {
      // 重试逻辑
      if (retries < this.maxRetries) {
        console.log(`分片${chunk.index}上传失败，${retries + 1}次重试...`);
        await this.delay(1000 * (retries + 1)); // 递增延迟
        return this.uploadChunk(uploadId, chunk, retries + 1);
      }
      throw error;
    }
  }

  /**
   * 并发控制上传
   */
  async uploadWithConcurrency(chunks, uploadId) {
    const results = [];
    let currentIndex = 0;

    // 上传完成或失败的标记
    const completed = new Set();
    const errors = [];

    // 上传下一个待上传的分片
    const processNext = async () => {
      while (currentIndex < chunks.length) {
        const chunk = chunks[currentIndex];
        const chunkIndex = currentIndex;
        currentIndex++;

        // 如果已有分片失败，跳过后续分片
        if (errors.length > 0) {
          completed.add(chunkIndex);
          continue;
        }

        try {
          const result = await this.uploadChunk(uploadId, {
            ...chunk,
            totalChunks: chunks.length,
            filename: this.filename
          });

          completed.add(chunkIndex);
          results[chunkIndex] = result;

          // 报告进度
          this.onProgress({
            completed: completed.size,
            total: chunks.length,
            percent: Math.round((completed.size / chunks.length) * 100),
            currentChunk: chunkIndex
          });

        } catch (error) {
          errors.push(error);
          this.onError(error);
        }
      }
    };

    // 启动并发上传
    const workers = [];
    for (let i = 0; i < this.maxConcurrent && i < chunks.length; i++) {
      workers.push(processNext());
    }

    await Promise.all(workers);

    if (errors.length > 0) {
      throw errors[0];
    }

    return results;
  }

  /**
   * 合并分片
   */
  async mergeChunks(uploadId) {
    const response = await fetch('/upload/merge', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({ uploadId })
    });

    if (!response.ok) {
      throw new Error('合并失败：' + response.status);
    }

    return await response.json();
  }

  /**
   * 开始上传
   */
  async upload(file) {
    this.filename = file.name;

    // 1. 计算分片
    const chunks = this.calculateChunks(file);
    console.log(`文件共${file.size}字节，分为${chunks.length}个分片`);

    // 2. 初始化上传
    const initResponse = await fetch('/upload/init', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({
        filename: file.name,
        fileSize: file.size,
        chunkSize: this.chunkSize,
        mimeType: file.type
      })
    });

    if (!initResponse.ok) {
      throw new Error('初始化上传失败');
    }

    const { uploadId } = await initResponse.json();
    console.log('获取上传ID：' + uploadId);

    // 3. 上传分片
    await this.uploadWithConcurrency(chunks, uploadId);

    // 4. 合并分片
    const mergeResult = await this.mergeChunks(uploadId);

    this.onComplete(mergeResult);
    return mergeResult;
  }

  /**
   * 延迟辅助函数
   */
  delay(ms) {
    return new Promise(resolve => setTimeout(resolve, ms));
  }
}

// 使用示例
const uploader = new ChunkUploader({
  chunkSize: 5 * 1024 * 1024, // 5MB
  maxConcurrent: 3,
  maxRetries: 3,

  onProgress: (progress) => {
    console.log(`上传进度：${progress.percent}% (${progress.completed}/${progress.total})`);
    // 更新UI进度条
    document.getElementById('progress').style.width = progress.percent + '%';
  },

  onComplete: (result) => {
    console.log('上传完成！', result);
    alert('上传成功！');
  },

  onError: (error) => {
    console.error('上传出错：', error);
    alert('上传失败：' + error.message);
  }
});

// 绑定上传按钮
document.getElementById('uploadBtn').addEventListener('click', () => {
  const fileInput = document.getElementById('fileInput');
  const file = fileInput.files[0];

  if (!file) {
    alert('请选择文件');
    return;
  }

  uploader.upload(file);
});
```

### 9.3 服务端合并

服务端接收分片后，需要按照正确的顺序合并成完整的文件。合并过程中需要处理并发写入、临时文件清理、错误恢复等问题。

合并的关键是按分片索引顺序写入文件，不能按到达顺序合并，否则最终文件会损坏。使用文件流的顺序写入可以保证正确性，但需要注意文件偏移量的计算。

服务端还应该处理合并过程中的异常情况：如果合并失败，应该删除已合并的部分并重置上传状态；如果服务器重启，应该能从断点恢复合并。一种做法是将分片信息持久化到数据库，重启后检查并恢复未完成的合并。

### 9.4 进度记录

分片上传的进度记录需要记录每个分片的完成状态。服务端可以维护一个上传记录，包含上传ID、文件名、文件大小、分片数量、已上传分片列表等信息。

存储上传记录的方式有多种：内存（不适合生产环境，重启会丢失）、文件系统（简单但不易扩展）、数据库（推荐，支持查询和统计）、Redis（高性能，适合高并发场景）。

进度查询接口应该返回已上传分片数和总分片数，前端据此计算百分比。由于分片可能乱序到达，已上传分片列表可能不连续，UI在显示进度时应该使用完成数而不是最大分片索引。

## 第十章 断点续传

### 10.1 已上传分片记录

断点续传的核心是记录每个分片的完成状态。服务端需要维护一个上传记录，包含哪些分片已经上传完成，哪些还需要上传。当上传中断后，客户端可以通过查询这个记录来确定从哪个分片继续上传。

已上传分片的记录方式直接影响续传的实现。一种方式是在数据库中存储已上传分片的索引列表，如一个整数数组[0, 1, 3, 4, 5]表示第0、1、3、4、5个分片已上传。客户端查询状态后，需要上传的是第2、6、7等分片。

分片状态的存储位置可以是Redis（高性能，适合高并发）、数据库（持久化，适合重要数据）或文件系统（简单，适合低频场景）。考虑到上传记录的即时性要求较高，Redis是常见的选择。

### 10.2 秒传原理：MD5

秒传是指当服务器上已经存在相同内容的文件时，直接返回成功而不再重复上传。这通过计算文件内容的MD5或SHA哈希值来实现。相同内容的文件一定有相同的哈希值。

秒传的流程是：客户端在开始上传前，先计算本地文件的MD5值（通常只计算整个文件的MD5，而不是每个分片的）；将MD5发送给服务器查询；服务器检查是否存在相同MD5的文件；如果存在，返回已有的文件信息，客户端无需真正上传；如果不存在，则开始正常上传流程。

计算大文件的MD5需要读取整个文件，对于超大文件可能耗时较长。可以使用spark-md5库实现分片计算MD5，将文件分割成小块分别计算，然后合并结果，这样可以边读取边计算，避免一次性加载整个文件到内存。

秒传功能需要注意的问题：MD5碰撞（虽然概率极低）、文件存在但已被删除的情况、权限问题（用户A上传的文件，用户B不应该直接使用）。实际实现时应该验证文件的所有权和权限。

### 10.3 续传逻辑

续传逻辑是断点续传功能的核心。客户端在开始上传或恢复上传时，首先查询服务器端已上传的分片列表，然后只上传尚未上传的分片。

续传的完整流程如下：首先，客户端初始化上传，获取上传ID和文件信息；如果服务器返回uploadId已存在，说明是续传；客户端查询当前上传状态，获取已上传分片列表；客户端计算未上传的分片列表；客户端按顺序上传未上传的分片；上传完成后触发合并。

在续传过程中，还应该处理分片大小变化的情况。如果服务器配置的 chunkSize 与客户端不同，可能会导致分片索引对应关系错乱。因此，初始化时应该确保客户端和服务器使用相同的分片大小。

网络中断后的自动重连也是续传的一部分。可以使用指数退避策略来实现重连：首次失败后等待1秒重试，如果再次失败等待2秒，然后4秒，以此类推，最大等待时间如32秒。这样可以避免频繁重试对服务器造成压力。

### 10.4 并发控制

并发控制是分片上传的重要参数。设置合理的并发数可以在充分利用带宽的同时，避免对服务器造成过大压力，也避免浏览器资源竞争。

并发数的选择需要考虑几个因素：网络带宽（带宽越高，并发可以越大）、服务器处理能力（需要根据服务器配置调整）、浏览器限制（不同浏览器对同域名并发请求数有限制，通常是6个）、文件大小（对于小文件，过高并发反而增加开销）。

一种自适应并发策略是：根据上传成功率动态调整并发数。如果连续成功，降低并发数尝试提高速度；如果出现失败，降低并发数减少压力。这种策略需要在客户端实现一个简单的反馈循环。

实现并发控制可以使用Promise的并发控制模式，如著名的p-limit库，或者手动实现一个带并发限制的Promise池。

## 第十一章 大文件优化

### 11.1 流式处理

流式处理是处理大文件的核心技术。Node.js的流API允许边读边写，不需要将整个文件加载到内存中。这对于处理GB级别的文件至关重要。

在文件上传场景中，HTTP请求体就是一个可读流，文件写入是可写流。通过pipeline或pipe方法，可以将请求流直接导向文件流，实现边接收边写入。这种方式内存占用恒定，不随文件大小增长。

使用multer的diskStorage时，已经默认使用了流式写入。但对于需要处理的场景，如图片压缩，可以创建转换流（transform stream）插入到管道中：请求流 -> 转换流 -> 文件流。这种方式可以一边接收数据一边处理，不需要等整个文件接收完才开始处理。

### 11.2 管道传输

管道（pipe）是Node.js流处理的核心概念，用于连接流组件。Readable流通过pipe方法将数据导向Writable流，数据会自动流动，开发者不需要手动管理。

pipeline是更现代的管道API，相比pipe有更好的错误处理。使用pipe时，如果目标流发生错误，源流不会自动停止，可能导致资源泄漏。使用pipeline，目标流关闭或发生错误时，源流会自动解耦并清理。

```javascript
// pipeline-example.js - 管道处理示例

const fs = require('fs');
const { pipeline } = require('stream/promises');
const sharp = require('sharp');

/**
 * 流式处理大图片：边接收边压缩
 */
async function streamProcessLargeImage(inputStream, outputPath, options = {}) {
  const { width, height, quality = 80, format = 'jpeg' } = options;

  // 创建转换流
  const imageProcessor = sharp()
    .resize(width, height, { fit: 'inside', withoutEnlargement: true })
    [format]({ quality });

  // 使用pipeline进行流式处理
  await pipeline(
    inputStream,      // 输入流
    imageProcessor,    // 转换流
    fs.createWriteStream(outputPath)  // 输出流
  );

  return outputPath;
}

/**
 * 复制大文件
 */
async function streamCopyFile(sourcePath, destPath) {
  await pipeline(
    fs.createReadStream(sourcePath),
    fs.createWriteStream(destPath)
  );
}

/**
 * 合并多个文件
 */
async function streamMergeFiles(sources, destPath) {
  const { pipeline } = require('stream/promises');

  const dest = fs.createWriteStream(destPath);

  for (const source of sources) {
    const sourceStream = fs.createReadStream(source);
    sourceStream.pipe(dest, { end: false });

    await new Promise((resolve, reject) => {
      sourceStream.on('end', resolve);
      sourceStream.on('error', reject);
    });
  }

  dest.end();

  await new Promise((resolve, reject) => {
    dest.on('finish', resolve);
    dest.on('error', reject);
  });
}
```

### 11.3 内存优化

大文件处理中的内存优化至关重要。Node.js的内存限制通常在1-2GB，如果处理大文件时内存使用不当，很容易触发内存溢出错误。

内存优化的关键策略包括：避免bufferAll()等一次性加载整个文件的方法；使用流式处理，边读边写；及时释放不再需要的资源，如关闭文件描述符；使用对象池复用对象，减少GC压力。

对于图片处理，sharp库本身就做了大量内存优化。但如果需要处理超大型图片（如全景图），可能还需要进一步优化，如分块处理、将处理链拆分等。

服务器端的内存配置也很重要。Node.js默认堆内存限制约为1.4GB，可以通过--max-old-space-size参数调整。对于需要处理大文件的服务器，可以适当增加这个限制，但也要考虑实际可用的物理内存。

### 12.1 视频转码

视频上传后的转码处理是视频类应用的核心功能。转码包括格式转换（如上传MKV，转码为MP4）、分辨率调整、码率控制、生成缩略图等。视频转码通常需要借助FFmpeg工具完成。

FFmpeg是开源的音视频处理工具，Node.js中可以使用fluent-ffmpeg或node-fluent-ffmpeg库来调用FFmpeg。转码是CPU密集型操作，对于大文件可能需要较长时间，通常作为异步任务处理。

视频转码的完整流程：用户上传视频 -> 服务器保存原始文件 -> 创建转码任务 -> 执行转码（FFmpeg处理） -> 生成多个分辨率版本 -> 生成缩略图 -> 任务完成通知。转码过程中应该记录进度，支持查询。

```javascript
// video-transcoder.js - 视频转码示例

const express = require('express');
const path = require('path');
const fs = require('fs');
const { exec } = require('child_process');
const { promisify } = require('util');

const execAsync = promisify(exec);

const app = express();
const PORT = 3000;

const UPLOAD_DIR = path.join(__dirname, 'video-uploads');
const OUTPUT_DIR = path.join(__dirname, 'video-output');

// 确保目录存在
[UPLOAD_DIR, OUTPUT_DIR].forEach(dir => {
  if (!fs.existsSync(dir)) {
    fs.mkdirSync(dir, { recursive: true });
  }
});

/**
 * 视频转码任务
 */
class VideoTranscoder {
  constructor(inputPath, outputPath, options = {}) {
    this.inputPath = inputPath;
    this.outputPath = outputPath;
    this.options = {
      codec: options.codec || 'libx264',
      bitrate: options.bitrate || '1000k',
      resolution: options.resolution || null,
      fps: options.fps || null,
      audioCodec: options.audioCodec || 'aac',
      audioBitrate: options.audioBitrate || '128k',
      ...options
    };
    this.progress = 0;
  }

  /**
   * 获取视频信息
   */
  async getInfo() {
    try {
      const { stdout } = await execAsync(
        `ffprobe -v quiet -print_format json -show_format -show_streams "${this.inputPath}"`
      );
      return JSON.parse(stdout);
    } catch (error) {
      throw new Error('获取视频信息失败：' + error.message);
    }
  }

  /**
   * 生成缩略图
   */
  async generateThumbnail(outputPath, timestamp = '00:00:01') {
    const command = `ffmpeg -i "${this.inputPath}" -ss ${timestamp} -vframes 1 -q:v 2 "${outputPath}"`;

    try {
      await execAsync(command);
      return outputPath;
    } catch (error) {
      throw new Error('生成缩略图失败：' + error.message);
    }
  }

  /**
   * 生成GIF
   */
  async generateGif(outputPath, startTime = '00:00:01', duration = 5) {
    const command = `ffmpeg -i "${this.inputPath}" -ss ${startTime} -t ${duration}
      -vf "fps=10,scale=320:-1:flags=lanczos,split[s0][s1];[s0]palettegen[p];[s1][p]paletteuse"
      -loop 0 "${outputPath}"`;

    try {
      await execAsync(command);
      return outputPath;
    } catch (error) {
      throw new Error('生成GIF失败：' + error.message);
    }
  }

  /**
   * 转码（Promise版本）
   */
  async transcode() {
    return new Promise((resolve, reject) => {
      // 构建FFmpeg命令
      let command = `ffmpeg -i "${this.inputPath}" -c:v ${this.options.codec} -b:v ${this.options.bitrate}`;

      // 添加视频滤镜
      if (this.options.resolution) {
        command += ` -vf scale=${this.options.resolution}`;
      }

      // 添加帧率控制
      if (this.options.fps) {
        command += ` -r ${this.options.fps}`;
      }

      // 音频配置
      command += ` -c:a ${this.options.audioCodec} -b:a ${this.options.audioBitrate}`;

      // 输出
      command += ` -y "${this.outputPath}"`;

      console.log('开始转码：', command);

      exec(command, (error, stdout, stderr) => {
        if (error) {
          reject(new Error('转码失败：' + error.message));
          return;
        }
        resolve({
          success: true,
          outputPath: this.outputPath
        });
      });
    });
  }

  /**
   * 转码（带进度回调版本）
   */
  async transcodeWithProgress(onProgress) {
    return new Promise((resolve, reject) => {
      let command = `ffmpeg -i "${this.inputPath}" -c:v ${this.options.codec} -b:v ${this.options.bitrate}`;

      if (this.options.resolution) {
        command += ` -vf scale=${this.options.resolution}`;
      }

      command += ` -c:a ${this.options.audioCodec} -b:a ${this.options.audioBitrate}`;
      command += ` -progress pipe:1 -y "${this.outputPath}"`;

      const ffmpeg = exec(command);

      ffmpeg.stdout.on('data', (data) => {
        // 解析进度信息
        const output = data.toString();
        const timeMatch = output.match(/out_time_ms=(\d+)/);

        if (timeMatch && this.duration) {
          const currentTimeMs = parseInt(timeMatch[1]) / 1000;
          this.progress = Math.min(100, (currentTimeMs / this.duration) * 100);
          onProgress(this.progress);
        }
      });

      ffmpeg.stderr.on('data', (data) => {
        // 尝试从错误输出解析时长
        const output = data.toString();
        const durationMatch = output.match(/Duration: (\d{2}):(\d{2}):(\d{2})/);

        if (durationMatch) {
          this.duration =
            parseInt(durationMatch[1]) * 3600 +
            parseInt(durationMatch[2]) * 60 +
            parseInt(durationMatch[3]);
        }
      });

      ffmpeg.on('close', (code) => {
        if (code === 0) {
          resolve({ success: true, outputPath: this.outputPath });
        } else {
          reject(new Error('转码失败，退出码：' + code));
        }
      });

      ffmpeg.on('error', reject);
    });
  }
}

  const inputPath = path.join(UPLOAD_DIR, videoId + '.mp4');

    return res.status(400).json({
      success: false,
      message: '视频文件不存在'
    });
  }

  const transcoder = new VideoTranscoder(inputPath,
    path.join(OUTPUT_DIR, videoId + '-transcoded.mp4'));

  try {
    // 获取信息
    const info = await transcoder.getInfo();

    // 生成缩略图
    const thumbnailPath = path.join(OUTPUT_DIR, videoId + '.jpg');
    await transcoder.generateThumbnail(thumbnailPath);

    // 转码
    await transcoder.transcodeWithProgress((progress) => {
      console.log(`转码进度：${progress.toFixed(1)}%`);
    });

    res.json({
      success: true,
      message: '转码完成',
      output: {
        video: path.join(OUTPUT_DIR, videoId + '-transcoded.mp4'),
        thumbnail: thumbnailPath
      }
    });

  } catch (error) {
    res.status(500).json({
      success: false,
      message: error.message
    });
  }
});

app.listen(PORT, () => {
  console.log('视频处理服务器已启动，端口：' + PORT);
});

module.exports = app;
```

### 12.2 进度反馈

视频上传和处理的进度反馈对用户体验至关重要。用户在上传大文件或等待视频转码时，需要知道当前进度。进度反馈需要覆盖上传进度和处理进度两部分。

上传进度可以通过XMLHttpRequest的upload.onprogress事件获取，这个在前面已经讲过。对于分片上传，每上传一个分片就知道完成了多少，通过已上传分片数除以总分片数即可计算百分比。

处理进度（如转码）通常无法直接获取，除非处理工具提供进度API。FFmpeg通过输出流提供进度信息，包括当前处理时间戳和总时长。可以解析这些信息来计算百分比。也可以使用轮询方式查询服务端状态，由服务端维护处理进度。

实时进度推送可以使用WebSocket实现。当服务端进度变化时，主动推送更新给客户端。这比客户端轮询更高效，也能更快响应进度变化。

### 12.3 断点续传

视频文件通常很大（如几个GB），断点续传是必需的。视频上传的断点续传实现与前面章节介绍的分片上传+断点续传方案完全一致，核心是：分片上传、记录分片状态、续传时只上传缺失的分片。

对于视频上传，还应该考虑与视频处理的衔接。上传完成后自动触发转码任务，转码完成后通知用户。这些可以通过消息队列（如Redis队列、BullMQ）来解耦上传和处理流程。

视频上传的断点续传还有一些特殊考虑：由于视频文件大，分片数量多，每个分片的元数据管理需要高效；视频文件可能有特定的格式要求，上传前可以进行初步验证（如检查文件头）；视频上传的并发控制需要更保守，避免占用过多资源。

### 12.4 CDN分发

视频上传并处理完成后，需要通过CDN分发到用户。CDN（Content Delivery Network）通过在全球部署边缘节点，让用户从最近的节点获取内容，减少加载时间。

CDN分发的基本流程：源站（OSS/S3）存储原始视频和转码后的多个版本；CDN配置源站回源规则；用户访问视频URL时，CDN判断是否命中缓存，命中则直接返回，未命中则回源拉取并缓存。

视频CDN分发还需要考虑的问题：防盗链（防止他人盗用视频内容）、HLS/DASH流媒体（将视频切成小片段，支持自适应码率）、预热（提前将热门视频推送到CDN节点）。对于点播场景，通常使用HLS或DASH协议，将视频切成几秒到十几秒的TS片段，配合m3u8播放列表，实现流畅的观看体验。

### 12.5 我的思考：上传体验的重要性

文件上传看似是一个简单的功能，但实际上是用户体验的重要体现。一个稳定、快速、易用的上传功能可以显著提升用户满意度，而一个经常失败、体验差的上传功能则会让用户望而却步。

在设计上传功能时，应该考虑以下几个方面：

首先，进度反馈是基础。用户上传一个大文件时，如果没有任何进度提示，用户会感到焦虑。他们不知道上传是否在进行、是否卡住了、还需要多久。特别是对于视频等大文件，缺少进度反馈几乎是不可接受的。除了基本的进度条，还应该显示上传速度、剩余时间等辅助信息，让用户对上传状态有清晰的了解。

其次，断点续传是必需功能。网络不稳定是常态，特别是在移动网络环境下。如果用户上传一个2GB的视频，进行到95%时网络中断，需要从头开始，这会让用户崩溃。支持断点续传可以让用户在任何网络条件下都能顺利完成上传，即使中断也能从断点继续。

第三，错误处理要友好。当上传失败时，应该给用户明确的错误信息和重试选项。是一般性错误还是特定问题（如文件过大、格式不支持），应该区分处理。对于可恢复的错误（如网络波动），应该自动重试；对于不可恢复的错误（如文件格式不支持），应该给用户清晰的提示和解决方案。

第四，上传方式要灵活。不同的场景需要不同的上传方式。头像上传等小文件可以简单直传；大文件应该支持分片上传；需要安全验证的场景可能需要后端签名URL直传到云存储。选择合适的上传方式可以平衡性能、安全性和实现复杂度。

最后，后端处理要及时。用户上传完成后，应该尽快完成后续处理（如视频转码、图片压缩），让内容可以尽快被访问。对于大文件的处理，可以使用消息队列来异步处理，避免阻塞主流程。处理进度也应该及时反馈给用户。

## 总结

本指南系统地介绍了文件上传的各个方面，从基础的HTML表单上传到高级的断点续传、云存储集成。文件上传看似简单，但涉及的知识面很广：HTTP协议、文件系统、流处理、网络安全、图片处理、云服务等。

在实际开发中，应该根据业务需求选择合适的技术方案：小规模应用可以直接使用本地存储；中等规模应用可以使用云存储的标准模式；大规模应用需要考虑分片上传、断点续传、CDN分发等高级功能。

无论选择哪种方案，安全始终是首要考虑。文件上传是Web安全中最常见的攻击向量之一，必须做好充分的验证和防护。文件名验证、MIME类型检查、魔数验证、存储位置隔离、不可执行设置等多层防护缺一不可。

希望本指南能帮助开发者全面掌握文件上传技术，在实际项目中实现稳定、安全、高效的文件上传功能。
