# 即时通讯系统实战完全指南

## 概述

即时通讯（Instant Messaging，IM）系统是现代互联网应用中最重要的场景之一。从微信、钉钉到在线客服、实时协作工具，IM技术无处不在。本文档将系统性地讲解即时通讯系统的需求分析、技术方案、架构设计、前后端实现、性能优化和运维监控，通过实际项目源码（FastDocument）展示完整的IM系统实现。

## 一、即时通讯系统的需求分析

### 1.1 什么是即时通讯

即时通讯的核心目标是**消息的实时传递**，但一个完整的IM系统远不止"发消息"这么简单。从技术角度来看，IM系统需要解决三个核心问题：

**消息传递**：这是IM最基础的功能。用户A发送一条消息，系统需要确保这条消息能够在最短时间内送达用户B。在理想情况下，延迟应该控制在100毫秒以内；在网络波动时，也应该通过重试机制确保消息最终送达。

**状态同步**：除了消息内容，IM还需要同步各种状态信息。用户在线状态（在线、离线、忙碌）、消息状态（已发送、已读）、输入状态（对方正在输入）都属于状态同步的范畴。状态同步的难点在于**一致性**——如何在多个设备、多个客户端之间保持状态的统一。

**离线推送**：当用户不在线时，系统需要暂存消息，等待用户上线后推送。这涉及到消息的持久化存储和推送策略。

### 1.2 核心功能拆解

让我们从功能维度拆解一个完整的IM系统：

**基础消息功能**包括私聊（点对点消息）、群聊（多人会话）、系统通知（系统下发的消息，如系统公告）。消息类型上，需要支持文本消息、图片消息、文件消息、表情消息，以及后续可能扩展的语音、视频消息。

**用户状态功能**包括在线状态感知（知道对方何时在线/离线）、最后活跃时间（显示用户最后活跃时间）、自定义状态（如忙碌、开会中）。

**消息管理功能**包括消息历史（查看历史消息）、消息搜索（搜索消息内容）、消息撤回（撤销已发送的消息）、消息删除（删除本地或全局消息）。

**群组功能**包括创建群组、加入/退出群组、群成员管理、群公告、群消息免打扰模式。

### 1.3 技术挑战分析

即时通讯系统的技术挑战主要来自三个方面：

**实时性挑战**：网络延迟是不可预测的，在跨地域、跨运营商场景下，延迟可能高达数百毫秒甚至秒级。解决方案包括：就近接入（CDN加速）、协议优化（WebSocket优于HTTP轮询）、消息优先级（重要消息优先发送）。

**可靠性挑战**：消息不能丢失，这是IM系统的硬性要求。但网络会中断、服务器会宕机、客户端会崩溃。解决方案包括：消息持久化（数据库存储）、消息确认机制（ACK应答）、重试补偿（自动重发）。

**一致性挑战**：在多设备登录、消息同步、已读未读状态等场景下，如何保证数据一致性？解决方案包括：唯一消息ID（全局递增ID）、版本向量（多设备同步）、最终一致性（允许短暂不一致）。

### 1.4 我的思考：为什么不要重复造轮子 vs 为什么要造轮子

这是一个经典的工程师困境。在IM领域，腾讯云IM、环信、融云等第三方服务已经非常成熟，提供了完整的SDK和API。为什么我们还要自己实现IM系统？

**选择第三方服务的理由**：

第一，开发成本低。第三方服务提供了完整的解决方案，从客户端SDK到服务端API，从消息下发到推送通知，拿来即用。中小型项目的首选。

第二，稳定性有保障。成熟的IM服务经过了海量用户的验证，在各种网络环境下都有良好的表现。不用担心服务器扛不住并发。

第三，运维工作量小。不需要自己维护WebSocket连接池、不需要处理断线重连、不需要担心消息存储的容量规划。

**选择自建IM系统的理由**：

第一，数据安全要求高。金融、医疗、政务等行业对数据有严格的合规要求，不能使用第三方服务。

第二，功能定制化强。通用IM服务很难满足特殊的业务需求，如特殊的消息类型、工作流集成、审批流程等。

第三，成本控制需求。第三方服务按消息量收费，在用户量巨大时成本可能成为瓶颈。自建系统在初期投入后，边际成本趋近于零。

**我的建议**：对于大多数Web应用，除非有特殊的定制化需求，否则应该优先考虑Socket.io配合简单的消息存储来实现基础IM功能。只有在功能复杂、并发要求高、数据安全要求严苛的场景下，才需要考虑完整的自建IM系统。

## 二、技术方案对比

### 2.1 通信协议对比：WebSocket vs SSE vs 长轮询

实现实时通信有多种方案，每种方案都有其适用场景和局限性。

**WebSocket**是目前最主流的实时通信方案。它在客户端和服务器之间建立一个持久的TCP连接，允许服务器主动向客户端推送数据。WebSocket的优势在于：全双工通信（客户端和服务器可以同时发送数据）、低延迟（无需HTTP握手的开销）、轻量级协议头（相比HTTP大幅减少 overhead）。WebSocket的缺点是：需要特殊的服务器支持、复杂的代理处理（部分企业网络可能限制WebSocket）。

**SSE（Server-Sent Events）**是一种服务端推送的技术，它基于HTTP协议，服务器通过特殊的MIME类型（text/event-stream）向客户端发送事件。SSE的优势在于：基于HTTP协议，兼容性好、实现简单。缺点是：只支持服务端向客户端推送，不支持客户端向服务端推送、需要特殊的浏览器支持。

**长轮询（Long Polling）**是一种模拟实时通信的HTTP轮询方案。客户端发送请求后，服务器在没有新消息时保持连接，直到有消息或超时才返回。优点是实现简单、兼容性好。缺点是：延迟不确定（取决于轮询间隔）、服务器压力大（每个客户端占用一个连接）、浪费带宽。

**为什么选择WebSocket**：对于IM系统来说，WebSocket是最佳选择。因为IM系统需要双向通信（客户端发送消息、服务器推送消息）、低延迟响应、以及高效的连接管理。虽然SSE在某些简单场景下可以使用，但对于聊天、协作编辑等复杂场景，WebSocket是必然选择。

### 2.2 Socket.io vs 原生WebSocket

即使选择了WebSocket，开发者还面临一个选择：使用原生WebSocket还是Socket.io？

**原生WebSocket**的优势在于：轻量级，没有额外的抽象层、性能好，直接操作底层协议、灵活，可以精确控制每一个细节。原生WebSocket的缺点是：需要自己实现重连逻辑、需要自己实现心跳机制、浏览器兼容性需要polyfill。

**Socket.io**是一个封装了WebSocket的库，它提供了：自动重连（内置指数退避重连）、消息确认（消息可靠传输）、房间概念（方便实现群聊）、跨浏览器兼容（自动降级到轮询）。但这些特性也带来了代价：额外的协议开销、库体积增加、以及一定的灵活性牺牲。

**为什么选择Socket.io**：在FastDocument项目中，我们选择了Socket.io。原因有三：第一，项目需要跨房间的消息分发（群聊功能），Socket.io的房间概念让这变得简单；第二，自动重连和消息确认让我们可以专注于业务逻辑，而不用处理网络异常；第三，Socket.io的API设计优雅，易于使用和维护。

### 2.3 自建 vs 第三方服务（腾讯云IM）

**第三方IM服务（如腾讯云IM）**的优势：

一站式解决方案，从客户端SDK到服务端API完整提供，集成周期短，通常只需要几天时间。功能丰富，支持音视频通话、消息已读回执、消息撤回、离线推送等高级功能。全球接入，基于腾讯云的基础设施，提供全球就近接入和高质量的传输线路。成本模型清晰，按消息量或按月订阅，长期成本可预估。

**自建IM系统的优势**：

完全可控，从协议设计到服务器部署，每一步都了如指掌。数据私有，所有消息数据都存储在自己的服务器上，适合对数据安全有严格要求的场景。定制灵活，可以根据业务需求自由扩展消息类型、消息处理逻辑、存储方案。长期成本低，在用户量巨大时，边际成本远低于第三方服务。

**FastDocument的选择**：我们采用了自建方案，使用Socket.io配合Redis实现实时消息分发。主要考虑是项目需要与文档协作功能深度集成，第三方服务难以满足这种定制化需求。同时，项目的用户规模预计在几千到几万量级，自建系统完全能够支撑。

### 2.4 我的思考：什么时候该用第三方服务

根据我的经验，以下情况适合使用第三方IM服务：

第一，项目时间紧迫，需要快速上线IM功能。第三方服务可以在一周内完成集成，而自建系统可能需要一个月甚至更长时间。

第二，技术团队规模小，没有专门的IM开发经验。IM系统的复杂性远超表面看起来的样子，维护一个生产级别的IM系统需要专职的工程师。

第三，用户分布在海外，对延迟敏感。第三方服务通常有全球CDN节点，可以提供更好的跨国用户体验。

第四，功能需求与通用IM服务高度吻合。如果你的需求就是聊天、群组、离线推送，没有太多特殊定制，第三方服务是更好的选择。

反之，如果你的业务需要深度定制（如与审批流程集成、与文档协作深度绑定）、有严格的数据安全要求（如金融、医疗行业）、或者预估用户量巨大（百万级以上），自建系统是更合理的选择。

## 三、消息系统设计

### 3.1 消息存储：为什么需要离线存储

消息存储是IM系统的基础设施。当用户在线时，消息通过WebSocket实时送达；当用户离线时，消息需要持久化存储，等待用户上线后推送。

**存储方案选择**：在FastDocument项目中，我们使用PostgreSQL存储消息，使用Redis作为消息缓存和在线状态存储。PostgreSQL的优势在于：支持复杂查询（消息搜索、分页加载）、支持事务（消息状态的原子更新）、成熟稳定。Redis的优势在于：高性能（适合实时消息缓存）、支持发布订阅（消息分发）、支持过期TTL（会话管理）。

**消息存储设计**：一条消息记录需要包含以下字段：

```sql
-- 消息主表
CREATE TABLE messages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    conversation_id UUID NOT NULL,          -- 会话ID（单聊为用户对，群聊为群ID）
    sender_id UUID NOT NULL,                -- 发送者用户ID
    content TEXT NOT NULL,                   -- 消息内容（JSON格式，支持多种类型）
    content_type VARCHAR(20) DEFAULT 'text', -- 内容类型：text/image/file/system
    metadata JSONB,                          -- 扩展信息：提及、回复、表情等
    status VARCHAR(20) DEFAULT 'sent',       -- 消息状态：sending/sent/delivered/read
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    deleted_at TIMESTAMP,                    -- 软删除时间
    INDEX idx_conversation_created (conversation_id, created_at DESC)
);

-- 消息状态表（用于已读未读同步）
CREATE TABLE message_status (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    message_id UUID NOT NULL REFERENCES messages(id),
    user_id UUID NOT NULL,                  -- 阅读者用户ID
    status VARCHAR(20) NOT NULL,             -- read/delivered
    read_at TIMESTAMP,                       -- 阅读时间
    delivered_at TIMESTAMP,                  -- 送达时间
    UNIQUE(message_id, user_id)
);

-- 会话表
CREATE TABLE conversations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    type VARCHAR(20) NOT NULL,               -- 单聊：direct，群聊：group
    name VARCHAR(255),                        -- 群名称
    avatar_url VARCHAR(512),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    last_message_id UUID,                     -- 最后一条消息ID
    last_message_at TIMESTAMP                 -- 最后消息时间
);
```

**离线消息的处理流程**：当用户发送消息时，系统首先将消息持久化到PostgreSQL；然后通过Socket.io推送给当前在线的用户；如果目标用户不在线，则通过离线推送服务（如APNs、FCM）发送推送通知；用户下次上线时，从数据库拉取离线期间的消息。

### 3.2 消息状态：发送中、已发送、已读

消息状态是IM系统中另一个核心概念。从发送者的视角，一条消息会经历以下状态：

**sending（发送中）**：消息已写入本地UI，正在等待服务器确认。这是乐观更新的体现——用户发送消息后，UI立即显示消息，无需等待服务器响应。

**sent（已发送）**：服务器已收到消息，并返回了确认。这条消息已经被持久化到数据库。

**delivered（已送达）**：服务器已确认消息送达接收者。对于在线用户，这意味着接收者的客户端已经收到消息；对于离线用户，这一步会在消息下次送达时完成。

**read（已读）**：接收者已经查看了这条消息。已读状态通常由接收者的客户端在打开会话时上报。

```typescript
// FastDocument 中的消息状态类型定义
export type MessageStatus = 'sending' | 'sent' | 'delivered' | 'read' | 'failed';

/**
 * 消息接口
 */
export interface Message {
  id: string;
  conversationId: string;
  senderId: string;
  senderName: string;
  content: string;
  contentType: 'text' | 'image' | 'file' | 'system';
  status: MessageStatus;
  mentions?: string[];        // 提及的用户
  replyTo?: string;           // 回复的消息ID
  reactions?: Reaction[];      // 表情反应
  createdAt: number;
  deliveredAt?: number;
  readAt?: number;
}

/**
 * 表情反应
 */
export interface Reaction {
  emoji: string;
  userId: string;
  userName: string;
  addedAt: number;
}
```

### 3.3 消息同步：多设备同步问题

多设备同步是IM系统中最复杂的问题之一。当用户在手机、平板、电脑多个设备登录时，一条消息需要在所有设备上保持一致。

**同步策略**：我们采用"时间戳 + 版本号"的同步策略。每条消息携带创建时间戳，客户端根据时间戳判断消息的先后顺序。对于多设备同步，使用增量同步——客户端记录最后一次同步的时间戳，下次同步时只拉取该时间戳之后的消息。

```typescript
/**
 * 消息同步请求参数
 */
interface SyncMessagesParams {
  conversationId: string;
  since?: string;           // 最后同步时间戳
  limit?: number;            // 每次最多拉取消息数
}

/**
 * 消息同步响应
 */
interface SyncMessagesResponse {
  messages: Message[];
  hasMore: boolean;
  syncTimestamp: string;     // 本次同步的时间戳，用于下次增量同步
}
```

**冲突处理**：在极端情况下（如网络分区），不同设备可能对同一条消息有不同的状态（如已读/未读）。我们的处理策略是：以服务器状态为准。当客户端同步时，服务器返回的消息状态会覆盖客户端的本地状态。这样做的好处是简化逻辑，代价是可能的短暂不一致。

### 3.4 我的思考：消息系统最复杂的地方在哪

很多人认为IM系统的难点在于实时性——WebSocket连接管理、消息推送的高效分发。但经过实际项目实践，我认为**消息状态的同步和一致性**才是最复杂的地方。

**状态爆炸问题**：考虑一个群聊场景。一个500人的群，一条消息需要更新500个用户的未读计数。如果每个用户有3个设备登录，还需要考虑跨设备的状态同步。这个问题的本质是：状态变更的复杂度与用户数和设备数的乘积成正比。

**已读回执的性能问题**：已读回执需要精确地记录"谁在什么时间读了哪条消息"。在大群场景下，一条消息可能有几百个已读状态记录。如果用户快速滑动聊天记录，会有大量的已读状态需要更新，这对数据库的压力是巨大的。

**解决方案的权衡**：在实际项目中，我们采用了"弱一致性"的策略。不追求精确的已读未读状态，而是使用"大致准确"的未读计数。例如，群消息的未读数可以容忍一定的误差，因为用户真正关心的是"有没有新消息"，而不是精确的"第几条消息未读"。

## 四、实时同步实现

### 4.1 在线状态：如何知道对方是否在线

在线状态是IM系统的基础功能之一。FastDocument项目通过Socket.io的房间机制实现在线状态感知。

**实现原理**：每个Socket.io连接对应一个用户。当用户连接到服务器时，在`handleConnection`回调中记录用户的在线状态；当用户断开连接时，在`handleDisconnect`回调中清除状态。用户加入文档房间时，服务器维护一个房间用户列表，并向房间内所有用户广播在线用户更新。

```typescript
// FastDocument 后端 - documents.gateway.ts
// 在线用户管理
private roomUsers: Record<string, { id: string; name: string }[]> = {};

// 用户连接时记录在线状态
handleConnection(client: Socket) {
  console.log(`用户连接: ${client.id}`);
}

// 用户断开时清除在线状态
async handleDisconnect(client: Socket) {
  const userId = client.id;
  // 清除用户在 Redis 中的光标数据
  await this.redis.del(`cursor:${userId}`);

  // 从所有记录的房间中移除该用户
  for (const docId in this.roomUsers) {
    const initialLength = this.roomUsers[docId].length;
    this.roomUsers[docId] = this.roomUsers[docId].filter(
      (u) => u.id !== userId,
    );

    // 如果房间人数发生变化，向该房间广播最新的用户列表
    if (this.roomUsers[docId].length !== initialLength) {
      this.server.to(docId).emit('onlineUsersUpdate', this.roomUsers[docId]);
    }
  }

  console.log(`用户断开: ${userId}`);
}

// 用户加入房间时更新在线列表
@SubscribeMessage('joinDocument')
async handleJoinRoom(
  @MessageBody() data: { docId: string; userName?: string; token?: string },
  @ConnectedSocket() client: Socket,
) {
  const { docId, userName, token } = data;

  // ... 认证逻辑省略

  client.join(docId);

  // 初始化房间用户列表
  if (!this.roomUsers[docId]) {
    this.roomUsers[docId] = [];
  }

  // 记录用户信息
  const existingUserIndex = this.roomUsers[docId].findIndex((u) => u.id === userId);
  if (existingUserIndex >= 0) {
    this.roomUsers[docId][existingUserIndex].name = validatedName;
  } else {
    this.roomUsers[docId].push({
      id: userId,
      name: validatedName,
    });
  }

  // 向房间内所有人广播当前在线用户列表
  this.server.to(docId).emit('onlineUsersUpdate', this.roomUsers[docId]);
}
```

**在线状态的前端处理**：前端通过监听Socket.io的连接状态事件来感知服务器连接状态，通过监听`onlineUsersUpdate`事件来感知房间内其他用户的在线情况。


### 4.2 消息推送：如何保证消息不丢失

消息不丢失是IM系统的核心要求。我们通过以下机制确保消息的可靠传递：

**消息持久化**：所有消息首先写入数据库，确保消息不会因服务器重启而丢失。

**消息确认机制**：当服务器收到客户端发送的消息时，会向客户端返回一个确认（ACK）。如果客户端在一定时间内没有收到确认，会触发重试逻辑。

**服务器广播优化**：在FastDocument中，我们采用了"立即广播 + 异步持久化"的策略。当服务器收到块更新时，先立即广播给其他用户（低延迟体验），然后异步持久化到数据库（可靠性保障）。

```typescript
// FastDocument 后端 - 消息处理
@SubscribeMessage('updateBlock')
async handleUpdateBlock(
  @MessageBody()
  data: {
    docId: string;
    blockId: string;
    content: string;
    type?: string;
    properties?: any;
  },
  @ConnectedSocket() client: Socket,
) {
  try {
    // 更新房间活动状态
    this.updateRoomActivity(data.docId);

    // 1. 立即广播变更给房间内的其他用户（实现低延迟体验）
    client.to(data.docId).emit('blockUpdated', data);

    // 2. 异步将变更持久化到数据库
    // 即使数据库写入失败，消息也已经广播给在线用户
    await this.documentsService.updateBlock(
      data.docId,
      data.blockId,
      data.content,
      data.type,
      data.properties,
    );
  } catch (error) {
    console.error('[DocumentsGateway] 更新块失败:', error);
    // 通知客户端更新失败
    client.emit('error', {
      message: '保存块失败',
      blockId: data.blockId,
    });
  }
}
```

### 4.3 心跳机制：为什么需要心跳

心跳（Heartbeat）是维持连接活跃的重要机制。它的作用有三个：

**检测连接的存活状态**：TCP连接在某些情况下可能处于"半开"状态——一方认为连接正常，另一方认为连接已断开。心跳可以帮助检测这种状态。

**保持代理/负载均衡器的连接**：企业网络中的HTTP代理、nginx等中间件，通常会对空闲连接设置超时时间。心跳可以保持连接的活跃，避免被中间件断开。

**及时发现网络故障**：如果连续几次心跳失败，可以判定网络出现故障，触发重连逻辑。

```typescript
// Socket.io 内置了心跳机制，配置如下：
this.socket = io(SOCKET_URL, {
  reconnectionAttempts: 5,
  timeout: 10000,
  // Socket.io 默认心跳间隔是 25 秒，超时是 60 秒
  // 这些值可以通过以下选项自定义
  // pingInterval: 25000,
  // pingTimeout: 60000,
});

// 前端可以监听心跳事件来显示连接状态
this.socket.on("ping", () => {
  console.log("心跳发送");
});

this.socket.on("pong", () => {
  console.log("心跳响应");
});
```

### 4.4 断线重连：如何优雅重连

网络中断是常态，不是异常。IM系统必须能够优雅地处理断线重连。

**Socket.io内置重连**：Socket.io已经内置了自动重连机制，会在连接断开后自动尝试重连。默认采用指数退避策略：第一次重连等待1秒，第二次2秒，第三次4秒，以此类推，最大等待时间不超过60秒。

**重连时的状态恢复**：重连后需要恢复之前的订阅状态。Socket.io提供了`io.of("/").on("reconnect", ...)`事件，但更可靠的方式是保存当前的会话状态，在重连后重新join房间。

```typescript
// 前端重连管理
class SocketClient {
  private socket: Socket | null = null;
  private currentDocId: string | null = null;  // 保存当前房间ID

  connect(onStatusChange?: (online: boolean) => void) {
    if (!this.socket) {
      this.socket = io(SOCKET_URL, {
        reconnectionAttempts: 5,
        timeout: 10000,
      });

      this.socket.on("connect", () => {
        console.log("成功连接至 WebSocket 服务器");
        if (onStatusChange) onStatusChange(true);

        // 重连后恢复之前的房间订阅
        if (this.currentDocId) {
          this.joinDocument(this.currentDocId, this.currentUserName);
        }
      });

      this.socket.on("disconnect", () => {
        logWarn("与 WebSocket 服务器断开连接");
        if (onStatusChange) onStatusChange(false);
      });

      // 监听重连事件
      this.socket.on("reconnect_attempt", (attemptNumber) => {
        console.log(`重连尝试 ${attemptNumber}`);
      });

      this.socket.on("reconnect_failed", () => {
        console.error("重连失败");
        // 可以提示用户检查网络
      });
    }
    return this.socket;
  }

  joinDocument(docId: string, userName: string) {
    this.currentDocId = docId;  // 保存状态
    this.currentUserName = userName;
    if (this.socket) {
      this.socket.emit("joinDocument", { docId, userName });
    }
  }
}
```

## 五、群聊系统

### 5.1 群消息分发：扩散点 vs 扩散读

群聊的消息分发有两种常见模式：

**扩散写（Write Fan-out）**：发送消息时，服务器将消息分别推送给每个群成员。这是最简单的实现，但缺点是当群成员数量巨大时，服务器需要同时处理大量推送，性能压力大。

**扩散读（Read Fan-out）**：发送消息时，消息只写入一个共享的收件箱（通常是数据库表），每个群成员从自己的收件箱读取消息。这种方式在成员数量大时有更好的扩展性，但增加了读取延迟。

```sql
-- 扩散写模式：消息表
-- 每条消息只存储一条记录，通过查询获取所有群成员
CREATE TABLE group_messages (
    id UUID PRIMARY KEY,
    group_id UUID NOT NULL,
    sender_id UUID NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 扩散读模式：用户消息收件箱
-- 每条消息为每个接收者创建一条记录
CREATE TABLE user_inbox (
    id UUID PRIMARY KEY,
    user_id UUID NOT NULL,
    group_id UUID NOT NULL,
    message_id UUID NOT NULL REFERENCES group_messages(id),
    is_read BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP DEFAULT NOW()
);
```

FastDocument项目采用扩散写模式，因为文档协作的成员数量通常较小（通常不超过几十人），扩散写的低延迟优势更加明显。

### 5.2 历史消息：为什么要分页加载

当用户打开一个历史悠久的群聊时，不可能一次性加载所有消息。分页加载是必然选择。

**分页策略**：我们采用"时间倒序、分页加载"的策略。用户打开群聊时，首先加载最近的N条消息；向上滚动时，再加载更早的消息。

```typescript
      skip: options.offset,
```

```typescript
  @ConnectedSocket() client: Socket,

  const { groupId, userId } = data;

  // 检查用户是否已被禁言
  const isMuted = await this.redis.get(`group:${groupId}:muted:${userId}`);

  client.leave(groupId);

    userId: targetUserId,
    action: 'mute',
    operatorId: client.id,
    timestamp: Date.now(),
  });
```

### 5.4 我的思考：群聊比单聊复杂多少

群聊的复杂度远超单聊，主要体现在以下几个方面：

**消息分发的复杂度**：单聊中，一条消息只需要发送给一个人。群聊中，一条消息需要发送给所有群成员。在一个有500人的群里，服务器需要处理500次消息推送。即使每次推送只需要几毫秒，500次也是数百毫秒的延迟。

**状态管理的复杂度**：单聊中，在线状态只有两个人。群聊中，需要管理几十甚至几百个人的在线状态。当用户列表变化时，需要广播给所有群成员。这个广播的开销随着群成员数量线性增长。

**权限控制的复杂度**：单聊中，发送者和接收者是平等的关系。群聊中，需要区分群主、管理员、普通成员。不同角色有不同的权限——只有群主可以禁言、只有管理员可以踢人、敏感消息可能需要特定权限才能查看。

**我的建议**：在设计群聊系统时，应该设置合理的群成员上限。微信群最大500人、钉钉群最大2000人，这些限制不仅是用户体验的考虑，更是系统性能的保障。对于超大规模群组（如直播弹幕），应该考虑采用不同的技术方案（如写扩散改为读扩散）。

## 六、前端实现

### 6.1 聊天界面设计：为什么要用虚拟列表

当聊天记录达到成百上千条时，如何高效渲染消息列表成为性能瓶颈。传统的`map`渲染所有消息，会导致大量的DOM节点，内存占用高，滚动卡顿。

**虚拟列表（Virtual List）**是解决方案。虚拟列表的核心思想是：只渲染当前可视区域内的消息，其他消息在滚动时动态替换。这样，无论消息总数有多少，DOM节点数量始终保持在一个可控范围内。

```typescript
import { useState, useEffect, useRef, useCallback } from 'react';

/**
 * 虚拟列表 Hook
 * 用于高效渲染大量聊天消息
 */
export function useVirtualList<T>({
  items,                // 完整的数据列表
  itemHeight,           // 每项的高度（px）
  containerHeight,      // 容器高度
  overscan = 5,         // 额外渲染的项目数（上下各5个）
}: {
  items: T[];
  itemHeight: number;
  containerHeight: number;
  overscan?: number;
}) {
  const [scrollTop, setScrollTop] = useState(0);
  const containerRef = useRef<HTMLDivElement>(null);

  // 计算可视区域能显示多少个项目
  const visibleCount = Math.ceil(containerHeight / itemHeight);

  // 计算起始索引（考虑 overscan）
  const startIndex = Math.max(0, Math.floor(scrollTop / itemHeight) - overscan);

  // 计算结束索引
  const endIndex = Math.min(
    items.length - 1,
    startIndex + visibleCount + overscan * 2
  );

  // 获取可视区域内的项目
  const visibleItems = items.slice(startIndex, endIndex + 1);

  // 计算总高度（用于创建滚动空间）
  const totalHeight = items.length * itemHeight;

  // 计算偏移量（用于定位可视区域）
  const offsetY = startIndex * itemHeight;

  // 处理滚动事件
  const handleScroll = useCallback((e: React.UIEvent<HTMLDivElement>) => {
    setScrollTop(e.currentTarget.scrollTop);
  }, []);

  return {
    containerRef,
    visibleItems,
    totalHeight,
    offsetY,
    startIndex,
    endIndex,
    handleScroll,
    // 滚动到底部的方法
    scrollToBottom: () => {
      if (containerRef.current) {
        containerRef.current.scrollTop = containerRef.current.scrollHeight;
      }
    },
    // 滚动到指定索引的方法
    scrollToIndex: (index: number) => {
      if (containerRef.current) {
        containerRef.current.scrollTop = index * itemHeight;
      }
    },
  };
}

/**
 * 聊天消息列表组件
 */

      // 注意：可能需要防抖处理
    }
```

### 6.3 输入组件：为什么需要草稿功能

用户在输入消息时，可能会因为各种原因切换到其他界面（如查看文档、回复其他消息）。如果输入的内容丢失，会严重影响用户体验。

**草稿功能**通过`localStorage`或`sessionStorage`保存用户的输入内容，在用户返回时恢复。

```typescript
/**
 * 聊天输入框组件（带草稿功能）
 */
function ChatInput({
  conversationId,
  onSend,
  onTyping,
}: {
  conversationId: string;
  onSend: (content: string, mentions?: string[]) => void;
  onTyping?: () => void;
}) {
  const [content, setContent] = useState('');
  const [mentions, setMentions] = useState<string[]>([]);
  const textareaRef = useRef<HTMLTextAreaElement>(null);
  const typingTimeoutRef = useRef<NodeJS.Timeout>();

  // 加载草稿
  useEffect(() => {
    const draft = localStorage.getItem(`chat_draft_${conversationId}`);
    if (draft) {
      const { content: draftContent, mentions: draftMentions } = JSON.parse(draft);
      setContent(draftContent);
      setMentions(draftMentions || []);
    }
  }, [conversationId]);

  // 保存草稿（防抖）
  useEffect(() => {
    const timeout = setTimeout(() => {
      if (content.trim()) {
        localStorage.setItem(
          `chat_draft_${conversationId}`,
          JSON.stringify({ content, mentions })
        );
      } else {
        localStorage.removeItem(`chat_draft_${conversationId}`);
      }
    }, 500);  // 500ms 防抖

    return () => clearTimeout(timeout);
  }, [content, mentions, conversationId]);

  // 发送消息后清除草稿
  const handleSend = () => {
    if (content.trim()) {
      onSend(content, mentions);
      setContent('');
      setMentions([]);
      localStorage.removeItem(`chat_draft_${conversationId}`);
    }
  };

  // 输入变化时触发正在输入状态
  const handleChange = (e: React.ChangeEvent<HTMLTextAreaElement>) => {
    setContent(e.target.value);

    // 检测 @ 提及
    const mentionMatches = e.target.value.match(/@(\w+)/g);
    if (mentionMatches) {
      setMentions(mentionMatches.map(m => m.slice(1)));
    }

    // 通知正在输入
    if (onTyping) {
      clearTimeout(typingTimeoutRef.current);
      onTyping();
      typingTimeoutRef.current = setTimeout(() => {
        // 停止输入通知
      }, 3000);
    }
  };

  // 自动调整高度
  const handleKeyDown = (e: React.KeyboardEvent) => {
    if (e.key === 'Enter' && !e.shiftKey) {
      e.preventDefault();
      handleSend();
    }
  };

  return (
    <div style={{ display: 'flex', gap: '8px', padding: '12px' }}>
      <textarea
        ref={textareaRef}
        value={content}
        onChange={handleChange}
        onKeyDown={handleKeyDown}
        placeholder="输入消息..."
        style={{
          flex: 1,
          minHeight: '40px',
          maxHeight: '120px',
          padding: '8px 12px',
          borderRadius: '20px',
          border: '1px solid #d9d9d9',
          resize: 'none',
        }}
      />
      <button onClick={handleSend} disabled={!content.trim()}>
        发送
      </button>
    </div>
  );
}
```

## 七、性能与优化

### 7.1 消息渲染优化：为什么消息多会卡

当聊天记录达到成千上万条时，即使使用虚拟列表，仍然可能遇到性能问题。常见原因包括：

**频繁的状态更新**：每当收到新消息或状态变化时，React组件会重新渲染。如果使用不当，可能导致整棵树重新渲染。

**复杂的消息气泡组件**：每条消息都包含头像、用户名、消息内容、时间戳、状态图标、表情回复等元素，如果每个元素都是独立的组件，渲染开销会很大。

**图片消息的解码和布局**：图片消息需要解码、缩放、布局计算，这些操作可能阻塞主线程。

**优化策略**：

```typescript
// 1. 使用 React.memo 避免不必要的重渲染

  return (
```

### 7.2 网络优化：为什么需要消息压缩

消息压缩主要有两个目的：减少带宽消耗、降低传输延迟。

**文本压缩**：对于JSON格式的消息体，可以使用gzip或brotli压缩。Socket.io支持自动压缩，但也可以在应用层对消息内容进行编码优化（如使用更短的消息类型枚举）。

**增量同步**：不要每次都拉取全量消息，只拉取增量。这在消息量大时效果显著。

```typescript
// 增量同步实现
class MessageService {
  private lastSyncTimestamps: Map<string, number> = new Map();

  /**
   * 增量同步消息
   */
  async syncMessages(conversationId: string): Promise<Message[]> {
    const lastSyncTime = this.lastSyncTimestamps.get(conversationId);

    // 如果有上次同步时间，只拉取该时间之后的消息
    if (lastSyncTime) {
      const messages = await this.fetchMessagesSince(conversationId, lastSyncTime);
      // 更新同步时间戳
      this.lastSyncTimestamps.set(conversationId, Date.now());
      return messages;
    } else {
      // 首次同步，只拉取最近的N条消息
      const messages = await this.fetchRecentMessages(conversationId, 50);
      this.lastSyncTimestamps.set(conversationId, Date.now());
      return messages;
    }
  }

  /**
   * 获取最近的消息（首次同步）
   */
  private async fetchRecentMessages(conversationId: string, limit: number): Promise<Message[]> {
    // 从数据库获取最近的消息
    return db.messages.find({
      where: { conversationId },
      order: { createdAt: 'DESC' },
      take: limit,
    });
  }

  /**
   * 获取增量消息
   */
  private async fetchMessagesSince(conversationId: string, since: number): Promise<Message[]> {
    // 从数据库获取指定时间之后的增量消息
    return db.messages.find({
      where: {
        conversationId,
        createdAt: { gte: since },
      },
      order: { createdAt: 'ASC' },
    });
  }
}
```

### 7.3 存储优化：为什么要本地存储

本地存储在IM系统中扮演重要角色，它可以：

**加速首屏加载**：用户打开聊天界面时，不需要等待服务器返回消息，本地存储中的消息可以立即显示。

**支持离线浏览**：用户断网时，可以浏览已经加载过的消息，提供更好的离线体验。

**减少服务器压力**：很多查询可以在本地完成，不需要每次都请求服务器。

```typescript
// 基于 IndexedDB 的本地消息存储
class LocalMessageStore {
  private dbName = 'chat_messages';
  private dbVersion = 1;
  private db: IDBDatabase | null = null;

  async initialize(): Promise<void> {
    return new Promise((resolve, reject) => {
      const request = indexedDB.open(this.dbName, this.dbVersion);

      request.onerror = () => reject(request.error);
      request.onsuccess = () => {
        this.db = request.result;
        resolve();
      };

      request.onupgradeneeded = (event) => {
        const db = (event.target as IDBOpenDBRequest).result;

        // 创建消息存储
        if (!db.objectStoreNames.contains('messages')) {
          const store = db.createObjectStore('messages', { keyPath: 'id' });
          store.createIndex('conversationId', 'conversationId', { unique: false });
          store.createIndex('createdAt', 'createdAt', { unique: false });
        }
      };
    });
  }

  /**
   * 保存消息到本地
   */
  async saveMessages(conversationId: string, messages: Message[]): Promise<void> {
    if (!this.db) await this.initialize();

    const transaction = this.db!.transaction(['messages'], 'readwrite');
    const store = transaction.objectStore('messages');

    for (const message of messages) {
      store.put({ ...message, conversationId });
    }

    return new Promise((resolve, reject) => {
      transaction.oncomplete = () => resolve();
      transaction.onerror = () => reject(transaction.error);
    });
  }

  /**
   * 获取本地消息
   */
  async getMessages(
    conversationId: string,
    options: { before?: number; limit?: number } = {}
  ): Promise<Message[]> {
    if (!this.db) await this.initialize();

    return new Promise((resolve, reject) => {
      const transaction = this.db!.transaction(['messages'], 'readonly');
      const store = transaction.objectStore('messages');
      const index = store.index('conversationId');

      const messages: Message[] = [];
      const request = index.openCursor(IDBKeyRange.only(conversationId), 'prev');

      let count = 0;
      const maxCount = options.limit || 50;

      request.onsuccess = (event) => {
        const cursor = (event.target as IDBRequest).result;
        if (cursor && count < maxCount) {
          // 如果指定了 before 时间过滤
          if (!options.before || cursor.value.createdAt < options.before) {
            messages.push(cursor.value);
            count++;
          }
          cursor.continue();
        } else {
          resolve(messages);
        }
      };

      request.onerror = () => reject(request.error);
    });
  }

  /**
   * 清理旧消息（保留最近的消息）
   */
  async cleanupOldMessages(keepCount: number = 1000): Promise<void> {
    if (!this.db) await this.initialize();

    // 获取所有会话ID
    const conversationIds = await this.getAllConversationIds();

    for (const conversationId of conversationIds) {
      const messages = await this.getMessages(conversationId, { limit: keepCount * 2 });

      if (messages.length > keepCount) {
        // 删除多余的消息（保留最新的）
        const toDelete = messages.slice(keepCount);
        const transaction = this.db!.transaction(['messages'], 'readwrite');
        const store = transaction.objectStore('messages');

        for (const message of toDelete) {
          store.delete(message.id);
        }
      }
    }
  }
}
```

### 7.4 我的思考：IM的性能优化有哪些坑

在IM系统的性能优化过程中，有几个常见的坑需要避免：

**过早优化**：很多性能问题在用户量小、数据量小时根本不会出现。在没有实际数据支撑的情况下，不要花大量时间优化可能根本不存在的性能问题。先让产品跑起来，再根据实际监控数据进行优化。

**过度优化**：为了减少几毫秒的渲染时间，引入了大量复杂的优化代码，导致代码可维护性下降，这是不值得的。优化的目的是提升用户体验，而不是炫技。

**忽视移动端**：桌面浏览器的性能通常远好于移动设备。很多在桌面端流畅运行的功能，在低端手机上可能严重卡顿。务必在真实移动设备上测试IM系统的性能。

**内存泄漏**：WebSocket连接、事件监听器、定时器如果没有正确清理，会导致内存泄漏。在SPA中，用户频繁切换页面时尤其容易出现这个问题。务必养成在组件卸载时清理所有资源的习惯。

**我的建议**：性能优化的正确姿势是：首先使用Chrome DevTools的Performance和Memory面板定位瓶颈；然后针对性地优化影响最大的地方；最后用数据验证优化效果。切忌凭感觉优化。

## 八、上线与运维

### 8.1 为什么IM需要独立服务

在传统的Web应用架构中，所有功能通常部署在同一个应用中。但IM系统有其特殊性，建议独立部署：

**资源隔离**：WebSocket连接是长连接，会占用服务器资源。如果IM和其他业务混部，当IM消息量突增时，可能影响其他业务。

**独立扩展**：IM的扩展需求可能与其他业务不同。深夜时，IM的连接数可能远高于白天；活动期间，IM的消息量可能暴涨。独立部署可以针对IM的特点进行容量规划。

**版本独立**：IM功能相对稳定，不需要频繁发布。独立部署可以减少发布其他业务时对IM的影响。

**故障隔离**：当其他服务出现故障时，IM仍然可以正常工作。这对于需要保持沟通的场景（如客服系统）尤为重要。

### 8.3 容量预估：支持多少人同时在线

容量预估是IM系统上线前的必备工作。主要考虑以下因素：

**连接数限制**：一个Node.js进程默认可以支持几千到一万个WebSocket连接。如果需要支持更多连接，需要多进程部署（利用Node.js的cluster模块或PM2）。

**内存限制**：每个WebSocket连接会占用一定的内存（通常几十KB）。一个4GB内存的服务器，大约可以支持4万到10万个并发连接（具体取决于消息处理逻辑的复杂度）。

**带宽限制**：假设平均每个用户每秒接收1条消息，每条消息平均1KB，那么1万个在线用户需要约10MB/s的出口带宽。

```typescript
// 容量计算示例
interface CapacityPlan {
  maxConnections: number;      // 最大连接数
  maxMessagesPerSecond: number; // 最大每秒消息量
  requiredBandwidth: number;    // 所需带宽（MB/s）
  requiredMemory: number;       // 所需内存（GB）
  instanceCount: number;         // 需要的服务实例数
}

// 基于配置计算容量
function calculateCapacity(targetUsers: number): CapacityPlan {
  // 假设参数
  const avgConnectionsPerUser = 2;  // 每个用户平均2个设备
  const peakFactor = 3;              // 峰值是在线的3倍
  const avgMessagesPerUserPerMinute = 10;
  const avgMessageSizeKB = 2;
  const memoryPerConnectionMB = 0.05;

  const peakConnections = targetUsers * avgConnectionsPerUser * peakFactor;
  const messagesPerSecond = (targetUsers * avgMessagesPerUserPerMinute) / 60;
  const bandwidthMBps = (peakConnections * avgMessageSizeKB) / 1024;
  const memoryGB = (peakConnections * memoryPerConnectionMB) / 1024;

  // 每个实例支持约1万连接
  const connectionsPerInstance = 10000;
  const instanceCount = Math.ceil(peakConnections / connectionsPerInstance);

  return {
    maxConnections: peakConnections,
    maxMessagesPerSecond: messagesPerSecond,
    requiredBandwidth: bandwidthMBps,
    requiredMemory: memoryGB,
    instanceCount,
  };
}

// 示例：预估1万用户的容量
const plan = calculateCapacity(10000);
console.log(`
容量预估（1万用户）：
- 峰值连接数: ${plan.maxConnections}
- 最大消息量: ${plan.maxMessagesPerSecond} 条/秒
- 所需带宽: ${plan.requiredBandwidth.toFixed(2)} MB/s
- 所需内存: ${plan.requiredMemory.toFixed(2)} GB
- 推荐实例数: ${plan.instanceCount}
`);
```

### 8.4 我的思考：IM运维和普通服务有什么区别

IM系统的运维比普通Web服务更复杂，主要体现在以下几点：

**长连接维护**：普通HTTP请求是短连接，请求结束就断开。WebSocket是长连接，连接会持续很长时间。这意味着服务器需要维护大量的连接状态，对服务器的资源管理提出更高要求。

**消息顺序性**：在分布式系统中，消息的顺序性很难保证。两个用户可能因为网络延迟不同，看到的消息顺序不一致。运维时需要考虑消息顺序的监控和告警。

**峰值压力**：普通Web服务的流量曲线通常比较平滑。IM服务则可能在活动期间出现爆发式增长（如明星粉丝群、直播弹幕）。运维需要提前做好容量规划，并准备扩容预案。

**客户端兼容性**：不同版本的客户端可能行为不一致。某些版本可能有bug导致消息发送失败、连接异常等。运维需要支持多版本并存，并能够追踪特定版本的错误。

**我的建议**：IM系统的运维应该建立完善的监控告警体系，包括连接数告警、消息延迟告警、错误率告警等。同时，建立日常巡检制度，定期检查服务器的连接状态分布、消息队列积压情况、Redis内存使用等。

### 9.2 后端Socket.io网关完整实现

以下是FastDocument项目中后端Socket.io网关的完整实现：

```typescript
// ============================================
// WebSocket 网关 - documents.gateway.ts
// ============================================

import {
  WebSocketGateway,
  SubscribeMessage,
  MessageBody,
  ConnectedSocket,
  WebSocketServer,
  OnGatewayInit,
  OnGatewayConnection,
  OnGatewayDisconnect,
} from '@nestjs/websockets';
import { Server, Socket } from 'socket.io';
import { DocumentsService } from './documents.service';
import Redis from 'ioredis';
import { appEnv } from '../config/env';
import * as jwt from 'jsonwebtoken';

/**
 * 实时协作网关
 * 集成多端内容同步、光标状态追踪、聊天消息处理
 */
const ALLOWED_ORIGINS = appEnv.allowedOrigins;

@WebSocketGateway({
  cors: {
    origin: ALLOWED_ORIGINS,
    methods: ['GET', 'POST'],
    credentials: true,
  },
})
export class DocumentsGateway
  implements OnGatewayInit, OnGatewayConnection, OnGatewayDisconnect
{
  @WebSocketServer()
  server: Server;

  private redis: Redis;
  private readonly jwtSecret: string;
  private readonly MAX_USERNAME_LENGTH = 50;
  private readonly ROOM_TTL = 30 * 60 * 1000;
  private roomLastActivity: Record<string, number> = {};

  // 节流配置
  private typingThrottle: Record<string, number> = {};
  private cursorThrottle: Record<string, number> = {};
  private readonly THROTTLE_INTERVAL = 100;

  constructor(private readonly documentsService: DocumentsService) {
    this.jwtSecret = appEnv.auth.jwtSecret;

    // 初始化 Redis 连接
    this.redis = new Redis({
      host: appEnv.redis.host,
      port: appEnv.redis.port,
      retryStrategy: (times: number) => {
        if (times > 10) {
          console.error('[DocumentsGateway] Redis 重连失败次数过多');
          return null;
        }
        const delay = Math.min(times * 200, 5000);
        return delay;
      },
      maxRetriesPerRequest: 3,
      connectTimeout: 10000,
    });

    this.redis.on('connect', () => console.log('[DocumentsGateway] Redis 连接已建立'));
    this.redis.on('ready', () => console.log('[DocumentsGateway] Redis 已就绪'));
    this.redis.on('error', (err: Error) => console.error('[DocumentsGateway] Redis 错误:', err.message));

    // 定期清理不活跃房间
    setInterval(() => this.cleanupInactiveRooms(), 5 * 60 * 1000);
  }

  afterInit(server: Server) {
    console.log('协作网关已初始化');
  }

  private roomUsers: Record<string, { id: string; name: string }[]> = {};

  // 清理不活跃房间
  private cleanupInactiveRooms(): void {
    const now = Date.now();
    const inactiveRooms: string[] = [];

    for (const [docId, lastActivity] of Object.entries(this.roomLastActivity)) {
      if (now - lastActivity > this.ROOM_TTL) {
        inactiveRooms.push(docId);
      }
    }

    for (const docId of inactiveRooms) {
      if (this.roomUsers[docId] && this.roomUsers[docId].length === 0) {
        delete this.roomUsers[docId];
        delete this.roomLastActivity[docId];
        console.log(`[DocumentsGateway] 清理不活跃房间: ${docId}`);
      }
    }
  }

  // 更新房间活动状态
  private updateRoomActivity(docId: string): void {
    this.roomLastActivity[docId] = Date.now();
  }

  // 验证 JWT token
  private validateToken(token: string): { id: string; name: string } | null {
    try {
      const decoded = jwt.verify(token, this.jwtSecret) as any;
      return {
        id: decoded.sub || decoded.id,
        name: decoded.username || decoded.name || '未知用户',
      };
    } catch (error) {
      console.error('[DocumentsGateway] JWT 验证失败:', error);
      return null;
    }
  }

  // 检查用户权限
  private async checkUserAccess(docId: string, userId: string): Promise<boolean> {
    try {
      return await this.documentsService.checkUserAccess(docId, userId);
    } catch (error) {
      console.error('[DocumentsGateway] 检查用户权限失败:', error);
      return false;
    }
  }

  // 过滤用户输入
  private sanitizeInput(input: string, maxLength: number): string {
    const stripped = input.replace(/<[^>]*>/g, '');
    return stripped.substring(0, maxLength).trim() || '匿名用户';
  }

  // 连接处理
  handleConnection(client: Socket) {
    console.log(`用户连接: ${client.id}`);
  }

  // 断开连接处理
  async handleDisconnect(client: Socket) {
    const userId = client.id;
    await this.redis.del(`cursor:${userId}`);

    for (const docId in this.roomUsers) {
      const initialLength = this.roomUsers[docId].length;
      this.roomUsers[docId] = this.roomUsers[docId].filter((u) => u.id !== userId);

      if (this.roomUsers[docId].length !== initialLength) {
        this.server.to(docId).emit('onlineUsersUpdate', this.roomUsers[docId]);
      }
    }

    console.log(`用户断开: ${userId}`);
  }

  // 加入文档房间
  @SubscribeMessage('joinDocument')
  async handleJoinRoom(
    @MessageBody() data: { docId: string; userName?: string; token?: string },
    @ConnectedSocket() client: Socket,
  ) {
    const { docId, userName, token } = data;

    let userId: string;
    let validatedName: string;

    if (token) {
      const userInfo = this.validateToken(token);
      if (!userInfo) {
        client.emit('error', { message: '无效的认证令牌' });
        return;
      }
      userId = userInfo.id;
      validatedName = userInfo.name;

      const hasAccess = await this.checkUserAccess(docId, userId);
      if (!hasAccess) {
        client.emit('error', { message: '您没有权限访问此文档' });
        return;
      }
    } else {
      userId = client.id;
      validatedName = this.sanitizeInput(userName || '', this.MAX_USERNAME_LENGTH);
    }

    client.join(docId);

    if (!this.roomUsers[docId]) {
      this.roomUsers[docId] = [];
    }

    this.updateRoomActivity(docId);

    const existingUserIndex = this.roomUsers[docId].findIndex((u) => u.id === userId);
    if (existingUserIndex >= 0) {
      this.roomUsers[docId][existingUserIndex].name = validatedName;
    } else {
      this.roomUsers[docId].push({ id: userId, name: validatedName });
    }

    const doc = await this.documentsService.findOne(docId, userId);
    if (doc) {
      client.emit('documentLoaded', doc);
    }

    this.server.to(docId).emit('onlineUsersUpdate', this.roomUsers[docId]);
    console.log(`用户 ${validatedName} (${userId}) 加入房间 ${docId}`);
  }

  // 处理块更新
  @SubscribeMessage('updateBlock')
  async handleUpdateBlock(
    @MessageBody()
    data: {
      docId: string;
      blockId: string;
      content: string;
      type?: string;
      properties?: any;
    },
    @ConnectedSocket() client: Socket,
  ) {
    try {
      this.updateRoomActivity(data.docId);

      // 立即广播给其他用户
      client.to(data.docId).emit('blockUpdated', data);

      // 异步持久化
      await this.documentsService.updateBlock(
        data.docId,
        data.blockId,
        data.content,
        data.type,
        data.properties,
      );
    } catch (error) {
      console.error('[DocumentsGateway] 更新块失败:', error);
      client.emit('error', { message: '保存块失败', blockId: data.blockId });
    }
  }

  // 批量更新块
  @SubscribeMessage('updateBlocksBatch')
  async handleUpdateBlocksBatch(
    @MessageBody()
    data: {
      docId: string;
      blocks: { id: string; content: string; type?: string; properties?: any }[];
    },
    @ConnectedSocket() client: Socket,
  ) {
    try {
      this.updateRoomActivity(data.docId);

      client.to(data.docId).emit('blocksUpdated', { docId: data.docId, blocks: data.blocks });

      await this.documentsService.updateBlocksBatch(data.docId, data.blocks);
    } catch (error) {
      console.error('[DocumentsGateway] 批量更新块失败:', error);
      client.emit('error', { message: '批量保存块失败' });
    }
  }

  // 分块加载
  @SubscribeMessage('loadBlocksChunked')
  async handleLoadBlocksChunked(
    @MessageBody() data: { docId: string; offset: number; limit: number },
    @ConnectedSocket() client: Socket,
  ) {
    const result = await this.documentsService.findBlocksChunked(data.docId, {
      offset: data.offset,
      limit: Math.min(data.limit, 200),
      includeTotal: true,
    });
    client.emit('blocksChunked', { docId: data.docId, ...result });
  }

  // 增量同步
  @SubscribeMessage('syncBlocks')
  async handleSyncBlocks(
    @MessageBody() data: { docId: string; since: string },
    @ConnectedSocket() client: Socket,
  ) {
    const lastSyncTime = new Date(data.since);
    const result = await this.documentsService.findBlocksSince(data.docId, lastSyncTime);
    client.emit('blocksSynced', { docId: data.docId, ...result });
  }

  // 正在输入状态
  @SubscribeMessage('typing')
  async handleTyping(
    @MessageBody() data: { docId: string; userId: string; userName: string },
    @ConnectedSocket() client: Socket,
  ) {
    const throttleKey = `typing:${data.docId}:${data.userId}`;
    const now = Date.now();
    const lastTime = this.typingThrottle[throttleKey] || 0;

    if (now - lastTime < this.THROTTLE_INTERVAL) {
      return;
    }
    this.typingThrottle[throttleKey] = now;

    this.updateRoomActivity(data.docId);
    client.to(data.docId).emit('userTyping', data);
  }

  // 处理聊天消息
  @SubscribeMessage('chatMessage')
  handleChatMessage(
    @MessageBody()
    data: {
      docId: string;
      userId: string;
      userName: string;
      content: string;
      timestamp: number;
      mentions?: string[];
    },
    @ConnectedSocket() client: Socket,
  ) {
    this.server.to(data.docId).emit('chatMessage', data);
  }

  // 处理消息反应
  @SubscribeMessage('messageReaction')
  handleMessageReaction(
    @MessageBody()
    data: {
      docId: string;
      messageIndex: number;
      emoji: string;
      userName: string;
      action: 'add' | 'remove';
    },
    @ConnectedSocket() client: Socket,
  ) {
    this.server.to(data.docId).emit('messageReaction', data);
  }

  // 光标移动
  @SubscribeMessage('cursorMove')
  handleCursorMove(
    @MessageBody()
    data: { docId: string; userId: string; position: { x: number; y: number } },
    @ConnectedSocket() client: Socket,
  ) {
    const throttleKey = `cursor:${data.docId}:${data.userId}`;
    const now = Date.now();
    const lastTime = this.cursorThrottle[throttleKey] || 0;

    if (now - lastTime < this.THROTTLE_INTERVAL) {
      return;
    }
    this.cursorThrottle[throttleKey] = now;

    this.updateRoomActivity(data.docId);
    client.to(data.docId).emit('cursorMoved', data);
  }
}
```

### 9.3 离线存储与消息同步

```typescript
// ============================================
// 离线存储模块 - offline-storage.ts
// ============================================

import * as Y from "yjs";
import { IndexeddbPersistence } from "y-indexeddb";
import { logError, logInfo, logWarn } from "./logger";

/** 同步状态 */
export type SyncStatus = "idle" | "syncing" | "synced" | "error" | "offline";

/** 网络状态 */
export type NetworkStatus = "online" | "offline" | "slow";

/**
 * 离线存储管理器
 * 提供完整的离线支持，包括数据持久化、同步状态管理、冲突解决
 */
class OfflineStorageManager {
  private persistences: Map<string, IndexeddbPersistence> = new Map();
  private documents: Map<string, Y.Doc> = new Map();
  private syncStatuses: Map<string, SyncStatus> = new Map();
  private listeners: Set<SyncEventListener> = new Set();
  private networkStatus: NetworkStatus = "online";
  private pendingSyncQueue: string[] = [];
  private isInitialized: boolean = false;

  constructor() {
    if (typeof window !== "undefined") {
      window.addEventListener("online", this.handleOnline.bind(this));
      window.addEventListener("offline", this.handleOffline.bind(this));
      this.networkStatus = navigator.onLine ? "online" : "offline";
    }
  }

  /**
   * 初始化离线存储
   */
  async initialize(): Promise<void> {
    if (this.isInitialized) return;

    try {
      if (!this.isIndexedDBSupported()) {
        throw new Error("IndexedDB 不支持");
      }

      await this.cleanupOldData();
      this.isInitialized = true;
      logInfo("离线存储初始化成功");
    } catch (error) {
      logError("离线存储初始化失败:", error);
      throw error;
    }
  }

  /**
   * 创建或获取离线文档
   */
  async getDocument(docId: string): Promise<Y.Doc> {
    const existing = this.documents.get(docId);
    if (existing) return existing;

    const doc = new Y.Doc();
    const persistence = new IndexeddbPersistence(docId, doc);

    this.documents.set(docId, doc);
    this.persistences.set(docId, persistence);

    await new Promise<void>((resolve, reject) => {
      persistence.once("synced", () => {
        this.updateSyncStatus(docId, "synced");
        resolve();
      });

      persistence.once("error", (error: Error) => {
        this.updateSyncStatus(docId, "error");
        reject(error);
      });

      setTimeout(() => {
        logWarn(`文档 ${docId} 加载超时，使用空文档`);
        resolve();
      }, 5000);
    });

    this.setupDocumentObservers(docId, doc);
    return doc;
  }

  /**
   * 设置文档观察者
   */
  private setupDocumentObservers(docId: string, doc: Y.Doc): void {
    doc.on("update", () => {
      this.markPendingChanges(docId, true);
      this.updateSyncStatus(docId, "syncing");

      if (this.networkStatus === "online") {
        this.queueSync(docId);
      }
    });
  }

  /**
   * 处理上线事件
   */
  private handleOnline(): void {
    this.networkStatus = "online";
    logInfo("网络已连接，开始同步待处理文档");
    this.processSyncQueue();
  }

  /**
   * 处理离线事件
   */
  private handleOffline(): void {
    this.networkStatus = "offline";
    logInfo("网络已断开，进入离线模式");

    this.syncStatuses.forEach((_, docId) => {
      this.updateSyncStatus(docId, "offline");
    });
  }

  /**
   * 检查是否在线
   */
  isOnline(): boolean {
    return this.networkStatus === "online";
  }

  /**
   * 获取同步状态
   */
  getSyncStatus(docId: string): SyncStatus {
    return this.syncStatuses.get(docId) || "idle";
  }

  /**
   * 加入同步队列
   */
  private queueSync(docId: string): void {
    if (!this.pendingSyncQueue.includes(docId)) {
      this.pendingSyncQueue.push(docId);
      this.processSyncQueue();
    }
  }

  /**
   * 处理同步队列
   */
  private processSyncQueue(): void {
    if (this.networkStatus !== "online") return;

    while (this.pendingSyncQueue.length > 0) {
      const docId = this.pendingSyncQueue.shift();
      if (docId) {
        this.syncDocument(docId);
      }
    }
  }

  /**
   * 同步单个文档
   */
  private async syncDocument(docId: string): Promise<void> {
    const persistence = this.persistences.get(docId);
    if (!persistence) return;

    this.updateSyncStatus(docId, "syncing");

    try {
      await new Promise<void>((resolve) => {
        persistence.once("synced", () => resolve());
      });

      this.updateSyncStatus(docId, "synced");
      this.markPendingChanges(docId, false);
    } catch (error) {
      this.updateSyncStatus(docId, "error");
      logError(`文档 ${docId} 同步失败:`, error);
    }
  }

  /**
   * 更新同步状态
   */
  private updateSyncStatus(docId: string, status: SyncStatus): void {
    this.syncStatuses.set(docId, status);
    this.emitEvent({
      type: status === "synced" ? "syncComplete" : status === "error" ? "syncError" : "syncStart",
      documentId: docId,
    });
  }

  /**
   * 标记待同步变更
   */
  private markPendingChanges(docId: string, pending: boolean): void {
    if (pending && !this.pendingSyncQueue.includes(docId)) {
      this.pendingSyncQueue.push(docId);
    } else if (!pending) {
      this.pendingSyncQueue = this.pendingSyncQueue.filter((id) => id !== docId);
    }
  }

  /**
   * 清理旧数据（超过30天未访问的文档）
   */
  private async cleanupOldData(): Promise<void> {
    const thirtyDaysAgo = Date.now() - 30 * 24 * 60 * 60 * 1000;

    const docIds: string[] = [];
    this.persistences.forEach((_, docId) => {
      docIds.push(docId);
    });

    for (const docId of docIds) {
      const meta = await this.getDocumentMeta(docId);
      if (meta && meta.lastModified < thirtyDaysAgo) {
        await this.deleteDocument(docId);
        logInfo(`清理旧文档: ${docId}`);
      }
    }
  }

  /**
   * 删除离线文档
   */
  async deleteDocument(docId: string): Promise<void> {
    const persistence = this.persistences.get(docId);
    if (persistence) {
      persistence.destroy();
      this.persistences.delete(docId);
    }

    const doc = this.documents.get(docId);
    if (doc) {
      doc.destroy();
      this.documents.delete(docId);
    }

    this.syncStatuses.delete(docId);
    await this.clearDocumentData(docId);
  }

  /**
   * 清空文档数据
   */
  private async clearDocumentData(docId: string): Promise<void> {
    if (typeof window === "undefined") return;

    return new Promise((resolve, reject) => {
      const request = indexedDB.deleteDatabase(docId);
      request.onsuccess = () => resolve();
      request.onerror = () => reject(request.error);
    });
  }

  /**
   * 获取存储统计
   */
  async getStorageStats(): Promise<StorageStats> {
    const stats: StorageStats = {
      totalDocuments: this.documents.size,
      totalSize: 0,
      pendingSync: this.pendingSyncQueue.length,
      lastSyncTime: null,
      oldestPendingChange: null,
    };

    this.documents.forEach((doc) => {
      const size = this.estimateDocumentSize(doc);
      stats.totalSize += size;
    });

    return stats;
  }

  /**
   * 估算文档大小
   */
  private estimateDocumentSize(doc: Y.Doc): number {
    try {
      const encoded = Y.encodeStateAsUpdate(doc);
      return encoded.byteLength;
    } catch {
      return 0;
    }
  }

  /**
   * 添加事件监听器
   */
  addListener(listener: SyncEventListener): () => void {
    this.listeners.add(listener);
    return () => this.listeners.delete(listener);
  }

  /**
   * 发送事件
   */
  private emitEvent(event: SyncEvent): void {
    this.listeners.forEach((listener) => {
      try {
        listener(event);
      } catch (error) {
        logError("事件监听器错误:", error);
      }
    });
  }

  /**
   * 销毁
   */
  destroy(): void {
    if (typeof window !== "undefined") {
      window.removeEventListener("online", this.handleOnline.bind(this));
      window.removeEventListener("offline", this.handleOffline.bind(this));
    }

    this.documents.forEach((doc) => doc.destroy());
    this.persistences.forEach((persistence) => persistence.destroy());

    this.documents.clear();
    this.persistences.clear();
    this.syncStatuses.clear();
    this.listeners.clear();
    this.pendingSyncQueue = [];
  }
}

// 导出单例
export const offlineStorage = new OfflineStorageManager();

/**
 * 使用离线存储状态的 Hook
 */
export function useOfflineStatus(): {
  isOnline: boolean;
  networkStatus: NetworkStatus;
  pendingSync: number;
} {
  const [state, setState] = useState({
    isOnline: offlineStorage.isOnline(),
    networkStatus: offlineStorage.getNetworkStatus(),
    pendingSync: 0,
  });

  useEffect(() => {
    const updateState = () => {
      setState((prev) => ({
        ...prev,
        isOnline: offlineStorage.isOnline(),
        networkStatus: offlineStorage.getNetworkStatus(),
      }));
    };

    window.addEventListener("online", updateState);
    window.addEventListener("offline", updateState);

    const interval = setInterval(async () => {
      const stats = await offlineStorage.getStorageStats();
      setState((prev) => ({ ...prev, pendingSync: stats.pendingSync }));
    }, 5000);

    return () => {
      window.removeEventListener("online", updateState);
      window.removeEventListener("offline", updateState);
      clearInterval(interval);
    };
  }, []);

  return state;
}
```

## 十、总结

即时通讯系统是Web应用中最复杂的场景之一，它涉及到实时通信、数据一致性、高性能渲染、离线支持等多个技术领域。通过FastDocument项目的实际代码，我们可以看到一个完整的IM系统是如何实现的。

**核心要点回顾**：

一、技术选型方面：WebSocket是实时通信的首选协议，Socket.io提供了更完善的功能封装。在大多数场景下，自建IM系统的成本远高于使用第三方服务，除非有特殊需求。

二、消息系统设计：消息需要持久化存储以支持离线推送，消息状态（发送中、已发送、已读）需要精心设计，多设备同步是IM系统最复杂的挑战之一。

三、前端实现：虚拟列表是处理大量消息的必备技术，消息状态需要在UI上清晰展示，草稿功能可以极大提升用户体验。

四、性能优化：避免过早优化，用数据驱动优化决策，注意移动端兼容性和内存泄漏问题。

五、运维监控：IM服务建议独立部署，建立完善的监控告警体系，提前做好容量规划。

**进一步学习建议**：

1. 研究专业的IM系统架构，如微信弱网优化策略、钉钉的分布式消息系统设计。

2. 探索WebRTC在IM中的应用，特别是音视频通话功能。

3. 学习CRDT（无冲突复制数据类型）在协作编辑中的应用，这是实现无冲突协作编辑的基础。

4. 深入了解消息队列（如Kafka、RocketMQ）在IM系统中的使用场景。
