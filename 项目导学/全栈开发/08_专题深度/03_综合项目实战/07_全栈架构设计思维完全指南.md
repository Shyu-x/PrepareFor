# 全栈架构设计思维完全指南

## 概述

架构设计是软件工程中最具挑战性的工作之一。它要求工程师不仅理解技术，还要理解业务、团队、成本和时间的约束。作为全栈工程师，我们既要掌握前端架构设计，也要理解后端架构设计，更要在系统层面进行整体思考。

本文档将从实战角度出发，系统讲解全栈架构设计的各个方面，帮助读者建立完整的架构思维体系。

---

## 一、架构设计的本质

### 1.1 什么是架构：顶层设计

架构（Architecture）这个词最初来自建筑领域，指的是建筑物的结构设计和整体规划。引入软件工程后，架构指的是软件系统的**顶层设计**，它定义了系统的组成部分、各部分之间的关系以及指导其设计与演化的原则。

架构的本质是**抽象**。通过对系统进行高层抽象，我们能够在纷繁复杂的实现细节中抓住核心问题。正如建筑师不需要关心每块砖的精确位置，软件架构师也不需要关心每行代码的具体实现。

架构设计需要回答的核心问题包括：

```
┌─────────────────────────────────────────────────────────────────┐
│                        架构设计核心问题                           │
├─────────────────────────────────────────────────────────────────┤
│  1. 系统由哪些部分组成？（组成部分）                              │
│  2. 各部分之间如何协作？（关系与交互）                             │
│  3. 什么原则指导设计？（设计原则）                                │
│  4. 如何应对变化和增长？（演进策略）                              │
└─────────────────────────────────────────────────────────────────┘
```

一个好的架构应该具备以下特征：

| 特征 | 说明 | 重要性 |
|------|------|--------|
| **正确性** | 能够正确实现业务需求 | 必须 |
| **性能** | 满足系统的性能指标要求 | 关键 |
| **可扩展性** | 能够应对业务增长和变化 | 重要 |
| **可维护性** | 便于团队理解和维护 | 重要 |
| **可靠性** | 系统能够稳定运行 | 必须 |
| **安全性** | 保护数据安全和业务安全 | 必须 |

### 1.2 架构 vs 设计：分不清就别做

很多人容易混淆架构（Architecture）和设计（Design）这两个概念。虽然它们都是软件工程的重要活动，但层次和关注点不同。

```
┌─────────────────────────────────────────────────────────────────┐
│                        架构 vs 设计                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   架构（Architecture）                                           │
│   ├── 层次：高层次、全局性                                       │
│   ├── 关注点：系统结构、组件划分、关键技术决策                     │
│   ├── 产出物：架构文档、技术选型报告、系统拓扑图                   │
│   ├── 决策者：架构师、技术负责人                                  │
│   └── 变更成本：高（影响全局）                                    │
│                                                                 │
│   设计（Design）                                                 │
│   ├── 层次：低层次、局部性                                       │
│   ├── 关注点：具体实现、类/函数设计、接口定义                     │
│   ├── 产出物：详细设计文档、类图、时序图                          │
│   ├── 决策者：开发工程师                                         │
│   └── 变更成本：低（影响局部）                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

举一个具体的例子来说明：

**架构层面的决策：**
- 选择微服务架构还是单体架构
- 采用什么数据库（MySQL、PostgreSQL、MongoDB）
- 如何拆分系统服务
- 选择什么缓存方案

**设计层面的决策：**
- 一个类应该有哪些方法
- 函数如何命名和封装
- 接口返回什么字段
- 异常如何处理和传播

架构是**做正确的事**，设计是**正确地做事**。如果架构错了，设计再好也无法弥补；反之，如果设计很差，即使架构正确，系统也难以实现。

### 1.3 我的思考：架构是取舍的艺术

架构设计从来不是追求最优解，而是**在各种约束条件下寻找平衡点**。这是一个取舍（Trade-off）的过程。

**常见的架构取舍维度：**

```
性能 vs 可维护性
├── 高性能往往意味着高度定制化和复杂性
├── 高可维护性意味着标准化和抽象
└── 例子：手写SQL vs ORM框架

扩展性 vs 成本
├── 水平扩展需要无状态设计，增加了复杂度
├── 垂直扩展成本高但实现简单
└── 例子：分布式缓存 vs 单机缓存

一致性 vs 可用性
├── 强一致性会牺牲可用性（如分布式锁）
├── 最终一致性提高可用性但可能返回旧数据
└── 例子：ZooKeeper vs Cassandra

开发效率 vs 运行效率
├── 脚本语言开发快但运行效率低
├── 编译型语言开发慢但运行效率高
└── 例子：Node.js vs Go

耦合 vs 独立
├── 高耦合开发简单但难以维护和扩展
├── 低耦合需要更多抽象和接口设计
└── 例子：单体架构 vs 微服务架构
```

**架构没有银弹**。任何架构方案都有其优缺点，关键是找到最适合当前场景的方案。评价一个架构好不好，不在于它使用了多么先进的技术，而在于它是否：

1. **满足业务需求**：这是最基本的要求
2. **适应团队能力**：再好的架构如果团队无法驾驭也是白搭
3. **符合成本约束**：包括开发成本、运维成本、基础设施成本
4. **能够演进**：架构不是一成不变的，要能够适应业务变化

作为全栈工程师，我们要认识到：架构是**动态演进**的。Google的《SRE》中有句话说得很好："软件设计是一个持续的过程，而不是一个最终产物。"好的架构应该能够随着业务和技术的发展而演进，而不是一开始就追求完美。

## 二、从需求到架构

### 2.1 需求分析：抓住核心诉求

需求分析是架构设计的起点，也是最关键的环节之一。很多架构问题本质上是对需求理解不深导致的。

**需求分析的层次：**

```
┌─────────────────────────────────────────────────────────────────┐
│                        需求分析层次                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  业务需求层（Why）                                               │
│  ├── 解决什么业务问题？                                          │
│  ├── 目标用户是谁？                                              │
│  └── 业务价值是什么？                                            │
│                                                                 │
│  功能需求层（What）                                              │
│  ├── 系统需要实现哪些功能？                                      │
│  ├── 功能的优先级是什么？                                        │
│  └── 功能的边界在哪里？                                          │
│                                                                 │
│  非功能需求层（How Well）                                        │
│  ├── 性能要求：响应时间、吞吐量、TPS                              │
│  ├── 可用性要求： uptime、降级策略                               │
│  ├── 扩展性要求：用户增长、数据增长                                │
│  └── 安全要求：认证授权、数据保护                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**需求分析的常用技巧：**

**1. 追问"为什么"**

用户说"我要一个能承载100万人同时在线的系统"，这不是需求，是解决方案。真正的需求可能是"我们预计半年内用户会增长到100万，届时系统不能崩溃"。追问为什么能帮助我们理解真正的目标。

**2. 区分需求和方案**

用户说"我要用微服务架构"，这可能是方案，不是需求。真正的需求可能是"我想快速迭代、独立部署"。区分需求和方案可以避免过早陷入技术细节。

**3. 量化非功能需求**

模糊的需求如"系统要快"没有意义。需要量化：
- 登录接口响应时间 < 200ms
- 首页加载时间 < 3秒（3G网络）
- 支持 10000 TPS

**4. 识别约束条件**

- 时间约束："3个月内必须上线"
- 技术约束："必须使用Java技术栈"
- 资源约束："只有2个开发人员"
- 合规约束："必须符合GDPR"

**需求分析的输出文档：**

| 文档 | 内容 | 用途 |
|------|------|------|
| 业务需求文档 | 业务背景、目标、价值 | 沟通确认 |
| 功能需求规格 | 功能清单、优先级、边界 | 开发依据 |
| 非功能需求文档 | 性能、可用性、安全等指标 | 架构设计依据 |
| 用户故事地图 | 用户角色、功能流程 | 理解用户体验 |

### 2.2 容量预估：TPS、并发、存储

容量预估是连接业务需求和技术架构的桥梁。没有容量预估的架构设计就像盲人摸象。

**核心概念解析：**

| 概念 | 定义 | 示例 |
|------|------|------|
| **TPS** | Transactions Per Second，每秒事务数 | 系统每秒处理1000个订单 |
| **QPS** | Queries Per Second，每秒查询数 | 查询接口每秒处理5000次 |
| **并发用户数** | 同时发起请求的用户数量 | 1万个用户同时在线 |
| **响应时间** | 请求到响应的时间 | API响应时间 < 100ms |
| **吞吐量** | 单位时间处理的数据量 | 每秒处理100MB数据 |

**容量预估的计算方法：**

```
┌─────────────────────────────────────────────────────────────────┐
│                      容量预估公式                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 基于日活用户预估                                            │
│     日活用户 = 总用户数 × 日活率                                  │
│     假设：100万用户，10%日活率 → 日活 = 10万用户                   │
│                                                                 │
│  2. 基于用户行为预估TPS                                         │
│     TPS = 日活用户 × 每日操作次数 / 每日秒数                      │
│     假设：10万用户，每人每天100次操作                             │
│     TPS = 100000 × 100 / 86400 ≈ 115 TPS                        │
│                                                                 │
│  3. 峰值TPS预估                                                  │
│     峰值系数 = 最大QPS / 平均QPS                                 │
│     通常取 3-10 倍                                               │
│     峰值TPS = 115 × 5 = 575 TPS                                  │
│                                                                 │
│  4. 存储容量预估                                                │
│     日增量 = 用户数 × 操作次数 × 数据大小                        │
│     总容量 = 日增量 × 保留天数 × 冗余系数                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**实际预估示例：电商秒杀系统**

假设：
- 总用户：500万
- 日活率：20%，即100万日活
- 秒杀活动参与率：10%，即10万人参与
- 活动持续时间：1小时
- 每人下单次数：平均2次
- 每人订单数据：约2KB

计算：
- 峰值QPS = 100000 × 2 / 3600 ≈ 56 TPS（纯下单）
- 加上详情页浏览、库存查询等 → 总QPS约 500-1000
- 订单总量 = 10万 × 2 = 20万单
- 日存储增量 = 20万 × 2KB = 400MB
- 考虑3年存储 + 3倍冗余 = 400MB × 365 × 3 × 3 ≈ 1.3TB

**容量预估的注意事项：**

1. **留足余量**：实际流量往往超出预估，至少预留 2-3 倍余量
2. **考虑峰值**：大促、节假日、热点事件会导致流量激增
3. **分阶段预估**：不要试图一次预估准确，分阶段验证
4. **建立监控**：上线后根据实际数据调整预估模型

### 2.3 技术选型：没有标准答案

技术选型是架构设计中最容易引发争论的环节。"PHP是世界上最好的语言"这种争论毫无意义，因为**技术选型必须基于具体场景**。

**技术选型的决策框架：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     技术选型决策框架                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  第一层：技术是否成熟                                            │
│  ├── 是否已经大规模生产验证？                                    │
│  ├── 社区是否活跃？                                              │
│  ├── 版本是否稳定？版本生命周期？                                 │
│  └── 遇到问题能否找到解决方案？                                   │
│                                                                 │
│  第二层：技术是否适合业务                                        │
│  ├── 性能是否满足需求？                                          │
│  ├── 是否易于开发和维护？                                        │
│  ├── 团队是否掌握或能快速学习？                                   │
│  └── 生态是否完善？                                              │
│                                                                 │
│  第三层：成本与收益                                              │
│  ├── 引入技术的研发成本                                          │
│  ├── 基础设施和运维成本                                          │
│  ├── 人员培训成本                                                │
│  └── 未来迁移和替换成本                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**常见技术选型对比：**

**1. 后端语言选型**

| 语言 | 优势 | 劣势 | 适用场景 |
|------|------|------|----------|
| Java | 生态完善、性能优秀、框架成熟 | 语法啰嗦、启动慢 | 企业级应用、大规模系统 |
| Go | 性能高、并发简单、部署方便 | 泛型较弱、生态较新 | 云计算、微服务、高并发 |
| Python | 开发效率高、数据处理强 | 性能较差、GIL限制 | 数据分析、AI、爬虫、Web |
| Node.js | 前后端统一、异步IO | 单线程、CPU密集弱 | 实时应用、轻量API |
| Rust | 性能极强、安全性高 | 学习曲线陡、生态较新 | 系统编程、嵌入式 |

**2. 数据库选型**

| 数据库 | 类型 | 优势 | 劣势 | 适用场景 |
|--------|------|------|------|----------|
| PostgreSQL | 关系型 | 功能强大、扩展性好 | 复杂场景调优困难 | 复杂查询、事务型 |
| MySQL | 关系型 | 生态成熟、运维简单 | 复杂查询较弱 | Web应用、OLTP |
| MongoDB | 文档型 | 灵活 Schema、高扩展 | 事务能力弱 | 内容管理、日志 |
| Redis | KV型 | 性能极高、丰富数据结构 | 内存成本高 | 缓存、Session、队列 |
| Elasticsearch | 搜索型 | 全文搜索强、聚合分析 | 写入性能一般 | 搜索、日志分析 |

**3. 缓存选型**

| 缓存 | 架构 | 数据存储 | 适用场景 |
|------|------|----------|----------|
| Redis | 单机/集群 | 内存 + 磁盘持久化 | 通用缓存、分布式锁 |
| Memcached | 分布式 | 纯内存 | 简单 KV 缓存 |
| CDN | 分布式边缘 | 静态资源 | 静态文件、加速分发 |

**技术选型的常见误区：**

| 误区 | 说明 | 正确做法 |
|------|------|----------|
| 追新弃旧 | 盲目使用最新技术 | 技术选型以稳定成熟为主 |
| 性能至上 | 只考虑性能指标 | 综合考虑团队、成本、维护 |
| 一刀切 | 所有场景用同一技术 | 根据场景选择合适技术 |
| 忽视团队 | 不考虑团队技术栈 | 优先选择团队熟悉的技术 |
| 完美主义 | 追求完美方案 | 选择够用方案，快速验证 |

### 2.4 我的思考：架构设计的第一步是什么

很多人拿到需求就开始画架构图、选技术栈，这是非常错误的做法。**架构设计的第一步是理解业务本质**。

**理解业务本质的三个问题：**

1. **业务的核心价值是什么？**
   - 这个系统为什么存在？
   - 用户为什么需要它？
   - 它解决了什么痛点？

2. **业务的增长模式是什么？**
   - 用户增长预期是多少？
   - 数据增长预期是多少？
   - 业务模式会不会变化？

3. **业务的技术敏感点在哪里？**
   - 什么操作最频繁？
   - 什么操作最关键？
   - 什么变化最可能发生？

举一个例子：

同样是"用户系统"，社交App的账号系统和银行系统对架构的要求完全不同：

| 维度 | 社交App账号系统 | 银行账号系统 |
|------|----------------|--------------|
| 核心诉求 | 登录便捷、社交关系 | 安全、合规、资金安全 |
| 性能要求 | 高并发、快速响应 | 事务一致性、审计追溯 |
| 技术重点 | SSO、单点登录 | 加密、风控、合规 |
| 架构侧重 | 水平扩展、高可用 | 数据安全、事务强一致 |

不理解业务本质，做出的架构可能会"用力过猛"或"南辕北辙"。

**架构设计的正确姿势：**

```
1. 深入理解业务需求（30%时间）
   └── 与业务方充分沟通，理解背景、目标、约束

2. 进行容量预估（20%时间）
   └── 量化需求，建立基线

3. 设计高层架构（20%时间）
   └── 确定系统划分、技术选型

4. 评审和迭代（20%时间）
   └── 团队评审，根据反馈调整

5. 输出架构文档（10%时间）
   └── 记录架构决策和设计理由
```

记住：**好的架构源于对业务的深刻理解**。花时间理解业务，永远是值得的。

## 三、前端架构设计

### 3.1 应用架构：单体还是微前端

前端架构设计首先要解决的是应用架构问题：整个前端应该是一个单体，还是拆分成多个独立的部分（微前端）？

**单体前端架构：**

```
┌─────────────────────────────────────────────────────────────────┐
│                       单体前端架构                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                    单一部署单元                           │   │
│   │  ┌───────────┬───────────┬───────────┬───────────┐      │   │
│   │  │  登录模块  │  主页模块  │  订单模块  │  用户模块  │      │   │
│   │  └───────────┴───────────┴───────────┴───────────┘      │   │
│   │  ┌─────────────────────────────────────────────────┐    │   │
│   │  │              共享组件 / 路由 / 状态               │    │   │
│   │  └─────────────────────────────────────────────────┘    │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│   优势：简单、一致、易于调试                                     │
│   劣势：团队耦合、部署不灵活、技术栈限制                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**微前端架构：**

```
┌─────────────────────────────────────────────────────────────────┐
│                       微前端架构                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────┐  ┌─────────────┐  ┌─────────────┐             │
│   │  登录子应用  │  │  主页子应用  │  │  订单子应用  │             │
│   └──────┬──────┘  └──────┬──────┘  └──────┬──────┘             │
│          │               │               │                     │
│   ┌──────┴───────────────┴───────────────┴──────┐              │
│   │              应用容器 / 路由编排              │              │
│   └─────────────────────┬───────────────────────┘              │
│                         │                                        │
│   ┌─────────────────────┴───────────────────────┐              │
│   │              共享依赖（React/Vue等）           │              │
│   └───────────────────────────────────────────────┘              │
│                                                                 │
│   优势：独立部署、技术多样、团队自治                              │
│  劣势：复杂度高、共享代码冲突、一致性挑战                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**如何选择：**

| 因素 | 倾向于单体 | 倾向于微前端 |
|------|------------|--------------|
| 团队规模 | < 10人 | > 20人 |
| 应用复杂度 | 功能简单 | 功能复杂，多团队并行 |
| 迭代速度 | 快速迭代 | 需要独立发布不同模块 |
| 技术需求 | 统一技术栈 | 不同模块需要不同技术 |
| 变更频率 | 整体频繁变更 | 模块相对稳定 |

**我的经验**：大多数项目一开始选择单体架构是正确的。微前端的复杂度很高，只有在真正需要的时候才考虑。大多数团队遇到的"单体问题"，其实不是架构问题，而是代码组织和团队协作问题。

### 3.2 状态管理：集中还是分散

状态管理是前端架构中最核心的问题之一。状态应该集中管理还是分散到各个组件？

**状态管理的演进：**

```
┌─────────────────────────────────────────────────────────────────┐
│                    状态管理演进历程                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 本地状态（Local State）                                      │
│     └── useState, useRef                                       │
│     适用：组件私有状态                                           │
│                                                                 │
│  2. 提升状态（Lifted State）                                      │
│     └── 状态提升到父组件，通过 props 传递                        │
│     适用：相关组件共享状态                                       │
│                                                                 │
│  3. Context 状态（Context State）                                │
│     └── React Context，跨组件共享                                │
│     适用：主题、语言、认证状态                                   │
│                                                                 │
│  4. 外部状态管理（External State）                                │
│     └── Redux, Zustand, Jotai                                   │
│     适用：复杂应用状态                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**状态分类与存储策略：**

| 状态类型 | 存储位置 | 示例 | 说明 |
|----------|----------|------|------|
| **UI状态** | 组件内部 | 弹窗开关、折叠展开 | 尽量本地管理 |
| **共享UI状态** | Context | 主题、语言设置 | 使用 Context |
| **服务端状态** | 缓存库 | 用户信息、列表数据 | 使用 SWR/React Query |
| **业务状态** | 状态管理库 | 购物车、订单 | 使用 Zustand/Redux |
| **持久化状态** | LocalStorage | 表单草稿、用户偏好 | 需要序列化 |

### 3.3 路由设计：路由要不要分权限

路由设计不仅要考虑页面导航，还要考虑权限控制、懒加载、SEO等问题。

**路由设计的核心问题：**

```
┌─────────────────────────────────────────────────────────────────┐
│                       路由设计核心问题                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 路由结构                                                     │
│     ├── 静态路由 vs 动态路由                                     │
│     ├── 嵌套路由 vs 平铺路由                                     │
│     └── 路由分组                                                 │
│                                                                 │
│  2. 路由权限                                                     │
│     ├── 游客访问                                                 │
│     ├── 登录访问                                                 │
│     └── 角色权限                                                 │
│                                                                 │
│  3. 路由加载                                                     │
│     ├── 预加载 vs 懒加载                                         │
│     └── 加载状态                                                 │
│                                                                 │
│  4. 路由与SEO                                                    │
│     ├── 静态生成（SSG）                                          │
│     ├── 服务端渲染（SSR）                                        │
│     └── 客户端渲染（CSR）                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4 我的思考：前端架构容易被忽视

前端架构在很多团队中不受重视，很多人认为"前端就是切图写样式"。实际上，前端架构同样重要，而且前端架构的复杂度在不断上升。

**前端架构面临的挑战：**

1. **复杂度增加**：现代前端应用已经不再是简单的页面，而是复杂的单页应用（SPA）
2. **状态爆炸**：前端状态越来越多，状态管理成为难题
3. **性能要求**：用户对性能的要求越来越高，首屏加载、交互响应都有严格要求
4. **团队协作**：大团队需要良好的架构来支撑并行开发

**前端架构需要关注的核心指标：**

| 指标 | 说明 | 优化方向 |
|------|------|----------|
| **首屏加载时间** | 用户看到内容的首次时间 | SSR/SSG、代码分割、CDN |
| **交互响应时间** | 用户操作到反馈的时间 | 骨架屏、预加载、缓存 |
| **JS Bundle 大小** | 客户端下载的 JS 体积 | Tree Shaking、懒加载 |
| **重渲染次数** | 组件不必要的重渲染 | React.memo、useMemo |
| **可访问性** | 对特殊用户的支持 | ARIA、语义化标签 |

## 四、后端架构设计

### 4.1 服务分层：Controller-Service-DAO

后端架构最经典的分层模式是 Controller-Service-DAO 三层架构。这种分层清晰明了，适合大多数业务系统。

**三层架构详解：**

```
┌─────────────────────────────────────────────────────────────────┐
│                    Controller-Service-DAO 三层架构               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                    Controller 层                        │   │
│   │                                                          │   │
│   │   职责：处理请求、参数校验、调用 Service                    │   │
│   │   输入：HTTP 请求                                         │   │
│   │   输出：HTTP 响应                                         │   │
│   │                                                          │   │
│   │   示例：                                                   │   │
│   │   @Post('/documents')                                    │   │
│   │   async createDocument(@Body() body: CreateDto) {       │   │
│   │     return this.documentService.create(body);           │   │
│   │   }                                                      │   │
│   └─────────────────────────────────────────────────────────┘   │
│                              ↓                                   │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                    Service 层                           │   │
│   │                                                          │   │
│   │   职责：业务逻辑处理、事务管理、调用多个 DAO                │   │
│   │   输入：DTO / Domain 对象                                 │   │
│   │   输出：Domain 对象 / 聚合根                              │   │
│   │                                                          │   │
│   │   示例：                                                   │   │
│   │   async createDocument(dto: CreateDto): Promise<Doc> {  │   │
│   │     // 业务校验                                            │   │
│   │     if (await this.hasDuplicateTitle(dto.title)) {      │   │
│   │       throw new BadRequestException('标题已存在');       │   │
│   │     }                                                    │   │
│   │     // 创建文档                                            │   │
│   │     const doc = await this.documentDao.create(dto);     │   │
│   │     // 发送通知                                            │   │
│   │     await this.notificationService.notify(doc);         │   │
│   │     return doc;                                          │   │
│   │   }                                                      │   │
│   └─────────────────────────────────────────────────────────┘   │
│                              ↓                                   │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                    DAO / Repository 层                   │   │
│   │                                                          │   │
│   │   职责：数据访问、数据库操作                               │   │
│   │   输入：Entity / 查询条件                                 │   │
│   │   输出：Entity / Entity 列表                             │   │
│   │                                                          │   │
│   │   示例：                                                   │   │
│   │   async create(data: CreateDto): Promise<Document> {    │   │
│   │     return this.repository.save(data);                  │   │
│   │   }                                                      │   │
│   │   async findById(id: string): Promise<Document> {       │   │
│   │     return this.repository.findOne({ where: { id } });  │   │
│   │   }                                                      │   │
│   └─────────────────────────────────────────────────────────┘   │
│                              ↓                                   │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                      数据库层                              │   │
│   │                                                          │   │
│   │   MySQL / PostgreSQL / MongoDB / Redis                   │   │
│   │                                                          │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**三层架构的变体：**

| 变体 | 说明 | 适用场景 |
|------|------|----------|
| **Controller-Service-DAO** | 标准三层 | 大多数业务系统 |
| **Controller-Service-Repository** | 经典Java风格 | 企业应用 |
| **Controller-Command-Query** (CQRS) | 命令查询分离 | 复杂业务场景 |
| **Controller-Service-DAO-Converter** | 增加转换层 | 多端API场景 |

### 4.2 模块划分：按功能还是按领域

模块划分是架构设计中的重要决策。好的模块划分应该高内聚低耦合。

**两种主要的模块划分方式：**

```
┌─────────────────────────────────────────────────────────────────┐
│                    模块划分方式对比                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  按功能划分（水平分层）                                          │
│  ┌─────────────┬─────────────┬─────────────┬─────────────┐     │
│  │   模块A     │   模块B     │   模块C     │   模块D     │     │
│  │  用户管理   │   订单管理   │   支付管理   │   统计模块   │     │
│  ├─────────────┼─────────────┼─────────────┼─────────────┤     │
│  │            所有层         │            所有层         │     │
│  └─────────────┴─────────────┴─────────────┴─────────────┘     │
│                                                                 │
│  按领域划分（垂直分割）                                          │
│  ┌─────────────┬─────────────┬─────────────┬─────────────┐     │
│  │   用户域     │   订单域     │   支付域     │   内容域     │     │
│  │  ┌───────┐  │  ┌───────┐  │  ┌───────┐  │  ┌───────┐  │     │
│  │  │Controller│ │Controller│ │Controller│ │Controller│  │     │
│  │  │Service │  │  │Service │  │  │Service │  │  │Service │  │     │
│  │  │Repository│ │Repository│ │Repository│ │Repository│  │     │
│  │  │  Entity │  │  │  Entity │  │  │  Entity │  │  │  Entity │  │     │
│  │  └───────┘  │  └───────┘  │  └───────┘  │  └───────┘  │     │
│  └─────────────┴─────────────┴─────────────┴─────────────┘     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**按功能划分的特点：**
- 优点：职责清晰，容易理解
- 缺点：跨功能调用复杂，领域逻辑分散

**按领域划分的特点：**
- 优点：领域内聚，边界清晰，符合 DDD 思想
- 缺点：需要有明确的领域边界划分能力

### 4.3 数据库设计：范式还是反范式

数据库设计是后端架构的核心环节。关系型数据库设计有严格的范式要求，但在实际项目中需要根据场景选择。

**数据库范式：**

| 范式 | 要求 | 示例 |
|------|------|------|
| **1NF** | 原子性，每列不可再分 | 地址字段拆分为省/市/区 |
| **2NF** | 满足1NF + 非主键列完全依赖主键 | 订单明细表避免部分依赖 |
| **3NF** | 满足2NF + 非主键列不传递依赖主键 | 用户表不存储部门名称 |
| **BCNF** | 满足3NF + 主键内部字段无依赖 | 多对多关系优化 |

**范式 vs 反范式：**

```
┌─────────────────────────────────────────────────────────────────┐
│                      范式设计权衡                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  范式化设计（Normalized）                                       │
│  ├── 优点：数据冗余小、更新快、存储省                            │
│  ├── 缺点：查询需要多表 JOIN、性能差                            │
│  └── 适用：写多读少、事务要求高                                  │
│                                                                 │
│  反范式化设计（Denormalized）                                   │
│  ├── 优点：查询快、无需 JOIN                                    │
│  ├── 缺点：数据冗余、更新复杂、存储大                            │
│  └── 适用：读多写少、需要高性能查询                              │
│                                                                 │
│  实践中的选择：                                                  │
│  ├── 通常先按范式设计                                           │
│  ├── 对性能敏感的关键查询做反范式优化                           │
│  └── 注意保持数据一致性                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.4 我的思考：后端架构要稳

后端架构和前端架构的最大区别是：**后端架构更注重稳定性**，因为后端是系统的基础，一旦出问题影响面巨大。

**后端架构的核心原则：**

1. **稳定性优先**
   - 性能可以慢慢优化，但系统崩溃直接影响业务
   - 优先考虑限流、熔断、降级等保护措施

2. **事务一致性**
   - 金融、订单等场景必须保证事务一致性
   - 不要为了性能轻易放弃强一致性

3. **可回滚**
   - 所有变更必须可回滚
   - 数据库变更要支持回滚脚本

4. **渐进式演进**
   - 大版本架构调整要分阶段
   - 保持向后兼容，支持灰度发布

## 五、系统交互设计

### 5.1 同步 vs 异步：什么时候选哪个

系统间的交互分为同步和异步两种方式。不同的场景适合不同的方式。

**同步交互特点：**
- 请求-等待-响应模式
- 实时性强，调用方等待结果
- 简单直接，易于理解和调试
- 适合：核心流程、状态敏感操作

**异步交互特点：**
- 发送-确认模式，不等待处理完成
- 解耦发送方和处理方
- 适合：耗时操作、批量处理、事件通知

```
┌─────────────────────────────────────────────────────────────────┐
│                      同步 vs 异步交互                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  同步调用：                                                      │
│                                                                 │
│  ┌─────┐    调用     ┌─────┐    执行     ┌─────┐                 │
│  │调用方│ ──────────→│服务A│ ──────────→│数据库│                 │
│  └─────┘            └─────┘            └─────┘                 │
│      ↑                  │                                        │
│      │                  ↓                                        │
│      │               返回结果                                     │
│      │                  │                                        │
│      └──────────────────┘                                        │
│                                                                 │
│  异步调用（消息队列）：                                           │
│                                                                 │
│  ┌─────┐    发送     ┌─────┐    消费     ┌─────┐                 │
│  │调用方│ ──────────→│  MQ │ ──────────→│服务B │                 │
│  └─────┘            └─────┘            └─────┘                 │
│      │                  │                                        │
│      │                  ↓                                        │
│      │               存储消息                                     │
│      │                  │                                        │
│      │               持久化                                       │
│      │                  │                                        │
│      └────────────────────────────────────── (不等待结果)        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**选择同步还是异步的决策树：**

```
需要同步还是异步？
│
├─ 是否需要实时返回结果？
│   │
│   ├─ 是 → 同步调用
│   │   示例：查询余额、获取用户信息、验证token
│   │
│   └─ 否 → 继续判断
│
├─ 操作是否耗时（> 1秒）？
│   │
│   ├─ 是 → 异步处理
│   │   示例：文件处理、批量导出、复杂计算
│   │
│   └─ 否 → 继续判断
│
├─ 是否允许失败重试？
│   │
│   ├─ 是 → 考虑异步（消息队列）
│   │
│   └─ 否 → 同步调用，但做好异常处理
│
└─ 是否需要多消费者？
    │
    ├─ 是 → 异步（消息队列）
    │   示例：事件通知、日志收集
    │
    └─ 否 → 继续判断
```

**实际项目中的使用场景：**

| 场景 | 方式 | 原因 |
|------|------|------|
| 用户登录验证 | 同步 | 需要立即返回结果 |
| 发送短信/邮件 | 异步 | 不影响主流程、可重试 |
| 订单创建 | 同步 | 需要确认结果 |
| 日志收集 | 异步 | 高频、允许延迟 |
| 支付回调 | 异步 | 外部系统触发、需要处理 |
| 数据同步 | 异步 | 允许延迟、批量处理 |

### 5.2 REST vs RPC：风格选择

系统间的远程调用主要有 REST 和 RPC 两种风格。

**REST（Representational State Transfer）：**

| 特点 | 说明 |
|------|------|
| 资源导向 | 以资源为中心，用名词表示 |
| HTTP 方法语义 | GET查、POST增、PUT/PATCH改、DELETE删 |
| 无状态 | 每个请求包含所有必要信息 |
| 标准媒体类型 | JSON、XML 等 |

**RPC（Remote Procedure Call）：**

| 特点 | 说明 |
|------|------|
| 动作导向 | 以动作为中心，用动词表示 |
| 简单直接 | 像调用本地函数一样调用远程服务 |
| 高效 | 通常使用二进制协议，性能更好 |
| 紧耦合 | 客户端需要知道服务端接口 |

**REST vs RPC 对比：**

| 维度 | REST | RPC |
|------|------|-----|
| 耦合度 | 低（标准HTTP） | 高（接口定义） |
| 性能 | 较低（文本协议） | 高（二进制协议） |
| 易用性 | 好理解 | 函数调用直观 |
| 调试 | 可用 HTTP 日志 | 需要专门工具 |
| 适用场景 | 外部API、Web端 | 内部服务调用 |
| 生态 | 丰富（工具链） | 单一（主要为gRPC） |

**我的经验：**
- 对外API使用REST，易于理解和使用
- 内部服务调用使用gRPC，性能好且有IDL约束
- 简单场景可以用JSON-RPC

### 5.3 消息队列：什么时候引入

消息队列是系统解耦和异步处理的重要工具，但引入消息队列会增加复杂度，需要谨慎。

**消息队列的核心应用场景：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     消息队列应用场景                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 异步处理                                                      │
│     ├── 场景：发送短信、邮件、推送通知                            │
│     └── 效果：不阻塞主流程，提升响应速度                          │
│                                                                 │
│  2. 系统解耦                                                      │
│     ├── 场景：订单完成后触发库存、物流、积分等多个下游            │
│     └── 效果：上游不关心下游实现，单点故障不传递                  │
│                                                                 │
│  3. 流量削峰                                                      │
│     ├── 场景：秒杀活动、突发流量                                │
│     └── 效果：消息队列缓存请求，consumer按能力消费              │
│                                                                 │
│  4. 事件驱动                                                      │
│     ├── 场景：状态变更触发后续处理                              │
│     └── 效果：构建事件总线，支持复杂业务流程                     │
│                                                                 │
│  5. 数据同步                                                      │
│     ├── 场景：主数据库变更同步到搜索、缓存                      │
│     └── 效果：解耦数据源，构建读写分离架构                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**常见消息队列对比：**

| 特性 | Kafka | RabbitMQ | RocketMQ |
|------|-------|----------|----------|
| **吞吐量** | 百万级/秒 | 万级/秒 | 十万级/秒 |
| **延迟** | 毫秒级 | 微秒级 | 毫秒级 |
| **消息持久化** | 支持 | 支持 | 支持 |
| **消息回溯** | 支持 | 不支持 | 支持 |
| **顺序消息** | 分区有序 | 单队列有序 | 支持 |
| **事务消息** | 支持 | 支持 | 支持 |
| **适用场景** | 日志、大数据 | 小消息、灵活路由 | 电商交易 |

### 5.4 我的思考：系统间的边界最难划

系统设计中最大的难题之一是**划分子系统的边界**。边界划得不好，会导致系统间高度耦合、修改困难。

**边界划分的基本原则：**

```
1. 单一职责原则
   └── 每个服务只负责一件事

2. 高内聚低耦合
   └── 强相关的功能放在一起

3. 业务边界清晰
   └── 按照业务领域划分，而非技术层次

4. 独立部署
   └── 能独立部署和发布
```

**常见的边界划分问题：**

| 问题 | 表现 | 解决方案 |
|------|------|----------|
| **过早拆分** | 服务数量多但调用复杂 | 初期用模块，逐步拆分为服务 |
| **边界模糊** | 功能不知道放哪个服务 | 明确业务边界，制定边界协议 |
| **过度耦合** | 服务间循环依赖 | 重新设计领域边界 |
| **职责不清** | 一个服务做太多事 | 拆分为多个服务 |

**判断边界是否合理的标准：**

1. **能否独立开发**？修改一个服务是否会影响其他服务
2. **能否独立部署**？能否单独发布而不需要同时发布其他服务
3. **能否独立扩展**？能否只扩展瓶颈服务
4. **数据是否独立**？是否有共享数据库

### 6.2 缓存层：Redis怎么用

缓存是提升系统性能的关键手段。Redis是最常用的分布式缓存解决方案。

**Redis 数据结构与适用场景：**

| 数据结构 | 命令 | 适用场景 |
|----------|------|----------|
| **String** | SET/GET | 简单KV、Session、验证码 |
| **Hash** | HSET/HGET | 对象存储、配置缓存 |
| **List** | LPUSH/RPOP | 队列、消息列表、最新N条 |
| **Set** | SADD/SMEMBERS | 标签、去重、关注列表 |
| **Sorted Set** | ZADD/ZRANGE | 排行榜、延迟队列、有序集合 |
| **Bitmap** | SETBIT | 用户签到、布隆过滤器 |
| **HyperLogLog** | PFADD | UV统计 |

**Redis 缓存策略：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     Redis 缓存策略                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. Cache-Aside（旁路缓存）- 最常用                              │
│     读：先读缓存，缓存不存在则读DB并写入缓存                      │
│     写：先写DB，再删除缓存（注意不是更新）                        │
│                                                                 │
│  2. Read-Through（读穿透）                                       │
│     读：缓存不存在，由缓存层自动加载                              │
│                                                                 │
│  3. Write-Through（写穿透）                                      │
│     写：同时写缓存和DB                                           │
│                                                                 │
│  4. Write-Behind（写回）                                          │
│     写：只写缓存，异步批量写DB                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.4 我的思考：数据是系统的核心

数据是系统最重要的资产，**数据架构设计必须谨慎**。

**数据架构设计的核心原则：**

1. **数据一致性是底线**
   - 金融、订单等核心数据必须保证一致性

2. **数据安全是生命线**
   - 敏感数据必须加密
   - 数据访问必须有审计日志

3. **数据备份是保险**
   - 定期备份，异地容灾
   - 定期演练恢复流程

4. **数据成本要控制**
   - 不是所有数据都需要实时
   - 冷数据可以归档

### 7.1 冗余设计：没有单点

### 7.2 降级熔断：保命用的

降级和熔断是保护系统的重要手段。当系统负载过高或依赖服务不可用时，通过降级非核心功能或熔断调用来保护核心功能。

**降级（Degradation）：**

| 策略 | 说明 | 示例 |
|------|------|------|
| **功能降级** | 关闭非核心功能 | 大促时关闭评论、推荐 |
| **返回缓存** | 缓存过期时返回旧数据 | 首页返回1小时前的缓存 |
| **返回默认值** | 服务不可用时返回预设值 | 返回空列表而非报错 |
| **人工降级** | 手动关闭部分功能 | 后台开关控制 |

**熔断（Circuit Breaker）：**

## 八、安全架构

### 8.1 认证授权：JWT还是Session

认证和授权是安全架构的基础。需要根据场景选择合适的方案。

**认证方案对比：**

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **Session** | 服务端存储会话，客户端存SessionID | 容易控制、会话管理方便 | 分布式麻烦、占用服务端资源 | 传统Web应用 |
| **JWT** | 令牌包含用户信息，签名验证 | 无状态、易于扩展 | 无法主动失效、令牌体积大 | 前后端分离、微服务 |
| **OAuth2** | 第三方授权 | 支持第三方登录 | 复杂度高 | 开放平台、社交登录 |

### 8.2 接口防护：防刷防爬

接口安全是系统的第一道防线。需要防范常见的攻击手段。

**常见接口攻击及防护：**

```
┌─────────────────────────────────────────────────────────────────┐
│                    常见接口攻击及防护                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. CSRF（跨站请求伪造）                                          │
│     ├── 攻击：在第三方网站诱导用户发起请求                        │
│     └── 防护：CSRF Token、SameSite Cookie                        │
│                                                                 │
│  2. XSS（跨站脚本攻击）                                           │
│     ├── 攻击：在页面注入恶意脚本                                  │
│     └── 防护：输入过滤、输出转义、CSP                             │
│                                                                 │
│  3. SQL 注入                                                      │
│     ├── 攻击：拼接恶意SQL                                        │
│     └── 防护：参数化查询、输入验证                                │
│                                                                 │
│  4. 接口刷量                                                      │
│     ├── 攻击：高频调用接口                                        │
│     └── 防护：限流、验证码、行为检测                              │
│                                                                 │
│  5. 暴力破解                                                      │
│     ├── 攻击：尝试大量密码登录                                    │
│     └── 防护：限流、验证码、账号锁定                              │
│                                                                 │
│  6. 重放攻击                                                      │
│     ├── 攻击：重放合法请求                                        │
│     └── 防护：请求签名、时间戳、一次性Token                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 8.3 数据安全：敏感数据加密

敏感数据必须加密存储，防止数据泄露后被利用。

**敏感数据分类与处理：**

| 数据类型 | 示例 | 处理方式 |
|----------|------|----------|
| **密码** | 用户密码 | BCrypt/SHA256 哈希 |
| **身份证** | 身份证号 | 脱敏或加密 |
| **手机号** | 手机号码 | 脱敏或加密 |
| **银行卡** | 卡号、CVV | 加密 + 脱敏展示 |
| **密钥** | API Key、Token | 加密存储 |
| **传输** | 网络传输 | HTTPS/TLS |

### 8.4 我的思考：安全不能事后考虑

安全架构必须在系统设计阶段就考虑，而不是上线后再补救。

**安全设计的原则：**

1. **纵深防御**：多层安全防护，一层被破还有下一层
2. **最小权限**：只授予必要的权限
3. **默认安全**：系统默认状态就是安全的
4. **简化设计**：复杂的安全设计更容易出错

**安全检查清单：**

```
□ 认证：强密码策略、多因素认证
□ 授权：基于角色的访问控制、最小权限
□ 会话管理：安全Token、合理的过期时间
□ 输入验证：所有用户输入都验证
□ 输出编码：防止XSS
□ SQL注入：参数化查询
□ 加密：敏感数据加密存储、传输加密
□ 日志：记录关键操作、异常事件
□ 限流：防止暴力破解、DDoS
□ 安全头：X-Frame-Options、CSP等
```

### 9.4 我的思考：可观测性是运维的基础

可观测性不是可选项，而是必须项。没有可观测性，系统出问题就是盲人摸象。

**可观测性建设的三个层次：**

1. **Metrics（指标）**：系统运行状态的量化数据
2. **Logs（日志）：** 离散的事件记录

```
   └── QPS、响应时间、错误率

3. 应用业务指标
   └── 核心业务动作计数

4. 日志收集
   └── 结构化日志、统一收集

5. 链路追踪
   └── 全链路请求追踪

6. 告警
   └── 异常自动通知
```

### 10.4 踩坑与迭代

**第一版架构的问题：**

| 问题 | 现象 | 原因 | 解决方案 |
|------|------|------|----------|
| 数据库连接过多 | 服务启动慢、高并发时连接耗尽 | 每个服务独立连接池 | 引入 PGBouncer 连接池 |
| 协作延迟高 | 编辑操作有感知延迟 | 协作服务与文档服务跨机房 | 就近部署协作服务 |
| 搜索不一致 | 文档更新后搜索不到 | 同步延迟 | 增加搜索同步确认 |
| 审计日志丢失 | 部分操作没记录 | 异步写入失败丢失 | 改为同步写入 + 定期补录 |

### 10.5 我的思考：好的架构是演进出来的

架构不是一蹴而就的，而是**根据业务发展逐步演进**的。

**架构演进的判断标准：**

1. **什么时候拆分服务？**
   - 单体代码超过 10 万行
   - 团队超过 10 人并行开发
   - 不同模块有不同扩展需求
   - 团队之间协调成本过高

2. **什么时候引入新技术？**
   - 当前方案有明显瓶颈
   - 新技术能解决实际问题
   - 团队有能力驾驭新技术
   - 收益大于成本

3. **什么时候重构？**
   - 架构限制了业务发展
   - 代码维护成本过高
   - 存在明显的技术债务

**架构设计的正确姿势：**

```
1. 始于简单
   └── 不要过度设计，先解决当前问题

2. 持续演进
   └── 根据业务发展调整架构

3. 数据驱动
   └── 用监控数据指导优化方向

4. 团队适配
   └── 架构要适配团队能力
```

### 11.2 评审常见问题

**架构评审中常见的问题：**

| 问题类型 | 具体问题 | 评审要点 |
|----------|----------|----------|
| **过度设计** | 微服务拆分过细、引入不需要的技术 | 是否有明确的收益？ |
| **设计不足** | 单点故障、扩展性差 | 未来3年需求能满足吗？ |
| **边界模糊** | 服务职责不清、循环依赖 | 模块是否可以独立部署？ |
| **忽略非功能** | 没考虑性能、安全、运维 | 非功能需求是否验证过？ |
| **缺少预案** | 没有降级、容灾、恢复方案 | 单点故障如何处理？ |

### 11.3 如何给出有价值的评审意见

**评审意见的结构：**

```
评审意见 = 问题描述 + 影响分析 + 建议方案 + 优先级
```

**评审意见示例：**

```markdown
## 评审意见

### 1. 【高优】协作服务单点问题

**问题描述：**
协作服务只有一个实例，存在单点故障风险。

**影响分析：**
- 协作服务宕机将导致所有实时协作功能不可用
- 当前架构无法水平扩展协作服务

**建议方案：**
1. 短期：部署多个协作服务实例，使用Redis Pub/Sub同步状态
2. 中期：引入协作服务集群，支持分区隔离
3. 验证方案：用ChaosEngineering测试故障场景

**优先级：P0（上线前必须解决）**
```

**评审的原则：**

| 原则 | 说明 |
|------|------|
| **建设性** | 指出问题也要给出解决方案 |
| **区分层次** | 分清关键问题和次要问题 |
| **考虑约束** | 理解业务和时间约束 |
| **达成共识** | 评审是为了达成一致，不是为了挑刺 |

### 11.4 我的思考：架构评审是团队成长的机会

架构评审不仅是评审架构，更是**团队知识共享和技术提升的机会**。

**评审的价值：**

1. **知识传播**
   - 让团队理解架构设计决策
   - 统一团队的技术认知

2. **风险发现**
   - 提前发现设计问题
   - 减少后期修复成本

3. **技术积累**
   - 记录评审决策和理由
   - 形成团队的技术资产

4. **人才培养**
   - 年轻工程师学习架构思维
   - 培养架构师后备力量

**如何做好架构评审：**

1. **评审前准备充分**
   - 提前发出评审材料
   - 让评审者有思考时间

2. **评审中充分讨论**
   - 鼓励提问和挑战
   - 记录讨论要点

3. **评审后跟踪落实**

   - 定期跟踪进度
