## 2. 核心战役一：前端底层与框架引擎

### 2.1 现代浏览器与性能 (INP 的崛起)
- **概念**：INP (Interaction to Next Paint) 已经完全取代 FID。它衡量整个页面生命周期中最长的交互延迟。
- **底层杀招**：主线程被长任务 (Long Task, >50ms) 锁死。
- **解法**：使用 `scheduler.yield()` 让出控制权，或者利用 **OffscreenCanvas + WebGPU** 将重度渲染转移到 Worker 线程。

### 2.2 React 19 核心引擎 (Fiber vs Compiler)
- **Fiber 架构**：本质上是一个 JS 对象链表，实现了 Render 阶段的**异步可中断**。`useState` 在底层是存在 `memoizedState` 上的单向链表，这就是为什么 Hook 不能写在 `if` 里的物理原因。
- **React Compiler**：通过 **AST 转换为 HIR (高级中间表示)**，利用 **SSA (静态单赋值)** 精准追踪变量版本。它在编译期自动生成 `_c(n)` 缓存数组，从根本上淘汰了手动写的 `useMemo`。
- **Action 与乐观更新**：`useOptimistic` 的本质是把更新挂载到了最高优先级的 **SyncLane**，而把真实请求挂载到了 **TransitionLane**，实现了视觉上的零延迟。

### 2.3 Vue 3.5 与响应式终局 (TC39 Signals)
- **Proxy 陷阱**：对 `reactive` 对象进行解构，会脱离 Proxy 包装变成纯值，导致响应式失效（需用 `toRefs` 或编译器宏）。
- **终局演进**：2026 年前端界正在被 **TC39 Signals 提案** 大一统。Vue 和 MobX 都在向 C++ 引擎级的原生 Signal 靠拢，以突破 JS 闭包依赖追踪的性能天花板。

## 3. 核心战役二：Node.js 引擎与高性能架构

### 3.1 V8 内存与垃圾回收 (Orinoco)
- **代际假说**：新生代用 **Scavenge (复制算法)** 解决碎片；老生代用 **Mark-Sweep (标记-清除)**。
- **2026 黑科技**：**增量标记 (Incremental Marking)** 和 **并发回收 (Concurrent)**，把 Stop-The-World (全停顿) 时间压缩到了几毫秒。
- **内存泄漏杀手**：未清理的定时器、全局变量、以及被 JS 变量死死抓住的**脱离 DOM 树节点 (Detached DOM)**。

### 3.2 异步 I/O 底层 (libuv 与背压)
- **文件 I/O 的谎言**：在 Linux 下，网络 I/O 是真异步（基于 epoll），但文件 I/O 无法 epoll。Node.js 是通过 libuv 的 **Thread Pool (默认 4 线程)** 模拟出来的伪异步。
- **Stream 背压 (Backpressure)**：如果写入速度慢于读取速度，达到 `highWaterMark` 时，`write()` 返回 false。底层会调用 `uv_read_stop`，最终引发系统内核级 TCP 接收窗口缩小至 0，从物理层逼迫发送端停手。

### 3.3 进程间通信 (IPC)
- **Cluster 的魔法**：为什么多个 Worker 进程可以同时监听 80 端口而不冲突？因为 Master 进程监听端口后，通过底层 Unix Domain Socket，将**底层 TCP 句柄 (File Descriptor)** 直接发送给 Worker，实现了内核级的负载均衡。

## 4. 核心战役三：数据库与高并发存储

### 4.1 B-Tree vs LSM-Tree
- **B-Tree (PostgreSQL)**：高度平衡的树，读取极快。但因为 SSD 不能覆盖写，更新 1 个字节会导致整页 8KB 甚至更多的**写入放大 (Write Amplification)**。
- **LSM-Tree (RocksDB)**：化随机写为**顺序写**，写入极快。但后台的 **Compaction (压缩合并)** 会导致 I/O 尖刺和严重的读放大。
- **优化黑科技**：Bloom Filter (布隆过滤器) 用于 LSM-Tree 极速判断“Key 不存在”；2026 年结合 ZNS 固态盘实现硬件级垃圾回收。

### 4.2 Serverless 连接池击穿 (Prisma)
- **痛点**：Serverless 瞬间横向扩容出 1000 个 Lambda 进程，每个进程建立一次 TCP 握手，瞬间打爆 PostgreSQL 默认的 100 个 `max_connections`。
- **解法**：必须在中间架设 HTTP 代理层（如 Prisma Accelerate 或 PgBouncer）。

## 5. 核心战役四：安全与网络基建

### 5.1 XSS 与 CSP Level 3
- 传统的转义防不胜防。2026 标准是 **Strict CSP**：废弃域名白名单，全量使用 **Nonce (随机数)** + **`strict-dynamic`**。
- **Trusted Types**：从浏览器 C++ 底层锁死 `innerHTML`，必须使用 Policy 过滤过的 `TrustedHTML` 对象才能赋值，彻底免疫 DOM XSS。

### 5.2 CSRF 与 OAuth 2.1
- **CSRF 终结者**：Cookie 的 `SameSite=Lax/Strict` 属性。
- **OAuth 2.1**：废除了存在被拦截风险的隐式流，强制所有终端使用 **PKCE** (利用 Code Verifier 和 Challenge 保证授权码哪怕被偷走也换不到 Token)。
- **WebAuthn**：基于硬件 TPM 的无密码登录。因为签名里混入了当前 Origin 域名，所以从物理原理上 100% 免疫钓鱼攻击。

## 6. 面试沟通必杀技 (STAR 法则)

当面试官问你“有没有做过什么复杂的性能优化”时，套用这个模板：
- **S (情境)**：“在我们 2026 年重构的那个协同看板项目中，初期发现随着组件数量超过 5000，页面出现了严重的输入卡顿。”
- **T (任务)**：“我的任务是把 P99 的 INP 指标从 800ms 降到 200ms 以内。”
- **A (行动)**：“我没有盲目加 `useMemo`。我首先抓取了 **Chrome Performance 火焰图**，发现主要时间耗在了一整块黄色的 JS 执行（Long Task）上。我采用了两步走：1. 将同步的巨量数据清洗逻辑放到 **Web Worker** 里。2. 在主线程渲染时，使用 **`scheduler.yield()`** 主动让出时间片给用户输入。”
- **R (结果)**：“最终 INP 稳定在 80ms，不仅通过了 Lighthouse 的严苛测试，连低端 Android 设备的转化率都提升了 15%。”
