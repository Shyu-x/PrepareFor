# JavaScript 内存模型与 V8 垃圾回收机制深度解析 (2026版)

## 1. 概述

JavaScript 作为一门自动管理内存的语言，让开发者免去了手动 `malloc` 和 `free` 的痛苦。然而，在复杂的大型 Web 应用（如数据可视化平台、长时间运行的 Node.js 服务）中，如果我们不了解底层的内存机制，极其容易引发**内存泄漏 (Memory Leak)**，导致页面卡顿崩溃或服务器 OOM (Out Of Memory)。

本指南将深入 Google V8 引擎的最底层，剖析栈与堆的区别、对象的内存分配布局，以及最新的 Orinoco 垃圾回收器是如何做到“无感回收”的。

---

## 2. V8 引擎的内存空间划分

在宿主环境（操作系统）分配给 V8 进程的物理内存中，主要被划分为两大区域：**栈空间 (Stack)** 和 **堆空间 (Heap)**。

### 2.1 栈空间 (Call Stack)
- **用途**：存储基础数据类型（`Number`, `String`, `Boolean`, `Null`, `Undefined`, `Symbol`, `BigInt`），以及指向堆内存对象的**指针引用**。它同时维护着执行上下文（Execution Context）的状态。
- **特性**：**空间极小（通常只有几 MB）、分配速度极快**。它由操作系统自动分配和释放。
- **释放机制**：当函数执行完毕，ESP（Extended Stack Pointer，栈顶指针）向下移动，该函数的执行上下文被销毁，其占据的栈空间瞬间被释放。**栈空间不存在所谓的垃圾回收算法，它是线性且确定的。**

### 2.2 堆空间 (Heap)
- **用途**：存储复杂的数据结构，如对象 (Object)、数组 (Array) 和函数 (Function)——即所谓的值类型在栈中，引用类型在堆中。
- **特性**：**空间巨大（可达几十 GB）、分配和读取相对缓慢**。所有存储在堆中的对象，其在栈中都只保留一个 32位 或 64位 的内存地址指针。
- **释放机制**：堆内存是无序的。当没有任何指针指向堆中的某个对象时，这个对象就变成了“孤岛（孤儿）”。这就是 V8 的**垃圾回收器 (Garbage Collector, GC)** 需要大显身手的地方。

## 3. 代际假说与 V8 的分代式垃圾回收

V8 的垃圾回收并非是对整个堆内存“一锅端”，它基于一个在计算机科学中被无数次验证的经验法则：**代际假说 (The Generational Hypothesis)**：
1. **朝生夕死**：绝大部分对象一经分配内存，很快就会变得不可达（比如函数内部的局部变量）。
2. **长生不死**：活得越久的对象，越倾向于继续存活（比如全局对象、DOM 节点引用）。

基于此，V8 将堆内存物理切割为两个区域：**新生代 (New Space)** 和 **老生代 (Old Space)**。

### 3.1 新生代：Scavenge (清道夫) 算法
新生代空间很小（通常只有 1-8 MB），专门存放那些“朝生夕死”的短期对象。

**运行机制 (Cheney 算法)：**
1. 新生代被一分为二：**From 空间** 和 **To 空间**。
2. 新创建的对象总是分配在 From 空间。
3. 当 From 空间快满时，触发一次 Minor GC（次级垃圾回收）。
4. V8 遍历 From 空间，找到所有存活（依然有引用）的对象，将它们**紧凑地复制 (Copy) 到 To 空间**。（这个复制过程顺便解决了内存碎片问题）。
5. 直接清空整个 From 空间！
6. **角色翻转**：原来的 To 空间变成下一次的 From 空间，原来的 From 变成 To。

**晋升机制 (Promotion)：**
如果一个对象在经历过一次 Scavenge 复制后依然存活，或者 To 空间的空间占用超过了 25%，这个对象就会被直接移动到**老生代**中。

### 3.2 老生代：标记-清除 (Mark-Sweep) 算法
老生代空间很大，存放着晋升过来的长寿对象。如果在这里使用复制算法，巨大的对象拷贝开销将是一场灾难。因此老生代使用 **标记-清除** 和 **标记-整理 (Mark-Compact)**。

1. **标记 (Marking)**：
   从根节点（Root，如全局 `window`、调用栈上的本地变量）出发，深度优先遍历整棵对象引用树。凡是能访问到的对象打上标记（存活）。
2. **清除 (Sweeping)**：
   遍历结束后，没有标记的对象就是垃圾。V8 将这些对象占用的内存记录到一个空闲列表（Free List）中，供下次分配使用。
3. **整理 (Compacting)**：
   单纯的清除会产生大量的内存碎片（就像一块破洞百出的奶酪）。如果此时要存入一个巨大的数组，可能会因为没有连续空间而失败。因此，V8 会偶尔执行整理，将所有存活对象向内存的一端滑动移动，把碎片挤出去。

## 4. 2026 前沿：Orinoco 垃圾回收器的黑科技

在传统的 GC 机制中，执行标记和清除时，必须停止整个 JS 引擎的运行——这被称为 **全停顿 (Stop-The-World, STW)**。如果在老生代执行一次 500ms 的 GC，页面就会死机半秒。

V8 的现代 GC 引擎 **Orinoco** 引入了三大黑科技，将停顿时间从几百毫秒压缩到了**几毫秒甚至无感**级别：

### 4.1 并行回收 (Parallel)
主线程在执行 GC 时，会唤醒多个辅助线程（Helper Threads）。大家一起分担标记或复制的工作。这虽然还是会 STW，但因为多核协作，时间大幅缩短（主要用于新生代的 Scavenge）。

### 4.2 增量标记 (Incremental Marking)
将原本漫长的老生代标记过程，**切分成无数个小碎片**。主线程执行一会儿 JS 代码，就暂停一下做一小段标记；然后再执行 JS 代码，再标记一小段。
- **三色标记法 (Tri-color Marking)**：V8 使用白、灰、黑三种颜色来记录标记进度，使得标记过程可以随时暂停和恢复，防止 JS 在执行期间修改了对象的引用关系导致标记错乱（写入屏障 Write Barrier 技术）。

### 4.3 并发回收 (Concurrent)
最极端的优化：主线程完全不停止，继续执行 JS 业务逻辑。辅助线程在后台默默地进行遍历标记和清理。目前 V8 的老生代标记和新生代的大部分清理已经实现了高度的并发化。

## 5. 导致内存泄漏的四大隐蔽杀手

如果你发现页面的内存占用像爬楼梯一样只增不减，通常是犯了以下禁忌：

**1. 意外的全局变量**
在非严格模式下，函数内遗漏 `let/const` 声明，会导致变量挂载到 `window` 上。由于 `window` 是 GC 的 Root 节点，它挂载的属性**永远不会被回收**。
*对策：永远使用严格模式 `'use strict'` 和模块化 ESM。*

**2. 被遗忘的定时器或事件监听**
```javascript
useEffect(() => {
  window.addEventListener('resize', largeDataProcessingFn);
  // 致命错误：没有 return () => window.removeEventListener(...)
}, []);
```
如果不移除监听，`window` 会一直持有 `largeDataProcessingFn` 的引用，进而导致该函数闭包作用域内的所有数据永远驻留老生代。

**3. 闭包的无意捕获**
如果一个大的局部变量在闭包环境中被引用，即使闭包没有直接使用它，由于 V8 的上下文实现机制，同一个作用域下的闭包共享同一个上下文对象，可能导致整个大对象无法回收。
*对策：在离开作用域前，将不需要的巨大对象手动置为 `null`，切断引用。*

**4. 分离的 DOM 节点 (Detached DOM)**
在 React/Vue 卸载组件后，如果你在外部的 JS 变量（如 Redux store 或一个全局 Map）中依然保留了对某个 DOM 节点的引用。那么这个节点不仅它自身不会被回收，它在 DOM 树上的所有子节点也都无法被回收！
