## 二、WebEnv-OS 项目深挖

### 2.1 项目背景与技术选型

**面试官可能这样问**：请介绍一下 WebEnv-OS 这个项目？你为什么选择做这样一个项目？

**回答思路**：从项目动机、技术目标、个人成长三个维度展开。

**参考回答**：

WebEnv-OS 是一个尝试在浏览器中从零构建类桌面操作系统交互界面的项目。之所以选择做这个项目，主要有两个原因：第一，在学习 React 过程中，我发现市面上很多教程都停留在"组件堆砌"的层面，我想通过一个具有挑战性的项目，深入理解浏览器原生能力与 React 状态驱动机制；第二，当时我正在学习 Zustand 状态管理库，需要一个合适的场景来实践。

这个项目的技术目标很明确：不使用任何 UI 库（如 Ant Design、MUI 等），完全依靠 HTML5 与 CSS3 搭建视窗结构，深入处理复杂的多窗口层级管理和强 DOM 交互逻辑。同时引入 Zustand 进行系统级状态管理，将 React 的声明式更新与原生 DOM 操作结合起来。

### 2.2 多窗口 z-index 管理机制

**面试官可能这样问**：你是如何管理多个窗口的层级关系的？如何确保新激活的窗口始终在最上层？

**回答思路**：先讲基本思路，再深入到实现细节，最后讨论潜在问题和优化方案。

**参考回答**：

我使用 Zustand 维护了一套完整的窗口层级管理系统。核心思路是：每次激活窗口时，将全局最高 z-index 加 1，确保新窗口始终处于最上层。

```javascript
// Zustand store 定义
import { create } from 'zustand';

const useOSStore = create((set, get) => ({
    // 窗口列表
    windows: [],
    // 当前激活窗口 ID
    activeWindowId: null,
    // 全局最高 z-index
    highestZIndex: 100,

    // 初始化窗口
    initWindow: (windowId, initialPosition, initialSize) => {
        set((state) => ({
            windows: [...state.windows, {
                id: windowId,
                position: initialPosition,
                size: initialSize,
                zIndex: state.highestZIndex + 1,
                isMinimized: false,
                isMaximized: false
            }],
            highestZIndex: state.highestZIndex + 1
        }));
    },

    // 窗口获取焦点（置顶）
    bringToFront: (windowId) => {
        set((state) => {
            const newHighestZIndex = state.highestZIndex + 1;
            return {
                windows: state.windows.map(win => 
                    win.id === windowId 
                        ? { ...win, zIndex: newHighestZIndex, isMinimized: false }
                        : win
                ),
                activeWindowId: windowId,
                highestZIndex: newHighestZIndex
            };
        });
    },

    // 窗口失焦
    handleBlur: (windowId) => {
        set({ activeWindowId: null });
    }
}));
```

**深入追问 1**：z-index 一直增加会不会有问题？

**可能的追问回答**：

理论上 z-index 会无限增加，长期运行可能会出现数值溢出的风险。但在实际项目中，这个问题很难出现，因为用户同时打开的窗口数量有限。不过，更优雅的解决方案是使用循环重置：当 z-index 超过某个阈值（比如 10000）时，将所有窗口的 z-index 减去一个偏移量。不过在我的项目中没有遇到这个问题，所以没有实现。

**深入追问 2**：点击窗口标题栏和点击窗口内部，处理方式有什么不同？

**可能的追问回答**：

这是一个很好的细节问题。我的实现是：点击窗口标题栏或窗口任意位置都会触发 bringToFront，但点击窗口外部（如桌面区域）不会改变窗口层级。这样设计的考虑是：用户点击窗口内部通常是为了与该窗口交互，自然希望该窗口置顶；而点击桌面可能是为了查看桌面图标或其他操作，不应该改变窗口层级。

```javascript
// 窗口组件中的事件处理
function Window({ id, children }) {
    const { bringToFront, activeWindowId } = useOSStore();
    const isActive = activeWindowId === id;

    const handleMouseDown = (e) => {
        // 只有非最小化状态才置顶
        bringToFront(id);
    };

    return (
        <div 
            className={`window ${isActive ? 'active' : ''}`}
            onMouseDown={handleMouseDown}
            style={{ zIndex: getWindowZIndex(id) }}
        >
            <WindowTitleBar id={id} />
            <div className="window-content">
                {children}
            </div>
        </div>
    );
}
```

### 2.3 窗口拖拽实现原理

**面试官可能这样问**：窗口拖拽是如何实现的？拖拽过程中如何保证性能？

**回答思路**：从基本原理讲起，重点说明鼠标事件处理、坐标计算、性能优化三个层面。

**参考回答**：

窗口拖拽的核心原理是：监听鼠标按下、移动、抬起三个事件，通过计算鼠标位移来更新窗口位置。我的实现采用了"三阶段"处理方式：

**第一阶段：鼠标按下（Drag Start）**

```javascript
// 拖拽 Hook 实现
function useWindowDrag(windowId) {
    const [isDragging, setIsDragging] = useState(false);
    const [offset, setOffset] = useState({ x: 0, y: 0 });
    
    // 使用 ref 存储拖拽起始点，避免闭包陷阱
    const dragStartRef = useRef({ x: 0, y: 0, windowX: 0, windowY: 0 });

    const handleMouseDown = (e) => {
        // 只响应标题栏的拖拽
        if (!e.target.closest('.window-titlebar')) return;
        
        e.preventDefault(); // 防止选中文本
        setIsDragging(true);
        
        // 记录鼠标起始位置和窗口当前位置
        const currentPosition = getWindowPosition(windowId);
        dragStartRef.current = {
            x: e.clientX,
            y: e.clientY,
            windowX: currentPosition.x,
            windowY: currentPosition.y
        };
    };
```

**第二阶段：鼠标移动（Dragging）**

```javascript
    // 使用 useCallback 优化，配合 useEffect 依赖
    const handleMouseMove = useCallback((e) => {
        if (!isDragging) return;

        // 计算鼠标位移差值
        const deltaX = e.clientX - dragStartRef.current.x;
        const deltaY = e.clientY - dragStartRef.current.y;

        // 计算新位置 = 起始窗口位置 + 鼠标位移
        const newX = dragStartRef.current.windowX + deltaX;
        const newY = dragStartRef.current.windowY + deltaY;

        // 更新窗口位置
        updateWindowPosition(windowId, { x: newX, y: newY });
    }, [isDragging, windowId]);
```

**第三阶段：鼠标抬起（Drag End）**

```javascript
    const handleMouseUp = useCallback(() => {
        setIsDragging(false);
    }, []);

    // 事件绑定与解绑
    useEffect(() => {
        if (isDragging) {
            window.addEventListener('mousemove', handleMouseMove);
            window.addEventListener('mouseup', handleMouseUp);
        }
        
        return () => {
            window.removeEventListener('mousemove', handleMouseMove);
            window.removeEventListener('mouseup', handleMouseUp);
        };
    }, [isDragging, handleMouseMove, handleMouseUp]);

    return { isDragging, handleMouseDown };
}
```

**性能优化策略**：

1. **使用 transform 替代 top/left**：我使用了 CSS transform: translate(x, y) 而不是 top/left 属性来移动窗口。transform 不会触发布局重排（Reflow），只会触发合成（Composite），由 GPU 加速处理，渲染效率更高。

```css
.window {
    position: absolute;
    /* 使用 transform 替代 top/left */
    transform: translate(var(--window-x), var(--window-y));
    /* 开启 GPU 加速 */
    will-change: transform;
}
```

2. **使用 requestAnimationFrame**：在拖拽过程中，鼠标移动事件触发频率可能很高（高达数百次/秒），直接更新状态可能导致 React 组件频繁渲染。更优的做法是将位置更新逻辑放在 requestAnimationFrame 中：

```javascript
const handleMouseMove = useCallback((e) => {
    if (!isDragging) return;
    
    // 使用 requestAnimationFrame 节流
    requestAnimationFrame(() => {
        const deltaX = e.clientX - dragStartRef.current.x;
        const deltaY = e.clientY - dragStartRef.current.y;
        const newX = dragStartRef.current.windowX + deltaX;
        const newY = dragStartRef.current.windowY + deltaY;
        updateWindowPosition(windowId, { x: newX, y: newY });
    });
}, [isDragging, windowId]);
```

3. **使用 useRef 避免闭包陷阱**：我在拖拽起始点存储在 ref 中，而不是 state。这是因为如果在 useEffect 中直接使用 state，会遇到闭包过期的问题——事件回调中读取的总是旧的 state 值。useRef 的值在组件整个生命周期内保持一致，解决了我在开发过程中遇到的"拖拽位置跳变"的 bug。

### 2.4 窗口缩放实现

**面试官可能这样问**：窗口缩放功能是如何实现的？拖拽窗口边缘 resize 时需要注意什么？

**参考回答**：

窗口缩放与拖拽的原理类似，但有几个关键区别：

**边缘检测与方向判断**

```javascript
function useWindowResize(windowId) {
    const RESIZE_EDGE = 8; // 边缘检测宽度
    
    const handleMouseMove = useCallback((e) => {
        const rect = getWindowRect(windowId);
        const { clientX, clientY } = e;
        
        // 计算鼠标相对于窗口边缘的距离
        const distanceToRight = rect.right - clientX;
        const distanceToBottom = rect.bottom - clientY;
        const distanceToLeft = clientX - rect.left;
        const distanceToTop = clientY - rect.top;
        
        // 判断是否在边缘区域
        const isRightEdge = distanceToRight < RESIZE_EDGE;
        const isBottomEdge = distanceToBottom < RESIZE_EDGE;
        const isLeftEdge = distanceToLeft < RESIZE_EDGE;
        const isTopEdge = distanceToTop < RESIZE_EDGE;
        
        // 设置光标样式
        let cursor = 'default';
        if (isRightEdge && isBottomEdge) cursor = 'se-resize';
        else if (isLeftEdge && isTopEdge) cursor = 'nw-resize';
        else if (isRightEdge) cursor = 'e-resize';
        else if (isLeftEdge) cursor = 'w-resize';
        else if (isBottomEdge) cursor = 's-resize';
        else if (isTopEdge) cursor = 'n-resize';
        
        setCursor(cursor);
    }, [windowId]);
}
```

**位置与尺寸的联动处理**

窗口缩放与拖拽最大的不同在于：拖拽只改变位置，缩放需要同时改变位置和尺寸（从边缘向内/向外拖动时，位置也要相应调整）。我的实现是：

- 从右/下边缘 resize：只改变尺寸
- 从左/上边缘 resize：改变尺寸的同时等量移动位置（保持右下角不动）

```javascript
const handleResize = (direction, deltaX, deltaY) => {
    const currentRect = getWindowRect(windowId);
    let newRect = { ...currentRect };

    switch (direction) {
        case 'right':
            newRect.width = Math.max(MIN_WIDTH, currentRect.width + deltaX);
            break;
        case 'left':
            const widthDelta = Math.min(deltaX, currentRect.width - MIN_WIDTH);
            newRect.width = currentRect.width - widthDelta;
            newRect.x = currentRect.x + widthDelta;
            break;
        case 'bottom':
            newRect.height = Math.max(MIN_HEIGHT, currentRect.height + deltaY);
            break;
        case 'top':
            const heightDelta = Math.min(deltaY, currentRect.height - MIN_HEIGHT);
            newRect.height = currentRect.height - heightDelta;
            newRect.y = currentRect.y + heightDelta;
            break;
    }

    updateWindowRect(windowId, newRect);
};
```

### 2.5 为什么选择 Zustand？和 Redux 有什么区别？

**面试官可能这样问**：你为什么选择 Zustand 作为状态管理方案？它和 Redux 相比有什么优劣？

**参考回答**：

选择 Zustand 主要基于以下几个考量：

**1. 极简的 API 设计**

Zustand 的 API 非常简洁，只需要创建一个 store 并导出 hook 即可使用，不需要 Provider 嵌套：

```javascript
// Zustand
import { create } from 'zustand';

const useOSStore = create((set) => ({
    windows: [],
    activeWindowId: null,
    addWindow: (window) => set((state) => ({ 
        windows: [...state.windows, window] 
    })),
    setActiveWindow: (id) => set({ activeWindowId: id })
}));

// 在组件中使用 - 无需 Provider
function Taskbar() {
    const windows = useOSStore((state) => state.windows);
    return <div>{windows.length} 个窗口</div>;
}
```

对比 Redux，需要创建 store、编写 reducer、配置中间件（如果需要）、用 Provider 包裹应用：

```javascript
// Redux
import { createStore } from 'redux';
import { Provider, useDispatch, useSelector } from 'react-redux';

// 定义 reducer
function osReducer(state = { windows: [], activeWindowId: null }, action) {
    switch (action.type) {
        case 'ADD_WINDOW':
            return { ...state, windows: [...state.windows, action.payload] };
        case 'SET_ACTIVE':
            return { ...state, activeWindowId: action.payload };
        default:
            return state;
    }
}

const store = createStore(osReducer);

// Provider 包裹
function App() {
    return (
        <Provider store={store}>
            <Taskbar />
        </Provider>
    );
}

// 组件中使用
function Taskbar() {
    const windows = useSelector(state => state.windows);
    const dispatch = useDispatch();
    return <div onClick={() => dispatch({ type: 'ADD_WINDOW' })}>添加窗口</div>;
}
```

**2. 极小的体积**

Zustand 体积仅约 1KB（gzip 后），而 Redux + React-Redux 体积约 7KB。对于我这样的练手项目，Zustand 的轻量特性非常合适。

**3. 支持 React 外部更新状态**

Zustand 允许在 React 组件外部更新状态，这在某些场景下非常有用：

```javascript
// 在非组件环境中更新状态
import { create } from 'zustand';

constuseStore = create((set) => ({
    count: 0,
    increment: () => set((state) => ({ count: state.count + 1 }))
}));

// 在事件处理器中直接使用
document.getElementById('btn').addEventListener('click', () => {
    useStore.getState().increment(); // 无需 useDispatch
});
```

**4. 基于 Proxy 的细粒度更新**

Zustand 使用 Proxy 实现状态监听，可以做到组件级别的精确更新，避免不必要的渲染：

```javascript
// 只有 count 变化时才重新渲染 Counter 组件
function Counter() {
    const count = useStore((state) => state.count);
    return <div>{count}</div>;
}

// 如果需要多个状态，可以分别选择
function Info() {
    const name = useStore((state) => state.name);
    const age = useStore((state) => state.age);
    return <div>{name} - {age}</div>;
}
```

**Zustand 的局限性**：

当然，Zustand 也有它的局限：缺乏 Redux 那样成熟的中间件生态（如 redux-thunk、redux-saga、redux-devtools），不适合需要复杂异步流程管理的项目。对于 WebEnv-OS 这样以本地状态为主的项目，Zustand 是合适的选择；如果是一个大型企业级应用，可能还需要更强大的状态管理方案。

### 2.6 Zustand 的发布订阅模式

**面试官可能这样问**：你提到 Zustand 使用了发布订阅模式，能详细解释一下吗？

**参考回答**：

Zustand 的核心确实基于发布订阅模式。让我从源码角度分析一下：

**基本原理**

```javascript
// Zustand 核心实现简化版
function create(set, get) {
    let state;
    const listeners = new Set();

    const setState = (partial) => {
        // 计算新状态
        const nextState = typeof partial === 'function' 
            ? partial(state) 
            : partial;
        
        // 如果状态没变，不触发更新
        if (nextState !== state) {
            const previousState = state;
            state = Object.assign({}, state, nextState);
            
            // 发布：通知所有监听者
            listeners.forEach(listener => listener(state, previousState));
        }
    };

    const getState = () => state;

    const subscribe = (listener) => {
        listeners.add(listener);
        // 返回取消订阅函数
        return () => listeners.delete(listener);
    };

    const destroy = () => {
        listeners.clear();
    };

    // 返回 API
    return {
        setState,
        getState,
        subscribe,
        destroy,
        // 暴露给用户的 store
        ...api
    };
}
```

**React 集成**

```javascript
// useStore hook 的实现
function useStore(selector = (state) => state) {
    const [, forceUpdate] = useState(0);
    
    // 获取当前状态
    const state = useSyncExternalStore(
        api.subscribe,          // 订阅函数
        () => selector(api.getState()), // 获取当前状态
        () => selector(initialState)    // 服务端渲染支持
    );

    return state;
}
```

**发布订阅在项目中的应用**

在 WebEnv-OS 中，我利用 Zustand 的订阅机制实现了窗口状态变化的监听：

```javascript
// 监听窗口变化，执行额外逻辑
useEffect(() => {
    const unsubscribe = useOSStore.subscribe((state, prevState) => {
        // 当窗口列表变化时，保存到 localStorage
        if (state.windows !== prevState.windows) {
            localStorage.setItem('os-windows', JSON.stringify(state.windows));
        }
        
        // 当活动窗口变化时，触发焦点事件
        if (state.activeWindowId !== prevState.activeWindowId) {
            handleWindowFocusChange(state.activeWindowId);
        }
    });
    
    return () => unsubscribe();
}, []);
```

### 2.7 Monaco Editor 集成与排错

**面试官可能这样问**：Monaco Editor 集成过程中遇到过什么问题？你是如何解决的？

**参考回答**：

Monaco Editor 是 VS Code 的核心编辑器组件，功能强大但集成起来确实有不少坑。我主要遇到了以下问题：

**问题一：Monaco 体积过大，首屏加载慢**

Monaco Editor 完整包体达几 MB，直接引入会影响首屏加载。我的解决方案是使用动态导入和按需加载：

```javascript
// 使用 @monaco-editor/react 的懒加载
import Editor from '@monaco-editor/react';

// 配置项：首屏显示加载状态，实际使用时再加载 Monaco
<Editor
    height="100%"
    defaultLanguage="javascript"
    loading={<div>编辑器加载中...</div>}
    options={{
        // 禁用不需要的功能，减少加载体积
        minimap: { enabled: false },
        folding: false,
        lineNumbers: 'on',
        scrollBeyondLastLine: false,
        automaticLayout: true
    }}
/>
```

**问题二：与 React 状态管理冲突**

Monico Editor 有自己的内部状态管理，与 React 状态同步时容易出现问题。我的解决方案是使用 useRef 保存 Editor 实例：

```javascript
function CodeEditorWindow({ windowId, initialCode }) {
    const editorRef = useRef(null);
    const [code, setCode] = useState(initialCode || '');

    const handleEditorDidMount = (editor, monaco) => {
        editorRef.current = editor;
        
        // 自定义主题
        monaco.editor.defineTheme('os-dark', {
            base: 'vs-dark',
            inherit: true,
```

### 2.8 浏览器渲染流水线

**面试官可能这样问**：你在简历中提到了解浏览器渲染流水线，能详细说一下吗？

**参考回答**：

浏览器渲染流水线是前端性能优化的理论基础。我在项目中处理拖拽和动画时，多次涉及到这个知识点。

**渲染流水线的完整流程**

```
1. JavaScript 执行 → 2. 样式计算 (Style) → 3. 布局 (Layout/Reflow) → 4. 绘制 (Paint) → 5. 合成 (Composite)
```

**各阶段详解**

1. **JavaScript 执行**：JS 线程执行 JS 代码，可能修改 DOM 结构或 CSS 样式
2. **样式计算**：浏览器计算每个 DOM 元素的最终样式（CSS 规则匹配、继承、CSS 变量解析）
3. **布局（Layout/Reflow）**：根据样式计算每个元素的几何位置和尺寸
4. **绘制（Paint）**：将每个元素绘制到多个图层上（填充颜色、边框、阴影等）
5. **合成（Composite）**：将多个图层合成为最终页面

**关键优化点：避免触发 Layout 和 Paint**

在 WebEnv-OS 的拖拽实现中，我特别注意了这一点：

```javascript
// ❌ 错误：修改 top/left 会触发 Layout
function handleDragBad(deltaX, deltaY) {
    const element = document.getElementById('window');
    element.style.left = `${currentX + deltaX}px`;
    element.style.top = `${currentY + deltaY}px`;
}

// ✅ 正确：使用 transform 只触发 Composite
function handleDragGood(deltaX, deltaY) {
    const element = document.getElementById('window');
    element.style.transform = `translate(${currentX + deltaX}px, ${currentY + deltaY}px)`;
}
```

**will-change 优化**

对于频繁变化的属性，可以提前告知浏览器进行优化：

```css
.window.dragging {
    /* 提示浏览器提前为 transform 创建独立图层 */
    will-change: transform;
}
```

但需要注意：will-change 不能滥用，否则会创建过多图层，消耗过多内存。

**图层的概念**

浏览器会为某些元素创建独立的合成图层（Compositing Layer）：

- 拥有 CSS 3D transform 的元素
- 使用 CSS position: fixed 的元素
- 使用 CSS will-change: transform 的元素
- video、canvas 元素
- 动画元素

每个图层由 GPU 独立渲染，这就是 transform 和 opacity 能够由 GPU 加速的原因。

```javascript
// 需要手动维护操作历史
class HistoryManager {
    constructor() {
        this.stack = [];
        this.index = -1;
    }
    
    push(operation) {
        // 截断当前索引后的所有操作
        this.stack = this.stack.slice(0, this.index + 1);
        this.stack.push(operation);
        this.index++;
    }
    
    undo() {
        if (this.index >= 0) {
            const operation = this.stack[this.index];
            this.applyInverse(operation);
            this.index--;
        }
    }
}
```

### 3.3 TipTap 编辑器扩展：Markdown 快捷输入

**面试官可能这样问**：你是如何在 TipTap 中扩展 Markdown 快捷输入功能的？

**参考回答**：

Markdown 快捷输入是提升编辑器体验的关键功能。我在 TipTap 中扩展了这个功能，当用户输入特定字符时，自动将其转换为对应的格式。

**实现原理**

TipTap 提供了 InputRule 和 PasteRule 两种规则机制：

```javascript
import { InputRule } from '@tiptap/core';

// 自定义 Markdown 快捷输入扩展
import { Extension } from '@tiptap/core';
import { Plugin, PluginKey } from '@tiptap/pm/state';

const MarkdownShortcuts = Extension.create({
    name: 'markdownShortcuts',

    addInputRules() {
        return [
            // 标题：# 空格 → <h1>
            new InputRule({
                find: /^#\s$/,
                handler: ({ state, range, chain }) => {
                    chain().deleteRange(range).setNode('heading', { level: 1 }).run();
                },
            }),
            // 加粗：**text** → <strong>text</strong>
            new InputRule({
                find: /\*\*([^*]+)\*\*$/,
                handler: ({ state, range, match }) => {
                    const start = range.from;
                    const end = range.to;
                    const text = match[1];
                    
                    state.tr.replaceWith(
                        start - 2, // 去掉 **
                        end + 2,
                        state.schema.nodes.strong.create(null, [
                            state.schema.text(text)
                        ])
                    );
                },
            }),
            // 链接：[text](url) → <a href="url">text</a>
            new InputRule({
                find: /\[([^\]]+)\]\(([^)]+)\)$/,
                handler: ({ state, range, match }) => {
                    const start = range.from;
                    const end = range.to;
                    const text = match[1];
                    const url = match[2];
                    
                    state.tr.replaceWith(
                        start - match[0].length,
                        end,
                        state.schema.nodes.link.create({ href: url }, [
                            state.schema.text(text)
                        ])
                    );
                },
            }),
            // 代码：`` → <code>
            new InputRule({
                find: /`([^`]+)`$/,
                handler: ({ state, range, match }) => {
                    const start = range.from;
                    const end = range.to;
                    const text = match[1];
                    
                    state.tr.replaceWith(
                        start - 1,
end + 1,
                        state.schema.nodes.code.create(null, [
                            state.schema.text(text)
                        ])
                    );
                },
            }),
        ];
    },
});
```

### 3.5 WebSocket 实时同步机制

**面试官可能这样问**：FastDocument 的实时同步是如何实现的？WebSocket 连接是如何管理的？

**参考回答**：

**Socket.io 基础架构**

```javascript
// 前端 Socket.io 客户端
import { io } from 'socket.io-client';

class DocumentSyncManager {
    constructor(documentId, onContentChange, onCursorChange) {
        this.documentId = documentId;
        this.onContentChange = onContentChange;
        this.onCursorChange = onCursorChange;
        this.socket = null;
        this.connected = false;
        this.operationQueue = []; // 操作队列，用于断线重连
    }

    connect() {
        this.socket = io('http://localhost:3000', {
            reconnection: true,          // 启用自动重连
            reconnectionAttempts: 5,     // 最多重连 5 次
            reconnectionDelay: 1000,     // 初始重连延迟 1s
            reconnectionDelayMax: 5000,  // 最大重连延迟 5s
            timeout: 20000,             // 连接超时 20s
        });

        this.setupEventHandlers();
    }

    setupEventHandlers() {
        // 连接成功
        this.socket.on('connect', () => {
            console.log('WebSocket connected');
            this.connected = true;
            
            // 加入文档房间
            this.socket.emit('join-document', this.documentId);
            
            // 重连后同步本地操作队列
            this.flushOperationQueue();
        });

        // 断开连接
        this.socket.on('disconnect', (reason) => {
            console.log('WebSocket disconnected:', reason);
            this.connected = false;
        });

        // 连接错误
        this.socket.on('connect_error', (error) => {
            console.error('Connection error:', error);
        });

        // 文档内容变化（来自服务器广播）
        this.socket.on('document-change', (change) => {
            this.onContentChange(change);
        });

        // 光标位置变化（来自其他用户）
        this.socket.on('cursor-update', (data) => {
            this.onCursorChange(data);
        });
    }

    // 发送文档变化
    sendChange(change) {
        if (!this.connected) {
            // 未连接时加入操作队列
            this.operationQueue.push(change);
            return;
        }

        this.socket.emit('document-change', {
            documentId: this.documentId,
            change: change,
            timestamp: Date.now()
        });
    }

    // 发送光标更新
    sendCursorUpdate(cursorData) {
        if (!this.connected) return;
        
        this.socket.emit('cursor-update', {
            documentId: this.documentId,
            userId: this.getCurrentUserId(),
            position: cursorData.position,
            userInfo: this.getCurrentUserInfo()
        });
    }

    // 重连后同步操作队列
    flushOperationQueue() {
        while (this.operationQueue.length > 0) {
            const change = this.operationQueue.shift();
            this.sendChange(change);
        }
    }

    disconnect() {
        if (this.socket) {
            this.socket.disconnect();
        }
    }
}
```

**后端简单的广播实现**

```javascript
// 服务端（Node.js + Socket.io）
const io = require('socket.io')(httpServer, {
    cors: {
        origin: '*',
    }
});

// 文档房间管理
const documentRooms = new Map();

io.on('connection', (socket) => {
    let currentDocumentId = null;

    // 加入文档房间
    socket.on('join-document', (documentId) => {
        currentDocumentId = documentId;
        socket.join(documentId);
        
        // 记录房间人数
        if (!documentRooms.has(documentId)) {
            documentRooms.set(documentId, new Set());
        }
        documentRooms.get(documentId).add(socket.id);
        
        // 通知房间内其他用户
        socket.to(documentId).emit('user-joined', {
            userId: socket.id,
            userCount: documentRooms.get(documentId).size
        });
    });

    // 文档内容变化广播
    socket.on('document-change', (data) => {
        // 广播给房间内除发送者外的所有人
        socket.to(data.documentId).emit('document-change', {
            change: data.change,
            userId: socket.id,
            timestamp: data.timestamp
        });
    });

    // 光标位置广播
    socket.on('cursor-update', (data) => {
        socket.to(data.documentId).emit('cursor-update', {
            userId: socket.id,
            position: data.position,
            userInfo: data.userInfo
        });
    });

    // 离开房间
    socket.on('leave-document', () => {
        if (currentDocumentId) {
            socket.leave(currentDocumentId);
            if (documentRooms.has(currentDocumentId)) {
                documentRooms.get(currentDocumentId).delete(socket.id);
            }
        }
    });

    // 断开连接
    socket.on('disconnect', () => {
        if (currentDocumentId && documentRooms.has(currentDocumentId)) {
            documentRooms.get(currentDocumentId).delete(socket.id);
            // 通知其他用户
            socket.to(currentDocumentId).emit('user-left', {
                userId: socket.id
            });
        }
    });
});
```

### 3.7 LiveKit 视频会议集成

**面试官可能这样问**：LiveKit 是如何集成的？多人视频会议的核心功能怎么实现？

**参考回答**：

LiveKit 是一个开源的实时音视频（WebRTC）引擎，我在 FastDocument 中探索性地集成了基础的多人会议功能。

**基本架构**

```javascript
import { Room, VideoPresets, ConnectionState } from 'livekit-client';

class VideoConference {
    constructor() {
        this.room = null;
        this.localTracks = [];
        this.remoteParticipants = new Map();
    }

    // 加入房间
    async joinRoom(url, token) {
        // 创建 Room 实例
        this.room = new Room({
            adaptiveStream: true,
            dynacast: true,
        });

        // 监听房间事件
        this.setupRoomEventHandlers();

        // 连接
        await this.room.connect(url, token);
        
        // 开启本地音视频
        await this.room.localParticipant.enableCameraAndMicrophone();
        
        return this.room;
    }

    setupRoomEventHandlers() {
        // 参与者加入
        this.room.on('participantConnected', (participant) => {
            console.log('参与者加入:', participant.identity);
            this.remoteParticipants.set(participant.identity, participant);
            this.updateParticipantList();
        });

        // 参与者离开
        this.room.on('participantDisconnected', (participant) => {
            console.log('参与者离开:', participant.identity);
            this.remoteParticipants.delete(participant.identity);
            this.updateParticipantList();
        });

        // 轨道订阅成功（收到远程视频/音频）
        this.room.on('trackSubscribed', (track, publication, participant) => {
            console.log('订阅轨道:', track.kind, 'from', participant.identity);
            this.attachTrack(track, participant.identity);
        });

        // 轨道取消订阅
        this.room.on('trackUnsubscribed', (track, publication, participant) => {
            this.detachTrack(track);
        });

        // 连接状态变化
        this.room.on('connectionStateChanged', (state) => {
            console.log('连接状态:', state);
            if (state === ConnectionState.Disconnected) {
                this.handleDisconnection();
            }
        });
    }

    // 将轨道添加到 DOM
    attachTrack(track, participantId) {
        const element = track.attach();
        element.id = `track-${track.kind}-${participantId}`;
        
        // 根据轨道类型添加到不同容器
        if (track.kind === 'video') {
            document.getElementById('video-container').appendChild(element);
        } else if (track.kind === 'audio') {
            document.getElementById('audio-container').appendChild(element);
        }
    }

    // 从 DOM 移除轨道
    detachTrack(track) {
        track.detach().forEach((el) => el.remove());
    }

    // 离开房间
    async leaveRoom() {
        if (this.room) {
            await this.room.disconnect();
            this.room = null;
            this.localTracks = [];
            this.remoteParticipants.clear();
        }
    }

    // 切换麦克风
    toggleMicrophone() {
        const audioEnabled = this.room.localParticipant.isMicrophoneEnabled;
        if (audioEnabled) {
            this.room.localParticipant.disableMicrophone();
        } else {
            this.room.localParticipant.enableMicrophone();
        }
    }

    // 切换摄像头
    toggleCamera() {
        const videoEnabled = this.room.localParticipant.isCameraEnabled;
        if (videoEnabled) {
            this.room.localParticipant.disableCamera();
        } else {
            this.room.localParticipant.enableCamera();
        }
    }
}
```

### 3.8 CRDT 与 OT 算法初步了解

**面试官可能这样问**：实时协作编辑器的核心算法了解多少？能介绍一下 CRDT 或 OT 吗？

**参考回答**：

这是一个非常深入的问题。我在项目中主要使用了简单的"最后写入胜出"策略，但确实了解过 CRDT 和 OT 这两种主流的协作算法。

**OT（Operational Transformation）**

OT 是最早被广泛使用的协作编辑算法，Google Docs 早期使用的就是 OT：

- **核心思想**：将用户的编辑操作（插入、删除）进行转换，使得多个操作可以按正确顺序执行
- **关键挑战**：操作转换的复杂性，尤其是涉及并发插入和删除时
- **特点**：需要中心服务器协调操作顺序

```javascript
// OT 操作示例
const ops = [
    { type: 'insert', position: 5, text: 'hello' },
    { type: 'delete', position: 3, length: 2 }
];

// 转换函数示例
function transform(op1, op2) {
    // 如果 op2 在 op1 之前，op1 的位置需要调整
    if (op2.position <= op1.position) {
        return {
            ...op1,
            position: op1.position + getOpLength(op2)
        };
    }
    return op1;
}
```

**CRDT（Conflict-free Replicated Data Types）**

CRDT 是更现代的、去中心化的协作算法：

- **核心思想**：设计数据结构，使得并发修改可以自动合并，不会产生冲突
- **优势**：去中心化、最终一致性、离线支持
- **应用**：Figma、Notion 等使用 Yjs（基于 CRDT）

```javascript
// Yjs - 最流行的 CRDT 实现
import * as Y from 'yjs';
import { WebsocketProvider } from 'y-websocket';

const ydoc = new Y.Doc();
const ytext = ydoc.getText('document');

// 本地修改
ytext.insert(0, 'Hello World');

// 监听远程修改
ytext.observe((event) => {
    console.log('远程修改:', event.delta);
});

// 通过 WebSocket 同步
const provider = new WebsocketProvider(
    'ws://localhost:1234',
    'document-name',
    ydoc
);
```

### 3.9 项目中遇到的挑战与解决

**面试官可能这样问**：在 FastDocument 项目中，你遇到的最大挑战是什么？如何解决的？

**参考回答**：

我在这个项目中遇到了多个挑战，这里分享几个印象深刻的：

**挑战一：编辑器内容与状态同步**

TipTap 编辑器内部维护了自己的状态，如何将其与 React 状态同步是一个难题。最初的方案是在 onChange 回调中更新 state，但很快发现这会导致无限循环（编辑器变化触发 state 更新，state 更新又触发编辑器变化）。

```javascript
// ❌ 错误方案：导致无限循环
const [content, setContent] = useState('');
const editor = useEditor({
    onUpdate: ({ editor }) => {
        setContent(editor.getHTML()); // 触发重渲染
    }
});

// 每次 editor.getHTML() 返回新引用，编辑器重新渲染，触发 onUpdate...
```

解决方案是使用适当的方式同步，或者接受编辑器内部状态作为"真实来源"：

```javascript
// ✅ 方案一：只在需要时同步
const editor = useEditor({
    onUpdate: ({ editor }) => {
        // 使用防抖
        debouncedSave(editor.getHTML());
    }
});

// ✅ 方案二：使用 editorRef 而不是 state
const editorRef = useRef(null);
const editor = useEditor({
    onUpdate: ({ editor }) => {
        editorRef.current = editor; // 只存引用，不触发重渲染
    }
});

// 在需要时获取内容
const handleSave = () => {
    const content = editorRef.current?.getHTML();
    saveToServer(content);
};
```

**挑战二：WebSocket 断线重连后的数据同步**

实际使用中发现，如果网络中断一段时间，重新连接后可能出现数据不一致。

解决方案是实现完整的"操作日志 + 重放"机制：

```javascript
class SyncManager {
    constructor() {
        this.operationLog = []; // 本地操作日志
        this.lastSyncTime = 0;
    }

    // 记录操作
    recordOperation(op) {
        this.operationLog.push({
            op,
            timestamp: Date.now()
        });
        
        // 定期清理太老的日志
        if (this.operationLog.length > 1000) {
            this.operationLog = this.operationLog.slice(-500);
        }
    }

    // 重放操作
    replayOperations(fromTimestamp) {
        const ops = this.operationLog
            .filter(log => log.timestamp > fromTimestamp)
            .map(log => log.op);
        
        return ops;
    }

    // 重连后同步
    async handleReconnect() {
        // 获取服务器上的最后修改时间
        const serverLastUpdate = await fetchServerLastUpdate();
        
        // 重放本地操作
        const localOps = this.replayOperations(serverLastUpdate);
        for (const op of localOps) {
            await sendToServer(op);
        }
    }
}
```

## 四、计算机网络基础（简历技能点延伸）

### 4.1 TCP/IP 与 HTTP 协议

**面试官可能这样问**：你提到熟悉 TCP/IP 和 HTTP 协议，能解释一下 TCP 三次握手和四次挥手吗？

**参考回答**：

**TCP 三次握手**

三次握手是建立 TCP 连接的过程，目的是同步双方的序列号：

```
客户端                    服务器
  |                         |
  |------- SYN = 1 -------->|
  |    seq = x (随机)       |
  |                         |
  |<---- SYN = 1, ACK = 1 --|
  |    seq = y, ack = x+1   |
  |                         |
  |------- ACK = 1 -------->|
  |    seq = x+1, ack = y+1 |
  |                         |
         连接建立
```

**各步骤详解**

1. **第一次握手（SYN）**：客户端发送 SYN 包（SYN=1），随机生成序列号 seq=x。此时客户端进入 SYN_SENT 状态。

2. **第二次握手（SYN+ACK）**：服务器收到 SYN 包后，发送 SYN+ACK 包（SYN=1, ACK=1），随机生成序列号 seq=y，确认号 ack=x+1。此时服务器进入 SYN_RCVD 状态。

3. **第三次握手（ACK）**：客户端收到服务器的 SYN+ACK 包后，发送 ACK 包（ACK=1），确认号 ack=y+1。此时客户端和服务器都进入 ESTABLISHED 状态。

**为什么需要三次？**

两次握手无法确认双方的接收和发送能力都正常：第二次握手只能确认服务器收到了客户端的 SYN，但客户端无法确认服务器是否准备好。只有第三次握手后，双方才能确认彼此的收发能力正常。

**TCP 四次挥手**

四次挥手是关闭 TCP 连接的过程：

```
客户端                    服务器
  |                         |
  |------- FIN = 1 -------->|
  |    seq = u              |
  |                         |<---- 继续发送数据 ----
  |<---- ACK = 1 -----------|---- seq = v, ack = u+1
  |    ack = u+1            |
  |                         |
  |<---- FIN = 1, ACK = 1 --|
  |    seq = w, ack = u+1   |
  |                         |
  |------- ACK = 1 -------->|
  |    seq = u+1, ack = w+1 |
  |                         |
         连接关闭
```

**为什么需要四次？**

因为 TCP 是全双工通信，关闭连接需要双方分别发送 FIN 包。服务器收到客户端的 FIN 后，可能还有数据要发送，所以先回复 ACK，等数据发送完毕后再发送 FIN。

### 4.2 HTTP 缓存机制

**面试官可能这样问**：HTTP 缓存是如何工作的？请介绍一下强缓存和协商缓存。

**参考回答**：

**HTTP 缓存分类**

```
┌─────────────────────────────────────────┐
│              HTTP 缓存                  │
├──────────────────┬──────────────────────┤
│    强缓存         │     协商缓存         │
│  (本地缓存)       │   (服务器验证)        │
├──────────────────┴──────────────────────┤
│  Cache-Control / Expires               │
│               vs                        │
│  ETag / Last-Modified                  │
└─────────────────────────────────────────┘
```

**强缓存**

强缓存阶段：浏览器直接使用本地缓存，不请求服务器。

```
响应头：
Cache-Control: max-age=3600        // 相对时间，秒
Expires: Wed, 26 Feb 2026 12:00:00 GMT  // 绝对时间
```

- Cache-Control 优先级高于 Expires
- 常见值：max-age、no-cache（需协商）、no-store（不缓存）

**协商缓存**

协商缓存阶段：浏览器发送请求，服务器判断资源是否变更，决定返回 304 还是新资源。

```
首次请求：
响应头：
ETag: "abc123"                      // 资源版本标识
Last-Modified: Wed, 26 Feb 2026 10:00:00 GMT

后续请求：
请求头：
If-None-Match: "abc123"             // 对应 ETag
If-Modified-Since: Wed, 26 Feb 2026 10:00:00 GMT  // 对应 Last-Modified

服务器响应：
- 资源未变更：304 Not Modified（使用缓存）
- 资源已变更：200 OK + 新资源
```

**ETag vs Last-Modified**

- ETag：基于内容生成的唯一标识，精度更高，能检测到秒级以下的变化
- Last-Modified：基于修改时间，精度为秒，精度较低

### 4.3 HTTPS 加密流程

**面试官可能这样问**：HTTPS 是如何保证通信安全的？

**参考回答**：

**HTTPS 核心：TLS/SSL 握手**

```
┌────────────────────────────────────────────────────────┐
│                   TLS 握手过程                         │
├────────────────────────────────────────────────────────┤
│ 1. Client Hello                                        │
│    - TLS 版本                                          │
│    - 加密套件列表                                       │
│    - 随机数                                            │
│                                                        │
│ 2. Server Hello                                        │
│    - 选定加密套件                                      │
│    - 随机数                                            │
│    - 服务器证书                                        │
│                                                        │
│ 3. 证书验证 + 密钥交换                                 │
│    - 客户端验证证书                                    │
│    - 生成预主密钥                                      │
│    - 用服务器公钥加密发送                              │
│                                                        │
│ 4. 握手完成                                            │
│    - 双方生成会话密钥                                  │
│    - 后续使用对称加密通信                              │
└────────────────────────────────────────────────────────┘
```

**证书验证流程**

```
1. 浏览器内置受信任 CA 根证书
2. 服务器证书由 CA 签发，包含：
   - 服务器公钥
   - 服务器域名
   - CA 数字签名
3. 浏览器验证：
   - 证书是否过期
   - 证书域名是否匹配
   - CA 签名是否有效
4. 验证通过后，提取公钥
```

**对称加密 vs 非对称加密**

- **非对称加密**（RSA）：公钥加密，私钥解密。用于握手阶段，安全性高但计算慢。
- **对称加密**（AES）：相同密钥加解密。用于握手后的数据传输，效率高。

- 这个方案的局限性是什么？

### 6.3 不会的问题如何应对

**策略一：诚实承认 + 思考方向**

"这个问题让我想到了……（你熟悉的知识点），虽然细节不同，但核心思路应该是相通的。"

**策略三：展现学习能力**

"这个问题我在项目中没有遇到，但根据我之前的学习，如果遇到类似问题，我的解决思路是……"
