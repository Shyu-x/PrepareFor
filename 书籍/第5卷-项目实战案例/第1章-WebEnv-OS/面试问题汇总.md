### 1.1 核心技术难点深度分析

### 1.1 窗口管理系统核心难点

#### 1.1.1 拖拽性能优化

**难点描述**:
在高频鼠标移动事件（ mousemove）中更新窗口位置，如果直接操作 DOM 会导致严重的性能问题。普通实现可能只有 30fps，使用优化后可以达到 60fps。

**技术挑战**:
1. **事件频率**: mousemove 事件每秒可触发数百次，每次都操作 DOM 会导致浏览器大量重排（reflow）和重绘（repaint）
2. **坐标计算**: 需要实时计算窗口位置，同时处理边界检测、对齐辅助线等功能
3. **状态同步**: 窗口位置变化需要同步到 Zustand store，影响其他组件（如任务栏预览）

**面试追问**:
- 为什么 transform 比 left/top 性能更好？
- 什么是 GPU 合成层？
- will-change 属性的副作用是什么？

**回答要点**:
- `transform` 不会触发布局重排，只触发合成
- GPU 合成层独立于主线程，不阻塞 JS 执行
- `will-change` 会占用额外内存，不能滥用

#### 1.1.2 窗口层级管理（Z-Index）

**面试追问**:
- 为什么不用 zIndex 管理，而用 transform？
- 如何处理两个窗口同时点击的竞态条件？

#### 1.1.3 窗口状态持久化

**难点描述**:
用户刷新页面后，窗口的位置、大小、最大化状态需要恢复。需要在性能和一致性之间取得平衡。

### 2.1 浏览器渲染原理与窗口系统

#### CSS Transform vs Position

| 属性 | 触发重排 | 触发重绘 | GPU加速 |
|------|----------|----------|---------|
| `left/top` | 是 | 是 | 否 |
| `transform` | 否 | 否 | 是 |

**浏览器渲染流程**:
1. **DOM 树构建** -> 2. **CSS 树构建** -> 3. **渲染树** -> 4. **布局(Layout)** -> 5. **绘制(Paint)** -> 6. **合成(Composite)**

使用 `left/top` 修改位置会触发步骤 4（布局），这是最耗时的操作。使用 `transform: translate()` 只触发步骤 6（合成），因为它在一个独立的合成层中运行。

**回答框架**:

2. 尝试的解决方案

**示例回答**:
> 在实现大文件上传时，遇到浏览器内存溢出的问题。最初尝试一次性读取整个文件，但 500MB 以上的文件会导致浏览器崩溃。
>
> 后来采用分块上传的方式，将文件切分为 1MB 的小块，逐块读取和上传。同时使用 IndexedDB 缓存已上传的分块，支持断点续传。
>
> 最终成功支持了 2GB 大文件的上传，而且上传中断后可以从断点继续。这个过程让我深刻理解了浏览器存储限制和流式处理的重要性。

### 8.3 3 分钟深挖版本

1. **业务场景与价值**
   - 一体化研发工作台：代码、文档、终端、协作在同一环境内闭环。
   - 降低环境成本：减少"本地装环境 + 团队对齐成本"。
   - 提升协作效率：统一权限、统一入口、统一审计。

2. **核心技术拆解**
   - 桌面层：`Desktop` + `Dock` + `Taskbar` + `WindowManager` 管理多应用窗口。
   - 内核层：`Kernel` 统一进程、文件系统、运行时、系统注册。
   - IDE 层：`IDELayout` 聚合编辑器、面板、搜索、调试与 Git。
   - 后端层：NestJS 模块化服务承接认证、文件、终端、协作、代理。

3. **关键难点与方案**
   - 难点 A：多窗口并发导致状态错乱。
     - 方案：窗口实体化 + 明确状态机（创建/聚焦/最小化/关闭）+ z-index 规则。
   - 难点 B：浏览器环境模拟 OS 能力边界。
     - 方案：内核抽象分层，运行时与文件系统能力可替换。
   - 难点 C：IDE 与桌面互相耦合导致迭代慢。
     - 方案：store 按域拆分、组件职责下沉、协议稳定化。

4. **结果与复盘**
   - 系统稳定性从"功能可跑"提升到"异常可控、可恢复"。
   - 问题定位路径清晰，能在高压演示场景快速止血。
   - 后续可演进为多租户、企业权限、审计合规能力。
