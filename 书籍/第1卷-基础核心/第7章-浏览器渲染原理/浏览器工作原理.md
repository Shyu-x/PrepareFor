### 1.2 关键渲染阶段详解

#### 阶段一：DOM 与 CSSOM 构建

**DOM（Document Object Model）构建：**
- 浏览器将 HTML 字节流转换为 Token（标记）
- Token 解析为节点（Node）
- 节点构建为 DOM 树
- 解析过程中遇到 `<script>` 会阻塞（除非 async/defer）

**CSSOM（CSS Object Model）构建：**
- 类似 DOM 构建过程
- CSS 解析不阻塞 DOM 构建，但会阻塞渲染
- 外部 CSS 必须加载完成才能进入下一阶段

#### 阶段二：渲染树（Render Tree）构建

渲染树 = DOM 树 + CSSOM 树

**特点：**
- 只包含可见节点（`display: none` 的元素不包含）
- 每个节点包含 CSS 样式信息
- 与 DOM 节点一一对应

#### 阶段三：布局（Layout/Reflow）

**什么是重排（Reflow）：**
- 计算每个节点的几何信息（位置、尺寸）
- 从根节点遍历渲染树
- 是性能消耗最大的操作

**导致重排的操作：**
- 页面首次渲染
- 浏览器窗口大小变化
- 元素尺寸、位置变化
- 添加/删除 DOM 元素
- 读取某些布局属性（offsetWidth, offsetHeight 等）

#### 阶段四：绘制（Paint）

**什么是重绘（Repaint）：**
- 将布局信息转换为像素
- 不改变元素位置，只改变外观
- 比重排消耗小，但仍影响性能

**导致重绘的操作：**
- 颜色、背景变化
- 边框圆角变化
- 阴影变化

#### 阶段五：合成（Composite）

**合成层（Compositing Layers）：**
- 拥有独立层的元素可以独立渲染
- 不会影响其他元素

**触发合成层的属性：**
```css
transform: translateZ(0);
transform: rotate(45deg);
will-change: transform;
opacity: 0.5;
position: fixed;
```

### 1.3 关键渲染路径（Critical Rendering Path）

关键渲染路径是指浏览器从接收 HTML、CSS、JavaScript 到渲染出页面的过程。

**优化关键渲染路径可以显著提升首屏加载速度。**

```
关键路径优化策略：

1. 最小化阻塞资源
   - CSS 放在 <head> 中，尽早加载
   - JavaScript 使用 defer/async
   - 关键 CSS 内联

2. 减少关键资源数量
   - 合并 CSS/JS 文件
   - 代码分割按需加载

3. 减少关键路径长度
   - 减少依赖层级
   - 避免同步 XHR
```

### 1.4 HTML、CSS、JavaScript 的相互作用

浏览器渲染过程中，HTML、CSS、JavaScript 三者相互配合又相互影响。

#### 1.4.1 渲染阻塞关系

```
┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓ 
┃                      渲染阻塞关系图                                      ┃
┃                                                                         ┃ 
┃   HTML 解析                                                            ┃  
┃       ┃                                                                ┃  
┃       ┣━━ 遇到 <link CSS> ━━▶ 异步下载，不阻塞 HTML 解析              ┃   
┃       ┃                  (但会阻塞渲染)                                  ┃
┃       ┃                                                                ┃  
┃       ┗━━ 遇到 <script> ━━▶ 停止解析，等待 JS 下载并执行              ┃   
┃                              (因为 JS 可能修改 DOM/CSSOM)                ┃
┃                                                                         ┃ 
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛ 

```

**关键点：**

1. **CSS 不会阻塞 HTML 解析，但会阻塞渲染**
   - 浏览器有预解析线程，专门用于下载 CSS
   - 但必须等 CSSOM 构建完成才能开始渲染

2. **JavaScript 会阻塞 HTML 解析**
   - 因为 JS 可能修改 DOM 树或 CSSOM 树
   - DOM 构建必须暂停，等待 JS 执行完成

#### 1.4.2 script 标签的 async 与 defer

```html
<!-- 1. 普通 script：阻塞 HTML 解析 -->
<script src="jquery.js"></script>

<!-- 2. async：异步加载，立即执行，仍会阻塞渲染 -->
<script async src="jquery.js"></script>

<!-- 3. defer：异步加载，HTML 解析完成后执行 -->
<script defer src="jquery.js"></script>
```

**执行时机对比：**

| 特性 | 普通 script | async | defer |
|------|-------------|-------|-------|
| 下载时机 | 阻塞 | 异步 | 异步 |
| 执行时机 | 阻塞解析 | 立即执行 | 解析完成后 |
| 顺序 | 不保证 | 不保证 | 按顺序 |
| DOMContentLoaded | 阻塞 | 可能之前 | 之后 |

#### 1.4.3 DOM 操作对渲染的影响

**JavaScript 修改 DOM 会触发连锁反应：**

```
JS 修改 DOM
     ┃
     ▼
DOM 树变化 ━━▶ 重新构建渲染树 ━━▶ 重新布局 ━━▶ 重绘 ━━▶ 合成
```

**优化技巧：**

```javascript
// 1. 使用 DocumentFragment 批量操作 DOM
const fragment = document.createDocumentFragment();
for (let i = 0; i < 1000; i++) {
  const div = document.createElement('div');
  div.textContent = i;
  fragment.appendChild(div);
}
document.body.appendChild(fragment);

// 2. 使用 CSS class 批量修改
element.classList.add('active', 'visible', 'animated');
```

## 第2章 Core Web Vitals 性能指标

Google 提出的 Core Web Vitals 是衡量用户体验的核心指标：

| 指标 | 全称 | 含义 | 良好阈值 | 优化目标 |
|------|------|------|----------|----------|
| LCP | Largest Contentful Paint | 最大内容绘制 | <=2.5s | 首屏加载速度 |
| FID | First Input Delay | 首次输入延迟 | <=100ms | 交互响应速度 |
| CLS | Cumulative Layout Shift | 累积布局偏移 | <=0.1 | 视觉稳定性 |

### LCP 优化方案

```html
<!-- 1. 预加载关键资源 -->
<link rel="preload" href="hero.jpg" as="image">

<!-- 2. 预连接关键域名 -->
<link rel="preconnect" href="https://cdn.example.com">

<!-- 3. 使用现代图片格式 -->
<picture>
  <source srcset="image.avif" type="image/avif">
  <source srcset="image.webp" type="image/webp">
  <img src="image.jpg" alt="Hero">
</picture>
```

### FID 优化方案

```javascript
// 1. 拆分长任务
function processBigTask() {
  const items = getAllItems();
  const chunkSize = 10;

  function processChunk(start) {
    const end = Math.min(start + chunkSize, items.length);
    for (let i = start; i < end; i++) {
      processItem(items[i]);
    }
    if (end < items.length) {
      requestIdleCallback(() => processChunk(end));
    }
  }
  requestIdleCallback(() => processChunk(0));
}

// 2. 使用 Web Worker 处理计算密集任务
const worker = new Worker('heavy-task.js');
worker.postMessage(data);
worker.onmessage = (e) => console.log(e.data);
```

### CLS 优化方案

```html
<!-- 1. 为图片和视频指定尺寸 -->
<img src="image.jpg" width="800" height="600" alt="...">

<!-- 2. 为插入内容预留空间 -->
<div id="ads-container" style="min-height: 250px;">
  <!-- 广告加载后不会影响下方内容 -->
</div>

<!-- 3. 使用 font-display 避免字体加载闪烁 -->
<style>
  @font-face {
    font-family: 'CustomFont';
    src: url('font.woff2') format('woff2');
    font-display: swap;
  }
</style>
```

## 第3章 浏览器渲染优化实战

### 使用 Chrome DevTools Performance 面板分析

1. 打开 Chrome DevTools (F12)
2. 切换到 Performance 面板
3. 点击录制按钮
4. 执行操作后停止
5. 分析各阶段耗时

### 性能优化技巧

| 优化手段 | 作用 | 示例 |
|---------|------|------|
| 避免重排 | 减少布局计算 | 使用 transform 代替 left/top |
| 使用合成层 | GPU 加速 | transform: translateZ(0) |
| 减少重绘 | 降低绘制开销 | 使用 CSS3 动画 |
| will-change | 提前创建层 | will-change: transform |
| content-visibility | 懒渲染 | content-visibility: auto |
